# JVM 내부 구조와 Garbage Collection 저술 계획

## 제목 후보
1. **JVM을 알아야 서버가 산다** — 톤: 실무 중심의 직설적 제목, 포지셔닝: "왜 알아야 하는가"에 대한 동기 부여형. 백엔드 개발자의 위기감과 호기심을 동시에 자극한다.
2. **GC를 멈추면 서버도 멈춘다** — 톤: 기술적 위트가 담긴 경고형, 포지셔닝: Stop-The-World라는 GC의 핵심 개념을 제목에 녹여 기술적 깊이를 암시한다. GC에 이미 관심이 있는 독자에게 강하게 어필한다.
3. **백엔드 개발자를 위한 JVM 깊이 보기** — 톤: 차분하고 신뢰감 있는 기술서 제목, 포지셔닝: 대상 독자를 명확히 하고, "깊이 보기"로 입문서 이상의 가치를 약속한다. 시리즈화에도 유리하다.

**추천:** "JVM을 알아야 서버가 산다" — 백엔드 개발자라는 타깃에게 가장 직접적으로 와닿는 제목이다. "서버가 산다"는 표현이 실무적 가치를 명확히 전달하며, 토비 스타일의 실용주의적 톤과도 잘 맞는다. 주니어~미드 레벨 개발자가 서점이나 목록에서 "이건 나를 위한 책이구나"라고 즉시 인식할 수 있다.

## 책 특성
- **장르:** 실무 중심 기술서 (에세이형 기술서에 가까운 서술 방식)
- **분량:** 약 80페이지 (약 56,000자, 페이지당 약 700자 기준)
- **난이도:** 초중급 (Java/Kotlin으로 서버 개발 경험이 있는 주니어~미드 레벨)
- **독자 여정:** "JVM? GC? 면접용으로만 알고 있다" → "프로덕션에서 GC 로그를 읽고, 적절한 GC를 선택하고, 문제 상황을 진단할 수 있다"

## 내러티브 아크

이 책은 "왜?"에서 출발하여 "어떻게?"를 거쳐 "직접 해보자"로 착지하는 구조를 따른다.

1장은 독자의 일상에서 시작한다. API 응답 시간이 갑자기 튀고, 서버가 OOM으로 죽고, 배포 후 CPU가 치솟는 상황. 이 모든 문제의 뿌리에 JVM과 GC가 있다는 것을 보여주며, "왜 백엔드 개발자가 JVM 내부를 알아야 하는가"라는 질문에 답한다. JVM이 무엇인지 30초 안에 훑되, 기술적 상세는 다음 장에 맡긴다. 이것이 책 전체의 동기 부여다.

2장에서 JVM의 내부 구조를 본격적으로 펼친다. 바이트코드와 JIT 컴파일러가 코드를 실행하는 방식을 먼저 짚고, 이어서 힙, 스택, 메서드 영역이 각각 어떤 역할을 하는지, 코드에서 생성한 객체가 메모리 어디에 놓이는지를 실제 코드와 함께 추적한다. 1장에서 가볍게 건넨 조감도를 여기서 정밀 지도로 확장하는 셈이다.

3장은 GC의 핵심 원리를 다룬다. "쓰레기를 어떻게 찾고, 어떻게 치우는가"라는 단순한 질문에서 출발하여 Mark-and-Sweep, 세대별 수집, Stop-The-World라는 세 가지 핵심 개념을 단단히 잡는다. GC가 왜 필요한지, 왜 완벽할 수 없는지를 이해하는 장이다.

4장에서 실전의 문을 연다. CMS의 한계가 왜 G1을 불러왔는지 역사적 맥락을 짚은 뒤, G1부터 ZGC, Shenandoah, Generational ZGC까지 각 알고리즘을 비교한다. Generational ZGC는 이 장에서 알고리즘적 특성과 성능 데이터를 충분히 다룬다. 장 말미에는 "G1이면 충분한가?"라는 독자의 예상 질문에 정면으로 답하며, 워크로드별 의사결정 프레임워크를 제시한다.

5장은 GC 모니터링과 튜닝의 실전이다. 도구를 나열하는 대신, "GC 로그 수집 → GCeasy로 병목 식별 → JFR로 상세 분석"이라는 3단계 실습 워크플로우를 중심으로 구성한다. 측정 결과를 바탕으로 힙 사이징과 JVM 옵션을 조정하는 구체적 절차를 안내하고, Uber와 LinkedIn의 튜닝 사례로 마무리한다.

6장은 프로덕션의 전쟁터를 다룬다. OOM 에러 유형별 진단과 해결, 메모리 릭 추적을 실전 트러블슈팅 형식으로 풀어낸 뒤, 별도 섹션으로 컨테이너 환경을 집중 조명한다. OOM Killer, UseContainerSupport, 힙+비힙 메모리 합산 문제를 독립적으로 다루어, 2026년 현재 대부분의 서비스가 운영되는 컨테이너 환경에서의 실수를 방지한다.

7장은 시선을 미래로 돌린다. Virtual Threads가 GC에 미치는 영향, GraalVM 네이티브 이미지의 제약, 자동 힙 사이징 JEP와 ML 기반 자동 튜닝의 가능성, 그리고 기본 GC 전환 로드맵까지. Generational ZGC의 알고리즘적 설명은 4장에서 끝냈으므로, 여기서는 "앞으로 GC가 어디로 가는가"라는 방향성에만 집중한다.

전체적으로 각 장은 이전 장의 지식 위에 한 층씩 쌓이면서도, 4장 이후의 실전 챕터는 독립적으로 참고할 수 있도록 설계한다. "동기(1장) → 구조 이해(2장) → 원리(3장) → 선택과 비교(4장) → 모니터링과 튜닝(5장) → 트러블슈팅(6장) → 미래 전망(7장)"이라는 흐름이 자연스럽게 독자의 역량을 끌어올린다.

## 챕터 목록

### 1장. 왜 백엔드 개발자가 JVM을 알아야 하는가
- **핵심 질문:** JVM 내부를 몰라도 서버 개발은 되지 않나? 왜 굳이 알아야 하는가?
- **주요 내용:**
  - 프로덕션에서 만나는 JVM 관련 장애 시나리오 (응답 지연 스파이크, OOM 크래시, CPU 폭주)
  - "GC가 뭔지 모르면서 GC 때문에 서버가 죽는" 아이러니
  - JVM 30초 오버뷰: 클래스 로더 → 런타임 데이터 영역 → 실행 엔진 (기술 상세는 2장으로)
  - 이 책에서 다룰 범위와 읽는 방법 안내
- **독자가 얻는 것:** JVM 내부를 배울 명확한 동기와 전체 그림에 대한 조감도
- **예상 분량:** 6페이지 (약 4,200자)

### 2장. JVM의 구조 — 바이트코드에서 메모리까지
- **핵심 질문:** Java/Kotlin 코드가 어떻게 실행되고, 내가 만든 객체는 메모리 어디에 저장되는가?
- **주요 내용:**
  - 바이트코드와 JIT 컴파일러: 소스 코드가 실행되기까지의 여정 (1장에서 넘겨받은 내용)
  - 힙(Heap): 객체가 태어나고 살고 죽는 곳. Young/Old Generation 구분과 이유
  - 스택(Stack): 메서드 호출의 역사가 쌓이는 곳. 스레드별 독립 존재의 의미
  - 메서드 영역과 Metaspace: 클래스 메타데이터의 집. PermGen에서 Metaspace로의 전환
  - 변수 유형별 저장 위치 정리 (지역 변수 → 스택, 인스턴스 변수 → 힙, 정적 변수 → 메서드 영역)
  - 간단한 코드 예제로 메모리 배치 추적해보기
- **독자가 얻는 것:** JVM 실행 모델에 대한 이해와 메모리의 전체 지도. "어떤 코드가 어디에 영향을 주는지" 감각
- **예상 분량:** 12페이지 (약 8,400자)

### 3장. Garbage Collection의 원리 — 쓰레기는 어떻게 찾아 치우는가
- **핵심 질문:** GC는 어떻게 "더 이상 쓰지 않는 객체"를 판별하고 회수하는가?
- **주요 내용:**
  - Reference Counting의 아이디어와 한계 (순환 참조 문제)
  - Mark-and-Sweep: GC Root에서 시작하는 추적 탐색. "닿을 수 있으면 살리고, 닿을 수 없으면 치운다"
  - Mark-Sweep-Compact: 단편화 해결을 위한 압축 단계
  - 세대별 GC와 약한 세대 가설: "대부분의 객체는 빨리 죽는다"는 통찰
  - Eden, Survivor, Old 영역의 객체 이동 흐름
  - Stop-The-World: GC가 피할 수 없는 대가. 왜 모든 스레드를 멈춰야 하는가
  - Minor GC vs Major/Full GC의 차이와 발생 조건
- **독자가 얻는 것:** GC의 동작 원리에 대한 직관적 이해. "GC 로그에 나오는 용어가 뭘 뜻하는지" 알게 된다
- **예상 분량:** 12페이지 (약 8,400자)

### 4장. GC 알고리즘 선택 가이드 — 내 서비스에는 어떤 GC가 맞는가
- **핵심 질문:** Serial, Parallel, G1, ZGC, Shenandoah 중 내 워크로드에 최적인 GC는 무엇인가?
- **주요 내용:**
  - GC 성능 삼각형: 지연(Latency) vs 처리량(Throughput) vs 메모리(Footprint) — 셋 다 잡을 수 없다
  - CMS의 등장과 퇴장: Concurrent Mark Sweep이 해결한 것과 해결하지 못한 것 (deprecated 배경, G1 등장의 맥락)
  - Serial GC: 단일 스레드의 단순함, 소형 컨테이너에서의 가치
  - Parallel GC: 배치 처리의 왕, 처리량 극대화 전략
  - G1 GC: 기본값의 이유. 리전 기반 증분 수집의 장점과 한계
  - ZGC와 Generational ZGC: Colored Pointers, Load Barriers, 그리고 세대별 수집 도입으로 메모리 75% 감소 + 처리량 4배 향상의 실측 데이터
  - Shenandoah: Brooks Pointers와 write-barrier 접근의 차이
  - 워크로드별 GC 선택 의사결정 트리 (마이크로서비스, 데이터 파이프라인, 초저지연 서비스 등)
  - Netflix의 Generational ZGC 마이그레이션 사례
  - "G1이면 충분한가?" — 대부분의 서비스에서 G1이 적정선인 이유와, ZGC/Shenandoah로 넘어가야 하는 분기점
- **독자가 얻는 것:** CMS에서 ZGC까지의 진화 맥락을 이해하고, 자기 서비스의 특성에 맞는 GC를 선택할 수 있는 판단 기준과 근거
- **예상 분량:** 16페이지 (약 11,200자)

### 5장. GC 모니터링과 튜닝 — 측정하고, 이해하고, 조정하라
- **핵심 질문:** GC 성능이 부족할 때, 무엇을 측정하고 어떤 순서로 튜닝해야 하는가?
- **주요 내용:**
  - 튜닝의 제1원칙: "기본값으로 시작하고, 측정 후에만 튜닝한다"
  - 3단계 모니터링 실습 워크플로우:
    - Step 1: GC 로그 수집 — `-Xlog:gc*` 활성화와 출력 해석법
    - Step 2: GCeasy 분석 — 로그 업로드 후 병목 지점 식별 (pause 분포, 승격 패턴, 할당률)
    - Step 3: JFR 심층 분석 — Java Flight Recorder로 GC 이벤트, 할당 핫스팟, 스레드 상태 상세 추적
  - 핵심 JVM 옵션 해부: -Xms/-Xmx, MaxGCPauseMillis, IHOP
  - 힙 사이징 가이드라인: 전체 힙, Young/Old 비율
  - 코드 최적화 vs GC 튜닝: 어디에 먼저 투자할 것인가 (Uber의 1ms backoff 버그 사례)
  - Uber HDFS NameNode 튜닝 사례: GC 시간 22초 → 1.5초
  - LinkedIn의 OS 수준 GC 최적화 사례: AlwaysPreTouch, vm.swappiness
- **독자가 얻는 것:** 도구 나열이 아닌 실습 가능한 모니터링 워크플로우와 GC 튜닝의 구체적 절차
- **예상 분량:** 14페이지 (약 9,800자)

### 6장. 프로덕션 트러블슈팅 — 서버가 죽었을 때 살리는 법
- **핵심 질문:** OOM, GC 루프, 메모리 릭이 발생했을 때 어떻게 원인을 찾고 해결하는가?
- **주요 내용:**
  - OOM 에러 유형별 진단 가이드: heap space, GC overhead limit, Metaspace, native thread
  - 힙 덤프 분석 기초: HeapDumpOnOutOfMemoryError 설정과 분석 도구
  - 메모리 릭 패턴: 정적 컬렉션, 리스너 미해제, 캐시 무한 성장
  - Netflix의 jvmquake: GC 부채 모델로 GC 루프를 사전 차단하는 접근
  - 프로덕션 OOM 사례: IHOP 설정 오류로 하루 3회 OOM 발생과 해결
  - **[섹션] 컨테이너 환경에서의 JVM 메모리 관리**
    - OOM Killer의 작동 원리: 커널이 보는 메모리와 JVM이 보는 메모리의 괴리
    - UseContainerSupport와 MaxRAMPercentage: 컨테이너 메모리 제한을 JVM에 전달하는 법
    - 힙 + 비힙(Metaspace, 스레드 스택, 네이티브 메모리) 합산 계산: 컨테이너 limit의 70~80% 규칙
    - 컨테이너 환경 메모리 설정 체크리스트
  - 트러블슈팅 체크리스트: "서버가 죽었다" 신고를 받았을 때 확인할 순서
- **독자가 얻는 것:** 프로덕션 장애 상황에서 당황하지 않고 원인을 추적하는 체계적 접근법. 컨테이너 환경 특유의 함정을 사전에 방지하는 실무 지식
- **예상 분량:** 12페이지 (약 8,400자)

### 7장. JVM의 내일 — Virtual Threads, 자동 튜닝, 그리고 그 너머
- **핵심 질문:** JVM과 GC 기술은 어디로 향하고 있으며, 지금 무엇을 준비해야 하는가?
- **주요 내용:**
  - Virtual Threads와 GC의 관계: GC pause 14회 → 2회의 의미와 한계
  - Java 25 이후의 GC 로드맵: G1 기본 유지, ZGC 기본화 가능성, 자동 힙 사이징 JEP의 방향
  - GraalVM 네이티브 이미지: Serial GC/G1만 지원하는 제약, 런타임 컴파일 활성화 시 가상 스레드 미지원 등 구체적 한계
  - ML 기반 GC 자동 튜닝: "플래그 제로" GC를 향한 움직임
  - 백엔드 개발자가 지금 취해야 할 액션: JDK 버전 업그레이드 전략, GC 전환 판단 기준
  - 이 책에서 배운 것의 정리와 다음 학습 방향 안내
- **독자가 얻는 것:** JVM/GC 기술의 방향성에 대한 감각과 커리어에서 활용할 구체적 액션 아이템
- **예상 분량:** 8페이지 (약 5,600자)
