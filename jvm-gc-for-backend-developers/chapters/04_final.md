# 4장. GC 알고리즘 선택 가이드 — 내 서비스에는 어떤 GC가 맞는가

금요일 오후, 팀 리더가 슬랙에 메시지를 남긴다. "우리 서비스 P99 응답 시간이 간헐적으로 2초를 넘기는데, GC 때문인 것 같다. 다음 주까지 GC 바꿔보고 결과 공유해줘." 막막하다. Serial, Parallel, G1, ZGC, Shenandoah — 이름은 들어봤지만, 어떤 기준으로 골라야 하는지 감이 안 잡힌다. 구글링하면 각 GC의 구조를 설명하는 글은 넘쳐나지만, 정작 "내 서비스에는 뭘 써야 하는가"에 답하는 글은 찾기 어렵다.

이번 장에서는 각 GC 알고리즘을 학술적으로 나열하는 대신, 실무에서 마주치는 워크로드 유형별로 어떤 GC가 왜 적합한지를 따져본다. 3장에서 살펴본 GC의 원리가 실제 알고리즘에서 어떻게 다르게 구현되는지, 그 차이가 우리 서비스에 어떤 영향을 주는지 함께 살펴보자.

## GC 성능 삼각형: 셋 다 잡을 수는 없다

GC를 선택하기 전에 하나 분명히 해두자. GC 성능에는 세 가지 축이 있다.

- **지연(Latency)**: GC가 애플리케이션을 멈추는 시간. Stop-The-World pause가 짧을수록 좋다.
- **처리량(Throughput)**: 전체 실행 시간 중 애플리케이션이 실제 일을 하는 비율. GC에 빼앗기는 시간이 적을수록 좋다.
- **메모리(Footprint)**: GC가 제대로 작동하기 위해 필요한 추가 메모리. 적을수록 좋다.

문제는 이 세 가지를 동시에 최적화하는 것이 불가능하다는 점이다. 둘을 선택하면 나머지 하나는 반드시 희생된다. Parallel GC가 처리량을 극대화하는 대신 긴 pause를 감수하는 것, ZGC가 1ms 미만의 pause를 달성하는 대신 20~30%의 메모리 오버헤드를 요구하는 것 — 전부 이 삼각형의 법칙을 따른다.

그렇다면 우리가 해야 할 일은 명확하다. 내 서비스에서 가장 중요한 두 가지가 무엇인지 먼저 정하는 것이다. 이걸 정하지 않고 GC를 고르는 건, 목적지 없이 지도를 펼치는 것과 다를 바 없다.

## CMS의 등장과 퇴장: 역사에서 배우는 것

본격적으로 현재의 GC들을 살펴보기 전에, 한때 "저지연 GC의 대명사"였던 CMS(Concurrent Mark Sweep)를 잠깐 돌아보자. CMS가 왜 등장했고 왜 퇴장했는지를 이해하면, 현재 GC들의 설계 철학이 훨씬 선명하게 보인다.

CMS는 대부분의 마킹 작업을 애플리케이션 스레드와 동시에 수행했다. 당시로서는 혁신적이었다. STW 시간을 획기적으로 줄였으니까. 하지만 CMS에는 치명적인 문제가 있었다. 메모리 단편화다. CMS는 Sweep만 하고 Compact를 하지 않았기 때문에, 오래 돌릴수록 힙 안에 자잘한 빈 공간이 흩어졌다. 마치 책장에 책을 빼고 꽂기를 반복하다 보면 빈 칸이 군데군데 생기는 것과 비슷하다. 결국 연속된 큰 공간이 필요한 객체를 할당할 수 없게 되면, CMS는 어쩔 수 없이 Full GC를 발동했다. 이때의 pause는 CMS를 쓰지 않는 것보다 더 길어지기도 했다.

CPU 오버헤드도 문제였다. 동시 수집을 위해 애플리케이션과 GC가 CPU를 나눠 써야 했으니, 처리량이 눈에 띄게 떨어졌다. 결국 CMS는 Java 9에서 deprecated되고 Java 14에서 완전히 제거되었다.

CMS의 실패가 남긴 교훈은 두 가지다. 첫째, 동시 수집은 올바른 방향이지만, 단편화를 해결하지 않으면 결국 무너진다. 둘째, CPU 오버헤드를 무시할 수 없다. G1은 바로 이 두 가지 교훈 위에 설계되었다.

## Serial GC: 단순함이 무기가 되는 순간

Serial GC를 무시하는 개발자가 많다. 단일 스레드로 GC를 수행하니 당연히 느릴 거라고 생각한다. 대부분의 경우 그 판단이 맞다. 하지만 "대부분"이 아닌 경우가 있다.

힙이 100MB 미만이고 CPU가 1~2개뿐인 소형 컨테이너를 생각해보자. 요즘 쿠버네티스 환경에서 사이드카 컨테이너나 경량 유틸리티 서비스를 이런 스펙으로 띄우는 경우가 드물지 않다. 이런 환경에서 G1이나 ZGC를 쓰면 오히려 난감해진다. 멀티스레드 GC는 여러 스레드를 조율하는 데 오버헤드가 발생하는데, CPU가 1개뿐이면 그 조율 비용이 실제 수집 비용보다 클 수 있다. G1의 리전 관리 메타데이터, ZGC의 Colored Pointer 자료구조 — 이런 것들이 100MB 힙에서는 배보다 배꼽이 커진다.

Serial GC는 `-XX:+UseSerialGC` 한 줄이면 된다. 구조가 단순하니 예측 가능하고, 메모리 오버헤드가 최소다. 작은 힙에서는 pause 자체가 수 밀리초에 불과하니 STW가 문제되지도 않는다.

기억해두자. GC 선택은 "가장 좋은 GC"를 고르는 게 아니라 "내 환경에 맞는 GC"를 고르는 것이다. 리소스가 극히 제한된 컨테이너라면 Serial GC가 정답일 수 있다.

## Parallel GC: 배치 처리의 왕

Parallel GC는 이름 그대로 여러 스레드가 동시에 GC를 수행한다. STW는 여전히 발생하지만, 여러 스레드가 달려들어 일을 처리하니 전체 GC 시간은 짧아진다. CPU 코어 수에 비례해서 성능이 확장되기 때문에, 코어가 넉넉한 서버에서 빛을 발한다.

어떤 상황에서 Parallel GC가 적합할까? 야간 배치 작업을 떠올려보자. 수백만 건의 데이터를 읽어서 변환하고 적재하는 ETL 파이프라인. 이런 워크로드에서 중요한 건 개별 요청의 응답 시간이 아니라, 전체 작업이 얼마나 빨리 끝나느냐다. GC pause가 500ms든 1초든, 전체 처리 시간이 줄어들면 그만이다.

Apache Spark나 Flink 같은 데이터 처리 프레임워크를 쓴다면 Parallel GC를 먼저 고려하는 편이 낫다. 이런 시스템은 대량의 데이터를 빠르게 처리하는 데 최적화되어 있고, GC도 그 철학에 맞춰야 한다. `-XX:+UseParallelGC`로 설정하면 된다.

물론 Parallel GC에도 찜찜한 지점이 있다. pause 시간을 예측하기 어렵다는 것이다. 힙이 커질수록 Full GC 시간이 비례적으로 늘어나기 때문에, 실시간 API 서버에 쓰면 간헐적으로 수 초짜리 멈춤이 발생할 수 있다. 사용자가 기다리고 있는 서비스라면, Parallel GC는 피하는 것이 바람직하다.

## G1 GC: 기본값이 된 데에는 이유가 있다

G1(Garbage-First)은 JDK 9부터 기본 GC로 채택되었다. 그리고 Java 25(JEP 523)에서는 이전에 Serial로 폴백하던 소형 환경까지 포함해 모든 환경의 기본 GC로 확대된다. 왜 이렇게까지 밀어주는 걸까?

G1의 핵심 아이디어는 힙을 고정 크기의 리전(Region)으로 나누는 것이다. 전통적인 GC가 Young/Old를 거대한 연속 영역으로 관리했다면, G1은 힙을 수백~수천 개의 작은 리전으로 쪼개고, 각 리전에 Young/Old/Humongous 같은 역할을 동적으로 부여한다.

이렇게 하면 뭐가 좋을까? 전체 힙을 한꺼번에 수집하지 않아도 된다. G1은 쓰레기가 가장 많은 리전부터 우선적으로 수집한다 — 그래서 이름이 "Garbage-First"다. 이 증분 수집(incremental collection) 덕분에 pause 시간을 목표치 이내로 제어할 수 있다. `-XX:MaxGCPauseMillis=200`으로 목표를 설정하면, G1은 그 시간 안에 수집할 수 있는 만큼만 리전을 선택해서 처리한다.

G1의 강점은 균형에 있다. 처리량이 Parallel GC에 미치지는 못하지만 크게 뒤지지도 않고, 지연 시간이 ZGC만큼 낮지는 않지만 대부분의 서비스에서 허용 가능한 수준이다. 메모리 효율성은 ZGC나 Shenandoah보다 우수하다. 추가 메타데이터가 상대적으로 적기 때문이다.

물론 G1에도 한계가 있다. 동시 마킹 단계를 포함하지만, 대부분의 실제 수집 작업은 여전히 STW로 이루어진다. 힙이 수십 GB를 넘어가면 pause 시간이 수백 ms까지 올라갈 수 있고, remembered set 처리 과정에서 간헐적인 pause 스파이크가 발생하기도 한다. P99 응답 시간이 50ms 이내여야 하는 초저지연 서비스에서는 G1으로 충분하지 않을 수 있다.

그래도 대부분의 마이크로서비스, 웹 API, 일반적인 Spring Boot 애플리케이션이라면 G1이 가장 무난하고 안전한 선택이다. 별도 설정 없이 `-XX:+UseG1GC`만으로 — 사실 JDK 9 이후라면 이것조차 생략 가능하다 — 합리적인 성능을 얻을 수 있다. 이게 "기본값"이 된 이유다.

## ZGC와 Generational ZGC: 1ms의 약속

G1으로 부족한 순간이 온다면, 다음 선택지는 ZGC다. Oracle이 개발한 ZGC는 단 하나의 목표를 향해 설계되었다. 힙 크기와 무관하게 pause time을 1ms 미만으로 유지하는 것.

이게 어떻게 가능할까? ZGC는 두 가지 핵심 메커니즘을 사용한다.

**Colored Pointers**는 64비트 객체 참조의 상위 비트에 메타데이터를 인코딩하는 기법이다. 일반적으로 객체 참조는 메모리 주소만 담고 있지만, ZGC는 여기에 Marked0, Marked1, Remapped, Finalizable 같은 상태 정보를 함께 실어 보낸다. 덕분에 객체의 GC 상태를 별도 자료구조 조회 없이 참조 자체만으로 확인할 수 있다.

**Load Barriers**는 객체 참조를 로드할 때마다 실행되는 작은 코드 조각이다. GC가 객체를 재배치하고 있는 도중에도, 애플리케이션이 해당 객체를 읽으려 하면 Load Barrier가 참조를 자동으로 새 위치로 수정해준다. 이 메커니즘 덕분에 ZGC는 객체 재배치를 애플리케이션과 동시에(concurrently) 수행할 수 있다. 재배치를 위해 세상을 멈출 필요가 없는 것이다.

그런데 초기 ZGC에는 아쉬운 점이 있었다. 세대(Generation) 구분이 없었다. 3장에서 살펴본 "대부분의 객체는 금방 죽는다"는 세대 가설을 활용하지 않고, 모든 객체를 동일하게 취급했다. 이 때문에 메모리 사용량이 높고, 단기 객체를 처리하는 효율이 떨어졌다.

JDK 21에서 등장한 **Generational ZGC**가 이 문제를 해결했다. Young과 Old 세대를 도입하면서, 비세대 ZGC 대비 놀라운 수치를 보여주었다.

- 메모리 사용량 75% 감소
- 처리량 4배 향상
- pause time 1ms 미만은 그대로 유지

Apache Cassandra 테스트에서는 275개 동시 클라이언트 환경에서도 일관된 pause time을 유지했다. 비세대 ZGC는 같은 조건에서 성능이 눈에 띄게 저하되었다. JDK 23부터는 Generational 모드가 ZGC의 기본으로 전환되었으니, 이제 ZGC를 쓴다면 자연스럽게 Generational ZGC를 쓰게 된다.

다만 ZGC의 메모리 오버헤드는 여전히 주의가 필요하다. Colored Pointers와 동시 수집 자료구조를 위해 20~30%의 추가 메모리가 필요하다. compressed oops도 지원하지 않는다. 힙 4GB짜리 서비스에 ZGC를 적용하면 실질적으로 5~5.2GB의 메모리가 필요해질 수 있다. 메모리가 빠듯한 환경이라면 이 오버헤드가 부담스럽다.

ZGC 설정은 `-XX:+UseZGC`로 간단하다. JDK 21 이상이라면 Generational 모드가 기본이므로 추가 플래그가 필요 없다.

## Shenandoah: 또 다른 저지연의 길

ZGC만이 저지연 GC의 유일한 선택지는 아니다. Red Hat이 개발한 Shenandoah도 비슷한 목표 — 10ms 미만의 일관된 pause time — 를 추구하지만, 접근 방식이 다르다.

ZGC가 Load Barrier(읽기 시점 배리어)를 사용한다면, Shenandoah는 **Brooks Pointers**와 write-barrier(쓰기 시점 배리어) 중심으로 동작한다. Brooks Pointer는 각 객체에 추가되는 포인터 필드로, 항상 해당 객체의 "현재 위치"를 가리킨다. 객체가 재배치되면 이 포인터만 새 위치로 갱신하면 되니, 애플리케이션은 항상 Brooks Pointer를 통해 투명하게 객체에 접근할 수 있다.

이 설계 차이가 성능 특성에도 영향을 준다. 극한의 메모리 압력 상황에서는 ZGC가 25~40배 더 나은 pause time을 보여준다. 반면 중간 수준의 메모리 사용 시나리오에서는 Shenandoah가 더 높은 처리량을 보이는 경우가 있다. 지연과 처리량 사이의 균형이 ZGC보다 나은 지점이 존재하는 것이다.

메모리 오버헤드 면에서도 차이가 있다. Shenandoah는 객체당 추가 포인터 필드 하나가 필요해서 10~20%의 오버헤드가 발생하는데, 이는 ZGC의 20~30%보다 적다.

하나 주의할 점이 있다. Shenandoah는 Oracle JDK에 포함되어 있지 않다. OpenJDK, AdoptOpenJDK, Red Hat 빌드 등에서만 사용 가능하다. Red Hat 생태계(OpenShift 등)에서 운영 중이라면 자연스러운 선택이 될 수 있지만, Oracle JDK를 쓰고 있다면 JDK 변경까지 고려해야 한다. 이건 GC 변경보다 훨씬 큰 결정이다.

설정은 `-XX:+UseShenandoahGC`다.

## 워크로드별 GC 선택 의사결정 트리

각 GC의 특성을 살펴봤으니, 이제 실제로 어떻게 선택할지 정리해보자. 복잡한 비교표보다 의사결정 트리가 더 실용적이다.

**1단계: 리소스부터 확인한다.**
CPU 2개 미만, 힙 2GB 미만의 제한된 환경인가? 그렇다면 **Serial GC**다. 더 복잡한 GC를 올려봤자 오버헤드만 늘어난다.

**2단계: 워크로드의 성격을 파악한다.**
사용자의 실시간 요청을 처리하는 서비스인가, 아니면 야간 배치처럼 전체 처리 시간이 중요한 작업인가? 배치 처리 우선이라면 **Parallel GC**가 처리량을 극대화해준다.

**3단계: 지연 시간 요구사항을 확인한다.**
P99 응답 시간이 수백 ms 이내면 충분한 일반적 서비스인가? 그렇다면 **G1 GC**가 가장 안전한 선택이다. 설정도 단순하고, 커뮤니티 경험과 자료도 풍부하다.

**4단계: 초저지연이 필요한가?**
JDK 21 이상을 사용 가능하고, 대규모 힙에서 1ms 미만의 pause가 필요하다면 **Generational ZGC**다. Oracle JDK가 아닌 OpenJDK/Red Hat 빌드를 사용하고 있다면 **Shenandoah**도 고려할 수 있다.

정리하면 이렇다.

| 워크로드 유형 | 권장 GC | 핵심 이유 |
|---|---|---|
| 소형 컨테이너 (100MB 미만 힙) | Serial GC | 최소 오버헤드, 단순함 |
| 데이터 처리 (Spark/Flink) | Parallel GC | 처리량 극대화 |
| 일반 웹 서비스 (Spring Boot 등) | G1 GC | 지연-처리량 균형, 안정성 |
| 초저지연 서비스 (JDK 21+) | Generational ZGC | 1ms 미만 pause |
| 초저지연 + OpenJDK 환경 | Shenandoah | 낮은 메모리 오버헤드 |

## Netflix의 Generational ZGC 마이그레이션: 현실의 검증

이론만으로는 찜찜하다. 실제로 대규모 서비스에서 GC를 바꿔본 사례를 살펴보자.

Netflix는 핵심 스트리밍 비디오 서비스의 절반 이상을 JDK 21 + Generational ZGC로 운영하고 있다. G1에서 ZGC로 전환한 결과, 동일하거나 더 나은 CPU 활용률에서 평균 및 P99 지연 시간이 모두 개선되었다. 주목할 점은 "성능 트레이드오프 없이" 전환에 성공했다는 것이다. 보통 GC를 바꾸면 한쪽이 좋아지는 대신 다른 쪽이 나빠지기 마련인데, Netflix의 경우 양쪽 모두 개선되었다.

Netflix가 개발한 **jvmquake**라는 도구도 흥미롭다. GC에 소비된 시간을 "부채(debt)"로 모델링하는 개념이다. GC에 200ms가 소비되면 부채 카운터에 200ms가 추가되고, 애플리케이션이 실행되는 시간이 이 부채를 상환한다. 부채가 임계값을 초과하면 JVM을 사전적으로 종료해서 GC 루프(GC가 거의 모든 시간을 차지하는 상황)에 빠지는 것을 방지한다. GC 자체를 바꾸는 것뿐 아니라, GC의 비정상 상태를 감지하고 대응하는 체계까지 갖춘 것이다.

한편 Uber의 사례는 조금 다른 교훈을 준다. Uber는 GC 최적화를 통해 CPU 소비 7만 코어를 절감하고 수백만 달러를 절약했다. HDFS NameNode에서는 힙과 Young Generation 크기를 조정하는 것만으로 최대 GC 시간을 22초에서 1.5초로 줄였다. 하지만 더 인상적인 교훈은 따로 있다. 메트릭 수집기의 backoff 설정이 1초가 아닌 1ms로 잘못 설정되어 400Mbps의 가비지 생성율을 유발했고, 이것이 2,258회의 GC pause(평균 177ms)로 이어진 것이다. 아무리 GC를 잘 튜닝해도, 코드 수준에서 가비지를 쏟아내면 의미가 없다.

## "G1이면 충분한가?" — 대부분의 개발자가 가장 궁금한 질문

솔직히 말하면, 대부분의 서비스에서 G1이면 충분하다. 이것은 보수적인 답변이 아니라, 실용적인 판단이다.

G1이 충분한 이유를 정리해보자. 첫째, JDK 9부터 기본 GC로 수년간 검증되었고, Java 25에서는 모든 환경의 기본으로까지 확대된다. 둘째, 지속적으로 개선 중이다. remembered set 처리와 mixed GC pause 스파이크가 버전마다 나아지고 있다. 셋째, 메모리 효율성이 ZGC나 Shenandoah보다 우수하다. 넷째, 튜닝이 비교적 단순하다. `-XX:MaxGCPauseMillis`와 `-XX:InitiatingHeapOccupancyPercent` 정도만 알면 대부분의 상황에 대응할 수 있다.

그렇다면 ZGC로 넘어가야 하는 분기점은 어디일까? 다음 조건 중 하나라도 해당되면 전환을 진지하게 검토하는 편이 낫다.

- **P99/P999 지연 시간이 비즈니스 핵심 지표일 때.** 결제 시스템, 실시간 입찰 시스템, 게임 서버처럼 꼬리 지연(tail latency)이 곧 매출이나 사용자 경험에 직결되는 경우다. G1의 간헐적 수백 ms pause가 허용되지 않는 서비스라면, ZGC의 1ms 미만 pause는 확실한 차이를 만든다.
- **힙이 수십 GB를 넘어갈 때.** 힙이 커질수록 G1의 Full GC 비용이 높아진다. 대규모 캐시를 메모리에 올려두는 서비스, 인메모리 데이터 그리드를 운영하는 서비스라면 ZGC가 더 안정적인 pause를 보장한다.
- **JDK 21 이상을 이미 사용하고 있을 때.** Generational ZGC의 성숙으로 이전의 메모리 오버헤드와 처리량 우려가 크게 완화되었다. JDK 21+라면 전환 비용이 낮다.

반대로, 다음 상황이라면 G1에 머무는 것이 현명하다.

- JDK 17 이하를 쓰고 있고, JDK 업그레이드가 가까운 시일 내에 어렵다면.
- 서비스의 메모리 여유가 넉넉하지 않다면. ZGC의 20~30% 메모리 오버헤드를 감당하기 어려울 수 있다.
- GC pause보다 전체 처리량이 더 중요한 서비스라면.

Netflix 사례가 보여주듯, ZGC 전환이 반드시 트레이드오프를 수반하지는 않는다. 하지만 "남들이 바꾸니까 나도 바꿔야지"라는 접근은 위험하다. 먼저 현재 GC 로그를 분석해서 pause가 실제로 문제가 되는지 확인하자. GC 로깅은 `-Xlog:gc*:file=gc.log:time,uptime,level,tags`로 활성화할 수 있다. 문제가 확인되면, 그때 ZGC를 스테이징 환경에서 먼저 테스트하고, 프로덕션 트래픽과 유사한 부하로 검증한 뒤 전환하는 것이 바람직하다.

## 마무리

GC 선택은 결국 "내 서비스에서 가장 중요한 것이 무엇인가"에 대한 답에서 시작한다. 지연인지, 처리량인지, 메모리인지. 이 세 가지 중 두 개를 선택하고, 그 선택에 맞는 GC를 고르면 된다.

대부분의 서비스에서는 G1이면 충분하다. 하지만 P99 지연이 비즈니스 핵심이거나, 힙이 수십 GB를 넘어간다면 Generational ZGC를 진지하게 검토하자. 소형 컨테이너에서는 Serial GC가 오히려 최적이고, 배치 워크로드에서는 Parallel GC가 여전히 강력하다.

잊지 말자. GC를 바꾸기 전에, 코드 수준에서 불필요한 객체 할당을 먼저 점검하는 것이 순서다. Uber 사례가 보여주듯, 1ms짜리 설정 실수가 수천 번의 GC pause를 유발할 수 있다.

다음 장에서는 GC를 선택한 뒤 실제로 어떻게 튜닝하는지를 다룬다. 힙 크기 설정, GC 로그 분석, 모니터링 도구 활용까지 — 선택한 GC의 성능을 최대한 끌어내는 방법을 함께 살펴보자.
