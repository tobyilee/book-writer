# 5장. GC 모니터링과 튜닝 — 측정하고, 이해하고, 조정하라

금요일 오후, 배포 직후 대시보드의 응답 시간 그래프가 울퉁불퉁해지기 시작한다. P99 레이턴시가 평소의 세 배를 찍고, 슬랙 채널에 알림이 쏟아진다. "GC 튜닝 좀 해야 하는 거 아닌가?" 누군가 말한다. 그런데 정작 무엇을 어떻게 측정해야 하는지, 어디부터 손을 대야 하는지 아는 사람은 드물다.

GC 튜닝은 신비로운 기술이 아니다. 측정하고, 이해하고, 조정하는 반복이다. 이번 장에서는 이 세 단계를 실습 워크플로우로 풀어보자.

## 튜닝의 제1원칙: 측정 없이 손대지 않는다

GC 튜닝에 관해 가장 먼저 기억해둘 원칙이 하나 있다. **기본값으로 시작하고, 측정 후에만 튜닝한다.**

이 원칙이 왜 중요할까? JVM의 기본 GC 설정은 수많은 워크로드에서 검증된 값이다. JDK 9 이후 G1이 기본 GC로 자리잡은 이후, 대부분의 일반적인 서버 애플리케이션은 별도 튜닝 없이도 합리적으로 동작한다. 근거 없이 인터넷에서 주워온 옵션을 복사해 붙이면, 오히려 기본값보다 나빠지는 경우가 허다하다.

그렇다면 언제 튜닝을 시작해야 할까? 답은 단순하다. **문제가 측정으로 확인되었을 때.** 응답 시간이 SLA를 위반하거나, GC pause가 비정상적으로 길어지거나, OOM이 반복될 때다. 감이 아니라 숫자로 확인된 문제만 다룬다.

## Step 1: GC 로그 수집 — 모든 것은 로그에서 시작된다

GC 모니터링의 첫 단계는 놀랍도록 간단하다. GC 로그를 켜는 것이다.

JDK 9 이상에서는 통합 로깅 프레임워크(Unified Logging)를 사용한다. 다음 옵션 하나면 충분하다.

```bash
-Xlog:gc*:file=gc.log:time,uptime,level,tags
```

이 한 줄이 무엇을 하는지 살펴보자.

- `gc*`: GC 관련 모든 태그의 로그를 수집한다
- `file=gc.log`: 출력을 파일로 저장한다
- `time,uptime,level,tags`: 각 로그 라인에 시각, 가동 시간, 레벨, 태그를 데코레이터로 붙인다

프로덕션 환경이라면 로그 로테이션도 함께 설정하는 편이 낫다.

```bash
-Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=100m
```

로그를 켜는 것 자체는 성능에 거의 영향을 주지 않는다. 찜찜해서 로그를 꺼두는 팀이 의외로 많은데, 장애가 터진 뒤에 "로그가 없어서 원인을 모르겠다"고 말하게 되는 것이 훨씬 끔찍한 일이다. 프로덕션에서는 항상 GC 로그를 활성화해두자.

### GC 로그 읽기: 핵심만 짚자

G1 GC 로그의 한 줄을 보자.

```
[2024-03-15T14:23:01.234+0900][12.456s] GC(42) Pause Young (Normal) (G1 Evacuation Pause) 1024M->512M(2048M) 15.234ms
```

이 한 줄에서 읽어낼 수 있는 정보는 다음과 같다.

- `Pause Young (Normal)`: Young Generation의 일반적인 수집이다
- `1024M->512M(2048M)`: 힙 사용량이 1024MB에서 512MB로 줄었고, 전체 힙은 2048MB다
- `15.234ms`: 이 GC pause에 걸린 시간이다

여기서 주목할 숫자는 세 가지다. **pause 시간**(15ms), **회수량**(512MB), **힙 점유율 변화**. 이 세 가지의 추이를 시간 축 위에 놓으면 GC의 건강 상태가 보인다.

pause 시간이 점점 길어지고 있다면? 회수량이 줄어들고 있다면? 힙 사용량이 GC 후에도 계속 높은 수준을 유지한다면? 각각 다른 문제를 가리킨다. 하지만 수백, 수천 줄의 로그를 눈으로 읽는 것은 비현실적이다. 이제 도구의 힘을 빌릴 차례다.

## Step 2: GCeasy 분석 — 병목을 시각화하다

GCeasy는 GC 로그를 업로드하면 자동으로 분석해주는 웹 기반 도구다. 무료 플랜으로도 핵심 정보를 충분히 얻을 수 있다.

GC 로그 파일을 업로드하면 GCeasy가 제공하는 핵심 분석 항목은 다음과 같다.

**Pause 시간 분포**: 전체 GC pause의 분포를 히스토그램으로 보여준다. 대부분의 pause가 50ms 이내인데 간헐적으로 500ms짜리가 튄다면, 그 원인을 추적해야 한다. Mixed GC인지, Full GC인지, 혹은 Concurrent Marking이 밀린 것인지.

**힙 사용량 추이**: 시간에 따른 힙 점유율 그래프다. GC 후에도 Old Generation의 점유율이 계속 상승하는 패턴이 보인다면 메모리 릭을 의심해야 한다.

**할당률(Allocation Rate)**: 애플리케이션이 얼마나 빠르게 객체를 생성하는지 보여준다. 할당률이 비정상적으로 높다면, GC 튜닝보다 코드를 먼저 살펴봐야 한다.

**승격률(Promotion Rate)**: Young에서 Old로 승격되는 객체의 양이다. 승격률이 높다면 Young Generation이 너무 작거나, 단기 객체가 아닌 중기 수명 객체가 많다는 신호다.

GCeasy는 진단의 출발점이다. "어디가 아픈지"를 빠르게 파악하는 용도로 쓰고, 더 깊이 들어가야 할 때는 다음 단계로 넘어간다.

## Step 3: JFR 심층 분석 — 속을 들여다보다

Java Flight Recorder(JFR)는 JDK에 내장된 프로파일링 도구다. JDK 11부터 상용 라이선스 없이 무료로 사용할 수 있다. GC 로그가 "무엇이 일어났는가"를 알려준다면, JFR은 "왜 일어났는가"까지 파고들 수 있게 해준다.

JFR을 시작하는 가장 간단한 방법은 JVM 옵션으로 활성화하는 것이다.

```bash
-XX:StartFlightRecording=duration=60s,filename=recording.jfr
```

이미 실행 중인 JVM이라면 jcmd로도 가능하다.

```bash
jcmd <pid> JFR.start duration=60s filename=recording.jfr
```

JFR이 수집하는 GC 관련 이벤트 중 특히 유용한 것들을 살펴보자.

- **jdk.GarbageCollection**: 각 GC 이벤트의 원인, 유형, 소요 시간
- **jdk.GCHeapSummary**: GC 전후의 힙 상태 상세
- **jdk.ObjectAllocationInNewTLAB / jdk.ObjectAllocationOutsideTLAB**: 어떤 클래스의 객체가 얼마나 할당되는지, 할당이 발생하는 코드 위치(스택 트레이스)까지 추적

세 번째 항목이 핵심이다. GC 로그나 GCeasy에서 "할당률이 높다"는 것을 확인했다면, JFR로 **어떤 코드가 할당을 유발하는지** 정확히 짚을 수 있다. 이것이 GC 로그 분석과 JFR 분석의 결정적 차이다.

수집된 .jfr 파일은 JDK Mission Control(JMC) 또는 IntelliJ IDEA의 프로파일러로 열어볼 수 있다.

## 핵심 JVM 옵션 해부

모니터링으로 병목을 확인했다면, 이제 조정할 차례다. GC 튜닝에서 가장 자주 다루는 옵션 세 가지를 살펴보자.

### -Xms와 -Xmx: 힙의 바닥과 천장

```bash
-Xms4g -Xmx4g
```

`-Xms`는 초기 힙 크기, `-Xmx`는 최대 힙 크기다. 프로덕션에서는 **둘을 같은 값으로 설정하는 편이 낫다.** 왜 그럴까? 힙이 동적으로 확장/축소될 때마다 JVM은 메모리를 재할당하고 재배치하는 비용을 지불한다. 처음부터 필요한 만큼 확보해두면 이 비용을 피할 수 있다.

### MaxGCPauseMillis: 목표를 선언하라

```bash
-XX:MaxGCPauseMillis=200
```

G1 GC에게 "pause를 200ms 이내로 유지해달라"고 요청하는 옵션이다. 주의할 점은, 이것이 **보장이 아니라 목표**라는 것이다. G1은 이 목표를 달성하기 위해 Young Generation 크기를 동적으로 조절하고, 한 번에 수집할 리전 수를 조절한다.

이 값을 너무 낮게 잡으면 어떻게 될까? G1이 한 번에 처리하는 양을 줄여 pause는 짧아지지만, GC 빈도가 증가하고 전체 처리량이 떨어진다. 너무 높게 잡으면 pause가 길어지지만 처리량은 올라간다. 서비스의 SLA에 맞는 적정값을 찾아야 한다.

### IHOP: Concurrent Marking의 시작점

```bash
-XX:InitiatingHeapOccupancyPercent=45
```

IHOP(Initiating Heap Occupancy Percent)는 Old Generation의 점유율이 이 비율에 도달하면 Concurrent Marking을 시작하라는 신호다. 기본값은 45%다.

이 값이 왜 중요할까? Concurrent Marking이 너무 늦게 시작되면, 마킹이 끝나기 전에 Old Generation이 가득 차서 Full GC가 발생한다. Full GC는 전체 힙을 대상으로 하며, G1에서는 Single-threaded로 수행되어 수 초에서 수십 초까지 걸릴 수 있다. 난감한 상황이다.

실제 프로덕션 사례를 보자. 한 시스템에서 IHOP를 70%로 설정했더니, Concurrent Marking이 완료되기 전에 힙이 차면서 하루에 3회 이상 OOM이 발생했다. IHOP를 낮춰 마킹을 일찍 시작하게 하는 것이 해결책이었다. 다만, Working Set 크기와 동일하게 설정하면 GC가 상시 실행되어 CPU를 과도하게 소진하므로 균형점을 찾아야 한다.

## 힙 사이징 가이드라인

힙 크기를 어떻게 잡아야 할까? Uber의 대규모 서비스 운영 경험에서 추출된 가이드라인을 참고해보자.

- **전체 힙은 가용 RAM의 75% 이하로 설정한다.** 나머지는 OS, 네이티브 메모리(Metaspace, 스레드 스택, NIO 버퍼 등)를 위해 남겨두어야 한다.
- **전체 힙은 측정된 최대 메모리 풋프린트보다 20% 크게 설정한다.** GC가 일할 여유 공간이 필요하다.
- **Young Generation은 전체 힙의 20~50% 범위에서 조절한다.**
- **전체 힙을 늘릴 때는 Young Generation도 비례적으로 늘린다.** 이것을 놓치면 난감한 일이 벌어진다.

마지막 항목이 특히 중요하다. Uber에서 실제로 겪은 사례인데, 전체 힙을 늘리면서 Young Generation 크기를 그대로 두었더니 ParNew GC 시간이 오히려 35% 증가했다. Old Generation만 커진 상태에서 Young이 상대적으로 작아지면, Young GC가 더 자주 발생하고 승격 빈도도 높아지기 때문이다.

## 코드 최적화 vs GC 튜닝: 어디에 먼저 투자할 것인가

GC pause가 길다는 진단이 나왔을 때, 바로 JVM 옵션을 만지작거리고 싶은 충동이 든다. 하지만 잠깐 멈추고 생각해보자. 정말 GC 옵션의 문제인가, 아니면 애플리케이션 코드가 쓰레기를 너무 많이 만들고 있는 것인가?

Uber에서 실제로 있었던 일이다. 한 서비스의 GC가 비정상적으로 자주 발생했다. 조사해보니, 메트릭 수집기의 backoff 설정이 1초가 아니라 1ms로 잘못 설정되어 있었다. 이 하나의 버그가 **400Mbps의 가비지 생성률**을 만들어냈고, 2,258회의 GC pause(평균 177ms)를 유발했다. 아무리 GC를 튜닝해도 이런 수준의 할당률은 감당할 수 없다.

실무에서의 합의는 이렇다.

1. **먼저 JFR로 할당 핫스팟을 확인한다.** 비정상적으로 높은 할당률의 원인이 코드에 있다면, 그것을 먼저 고친다.
2. **코드 수준의 과도한 할당이 없다면, 그때 GC 옵션을 조정한다.** 적절한 GC 선택, 힙 사이징, pause 목표 설정 순서로.

물론, 레거시 코드를 당장 고칠 수 없는 상황도 있다. 그럴 때는 GC 튜닝이 현실적인 대안이 된다. JVM 옵션 변경은 코드 변경 대비 리스크와 비용이 낮으니까. 하지만 그것이 근본적 해결이 아니라는 점은 잊지 말자.

## 실전 사례 1: Uber HDFS NameNode — GC 시간 22초에서 1.5초로

Uber의 HDFS NameNode는 160GB 힙을 사용하는 대규모 서비스다. GC 최적화를 통해 CPU 소비 7만 코어를 절감하고 수백만 달러를 절약한 사례에서, NameNode 튜닝은 가장 극적인 결과를 보여준다.

튜닝 전 상황을 보자.

- 최대 GC 시간: **22초** (서비스가 22초간 멈춘다는 뜻이다)
- RPC 큐 대기 시간: 500ms
- GC 횟수: 90회

적용한 옵션은 다음과 같다.

```bash
-Xmx160g -Xmn16g -XX:ParGCCardsPerStrideChunk=32k
```

핵심은 `-Xmn16g`로 Young Generation을 전체 힙의 10%인 16GB로 명시한 것과, `ParGCCardsPerStrideChunk`로 카드 테이블 스캔의 병렬 처리 단위를 키운 것이다.

결과는 놀랍다.

- 최대 GC 시간: **1.5초** (22초에서 93% 감소)
- RPC 큐 대기 시간: 400ms (20% 개선)
- GC 횟수: 70회

옵션 몇 줄의 변경이 서비스의 안정성을 근본적으로 바꾼 것이다. 물론 이것은 160GB 힙이라는 극단적 환경에서의 사례다. 일반적인 4~8GB 힙에서는 이 정도의 극적 효과를 기대하기 어렵다. 하지만 "측정하고, 원인을 파악하고, 적합한 옵션을 찾는다"는 절차는 동일하다.

## 실전 사례 2: LinkedIn의 OS 수준 GC 최적화

LinkedIn의 사례는 JVM 옵션뿐 아니라 운영체제 수준의 설정도 GC 성능에 영향을 준다는 것을 보여준다.

환경은 32GB 힙(Young 6GB)에서 시작하여 40GB로 확장한 Java 서비스였다. CMS GC를 사용하고 있었다.

LinkedIn이 적용한 최적화 중 특히 주목할 두 가지가 있다.

### AlwaysPreTouch

```bash
-XX:+AlwaysPreTouch
```

JVM이 시작할 때 힙 전체를 한 번 터치하여 OS로부터 물리 메모리 페이지를 미리 할당받는 옵션이다. 이 옵션이 없으면, JVM이 처음 메모리 영역에 접근할 때마다 OS가 페이지를 할당하고 제로화하는 비용이 발생한다. GC가 새로운 메모리 영역으로 객체를 이동시킬 때마다 이 비용을 치르게 되면, GC pause가 길어진다.

### vm.swappiness=0

```bash
sysctl vm.swappiness=0
```

리눅스 커널의 스왑 성향을 최소로 낮추는 설정이다. 스왑이 발생하면 디스크 I/O가 개입하며, 이것이 GC pause 동안에 일어나면 pause 시간이 수십 배로 뛸 수 있다.

LinkedIn은 하나 더 흥미로운 발견을 했다. 백그라운드 IO 트래픽(OS 페이지 캐시의 writeback)이 JVM의 GC 로깅을 블로킹하여 대규모 STW pause를 유발한 것이다. GC 자체는 빠르게 끝났지만, 로그를 쓰는 동안 스레드가 블로킹되어 전체 pause가 길어지는 기묘한 상황이었다.

이러한 최적화의 결과는 다음과 같다.

- Young GC pause: 80ms에서 40~60ms로 감소
- P99.9 레이턴시: 100ms에서 60ms로 감소

## 마무리

GC 모니터링과 튜닝의 핵심을 다시 정리해보자.

1. **기본값으로 시작하고, 문제가 측정으로 확인되었을 때만 튜닝한다.**
2. **3단계 워크플로우를 따른다:** GC 로그 수집 → GCeasy로 병목 시각화 → JFR로 원인 추적.
3. **코드를 먼저 의심한다.** 할당률이 비정상이라면 GC 옵션이 아니라 코드를 고쳐야 한다.
4. **힙 사이징은 측정 기반으로, Young과 Old의 비율까지 신경 쓴다.**
5. **JVM 밖도 살핀다.** OS 수준의 설정(AlwaysPreTouch, swappiness)도 GC 성능에 영향을 준다.

측정 → 이해 → 조정. 이 순환을 성실히 반복하면, GC는 더 이상 블랙박스가 아니라 예측 가능한 시스템 컴포넌트가 된다.

다음 장에서는 GC가 예상대로 작동하지 않을 때, 즉 OOM이 터지고 서버가 죽었을 때 어떻게 원인을 추적하고 살려내는지를 다룬다. 프로덕션 트러블슈팅의 세계로 들어가보자.
