# 2장. JVM의 구조 — 바이트코드에서 메모리까지

Spring Boot 애플리케이션을 하나 띄웠다고 해보자. 컨트롤러를 하나 만들고, 서비스 객체를 주입받고, DB에서 데이터를 꺼내 DTO로 변환해서 응답한다. 이 흔한 요청 하나를 처리하는 동안, 내가 만든 객체들은 도대체 메모리 어디에 살고 있을까? 지역 변수로 선언한 `List<User>`는 어디에 있고, `@Service`로 등록한 싱글턴 빈은 또 어디에 있는 걸까?

1장에서 JVM의 전체 구조를 30초 만에 훑어봤다. 이번 장에서는 그 조감도를 정밀한 지도로 확장한다. 바이트코드가 실행되는 과정부터, 힙과 스택이 각각 무엇을 책임지는지, 그리고 내 코드의 변수 하나하나가 메모리의 어느 칸에 놓이는지를 추적해보자.

## 소스 코드에서 실행까지: 바이트코드와 JIT 컴파일러

Java나 Kotlin으로 작성한 코드가 실행되기까지는 두 번의 변환을 거친다. 첫 번째는 컴파일러가 소스 코드를 바이트코드(.class 파일)로 변환하는 단계이고, 두 번째는 JVM이 이 바이트코드를 실제 기계어로 바꿔 실행하는 단계다.

왜 바로 기계어로 컴파일하지 않고, 굳이 바이트코드라는 중간 단계를 둘까? 이유는 간단하다. 바이트코드는 특정 운영체제나 CPU에 종속되지 않는다. 같은 .class 파일을 리눅스 서버에서도, 맥북에서도, 윈도우 PC에서도 실행할 수 있다. "Write Once, Run Anywhere"라는 Java의 오래된 약속이 바로 이 구조 덕분에 가능하다.

JVM의 실행 엔진은 바이트코드를 두 가지 방식으로 처리한다.

**인터프리터**는 바이트코드를 한 줄씩 읽어서 즉시 실행한다. 시작은 빠르지만, 같은 코드를 반복 실행할 때마다 매번 해석해야 하니 느리다.

**JIT(Just-In-Time) 컴파일러**는 여기서 등장한다. JVM은 메서드가 얼마나 자주 호출되는지를 추적하고 있다가, 호출 횟수가 일정 임계값을 넘으면 "이건 자주 쓰는 코드니까 아예 기계어로 컴파일해두자"라고 판단한다. 한 번 컴파일된 네이티브 코드는 이후 호출 시 인터프리터를 거치지 않고 바로 실행된다.

이 방식이 영리한 점은 "모든 코드를 처음부터 컴파일하지 않는다"는 데 있다. 실행 초기에는 인터프리터가 빠르게 시작하고, 자주 호출되는 핫 코드만 선별적으로 최적화한다. 실행 시간이 길어질수록 애플리케이션은 점점 빨라진다. 서버 애플리케이션처럼 오래 실행되는 프로그램에 JVM이 잘 맞는 이유가 바로 이것이다.

## 힙(Heap): 객체가 태어나고 살고 죽는 곳

`new` 키워드로 객체를 생성하면, 그 객체는 힙에 자리를 잡는다. `new ArrayList<>()`, `new UserDto()`, `new StringBuilder()` — 우리가 만드는 거의 모든 객체가 힙에 산다.

힙은 모든 스레드가 공유하는 영역이다. 스레드 A에서 만든 객체를 스레드 B가 참조할 수 있는 이유가 바로 힙이 공유 공간이기 때문이다. 그리고 이 공유 영역이 바로 Garbage Collector가 관리하는 주된 대상이기도 하다.

여기서 한 가지 중요한 구분이 있다. 힙은 내부적으로 **Young Generation**과 **Old Generation**으로 나뉜다.

왜 굳이 나눌까? "대부분의 객체는 금방 죽는다"는 경험적 사실 때문이다. API 요청을 처리하면서 만들어지는 DTO, 임시 리스트, 문자열 조합용 StringBuilder 같은 것들은 요청이 끝나면 더 이상 참조되지 않는다. 이런 단명 객체가 전체 객체의 대다수를 차지한다.

반면, 스프링 빈처럼 애플리케이션이 살아있는 동안 계속 참조되는 장수 객체도 있다. 단명 객체와 장수 객체를 같은 공간에 두고 관리하면 비효율적이다. 짧게 살다 죽는 객체들만 모아놓고 빠르게 정리하는 편이 훨씬 효율적이다.

그래서 힙을 이렇게 나눈다:

- **Young Generation**: 새로 생성된 객체가 처음 자리잡는 곳. 여기서 빈번하고 빠른 GC(Minor GC)가 일어난다.
  - **Eden 영역**: 객체가 최초로 할당되는 공간
  - **Survivor 영역(S0, S1)**: Minor GC에서 살아남은 객체가 잠시 머무는 대기실
- **Old Generation**: Young 영역에서 여러 차례 GC를 살아남은 객체가 승격(Promotion)되어 오는 곳. GC 빈도가 낮지만, 한 번 일어나면 시간이 오래 걸린다.

이 구조가 왜 중요한지는 3장에서 GC 알고리즘을 다룰 때 더 분명해진다. 지금은 "힙은 단순한 하나의 공간이 아니라, 객체의 수명에 따라 나뉜 구조"라는 점만 기억해두자.

## 스택(Stack): 메서드 호출의 역사가 쌓이는 곳

힙이 "객체의 세계"라면, 스택은 "실행 흐름의 기록"이다.

메서드를 호출할 때마다 JVM은 스택에 **스택 프레임** 하나를 쌓는다. 이 프레임 안에는 그 메서드의 지역 변수, 전달받은 인자, 그리고 메서드가 끝난 뒤 돌아갈 주소가 담긴다. 메서드 실행이 끝나면 해당 프레임은 자동으로 제거된다. LIFO(Last-In-First-Out) — 마지막에 호출된 메서드가 가장 먼저 끝나는 구조다.

```
main() 호출       → [main 프레임]
  processOrder() 호출  → [main 프레임][processOrder 프레임]
    calculateTotal() 호출 → [main 프레임][processOrder 프레임][calculateTotal 프레임]
    calculateTotal() 종료 → [main 프레임][processOrder 프레임]
  processOrder() 종료  → [main 프레임]
```

스택에서 가장 중요한 특성은 **스레드별로 독립적**이라는 점이다. 스레드가 하나 생성되면 그 스레드만의 스택이 함께 만들어진다. 다른 스레드의 스택에는 접근할 수 없다.

이게 실무에서 어떤 의미일까? 메서드 안에서 선언한 지역 변수는 다른 스레드가 건드릴 수 없다. 스레드 안전성을 고민할 때 "이 변수는 지역 변수니까 동기화가 필요 없다"고 판단할 수 있는 근거가 바로 여기에 있다. 반면, 힙에 있는 객체를 여러 스레드가 동시에 수정하면 문제가 생길 수 있다. 그래서 `synchronized`나 `volatile` 같은 동기화 메커니즘이 필요한 것이다.

한 가지 더. `StackOverflowError`를 만나본 적이 있을 것이다. 재귀 호출이 너무 깊어져서 스택 프레임이 한계를 넘으면 발생하는 에러다. 스택은 크기가 제한되어 있기 때문이다. 이 에러를 만나면 찜찜한 기분이 들기 마련인데, 원인을 알고 나면 대처는 명확하다 — 재귀 깊이를 줄이거나, 반복문으로 바꾸면 된다.

## 메서드 영역과 Metaspace: 클래스 메타데이터의 집

객체가 힙에 살고, 실행 흐름이 스택에 기록된다면, 클래스 자체의 정보는 어디에 저장될까? 메서드 영역(Method Area)이다.

메서드 영역에는 클래스의 구조 정보, 메서드 코드, 상수 풀(Constant Pool), 그리고 정적(static) 변수가 저장된다. 힙과 마찬가지로 모든 스레드가 공유하는 영역이다.

여기서 역사적 맥락을 하나 짚어보자. Java 7 이하에서는 메서드 영역이 **PermGen(Permanent Generation)**이라는 힙의 일부 영역에 구현되어 있었다. 문제는 PermGen의 크기가 고정되어 있었다는 것이다. 클래스를 많이 로딩하는 애플리케이션 — 예를 들어 동적 프록시를 대량으로 생성하거나, 핫 리로드를 반복하는 개발 환경 — 에서는 `java.lang.OutOfMemoryError: PermGen space`라는 난감한 에러를 자주 만났다.

Java 8부터는 PermGen이 사라지고 **Metaspace**가 그 자리를 대신한다. Metaspace는 힙이 아니라 네이티브 메모리(OS가 직접 관리하는 메모리)에 위치한다. 기본적으로 크기 제한이 없어서 시스템의 가용 메모리만큼 확장된다. 덕분에 PermGen 시절의 고질적인 OOM 문제는 크게 줄었다.

물론 제한이 아예 없으면 그것도 문제가 될 수 있다. 클래스 로딩에 이상이 생겨 Metaspace가 무한히 커지면 OS 레벨에서 메모리 문제가 발생한다. 그래서 프로덕션에서는 `-XX:MaxMetaspaceSize`로 상한선을 걸어두는 편이 낫다.

## 변수는 어디에 사는가: 저장 위치 정리

지금까지 살펴본 메모리 영역을 변수의 관점에서 정리해보자.

| 변수 유형 | 선언 위치 | 저장 영역 | 생명주기 |
|-----------|----------|----------|----------|
| 지역 변수 | 메서드 내부 | 스택 | 메서드 실행 동안 |
| 인스턴스 변수 | 클래스의 필드 (non-static) | 힙 (객체와 함께) | 객체가 GC될 때까지 |
| 정적 변수 | 클래스의 static 필드 | 메서드 영역 | 클래스가 언로드될 때까지 |

이 구분이 왜 중요할까? 메모리 문제를 진단할 때 "이 데이터가 어디에 있는지"를 아는 것이 출발점이기 때문이다. 힙 덤프를 분석할 때 보이는 것은 힙에 있는 객체뿐이고, 스택 트레이스에서 보이는 것은 스택의 프레임뿐이다. 어떤 변수가 어디에 속하는지 감각이 있어야 도구가 보여주는 정보를 제대로 읽을 수 있다.

## 코드로 추적하는 메모리 배치

구체적인 코드를 하나 놓고 메모리 배치를 추적해보자.

```java
public class OrderService {
    private static final int MAX_RETRY = 3;          // (1)
    private final OrderRepository repository;         // (2)

    public OrderService(OrderRepository repository) {
        this.repository = repository;
    }

    public OrderDto getOrder(Long orderId) {
        Order order = repository.findById(orderId);   // (3)
        List<ItemDto> items = new ArrayList<>();       // (4)

        for (OrderItem item : order.getItems()) {
            items.add(new ItemDto(item.getName(), item.getPrice())); // (5)
        }

        return new OrderDto(order.getId(), items);    // (6)
    }
}
```

이 코드가 실행될 때 메모리에서 무슨 일이 벌어지는지 살펴보자.

**(1) `MAX_RETRY`** — `static final` 변수다. 메서드 영역에 저장된다. 클래스가 로딩될 때 한 번 할당되고, 애플리케이션이 종료될 때까지 그 자리에 있다.

**(2) `repository`** — 인스턴스 변수다. `OrderService` 객체가 힙에 생성될 때 그 객체의 일부로 함께 힙에 자리잡는다. 스프링 빈이라면 애플리케이션 컨텍스트가 살아있는 동안 힙에 머문다.

**(3) `order`** — `getOrder` 메서드의 지역 변수다. 이 변수 자체(참조값)는 스택 프레임에 저장된다. 하지만 이 변수가 가리키는 `Order` 객체는 힙에 있다. 메서드가 끝나면 스택의 참조는 사라지고, 힙의 `Order` 객체는 더 이상 참조하는 곳이 없으면 GC 대상이 된다.

**(4) `items`** — 역시 지역 변수다. 참조값은 스택에, `ArrayList` 객체 자체는 힙에 생성된다.

**(5) `new ItemDto(...)`** — 반복문을 돌 때마다 힙에 새로운 `ItemDto` 객체가 생성된다. 주문 항목이 100개라면 100개의 `ItemDto`가 힙에 만들어진다. 이런 단명 객체들이 Young Generation의 Eden 영역을 채운다.

**(6) `new OrderDto(...)`** — 응답으로 내보낼 DTO 객체가 힙에 생성된다. 이 객체는 HTTP 응답이 완료되면 참조가 끊기고 GC 대상이 된다.

이 과정을 시각화하면 이렇다:

```
[스택 - 스레드 #1]              [힙]                    [메서드 영역]
┌─────────────────┐          ┌──────────────┐        ┌──────────────┐
│ getOrder 프레임  │          │ Order 객체    │        │ MAX_RETRY: 3 │
│  orderId: 42    │          │ ArrayList    │        │ OrderService │
│  order: ──────────────────→│ ItemDto #1   │        │  클래스 정보  │
│  items: ──────────────────→│ ItemDto #2   │        └──────────────┘
│                 │          │ OrderDto     │
└─────────────────┘          │ OrderService │
                             │  (싱글턴 빈)  │
                             └──────────────┘
```

하나의 API 요청을 처리하는 동안에도 힙에는 여러 객체가 생겼다가 사라진다. 트래픽이 초당 수천 건이라면, 매초 수만 개의 객체가 Eden 영역에서 태어나고 죽는 셈이다. 왜 GC가 바쁜지, 왜 세대별로 나눠서 효율적으로 관리해야 하는지가 이 그림에서 느껴지지 않는가.

## 마무리

이번 장에서 JVM 메모리의 전체 지도를 그려봤다. 바이트코드가 인터프리터와 JIT를 통해 실행되고, 객체는 힙에, 실행 흐름은 스택에, 클래스 정보는 메서드 영역에 저장된다. 그리고 힙은 단일 공간이 아니라 Young과 Old로 나뉘어 있다.

이 지도를 머릿속에 갖고 있으면, 앞으로 만날 GC 로그와 메모리 에러가 훨씬 덜 난감해질 것이다. "Eden space 95% used"라는 로그가 뭘 뜻하는지, "Old Gen이 가득 찼다"는 경고가 왜 심각한지, 이제 감이 잡힌다.

다음 장에서는 이 힙 구조 위에서 GC가 어떻게 동작하는지를 본격적으로 다룬다. "쓰레기를 어떻게 찾고, 어떻게 치우는가" — Mark-and-Sweep부터 세대별 수집까지, GC의 핵심 원리를 파헤쳐보자.
