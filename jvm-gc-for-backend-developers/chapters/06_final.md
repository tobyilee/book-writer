# 6장. 프로덕션 트러블슈팅 — 서버가 죽었을 때 살리는 법

새벽 3시, 잠결에 울리는 PagerDuty 알림. 눈을 비비며 슬랙을 열어보니 "서버 3대 중 2대 OOM으로 죽었습니다"라는 메시지가 찍혀 있다. 대시보드를 열면 에러율은 천장을 뚫고 있고, 남은 1대마저 응답 시간이 10초를 넘기고 있다. 이 순간 무엇부터 해야 할까?

당황스럽다. 하지만 당황할 시간이 없다. 이번 장에서는 서버가 죽었을 때 — 정확히는 JVM이 비정상 종료했을 때 — 체계적으로 원인을 추적하고 해결하는 방법을 살펴보자.

## OOM 에러, 같은 이름 다른 원인

`java.lang.OutOfMemoryError`라는 이름은 하나지만, 뒤에 붙는 메시지에 따라 원인과 대응이 완전히 다르다. 하나씩 살펴보자.

**Java heap space**

가장 흔한 유형이다. 힙에 객체를 할당하려는데 공간이 부족하고, GC를 돌려도 충분한 메모리를 확보하지 못할 때 발생한다. 원인은 크게 두 가지다. 첫째, 단순히 힙이 작은 경우 — `-Xmx`를 늘리면 해결된다. 둘째, 메모리 릭이 있는 경우 — 힙을 아무리 늘려도 결국 다시 터진다. 찜찜하게도, 두 경우 모두 같은 에러 메시지를 뱉기 때문에 힙 덤프를 떠봐야 구분할 수 있다.

**GC overhead limit exceeded**

GC가 전체 실행 시간의 98% 이상을 차지하면서도 힙의 2% 미만만 회수하는 상황이 반복될 때 JVM이 스스로 포기하며 던지는 에러다. 사실상 "힙이 거의 가득 찼는데 GC가 아무것도 못 치우고 있다"는 뜻이다. 힙을 늘리거나, 객체 할당률 자체를 줄여야 한다.

**Metaspace**

Java 8 이후 클래스 메타데이터가 저장되는 Metaspace 영역이 가득 찼을 때 발생한다. 동적 프록시를 과도하게 생성하거나, 애플리케이션을 빈번하게 재배포(핫 디플로이)할 때 자주 만난다. `-XX:MaxMetaspaceSize`를 조정하되, 근본 원인인 클래스 과다 로딩을 해결하는 편이 낫다.

**Unable to create new native thread**

힙과는 무관하게, OS 레벨에서 새 스레드를 생성할 수 없을 때 발생한다. 컨테이너 환경에서 메모리 제한에 걸리거나, 스레드 풀 크기를 제한하지 않아 수천 개의 스레드가 만들어지는 경우다. 스레드 풀을 바운딩하고, 필요하면 `-Xss`(스레드 스택 크기)를 줄이는 것이 해법이다.

## 힙 덤프 분석 — 범인을 찾는 첫 번째 단서

OOM이 발생했다면, 가장 먼저 확인해야 할 것은 힙 덤프다. 그런데 문제는 OOM이 터진 후에는 이미 프로세스가 죽어있을 수 있다는 점이다. 그래서 사전에 이 옵션을 반드시 설정해두자.

```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dumps/
```

이 두 줄만 JVM 옵션에 추가해두면, OOM이 발생하는 순간 자동으로 힙 덤프 파일(.hprof)이 생성된다. 새벽에 서버가 죽어도 "범행 현장 사진"이 남아 있는 셈이다.

힙 덤프를 분석할 때는 Eclipse MAT(Memory Analyzer Tool)이나 JVisualVM을 사용한다. 핵심은 "Dominator Tree"를 보는 것이다. 어떤 객체가 가장 많은 메모리를 점유하고 있는지, 그 객체를 누가 참조하고 있는지를 추적하면 릭의 출발점을 찾을 수 있다.

한 가지 주의할 점이 있다. 프로덕션 서버의 힙 덤프는 수 GB에 달할 수 있다. 4GB 힙을 쓰는 서버라면 힙 덤프도 4GB 이상이 된다. 분석 머신에 충분한 메모리가 있는지 미리 확인해두자.

## 메모리 릭의 단골 패턴

메모리 릭은 대개 몇 가지 반복되는 패턴에서 발생한다. 알아두면 힙 덤프를 보기도 전에 "아, 혹시 이거 아닐까?" 하고 감을 잡을 수 있다.

**정적 컬렉션의 무한 성장**

`static Map`이나 `static List`에 데이터를 넣기만 하고 빼지 않는 경우다. 캐시를 직접 구현한다며 `HashMap`에 결과를 담아두는데, 만료 정책이 없으면 서버가 살아있는 한 계속 쌓인다. 난감한 것은, 트래픽이 적을 때는 멀쩡하다가 프로덕션에서 며칠 운영한 뒤에야 터진다는 점이다.

**리스너/콜백 미해제**

이벤트 리스너를 등록해놓고 해제하지 않으면, 리스너가 참조하는 객체 전체가 GC 대상에서 제외된다. 특히 익명 클래스나 람다로 리스너를 등록할 때 주의해야 한다. 외부 클래스의 `this` 참조를 암묵적으로 잡고 있기 때문이다.

**캐시 무한 성장**

위의 정적 컬렉션과 비슷하지만, "우리는 캐시 라이브러리를 쓰니까 괜찮다"고 안심하는 경우가 있다. 그런데 `maximumSize`나 `expireAfterWrite` 같은 퇴출 정책을 설정하지 않았다면, 라이브러리를 쓰든 직접 만들든 결과는 같다.

이 세 가지 패턴을 기억해두자. 힙 덤프에서 비정상적으로 큰 컬렉션이 보이면, 그 컬렉션이 어디에 선언되어 있고 누가 원소를 추가하는지를 추적하면 대부분 원인에 도달한다.

## Netflix의 jvmquake — GC 루프를 사전에 차단하다

GC overhead limit exceeded 에러가 터지기 전에, JVM은 한동안 "GC 루프" 상태에 빠져 있다. GC를 돌리고, 조금 회수하고, 또 돌리고, 조금 회수하고 — 애플리케이션은 사실상 멈춘 것이나 다름없는데 JVM은 아직 "살아있다"고 보고한다. 로드밸런서는 이 인스턴스에 계속 트래픽을 보내고, 사용자는 타임아웃을 맞는다.

Netflix는 이 문제를 해결하기 위해 jvmquake라는 도구를 만들었다. 핵심 아이디어는 "GC 부채(debt) 모델"이다. GC에 200ms를 소비하면 부채 카운터에 200ms를 추가하고, 애플리케이션이 실제로 실행된 시간만큼 부채를 상환한다. 부채가 설정한 임계값을 초과하면 JVM을 강제로 종료시킨다.

왜 이게 필요할까? JVM이 스스로 OOM을 던지기까지 기다리면, 그 사이에 수십 초에서 수분간 서비스가 먹통이 된다. jvmquake를 쓰면 GC 루프에 빠진 인스턴스를 빠르게 죽이고, 오케스트레이터(k8s 등)가 새 인스턴스를 띄우도록 유도할 수 있다. "느리게 죽는 것보다 빠르게 죽고 빠르게 살아나는 편이 낫다"는 철학이다.

## 프로덕션 OOM 사례 — IHOP 설정이 하루를 망치다

실제 프로덕션 사례를 하나 살펴보자. 4대의 서버로 구성된 시스템에서 하루에 3회 이상 OOM이 발생했다. 힙 덤프를 분석해도 명확한 릭은 보이지 않았다. 그렇다면 무엇이 문제였을까?

원인은 G1 GC의 IHOP(InitiatingHeapOccupancyPercent) 설정이었다. IHOP는 "힙 사용률이 이 비율을 넘으면 concurrent marking을 시작하라"는 임계값이다. 이 값이 70%로 설정되어 있었는데, 워킹 셋(실제로 사용 중인 객체들의 총량)이 힙의 65%를 차지하고 있었다.

무슨 일이 벌어졌을까? 힙 사용률이 70%를 넘어야 concurrent marking이 시작되는데, 이미 65%가 살아있는 객체다. GC가 시작되어도 회수할 수 있는 건 고작 5% 남짓. 그 사이에 새로운 할당이 들어오면 힙이 가득 차고, Full GC가 발동하고, 그래도 안 되면 OOM이 터진다.

해결은 IHOP 값을 워킹 셋 크기보다 낮게 조정하는 것이었다. 단, 워킹 셋과 동일하게 설정하면 GC가 항상 돌아가며 CPU를 소진하므로, 적절한 여유분을 두는 것이 중요하다. 기억해두자 — IHOP는 "GC가 일을 시작하는 시점"을 결정하는 값이고, 이 시점이 너무 늦으면 GC가 할당 속도를 따라잡지 못한다.

## 컨테이너 환경에서의 JVM 메모리 관리

요즘 대부분의 서비스가 쿠버네티스 위에서 돌아간다. 그런데 컨테이너 환경은 JVM 메모리 관리에 또 다른 층위의 복잡성을 추가한다. 왜 그럴까?

### OOM Killer — JVM 밖에서 날아오는 사형선고

컨테이너에는 메모리 limit이 설정되어 있다. 프로세스가 이 limit을 초과하면, 리눅스 커널의 OOM Killer가 해당 프로세스를 즉사시킨다. JVM 입장에서는 `OutOfMemoryError`를 던질 기회조차 없이 SIGKILL(kill -9)을 맞는 셈이다.

찜찜한 것은, 이때 힙 덤프도 남지 않는다는 점이다. `HeapDumpOnOutOfMemoryError` 옵션은 JVM이 OOM을 인식했을 때만 동작한다. 커널이 프로세스를 죽이면 JVM은 아무것도 모른 채 사라진다. `dmesg`나 쿠버네티스 이벤트에서 "OOMKilled" 상태를 확인해야 비로소 원인을 알 수 있다.

### UseContainerSupport와 MaxRAMPercentage

Java 8u191 이후, JVM은 `-XX:+UseContainerSupport` 옵션(기본 활성화)을 통해 컨테이너의 메모리 limit을 인식한다. 이전에는 호스트 전체 메모리를 기준으로 힙 크기를 계산했기 때문에, 컨테이너 limit보다 큰 힙을 잡아버리는 문제가 있었다.

`-XX:MaxRAMPercentage`를 사용하면 컨테이너 메모리 limit의 특정 비율을 최대 힙으로 설정할 수 있다. 예를 들어 `-XX:MaxRAMPercentage=70.0`이면 컨테이너 limit의 70%를 최대 힙으로 잡는다.

### 힙 + 비힙 합산 — 70~80% 규칙

그런데 JVM이 사용하는 메모리는 힙만이 아니다. Metaspace, 스레드 스택, NIO Direct Buffer, JIT 컴파일된 코드 캐시 등 비힙(non-heap) 메모리도 상당량을 차지한다. 이 모든 것의 합이 컨테이너 limit을 넘으면 OOM Killer가 발동한다.

실무에서는 힙을 컨테이너 limit의 70~80%로 설정하고, 나머지 20~30%를 비힙 메모리 여유분으로 남겨두는 것이 안전하다. 스레드를 많이 사용하는 서비스라면 70%, 스레드 수가 적고 NIO를 별로 안 쓴다면 80%까지 올려도 된다.

### 컨테이너 환경 메모리 설정 체크리스트

컨테이너에서 JVM을 운영할 때, 다음 항목들을 확인해보자.

1. `-XX:+UseContainerSupport`가 활성화되어 있는가? (Java 10+ 기본값이지만, 확인하는 편이 낫다)
2. 힙 크기가 컨테이너 limit의 70~80% 이내인가?
3. `-Xms`와 `-Xmx`를 동일하게 설정했는가? (프로덕션에서는 힙 리사이징 오버헤드를 피하는 것이 좋다)
4. Metaspace 상한(`-XX:MaxMetaspaceSize`)을 설정했는가?
5. 스레드 수 x 스레드 스택 크기(`-Xss`, 기본 1MB)를 비힙 메모리 예산에 포함했는가?
6. NIO Direct Buffer를 사용한다면 `-XX:MaxDirectMemorySize`를 설정했는가?
7. 컨테이너의 request와 limit을 동일하게 설정했는가? (다르면 노드 pressure 시 예측 불가능한 동작이 발생할 수 있다)

## 트러블슈팅 체크리스트 — "서버가 죽었다" 신고를 받았을 때

마지막으로, 새벽에 알림을 받았을 때 당황하지 않고 따라갈 수 있는 순서를 정리해두자.

**1단계: 사망 원인 확인**
- 쿠버네티스 이벤트에서 Pod 상태 확인: OOMKilled인가, 에러 코드가 있는가?
- JVM 로그에서 `OutOfMemoryError` 메시지 확인: 어떤 유형인가?
- GC 로그 확인: 마지막 수분간 GC가 비정상적으로 자주, 길게 돌았는가?

**2단계: 증거 수집**
- 힙 덤프 파일이 남아있는가? (`HeapDumpOnOutOfMemoryError` 설정 여부)
- GC 로그에서 힙 사용량 추이 확인: 점진적 증가(릭) vs 갑작스러운 폭발(버스트 할당)?
- 장애 시점의 트래픽 패턴: 평소 대비 급증이 있었는가?

**3단계: 원인 분류**
- 힙 덤프에서 비정상적으로 큰 객체/컬렉션이 보이면 → 메모리 릭 추적
- GC 로그에서 concurrent marking이 늦게 시작되면 → IHOP 등 GC 설정 검토
- OOMKilled이면서 JVM OOM은 아닌 경우 → 비힙 메모리 합산 초과, 컨테이너 설정 검토
- 특정 시점에 갑작스러운 할당 폭발이 보이면 → 해당 시점의 요청/배치 작업 확인

**4단계: 임시 조치와 근본 해결**
- 임시: 힙 크기 증가, 인스턴스 추가, 문제 엔드포인트 차단
- 근본: 릭 수정, GC 설정 조정, 코드 레벨 할당률 감소

## 마무리

서버가 죽는 순간은 누구에게나 긴장되는 일이다. 하지만 OOM의 유형을 구분할 줄 알고, 힙 덤프를 미리 설정해두고, 메모리 릭의 단골 패턴을 머릿속에 넣어두면 — 적어도 "뭘 봐야 하는지 모르겠다"는 상황은 피할 수 있다. 특히 컨테이너 환경에서는 JVM 바깥(커널 OOM Killer)에서 날아오는 사형선고도 있다는 점을 잊지 말자.

체계적인 접근법이 있다면, 새벽 3시 알림도 조금은 덜 무섭다. 다음 장에서는 이러한 문제를 사전에 방지하기 위한 GC 모니터링과 튜닝 전략을 살펴보자.
