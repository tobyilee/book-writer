# 1장. 왜 백엔드 개발자가 JVM을 알아야 하는가

금요일 오후 5시, 배포 직후 대시보드에 빨간 불이 들어온다고 상상해보자. API 응답 시간이 평소 50ms에서 갑자기 3초로 튀어오른다. 로그를 열어봐도 에러는 없다. 코드 변경은 단순한 문자열 처리 로직 하나뿐이었다. 동료들이 하나둘 퇴근하는 사이, 화면을 붙잡고 있는 개발자의 심정은 어떨까?

원인은 GC였다. Full GC가 수 초간 애플리케이션을 멈춰 세운 것이다. 그런데 난감한 건, 당사자가 GC가 뭔지 정확히 모른다는 점이다.

## 프로덕션에서 만나는 JVM의 그림자

JVM 내부를 몰라도 서버 개발은 된다. Spring Boot 프로젝트를 만들고, API를 작성하고, 테스트를 돌리는 데 JVM 메모리 구조를 알 필요는 없어 보인다. 그렇다면 왜 굳이 알아야 할까?

답은 간단하다. 프로덕션은 로컬 환경과 다르기 때문이다. 트래픽이 몰리고, 메모리가 한정되고, 수십 개의 스레드가 동시에 돌아가는 환경에서는 JVM이 우리 코드의 생사여탈을 쥐고 있다. 구체적인 시나리오를 살펴보자.

**응답 지연 스파이크.** 평소에는 멀쩡하던 서비스가 주기적으로 수백 밀리초에서 수 초씩 멈칫한다. 모니터링 도구를 열어보면 CPU 사용률이 치솟는 구간과 정확히 일치한다. GC가 "Stop-The-World"를 발동시켜 모든 애플리케이션 스레드를 정지시킨 것이다. Uber의 HDFS NameNode에서는 이 STW가 최대 22초까지 발생했다. 22초 동안 서버가 응답을 못 하는 상황을 생각해보자. 찜찜한 정도가 아니라 끔찍한 일이다.

**OOM 크래시.** 새벽 3시에 알림이 울린다. 서버가 죽었다. 로그를 보면 `java.lang.OutOfMemoryError: Java heap space`라는 메시지가 덩그러니 남아 있다. 힙 메모리가 가득 차서 새로운 객체를 할당할 수 없게 된 것이다. 서버를 재시작하면 당장은 살아나지만, 며칠 후 또 같은 일이 반복된다. 메모리 릭인지, 힙 설정이 부족한 건지, 아니면 GC 설정에 문제가 있는 건지 판단할 지식이 없으면 밤잠을 설치는 나날이 이어진다.

**CPU 폭주.** 서버 CPU가 100%를 찍고 있는데, 실제 요청 처리량은 바닥이다. GC가 힙을 회수하려고 안간힘을 쓰고 있지만 충분히 회수하지 못하는 상태다. 이른바 "GC 루프"에 빠진 것이다. Netflix는 이 문제를 해결하기 위해 jvmquake라는 도구를 만들었다. GC에 소비된 시간을 부채처럼 누적하다가 임계값을 넘으면 JVM을 아예 종료시켜버리는 방식이다. GC 루프에 빠져 좀비처럼 떠 있는 서버보다는 깔끔하게 죽고 다시 뜨는 편이 낫다는 판단이었다.

## GC를 모르면서 GC 때문에 죽는 아이러니

재미있는 사실이 있다. 대부분의 백엔드 개발자는 GC의 존재를 알고 있다. "자바는 메모리를 자동으로 관리해준다"는 사실도 안다. 그런데 정작 GC가 언제 돌아가는지, 왜 오래 걸리는지, 어떻게 하면 덜 부담을 줄 수 있는지는 모르는 경우가 많다.

이건 마치 자동차 운전자가 "엔진오일이 자동으로 교체된다"고 믿고 있는 것과 비슷하다. 물론 GC는 자동이다. 하지만 자동이라고 해서 공짜는 아니다.

Uber의 사례를 보자. 메트릭 수집 라이브러리의 재시도 간격이 1초가 아니라 1밀리초로 설정되어 있었다. 이 작은 설정 하나가 초당 400MB의 가비지를 생성했고, 결과적으로 2,258회의 GC pause를 유발했다. 평균 177ms씩, 합치면 어마어마한 시간이다. 코드 한 줄의 실수가 GC를 통해 증폭되어 서비스 전체를 흔든 것이다.

그렇다면 어떻게 해야 할까? GC의 동작 원리를 이해하고 있었다면, 객체 할당률이 비정상적이라는 사실을 모니터링만으로도 금방 잡아낼 수 있었을 것이다. JVM 내부를 아는 것은 장애 대응 시간을 단축시키는 가장 확실한 무기다.

## JVM 30초 오버뷰

깊은 이야기는 다음 장부터 풀어갈 테니, 여기서는 전체 그림만 빠르게 훑어보자. JVM은 크게 세 덩어리로 구성된다.

**클래스 로더 서브시스템.** 우리가 작성한 `.class` 파일을 메모리에 올리는 역할을 한다. 런타임에 동적으로 클래스를 로딩하고, 검증하고, 초기화한다. "내가 짠 코드가 어떻게 실행 가능한 상태가 되는가"에 대한 답이 여기에 있다.

**런타임 데이터 영역.** 프로그램이 실행되는 동안 데이터가 머무는 공간이다. 힙, 스택, 메서드 영역 등으로 나뉘는데, GC가 주로 활동하는 무대가 바로 힙이다. 객체가 태어나고, 살다가, 죽는 곳이다. 이 영역의 구조를 이해하면 메모리 문제의 절반은 해결된 셈이다.

**실행 엔진.** 바이트코드를 실제 기계어로 바꿔 실행하는 부분이다. 인터프리터가 한 줄씩 해석하다가, 자주 호출되는 메서드는 JIT 컴파일러가 네이티브 코드로 변환해 성능을 끌어올린다. 그리고 가비지 컬렉터가 여기 속해 있다. 더 이상 쓰이지 않는 객체의 메모리를 회수하는 조용하지만 강력한 존재다.

이 세 요소가 맞물려 돌아가는 방식을 이해하면, 프로덕션에서 벌어지는 기이한 현상들이 설명 가능한 것으로 바뀐다. 마법이 아니라 메커니즘으로 보이기 시작한다.

## 이 책에서 다룰 것들

이 책은 JVM 스펙 문서를 옮겨놓은 교과서가 아니다. 백엔드 개발자가 프로덕션에서 실제로 부딪히는 문제들을 중심으로, 필요한 만큼의 깊이로 JVM과 GC를 파헤친다.

메모리 구조부터 시작해서, 세대별 GC가 왜 그렇게 설계되었는지 살펴보고, G1부터 ZGC까지 현대적인 GC 알고리즘들의 작동 원리를 비교한다. Netflix, Uber, LinkedIn 같은 기업들이 실전에서 어떻게 GC를 다루었는지도 함께 본다. 단순히 "이런 옵션이 있다"가 아니라, "이 상황에서 왜 이 선택이 나왔는가"를 추적하는 방식으로 전개한다.

읽는 방법은 자유롭다. 순서대로 읽어도 좋고, 당장 GC 튜닝이 급하다면 해당 장으로 바로 넘어가도 된다. 다만 2장의 메모리 구조를 먼저 읽어두면 이후 내용이 훨씬 수월하게 들어올 것이다.

## 마무리

JVM을 모르는 백엔드 개발자는 좋은 날씨에만 운전할 수 있는 운전자와 비슷하다. 평소에는 문제없이 달리지만, 폭우가 쏟아지는 순간 — 트래픽이 폭증하거나, 메모리가 한계에 다다르거나, GC가 예상치 못한 타이밍에 걸릴 때 — 속수무책이 된다.

이 책을 덮을 즈음에는 GC 로그를 읽고, 적절한 GC를 선택하고, 메모리 문제를 체계적으로 진단할 수 있는 개발자가 되어 있을 것이다. 그러면 금요일 오후 5시의 장애도, 새벽 3시의 OOM도 더 이상 공포가 아니라 풀어야 할 퍼즐이 된다.

다음 장에서는 JVM 메모리 구조를 본격적으로 파헤쳐보자. 객체가 태어나서 죽기까지의 여정을 따라가다 보면, GC가 왜 그렇게 행동하는지 자연스럽게 이해하게 될 것이다.
