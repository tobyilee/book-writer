## 라운드 1

### 1. 종합 평가

**강점 3개**

1. **"같은 mini_agent를 세 번 짓는다"는 학습 장치가 계획서 전반에 일관되게 박혀 있다.** 1장의 길 안내 → Part 1 도입글 → 6장 헤딩 → 9장 헤딩 → 8절 표까지 매번 같은 약속을 반복한다. 이건 책 차별화의 핵심이고, 흔들리지 않게 잘 못 박았다.
2. **"책에서 정한 입장" 6개를 명시적으로 표로 정리했다.** reference §9의 6개 논쟁점에 1:1로 답을 정해놓고, 그 입장이 어느 챕터에서 노출되는지 추적 가능하다. 입문서가 흔히 빠지는 "모든 의견을 공평하게 소개하다 입장이 없어지는 함정"을 피했다.
3. **챕터 사이의 연결고리(이전→다음)가 모든 챕터에 1줄로 명시되어 있다.** 2장 "한 바퀴 round-trip" → 3장 "while로 감싼다", 5장 끝 "이걸 매번 다시 짤 수는 없겠다" → 6장 등 내러티브 동력이 손에 잡힌다.

**약점 3개**

1. **3장이 과적되어 있다.** 11,000자 챕터 하나에 ReAct 루프 직접 구현 + 도구 3개 추가 + 종료 조건 분류 + Plan-and-Solve/Reflexion/ToT 비교까지 들어간다. ReAct 본문만으로 11,000자가 빠듯한데 학술 패턴 3개를 "한 단락씩"으로 처리하면 둘 다 망친다 — 코드 흐름은 끊기고, 학술 비교는 얕다.
2. **reference §6.6(Claude Code/Cursor/Devin 아키텍처)이 챕터 수준의 자산인데 계획서가 흩뿌려 쓴다.** 1장 한 줄, 4장 한 줄, 5장 hook으로만 등장한다. 이 자료는 "에이전트는 결국 단순 루프 + 좋은 도구 + 좋은 컨텍스트"라는 책 전체 입장의 *실증 근거*다. 책의 입장을 떠받치는 가장 강한 외부 증거를 양념처럼 흘리고 있다.
3. **Anthropic의 5가지 워크플로 패턴(prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer)이 계획에서 사라졌다.** reference §2의 mid-callout으로 "책 전반에 반복 등장하므로 함께 못 박는다"고 명시되어 있는데, 정작 plan의 어느 챕터에도 이 5종을 정면으로 다루는 슬롯이 없다. 1장에서 "Agent vs Workflow 스펙트럼"으로만 압축되었다.

**한 줄 평결: 소규모 수정.** 12장 구조와 내러티브 골격은 그대로 두되, 3장 분할 / Claude Code 위상 격상 / 워크플로 5종 명시적 자리 잡기 — 이 셋만 반영하면 완성도가 한 단계 올라간다.

### 2. 구체적 피드백

> **[3장: ReAct 루프 짓기] — 심각도: 높음**
> 문제: 11,000자에 (a) ReAct 루프 직접 구현, (b) 도구 3개로 확장, (c) hallucinated tool name·malformed JSON 같은 실패 모드, (d) 종료 조건 3분류, (e) Plan-and-Solve/Reflexion/ToT/Voyager까지 들어간다. (e)는 정량 근거(ToT 4%→74%, Reflexion HumanEval 91%)가 reference §3.4에 정리되어 있을 만큼 비중 있는 학술 자산인데 "한 단락씩"으로 처리하면 코드 흐름은 끊기고 학술 패턴은 표면만 긁힌다.
> 제안: 3장은 "ReAct 루프 + mini_agent v1 + 종료 조건"까지로 좁히고 9,000자로 줄인다. Plan-and-Solve / Reflexion / ToT / Voyager / LATS는 4장 다음에 **신규 5장 "루프 모양의 변종들 — Plan-and-Solve, Reflexion, ToT"**(7,000~8,000자)을 끼우거나, 12장이 너무 무거우면 부록 C로 옮긴다. 본문에 두는 게 낫다 — 입문자가 "ReAct만 알면 끝인가?"라는 의문을 한 챕터로 정리해주는 효과가 크다.
> 근거: reference §3.4의 정량 근거 밀도(ToT 4%→74%, Reflexion 91%, ALFWorld +34%p)는 "한 단락"으로 다 못 담는다. 또한 입문자가 "내 에이전트가 막혔을 때 ReAct 말고 뭐가 있나"를 책 한가운데서 한번 답해주는 안전판 역할을 한다.

> **[1장 / 5장 hook / 책 전반: Claude Code·Cursor·Devin의 위상] — 심각도: 높음**
> 문제: reference §6.6은 "가장 잘 동작하는 에이전트조차 단순 루프 + 좋은 도구 + 좋은 컨텍스트"라는 책의 핵심 주장을 떠받치는 실증 자료다. 그런데 계획서에서는 1장의 한 줄 ("Claude Code, Cursor, Devin이 공통으로 가진 모양 — 단순 루프 + 좋은 도구 + 좋은 컨텍스트")로 압축되었고, 4장에서 "Claude Code는 vector DB가 아니라 grep을 쓴다"는 한 줄 시사가 떠다닌다. 흩뿌리는 만큼 영향력이 약해진다.
> 제안: 1장 "주요 내용"의 "2026년 기준 가장 잘 동작하는 에이전트들" 항목을 별도 섹션으로 격상(1장 분량 7,000자 → 8,500자, 그중 1,500자를 이 사례 분석에 할애). Claude Code 3단계 루프(gather context → take action → verify), Cursor 인라인+백그라운드, Devin 비동기 격리 환경을 하나씩 그림 한 장으로 묘사. "이 책에서 짤 mini_agent도 결국 이 모양에 수렴한다"를 1장 끝의 약속으로 만든다.
> 근거: reference §6.6의 자료 밀도(Augment 가이드, Mindstudio, 공식 문서 3건)는 한 줄로 끝낼 양이 아니다. 또한 입문자가 1장을 덮을 때 "이게 진짜 현실의 모양이구나"를 손에 쥐고 가야 Part 1 5장까지 끈기 있게 따라온다.

> **[1장 / Part 1 도입 / 9장: Anthropic 5가지 워크플로 패턴] — 심각도: 중간**
> 문제: reference §2 끝에 "다섯 가지 워크플로 패턴(prompt chaining, routing, parallelization, orchestrator-workers, evaluator-optimizer)은 책 전반에 반복 등장하므로 함께 못 박는다"고 명시되어 있다. 그런데 plan은 이걸 1장의 "Agent vs Workflow 스펙트럼"으로 한 줄 처리했고, 그 5종이 본문 어디에서 다시 호출되는지 추적이 안 된다.
> 제안: 두 가지 길 중 하나.
> - (A) 1장 "주요 내용"에 "5종 워크플로 패턴을 한 표로 압축, 책 본문에서 이 표를 참조 좌표로 쓴다"를 명시. 그리고 9장 도입에서 "이 5종 중 orchestrator-workers와 evaluator-optimizer가 LangGraph가 필요한 자리다"를 반복 인용.
> - (B) Part 1 도입글 또는 6장(LangChain LCEL)에서 5종 중 prompt chaining·routing·parallelization을 LCEL의 `RunnableParallel`·conditional 분기와 1:1 매핑해서 보여준다. 그러면 "추상화가 무엇을 대신해 주는지"가 5종 패턴 어휘로 깔끔하게 정리된다.
> (A)와 (B)를 합쳐도 200~300자 안에 들어간다. 추가 분량 거의 없음.
> 근거: 5종 워크플로 어휘는 2025~2026년 에이전트 담론의 표준 어휘다. 입문서가 이걸 *명시적으로 호출*하지 않으면 독자가 다른 자료를 읽을 때 어휘 매핑을 못 한다.

> **[6장: LangChain·LCEL 재구현] — 심각도: 중간**
> 문제: "200줄 → 90줄, 단 import 6줄" 같은 수치가 본문에 박혀 있는데 이게 *실측치인지 예상치인지* 계획서에서 알 수 없다. 또한 "줄어든 줄 수와 늘어난 추상의 깊이를 같이 본다"라는 약속이 가장 추상적이다 — 줄 수는 측정 가능하지만 "추상의 깊이"는 무엇으로 보여줄지 합의가 안 됐다.
> 제안: 6장 핵심 코드 예제 약속을 좀 더 구체화한다.
> - (a) `mini_agent.py`(SDK)와 `mini_agent_lc.py`(LangChain)의 **import 그래프 깊이 비교**(직접 의존성 트리)
> - (b) 한 동작(예: tool 등록)이 SDK에서는 N줄, LangChain에서는 M줄이지만, LangChain 쪽 M줄을 따라가면 `BaseTool` → `Runnable` → ABC 추상 트리가 몇 단 더 깊다는 사실을 *코드 트레이스*로 보여주기
> - (c) "추상이 늘어난 자리"를 한 챕터에 3~4개 구체 예시로 못 박기 (예: `@tool` 데코레이터의 inspect 마법, LCEL `__or__` 오버로딩의 부작용, `RunnableConfig` 전파)
> 줄 수 수치는 저술 단계에서 실측한 뒤 그 숫자로 갱신 (예상치 박아두고 가지 말 것).
> 근거: 입문자가 6장에서 "그래서 LangChain은 좋은 거야 나쁜 거야"를 결정하려면 측정 가능한 비교 좌표가 필요하다.

> **[9장: LangGraph 도입] — 심각도: 중간**
> 문제: "단순 ReAct는 그래프가 과한가 — 그렇다. 의도적으로 과하게 짜본다"는 메타 코멘트는 좋다. 그런데 이게 9장 본문에서 *내러티브적으로* 어떻게 풀리는지 막연하다. 독자가 9장 중반에서 "왜 이걸 굳이 LangGraph로 짜고 있지?"라고 의문을 가질 위험이 크다 — Part 3 도입글에서 "사이클·HITL이 필요한 진짜 사례"를 약속해놓고 9장에서 똑같은 ReAct를 그래프로 옮기기만 한다면.
> 제안: 9장 후반에 의도적 페인포인트 하나를 끼운다 — 예: "사용자 입력이 빈 검색 결과를 돌려받으면 재시도 한 번 한다"는 요구사항을 추가하고, 이걸 LCEL로 짜려 하면 막힌다는 사실(DAG 한계)을 30~50줄짜리 코드로 *실제로 막혀보게* 만든 뒤, 그래프로 푸는 모습을 보여준다. 그래야 "그래프가 과하다"가 "그래프가 마침 필요해진다"로 자연스럽게 넘어간다.
> 근거: 9장의 핵심 질문이 "chain만으로 안 되는 진짜 상황은 무엇인가"인데, 그 상황을 직접 만나본 적 없이 그래프로 갈아탄다면 학습이 일어나지 않는다. reference §5.1의 "사이클" 항목이 이 자리의 정확한 자료다.

> **[10장: 멀티 에이전트] — 심각도: 중간**
> 문제: 9,000자에 (a) 4종 패턴 분류, (b) Supervisor 데모 코드(리서처/라이터/에디터), (c) 핸드오프 함정, (d) AutoGen/MetaGPT/CrewAI 비교표, (e) Grid Dynamics 후일담, (f) "supervisor + 1~2 worker가 입문 한계" 권고, (g) 멀티가 정말 필요한 두 가지가 들어간다. 9,000자에 7가지는 빡빡하다.
> 제안: (d) 비교표를 부록 B(다루지 않은 길로 가는 지도)로 옮기거나 1페이지짜리 박스 처리. 또는 "리서치/라이터/에디터" Supervisor 데모를 50~80줄에서 *진짜로 50~80줄*로 줄이고, 데모 코드 옆에 핸드오프 함정 시각화(같은 정보를 3번 다시 인용하는 ping-pong) 1개를 끼운다. 어느 쪽이든 9,000자는 유지 가능하지만 *밀도 분배*를 미리 정해두지 않으면 저술 단계에서 늪에 빠진다.
> 근거: 8절 분량표가 9,000자인데, 위 7개를 다 쓰면 13,000자가 된다. plan-review의 임무 중 하나는 "저술 단계에서 늘어날 게 뻔한 챕터를 미리 표시"하는 것이다.

> **[12장: 평가·관찰성·보안·비용 — 마무리] — 심각도: 중간**
> 문제: 10,000자에 (a) 평가 4종 분류, (b) 도구 5종 비교표(LangSmith/Braintrust/Ragas/DeepEval/Arize Phoenix), (c) 추천 조합 2종, (d) 관찰성 3축, (e) 보안 4종 + 완화 4종, (f) lethal trifecta 박스, (g) 비용 캡 4종(5장 재인용), (h) 마지막 한 문단의 송사. 8개 토픽이 10,000자에 들어간다.
> 제안: 두 가지 길.
> - (A) 보안을 12장에서 분리해 **11장과 12장 사이 별도 챕터**로 두기 (책 13장 구조로 확장). 분량 6,000~7,000자. lethal trifecta + OWASP 4종 + 완화 4종 + dual-LLM 패턴이 한 챕터를 받을 만하다.
> - (B) 12장 분량을 12,000~13,000자로 늘리고, 평가 도구 비교표를 표 자체만 부록으로 옮긴다. 본문에서는 "(부록 C에 도구 비교표) 추천 조합 두 가지만 본문에서 다룬다"로 압축.
> 둘 중 어느 쪽이든 OK인데, "10,000자에 8토픽"은 그대로 두면 마무리 챕터가 자료 카탈로그로 끝날 위험이 있다.
> 근거: 보안은 책의 "프로덕션 근처에 둘 수 있겠다"는 마무리 약속의 절반을 차지하는 주제다. 비용 통제와 함께 묶여 마지막 챕터에 압축되면 둘 다 짧아진다. reference §6.3-6.4 자료 밀도(OWASP 3건, Lakera, Datadog, AppSec Santa)는 한 챕터로 충분히 갈 양이다.

> **[reference §6.5 벤치마크 ToolBench·API-Bank의 처리] — 심각도: 낮음**
> 문제: "다루지 않는 것" 절에서 "ToolBench·API-Bank는 한 단락 소개. 측정 방법 자세히는 안 다룸"이라고 명시했다. 그런데 본문에서 이 한 단락이 어느 챕터에 들어가는지 plan에 표시가 없다.
> 제안: 12장 평가 4종 분류에서 "task eval" 항목 옆에 "공개 벤치마크로는 ToolBench, API-Bank가 있다 (자세히는 다루지 않는다)" 한 줄을 명시. 또는 부록 B에 흡수.
> 근거: 입문자가 "내 에이전트를 어떤 벤치마크에 돌려볼 수 있나"를 책에서 한 번도 못 본다면, "다루지 않는다"는 결정 자체를 모른다.

> **[제목 / 컨셉 일관성] — 심각도: 낮음**
> 문제: 추천 제목 "맨손으로 짓는 AI 에이전트"는 Part 1 컨셉과는 완벽히 맞는데, 책 *전체*가 LangChain·LangGraph까지 가는 책이라는 사실과는 약간 어긋난다. 부제 "Python SDK에서 LangChain, LangGraph까지"가 보충하지만, 짧은 SNS 인용·검색 결과에서 부제가 잘리면 "맨손으로 짓는 책 = LangChain 안 쓰는 책"으로 오해할 위험이 있다.
> 제안: 부제를 본 제목 가까이 묶거나, 표지 디자인 단계에서 부제 폰트를 본 제목 80% 정도로 크게. (제목 자체는 유지 추천)
> 근거: 표지·서점 검색 노출 관점.

### 3. 확인 요청 3건에 대한 의견

**(1) 제목 결정 — A vs C**

**A "맨손으로 짓는 AI 에이전트" 추천.** 이유:

- "맨손"은 차별화 단어다. 같은 분야 책 80%가 "LangChain·LangGraph로 만드는…"으로 시작한다. 그 사이에 검색·서가에서 한 번에 식별된다.
- "세 번 다시 짓는다"라는 책의 핵심 학습 장치를 부제가 광고한다. C는 "본질에서 시작해 추상화까지"라는 표현이 좀 학구적이고, 정작 책의 가장 큰 차별점(같은 코드를 세 번)을 광고하지 못한다.
- 토비 문체 — 청유형 "함께 짓자"와 "맨손" 의 결합이 자연스럽다. C의 "본질에서 시작해 추상화까지"는 평어체보다는 격식체와 어울리는 톤이다.

C의 장점은 "검색 친화"라는 plan의 자체 진단인데, 부제에 "Python SDK", "LangChain", "LangGraph" 키워드가 다 들어가면 검색 측면도 충분히 보완된다.

**(2) 챕터 수 12개의 적정성**

**12장 + 부록 2개는 적정. 단, 위에서 제안한 대로 (a) 3장에서 학술 패턴(Plan-and-Solve/Reflexion/ToT)을 분리해 신규 챕터로 또는 (b) 12장에서 보안을 분리해 별도 챕터로 만든다면 13장이 된다. 둘 중 하나는 진지하게 검토할 가치가 있다.**

근거:
- 8절 분량표를 합산해보면 115,000자인데, 3장(11,000)·9장(11,000)이 위에서 지적한 대로 과적이라 저술 단계에서 13,000~14,000자로 늘어날 가능성이 높다. 12장도 8토픽 10,000자라 12,000자로 늘어난다.
- 실제 본문 분량은 130,000~135,000자에 수렴할 가능성. 목표 범위(80k~150k)의 상단으로 간다.
- 챕터 분할 없이 "분량 박스" 안에 욱여넣으면 어느 챕터에서 무엇이 잘려나갈지 모른다. 분량이 좀 늘더라도 챕터 한 개 추가가 *예측 가능한* 선택이다.

**(3) smolagents·CrewAI·AutoGen 노출 정도**

**현재 계획(비교표 한 장 + smolagents 한 페이지 본문) 유지에 동의.** 이유:

- 책의 정체성이 "맨손 → LangChain → LangGraph"인데, 여기에 AutoGen/CrewAI 챕터를 끼우면 정체성이 흔들린다. 입문자에게 "그래서 뭘 골라야 하지?"의 안개를 더 키운다.
- reference §5.4의 학술 정리는 비교표 한 장 분량에 정확히 맞는 양이다. 더 깊이 가면 출처가 부족해진다 (AutoGen·CrewAI 후일담은 reference에 거의 없다).
- 단, **smolagents는 본문(8장 한 페이지)에 두는 게 맞다.** "LangChain의 미니멀 대안"이라는 위치를 차지하므로 책의 입장 정리에 기여한다. AutoGen/CrewAI/MetaGPT는 "다른 길"이지 "대안"이 아니므로 비교표로 충분.

다만 부록 B "다루지 않은 길로 가는 지도"에서 AutoGen·CrewAI 각각 한 단락("이걸로 시작하려면 여기를 읽으세요")을 명시적으로 적어주면 좋다. 현재 계획에 "각각 한 단락씩"이라고 적혀 있어 OK.

### 4. book-planner에게 보낼 액션 아이템 (high-priority)

다음 5개를 `02_plan.md`에 반영하기를 권한다. 1~3은 반드시, 4~5는 가능하면.
