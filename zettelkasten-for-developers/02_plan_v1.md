# 저술 계획: 개발자를 위한 옵시디언 제텔카스텐

## 책 제목 후보 3개

### 후보 1. **개발자의 두 번째 뇌 — 옵시디언으로 짓는 제텔카스텐**
- **톤:** 따뜻하고 입문 친화적. "두 번째 뇌"라는 익숙한 메타포로 진입 장벽을 낮춤.
- **포지셔닝:** Tiago Forte(BASB)의 키워드를 차용하면서도 본문에서는 "BASB는 자료, Zettelkasten은 사고"라는 차별점을 명확히 함. 일반 PKM 독자도 흡수할 수 있는 가장 넓은 그물.
- **약점:** "두 번째 뇌" 키워드는 이미 포화. 차별화 메시지가 부제에 의존.

### 후보 2. **연결되는 노트 — 개발자가 옵시디언으로 생각을 키우는 법**
- **톤:** 시적이고 메타포 중심. "연결"이라는 Zettelkasten의 핵심을 제목에 박음.
- **포지셔닝:** 도구(옵시디언)보다 작용(연결·성장)을 앞세워 5년 후에도 유효한 제목. 개발자가 "키운다"는 능동성과 어울림.
- **약점:** 검색 키워드(Zettelkasten, 제텔카스텐) 노출이 약함. SEO에 불리.

### 후보 3. **개발자를 위한 제텔카스텐 — 옵시디언으로 코드와 글을 함께 키우는 법** (추천)
- **톤:** 실용적이고 구체적. 제목에 대상(개발자)·방법(제텔카스텐)·도구(옵시디언)·결과물(코드와 글)이 모두 들어감.
- **포지셔닝:** 한국 시장에 출간된 『세컨드 브레인을 구축하는 제텔카스텐 & 옵시디언』(생산적생산자, 2024)과 명확히 구분. 그 책이 "일반 지식 노동자용"이라면 이 책은 "개발자 전용". 부제의 "코드와 글을 함께"는 ADR/TIL/블로그/책쓰기까지 묶는 약속.
- **약점:** 제목이 길다. 표지 디자인에서 "개발자를 위한 제텔카스텐" + 부제 분리가 필요.

**추천: 후보 3.** 개발자 시장에서는 모호한 메타포보다 "내 직무가 호명되는" 직설적 제목이 강하다. 검색 키워드(제텔카스텐, 옵시디언, 개발자)가 모두 들어가 발견성도 좋다. 부제의 "코드와 글을 함께 키우는 법"이 이 책의 차별점(코드 리뷰·ADR·블로그·발표·책쓰기로 확장)을 한 줄에 압축한다.

## 책 특성

- **장르:** 에세이형 기술서 (실용 기술서 + 개발자 자기계발). 코드 예제보다 워크플로우·템플릿·사고 도구 중심. Toby의 회고적 어조와 결합.
- **예상 분량:** 9개 챕터 × 평균 17페이지 = 약 153페이지. 한글 글자 수로는 약 18~22만 자.
- **난이도:** 중급. 옵시디언을 한 번도 써보지 않은 개발자도 따라올 수 있되, 3년차 이상이면 실무 통합 사례에서 더 큰 가치를 느낄 수 있도록 두 층위로 설계.
- **독자 여정:**
  - **시작점:** "노트를 많이 쌓아두지만 다시 안 본다", "TIL이나 블로그를 써보려 했지만 흐지부지", "GPT/Claude를 쓰지만 컨텍스트가 매번 사라진다", "옵시디언은 들어봤지만 어디서 시작할지 모르겠다"
  - **도달점:** "매일 30분으로 atomic note 1~2개를 만들고, 그게 쌓여서 자연스럽게 ADR·블로그·컨퍼런스 발표·책 한 챕터로 흘러나오는" 개인 지식 시스템을 자기 손으로 운영
- **톤앤매너:**
  - **평어체 + 청유형 적극 활용** ("같이 해보자", "한 번 시도해 보자")
  - **수사적 질문**으로 독자의 사고를 끌어들임 ("그런데 왜 우리는 노트를 다시 안 볼까?")
  - **저자의 실패담을 먼저 꺼냄** — Toby도 처음엔 Collector's Fallacy에 빠졌고, 플러그인 튜닝에 1주를 날렸고, atomic note를 너무 잘게 쪼개 길을 잃은 경험을 공유
  - **단정 대신 권유** — "이렇게 해야 한다"가 아니라 "이렇게 해보면 무슨 일이 일어나는지 알게 된다"
  - **개발자 은유 적극 활용** — Permanent note를 "API의 안정적 인터페이스"로, MOC를 "라우팅 레이어"로, fleeting note를 "stdin 버퍼"로 비유
  - **불확실성 명시** — Zettelkasten 효과 RCT가 없다는 사실을 숨기지 않음. "현재 합의 부족"을 인정하면서 독자에게 자기 실험을 권함

### 챕터 4. 매일 30분 워크플로우 — Fleeting부터 Permanent까지

- **핵심 질문:** 하루 중 언제, 무엇을, 어떻게 쪼개서 노트로 만드는가?
- **주요 내용:**
  - 하루 흐름 시뮬레이션 — 출근길 fleeting → 점심 후 literature → 저녁 30분 permanent
  - Fleeting note의 3가지 통로 — 모바일 캡처 앱(Drafts/Fleeting Notes), Apple Shortcut, 음성→Whisper
  - "한 시간 읽었으면 두 시간 가공" 휴리스틱
  - Literature note 쓰는 법 — 한 자료 = 한 노트, 자기 언어 재구성, 출처 메타데이터
  - Permanent note의 작성 순간 — "일주일 후의 나도 이해하는가" 체크
  - atomic의 적정 입자 — "한 주장(claim)" 정의로 파편화 불안 해결
  - 주간 리뷰 30분 — Inbox 비우기, 끊긴 연결 잇기, 한 주의 baby-MOC 만들기
- **예상 분량:** 19페이지
- **포함할 사례/예시:** Toby의 실제 하루 노트 흐름 스크린샷 시리즈, fleeting → literature → permanent 변환 사례 1건 (예: "RAG vs Fine-tuning" 관련 영상에서 atomic note 3장이 만들어지는 과정), 주간 리뷰 체크리스트 템플릿

### 챕터 5. 개발자의 일상에 녹이기 — 코드 리뷰·ADR·TIL·회고

- **핵심 질문:** 업무 시간에 짬내서 노트할 수 있을까, 그리고 그게 회사 일에 도움이 될까?
- **주요 내용:**
  - **코드 리뷰 노트:** PR을 읽으며 떠오른 패턴을 즉시 atomic note로. 같은 패턴이 N번 반복되면 baby-MOC로 승격
  - **ADR(Architecture Decision Record):** Matteo Paoli의 MADR + Templater 자동화. 결정 카드는 백링크로 영향받는 시스템 노트와 연결되어 "결정의 진화" 추적 가능
  - **TIL과 Permanent 폴더 분리** — TIL은 학습 기록, Permanent는 자기 주장. 한국 개발자 문화에 잘 맞는 두 층위
  - **회의록·일감 관리:** 송요창 패턴 — 이모지 메타데이터(📅 due, 🏃 in-progress) + 백링크
  - **온콜·장애 대응 회고:** 5 Whys 루트 분석을 atomic note로 분해해 다음 장애에서 재사용
  - **오픈소스 기여 노트:** 라이브러리 internals를 읽으며 만든 atomic note는 다음 PR·이슈 토론에서 인용
  - **회사 vault vs 개인 vault** — 분리 운영, 보안 체크리스트
- **예상 분량:** 20페이지
- **포함할 사례/예시:** ADR 템플릿 전문(MADR), Toby의 PR 리뷰에서 만들어진 atomic note 3장 시리즈, "Postgres VACUUM 학습 사건" 케이스 스터디(TIL → permanent → 동료 공유 → 사내 발표로 진화), 회사 vault 보안 체크리스트

### 챕터 6. AI와 함께 — Claude·Claude Code·NotebookLM의 자리

- **핵심 질문:** AI에게 어디까지 맡기고 어디서부터는 내가 해야 사고가 죽지 않는가?
- **주요 내용:**
  - **3가지 입장 정직하게 제시** — 회의론(사고 대체), 활용론(분업), 반-Zettelkasten론(raw notes + LLM 검색)
  - **이 책의 입장 — 의도적 분업** — AI는 캡처·요약·연결 추천, 사람은 atomic claim 결정과 영구 노트 작성
  - **NotebookLM 패턴:** 긴 논문/영상을 NotebookLM에 업로드 → 출처 추적 가능한 Q&A → 자기 언어로 atomic note 작성. vault Markdown을 Google Drive 동기화하는 셋업
  - **Smart Connections:** vault 임베딩 → 작성 중 노트와 의미적으로 가까운 노트 자동 표시. "내가 잊고 있던 관련 노트" 발견용
  - **Claude Desktop + Obsidian MCP Tools:** Claude가 vault 파일을 직접 읽고 쓸 수 있음 — Permanent note 초안 작성을 Claude에게 위임 후 사람이 검수
  - **Claude Code + Claudian:** Claude Code를 vault 안의 "에이전트 협업자"로. 코드 작업 컨텍스트를 vault에서 가져와 일하고, 결과를 vault에 다시 저장
  - **위험 신호 3가지:** AI가 만든 atomic note 그대로 저장 / 노트 사이 연결을 AI에게 위임 / hallucination이 영구 노트로 진입
  - 한국 개발자의 빠른 AI 채택 흐름 (디지털투데이·청년개발자신문 보도)
- **예상 분량:** 19페이지
- **포함할 사례/예시:** NotebookLM에 논문 업로드 → atomic note 산출 워크플로우 캡처, MCP로 Claude가 vault에 permanent note 초안을 만드는 실제 대화 로그, Smart Connections가 발견한 "잊고 있던 연결" 사례, AI hallucination이 침투한 사례와 차단법

### 챕터 7. 노트가 글이 되고 글이 책이 되는 — 출력의 단계들

- **핵심 질문:** atomic note가 어느 시점에 블로그 글이 되고, 어느 시점에 발표·책으로 자라는가?
- **주요 내용:**
  - **출력 5단계 모델** — atomic → cluster → MOC → 블로그 글 → 컨퍼런스 발표/책 한 챕터
  - **velog/티스토리 직전 단계로 Permanent Note 활용** — 한국 개발자에게 블로그는 곧 포트폴리오. Permanent note 5~7장을 묶으면 블로그 글 한 편이 자연스럽게 떠오름
  - **컨퍼런스 발표 준비:** atomic note 5~7개 → 발표 1편. 실제 한국 컨퍼런스 발표 준비 사례
  - **개발 책·전자책 쓰기:** atomic note → 챕터 → 책. 이 책 자체가 그 산물임을 메타적으로 공개
  - **Excalidraw·Advanced Slides:** 다이어그램·발표 자료를 vault 안에서 함께 관리
  - **외부 공개 vault 사례** — Andy Matuschak's notes, MolecularNotes — 영감과 한계
  - **인용·라이선스 위생** — 자기 언어 재구성과 표절의 경계, 출처 추적
- **예상 분량:** 18페이지
- **포함할 사례/예시:** Toby의 atomic note 묶음이 블로그 글로 변환된 실제 사례(diff 형태), 컨퍼런스 발표 슬라이드 → 어떤 atomic note에서 왔는지 매핑, 이 책의 한 챕터가 어떤 atomic note들에서 자라났는지 보여주는 트리

### 챕터 8. 임계점을 넘기 위한 100일 — 좌절 구간 통과법

- **핵심 질문:** 200~500개 노트가 쌓이기 전 ROI가 안 보이는 좌절 구간을 어떻게 넘기는가?

## 챕터 간 흐름 (내러티브 아크)

이 책은 **공감 → 이해 → 실행 → 통합 → 지속 → 의미**의 6악장 구조를 따른다.

- **공감 악장(1장):** 독자의 현재 고통(노트 묘지)을 호명한다. 추상적 PKM 이론으로 시작하지 않고 "당신의 Notion DB가 죽어 있는 이유"부터 짚는다. 이 한 장으로 "이 책은 나를 위한 책"이라는 신호를 보낸다.

- **이해 악장(2장):** 너무 길어지기 전에 Zettelkasten의 핵심을 한 호흡에 끝낸다. 학술적 깊이는 인지과학 4효과 + Extended Mind까지만, 그 이상은 의도적으로 자제. "왜 작동하는가"의 질문에 답하되 학자처럼 다루지 않는다.

- **실행 악장(3·4장):** 이론에서 손으로 빠르게 내려간다. 3장은 셋업(공간을 마련), 4장은 일상 루틴(시간을 마련). 두 장이 끝나면 독자는 이미 "이번 주말부터 할 수 있다"는 감각을 가져야 한다.

- **통합 악장(5·6장):** 개발자 일상에 녹이는 단계. 5장은 인간 협업(코드 리뷰·ADR·TIL·회의록), 6장은 AI 협업(Claude·NotebookLM·MCP). 이 책의 가장 굵은 차별점이 여기에 있고, 한국 개발자의 2026년 현실(Claude Code·MCP 빠른 채택)을 반영한다.

- **지속 악장(7·8장):** 산출이 보이기 시작하는 단계와 좌절 구간 통과 단계. 순서가 직관과 반대인 이유는 — 7장에서 "이렇게 글·발표·책이 나온다"는 약속을 먼저 보여줘야, 8장의 "100일 좌절 구간을 넘는 법"이 추상적 인내가 아닌 구체적 보상을 향한 인내가 된다.

- **의미 악장(9장):** 기술서의 클리셰인 "당신만의 시스템을 만드세요"로 끝내지 않는다. Extended Mind의 철학적 질문을 다시 꺼내고, 도구 이전성과 직업 수명의 5년 관점을 짚으며, 마지막엔 "이 책을 덮은 후 첫 24시간"이라는 작은 실천 카드 하나로 마무리. 책을 닫는 동시에 vault를 여는 동선.

이 흐름의 핵심은 **각 장이 단독으로도 가치가 있되, 이어 읽으면 "공감 → 행동 → 의미"의 곡선이 그려진다**는 점이다. 5장만 읽고 ADR을 도입하는 독자도 있고, 6장만 읽고 NotebookLM 워크플로우를 시작하는 독자도 있다. 그래도 책 전체의 약속(노트 → 글 → 책)은 1·7·9장에 분산되어 흐름을 잃지 않는다.

## 책의 독자 약속

이 책을 다 읽고 100일이 지난 독자는 다음을 할 수 있다.

1. **매일 30분으로** fleeting note 3개를 캡처하고, 주간 리뷰 시간에 atomic permanent note 1~2개를 만들어 vault에 영구 보관한다.
2. **자기 손으로** 옵시디언 + GitHub Private 저장소 + 모바일 캡처를 묶은 동기화 환경을 운영하고, 회사 vault와 개인 vault를 분리해 보안 위험 없이 업무에 녹인다.
3. **코드 리뷰·ADR·TIL·회의록**을 같은 vault 안에서 운영하며, 결정과 학습이 백링크로 연결된 "팀의 외장된 마음"을 개인 차원에서 구축한다.
4. **Claude·Claude Code·NotebookLM·Smart Connections**를 의도적 분업 원칙(AI는 캡처·요약·연결 추천, 사람은 atomic claim 결정)으로 vault에 연결해 사고를 잃지 않으면서 가속한다.
5. **블로그 글 1편, 컨퍼런스 발표 1회, 또는 전자책 한 챕터**를 자기 atomic note 묶음에서 자연스럽게 길어 올린다. 그 경험을 바탕으로 "어떤 마감이 있을 때 vault가 살아나는지" 알게 된다.
6. **Collector's Fallacy·Tinkering Trap·파편화 불안·임계점 전 절망·모바일 캡처 마찰**의 5가지 함정을 이름과 처방으로 알아보고, 자기 vault의 건강 상태를 분기별로 진단한다.
7. **5년 후 직무가 바뀌어도** 마크다운 파일로 남은 vault를 다음 도구·다음 직무로 이주시키며, "도구는 바뀌어도 사고는 따라온다"는 감각을 갖는다.

이 책이 약속하지 않는 것도 분명히 한다 — Zettelkasten의 학습 효과를 RCT로 보장하지 않는다. 그러나 인접 인지과학 증거(Testing Effect, Spacing, Writing-to-Learn, Extended Mind)와 100명+ 개발자의 실증 사례를 묶어, **"독자 자기 손으로 100일을 실험해 볼 가치가 있다"**는 정도까지는 책임지고 안내한다.
