## 리뷰 라운드 1

### 총평

전체적으로 **계획의 완성도는 상당히 높다**. 레퍼런스의 핵심 주장(Ahrens 3종 노트, Doto의 atomic claim 정의, Matuschak evergreen, Extended Mind, 인지과학 4효과, 5가지 실패 패턴, AI 분업론)이 모두 챕터에 매핑되어 있고, 한국 개발자 맥락(GitHub Private + Obsidian Git, 송요창 패턴, 디지털투데이 보도, 단행본 문화)도 빠짐없이 흡수되었다. 내러티브 아크(공감→이해→실행→통합→지속→의미) 6악장 구조는 설득력 있고, 7장과 8장의 순서 역전(출력 약속 먼저 → 좌절 통과)에 대한 설명도 타당하다.

다만 **개발자 독자가 "바로 적용할 수 있다"는 감각**을 갖기 위해서는 몇 가지 구조적 약점이 있다. 우선순위는 Critical / Should / Nice-to-have로 라벨링한다.

### Critical (반드시 수정)

#### C1. 노트 ID 시스템 누락 — Zettelkasten의 정체성 결락

레퍼런스 §1.1에서 **Luhmann의 분기 번호(1, 1a, 1a1, 1b, 2…)** 가 Zettelkasten의 핵심 정체성 중 하나로 강조되는데, 챕터 2에서는 "다이어그램"으로 한 번 보여주고 끝난다. 그런데 **개발자 독자에게 가장 먼저 떠오르는 실무 질문은 "그래서 노트 파일명/ID는 어떻게 짓나?"** 다. Obsidian 커뮤니티에서 끝나지 않는 논쟁:
- 타임스탬프 ID (`202605101430-mvcc.md`) — Zettlr 전통
- Luhmann 분기 번호 (`1.2.3a-mvcc.md`) — 원형
- 주제어 슬러그 (`mvcc-postgres-vacuum.md`) — Andy Matuschak / Obsidian 표준
- 하이브리드 (날짜 + 슬러그)

**이 결정 없이는 첫 노트를 못 쓴다.** 챕터 3 "30분 셋업" 또는 챕터 4 "매일 30분 워크플로우"에 **"노트 파일명 정하기 — 3가지 방식과 추천"** 절을 명시적으로 추가해야 한다. Bob Doto는 "제목이 곧 ID"를, Matuschak는 "API 인터페이스로서의 제목"을, 한국 한국어 슬러그(영문 표기) 처리 등 한국 개발자 특수 고려사항도 함께 다뤄야 한다.

#### C2. 링크 vs 태그 vs 폴더 전략 논쟁이 설명되지 않고 결과만 적용됨

레퍼런스 §4.3에 명확한 합의("주제별 폴더는 안티패턴, 링크가 의미·태그는 메타데이터")가 있고, 이는 **Obsidian 커뮤니티에서 가장 격렬한 논쟁** 중 하나다. 그런데 챕터 3에서는 "폴더 3개만"으로 결과만 던지고, 챕터 어디에도 "왜 주제별 폴더가 안티패턴인지", "태그를 어떻게 쓸 것인지"가 별도 섹션으로 다뤄지지 않는다.

Notion·Apple 메모에서 넘어온 개발자는 **본능적으로 "주제별 폴더"를 만들려 한다**. 이 본능을 명시적으로 꺾어주지 않으면 챕터 3의 "폴더 3개만"이 자의적 규칙처럼 느껴진다. 챕터 2 또는 챕터 4에 **"링크가 의미를 담고 태그가 유형을 담는다 — 폴더는 종류만 구분"** 절을 추가해야 한다.

#### C3. 회사 vault 보안/프라이버시가 챕터 5의 한 글머리표로 축소

레퍼런스에서는 "한국 개발자 = GitHub Private 표준"이 강하게 호소되고, 사용자 지정 추가 검토 항목에도 **"노트 보안/프라이버시 (회사 정보 처리)"** 가 명시되어 있다. 그런데 챕터 5에서 이를 마지막 글머리표 한 줄("회사 vault vs 개인 vault — 분리 운영, 보안 체크리스트")로 처리하고 있다.

이 주제는 **챕터 한 절(2~3페이지) 분량으로 격상**해야 한다. 다뤄야 할 것:
- 회사 코드/고객 데이터를 vault에 적어도 되는 경계
- GitHub Private 저장소가 "안전"한가에 대한 현실적 평가 (회사 정책에 따라 외부 Git 금지)
- 회사 vault의 로컬 전용 운영 (동기화 자체를 포기)
- 퇴사 시 vault 처리 — 어디까지 가져가도 되나
- AI에 vault 노출 시 (NotebookLM에 회사 자료 업로드) 주의점

이는 **한국 IT 기업의 보안 정책 현실**(원격 접근 차단, 외부 클라우드 금지 등)과 직결되어 한국 독자에게 특히 중요하다. 챕터 5를 20p → 22~24p로 확장하거나, 챕터 5에서 분리해 챕터 5b로 만드는 것도 고려.

### Should (개선 권장)

#### S1. 모바일 캡처가 챕터 3·4·8에 흩어져 통합적 설명이 없음

레퍼런스 §3.3에서 모바일 워크플로우는 별도 섹션으로 다뤄지고, 5가지 실패 패턴 중 하나가 "모바일 캡처 마찰"이다. 그런데 계획에서는:
- 챕터 3 — 모바일 동기화 우선순위 (한 줄)
- 챕터 4 — fleeting 3가지 통로 중 하나로 모바일 캡처 (한 줄)
- 챕터 8 — 5가지 함정 중 모바일 마찰

세 곳에 분산되어 있어 **"모바일을 어떻게 통합 운영하는가"의 그림이 한 번에 안 잡힌다**. 챕터 4의 "Fleeting note 3가지 통로" 글머리표를 **반페이지짜리 별도 섹션으로 격상**하고, iOS Drafts → Shortcuts → vault Inbox.md append 워크플로우를 코드(Shortcut 스크린샷·Drafts action)로 보여주는 게 좋다. 안드로이드 사용자(Termux + Fleeting Notes)도 동등 비중으로.

#### S2. 챕터 9가 14p로 가장 짧지만 부담이 가장 무거움

챕터 9는 14p에 다음을 다 담는다 — Extended Mind 재론, 도구 이전성, AI 시대 atomic 생존, 직업 수명, vault 공개 여부, 마지막 과제, 부록 안내. **밀도가 너무 높다**. 두 갈래 중 선택:
- (a) 챕터 9 분량을 16~17p로 늘리고 부록 안내는 별도 "부록 인트로"로 빼기
- (b) "vault 공개 여부"와 "AI 시대 atomic 생존"은 챕터 6/7에 흡수하고 챕터 9는 Extended Mind + 직업 수명 + 마지막 과제로 슬림화

기술서 결말이 너무 빽빽하면 "철학적 마무리"가 헐떡임으로 느껴진다. (b)를 추천한다.

#### S3. 플러그인 생태계의 깊이가 얕음 — Dataview·Templater 활용 사례 부족

사용자 지정 검토 항목에 **"Obsidian 플러그인 생태계 (Dataview, Templater, Git 등)"** 가 명시되어 있다. 레퍼런스 §3.2에 코어 vs 플러그인 단계별 도입 가이드가 잘 정리되어 있는데, 계획에서는:
- 챕터 3 — "지금은 깔지 마세요" 목록 (방어 모드)
- 챕터 5 — Templater + MADR (한 글머리표)
- 챕터 9 — 부록 25개 (이름만)

**"Dataview로 ADR 인덱스 자동 생성", "Templater로 fleeting → permanent 변환 자동화"** 같은 실제 활용 사례가 챕터 5 또는 챕터 6에 한 절로 들어가야 한다. 개발자 독자는 "왜 이 플러그인이 좋은지"보다 "어떤 쿼리/스크립트를 짜는지"를 보고 싶어한다. Dataview 쿼리 예시 1~2개(예: "지난 30일에 만든 atomic note 중 백링크 0개인 것")는 강력한 구체성을 준다.

#### S4. 챕터 4와 챕터 5의 ADR 등 워크플로우 경계가 모호

챕터 4 "매일 30분 워크플로우"는 일상 루틴을, 챕터 5 "개발자의 일상에 녹이기"는 업무 통합을 다룬다. 그런데 둘 다 "fleeting → permanent 흐름"을 다루는데 챕터 4에서는 "RAG vs Fine-tuning 영상"을, 챕터 5에서는 "Postgres VACUUM 사건"을 사례로 든다. **읽다 보면 "어 이거 4장에서 본 것 같은데"가 발생할 수 있다.**

명확한 경계 기준 제시 권장:
- 챕터 4 = **개인 학습** 흐름 (논문·영상·책)
- 챕터 5 = **업무 산출물**과 통합 (ADR·코드 리뷰·회의록)

이 경계를 챕터 5 도입부에서 명시("4장이 학습 루틴이었다면 5장은 업무 산출물과의 결합")하면 중복 인지가 사라진다.

#### S5. 챕터 6 AI 통합이 19p에 너무 많은 도구를 담음

챕터 6은 19p에 NotebookLM, Smart Connections, Claude Desktop + MCP, Claude Code + Claudian, hallucination 차단, 한국 채택 흐름까지 다룬다. 도구 5개 × 워크플로우 = 가독성 부담. 추천:
- 도구별로 1.5~2p씩 균등 배분
- 각 도구마다 **"이걸로 뭘 만들었나"의 한 사례** 고정 (도구 소개 → 사례 → 위험 패턴)
- 한국 채택 흐름은 사이드바(0.5p)로 분리

또는 6장을 21~22p로 늘리는 것도 가능. 19p에 5개 도구는 빡빡하다.

### Nice-to-have (선택적 개선)

#### N1. 챕터 1의 "노트 묘지 4가지"가 한국 개발자에게 더 강하게 호소되도록

현재: Slack 저장 메시지, GitHub Star, Apple 메모, Notion DB. **한국 개발자 추가 후보**:
- 카카오톡 "나에게 보내기"에 쌓인 코드 스니펫
- 노션 "데일리 일지" 페이지가 빈 채로 멈춰 있음
- 벨로그 임시저장 글 12개

한국 독자에게는 카톡 "나에게 보내기"가 가장 흔한 노트 묘지일 수 있다. 사례 한두 개 한국화하면 첫 페이지부터 공감이 강해진다.

#### N2. ID 시스템과 함께 "노트 제목 잘 쓰는 법"도

Matuschak의 "제목을 API처럼"은 레퍼런스에 있지만 계획에는 챕터 2의 한 줄로만 등장. 챕터 4의 "permanent note의 작성 순간"에 **"제목 = 한 줄 요약 = 향후 검색 키워드"** 라는 실무 가이드를 한 절로 넣으면 좋다. "MVCC가 뭔지"보다 "Postgres MVCC가 동시성을 어떻게 처리하나" 같은 명제 형식 제목 가이드.

#### N3. 챕터 8 "100일" 프레임을 챕터 1에 한 번 미리 노출

챕터 8에서 100일 KPI가 등장하는데, 챕터 1의 "이 책이 약속하는 것"에 미리 "100일 후 당신은…" 한 줄을 넣으면 독자가 첫 장부터 시간 척도를 잡을 수 있다. 8장에서야 100일이 등장하면 "그럼 그동안 뭐 한 거지" 느낌.

#### N4. 챕터 7의 "외부 공개 vault" 절을 챕터 9의 "공개 여부" 논의와 통합

7장 "Andy Matuschak's notes, MolecularNotes — 영감과 한계"와 9장 "vault를 동료에게·후배에게 공개할 것인가"가 의미상 겹친다. 7장에서 영감 사례를 짧게 보여주고, 9장에서 결정 가이드를 다루도록 분담을 명확히.

### Critical — 모두 반영

- **C1 (노트 ID 시스템):** 챕터 3에 `§3.2 노트 ID 시스템 — 3가지 방식과 개발자 추천` 절을 신설했다. 타임스탬프/Luhmann 분기 번호/주제어 슬러그/하이브리드 4가지를 표로 비교하고, 추천은 **"주제어 슬러그 + Matuschak의 'API 인터페이스로서의 제목' 원칙"** 으로 명시. 한국어 슬러그 처리 가이드(영문 표기 / 한글 그대로 / 케밥 케이스) 포함. 챕터 3 분량 18p → 22p로 +4p.
- **C2 (링크 vs 태그 vs 폴더 전략):** 챕터 3에 두 절(`§3.1 폴더 3개만 — 그리고 왜 "주제별 폴더"가 안티패턴인가`, `§3.3 링크 vs 태그`)을 추가해 Notion·Apple 메모 출신 본능을 명시적으로 꺾는다. "폴더는 종류·태그는 유형·링크는 의미"의 3원칙을 한국 개발자가 자주 빠지는 "태그 트리 지옥" 사례와 함께 다룬다. (리뷰어가 챕터 2를 권했으나 챕터 3 셋업 흐름과 합치는 게 자연스러워 챕터 3에 배치 — 챕터 2는 "이론", 챕터 3은 "실무 결정"으로 역할 분리)

### Should — 모두 수용
