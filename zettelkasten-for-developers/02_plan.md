## 책 제목 후보 3개

### 후보 1. **개발자의 두 번째 뇌 — 옵시디언으로 짓는 제텔카스텐**
- **톤:** 따뜻하고 입문 친화적. "두 번째 뇌"라는 익숙한 메타포로 진입 장벽을 낮춤.
- **포지셔닝:** Tiago Forte(BASB)의 키워드를 차용하면서도 본문에서는 "BASB는 자료, Zettelkasten은 사고"라는 차별점을 명확히 함. 일반 PKM 독자도 흡수할 수 있는 가장 넓은 그물.
- **약점:** "두 번째 뇌" 키워드는 이미 포화. 차별화 메시지가 부제에 의존.

### 후보 2. **연결되는 노트 — 개발자가 옵시디언으로 생각을 키우는 법**
- **톤:** 시적이고 메타포 중심. "연결"이라는 Zettelkasten의 핵심을 제목에 박음.
- **포지셔닝:** 도구(옵시디언)보다 작용(연결·성장)을 앞세워 5년 후에도 유효한 제목. 개발자가 "키운다"는 능동성과 어울림.
- **약점:** 검색 키워드(Zettelkasten, 제텔카스텐) 노출이 약함. SEO에 불리.

### 후보 3. **개발자를 위한 제텔카스텐 — 옵시디언으로 코드와 글을 함께 키우는 법** (추천)
- **톤:** 실용적이고 구체적. 제목에 대상(개발자)·방법(제텔카스텐)·도구(옵시디언)·결과물(코드와 글)이 모두 들어감.
- **포지셔닝:** 한국 시장에 출간된 『세컨드 브레인을 구축하는 제텔카스텐 & 옵시디언』(생산적생산자, 2024)과 명확히 구분. 그 책이 "일반 지식 노동자용"이라면 이 책은 "개발자 전용". 부제의 "코드와 글을 함께"는 ADR/TIL/블로그/책쓰기까지 묶는 약속.
- **약점:** 제목이 길다. 표지 디자인에서 "개발자를 위한 제텔카스텐" + 부제 분리가 필요.

**추천: 후보 3.** 개발자 시장에서는 모호한 메타포보다 "내 직무가 호명되는" 직설적 제목이 강하다. 검색 키워드(제텔카스텐, 옵시디언, 개발자)가 모두 들어가 발견성도 좋다. 부제의 "코드와 글을 함께 키우는 법"이 이 책의 차별점(코드 리뷰·ADR·블로그·발표·책쓰기로 확장)을 한 줄에 압축한다.

## 책 특성

- **장르:** 에세이형 기술서 (실용 기술서 + 개발자 자기계발). 코드 예제보다 워크플로우·템플릿·사고 도구 중심. Toby의 회고적 어조와 결합.
- **예상 분량:** 9개 챕터 × 평균 17.4페이지 = 약 156페이지. 한글 글자 수로는 약 19~22만 자.
- **난이도:** 중급. 옵시디언을 한 번도 써보지 않은 개발자도 따라올 수 있되, 3년차 이상이면 실무 통합 사례에서 더 큰 가치를 느낄 수 있도록 두 층위로 설계.
- **독자 여정:**
  - **시작점:** "노트를 많이 쌓아두지만 다시 안 본다", "TIL이나 블로그를 써보려 했지만 흐지부지", "GPT/Claude를 쓰지만 컨텍스트가 매번 사라진다", "옵시디언은 들어봤지만 어디서 시작할지 모르겠다"
  - **도달점:** "매일 30분으로 atomic note 1~2개를 만들고, 그게 쌓여서 자연스럽게 ADR·블로그·컨퍼런스 발표·책 한 챕터로 흘러나오는" 개인 지식 시스템을 자기 손으로 운영
- **톤앤매너:**
  - **평어체 + 청유형 적극 활용** ("같이 해보자", "한 번 시도해 보자")
  - **수사적 질문**으로 독자의 사고를 끌어들임 ("그런데 왜 우리는 노트를 다시 안 볼까?")
  - **저자의 실패담을 먼저 꺼냄** — Toby도 처음엔 Collector's Fallacy에 빠졌고, 플러그인 튜닝에 1주를 날렸고, atomic note를 너무 잘게 쪼개 길을 잃은 경험을 공유
  - **단정 대신 권유** — "이렇게 해야 한다"가 아니라 "이렇게 해보면 무슨 일이 일어나는지 알게 된다"
  - **개발자 은유 적극 활용** — Permanent note를 "API의 안정적 인터페이스"로, MOC를 "라우팅 레이어"로, fleeting note를 "stdin 버퍼"로 비유
  - **불확실성 명시** — Zettelkasten 효과 RCT가 없다는 사실을 숨기지 않음. "현재 합의 부족"을 인정하면서 독자에게 자기 실험을 권함

  - **§3.1 폴더 3개만 — 그리고 왜 "주제별 폴더"가 안티패턴인가** — `00_Inbox/`, `10_Permanent/`, `20_Reference/` (`30_Daily/`, `40_Projects/`는 나중에). Notion·Apple 메모에서 넘어온 본능을 명시적으로 꺾는 한 절. "폴더는 종류를 구분하고, 의미는 링크가, 유형은 태그가 담는다"는 3원칙 (C2 반영)

### 챕터 4. 매일 30분 워크플로우 — Fleeting부터 Permanent까지 (개인 학습 흐름)

- **핵심 질문:** 하루 중 언제, 무엇을, 어떻게 쪼개서 노트로 만드는가? (논문·영상·책 등 **개인 학습** 자료를 다루는 루틴)
- **주요 내용:**
  - **장 도입부에 4·5장 경계 명시** — "이 장은 개인 학습(논문·영상·책) 흐름을, 다음 장(5장)은 업무 산출물(ADR·코드 리뷰·회의록) 통합을 다룬다." (S4 반영)
  - 하루 흐름 시뮬레이션 — 출근길 fleeting → 점심 후 literature → 저녁 30분 permanent
  - Fleeting note 캡처 — 모바일은 §3 인서트 박스 참조, 데스크톱에서는 Quick Switcher와 Daily Note
  - "한 시간 읽었으면 두 시간 가공" 휴리스틱
  - Literature note 쓰는 법 — 한 자료 = 한 노트, 자기 언어 재구성, 출처 메타데이터
  - **Permanent note의 작성 순간 — 제목 = 한 줄 요약 = 향후 검색 키워드** — Matuschak의 "제목을 API처럼" 원칙을 실무 가이드로. "MVCC가 뭔지"보다 "Postgres MVCC가 동시성을 어떻게 처리하나" 같은 명제 형식 제목 (N2 반영)
  - "일주일 후의 나도 이해하는가" 체크
  - atomic의 적정 입자 — "한 주장(claim)" 정의로 파편화 불안 해결
  - 주간 리뷰 30분 — Inbox 비우기, 끊긴 연결 잇기, 한 주의 baby-MOC 만들기
- **예상 분량:** 19페이지
- **포함할 사례/예시:** Toby의 실제 하루 노트 흐름 스크린샷 시리즈, fleeting → literature → permanent 변환 사례 1건 (예: "RAG vs Fine-tuning" 관련 영상에서 atomic note 3장이 만들어지는 과정 — **개인 학습 자료**), 주간 리뷰 체크리스트 템플릿, 명제 형식 제목 vs 단어 형식 제목 비교 표

### 챕터 5. 개발자의 일상에 녹이기 — 코드 리뷰·ADR·TIL·회고 (업무 산출물 통합)

- **핵심 질문:** 업무 시간에 짬내서 노트할 수 있을까, 그리고 그게 회사 일에 도움이 될까? 회사 보안 정책과 어떻게 공존하는가?
- **주요 내용:**
  - **장 도입부에 4·5장 경계 명시** — "4장이 학습 루틴이었다면 5장은 업무 산출물과의 결합이다." (S4 반영)
  - **§5.1 코드 리뷰 노트:** PR을 읽으며 떠오른 패턴을 즉시 atomic note로. 같은 패턴이 N번 반복되면 baby-MOC로 승격
  - **§5.2 ADR(Architecture Decision Record):** Matteo Paoli의 MADR + Templater 자동화. 결정 카드는 백링크로 영향받는 시스템 노트와 연결되어 "결정의 진화" 추적 가능
  - **§5.3 Dataview·Templater 실전 쿼리 (3페이지로 강화)** — (S3 반영)
    - **Templater 스니펫 1:** fleeting → permanent 변환 자동화 (frontmatter 자동 생성, 백링크 placeholder, 작성 일자)
    - **Templater 스니펫 2:** ADR 새 카드 — MADR 형식 자동 채움 + 결정 ID 자동 부여
    - **Dataview 쿼리 1:** "지난 30일에 만든 atomic note 중 백링크 0개인 것" — 고립 노트 발견
    - **Dataview 쿼리 2:** ADR 인덱스 자동 생성 — `status: accepted | superseded` 필드별 그룹핑
    - **Dataview 쿼리 3:** "이번 주 학습한 TIL 목록을 블로그 초안 후보로 모으기"
    - 각 쿼리는 실제 Markdown 코드 블록으로 보여주고, 결과 화면 캡처 함께
  - **§5.4 TIL과 Permanent 폴더 분리** — TIL은 학습 기록, Permanent는 자기 주장. 한국 개발자 문화에 잘 맞는 두 층위
  - **§5.5 회의록·일감 관리:** 송요창 패턴 — 이모지 메타데이터(📅 due, 🏃 in-progress) + 백링크
  - **§5.6 온콜·장애 대응 회고:** 5 Whys 루트 분석을 atomic note로 분해해 다음 장애에서 재사용
  - **§5.7 오픈소스 기여 노트:** 라이브러리 internals를 읽으며 만든 atomic note는 다음 PR·이슈 토론에서 인용
  - **§5.8 회사 vault vs 개인 vault — 보안·프라이버시 절 (3페이지로 격상)** — (C3 반영)
    - 회사 코드/고객 데이터를 vault에 적어도 되는 경계 — "재현 가능한 일반 패턴은 OK, 식별 가능한 데이터·키는 NO"
    - **GitHub Private 저장소가 "안전"한가에 대한 현실적 평가** — 회사 정책에 따라 외부 Git 금지인 경우(금융·공공·일부 대기업)

    - **퇴사 시 vault 처리** — 어디까지 가져가도 되나, 사내 자료 분리 셋업 (별도 폴더 + 퇴사 시 자동 제외 스크립트)

## 챕터 간 흐름 (내러티브 아크)

이 책은 **공감 → 이해 → 실행 → 통합 → 지속 → 의미**의 6악장 구조를 따른다.

- **공감 악장(1장):** 독자의 현재 고통(노트 묘지)을 호명한다. 추상적 PKM 이론으로 시작하지 않고 "당신의 Notion DB와 카톡 '나에게 보내기'가 죽어 있는 이유"부터 짚는다. 이 한 장으로 "이 책은 나를 위한 책"이라는 신호를 보내고, **100일 후 도달점을 미리 한 단락으로 노출**해 시간 척도를 깐다.

- **이해 악장(2장):** 너무 길어지기 전에 Zettelkasten의 핵심을 한 호흡에 끝낸다. 학술적 깊이는 인지과학 4효과 + Extended Mind까지만, 그 이상은 의도적으로 자제. "왜 작동하는가"의 질문에 답하되 학자처럼 다루지 않는다.

- **실행 악장(3·4장):** 이론에서 손으로 빠르게 내려간다. 3장은 셋업(공간을 마련) — **노트 ID·링크/태그/폴더·모바일 캡처라는 "첫 노트를 쓰기 직전에 반드시 결정해야 하는 3대 질문"을 한 챕터에서 모두 해결**해 독자가 셋업 단계에서 길을 잃지 않도록 한다. 4장은 일상 루틴(시간을 마련)으로, **개인 학습 자료**(논문·영상·책) 다루는 흐름. 두 장이 끝나면 독자는 이미 "이번 주말부터 할 수 있다"는 감각을 가져야 한다.

- **통합 악장(5·6장):** 개발자 일상에 녹이는 단계. 5장은 인간 협업(코드 리뷰·ADR·TIL·회의록) — **업무 산출물 통합** 흐름이며 4장과의 경계가 명시된다. **Dataview·Templater 실전 쿼리를 한 절로 강화**해 "왜 옵시디언이 개발자에게 좋은가"의 구체성을 확보하고, **회사 vault 보안을 한 절(3p)로 격상**해 한국 IT 보안 정책 현실에 답한다. 6장은 AI 협업(Claude·NotebookLM·MCP)으로 **도구 4개를 균등 배분**해 가독성을 확보. 한국 개발자의 2026년 현실(Claude Code·MCP 빠른 채택)은 사이드바로 처리한다.

- **지속 악장(7·8장):** 산출이 보이기 시작하는 단계와 좌절 구간 통과 단계. 순서가 직관과 반대인 이유는 — 7장에서 "이렇게 글·발표·책이 나온다"는 약속을 먼저 보여줘야, 8장의 "100일 좌절 구간을 넘는 법"이 추상적 인내가 아닌 구체적 보상을 향한 인내가 된다.

- **의미 악장(9장):** 기술서의 클리셰인 "당신만의 시스템을 만드세요"로 끝내지 않는다. **4개 절(Extended Mind 재론 / 도구 이전성·직업 수명 / vault 공개 결정 가이드 / 24시간 마지막 과제)로 슬림화**하여 결말의 호흡을 가볍게 가져간다. 책을 닫는 동시에 vault를 여는 동선.

이 흐름의 핵심은 **각 장이 단독으로도 가치가 있되, 이어 읽으면 "공감 → 행동 → 의미"의 곡선이 그려진다**는 점이다. 5장만 읽고 ADR을 도입하는 독자도 있고, 6장만 읽고 NotebookLM 워크플로우를 시작하는 독자도 있다. 그래도 책 전체의 약속(노트 → 글 → 책)은 1·7·9장에 분산되어 흐름을 잃지 않는다.

## 책의 독자 약속

이 책을 다 읽고 100일이 지난 독자는 다음을 할 수 있다.

1. **매일 30분으로** fleeting note 3개를 캡처하고, 주간 리뷰 시간에 atomic permanent note 1~2개를 만들어 vault에 영구 보관한다.
2. **자기 손으로** 옵시디언 + GitHub Private 저장소 + 모바일 캡처(iOS Drafts/Android Termux)를 묶은 동기화 환경을 운영하고, 회사 vault와 개인 vault를 분리해 보안 위험 없이 업무에 녹인다. 한국 IT 보안 정책 13항목 체크리스트로 자기 회사 환경을 진단한다.
3. **노트 ID·링크/태그/폴더 전략**을 자기 언어로 설명할 수 있고, 자기 vault에 일관된 명명 규칙을 적용해 1년 후의 자신도 그 노트를 찾을 수 있게 한다.
4. **코드 리뷰·ADR·TIL·회의록**을 같은 vault 안에서 운영하며, Dataview 쿼리로 ADR 인덱스·고립 노트·블로그 후보를 자동 생성하고, Templater로 fleeting → permanent 변환을 자동화한다.
5. **Claude·Claude Code·NotebookLM·Smart Connections**를 의도적 분업 원칙(AI는 캡처·요약·연결 추천, 사람은 atomic claim 결정)으로 vault에 연결해 사고를 잃지 않으면서 가속한다.
6. **블로그 글 1편, 컨퍼런스 발표 1회, 또는 전자책 한 챕터**를 자기 atomic note 묶음에서 자연스럽게 길어 올린다. 그 경험을 바탕으로 "어떤 마감이 있을 때 vault가 살아나는지" 알게 된다.
7. **Collector's Fallacy·Tinkering Trap·파편화 불안·임계점 전 절망·모바일 캡처 마찰**의 5가지 함정을 이름과 처방으로 알아보고, 자기 vault의 건강 상태를 분기별로 진단한다.
8. **5년 후 직무가 바뀌어도** 마크다운 파일로 남은 vault를 다음 도구·다음 직무로 이주시키며, "도구는 바뀌어도 사고는 따라온다"는 감각을 갖는다.

이 책이 약속하지 않는 것도 분명히 한다 — Zettelkasten의 학습 효과를 RCT로 보장하지 않는다. 그러나 인접 인지과학 증거(Testing Effect, Spacing, Writing-to-Learn, Extended Mind)와 100명+ 개발자의 실증 사례를 묶어, **"독자 자기 손으로 100일을 실험해 볼 가치가 있다"**는 정도까지는 책임지고 안내한다.
