## 한 줄 총평

레퍼런스의 95%를 차지하는 1차/2차 사료를 빠짐없이 11챕터에 분배해낸 야무진 구조이며 사건→해부→사회→비판→응용→메타→시야의 리듬도 명료하지만, **3장의 워크스루 리스크·7장의 한국 챕터 정체성·10장 응용 루프의 과수·11장의 자기과시 함정**이라는 네 개의 약한 관절을 손보지 않으면 시니어 독자가 중간에 책을 닫을 위험이 크다.

## 강점

- **레퍼런스 커버리지 우수.** §1~§8의 1차/2차 출처가 거의 누락 없이 매핑되어 있다. 특히 §6의 비판 8종을 8장 한 곳에 모은 것은 책의 신뢰도를 만든다.
- **내러티브 ring 구조.** 1장 README 픽션 인용 → 에필로그 "10,205세대"로 돌아오는 ring은 독자에게 정서적 닫힘을 준다. epigraph 3개의 배치 설계도 깔끔하다.
- **응용 챕터의 비중.** 책의 28%를 Part 4에 할애한 비율 자체는 적정하다 — "패턴을 옮겨가라"는 책의 주 메시지를 분량으로 뒷받침한다.
- **비판 챕터(8장)의 분량 확보.** 7,000자로 최장 챕터를 비판에 배정한 것은 hype-book의 함정을 피하는 좋은 결정이다.
- **편집 가능 아티팩트 + frozen metric + ratchet의 3축**을 2·4·5장으로 명확히 분리해 일반화 공식이 9장에서 자연스럽게 도출되도록 설계했다.

## 우려 사항 (Critical / Should / Nice 라벨링)

### Critical 1. 3장은 기술 워크스루로 미끄러질 운명이다
- **현재 계획:** Muon (Newton-Schulz 5스텝 수렴, 직교화) + ResFormer (value residual, 16% 파라미터 절감) + Sliding Window SSSL + layer-wise scalar + 한국 시각 메모를 7,000자에 욱여넣음.
- **문제:** 책의 약속(§29줄, "수식·구현 디테일로 빠지지 않는다")과 정면 충돌한다. 시니어 엔지니어라도 Muon 옵티마이저 본질을 처음 듣는 독자가 다수다. 한 챕터 안에서 옵티마이저·attention 패턴·residual·layer scalar 네 가지 ML 디테일을 동시에 다루면 책 톤이 nanoGPT 튜토리얼로 미끄러진다 — 1장에서 "코드 워크스루로 빠지지 않겠다"고 약속한 직후라 깨지면 신뢰가 무너진다.
- **근거:** 레퍼런스 §2.1~§2.6은 자료 자체로 매우 짙다. 박제창의 "human-above-the-loop" 프레임(§4.6)은 본질이 "정교한 추론이 아니라 문제 재구조화"라고 명시한다 — 책의 메시지는 "제약 → 선택"이지 "옵티마이저 자체"가 아니다.
- **제안:**
  1. 3장 분량을 7,000 → **4,500자**로 축소.
  2. Muon은 "Newton-Schulz가 bfloat16에 안정적이고 5스텝 수렴 → 5분 예산이라는 제약이 강제한 선택" **한 단락**으로 압축. 수식·계수·NanoGPT 스피드런 35% 일화는 각주로.
  3. ResFormer / SSSL / resid_lambdas는 **표 한 개**에 "제약 → 결정 → 부수효과 → 비판 가능 지점" 4열로 정리. 본문은 각 항목 2~3문장.
  4. 절약한 2,500자를 두 곳에 분배: **(a) 2장(+1,000자)** "제약 → 디자인 결정" 어휘를 더 깊게 다지기, **(b) 9장(+1,500자)** 일반화 공식 챕터에서 "내 도메인의 제약은 무엇인가" 사고 실험을 늘리기. 이렇게 하면 3장의 본질이 "기술 해설"이 아니라 "제약 읽기 연습"이 된다.

### Critical 2. 7장의 한국 챕터는 독립으로 두면 비어 보인다
- **현재 계획:** 5,500자 분량으로 DLI Lab, SOTAAZ, Dreamwalker, GPTers, Jangwook + "한국 커뮤니티의 직접 댓글 후기는 빈약하다는 정직한 보고" + 한국 실무자가 시작한다면.
- **문제:** 레퍼런스 §9 자체가 "한국어 커뮤니티의 직접 댓글은 검색 노출이 적어 일부만 확보"라고 인정한다. 1차 자료가 4~5개뿐이고 그중 박재홍 블로그는 fetch 실패다. **5,500자를 채우려면 자료를 늘여 쓰거나 같은 논점을 두 번씩 말해야 한다.** 더 큰 문제: 7장이 독립으로 서면 "한국 챕터는 사실 채울 거리가 없었음"이라는 메타 메시지가 책 자체에서 드러난다 — 한국 독자에게 오히려 모욕적이다.
- **근거:** 레퍼런스 §4.6 (4명), §8.6 (6개 링크, 1개 fetch 실패), §9-3 (정직한 한계 자백). 영어권 1차 후기(§5.2 Abhishek Nair, Karpathy, Varun Mathur)와 비교하면 풍성함의 격차가 너무 크다.
- **제안:** 7장을 독립 챕터에서 **빼고 분산**한다.
  - DLI Lab의 "metric optimization paradox"는 **4장**(BPB·frozen metric)에 흡수 → 봉인된 메트릭의 통계적 오염 논의에 한국 학계 시각을 더한다.
  - Dreamwalker의 "human-above-the-loop"은 **2장**(디자인 슬로건 해부)에 흡수 → 이미 계획에 명시되어 있으므로 7장 중복을 제거.
  - SOTAAZ Part 1·3 (해부 + 일반화)은 **9장**(일반화 공식)의 사례 지도에 흡수.
  - GPTers/Jangwook 사용법은 **10장**(응용 루프)의 사이드바 박스 한 개로.
  - "한국 실무자가 시작한다면" 사고 실험은 **9장** 후반에 "한국 독자를 위한 사고 실험" 단락으로 이미 계획되어 있음 — 7장 중복.
- **그 자리에 둘 챕터 (신설 7장 후보):** **"산업 의제로의 승격 — Anthropic·ICLR RSI 워크숍·Sakana AI Scientist와 나란히 놓고 보기"** (5,500~6,000자)
  - 5월 19일 Anthropic 합류 사건의 깊이 (§4.2)
  - ICLR 2026 RSI Workshop (§4.3) — corrigibility·co-improvement 제안
  - Sakana AI Scientist (§8.2 arxiv 2502.14297) / Agent Laboratory / AgentRxiv — autoresearch가 진공 속에서 나온 게 아니라 같은 의제의 다른 시도들과 함께 있다는 풍경
  - Jack Clark "2028년 60% RSI" 베팅, Karpathy의 "tipping point December 2025" 발언
  - 이 챕터는 책의 산업적 권위를 만든다 — 시니어 독자가 "이건 단순 화제가 아니라 시대의 의제"라고 인식하게 만드는 다리 역할.

### Critical 3. 10장의 6개 응용 루프는 셋으로 줄여야 한다
- **현재 계획:** 7,500자에 핫경로 성능·flaky test·테스트 커버리지·RAG·시스템 프롬프트·성능 회귀 자동 격리 6개 루프, 각각 6항목(대상/파일/메트릭/게이트/실패/시작) × 6루프 = 36개 박스.
- **문제:** 7,500 / 6 = 1,250자/루프. 항목당 200자가 못 된다. **응용 루프는 "오, 이거 내일 시작할 수 있겠다"를 느끼게 해야 의미가 있는데, 200자짜리 6개는 카탈로그 페이지가 된다.** 책 후반의 가장 중요한 챕터가 평면화된다.
- **근거:** 레퍼런스 §7에는 7.1~7.10까지 10개 후보가 있다. 모두 가치 있으나 책에서 다 깊이 다룰 수는 없다. 가장 검증된 사례(Tobi Liquid §4.4, az9713 §5.4, idealo Search Ranking §5.4)에 무게를 두는 편이 권위 있다.
- **제안:**
  1. 6개 → **3개 루프**로 축소. 각 루프 ~2,300자, 총 7,000자.
  2. **선택 기준:** "1차 사례가 검증된 것" + "Claude Code 환경에서 즉시 시작 가능한 것" + "서로 다른 도메인 (코드·인프라·에이전트 자체)".
  3. **추천 3개:**
     - **루프 1 — 핫경로 함수 성능 최적화** (Tobi Liquid 사례를 깊게 워크스루, Maintainability Index 보조 게이트 추가, Josh Moody 비판의 교훈)
     - **루프 2 — RAG 파이프라인 튜닝** (idealo Search Ranking 37% 사례, held-out leakage 검증 의식)
     - **루프 3 — 시스템 프롬프트 진화** (az9713 74.72%→100% 사례, adversarial test set 디자인)
  4. 나머지 3개(flaky test, 테스트 커버리지, 성능 회귀)는 챕터 말미에 **"추가 후보 3가지"** 박스로 각각 200자 압축 — "이런 것도 가능하다"의 카탈로그로 격하.
  5. 이렇게 하면 챕터의 톤이 "카탈로그"에서 "워크스루 + 카탈로그"로 바뀌어, 독자가 한 루프를 따라가는 체험을 얻는다.

### Should 1. 11장의 메타 응용은 자기과시 트랩을 안고 있다
- **현재 계획:** EvoSkill/Skill Forge v2/AutoSkill + Sibyl Research System + 책 저술 하네스 자기 적용 + corrigibility + Karpathy Anthropic 미션을 6,500자에 한꺼번에.
- **문제:** "이 책이 책 저술 하네스로 만들어졌다"는 메시지는 강력하지만, 한 챕터 안에서 자기참조가 너무 길면 독자가 "결국 자기 하네스 자랑이었네"로 받아들일 위험이 있다. 또 corrigibility·RSI 안전성은 Critical 2의 신설 7장에서 다뤘다면 11장에서 중복된다.
- **근거:** 계획 §187-188에 corrigibility + ICLR RSI 워크숍이 11장 말미로 가 있는데, 만약 7장이 신설되면 이 부분이 중복된다.
- **제안:**
  1. 11장은 **메타 응용 사례 (외부) + 책 저술 하네스 (자기) 두 갈래**로 한정. 6,500 → **5,500자**.
  2. corrigibility / RSI 안전성 / Karpathy Anthropic 미션 = 신설 7장으로 이동.
  3. 책 저술 하네스의 자기 적용은 "디자인 스케치"가 아니라 **"이 책 챕터 N개를 베이스라인으로 잡고 style-guardian rubric을 frozen metric으로 봉인하면 어떻게 되는가"**의 구체 사고 실험으로 한 페이지. 자랑이 아니라 메타 패턴의 실재성 증거가 되도록.
  4. 절약한 1,000자는 신설 7장(Anthropic + RSI 워크숍 + Sakana)로 이동.

### Should 2. 8장의 9개 소주제는 분류가 필요하다
- **현재 계획:** Shopify Liquid 양면 / 2026 MSR conf 56.1% / Gomoku reward hacking / 짧은 호라이즌 오버피팅 / 평가의 통계적 오염 / prompt injection / 도메인 한계 / "innovative idea 못 찾는다" 9개를 7,000자에 평면 나열.
- **문제:** 9개의 "부서지는 지점"이 평면 리스트로 늘어서면 독자가 어느 게 가장 무서운지 가늠이 안 된다. 비판 자체가 메타-카탈로그가 된다.
- **근거:** 레퍼런스 §6.1~§6.10. 사실 §6은 자연스럽게 (a) 알고리즘 한계, (b) 메트릭 한계, (c) 보안/거버넌스, (d) 도메인 한계의 4그룹으로 나뉜다.
- **제안:** 8장을 4섹션으로 묶는다.
  - **§8.1 알고리즘이 못 보는 것 —** 단방향 ratchet, 짧은 호라이즌 오버피팅, innovative idea 빈곤
  - **§8.2 메트릭이 자기 자신을 망가뜨리는 법 —** Shopify Liquid 53% 양면, Gomoku reward hacking, 통계적 오염
  - **§8.3 코드 품질 회귀의 정량적 증거 —** 2026 MSR conf 56.1% (이게 중요한 외부 검증인데 현 계획에서는 한 줄로 끝남)
  - **§8.4 보안과 도메인 한계 —** prompt injection (Issue #64), wet lab 사각지대, 에이전트 신뢰성 (Codex vs Claude)
- 이렇게 4섹션으로 묶으면 독자가 "어디에 가장 조심해야 하는지"의 지도를 얻는다. 분량은 그대로 7,000자.

### Should 3. 1장이 시니어 독자에게 너무 가볍게 시작한다
- **현재 계획:** Karpathy 픽션 인용 + autoresearch 30초 요약 + 3만 stars + 5월 19일 Anthropic 합류 + Sequoia 발언 + 책의 약속 (5,500자).
- **문제:** 시니어 엔지니어가 "이건 그 화제의 그것"을 이미 안다. 1장이 "사건 소개"만 하면 첫 5분에 책을 닫을 위험이 있다 — "내가 모르는 게 뭐냐"가 1장 안에 한 번은 등장해야 한다.
- **근거:** 레퍼런스 §1.4 (nanochat 계보 char-rnn→minGPT→nanoGPT→nanochat→autoresearch)와 §4.1 (Software 3.0 프레임)은 시니어 독자도 처음 듣는 경우가 많다.
- **제안:** 1장 후반에 두 개의 "낯섦 hook"을 더한다.
  - **(a) nanochat 계보 한 단락** — autoresearch는 Karpathy의 미니멀 교육 코드 13년 계보의 마지막 마디이며, 이번엔 "코드를 가르치는 것"이 아니라 "에이전트에게 코드를 맡기는 것"이라는 질적 전환.
  - **(b) Software 3.0 → 3.5 한 단락** — 3.0에서 context window = program이라면, autoresearch는 program.md = program이고 train.py는 결과물이라는 한 칸 더 위의 추상화. 시니어 독자가 "오, 이건 단순 트릭이 아니라 추상화 층의 이동이구나"를 느낀다.
- 분량은 5,500 → 6,000자.

### Should 4. 2장과 5장의 경계가 모호해질 위험
- **현재 계획:** 2장(디자인 슬로건 해부)에서 "단일 편집 가능 파일·단일 메트릭·human-above-the-loop"을 다루고, 5장(keep/discard)에서 "LOOP FOREVER·git ratchet·simplicity criterion·도구 의존성"을 다룬다.
- **문제:** 2장에서 "권한 분할 표"를 그리면 5장의 git 운영 부분과 겹친다. 독자가 "이거 아까 봤는데?"를 느낀다.
- **제안:**
  - 2장의 "3-파일 구조와 권한 분할 표"는 **표만** 두고, git 워크플로 디테일(LOOP FOREVER 9단계, results.tsv 5컬럼, autoresearch/<tag> 브랜치)은 **5장 전속**으로.
  - 2장은 "**무엇이 봉인되고 무엇이 열려 있는가**"의 디자인 추상으로 한정.
  - 5장은 "**그래서 이게 git 위에서 어떻게 굴러가는가**"의 운영 구체로 한정.
- 분량 변경 없음.

### Should 5. 6장의 한국 시각 단락은 7장 분산 시 어떻게 되는가
- **현재 계획:** 6장 (포크의 풍경) 한국 시각 단락 — "macOS·MLX 포크가 한국 ML 커뮤니티에서 받은 관심, 국내 GPU 인프라 격차".
- **문제:** Critical 2에서 7장 한국 챕터를 분산하기로 했다면, 6장의 한국 시각 단락은 유지해야 한다 (분산의 의미). 다만 레퍼런스 §5.1~§5.2에 "한국 ML 커뮤니티가 macOS 포크에 보인 관심"이라는 직접 근거가 약하다.
- **제안:** 6장의 한국 시각 단락을 "한국 ML 인프라 환경 (스타트업 단일 GPU 환경, KISTI 등 공공 클러스터)에서 어떤 포크가 가장 현실적인가"라는 **실용적 추천 한 문단**으로 바꾼다. 검증 가능한 사실(M2 후기·Windows-RTX·MLX 20% 우위)만 인용하고 한국 커뮤니티의 반응이라는 검증 어려운 주장은 피한다.

### Nice 1. 9장 일반화 공식의 4기준이 4장과 중복
- 4장에서 "binary, locked, compact, gaming-resistant" 4기준을 다루고, 9장에서 "binary, locked, fast, gaming-resistant" 4기준을 다시 다룬다. (4장: compact, 9장: fast — 거의 같다)
- **제안:** 4장에서는 **메트릭 디자인의 디테일**(왜 BPB가 perplexity보다 나은가, frozen이 reward hacking을 어떻게 막는가)에 집중하고, 9장에서는 4기준을 한 줄로만 회상한 뒤 **체크리스트**(자기 도메인 적용 판정용)에 집중. 9장이 4장의 요약이 되지 않도록.

### Nice 2. epigraph #2 ("indefinitely and without any of your own involvement")는 책 제목과 약간 어긋난다
- 책 제목 부제가 "**사람이 자는 동안 진화하는 코드**"이고 epigraph #2는 "**without any of your own involvement**"이다. 두 표현은 가깝지만 책의 마지막 메시지("측정의 토대는 늘 사람의 손에서 시작")와 epigraph #2 사이에 미묘한 충돌이 있다 — epigraph만 보면 "사람을 빼겠다"는 책처럼 읽힌다.
- **제안:** epigraph #2를 그대로 두되, 그 아래 책 전체 epigraph로 **카운터 인용**을 더한다.
  > "1만번째 세대의 코드를 향한 첫 번째 commit은 늘 사람의 손에서 시작한다." — 본서 에필로그
- 두 인용을 나란히 박으면 책의 긴장(자율 vs 사람의 자리)이 표지 직후 페이지에 노출된다.

### Nice 3. Karpathy 트윗 본문 미확보의 정직한 표기
- 레퍼런스 §9-1에서 Karpathy의 두 원본 트윗 본문이 X 인증 제한으로 미확보임을 자백한다.
- **제안:** 1장과 에필로그에서 Karpathy 인용은 "README에 인용된 표현" 또는 "Sequoia 발언 (2차 인용)" 식으로 출처 정직하게 표기. 책의 권위는 정직함에서 온다.

## book-planner의 두 질문에 대한 명확한 답

### Q1. 3장(Muon·ResFormer·Sliding Window)이 기술 워크스루로 미끄러질 위험 — 분량을 줄이고 다른 챕터로 분배하는 게 나은가?

**그렇다. 7,000 → 4,500자로 줄여라.** 이유는 세 가지다.

1. **책의 약속 위반 위험.** 계획 §29줄에서 "수식·구현 디테일로 빠지지 않는다"고 약속했다. 7,000자에 옵티마이저·attention 패턴·residual·layer scalar 네 개 ML 디테일을 동시 진열하면 약속이 깨진다.
2. **본질이 다른 곳에 있음.** 박제창의 "human-above-the-loop"(레퍼런스 §4.6)이 정리하듯, autoresearch의 본질은 정교한 추론이 아니라 **문제 재구조화**다. 3장이 다뤄야 하는 것은 Muon 자체가 아니라 "5분 예산이 어떤 옵티마이저를 강제하는가"의 매핑이다.
3. **9장의 일반화 공식이 더 강해진다.** 3장에서 절약한 2,500자 중 1,500자를 9장의 "내 도메인의 제약은 무엇인가" 사고 실험으로 옮기면, 책 후반의 응용 챕터가 두 배로 단단해진다. 3장은 "기술 카탈로그"에서 "제약 읽기 연습"으로 격상된다.

**구체 실행:** Muon은 표 한 칸으로 압축 (제약: 5분 빠른 수렴 / 결정: Newton-Schulz 5스텝 / 부수효과: bfloat16 안정성 / 비판: ?). ResFormer·SSSL·resid_lambdas도 같은 표의 다른 행으로. 본문은 표를 해설하는 정도로만.

### Q2. 7장(한국 시각)을 독립 챕터로 두는 게 나은가, 분산이 나은가?

**분산이 낫다. 그 자리에는 산업 의제 풍경 챕터를 새로 둬라.** 이유는 세 가지다.

1. **자료 부족이 메타 메시지로 노출됨.** 레퍼런스 §9-3가 "한국어 커뮤니티의 직접 댓글은 검색 노출이 적어 일부만 확보"라고 인정하고, 박재홍 블로그는 fetch 실패다. 5,500자를 채우려면 자료를 늘여 쓰거나 같은 논점을 반복해야 한다 — 이 자체가 "한국 챕터는 사실 채울 거리가 없었음"이라는 메타 메시지를 만들어 한국 독자에게 오히려 모욕적이다.
2. **분산이 한국 시각의 권위를 더 키운다.** DLI Lab의 metric paradox를 4장(BPB·frozen metric)에 흡수하면 그 통찰이 책의 핵심 논의에 직접 박힌다. Dreamwalker의 "human-above-the-loop"을 2장에 흡수하면 디자인 슬로건 해부의 어휘가 된다. 독립 챕터에서 한 번 다루는 것보다 본문 곳곳에서 한국 학계의 이름이 인용되는 편이 더 권위 있다.
3. **신설 7장이 책의 산업적 권위를 만든다.** Anthropic 합류(§4.2) + ICLR 2026 RSI 워크숍(§4.3) + Sakana AI Scientist / Agent Laboratory / AgentRxiv(§8.2)를 함께 다루면 "autoresearch는 진공에서 나온 게 아니라 같은 의제의 여러 시도 중 가장 미니멀한 것"이라는 큰 그림이 그려진다. 시니어 독자가 "이건 시대의 의제구나"를 느끼는 결정적 다리다.

**구체 실행:** 분산은 위 Critical 2의 매핑 표대로. 신설 7장 제목 후보: **"같은 의제의 다른 풍경 — Anthropic·ICLR RSI 워크숍·AI Scientist 옆에서 autoresearch 읽기"**.

## 모든 제안이 반영된 책의 모습 (한 단락)

총 12개 본문 단위(11장 + 에필로그)·약 73,000자. 1장은 "사건 소개"가 아니라 Software 3.0 한 칸 위로 가는 추상화의 이동을 시니어 독자에게 보여주며 시작한다. 2~5장은 디자인 슬로건·옵티마이저(축약)·frozen metric·git ratchet 4개의 핵심 디자인 결정을 차례로 해부한다. 3장은 워크스루가 아닌 "제약 → 결정" 매핑 표가 중심이고, 한국 학계의 metric paradox는 4장에 자연스럽게 흡수된다. 6장 포크의 풍경에 이어 신설 7장이 책의 시야를 한 단계 확장한다 — autoresearch는 같은 의제의 한 진영이며, Anthropic·ICLR RSI·Sakana가 그 옆에 있다. 8장 비판은 4섹션으로 정돈되어 독자가 "어디에 가장 조심해야 하는지"의 지도를 얻는다. 9장 일반화 공식이 책의 무게 중심이 되어 8,000자에 한국 도메인 사고 실험까지 더해진다. 10장은 3개 깊은 워크스루(Tobi Liquid·idealo RAG·az9713 프롬프트) + 3개 카탈로그 박스로 "내일 시작할 수 있다"를 정말 느끼게 한다. 11장은 메타 응용과 책 자기 적용 두 갈래로 한정되어 자기과시 함정을 피하고, corrigibility 논의는 7장으로 이미 옮겨가 있다. 에필로그는 ring 구조의 닫힘으로 "측정의 토대는 늘 사람의 손에서 시작"을 남긴다. 한국 학계의 이름들(DLI Lab·SOTAAZ·박제창)은 챕터 곳곳에 분산되어, 한 챕터에서 한 번 다뤄지는 것보다 더 권위 있게 책 전체에 새겨진다.
