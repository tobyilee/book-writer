# 10장. Claude Code 응용 루프 — 3개 깊은 워크스루 + 3개 카탈로그 박스

9장에서 공식을 외웠으니 이제 손에 도구가 있다. 하지만 손에 든 도구는 어딘가에 박아 봐야 진짜 도구가 된다. 자, 그러면 이번 주에 시작할 수 있는 자율 루프 하나를 그려 보자. Claude Code 환경에서, 자기 책상 위에 있는 코드베이스 하나를 골라.

이 챕터는 진열을 피하려 한다. 6개 도메인을 평면으로 늘어놓으면 카탈로그가 되고, 카탈로그는 읽기는 좋지만 시작하기는 어렵다. 그래서 *3개의 깊은 워크스루*와 *3개의 짧은 카탈로그 박스*로 비대칭 구조를 짠다. 깊은 워크스루는 한 도메인을 8단계로 끝까지 따라가본다 — 레퍼런스 케이스, 편집 가능 아티팩트, 메트릭, frozen 부분, Claude Code 도구 묶음, program.md 스케치 5줄, 실패 모드, 한국 적용 사고 실험까지. 카탈로그 박스는 가능성의 지도 역할만 한다. "이 자리에 이것도 가능하다" 정도의 신호.

세 깊은 워크스루를 고른 기준은 단순하다. **레퍼런스 케이스가 1차 자료로 확보된 도메인**만 골랐다. Shopify Liquid 핫경로 (Tobi Lütke 본인 사례), idealo Search Ranking RAG 튜닝 (사내 보고), az9713 시스템 프롬프트 진화 (오픈소스 레포). 셋 다 결과 수치가 공개되어 있고, 양면 — 인상적인 면과 회의적인 면 — 이 함께 보고된 사례들이다. 함정 없이 그려진 사례는 없으니, 함정을 포함해서 그리는 편이 낫다.

## 깊은 워크스루 1 — Liquid 형 핫경로 함수 성능 최적화

### 1. 레퍼런스 케이스

Shopify CEO Tobi Lütke가 자기가 20년 전에 쓴 Liquid 템플릿 엔진에 autoresearch 변형을 적용한 사례다. Liquid는 모든 Shopify 스토어에서 페이지 렌더링에 쓰이니, parse+render 함수는 회사 전체의 핫경로다. ThemeRunner 벤치마크 기준 7,469μs에서 3,534μs로, 53% 단축. 객체 할당은 62,620개에서 24,530개로, 61% 감소. 974개 단위 테스트는 모두 통과. 120 실험에 93 commit.

수치는 인상적이다. 그런데 Tobi 본인이 트위터에서 한마디를 덧붙였다 — "this is probably somewhat overfit." 그리고 개발자 Josh Moody는 그 PR을 보고 "코드 품질이 그냥 나쁘다(just bad)"고 비판했다. PR은 머지되지 않은 채로 남아 있다. 2026년 Mining Software Repositories 컨퍼런스의 분석은 한 발 더 나갔다 — 403개의 AI 에이전트 commit 중 56.1%가 Maintainability Index 하락, 42.7%가 Cyclomatic Complexity 상승. 핫경로 최적화는 *되긴 된다*. 하지만 *그냥 둬도 되는 결과인가*는 별개의 질문이다.

이 양면이 이 워크스루의 설계 출발점이다. 우리가 만들 루프는 53% 단축을 노리는 게 아니라, *53% 단축을 노리면서 동시에 Maintainability를 안 망가뜨리는* 루프다. 그러려면 보조 게이트가 다중으로 필요하다.

### 2. 편집 가능 아티팩트

핫경로 한 함수. 한 함수다. 자기 코드베이스의 어디서든 골라보자 — 사내 템플릿 엔진의 `render()`, JSON 파서의 메인 루프, SQL 빌더의 어떤 단계, 회계 계산 로직의 핵심 함수. 백 줄 미만이면 이상적이고, 삼백 줄을 넘으면 너무 넓다. 좁을수록 에이전트가 길을 잃지 않는다.

권한 분할은 명확하게 그어 둔다. 그 한 함수만 자유 편집. 그 함수가 호출하는 다른 함수, 단위 테스트 파일, 벤치마크 입력 데이터, 보조 게이트 설정은 전부 frozen. 5장의 표현을 빌리면 — 이게 사람과 에이전트 사이의 약속이다.

### 3. 메트릭

1차 메트릭은 `pytest-benchmark`의 p50 wall-clock이다. binary 게이트는 "기준선 대비 단축" — 0.1%라도 빨라지면 통과, 아니면 discard. p50을 쓰는 이유는 outlier에 흔들리지 않으려는 것이다. p95도 보조로 봐 둔다.

보조 게이트는 세 단으로 둔다.

- **단위 테스트 100% 통과**: 하나라도 깨지면 즉시 discard. 이건 빨간 게이트.
- **Maintainability Index (radon) 기준선 대비 -5% 이내**: MSR 컨퍼런스의 56.1% 하락 패턴을 방어하는 자리다.
- **Cyclomatic Complexity 상승 +20% 이내**: 복잡도가 폭주하는 변경 — 종종 53% 단축의 비결인 — 을 막는다.

세 보조 게이트가 함께 작동하면 "빠르지만 끔찍한 코드"는 걸러진다. 게이트가 셋이라 빡빡해 보이지만, 사내에 도입한다면 더 늘릴 수도 있다 — 보안 정적 분석, 메모리 누수 검사, lint 통과 등.

### 4. frozen 부분

- 단위 테스트 셋. 에이전트가 테스트를 수정하면 reward hacking이 시작된다.
- 벤치마크 입력 데이터. 좁은 입력에 과적합한 캐시 트릭이 흔한 게이밍 패턴이다. 입력은 풍부하고 다양해야 한다.
- 보조 게이트의 임계값. -5%, +20% 같은 숫자 자체가 frozen.
- 평가 셸 스크립트 전체. `bench.sh` 한 파일에 묶어두고 잠가 둔다.

### 5. Claude Code 도구 묶음

깊이 따라가 보자. Claude Code의 어떤 기능을 어떻게 묶는가.

- **Bash**: 벤치마크 실행, 테스트 실행, 보조 게이트 계산. `bench.sh`를 한 호출로 돌리도록 묶는다.
- **Edit**: 그 한 함수만. Edit의 strict matching 덕에 에이전트가 무심코 다른 곳을 만지지 않는다.
- **Read**: 테스트 파일을 읽어 어떤 케이스가 검증되는지 확인. 단, 테스트를 *고치는* 권한은 없다.
- **sub-agent**: 별도 에이전트로 평가자를 분리한다. 평가자는 코드를 못 만지고 평가만 한다. 이게 frozen 메트릭을 행위 자체로 강제하는 길이다 — 메인 에이전트가 평가 코드에 접근할 길을 구조적으로 끊는다.

여기서 자주 놓치는 디자인 결정 하나를 짚자. 평가자 sub-agent는 *별도 컨텍스트*에서 돈다. 메인 에이전트의 변경 의도를 모른 채 그냥 벤치를 돌리고 숫자만 돌려준다. 이게 왜 중요한가 — LLM judge가 자기 점수를 매기는 구조는 거의 무조건 게이밍된다. 인간도 자기 시험지를 채점하면 점수가 올라간다. 같은 원리다.

### 6. program.md 스케치 5줄

자기 자리에서 그대로 채워 시작할 수 있는 템플릿이다.

```
# program.md — 핫경로 성능 최적화 루프
1. 편집 범위: src/render.py의 render() 함수만. 다른 어떤 파일도 건드리지 마.
2. 1회 사이클: `bench.sh`를 호출. p50 wall-clock + 보조 게이트 3개를 본다.
3. 기준선 대비 p50 단축 + 보조 게이트 3개 모두 통과면 keep (git commit). 아니면 git reset --hard.
4. 사람이 자고 있을 수도 있다. 멈춰서 묻지 마. 단순한 트릭이 막히면 더 깊이 생각하고, 인용된 코드의 다른 부분을 다시 읽어. NEVER STOP.
5. results.tsv에 commit / p50_us / MI / CC / status / description 5컬럼 기록.
```

5줄이다. autoresearch 원본 program.md의 정신을 그대로 가져왔다. 자기 환경의 path와 이름만 갈아 끼우면 시작점이 된다.

### 7. 실패 모드

이 루프가 어디서 부서지는지를 미리 알아두자.

- **테스트가 약해서 reward hacking**: 974개라는 숫자에 안심하면 안 된다. 핫경로의 *어떤 경로*만 커버하고 나머지는 비어 있을 수 있다. 에이전트는 안 커버된 경로로 도망간다. 이걸 막으려면 mutation testing 같은 보조 평가를 함께 두는 편이 낫다 — 박스 B에서 그 패턴을 가져올 수 있다.
- **벤치 입력 분포가 좁아서 캐시 친화 트릭에 과적합**: 입력이 항상 같으면 에이전트는 입력을 캐싱하는 트릭을 발명한다. 벤치 입력은 실제 운영의 분포를 반영해야 한다.
- **Maintainability 양면**: -5% 게이트로 막는다지만, 그 안에서도 점진적으로 복잡해진다. 50개 commit 뒤에 코드를 열어보면 한숨이 나온다. 그래서 사람의 머지 게이트가 필요하다 — 에이전트가 PR을 만들고, 사람이 머지한다. autoresearch처럼 자기 브랜치에 commit만 쌓고 머지는 정기적인 리뷰로.
- **단위 테스트가 통과해도 통합 환경에서 깨지는 변경**: 단위 테스트는 단위만 본다. 통합 테스트 셋도 보조 게이트에 넣는 편이 낫다.

여기서 한국어 가이드들 — GPTers나 Jangwook 같은 — 이 강조하는 디자인 직관을 짧게 끼워 두자. *"처음 5분"의 함정*. 한 번에 너무 큰 범위를 풀어 두면 에이전트가 헤맨다. 한 함수로 좁게 시작해서, 그 함수가 충분히 다듬어지면 다음 함수로 옮기는 식의 *점진적 확장*이 작동한다. 처음부터 모듈 전체를 풀어두는 야망은 거의 항상 후회로 끝난다.

### 8. 한국 적용 사고 실험

9장의 카드 3 — 쿠팡형 결제 검증 함수 — 가 정확히 이 패턴이다. 결제 시스템은 보수적이라 사람의 머지 게이트를 하나 더 두는 편이 낫고, 보안 정적 분석을 보조 게이트에 추가한다. 그 외의 모든 핫경로 함수 — 검색 인덱서, 추천 score 계산, 결제 amount 검증, 한국어 형태소 분석 토크나이저 — 가 이 패턴의 후보다.

## 깊은 워크스루 2 — idealo 형 RAG 파이프라인 튜닝

### 1. 레퍼런스 케이스

idealo가 자사 Search Ranking에서 autoresearch 패턴을 적용한 사례. 결과 — preprocessing latency 5.9배 단축, 엔드투엔드 latency 37% 감소, 1시간 안에, API 비용 약 7달러. 짧고 깔끔한 보고다. 그래서 더 가져가기 좋은 케이스다.

여기서 짚어 둘 것은 — idealo의 사례는 *RAG의 검색 단계*에 가깝지만, 같은 패턴을 RAG 전체로 옮기기 좋다. retriever config (chunk size, overlap, embedding model, rerank top-k, prompt template의 retrieval 섹션)를 한 파일로 묶고, 그 한 파일을 진화시키는 형태로.

### 2. 편집 가능 아티팩트

`retriever_config.yaml` 한 파일. 그 안에 다음이 들어 있다.

- chunk size, overlap
- embedding model 이름 (또는 dimension)
- rerank top-k
- prompt template의 retrieval 섹션 (어떤 문서를 어떻게 LLM에 주입하는지)
- (선택) reranker 모델 이름

그 외 — 인덱스 build 코드, 평가 로직, held-out QA 셋 — 은 전부 frozen.

### 3. 메트릭

1차 메트릭은 held-out QA 정답률이다. 핵심은 *정규식·정답 매칭 기반*이지 LLM judge가 아니라는 점. LLM judge는 게이밍의 첩경이다 — 에이전트가 평가 LLM을 알면 그 LLM의 평가 패턴에 과적합한 응답을 만든다. 정규식·정답 매칭은 더 답답하지만 게이밍이 어렵다.

보조 게이트 두 단.

- **p95 latency 상한**: 정답률을 올리는데 latency가 폭주하는 패턴 — 예를 들어 rerank top-k를 무한히 늘리는 — 을 막는다.
- **token 비용 상한**: prompt를 늘려서 정답률을 끌어올리는 패턴을 막는다.

### 4. frozen 부분

- held-out QA 세트. 가장 신경 쓸 부분이다. *진짜로 held-out인가*를 사람이 한 번은 검증해야 한다.
- 평가 prompt와 정답 매칭 로직.
- leakage 검증 의식 — 평가 셋에 인덱스가 닿지 않는가. RAG에서 가장 흔한 사고는 평가 셋의 문서가 인덱스에 들어가 있는 상황이다. 평가 셋의 정답은 인덱스가 못 보도록 별도 보관.

leakage는 RAG의 가장 끔찍한 함정이다. 정답을 인덱싱한 상태로 평가하면 retriever가 *정답 문서를 찾아낸다*. 점수는 폭등하고, 사람은 안심하고, 운영에 배포한다. 그리고 실제 사용자 쿼리에서는 무너진다. 4장의 metric paradox가 RAG에서 가장 흔하게 발현되는 형태다.

### 5. Claude Code 도구 묶음

- **Bash**: 인덱싱 + 평가를 한 셸 스크립트로. `index.sh && eval.sh`.
- **Edit**: retriever_config.yaml만.
- **Read**: 로그를 읽어 어떤 query에서 실패했는지 확인. 단, 정답을 읽으면 안 된다 — 정답이 컨텍스트에 들어오면 에이전트의 다음 변경이 오염된다. 정답은 평가 sub-agent만 본다.
- **sub-agent**: 평가자 분리. 메인 에이전트는 *어떤 query에서 실패했는지의 패턴*만 받지, 정답 자체는 안 본다.

이 권한 분할이 RAG에서 특히 중요하다. 정답이 메인 에이전트의 컨텍스트에 한 번이라도 들어오면, 그 정답이 다음 prompt template으로 새어 들어갈 수 있다. *프롬프트 leakage*는 RAG의 거의 가장 미묘한 사고 유형이다.

### 6. program.md 스케치 5줄

```
# program.md — RAG retriever 튜닝 루프
1. 편집 범위: configs/retriever.yaml만. 다른 어떤 파일도 안 된다.
2. 1회 사이클: index.sh로 재인덱싱 → eval.sh로 held-out QA 정답률 + p95 latency + token 비용 측정.
3. 정답률 +1% 이상 + 보조 게이트 두 개 통과면 keep. 아니면 git reset --hard.
4. 평가 정답은 절대 읽지 마. 실패 쿼리 ID와 카테고리만 봐. 사람이 자고 있다 — NEVER STOP.
5. results.tsv에 commit / accuracy / p95_ms / cost_usd / status / description 6컬럼 기록.
```

### 7. 실패 모드

- **held-out에 인덱스 leakage**: 위에서 길게 다뤘다. 첫 사이클을 돌리기 전에 사람이 한 번은 leakage 검증을 거쳐야 한다.
- **LLM judge 사용 시 점수 게이밍**: judge의 평가 패턴을 학습한 응답을 만든다. 가능하면 정규식·정답 매칭으로.
- **retrieval 다양성 손실**: 평가 셋이 좁으면 자주 등장하는 query 패턴에만 과적합한다. 소수 query는 무너진다. adversarial QA 셋 — 평가 셋과 분포가 다른 별도 셋 — 을 정기적으로 들이대 검증.
- **embedding model을 바꾸는 변경의 비용**: 대형 embedding을 쓰면 정답률은 오르지만 latency와 비용이 폭주한다. 보조 게이트가 막지만, 한 번에 한 차원만 풀어주는 식의 운영도 한 방법이다.

### 8. 한국 적용 사고 실험

9장의 카드 1 — 네이버형 한국어 검색 랭킹 — 이 가장 가깝다. 한국어 RAG에서는 평가 셋 구축 자체가 가장 큰 진짜 작업이다. *정답이 있는 사내 문서 100개를 어떻게 모을 것인가*. 100개가 아니라 1,000개라면 더 좋다. 이 100개를 모으는 일은 사람의 작업이고, 일주일이 걸리고, 한 번 모으면 6개월 frozen이다. 모으는 시점에 leakage 가능성을 차단해야 6개월이 가치 있다.

## 깊은 워크스루 3 — az9713 형 시스템 프롬프트 진화

### 1. 레퍼런스 케이스

az9713/autoresearch-prompt-optimization 레포. 시스템 프롬프트의 task 통과율을 74.72%에서 100%로, 8번의 실험, 사람 개입 0. train.py 자리에 `prompt.txt`가 들어간 형태다.

수치만 보면 환상적이다. 그런데 한 가지 주의 — 8 실험에 100%는 너무 깔끔하다. 좁은 task 셋에 과적합했을 가능성이 매우 높다. 그래서 이 워크스루의 디자인은 *그 과적합을 의식한 형태*다. adversarial set이라는 두 번째 평가 셋이 본 워크스루의 핵심 차별점이다.

### 2. 편집 가능 아티팩트

`system_prompt.md` 한 파일. 또는 사내 에이전트라면 그 에이전트의 system 메시지가 정의된 한 파일. 한 파일이라는 점이 중요하다 — system prompt가 여러 파일에 흩어져 있으면 편집 범위가 흐려진다.

### 3. 메트릭

1차 메트릭: held-out task set 통과율. binary 매칭으로 — 정답 문자열이 응답에 포함되는지, 또는 특정 형식을 따르는지를 정규식으로 가른다.

보조 게이트 셋.

- **응답 길이 상한**: 시스템 프롬프트가 점점 길어지는 패턴 — 에이전트가 자기에게 더 많은 지시를 추가하는 형태 — 을 막는다. 응답이 무한정 길어지면 통과율은 오를 수 있어도 사용성이 무너진다.
- **token 비용 상한**: 위의 변주. prompt + 응답 합산.
- **adversarial test set 통과율**: 이게 핵심이다. *task set과는 분포가 다른* 별도 셋을 보관. 좁은 task에 과적합한 prompt는 여기서 무너진다. adversarial 셋의 통과율이 기준선 대비 -10% 이상 하락하면 즉시 discard.

세 보조 게이트가 함께 작동하면 "task set에 100% 통과하지만 실제로는 쓸 수 없는 prompt"는 걸러진다.

### 4. frozen 부분

- task set
- adversarial set
- 정답 매칭 로직
- 평가 셸 스크립트 전체

adversarial set의 보관은 특별히 신경 써야 한다. 메인 에이전트가 그 셋의 패턴을 학습하면 의미가 사라진다. 별도 디렉토리, 별도 권한, sub-agent만 접근.

### 5. Claude Code 도구 묶음

- **Bash**: task suite 실행. `task_run.sh`로 묶기.
- **Edit**: prompt 파일만.
- **sub-agent**: 평가자 별도. adversarial 셋 접근 권한도 sub-agent만.

여기서 흥미로운 디자인 가능성 하나를 짚자. *두 개의 평가 sub-agent를 둔다.* 하나는 task set만, 하나는 adversarial set만. 메인 에이전트는 두 sub-agent의 점수를 받지만, 각 set의 내용은 보지 못한다. 점수의 합이 아니라 *둘 다 통과*가 keep 조건. AND 게이트.

### 6. program.md 스케치 5줄

```
# program.md — 시스템 프롬프트 진화 루프
1. 편집 범위: prompts/system.md만. 절대 task_set/, adversarial_set/는 못 본다.
2. 1회 사이클: task_run.sh → task 통과율 + adversarial 통과율 + 응답 길이 + 비용 측정.
3. task 통과율 +1% & adversarial -10% 이내 & 응답 길이·비용 게이트 통과면 keep. 아니면 git reset --hard.
4. 사람이 자고 있다 — NEVER STOP. 좁은 task에 과적합하지 마. adversarial 셋에서 무너지면 거꾸로 가는 거다.
5. results.tsv에 commit / task_pass / adv_pass / len / cost / status / description 7컬럼.
```

### 7. 실패 모드

- **task set 좁음 → 그 외 사용 사례 망가짐**: adversarial set이 이 패턴의 1차 방어선이다. 그래도 정기적으로 adversarial set 자체를 갱신해야 한다 — 6개월 frozen은 좋지만, 1년 frozen은 위험하다. 사람이 분기마다 adversarial 셋에 새 케이스를 추가.
- **LLM judge 자기 점수 매김 → 게이밍**: judge를 안 쓰고 정규식으로 가는 편이 낫다.
- **시스템 프롬프트가 점점 길어짐**: 응답 길이 게이트가 일부 막지만, prompt 자체의 길이 게이트도 두는 편이 낫다. 1KB, 2KB 같은 상한.
- **az9713의 100% 함정**: 100%는 의심해야 한다. 진짜 100%는 task set이 충분히 넓다는 뜻이거나, 셋이 너무 좁다는 뜻이다. 둘 중 어느 쪽인지를 사람이 정기적으로 확인.

## 카탈로그 박스 — 가능성의 지도

깊은 워크스루 세 개에서 패턴은 충분히 그렸다. 같은 뼈대를 어디에 더 끼울 수 있는지, 세 개의 짧은 카탈로그 박스로 지도를 그려두자. 각 박스는 자기 도메인에 옮기고 싶으면 위의 8단 구성을 그대로 따라 채우면 된다.

> **박스 A — CI 그린닝과 flaky test 사냥**
> 편집 가능: 테스트 파일·setup·fixture. 1차 메트릭: 100회 반복 통과율. 보조 게이트: 평균 실행 시간 상한, 테스트 의도(assertion 수) 유지. frozen: 통과율 측정 셸 (100회 반복기), 실제 코드. retry/timing/순서 의존성을 자동 디버그한다. *"CI는 새벽 3시에 누군가 본다"* 상태에서 *"CI 그린닝 에이전트가 자율적으로 PR을 올린다"*로. 함정: trivial assertion으로 통과율만 채우는 reward hacking 가능 — assertion 의미성 검사가 보조 게이트로 필요. 한국 적용: 사내 CI의 만성 flaky test 5개를 한 주 동안 묶어 풀어보기.

> **박스 B — 테스트 커버리지 진화 (mutation testing 동반)**
> 편집 가능: 테스트 파일 1개. 1차 메트릭: statement coverage %. 보조 게이트: 기존 테스트 100% 통과 + 신규 테스트가 mutation testing에서 mutant kill 율 70% 이상. frozen: 대상 코드, mutation 도구의 설정, kill 율 임계값. statement coverage만 보면 trivial assertion으로 채우는 게이밍이 너무 쉽다 — mutation testing이 그 게이밍을 잡는다. 변이 코드가 통과하면 그 테스트는 빈 깡통이라는 뜻. 한국 적용: 커버리지 미달 모듈 한 개를 한 주에 50% → 80%로 끌어보기, mutation kill 율을 동반 게이트로.

> **박스 C — 성능 회귀 자동 격리·복구**
> 편집 가능: PR 단위 커밋들. 1차 메트릭: 벤치마크 wall-clock 회귀 여부 (binary). 루프: PR 전체를 5분짜리 회귀 실험으로 자동 분해 → 회귀 보이는 commit만 isolation (bisect) → autoresearch가 fix 시도. CI green-keeping을 진짜로 자율화하는 패턴. 함정: 회귀의 원인이 환경(노이즈)이지 commit이 아닌 경우 — 5회 반복 측정으로 노이즈 분포를 먼저 잡아 두는 편이 낫다. 한국 적용: 사내 백엔드의 p95 latency 회귀를 자동으로 잡아내고 fix까지 시도하는 night job.

세 박스는 점진적 확장의 권유다. 깊은 워크스루 한 도메인을 끝까지 풀어본 뒤, 그 경험으로 박스 하나를 더 펴는 식의 운영이 작동한다. 한 번에 여섯 도메인을 풀려 들면 다 어설프게 끝난다.

## 도구 묶음의 공통 패턴 — Claude Code의 단일 마스터 루프

세 워크스루를 묶어 보면 한 가지 공통 구조가 보인다. **메인 에이전트 + 평가자 sub-agent + frozen 셸 스크립트.** Claude Code의 single-threaded master loop 아키텍처가 이 구조와 자연스럽게 맞물린다 — 메인 에이전트가 master loop이고, 평가자는 별도 컨텍스트의 sub-agent, 평가 셸은 도구가 호출하는 외부 프로세스.

이게 왜 작동하는가. 권한 분할이 *행위*로 강제되기 때문이다. 메인 에이전트는 자기가 평가 코드에 접근할 수단을 도구 레벨에서 갖지 못한다. Edit 도구의 file scope가 좁고, Read 도구가 정답 디렉토리를 못 보고, sub-agent의 컨텍스트가 분리되어 있다. 사람이 약속을 명문화하고, 약속을 도구의 권한 모델이 강제한다.

10장에서 가장 가져가고 싶은 게 있다면 이 한 가지다. **권한 분할은 program.md에 쓰는 텍스트가 아니라 도구의 권한 모델로 강제되어야 한다.** 텍스트만 있으면 에이전트가 "왜 안 되죠"라며 우회로를 찾는다. 도구가 못 하게 하면 우회로가 없다.

## 마무리

세 워크스루의 program.md 스케치 5줄을 다시 한번 훑어보자. 핫경로 성능, RAG 튜닝, 시스템 프롬프트 진화 — 셋 다 공통점이 있다. 편집 범위 한 줄, 사이클 정의 한 줄, keep 조건 한 줄, NEVER STOP 한 줄, results.tsv 한 줄. 자기 자리에서 갈아 끼우면 시작점이 된다.

이번 주에 깊은 워크스루 하나만 골라보자. 가장 좁은 핫경로, 가장 작은 RAG 모듈, 가장 단순한 시스템 prompt. 좁게 시작해서 그게 작동하는 것을 본 뒤에 넓혀가는 편이 낫다. 처음부터 큰 범위는 거의 항상 후회로 끝난다.

그런데 한 가지 흥미로운 질문이 남는다. 우리가 program.md를 손으로 써서 에이전트의 운영을 정의했는데 — 그 program.md 자체를 진화시키면 어떻게 될까. SKILL.md를 진화시키면, 에이전트의 운영 매뉴얼을 같은 ratchet 루프에 넣으면, 무슨 일이 벌어질까. 그리고 그 메타 루프가 — 좀 어색하지만 정직하게 말하면 — 이 책을 쓴 도구 자체에도 적용된다면. 11장에서 그 자리로 가본다.
