## 제목 후보

1. **DDD는 죽지 않는다 — AI 에이전틱 코딩 시대에 다시 묻는 도메인 주도 설계**
   - 톤: 단언형, 도전적
   - 포지셔닝: "DDD 무용론에 맞서는 시니어의 반론서". 표지에서부터 입장을 표명. AI 시대에 흔들리는 시니어가 손에 잡고 "그래, 이게 내가 찾던 답이다" 느끼게 한다.
   - 약점: 단언형이라 본문의 "변형론·축소론도 정직하게 받아들인다"는 균형과 충돌. v2에서 무용론 분량을 30% 늘리기로 했으므로 더 두드러진 약점.

2. **AI는 도메인을 모른다 — 에이전틱 코딩 시대, 우리가 다시 짚어야 할 것들**
   - 톤: 시적·관조적
   - 포지셔닝: "에세이형 기술서". AI의 한계와 사람의 역할을 동시에 환기. 시니어가 야간에 위스키 한 잔과 함께 읽고 싶을 만한 책.
   - 약점: 실무 적용서를 기대한 독자에게 추상적으로 보일 위험.

3. **에이전트의 시대, DDD는 여전히 유효한가? — 변하는 것, 살아남는 것, 사라지는 것**
   - 톤: 질문형, 분석적
   - 포지셔닝: "현재진행형 논쟁의 정리·재구성서". 부제가 5~9장 패턴 분석부의 기조를 미리 공표. 컨퍼런스 발표를 책으로 풀어낸 듯한 진중함.
   - 약점: 제목이 길고(공백 포함 약 35자), 의문문이라 결단력 있는 답을 원하는 독자에게 약하게 보일 수 있음.

**추천: 3번 — 에이전트의 시대, DDD는 여전히 유효한가? — 변하는 것, 살아남는 것, 사라지는 것**

이유:
- 책의 5개 핵심 명제가 모두 "어디까지 유효하고 어디는 변하는가"라는 질문에 답하는 구조다. 제목이 책의 골격을 그대로 드러내야 한다.
- 부제 "변하는 것, 살아남는 것, 사라지는 것"이 5~9장 패턴 분석부의 기조를 공표한다 — 독자가 "이 책은 분류와 분석을 해주는구나" 하고 기대 정렬.
- 시니어·아키텍트는 결론을 강요하는 책보다 자기 사고를 도와주는 책을 선호한다.
- v2에서 무용론 비중이 늘어났으므로 1번 단언형보다 3번 질문형이 본문과 더 정합.

> 리뷰 피드백 S14·S15에 대한 응답: 본 제목이 의문문이라는 약점은 인정하되, v2에서 무용론을 더 진지하게 다루기로 한 결정이 단언형 제목을 더 약화한다. 부제의 "변하는 것·살아남는 것·사라지는 것"이 결단의 시그널을 충분히 준다고 판단해 3번 유지.

## 내러티브 아크

이 책은 **의문 제기 → 시대 진단 → 기초 재해석 → 논쟁 정리 → 패턴별 정밀 분석 → 실증 사례(분할) → 거버넌스 → 사람과 팀의 영역 → 적용 전략 → 미래 전망**의 10단 구조로 흐른다. v1의 9단에서 거버넌스 단계가 추가되어 10단이 됐다.

1~2장은 **의문 제기와 시대 진단**이다. 시니어 개발자가 AI 도구로 일하면서 느끼는 모호한 불안을 첫 화두로 삼고("뭔가 빨라진 것 같은데, 정말 내가 더 잘하고 있는 것일까?"), 그 불안의 정체를 vibe coding·agentic coding·context engineering·harness engineering이라는 시대 언어로 해부한다. METR 19% 격차, Anthropic 0~20% 위임 가능성 같은 수치가 독자의 직감을 데이터로 확정한다. **v2에서 2장은 4용어로 슬림화** — vibe modeling은 9장(Event Storming 부활)으로 이전한다.

3장은 **DDD 기초 재해석**이다. v1의 "재방문"에서 "재해석"으로 격상했다. 패턴 카탈로그 정리는 표·박스로 압축하고, 본문 절반 이상을 **"어떤 패턴이 23년 동안 닳지 않았고 어떤 패턴은 시대적 색이 짙게 입혀졌는가"**의 재해석에 할애한다. 시니어 독자가 "내가 아는 걸 또 짚느냐"로 이탈하지 않도록.

4장은 **논쟁의 한복판으로 진입**한다. 강화론·변형론·축소론·흡수론의 네 진영을 차분히 펼친다. **v2에서 축소론(DDD 무용론)의 분량을 약 30%로 확보**하고, "내가 무용론자라면" 시뮬레이션 박스를 신설한다. 무용론의 강한 논거(과한 ceremony, 작은 도메인 오버킬, AI가 대체할 수 있는 부분)를 정직하게 인정한 후 변형론으로 응답하는 구조다.

5~9장은 **패턴별·자산별 정밀 분석**이다. 5장 ubiquitous language의 격상, 6장 bounded context와 agent의 매핑, 7장 aggregate와 invariant — 사람의 마지막 영역, 8장 context map과 ACL의 새로운 의미, 9장 domain event와 멀티 에이전트 통신 + **vibe modeling**. 각 장은 "이 패턴이 어떻게 변하는가 / 강화되는가 / 사라지는가"를 결판낸다. 9장은 v1보다 두꺼워진다.

10·11장은 **결제 게이트웨이 가상 프로젝트의 분할 사례부**다. v1의 16,000자 단일 챕터를 둘로 쪼갰다. 10장은 **디스커버리·전략** — vibe modeling 워크숍, ULang 정렬, bounded context 식별, context map 설계까지. 11장은 **전술·실행** — aggregate·invariant, domain event 카탈로그, 에이전트 토폴로지, ACL, 휴먼 체크포인트, 측정 지표까지. 두 챕터가 짝을 이루며, 각 단계 말미에 **"다른 도메인에서는?"** 사이드바(의료·게임·물류 중 1개를 단계마다 1문단 비교)와 **Toss·당근·Stripe 공개 자료 인용**이 들어간다.

12장은 **AI 시대의 거버넌스 — 보안·환각·비용·관측성·책임**이다. v1에서 한 챕터도 못 받았던 운영 거버넌스 트랙을 신설한다. 박정현 2.74×, arXiv 26.1%, AI 환각의 도메인 모델링적 결과, 비용·관측성·평가(eval), IP·라이선스·도메인 데이터 윤리 — 다섯 기둥. 11장 결제 사례의 측정 지표를 거버넌스 챕터로 자연 확장한다.

13·14장은 **사람과 팀의 영역**이다. 13장은 팀 구성과 역할 재편(PO·아키텍트·시니어·주니어), 한국 조직의 사일로 문제. **부제에 "한국 조직의 사일로 문제를 정면으로"** 명시. 카카오 헤어샵 사례를 5장과 다르게 재인용(5장 = ULang 한국형 모범 / 13장 = BC 못 나눈 조직의 한계). 14장은 시니어 정체성과 학습의 역설.

15장은 **적용 전략**이다. 책을 덮자마자 월요일 아침에 무엇을 할지 — `docs/glossary.md`부터 첫 bounded context, 휴먼 체크포인트, 측정 지표까지 구체적 로드맵. 도구 추천을 "도구 이름이 아니라 역할(Schema 검증기·글로사리 운영기·Event Storming 보드)"로 재구성해 시간 내성을 확보한다. **Legacy 마이그레이션 절(S1 해결)**을 추가해 grass-field 가정에서 벗어난다.

16장은 **미래 전망과 닫는 글**이다. DDD의 진화 시나리오 세 갈래(흡수·강화·잔존), 각 시나리오마다 "시니어가 가장 먼저 잃을 직무 vs 가장 강해질 직무" 1문단 추가. 닫는 문단에 **15장 1주차 행동 3개를 한 줄로 재호명**("어휘 30개 등재, 도메인 전문가 30분 인터뷰, BC 종이 스케치 — 이것이 월요일이다") 후 "월요일 아침에 `docs/glossary.md`를 열자"로 마감.

10단 곡선: **의문 → 진단 → 기초(재해석) → 논쟁(무용론 강화) → 분석×5 → 종합(분할 사례×2) → 거버넌스 → 사람×2 → 행동 → 전망**. v1의 9단에서 (a) 거버넌스 단계가 추가됐고 (b) 종합 단계가 분할 사례 둘로 늘어났다.

## 챕터 목록

### 1장. AI가 코드를 짜는데, 나는 왜 더 불안해질까?

- **핵심 질문:** AI 도구를 도입한 시니어 개발자가 느끼는 모호한 불안의 정체는 무엇인가? 그리고 그 불안이 우리에게 무엇을 알려주려 하는가?
- **주요 내용:**
  - 박정현(NDS Cloud Tech, 2026-03)의 8년차 AI 엔지니어 사례 — 채팅 기능 하나 더 붙이려다 기존 코드가 깨졌다는 증언으로 책을 연다.
  - METR 2025의 충격적 결과 — 경험 많은 개발자가 AI 도구로 실제 19% 더 느려졌는데 본인은 20% 빨라졌다고 인식했다. 인지와 실제의 격차.
  - "AI 공저 코드의 보안 취약점이 사람 코드의 2.74×"라는 NDS Cloud Tech 데이터 (12장 거버넌스 챕터의 첫 복선).
  - 시니어가 느끼는 세 가지 불안 — 코드 검증 비용 폭증, 도메인 직관의 마모, 팀 내 자신의 위치 모호.
  - 이 책이 다루는 핵심 질문 다섯 가지 선언 — ① DDD의 의식과 본질, ② Bounded Context와 Agent, ③ Ubiquitous Language의 격상, ④ Aggregate와 invariant의 인간 영역, ⑤ AI를 잘 쓰는 개발자의 정의. (v2에서 ⑥ "거버넌스를 어떻게 구성하는가"가 12장 신설을 미리 알리는 보너스 질문으로 등장.)
  - 1장 톤은 "감정의 불안" — 4장 "관점의 정리"와 의식적으로 차별화. (N1 해결)
  - "이 책을 다 읽고 나면 무엇이 달라질 것인가" — 독자에게 던지는 약속.
- **독자가 얻는 것:** 자신이 느끼던 불안의 정체를 데이터와 사례로 확인하는 위로. 동시에, 그 불안이 곧 다음 학습의 출발점이라는 자각.
- **예상 분량:** 약 10,000자
- **사용할 reference 섹션:** 2.2 생산성·역할 변화 데이터 / 5.6 한국 vibe coding 포기 사례 / 7.3 한국 실무 사례·증언

### 2장. Vibe coding부터 Harness engineering까지 — 우리는 지금 어디 서 있나?

- **핵심 질문:** AI 에이전틱 코딩 시대를 둘러싼 용어 네 개(vibe coding, agentic coding, context engineering, harness engineering)는 각각 무엇이고, 그 변천이 우리에게 말해 주는 것은 무엇인가?
- **주요 내용:**

  - Dobrohotov의 "Vibe Coding Is Dead. Long Live Agentic Coding" (2026-02) — 본인 사례로 본 사망 선언과 후행 평가.
  - Anthropic 2026 Trends Report — agentic coding의 정의: 다수 에이전트가 시간·일 단위로 자율 실행, 엔지니어는 조율자.

### 3장. DDD를 다시 펼치자 — 23년의 정전, 무엇이 닳지 않았고 무엇은 색을 입었나

- **핵심 질문:** Eric Evans의 2003년 책이 정립한 DDD의 핵심 어휘는 지금 우리에게 여전히 무엇을 의미하는가? 어떤 패턴이 23년 동안 닳지 않았고, 어떤 패턴은 시대적 색이 짙게 입혀졌는가?
- **주요 내용:**
  - **(전반 30% — 패턴 카탈로그 압축 정리)** 표·박스로 빠르게: Domain·Subdomain(Core/Supporting/Generic), Bounded Context, Context Map, Ubiquitous Language / Entity, Value Object, Aggregate, Repository, Factory, Domain Service, Domain Event, Anti-Corruption Layer. 한국어 번역어 분기(바운디드 컨텍스트 vs 경계 컨텍스트, 애그리거트 vs 집합체)와 이 책의 채택안.
  - **(후반 70% — 재해석)** **닳지 않은 패턴 vs 색이 입혀진 패턴**의 정밀 분류:
    - **닳지 않은 것 (essence):** Ubiquitous Language (오히려 격상), Bounded Context의 식별 원리, Aggregate의 invariant 책임, Core/Supporting/Generic 분류의 시간 재배분 가치, Domain Event의 fact-as-message 의미.
    - **색이 입혀진 것 (form):** Context Map의 다이어그램 형식(→ 실행 가능 코드로), ACL의 데이터 변환 의미(→ semantic firewall로), Repository/Factory의 수작업 코딩(→ AI 위임 가능), 폴더 구조·계층 카탈로그 같은 형식주의.
    - **거의 닳은 것 (ceremony):** 폭포수형 모델링 워크숍의 무게, 거대 문서화, 패턴 카탈로그를 그대로 따르는 의식.
  - Evans 본인의 "은탄환 아님" 단서가 23년 후 어떻게 적중했는가의 회고.
  - 이 분류표가 5~9장 패턴 분석의 큰 그림 미리보기 역할.
  - **v2 변경: 단순 재정리에서 재해석으로 격상.** 본문 50% 이상이 "무엇이 닳고 무엇은 색을 입었나"의 분류·논증에 할애된다. (S5 해결)
- **독자가 얻는 것:** 5년 전 *DDD Start!* 를 본 독자가 손때 묻은 기억을 단숨에 회복하면서도, 동시에 "내가 알던 DDD가 어떻게 다시 보이는가"의 첫 깨달음.
- **예상 분량:** 약 10,000자
- **사용할 reference 섹션:** 1.1 DDD 핵심 개념 / 7.1 한국어 DDD 표준 어휘 / 7.2 한국 입문 표준 / 1.1 Evans "은탄환 아님" / 4.1·4.2 패턴별 변화 (미리보기로 일부 차용)

### 4장. 네 진영의 격돌 — DDD는 강화되는가, 사라지는가, 변형되는가, 흡수되는가?

### 5장. Ubiquitous Language의 격상 — LLM 시대의 1급 자산

- **핵심 질문:** Ubiquitous Language는 DDD의 한 패턴이었지만, LLM 시대에 왜 갑자기 모든 것의 출발점으로 격상되는가? 그리고 이를 어떻게 살아있는 자산으로 운영할 것인가?
- **주요 내용:**
  - Daniel Schleicher의 핵심 진단 — "LLMs are amplifiers. 'order'가 12가지 의미일 때 AI에게 모호한 명령을 주면 카오스가 증폭된다."
  - Paul Iusztin의 6-Agent 팀 — `docs/glossary.md`가 모든 산출물의 단일 진실. 코드, OpenAPI, DB 컬럼, 고객 인터페이스 모두.
  - 자동화의 진척 — arXiv 2509.00140: LLM이 SE 표준 문서에서 zero-shot으로 triple을 추출해 ontology 자동 생성.
  - 카카오 헤어샵 사례 재해석 (5장 인용 = ULang의 한국형 모범) — 예약 상태 enum이 운영진의 raw 비즈니스 어휘 그대로 코드까지 통과. AI 시대에 같은 시스템을 다시 만든다면 에이전트들에게도 같은 어휘로 컨텍스트를 주겠다. (S11 — 13장 인용과의 차별화 명시)
  - 살아있는 글로사리 운영 패턴 — 자동 추출 → 사람 검토 → 글로사리 등재의 하이브리드. 핵심 어휘일수록 사람 비중 ↑. (논쟁 6의 명시적 응답)
  - 한국 실무의 현실 — Kerry Kim(velog, 2026-01): "도메인 전문가와 개발자가 모여서 같은 언어를 쓰도록 정리하는 것이 현실적으로 어렵다." 1주일간 SDR 업무를 체험한 후에야 진짜 시작됐다는 증언.
  - 실행 체크리스트 (요약) — `docs/glossary.md` 시작 템플릿의 핵심 5필드, 용어 등재 워크플로, 에이전트 시스템 프롬프트에 글로사리 주입하는 패턴, CI에서 글로사리-코드 일관성 검증하는 방법.
  - **부록 A 약속 명시** — 글로사리 전체 템플릿·운영 가이드는 부록 A에 분리. (S7 해결)
  - Spec Ambiguity Resolver 류 도구의 위치 — AI는 후보만 제시, 사람이 결정.
- **독자가 얻는 것:** 월요일 아침부터 `docs/glossary.md`를 만들 수 있는 구체적 시작점. ULang이 왜 다른 모든 패턴의 기초가 되는지에 대한 깊은 이해.

### 6장. Bounded Context = Agent 1개 — 사실상의 산업 표준 매핑

- **핵심 질문:** "에이전트 1개 = bounded context 1개"라는 매핑이 왜 업계의 사실상 표준이 되었는가? 그것이 실무에 의미하는 바는 무엇인가? 그리고 마이크로서비스 시대의 "BC=서비스"와 어떻게 다른가?
- **주요 내용:**
  - James Croft의 e-Commerce 사례 — Inventory Management 에이전트, Order Processing 에이전트, Regulatory Compliance 에이전트. 각 에이전트는 도메인 전문가 어휘로 일관된 ULang.
  - Paul Iusztin — "코드를 파일 타입이 아니라 actionability(bounded context) 기준으로 조직하라." 에이전트가 한 컨텍스트에 매여 있으면 reasoning이 정확해진다.
  - Nikita Golovko의 Siemens 사례 — 6개월차 prompt spaghetti: 에이전트당 3,000+ 토큰 중 85%가 파싱·통합. DDD 4패턴 적용 후 ~500 토큰, 90%가 도메인 로직. **정량적 KPI** — 12장 거버넌스 측정 지표의 한 축.
  - 멀티 에이전트 토폴로지(chain, star, mesh, workflow graph; arXiv 2601.12560)가 결국 Context Map의 한 형태.
  - 에이전트 책임 분리의 강한 규칙(Iusztin) — "No agent both writes code and decides whether the code is correct."
  - **신설 절: 모놀리스 vs 마이크로서비스 vs 멀티 에이전트의 위계 (S2 해결, ~1,500자)** — BC=Agent라는 매핑이 BC=서비스와 어떻게 다른가. 라이프사이클·배포 단위·실패 도메인·관측 단위의 비교표.
  - 에이전트 = 비즈니스 capability와 매핑 (Croft). 기술 편의로 임의 분할 금지.
  - 한계 자기 인정 — Iusztin도 "에이전트가 글로사리를 일관되게 활용하지 못하는 경우가 있다"고 인정.
  - 실행 체크리스트 — 첫 bounded context 식별, 에이전트 책임 정의, 에이전트 간 호출 패턴, 토큰·도메인 로직 비중 측정.
- **독자가 얻는 것:** "내일 우리 팀의 6-agent 구성을 어떻게 설계할 것인가"에 대한 청사진. MSA 경험을 멀티 에이전트로 옮기는 인지 다리. 에이전트당 토큰·도메인 로직 비중이라는 새 KPI.
- **예상 분량:** 약 12,000자
- **사용할 reference 섹션:** 4.1 Strategic Patterns / 5.1 6-Agent / 5.4 Croft / 5.5 Siemens Golovko / 8.4 패턴 적용 / 논쟁점 4 (BMAD vs DDD)

### 7장. Aggregate와 invariant — 사람의 마지막 영역

- **핵심 질문:** AI가 자동화에 거의 성공한 영역과 여전히 실패하는 영역의 경계는 어디인가? 그 경계가 우리의 일과 책임에 의미하는 바는 무엇인가?
- **주요 내용:**
  - Eisenreich et al.(arXiv 2603.26244, 2026) FTAPI 사례 — DDD 5단계 자동화 중 1-3단계는 신뢰성 있게 작동, 4-5단계(aggregate 설계, 기술 매핑)는 누적 오류로 실용성 떨어짐. "LLMs can enhance, but not replace, architectural expertise."
  - Wiegand et al.(arXiv 2601.20909, 2026) — 도메인 메타모델 JSON 생성은 Code Llama 4-bit + LoRA로도 잘 한다. Entity·VO 정의는 AI 자동화 가능 영역.
  - 자동화 가능 — Entity·VO·Repository·Factory·메타모델 생성. AI-DLC 한국어 백서도 이를 명시.
  - 자동화 어려움 — Aggregate 경계, invariant, 트랜잭션 일관성, cross-aggregate 정합성.
  - 왜 사람의 영역인가 — aggregate는 비즈니스 규칙의 응집이고, 그 규칙은 도메인 전문가의 머릿속과 비공식 문서에 분산. 5층 추론 chain에서 누적 오류는 폭발한다.
  - **AI 환각의 도메인 모델링적 결과 — 작은 절 (~1,200자)** — 잘못된 invariant 제안("Refund는 Payment 금액의 두 배까지 가능"), 가짜 ULang 후보, 허구의 외부 API 가정. 12장 거버넌스 챕터의 환각 본격 분석을 위한 복선.
  - Mimul의 12원칙 중 "비즈니스 규칙은 반드시 도메인 모델 내부에 명시적으로 표현되어야 한다"의 재해석.
  - Hexagonal 원칙 엄수 — "도메인 계층은 표준 라이브러리 외 외부 의존성 0" (Khosravi).
  - **테스트 전략 위계 (S3 일부 해결)** — 사람이 작성하는 invariant 단위 테스트 / AI가 생성하는 적대적 케이스 / Tester 에이전트의 E2E. 누가 무엇을 책임지는가의 분업표. 본격적 테스트 거버넌스는 12장으로.

### 8장. Context Map과 ACL의 새로운 의미 — Schema-as-Contract와 Semantic Firewall

- **핵심 질문:** 다이어그램으로 그려지던 Context Map과 데이터 변환 계층이던 ACL은 AI 시대에 어떻게 변했는가? 그리고 이 변화가 멀티 에이전트 아키텍처에 무엇을 가져오는가?
- **주요 내용:**
  - Golovko의 Context Map 격상 — 더 이상 다이어그램이 아니라 **실행 가능한 코드** (config + adapter + schema). upstream/downstream, contract version, adapter 구현체 명시. CI에서 검증.
  - Schema-as-Contract 패턴 — Pydantic, JSON Schema. 자연어 API 금지. 자동 검증·버저닝·셀프 도큐먼테이션.
  - ACL의 새 의미 — Semantic Firewall — 같은 단어가 컨텍스트마다 다른 의미일 때 의미 변환까지 책임진다. 예: "function"이 코드 생성 에이전트엔 "signature + implementation", 테스팅 에이전트엔 "behavior contract".
  - Customer-Supplier, Conformist, Shared Kernel, Published Language 같은 고전 Context Map 패턴이 에이전트 간 관계로 자연 매핑.
  - 에이전트 토폴로지(chain, star, mesh, workflow graph)와 Context Map 패턴의 대응표.
  - Anti-Corruption Layer의 한국 사례 부재 — 카카오 헤어샵이 비즈니스 어휘를 raw로 코드에 통과시킨 것은 ACL 없는 시스템의 좋은 면과 나쁜 면을 동시에 보여준다.
  - **계약 테스트(contract test) 절 (~800자)** — schema가 변할 때 누가 어떻게 검증하는가. 7장 테스트 위계의 BC 간 짝.
  - 실행 패턴 — 에이전트 간 첫 schema 정의, 버저닝 전략, contract test, semantic firewall이 필요한 경계의 식별 휴리스틱.
  - 한계 — schema가 너무 엄격하면 에이전트의 자유도가 줄고, 너무 느슨하면 spaghetti로 회귀. 균형점 찾기.
- **독자가 얻는 것:** "내 시스템의 에이전트들이 자연어로 대화하던 부분을 어디부터 schema로 바꿀 것인가"의 명확한 우선순위. Context Map을 도면이 아닌 코드로 진화시키는 첫 걸음.
- **예상 분량:** 약 11,000자
- **사용할 reference 섹션:** 4.1 Context Map / 4.2 ACL / 4.3 Schema-as-Contract / 5.5 Siemens Golovko / papers.md 논문 6

### 9장. Domain Event와 멀티 에이전트 통신 — Event Storming과 Vibe Modeling의 부활

- **핵심 질문:** Domain Event는 항상 DDD의 매력적인 패턴이었지만 실무 도입은 망설여졌다. 멀티 에이전트 환경에서 Domain Event는 왜 갑자기 필수가 되는가? 그리고 코딩 전 단계의 vibe modeling은 무엇을 가능하게 하는가?
- **주요 내용:**
  - Domain Event가 에이전트 간 통신의 기본 단위가 되는 이유 — 비동기, 느슨한 결합, 명시적 사실(fact), 관찰 가능성.
  - Croft·Iusztin의 멀티 에이전트 사례에서 Domain Event의 위치 — "OrderPlaced", "InventoryReserved", "PaymentAuthorized" 같은 이벤트가 에이전트들을 엮는다.
  - Event Sourcing + CQRS가 에이전트 환경에서 갖는 새 의미 — Vaughn Vernon의 관심. 이벤트 로그가 곧 에이전트 행동의 audit trail.
  - **Vibe Modeling 절 — v2 신설 (~3,000자)** — 2장에서 이전. 코드 생성 전 도메인 이벤트·경계·사용자 흐름을 시각 보드에서 AI와 함께 탐색. "vibe coding의 차량, vibe modeling은 어디로 갈지 아는 것" (vibemodeling.app). Event Storming의 디지털·AI 보조 재발견으로 자리매김. (S6 해결)
  - Event Storming의 AI 보조 가속 — Qlerify, tanstorm, vibemodeling.app. 분산 팀 실시간 협업.
  - Just-in-Time Context와의 결합 — 모든 이벤트 히스토리를 미리 주입하지 않고 lightweight identifier로 두고 런타임에 동적 로드 (Anthropic Engineering).
  - 실행 패턴 — 첫 Event Storming 세션을 AI와 함께 여는 방법, 도메인 이벤트 카탈로그 만들기, 에이전트 통신을 event-driven으로 전환하는 마이그레이션 패턴.
  - 한국 실무 함의 — 도메인 전문가 워크숍의 한국형 운영 (Kerry Kim의 SDR 체험 사례를 확장).
- **독자가 얻는 것:** 한 번도 시도해 보지 못한 Event Storming을 AI 도구의 도움으로 시작할 수 있는 자신감. Vibe modeling이 vibe coding의 위험을 줄이는 사전 단계라는 깨달음. Domain Event 중심 설계로 가는 첫 발.
- **예상 분량:** 약 12,000자 (v1의 10,000자에서 vibe modeling 흡수로 +2,000자)
- **사용할 reference 섹션:** 4.2 Domain Event / 4.3 Vibe Modeling / 1.2 Vibe Modeling / 3.5 거장 시각 (Vernon) / 8.3 도메인 모델링

### 11장. 가상 프로젝트 (후반) — 결제 게이트웨이의 전술과 실행

- **핵심 질문:** 디스커버리·전략이 끝나면 무엇이 실제 코드와 운영으로 흘러야 하는가? Aggregate·invariant·event·schema·휴먼 체크포인트가 어떻게 짜이고, 측정 지표는 어떻게 작동하는가?
- **주요 내용:**

  - 11장 마지막 — **12장 브릿지** — "이 6-Agent 팀이 한 달, 6개월, 1년을 돌아가려면 무엇이 더 필요한가? 보안·환각·비용·관측성·IP 거버넌스의 다섯 기둥을 다음 장에서 다룬다."
- **독자가 얻는 것:** 디스커버리에서 시작한 가상 프로젝트가 코드와 운영으로 완결되는 모습. 측정 지표라는 운영 자산이 거버넌스로 자연스럽게 흘러감.

### 12장. AI 시대의 거버넌스 — 보안·환각·비용·관측성·책임 (v2 신설)

- **핵심 질문:** 멀티 에이전트 시스템이 한 달, 6개월, 1년을 안정적으로 돌아가려면 어떤 거버넌스 다섯 기둥을 세워야 하는가? 보안 취약점 2.74×, 환각, 비용 폭증, 관측성, IP·라이선스 — 이 다섯이 시니어/아키텍트의 일상이 된다.

  - **기둥 2 — 환각 (~2,500자)** — AI 환각의 도메인 모델링적 결과. 7장에서 복선으로 깔린 "잘못된 invariant 제안", "가짜 ULang 후보", "허구의 외부 API 가정", "존재하지 않는 함수 시그니처 호출"이 본격 분석. 환각 탐지 패턴: schema validation, 글로사리 대조, 외부 의존 화이트리스트, 적대적 테스터 에이전트.

### 13장. AI 시대의 팀과 역할 — 한국 조직의 사일로 문제를 정면으로

### 14장. AI를 잘 쓰는 개발자가 된다는 것 — 시니어 정체성과 학습의 역설

- **핵심 질문:** "AI를 가장 잘 활용할 수 있는 개발자는 AI 없이도 코드를 판단할 수 있는 개발자다." 이 역설을 어떻게 받아들이고, 어떻게 자신의 학습 전략에 녹여낼 것인가?
- **주요 내용:**
  - Evan Moon의 진단(2026-04) — "뇌는 편하면 기억하지 않는다", "AI에 의존할수록 코드 판단 청크가 형성되지 않는다", "30분 끙끙대며 짠 코드가 AI가 3초 만에 만든 코드보다 기억에 깊이 남는다."
  - 시니어가 직면하는 정체성 위기 — "내가 시니어인데 주니어보다 빨라야 하나, 더 잘 판단해야 하나, 둘 다인가?"
  - Generation effect — 인지심리학의 고전. 자기가 직접 만든 정보가 외부에서 받은 정보보다 기억과 이해에 강하다.
  - 실천 전략 1 — AI 사용 전 자기 설계안 먼저: 5분이라도 자기 머리로 풀어보고 AI에게 가서 비교.
  - 실천 전략 2 — Core subdomain은 직접 짠다: Generic은 AI에 풀고, 시간을 Core에 더 쓴다.
  - 실천 전략 3 — 의도적 딥 워크 시간 확보: AI 끄고 도메인 책 읽기, 도메인 전문가와 대화, 직접 손으로 모델링.
  - 실천 전략 4 — AI 산출물의 의도적 비평: PR Reviewer 에이전트 외에 자기 자신이 매일 1개는 깊이 본다.
  - 실천 전략 5 — 도메인 깊이 우선의 학습: 새 프레임워크보다 도메인 깊이가 더 오래 간다.
  - Anthropic 2026 Trends의 27% 데이터 재해석 — "AI 없으면 시도조차 안 했을 작업"이 늘어나는 것은 좋은 일이지만, "사람이 깊이 이해해야 하는 작업"은 줄어들지 않는다.
  - **시니어의 자기 점검 질문 5~7개 (v1의 10개에서 축소, N2 해결)** — 체크리스트 인플레이션 경계. "이번 주 내 머리로 푼 문제는 몇 개인가?", "이번 주 도메인 전문가와 대화한 시간은?", "마지막으로 AI 끄고 코드를 짠 때는?" 평어체 회고록 톤으로 풀어 자기계발서 톤에서 분리.
  - Mimul의 12원칙 중 "코드가 명확하고 의도적으로 작성되었는지가 AI의 추론 품질과 직결된다"의 재해석.
- **독자가 얻는 것:** 자기 학습 전략의 재정렬. 시니어 정체성의 새 정의 — 코드 한 줄보다 시스템·도메인·판단·멘토링.

### 15장. 월요일 아침의 적용 전략 — 어디서부터 시작할 것인가

- **핵심 질문:** 책을 덮자마자 우리 팀에서 무엇을 시작할 것인가? 1주, 1개월, 3개월의 구체적 로드맵은? 그리고 grass-field가 아닌 legacy 모놀리스에서는 어떻게 시작하나?
- **주요 내용:**
  - **1주차** — `docs/glossary.md`를 시작한다. 핵심 어휘 30개 등재. 도메인 전문가 1명과 30분 인터뷰.
  - **2주차** — 우리 시스템의 Bounded Context 후보를 종이에 그린다. 현재 모놀리스라도 논리적 경계를 선언.
  - **1개월차** — 한 BC를 골라 첫 에이전트화 시도. 시스템 프롬프트에 글로사리 주입. 측정 지표(에이전트당 토큰, 도메인 로직 비중) 설정.
  - **2개월차** — Aggregate·invariant 점검 워크숍. AI에게 적대적 케이스 생성 요청. 사람이 invariant 명세화.
  - **3개월차** — 휴먼 체크포인트 2개 + 재시도 상한 5회의 거버넌스 안착. PR Reviewer 에이전트 도입. 12장 거버넌스 다섯 기둥의 첫 두 기둥(보안 + 비용) 모니터링 시작.
  - **신설 절: Legacy 마이그레이션 (~1,500자, S1 해결)** — 모놀리스/사일로 코드베이스에서 시작하는 시니어를 위한 단계: ① 글로사리만 먼저 도입(코드는 그대로) ② 한 BC를 강제로 끊어 ACL로 격리 ③ 그 BC에만 첫 에이전트 적용 ④ 6개월에 걸쳐 BC 1개씩 추가. "한 번에 6-Agent로 가지 말라"는 안티패턴 명시.
  - **안티패턴 8가지** — ① 글로사리 없이 에이전트 늘리기 ② 한 에이전트에 여러 BC 떠넘기기 ③ schema 없이 자연어 API ④ aggregate 자동화 위임 ⑤ 코드 짠 에이전트에게 검증 맡기기 ⑥ Core subdomain을 vibe coding으로 ⑦ 보안 리뷰 생략 ⑧ 시니어가 직접 코드 안 짜기.
  - **측정 지표 카드** — 에이전트당 토큰 ~500 목표 / 도메인 로직 비중 ~90% / 글로사리-코드 일관도 / AI 공저 코드 보안 취약점 비율 / METR 19% 격차 자기 점검 / 비용 추이 / 환각 발생 빈도. (12장 거버넌스와 정합)
  - **도구 추천 — "도구가 아니라 역할로" (N4 해결)** — Schema 검증기(Pydantic/JSON Schema류), 글로사리 운영기(텍스트 + CI 검증), Event Storming 보드(Qlerify/vibemodeling.app류), 에이전트 IDE(Claude Code/Cursor/Copilot/Devin류), Eval harness. 도구 이름을 종속변수로 두고 역할을 독립변수로 — 6개월 후 도구가 바뀌어도 책의 권고가 살아남는 구조.
  - 한국 조직의 도입 순서 제안 — PO·기획자 워크숍 → ULang 정렬 → 시니어 1명이 PoC → 팀 전파.
  - 실패해도 괜찮은 영역과 실패하면 큰일나는 영역의 구분 — 사이드 프로젝트 vs Core 도메인.
- **독자가 얻는 것:** 책을 덮자마자 행동에 옮길 수 있는 구체적 단계. 측정 지표로 자기 점검하는 자신감. Legacy 환경에서 grass-field 가정 없이 시작하는 안전한 경로.
- **예상 분량:** 약 11,000자

### 16장. DDD의 진화 시나리오와 우리에게 남은 일

- **핵심 질문:** 5년 후, 10년 후 DDD는 어떤 모습일까? 그리고 그 미래에서 역산해 지금 우리가 무엇을 준비해야 할까?
- **주요 내용:**
  - **시나리오 1 — 흡수** — DDD는 더 이상 별도 방법론으로 불리지 않고 AI 에이전틱 시스템의 기본 골격이 된다. Bounded Context, ULang, Aggregate, Context Map이 도구 자체에 내장. **이 미래에서 시니어가 가장 먼저 잃을 직무 vs 가장 강해질 직무 (S8 해결)** — 잃는 것: 패턴 적용 reviewer / 강해지는 것: 도메인 전문가-에이전트 사이의 통역자.
  - **시나리오 2 — 강화** — 멀티 에이전트 시스템 폭증과 함께 DDD가 핵심 거버넌스 방법론으로 격상. "AI 시대 시니어 = DDD를 깊이 아는 개발자"라는 등식. **잃는 것·강해지는 것**: 잃는 것: 단일 서비스 아키텍트 / 강해지는 것: 멀티 에이전트 토폴로지 설계자.
  - **시나리오 3 — 잔존** — 복잡 도메인에서만 살아남고, 단순 CRUD·자동화·prototype은 SDD/BMAD/vibe coding이 차지. DDD는 enterprise·금융·의료·정부의 방법론으로 좁아진다. **잃는 것·강해지는 것**: 잃는 것: 범용 generalist / 강해지는 것: 복잡 도메인 specialist.
  - 세 시나리오의 공통점 — 어느 쪽이든 도메인 전문성·ULang·BC 식별의 본질은 살아남는다. 형식주의만 다른 옷을 입을 뿐.
  - 책의 미해결 영역 (의도적 한계 인정) — HN raw thread, Reddit deep dive, OKKY 라이브 토론의 직접 인용은 후속 보강. Vaughn Vernon, Alberto Brandolini, Mathias Verraes, Nick Tune의 최신 발언은 더 깊이 파야 한다. 도구별 DDD 친화도 비교 매트릭스, 동아시아(일본·인도) 시각은 후속 책의 몫.
  - **닫는 글 — 15장 1주차 행동 콜백 (S16 해결)** — "DDD는 죽지 않는다. 그 의식은 변하고, 본질은 더 무거워진다. 그리고 그 무게를 어깨에 지는 것은 결국 우리 — 시니어 개발자·테크 리드·아키텍트다. 어휘 30개 등재, 도메인 전문가 30분 인터뷰, BC 종이 스케치 — 이것이 월요일이다. 월요일 아침에 `docs/glossary.md`를 열자."
  - 감사의 말 / 후속 자료 / 참고문헌 안내 / 부록 A 글로사리 템플릿 안내.
- **독자가 얻는 것:** 미래에 대한 확정된 지도는 아니지만, 어떤 미래가 와도 통할 시니어의 자기 정체성과 다음 발걸음.
- **예상 분량:** 약 10,000자
- **사용할 reference 섹션:** 10 리서치 한계 / 3.1~3.5 핵심 관점들 종합 / 1.1 Evans "은탄환 아님"
