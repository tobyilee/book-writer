# 3장. DDD를 다시 펼치자 — 23년의 정전, 무엇이 닳지 않았고 무엇은 색을 입었나

5년 전, 책장 한구석에 꽂아두었던 *DDD Start!* 를 다시 꺼낸다고 상상해보자. 표지 모서리가 닳아 있고, 형광펜으로 그어놓은 자국이 군데군데 보인다. "애그리거트는 트랜잭션 일관성의 경계", "바운디드 컨텍스트마다 모델은 따로", "유비쿼터스 언어를 코드에 그대로 흡수하라" — 이런 문장들에 줄이 그어져 있다. 그때는 분명 마음에 새기고 코드에 옮기려 애썼다. 그런데 지금은? 솔직히 손때 묻은 기억 절반은 흐릿하고, 나머지 절반은 "그때는 그게 무슨 의미였더라" 싶다.

여기서 한 가지 의문이 생긴다. **2003년에 Eric Evans가 정립한 DDD의 핵심 어휘는 23년이 지난 지금 우리에게 무엇을 의미할까?** 어떤 패턴은 시대가 바뀌어도 닳지 않았고, 어떤 패턴은 그 위에 시대적 색이 짙게 입혀졌다. 그리고 어떤 패턴은 이제 거의 의식(ceremony)으로만 남아 있다. 5~9장에서 패턴 하나하나를 정밀하게 다루기 전에, 이 장에서 큰 지도를 먼저 그려보자.

서론을 길게 끌고 싶은 마음은 없다. 시니어 독자가 이미 알고 있는 패턴 카탈로그를 처음부터 다시 설명하는 것만큼 지루한 일도 없다. 그러니 전반부는 표 몇 개로 빠르게 복기하고, 본격적인 이야기는 후반부의 재해석에 할애한다.

## 패턴 카탈로그 — 짧게 복기하자

Evans가 *Domain-Driven Design: Tackling Complexity in the Heart of Software* 에서 정립한 어휘는 크게 두 갈래로 나뉜다. 거시(strategic)와 미시(tactical)다. 한국에서는 최범균의 *DDD Start!* 가 표준 입문서로 자리 잡았고, 인프런의 "도메인 주도 설계 마이크로서비스" 강의도 여전히 인기다. 우리는 이미 한 번씩 이 어휘들을 지나왔다.

| 구분 | 개념 | 핵심 의미 |
|------|------|----------|
| Strategic | **Domain** | 소프트웨어로 해결하려는 문제 영역 |
| Strategic | **Subdomain** | Core / Supporting / Generic 세 갈래로 분할 |
| Strategic | **Bounded Context** | 한 모델·한 공용어가 일관되게 적용되는 경계 |
| Strategic | **Context Map** | BC 간의 관계(Customer-Supplier, Conformist, ACL, Shared Kernel, Published Language)를 명시한 지도 |
| Strategic | **Ubiquitous Language** | 도메인 전문가와 개발자가 함께 쓰는, 한 BC 안에서 일관된 공용어 |
| Tactical | **Entity** | 식별자로 구분되는, 상태와 라이프사이클을 가진 도메인 객체 |
| Tactical | **Value Object** | 식별자 없이 값으로 동등성을 판단하는, 개념적으로 묶인 데이터 |
| Tactical | **Aggregate** | 루트 엔티티를 통해 외부와 소통하고 내부 일관성을 보장하는 도메인 객체 묶음 |
| Tactical | **Repository** | 애그리거트 단위로 영속성을 추상화하는 컬렉션 같은 인터페이스 |
| Tactical | **Factory** | 복잡한 객체 생성 로직을 도메인에서 분리해 캡슐화 |
| Tactical | **Domain Service** | 엔티티·VO에 자연스럽게 속하지 않는 도메인 로직 |
| Tactical | **Domain Event** | 도메인에서 발생한 사실(fact). 다른 컨텍스트가 반응할 수 있도록 발행 |
| Strategic | **Anti-Corruption Layer** | 외부 컨텍스트의 모델이 침투해 내 모델을 오염시키는 것을 막는 번역 계층 |

이쯤 보면 손때 묻은 기억이 천천히 돌아올 것이다. "맞아, 이런 거였지." 다만 한 가지 짚고 넘어가자. 한국어로 옮길 때 번역어가 갈리는 부분이 꽤 있다.

### 한국어 번역어 분기와 이 책의 채택

| 영어 | 한국에서 통용되는 표기 | 이 책의 채택 |
|------|----------------------|-------------|
| Bounded Context | 바운디드 컨텍스트(주류) / 경계 컨텍스트 / 제한된 맥락 | **바운디드 컨텍스트** (필요 시 영문 BC 병기) |
| Aggregate | 애그리거트(주류) / 집합체 / 어그리게이트 | **애그리거트** |
| Ubiquitous Language | 유비쿼터스 언어 / 공용어 / 보편 언어 | **유비쿼터스 언어** (문맥에 따라 "공용어" 병기) |
| Value Object | 밸류 / 값 객체 / VO | **밸류 객체** 또는 **VO** |
| Repository | 리포지터리 / 저장소 | **리포지터리** |
| Anti-Corruption Layer | 부패 방지 계층 / 안티커럽션 레이어 / ACL | **ACL** (한국 실무에서 가장 짧고 통용) |
| Domain Event | 도메인 이벤트 (거의 통일) | **도메인 이벤트** |

번역어 통일을 깐깐히 따지는 책도 있지만, 이 책은 **한국 실무 현장에서 가장 자주 쓰이는 표기를 우선**한다. 회의에서 "바운디드 컨텍스트" 대신 "경계 컨텍스트"라고 말하면 묘하게 학술적인 분위기가 풍기는데, 한 자리에서 같이 일하는 동료들과 자연스럽게 통하는 단어가 중요하기 때문이다.

여기까지가 카탈로그 복기다. 자, 이제 진짜 이야기로 들어가자.

## 닳지 않은 것 — 본질(essence)

23년이 지나도 닳지 않은 것이 있다면 무엇일까? "은탄환이 아니다"라며 Evans가 입버릇처럼 던졌던 단서는 23년 후 묘하게 적중했다. 시대가 바뀌면서 패턴 카탈로그의 표면은 색이 바뀌었지만, 그 안의 어떤 알맹이는 오히려 더 단단해졌다. 다섯 가지만 짚어보자.

### 유비쿼터스 언어 — 오히려 격상되었다

가장 먼저 짚을 것은 유비쿼터스 언어다. 이것은 닳지 않은 정도가 아니라 **시대적으로 격상**되었다. 왜 그럴까?

LLM은 자연어로 일한다. 우리가 "order"라고 적은 한 단어를 보고 LLM이 무엇을 추론할까? 이커머스라면 주문일까, 정렬일까, 명령일까? 동일한 코드베이스 안에서도 한 BC에서는 "Order = 결제 직전의 장바구니 확정 객체"이고, 다른 BC에서는 "Order = 출고 지시서"일 수 있다. 사람이라면 문맥을 읽고 알아서 구분한다. 그런데 LLM은 그 모호함을 인식하지 못한 채 코드를 생성한다. 이 카오스가 증폭되면 어떻게 될까? 정말 끔찍한 일이다.

그래서 AI 시대의 유비쿼터스 언어는 더 이상 "있으면 좋은 것"이 아니다. `docs/glossary.md` 같은 살아있는 용어집이 모든 에이전트가 참조하는 단일 진실이 된다. Paul Iusztin의 6-에이전트 팀에서도 1번 컨텍스트가 글로사리다. 시니어가 가장 먼저 만들어야 할 것이 무엇이냐 물으면, 어느 시대든 답은 같다. 어휘부터다. 5장에서 이 격상의 의미를 깊이 파볼 텐데, 지금은 이것만 기억하자. **유비쿼터스 언어는 닳지 않았다. 오히려 1급 자산으로 진급했다.**

### 바운디드 컨텍스트의 식별 원리

두 번째는 바운디드 컨텍스트의 식별 원리다. 다이어그램으로 그리는 방식이나 도구는 시대에 따라 바뀌지만, **"한 모델·한 공용어가 일관되게 적용되는 경계를 어떻게 찾을 것인가"** 라는 질문 자체는 23년 전이나 지금이나 같다.

오히려 AI 시대에는 이 식별 원리가 더 결정적이 된다. 멀티 에이전트 시스템을 설계해본 사람은 안다. **에이전트 1개를 어디까지 책임지게 할 것인가** — 이 질문은 바운디드 컨텍스트를 어디까지 그을 것인가와 사실상 같은 질문이다. 한 에이전트가 너무 많은 것을 떠안으면 컨텍스트 윈도우가 터지고, 추론이 오락가락한다. 너무 잘게 쪼개면 에이전트끼리 끝없이 통신하느라 비용과 지연이 폭증한다. 결국 도메인의 자연스러운 솔기를 찾아야 하고, 그 솔기를 찾는 안목은 폭포수 시절이나 지금이나 사람의 영역이다.

6장에서 "에이전트 1개 = 바운디드 컨텍스트 1개"라는 사실상의 산업 표준을 본격적으로 다룬다. 미리 보면 이렇다. James Croft가 멀티 에이전트 이커머스 사례에서 보여준 "Inventory Management 에이전트가 모든 재고 상호작용을 캡슐화한다"는 매핑은 우연이 아니다. Nikita Golovko의 Siemens 사례도, Iusztin의 6-에이전트 팀도 동일한 매핑을 채택했다. 같은 결론에 독립적으로 도달했다면, 그 원리는 시대를 견딘 것이다.

### 애그리거트의 invariant 책임

세 번째는 애그리거트의 invariant(불변식) 책임이다. 이 부분은 7장 전체를 할애할 만큼 이 책에서 가장 단단하게 살아남은 패턴이다.

엔티티 정의나 VO 추출은 AI가 거의 다 한다. arXiv 2601.20909는 작은 fine-tuned 모델로도 도메인 메타모델 JSON을 syntactically correct하게 생성할 수 있음을 보였다. 그런데 Eisenreich와 동료들이 2026년에 진행한 DDD 5단계 LLM 자동화 연구(arXiv 2603.26244)는 묘한 결과를 보고했다. **애그리거트 설계 단계만 누적 오류로 실용성이 떨어졌다.** 다른 단계는 어느 정도 작동했는데, 유독 애그리거트만 사람이 다시 손봐야 했다.

왜 그럴까? 잠시 멈추고 생각해보자. 애그리거트의 본질은 무엇인가? 단순히 객체를 묶어놓은 것이 아니다. **"어떤 규칙이 깨지면 도메인이 무너지는가"** 라는 질문에 답하는 작업이다. "주문 총액은 라인 아이템 합과 항상 같아야 한다", "재고가 0인 상태에서 출고 지시를 내릴 수 없다", "결제 승인이 떨어지지 않은 주문은 출고 단계로 넘어가지 못한다" — 이런 invariant는 도메인 전문가의 머릿속에 있다. 코드 어디에도 명시되지 않은 채로. AI가 코드의 표면적 패턴은 흉내 낼 수 있지만, 이 무엇이 깨지면 안 되는가의 판단은 여전히 사람의 영역이다.

그래서 애그리거트의 경계와 invariant 정의는 "사람의 마지막 영역"이라 부를 만하다. 7장에서 이 표현을 책의 굵직한 입장으로 다시 꺼낼 것이다.

### Core / Supporting / Generic 분류

네 번째는 서브도메인을 Core·Supporting·Generic으로 나누는 분류다. 이것도 23년 동안 표면적으로는 변하지 않았다. 그런데 AI 시대에 이 분류의 **시간 재배분** 가치는 폭발적으로 커졌다.

Generic 서브도메인이 무엇인가? 인증, 로깅, 표준 CRUD, 알림 발송 같은 것이다. 시장에 이미 잘 만들어진 라이브러리·SaaS가 널려 있고, AI에게 맡겨도 큰 문제가 없는 영역이다. Supporting은 비즈니스에 필요하지만 차별화 요인은 아닌 영역. Core는 우리 회사를 우리 회사답게 만드는, 경쟁사가 흉내 내지 못하는 영역이다.

Martin Fowler와 Daniel Westheide 모두 같은 결론에 도달한다. **"Generic은 점점 더 AI에게 위임 가능해진다. 그러나 Core는 오히려 사람의 판단·도메인 이해 비중이 늘어난다."** 시니어의 시간을 어디에 쓸 것인가의 답이 명확해진 셈이다. 23년 전에는 "Core에 인력을 집중하라"는 권유였다면, 지금은 "Generic은 에이전트에게 넘기고 Core에 사람 시간을 재배분하라"는 작전 명령에 가깝다. 분류 자체는 그대로지만, 그 활용 가치는 시대와 함께 격상되었다.

### 도메인 이벤트의 fact-as-message

다섯 번째는 도메인 이벤트의 "사실로서의 메시지(fact-as-message)" 의미다.

도메인 이벤트의 정의를 다시 떠올려보자. **도메인에서 발생한 사실. 다른 컨텍스트가 반응할 수 있도록 발행된다.** 23년 전에는 주로 단일 프로세스 안의 옵저버 패턴이거나, 메시지 큐를 통한 마이크로서비스 간 통신이었다. 그런데 멀티 에이전트 환경으로 무대를 옮기면? 도메인 이벤트는 **에이전트 간 통신의 기본 단위**가 된다. Croft와 Iusztin이 입을 모으는 지점이다.

"OrderPlaced"라는 사실이 발생하면, 재고 에이전트는 재고 차감을 시작하고, 결제 에이전트는 결제 시퀀스를 가동하며, 알림 에이전트는 고객에게 이메일을 보낸다. 이 모든 일이 직접 호출이 아니라 사실의 발행과 구독으로 일어난다. 이는 도메인 이벤트의 본질이 변한 것이 아니라, **본질이 새 무대에서 더 빛난 것**이다. 9장에서 Event Storming의 부활과 함께 깊이 다룬다.

자, 다섯 가지가 닳지 않은 본질이다. 이제 시대적 색이 입혀진 패턴들로 넘어가보자.

## 색이 입혀진 것 — 형식(form)

본질은 살아남았지만, 형식은 시대와 함께 옷을 갈아입은 패턴들이 있다. 이 그룹이 가장 흥미롭다. 23년 전의 우리가 알던 것과 표면적으로는 같은 이름인데, 실제로 일하는 방식이 달라진 것들이다. 네 가지를 보자.

### 컨텍스트 맵 — 다이어그램에서 실행 가능한 코드로

23년 전 컨텍스트 맵은 화이트보드나 Visio 위의 다이어그램이었다. 사각형 몇 개와 그 사이를 잇는 선, "Customer-Supplier"·"Conformist"·"ACL" 같은 라벨. 회의실 벽에 붙여놓고 분기별로 한 번씩 갱신했다. 그런데 솔직히 말하자. 그 다이어그램이 실제 코드와 동기화된 적이 얼마나 있었는가? 다이어그램은 다이어그램대로, 코드는 코드대로 떠돌다가, 1년 후에 보면 둘은 완전히 다른 시스템을 묘사하고 있었다. 찜찜한 일이었다.

AI 시대에는 컨텍스트 맵이 **실행 가능한 코드(config + adapter + schema)** 로 격상된다. Golovko의 표현을 빌리면 이렇다. upstream과 downstream, contract version, adapter 구현체를 명시적으로 기술하고, CI가 매 푸시마다 검증한다. 멀티 에이전트 시스템의 토폴로지(chain·star·mesh·workflow graph)가 컨텍스트 맵 패턴과 자연 매핑된다(arXiv 2601.12560). 다이어그램이 죽고 코드가 산다.

이게 무슨 의미일까? **본질(BC 간 관계를 명시적으로 다스린다)은 그대로지만, 형식(다이어그램)은 폐기되고 실행 가능 코드로 대체되었다.** 컨텍스트 맵이라는 이름은 살아남았지만, 우리가 그것을 다루는 방식은 완전히 달라졌다. 8장에서 Schema-as-Contract 패턴과 함께 본격 다룬다.

### ACL — 데이터 변환에서 semantic firewall로

ACL도 색이 짙게 입혀진 패턴이다. 23년 전 ACL은 외부 시스템의 데이터 모델이 우리 도메인을 오염시키지 않도록 막는 번역 계층이었다. 결제 게이트웨이 응답이 우리 주문 모델을 점령하지 않도록 사이에 어댑터를 두는 식이었다. 본질은 "외부 모델로부터 내 모델을 보호하기"다.

AI 시대에는 이 본질이 더 깊은 의미를 얻는다. 멀티 에이전트 환경에서는 같은 단어가 에이전트마다 다른 의미를 갖는다. Golovko의 예시를 보자. "function"이라는 단어가 코드 생성 에이전트에게는 "signature + implementation"을 의미하지만, 테스팅 에이전트에게는 "behavior contract"를 의미한다. 같은 단어, 다른 세계. 이 사이에 무엇이 필요한가? **semantic firewall** 이다. 의미를 번역하는 계층 말이다.

ACL이라는 패턴 이름은 살아남았다. 그런데 다루는 대상이 "데이터 형식 변환"에서 "의미 변환"으로 옮겨갔다. 이 변화는 작은 색칠이 아니라, 패턴의 작동 무대 자체가 바뀐 것에 가깝다. 8장 후반부의 주제다.

### 리포지터리 / 팩토리 — 수작업에서 AI 위임으로

리포지터리와 팩토리는 가장 일관되게 형식이 바뀐 패턴들이다. 23년 전, 시니어 개발자는 도메인 모델에 맞춰 리포지터리 인터페이스를 설계하고, JPA·MyBatis·MongoDB 매핑을 한 줄 한 줄 손으로 짰다. 팩토리도 마찬가지였다. 복잡한 객체 생성 로직을 캡슐화하는 것이 시니어의 자존심이었다.

AI 시대에는? AI-DLC 한국어 백서는 AI가 자율적으로 Repository와 Factory를 생성하는 워크플로를 명시한다. 도메인 메타모델이 정의되면, 그에 맞는 영속성 어댑터와 생성 로직은 사실상 자동으로 따라온다. 시니어가 손으로 짤 필요가 점점 사라진다. 그렇다고 패턴 자체가 사라지는가? 아니다. **Hexagonal 관점에서 리포지터리는 secondary port의 한 구현이고, 도메인 계층이 외부에 의존하지 않도록 경계를 유지하는 책임은 여전히 사람에게 있다**(Khosravi). 본질은 살아남고, 손가락 노동은 AI가 가져간다.

이 변화에 대한 시니어의 감정이 묘하다. 한편으로는 시원하다. 그 지루한 매핑 코드를 안 짜도 된다니. 다른 한편으로는 찜찜하다. "내가 잘 짜던 그것을 AI가 한다고? 그럼 나는 뭘 하지?" 이 정체성 질문에 대한 답은 14장에서 다룬다. 미리 한 줄만 적자면, **본질을 정의하는 사람과 형식을 채우는 사람의 분업이 새로 그어진다**.

### 폴더 구조와 계층 카탈로그 — 형식주의의 풍화

마지막은 폴더 구조와 계층 카탈로그다. 솔직히 인정하자. DDD 책 한 권 읽고 가장 먼저 한 일이 무엇이었던가? `domain/entity`, `domain/vo`, `domain/service`, `application/usecase`, `infrastructure/repository` — 이런 폴더를 만든 일 아니었던가. 그리고 그 폴더 안에 적당히 클래스를 채워 넣고 "DDD 적용했다"고 말해버린 경험.

Daniel Westheide가 직설적으로 짚었다. "BMAD wouldn't save you" — 무슨 방법론이든 형식만 따른다고 살아나지 않는다. **폴더 구조·패턴 카탈로그 같은 형식주의는 가치가 빠르게 떨어진다.** 도메인 전문성·유비쿼터스 언어·BC 식별 같은 본질은 더 결정적이 되는데 말이다. AI 시대에 이 차이는 더 또렷해진다. AI는 폴더 구조를 흉내 내는 코드를 분 단위로 생산할 수 있지만, 그 안에 정말 도메인이 들어 있는지는 별개 문제다.

폴더 구조 자체가 나쁘다는 말은 아니다. 다만, 폴더가 곧 DDD라는 착각은 시대와 함께 풍화되어야 한다. Iusztin의 말을 빌리면 더 명확하다. **"코드를 파일 타입이 아니라 actionability(bounded context) 기준으로 조직하라."** 폴더의 이름이 `entity`·`service`·`repository`가 아니라, `payment`·`inventory`·`shipping`이 되어야 한다는 것이다. 형식이 본질을 호명하도록.

## 거의 닳은 것 — 의식(ceremony)

마지막 분류는 거의 닳은 것들이다. 이 그룹은 패턴 자체가 사라졌다기보다는, **그 패턴을 둘러싼 의식적 절차의 무게가 시대와 어울리지 않게 된 것들**이다. 무용론자들이 가장 강하게 공격하는 지점이기도 하다.

### 폭포수형 모델링 워크숍의 무게

DDD 워크숍이라 하면 어떤 장면이 떠오르는가? 도메인 전문가와 개발자들이 회의실에 모여 며칠 동안 화이트보드 앞에서 포스트잇을 붙이는 풍경. Event Storming이라는 이름으로 6시간짜리 세션을 한 번에 진행하는 경우도 있다. 한 번에 며칠을 빼는 게 가능한 조직도 있지만, 솔직히 한국 조직에서 그 시간을 확보하는 것이 얼마나 난감한 일인지 우리는 잘 안다.

이 무거운 의식이 AI 시대 속도와 충돌한다. 솔로 expert가 BMAD나 SDD를 옹호하는 이유 중 하나가 이것이다. "DDD 워크숍 한 번 할 시간에 나는 프로토타입 세 개 만든다." 이 주장의 정직한 부분은 인정하자. 일정 압박이 강하고 도메인 전문가가 회의에 잘 나오지도 못하는 조직에서, 폭포수형 모델링 의식은 실제로 비용이 너무 크다.

물론 그렇다고 모델링 자체가 사라지지는 않는다. 다만 형식이 바뀐다. 9장에서 다룰 **Vibe Modeling** — 코드 생성 전에 도메인 이벤트·경계·사용자 흐름을 AI와 함께 시각 보드에서 탐색하는 방식 — 이 그 대안이다. 워크숍의 본질(같이 모델을 그려보기)은 살리되, 의식의 무게는 덜어내는 시도다.

### 거대 문서화

두 번째 의식은 거대 문서화다. 컨텍스트 맵 다이어그램 100페이지짜리, 도메인 모델 사전 200페이지짜리, 아키텍처 결정 기록(ADR) 수백 건 — 이런 문서를 본 적이 있다면, 그것이 6개월 후에 얼마나 빨리 거짓말이 되는지도 알 것이다. 거대 문서는 작성된 순간부터 코드와 멀어지기 시작한다.

AI 시대의 답은 다시 한 번 "실행 가능한 형태"다. `docs/glossary.md`처럼 가볍게 살아 있는 용어집. CI에서 검증되는 schema. 코드와 함께 진화하는 ADR. 200페이지짜리 PDF가 아니라, 200줄짜리 마크다운이 더 강력하다. Mimul의 "언어 독립적 도메인 중심 코딩 12원칙"에서 한 줄을 빌리자. **"AI가 코드를 생성하고 수정하는 빈도가 높아질수록 코드가 얼마나 명확하고 의도적으로 작성되었는지가 AI의 추론 품질과 직결된다."** 거대 문서가 아니라 명확한 코드와 가벼운 글로사리. 이것이 새 시대의 형태다.

### 패턴 카탈로그 추종

세 번째이자 가장 미묘한 의식은 패턴 카탈로그 추종이다. "DDD를 한다"는 말이 "엔티티·VO·애그리거트·리포지터리·팩토리·도메인 서비스를 다 갖춰서 코드를 짠다"는 의미로 변질된 경우가 얼마나 많은가. 도메인이 단순한 CRUD인데도 패턴을 다 갖춰 넣어 코드가 비대해지면, 그것은 DDD가 아니라 DDD의 흉내다.

Hacker News의 "DDD Is Overrated" 류 떡밥이 매번 다시 떠오르는 이유가 여기에 있다. 단순 CRUD에 DDD를 강요한 시니어들의 실패담이 누적되어 있다. 3년 미만 시스템에 패턴 카탈로그를 다 갖춰 넣는 것은 ROI가 안 나온다는 솔로 창업자들의 자기 변호도 정직한 근거가 있다. 이 무용론의 강한 논거를 4장에서 본격적으로 받아 풀 텐데, 미리 한 가지만 짚자. **패턴 카탈로그를 그대로 따르는 의식은 시대와 함께 닳았다.** 단, 그 안의 본질(도메인 전문성·BC 식별·invariant 책임)은 닳지 않았다. 의식과 본질을 분리하는 안목이 시니어의 새 무기다.

## 한눈에 보는 분류표

지금까지 본 내용을 한 표로 압축해두자. 5~9장에서 패턴별로 깊이 들어갈 때 다시 돌아와 펼칠 지도다.

| 분류 | 패턴 | 본질 | 시대적 변화 | 본격 다룰 장 |
|------|------|------|------------|-------------|
| **닳지 않은 것** | Ubiquitous Language | 도메인 전문가와 개발자의 공용어 | 격상 — 1급 자산 | 5장 |
| **닳지 않은 것** | Bounded Context 식별 | 한 모델·한 공용어의 경계 찾기 | Agent 1개 = BC 1개로 매핑 | 6장 |
| **닳지 않은 것** | Aggregate invariant | 깨지면 안 되는 도메인 규칙 책임 | 사람의 마지막 영역 | 7장 |
| **닳지 않은 것** | Core/Supporting/Generic | 도메인의 가치 분류 | 시간 재배분 작전 명령 | 5·7·15장 |
| **닳지 않은 것** | Domain Event (fact-as-message) | 도메인에서 발생한 사실 | 에이전트 간 통신의 기본 단위 | 9장 |
| **색이 입혀진 것** | Context Map | BC 간 관계의 명시 | 다이어그램 → 실행 가능 코드 | 8장 |
| **색이 입혀진 것** | ACL | 외부로부터 내 모델 보호 | 데이터 변환 → semantic firewall | 8장 |
| **색이 입혀진 것** | Repository / Factory | 영속성·생성의 추상화 | 수작업 → AI 위임 | 7·8장 |
| **색이 입혀진 것** | 폴더 구조 | 코드 조직의 형식 | 파일 타입 → BC 기준 | 6장 |
| **거의 닳은 것** | 폭포수 모델링 워크숍 | 도메인 같이 그리기 | 의식 무게 풍화, Vibe Modeling으로 대체 | 9장 |
| **거의 닳은 것** | 거대 문서화 | 모델·결정의 기록 | 실행 가능한 가벼운 형태로 | 5·8장 |
| **거의 닳은 것** | 패턴 카탈로그 추종 | 카탈로그 그대로 적용 | 본질과 의식 분리하는 안목 필요 | 4장(무용론) |

이 표가 5~9장의 큰 지도다. 한 번 마음에 새겨두고, 각 장에서 깊이 들어갈 때 펼쳐보자.

## Evans의 "은탄환 아님" 단서, 23년 후의 적중

여기서 한 가지 짚고 가자. Evans 본인이 *Domain-Driven Design* 출간 후로도 "DDD는 은탄환이 아니다"라고 끊임없이 강조해왔다. 2024년 InfoQ 인터뷰에서도 같은 말을 반복했다. 이 단서가 지금 어떻게 적중했는가?

당시에는 "DDD를 모든 프로젝트에 적용하지 말라"는 정도의 의미였을 것이다. 그런데 23년 후, 이 단서는 더 정밀하게 적중한다. **DDD의 어떤 부분은 닳지 않고 더 강해졌고, 어떤 부분은 색이 입혀져 다른 모양이 되었으며, 어떤 부분은 의식으로만 남아 풍화되었다.** Evans가 예견했든 안 했든, 그의 패턴 카탈로그는 처음부터 "은탄환이 아닌 것"이었고, 그래서 시대와 함께 분류·재해석·재배치되는 운명을 견딜 수 있었다.

이 책의 핵심 질문 — "AI 시대에 DDD는 여전히 유효한가?" — 도 그래서 더 정밀하게 다시 물어야 한다. **"어디까지 유효하고, 어디는 변하며, 어디는 사라지는가?"** 4장에서 네 진영(강화론·변형론·축소론·흡수론)의 격돌을 정리할 때 이 질문이 다시 나온다.

## 마무리 — 5~9장으로 가는 다리

자, 큰 지도가 그려졌다. 닳지 않은 본질 다섯, 색이 입혀진 형식 넷, 거의 닳은 의식 셋. 이 분류표가 앞으로 다섯 장(5~9장)의 길잡이가 될 것이다.

다음 장(4장)에서는 잠시 패턴 분석에서 한 발 물러나, **DDD의 운명에 대한 네 진영의 격돌**을 들여다본다. 강화론·변형론·축소론·흡수론 — 각각의 논리적 뼈대가 무엇이고, 무용론의 강한 논거를 정직하게 인정한 후 우리는 어디로 응답할 것인가. 무용론자의 입장을 책이 가장 강한 형태로 대변하는 시뮬레이션도 들어 있다. 솔직한 책이 되고 싶기 때문이다.

5장부터 9장까지는 이 지도를 들고 패턴 하나하나의 정밀 분석으로 들어간다. 5장 유비쿼터스 언어의 격상, 6장 바운디드 컨텍스트와 에이전트의 매핑, 7장 애그리거트와 사람의 마지막 영역, 8장 컨텍스트 맵과 ACL의 새로운 의미, 9장 도메인 이벤트와 멀티 에이전트 통신.

마지막으로 한 가지 당부. 이 분류표를 너무 단단하게 쥐지 말자. 어떤 패턴이 어느 분류에 속하는지는 시간과 맥락에 따라 다시 미끄러진다. 우리가 정말 길러야 할 것은 분류표 그 자체가 아니라, **본질과 형식과 의식을 분리해서 보는 안목**이다. 그 안목이 있다면 23년 후 또 어떤 시대가 와도 닳지 않을 것이다. 기억해두자.
