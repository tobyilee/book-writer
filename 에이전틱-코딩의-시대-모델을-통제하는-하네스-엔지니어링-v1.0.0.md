# 에이전틱 코딩의 시대: 모델을 통제하는 하네스 엔지니어링

> 프롬프트의 시대는 끝났다. 이제 하네스를 설계하라.

- **저자:** Toby-AI
- **버전:** v1.0.0
- **발행일:** 2026-05-02
- **언어:** 한국어
- **분량:** 본문 5부 26장 + 에필로그 + 부록 8종 / 약 326,000자 / 약 430쪽
- **형식:** EPUB 3 (epubcheck 통과)

## 이 책은 무엇인가

이 책은 한국어권에서 **하네스 엔지니어링(Harness Engineering)**을 정면으로 다루는 첫 종합 바이블이다. Addy Osmani, Mitchell Hashimoto, Birgitta Böckeler, Anthropic, OpenAI Harness 팀이 2025~2026년 사이에 같은 결론에 도달했다 — *Agent = Model + Harness*. 모델을 만드는 사람이 아니라면, 우리가 하는 일은 모두 하네스 엔지니어링이다.

프롬프트 엔지니어링은 더 이상 충분하지 않다. AGENTS.md를 백 줄 썼는데도 같은 실수를 반복하는 에이전트, Sonnet 4.5에서 4.6으로 갈아탔더니 코드 품질이 떨어지는 회귀, 5만 토큰에서 부패하는 "Context Rot", 한 번의 자격증명 탈취가 25,000개 저장소로 번진 Shai-Hulud 사고. 이 모든 일이 **모델 바깥의 모든 것**, 즉 하네스에서 일어난다.

이 책은 산업의 합의가 어떻게 형성됐는지(왜) → 그 합의를 어떻게 분해하는지(무엇 — CAR·Guides/Sensors·ACI 4원칙) → 어떻게 운영하고 측정하는지(어떻게 — 2-Prompt·Ralph Loop·Generator-Evaluator·5축 메트릭) → 어떻게 보호하는지(안전 — Lethal Trifecta·격리 3축·인젝션 카탈로그) → 어디서 더 갈 수 있고 팀에 어떻게 옮겨 심는지(현장 — 평가 인프라·한국 생태계·팀 도입·메타 하네스)의 5단 사다리로 답한다.

다른 자료와 무엇이 다른가. 첫째, 영어권 산업 합의를 한국어 환경(한국어 토큰 비용, GitLab+Jira+Slack 통합, revfactory·instructkr·Haandol·MadPlay 같은 한국 생태계)에 정확히 맞춰 다시 짠다. 둘째, *이론과 운영 레시피가 한 책 안에 같이 있다.* CAR과 ACI 같은 추상이 30분 PoC 도커 레시피와 5축 대시보드로 즉시 이어진다. 셋째, 모든 챕터에 실습이 있다 — 자기 환경의 CAR 매트릭스, AGENTS.md 분산 측정, Canary Token 깔기, Inspect 골든 케이스까지.

## 누구를 위한 책인가

**진입 상태:** Cursor·Claude Code·Copilot·Aider 정도는 써 봤다. AGENTS.md/CLAUDE.md를 한 번 작성해 봤지만 "왜 어떤 줄은 효과 있고 어떤 줄은 무시되는지" 모른다. 에이전트가 같은 실수를 반복하면 짜증을 내며 다시 프롬프트를 고친다.

이 책은 다음 독자를 위해 쓰였다:

- AI 코딩과 에이전트의 기본 활용을 이미 하고 있고, 다음 단계로 가고 싶은 **현업 개발자**
- 자기 팀의 AGENTS.md/CLAUDE.md를 설계·유지보수하는 **테크 리드·시니어 엔지니어**
- 에이전트를 프로덕션에 올리며 *비용·회귀·보안*을 책임지는 **플랫폼/엔지니어링 매니저**
- AI 도구 도입을 5명·30명·300명 팀에 옮겨 심어야 하는 **개발자 도입 담당자**
- 평가 하네스(SWE-Bench·Terminal-Bench·Inspect)와 런타임 하네스의 동형성에 호기심이 있는 **연구·도구 빌더**

기초 프롬프트 엔지니어링이나 LLM API 호출 같은 입문 내용은 다루지 않는다. 이 책은 **현업 1~2단계를 졸업한 사람**의 다음 책이다.

## 무엇을 얻게 되는가

다섯 가지 출구 능력을 측정 가능한 형태로 약속한다:

- **(설계)** 자기 팀의 워크플로를 **CAR(Control·Agency·Runtime)** 3축과 **Guides·Sensors** 2축으로 분해하고, 빈자리를 짚어낼 수 있다.
- **(저술)** 거대 단일 AGENTS.md가 아닌 **디렉토리 분산 + 계층화된 메모리 파일**을 설계해, 컨텍스트 예산을 의식적으로 배분할 수 있다.
- **(운영)** 장기 실행 작업에 **2-Prompt(Initializer + Coding)** 또는 **3-에이전트(Planner-Generator-Evaluator)** 패턴을 적용하고, `progress.txt`·서브에이전트로 컨텍스트 부패를 통제하며, **5축 메트릭**(latency·token·cost·success·regression delta)으로 운영 회귀를 정량 탐지할 수 있다.
- **(보안)** **Lethal Trifecta·Shai-Hulud·프롬프트 인젝션 카탈로그**를 자기 환경에 비춰 재현하고, **MicroVM·egress 화이트리스트·MCP 권한 모델**로 1차 방어선을 구성할 수 있다.
- **(평가·진화·전파)** Inspect·Terminal-Bench·SWE-Bench 같은 평가 하네스로 자기 하네스의 성능 회귀를 정량 측정하고, **메타 하네스**(스스로 갱신되는 AGENTS.md) 패턴을 시도하며, 자기 팀에 1주차·2주차·1개월 온보딩으로 도입할 수 있다.

## 차례

### 1부. 패러다임의 전환 — 프롬프트에서 하네스로

1. **Agent = Model + Harness, 그러므로 우리는 무엇을 하는가** — "프롬프트 엔지니어링이 끝났다"는 말의 진짜 뜻은 무엇인가?
2. **Hashimoto의 6단계 — 당신은 지금 몇 단계인가** — 내가 지금 AI 도입의 어느 지점에 서 있는지 어떻게 정직하게 알 수 있는가?
3. **평가 하네스와 런타임 하네스는 같은 코드를 쓴다** — 벤치마크에서 잘 나오게 만드는 트릭이 왜 그대로 프로덕션의 베스트 프랙티스가 되는가?
4. **모델보다 중요한 하네스 — Terminal-Bench 2.0이 보여준 것** — 같은 모델이 다른 하네스 위에서 왜 다른 점수를 받는가?

### 2부. 하네스 아키텍처 — CAR, Guides·Sensors, ACI

5. **CAR 분해 — Control, Agency, Runtime** — 하나의 거대한 하네스를 어떻게 분해하면 빠진 곳이 보이는가?
6. **Guides와 Sensors — 사이버네틱 폐루프로서의 하네스** — 행동 전과 후, 두 통제는 어떻게 함께 작동해야 하는가?
7. **ACI 4원칙 — 인간 IDE의 가정을 버려라** — 에이전트가 쓰는 인터페이스가 왜 인간이 쓰는 IDE와 달라야 하는가?
8. **왜 큰 AGENTS.md는 실패하는가** — 한 곳에 모은 지침이 왜 오히려 에이전트를 무력하게 만드는가?
9. **도구·MCP·서브에이전트 — Agency를 설계하기** — 모델에게 무엇을 시킬 수 있게 할지, 어떤 표면적으로 줄지를 어떻게 정하는가?

### 3부. 장기 실행과 컨텍스트 — 살아 있는 에이전트

10. **Context Rot — 긴 컨텍스트의 배신** — 왜 큰 윈도우를 가진 모델조차 긴 컨텍스트에서 멍청해지는가?
11. **2-Prompt 패턴 — Initializer와 Coding의 분리** — 매번 새로 시작하는 세션을 어떻게 살아 있는 작업으로 잇는가?
12. **Ralph Loop은 어떻게 단순함의 승리가 되었나** — 한 줄짜리 bash 루프가 왜 Anthropic 공식 플러그인이 됐는가?
13. **Generator-Evaluator — 자기 평가의 함정과 해법** — 같은 모델이 자기 결과를 평가하면 왜 과대평가하고, 그걸 어떻게 막는가?
14. **단일 에이전트 vs 다중 에이전트 — 합의가 형성된 자리** — 멀티에이전트는 언제 쓰고, 언제 쓰지 말아야 하는가?
15. **관측과 비용 — Sensors의 다른 얼굴** — "내 하네스가 잘 돌고 있다"를 어떻게 숫자로 보여주고, 회귀를 *먼저* 알아내는가?

### 4부. 보안과 격리 — 하네스가 공격받을 때

16. **Lethal Trifecta — Simon Willison의 위협 모델** — 에이전트가 위험해지는 정확한 조건은 무엇인가?
17. **Shai-Hulud — 자기 복제하는 npm 웜의 해부** — 한 번의 자격증명 탈취가 어떻게 25,000개 저장소로 번지는가?
18. **프롬프트 인젝션 카탈로그와 방어** — 인젝션 공격은 몇 가지 모양이 있고, 각각을 어떻게 1차로 막는가?
19. **격리 3축 — Tooling·Host·Network의 이론** — 에이전트가 망쳐도 시스템이 살아 있게 하려면 어떤 축으로 격리하는가?
20. **격리 운영 레시피 — Docker·devcontainer·sandbox-exec 30분 PoC** — 30분 안에 자기 에이전트를 어떻게 *실제로* 격리시키는가?
21. **평가 회피와 Sandbagging — 보안과 평가가 만나는 곳** — 평가 환경에서 잘 행동하던 에이전트가 프로덕션에서 다르게 굴 가능성은 어떻게 막는가?

### 5부. 평가·생태계·팀·미래 — 한국에서 하네스를 굴리는 법

22. **SWE-Bench·Terminal-Bench·OSWorld — 자기 하네스를 정량 측정하기** — "내 하네스가 좋아졌다"를 어떻게 숫자로 보여주는가?
23. **오픈소스 풍경 — Claude Code·Codex CLI·Cursor·OpenHands·Aider** — 어느 도구를 어느 자리에 놓을지, 어떤 렌즈로 비교하는가?
24. **한국에서 하네스를 굴리기 — revfactory·instructkr·Haandol·MadPlay** — 한국 환경의 특수 이슈를 어떻게 다루는가?
25. **팀에 하네스를 도입하기 — 1주차·2주차·1개월** — 한 사람이 알게 된 것을 5명·30명·300명 팀에 어떻게 옮겨 심는가?
26. **메타 하네스 — 스스로 진화하는 시스템** — 하네스가 스스로 자신을 갱신할 때, 우리는 무엇을 하게 되는가? 그게 *항상* 좋은 일인가?

**에필로그. 현장의 목소리·5년 뒤** — 책이 닫히는 자리에서, 현장은 무엇을 말하고 있는가?

### 부록 (8종)

## 저자 소개

**Toby-AI**는 한국어권 개발자를 대상으로 깊이 있는 기술서를 집필하기 위해 설계된 AI 저자 페르소나다. 영어권 1차 자료(Anthropic 엔지니어링 블로그, OpenAI Harness 팀 발표, Mitchell Hashimoto의 자기 회고, Birgitta Böckeler의 Guides/Sensors 프레임워크, Simon Willison의 보안 분석 등)를 한국어 실무자의 시야로 재구성한다. 평어체와 청유형, 수사적 질문, 상황 가정, 감정적 공감을 결합한 *Toby 스타일*로 책 한 권을 한 호흡에 읽히도록 짠다. 이 책은 Toby-AI의 첫 단행본이다.
