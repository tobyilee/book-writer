# 1장. 컴퓨터는 왜 이런 모습인가

아침에 출근해서 IDE를 열고, 평소처럼 Kotlin 프로젝트를 빌드한다고 상상해보자. `./gradlew build`를 입력하는 순간, JVM이 떠오르고, 그 위에 우리 코드가 얹히고, 그 아래 어딘가에서 CPU가 명령어를 하나씩 꺼내 실행한다. 그런데 한 번쯤 이런 질문을 던져본 적이 있을 것이다. 그 CPU는 도대체 왜 그런 식으로 동작할까? 왜 코드와 데이터가 같은 메모리에 섞여 있을까? 왜 8비트, 16비트, 32비트, 64비트라는 단위로 자라 왔을까? 왜 우리가 짠 함수 호출이 결국 `CALL`이라는 네 글자짜리 어셈블리어로 환원될 수 있는가?

이 질문들을 한 번이라도 진지하게 들여다보면 우리는 곧 어떤 막막함과 마주친다. 컴퓨터 구조 교과서는 무겁고, 게이트와 트랜지스터 다이어그램은 친절하지 않으며, 4년 전 학부 수업에서 들은 "von Neumann 아키텍처"라는 단어는 시험 직후 기억의 서랍 어딘가에 잠들어 있다. 다시 꺼내 보려고 하면 어딘지 모르게 찜찜하다. 알 것 같으면서 정확히 설명하지는 못하는, 그런 어정쩡한 상태.

그 어정쩡함을 풀어내는 길은 여러 가지가 있다. 슬라이드와 시험 문제로 푸는 길도 있고, **우리 손으로 직접 Kotlin 코드를 짜서 8비트 CPU를 짓는** 길도 있다. 우리는 후자를 택한다. 우리가 짠 어셈블러로, 우리가 짠 프로그램을, 우리가 짠 CPU 위에서 한 번 돌려보자. 만들어 봐야 안다. 만들지 않은 것은 진짜로 안다고 말하기 어렵다.

본격적으로 첫 줄의 Kotlin 코드를 쓰기 전에, 잠깐 멈춰서 한 가지를 함께 살펴보자. 우리가 앞으로 15주 동안 만들 그 CPU의 모습은 도대체 어디서 온 것일까? 왜 명령어와 데이터가 같은 메모리에 머무는가? 왜 8비트로 시작하는가? 이 장을 다 읽고 나면, 다음 장에서 만날 30줄짜리 Kotlin 미니 CPU의 골격이 머릿속에 미리 그려질 것이다. 거기까지 함께 가 보자.

## 1.1 1945년의 한 보고서, 그리고 오늘의 CPU

1945년 6월, 펜실베이니아 대학 무어 공과대학에서 한 편의 보고서가 세상에 나왔다. 제목은 *First Draft of a Report on the EDVAC*. 표지에는 단 한 사람의 이름이 적혀 있었다 — John von Neumann. 헝가리에서 태어나 미국으로 건너간 수학자. 이 100여 페이지짜리 미완성 원고가 그 이후 80년 동안 만들어진 거의 모든 일반 컴퓨터의 골격을 정한다. 우리 노트북도, 서버도, 휴대전화도, 우리가 곧 만들 SAP-1도 — 모두 이 보고서의 직계 후손이다.

이 보고서가 정한 골격은 의외로 단순하다. 컴퓨터는 세 덩어리로 이루어져 있다. **CPU**(연산과 제어), **메모리**(데이터와 명령어를 함께 담는 통합 저장소), 그리고 **I/O**(바깥 세상과 주고받는 통로). 이 셋이 버스로 묶여서 한 박자, 한 박자 함께 움직인다. 우리는 이것을 "von Neumann 아키텍처"라고 부른다.

이 모델의 진짜 핵심은 어디에 있을까? 세 덩어리로 나뉜다는 사실 자체가 아니다. **명령어와 데이터가 같은 메모리에 머문다**는 한 줄에 핵심이 응축돼 있다. von Neumann의 원문은 이렇게 적고 있다.

> "A uniform memory containing both numbers (data) and orders (instructions)."
>
> — von Neumann, *First Draft of a Report on the EDVAC*, 1945

명령어를 숫자로 적고, 그 숫자를 데이터와 같은 메모리 칸에 둔다. 한 번 들으면 너무 당연해 보이는 이 문장이 왜 그렇게 혁명적이었을까? 한번 그 반대를 상상해보자. 만약 명령어와 데이터가 분리된 저장소에 살고 있다면 어떨까? 명령어는 회로에 미리 박혀 있는, 변경 불가능한 무엇이라면? 실제로 ENIAC이 그런 구조였다. ENIAC에서 새 프로그램을 돌리려면 수십 명의 작업자가 며칠 동안 케이블을 다시 꽂고 스위치를 다시 맞춰야 했다. 그것은 "프로그램을 짠다"기보다는 **기계를 매번 새로 만든다**는 쪽에 가까웠다. 끔찍한 일이다.

von Neumann의 한 줄이 무엇을 바꾸었는가? 명령어가 데이터와 같은 종류의 시민이 되었다는 사실이다. 우리는 한 메모리 칸을 가리키며 "이건 숫자 7이야"라고 말할 수도 있고, 같은 비트 패턴을 두고 "이건 `ADD A, B`라는 명령어야"라고 말할 수도 있게 됐다. 컴퓨터의 동작을 바꾸고 싶으면 메모리의 비트만 다시 적으면 된다. 케이블을 빼지 않아도 된다. 이 한 발의 차이가 80년치 컴퓨터 산업의 출발선이다.

조금 신비롭게 들리겠지만, 정작 우리는 이 사실에 이미 너무 익숙해서 신비함을 잊고 산다. 우리가 매일 짠 Kotlin 함수가 JIT 컴파일러를 거쳐 바이트코드가 되고, 결국 CPU가 메모리에서 꺼내 실행하는 일련의 비트로 환원된다. 그 비트들이 데이터 영역에 있는 정수 변수와 같은 종류의 메모리 칸에 살고 있다는 점이 — 정확히 1945년의 그 한 줄에서 비롯된 일이라는 점이 — 의식되지 않을 뿐이다.

> **메모: von Neumann이 진짜 발명자인가?**
>
> 사실 stored program 아이디어 자체는 von Neumann이 처음 떠올린 것이 아니다. ENIAC을 설계한 J. Presper Eckert와 John Mauchly가 이미 같은 방향으로 가고 있었고, von Neumann은 그들과의 토론을 정리해 보고서로 묶었다는 시각이 있다. 그러나 보고서의 표지에 자기 이름만 적힌 채 출간되면서 "von Neumann 아키텍처"라는 이름이 굳어졌고, 이후 Eckert와 Mauchly와의 관계는 평생 어색하게 남았다. 학술적 논쟁은 더 길지만, 우리는 표준 용어를 따르되 부록 D에 이 짧은 역사를 따로 정리해두었다. 관심 있는 독자는 부록 D에서 만나자.

## 1.2 그렇다면 Harvard 아키텍처는?

물론 von Neumann만이 유일한 길은 아니다. 같은 시기에 등장해 평행선을 달려온 다른 길이 있다 — **Harvard 아키텍처**. 명령어 메모리와 데이터 메모리가 물리적으로 분리된 구조. 1944년 하버드 대학의 Mark I가 그 원형이라고 알려져 있다.

Harvard 길의 매력은 무엇일까? 명령어를 읽는 일과 데이터를 읽는 일이 같은 메모리 통로를 두고 다투지 않는다는 점이다. 한 사이클에 명령어도 가져오고 데이터도 가져올 수 있다. 그래서 임베디드 마이크로컨트롤러나 DSP, 그리고 현대 CPU 안의 L1 캐시 같은 **속도가 중요한 좁은 영역**에서는 여전히 Harvard 또는 그 개량형이 살아남아 있다.

교육용 CPU의 세계에서도 Harvard 길을 택한 유명한 사례가 있다. 바로 **nand2tetris**의 Hack CPU다. NAND 게이트만 가지고 16비트 컴퓨터를 짓는 이 유명한 교과서는 명령어 ROM과 데이터 RAM을 깨끗하게 갈라놓은 Harvard 모델을 택했다. 왜 그랬을까? 단순함 때문이다. 명령어와 데이터가 같은 메모리에 섞여 있으면, "지금 이 비트 패턴은 명령어인가 데이터인가"를 결정하는 책임이 컨트롤러에게 떨어진다. Harvard로 가르면 그 책임이 줄어든다. 게이트로 처음부터 짜는 교과서에게 그것은 큰 단순화다.

그렇다면 우리는 왜 Harvard 길을 따라가지 않는가? 두 가지 이유가 있다. 첫째, **자기 자신을 수정하는 코드**가 가능한 모델을 직접 만져보는 경험이 컴퓨터 구조의 이해에 결정적이라고 생각하기 때문이다. 명령어와 데이터가 같은 비트로 표현된다는 사실, 그래서 어셈블러라는 프로그램이 다른 프로그램을 만들어낼 수 있다는 사실, 그것이 컴파일러와 인터프리터로 가는 가장 자연스러운 다리다. 둘째, **이 모델이 8비트 시대의 실제 칩들**(8080, 6502, Z80)이 따른 길이기도 하기 때문이다. 우리가 만든 SAP-1과 SAP-2가 8080 ISA의 친척이 되는 것은 von Neumann 모델 위에 서 있기 때문이다. 11장에서 8080, 6502, Z80, 8086의 어셈블리를 비교할 때 이 친척 관계가 또렷이 드러날 것이다.

기억해두자. **우리가 짤 모든 SAP는 von Neumann이다.** 메모리에서 명령어를 꺼내고, 같은 메모리의 다른 칸을 데이터로 읽고, 한 사이클 뒤에 또 한 칸을 명령어로 꺼낸다. Harvard와 von Neumann의 갈림길에 대해 더 알고 싶으면 nand2tetris의 1~5장이 좋은 짝꿍이다. 같은 산을 다른 길로 오르는 두 책을 나란히 펴 보는 것은 늘 즐거운 일이다.

## 1.3 fetch-decode-execute — CPU의 가장 작은 심장 박동

von Neumann 아키텍처가 무엇을 정했는지를 한 발 더 들어가서 보자. 명령어와 데이터가 같은 메모리에 산다고 했을 때, CPU는 그 메모리에서 어떻게 명령어를 하나씩 꺼내 실행하는 걸까?

답은 단 세 박자로 정리된다. **꺼내고**(fetch), **풀고**(decode), **수행한다**(execute). 이 세 박자가 끝나면 CPU는 다음 명령어를 꺼내려고 메모리 주소를 한 칸 옮긴다. 그러고는 또 꺼내고, 풀고, 수행한다. 컴퓨터의 전원이 켜져 있는 동안 — 우리가 키보드 앞에 앉아 있는 매 순간, JVM이 우리 코드를 실행하는 매 순간 — 이 사이클은 멈추지 않는다.

> "The Fetch-Execute cycle was first introduced by John von Neumann. This cycle starts when the computer is turned on and keeps repeating until the computer is shut down."
>
> — Baeldung CS, *The Fetch-Execute Cycle*

자세히 들여다보면 세 박자 안에는 더 작은 박자들이 숨어 있다. fetch 단계만 떼어 봐도 이렇다. 먼저 **PC**(Program Counter)에 적힌 주소를 **MAR**(Memory Address Register)로 옮긴다. MAR이 가리키는 메모리 칸의 값을 읽어 **MDR**(Memory Data Register)에 담는다. MDR의 값을 **CIR**(Current Instruction Register, 또는 단순히 IR)로 옮긴다. 그러고 나서 PC를 1 증가시킨다. 다음 명령어를 가리키도록.

말로 풀어 쓰니까 한 단락이 됐지만, 이걸 그림으로 그려보면 그저 작은 화살표 몇 개다. PC → MAR → 메모리 → MDR → IR → PC++. 이 화살표들이 우리가 앞으로 8개 챕터에 걸쳐 만들 모든 SAP CPU의 뼈대다. 다음 2장에서 우리는 이 화살표를 곧장 30줄짜리 Kotlin 함수로 옮긴다. 어떻게 옮기는지 짧게 미리 보면 이렇다.

```kotlin
class MiniCpu(val memory: IntArray) {
    var pc: Int = 0
    var a: Int = 0           // 누산기
    var halted: Boolean = false

    fun step() {
        val opcode = fetch()      // 1. 꺼낸다
        decode(opcode)            // 2. 풀고 → 수행한다
    }

    private fun fetch(): Int {
        val byte = memory[pc] and 0xFF
        pc = (pc + 1) and 0xFF
        return byte
    }

    private fun decode(opcode: Int) {
        when (opcode and 0xF0) {
            0x10 -> a = (a + (opcode and 0x0F)) and 0xFF   // ADD
            0xF0 -> halted = true                          // HLT
            else -> error("unknown opcode: $opcode")
        }
    }
}
```

낯설어 보이는 코드가 한 줄도 없다. `IntArray`로 메모리를 흉내내고, `var`로 레지스터를 만들고, `when`으로 명령어를 분기한다. CPU란 결국 이런 것이다. 정확히 말하면, CPU의 **가장 단순한 모습**이 이런 것이다. 우리는 다음 장에서 이 30줄을 처음부터 끝까지 손으로 짜고 8개 테스트로 검증할 것이다. 첫 번째 작은 성공.

> **사이드바: 왜 `Int`인가, `UByte`가 아닌가?**
>
> 눈썰미가 좋은 독자라면 `IntArray`에 8비트 값을 담는다는 표현이 이상하게 느껴졌을 수 있다. Kotlin에는 `UByte`라는 자연스러운 타입이 있는데 왜 `Int` + `and 0xFF`라는 번거로운 마스킹을 쓰는가? 짧게 답하면, JVM에서 `Int`가 가장 빠르고 호환성이 좋기 때문이다. 그리고 정직하게 답하면, `UByte`는 매력적이지만 비트 시프트 연산자(`>>`)가 좁은 타입에서 아직 미지원이라 매번 `.toInt()`로 변환해야 하는 번거로움이 있다. 4장에서 이 결정을 정면으로 다룬다. 책 전체에서 우리가 어느 챕터에서 무엇을 쓸지의 정책도 그곳에서 선언한다. 지금은 일단 "`Int` + `and 0xFF`"가 우리의 기본 표기라는 점만 기억해두자.

## 1.4 왜 8비트인가 — 단순함을 만드는 능력

이쯤에서 한 가지 의문이 생긴다. 좋다, von Neumann 모델을 따라간다. fetch-decode-execute로 동작한다. 그런데 굳이 **8비트**여야 할 이유는 무엇인가? 우리가 매일 쓰는 CPU는 64비트인데, 굳이 절반의 절반의 절반 크기인 8비트를 만져야 할 까닭이 있을까?

이 질문에 대한 답은 두 갈래로 갈라진다.

첫 번째 답은 **단순함이 가르치는 힘**이다. r/EmuDev라는 17,000명 규모의 emulator 개발 커뮤니티에서는 거의 만장일치로 "첫 emulator는 Chip-8부터 시작하라"고 권한다. emudev.org도 같은 충고를 한다. 왜일까? 단순한 시스템이 그 자체로 단순해서가 아니다. **단순한 시스템이 가르쳐 주는 것이 더 많기 때문**이다. Chip-8은 명령어가 35개, 메모리가 4KB다. 그 좁은 우주 안에서 학습자는 명령어 디코딩, 메모리 매핑, 입출력 처리, 그리고 가장 중요한 — fetch-decode-execute 사이클의 본질 — 을 한 번에 만진다. 복잡한 x86이나 ARM을 처음 짜려고 들면 명령어 디코딩에서만 몇 달이 흐른다. 본질에 닿기 전에 지친다. 정말 아찔한 일이다.

SAP-1은 그 점에서 Chip-8과 동등한 진입 수준에 있다. 명령어가 5개, 메모리가 16바이트. 16비트도 아니고 16**바이트**다. 한 줄에 들어가는 트윗보다 작은 메모리에서 컴퓨터가 동작한다. 이 좁은 우주에 시간을 들이면, 거기에서 본 모든 화살표 — PC가 어떻게 증가하는지, MAR이 어떻게 쓰이는지, ALU가 어떻게 두 수를 더하는지 — 가 손 안에 자연스럽게 들어온다. ChampSim이라는 페다고지 시뮬레이터를 만든 학자들이 한 말을 빌리면 이렇다.

> "ChampSim simulator's design does not necessarily attempt to precisely replicate all hardware functions, but to emulate it in a readable way, allowing a student to rapidly grasp its functionality."
>
> — *ChampSim*, SIGCSE 발표 [논문-12]

읽기 쉽게, 빨리 잡힐 수 있도록. 이것이 단순함 우선 관점의 본질이다. 정확함을 좇느라 가르침의 효용을 잃지 말자.

두 번째 답은 **8비트가 컴퓨터 아키텍처 어휘의 정립기**라는 사실에서 온다. 1970년대 중반부터 1980년대 초반은 8비트 CPU들이 세상을 새로 쓴 시대다. 1974년의 Intel 8080, 1975년의 MOS 6502, 1976년의 Zilog Z80, 1978년의 Intel 8086. 이 칩들 중 6502는 단돈 25달러였고 — 그 가격으로 Apple II, Commodore 64, NES, Atari 2600 같은 기기들이 시장에 나왔다. 6502 가격 혁명 한 줄로 컴퓨터가 학교에 들어가고, 거실에 들어가고, 가족 오락이 되었다는 흐름을 정리한 글이 있다.

> "The Apple II put computing in schools. The Commodore 64 put it in living rooms. The NES made it entertainment for an entire generation."
>
> — Harlepengren, 8-bit CPU history [웹-자료12]

그 시기에 누산기, 인덱스 레지스터, 스택 포인터, 어드레싱 모드, 마이크로코드, 인터럽트 같은 단어들이 표준 어휘로 정착했다. 우리가 오늘 RISC-V 명세를 보면서 자연스럽게 이해하는 거의 모든 개념이 그 시대의 8비트 칩들에서 처음 또렷한 형태로 굳어졌다. 8비트 CPU를 만든다는 것은 박물관 견학이 아니다. **컴퓨터 아키텍처의 어휘를 손으로 익히는 일이다.** 그 어휘를 우리 손으로 한 번 익혀보자.

그렇다면 왜 굳이 **SAP-1**인가? 다른 좋은 선택지도 있다. nand2tetris의 Hack(16비트), LC-3(16비트), RISC-V(32비트). 그러나 SAP-1은 8비트, 명령어 5개, 메모리 16바이트라는 단순함의 극단으로 우리를 데려간다. 거기서 위로 한 단계씩 올라간다. SAP-1 → SAP-2(64KB로 폭증, 39개 명령어) → 8비트 RISC(우리 손으로 직접 설계한 ISA, 30개 명령어). 이 점진적 진화의 줄이 다른 어떤 교과서에도 정확히 같은 형태로 존재하지 않는다. 이 책의 자리가 거기다.

## 1.5 우리 앞에 놓인 15주 — 5부 짜리 학습 곡선 지도

이 책은 다섯 부분으로 되어 있다. 한 부씩 통과할 때마다 손에 잡히는 무엇이 늘어난다. 책의 끝에 도달했을 때 GitHub 저장소에는 다음과 같은 것들이 쌓여 있을 것이다 — 우리가 짠 SAP-1 시뮬레이터, 그 위에서 돌아가는 우리가 짠 어셈블러, 더 큰 SAP-2, 디버거, 그리고 마지막으로 우리가 직접 ISA를 설계한 8비트 RISC.

지도를 한 번 펴 보자.

| 부 | 주제 | 끝나면 만지는 것 |
|----|-----|---------|
| 1부. 출발 | von Neumann, 미니 CPU 30줄 | 30줄짜리 ADD/HLT 만 도는 미니 CPU |
| 2부. SAP-1 짓기 | 레지스터, ALU, 메모리, 어셈블러 | 자기가 짠 SAP-1 위에서 자기가 짠 어셈블러로 어셈블한 프로그램이 돌아간다 |
| 3부. SAP-2로 진화 | 64KB, 39개 명령어, 디버거 | 자기 CPU 위에서 함수 호출과 디버거가 돌아간다 |
| 4부. 실제 칩 만나기 | 8080·6502·Z80·8086 어셈블리 비교 | 같은 Fibonacci를 4가지 어셈블리로 읽고 쓴다 |
| 5부. 8비트 RISC | ISA 직접 설계, cycle-accurate | 자기가 설계한 ISA, 자기가 짠 RISC가 cycle-accurate로 돌아간다 |

각 부의 첫 챕터 앞에는 "여기서 우리는 어디에 있는가" 환기 페이지가 짧게 들어간다. 길을 잃었다는 느낌이 들면 그 페이지를 한 번씩 다시 펴 보자. 산행에서 능선 위에 잠깐 멈춰 서서 지나온 길과 갈 길을 함께 바라보는 시간 같은 것이다.

핵심은 이 줄에 있다. **점진적 진화.** SAP-1을 짓고 나면 거기에 살이 붙어 SAP-2가 된다. SAP-2를 짓고 나면 다시 그 살이 다시 짜인다 — 이번에는 우리가 직접 설계한 ISA로. 한 번에 모든 것을 짜지 않는다. 한 번에 모든 것을 이해하려는 시도는 거의 항상 실패한다. 단계마다 작은 우주에서 작은 성공을 거두고, 그 성공을 짚으며 다음 우주로 넘어가는 편이 낫다. nand2tetris 12개 프로젝트도, Crafting Interpreters의 "같은 인터프리터를 두 번 짓는 구조"도 같은 페다고지에 기대고 있다.

> "This dual implementation strategy isn't just a structural gimmick. It serves a deep pedagogical purpose, allowing readers to first grasp the conceptual architecture in a high-level language before rebuilding everything from raw memory."
>
> — *Crafting Interpreters*에 대한 후기 [웹-자료19]

우리의 SAP-1 → SAP-2 → RISC도 같은 의미의 "한 인터프리터를 세 번 짓기"다. 같은 동물의 진화 단계를 한 줄로 놓고 보는 느낌이라고 해도 좋다.

## 1.6 우리 손에 들린 도구 — Kotlin 한 줄의 변호

다음 장에서 우리가 처음 만질 것은 Kotlin 코드다. 왜 Kotlin인가? 이 책을 손에 든 독자 대부분이 Kotlin이나 Java를 일상에서 쓰는 중급 이상의 개발자다. **emulator를 배우는 게 본업이지 새 언어를 배우는 게 본업이 아니다.** 이미 익숙한 언어로 본질에 집중하자는 것이 이 책의 입장이다.

tobiasvl이라는 Chip-8 가이드 저자가 한 짧은 충고가 있다.

> "Pick a programming language you're familiar with such as C/C++ or Java."
>
> — tobiasvl Chip-8 guide

C/C++ 또는 Java라고 했지만, Kotlin은 이 충고에 정확히 부합한다. JVM 위에서 돌고, 자바와 상호운용되고, sealed class와 data class와 `when` 표현식 같은 도구가 명령어 디코더를 자연스럽게 짜기에 좋다. 비트 처리도 충분히 한다. 한 가지 함정만 빼면.

함정은 **signed byte**다. Kotlin의 `Byte`는 -128부터 127까지의 부호 있는 8비트다. 그런데 8비트 CPU에서 우리가 다루는 비트 패턴은 보통 0부터 255까지의 부호 없는 표현으로 생각하는 것이 자연스럽다. `Byte`를 그대로 쓰면 비트 시프트나 비교 연산에서 부호 확장 버그가 끊임없이 따라온다. 정말 찜찜한 일이다.

이 함정을 피하는 길은 두 가지다. **`Int` + 마스킹**(`value and 0xFF`)으로 호환성과 속도를 챙기든가, **`UByte`**로 자연스러운 0~255 표현을 챙기든가. 두 길은 모두 정당하다. Kotlin 공식 문서가 정리한 한 줄이 이걸 잘 보여준다.

> "There is no significant overhead when using unsigned integer types on the JVM. Under the covers, all of the unsigned types in Kotlin are implemented as inline classes."
>
> — Kotlin 공식 문서, Unsigned integer types

우리는 어느 길을 택할까? **앞쪽 챕터는 `Int` + 마스킹**으로 시작한다. 가장 빠르고, 가장 호환성이 좋고, Java로 옮기기도 쉽다. **뒤쪽 챕터에서 `UByte`로 마이그레이션**한다. 같은 ALU를 두 가지 표현으로 짜 보고 차이를 직접 체감한다. 이 결정의 자세한 정당화는 4장에서 다시 만나자.

> **사이드바: ksim65와 kNES의 길**
>
> 이 책을 쓰면서 가장 자주 들여다본 두 개의 Kotlin/JVM 사례가 있다. **ksim65**는 6502/65C02 시뮬레이터로, Commodore 64 ROM을 그대로 돌릴 만큼 성숙한 instruction-accurate 구현체다. 의도적으로 cycle-exact를 포기하면서 단순함을 택했다. **kNES**는 Java의 vNES를 Kotlin으로 포팅한 cycle-accurate NES 에뮬레이터다. 두 프로젝트 모두 우리가 앞으로 자주 빌릴 패턴 — `when`을 명령어 디스패처로 쓰는 방식, sealed class로 명령어를 모델링하는 방식 — 을 잘 보여준다. 책의 부록에서 두 프로젝트를 다시 만난다. 지금은 그저 "8비트 CPU를 Kotlin으로 짜는 일은 우리가 처음이 아니다, 그러나 같은 방식으로 짠 사람도 없다"는 점만 기억해두면 된다.

## 1.7 정확하지 않을 권리 — 시뮬레이터 정신

마지막으로 한 가지 짧은 입장을 정리하고 다음 장으로 넘어가자. 시뮬레이터의 세계에는 오래된 논쟁이 있다 — **cycle-accurate**인가, **instruction-accurate**인가.

cycle-accurate 진영은 이렇게 말한다. 8비트 시스템, 특히 NES나 Game Boy 같은 기기에서는 CPU와 PPU(그래픽 칩)가 사이클 단위로 동기화된다. 만약 한 명령어를 통째로 한 번에 끝낸다면(instruction-accurate), 그 사이의 미세한 타이밍이 깨져서 게임 화면이 비뚤어진다. 그래서 BeesNES나 bsnes 같은 프로젝트는 sub-cycle 단위까지 정확하게 흉내내는 데 자존심을 건다.

instruction-accurate 진영은 다르게 말한다. 학습용 시뮬레이터의 본질은 **읽기 쉽고 빠르게 잡히는 것**이다. ksim65는 의도적으로 cycle-exact를 포기하며 "for simplicity reasons"라고 적었다. 정확도는 학습에 필요한 만큼만 가지면 된다.

우리는 어느 편에 설까? **앞쪽 4부까지는 instruction-accurate**로 간다. 한 명령어를 한 호출로 끝내고 다음으로 넘어간다. 그래야 가독성이 산다. **5부에서 cycle-accurate로 리팩터링**한다. 같은 시뮬레이터를 두 가지 추상화로 짜 보면 사이클이라는 단위가 손에 어떻게 다르게 잡히는지가 또렷이 드러난다. 그 비교를 위해서라도 cycle-accurate를 한 번 직접 만져 볼 필요가 있다.

이것은 단순함 우선 관점과 정확도 우선 관점 사이에서 우리가 잡은 자리다. 두 길이 모두 정당하다. 다만 학습이 본업인 책이라면, 가독성을 먼저 잡고 정확도를 나중에 입히는 편이 낫다. ChampSim의 학자들도, ksim65의 저자도 그렇게 결정했다. 우리도 그 자리에 선다.

## 1.8 다음 장으로 — 30줄의 약속

이 장을 정리해보자. 우리가 함께 짚은 것은 네 가지다.

첫째, 우리가 매일 쓰는 컴퓨터의 모습은 1945년 한 보고서가 정한 von Neumann 아키텍처다. 그 핵심은 **명령어와 데이터가 같은 메모리에 산다**는 한 줄이다. 이 한 줄이 자기 수정 코드, 어셈블러, 컴파일러, 인터프리터로 가는 모든 길을 연다.

둘째, CPU의 가장 작은 심장 박동은 **fetch-decode-execute**라는 세 박자다. 그 안에 PC, MAR, MDR, IR이라는 작은 화살표들이 살고 있다. 다음 장에서 우리는 이 화살표들을 30줄짜리 Kotlin 함수로 직접 옮긴다.

셋째, 8비트는 박물관 견학이 아니라 **컴퓨터 아키텍처 어휘의 정립기**다. SAP-1이라는 단순함의 극단에서 시작해 SAP-2, 그리고 우리가 직접 설계한 8비트 RISC까지 점진적으로 올라간다. 이 진화의 줄은 다른 어느 교과서에도 정확히 같은 형태로 있지 않다.

넷째, 우리 손에 들린 도구는 Kotlin이다. `Int` + 마스킹으로 시작해 `UByte`로 마이그레이션한다. instruction-accurate로 시작해 cycle-accurate로 진화한다. 단순함을 먼저 잡고 정확도를 나중에 입히는 편이 낫다.

기억해두자. 우리는 이 모든 것을 한 번에 이해할 필요가 없다. 매 장마다 작은 우주에서 작은 성공을 거두는 식으로 간다. 다음 장에서 만날 첫 작은 성공은 30줄짜리 Kotlin 미니 CPU다. ADD와 HLT 단 두 개의 명령어만 도는, 그러나 분명히 "동작하는" CPU. 자기 컴퓨터의 터미널에서 `./gradlew :chapter-02:run`을 입력하면 우리가 짠 그 CPU가 메모리에서 명령어를 꺼내 실행하는 모습을 직접 본다. 그 30줄 안에 von Neumann의 1945년이 응축돼 있다는 사실은, 그 줄을 직접 짜 본 사람만이 진짜로 안다.

그러니까 이제 슬슬 IDE를 열어보자.
