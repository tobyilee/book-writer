## 리뷰 라운드 1

### 종합 평가

계획서는 레퍼런스의 §1~§7을 매우 충실하게 챕터별로 매핑했고, 5부 14장 구조에 명확한 정서적 보상점("첫 SAP-1이 돌아간다", "자기 어셈블러로 자기 CPU 위에서 돌린다", "자기 CPU에 디버거가 생긴다")을 심어 두었다. 토비 문체와 결합할 청유형 키워드("정면돌파", "묘사한다", "짓는다")도 챕터 제목에 잘 녹였다. 그러나 **(1) 4장의 과부하**, **(2) 11→12장 순서의 어색함**, **(3) 인터럽트/메모리 매핑 I/O 같은 핵심 토픽이 사이드바조차 없이 사라진 점**, **(4) 1장과 2장의 역할 분배에서 1장이 너무 무겁다는 점**, **(5) 테스트 전략의 독립 챕터 부재**가 책 전체의 페다고지 효율을 떨어뜨릴 위험이 있다. 차별점 포지셔닝("nand2tetris와 Ben Eater 사이")의 환기도 두 군데에 그쳐, 5부 구조의 각 부 도입부에 일관되게 회수돼야 한다. 우선순위 순으로 8개 이슈와 추가 토픽 3개를 제안한다.

---

### 강점

- **레퍼런스 커버리지 자체는 거의 100%다.** §1 모든 개념 정의가 챕터로 분산됐고 §4 논쟁점 4개는 본문 또는 사이드바로 모두 배치됐다. §5 실무 팁은 본문 코드 패턴으로 녹였다 — 자기 점검 체크리스트가 정직하게 작동했다.
- **5부 구조가 정서적 보상의 단위와 일치한다.** 1부 끝(첫 미니 CPU), 2부 끝(SAP-1 + 어셈블러), 3부 끝(SAP-2 + 디버거), 4부 끝(RISC + 8086까지 닿음), 5부(장인 마무리). 매 부마다 "한 단계 더 깊어졌다"는 명시적 보상이 있다.
- **본 책 고유의 합성 8비트 RISC ISA를 11장의 중심으로 못 박은 점은 시장 차별점과 정확히 일치한다.** 레퍼런스 §7.1 한계 항목("8비트 RISC ISA의 구체 설계 — 직접 영감원이 없다")을 정면으로 흡수해 책의 가장 창의적 산물로 만들었다.
- **챕터마다 "독자가 얻는 것"이 손에 잡히게 적혔다.** 추상적 학습 목표가 아니라 "Fibonacci가 자기 SAP-2에서 돈다", "자기 CPU에 breakpoint가 생긴다" 같은 산출물 단위. GitHub 저장소가 한 단계씩 발전한다는 약속이 챕터별로 구체적이다.
- **부록 4개의 분배가 합리적이다.** A(환경)·B(비트 연산 치트시트)·C(8비트 비교 확장판)·D(더 읽을거리)가 본문에서 흐름을 끊을 만한 잡일을 흡수한다.

### 개선이 필요한 이슈 (우선순위 순)

#### 이슈 1 (Critical) — 챕터 11과 12의 순서가 흐름을 끊는다

- **무엇이 문제인가:** 4부 "8비트 RISC로 점프"의 챕터 배치가 10장(RISC 역사) → 11장(우리의 RISC 설계) → 12장(실제 8비트 8080/6502/Z80/8086 비교)이다. 12장은 사실상 CISC 계보(8080·6502·Z80·8086 모두 CISC 또는 RISC와 CISC 중간) 비교 챕터인데, "RISC 패러다임 전환"이라는 4부 제목 아래 마지막에 놓이면서 패러다임이 다시 뒤로 돌아간다.
- **왜 문제인가 (내러티브 관점):** 4부의 정서적 절정은 11장 "자기 RISC ISA로 Fibonacci를 SAP-2 버전보다 짧게 짠다"여야 한다. 그런데 12장이 8080/Z80/8086 어셈블리로 다시 끌고 가면 RISC 전환의 감격이 흐려진다. 또한 7장(SAP-2가 8080 subset)에서 "실제 8080은 어떻게 생겼지?"라는 호기심이 떴는데 5개 챕터 뒤 12장까지 답이 미뤄진다.
- **제안:** **12장을 9장과 10장 사이(즉 새 10장)로 이동.** 순서는 → 9장(SAP-2 + 디버거) → **새 10장(같은 Fibonacci, 네 어셈블리 — SAP-2의 친척들)** → 11장(RISC 역사) → 12장(우리의 8비트 RISC 설계). 이렇게 하면: (a) SAP-2(8080 subset)를 짠 직후 실제 8080/Z80을 만나 친연성을 즉시 회수하고, (b) "CISC 계보의 끝"을 본 다음 RISC 패러다임 전환으로 자연스럽게 넘어가며, (c) 4부의 절정이 "우리의 RISC 설계"로 끝나면서 패러다임 전환의 감격이 흐려지지 않는다. 4부 명칭도 살짝 조정 — "8비트 RISC로 점프" 그대로 두되 첫 챕터(RISC 역사)에서 "지금까지 짠 SAP-2가 CISC 계보였다"를 명시.

#### 이슈 2 (Critical) — 챕터 4가 과부하다 (입문자의 무덤 자리)

- **무엇이 문제인가:** 4장(26~30p)이 다루는 것: (1) 레지스터 A·B 클래스화, (2) 2의 보수, (3) 가산기/감산기, (4) Z/C 플래그, (5) ADC/SBC overflow 정면돌파, (6) 512케이스 parameterized test, (7) Ken Shirriff 인용, (8) Java signed byte → UByte 사이드바. 8개 토픽이 한 챕터에 적층돼 있다.
- **왜 문제인가 (독자 적합도):** 레퍼런스 §5.2와 [커뮤니티-패턴3]가 명시한다 — ADC/SBC가 "90%의 입문자가 막히는 지점"이다. 거기에 부호/부호 없음 비트 표현 패러다임 전환(Java signed byte vs Kotlin UByte)을 동시에 던지면 학습 부하가 임계점을 넘는다. 중급 Kotlin 개발자라도 비트 연산에 익숙하지 않은 비율이 높다(레퍼런스 §5.1, [웹-자료10]).
- **제안:** **4장을 두 챕터로 쪼개라.**
  - **새 4장. 레지스터와 8비트 산술 — 비트로 다시 익히는 덧셈:** A/B 레지스터, 2의 보수, 가산기/감산기, Z/C 플래그까지. UByte vs Int 마스킹 결정도 여기. 18~22p.
  - **새 5장. overflow라는 무덤 — ADC/SBC를 테스트로 잡는다:** ADC/SBC overflow 플래그 정면돌파, 512케이스 테이블, Ken Shirriff 수학 해설, Java signed byte 함정. 20~24p. **이 챕터는 사실상 본 책의 "테스트 주도 emulator 개발" 도입 챕터로 자리매김** (이슈 5와 연결).
  - 후속 챕터 번호가 한 칸씩 밀린다 — 5부 → 14장 → 15장. 총 분량이 18~22p 증가하지만 목표(300p+ "두꺼운 한 권")에 부합한다. 자기 점검 체크리스트의 "두꺼운 한 권"을 더 적극 활용.

#### 이슈 3 (Should) — 인터럽트 메커니즘이 본문에서 빠졌다

- **무엇이 문제인가:** 인터럽트가 14장 "다음 단계"의 한 줄로만 등장한다. 그러나 SAP-2의 I/O 포트(8장)와 12장의 8080·Z80·6502 비교는 인터럽트 없이는 반쪽이다.
- **왜 문제인가 (커버리지 관점):** 8비트 CPU에서 인터럽트는 핵심이다 — Z80의 IM 0/1/2 모드, 6502의 IRQ/NMI/BRK, 8080의 RST 명령은 모두 "I/O와 CPU가 어떻게 협력하는가"의 답이다. 13장에서 cycle-accurate를 다루면서도 "왜 cycle-accurate가 필요한가"의 가장 큰 이유(인터럽트 타이밍)가 본문에 없으면 동기가 약하다. 또한 14장 "다음 단계"에서 인터럽트를 "여기서 안 다룬 토픽"으로 던지면 책의 범위가 너무 좁아 보인다.
- **제안:**
  - **8장(조건 분기·서브루틴·I/O)에 "인터럽트 첫 만남" 절을 추가하라.** SAP-2 명세에 인터럽트가 없는 점은 인정하되, "본 책은 SAP-2에 polled I/O만 두고 인터럽트는 합성 8비트 RISC(11장 또는 새 12장)에서 도입한다"고 명시. 한 페이지 분량 사이드바: 인터럽트 vs polling, vector table, ISR의 정의.
  - **새 12장(이슈 1 이후 우리의 8비트 RISC 설계)에 인터럽트 벡터를 ISA 설계 결정으로 포함하라.** "RISC는 단순함이 미덕인데 인터럽트는 어떻게 단순하게 처리할 것인가" — RISC-V의 trap 메커니즘을 8비트로 축소한 합성 설계. 책의 차별점이 강화된다.
  - **13장(cycle-accurate)에 "cycle-accurate가 필수가 되는 진짜 이유 — 인터럽트 타이밍" 절을 1~2p 추가.** Game Boy/NES 사례.

#### 이슈 4 (Should) — 챕터 1이 너무 무겁고, 2장에 코드가 몰린다

- **무엇이 문제인가:** 1장(18~22p)이 다루는 것: (a) von Neumann 아키텍처, (b) Harvard 대비, (c) 8비트 선택 정당화, (d) 학습 곡선 지도, (e) von Neumann 발명자 논쟁. 코드 0줄. 첫 GitHub 커밋은 2장 끝에 가야 등장한다.
- **왜 문제인가 (독자 적합도/내러티브):** 본 책의 차별점은 "코드로 짓는다"인데 첫 챕터가 18~22p 동안 코드 없이 역사·이론만 다룬다. 중급 개발자가 "또 컴퓨터구조 교과서네"라는 인상을 받을 위험이 있다. 토비 문체(평어체+청유형)는 첫 챕터부터 코드와 같이 가야 자기 정체성을 드러낸다.
- **제안:**
  - **1장을 12~14p로 축소하고, 코드 한 조각을 끌어다 놓아라.** "이 책이 무엇인가"의 메타 설명을 줄이고, 1장 끝에 "다음 챕터에서 우리가 짤 30줄 코드의 골격"이라며 fetch-decode-execute의 의사 코드 5~6줄을 미리 보여준다. 학습 곡선 지도는 5부 구조 다이어그램으로 1p에 압축.
  - **2장은 그대로 유지(20~24p) — 첫 GitHub 커밋의 정서적 무게가 유지되어야 한다.**
  - von Neumann 발명자 논쟁(§4.6)은 1장 본문에서 빼고 부록 D(더 읽을거리)나 1장 끝 메모로 이관. 5축 중 "독자 적합도"를 위한 양보.

#### 이슈 5 (Should) — 테스트 전략의 독립 챕터가 없다

- **무엇이 문제인가:** 레퍼런스 §5.5는 명시한다 — "처음부터 TDD로 SAP-1을 구현 — emulator 개발의 가장 안전한 방법." 그런데 계획서에서는 4장에 parameterized test 한 단락, 6장 어셈블러에서 슬쩍 언급되는 정도. test ROM 개념(klaus2m5/6502_65C02_functional_tests, nestest)이 본문에 없다.
- **왜 문제인가 (커버리지 + 차별점):** 본 책의 차별점 중 하나는 "Kotlin 생태계의 테스트 도구로 emulator를 검증한다"이다. ksim65도 이 점을 강조한다. 그런데 테스트가 4장 ADC 한 챕터에 갇혀 있으면 차별점이 약해진다.
- **제안 (이슈 2와 연동):**
  - **이슈 2의 새 5장(overflow라는 무덤)에서 본 책의 테스트 전략 전체를 도입한다.** kotest 설치, `@ParameterizedTest`, table-driven test, property-based testing, test ROM 개념. 이 챕터를 본 책의 "테스트 주도 emulator 개발의 도입부"로 자리매김.
  - **9장(마이크로코드와 디버거)에 "테스트 ROM으로 자기 SAP-2를 검증한다" 절을 1~2p 추가.** 6502 functional test의 SAP-2 축소판을 책에서 직접 작성. 디버거와 자연스럽게 묶인다.

#### 이슈 6 (Should) — "nand2tetris와 Ben Eater 사이"의 환기 빈도가 부족하다

- **무엇이 문제인가:** 차별점 포지셔닝("nand2tetris의 추상 게이트와 Ben Eater의 실물 빌드 사이")이 1장 도입부 한 번, 3장 사이드바 한 번, 13장 사이드바 한 번 등장한다. 즉 5부 중 1·2·5부에서만 회수되고 3부(SAP-2)와 4부(RISC)에서는 잊혀진다.
- **왜 문제인가 (차별점 일관성):** 5부의 매 도입부에서 "왜 우리가 nand2tetris도 Ben Eater도 아닌 자기만의 길을 가는가"를 환기시켜야 차별점이 책의 척추로 작동한다. 특히 4부 RISC는 nand2tetris의 Hack CPU나 RISC-V 학습서와 직접 비교될 수 있는 영역이라 회수가 필요하다.
- **제안:**
  - **각 부 도입 짧은 페이지(2~3p)를 추가하라.** 부 도입부에서 (a) 이번 부의 정서적 보상, (b) nand2tetris/Ben Eater와 본 책의 위치 차이, (c) 이번 부의 GitHub 저장소 상태 변화를 1~2단락으로. 5개 부 × 2~3p = 10~15p 증가. 또한 이는 5부 구조의 "한 단계 도약" 느낌을 시각적으로 강화한다.
  - 또는 더 가벼운 대안 — 각 부의 첫 챕터 도입부 1단락에서 차별점을 회수.

#### 이슈 7 (Should) — 메모리 매핑 I/O vs 별도 포트 공간이 한 줄로 처리된다

- **무엇이 문제인가:** 8장의 "I/O 포트 IN/OUT — 메모리 매핑 vs 별도 포트 공간"이 한 줄에 그친다. 그러나 이것은 6502(메모리 매핑)와 8080·Z80·8086(별도 포트, IN/OUT 명령) 사이의 가장 큰 철학 차이 중 하나다.
- **왜 문제인가 (커버리지 + 차별점):** 12장의 정량 비교에서 "어드레싱 모드" 차이만 다루고 I/O 모델 차이를 빼면 6502 vs Z80 비교가 절반밖에 안 보인다. 또 14장의 "다음 단계 — I/O 디바이스"가 이 토픽을 받아주려면 본문에서 한 번 짚어야 한다.
- **제안:** **8장에 "메모리 매핑 I/O vs 별도 포트 공간" 절을 1.5~2p로 확장하라.** SAP-2가 별도 포트(8080 subset이라)를 쓰는 결정의 의미, 6502가 메모리 매핑을 선택한 의미, NES PPU가 메모리 매핑 레지스터를 쓰는 사례. 12장 비교 표(이슈 1 이후 새 10장)에 "I/O 모델" 컬럼 추가.

#### 이슈 8 (Nice-to-have) — 챕터별 GitHub 저장소 발전이 부록 A로 밀려 있다

- **무엇이 문제인가:** "매 챕터마다 GitHub 저장소가 한 단계 발전한다"는 약속은 5부 구조 설명에 있지만, 구체적인 폴더 구조(`chapter-01/`, `chapter-02/`...)는 부록 A에 묻혀 있다. 그리고 각 챕터의 "독자가 얻는 것" 항목이 코드 산출물이 아니라 추상적 학습 결과로 적힌 챕터(특히 1, 7, 10장)가 있다.
- **왜 문제인가 (실습 강도):** 본 책의 정체성이 "코드로 짓는다"인데, 챕터마다 "이번 챕터의 산출물: 어떤 파일들, 몇 개의 테스트, 어떤 명령어로 돌릴 수 있는지"가 본문 끝에 명시되지 않으면 GitHub 저장소가 어떻게 자라는지가 흐려진다.
- **제안:** **각 챕터 마지막에 "이번 챕터의 GitHub 커밋" 박스를 표준 패턴으로 명시하라.** 예:
  ```
  이번 챕터의 GitHub: chapter-02/
    src/main/kotlin/MiniCpu.kt (40줄)
    src/test/kotlin/MiniCpuTest.kt (8개 테스트)
    실행: ./gradlew :chapter-02:run
  ```
  계획서의 챕터 항목에는 한 줄로 명시해도 충분 — 본 저술 단계에서 챕터 작가가 따를 패턴이 정해진다. 1장도 코드는 없지만 "지도 문서 `00-roadmap.md`"라도 산출.

### 누락 또는 추가 필요 토픽

#### 토픽 A. ASCII와 8비트 OUT의 표시 — 5장 또는 8장에 짧게 추가

- 8비트 CPU의 OUT 명령이 출력하는 게 무엇인가? 7-segment 디스플레이(SAP-1 원전)? ASCII 문자(SAP-2 I/O 포트)? Kotlin 시뮬레이터에서는 어떻게 보여줄 것인가?
- 레퍼런스에서는 명시되지 않았지만 본 책이 "Kotlin 시뮬레이터"인 만큼 콘솔 출력의 표현 결정이 코드 결정과 직결된다. 5장의 첫 SAP-1 프로그램 OUT 결과를 어떻게 보여줄지가 분명해야 한다.
- **제안:** 5장 끝 또는 8장 I/O 절에 0.5~1p — "OUT은 무엇을 출력하는가, ASCII와 8비트의 만남."

#### 토픽 B. Kotlin 시뮬레이터의 성능 측정 — 13장 보강

- 13장 "instruction-accurate → cycle-accurate" 리팩토링에서 자연스럽게 따라오는 질문 — Kotlin/JVM 시뮬레이터의 성능. 8비트 CPU를 시뮬레이션하는데 native 8비트 칩보다 몇 배 느린가/빠른가, JIT의 영향. 레퍼런스 §2.5("ksim65의 가독성 우선")의 반대편 — 가독성을 포기하면 어디까지 갈 수 있는가.
- **제안:** 13장에 "Kotlin 시뮬레이터는 얼마나 빠른가" 절을 2~3p 추가. JIT 워밍업, `Int` vs `UByte` 성능 비교, instruction count/sec 벤치. 14장 "다음 단계"의 dynamic recompilation 떡밥과 연결.

#### 토픽 C. 부동소수점·문자열의 부재가 의미하는 것 — 14장 다음 단계에 한 단락

- 8비트 CPU는 부동소수점도 문자열도 없다. 그게 무엇을 의미하는가? 소프트웨어 FPU(소프트 부동소수점)의 역사, 8087 코프로세서, 문자열 처리에 ASCII가 어떻게 등장하는가.
- **제안:** 14장 "다음 단계" 청사진에 한 단락 추가 — "우리가 짠 8비트 CPU에는 부동소수점도 문자열도 없다. 이걸 더하면 다음 챕터가 시작된다." 후속 책 떡밥.

### 갱신 권고 — book-planner에게 핵심 액션 아이템 5개

1. **챕터 11과 12의 순서를 뒤집어라.** 새 10장 = 8080/6502/Z80/8086 비교(현 12장), 새 11장 = RISC 역사(현 10장), 새 12장 = 우리의 8비트 RISC 설계(현 11장). 4부의 정서적 절정이 "우리의 RISC"로 끝나게 한다.
2. **4장을 두 챕터로 분할하라.** 새 4장 = 레지스터·2의 보수·가산기·Z/C 플래그, 새 5장 = ADC/SBC overflow + 테스트 전략 도입. 후속 챕터 번호는 한 칸씩 밀린다(총 15장 + 부록 4). 두꺼운 한 권 목표에 부합.
3. **인터럽트를 본문에 포함하라.** 8장에 1p 사이드바("인터럽트 첫 만남"), 새 12장(우리의 8비트 RISC) ISA 설계에 인터럽트 벡터 결정 포함, 13장에 "cycle-accurate가 필수가 되는 진짜 이유 — 인터럽트 타이밍" 절 추가.
4. **각 부 도입부에 차별점 환기 페이지(2~3p)를 추가하라.** 부의 정서적 보상 + nand2tetris/Ben Eater와 본 책의 위치 차이 + GitHub 저장소 상태 변화. 5부 구조의 척추가 강화된다.
5. **챕터 1을 12~14p로 축소하고, 챕터별 GitHub 산출물 박스를 표준화하라.** 1장의 von Neumann 발명자 논쟁은 부록 D로 이동. 각 챕터 끝에 "이번 챕터의 GitHub 커밋" 박스(파일·테스트·실행 커맨드)를 통일.
