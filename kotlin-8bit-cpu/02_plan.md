# Kotlin으로 만드는 8비트 CPU 시뮬레이터 — 저술 계획

## 제목 후보

### 1. **코드로 짓는 CPU: Kotlin으로 SAP-1에서 8비트 RISC까지**
- **슬러그:** `kotlin-8bit-cpu` (기본)
- **부제:** 컴퓨터의 어휘를 손으로 다시 익히는 15주
- **톤:** 실습 중심·서사형. "직접 짓는다"가 핵심 메타포. 학부 시절 못 배운 것을 일터에서 다시 마주한 중급 개발자에게 말 거는 톤.
- **포지셔닝:** 실전 코딩 가이드인 동시에, "왜 컴퓨터는 이렇게 동작하는가"를 풀어내는 에세이 톤 기술서. 한국어 시장에서 nand2tetris(영문 추상 게이트)와 Ben Eater(실물 빌드) 사이의 공백을 메운다.

### 2. **8비트 CPU의 해부학: Kotlin으로 SAP-1·SAP-2·RISC를 짓다**
- **슬러그:** `kotlin-8bit-cpu`
- **부제:** 16바이트에서 시작하는 컴퓨터 아키텍처 여행
- **톤:** 좀 더 학술·교과서 톤. "해부학"이라는 단어가 진지함을 강조.
- **포지셔닝:** 학부 컴퓨터구조 보충 교재로 자리 잡고자 하는 포지셔닝. 그러나 토비 문체와 맞지 않을 수 있음 — 너무 딱딱하다.

### 3. **CPU는 어떻게 동작하는가: Kotlin으로 만드는 8비트 컴퓨터**
- **슬러그:** `kotlin-8bit-cpu`
- **부제:** SAP-1에서 8비트 RISC까지, 그리고 그 너머로
- **톤:** "어떻게 동작하는가" 류 입문서 톤. 호기심을 자극하는 질문 형.
- **포지셔닝:** 좀 더 입문층 친화적. 다만 "Kotlin/Java 중급 개발자"라는 대상 독자에게는 너무 가벼워 보일 위험.

**추천: 1번 — "코드로 짓는 CPU: Kotlin으로 SAP-1에서 8비트 RISC까지"**

이유:
- 토비 문체(평어체·청유형·수사적 질문)와 "짓는다"는 동사가 자연스럽게 어울린다
- 부제가 책의 점진적 학습 곡선(SAP-1 → SAP-2 → 8비트 RISC)을 명시한다 — 시장 공백을 한 줄로 드러낸다
- "코드로 짓는다"가 본 책의 차별점(실물 브레드보드도 아니고 추상 게이트도 아닌 — Kotlin 코드로 짓는다)을 압축한다
- 대상 독자(중급 개발자)에게 "지적 호기심을 자극하되 너무 가볍지 않은" 균형을 잡는다

## 내러티브 아크

이 책은 **5부 15장**으로 구성된다. 각 부 시작에는 **2~3페이지의 환기 페이지("여기서 우리는 어디에 있는가")** 가 들어간다. 부의 정서적 보상, nand2tetris/Ben Eater와 본 책의 위치 차이, 그 부 동안 GitHub 저장소가 어떻게 변할지를 1~2단락으로 정리한다. 이 환기가 5부 구조의 척추가 되어 차별점을 매 단계에 회수한다.

**1부: 출발 (1~2장).** "왜 컴퓨터는 이렇게 동작하는가"라는 질문에서 출발한다. 1장은 가볍게 von Neumann과 학습 지도를 12~14페이지로 압축하고, 마지막에 "다음 챕터에서 짤 30줄"의 미리 보기를 둔다. 2장에서 fetch-decode-execute를 Kotlin 30줄로 구현해 손에 잡히게 만든다. 1부를 다 읽고 나면 독자의 GitHub 저장소에 "돌아가는 무엇"이 있다. 이 작은 성공이 나머지 13개 챕터를 끌고 가는 동력이다.

**2부: SAP-1 짓기 (3~7장).** Malvino의 SAP-1을 Kotlin으로 한 모듈씩 짓는다. 3장에서 청사진과 첫 모듈로 시작해, 4장에서 ALU의 핵심(레지스터·2의 보수·가산기·기본 플래그·UByte 결정)을 다룬다. 그리고 **5장 — 본 책의 무덤이자 봉우리 — "ADC/SBC overflow를 테스트로 잡는다"**. 입문자 90%가 막히는 지점을 정면돌파하면서 동시에 본 책의 "테스트 주도 emulator 개발" 전략 전체를 도입한다(kotest, parameterized test, table-driven, test ROM 개념). 6장에서 메모리·버스·컨트롤러로 SAP-1이 처음 돌아간다. 7장에서 첫 어셈블러를 짠다 — 손으로 짠 SAP-1 위에서 자기가 짠 어셈블러로 어셈블한 프로그램이 돌아간다. 여기까지가 책의 첫 큰 봉우리.

**3부: SAP-2로 진화하기 (8~10장).** SAP-1의 한계(16바이트, 5개 명령어)를 직접 부딪힌다. 8장에서 SAP-2의 근본 재설계 의의를 본 다음, 9장에서 메모리를 64KB로 확장하고 조건 점프·CALL/RET·I/O 포트·**인터럽트 첫 만남**·**메모리 매핑 I/O vs 별도 포트 공간**·**ASCII와 8비트 OUT**을 다룬다. 10장에서 컨트롤러를 하드와이어드에서 마이크로코드 ROM 기반으로 재설계하고 디버거를 짠다 — 자기가 만든 CPU에 breakpoint·step-over와 테스트 ROM 검증이 생긴다.

**4부: 8비트 RISC로 점프 (11~13장).** **여기서 부 구조가 재배치되었다.** 먼저 11장에서 실제 8비트 CPU 친척들(8080·6502·Z80·8086)을 같은 Fibonacci로 비교한다 — SAP-2가 8080 subset이었다는 친연성이 즉시 회수되고, 동시에 "CISC 계보의 끝"을 본다. 그러고 나서 12장에서 1980년대 RISC 혁명의 격렬한 논쟁("지금까지 짠 SAP-2가 CISC 계보였다"를 명시)을 따라간 뒤, 13장에서 본 책의 합성 8비트 RISC ISA를 직접 설계한다 — 인터럽트 벡터 결정까지 포함해서. **4부의 절정은 13장 "우리의 RISC가 완성되는 순간"** 이다. SAP-2 버전 Fibonacci보다 짧은 RISC 코드. 패러다임 전환의 감격이 다른 무엇으로도 흐려지지 않는다.

**5부: 정확도와 그 너머 (14~15장).** 마지막 두 챕터는 "장인의 마무리"와 "다음 여정". 14장에서 지금까지 instruction-accurate로 짠 시뮬레이터를 cycle-accurate로 리팩토링하면서 "정확도"의 의미를 손으로 체험한다 — 인터럽트 타이밍이 cycle-accurate가 필수가 되는 진짜 이유로 등장하고, Kotlin/JVM 시뮬레이터의 성능 측정 사이드바도 함께. 15장은 청사진 — CPU → 메모리 시스템 → I/O 디바이스 → 인터럽트 → 커널 → OS로 이어지는 다음 12개월의 로드맵. 부동소수점과 문자열의 부재가 무엇을 의미하는지의 한 단락이 후속 책 떡밥이 된다.

**왜 이 순서인가?** 손에 잡히는 작은 성공(1~2장) → 한 시스템을 끝까지 짓는 경험(SAP-1, 3~7장, 중간에 테스트 전략 도입) → 진화시키는 경험(SAP-2, 8~10장) → CISC 계보를 본 뒤 패러다임을 바꾸는 경험(11~13장) → 정확도를 높이는 장인 정신(14장) → 다음 지평(15장). 매 부마다 환기 페이지와 정서적 절정이 있다. 그리고 매 챕터마다 GitHub 저장소가 한 단계 발전한다 — 각 챕터 끝의 "GitHub 커밋" 박스가 파일·테스트·실행 커맨드를 표준화한다.

#### 1장. 컴퓨터는 왜 이런 모습인가
- **핵심 질문:** 우리가 매일 쓰는 컴퓨터의 모습은 어떻게 결정되었나? 왜 명령어와 데이터가 같은 메모리에 머무는가?
- **주요 내용:**
  - von Neumann 아키텍처: 1945년 EDVAC 보고서가 정한 골격 (§1.1)
  - "uniform memory containing both numbers (data) and orders (instructions)"의 의미
  - Harvard 아키텍처와의 짧은 대비 — nand2tetris의 Hack CPU는 왜 다른 길을 갔는가
  - 왜 8비트인가, 왜 SAP-1에서 시작하는가 — 단순함을 만드는 능력 (§2.2)
  - 본 책의 학습 곡선 지도: 1페이지 다이어그램으로 5부 구조 압축
  - **마지막 절: "다음 챕터의 30줄 미리 보기"** — fetch-decode-execute의 의사 코드 5~6줄
  - von Neumann 발명자 논쟁(§4.6)은 본문에서 빼고 짧은 메모로 부록 D로 안내
- **독자가 얻는 것:** CPU 아키텍처를 손으로 만들어가는 15주의 지도. 다음 챕터의 코드 골격이 머릿속에 미리 그려진다.
- **사용 레퍼런스:** §1.1, §1.3, §2.2, §3.4
- **예상 분량:** 12~14페이지 (이슈 4 반영, 축소)
- **GitHub 산출물:** `chapter-01/00-roadmap.md` (학습 지도 문서)

#### 2장. fetch-decode-execute를 Kotlin 30줄로
- **핵심 질문:** CPU의 가장 작은 심장 박동은 무엇인가? 그것을 Kotlin 코드로 보여줄 수 있는가?
- **주요 내용:**
  - fetch-decode-execute 사이클의 분해 (§1.2)
  - PC → MAR → 메모리 read → MDR → CIR → decode → execute의 흐름
  - Kotlin으로 짜는 미니 CPU — 단 30줄. ADD/HLT 두 명령어, 메모리 4바이트.
  - `while(true)` 루프 안에 fetch/decode/execute 메서드 호출이 어떻게 매핑되는가
  - Kotlin/JVM에서 8비트 값을 어떻게 표현할 것인가 — `Int` + 마스킹 vs `UByte` (§5.1, [커뮤니티-패턴4]). 본격 결정은 4장으로 미루되 첫 인상.
  - 첫 GitHub 커밋: 이 챕터를 다 읽으면 "돌아가는 무엇"이 있다
- **독자가 얻는 것:** "CPU가 동작한다"는 추상적 명제가 자기 코드의 어느 줄에 매핑되는지에 대한 손 감각. 첫 작은 성공.
- **사용 레퍼런스:** §1.2, §5.1, §5.4
- **예상 분량:** 20~24페이지
- **GitHub 산출물:** `chapter-02/MiniCpu.kt`, `MiniCpuTest.kt` (8개 테스트), `./gradlew :chapter-02:run`

### 3부. SAP-2로 진화하기

**3부 환기 페이지(2~3p):** "5개에서 39개로" — SAP-1의 한계와 SAP-2의 의미. nand2tetris의 Hack(16비트 + 2 레지스터)과 본 책의 위치 차이(8비트 + 3 레지스터로 8080 친연성). 3부 끝에 자기 CPU 위에서 함수 호출과 디버거가 돌아간다는 약속.

#### 8장. 16바이트의 한계 — SAP-2는 왜 다시 설계되었는가
- **핵심 질문:** 5개 명령어와 16바이트로는 무엇을 할 수 없는가? Malvino는 왜 SAP-2를 "단순 확장"이 아니라 "근본 재설계"라고 불렀는가?
- **주요 내용:**
  - SAP-1의 한계를 직접 부딪히는 실습: "Fibonacci를 SAP-1으로 짤 수 있는가?" — 못 한다는 발견
  - SAP-2의 변화 일람 (§1.4): 16바이트 → 64KB, 4비트 주소 → 16비트, 5개 → 39개 명령어
  - 누산기 1개 → 범용 레지스터 3개 (A/B/C)
  - 새로운 명령어 클래스 개요: 조건 점프(JZ/JNZ/JM), CALL/RET, I/O 포트 (구체 구현은 9장에)

  - 메타 코멘트: Austin Morlan이 "highly awkward and strange"라고 평한 SAP-2 명세의 한계 (§1.4, §4.5). 본 책은 모호한 부분을 어떻게 해석하는가
- **독자가 얻는 것:** "더 큰 컴퓨터"가 단순히 "수만 늘린 것"이 아니라 "다시 설계한 것"이라는 통찰. ISA 설계라는 개념의 첫 등장.

#### 9장. 조건 분기·서브루틴·I/O — 프로그램다운 프로그램
- **핵심 질문:** 분기가 가능해진 CPU는 어떻게 다른가? `if`와 함수 호출이 기계어 수준에서 어떻게 살아 움직이는가? CPU와 바깥 세계는 어떻게 대화하는가?

### 4부. 8비트 RISC로 점프

**4부 환기 페이지(2~3p):** "패러다임이 바뀐다" — 지금까지 짠 SAP-2가 CISC 계보였다는 명시. 실제 친척들(8080·6502·Z80·8086)을 본 다음 RISC 패러다임 전환으로 넘어간다는 4부의 형 구조. nand2tetris의 Hack은 단일 사이클 RISC-like지만 본 책은 패러다임 전환 자체를 정면으로 다룬다는 위치. 4부 끝에 본 책 고유의 8비트 RISC가 완성된다는 절정 예고.

#### 11장. 같은 Fibonacci, 네 가지 어셈블리 — 8080·6502·Z80·8086 비교

#### 12장. RISC의 약속 — 1980년대의 격렬한 논쟁
- **핵심 질문:** 왜 1980년대의 컴퓨터 과학자들은 "명령어를 줄이자"고 외쳤는가? 그 외침은 어떻게 오늘날 Apple Silicon이 되었는가?
- **주요 내용:**
  - "우리가 지금까지 짠 SAP-2와 어제 본 8080·Z80·6502는 모두 CISC 계보였다"를 첫 단락에서 회수
  - RISC의 정의: load/store, 고정 길이, 단일 사이클, 큰 레지스터 파일, 컴파일러 의존 (§1.5)
  - 역사: IBM 801 (1975, John Cocke) → Berkeley RISC-I (1981, Patterson) → Stanford MIPS (1981, Hennessy) → SPARC/ARM/RISC-V (§1.5)
  - **RISC vs CISC 논쟁** (§4.1): Patterson & Ditzel 1980 vs Clark & Strecker 1980. 양쪽 논점을 균형 있게.
  - 현대의 통합 관점: x86은 internal하게 RISC-like micro-op으로 변환된다. "RISC 정신이 ISA 설계의 표준 어휘가 되었다."
  - 사이드바: Radin 1983의 명문 — "everyone else is busy moving software into hardware and we are clearly moving hardware into software" (§1.5)
  - 다음 챕터에서 짤 본 책 고유의 8비트 RISC ISA의 설계 목표 선언
- **독자가 얻는 것:** "RISC"라는 단어가 면접 질문 단어에서 살아 있는 역사가 된다. ISA 설계라는 활동의 무게감.
- **사용 레퍼런스:** §1.5, §1.6, §4.1
- **예상 분량:** 22~26페이지
- **GitHub 산출물:** `chapter-12/notes/risc-cisc-debate.md` (논쟁 정리), 다음 챕터 ISA 설계 명세 초안

#### 13장. 우리의 8비트 RISC를 설계하다
- **핵심 질문:** RISC 정신을 8비트로 축소하면 어떤 ISA가 나오는가? 본 책의 합성 ISA는 어떤 결정을 내렸는가? 인터럽트는 어떻게 RISC답게 단순할 수 있는가?
- **주요 내용:**
  - 본 책 고유의 합성 8비트 RISC ISA 설계 — **이 책의 가장 창의적인 산물이자 4부의 정서적 절정** (§7.1 한계 항목과 맞물림)
  - 설계 결정: 16개 레지스터(R0~R15), 16비트 고정 길이 명령어, load/store 아키텍처
  - 명령어 집합: 산술(ADD/SUB), 논리(AND/OR/XOR), 시프트, load/store, 분기, JAL/JR
  - 명령어 인코딩: R-type, I-type, J-type (MIPS/RISC-V를 8비트로 축소)
  - **인터럽트 벡터 결정** (이슈 3 반영): "RISC는 단순함이 미덕인데 인터럽트는 어떻게 단순하게 처리할 것인가" — RISC-V의 trap 메커니즘을 8비트로 축소한 합성 설계. 단일 trap vector + cause register 패턴.
  - 어드레싱 모드를 일부러 단순화한 이유 — Berkeley RISC-I의 정신
  - Kotlin으로 ISA 구현: sealed class 계층, `when` 디스패치, 컴파일러가 만든 합성 코드 가독성 비교
  - 자기 RISC ISA용 어셈블러로 Fibonacci 다시 짜기 — SAP-2 버전과 코드 라인 수 비교
  - 11장의 8080·Z80·6502·8086 Fibonacci와 같은 표에 본 책 RISC를 한 줄 추가 — 패러다임 전환의 시각적 증거
- **독자가 얻는 것:** "ISA를 설계한다"는 활동을 손으로 한 번 해본 경험. RISC 원칙이 어떤 트레이드오프를 만드는지. 자기 만의 8비트 RISC 시뮬레이터가 GitHub에 있다.
- **사용 레퍼런스:** §1.5, §3.4, §7.1
- **예상 분량:** 30~34페이지
- **GitHub 산출물:** `chapter-13/risc/{IsaSpec.md, RiscCore.kt, RiscAssembler.kt, TrapVector.kt}`, Fibonacci e2e 테스트, ISA 설계 결정 ADR

### 5부. 정확도와 그 너머

**5부 환기 페이지(2~3p):** "장인의 마무리, 다음 여정" — 14장에서 무엇을 어떻게 손볼지, 15장에서 어디로 손이 향하게 될지. nand2tetris의 Hack은 단일 추상 수준에서 멈추지만 본 책은 정확도의 사다리와 다음 12개월의 청사진을 명시한다는 위치.

#### 14장. instruction-accurate에서 cycle-accurate로
- **핵심 질문:** "정확도"는 무슨 뜻인가? 우리가 짠 시뮬레이터는 얼마나 정확한가, 그리고 더 정확하게 만들려면 무엇이 필요한가? Kotlin/JVM 시뮬레이터는 얼마나 빠른가?
- **주요 내용:**
  - instruction-accurate vs cycle-accurate vs gate-level (§1.8, §4.2)
  - 왜 NES·Game Boy emulator는 cycle-accurate가 필수인가 — PPU/CPU 동기 (§4.2)
  - M-cycle(machine cycle) 추상화: 1 M-cycle = 4 T-cycle. 정확도-성능 절충 (§1.8)
  - **cycle-accurate가 필수가 되는 진짜 이유 — 인터럽트 타이밍** (이슈 3 반영, 1~2p): IRQ가 명령어 중간에 들어오면 어떻게 되는가, Game Boy의 IME 토글 타이밍 사례
  - **본 책 SAP-2 시뮬레이터를 cycle-accurate으로 리팩토링하는 실습**
  - ksim65의 의도적 cycle-exact 포기와 그 정당화 (§3.2, §4.2)
  - 대조 사례: bsnes/higan, BeesNES의 sub-cycle-accurate 철학
  - **사이드바: Kotlin/JVM 시뮬레이터의 성능 측정** (토픽 B 반영, 2~3p): JIT 워밍업, `Int` vs `UByte` 성능 비교, instruction count/sec 벤치, jmh 소개. "가독성을 포기하면 어디까지 빨라지는가"의 정량 답
  - 메타 코멘트: "make it work → make it right → make it fast"의 emudev 합의에서 본 책은 어디까지 가는가 (§2.3)
  - 사이드바: 실물 빌드 vs 시뮬레이터 논쟁과 Ben Eater 추천 (§4.3) — 3단계 학습의 마지막 단계로 안내
- **독자가 얻는 것:** "정확도"라는 단어가 다층적 개념이라는 인식. 자기 시뮬레이터의 정확도 등급을 한 단어로 말할 수 있다. JVM 성능 측정의 도구 어휘.
- **사용 레퍼런스:** §1.8, §2.3, §2.5, §3.2, §4.2, §4.3
- **예상 분량:** 26~30페이지
- **GitHub 산출물:** `chapter-14/sap2-cycle-accurate/`, jmh 벤치마크, M-cycle 정확도 회귀 테스트

#### 15장. 여기서 어디로 — CPU에서 운영체제까지의 청사진
- **핵심 질문:** 8비트 CPU를 짠 다음, 컴퓨터 전체를 짓는 길은 어떻게 생겼는가? 다음 12개월의 학습 로드맵은?
- **주요 내용:**
  - 본 책의 끝에서 독자가 가진 것 정리 — SAP-1·SAP-2·8비트 RISC 시뮬레이터, 어셈블러·디스어셈블러·디버거, 테스트 ROM, cycle-accurate 리팩토링 경험
  - **다음 단계 청사진** (§3.5):
    - 메모리 시스템 — 캐시·MMU·가상 메모리
    - I/O 디바이스 — UART·SPI·인터럽트 컨트롤러 (본 책 9·13장에서 시작한 인터럽트의 확장)
    - 인터럽트와 trap — 동기 vs 비동기
    - 커널 — 스케줄러·메모리 관리·시스템 콜
    - OS — 파일 시스템, 사용자 공간
  - **짧은 단락: 부동소수점과 문자열의 부재가 의미하는 것** (토픽 C 반영, 1p): 8비트 CPU에는 부동소수점도 문자열 처리 명령어도 없다. 소프트웨어 FPU의 역사, 8087 코프로세서, ASCII에서 UTF-8로. "이걸 더하면 다음 챕터가 시작된다"는 후속 책 떡밥.
  - 후속 자료 큐레이션 (§3.5):
    - nand2tetris (NAND → CPU → OS, 게이트 레벨 보강)
    - *Computer Systems: A Programmer's Perspective* — layered abstraction
    - *Operating Systems: Three Easy Pieces* (OSTEP)
    - *Crafting Interpreters* — 두 번 구현 페다고지, 언어 처리계로
    - *Operating Systems from 0 to 1*
  - 한국어 학습자를 위한 자료 큐레이션 — POCU, OKKY, velog 등 (§6.3 한국어 자료, §7.1 한국어 1차 자료 공백)
  - 닫는 단락: "이 책은 시작이다"라는 메시지. Hennessy & Patterson의 "new golden age for computer architecture" 인용 (§2.4)
- **독자가 얻는 것:** 다음 주말에 무엇을 만들지가 정해진다. 후속 학습의 자신감.
- **사용 레퍼런스:** §2.4, §3.5, §6.1, §6.3, §7.1
- **예상 분량:** 20~24페이지

## 변경 이력 (Phase 3 리뷰 반영)

**라운드 1 반영 (2026-05-13):**
- **[Critical 이슈 1]** 4부 챕터 재배치 — 실제 8비트 CPU 비교(현 12장)를 새 11장으로 앞당기고, RISC 역사(현 10장)를 12장, 우리 RISC ISA(현 11장)를 13장으로. 4부 절정이 "우리의 RISC 완성"에 위치하도록 함. 부 경계 재조정: 3부 끝 = SAP-2 + 디버거(10장), 4부 = 실제 비교 → RISC 역사 → 우리 RISC(11~13장).
