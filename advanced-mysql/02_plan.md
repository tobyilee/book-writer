# 자바 스프링 개발자를 위한 고급 MySQL DB 기술 종합서 저술 계획 (v2)

> **버전 메모:** 라운드 1 리뷰(`03_review_log.md`) 반영본. HIGH 3건은 모두 반영했고, SHOULD 4건과 NICE 2건도 본문에 반영했다. 라운드 1과의 변경 요약은 문서 끝 `## v2 변경 요약` 절에 정리한다.

## 1. 제목 후보

1. **DBA처럼 생각하는 스프링 개발자** — 슬러그: `dba-mindset-spring`
   - 부제: *MySQL 8.x를 다루는 13가지 관점*
   - 톤: 단단한 멘토링형. "지금까지는 ORM 뒤에 숨었지만, 이제는 데이터베이스를 직시하자"는 호명.
   - 왜: 독자의 자기 정체성("나는 스프링 개발자다")을 유지하면서도 한 단계 위로 끌어올린다는 약속이 제목 자체에 담긴다.

2. **MySQL을 정면으로 다루기** — 슬러그: `facing-mysql`
   - 부제: *InnoDB, 인덱스, 복제, JPA — 도망치지 않는 13개 챕터*
   - 톤: 도전적·고백적. "MySQL을 그동안 어렴풋이 알고 썼다"는 흔한 불안을 정면에서 받아낸다.
   - 왜: '정면으로'라는 표현이 책의 태도를 곧장 드러낸다. 입문서가 아니라 한 번 갈무리하고 가자는 책이라는 점이 분명하다.

3. **고급 MySQL: 스프링 개발자를 위한 13개 챕터** — 슬러그: `advanced-mysql-spring`
   - 부제: *InnoDB부터 RDS·Aurora 운영까지*
   - 톤: 정통 기술서. 서가에서 키워드로 찾기 쉬운 직설적 제목.
   - 왜: '고급'이라는 단어가 선수 지식 수준을 명확히 신호한다. 검색·추천 시스템 친화적.

**추천:** **1번 — "DBA처럼 생각하는 스프링 개발자"**.
이유는 두 가지다. 첫째, 독자의 출구 상태(자바 개발자의 정체성을 유지하면서 DBA 수준의 의사결정을 내릴 수 있는 사람)를 제목이 정확히 가리킨다. 둘째, '생각하는'이라는 단어가 책의 톤 — 기능 나열이 아니라 왜 그렇게 해야 하는지를 풀어내는 — 과 맞물린다. 본문 평어체("~한다", "~하자")와도 자연스럽게 이어진다.

## 2. 책 특성

- **장르:** 기술 실무서. 입문서가 아니라 "한 번 갈무리하고 가는" 중·고급 종합서. 레퍼런스보다는 멘토링형에 가깝다.
- **분량:** 본문 13개 챕터 + 서장 + 종장 = 15개 단위. 약 290페이지(한국어 EPUB 기준 약 28.4만 자). 챕터별 분량은 §4 합계 표에 명시.
- **난이도:** 중·고급. 선수 지식은 (a) 자바·스프링 부트로 CRUD 애플리케이션을 짜본 경험, (b) `SELECT/INSERT/UPDATE/JOIN` 수준의 SQL, (c) JPA 엔티티를 만들어 본 경험. `EXPLAIN`을 본 적 없거나 트랜잭션 격리수준이라는 단어가 낯선 독자도 1~3장에서 흡수 가능하도록 디자인한다.
- **독자 여정 (1인칭으로):**
  - 진입 — "JPA 뒤에 숨어 살았다. EXPLAIN을 보면 어지럽고, RR/RC 차이가 헷갈리고, Aurora를 쓰라는 말은 듣지만 왜 그래야 하는지 모르겠다. 슬로우 쿼리가 떴을 때 누군가에게 묻지 않고는 손을 못 댄다."
  - 출구 — "이제 EXPLAIN ANALYZE 트리를 안에서 바깥으로 읽는다. 데드락 로그를 받으면 어느 락이 어디서 부딪쳤는지 5분 안에 짚는다. RDS와 Aurora 사이의 분기점을 우리 워크로드 기준으로 설명할 수 있다. JPA와 native SQL 사이의 경계를 내가 그을 줄 안다. 백업·PITR 리허설을 분기마다 돌릴 수 있고, 메이저 업그레이드 일정이 떨어져도 무엇부터 점검할지 안다."
- **스타일:** Toby 평어체. "~한다", "~하자"가 기본. 청유형으로 독자를 끌어들이고("EXPLAIN을 한 줄씩 같이 읽어보자"), 수사적 질문으로 챕터를 열고("InnoDB는 안에서 무슨 일을 벌이고 있을까?"), 상황 가정으로 페이지를 굴린다("월요일 아침, 슬랙에 '결제 페이지가 느려요' 알람이 떴다고 해보자"). 사령형·지시형은 쓰지 않는다.

## 3. 내러티브 아크

1~2장은 **왜 지금 이 책인가**를 푼다. 1장에서 ORM 뒤에 숨어 살던 스프링 개발자가 왜 DB를 직시해야 하는지를 짧고 밀도 있게 풀고, 2장에서 MySQL 8.x가 만든 변화(8.0 신기능과 8.4 LTS의 디폴트 변화)가 왜 이 책의 출간 타이밍을 정당화하는지를 보여준다. 1장은 *지금 이 책*의 정당화에 좁혀 분량을 가볍게 둔다.

3~5장은 **기반**이다. 3장에서 InnoDB 내부(MVCC, redo/undo, 버퍼풀, B+Tree)를 시각적 은유로 깔고, 4장에서 그 위에 트랜잭션·격리수준·**락 메커니즘**을 얹는다. 4장은 락의 *메커니즘*만 다룬다 — 락 패턴은 10장의 영토다. 5장은 인덱스 — 클러스터링/세컨더리/커버링/복합/JSON functional — 를 같은 InnoDB 그림 위에서 풀어낸다. 이 세 장이 책의 뼈대다.

6~7장은 **읽고 다듬는 기술**이다. 6장은 EXPLAIN ANALYZE와 옵티마이저 트레이스로 쿼리를 안에서 바깥으로 읽는 법을, 7장과 8장은 SQL 자체의 표현력을 끌어올린다 — 7장은 윈도우 함수·CTE, 8장은 JSON 데이터 모델링과 인덱싱. 6장이 진단이라면 7·8장은 무기다.

9~11장은 **스프링과의 접점**이다. 9장은 도메인 모델링(애그리거트, soft delete 논쟁, ID 전략)에서 시작해 JPA를 위한 스키마 설계를 다루고, 10·11장은 JPA 챕터를 둘로 나누어 10장에서 영속성 기본기와 함정을, 11장에서 성능·동시성·**락 패턴**을 다룬다. 분산락(named lock)을 비롯한 *애플리케이션 락 패턴*은 모두 11장으로 몰아 4장(메커니즘)과의 책임을 명확히 가른다.

12~13장은 **운영의 세계로**의 도약이다. 12장은 RDS와 Aurora 선택, GTID·반동기·MGR, 파티셔닝과 샤딩의 경계를 그리고, 13장은 관측·보안과 함께 **백업·복구·PITR·DR 일상 운영**까지를 끌어안는다. 토스 SLASH 21 DR 사례는 단순 인프라 선택이 아니라 운영 노하우이므로 13장에 둔다.

14장과 종장은 **통합과 다음 걸음**이다. 14장은 가상의 결제 시스템 한 케이스를 1~13장의 도구로 끝까지 분해해 보여주는 통합 사례이며, 각 절 끝에 "다른 도메인으로 옮긴다면" 한 단락을 박아 도메인 일반화를 본문에서 직접 시연한다. 종장 직전 15장은 메이저 업그레이드 의사결정 체크리스트와 "다음에 무엇을 더 공부할지"의 지도를 그리고, 종장은 다음 주 월요일의 실험과 감사의 말로 닫는다.

처음엔 막연했던 독자가 마지막 장에서는 **"우리 서비스의 다음 분기 DB 운영 로드맵을 내 손으로 그려낼 수 있다"** 고 말할 수 있게 만드는 흐름이다.

## 4. 챕터 목록

### 서장. ORM 뒤에 숨어 살던 우리에게
- **번호·슬러그:** 0 · `prologue`
- **핵심 질문:** *우리는 왜 MySQL을 다시 정면으로 봐야 하는가?*
- **주요 내용:**
  - 스프링 개발자의 일상 — JPA가 만들어준 추상의 편안함
  - 그 편안함이 깨지는 순간들: 슬로우 쿼리 알람, 새벽 데드락, "Aurora로 옮기자"는 회의
  - 이 책의 약속: DBA가 되라는 것이 아니라 **DBA처럼 생각하는 스프링 개발자**가 되자
  - 13개 챕터의 지도와 읽는 순서 가이드
  - 무엇을 다루지 않는가(스코프 선언) — HeatWave, NDB Cluster, 시계열 OLAP은 다루지 않음
- **사용할 리서치 소스:** 0장 토픽 지도, 4장 논쟁점 표, 7장 리서치 한계
- **독자가 얻는 것:** 이 책을 읽을 자기 동기, 읽는 길의 지도
- **예상 분량:** 약 6,000자 (10페이지)

### 1장. MySQL 8.x가 바꾼 풍경 — 이 책이 *지금* 출간되는 이유
- **번호·슬러그:** 1 · `mysql-8x-landscape`
- **핵심 질문:** *왜 지금 한 번 갈무리해야 하는가?*
- **주요 내용:**
  - MySQL 8.0이 가져온 것들 — 윈도우 함수, CTE, JSON 강화, 히스토그램, descending/functional index, EXPLAIN ANALYZE의 등장(개요 수준)
  - 8.0.30 동적 redo 로그 리사이즈처럼 운영을 바꾼 디테일 한 컷
  - 8.4 LTS의 다섯 가지 놀라움 — 이름과 운영 영향만 빠르게 소개, **깊은 분석은 3장 끝(메커니즘 측면)과 15장(업그레이드 측면)에서 다시 본다**고 명시
  - 8.0 → 8.4 다운그레이드 불가의 무게 한 단락
  - 우리가 이 책에서 기준으로 잡는 버전과 그 이유
- **사용할 리서치 소스:** §1.1 (개요만), §2.6, §5.9, Skeema 블로그
- **독자가 얻는 것:** 자신이 운영하는 MySQL 버전이 어디쯤에 있는지의 좌표
- **예상 분량:** 약 12,000자 (12페이지) — *라운드 1 SHOULD 5 반영, 16K → 12K로 압축*

### 2장. InnoDB라는 엔진을 그림으로 이해하자
- **번호·슬러그:** 2 · `innodb-internals`
- **핵심 질문:** *InnoDB는 안에서 무슨 일을 벌이고 있을까?*
- **주요 내용:**
  - 버퍼풀 — 디스크 페이지의 인메모리 캐시, 가용 RAM의 70~80% 휴리스틱
  - WAL과 redo 로그 — "데이터보다 먼저 로그를 쓴다"는 원칙의 의미
  - Undo 로그와 MVCC — 각 row의 `DB_TRX_ID`/`DB_ROLL_PTR`가 만드는 다중 버전
  - B+Tree와 페이지 구조 — Jeremy Cole의 그림을 텍스트와 ASCII로 재구성
  - Adaptive Hash Index — 왜 8.4에서 기본 OFF가 됐는가 (1장에서 예고한 분석을 여기서 풀이)
  - Change buffer와 디스크 IO 절약 패턴, 8.4 기본 OFF의 의미
  - **시각 자료 정책:** 핵심 6개 그림은 ASCII로 우선 시도하고, 시범 작성 결과 표현이 부족하다고 판단되면 EPUB의 SVG/PNG 컬러 다이어그램으로 보강한다 (§5 외부 의존 항목 참고)
- **사용할 리서치 소스:** §1.1, §1.3, 6.4 Jeremy Cole 시리즈, 카카오 InnoDB Log/AHI 글, Skeema 8.4 surprises
- **독자가 얻는 것:** 이후 모든 챕터의 그림이 되는 InnoDB의 멘탈 모델
- **예상 분량:** 약 20,000자 (20페이지)

### 3장. 트랜잭션, 격리수준, 그리고 락의 메커니즘
- **번호·슬러그:** 3 · `transaction-isolation-lock-mechanism`
- **핵심 질문:** *InnoDB는 어떤 락을 언제 어디에 거는가?*
- **챕터 도입부 한 줄:** *"이 챕터는 락의 **메커니즘**을 다룬다. 락을 코드에서 어떻게 거는지(애플리케이션 패턴)는 11장에서 본다."*
- **주요 내용:**
  - 격리수준 4단계와 각각이 허용/차단하는 이상 현상
  - InnoDB의 RR — next-key lock으로 phantom을 막는 방식
  - RC를 선택하면 잃는 것과 얻는 것
  - 레코드 락, 갭 락, 넥스트키 락, 인텐션 락 — 호환성 매트릭스와 함께 한 번에 정리
  - InnoDB가 어떤 SQL에 어떤 락을 거는가 — `SELECT ... FOR UPDATE`, `INSERT`, `UPDATE`, 외래키 검증 시
  - 데드락 사이클 분석 — `SHOW ENGINE INNODB STATUS`/`information_schema.INNODB_TRX`로 사이클을 그려내는 절차
  - 낙관적/비관적 락은 *개념 정의*만 한 단락으로 짚고, *코드 패턴은 11장으로 이관*함을 명시
  - Jepsen 8.0.34 분석과 PostgreSQL 진영의 다른 시각 — RR vs RC + SERIALIZABLE 비교 한 단락, Cahill 2009 SI 논문은 종장 참고문헌으로 안내
- **사용할 리서치 소스:** §1.2, §2.1, §5.1, Jepsen 분석, MySQL 17.7.x 공식 문서, Cahill 2009
- **독자가 얻는 것:** 락 메커니즘에 대한 단단한 멘탈 모델 + 데드락 로그를 읽는 손
- **예상 분량:** 약 22,000자 (22페이지)

### 4장. 인덱스 설계는 데이터 구조 설계다
- **번호·슬러그:** 4 · `index-design`
- **핵심 질문:** *왜 같은 컬럼 조합인데 어떤 순서는 빠르고 어떤 순서는 느린가?*
- **주요 내용:**
  - 클러스터링 인덱스 = PK = 테이블 자체라는 InnoDB의 본질
  - 세컨더리 인덱스의 back-to-table과 그 비용
  - 복합 인덱스와 leftmost prefix rule — equality → range → sort의 순서 휴리스틱
  - 커버링 인덱스로 back-to-table을 없애는 법
  - 프리픽스·descending·functional 인덱스의 자리. **JSON functional index는 인덱스 도구함에 있다는 사실만 짚고, 1mg 5.7초→280ms 사례 같은 *JSON 데이터 모델링* 관점의 본격 풀이는 8장**이라고 명시
  - Invisible index로 안전하게 인덱스 제거 시나리오 테스트
  - 인덱스 3~5개 휴리스틱과 인덱스가 너무 많을 때의 INSERT/UPDATE 비용
- **사용할 리서치 소스:** §1.3, §3.1, §3.2, §5.2, velog 슬로우 쿼리 개선기, OKKY 11초→0.1초
- **독자가 얻는 것:** "이 쿼리에는 어떤 인덱스를 어떻게 만들어야 할까?"에 대한 직관
- **예상 분량:** 약 22,000자 (22페이지)

### 5장. EXPLAIN ANALYZE를 안에서 바깥으로 읽기
- **번호·슬러그:** 5 · `explain-analyze`
- **핵심 질문:** *EXPLAIN ANALYZE 트리를 어디서부터 어디로 읽어야 하는가?*
- **주요 내용:**
  - EXPLAIN과 EXPLAIN ANALYZE의 차이 — 추정 plan과 실제 실행 트리
  - 트리 표기를 안쪽 노드부터 읽는 절차 — actual time/rows의 의미
  - estimate와 actual의 괴리 — 통계 갱신, 히스토그램, optimizer_trace
  - 인덱스가 안 쓰이는 세 가지 원인 — leftmost 위반, 함수 적용, cost 오추정
  - `Using filesort`/`Using temporary`가 나오면 무엇을 다시 보아야 하는가

### 10장. JPA를 넘어 — 성능·동시성·락 패턴·배치
- **번호·슬러그:** 10 · `jpa-performance-concurrency`
- **핵심 질문:** *JPA가 한계를 보일 때 우리는 어디로 내려가야 할까?*
- **챕터 도입부 한 줄:** *"이 챕터는 락의 **애플리케이션 패턴**을 다룬다. 락 메커니즘은 3장에서 보았다."*
- **주요 내용:**
  - 배치 INSERT 100배 — `jdbc.batch_size`, `order_inserts`, `rewriteBatchedStatements` 3종 (우아한형제들 §3.6) + IDENTITY가 막는 이유, SEQUENCE/TABLE 전환
  - 수백만 row UPDATE/DELETE — `WHERE id BETWEEN ? AND ? LIMIT N` 청크 분할 패턴, 락 경합·복제 lag 완화
  - 트랜잭션 경계 — 외부 IO·API 호출을 트랜잭션 밖으로, `flush()`/`clear()`로 JPA pending 동기화

  - 낙관적 락 — `@Version` 코드 패턴과 재시도(`@Retryable` + idempotent payload)
  - 비관적 락 — `@Lock(PESSIMISTIC_WRITE)`/`SELECT ... FOR UPDATE` 코드 패턴
  - **MySQL named lock 분산락** — `GET_LOCK("ad:campaign:42")`, 우아한형제들 §3.3 사례, **락 획득용 커넥션 풀을 비즈니스 풀과 분리**가 왜 핵심인가
  - 분산락 매개체 비교 — MySQL named lock vs Redis Redlock의 분기점
  - 락 순서 일관성 — 정렬된 키 순서로 접근해 데드락 사이클 차단
- **사용할 리서치 소스:** §2.4, §3.3, §3.6, §5.1, §5.5, 우아한형제들 named lock/배치 글, 권남이 위키 User Lock, haon.blog named lock, Baeldung optimistic locking, CodeWiz Spring locking strategies
- **독자가 얻는 것:** JPA 한계 너머의 성능·동시성 패턴 도구함 + 분산락 매개체 선택의 기준
- **예상 분량:** 약 28,000자 (28페이지) — *라운드 1 HIGH 1·2 반영, 락 패턴 책임 이관 + 8장 분할*

## 5. 외부 의존·제약

- **시각 자료 정책 (라운드 1 SHOULD 4 반영):** 2장 핵심 그림은 (1) 버퍼풀 구조, (2) WAL 흐름, (3) MVCC 다중 버전 row, (4) B+Tree 페이지 + 페이지 디렉터리, (5) AHI 동작, (6) Change buffer 등 6개. ASCII로 우선 시도하되, **2장 본문 저술 *전*에 가장 어려운 2개(B+Tree 페이지 구조, MVCC 다중 버전)를 시범 작성해 표현 가능성 검증**을 둔다. 표현이 부족하면 EPUB의 SVG/PNG 컬러 다이어그램을 본문에 포함한다(EPUB 3 지원). 약속 톤도 "그림으로 이해하자"가 ASCII만으로 깨질 위험이 있으면 "그림과 은유로 이해하자"로 톤다운하고 외부 블로그 링크 + 핵심 직관 산문으로 대체한다. 그 외 챕터의 다이어그램(3장 락 호환성 매트릭스, 5장 EXPLAIN ANALYZE 트리, 11장 Aurora 스토리지, 13장 결제 도메인 ERD)은 ASCII로 충분히 표현 가능하다고 판단한다.
