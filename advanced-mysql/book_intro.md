# DBA처럼 생각하는 스프링 개발자

> MySQL 8.x를 다루는 13가지 관점

**JPA 뒤에 숨어 살던 자바 스프링 개발자가, DBA가 되지 않으면서도 DBA처럼 생각할 수 있게 되는 자리까지 함께 걷는 책.** 슬로우 쿼리 알람·새벽 데드락·"Aurora로 옮기자"는 회의 앞에서 더 이상 시니어를 부르지 않아도 되는 손을 손에 쥐자.

## 책 설명

MySQL 8.0이 나오고도 한참이 지났는데, 우리 안에는 여전히 "JPA가 알아서 해주겠지"로 덮어두는 영역이 있다. `EXPLAIN`을 열어도 어디서부터 읽어야 할지 모르겠고, `Repeatable Read`와 `Read Committed`의 차이를 물으면 "기본값이요"라고 답한다. Aurora로 옮기자는 회의에 들어가지만, 왜 옮기는지보다 옮긴 뒤에 뭐가 달라지는지가 더 궁금하다. 이 책은 그 풍경에서 출발한다.

이 책은 입문서가 아니다. SQL `SELECT/INSERT/JOIN` 정도를 쓸 수 있고 JPA 엔티티를 만들어 본 중·고급 개발자가, MySQL을 한 번 갈무리하고 가도록 설계됐다. 16개 단위(서장 + 14장 + 종장)로 약 290페이지. InnoDB 내부의 멘탈 모델에서 시작해 트랜잭션과 락의 메커니즘, 인덱스 설계, `EXPLAIN ANALYZE` 분해, 윈도우 함수·CTE·JSON 인덱싱, 도메인 모델링과 스키마, JPA의 일상 함정과 분산락 패턴, RDS와 Aurora의 분기점, 관측·보안·백업·DR, 그리고 메이저 업그레이드까지 — 한 권으로 묶었다.

본문은 평어체로 흐른다. "이렇게 하라"가 아니라 "같이 살펴보자"의 톤. 챕터마다 한국·해외 실무자들의 공개 경험(우아한형제들, 토스, LINE, 카카오, Vlad Mihalcea, Skeema, Percona 등)을 본문에 녹였고, 각 챕터 끝의 참고 자료에 출처를 명시했다. 마지막 13장은 가상의 결제 시스템 한 건을 1~12장의 도구로 끝까지 분해하는 통합 사례다. 각 절 끝의 "다른 도메인으로 옮긴다면" 단락이 이커머스·SaaS·B2B 배치로 일반화하는 다리를 놓는다.

이 책의 출간 타이밍은 MySQL 8.4 LTS의 다섯 가지 디폴트 변경(AHI/Change Buffer OFF, FK 엄격성, 인증 플러그인 등)이 운영을 흔드는 시점이다. 8.0 → 8.4 다운그레이드가 불가능하다는 무게도 함께 다룬다. 14장의 업그레이드 체크리스트가 다음 분기 일정에 맞닿을 것이다.

## 대상 독자

- 자바·Spring Boot로 CRUD 애플리케이션을 짜본 중·고급 개발자
- JPA 엔티티를 만들어 봤지만 N+1·OFFSET 페이지네이션·배치 INSERT의 함정을 한 번씩은 겪은 사람
- 슬로우 쿼리 알람 앞에서 어디서부터 손대야 할지 막막했던 사람
- "Aurora로 옮기자"는 회의에서 결정 근거를 자기 손으로 정리하고 싶은 사람
- 백업이 있긴 한데 복구 리허설은 안 해본 팀의 한 사람

기본 SQL과 JPA 엔티티 생성 경험이 선수 지식이다. `EXPLAIN`을 본 적 없거나 격리수준이라는 단어가 낯서도 1~3장에서 흡수 가능하도록 설계했다.

## 핵심 약속

- InnoDB의 버퍼풀·redo 로그·MVCC·B+Tree를 한 장의 그림으로 손에 쥔다
- `EXPLAIN ANALYZE` 트리를 안쪽 노드부터 읽어 슬로우 쿼리를 혼자 분해한다
- 격리수준과 락 메커니즘을 그림으로 이해하고, 데드락 로그를 5분 안에 짚는다
- JPA의 N+1, 깊은 OFFSET, 영속성 컨텍스트 폭발 같은 일상 함정과 출구를 안다
- 배치 INSERT 100배, 청크 분할 UPDATE, named lock 분산락 패턴을 코드로 손에 쥔다
- RDS와 Aurora의 분기점을 우리 워크로드 기준으로 직접 정리할 수 있다
- 분기마다 PITR 리허설을 돌릴 수 있는 자신감을 얻는다
- MySQL 8.0 → 8.4 메이저 업그레이드 체크리스트를 손에 쥐고 일정을 받는다
