# 11장 — 같은 Fibonacci, 네 가지 어셈블리

## 핵심 질문

같은 계산을 1974~1978년의 실제 8비트 CPU 네 종류는 어떻게 다르게 표현하는가?
그리고 우리가 손으로 짠 SAP-2는 이들 중 누구의 친척인가?

이 폴더에 있는 네 `.asm` 파일이 그 질문에 직접 답한다.
같은 알고리즘 — 첫 10개 Fibonacci 수 출력 — 을 네 CPU의 실제 어셈블리로 표현했다.

8080에서 시작해 Z80, 6502, 8086으로 읽으면 흐름이 보인다.

## 정량 비교 — 네 칩 한 표로

책 §11.1에서 다룬 수치를 여기에 요약해둔다.

| 항목 | SAP-2 (우리) | Intel 8080 | MOS 6502 | Zilog Z80 | Intel 8086 |
|------|-------------|------------|----------|-----------|------------|
| 출시 | 교육용 | 1974 | 1975 | 1976 | 1978 |
| 트랜지스터 | (시뮬레이션) | ~6,000 | **~3,510** | ~8,500 | **29,000** |
| 8비트 범용 레지스터 | A, B, C | A, B, C, D, E, H, L | A, X, Y | A,B,C,D,E,H,L + alt set | AL/BL/CL/DL (AX·BX·CX·DX 16비트) |
| 명령어 수 | 39 | ~244 | 56 (공식) | ~252 + prefix | ~117 base |
| 어드레싱 모드 | 단순 | 다양 | **13가지** | 다양 | 다양 |
| I/O 모델 | 별도 포트 (OUT) | 별도 포트 (OUT) | **메모리 매핑** | 별도 포트 (OUT) | 별도 포트 (OUT) |
| 출시 가격 | — | **$360** | **$25** | ~$50 | $86.65 (1979) |
| 대표 적용처 | 교실·이 책 | Altair 8800, CP/M | Apple II, C64, NES | TRS-80, MSX, ZX Spectrum | IBM PC (8088 변형) |

트랜지스터 수가 가격을 결정했다.
6502가 8080보다 14배 싸게 팔린 가장 직접적인 이유가 트랜지스터 수의 차이다.

## 코드 라인 수 비교

```
fib_8080.asm  — 코드 라인(주석·공백 제외): 14줄
fib_z80.asm   — 코드 라인: 13줄  (DJNZ 한 줄이 8080의 DCR+JNZ 두 줄을 대체)
fib_6502.asm  — 코드 라인: 17줄  (zero page 경유 로드/스토어가 추가됨)
fib_8086.asm  — 코드 라인: 13줄  (LOOP 한 줄이 CX 감소+분기를 대체)
```

SAP-2의 `fib.sap2`(책 9장 기준)는 약 12줄로 8080과 거의 같다.
8080 subset을 그대로 쓰기 때문이다.

6502만 유일하게 17줄로 길다.
레지스터가 3개뿐이라 변수를 매번 zero page 메모리를 거쳐 이동해야 하기 때문이다.
이것이 "작은 칩 → 코드가 약간 길어진다"는 트레이드오프의 실제 모습이다.

## I/O 모델 비교 — 별도 포트 vs 메모리 매핑

### 별도 포트 I/O (8080 / Z80 / 8086 / SAP-2)

전용 `IN`·`OUT` 명령어가 존재한다.
메모리 주소 공간과 I/O 포트 공간이 분리되어 있다.

```asm
; 8080
OUT 1               ; 포트 번호 1로 출력

; Z80
OUT (1), A          ; 괄호 표기로 포트임을 명시

; 8086
OUT 1, AL           ; 포트, 레지스터 순서

; SAP-2
OUT 1               ; 8080과 동일
```

256개의 포트 공간이 별도로 있고, 메모리 주소와 겹치지 않는다.
하드웨어 설계에서 I/O 장치와 메모리가 같은 버스 선을 공유하되
`/IORQ`(또는 `M/IO#`) 신호로 구분한다.

### 메모리 매핑 I/O (6502)

`IN`·`OUT` 명령어가 없다.
I/O 장치를 그냥 특정 메모리 주소로 할당한다.

```asm
; 6502
STA $D012           ; $D012가 RAM이 아니라 출력 장치에 연결되어 있다
LDA $D011           ; 읽으면 입력 장치에서 값이 온다
```

장점: 명령어 수가 줄어든다. 모든 메모리 연산(LDA, STA, CMP 등)이 I/O에도 그대로 적용된다.
단점: 64KB 주소 공간을 RAM과 I/O 장치가 나눠 써야 한다. Apple II, C64, NES가 각자 다른 메모리 맵을 가진 이유가 이것이다.

## 루프 명령어 비교

루프 카운터를 감소시키고 분기하는 패턴은 어셈블리에서 가장 자주 나타난다.
네 칩이 이 패턴을 어떻게 다루는지가 설계 철학의 차이를 가장 잘 보여준다.

| CPU | 루프 구현 | 명령어 수 | 특이사항 |
|-----|----------|----------|---------|
| SAP-2 | `SUB` + `JNZ` | 2 | 별도 decrement 없음 |
| 8080 | `DCR D` + `JNZ` | 2 | DCR은 전용 1바이트 감소 |
| 6502 | `DEX` + `BNE` | 2 | X 레지스터 전용 감소 |
| **Z80** | **`DJNZ`** | **1** | B 감소 + 분기를 한 명령어로 |
| **8086** | **`LOOP`** | **1** | CX 감소 + 분기를 한 명령어로 |

Z80의 `DJNZ`와 8086의 `LOOP`는 같은 아이디어를 서로 다른 레지스터로 구현했다.
"자주 쓰이는 패턴을 한 명령어로 흡수"하는 CISC의 정신이 여기서 보인다.
8비트 시대에 한 바이트, 한 클록을 줄이는 것이 게임 속도를 바꾸는 차이를 만들었다.

## 덧셈 명령어 비교

```asm
; 8080 (SAP-2도 동일)
ADD C               ; A ← A + C  (대상은 항상 A, 묵시적)

; Z80
ADD A, C            ; A ← A + C  (대상을 명시)

; 6502
CLC                 ; carry를 먼저 지운다  (필수!)
ADC $81             ; A ← A + mem[$81] + carry

; 8086
ADD AL, CL          ; AL ← AL + CL  (8비트 부분 레지스터 사용)
```

6502의 `ADC`(add with carry)는 유독 눈에 띈다.
6502에는 단순 `ADD`가 없고 항상 carry를 같이 더하는 `ADC`만 있다.
단순 덧셈에는 `CLC`로 carry를 0으로 만들고 시작해야 하는 번거로움이 있다.
그러나 2바이트(16비트) 덧셈에서는 강점이 된다.
하위 바이트 `ADC` → carry 자동 전파 → 상위 바이트 `ADC` 순서로 carry를 따로 처리할 필요 없이 멀티바이트 산술이 이어진다.

## 우리 SAP-2의 자리 — 8080의 직계 후예

책 8장에서 Malvino의 문장을 짚었다.

> "The instruction set is patterned after Intel 8080/8085."

이 폴더의 `fib_8080.asm`을 자기 SAP-2 코드와 한 화면에 놓고 보자.
`MVI`, `MOV`, `ADD`, `OUT`, `JNZ`, `HLT` — SAP-2의 핵심 명령어가 8080과 한 글자도 다르지 않다.

SAP-2 → 8080 → (Z80, 8086) 이 순서가 ISA 가족도의 한 줄기다.

```
Intel 8080 (1974, 244개 명령어)
    │
    ├── SAP-2 (교육용 subset, 39개 명령어)   ← 우리 위치
    │
    ├── Intel 8085 (1976, 8080 + 인터럽트 강화)
    │
    ├── Zilog Z80 (1976, 8080 binary 호환 + 확장)
    │        └── TRS-80, ZX Spectrum, MSX, Game Boy 클론
    │
    └── Intel 8086 (1978, 16비트 확장)
             └── IBM PC → 80286 → 80386 → x86-64 → 오늘날 내 노트북
```

Z80은 8080과 **binary 호환**이다.
8080 기계어를 그대로 Z80이 실행한다.
이 호환성이 CP/M 소프트웨어 자산을 수백 종 마이크로컴퓨터로 이식할 수 있게 해줬다.

## 6502 vs Z80 — 두 갈래 철학

같은 8비트 시대를 살았지만 두 칩의 설계 철학은 정반대다.

**Z80 (레지스터 풍부형)**
- CPU 안에 빠른 저장소를 많이 둔다
- 대체 레지스터 셋(A'~L'), IX/IY 인덱스 레지스터
- 코드가 짧고 빠르다. 트랜지스터 비용이 올라간다
- 게임기·홈컴퓨터 운영체제에서 강점

**6502 (메모리 의존형)**
- CPU 내부 레지스터를 3개로 줄인다
- zero page(0x00~0xFF)를 사실상 256개의 느린 레지스터로 활용
- 트랜지스터 수를 줄여 가격을 낮춘다
- 코드가 약간 길어지지만 칩이 싸다 → 대중 시장 장악

1977년 Apple II($1,298), 1982년 Commodore 64($595), 1985년 NES($199) —
6502의 $25 가격이 8비트 시대를 대중 시대로 끌어내린 결정적 요인이다.

---

## 8086으로 가는 16비트 다리

8086은 8비트 시대의 끝이자 16비트 시대의 시작이다.

- AX = AH(상위 8비트) : AL(하위 8비트) — 레지스터를 8비트 혹은 16비트로 혼용
- `ADD AX, BX` 한 명령어로 16비트 Fibonacci 계산 가능
- `LOOP` = `DJNZ`의 8086 버전(CX 레지스터 사용)
- 세그먼트 레지스터로 1MB(20비트) 주소 공간 접근 — 8비트 시대의 64KB 한계를 넘음

8086의 후손이 어디까지 이어지는지 알고 싶으면:

```
8086(1978) → 80286(1982) → 80386(1985, 32비트) → Pentium → Core → 오늘날 x86-64
```

자기 SAP-2 어셈블리에서 손에 잡힌 `MOV`, `ADD`, `JNZ`의 감각이
오늘날 자기 노트북의 x86-64 어셈블리와 직접 이어진다.

## 사이드바 — 6502는 RISC인가 CISC인가?

(이 물음이 12장의 출발점이다.)

RISC의 잣대로 6502를 보면:

| RISC 기준 | 6502 |
|-----------|------|
| load/store 전용 메모리 접근? | 아니오 — ADC가 메모리 주소 직접 받음 |
| 고정 길이 명령어? | 아니오 — 1~3바이트 가변 |
| 단일 사이클 실행? | 아니오 — 2~7사이클 |
| 많은 레지스터(16개+)? | 아니오 — A, X, Y 3개뿐 |
| 단순 어드레싱 모드? | 아니오 — 13가지 |

이 잣대로는 명백한 CISC다.

그러나 6502에는 RISC의 정신이 부분적으로 들어 있다.
명령어 수가 56개로 적고, 자주 쓰이는 명령어가 한 사이클에 가깝게 빠르다.
일부 컴퓨터 역사가들이 6502를 "proto-RISC" 또는 "early RISC-like CISC"라고 부르는 이유다.

그 분류보다 중요한 한 가지 —
8비트 시대에는 RISC와 CISC의 경계가 애초에 없었다.
"RISC"라는 단어 자체가 1980년에 Patterson & Ditzel의 논문이 나오면서 처음 등장한다.
6502가 출시된 1975년에는 모든 칩이 "그냥 칩"이었다.

**그 경계가 선명해지는 1980년대의 이야기가 12장의 주제다.**

## 읽는 순서 제안

1. `comparison/fib_8080.asm` — 자기 SAP-2 코드와 나란히 놓고 비교한다
2. `comparison/fib_z80.asm` — DJNZ 한 줄이 8080의 두 줄을 대체하는 지점을 찾는다
3. `comparison/fib_6502.asm` — OUT 명령어가 사라지고 STA가 I/O를 하는 방식을 본다
4. `comparison/fib_8086.asm` — LOOP가 어떻게 CX를 자동으로 다루는지, AL과 AX의 관계를 확인한다

각 파일 하단의 주석 블록이 해당 CPU의 핵심 차이를 요약해뒀다.
책 11장 본문과 함께 읽으면 맥락이 더 잘 잡힌다.
