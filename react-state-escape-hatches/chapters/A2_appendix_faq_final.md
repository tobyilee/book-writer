# 부록 B — FAQ 30선

스터디나 사내 코드 리뷰에서, 비슷한 질문이 몇 번이고 다시 돌아오는 것을 본다. 한 번 정리해두면 다음에 누군가 같은 길을 헤맬 때 손에 쥐어줄 수 있다. 이 부록은 본문에서 한 번씩 다뤘던 주제 중에서, "그래서 결론이 뭐야?"라고 짧게 다시 묻고 싶을 때 펼쳐 보는 용도다. 답은 일부러 짧게 쓴다. 깊이 들어가야 한다면 안내된 본문 챕터로 다시 돌아가자.

답 안에 인용한 외부 출처는 모두 1장의 참고문헌과 부록 C에 정리되어 있다. 출처가 명시되지 않은 단언은 이 책의 입장임을 분명히 해두었다. 그래야 독자가 나중에 "이거 누가 그랬더라?" 하고 다시 추적할 수 있다.

## 1군. 상태 모양 / 파생값 / 정규화 (1·2·14장 영역)

**Q1. 두 boolean이 동시에 true가 되면 안 되는데, 자꾸 그런 일이 생긴다. 어떻게 막을까?**
A. `isSending`과 `isSent`처럼 서로 모순되는 boolean을 두 개 두는 순간, 언젠가는 둘 다 true가 되거나 둘 다 false가 되는 불가능한 상태가 생긴다. 이런 경우엔 `status: 'typing' | 'sending' | 'sent'` 같은 문자열 enum 하나로 합치는 편이 낫다. 필요한 boolean은 거기서 파생하면 된다. 자세한 패턴은 2장의 "모순 가능한 boolean 합치기"를 참고하자.

**Q2. `firstName`, `lastName`을 받아서 `fullName`을 따로 useState로 보관하면 안 되나?**
A. 안 된다고까지는 못해도, 거의 항상 후회한다. 두 이름 중 하나만 바꾸고 fullName 갱신을 빠뜨리면 그 순간 화면이 거짓말을 한다. 렌더 중에 `const fullName = firstName + ' ' + lastName;`처럼 계산해서 쓰는 편이 낫다. 비싼 계산이라면 `useMemo`로 감싸자. 이 원칙은 2장 "파생값은 렌더 중에 계산하기"에 정리돼 있다.

**Q3. 선택된 항목을 state에 어떻게 저장할까? 객체 자체? id?**
A. id를 저장하는 편이 낫다. 객체 자체를 넣어두면, 원본 리스트가 갱신될 때 state 안의 객체는 옛날 참조로 남는다. `selectedId`만 state로 두고 `items.find(i => i.id === selectedId)`로 끄집어내자. 다중 선택은 `Set<id>`가 깔끔하다. 2장과 14장에서 함께 다뤘다.

**Q4. 깊이 중첩된 트리 데이터를 그대로 setState로 갱신하면 왜 끔찍해지나?**
A. 깊은 spread 체인이 끊임없이 늘어난다. `state.boards[i].columns[j].cards[k].title`을 바꾸려고 네 단계 spread를 적다 보면 어디 한 군데가 빠진다. 정규화(ID 룩업 테이블 + `childIds` 배열)로 펼쳐두면 갱신은 한 키만 다루면 끝난다. 14장에서 칸반 예제로 풀었다.

**Q5. Immer를 쓰면 정규화 안 해도 되지 않나?**
A. Immer는 "변경 표기"의 편의를 주지만, **데이터 모양 자체의 결함**은 가려준다. 깊이 중첩된 모양을 mutate처럼 적을 수 있게 해줄 뿐, 같은 데이터를 두 자리에서 동기화하느라 생기는 버그는 그대로 남는다. 모양을 먼저 정리하고, 그 다음에 Immer로 갱신을 편하게 만드는 순서가 바람직하다. 이는 이 책의 주장이다.

**Q6. 파생값을 위해 매번 useMemo를 써야 하나?**
A. 아니다. 대부분의 파생값은 그냥 변수로 계산하면 충분하다. `useMemo`는 (1) 측정해보니 명백히 비싼 계산이거나 (2) 참조 동일성이 다른 hook의 의존성에 영향을 주는 경우에만 권장한다. "혹시 모르니까"는 메모화 자체의 비용을 무시하는 태도라서 찜찜하다. 14장에서 메모화 판단 기준을 다뤘다.

## 2군. 상태 위치 / 끌어올림 / Context (3·6·7·8장)

**Q7. 어디까지 상태를 끌어올려야 할까?**
A. 그 상태를 **함께 봐야 하는 모든 자식의 가장 가까운 공통 부모**까지가 기본이다. 그 이상 올리면 트리 전체가 그 상태에 결합되고, 더 이상 필요 없어졌는데도 못 내리는 일이 생긴다. Kent C. Dodds의 표현으로 "Lift, then Drop" — 끌어올린 다음, 더 이상 필요 없으면 다시 내려라. 3장과 8장의 콜로케이션 절을 보자.

**Q8. props로 두세 단계만 내려가는 정도면 Context를 안 써도 되나?**
A. 그렇다. Context는 prop drilling이 진짜로 고통일 때만 꺼내자. Kent C. Dodds는 "Don't reach for context too soon"이라고 말한다. 두세 단계는 그냥 prop으로 내리는 편이 추적이 쉽다. 6장에서 다뤘다.

**Q9. Context value가 자주 바뀌면 왜 문제가 되나?**
A. Context는 selective subscription이 없다. value가 바뀌면 해당 Provider 아래의 모든 consumer가 리렌더한다. "이름과 이메일만 쓰는 헤더 컴포넌트"가 "장바구니 카운트가 바뀌었다고" 함께 리렌더된다면 난감하다. 이런 경우엔 7장에서 본 state/dispatch context 분리, 또는 8장의 Zustand/Jotai 같은 도구로 옮기는 편이 낫다.

**Q10. state context와 dispatch context를 왜 따로 두나?**
A. state는 자주 바뀌고, dispatch는 거의 안 바뀐다. 둘을 합쳐 두면, 상태가 바뀔 때마다 dispatch만 쓰는 컴포넌트까지 같이 리렌더된다. 따로 두면 dispatch만 구독하는 컴포넌트는 조용히 살아 있다. 7장의 "Reducer + Context 조합"에서 정확히 이 문제를 풀었다.

**Q11. 모든 전역 상태를 Zustand로 옮기는 게 답인가?**
A. 아니다. "정말로 글로벌인 것만" 위로 보내는 원칙이 먼저다. UI의 토글, 선택 인덱스, 폼 입력 같은 것은 그 화면에 사는 게 맞다. 8장에서 본 도구 선택 휴리스틱 — 자주 안 바뀌는 글로벌 → Context, 작은 앱의 변하는 글로벌 → Zustand, 대규모 팀 표준 → Redux Toolkit, 원자 단위 의존성 → Jotai — 가 출발점이다.

**Q12. Redux는 이제 죽은 기술인가?**
A. 그렇지 않다. PkgPulse "State of React State Management in 2026"이 정리한 것처럼, Zustand가 작은-중간 앱의 기본값으로 자리 잡았지만 Redux Toolkit은 여전히 엔터프라이즈에서 강하다. action 로그·DevTools·미들웨어로 큰 팀의 협업을 떠받치는 도구로 본다면 충분히 살아 있다. 8장에서 비교를 정리했다.

## 3군. useReducer / 도구 선택 (5·8장)

**Q13. useState vs useReducer, 어떤 기준으로 가른다?**
A. "여러 핸들러에 같은 종류의 갱신이 흩어져 있고, 디버깅 가치가 충분한가"가 임계점이다. 작은 폼은 useState가 더 명확하다. 협업 가능한 task list, 위저드, 카트, 보드처럼 사용자 의도가 분기되는 곳은 reducer가 깔끔하다. 5장에서 다뤘다.

**Q14. action을 너무 잘게 쪼개면 왜 손해인가?**
A. `setName`과 `setEmail`을 reducer로 옮겨 `SET_NAME`, `SET_EMAIL` action을 만들면 결국 useState 두 개와 다르지 않다. action은 "사용자 의도 단위" — `'submitted'`, `'changed_draft'`, `'reset_wizard'` — 로 작성하는 편이 낫다. 디버그 로그가 의미를 갖는 순간 reducer가 보상을 준다. 5장 후반부의 "사용자 의도 단위 액션" 절을 참고하자.

**Q15. Redux Toolkit의 createSlice를 useReducer 대신 쓰면 어떤가?**
A. 단일 트리에서는 그대로 useReducer 결과와 비슷하지만, RTK는 Immer 통합·DevTools·미들웨어를 함께 들고 온다. 즉 리듀서 스타일이 좋아서가 아니라 **부수 인프라가 필요해서** RTK를 고르는 것이 정직한 선택이다. 8장에서 도구 선택 시 어떤 기준으로 합리화했는지 보자.

**Q16. 서버 응답을 Redux/Context에 다 박아두면 왜 매번 깨지나?**
A. 서버 데이터는 비동기·공유·외부 변경·캐시 무효화가 본질이다. 이걸 손으로 reducer에 적기 시작하면 사실상 query cache를 직접 만들고 있는 셈이다. TanStack Query 공식 문서가 명시적으로 "이건 우리 영역"이라고 선언한 이유다. 서버 상태와 UI 상태는 다른 도구로 다루는 편이 바람직하다. 8장 후반부에서 정리했다.

**Q17. Jotai와 Zustand 중 어느 쪽이 정답인가?**
A. 정답은 없고, 사고 단위가 다르다. Zustand는 "store 단위로" 묶고, Jotai는 "atom 단위로" 쪼갠다. 컴포넌트가 매우 많고 각자 다른 부분만 구독해야 하는 디자인이라면 Jotai의 atom 단위 구독이 자연스럽다. "한 화면에 묶이는 큰 덩어리"라면 Zustand가 단순하다. inhaq의 2026 비교 글이 좋은 출발점이다. 8장에서 다뤘다.

**Q18. useReducer 안에서 비동기 호출을 직접 호출해도 되나?**
A. 안 된다. reducer는 순수 함수여야 한다. 비동기 호출은 핸들러나 effect에서 하고, reducer에는 결과 action만 dispatch한다. 5장에서 본 패턴이다.

## 4군. Ref / Effect / cleanup (10·11·15장)

**Q19. ref와 state의 차이를 한 줄로 요약하면?**
A. **state는 변경이 렌더를 부른다, ref는 안 부른다.** 화면에 보이는 값은 state, 화면과 무관한 값(타이머 ID, 이전 값, DOM 노드)은 ref. 10장에서 한국 커뮤니티의 "21회 렌더 → 2회 렌더" 비교(velog/skawnkk)도 다뤘는데, 거기서 핵심은 "동적 뷰가 필요한 경우엔 ref가 답이 아니다"라는 단서였다.

**Q20. effect에 cleanup이 의무인가? 빠뜨리면 무슨 일이 생기나?**
A. 외부 시스템과 동기화하는 effect라면 의무에 가깝다. 채팅 연결을 끊지 않으면 방을 바꿀 때 좀비 연결이 남는다. 개발 모드의 더블 인보케이션은 일부러 cleanup을 검증한다 — 이걸 짜증으로 받지 말고 "고마운 검사"로 받는 편이 낫다. 11장 "Effect는 동기화다" 절에서 정리했다.

**Q21. effect 안에서 fetch를 하는데 race condition이 자주 생긴다.**
A. 빠르게 바뀌는 query에 대해 늦게 도착한 응답이 최신 응답을 덮어쓰는 고전적 함정이다. `let ignore = false; ... return () => { ignore = true; }` 패턴이 최소 처방. 다만 race condition·재시도·캐시·낙관적 업데이트까지 손으로 쌓다 보면 결국 query 라이브러리를 만들고 있다. 그 시점이 오면 TanStack Query/SWR로 옮기는 편이 낫다. 11장과 8장에서 함께 다뤘다.

**Q22. effect 의존성 배열에 객체나 함수를 넣어도 되나?**
A. 가능은 하지만 거의 항상 후회한다. 매 렌더마다 새 참조가 만들어져 effect가 폭주한다. 객체를 effect 안으로 옮기거나, 원시값(`roomId`, `serverUrl`)으로 분해하거나, 함수형 setState로 의존성을 끊자. 13장에서 통째로 다룬 주제다.

**Q23. linter `exhaustive-deps`를 끄면 안 되나?**
A. 끄면 그 순간 stale closure 버그가 생긴다. Dan Abramov가 말한 대로, 의존성 배열은 hack이 아니라 "동기화 계약"이다. 거짓말하면 잘못된 값이 잡힌다. 끄지 말고 코드를 바꿔서 의존성을 진짜로 없애는 편이 바람직하다. 13장에서 다섯 가지 처방을 정리했다.

**Q24. `useSyncExternalStore`는 언제 쓰나?**
A. React 외부에 사는 store(브라우저 API, 비-React 라이브러리)를 구독할 때다. effect로 직접 `addEventListener`하면 SSR/concurrent 환경에서 tearing 같은 미묘한 버그가 생긴다. `online/offline` 같은 단순한 것도 `useSyncExternalStore`로 옮기는 편이 안전하다. 15장에서 다뤘다.

## 5군. "필요 없을 때" / 의존성 (12·13장)

**Q25. effect를 쓸지 말지, 가장 빠른 판단 기준은?**
A. 두 가지 질문을 차례로 던진다. (1) "이걸 렌더 중에 계산할 수 있는가?" — 그러면 useMemo나 그냥 변수다. (2) "이건 사용자 이벤트 때문에 일어나는 일인가?" — 그러면 이벤트 핸들러다. 둘 다 아니면 그제야 effect다. Dan Abramov가 메타 내부에서 샘플링한 결과 약 46%의 useEffect가 불필요했다는 보고(HN 토론)는 이 가설과 잘 들어맞는다. 12장의 절정 챕터를 참고하자.

**Q26. prop이 바뀔 때 state를 리셋해야 하는데 effect로 하면 왜 나쁜가?**
A. 한 박자 늦은 깜빡임이 생긴다 — "이전 값으로 한 번 렌더" 후 "리셋된 값으로 다시 렌더". 사용자가 0.x초 동안 잘못된 화면을 본다. 처방은 두 가지다. (1) 정체성으로 다루는 게 자연스러우면 `<Chat key={contactId} />`처럼 `key`로 리셋. (2) 일부만 리셋해야 하면 `[prevX, setPrevX] = useState(x); if (x !== prev) { setPrevX(x); setSel(null); }` 패턴. 4장과 12장에서 함께 다뤘다.

**Q27. 자식이 부모에게 변경을 알리려고 effect를 쓰면 왜 무한 루프가 나나?**
A. `useEffect(() => onChange(value))` 같은 코드는 "값이 바뀌면 부모에게 알린다"가 아니라 "렌더가 끝나면 부모에게 알린다"가 되기 쉽다. 부모 state가 바뀌면 자식이 다시 렌더되고, 그러면 effect가 또 실행된다. 같은 이벤트 핸들러 안에서 자기 state와 부모 콜백을 함께 호출하거나, fully controlled로 만들어 자식이 자체 state를 안 두게 하는 편이 낫다. 12장에서 다뤘다.

**Q28. 앱 초기화를 effect 안에서 하면 왜 두 번 실행되나?**
A. 개발 모드의 Strict Mode가 일부러 두 번 실행해서 안전성을 검증한다. 인증 토큰 두 번 발급 같은 것은 그래서 끔찍한 일이다. 처방은 (1) 모듈 최상단에서 한 번만 실행되는 코드로 옮기거나 (2) `if (didInit.current) return; didInit.current = true;` 가드를 두는 것. 그러나 더 근본적으로는 "정말로 effect의 일인가?"를 다시 묻는 편이 좋다. 12장과 11장에서 다뤘다.

**Q29. cascading effect — A→state→B→state→A — 는 왜 만들지 말라고 하나?**
A. 디버깅 지옥이기 때문이다. 어떤 state 변화가 어떤 effect를 깨우는지가 트리를 따라 흩어지면, 어디서부터 어디까지가 원인이고 결과인지 추적이 어려워진다. 한 핸들러 안에서 모든 갱신을 함께 끝내거나 reducer로 모으는 편이 바람직하다. 12장에서 본 안티패턴이다.

## 6군. 커스텀 훅 / React 19 (14장 + 부록 A)

**Q30. 커스텀 훅을 언제 추출해야 할까?**
A. Robin Wieruch의 휴리스틱은 "두 번 이상 같은 패턴이 보일 때." 미리 추출하지 말자. 이른 추상화는 나중에 쓰임새가 갈라질 때 더 큰 비용으로 돌아온다. 그리고 추출 전에 (a) 컴포넌트 합성으로 풀리는지, (b) JSX를 children으로 넘겨 풀리는지 먼저 시도하는 편이 낫다. 14장에서 정리했다.

**Q31. `useMount` 같은 라이프사이클 래퍼는 왜 만들지 말라고 하나?**
A. Effect의 "동기화" 의미를 흐리기 때문이다. `useMount(() => fetchData())`는 "마운트 시 한 번"이라는 라이프사이클 신호를 다시 도입한다. React 팀이 11장 내내 우리에게서 빼앗으려 했던 그 사고방식이다. 동기화 관점이 흐려진 코드는 의존성을 정직하게 따지지 못한다. 11장과 14장의 공식 문서 입장을 따르는 편이 바람직하다.

**Q32. React 19에서 `forwardRef`가 졸업했다는데 무슨 뜻인가?**
A. 함수 컴포넌트가 `ref`를 그냥 prop처럼 받을 수 있게 됐다는 뜻이다. 예전엔 `forwardRef`로 감싸야 했지만, 19부터는 `function MyInput({ ref }) { ... }`로 적는다. 부록 A에 마이그레이션 노트를 정리해두었다. 라이브러리 저자가 아닌 한, 대부분의 컴포넌트는 자연스럽게 단순해진다.

**Q33. `useEffectEvent`는 안정 API인가? 지금 써도 되나?**
A. 2026년 4월 시점에도 카나리/실험 단계다. 운영 코드에 도입할 때는 React 버전 합의·롤백 계획을 함께 두는 편이 낫다. 그러나 어휘로서의 가치 — "반응형(reactive) 코드와 비반응형 코드를 분리한다" — 는 이미 우리 사고에 들어와 있다. 13장에서 의존성 처방으로, 부록 A에서 위상 변화로 다뤘다.

**Q34. 커스텀 훅 반환은 객체와 배열 중 뭐가 좋은가?**
A. Wieruch는 "다중 값은 배열" — 호출자가 자유롭게 이름을 지을 수 있어서 — 를 권한다. 다만 반환 항목이 셋 이상이고 의미 있는 이름이 필요하다면 객체가 읽기 좋다. 어느 한쪽이 정답이 아니라 "호출자가 이름을 다시 짓고 싶은 빈도"가 결정한다. 이 책의 권장은 두세 항목까지 배열, 그 이상은 객체다. 14장에서 다뤘다.

**Q35. `use(promise)`는 지금 학습해야 하나?**
A. Suspense·React Server Components와 함께 묶이는 새로운 어휘다. 본문에서 깊게 다루지 않은 이유는, 이 책의 범위인 "Managing State + Escape Hatches" 안에서는 부분적으로만 등장하기 때문이다. 부록 A에서 미리 보기 수준으로 짚었고, 본격 학습은 부록 C의 "다음 학습 경로"를 따라가자. 학습 순서로는 Suspense → RSC → `use(promise)`가 자연스럽다.

## 마무리

여기에 적힌 답은 어디까지나 "자주 묻는다"라는 빈도에 의해 모인 것이다. 즉 깊이가 아니라 폭이다. 답 하나하나가 본문 한 챕터의 압축이라서, 짧게 보고 흐름이 잡히지 않는다면 본문으로 다시 돌아가는 편이 낫다. 그리고 동료가 같은 질문을 또 던지면, 이 표를 펼쳐서 같이 읽자. 같은 자리에서 같은 질문을 두 번 듣는 건 죄가 아니다. 답을 찾아 함께 본문으로 돌아갈 수 있다면, 그게 가장 빠른 학습이다.

다음 부록 C에서는 책 전반에 인용한 1차 출처와 그 다음으로 갈 학습 경로를 정리한다. 본문에서 미처 다 못 파고든 영역의 길잡이가 될 것이다.
