# 0장. 출발점 점검 — 우리가 안다고 믿는 것

## 들어가는 이야기

페어 프로그래밍을 하다 보면 같은 장면이 반복된다. 한쪽이 코드를 짜다가 멈춘다. "어, 잠깐만요. 이게 왜 이렇게 동작하지?" 옆자리에서 보던 사람도 같이 갸우뚱한다. 두 사람 다 React를 3년 넘게 쓴 사람들이다. 둘 다 공식 문서를 처음부터 끝까지 읽은 적이 있다. 그런데도 한 줄짜리 코드 앞에서 한참을 머문다. 정확히는, 한 줄을 보고 있는데 그 한 줄이 실제로 어떻게 실행될지가 머릿속에서 그려지지 않는다.

이 장면의 본질은 무력감이 아니다. 오히려 **우리가 안다고 믿는 것과 실제로 아는 것 사이의 작은 틈**이다. 그 틈은 평소엔 잘 보이지 않다가, 어느 날 갑자기 무한 루프로, race condition으로, 사라지지 않는 stale state로 돌아온다. 그 틈이 평소에 안 보이는 이유는 단순하다. 우리는 평소엔 잘 굴러가는 코드만 만나니까. 동작하는 코드는 우리 멘탈 모델을 검증하지 않는다. 깨질 때만 검증한다. 그래서 깨진 자리에 서 있는 우리는 자주 당황한다.

좋은 소식이 하나 있다. 그 틈은 "내가 어디에서 흔들리는지"를 한번 정직하게 마주하는 것만으로도 절반 이상 좁아진다. 이 장은 그 절반을 위한 자가 진단이다. 점수를 매기려는 게 아니다. 어느 영역에서 가장 흔들리는지를 자기 자신에게 보고하기 위한 것이다. 진단 결과에 따라 어느 챕터에서 가장 큰 보강을 받게 될지도 마지막 표에서 함께 정리한다.

여덟 문항이다. 각 문항은 짧은 코드 스니펫과 짧은 질문 한 줄로 시작한다. 답이 곧장 떠오르더라도 한 번 멈추자. **정답을 가리고 30초 정도 생각해보자.** 30초가 길게 느껴질 수도 있지만, 자기 멘탈 모델을 정직하게 보려면 그 30초가 꼭 필요하다. 떠오른 생각을 한 줄로 적어두면 더 좋다. 적어두지 않으면 풀이를 본 뒤에 "아, 나도 그렇게 생각했어"라며 자기 자신을 속이게 되니까. 이 책에서 가장 긴 30초가 되겠지만, 가장 값진 30초이기도 하다.

진단을 통과 의례처럼 빠르게 넘기지는 말자. 흐린 자리를 또렷하게 보는 일은 한 번에 끝나지 않는다. 책 전체를 다 읽고 다시 이 장으로 돌아왔을 때, 같은 문항을 다시 풀어보면 자기가 어떻게 변했는지가 보인다. 그래서 이 자가 진단은 책의 시작이자 책의 끝이기도 하다.

자, 시작해보자.

## 1번 — 상태 enum이 두 boolean보다 나은 이유

다음 두 코드 중 어느 쪽이 더 견고한가? 그리고 그 이유는 무엇인가?

```tsx
// A. boolean 두 개
const [isLoading, setIsLoading] = useState<boolean>(false);
const [isError, setIsError] = useState<boolean>(false);

// B. 상태 enum 한 개
type Status = 'idle' | 'loading' | 'error' | 'success';
const [status, setStatus] = useState<Status>('idle');
```

*정답을 가리고 30초 생각해보자.*

### 풀이

답은 B다. 이유는 단 하나다. **A에는 "있어서는 안 되는 모양"이 표현 가능하다.** `isLoading === true`이면서 `isError === true`인 상태. `isLoading === true`인데 응답 데이터까지 같이 채워져 있는 상태. 이런 모순은 코드 어딘가에서 set 호출을 한 번만 빠뜨려도 발생한다.

B는 그런 모순을 타입 차원에서 미리 막는다. `'loading'`과 `'error'`는 동시에 존재할 수 없다. 모델링 단계에서 잘못된 모양을 표현 불가능하게 만든다 — 이것이 React 팀이 "Choosing the State Structure"에서 강조하는 핵심이다.

조금 더 풀어보자. A 형태로 코드를 짜기 시작하면 핸들러는 늘 두 군데를 동시에 챙겨야 한다. fetch가 시작되면 `setIsLoading(true)`와 `setIsError(false)`를 같이 부르고, 실패하면 `setIsLoading(false)`와 `setIsError(true)`를 같이 부른다. 이 "같이 부르기"가 한 군데라도 빠지면 그 자리에서 모순이 시작된다. 코드가 자랄수록 빠뜨릴 자리는 늘어난다. B는 그 부담을 아예 없앤다. 한 번에 한 값만 set하면 되고, 그것이 곧 일관된 화면의 보장이다. 보일러플레이트가 줄어드는 게 아니라, 신경 써야 할 동시 조건이 줄어드는 것이다.

여기서 흔들렸다면 → **1장(UI는 상태의 함수다)** 과 **2장(무엇을 state에 두고 무엇을 두지 말 것인가)** 가 가장 큰 보강을 줄 영역이다.

## 2번 — `setState`를 같은 핸들러에서 세 번

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
    setCount(count + 1);
    setCount(count + 1);
  }

  return <button onClick={handleClick}>count: {count}</button>;
}
```

버튼을 한 번 클릭하면 `count`는 얼마가 되는가? 그리고 `setState`는 동기인가, 비동기인가?

*정답을 가리고 30초 생각해보자.*

### 풀이

`count`는 1만 증가한다. 3이 아니다. "비동기"라고 흔히 답하지만, 정확히는 **"이번 렌더의 스냅샷을 읽는다"**가 본질이다.

핸들러가 실행되는 동안 `count`는 이번 렌더에 캡처된 0이라는 값이다. 세 번의 `setCount(count + 1)`은 모두 `setCount(0 + 1)`, 즉 `setCount(1)`을 의미한다. React는 다음 렌더의 state를 1로 결정하고 끝낸다.

3을 만들고 싶다면 함수형 업데이트로 누적하자.

```tsx
setCount(prev => prev + 1);
setCount(prev => prev + 1);
setCount(prev => prev + 1);
```

이 형태는 "이전 값으로부터 어떻게 갱신할지"의 의도를 React에 위임한다. React는 큐에 쌓인 업데이트들을 순서대로 적용한다. 첫 번째 호출에서 `prev`는 0이라 결과는 1, 두 번째 호출에서 `prev`는 1이라 결과는 2, 세 번째에서 `prev`는 2라 결과는 3이다. 핸들러 안에서 우리가 직접 "현재 값"을 읽지 않고 "다음 값을 어떻게 만들지"의 함수만 던지는 것 — 이것이 함수형 업데이트의 정수다.

"비동기"라는 표현은 그래서 절반만 맞다. 실제로는 "렌더 단위로 묶어서 처리한다"는 표현이 본질에 가깝다. 한 핸들러 안의 모든 set 호출은 같은 렌더의 스냅샷을 본다. 그리고 다음 렌더가 시작되기 전에 한꺼번에 적용된다. 이 메커니즘을 "batching"이라고 부르며, React 18 이후로는 promise 안이나 setTimeout 안에서도 같은 batching이 적용된다. 즉 "비동기 콜백 안이라 따로따로 적용되지 않을까"라는 옛 직관도 더 이상 맞지 않는다.

여기서 흔들렸다면 → **1장(UI는 상태의 함수다)** 의 "렌더는 스냅샷이다" 절과 **12장(의존성 다루기)** 의 함수형 setState 활용을 주의 깊게 읽자.

## 3번 — 같은 자리, 같은 타입, 다른 사용자

```tsx
function App() {
  const [showAlice, setShowAlice] = useState(true);

  return (
    <>
      {showAlice ? (
        <Profile name="Alice" />
      ) : (
        <Profile name="Bob" />
      )}
      <button onClick={() => setShowAlice(s => !s)}>전환</button>
    </>
  );
}

function Profile({ name }: { name: string }) {
  const [draft, setDraft] = useState('');
  return (
    <>
      <h1>{name}</h1>
      <input value={draft} onChange={e => setDraft(e.target.value)} />
    </>
  );
}
```

Alice 화면에서 입력란에 "안녕"이라고 적어 두었다. 버튼을 눌러 Bob으로 전환했다. 입력란의 `draft`는 어떻게 되는가?

*정답을 가리고 30초 생각해보자.*

### 풀이

**`draft`는 그대로 보존된다.** Bob 화면으로 넘어가도 입력란에 "안녕"이 남아 있다. 의도와 다르다면 끔찍한 일이다.

이유는 React가 컴포넌트의 정체성을 **트리 위치 + 컴포넌트 타입**으로 판단하기 때문이다. 자리가 같고 타입이 같으면 — `Profile`이 같은 자리에 있으니 — React는 "같은 컴포넌트가 prop만 바뀐 상태"로 본다. 그래서 state를 보존한다. 사용자가 다른 사람으로 바뀌었다는 사실은 React가 알 길이 없다.

의도적으로 리셋하려면 React에게 "다른 컴포넌트야"라고 알려주어야 한다. `key`가 그 도구다.

```tsx
<Profile key={name} name={name} />
```

`key`가 바뀌면 React는 새 컴포넌트로 인식해 state를 새로 만든다. 같은 `Profile` 타입이라도 React가 보기엔 "위치는 같은데 정체성이 다른 컴포넌트"가 된 셈이다. 이전 사용자의 입력 흔적이 다음 사용자에게 새어나가지 않는다.

흔히 빠지는 함정은 두 가지다. 하나는 `key`로 배열의 index를 쓰는 것 — 항목 순서가 바뀌면 state가 엉뚱한 자리로 끌려간다. 다른 하나는 prop이 바뀔 때마다 effect로 state를 리셋하려는 시도다. 이건 한 박자 늦은 깜빡임을 만든다. prop이 바뀌어 컴포넌트가 한 번 렌더된 다음에야 effect가 실행되어 state가 바뀌고, 그 결과로 또 한 번 렌더가 일어난다. 사용자 입장에서는 "이전 사용자의 정보가 잠깐 보였다 사라지는" 끔찍한 자리다. `key`로 끝낼 일을 effect로 푸는 패턴은 4장에서 정면으로 다룬다.

여기서 흔들렸다면 → **4장(보존과 리셋 — `key`가 정말로 하는 일)** 이 정확히 이 지점을 다룬다.

## 4번 — `useEffect`로 mount 시 1회 fetch

```tsx
function SearchResults({ query }: { query: string }) {
  const [results, setResults] = useState<string[]>([]);

  useEffect(() => {
    fetch(`/api/search?q=${query}`)
      .then(res => res.json())
      .then(data => setResults(data));
  }, []);

  return <ul>{results.map(r => <li key={r}>{r}</li>)}</ul>;
}
```

이 코드의 위험은 무엇인가? 한 가지가 아니라 두 가지 이상을 떠올려보자.

*정답을 가리고 30초 생각해보자.*

### 풀이

위험은 셋 다 있다.

1. **의존성 거짓말.** 의존성 배열이 비어 있는데 본문에서는 `query`를 쓰고 있다. lint가 경고하는 자리다. `query`가 바뀌어도 effect는 다시 실행되지 않는다. 첫 검색어로 영원히 고정된다.
2. **race condition.** 사용자가 빠르게 검색어를 바꾸면 늦게 도착한 응답이 최신 응답을 덮어쓸 수 있다. 사용자 화면에는 한참 전에 친 검색어의 결과가 떠 있게 된다. 찜찜하다는 말로는 부족하다.
3. **double invocation.** 개발 모드(StrictMode)에서 effect는 두 번 실행된다. cleanup이 없으면 같은 fetch가 두 번 일어나고, 두 응답 중 어느 쪽이 살아남을지가 모호해진다.

처방을 한 줄씩 정리하면 — `query`를 의존성에 넣고, `let ignore = false`로 ignore 가드를 두고, cleanup에서 `ignore = true`로 표시한다.

```tsx
useEffect(() => {
  let ignore = false;
  fetch(`/api/search?q=${query}`)
    .then(res => res.json())
    .then(data => { if (!ignore) setResults(data); });
  return () => { ignore = true; };
}, [query]);
```

이 패턴은 외워두자. `query`가 바뀌면 React가 cleanup을 먼저 호출해 이전 effect의 `ignore`를 `true`로 만든다. 이전에 시작된 fetch가 늦게 도착해도 `if (!ignore)` 가드가 set을 막는다. 새 effect는 새로운 `ignore` 변수를 자기 클로저에 가지고 시작한다. 변수 한 줄로 race condition이 닫힌다는 사실이 처음엔 마법처럼 보이지만, 익숙해지고 나면 가장 단단한 패턴 중 하나다.

다만 한 가지 단서를 잊지 말자. 실무에서는 직접 effect로 fetch를 짜기보다 TanStack Query나 SWR 같은 라이브러리로 옮기는 것이 표준이다. 이런 라이브러리들은 race condition, cache, 재시도, 낙관적 업데이트를 모두 처리해준다. 본문에서는 effect의 동작 원리를 이해하기 위해 직접 짜는 형태를 보여주지만, 한 번 손에 익으면 그 다음부터는 라이브러리 쪽으로 옮기는 편이 낫다.

여기서 흔들렸다면 → **10장(Effect로 외부 시스템 동기화)** 과 **11장(Effect 라이프사이클)** 이 핵심 챕터다.

## 5번 — `useRef`는 `useState`보다 "빠른가"

다음 두 입력 폼 중 어느 쪽이 더 빠른가? 그리고 어느 쪽이 더 적절한가?

```tsx
// A. useState
function FormA() {
  const [value, setValue] = useState('');
  return <input value={value} onChange={e => setValue(e.target.value)} />;
}

// B. useRef
function FormB() {
  const ref = useRef<HTMLInputElement>(null);
  return <input ref={ref} />;
}
```

*정답을 가리고 30초 생각해보자.*

### 풀이

질문 자체가 함정이다. **"빠르다"의 정의를 묻지 않으면 답할 수 없다.**

렌더 횟수만 보면 B가 적게 일어난다. 한국 커뮤니티의 정량 비교에서도 같은 입력 폼이 useState 기준 21회 렌더, useRef 기준 2회 렌더로 줄어든 사례가 보고된다. 그러나 그 자체가 "빠름"의 정의는 아니다. 화면에 글자가 찍히는 속도를 사용자가 느끼는 데 미치는 영향은 보통 미미하다.

문제는 다른 자리에서 터진다. B는 입력값을 화면에 다시 그려야 하는 순간 — 입력 길이에 따라 버튼을 활성/비활성하거나, 검색 제안을 띄우거나, 글자 수 카운터를 보여주려는 순간 — 곧장 무너진다. ref는 변경해도 리렌더를 트리거하지 않으니 화면에 반영되지 않는다.

원칙을 한 줄로 잡자. **렌더 결과에 영향을 주는 값은 state, 그렇지 않은 값(타이머 ID, 외부 라이브러리 인스턴스, 이전 값 추적 등)은 ref.** "리렌더가 줄어드니까 ref가 좋다"는 직관은 거의 항상 잘못된 출구로 우리를 이끈다.

조금 더 풀어보자. ref는 "React 데이터 흐름 바깥으로 잠깐 나가는 escape hatch"다. 평소엔 React 안에서 props/state로 데이터가 흐르고, 바깥 세계 — 브라우저 API, 비-React 라이브러리, 타이머 ID, DOM 노드 — 와 만나는 경계에서만 ref를 꺼내는 편이 낫다. "성능 최적화"를 ref의 첫 번째 동기로 삼는 것은 대개 잘못된 출발점이다. 입력 폼이 21회 렌더 vs 2회 렌더라는 정량 비교는 그 자체로 흥미롭지만, 그 비교가 곧장 "ref로 옮기자"는 결론이 되어서는 안 된다. 사용자 화면에 동적으로 비치는 값이라면 21회 렌더는 사실상 비용이 아니다. React는 그 정도 렌더는 우습게 처리한다.

여기서 흔들렸다면 → **9장(Ref — 렌더 바깥의 세계)** 이 ref의 정당한 사용과 회피해야 할 사용을 정리해준다.

## 6번 — Context 리렌더 조건

```tsx
type Theme = { mode: 'light' | 'dark'; primary: string };
const ThemeContext = createContext<Theme>({ mode: 'light', primary: '#000' });

function App() {
  const [mode, setMode] = useState<'light' | 'dark'>('light');
  return (
    <ThemeContext value={{ mode, primary: '#000' }}>
      <Page />
    </ThemeContext>
  );
}
```

`mode`가 `'light'`에서 `'light'`로 set된 경우(같은 값을 다시 set한 경우), `useContext(ThemeContext)`를 쓰는 모든 consumer가 리렌더되는가? selective subscription은 가능한가?

*정답을 가리고 30초 생각해보자.*

### 풀이

리렌더된다. 그리고 selective subscription은 **공식 Context API에는 없다.**

이유를 풀어보자. `mode`가 `'light'`로 똑같이 set되면 React는 "state가 안 바뀌었다"고 판단해 컴포넌트를 다시 렌더하지 않는다. 그러나 `mode`를 다른 값으로 set하면 `App`이 리렌더되고, 그 순간 `value`로 넘긴 객체 리터럴 `{ mode, primary: '#000' }`은 새 참조가 된다. Context는 value의 참조 동등성(`Object.is`)을 기준으로 판단하므로, 객체 자체는 내용이 같더라도 참조가 다르면 모든 consumer가 리렌더 대상이 된다.

"primary는 안 바뀌니까 primary만 쓰는 컴포넌트는 리렌더 안 되겠지"라는 직관은 틀렸다. 그게 가능하려면 selective subscription이 있어야 하는데, 공식 Context는 그것을 제공하지 않는다. Zustand나 Jotai로 손이 가게 되는 가장 큰 이유 중 하나가 여기다.

처방은 두 갈래다. (1) value를 두 개의 Context로 쪼갠다 — 자주 바뀌는 것과 안 바뀌는 것을 분리. (2) 정말 selective가 필요하면 외부 store를 쓴다.

(1)의 좋은 사례는 reducer + context 스케일 패턴에서 나온다. `TasksContext`(state)와 `TasksDispatchContext`(dispatch)를 분리해두면, dispatch만 쓰는 컴포넌트는 state가 바뀌어도 리렌더되지 않는다. dispatch 함수의 참조는 안정적이기 때문이다. 큰 앱에서 이 분리 한 줄이 의외로 큰 차이를 만든다.

(2)의 사례는 Zustand의 selector다. 같은 store에서 여러 컴포넌트가 각자 자기가 관심 있는 부분만 구독한다. value 전체가 바뀌어도 자기 selector의 결과가 같으면 리렌더되지 않는다. Context는 이런 능력을 기본 제공하지 않는다는 점을 잊지 말자. "Context로 글로벌 state를 다 처리하면 되겠다"는 직관이 자주 깨지는 자리다.

여기서 흔들렸다면 → **7장(Context로 깊이 전달하기)** 와 **8장(Reducer + Context 스케일 패턴)** 이 함께 보강해준다.

## 7번 — 의존성 배열에 객체

```tsx
function ChatRoom({ serverUrl, roomId }: { serverUrl: string; roomId: string }) {
  useEffect(() => {
    const options = { serverUrl, roomId };
    const connection = createConnection(options);
    connection.connect();
    return () => connection.disconnect();
  }, [{ serverUrl, roomId }]);

  return <h1>채팅방 {roomId}</h1>;
}
```

이 코드에서 effect는 어떤 주기로 재실행되는가? 그리고 왜 그렇게 되는가?

*정답을 가리고 30초 생각해보자.*

### 풀이

**매 렌더마다 재실행된다.** 부모가 한 번 더 렌더하면 곧장 disconnect → connect가 한 번 더 일어난다.

이유는 의존성 배열에 들어간 `{ serverUrl, roomId }` 객체 리터럴이 **렌더할 때마다 새로 만들어지기 때문**이다. React는 의존성을 `Object.is`로 비교한다. 내용이 같아도 참조가 다르면 "바뀌었다"고 본다. 결과적으로 의존성이 매 렌더마다 다르다고 판정되고, effect가 매 렌더마다 재실행된다.

처방은 단순하다. **원시값으로 분해해서 의존성에 넣는다.**

```tsx
useEffect(() => {
  const options = { serverUrl, roomId };
  const connection = createConnection(options);
  connection.connect();
  return () => connection.disconnect();
}, [serverUrl, roomId]);
```

이제 React는 두 개의 문자열을 비교한다. 문자열은 값으로 비교되니 내용이 같으면 같다고 판정한다. 객체 생성은 effect 내부에 두자. 의존성에서 객체와 함수를 빼는 것은 12장의 핵심 작업이다.

함수도 같은 함정에 들어간다. 부모가 매 렌더마다 새 화살표 함수를 만들어 자식에게 prop으로 내려주면, 그 함수를 의존성에 넣은 자식의 effect는 매 렌더마다 다시 실행된다. 처방은 비슷하다 — 함수를 effect 내부로 옮기거나, 함수가 캡처하는 원시값들을 의존성으로 분해하거나, `useCallback`으로 참조를 안정화하거나, 더 좋은 방법으로 함수형 업데이트나 `useEffectEvent`로 의존을 끊는다. 어떤 길을 택하든 핵심은 "참조가 매 렌더마다 새로워지는 값을 의존성에 두지 않는다"는 한 줄이다.

여기서 흔들렸다면 → **12장(의존성 다루기)** 이 가장 큰 보강을 준다.

## 8번 — `useEffectEvent`는 의존성에 넣어야 하는가

```tsx
import { experimental_useEffectEvent as useEffectEvent } from 'react';

function ChatRoom({ roomId, theme }: { roomId: string; theme: 'light' | 'dark' }) {
  const onConnected = useEffectEvent(() => {
    showNotification('연결되었습니다', theme);
  });

  useEffect(() => {
    const connection = createConnection(roomId);
    connection.on('connected', () => onConnected());
    connection.connect();
    return () => connection.disconnect();
  }, [roomId]); // ← onConnected는 일부러 의존성에 안 넣음

  return <h1>채팅방 {roomId}</h1>;
}
```

`onConnected`를 의존성 배열에 넣어야 하는가? 아니라면 그 이유는 무엇인가? lint가 경고를 내지 않는가?

*정답을 가리고 30초 생각해보자.*

### 풀이

**넣지 않는다. 그리고 React lint도 의도적으로 경고하지 않는다.**

이유의 뿌리는 "반응형(reactive) vs 비반응형" 구분에 있다. `useEffect`의 의존성 배열은 "이 값이 바뀌면 동기화를 다시 해야 한다"는 계약이다. 의존성에 들어간 값은 모두 "반응형"이다 — 변하면 effect가 다시 실행된다는 뜻.

`useEffectEvent`로 만든 함수는 본질적으로 다르다. effect 안에서 호출되지만, 그 함수가 캡처한 최신 props/state는 **effect 재실행을 트리거하지 않는다.** 이것이 핵심이다. `theme`이 바뀌어도 chat connection이 재맺어지지 않는다. 그러나 `onConnected`가 호출되는 순간에는 최신 `theme` 값을 본다. "비반응형으로 최신 값에 접근"이라는 모순처럼 보이는 능력을 한 단어로 만든 것이 `useEffectEvent`다.

함정은 두 개다.

1. **lint silencer로 쓰지 말자.** "의존성 경고가 귀찮아서" 함수를 `useEffectEvent`로 감싸 회피하는 사람들이 있다. 이건 의도와 다르다. effect의 동기화 의미를 흐리는 짓이다.
2. **다른 컴포넌트에 prop으로 넘기지 말자.** `useEffectEvent`로 만든 함수는 자기 effect 안에서만 호출되도록 설계됐다. 밖으로 새어나가면 미묘한 버그가 생긴다.

2026년 시점에 이 hook은 여전히 실험적이다. 그러나 어휘 자체는 정착했다고 봐도 좋다. "이 코드가 반응형이어야 하는가, 아닌가"를 분리해서 묻는 사고 도구가 우리에게 새로 생긴 셈이다. 같은 effect 안의 코드라도 어떤 줄은 props/state 변화에 반응해야 하고, 어떤 줄은 그저 "지금 이 순간의 최신 값을 보고 싶을 뿐"이다. 이 둘을 한 함수 안에 뭉뜽그려두면 의존성 배열은 거짓말을 강요받는다. 거짓말한 의존성은 stale closure를 만들고, 정직한 의존성은 불필요한 재실행을 만든다. `useEffectEvent`는 그 둘 사이의 진짜 처방이다.

다만 본문에서는 이 hook을 만능열쇠처럼 다루지 않는다. 가능하면 "이게 정말 effect여야 하는가"부터 다시 묻는다. 사용자 이벤트로 일어나는 일은 핸들러로 옮기고, 파생값은 렌더 중 계산으로 옮기고, prop 변화에 반응한 리셋은 `key`로 옮긴다. 이런 정리가 끝난 다음에야 남은 진짜 effect들을 두고 `useEffectEvent`가 등장한다.

여기서 흔들렸다면 → **12장(의존성 다루기)** 의 후반부에서 본격적으로 다룬다.

## 자가 보고 — 어디서 가장 흔들렸는가

여기서 멈추고, "몇 개를 맞췄는가"보다 **"어느 영역에서 가장 흔들렸는가"**를 자기 자신에게 묻자. 점수보다 패턴이 중요하다. 여덟 문항이 무작위로 나열된 게 아니다. 1번~3번은 "상태의 모양과 정체성", 4번은 "외부 세계와의 동기화", 5번은 "ref와 state의 경계", 6번은 "Context의 리렌더 모델", 7번~8번은 "의존성 배열의 본질" — 이 다섯 영역이 우리가 React에서 가장 자주 미끄러지는 자리다. 자기가 어느 영역에서 가장 자주 멈췄는지를 표로 적어두자.

| 영역 | 해당 문항 | 자가 진단 (확신 / 미묘 / 흐림) | 가장 보강이 필요한 챕터 |
|---|---|---|---|
| 상태 모양 (모순 제거, 파생값) | 1번 | | 1장, 2장 |
| 렌더 스냅샷 (setState 의미) | 2번 | | 1장, 12장 |
| 트리 위치와 정체성 (key) | 3번 | | 4장 |
| Effect와 race condition | 4번 | | 10장, 11장 |
| Ref vs State 경계 | 5번 | | 9장 |
| Context 리렌더 모델 | 6번 | | 7장, 8장 |
| 의존성 배열의 참조 동등성 | 7번 | | 12장 |
| 반응형 vs 비반응형 분리 | 8번 | | 12장 |

세 칸 중에 "흐림"이 두 개 이상인 영역이 있다면, 그 영역의 챕터를 먼저 읽어도 좋다. 책은 1장부터 읽는 것이 기본 흐름이지만, 통증이 가장 큰 자리부터 가는 것도 권장할 만한 방법이다. 조급해지지는 말자. 흐린 자리가 많다는 것은 부끄러운 일이 아니다. 정직하게 본다는 것이 이미 절반이다.

한 가지 더 덧붙이자. 진단 결과에 너무 많은 의미를 부여하지도 말자. "확신"이라고 적었는데 막상 챕터를 읽다 보면 "어, 이거 내가 틀렸네" 하는 자리가 분명히 있다. 그 자리를 만나면 진단표를 다시 쓰자. 멘탈 모델은 한 번에 또렷해지지 않는다. 책을 통과하는 동안 여러 번 다시 그려진다. 진단표는 그 여정의 출발선일 뿐, 결승선이 아니다.

또 하나, "확신"인데 실제로는 흐린 영역도 있을 수 있다. 평소에 그 자리에서 버그를 만나지 않은 운 좋은 사람일 수도 있고, 자기 코드가 어쩌다 그 함정을 비켜갔을 수도 있다. 챕터를 읽는 도중에 "어? 나 이걸 알고 있다고 생각했는데" 하는 순간이 오면, 그 순간이 진짜 진단의 결과다. 그런 자리일수록 챕터의 풀이를 한 번 더 천천히 따라가는 편이 낫다.

## 핵심 정리

이 장에서 우리가 마주한 사실들을 짧게 모아두자. 각 항목은 본문의 어느 챕터에서 깊게 파고들지를 함께 적어둔다.

1. **모순 가능한 상태 모양은 모델링 단계에서 막는다.** 두 boolean보다 한 enum이 거의 항상 낫다. → 1장, 2장.
2. **렌더는 스냅샷이다.** 같은 핸들러 안에서 `setState`를 여러 번 부른다고 자동으로 누적되지 않는다. 누적이 필요하면 함수형 업데이트. → 1장, 12장.
3. **컴포넌트의 정체성은 트리 위치 + 타입이다.** 의도적인 리셋이 필요하면 `key`. effect로 prop 변화에 반응해 state를 리셋하는 패턴은 거의 항상 잘못된 길이다. → 4장.
4. **Effect는 동기화 계약이다.** 의존성 거짓말, race condition, double invocation은 한 묶음으로 다뤄야 한다. → 10장, 11장.
5. **Ref는 렌더 결과에 영향을 주지 않는 값에만 쓴다.** "리렌더 줄이려고" ref를 꺼내는 직관은 잊자. → 9장.
6. **Context는 참조 동등성으로 리렌더를 판단하며, selective subscription이 없다.** 자주 바뀌는 value와 안 바뀌는 value는 분리한다. → 7장, 8장.
7. **의존성 배열에는 원시값을 넣고, 객체와 함수는 넣지 않는다.** 객체는 effect 내부에서 만들거나 원시값으로 분해한다. → 12장.
8. **`useEffectEvent`는 lint silencer가 아니다.** "이 코드가 반응형이어야 하는가"를 결정한 뒤에 꺼내는 도구다. → 12장.

이 여덟 줄을 책상 옆에 붙여두어도 된다. 챕터를 읽어가면서 같은 줄로 돌아오는 자신을 자주 보게 될 것이다. 본문에서는 같은 원리를 여러 각도에서 반복해서 만난다 — 상태 모양 이야기에서 한 번, 끌어올림 이야기에서 한 번, effect 이야기에서 또 한 번. 지루하게 들릴 수도 있겠지만, 멘탈 모델은 같은 원리를 다른 각도에서 여러 번 마주쳐야 비로소 손에 새겨진다. "또 그 이야기네"라는 느낌이 들면 잘 가고 있다는 신호다.

## 진단을 통과하며 자주 보이는 패턴

여덟 문항을 채점해보면 한국 React 개발자에게 공통적으로 보이는 몇 가지 패턴이 있다. 자기 결과와 비교해보자.

**패턴 1 — "비동기"라는 단어로 모든 걸 설명하려 함.** 2번 문항에서 "setState는 비동기다"라는 답을 곧장 떠올린 사람이 많다. 비동기라는 단어는 절반만 맞고, 본질은 "렌더 단위로 묶어서 처리한다"는 batching이다. 비동기로 설명하기 시작하면 "왜 핸들러 안에서 set한 직후에 state를 읽어도 옛 값이 나오는가"는 설명되지만, "그래서 어떻게 누적시키는가"는 답이 안 나온다. 함수형 업데이트가 처방으로 곧장 떠오르지 않으면 이 패턴에 걸린 것이다.

**패턴 2 — 의존성 배열을 "재실행 트리거"로만 봄.** 7번 문항에서 객체 의존성 함정에 걸린 사람들의 직관은 대개 "내용이 같으니까 재실행 안 되겠지"다. 의존성 배열은 "내용 비교"가 아니라 "참조 비교"라는 한 줄을 잊으면 거의 모든 effect 버그가 여기서 시작된다. React가 사용하는 비교는 `Object.is`다. `===`보다 약간 엄밀한, 그러나 본질적으로 참조 동등성에 기반한 비교다.

**패턴 3 — Context를 "글로벌 state의 만능 그릇"으로 생각함.** 6번 문항에서 selective subscription이 가능하다고 답한 사람이 적지 않다. Context는 "트리 깊이 있는 자식에게 prop을 안 거치고 값을 주는 통로"이지, "세분화된 구독 관리자"가 아니다. 이 둘을 혼동하면 큰 앱에서 Context가 곧장 성능 함정으로 변한다.

**패턴 4 — ref를 "성능 탈출구"로 생각함.** 5번 문항에서 ref가 "더 빠르다"고 답하는 직관은 한국 커뮤니티에서 특히 자주 보인다. 정량 비교가 폭넓게 공유되어 있어서 그렇다. 그러나 그 비교는 거의 항상 "동적 뷰가 필요 없는 입력 폼"에 한정된다. 자기 화면의 어느 부분이 입력값에 따라 바뀌어야 하는가를 먼저 묻지 않으면, ref로 옮긴 코드는 정확히 그 자리에서 다시 깨진다.

**패턴 5 — `useEffectEvent`를 "lint silencer"로 받아들임.** 8번 문항에서 "그냥 의존성 경고가 귀찮아서 만든 hook 아닌가요?" 같은 답이 나오면 이 패턴에 걸린 것이다. 이 hook은 "이 코드가 반응형이어야 하는가, 아닌가"라는 질문을 분리하기 위한 도구지, 경고를 끄기 위한 단축키가 아니다. 본문에서 가장 신중하게 다루는 hook 중 하나다.

자기 답이 이 다섯 패턴 중 어디에 가까웠는지 한번 적어두자. 같은 패턴이 본문 곳곳에서 다시 등장한다. 챕터를 읽다가 "어, 이거 그 패턴이네" 하는 순간이 오면, 그게 바로 멘탈 모델이 또렷해지는 자리다.

## 진단표를 어떻게 활용할까

진단을 마쳤으면 한 가지 결정이 남는다. 책을 어떻게 읽을 것인가. 두 가지 길이 있다.

**기본 흐름 (권장)**: 1부 1장부터 차례대로 읽는다. 1부의 8개 챕터는 서로 빌드업되어 있어서 순서대로 읽을 때 가장 효율이 좋다. 2장의 ID 저장 패턴이 3장 끌어올림에서 다시 등장하고, 4장 `key`가 11장 effect 라이프사이클에서 다시 인용된다. 챕터 간 다리가 잘 깔려 있으니 흐름을 따라가자. 하루 한 챕터 페이스로 잡으면 본문 분량 기준 2주 안에 1부를 마칠 수 있다. 연습 문제까지 손으로 풀고 가면 한 달 정도가 적당한 페이스다. 너무 빨리 가려고 욕심내지 말자. React의 멘탈 모델은 한 번 손에 새기면 다음 5년의 코드가 달라지는 종류의 투자다.

**통증 우선 흐름**: 진단표에서 "흐림"이 두 개 이상인 영역의 챕터로 곧장 간다. 이 길은 빠르지만 위험이 있다 — 앞 챕터의 어휘를 모르면 풀이가 절반만 들어온다. 그래도 당장 실무에서 그 자리의 버그를 잡고 있다면 우선 그 챕터로 가는 편이 낫다. 다 읽고 처음으로 돌아와 1장부터 다시 읽으면 된다.

어느 길을 가든 한 가지는 지키자. **연습 문제는 머리로 풀지 말고 손으로 풀자.** 매 챕터 끝에 두세 개의 연습 문제가 있다. 답은 함께 제공되지만, 한 번 자기 손으로 짜본 다음에 답을 보는 편이 낫다. React의 멘탈 모델은 손가락이 키보드에 닿아본 적이 있는 자리에서 가장 빠르게 새겨진다. 한국어로 풀이를 읽고 영어 공식 문서로 동일 개념을 한 번 더 확인하면 서로 다른 어휘로 같은 그림을 두 번 그리는 셈이라 더 단단해진다. 본문에는 자주 react.dev의 해당 페이지로 가는 링크가 걸려 있다. 시간 여유가 있는 자리에서는 그 링크들을 따라가 보자. 같은 이야기를 다른 입을 통해 들으면 흐린 부분이 더 또렷해진다.

## 핵심 정리를 한 번 더 — 이번엔 부정문으로

위의 핵심 정리 여덟 줄을 긍정문으로 적었다. 같은 내용을 부정문으로 한 번 더 적어두자. 부정문은 "하지 말 것"의 형태라 평소 코드를 짤 때 머릿속에서 자동으로 발화되기 쉽다.

1. **두 boolean으로 시작하지 말자.** 의심스러우면 enum을 먼저 그려보고 시작하자.
2. **같은 핸들러 안에서 set을 여러 번 부를 때 누적을 가정하지 말자.** 누적이 필요하면 함수형 업데이트로 명시하자.
3. **prop이 바뀔 때 effect로 state를 리셋하지 말자.** `key`가 더 빠르고 더 단단하다.
4. **`useEffect(() => fetch(...), [])`로 끝내지 말자.** 의존성, ignore 가드, cleanup — 셋이 함께 가야 한다.
5. **"리렌더 줄이려고" `useState`를 `useRef`로 바꾸지 말자.** 화면에 비치는 값은 state가 답이다.
6. **Context가 selective subscription을 해줄 거라 기대하지 말자.** 그러지 않는다.
7. **의존성 배열에 객체 리터럴을 넣지 말자.** 함수도 마찬가지다.
8. **`useEffectEvent`를 lint 경고를 끄려고 꺼내지 말자.** 의도와 다르다.

긍정문과 부정문은 같은 원리를 다른 각도에서 본 것이다. 평소 코드를 짤 때는 부정문이 먼저 떠오르고, 동료에게 설명할 때는 긍정문이 먼저 떠오른다. 둘 다 손에 쥐고 있으면 같은 함정에 두 번 걸리지 않는다.

## 다음 장으로 가는 다리

자가 진단을 마쳤다는 것은 함정의 지도를 한 번 그렸다는 뜻이다. 지도가 있으니 다음 단계는 단단한 토대를 쌓는 일이다. **상태를 어떻게 모양 잡고, 어디에 두고, 무엇은 두지 말아야 하는가** — 1부의 첫 챕터부터 이 질문을 정면으로 다뤄보자.

1장에서 우리가 같이 풀 첫 문제는 의외로 작다. 퀴즈 폼 하나다. 사용자는 답을 입력하고 제출하면 채점이 일어나고 결과가 뜬다. 명령형으로 짜면 "버튼 활성화", "에러 메시지 숨기기", "성공 메시지 표시" 같은 단계 하나하나를 직접 토글하게 되는데, 그 코드는 빠르게 끔찍해진다. 같은 폼을 React 식으로, 즉 "각 상태에서 어떤 모습이어야 하는지"를 기술하는 식으로 다시 그려본다. 다섯 단계의 절차를 따라가면 같은 폼이 절반의 코드로, 두 배의 견고함으로 다시 태어난다. 1장 끝에서 우리는 이 절차를 자기 코드에 적용할 수 있는 도구로 가지고 있게 된다.

출발선에 선 우리에게 가장 필요한 건 거창한 결심이 아니라 가벼운 호흡이다. 진단표 위의 흐린 영역들 앞에서 한 번 숨을 고르고, 천천히 한 줄씩 가보자. 함정의 지도가 그려졌으니, 이제 멘탈 모델의 토대부터 다시 쌓아보자.
