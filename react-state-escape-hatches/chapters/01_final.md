# 1장. UI는 상태의 함수다 — 다섯 단계로 상태를 설계하기

## 들어가는 이야기 — boolean 다섯 개의 늪

금요일 오후, 폼 하나를 만들고 있다고 해보자. 사용자가 답을 적어 제출 버튼을 누르면 서버로 보내고, 응답이 오면 성공 화면을 보여주거나 에러 메시지를 띄워준다. 흔하디흔한 화면이다. 처음엔 이렇게 시작한다.

```tsx
const [answer, setAnswer] = useState('');
const [isLoading, setIsLoading] = useState(false);
const [isError, setIsError] = useState(false);
const [isSuccess, setIsSuccess] = useState(false);
const [isEmpty, setIsEmpty] = useState(true);
```

여기까진 견딜 만하다. 그런데 기획자가 다가와 한마디 한다. "제출 버튼은 입력이 비어 있을 때 비활성화돼야 해요. 아, 그리고 제출 중일 때도요. 성공한 뒤에는 입력창이 사라져야 하고, 에러가 났을 땐 입력은 그대로 두되 빨간 테두리를 보여주세요."

이제 핸들러가 이런 모습이 된다.

```tsx
async function handleSubmit() {
  if (answer.length === 0) return;
  setIsLoading(true);
  setIsError(false);
  setIsSuccess(false);
  try {
    await submitAnswer(answer);
    setIsSuccess(true);
    setIsEmpty(true);
    setAnswer('');
  } catch (e) {
    setIsError(true);
  } finally {
    setIsLoading(false);
  }
}
```

JSX는 또 어떤가. `disabled`는 `isLoading || isEmpty || isSuccess`이고, 빨간 테두리는 `isError && !isLoading`이며, 성공 메시지는 `isSuccess && !isError`다. 한 줄 한 줄 토글링 조건을 만들고 있노라면 머릿속에서 진실표가 그려진다. boolean이 다섯 개니 가능한 조합은 32가지. 그중에서 실제로 의미 있는 화면은 네댓 개고, 나머지는 "절대 일어나면 안 되는데 코드상으론 가능한" 상태들이다. `isLoading`이면서 `isError`가 동시에 true라면? `isSuccess`인데 `isEmpty`가 false라면? 생각만 해도 난감하다.

조금 지나면 QA가 버그 리포트를 가져온다. "성공 메시지가 뜬 채로 에러 토스트도 같이 보여요." 코드를 뒤져보면 어딘가에서 `setIsError(false)`를 빠뜨렸다. 한 줄 추가하면 끝나는 일처럼 보이지만, 같은 종류의 버그가 다음 주에 또 다른 모양으로 돌아온다. 우리가 잘못 짠 게 아니라, **상태를 boolean들의 조합으로 표현하기로 한 결정 자체가 잘못된 것이다.**

여기서 멈춰 한번 짚어보자. 우리가 새 컴포넌트를 만들 때 가장 먼저 하는 일이 무엇이었나? 십중팔구는 `useState`를 한 줄 적는 것이었다. "일단 isLoading 하나 두고…" 하면서 시작한다. 그런데 이 첫 줄이 거꾸로 흘러간 출발점이다. **상태 설계의 출발점은 useState가 아니라 화면이어야 한다.** 화면을 먼저 그리고, 그다음에 그 화면을 만들 수 있는 최소 상태가 무엇인지 결정한다. 이 순서가 뒤집히면 항상 boolean이 늘어나는 방향으로 코드가 흘러간다.

이 장에서는 디자이너가 화면을 설계하듯 상태를 설계하는 방법을 다룬다. 다섯 단계만 머릿속에 두면, 위와 같은 폼은 처음부터 다른 모양으로 태어난다. boolean 다섯 개가 아니라, 잘 정의된 enum 하나로. 그리고 한번 이 절차에 익숙해지면, 컴포넌트를 만들기 전에 종이 위에서 끝나는 일이 점점 늘어난다. 코드 에디터를 여는 시점이 늦춰지는 것이다. 늦춰지는 만큼 우리는 덜 고치게 된다.

## 디자이너가 화면을 그리는 방식 빌리기

디자이너가 같은 폼을 그렸다면 어땠을까? 십중팔구 피그마 위에 네 장의 아트보드가 펼쳐졌을 것이다.

1. **빈 폼** — 입력창은 비어 있고 제출 버튼은 회색이다.
2. **입력 중** — 사용자가 글자를 치고 있다. 제출 버튼은 활성화되어 있다.
3. **제출 중** — 입력창은 잠기고, 버튼 자리에 스피너가 돈다.
4. **성공** — 폼 자리에 "제출되었습니다" 메시지가 떠 있다.
5. **에러** — 입력은 그대로지만 빨간 테두리가 둘러쳐지고, 아래에 메시지가 보인다.

화면이 다섯 장이다. 다섯 장은 서로 배타적이다. 한 시점에 사용자는 정확히 하나의 화면만 본다. **이 사실이 바로 우리가 코드로 옮겨야 하는 진실이다.**

그런데 우리가 처음에 짠 코드는 어땠는가? 다섯 화면을 다섯 boolean으로 나눠 들고 있다가, 매번 핸들러에서 일일이 토글하고 있었다. boolean의 조합이 화면을 결정하도록 만든 것이다. 디자이너의 머릿속에는 "다섯 장의 그림"이 있었는데, 코드에는 "32가지 가능한 조합"이 들어앉아 있다. 그러니 실제로 그릴 수 있는 화면은 다섯 장이지만, 버그가 자라날 공간은 27가지다.

상태 설계의 출발점을 바꿔보자. **먼저 화면을 그리고, 그다음에 상태를 짓자.** 이름을 붙일 수 있는 모든 시각적 모습을 종이에 적고, 그 모습들이 어떻게 서로 옮겨가는지를 화살표로 그려본다. 그게 끝나고 나서야 코드 에디터를 연다. 이 순서가 핵심이다. 코드 위에서 상태를 발명하지 말고, 화면을 보고 상태를 발견하자.

이 표현이 좀 추상적으로 느껴진다면 예를 하나 더 들어보자. 알림 배지를 만든다고 해보자. 처음엔 "읽지 않은 알림이 있으면 빨간 점이 뜬다"가 전부였다. boolean 하나로 충분했다. 며칠 뒤 "읽지 않은 알림 개수도 보여주자"가 추가된다. boolean 옆에 number를 둔다. 또 며칠 뒤 "처음 들어온 사용자에겐 점도 숫자도 안 보여주고, 대신 '환영합니다' 풍선을 띄우자"가 추가된다. boolean 하나가 또 늘어난다. 그 다음 주엔 "동기화 중에는 점이 회색으로 깜빡이게 하자"가 추가된다. 또 boolean. 어느 순간 컴포넌트가 boolean 네댓 개와 number 하나를 들고 있는데, 가능한 조합 중 의미 있는 화면은 다섯 개뿐이다. **요구사항이 늘어날 때마다 boolean이 늘어나는 게 자연스러워 보였지만, 실제로는 시각 상태가 다섯 개로 늘어난 것이었다.** 처음부터 시각 상태로 세었다면 enum으로 표현하고 끝났을 일이다.

이 사고법은 디자이너의 사고법을 빌려오는 것이라고 했지만, 실은 더 오래된 뿌리가 있다. 상태 머신(state machine)이라는 개념이다. 컴포넌트가 가질 수 있는 시각 상태의 집합을 노드로 그리고, 그 사이의 전이를 화살표로 그리면 그게 곧 상태 머신이다. XState 같은 라이브러리를 도입할 정도로 본격적이지 않더라도, **머릿속에 작은 상태 머신을 그려두는 것**만으로도 코드가 달라진다. 우리가 다룰 다섯 단계는 그 작은 상태 머신을 그리는 절차에 가깝다.

## 명령형과 선언형 — 무엇을 다르게 보는가

명령형 코드와 선언형 코드의 차이는 어디에 있을까? 흔히 "어떻게 vs 무엇"이라는 식으로 말하지만, 폼 코드 위에서 보면 좀 더 구체적인 차이가 드러난다.

명령형 사고는 **변화의 단계**를 본다. "버튼이 눌렸으니 로딩을 켜고, 비활성화하고, 에러 메시지를 지우고…" 한 사건이 벌어지면 그것에 응답해 무엇무엇을 토글한다. UI는 이전 상태에 변화가 누적된 결과다. 이전 상태가 뭐였는지 항상 신경 써야 한다.

선언형 사고는 **각 상태의 모습**을 본다. "지금 status가 'submitting'이라면 화면은 이렇게 생겼어야 한다." 한 사건이 벌어지면 status를 다른 값으로 바꿀 뿐, 화면을 어떻게 바꿀지는 신경 쓰지 않는다. UI는 매 순간 현재 상태에 대한 함수다. 이전 상태가 뭐였는지는 잊어도 된다.

리액트의 슬로건 `UI = f(state)`는 농담이 아니다. 이 식은 "UI는 상태에 의해 *결정된다*"가 아니라 "UI는 상태의 *순수 함수*다"라고 말하고 있다. 같은 state면 같은 화면이 나와야 한다. 어제의 토글 횟수, 그제의 클릭 순서, 그런 건 화면에 영향을 줘선 안 된다. 상태 하나만 보면 화면 전체가 결정되어야 한다.

이 관점이 몸에 배면 디버깅이 달라진다. 화면이 이상하면 더 이상 "어떤 핸들러에서 어떤 토글을 빠뜨렸지?"를 추적하지 않는다. 그저 React DevTools를 열어 현재 state를 들여다보고, "이 state가 정말 이 화면을 의미해야 하나?"만 물어보면 된다. state가 진실이고, 화면은 그 진실을 비추는 거울일 뿐이다.

물론 명령형이 무조건 나쁘다는 얘기는 아니다. 캔버스에 직접 그림을 그리거나, 외부 비-React 라이브러리를 제어할 때는 명령형이 자연스럽다. 그런 경우는 ref와 effect로 따로 다룬다(이 책의 9장과 11장에서 본격적으로 등장한다). 다만 **컴포넌트 내부의 일반적인 상태 흐름**에서는 선언형이 기본값이다. 그렇게 살자.

조금 다르게 말해보자. 명령형 코드는 "사건의 일대기"를 적는다. "버튼이 눌리면 이걸 켜고, 응답이 오면 저걸 끄고…" 하는 식으로 시간 순서를 따라간다. 선언형 코드는 "상태의 사진첩"을 그린다. 각 사진(상태) 아래에 그 사진에 어울리는 화면을 적어두고, 사건이 일어나면 사진을 한 장 넘긴다. 시간 순서는 사진첩 바깥에 있고, 코드 안에는 사진들만 줄지어 있다. 어느 쪽이 디버깅하기 좋은가? 화면이 이상할 때 명령형은 "지금까지 어떤 사건들이 어떤 순서로 일어났더라"를 거꾸로 추적해야 한다. 선언형은 "지금 사진이 뭐지?"만 보면 된다. 시간이라는 차원이 없어진 것이다.

이 차이는 동료와 코드 리뷰할 때도 드러난다. 명령형 핸들러는 "여기서 setIsError(false) 빼먹은 거 아냐?" 같은 리뷰 코멘트가 자주 달린다. 다섯 줄 짜리 핸들러에 set 호출 네 개가 들어가면 그중 하나가 빠졌는지 검수하는 일이 사람의 몫이 된다. 선언형 핸들러는 "status를 'error'로 바꾼다" 한 줄이라 빠뜨릴 게 없다. **사람이 검수해야 할 일을 줄이는 것**이 선언형이 주는 진짜 선물이다. 우리는 매일 코드를 읽고, 그중 대부분은 한참 뒤에 다시 읽는 자기 자신이다. 미래의 자신에게 검수 부담을 덜 넘기는 코드를 쓰는 편이 낫다.

## 다섯 단계로 상태 설계하기

리액트 공식 문서에는 "Reacting to Input with State"라는 글이 있다. 거기 나오는 다섯 단계가 우리에게 필요한 절차다. 이름을 붙여 외워두자. 이 다섯 단계는 폼 하나, 모달 하나, 위저드 하나를 만들 때마다 매번 따라할 가치가 있다.

1. **시각 상태 식별하기** — 컴포넌트가 가질 수 있는 모든 모습에 이름을 붙인다.
2. **트리거 정의하기** — 각 상태 사이의 전환을 무엇이 일으키는지 적는다.
3. **state로 표현하기** — 식별한 상태를 표현할 수 있는 최소한의 state 변수를 둔다.
4. **비본질 제거하기** — 모순 가능한 state, 파생 가능한 state를 솎아낸다.
5. **핸들러 연결하기** — 각 트리거가 state를 어떻게 갱신할지 함수로 적는다.

이 순서가 중요하다. 많은 사람이 ③번부터 시작한다. "useState 어떤 거 쓰지?"부터 묻는 것이다. 그러면 아까 우리가 본 boolean 다섯 개의 늪에 빠진다. ①과 ②를 종이 위에서 먼저 끝낸 다음에 코드 에디터를 열자.

이제 퀴즈 폼 예제로 다섯 단계를 한 번씩 밟아보자.

### 1단계 — 시각 상태에 이름 붙이기

컴포넌트가 보일 수 있는 모든 모습을 적는다. 디자이너의 아트보드를 글로 옮기는 작업이다.

- **Empty (빈 상태)**: 폼이 막 열렸고 아직 아무것도 입력되지 않았다. 제출 버튼은 비활성화.
- **Typing (입력 중)**: 사용자가 답을 적고 있다. 제출 버튼 활성화.
- **Submitting (제출 중)**: 서버로 보내고 응답을 기다린다. 입력 잠김, 버튼에 스피너.
- **Success (성공)**: 제출이 끝났다. 폼 자리에 감사 메시지.
- **Error (에러)**: 제출이 실패했다. 입력은 그대로 두고 에러 메시지 노출.

다섯 개. 외울 수 있을 만큼 적다. 이름이 명사로 떨어진다는 점도 중요하다. 동사가 아니다. "loading", "submitting", "error" 같은 형용사·명사들이 자연스럽게 나오면 1단계는 잘 끝난 것이다.

여기서 한 가지 팁. **개발용 토글로 모든 상태를 한 번씩 그려볼 수 있게 해두자.** 디버깅용 props를 임시로 받아서 status를 강제로 바꾸는 식이다.

```tsx
type Status = 'empty' | 'typing' | 'submitting' | 'success' | 'error';

function QuizForm({ debugStatus }: { debugStatus?: Status }) {
  const [status, setStatus] = useState<Status>(debugStatus ?? 'empty');
  // ...
}
```

이러면 스토리북이나 작은 갤러리에 다섯 가지 status를 일렬로 띄워놓고 "각 상태에서 화면이 정말 이래야 하는지" 디자이너와 함께 확인할 수 있다. 한참 만들고 나서 "이런 상태를 빠뜨렸어요"라는 말을 듣는 것보다, 만들기 시작할 때 보여주는 편이 훨씬 낫다.

이름 붙이기를 한 번 더 강조하자. "loading", "submitting", "sending"은 비슷해 보이지만 미묘하게 다른 뉘앙스를 가진다. 같은 컴포넌트에서 같은 의미라면 한 단어로 통일하자. 다른 컴포넌트마다 단어가 다르면 검색이 어려워지고, 이 컴포넌트의 'submitting'과 저 컴포넌트의 'loading'이 같은 의미인지 다른 의미인지 매번 코드를 읽어야 한다. 팀 내에 가벼운 컨벤션을 두는 편이 좋다. 예를 들어 "데이터 가져오는 중"은 'loading', "사용자 입력을 서버에 보내는 중"은 'submitting' 식으로. 이런 작은 어휘 정렬이 누적되면 코드베이스 전체의 가독성이 달라진다.

또 하나, 시각 상태에 이름을 붙일 때 **사용자의 관점**을 따라가는 편이 자연스럽다. 'fetching_in_progress_with_optimistic_update'처럼 내부 구현을 노출하는 이름은 피하자. 'loading' 정도면 충분하다. 사용자에겐 "데이터를 가져오는 중인지"만 중요하고, 그게 낙관적 업데이트와 결합되었는지 아닌지는 부차적이다. 이름이 화면에 어떤 표시가 떠야 하는지를 곧장 떠올리게 해주면 좋은 이름이다.

### 2단계 — 트리거 정의하기

상태에서 상태로 넘어가는 길을 그려본다. 트리거는 두 종류로 나뉜다.

- **사람의 트리거 (Human input)**: 사용자가 일으키는 것. 키 입력, 버튼 클릭 등.
- **시스템의 트리거 (Computer input)**: 시스템 응답으로 일어나는 것. 네트워크 응답, 타임아웃 등.

이 둘을 분리해 적는 습관이 중요하다. 핸들러 코드를 짤 때 어디서 시작되는 일인지 구분이 안 되면 effect에 사용자 이벤트를 밀어 넣는 안티패턴(이건 11장에서 본격적으로 다룬다)이 자라기 시작한다. 일단은 종이에 적어두자.

퀴즈 폼의 트리거를 정리하면 이렇다.

| 트리거 | 종류 | 전이 |
|---|---|---|
| 입력창에 첫 글자가 들어옴 | 사람 | empty → typing |
| 입력창이 다시 비워짐 | 사람 | typing → empty |
| 제출 버튼 클릭 | 사람 | typing → submitting |
| 서버 응답 성공 | 시스템 | submitting → success |
| 서버 응답 실패 | 시스템 | submitting → error |
| 에러 후 다시 입력 | 사람 | error → typing |
| "다시 풀기" 클릭 | 사람 | success → empty |

표를 다 적고 나면 머릿속에 작은 상태 다이어그램이 그려진다. 이 다이어그램이 우리가 만들 코드의 청사진이다. 이걸 그려놓고 코드를 짜는 것과, 안 그리고 짜는 것의 차이는 크다. 그려놓으면 "여기서 여기로 가는 길이 있던가?"를 확인하면서 짜게 되고, 안 그려놓으면 그때그때 boolean을 추가하게 된다.

이 표를 잘 그렸는지 점검하는 좋은 방법이 있다. **각 상태에 대해, 그 상태에서 빠져나갈 수 있는 길이 적어도 하나는 있는지** 확인해보자. 만약 'submitting' 상태에서 빠져나가는 길이 표에 없다면, 사용자가 한 번 제출 버튼을 누른 뒤 영원히 'submitting'에 갇히는 화면이 나온다. 거꾸로, **들어갈 수 없는 상태**가 있는지도 본다. 어떤 트리거로도 도달할 수 없는 상태라면 그건 처음부터 시각 상태 목록에서 빼야 한다. 이 두 가지 점검만 해도 빠뜨린 길과 죽은 상태가 드러난다.

또 하나, 같은 사건이 어떤 상태에서는 의미가 있고 어떤 상태에서는 무시되어야 하는 경우를 잘 적어두자. 예를 들어 사용자가 'submitting' 중에 다시 제출 버튼을 누르면 어떻게 되어야 하는가? 무시? 큐잉? 이전 요청 취소 후 재시도? 표를 그릴 때 이런 경계 사례에 대한 결정을 미리 내려두면, 핸들러를 짤 때 헤매지 않는다. **결정은 종이에서, 코딩은 키보드에서.**

### 3단계 — state로 표현하기

이제 코드 에디터로 옮긴다. 가장 단순한 방식부터 시작해보자. 아직 비본질을 솎아내지 않은 형태다.

```tsx
// 일단 머릿속에 떠오른 대로 다 둬보자
const [answer, setAnswer] = useState('');
const [isEmpty, setIsEmpty] = useState(true);
const [isTyping, setIsTyping] = useState(false);
const [isSubmitting, setIsSubmitting] = useState(false);
const [isSuccess, setIsSuccess] = useState(false);
const [isError, setIsError] = useState(false);
const [error, setError] = useState<string | null>(null);
```

자, 이게 우리가 처음에 봤던 `isLoading`/`isError`/`isSuccess`/`isEmpty`의 풀-패키지 버전이다. 시각 상태 다섯 개를 그대로 boolean 다섯 개로 옮겨놨다. 일단 이 상태로 두고 4단계로 넘어가자. 우리는 이걸 줄일 것이다.

### 4단계 — 비본질 제거하기

여기가 진짜 작업이 일어나는 단계다. 우리는 state 변수 하나하나를 보면서 다음 세 가지 질문을 던진다.

- **이 변수는 모순을 만드는가?** `isTyping`과 `isSubmitting`이 동시에 true가 될 수 있는가? 코드상 가능하다면 모순이다.
- **이 변수는 다른 변수에서 파생할 수 있는가?** `isEmpty`는 `answer.length === 0`으로 계산된다. 굳이 state로 들 필요가 없다.
- **같은 정보를 두 곳에서 추적하고 있는가?** `isError`와 `error: string | null`은 같은 정보를 두 번 들고 있다. 둘 중 하나만 있으면 된다.

이 질문을 다 통과시켜보자.

- `isEmpty`는 `answer === ''`로 파생된다. 빼자.
- `isTyping`/`isSubmitting`/`isSuccess`/`isError` 네 boolean은 서로 배타적이다. 한 시점에 정확히 하나만 true여야 한다. 그렇다면 boolean 네 개가 아니라 **enum 하나**가 정확하다.
- `error`는 에러 메시지 문자열이다. 에러 상태가 아닐 땐 의미가 없다. status가 'error'일 때만 채워두면 된다.

이렇게 솎아내고 나면 남는 것은 셋이다.

```tsx
type Status = 'typing' | 'submitting' | 'success' | 'error';

const [answer, setAnswer] = useState('');
const [status, setStatus] = useState<Status>('typing');
const [error, setError] = useState<string | null>(null);
```

뭐? 시각 상태가 다섯 개였는데 status는 네 가지뿐이라고? 'empty'는 어디 갔는지 짐작이 갈 것이다. **'empty'와 'typing'은 같은 상태다.** answer가 빈 문자열이냐 아니냐의 차이일 뿐이다. 시각적으로는 둘이 살짝 다르지만, 그 차이는 `answer.length === 0`이라는 파생값으로 충분히 표현된다. 두 시각 상태가 하나의 status로 합쳐지고, 그 안의 미세한 차이는 `answer`에서 파생되는 것이다.

엄밀하게 말하면 시각 상태와 state 변수가 1:1일 필요는 없다. 시각 상태 다섯 개가 status enum 네 개 + answer 한 줄로 표현되었다. 더 줄어든 것이다. 이렇게 줄어드는 게 4단계의 목적이다.

여기서 잠깐, **모순 가능 상태가 왜 위험한지** 한 번 더 짚자. 처음 코드에서 `isSuccess`와 `isError`는 별개의 state였다. 그 말은 "둘 다 true"인 조합이 코드상 가능하다는 뜻이다. 실제로는 둘 다 true가 되어선 안 되는데, 그걸 막는 건 우리의 부지런함뿐이다. 핸들러마다 잊지 말고 둘 다 false로 초기화해줘야 한다. 한 번이라도 빠뜨리면 버그다.

`status` enum으로 합치는 순간 그 모순은 **타입 시스템 차원에서 제거**된다. status는 네 값 중 정확히 하나다. 'success'면서 동시에 'error'인 값은 존재할 수 없다. 컴파일러가 우리의 부지런함을 대신 떠맡아준다. 부지런함을 컴파일러에게 떠넘길 수 있을 때 떠넘겨두는 것이 좋다.

조금 더 욕심을 내자면 status와 그에 따라오는 데이터를 **discriminated union**으로 묶는 방법도 있다. status가 'success'일 때만 data가 의미 있고, 'error'일 때만 message가 의미 있다는 사실을 타입에 새겨둘 수 있다.

```ts
type FormState =
  | { status: 'typing'; answer: string }
  | { status: 'submitting'; answer: string }
  | { status: 'success' }
  | { status: 'error'; answer: string; message: string };
```

이렇게 두면 `state.message`는 status가 'error'일 때만 접근 가능하다. 'typing' 상태에서 message에 접근하려 하면 컴파일러가 막아준다. **존재하지 않아야 할 데이터에 손을 댈 수 없게 만드는 것**이 정렬된 상태 설계의 또 다른 이점이다.

물론 이렇게까지 갈 필요가 늘 있는 것은 아니다. 작은 컴포넌트에서는 `status` enum + 별도 변수 두엇으로도 충분하다. 컴포넌트가 자라면서 가능한 데이터 조합이 많아질 때, discriminated union을 떠올리자. 더 일찍 도입하면 boilerplate가 늘고, 더 늦게 도입하면 버그가 늘어난다. 이 균형점은 컴포넌트마다 조금씩 다르다.

### 5단계 — 핸들러 연결하기

이제 트리거에 코드를 붙인다. 2단계에서 그린 표를 보면서 한 줄씩 옮긴다.

```tsx
async function handleSubmit(e: React.FormEvent) {
  e.preventDefault();
  setStatus('submitting');
  try {
    await submitAnswer(answer);
    setStatus('success');
  } catch (err) {
    setStatus('error');
    setError(err instanceof Error ? err.message : '알 수 없는 오류');
  }
}

function handleAnswerChange(e: React.ChangeEvent<HTMLInputElement>) {
  setAnswer(e.target.value);
  // 에러 상태에서 다시 타이핑하면 에러 해제
  if (status === 'error') {
    setStatus('typing');
    setError(null);
  }
}

function handleRetry() {
  setAnswer('');
  setStatus('typing');
  setError(null);
}
```

JSX는 status 한 변수에 의존해 분기한다.

```tsx
if (status === 'success') {
  return <SuccessMessage onRetry={handleRetry} />;
}

return (
  <form onSubmit={handleSubmit}>
    <h2>퀴즈를 풀어보자</h2>
    <p>리액트는 무엇의 함수로 UI를 표현하는가?</p>
    <input
      value={answer}
      onChange={handleAnswerChange}
      disabled={status === 'submitting'}
      aria-invalid={status === 'error'}
    />
    <button
      type="submit"
      disabled={status === 'submitting' || answer.length === 0}
    >
      {status === 'submitting' ? '제출 중...' : '제출'}
    </button>
    {status === 'error' && error && (
      <p role="alert" className="error">{error}</p>
    )}
  </form>
);
```

조건문 하나하나를 읽어보자. `disabled={status === 'submitting'}`. 이 줄은 "제출 중일 때 입력을 잠근다"를 그대로 코드로 옮겼다. 디자이너가 한 말과 줄 코드가 같은 모양이다. `aria-invalid={status === 'error'}`도 마찬가지. status를 보면 화면이 보인다. **state가 화면의 진실이고, JSX는 그 진실을 번역한 결과**라는 게 이런 모습이다.

처음의 명령형 코드에 비해 무엇이 달라졌는지 다시 보자. boolean 다섯 개 → state 변수 두 개 + 파생값. 핸들러는 매번 다섯 개를 토글하지 않고, status 한 줄만 바꾼다. 모순 가능 상태가 0이다. JSX의 조건은 한 줄에 한 가지만 본다. 디버깅하다 화면이 이상하면 status 한 변수만 들여다보면 된다. 처음의 32가지 조합은 어디 갔을까? 사라진 게 아니라, 실제로는 처음부터 없었다는 사실이 드러난 것이다.

여기서 잠깐 한 가지 흔한 함정에 대해 짚어두자. 어떤 사람들은 status 전이를 effect로 처리하려는 유혹을 받는다. 예를 들면 이렇게.

```tsx
// 이렇게 하지 말자
useEffect(() => {
  if (status === 'submitting') {
    submitAnswer(answer)
      .then(() => setStatus('success'))
      .catch((e) => {
        setStatus('error');
        setError(e.message);
      });
  }
}, [status, answer]);

function handleSubmit() {
  setStatus('submitting'); // effect가 이걸 보고 fetch한다
}
```

겉으로는 깔끔해 보인다. status가 바뀌면 effect가 알아서 통신을 하는 모양새다. 그런데 이 패턴은 **사용자 이벤트로 일어나야 하는 일을 effect에 옮긴** 안티패턴이다. 폼이 처음 마운트될 때 status가 'submitting'으로 시작했다면? 사용자가 누르지도 않았는데 제출이 발사된다. 또는 어떤 이유로 status가 짧은 순간 'submitting'을 거쳐갔다 하면 두 번 발사될 수도 있다. 더 심각한 건, 같은 사건(제출 버튼 클릭)에 대한 코드가 핸들러와 effect에 쪼개져 있어서 디버깅 시 어디서 무엇이 시작됐는지 추적하기 어려워진다는 점이다.

원칙은 단순하다. **사용자가 일으킨 일은 핸들러에서 처리한다.** effect는 외부 시스템과의 동기화에만 쓴다. 위 코드에서 `submitAnswer` 호출은 사용자 클릭이 일으킨 일이므로, 클릭 핸들러 안에서 직접 호출하면 된다. 우리가 5단계에서 짠 `handleSubmit`이 정확히 그 모양이다. 이 안티패턴은 11장에서 본격적으로 해부할 텐데, 일단 지금은 "status 전이를 effect로 트리거하지 말자"만 기억하자.

## 왜 boolean 두 개로 시작하면 망하는가

여기서 한 번 더 짚자. boolean 두 개로 시작하면 거의 항상 후회한다.

`isSending`과 `isSent`로 폼을 만들었다고 해보자. 처음엔 잘 돌아간다. 'sending'이면 `isSending=true, isSent=false`, 끝나면 `isSending=false, isSent=true`. 두 boolean이 깔끔히 양분된다.

그러다 "에러" 시각 상태가 추가된다. boolean 하나 더, `isError`. 이젠 세 boolean이 만드는 8가지 조합 중 의미 있는 건 셋뿐이다. 코드 어딘가에서 set을 빠뜨리면 `isSent=true, isError=true`가 동시에 가능해지고, "보냈는데 실패했다"는 모순이 화면에 떠오른다.

여기서 또 "재시도 중" 상태가 추가된다. boolean 네 개. 16가지 조합. 의미 있는 건 다섯 개. 11가지가 버그 후보지다. 한 핸들러에서 set을 빠뜨릴 확률이 점점 커진다.

이 곡선의 모양이 핵심이다. **boolean 개수는 선형으로 늘지만, 가능한 조합은 지수적으로 폭발한다.** 우리는 다섯 개 화면을 만들고 싶었을 뿐인데 32가지 조합을 손으로 막아야 하는 코드를 짜게 된다. boolean 두 개로 시작했을 때 이 곡선의 출발 지점에 서 있는 것이다. 출발 지점에서는 별 차이가 안 보이지만, 화면이 다섯 개로 늘어나는 순간 코드가 무너진다.

그래서 권장 휴리스틱은 단순하다.

> **두 개 이상의 시각 상태가 서로 배타적이라면, boolean 여러 개가 아니라 하나의 status enum을 쓰자.**

이 한 줄이 1장 전체의 핵심 권고다. 외워두자.

물론 boolean이 적합한 자리도 있다. 진짜로 서로 독립적인 두 차원이라면 boolean 두 개여도 된다. 예를 들어 "다크 모드인가"와 "사이드바가 열려 있는가"는 무관한 두 차원이고, 네 가지 조합이 모두 의미를 가진다. 이런 건 boolean이 맞다. **헷갈리면 이렇게 묻자. "이 두 boolean이 모두 true인 상태가 의미가 있는가?"** 의미가 없다면, 그건 enum이 되어야 할 두 값을 boolean으로 쪼개놓은 것이다.

이 질문은 강력하다. 코드 리뷰를 할 때 boolean 두 개가 보이면 거의 자동으로 이 질문을 던지게 된다. `isLoading`과 `isError`가 동시에 true인 상황이 의미가 있나? 없다 → enum 후보. `isDarkMode`와 `isSidebarOpen`이 동시에 true인 상황이 의미가 있나? 있다 → boolean 두 개로 두자. 이 한 줄짜리 질문이 우리를 모순 상태로부터 지켜준다.

또 하나의 변형 질문은 이것이다. **"이 boolean들이 동시에 모두 false인 상태가 의미가 있는가?"** `isLoading`, `isSuccess`, `isError`가 모두 false라면 그건 어떤 화면인가? 답하기 어렵다면, 그것 역시 enum 후보다. 'idle' 같은 명시적 시작 상태가 누락된 것이다.

그리고 한 가지 더, 컴포넌트가 자라면서 boolean을 enum으로 마이그레이션할 때 자주 듣는 푸념이 있다. "이미 코드 곳곳에 `isLoading`이 박혀 있는데 어떻게 다 바꿔요?" 그럴 땐 한 번에 다 바꿀 필요는 없다. 외부 인터페이스(예: 자식 컴포넌트에 props로 내려주는 boolean)는 잠시 유지하면서 내부적으로만 enum을 쓰고, 파생값으로 boolean을 만들어 호환시키는 방식도 있다.

```tsx
const status: Status = /* ... */;
const isLoading = status === 'submitting';
const isError = status === 'error';
// isLoading, isError는 외부에 보낼 때만 쓰고, 내부 분기는 status로
```

이렇게 두면 단계적 마이그레이션이 가능하다. 한 번에 다 바꾸려다가 PR이 거대해져 리뷰가 막히는 것보다, 작은 PR로 안쪽부터 enum화하는 편이 훨씬 안전하다. 무엇을 바꾸느냐만큼 **어떻게 바꾸느냐**도 중요하다.

## 자주 마주치는 변형들

다섯 단계를 익히고 나면 비슷한 형태의 컴포넌트들이 한꺼번에 정리된다. 폼 외에도 일상에서 자주 만나는 패턴 몇 가지를 짧게 짚어두자. 매번 이 절차를 처음부터 밟지 않아도 되도록 머릿속에 준비된 모양을 만들어 두자는 의도다.

### 변형 1 — 데이터 페치 컴포넌트

서버에서 데이터를 가져와 보여주는 컴포넌트는 이 책에서 끊임없이 등장한다. 이 컴포넌트의 시각 상태는 거의 항상 같은 모양이다.

```tsx
type FetchStatus = 'idle' | 'loading' | 'error' | 'success';

const [status, setStatus] = useState<FetchStatus>('idle');
const [data, setData] = useState<User | null>(null);
const [error, setError] = useState<string | null>(null);
```

'idle'을 처음부터 두는 게 좋다. 어떤 컴포넌트는 마운트 즉시 fetch를 시작하지만, 어떤 컴포넌트는 사용자 액션을 기다린다. 그 둘을 같은 코드로 다루려면 'idle' 같은 시작 상태가 필요하다. 처음부터 'loading'으로 두면 "아직 시작 안 한" 화면이 사라져버린다.

물론 본격적인 데이터 페칭은 TanStack Query나 SWR 같은 라이브러리로 옮기는 편이 낫다는 얘기를 12장에서 자세히 할 텐데, 이런 라이브러리들이 내부적으로 들고 있는 상태도 결국 비슷한 enum이다. **상태 모델링의 멘탈 모델은 라이브러리를 쓰든 안 쓰든 같다.**

### 변형 2 — 모달 / 다이얼로그

모달도 시각 상태가 정확히 떨어진다.

```tsx
type ModalStatus = 'closed' | 'open' | 'closing' | 'opening';
```

대부분은 'closed'와 'open'만 있어도 된다. 그런데 애니메이션이 들어가는 순간 'closing', 'opening'이 추가로 필요해진다. 단순히 `isOpen` boolean만 두면 닫힘 애니메이션 도중 사용자가 열기 버튼을 다시 눌렀을 때 화면이 깜빡이거나 멈춰버리는 현상이 생긴다. 'closing' 동안엔 새로운 'open' 트리거를 무시해야 하는지, 즉시 'opening'으로 전환해야 하는지를 표에 적어두자.

### 변형 3 — 위저드 / 다단계 폼

여러 단계로 구성된 가입 위저드 같은 화면을 보자. 첫 충동은 `currentStep` number를 두고 1, 2, 3을 토글하는 것이다. 이 자체는 잘못된 게 아니다. 다만 단계마다 보유해야 하는 데이터의 모양이 다르다면, status가 단계 이름인 enum과 단계별 데이터를 묶는 편이 안전하다.

```tsx
type WizardState =
  | { step: 'account'; email: string }
  | { step: 'profile'; email: string; name: string }
  | { step: 'plan'; email: string; name: string; plan: 'free' | 'pro' }
  | { step: 'done' };
```

이렇게 두면 'plan' 단계에서는 `state.name`을 안전하게 읽을 수 있고, 'account' 단계에서 `state.plan`에 접근하려는 시도는 컴파일러가 막는다. 위저드는 18장에서 useReducer와 함께 다시 등장한다. 그때까지 이 모양을 머릿속에 두자.

### 변형 4 — 토글 + 통신을 동반하는 버튼

"좋아요" 버튼 같은 컴포넌트는 겉보기엔 boolean 하나처럼 보인다. 그런데 통신이 끼면 더 복잡해진다. 사용자가 버튼을 누르면 즉시 UI를 바꾸고(낙관적 업데이트), 서버에 요청을 보내고, 실패하면 되돌리는 식이다. 시각 상태는 다음과 같이 정리된다.

```tsx
type LikeStatus = 'idle' | 'pending' | 'rolledBack';
```

여기서 `liked: boolean`은 별도 변수로 둔다. liked는 화면이 보여줄 모습이고, status는 통신이 어디까지 왔는지를 추적한다. 이 둘이 분리되어 있는 게 핵심이다. 합치려고 하면 'idle-and-liked', 'idle-and-not-liked', 'pending-and-liked'… 식으로 enum 값이 폭발한다. **두 개의 독립 차원이 있을 땐 두 변수**가 정답이다. boolean이 적합한 자리는 바로 이런 곳이다.

이 네 변형은 책 후반부에서 다시 만난다. 지금은 "패턴이 반복된다"는 사실만 머리에 새겨두자. 새 컴포넌트를 만들 때 "어, 이거 어디서 본 모양인데" 하고 떠오르면 절반은 끝난 셈이다.

## 현장 사례 — 친구의 코드 한 컷

가까운 동료가 코드를 보여주며 도와달라고 했다고 해보자. 검색 화면이다.

```tsx
const [query, setQuery] = useState('');
const [results, setResults] = useState<Item[]>([]);
const [isSearching, setIsSearching] = useState(false);
const [isEmpty, setIsEmpty] = useState(true);
const [hasSearched, setHasSearched] = useState(false);
const [error, setError] = useState<string | null>(null);

useEffect(() => {
  if (query.trim().length === 0) {
    setIsEmpty(true);
    setResults([]);
    return;
  }
  setIsEmpty(false);
  setIsSearching(true);
  setHasSearched(true);
  search(query)
    .then((items) => {
      setResults(items);
      setError(null);
    })
    .catch((e) => setError(e.message))
    .finally(() => setIsSearching(false));
}, [query]);
```

처음 보면 "그럭저럭 돌아가는 코드"처럼 보일 수도 있다. 그런데 다섯 단계의 안경을 끼고 다시 보자.

먼저 1단계로 시각 상태를 적어본다. 사용자가 보게 될 화면은 무엇인가?

- 검색어가 비어 있고 아직 한 번도 검색 안 한 상태 — "검색어를 입력하세요"
- 검색 중 — 스피너
- 검색 결과 있음
- 검색 결과 없음 (빈 결과)
- 에러

다섯 개. 그런데 코드의 boolean들은 어떤가? `isSearching`, `isEmpty`, `hasSearched`, `error`. 게다가 `results`의 길이도 화면 결정에 쓰인다. 다섯 개의 시각 상태를 표현하기 위해 네 개의 독립적인 변수가 동원된 것이다. 모순 가능 조합이 적지 않다. 예를 들어 `isSearching=true`이면서 `error`가 남아 있는 경우엔 화면이 어떻게 그려질지 자기도 모른다.

또 한 가지 문제는 **사용자 이벤트(query 변경)로 일어나는 일을 effect에 넣었다**는 점이다. 디바운싱을 위해 effect가 자연스러워 보일 수도 있지만, race condition 처리가 없다는 더 큰 문제가 숨어 있다. 사용자가 'a'를 쳤다가 빠르게 'ab'로 바꾸면, 'a'에 대한 응답이 늦게 도착해 'ab'의 결과를 덮어쓸 수 있다. 이 race는 11장에서 본격적으로 다루지만, 일단 여기서는 다섯 단계 절차를 적용해 정리해보자.

다섯 단계로 다시 짠 버전은 이렇다.

```tsx
type SearchStatus = 'idle' | 'searching' | 'success' | 'error';

function SearchBox() {
  const [query, setQuery] = useState('');
  const [status, setStatus] = useState<SearchStatus>('idle');
  const [results, setResults] = useState<Item[]>([]);
  const [error, setError] = useState<string | null>(null);

  async function handleSearch(nextQuery: string) {
    setQuery(nextQuery);
    if (nextQuery.trim().length === 0) {
      setStatus('idle');
      setResults([]);
      return;
    }
    setStatus('searching');
    try {
      const items = await search(nextQuery);
      setResults(items);
      setStatus('success');
    } catch (e) {
      setError(e instanceof Error ? e.message : '검색 실패');
      setStatus('error');
    }
  }

  return (
    <div>
      <input value={query} onChange={(e) => handleSearch(e.target.value)} />
      {status === 'idle' && <p>검색어를 입력하세요</p>}
      {status === 'searching' && <Spinner />}
      {status === 'success' && results.length === 0 && <p>결과가 없습니다</p>}
      {status === 'success' && results.length > 0 && <ResultList items={results} />}
      {status === 'error' && error && <p role="alert">{error}</p>}
    </div>
  );
}
```

물론 이 코드도 race condition 문제가 남아 있다. 그건 14장 이후에 `ignore` 플래그나 `AbortController`로 처리할 것이다. 다만 보아야 할 건 **시각 상태가 status 한 변수로 깔끔히 떨어졌고, 핸들러는 한 번의 사용자 이벤트에 맞춰 한 자리에서 모든 일을 한다**는 점이다. boolean 세 개가 사라졌고, 모순 가능 조합도 사라졌다. 다섯 단계가 한 컴포넌트를 어떻게 정리하는지 감이 올 것이다.

이 정도 정리만 해도 코드 리뷰에서 "이 부분이 왜 이렇게 됐어요?" 같은 질문이 줄어든다. 처음부터 절차를 따라 짠 코드는 설명할 게 별로 없다. **읽는 사람의 머리에 부담을 덜 주는 코드**가 좋은 코드라면, 다섯 단계는 그 부담을 줄이는 절차다.

## 다섯 단계 체크리스트

작업 중에 옆에 두고 짚을 수 있게 정리하자.

- [ ] **① 시각 상태 식별:** 종이에 모든 시각 상태에 이름을 붙였다. 이름은 명사다.
- [ ] **② 트리거 정의:** 각 상태 사이 전이를 표로 적었다. 사람의 트리거와 시스템의 트리거를 분리했다.
- [ ] **③ state 표현:** 시각 상태를 state 변수로 옮겼다. 일단 단순하게.
- [ ] **④ 비본질 제거:**
  - [ ] 파생 가능한 값을 state에서 뺐다.
  - [ ] 모순 가능한 boolean들을 status enum 하나로 합쳤다.
  - [ ] 같은 정보를 두 곳에서 추적하지 않는다.
- [ ] **⑤ 핸들러 연결:** 트리거 표를 보고 한 줄씩 코드로 옮겼다. 각 핸들러는 status를 한 번만 바꾼다.

이 다섯 단계는 폼뿐 아니라 모달, 다이얼로그, 토스트, 위저드, 서버 페치 결과 표시까지, "여러 시각 상태를 갖는 컴포넌트"라면 어디든 적용된다. 한 번 적용해두면 이후 비슷한 컴포넌트를 만들 때마다 머릿속에 자동으로 다섯 단계가 떠오를 것이다. 여기까지가 멘탈 모델의 첫 토대다.

## 짧은 막간 — 왜 boolean이 그렇게 매혹적인가

여기서 잠시 호흡을 고르고, 한 가지 의문에 답해보자. 우리가 boolean 다섯 개의 늪을 그렇게 자주 만들면서도 매번 비슷하게 빠지는 이유는 뭘까? 이미 한두 번 호되게 당해본 사람도 다음 컴포넌트에서 또 boolean부터 적기 시작한다. 의지력 부족이라고 자책할 일은 아니다. 사실 boolean은 우리에게 너무 매혹적인 도구다.

첫째, **boolean은 즉답이 가능하다.** "지금 로딩 중인가?"라는 물음에 `isLoading` 한 단어로 답할 수 있다. 추가 비교 없이 그 자체가 결과다. 사람의 머리는 이런 즉답형 표현을 좋아한다. status enum은 비교 한 번을 거쳐야 한다. 매혹은 즉답에 있다.

둘째, **boolean은 한 번에 하나씩 추가된다.** "스피너를 보여줘야 한다" → `isLoading` 하나 추가. "에러 메시지가 필요하다" → `isError` 하나 추가. 한 번에 한 boolean씩 늘어나는 게 자연스러운 변화처럼 느껴진다. 그런데 이 자연스러움이 함정이다. 한 번에 한 개씩 추가되어 결국 다섯 개가 되었을 때 멈춰서 "이걸 enum으로 합쳐야 하지 않나?"를 묻는 사람이 적다. 이미 다섯 개가 차례로 들어와 있는데, 그걸 한꺼번에 묶기는 새로 짜는 것보다 어렵게 느껴진다.

셋째, **boolean은 코드 리뷰를 잘 통과한다.** 한 줄 추가에 대한 PR은 작고, 리뷰어도 부담 없이 승인한다. 다섯 PR이 누적되어 다섯 boolean이 됐을 때, 어느 PR도 "잠깐, 우리 이걸 enum으로 바꾸자"라는 말을 끌어내지 못한다. 변화가 점진적일수록 그 변화 안의 잘못된 방향성을 잡아내기가 어렵다.

이 세 가지 매혹은 인간적이고, 부정해서 이길 게 아니다. 그래서 우리에겐 **절차**가 필요하다. 매혹이 우리를 끌고 가지 않도록, 새 컴포넌트나 새 boolean을 추가하기 전에 잠시 멈춰 다섯 단계를 떠올리는 습관을 들여야 한다. 절차가 매혹을 이긴다.

또 하나, **이미 늪에 빠진 코드를 정리할 때 마음을 가볍게 갖자.** 이걸 일찍 발견하지 못한 자신을 자책하지 말자. boolean의 매혹은 강력하고, 모두가 한 번씩은 그 매혹에 끌려간다. 중요한 건 늪에서 나올 절차를 가지고 있느냐다. 다섯 단계가 그 절차다.

## 한 발짝 더 — 상태 머신과 짧게 만나기

다섯 단계 절차가 익숙해지면 자연스럽게 다음 질문이 따라온다. "이거 결국 상태 머신 아닌가?" 정확히 그렇다. 우리가 그린 시각 상태 목록은 상태 머신의 노드이고, 트리거 표는 전이(transition)다. 차이가 있다면 우리는 그것을 라이브러리 없이 평범한 useState와 enum으로 표현했다는 점뿐이다.

상태 머신을 더 본격적으로 다루는 방법으로 XState 같은 라이브러리가 있다. XState는 상태와 전이를 선언적으로 그리고, 잘못된 전이를 컴파일/런타임에서 막아준다. "이 상태에선 이 사건만 받을 수 있다"를 코드 차원에서 강제할 수 있다. 큰 폼이나 결제 흐름처럼 전이가 복잡한 경우에 진가가 드러난다.

그런데 이 책에서는 XState를 본격적으로 다루진 않는다. 이유는 단순하다. **대부분의 컴포넌트는 라이브러리 없이도 충분하다.** status enum + 핸들러 안에서의 status 갱신 정도로 95%의 화면이 깔끔히 정리된다. 그 95%를 잘 처리할 수 있게 되는 게 이 장의 목표다. 나머지 5%, 정말 복잡한 흐름을 만나면 그때 XState나 useReducer를 검토하면 된다.

여기서 강조하고 싶은 건 **어휘의 힘**이다. "상태 머신"이라는 어휘를 알고 있으면, boolean이 늘어나는 코드를 볼 때마다 "이건 상태 머신으로 보면 더 단순해질까?"라는 질문이 자동으로 떠오른다. 어휘가 안목을 만든다. 라이브러리를 도입하지 않더라도 어휘는 가져가자.

그리고 한 가지, 상태 머신의 관점에서 우리 다섯 단계를 다시 보면 절차가 더 분명해진다. 1단계는 노드 정의, 2단계는 엣지 정의, 3단계는 노드의 데이터 표현, 4단계는 노드와 엣지의 정규화, 5단계는 사용자 사건을 엣지에 연결하는 단계다. 학술적으로 보면 그래프 이론의 한 응용이지만, 실용적으로 보면 그저 잘 그린 그림에 코드를 붙이는 절차일 뿐이다. 어렵게 받아들일 필요가 없다.

## 다른 사람이 쓴 코드를 다섯 단계로 진단하기

새 코드를 처음부터 짜는 건 어쩌면 쉬운 일이다. 진짜 어려운 일은 **이미 있는 코드**를 어떻게 정리하느냐다. 동료가 짠 코드, 몇 달 전 자신이 짠 코드, 오픈소스 라이브러리의 컴포넌트 — 이런 것들을 만났을 때 "이 코드는 정렬할 가치가 있는가?"부터 판단해야 한다.

다음 신호가 보이면 다섯 단계로 다시 정렬할 가치가 있다.

- **boolean이 세 개 이상**이고, 그 boolean들이 같은 시각 변화를 다양하게 조합한다.
- **set 호출이 한 핸들러 안에서 네 개 이상** 일어난다.
- 핸들러 사이에 **같은 set 패턴이 반복**된다 — `setIsLoading(true); setIsError(false); setIsSuccess(false);` 같은 식.
- 버그 리포트가 **모순 조합**(예: "성공 메시지와 에러가 동시에 떠요")으로 들어온 적이 있다.
- JSX 안에 **`&&` 체인이 길거나** `!isError && !isLoading && hasData` 같은 부정의 결합이 보인다.
- 컴포넌트 props에 **boolean이 다섯 개 이상** 들어오고 있다 — 자식 컴포넌트도 이미 부모의 boolean 늪에 끌려가고 있을 가능성이 높다.

신호가 보이면 일단 시각 상태부터 적어보자. 종이에 다섯 가지 미만으로 떨어진다면 enum으로 거의 확실히 정리된다. 그 이상이라면 컴포넌트를 쪼갤 신호인지 따져본다. 정리 결과가 좋다면 PR을 작은 단위로 쪼개 보내자. 한 번에 모든 boolean을 다 바꾸기보다, 한 컴포넌트씩 정리해 PR 두세 개로 나눠 보내는 편이 리뷰가 빠르다.

진단할 때 한 가지 함정은 **"잘 돌아가는 코드를 굳이 건드릴 필요 있나?"**라는 의문이다. 답은 "버그가 나기 전엔 안 건드려도 된다"이다. 다섯 단계는 모든 코드를 강제로 정리하라는 명령이 아니다. 그저 정리할 가치가 보일 때 따라가면 되는 절차다. 이미 잘 돌아가는 작은 컴포넌트를 enum으로 옮기느라 하루를 쓰는 건 시간 낭비일 수 있다. **새로 자라는 컴포넌트와, 자주 버그가 나는 컴포넌트**를 우선순위에 두자.

## 자주 받는 질문들

이 다섯 단계를 처음 적용해보는 사람들이 자주 던지는 질문이 몇 가지 있다. 짚고 넘어가자.

### Q. status enum을 쓰면 useState 호출이 줄어드는데, 이게 성능에 영향을 주나?

리렌더 횟수 측면에서는 거의 차이가 없다. 리액트는 한 이벤트 핸들러 안에서 일어난 여러 set 호출을 batching해 한 번에 렌더한다. boolean 다섯 개를 한꺼번에 바꾸든, status 하나를 바꾸든 결과 렌더는 한 번이다. 다만 **개념적 단순함**과 **버그 회피**라는 측면에서 차이가 크다. 성능 때문에 enum을 권하는 게 아니라, 코드를 읽고 디버깅하는 비용 때문에 권하는 것이다. 성능 최적화는 다른 책 다른 장에서 다룰 일이다(이 책에서는 4장에서 useMemo를 다룰 때 잠시 등장한다).

### Q. status에 보관할 데이터가 많은데, 매번 모든 필드를 다시 set해야 하나?

discriminated union을 쓰면 그렇게 된다. 'success' 상태로 옮길 때 data를, 'error' 상태로 옮길 때 message를 함께 set해야 한다. 처음엔 번거롭게 느껴질 수 있는데, **그 번거로움이 곧 안전망**이다. data 없이 'success'로 갈 수 없게 강제하는 것이 모순 상태를 막아준다. 정말 빈번한 갱신이 일어난다면 useReducer로 옮겨 한 번의 액션으로 status와 데이터를 함께 바꾸는 패턴이 깔끔하다. 9장에서 자세히 다룬다.

### Q. 시각 상태가 너무 많이 식별되면 어떻게 하나? 열 개가 넘어가는데.

먼저 두 가지를 점검한다. 첫째, **정말 별개의 상태인지** — "버튼이 빨간색인 상태"와 "버튼이 빨간색이고 약간 흔들리는 상태"는 같은 상태일 가능성이 높다. 흔들림은 status에서 파생되는 일시적 효과지 별도 상태가 아니다. 둘째, **하위 컴포넌트로 쪼개야 할 신호**일 수도 있다. 한 컴포넌트가 가져야 할 시각 상태가 너무 많다면, 그 컴포넌트가 두세 개의 책임을 함께 지고 있다는 신호다. 모달 안의 폼은 모달과 폼이라는 두 컴포넌트로 쪼개고, 각자 자기 status를 갖게 하면 자연스럽게 정리된다.

### Q. 다섯 단계를 매번 종이에 그려야 하나? 너무 번거롭지 않나?

처음 한 달 정도는 종이에 그리는 편이 낫다. 손이 절차에 익숙해지는 데에 시간이 걸린다. 익숙해지고 나면 머릿속으로 빠르게 훑게 되고, 결국엔 useState를 적기 직전 0.5초쯤 멈춰서 "이게 정말 enum 후보가 아닌가?"를 묻는 정도가 된다. **번거로움은 익숙함이 되고, 익숙함은 안목이 된다.** 다만 정말 복잡한 컴포넌트(위저드, 결제 흐름 등)는 익숙해진 뒤에도 종이에 그리는 편이 좋다. 손으로 그리는 5분이 일주일 디버깅을 막아준다.

### Q. 기존 코드에 이 패턴을 도입하려는데 동료들이 동의를 안 한다.

큰 PR로 한 번에 바꾸지 말고, 한 컴포넌트를 골라 작은 PR로 보여주자. 코드가 줄어들고 버그 가능성이 사라지는 모습을 직접 보면 대부분 설득된다. 말로 설득하는 것보다 diff로 설득하는 편이 훨씬 효과적이다. 그리고 단어를 신중히 고르자. "boolean이 잘못됐다"가 아니라 "이 두 boolean의 모순 조합이 가능한데, enum으로 합치면 불가능해진다" 식으로, 구체적인 결함을 짚자.

## 흔한 반론과 답하기

이 절차를 동료들에게 소개하면 자주 나오는 반론들이 있다. 미리 짚어두면 토론할 때 도움이 된다.

### 반론 1 — "boolean이 더 직관적인데?"

`isLoading`이라는 단어는 짧고 영어로 자연스럽다. `status === 'submitting'`은 길고, 한 번 더 비교를 해야 한다. 이 직관은 사실이다. 한 줄만 보면 boolean이 더 짧다.

문제는 한 줄이 아니라 **컴포넌트 전체**를 볼 때 드러난다. boolean 다섯 개를 토글하는 핸들러 vs status 한 번 바꾸는 핸들러를 비교하면 어느 쪽이 짧은지 명확하다. JSX 안의 조건부 렌더링도 boolean 조합을 검사하느라 길어지기 일쑤다. 한 줄짜리 비교는 길어져도, 줄 수와 분기 수는 줄어든다. **국소적 단순함**과 **전역적 단순함**의 트레이드오프인데, 우리는 후자를 택하는 편이 낫다.

### 반론 2 — "타입스크립트 안 쓰는데 enum이 무슨 의미가 있어?"

JavaScript만 쓴다고 해도 의미가 있다. enum 값은 `'typing'`, `'submitting'` 같은 문자열일 뿐이다. 타입 체커가 없어도 **이름이 문서**가 된다. `setStatus('submitting')`이라는 한 줄을 보는 사람은 그게 무슨 의미인지 즉시 안다. boolean 다섯 개를 토글하는 코드는 그 자체로 의미가 드러나지 않는다. 게다가 런타임에서 `status === 'submitting'` 비교는 boolean 비교만큼이나 빠르다. 성능 부담은 없다. 이름이 주는 가독성을 챙기는 것이다.

물론 타입스크립트를 쓰면 더 강력해진다. 컴파일러가 모순 상태를 막아주고, discriminated union이 제 역할을 한다. 이 책의 코드는 기본적으로 타입스크립트로 적는데, 그 이유 중 하나가 바로 이 장에서 보여준 안전망이다. 자바스크립트로도 가능하지만, 타입스크립트가 부지런함을 더 많이 떠맡아준다.

### 반론 3 — "리액트답지 않은 거 아닌가? 그냥 useState로 단순하게 짜는 게 React Way 아니냐."

오해가 있다. **리액트답다는 건 useState를 많이 쓰는 게 아니라, UI를 상태의 함수로 표현하는 것**이다. 리액트 공식 문서의 "Reacting to Input with State" 글이 정확히 우리가 다룬 다섯 단계를 권한다. status enum 패턴은 React 팀이 직접 추천하는 방식이다. boolean을 늘리는 건 리액트가 제공하는 도구를 쓰는 것일 뿐, 리액트의 정신을 살리는 것은 아니다. 도구 사용과 정신은 다르다.

### 반론 4 — "결국 useReducer 쓰면 되는 거 아니냐?"

useReducer가 더 정돈된 답일 때가 많다. 그런데 useReducer로 가기 전에도 status enum + useState만으로 충분히 단정한 코드가 가능하다. 작은 폼은 useState로 시작하고, 핸들러가 늘어나거나 액션 단위로 정리할 가치가 보이면 useReducer로 옮기자. 9장에서 그 경계선을 자세히 다룬다. 핵심은 **enum 사고**가 useState든 useReducer든 어느 도구에서나 통한다는 점이다. 도구를 바꾸기 전에 사고법을 먼저 갖추자.

### 반론 5 — "이미 만들어 놓은 컴포넌트가 너무 많은데, 다 바꿔야 해?"

그렇지 않다. 이미 잘 돌아가고 자주 안 만지는 컴포넌트는 그대로 둬도 된다. 절차는 **새로 짜는 코드**와 **자주 만지는 코드**부터 적용하자. 절차를 어디에 적용할지 우선순위를 두는 안목도 함께 길러야 한다. 모든 코드를 한 번에 정리하려는 시도는 거의 항상 실패한다. 마이그레이션은 콩나물 자라듯 자연스럽게.

## 절차의 정신 — 다섯 단계 너머

다섯 단계는 절차다. 그런데 절차의 본질이 뭔지를 한 번 더 짚자. 우리가 정말로 하려는 일은 **머릿속의 화면과 코드의 상태를 같은 모양으로 맞추는 것**이다. 디자이너의 머리에는 다섯 장의 그림이 있고, 우리는 그 다섯 장을 다섯 가지 status로 옮긴다. 머릿속과 코드가 같은 모양일 때, 코드를 읽으면 화면이 떠오르고 화면을 보면 코드의 어디가 어떻게 돼야 하는지 떠오른다.

이 일치가 무너질 때 버그가 자란다. 머리에는 다섯 장이 있는데 코드에는 32가지 조합이 있다. 그 차이만큼이 버그의 서식지다. 다섯 단계는 그 차이를 0에 가깝게 만들기 위한 절차다.

그래서 다섯 단계를 기계적으로 따르기만 하는 것보다, **왜 이 단계가 필요한지**를 곱씹는 편이 좋다. ①은 머릿속의 그림에 이름을 붙이는 단계다. 이름이 붙어야 코드로 옮길 수 있다. ②는 그림 사이의 연결을 명시화하는 단계다. ③은 그림을 데이터로 바꾸는 단계, ④는 데이터에서 비본질을 솎아내는 단계, ⑤는 데이터 변경을 사용자 사건과 연결하는 단계다. 각 단계가 차곡차곡 쌓여 머릿속과 코드를 같은 모양으로 만든다.

여기서 한 가지 위로의 말을 보태자. **처음부터 완벽하게 짤 필요는 없다.** 위에서 본 검색 컴포넌트의 첫 버전도 동작은 했다. 다만 자라나면서 버그가 늘어났을 뿐이다. 컴포넌트가 작을 땐 boolean 두세 개로 시작해도 괜찮고, 자라기 시작할 때 enum으로 옮기면 된다. 다섯 단계는 새 코드에도 적용되지만, **자라나는 코드를 다시 정렬하는 도구**로도 강력하다. 매번 처음부터 완벽하려고 애쓰기보다, 신호가 보일 때마다 절차를 적용해 정리하자. 코드는 자라고, 우리의 안목도 자란다.

## 핵심 정리

기억해둘 만한 것들을 압축하자.

- **UI는 상태의 함수다.** `UI = f(state)`. 같은 state면 같은 화면이 나와야 한다.
- **명령형 사고는 변화의 단계를 보고, 선언형 사고는 각 상태의 모습을 본다.** 컴포넌트 안에서는 선언형이 기본값이다.
- **시각 상태에 먼저 이름을 붙이자.** 코드 위에서 상태를 발명하지 말고, 화면을 보고 상태를 발견한다.
- **트리거를 사람과 시스템으로 분리해 적자.** 사용자 이벤트로 일어나는 일은 핸들러에, 시스템 응답은 effect에. 둘이 섞이기 시작하면 디버깅 지옥이다.
- **모순 가능한 state는 타입으로 막는다.** 서로 배타적인 boolean 여러 개는 status enum 하나로 합치자.
- **파생 가능한 값은 state로 두지 말자.** 다음 장에서 본격적으로 다룬다.
- **state가 화면의 진실이고, JSX는 그 진실의 번역이다.** JSX 조건문은 한 줄에 한 가지만 본다.
- **boolean 개수는 선형, 가능한 조합은 지수적이다.** 출발 지점에서부터 enum을 고려하자.
- **다섯 단계 절차를 외우자.** 식별 → 트리거 → 표현 → 비본질 제거 → 핸들러.

## 연습 문제

종이와 코드 에디터 둘 다 옆에 두고 풀어보자. 정답이 하나는 아니지만, 다섯 단계를 따라가면서 풀었는지가 중요하다.

연습 문제는 책을 덮은 뒤 키보드를 만져보라고 떠밀기 위한 것이다. 답은 정해져 있지 않다. 다섯 단계 절차를 자기 손으로 한 번씩 밟아본 뒤 자신만의 답을 만들면 된다. 답을 다 짠 뒤에는 친구나 동료에게 보여주고 "어디서 모순이 가능한가?"를 물어보자. 누군가 모순 조합을 찾아낸다면 그건 아직 enum이 충분히 단단해지지 않았다는 신호다.

### 연습 1 [기초] — 페치 UI를 단일 enum으로 모델링하기

다음과 같은 컴포넌트가 있다고 해보자. 외부 API에서 사용자 프로필을 가져와 보여주는 화면이다.

```tsx
function UserProfile({ userId }: { userId: string }) {
  const [isLoading, setIsLoading] = useState(true);
  const [hasError, setHasError] = useState(false);
  const [user, setUser] = useState<User | null>(null);
  const [errorMessage, setErrorMessage] = useState('');
  // ... fetch 로직
}
```

다음을 수행하라.

1. 시각 상태에 이름을 붙여라. (idle / loading / error / success 정도가 자연스럽다.)
2. boolean 두 개와 user, errorMessage를 합쳐 단일 `status: 'idle' | 'loading' | 'error' | 'success'` enum으로 모델링하라.
3. data와 error는 status가 각각 'success', 'error'일 때만 의미를 갖도록 정리하라. 가능하다면 discriminated union으로 표현해보자.
4. 모순 가능한 조합이 0이 되었음을 직접 확인하라.

힌트:

```tsx
type FetchState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; message: string };
```

이렇게 하면 `state.data`는 status가 'success'일 때만 접근할 수 있다. 타입 시스템이 우리를 도와준다는 게 어떤 느낌인지 감이 올 것이다.

### 연습 2 [중] — 신호등 boolean 셋을 status enum으로 리팩토링

다음 코드는 신호등을 표현한다.

```tsx
function TrafficLight() {
  const [isRed, setIsRed] = useState(true);
  const [isYellow, setIsYellow] = useState(false);
  const [isGreen, setIsGreen] = useState(false);

  function next() {
    if (isRed) {
      setIsRed(false);
      setIsGreen(true);
    } else if (isGreen) {
      setIsGreen(false);
      setIsYellow(true);
    } else {
      setIsYellow(false);
      setIsRed(true);
    }
  }

  return (
    <div>
      <div className={`light red ${isRed ? 'on' : ''}`} />
      <div className={`light yellow ${isYellow ? 'on' : ''}`} />
      <div className={`light green ${isGreen ? 'on' : ''}`} />
      <button onClick={next}>다음</button>
    </div>
  );
}
```

다음을 수행하라.

1. 이 코드에서 모순 가능한 상태가 몇 가지인지 적어보라. (세 boolean이 만드는 8가지 조합 중 의미 있는 건 몇 개인가?)
2. `light: 'red' | 'yellow' | 'green'` 한 변수로 리팩토링하라.
3. `next` 핸들러가 한 번에 한 줄만 set 호출하도록 만들어라.
4. 빨강 → 초록 → 노랑 → 빨강 사이클이 그대로 유지되는지 확인하라.

생각해볼 거리: 만약 "고장 났음(off)" 상태가 추가된다면 어떻게 확장할 텐가? boolean 네 개 vs enum 네 값. 어느 쪽이 잘 늘어나는지 비교해보자.

### 연습 3 [도전] — 모달 시각 상태를 종이에 그린 뒤 state로 옮기기

이번엔 거꾸로 가본다. **코드를 먼저 짜지 말고**, 종이부터 꺼내자.

요구사항: "이 작업을 정말 삭제하시겠습니까?" 확인 모달을 만든다. 동작은 다음과 같다.

- 사용자가 삭제 버튼을 누르면 모달이 열린다.
- 모달 안에는 "확인" 버튼과 "취소" 버튼이 있다.
- 확인을 누르면 서버에 삭제 요청을 보내고, 그동안 모달은 잠긴 상태로 스피너를 보여준다.
- 삭제가 성공하면 모달이 닫히고 부모 컴포넌트에 알린다.
- 삭제가 실패하면 모달 안에 에러 메시지가 뜨고, 사용자는 다시 시도하거나 취소할 수 있다.

다음 단계로 진행하라.

1. **종이에** 가능한 시각 상태를 모두 그려라. (closed / confirming / deleting / error 정도가 떠오를 것이다. 더 있을 수도 있다.)
2. 트리거 표를 작성하라. 어떤 트리거가 어떤 전이를 일으키는가? 사람과 시스템을 분리하라.
3. status enum과 추가로 필요한 state를 정의하라. 어떤 값이 status에서 파생 가능한가?
4. 핸들러 셋(`handleOpen`, `handleConfirm`, `handleCancel`)과 JSX를 작성하라.
5. 마지막으로 자문하라. **"이 컴포넌트의 어떤 두 boolean이 동시에 true가 되는 모순이 가능한가?"** 0이어야 한다.

도전 과제: 사용자가 모달이 열린 채로 ESC 키를 누르면 닫히게 만들어라. 단, 'deleting' 상태에서는 닫히면 안 된다(요청 중이니까). 어디에 어떤 조건을 걸어야 할까?

## 한 번 더, 천천히 — 마지막 정리

이 장이 길었다. 한 번에 모든 걸 소화하기는 쉽지 않다. 한 번 더, 가장 중요한 흐름을 천천히 짚어보자.

우리는 boolean 다섯 개의 늪에서 시작했다. 폼 하나를 만들고 있었는데, 요구사항이 늘어날 때마다 boolean이 늘어났다. 가능한 조합은 지수적으로 늘어났고, 그중 의미 있는 조합은 일부였다. 의미 없는 조합 하나하나가 잠재적 버그의 서식지였다. 우리는 부지런함으로 그것을 막으려 했지만, 부지런함은 자주 흔들린다.

여기서 우리는 사고법을 바꿨다. 코드 위에서 boolean을 발명하지 말고, 화면 위에서 시각 상태를 발견하자. 디자이너의 사고법을 빌려 다섯 가지 화면에 이름을 붙이고, 화면 사이의 길을 표로 그리고, 그 그림을 코드로 옮겼다. boolean 다섯 개는 status enum 한 줄로 줄어들었고, 모순 가능 조합은 0이 되었다. 같은 일을 하는 코드의 줄 수가 줄어들었고, 디버깅 시 보아야 할 변수의 수도 줄어들었다.

이 변화의 핵심은 **state를 화면의 진실로 본 것**이다. state가 정해지면 화면이 정해진다. 화면이 이상하면 state를 의심한다. 사용자 사건이 일어나면 state를 바꾼다. 화면을 직접 바꾸는 명령형 사고에서, state를 통해 간접적으로 바꾸는 선언형 사고로 옮긴 것이다.

다섯 단계는 그 사고를 절차로 만든 도구다. 시각 상태 식별 → 트리거 정의 → state 표현 → 비본질 제거 → 핸들러 연결. 이 순서를 외우자. 새 컴포넌트를 만들 때마다, 자주 버그가 나는 컴포넌트를 정리할 때마다, 손에 펜을 들고 화면을 그려보자. 펜이 움직이는 5분이 일주일의 디버깅을 막아준다.

마지막으로 한 번 더. **boolean 두 개로 시작하면 거의 항상 후회한다.** 두 boolean이 모두 true인 상황이 의미가 없다면, 그건 enum이어야 했다. 이 한 줄짜리 진단을 머릿속에 품고 다음 장으로 넘어가자.

## 다음 장으로 넘어가기 전에

상태를 식별하는 안목이 조금 생겼다. 시각 상태에 이름을 붙이고, 트리거 표를 그리고, status enum으로 압축하는 흐름이 머릿속에 들어왔을 것이다. 그런데 아직 답하지 않은 질문이 하나 남았다. **그렇다면 어떤 값은 state로 두고, 어떤 값은 두지 말아야 할까?** 우리는 4단계에서 "파생 가능한 값은 state에서 빼라"고 했지만, 그 기준을 본격적으로 다루진 않았다. fullName은 정말 state여선 안 되는가? 검색 결과는? 정렬된 리스트는? `useMemo`는 또 언제 필요한가? 다음 장에서 그 경계를 그어보자. state는 컴포넌트의 기억이다. 기억할 가치가 있는 것과, 매번 다시 계산해도 되는 것을 가르는 안목이 다음 장의 주제다. 1장이 화면의 다섯 가지 모습을 정렬하는 안목이었다면, 2장은 그 다섯 가지 안에서 무엇을 기억하고 무엇을 잊을지 결정하는 안목이다. 두 안목이 합쳐질 때 컴포넌트의 골격이 정말로 단정해진다.
