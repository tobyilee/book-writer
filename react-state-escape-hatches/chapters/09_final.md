# 9장. 경계 — React 안과 밖

여기까지 잘 따라왔다면, 우리는 React 안에서 데이터가 어떻게 흐르는지를 한 번쯤은 자기 손으로 그려봤을 것이다. props가 위에서 아래로 흐르고, state는 가장 가까운 공통 부모에 모이고, 이벤트는 콜백을 따라 다시 위로 올라간다. 너무 깊어진 흐름은 컨텍스트로 가로질렀고, 그 컨텍스트가 무거워질 즈음에는 외부 store까지 손에 익혔다. 1부 마지막에서는 "그러면 어떤 도구를 언제 써야 하는가"라는 질문에 결정 트리로 답해보기도 했다.

여기까지가 React가 자기 손바닥 위에 올려놓고 굴리는 세계다.

그런데, 어느 순간 그 손바닥 밖에 있는 것들이 눈에 들어오기 시작한다. 모달이 열릴 때 첫 번째 입력란에 자동으로 포커스를 맞춰야 한다. 사용자가 검색어를 입력하다 멈추면 300밀리초 뒤에 서버에 요청을 보내야 한다. 차트 라이브러리가 jQuery 시절부터 끌고 온 명령형 API를 들고 있는데, 그놈을 React 컴포넌트와 묶어 써야 한다. WebSocket 하나가 채팅 메시지를 실시간으로 흘려보내고 있고, 우리는 그걸 받아서 화면에 띄워야 한다. 브라우저가 오프라인으로 바뀌었는지를 화면 모서리에서 알려줘야 한다.

이런 순간이 오면, 1부에서 익힌 도구만으론 좀 난감해진다. 끌어올리기로도, 컨텍스트로도, 외부 store로도 깔끔하게 풀리지 않는다. 왜 그럴까? 우리가 다루려는 대상이 더 이상 React가 아는 세계 안에 있지 않기 때문이다.

이 장에서 하고 싶은 이야기는 단순하다. **React에는 안과 밖이 있다. 그리고 그 둘 사이에는 경계가 있다.** 2부의 모든 챕터는 결국 이 경계에서 벌어지는 일을 다루게 된다. 본격적으로 도구 하나하나를 꺼내기 전에, 우리가 어떤 지형에 발을 딛고 있는지 한 번 둘러보자.

## 선언형이 잘 작동하는 영역

먼저 1부에서 우리가 무얼 했는지 짧게 회수해보자.

리스트가 있고, 검색창이 있고, 검색어가 바뀌면 리스트가 자동으로 필터된다. 우리가 적은 코드는 대충 이런 모양이었을 것이다.

```typescript
function SearchableList({ items }: { items: Item[] }) {
  const [query, setQuery] = useState('');
  const filtered = items.filter((item) =>
    item.name.toLowerCase().includes(query.toLowerCase()),
  );

  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <ul>
        {filtered.map((item) => (
          <li key={item.id}>{item.name}</li>
        ))}
      </ul>
    </>
  );
}
```

이 코드의 핵심은 우리가 "어떻게" 리스트를 갱신하는지를 적지 않았다는 점이다. DOM의 어떤 노드를 지우고, 어떤 노드를 새로 만들고, 텍스트를 어떻게 바꾸는지에 대해 우리는 한 글자도 쓰지 않았다. 단지 "지금 query가 이 값이라면, 화면은 이런 모양이어야 한다"라고 선언했을 뿐이다. 그 선언을 실제 DOM에 반영하는 일은 React가 알아서 처리한다.

이게 선언형 React가 잘하는 일이다. 입력에서 출력으로 가는 함수를 적어주면, React가 그 함수를 적절한 시점에 다시 호출해서 화면을 맞춰준다. 우리는 "최종 상태"만 묘사하면 된다.

상태가 여러 곳에 흩어져 있어도 똑같다. 끌어올리기로 한 군데 모아주면, 그 위치에서 흘러내려가는 데이터가 여러 자식의 화면을 한꺼번에 동기화한다. 컨텍스트로 가로지르면 깊이의 부담을 줄일 수 있고, 외부 store로 옮기면 selective subscription도 얻을 수 있다. 도구는 다양해도 작동 원리는 한 결이다 — **데이터가 바뀌면, React가 화면을 다시 그린다.**

여기서 잠시 멈춰서 묻고 싶다. 그렇다면 React가 잘하지 못하는 일은 뭘까?

## React가 잘하지 못하는 것들

선언형 모델이 우아하게 풀어주지 못하는 일들의 목록을 한번 적어보자.

- 사용자가 모달을 열었을 때, 모달 안의 첫 번째 입력란에 포커스를 주는 일
- 채팅 메시지가 새로 추가됐을 때, 스크롤을 가장 아래로 보내는 일
- DOM 요소의 실제 픽셀 너비를 측정해서 그에 맞춰 다른 요소의 크기를 결정하는 일
- 사용자가 입력을 멈춘 지 300밀리초가 지나면 검색을 보내는 일
- 백엔드가 보내는 WebSocket 메시지를 받아 화면에 흘려보내는 일
- 지도 라이브러리(Leaflet, Mapbox)나 차트 라이브러리(D3) 같은 비-React 위젯을 컴포넌트 안에 띄우는 일
- 브라우저가 오프라인으로 바뀌었는지 감지하는 일
- 페이지 제목(`document.title`)을 현재 화면 상태에 맞춰 바꾸는 일

이 목록을 가만히 들여다보면 공통점이 하나 보인다. **이것들은 전부 React가 직접 관리하지 않는 어떤 시스템과 우리 컴포넌트가 만나는 지점에서 벌어지는 일이다.** DOM 자체, 타이머, 네트워크 연결, 외부 라이브러리, 브라우저의 전역 상태 — 이것들은 React가 자기 손으로 만든 것도, 자기 손으로 통제하는 것도 아니다.

React가 화면을 그리는 일은 잘하지만, 화면 너머에 있는 무언가에 손을 뻗어야 하는 순간이 오면 선언형 모델은 자연스러운 답을 내놓지 못한다. "포커스를 맞춘다"라는 행위는 어떤 값으로도 표현되지 않는다. "스크롤을 내린다"도 그렇다. "지금 이 순간 한 번 일어나야 하는 명령형 동작" — 이런 것들은 "이 입력에 대해 화면이 어떻게 보여야 한다"라는 함수로는 잘 잡히지 않는다.

여기서부터가 2부의 영토다.

## 두 세계 — 안과 밖

이쯤에서 머릿속에 단순한 그림 하나를 그려두자.

원이 하나 있다. 그 원의 안쪽이 React가 아는 세계다. 컴포넌트, props, state, JSX, 그리고 React가 만들어낸 가상 DOM. 이 안에서는 모든 것이 React의 규칙을 따른다. 값이 바뀌면 렌더가 다시 일어나고, 렌더 결과는 화면에 자동으로 반영된다. 그리고 React는 자기가 만들어낸 모든 컴포넌트의 상태와 흐름을 정확히 알고 있다.

원의 바깥은 React가 모르는 세계다. 브라우저의 진짜 DOM(React가 만들어낸 가상 DOM이 아니라 그것을 반영한 결과물 자체), 타이머, 인터벌, 네트워크 연결, IndexedDB, localStorage, 차트 라이브러리 인스턴스, 지도 객체, 카메라 API, 클립보드 — 이런 것들이 거기에 있다. React는 이것들의 존재를 모르고, 이것들이 언제 변하는지도 모른다.

원의 경계 위에서 두 세계가 만난다. 그리고 그 경계는 그냥 추상적인 비유가 아니다. 우리가 짠 코드 어딘가에는 "여기서부터는 React 밖이다"라고 선을 그을 수 있는 지점이 분명히 존재한다. `inputRef.current.focus()`를 호출하는 줄, `setInterval`이 호출되는 줄, `socket.connect()`를 부르는 줄, 외부 라이브러리의 `chart.update(data)`를 호출하는 줄 — 이런 줄들이 전부 경계선이다.

문제는 이 경계가 양방향이라는 점이다.

- **React → 외부:** 우리 컴포넌트의 상태가 바뀌면, 외부 시스템도 그에 맞춰 바뀌어야 한다. 입력값이 바뀌면 검색어 디바운스 타이머도 새로 걸려야 하고, `roomId`가 바뀌면 채팅 연결도 다시 맺어져야 한다.
- **외부 → React:** 외부에서 무슨 일이 벌어지면, 그 결과가 우리 컴포넌트의 화면에 반영되어야 한다. 서버가 새 메시지를 보내면 메시지 리스트가 길어져야 하고, 브라우저가 오프라인이 되면 배너가 떠야 한다.

이 양쪽 흐름을 다루는 도구가 2부의 주제다.

## 경계에서 일어나는 일들

조금 더 구체적으로 들여다보자. 경계에서 자주 일어나는 일들을 몇 가지 분류해두면, 각 장에서 어떤 도구를 꺼낼지 미리 짐작할 수 있다.

**첫 번째, DOM 조작.** React가 만들어낸 DOM 노드 자체에 우리가 손을 뻗어야 할 때가 있다. 포커스를 옮기거나, 스크롤 위치를 조절하거나, 요소의 픽셀 크기를 측정하거나, `<video>` 엘리먼트를 재생/일시정지하는 일. 이런 일은 "어떤 값으로 표현된다"라기보다 "어느 시점에 한 번 일어나는 명령"에 가깝다.

**두 번째, 비-React 라이브러리 연동.** 지도, 차트, 에디터, 캐러셀, 이미지 크롭퍼, 비디오 플레이어. 이 라이브러리들은 거의 다 자기만의 인스턴스를 만들고, 자기만의 명령형 API(`chart.setData()`, `map.flyTo()`)를 제공한다. 그 인스턴스를 우리 React 컴포넌트의 라이프사이클과 맞춰주는 일이 필요하다.

**세 번째, 시간과 관련된 것들.** `setTimeout`, `setInterval`, `requestAnimationFrame` — 이런 타이머들은 React가 모르는 시간의 흐름이다. 우리가 시작했으니 우리가 정리해야 한다.

**네 번째, 네트워크 구독.** WebSocket, Server-Sent Events, 그리고 `fetch`를 통한 데이터 가져오기. 외부 세계에서 우리에게 흘러들어오는 데이터의 통로다.

**다섯 번째, 브라우저 API.** `navigator.onLine`, `window.matchMedia`, `IntersectionObserver`, `ResizeObserver`, 클립보드, 위치 정보. 이것들도 React가 모르는 전역 상태나 이벤트의 원천이다.

이 다섯 종류는 서로 다르게 보이지만, 한 발짝 떨어져서 보면 본질적으로 같은 모양이다. **React가 모르는 무언가가 저쪽에 있고, 우리는 그것과 동기화되어 살아가야 한다.**

## 탈출구라는 작명

여기서 React 공식 문서가 쓰는 단어를 한번 짚어두자. **Escape Hatch.** 한국어로 옮기면 "탈출구"다.

이 단어가 왜 나왔는지 잠깐 생각해보면, 거기엔 꽤 분명한 메시지가 담겨 있다. 평소엔 React 안에서 살아라. 우리가 만든 선언형 모델이 잘 작동하는 영역에 머물러라. 그러나 그 모델로는 다룰 수 없는 일이 분명히 있고, 그럴 땐 일시적으로 빠져나갈 길이 필요하다 — 그 길이 탈출구다.

탈출구를 다른 말로 풀어보면 이렇다. **선언형으로 표현하기 어려운 것을 React에 정직하게 알려주는 출입구.** 우리는 React에게 거짓말하지 않는다. "이 코드 안에서는 React가 모르는 일을 합니다, 그 일이 끝나면 정리도 해드릴게요"라고 정직하게 알려준다.

여기서 두 가지 마음가짐이 중요하다.

하나, **탈출구는 평소에 쓰는 도구가 아니다.** 1부에서 다룬 도구들 — useState, props, lifting, context — 이게 일상이고, 탈출구는 어쩔 수 없을 때 꺼내는 비상구다. 탈출구를 너무 자주 쓰고 있다면, 보통은 React 안에서 풀 수 있는 문제를 굳이 밖으로 끌고 나간 경우가 많다. (이 이야기는 12장에서 길게 한다.)

둘, **탈출구를 써야 할 때는 망설이지 않는다.** 모든 걸 선언형으로 풀려고 애쓰다 보면, 오히려 코드가 더 비뚤어지는 경우가 있다. focus를 state 변화로 우회해보겠다고 useEffect 안에 또 다른 state를 토글하는 식의, 그런 찜찜한 코드 말이다. 외부 세계와 만나야 할 때는 외부 세계와 만나는 도구를 솔직하게 쓰는 편이 낫다.

이 균형을 잡는 일이 2부 전체의 숨은 주제다.

## 두 가지 도구 — Ref와 Effect

자, 그러면 그 출입구는 구체적으로 어떻게 생겼을까. 사실 React가 우리에게 주는 경계 도구는 크게 두 개다. **Ref**와 **Effect**.

이 둘은 자주 혼동되는데, 역할이 꽤 다르다. 한 줄로 요약해보자.

- **Ref는 값을 보관하는 자리다.** 렌더링과 무관하게, 컴포넌트가 살아 있는 동안 어떤 값을 들고 있고 싶을 때 쓴다. 그 값이 바뀌어도 렌더는 다시 일어나지 않는다.
- **Effect는 외부 시스템과 동기화하는 자리다.** 컴포넌트가 어떤 상태에 있을 때, 외부 세계도 그 상태에 맞춰 있어야 한다 — 그 동기화 작업을 적는 곳이다.

다시 정리하면, ref는 "값"의 문제고 effect는 "행동"의 문제다. ref는 정적이고, effect는 동적이다. ref는 그 자리에 있고, effect는 외부와 통신한다.

같은 일을 두고 어느 도구가 맞는지 비교해보면 감이 더 잘 잡힌다. 짧은 카탈로그를 적어보자.

```typescript
// (a) state — React 안에서 화면에 영향을 준다
const [count, setCount] = useState(0);

// (b) ref — React 안에 살지만, 화면에 영향을 주지 않는 값을 보관
const timeoutRef = useRef<number | null>(null);

// (c) effect — React 밖의 시스템과 동기화한다
useEffect(() => {
  document.title = `Count: ${count}`;
}, [count]);
```

세 줄짜리 카탈로그지만, 이 세 줄에 2부의 핵심이 다 담겨 있다. (a)는 1부에서 우리가 다뤘던 영역이고, (b)와 (c)가 2부의 주인공이다.

조금 더 풀어보자. 디바운스 버튼을 만든다고 해보자. 사용자가 버튼을 빠르게 여러 번 눌러도, 마지막 클릭으로부터 300밀리초가 지난 뒤에야 실제 동작을 수행한다.

```typescript
function DebouncedButton({ onAction }: { onAction: () => void }) {
  const timeoutRef = useRef<number | null>(null);

  const handleClick = () => {
    if (timeoutRef.current !== null) {
      clearTimeout(timeoutRef.current);
    }
    timeoutRef.current = window.setTimeout(() => {
      onAction();
    }, 300);
  };

  return <button onClick={handleClick}>저장</button>;
}
```

여기서 `timeoutRef`가 들고 있는 건 `setTimeout`이 돌려준 ID 숫자다. 이 값은 화면에 그려지지도 않고, 컴포넌트 동작에 직접 영향을 주지도 않는다. 단지 다음번 클릭에서 이전 타이머를 취소하기 위해 잠시 보관해둘 뿐이다. **렌더와 무관한 값이고, 그래서 ref가 적합하다.**

state로 같은 걸 짜면 어떻게 될까? 매 클릭마다 ID가 바뀌어 setState가 호출되고, 컴포넌트는 의미 없이 다시 렌더된다. ID 자체는 화면 어디에도 보여지지 않는데도. 이건 좀 번거롭고 낭비다.

이번엔 effect 쪽 사례. 채팅방에 연결을 맺고 메시지를 받는다고 해보자.

```typescript
function ChatRoom({ roomId }: { roomId: string }) {
  useEffect(() => {
    const connection = createConnection(roomId);
    connection.connect();
    return () => {
      connection.disconnect();
    };
  }, [roomId]);

  return <h1>Welcome to {roomId}</h1>;
}
```

여기서 일어나는 일을 잘 보자. `roomId`가 'general'일 때, React 안의 상태와 React 밖의 채팅 연결이 동기화되어 있다. 우리는 'general' 방에 연결되어 있다. `roomId`가 'random'으로 바뀌는 순간, 동기화가 깨진다. cleanup 함수가 이전 연결을 끊고, 새 effect가 'random' 방에 새 연결을 맺어 다시 동기화를 맞춘다.

이게 effect의 본질이다. 라이프사이클(mount/update/unmount)이 아니라 **동기화 시작과 동기화 중지의 사이클**. 이 관점은 11장에서 더 깊이 파고든다.

## 탈출구를 너무 자주 쓰면 일어나는 일

여기서 한 가지를 못박아두자. 탈출구는 강력한 도구지만, 강력한 만큼 남용하기도 쉽다.

useEffect를 처음 손에 익힌 사람이 빠지기 쉬운 함정 하나를 들어보자. 검색창에 입력값이 들어올 때마다 그 값을 대문자로 바꾸고 싶다.

```typescript
// 안티패턴
function SearchBox() {
  const [query, setQuery] = useState('');
  const [upperQuery, setUpperQuery] = useState('');

  useEffect(() => {
    setUpperQuery(query.toUpperCase());
  }, [query]);

  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <p>{upperQuery}</p>
    </>
  );
}
```

이 코드는 동작은 한다. 하지만 어딘가 찜찜하다. `upperQuery`는 `query`로부터 100% 계산 가능한 값이다. 외부 시스템과 아무 관계도 없다. 그런데도 우리는 effect를 써서 두 state를 동기화시키고 있다. 매 입력마다 두 번의 렌더가 일어나고, 한 번 더 setState가 부른 또 한 번의 렌더가 따라온다.

이건 effect가 다룰 영역이 아니다. 렌더 중에 그냥 계산하면 끝나는 일이다.

```typescript
function SearchBox() {
  const [query, setQuery] = useState('');
  const upperQuery = query.toUpperCase();

  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <p>{upperQuery}</p>
    </>
  );
}
```

훨씬 깔끔하다. 그리고 동기화 버그도 원천적으로 사라진다. `query`가 진실의 출처고, `upperQuery`는 그것의 파생일 뿐이다.

이런 종류의 "쓸데없는 effect"가 실제 코드에 얼마나 흔한지에 대해 흥미로운 데이터가 하나 있다. Dan Abramov가 메타 내부에서 useEffect 사용 사례를 살펴봤더니, 약 절반 가까이가 사실은 effect가 필요 없는 코드였다는 이야기다. 이건 우리만의 문제가 아니라 React 커뮤니티 전체의 고질이다. (이 주제만으로 12장 한 챕터를 따로 뺐다. 안티패턴들을 하나씩 잡아낼 것이다.)

기억해두자. **Effect는 React와 외부 세계 사이의 경계용 도구다.** 도메인 로직이 effect 안으로 흘러들어갔다면, 십중팔구 잘못 쓰고 있는 것이다. 같은 이야기를 ref에도 적용할 수 있다. 렌더 결과에 영향을 줘야 하는 값을 ref에 넣는 순간, 우리는 React가 추적하지 못하는 그림자 상태를 만든 것이다. 화면이 안 맞을 때 어디를 봐야 하는지 알 수 없게 된다.

탈출구의 황금률을 한 줄로 적어두자.

> 평소엔 React 데이터 흐름 안에 머무르자. 외부 세계와 만나는 경계에서만, 정직하게 꺼내자.

## 2부의 지도

이제 우리가 어디로 가는지 그림을 미리 그려두자. 2부에서 만날 챕터들을 한 줄씩 적어보면 이렇다.

- **10장. Ref — 렌더와 무관한 값을 보관하기.** 가장 가벼운 탈출구부터 시작한다. ref가 뭔지, 언제 쓰는지, 그리고 무엇보다 **언제 쓰지 말아야 하는지**를 다룬다. DOM ref로 focus, scroll, measure를 하는 패턴, 타이머 ID나 외부 인스턴스를 보관하는 패턴, 그리고 "리렌더를 줄이려고 useState 대신 ref를 쓰자"는 흔한 유혹이 왜 함정인지 짚는다.

- **11장. Effect와 외부 시스템 동기화.** 채팅 연결, 외부 라이브러리 인스턴스, 구독·해제. 라이프사이클이 아니라 동기화의 관점으로 effect를 다시 보는 장이다. cleanup이 왜 의무인지, StrictMode가 왜 effect를 두 번 실행시키는지, 그 두 번이 왜 우리를 도와주는지 알아본다.

- **12장. Effect가 필요 없을 때.** 위에서 본 안티패턴을 본격적으로 다룬다. 파생값을 effect로 동기화하는 코드, 이벤트 핸들러로 처리할 일을 effect에 넣은 코드, 부모에게 알리려고 effect로 props를 거꾸로 흘리는 코드. 이걸 알아보고 정리하는 안목을 만든다.

- **13장. Effect의 의존성 — 닫힌 함수와 stale closure.** 의존성 배열은 hack이 아니라 동기화 계약이다. linter에 거짓말하면 어떤 버그가 따라오는지, `useEffectEvent`가 왜 등장했는지, 객체와 함수를 의존성에 넣을 때 왜 무한 루프가 도는지 다룬다.

- **14장. 커스텀 훅 — 패턴을 추출하기.** 같은 모양의 effect와 ref가 여러 컴포넌트에 흩어졌을 때, 그것을 하나의 훅으로 묶어내는 기술이다. `useOnlineStatus`, `useFetch`, `useDebounce` 같은 훅을 직접 만들어보고, 어디까지가 자기 코드로 풀 영역이고 어디부터가 TanStack Query 같은 라이브러리에 맡길 영역인지 가른다.

- **15장. 고급 API — useSyncExternalStore, useImperativeHandle, flushSync.** 평범한 effect로는 잘 처리되지 않는 영역들이다. concurrent 모드에서 안전하게 외부 store를 구독하기, 명령형 핸들을 부모에게 노출시키기, 동기 업데이트가 필요한 드문 순간에 대처하기. 평소엔 만질 일이 없지만, 만져야 할 때 모르면 곤란한 도구들이다.

이 여섯 장을 관통하는 한 줄을 다시 적어두자. **React의 안과 밖이 만나는 경계에서, 우리는 어떤 도구로 어떻게 동기화할 것인가.** 이 질문 하나가 2부 전체를 끌고 간다.

## 핵심 정리

1. React에는 안과 밖이 있다. 안은 컴포넌트·props·state로 이뤄진 선언형 세계, 밖은 DOM·타이머·네트워크·외부 라이브러리·브라우저 API로 이뤄진 명령형 세계다.
2. 선언형 모델은 "입력에서 출력으로 가는 함수"가 잘 정의될 때 우아하게 작동한다. 함수로 잡히지 않는 일들 — 한 번 일어나는 명령형 동작, 외부 시스템과의 동기화 — 에는 다른 도구가 필요하다.
3. 그 다른 도구가 **탈출구(escape hatch)**다. Ref와 Effect가 대표적인 두 출입구다.
4. **Ref는 값을 보관하는 자리.** 렌더와 무관한 값을 컴포넌트 단위로 격리해 들고 있고 싶을 때 쓴다.
5. **Effect는 외부 시스템과 동기화하는 자리.** 라이프사이클이 아니라 "동기화 시작 ↔ 동기화 중지"의 사이클로 이해하는 편이 낫다.
6. 탈출구는 자주 꺼내는 도구가 아니다. 평소엔 React 데이터 흐름 안에 머무르고, 외부와 만나는 경계에서만 정직하게 꺼낸다.
7. Effect를 자주 쓰고 있다면, 그중 절반은 사실 필요 없을 가능성이 높다. 파생값은 렌더 중에 계산하고, 사용자 동작은 이벤트 핸들러로 처리한다.
8. 2부의 챕터들은 전부 이 경계의 도구들을 하나씩 깊이 파헤치는 일이다. 각 도구가 무엇을 잘하고 무엇을 잘못하는지를 같이 본다.

## 연습 문제

**[기초] 자기 코드 진단**

자기 프로젝트에서 가장 최근에 쓴 useEffect 다섯 개를 꺼내보자. 각각에 대해 다음 두 질문에 답해보자.

1. 이 effect는 React 밖의 어떤 시스템과 동기화하고 있는가? (DOM, 타이머, 네트워크, 외부 라이브러리, 브라우저 API 중 어떤 것?)
2. 만약 답이 "딱히 없다, React 안의 값을 다른 React 안의 값으로 옮기는 중이다"라면, 그건 12장에서 다룰 안티패턴 후보일 가능성이 높다. 메모해두자.

이 작은 진단만으로도 자기 코드의 절반쯤을 다시 보게 될 수도 있다. 너무 많이 나와도 당황하지 말자. 누구나 한 번씩 거치는 길이다.

**[중] 안과 밖을 가르기**

다음 컴포넌트에서, 어느 줄이 React 안의 일이고 어느 줄이 React 밖과 만나는 일인지 표시해보자. 어디가 경계선인가?

```typescript
function ProfileEditor({ userId }: { userId: string }) {
  const [name, setName] = useState('');
  const inputRef = useRef<HTMLInputElement>(null);

  useEffect(() => {
    inputRef.current?.focus();
  }, [userId]);

  useEffect(() => {
    document.title = `${name} 편집 중`;
  }, [name]);

  const handleSave = () => {
    fetch(`/api/users/${userId}`, {
      method: 'PUT',
      body: JSON.stringify({ name }),
    });
  };

  return (
    <>
      <input
        ref={inputRef}
        value={name}
        onChange={(e) => setName(e.target.value)}
      />
      <button onClick={handleSave}>저장</button>
    </>
  );
}
```

힌트: state·setState·JSX는 안이다. `inputRef.current.focus()`, `document.title`, `fetch` — 이 셋은 어디일까? 그리고 이들 중 어떤 것이 ref와 관련 있고, 어떤 것이 effect와 관련 있는지도 같이 생각해보자.

**[심화] 시나리오 분류**

다음 네 시나리오를 보고, 각각이 (a) React 데이터 흐름 안에서 풀 일인지, (b) ref가 필요한 경계 작업인지, (c) effect로 외부와 동기화할 일인지 분류해보자.

1. 사용자가 "이름" 입력란에 입력하면, 그 아래 "안녕하세요, OO님" 인사말이 실시간으로 바뀌어야 한다.
2. 모달이 열리면, 모달 안 첫 번째 입력란에 자동으로 포커스가 가야 한다.
3. 컴포넌트가 화면에 보이는 동안에는 1초마다 서버 시계와 동기화하고, 사라지면 멈춘다.
4. 두 자식 컴포넌트가 동일한 카운터 값을 공유해야 한다.

답이 한눈에 떠오르지 않는다면, 그 망설임을 메모해두자. 망설였던 시나리오가 다음 몇 장에 걸쳐 다시 등장하게 된다.

## 마무리

이 장은 짧다. 도구를 가르치지도 않았고, 코드도 거의 만들지 않았다. 단지 우리가 앞으로 6개 챕터에 걸쳐 어디를 거니는지 지도를 한 장 펴봤을 뿐이다.

기억해두자. 2부에서 무얼 배우든, 그 도구는 항상 **React 안과 밖이 만나는 경계에서 쓰는 물건**이다. 안에서 풀 수 있는 문제를 굳이 밖으로 끌고 나가는 게 아니라, 밖에 있는 진짜 외부 시스템과 정직하게 만나기 위한 도구다. 이 한 줄을 챕터마다 한 번씩 떠올려주면, 어디서 길을 잃어도 다시 돌아올 좌표가 된다.

자, 그러면 가장 가벼운 탈출구부터 살펴보자. 우리가 처음으로 만날 친구는 ref다. 다음 장에서, 디바운스 버튼과 자동 포커스 입력란을 함께 만들어보면서 ref가 정확히 언제 어떻게 쓰이는지 — 그리고 어떤 유혹에 흔들리지 말아야 하는지 — 차분히 풀어가자.
