## A. 책 제목 후보 (3개)

### 후보 1 — **"리액트, 상태와 경계"**
- **부제:** 흔들리지 않는 멘탈 모델로 다시 쓰는 React
- **슬러그:** `react-state-and-boundaries`
- **포지셔닝:** 철학적 에세이형 기술서. 상태(state)와 경계(escape hatch)라는 이중 축을 책 전체의 메타포로 끌고 가는 톤.
- **타깃 통증:** "useEffect를 어디까지 써야 하는지 감이 안 잡힌다", "ref와 state 중에 뭘 골라야 할지 늘 흔들린다"
- **장점/리스크:** 톤이 우아하고 기억에 남음 / 입문자가 표지만 보고 "에세이인가?" 오해할 위험.

### 후보 2 — **"리액트 상태 관리, 다시 배우기"**
- **부제:** useState부터 useEffect, ref까지 — 멘탈 모델로 풀어내는 중급 가이드
- **슬러그:** `relearn-react-state`
- **포지셔닝:** 정통 학습형 기술서. 검색 친화적이고 서점 매대에서 이해하기 쉬운 제목.
- **타깃 통증:** "기본은 안다고 생각했는데 실무에서 자꾸 같은 곳에서 막힌다", "공식 문서를 다 봤는데도 머릿속에서 정리가 안 된다"
- **장점/리스크:** 시장에서 즉시 통하는 명확한 포지셔닝 / 차별화 포인트가 약함, 흔한 제목.

### 후보 3 — **"useEffect는 라이프사이클이 아니다"**
- **부제:** 리액트 상태와 탈출구를 다시 그리는 멘탈 모델
- **슬러그:** `useeffect-is-not-lifecycle`
- **포지셔닝:** 도발적이고 주장이 분명한 입장의 책. 잘못된 멘탈 모델을 고치는 것을 명시적으로 약속.
- **타깃 통증:** "componentDidMount 시절의 사고방식을 못 버려서 매번 무한 루프를 만든다"
- **장점/리스크:** SNS에서 화제가 될 강력한 카피, 메타 메시지가 분명 / 제목이 너무 좁아 보일 수 있음(state 관리 절반이 가려짐).

### 추천 — **후보 1: "리액트, 상태와 경계"**
이유:
- 책 구조가 "1부 상태(state)"와 "2부 경계(escape hatches)"의 두 기둥으로 명확하다. 제목이 책 구조를 바로 그려준다.
- Toby의 책 시리즈와 톤이 잘 맞는다(짧고 묵직한 한국어 명사구 + 부제로 보강).
- 부제 "흔들리지 않는 멘탈 모델"이 정확히 이 책이 약속하는 출구 상태다.
- 후보 3은 강력하지만 절반의 주제(상태 관리)를 가리고, 후보 2는 차별화가 약하다.

### 코드 예제 정책
- **기본 언어: TypeScript.** 독자가 TS에 익숙하다는 전제를 활용한다. `useReducer`의 액션 유니온 타이핑, `useRef<HTMLInputElement | null>`, `Dispatch<Action>` 등을 자연스럽게 등장시킨다.
- **JS-then-TS 룰 (통합):** 새로운 **개념을 처음 도입하는 예제**는 JS로 한 번 보여주고 같은 자리에서 TS 본격 버전을 옆에 둔다. 그 뒤로는 모두 TS. 한 챕터 안에서 같은 개념을 두 번 도입하지 않으므로 JS-then-TS는 챕터당 1~2회로 제한된다.
- **실행 가능한 스니펫:** 모든 챕터의 핵심 예제는 그대로 복붙해 React 19 + Vite 환경에서 돌아가도록 설계. 챕터 깃 저장소(부록 안내)와 Sandbox 링크로 보강.
- **연습 문제:** 챕터당 3~5개. 시작 코드 + "이렇게 풀어보자" 힌트 + 정답 한 개 이상의 풀이.
- **React 19 기준:** ref가 prop으로 직접 전달되는 새 규약, `flushSync`, `useSyncExternalStore` 등을 모두 React 19 기준으로 설명. `useEffectEvent`는 챕터 12에서 명시적으로 "실험적 API" 박스로 표시.

### 표기 규약
- **외래어 표기:** 한 번 한글-원어 병기 후 이후엔 통일. 예: 처음에 "상태(state)"로 쓰고 그 뒤로는 문맥에 따라 "상태" 또는 "state". Hook은 "훅"으로 통일하되, 코드 식별자(`useState`, `useEffect`)는 영문 그대로.
- **인용 처리:** 외부 관점은 출처를 본문 안에 자연스럽게 녹임 — "Dan Abramov는 ~라고 적었다", "Kent C. Dodds의 표현을 빌리면 ~". 각주는 권말 참고문헌으로 통합.
- **코드 블록 룰:**
  - 전체 컴포넌트는 30~40줄 안팎으로 제한. 그보다 길면 두 단계로 쪼개서 보여준다.
  - 문제의 코드 → 진단 → 고친 코드, 항상 세 박자로 보인다 (Toby 스타일).
  - "이 줄을 주목해보자" 같은 안내 문장으로 코드 블록을 둘러싼다.
- **박스/콜아웃:** "함정", "한 걸음 더", "TS 노트" 세 종류만 사용한다. 남발하지 않는다.

### 0장. 출발점 점검 — 우리가 안다고 믿는 것
- **핵심 질문:** 지금 이 순간 내 머릿속의 React 모델은 어디까지 정확한가?
- **학습 목표:**
  - 자주 틀리는 여덟 가지 멘탈 모델 함정을 미리 본다
  - 자가 진단 결과로 자신이 책의 어느 챕터에서 가장 큰 보강을 받게 될지 예측한다
- **자가 진단 8문항(확정):**
  1. **`'idle' | 'loading' | 'error' | 'success'` 같은 상태 enum이 두 boolean보다 나은 이유는 무엇인가?** (정답 단서: 모순 가능 상태 제거. → 1·2장)
  2. **`setState(state + 1)`을 같은 핸들러에서 세 번 부르면 state는 얼마나 증가하는가? 동기인가 비동기인가?** (정답 단서: 같은 렌더의 스냅샷을 읽으니 1만 증가. 함수형 setState로 누적해야 3 증가. → 1·12장)
  3. **같은 자리에 같은 타입의 컴포넌트가 들어왔지만 "다른 사용자"라면 state는 어떻게 되는가?** (정답 단서: 보존된다. 의도적 리셋엔 `key`. → 4장)
  4. **`useEffect(() => { fetch(...) }, [])`로 mount 시 1회 fetch하는 코드의 위험은?** (정답 단서: cleanup 없음 → race condition, 더블 인보케이션. → 10·11장)
  5. **`useRef`로 입력값을 보관하면 `useState`보다 "빠른가"?** (정답 단서: 리렌더는 없지만 그 자체가 "빠름"의 정의가 아니다. 동적 뷰엔 부적절. → 9장)
  6. **Context의 value가 바뀔 때 모든 consumer가 리렌더되는 조건은 무엇인가?** (정답 단서: value 참조 동일성 기준. selective subscription 없음. → 7·8장)
  7. **의존성 배열에 객체 `{ serverUrl, roomId }`를 넣으면 무엇이 일어나는가?** (정답 단서: 매 렌더 새 참조 → 재실행 폭주. 원시값 분해. → 12장)
  8. **`useEffectEvent`는 dependency 배열에 넣어야 하는가?** (정답 단서: 비반응형이라 안 넣는다. lint silencer로 쓰지 마라. → 12장)
- **주요 내용:**

  - 각 문항의 정답을 짧게 풀고, 자세한 설명은 매핑된 챕터로 미룬다
  - "지금 몇 개 맞췄는가"보다 "어느 영역에서 가장 흔들리는가"를 자가 보고하게 함

## 1부 — 상태를 다루는 법 (8개 챕터)

### 1장. UI는 상태의 함수다 — 다섯 단계로 상태를 설계하기
- **핵심 질문:** 명령형으로 disable/show/hide를 토글하다 폭발하는 코드를, 어떻게 처음부터 선언형으로 그릴 수 있을까?
- **학습 목표:**
  - 시각 상태(visual states)를 명사로 식별한다
  - 상태 변경의 트리거를 사람의 의도와 시스템의 응답으로 분리한다
  - 최소한의 state로 모든 시각 상태를 표현한다
  - 다섯 단계 절차를 자기 코드에 적용할 수 있다
- **주요 내용:**
  1. 디자이너가 화면을 먼저 그리는 방식을 빌려온다
  2. 명령형 vs 선언형: "어떻게 바꿀지" vs "각 상태에서 어떤 모습이어야 하는지"
  3. 상태 식별 → 트리거 정의 → state로 표현 → 비본질 제거 → 핸들러 연결
  4. 퀴즈 폼 예제: status 한 변수로 typing/submitting/success 표현
  5. "왜 boolean 두 개로 시작하면 망하는가"
  6. 다섯 단계를 체크리스트로 정리
- **사용할 코드 예제:** 패턴 3.1 (다섯 단계 모델링) — 퀴즈 폼을 단계별로 만들어가는 형태로 변형
- **다룰 안티패턴:** #3 (사용자 이벤트를 effect에) — 가볍게 살짝, 본격 논의는 11장
- **연습 문제:**
  1. 시드 1번 (`'idle' | 'loading' | 'error' | 'success'`로 페치 UI 모델링)
  2. (신규) 신호등 세 boolean → 단일 `status` enum 리팩토링
  3. (신규) 모달 컴포넌트의 시각 상태를 종이 위에 그려본 뒤 state 변수로 옮기기
- **분량:** 약 7,500 단어
- **다음 챕터로의 다리:** "상태를 식별하는 안목이 생겼으니, 이제 어떤 값을 state로 두고 어떤 값은 두지 말아야 하는지 보자."

### 2장. 무엇을 state에 두고, 무엇을 두지 말 것인가
- **핵심 질문:** 어떤 값이 state여야 하고, 어떤 값은 그저 렌더 중에 계산되어야 할까?
- **학습 목표:**
  - 파생값을 state로 보관하지 않는다
  - 모순 가능한 state 모양을 상태 enum으로 합친다
  - 객체 자체 대신 ID/index를 저장한다
  - **(정규화 깊이 다루기는 14장 통합 시나리오로 이전됨)**
- **주요 내용:**
  1. "Single Source of Truth" — 같은 진실을 두 군데 두지 말자
  2. fullName 예제: state로 둘 때와 파생값으로 둘 때
  3. `useMemo`는 언제 필요한가 (성능 측정 후)
  4. `isSending` + `isSent` 두 boolean의 비극 → status enum
  5. 선택된 객체를 state로 두면 stale 참조가 생기는 이유 → ID/Index 저장
  6. **(이전됨)** ~~트리/배열 정규화~~ → 14장 통합 시나리오에서 본격 등장. 본 장에서는 "깊은 트리는 평탄화 가능하다"는 한 단락 예고만.
- **사용할 코드 예제:** 패턴 3.2 (파생값), 3.3 (status enum), 3.4 (ID/Index)
- **다룰 안티패턴:** #1 (파생값을 state로), #2 (effect로 비싼 계산), #10 (prop 미러링)
- **연습 문제:** (정규화 시드 19번은 14장으로 이전)
  1. 시드 2번 (firstName/lastName/fullName 리팩토링)
  2. (신규) 상품 리스트 + 선택된 상품을 ID로 바꾸기
  3. 시드 11번 (`Set<id>` 다중 선택)
- **분량:** 약 6,500 단어 (정규화 분리로 7,500 → 6,500)
- **이전 연결:** 1장의 다섯 단계 중 "비본질 제거" 단계를 자세히 푸는 챕터
- **다음 챕터로의 다리:** "상태의 모양이 깔끔해졌으면, 이제 그것을 어디에 둘지가 다음 문제다."

### 3장. 상태를 어디에 둘 것인가 — 끌어올림과 콜로케이션
- **핵심 질문:** 두 컴포넌트가 같은 정보를 알아야 할 때, 그 정보는 어디에 살아야 하는가?
- **학습 목표:**
  - "Lifting State Up"의 정확한 절차를 안다
  - Controlled vs Uncontrolled 구분의 실제 의미를 안다
  - "필요 없으면 다시 내린다(drop)"는 원칙을 적용한다
- **주요 내용:**
  1. 아코디언 예제: 두 패널이 서로 알아야 하는 순간
  2. 끌어올리기 절차 — state를 부모로, props로 다시 내림
  3. Controlled / Uncontrolled의 진짜 의미: 누가 진실의 소유자인가
  4. Kent C. Dodds의 "Lift, then Drop" — 콜로케이션이 왜 중요한가
  5. 동기화된 인풋 예제 / 필터된 리스트 예제
  6. "끌어올리지 않고도 같은 결과"의 대안 — children 합성 (5장 예고)
- **사용할 코드 예제:** 패턴 3.6 (아코디언, 동기화 입력, 필터링)
- **다룰 안티패턴:** #18 (모든 state를 위로 올리는 만행)
- **연습 문제:**
  1. 시드 3번 (Counter 두 개 동기화)
  2. 시드 4번 (아코디언 한 패널만 열림)
  3. (신규) 검색창 + 결과 리스트 — query만 끌어올리고 결과는 파생
- **분량:** 약 6,500 단어
- **이전 연결:** 2장 ID 저장 패턴이 여기서 부모-자식 props로 재등장
- **다음 챕터로의 다리:** "끌어올리는 법은 알았으니, 반대로 'state를 강제로 처음으로 되돌리는 법'도 알아야 한다."

### 4장. 보존과 리셋 — `key`가 정말로 하는 일
- **핵심 질문:** 같은 컴포넌트 자리에 다른 사람이 들어왔을 때, React는 어떻게 그것을 "다른 컴포넌트"라고 인식하는가?
- **학습 목표:**
  - "트리 위치"가 React의 정체성 기준임을 이해한다
  - 의도적으로 state를 리셋하기 위한 `key` 사용법을 안다
  - prop이 바뀔 때 일부만 리셋하는 "렌더 중 set" 패턴을 안다
  - **(중요)** "prop 변화에 effect로 반응해 state를 리셋하면 안 된다"의 본격 처방을 4장에서 끝낸다 — 11장 회수에서는 짧은 박스로 다시 인용만.
- **주요 내용:**
  1. 같은 자리·같은 타입이면 state 보존, 다른 타입이면 파괴
  2. 채팅 상대를 바꿔도 초고가 남는 버그 → `<Chat key={contact.id}>`
  3. `key`로 array index를 쓰면 안 되는 이유 (안티패턴 #21)
  4. 리스트 렌더에서의 stable id의 무게
  5. prop이 바뀔 때 일부 state만 초기화: `if (x !== prevX) setX(x); setSelection(null);`
  6. "왜 effect로 리셋하면 안 되는가" — 한 박자 늦은 깜빡임 (안티패턴 #3 본격 처방)
  7. "컴포넌트 안에서 컴포넌트 정의"가 모든 state를 파괴하는 이유 (안티패턴 #11)
- **사용할 코드 예제:** 패턴 3.7 (key로 리셋, 렌더 중 set)
- **다룰 안티패턴:** #3 (effect로 prop 변화에 반응) — **본격 처방 완결**, #11 (안에서 컴포넌트 정의), #21 (index를 key로)
- **연습 문제:**
  1. 시드 8번 (채팅 상대 바꾸면 초고 리셋)
  2. (신규) 같은 모달이 다른 user에 대해 열릴 때 내부 state 리셋
  3. (신규) 정렬 가능한 리스트에서 잘못된 key 진단하기
- **분량:** 약 6,500 단어
- **이전 연결:** 3장의 "정보의 소유자" 개념이 "정체성의 기준"으로 확장됨
- **다음 챕터로의 다리:** "여러 곳에 흩어진 setState를 한 곳으로 모으고 싶다면, 다음 도구가 필요하다."

### 5장. 흩어진 setState를 한 자리에 모으고 싶을 때 — useReducer
- **핵심 질문:** 이벤트 핸들러마다 흩어진 갱신 로직을 어떻게 한 곳으로 모으고, 디버그할 수 있을까?
- **학습 목표:**
  - useState와 useReducer의 경계선을 안다
  - 액션을 "사용자 의도" 단위로 작성한다
  - reducer + TypeScript 액션 유니온 타이핑을 한다
  - "useReducer is the cheat mode of Hooks"의 의미를 안다
- **주요 내용:**
  1. Task list가 커질수록 setTasks(...)가 늘어나는 코드 냄새
  2. reducer는 `(state, action) => newState` 순수 함수
  3. 사용자 의도 단위 액션 — `'tasks/added'`, `'tasks/edited'` (잘게 쪼개지 말 것)
  4. TS 액션 유니온 + discriminated union으로 안전하게 만들기
  5. 함수형 update와의 관계 — reducer는 그 일반화
  6. `useImmerReducer`로 mutate 스타일 (선택)
  7. "useReducer가 효과를 보는 임계점" — 작은 폼은 useState가 낫다
- **사용할 코드 예제:** 패턴 3.8 (Tasks reducer)
- **다룰 안티패턴:** 액션을 잘게 쪼개 useState만도 못한 reducer (논쟁 5.5)
- **연습 문제:**
  1. 시드 9번 (Task List를 useState→useReducer로 마이그레이션)
  2. 시드 18번 (3단계 폼 위저드, 도전 과제)
  3. (신규) reducer에 TypeScript 액션 타입 붙이기
- **분량:** 약 7,000 단어
- **이전 연결:** 1장의 status enum이 reducer의 state shape으로 자연스럽게 진화
- **다음 챕터로의 다리:** "이제 깊은 트리에 그것을 어떻게 흘려보낼지를 보자."

### 6장. Context — prop drilling이 진짜 고통일 때
- **핵심 질문:** Context는 언제 필요하고, 언제 너무 빠른 선택인가?
- **학습 목표:**
  - prop drilling의 진짜 비용을 측정한다
  - Context의 의미와 Provider 콜로케이션을 안다
  - "Don't reach for context too soon"의 기준을 갖는다
- **주요 내용:**
  1. Section → Post → Heading 6단계 prop drilling의 비극
  2. `LevelContext`로 자기 위 level + 1 자동 계산
  3. Context의 비용: value가 바뀌면 모든 consumer 리렌더
  4. Provider를 트리 깊숙이 두어 영향 범위 제한 (콜로케이션)
  5. Context를 쓸 좋은 후보 / 나쁜 후보 (테마 vs 자주 바뀌는 폼 값)
  6. children 합성으로 Context 없이 푸는 길 — 1순위 시도
- **사용할 코드 예제:** 패턴 3.9 (LevelContext), 합성 대안
- **다룰 안티패턴:** Context 너무 일찍 쓰기 (Kent Dodds 관점)
- **연습 문제:**
  1. 시드 12번 (LevelContext 구현)
  2. (신규) 같은 문제를 children 합성으로 다시 푸는 비교 연습
  3. (신규) Context를 도입하는 것보다 props가 나은 한 가지 사례 만들기
- **분량:** 약 6,500 단어
- **이전 연결:** 3장의 "끌어올리고 다시 내림" 원칙이 Context 도입 결정의 기준이 됨
- **다음 챕터로의 다리:** "reducer와 context를 합치면, 큰 앱에서도 흔들리지 않는 상태 구조가 만들어진다."

### 7장. 앱이 커져서 상태가 발에 채일 때 — Reducer + Context
- **핵심 질문:** 앱이 커져도 흩어지지 않는 상태 구조를 어떻게 짤까?
- **학습 목표:**
  - state context와 dispatch context를 분리한다
  - `useTasks()`/`useTasksDispatch()` 같은 커스텀 훅으로 노출한다
  - 이 패턴의 한계 — selective subscription 부재 — 를 다음 챕터로 넘긴다
- **주요 내용:**
  1. reducer를 깊은 자식까지 보내는 두 가지 부담 (state, dispatch)
  2. 두 Context로 분리 → 자주 안 바뀌는 dispatch는 별도 Provider
  3. `TasksProvider` + 커스텀 훅 노출 패턴
  4. value 참조 안정화 — `useMemo`로 state 객체 묶기, dispatch는 reducer가 안정 참조 보장
  5. **한계 인식:** Context는 selective subscription이 없다. value가 바뀌면 모든 consumer가 리렌더된다 — 8장으로 다리
  6. (이전됨) ~~Zustand/Redux 결정 트리~~ → **신설 8장으로 본격 분리**
  7. (이전됨) ~~Server state 단락~~ → **신설 8장으로 본격 분리**
- **사용할 코드 예제:** 패턴 3.10 (TasksContext + TasksDispatchContext + 커스텀 훅)
- **다룰 안티패턴:** #18 (모든 걸 한 Context에)
- **연습 문제:**
  1. 시드 13번 (Tasks 앱 reducer + 두 context)
  2. (신규) 자주 바뀌는 값과 거의 안 바뀌는 값을 두 Context로 분리하기
  3. (신규) 같은 Provider value를 매 렌더 새 객체로 만들었을 때의 부작용 진단
- **분량:** 약 6,000 단어 (도구 선택·Server state 분리로 7,500 → 6,000)
- **이전 연결:** 5장 reducer + 6장 Context의 결합편
- **다음 챕터로의 다리:** "이 구조로도 부족할 때가 있다 — 자주 바뀌는 값이 큰 앱 전체에 흐를 때다. 그래서 도구 선택의 문제가 따로 등장한다."

### 8장(신설). 도구 선택 — Context, Zustand, Redux, Jotai, 그리고 Server State
- **핵심 질문:** Context로 충분한 곳, Zustand가 필요한 곳, Redux Toolkit이 등장하는 자리, 그리고 "이건 UI state가 아니다"라고 깨닫는 자리는 어디인가?
- **학습 목표:**
  - Context의 selective subscription 부재가 만드는 비용을 안다
  - Zustand / Jotai / Redux Toolkit의 자리를 한 페이지 결정 트리로 가진다
  - **Server State는 별개**라는 메시지를 내재화한다 — TanStack Query/SWR이 등장하는 이유
  - "React IS your state management library"와 "그러나 server state는 다른 도구가 필요하다"가 동시에 참인 이유를 설명할 수 있다
- **주요 내용:**
  1. 7장의 한계로 시작: value가 자주 바뀌는 거대 Context는 트리 전체를 깨운다
  2. Zustand의 selective subscription 메커니즘 — `useStore(s => s.count)` 한 줄
  3. **2026 시점 현실 결정 트리(한 페이지):**
     - 안 변하는 글로벌 (theme, locale, user) → Context
     - 작은 앱의 자주 변하는 클라이언트 상태 → Zustand
     - 팀 표준·시간 여행 디버깅이 필요한 대규모 → Redux Toolkit
     - 원자 단위 의존성·파생 그래프 → Jotai
  4. Kent C. Dodds 인용 — "React IS your state management library", "Don't reach for it too soon"
  5. **Server State 본격 분리:**
     - 비동기·공유·외부 변경·캐시 무효화 — UI state와 다른 네 가지 성질
     - TanStack Query(13.4KB) vs SWR(4.2KB) 비교 한 페이지
     - "Redux에 fetch 결과를 다 때려박는다 = query cache를 reinvent한다 = 거의 항상 깨진다"
  6. 안티패턴 본격 처방: #18(모든 걸 Context에 몰아넣기), #19(서버 캐시를 useState/Context로)
  7. **이 책이 라이브러리 사용법 책이 아닌 이유** — 도구는 결정 트리로, 사용법은 각 라이브러리 공식 문서로
- **사용할 코드 예제:**
  - 7장의 TasksContext가 자주 바뀌는 필드 때문에 큰 트리를 다 깨우는 모습
  - 같은 문제를 Zustand로 푼 4줄 비교
  - "이걸 useState로 들고 있다가 망가지는 fetch 코드" → TanStack Query 단 8줄로 대체되는 비교
- **다룰 안티패턴:** #18, #19 — 본격 처방 완결
- **연습 문제:**
  1. (신규) 7장 TasksContext에 "마지막 수정 시각"을 추가했을 때 모든 컨슈머가 리렌더되는 현상 재현
  2. (신규) 같은 앱의 상태를 (a) Context, (b) Zustand로 두 번 짜고 비교
  3. (신규) 본인 사이드 프로젝트에 가상의 결정 트리 적용 — Server state는 무엇이고 UI state는 무엇인가
- **분량:** 약 6,000 단어
- **이전 연결:** 7장의 "selective subscription 부재" 한계가 이 챕터의 시작점
- **1부 → 2부 다리:** "여기까지가 React 안에서 데이터가 흐르는 세계다. 이제부터 우리는 React가 모르는 것들 — 외부 시스템 — 과 만나야 한다. 마침 우리가 막 본 server state도 결국 외부 세계의 한 종류였다. 그 경계를 다루는 도구가 2부의 주제다."

### 9장. 경계 — React 안과 밖 (전환 챕터)
- **핵심 질문:** React가 직접 관리하지 않는 것들과는 어떻게 만나야 하는가?
- **학습 목표:**
  - "escape hatch"라는 단어를 받아들인다
  - React 안의 데이터 흐름과 밖의 세계(브라우저 API, 비-React 라이브러리, 타이머 ID, 외부 store)를 구분한다
  - 2부 전체의 지도를 머릿속에 그린다
- **주요 내용:**
  1. 1부 요약: 상태 관리는 "내부 진실"의 문제였다 — 그 위에 도구 선택의 문제가 얹혔다(8장)
  2. 외부 세계: DOM, 네트워크, 타이머, WebSocket, third-party 위젯, 브라우저 API(online/offline)
  3. 두 가지 경계 도구: ref(값 보관·DOM 접근), effect(외부 시스템 동기화)
  4. "평소엔 React 데이터 흐름 안에 머문다, 경계에서만 꺼낸다" — Escape hatch라는 작명의 의미
  5. 2부 전체 지도: ref(10장) → effect 동기화·라이프사이클(11장) → "Effect가 필요 없을 때"(12장) → 의존성(13장) → 커스텀 훅(14장) → 고급 API(15장)
- **사용할 코드 예제:** 짧은 비교 — 같은 기능을 (a) state로, (b) ref로, (c) effect로 풀었을 때 각각 어떻게 다른지 한 페이지 카탈로그
- **연습 문제:**
  1. (신규) 주어진 4개 시나리오를 "react 내부 / 경계" 둘로 분류
  2. (신규) 본인 코드의 useEffect 5개를 "정말 외부 동기화인가" 진단 (12장 예고)
- **분량:** 약 5,000 단어 (전환 챕터라 짧게)

- **다음 챕터로의 다리:** "가장 가벼운 escape hatch부터 살펴보자 — ref다."

### 10장. Ref — 렌더와 무관한 값을 보관하기
- **핵심 질문:** state도 effect도 아닌 ref는, 정확히 언제 꺼내야 하는가?
- **학습 목표:**
  - useRef와 useState의 결정 기준을 안다
  - DOM ref로 focus/scroll/measure 한다
  - 타이머 ID, 이전 값, 외부 라이브러리 인스턴스를 ref에 보관한다
  - 렌더 중에 ref를 읽지 않는다
  - **(P1-3 반영)** ref가 "useState보다 빠르다"는 정량 비교가 거짓이 되는 경우를 안다
- **주요 내용:**
  1. ref의 정의: `{ current: ... }`, 변경해도 리렌더 없음
  2. 모듈 스코프 변수 vs ref — 컴포넌트마다 격리되는 이유
  3. 디바운스 버튼 예제 (`timeoutRef`)
  4. DOM ref: `<input ref={inputRef} />`, React 19에서는 prop처럼 직접
  5. 동적 리스트의 ref callback + Map
  6. 측정·스크롤·focus 시나리오
  7. **Ref의 정당한 사용 vs 회피 영역** (논쟁 5.4)
     - 정당: focus/scroll/measure, 비-React 위젯, 타이머/연결 ID, 이전 값
     - 회피: 렌더 결과에 영향 주는 값을 ref에, 리렌더 줄이려고 useState 대신 useRef
     - **함정 박스(P1-3):** 한국 커뮤니티의 21회→2회 렌더 정량 비교는 정적 입력 폼 한정. **동적 검색 제안처럼 입력값이 즉시 화면에 반영되어야 하는 경우엔 ref가 답이 아니다.** "리렌더 줄이기"가 목적이 되면 안 된다.
     - **한 걸음 더(P1-5):** "ref는 의존성 배열 폭주를 막는 또 다른 도구이기도 하다. 13장에서 다시 만난다."
- **사용할 코드 예제:** 패턴 3.11 (디바운스 ref), 3.12 (DOM ref, 동적 ref callback)
- **다룰 안티패턴:** #13 (렌더 중 ref 읽기), #14 (React가 관리하는 DOM 직접 조작)
- **연습 문제:**
  1. 시드 5번 (자동 focus)
  2. 시드 7번 (디바운스 ref 격리)
  3. (신규) 입력 폼 — useState로 풀고, ref로도 풀고, **동적 검증 메시지 표시 요구가 추가됐을 때** 어느 쪽이 맞는지 직접 결정
- **분량:** 약 7,000 단어
- **이전 연결:** 2장의 "단일 진실 출처" 원칙이 ref에서도 살아있음을 짚어준다
- **다음 챕터로의 다리:** "ref가 '값의 컨테이너'였다면, 다음 도구는 '동작의 사이클'을 다룬다."

### 11장(확장). Effect — 동기화의 사이클과 Reactive Value
- **핵심 질문:** useEffect는 mount/update/unmount의 라이프사이클인가, 아니면 다른 무엇인가? 그리고 컴포넌트 라이프사이클과 Effect 라이프사이클은 어떻게 다른가?
- **학습 목표:**
  - Effect를 "외부 시스템과의 동기화"로 다시 정의한다
  - **Effect의 라이프사이클이 컴포넌트의 라이프사이클과 다르다**는 사실을 받아들인다 (P0-1)
  - **"Reactive Value"의 정의를 안다** — 컴포넌트 본체에서 선언된 모든 값(props/state/파생값) (P0-1)
  - cleanup의 의무성과 의미를 안다
  - 개발 모드 더블 인보케이션의 의도를 이해한다
  - race condition 방지 패턴을 손에 익힌다
- **주요 내용:**
  1. componentDidMount 사고방식의 잔재 — 한국 커뮤니티 통증의 출발점
  2. Dan Abramov 인용: "It's all about the destination, not the journey"
  3. 채팅 연결 예제 1단계: roomId가 바뀌면 끊고 다시 연결 — synchronization 사이클
  4. cleanup이 의무인 이유 — 언마운트만이 아니라 "다음 동기화 시작 전"
  5. **(신규, P0-1) Effect 라이프사이클 ≠ 컴포넌트 라이프사이클**
     - 컴포넌트는 한 번 마운트되고 한 번 언마운트되지만, Effect는 그 사이에 여러 번 시작·중지된다
     - 채팅 연결 예제 2단계: roomId·serverUrl 두 reactive value로 확장 — 둘 중 어느 하나가 바뀌어도 다시 동기화된다
     - "각 Effect는 자기만의 시작·중지 사이클을 가진다"
  6. **(신규, P0-1) Reactive Value 정의**
     - 컴포넌트 본체에서 선언된 모든 값 = reactive
     - 외부 상수, ref.current, 모듈 스코프 변수 = non-reactive
     - 의존성 배열은 "이 Effect가 어떤 reactive value들과 동기화되어야 하는가"의 선언
  7. 개발 모드 두 번 실행이 검증하는 것: cleanup의 정확성 + 라이프사이클 안전성
  8. fetch + race condition 방지 (`let ignore = false`)
  9. 실무에선 TanStack Query/SWR로 옮긴다는 단서 (8장 회수)
  10. "Reactive Value" 개념이 다음 두 챕터(12장 효과 빼기, 13장 의존성 정직성)의 토대임을 명시
- **사용할 코드 예제:** 패턴 3.13 (chat connection, 두 reactive value 확장 버전), 3.14 (race condition)
- **다룰 안티패턴:** #16 (cleanup 없는 fetch), #17 (외부 store를 effect로 구독 — 15장 예고)
- **연습 문제:**
  1. 시드 6번 (setInterval cleanup 누락 고치기)
  2. 시드 14번 (chat connection cleanup + 더블 인보케이션 분석)
  3. 시드 15번 (fetch + ignore 플래그)
  4. (신규, P0-1) 주어진 컴포넌트 코드에서 reactive value와 non-reactive value를 모두 분류
- **분량:** 약 9,500 단어 (P0-1 반영, 8,000 → 9,500)
- **이전 연결:** 10장 ref와의 차이 — "값"이 아니라 "사이클"을 다룬다
- **다음 챕터로의 다리:** "그런데, 우리가 effect를 너무 많이 쓰고 있는지도 모른다."

### 12장. Effect가 필요 없을 때 — 가장 자주 쓰는 안티패턴들
- **핵심 질문:** 메타 내부 표본의 약 46%가 불필요한 useEffect였다 — 우리도 그럴까?
- **학습 목표:**
  - 렌더 중에 계산 가능한 값은 effect로 동기화하지 않는다
  - 사용자 이벤트로 일어나는 일은 이벤트 핸들러로 처리한다
  - cascading effects의 무한 루프 패턴을 즉시 진단한다
  - "초기화"는 effect의 일이 아니다
- **챕터 구조 (P0-2 반영):**
  - **도입부 (약 1,500단어, 절정 메시지 분리):** 46% 통계 + "이런 가이드가 길다는 것 자체가 API 설계의 흠인가?"라는 비판과 옹호의 양쪽 인용 + 저자 입장 — "이 챕터는 도구 비판이 아니라 도구 사용 검정이다"
  - **묶음 A — "렌더 중에 풀 수 있는 문제" (약 2,500단어):**

    - #2 비싼 계산을 effect+state로 → useMemo
    - #5 cascading effects (A→state→B→state→A) → 한 핸들러에서 다 계산
  - **묶음 B — "이벤트 핸들러로 풀 수 있는 문제" (약 2,500단어):**
    - #4 사용자 이벤트(구매, POST)를 effect로 → 핸들러
    - #6 앱 초기화를 effect로 → 모듈 최상단 또는 didInit 가드
    - #7 자식→부모 통신을 effect로 → 같은 이벤트에서 부모 상태도 함께 갱신
  - **회수 박스 — #3 (prop 변화 리셋, 약 600단어):** "4장에서 이미 본격 처방했다. `key` prop과 렌더 중 set이 답이다 — 다시 짧게 회수만 한다."
  - **마무리 (약 400단어):** 한국 커뮤니티의 useEffect 무한 루프 패턴 셋 빠른 진단 + 12장으로의 다리

- **다룰 안티패턴:** #1, #2, **#3은 회수 박스로만**, #4, #5, #6, #7
- **연습 문제:**

  2. (신규) cascading effects 무한 루프 진단·고치기

- **분량:** 약 7,500 단어 (P0-2 반영, 8,500 → 7,500)

### 13장. 의존성 배열에 거짓말하지 않기
- **핵심 질문:** linter exhaustive-deps에 짜증날 때, 거기엔 사실 어떤 진실이 숨어 있는가?
- **학습 목표:**
  - 11장의 "Reactive Value" 정의를 의존성 배열에 그대로 적용한다
  - 의존성 배열은 hack이 아니라 동기화 계약임을 받아들인다
  - 객체/함수 의존성을 정직하게 줄이는 법을 안다
  - 함수형 업데이트와 `useEffectEvent`의 자리를 구분한다
- **주요 내용:**
  1. 의존성 배열의 의미 재정의 — 거짓말하면 stale closure (11장 Reactive Value 회수)
  2. 객체를 deps에 넣어서 발생하는 폭주 → 객체를 effect 안으로 이동, 또는 원시값 분해
  3. **(P1-5 회수)** ref를 활용해 의존성 배열에서 비반응형 참조를 빼내는 패턴 — "10장에서 본 ref가 여기서 다시 등장한다"
  4. 함수형 setState로 의존성 끊기 (`setMessages(prev => [...prev, m])`)
  5. **`useEffectEvent` 압축 박스 (약 800~1,000단어, P1-4 반영):** theme 변경 시 reconnect 없이 알림에만 최신 theme 반영. 챕터 후반 한 박스로 분리
  6. **실험적 API 박스:** 2026년 시점에도 `useEffectEvent`는 카나리. 책의 입장 — "어휘로는 가치 있다, 단 lint 침묵용으로 쓰지 마라" (논쟁 5.6)
  7. linter suppress가 만드는 미래의 디버그 비용
  8. **논쟁 회수:** linter exhaustive-deps는 표준이 아니라 버그를 강제한다는 davedx 비판 vs 옹호
- **사용할 코드 예제:** 패턴 3.16 (객체 deps 제거), 3.17 (함수형 setState), 3.15 (useEffectEvent — 박스로만)
- **다룰 안티패턴:** #8 (객체/함수 deps), #9 (lint suppress), #15 (의존성 누락 stale), #20 (useEffectEvent 오용)
- **연습 문제 (P1-4 반영, useEffectEvent 의존도 1개로):**
  1. 시드 21번 (theme + chat reconnect 회피, `useEffectEvent` 사용)
  2. (신규) `// eslint-disable-next-line react-hooks/exhaustive-deps`가 붙은 코드 한 덩어리를 진단해서 진짜 의존성 형태로 고치기
  3. (신규) 함수형 setState로 의존성 끊기 — 패턴 3.17 응용
  4. **(시드 22번 이전됨)** 외부 카메라 위젯 객체 deps 함정 → 14장 커스텀 훅 챕터의 도전 과제로 이동
- **분량:** 약 8,000 단어 (8,500 → 8,000, useEffectEvent 박스 압축)
- **이전 연결:** 12장의 "필요 없는 effect 제거" 다음 단계 — "필요한 effect를 정직하게 쓰기" + 11장 Reactive Value 정의 활용
- **다음 챕터로의 다리:** "같은 effect/state 패턴이 컴포넌트마다 반복된다면, 이제 추출할 시간이다."

### 14장. 커스텀 훅 — 로직을 공유한다, 상태를 공유하지 않는다
- **핵심 질문:** 커스텀 훅은 언제 추출하고, 무엇을 공유하는가?
- **학습 목표:**
  - "두 번 이상 같은 패턴이 보일 때 추출한다"는 기준을 적용한다 (Wieruch)
  - `use` 접두사 규칙의 의미를 안다
  - 같은 훅의 두 호출이 독립적인 state를 갖는 이유를 안다
  - "라이프사이클 래퍼" 안티패턴을 피한다
- **주요 내용:**
  1. useFormInput, useChatRoom, useDelayedValue 사례
  2. **로직 공유 ≠ 상태 공유** — 호출마다 독립된 state
  3. `use` 접두사는 강제다. Hook을 부르지 않는 함수에 `use`를 붙이지 마라
  4. 다중 값 반환 — 객체 vs 배열 (Wieruch 입장: 호출자가 이름 붙이게 배열)
  5. **추출 전에 시도할 것:** 컴포넌트 합성, JSX children
  6. **하지 마라:** `useMount` 같은 라이프사이클 래퍼 — Effect의 동기화 의미를 흐린다
  7. TS로 커스텀 훅 시그니처 깔끔하게
  8. **(P1-4 회수)** 외부 카메라 위젯의 객체 deps 함정을 커스텀 훅(`useCameraWidget`)으로 깔끔하게 해결하는 도전 과제

### 15장. 마지막 카드 세 장 — useSyncExternalStore, useImperativeHandle, flushSync
- **핵심 질문:** `useSyncExternalStore`, `useImperativeHandle`, `flushSync` — 언제 꺼내고, 왜 마지막인가?
- **학습 목표:**
  - 외부 store는 effect로 직접 구독하지 않고 `useSyncExternalStore`로 한다
  - `useImperativeHandle`로 자식의 ref API를 좁힌다
  - `flushSync`로 동기 DOM 업데이트가 필요한 정확한 순간을 안다
- **주요 내용:**
  1. `useSyncExternalStore` — `online/offline`, 외부 상태 라이브러리, SSR/concurrent 안전성
  2. `useImperativeHandle` — 모달의 `open()`/`close()`만 노출
  3. `flushSync` — 새 todo 추가 후 즉시 그 항목으로 스크롤
  4. 세 API 모두 "마지막 카드"라는 공통점 — 평소엔 꺼내지 않는다
  5. **(P0-5 회수) 통합 시나리오 — 작은 칸반 앱:** 1부의 reducer/context, 2부의 ref/effect/커스텀 훅, 그리고 **2장에서 보내준 정규화(lookup table)**를 column/card 데이터 모델에 적용. UI state는 useReducer + Context, server state는 별도(개념 설계). 정규화는 여기서 본격 등장한다.
- **사용할 코드 예제:** 패턴 3.18 (`useSyncExternalStore`), 3.20 (`useImperativeHandle`), 3.21 (`flushSync`), 3.5 (정규화 — 칸반 데이터 모델)
- **다룰 안티패턴:** #17 (외부 store를 effect로 직접 구독), #12 (중첩 state — 칸반 시나리오에서 본격 처방, P0-5)
- **연습 문제:**
  1. 시드 17번 (`useOnlineStatus`를 effect → `useSyncExternalStore`로 리팩토링)
  2. 시드 20번 (`flushSync` + 새 메시지 스크롤)

## D. 내러티브 아크

### 1부 안에서의 흐름 (왜 이 순서인가, 8개 챕터)
1장은 "상태를 식별하는 안목"이라는 가장 큰 그림에서 출발한다. 모든 후속 패턴은 결국 "어떤 시각 상태가 존재하고, 어떤 트리거가 그 사이를 잇는가"로 환원되기 때문이다. 2장에서 그 안목을 도구화한다 — 무엇을 state로 두고 무엇을 두지 않을지(정규화는 14장으로 연기). 3장은 시야를 한 컴포넌트에서 두 컴포넌트로 넓힌다(끌어올리기). 4장은 다시 한 컴포넌트로 좁히지만, 이번엔 "정체성"의 축으로 — `key`. 4장은 또한 "prop 변화에 effect로 반응하지 마라"의 본격 처방을 끝내, 12장이 가벼워지게 한다. 5장에서 처음으로 추상화 계단을 한 단 올린다(reducer). 6장은 데이터 흐름을 깊은 트리로 펼친다(context). 7장은 5장과 6장의 결합. 그리고 **새로 신설된 8장**이 7장의 한계(selective subscription 부재)를 타고 도구 선택의 문제 — Zustand/Redux/Jotai/Server State — 로 자연스럽게 이어진다. 즉 1부는 **"안목 → 모양 → 위치 → 정체성 → 모음 → 전파 → 결합 → 도구 선택"**의 여덟 박자다.

### 2부 안에서의 흐름 (왜 이 순서인가, 7개 챕터)
9장은 1부에서 2부로 시야를 넓히는 전환 챕터다. "React 안과 밖"이라는 메타포를 도입해 ref와 effect의 자리를 같은 지도 위에 그린다. 10장은 가장 가벼운 escape hatch인 ref부터 — 값의 보관과 DOM 접근. **11장은 확장돼** effect를 "라이프사이클이 아닌 동기화"로 다시 정의하면서, 동시에 **Effect 라이프사이클과 Reactive Value의 정의**를 이 챕터 안에서 끝낸다. 12장은 책의 절정 — "효과가 필요 없는 곳에 effect를 쓰는 여섯 가지 비극(+ 4장 회수 박스)"을 두 묶음으로 나눠 푼다. 12장이 음각이라면 13장은 양각 — "진짜 필요한 effect의 의존성 배열을 정직하게". 14장은 패턴이 반복될 때 추출하는 법. 15장은 손에 잘 잡히지 않던 세 API + 칸반 앱 통합 시나리오로 1·2부를 묶는다. 즉 2부는 **"지도 → 값 → 사이클·라이프사이클 → 빼기 → 정직 → 추출 → 마지막 카드 + 통합"**의 일곱 박자다.

### 1부 → 2부 전환 논리
순서는 의도적이다. **상태를 먼저 다루지 않으면 escape hatch의 의미가 살지 않는다.** "ref는 escape hatch다"라는 말은 "상태와 effect의 표준 흐름이 무엇인지 안 다음에야 의미가 있다." 12장의 "Effect가 필요 없을 때"가 책의 절정인 이유도, 1부에서 다진 상태 관리의 도구함이 충분히 채워진 다음에야 "이건 effect가 아니라 상태/이벤트다"라는 진단이 가능하기 때문이다. **새로 신설된 8장은 1부 → 2부 다리를 더 매끄럽게 만든다** — Server state라는 외부 세계 개념이 8장 후반에 등장하면서, 9장의 "React 안과 밖" 메타포가 이미 8장에서 한 발 디딘 셈이 된다.

### 책 전체 감정 곡선
- **서문~0장: 자가 진단의 따끔함** — "어, 나 이거 잘못 알고 있었나?"
- **1~3장: 안도감** — "아, 이렇게 생각하면 깔끔하구나."
- **4~5장: 작은 충격** — "key가 그런 거였어?", "reducer가 이렇게 풀리네."
- **6~7장: 자신감 상승** — "큰 앱도 이 구조면 무섭지 않다."
- **8장(신설): 시야 확장** — "도구는 결정 트리로 고른다, server state는 다른 세계다."
- **9장: 호흡 고르기** — 전환의 평지.
- **10~11장: 다시 까칠한 영역** — "ref/effect는 늘 자신 없었는데. 그리고 Reactive Value라는 어휘를 처음 갖는다."
- **12장: 정점의 통찰** — "내가 쓴 effect 절반이 필요 없었다."
- **13장: 책임감** — "의존성 배열은 계약이다."
- **14~15장: 통합과 자신감** — "이제 도구 카탈로그가 머릿속에 그려진다. 칸반 앱 시나리오에서 모든 것이 한 자리에 모인다."
- **닫는 글: 평온** — "월요일 아침에 쓸 수 있는 세 문장이 손에 남는다. 0장 8문항을 이번엔 다 풀 수 있다."

즉 **혼란 → 안도 → 작은 충격 → 시야 확장 → 정점의 통찰 → 평온**의 곡선.

## E. 챕터 간 콜백 / 재방문 계획 (P1-5 반영, 8개 → 6개)

좋은 책은 한 번 등장한 개념을 나중에 다른 각도에서 다시 보게 만든다. 미리 약속해둔 회수 지점들 (장식적이던 2개 콜백을 정리해 6개로 압축):

1. **1장의 status enum → 5장 reducer state shape으로 진화**
   - 5장 도입부에서 "1장에서 우리가 만들었던 typing/submitting/success enum, 기억하나? 그걸 reducer 안으로 옮겨보자."

2. **2장의 "선택은 ID로 저장" → 3장에서 부모-자식 props로 재등장**
   - 3장에서 `selectedId`를 부모에 두고 props로 내릴 때 "2장의 그 패턴이 두 컴포넌트 사이로 확장된 것"이라고 명시.

3. **4장의 "prop 변화에 effect로 반응하지 마라" 본격 처방 → 12장에서 회수 박스로 짧게 인용**
   - 12장 회수 박스에서 "4장에서 이미 본격 처방했다. `key` prop과 렌더 중 set이 답이다."로 한 페이지만 회수.

4. **6장의 "Context를 너무 일찍 쓰지 마라" → 14장에서 "커스텀 훅을 너무 일찍 추출하지 마라"로 평행 인용**
   - 14장에서 "Kent Dodds가 'Don't reach for context too soon'이라고 했다면, Wieruch는 '두 번 이상 패턴이 보일 때까지 기다려라'고 한다. 같은 정신이다."

5. **(P1-5 보강) 10장의 ref → 13장에서 "객체 의존성 회피 도구"로 명시 회수**
   - 10장 7번 소절 "한 걸음 더" 박스에서 "ref는 의존성 배열 폭주를 막는 도구이기도 하다 (13장 회수 예고)"를 미리 못 박는다. 13장에서 약속을 갚는다.

6. **10장의 race condition `let ignore = false` → 14장에서 `useFetch` 커스텀 훅으로 패키징**
   - 14장 시드 23번에서 "11장에서 손으로 풀었던 그 ignore 패턴, 이제 훅 안에 가둘 시간이다."

7. **(특수) 0장 자가 진단 8문항 → 닫는 글에서 "이제 같은 문항을 다시 풀어보자"로 회수**
   - 책의 시작과 끝을 같은 진단으로 묶는다. 8문항이 0장에서 명시적으로 확정되었으므로 회수 약속이 결과까지 보장된다(P0-4).

8. **(특수) 2장이 미룬 정규화 → 15장 칸반 앱 통합 시나리오에서 본격 등장**
   - 2장에서 "깊은 트리는 평탄화 가능하다"고 한 단락 예고만 한 후 15장에서 약속 이행. 더 이상 장식적이지 않다(P0-5 + P1-5 정리).

**제거된 콜백 (P1-5):** 기존 안 #5(9장 ref → 12장 객체 deps)는 9장에서 용도 명시가 약했고, 기존 안 #7(2장 정규화 → 14장)은 정규화 자체를 P0-5에서 옮긴 결과 자연 정리됨. 위 #5·#8로 흡수.

이 6개의 회수 지점이 책을 "챕터의 모음"이 아니라 "한 권의 책"으로 만든다.

### 일관성 위험과 통제
1. **TS-기본·JS-폴백 룰 통합 (P1-2):** B절의 코드 예제 정책을 한 줄로 통일했다 — "**개념을 처음 도입하는 예제는 JS 한 번 → TS 본격, 그 뒤로는 모두 TS**". 챕터 첫 등장 코드는 JS, 그다음 TS 박스가 같은 자리에서 따라온다.
2. **코드 길이 폭주 위험:** 30~40줄 룰을 깨는 예제는 두 단계로 쪼개거나 부록 깃 저장소로 옮긴다.
3. **레퍼런스 한계:** 한국 기업 블로그(우아한형제들/카카오/토스/네이버 D2) 1차 발췌가 부족하다. 챕터 작성 시 "한국 커뮤니티" 인용이 velog 위주로만 흘러갈 위험. **부록 B FAQ 30선 작성 규칙(P1-6)에 "최소 10개는 한국 기업 블로그 1차 출처"를 명시.** 1·5·11·12장 작성 전에 web-research 한 라운드 보강 권장 (orchestrator에 신호).
4. **`useEffectEvent` 위상 (P1-4 반영):** 2026년 시점에도 실험적. 13장에 명시적인 "실험 API" 박스를 두고, 챕터 후반의 800~1,000단어 박스 한 곳에만 등장한다. 본문 흐름이 이 API에 의존하지 않도록(연습 문제도 시드 21번 한 개로 제한) 구성.
5. **분량 폭주 위험:** 본 챕터 15개 평균 약 7,000 단어로 잡으면 본문 약 105,000 단어. 부록까지 합치면 113,000 단어 근처. 305~355페이지 예산 안. 챕터별 단어 수를 매 챕터 끝에서 측정.
6. **챕터 12가 너무 무거워질 위험:** 책의 절정이지만 안티패턴 6개(+ #3 회수 박스)를 한 챕터에 욱여넣으면 독자가 지친다. **묶음 A·B의 두 묶음 구조와 도입부 1,500단어 분리(P0-2)**로 통제. 각 안티패턴은 "30초 진단 → 1분 처방"의 동일 포맷으로 짧게 끊는다.
7. **15장 통합 시나리오의 분량 부담 (P0-5):** 정규화 + 칸반 앱이 한 챕터에 들어가면서 7,500 단어로 늘었다. 칸반은 "개념 설계까지만, 풀 구현은 부록 깃 저장소로" 룰을 명시.
