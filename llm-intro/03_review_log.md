## 리뷰 라운드 1

### Critical (반드시 반영)

- **[7장] "디코딩 전략"이 API 파라미터 절에 흡수되어 개념이 뭉개짐**
  - 근거: Reference 1.5는 Greedy / Beam / Temperature / Top-k / Top-p / Autoregressive를 **독립 개념 블록**으로 다루고 ([웹:HF-Decoding]), 6.5 "Temperature=0 is a Lie"의 충격은 **디코딩이 무엇인지를 이해한 뒤에야** 의미가 산다. 현재 계획 7장은 `temperature, top_p, max_tokens, stop, finish_reason`을 "Chat Completions API 요청/응답 스키마 해부"로 묶어버려서, 독자가 "왜 top_p가 nucleus sampling인지"를 모른 채 파라미터 값만 조정하게 된다. 4장에서 미니 GPT를 돌리면 `logits → softmax → sampling` 루프가 **눈앞에 있는데** 그걸 5장 직전에 설명 안 하고 7장 API 해부 안에서 얼버무리는 건 구조적 낭비.
  - 제안: **4장 말미에 "디코딩 루프" 절(1.5K~2K자)을 신설**. 직접 만든 모델의 출력층에서 Greedy → Temperature → Top-k → Top-p를 한 장면 안에서 실행하고 결과 비교. 7장에서는 "4장에서 본 것을 API로 쓰면 이렇다"로 받는다. 이 변경이 4장 32K → 33~34K로 커지지만 5장 도입의 "왜 Instruction Tuning이 필요한가"에도 연결고리가 생긴다 (base 모델 sampling의 한계를 몸으로 봤기 때문).

- **[10장] 12,000자로 6개 미래 토픽을 다루는 건 물리적으로 얕음**
  - 근거: 에이전트/멀티모달/심화 FT/추론 최적화/긴 컨텍스트/평가·안전 6개 토픽 × "3줄 소개 + 추천 1건"을 12K자에 넣으면 토픽당 2K자, 실질 본문은 1.5K. 저자의 "닫히지 않은 지도" 목적에는 맞지만, 독자가 **"그래서 내가 다음에 어디로 가야 하는지"를 판단할 근거가 없다.** 특히 에이전트(function calling, MCP)는 2025~2026 현장에서 "LLM 입문 다음"의 기본값이 되었는데 2K자로는 MCP 이름만 나열하고 끝난다.
  - 제안: **10장을 18,000~20,000자로 증량**. 토픽당 (a) 왜 이게 다음 단계인가 1문단, (b) 어떤 문제를 풀려고 나왔나 1문단, (c) 최소 재현 난이도 + 추천 자료 1건. 에이전트·멀티모달은 2025년 현장 중요도를 반영해 토픽당 3K, 나머지 4개는 각 2.5K 수준. 부록 C와 역할 분담이 겹치지 않도록 10장은 "방향성+1차 자료", 부록 C는 "난이도 태그 카탈로그"로 분리 명시.

- **[9장] 케이스 스터디와 거버넌스 둘 다 "보완 리서치 필요" 마크 — 18K자의 절반이 공급 리스크**
  - 근거: 9장은 "책의 모든 조각을 한 장의 플로차트로 소환하는 역인덱스"라고 선언했는데, 실제 내용 중 **(a) 거버넌스·red-teaming** (Reference 8.3), **(b) 한국어 실패담 케이스 스터디** (Reference 8.2) 두 절이 모두 미확보. 이 두 절이 9장의 감정적·실무적 피크를 담당하므로, 리서치가 실제로 들어오지 않으면 9장은 "4~8장 요약 + 플로차트"로 축소되어 독자가 덮게 된다.
  - 제안: 집필 착수 전 **보완 리서치 ③(한국어 실패담)과 ④(거버넌스)가 "제출 조건"임을 계획 문서 자체에 명시**. 만약 한국어 실패담 인터뷰 2~3건이 확보되지 않으면, 9장은 **"Toby 본인의 실패담 1건 + 공개된 한국어 프로덕션 포스트모템 2건"**(카카오·네이버·토스 기술 블로그의 서비스 회고)으로 대체 설계를 미리 준비. "보완 필요"를 계획에 남겨두는 건 현시점에선 리스크 이관이지 해결이 아니다.

### Should (반영 권장)

- **[6·7·8장 연속 실습 피로도] 저자가 제기한 리스크는 실재한다. 단, 해법은 "쉼터 챕터"가 아니라 "내부 리듬 교정"**
  - 근거: 26K+34K+26K = 86K자의 실습 연쇄는 분량 자체보다 **관점 회전 지점이 장 안에 숨어 있다는 게** 진짜 문제. 6장은 "모델 만들기 관점", 7장은 "모델 쓰기 관점", 8장은 "지식 주입 관점"으로 각 장마다 패러다임이 바뀌는데, 계획에는 이 전환이 "장 제목"으로만 표현됨. 독자는 장이 넘어갈 때마다 자기 위치를 다시 찾느라 피로가 누적된다.
  - 제안: (a) **각 장 도입부에 "지금 우리가 어디에 있나" 1페이지 지도** — 2~8장 전체 여정 도식에 현재 장을 하이라이트. (b) 별도 쉼터 챕터 **반대** — 10장짜리 책에서 쉼터를 끼우면 목차 리듬이 어그러진다. (c) 대신 **7장 중간에 "지금까지 만든 것 vs 지금부터 쓰는 것" 2페이지 회고 콜아웃**을 내장. 8장 말미에 "파인튜닝-RAG-프롬프트 3각을 한 장면에 겹쳐보기" 콜아웃. 이 두 콜아웃이 피로도 완화의 실질 장치.

- **[4장 ↔ 6·7장 코퍼스·모델 선택의 일관성 부족]**
  - 근거: 4장은 한국어 셰익스피어(세종실록/나무위키)로 from-scratch, 6장은 Gemma/Llama 3 8B 한국어 요약 QLoRA, 7장은 OpenAI/Anthropic API + Ollama 스왑. **세 장에서 "한국어 데이터"가 세 번 다 다르게 나타나는데 서로 연결되지 않는다.** 독자는 4장에서 만든 감각을 6장으로 가져갈 수 없다.
  - 제안: **실습 스토리 라인을 통일**. 예를 들어 "사내 개발 문서 요약/Q&A"라는 **한 태스크**를 설정하고, 4장은 그 태스크의 축소판(작은 코퍼스 from scratch), 6장은 Llama 3 8B QLoRA로 같은 태스크, 7장은 API로 같은 태스크, 8장은 RAG로 같은 태스크. 네 가지 해법이 **같은 문제에 대한 서로 다른 답**임이 몸으로 느껴진다. 이것이 9장 플로차트의 최강의 근거가 된다.

- **[부록 A~D] 4개는 과하지 않다. 단 D의 근거는 약함**
  - 근거: A(환경 세팅) B(용어집) C(더 읽을거리)는 기술 입문서의 표준이며 활용도가 높다. 반면 **D "논문 빠르게 읽는 법"은 Reference 6.7의 modulabs 인용 1건이 유일한 근거**이고, 책 본문에서 "논문을 직접 읽어야 한다"고 요구하는 챕터가 없다 (5장도 의사코드 위주, 7·8·10장도 1차 자료 포인터 수준). 독자가 부록 D를 "필요해서" 펼치는 시나리오가 성립 안 함.
  - 제안: **부록 D 제거, 10장의 "저자의 편지" 앞에 1.5페이지짜리 "논문 한 편 혼자 읽기" 팁 박스로 흡수**. Vaswani 2017 / InstructGPT / Chinchilla 세 편을 "제목→초록→도표→결론→본문" 순서로 짚는 시연을 그 박스 안에 압축. 이렇게 하면 부록은 A/B/C 3개로 줄어 밸런스 개선.

### Nice (여유 있으면)

- **[3장] "encoder-only / decoder-only / enc-dec 3종 표"에 BERT 분량이 애매**
  - 근거: Reference 1.4는 BERT를 명시하나, 이 책의 대상 독자(LLM 서비스 사용자)는 실무에서 BERT를 거의 만지지 않는다. 3장에서 BERT에 지면을 할애하면 "왜 decoder-only가 현대 LLM인가"의 논지가 희석될 수 있다.
  - 제안: 3종 비교표는 유지하되 **BERT는 "검색·임베딩 생성에 지금도 쓰인다"를 한 줄로만** 언급하고, 더 깊이 다루는 건 8장 RAG의 임베딩 모델 절(bge-m3 등)에서. 3장은 decoder-only에 80% 몰입.

- **[2장] Positional Encoding을 "수식 감각 먼저, 유도는 3장에서"로 쪼갰는데 2장에 약한 잉여가 생김**
  - 근거: 2장에 PE를 도입만 하고 3장에서 유도하겠다고 하면 2장 말미가 "숙제 남긴 수업"이 된다. 3장은 이미 Q/K/V, Multi-Head, 아키텍처 전체 분량이 큼.
  - 제안: PE는 **3장 도입부에서 한 번에** 설명. 2장은 토큰·임베딩의 "의미 벡터" 직관에만 집중하고, "위치는 어떻게 넣는가?"라는 질문을 3장의 열쇠로 넘긴다. 2장 20K → 18K로 축소, 3장은 28K 유지 (PE 들어와도 기존에 배분된 지면으로 소화).

### 저자 질문에 대한 답변

- **Q1 (6·7·8장 연속 실습 피로도):** (c) "현재 안 + 1~2페이지 쉼터"를 변주 채택. 별도 쉼터 챕터 반대(책 리듬 무너짐), 7↔8 순서 교체 반대(RAG가 API 응답·벡터 DB 전제라 7장 선행 필연). 대신 Should 첫째 항목대로 **각 장 도입부 "지도" + 7장 중간·8장 말미 회고 콜아웃**으로 해결. 근본 피로도는 분량이 아니라 **관점 회전이 숨어 있다는 것**이라는 점을 강조.

- **Q2 (4장 vs 5장 순서):** **현행(4→5) 유지 권장.** 논거: (a) 토비 문체는 "청유형으로 같이 걸어가기"인데 5장은 서사·역사·의사코드 중심이라 4장보다 먼저 두면 **"원리 먼저 코드 나중"의 bottom-up**이 되어 저자의 명시적 선언(서문의 "코드 먼저")과 정면 충돌. (b) 4장에서 Bigram → Self-Attention → Transformer의 학습 곡선을 본 독자는 5장 Scaling Laws의 "N·D·C 거듭제곱"이 **그래프가 아닌 경험**으로 읽힌다. (c) "1.3B가 175B를 이긴" 감정 피크가 5장에 오려면 독자가 "1.3B를 내 손으로 만드는 게 이런 거구나"를 이미 체감했어야 한다. 역전하면 감정 피크가 무뎌진다.

- **Q3 (4장 한국어 코퍼스):** **(b) AI Hub/모두의 말뭉치 공개 코퍼스 일부**. 근거: (a)는 한국어 독자에게 "내 언어로 돌아가는 순간"을 제공 못 하고, (c)는 저자 의존(재현 불가, 크기·품질 제어 어려움). 세종실록은 저작권은 OK지만 **고어·한자 혼용으로 교육용 코퍼스로는 난이도 스파이크**를 만든다. 권장은 **모두의 말뭉치(국립국어원) 현대어 일부 + 나무위키 덤프 소량(CC BY-NC-SA 2.0 라이선스 조건 본문에 명시)** 조합. 크기는 Karpathy Shakespeare의 ~1MB에 맞춰 1~3MB. 저장소 README에 다운로드·전처리 스크립트 포함해서 재현 가능성 확보.

- **Q4 (부록 D):** **흡수(10장으로).** Should 셋째 항목 참조. 부록으로 남기면 "있어도 안 펼치는 페이지"가 되고, 10장에 박스로 넣으면 "다음 걸음으로 논문을 읽어야 하는 독자"가 **맥락 안에서** 만나게 된다.

- **Q5 (Toby 문체와 코드 중심 장):** **(b) 구획 원칙 채택.** 근거: (a)는 `pip install` 앞에서 "같이 해봅시다"가 실제로 어색해지고, (c)는 책 전체 일관성을 잃는다. 권장 원칙 3조: (1) **장 도입/전환 문단은 청유형**("같이 돌려봅시다", "여기서 한 번 확인해봅시다"), (2) **코드 블록 바로 앞 1~2문단은 평어 서술**("아래 스크립트는 ~한다"), (3) **코드 주석·에러 해설·터미널 로그 설명은 담담한 매뉴얼체**("이 오류는 ~일 때 발생한다"). 한 장 안에서 문체가 3층으로 자연스럽게 내려가고, 장 말미 "독자가 얻는 것" 절은 다시 청유형으로 올라와 수미상관. 한국어 자료 45% 문제에 대한 보강: 이 구획은 **외부 한국어 자료 인용이 부족해도 저자 본인의 문체로 메워지는 구조**를 만들어준다. 즉 공감 인용은 서문·1·9·10장의 청유형 레이어에 집중 배치(보완 리서치 ③의 5~10건), 실습 장은 저자 체험담으로 충당.

### 전체 평

계획은 **전체 재설계가 필요할 만한 구조적 결함이 없다.** 레퍼런스 커버리지는 인상적으로 고르고(디코딩 전략 배치 1건을 제외하면), 10장 구성은 교재로서의 리듬을 지켰으며, Python/Java 75:25 비율과 4장 from-scratch + 6장 QLoRA의 Hybrid 선택은 커뮤니티 논쟁 4.6 현장 합의와 정확히 일치한다. 저자가 이미 인지한 리스크(6·7·8 피로도, 문체 마찰, 한국어 공급)에 대해 **진단은 정확하지만 해법이 추상적**이므로, 이번 리뷰는 그 해법을 구체화하는 데 집중했다. 치명적인 것은 (1) 디코딩 개념이 4장과 7장 사이에 길을 잃은 점, (2) 10장의 분량이 목적을 배신할 만큼 얇은 점, (3) 9장이 "보완 리서치 필요" 마크 두 개로 속이 비어 있다는 점 — 이 셋을 손보면 본 계획은 집필 착수에 충분하다. 한국어 실패담 공급(보완 리서치 ③)은 **계획 문서의 제출 조건**으로 격상시키는 편이 안전하다.
