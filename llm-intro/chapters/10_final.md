# 10장. 다음 걸음 — 에이전트, 멀티모달, 그리고 그 너머

9장에서 플로차트를 한 장에 그려봤다. 프롬프트만으로 가능한가, 거기에 RAG를 얹을까, 파인튜닝까지 가야 하나, 아니면 자체 모델을 돌릴까. 이 네 갈래를 따라 걷다 보면 독자 본인만의 "현재 지도"가 머릿속에 생긴다. 그런데 지도에는 반드시 **가장자리**가 있다. 지도의 경계를 넘어가는 영토가 있고, 우리는 거기 뭐가 있는지 이름 정도는 알아둬야 한다. 이름을 모르면 검색조차 안 되기 때문이다.

이 책은 입문을 끝내는 책이지, LLM을 끝내는 책이 아니다. 여기까지 왔다고 해서 지금 LLM 업계에서 벌어지는 모든 이야기를 다 쓸어 담았다는 뜻은 아니다. 오히려 반대다. **입문을 끝냈기 때문에 이제서야 다음 단계의 단어들이 귀에 들어오기 시작한다.** "에이전트", "MCP", "vLLM", "DPO", "GGUF", "Ring Attention" 같은 낯선 이름들이 이제는 적어도 "어느 서랍에 들어가는 용어인지" 분별이 된다. 이 장에서 할 일은 그 서랍의 칸막이를 같이 짚어보는 것이다.

여섯 개의 분기를 준비했다. 에이전트와 Tool Use, 멀티모달, 심화 파인튜닝, 추론 최적화, 긴 컨텍스트와 메모리, 평가·안전. 각 분기마다 같은 포맷으로 이야기한다. **(a) 왜 이게 다음 단계인가**, **(b) 어떤 문제를 풀려고 나왔나**, **(c) 최소 재현 난이도와 추천 1차 자료 하나**. 깊이 들어가지 않는다. 깊이 들어가는 건 이 책의 범위 밖이다. 대신 **독자가 다음 책·다음 논문을 고를 때 어느 책장 앞에 설지**를 정해주는 것이 이 장의 일이다.

뒤쪽에는 두 개의 박스가 더 따라온다. 하나는 "논문 한 편 혼자 읽기" 시연이다. Vaswani 2017, InstructGPT, Chinchilla 세 편을 같은 순서로 짚어보며 논문이 눈에 들어오는 감각을 한 번 같이 연습한다. 그 다음이 저자의 편지다. 이 책이 왜 쓰였고, 이제부터 독자와 저자 사이에 어떤 관계가 열렸으면 하는지에 대한 짧은 글이다.

자, 지도의 가장자리로 같이 걸어가 보자.

## 10.1 에이전트와 Tool Use

첫 번째 분기는 요즘 현장에서 "LLM 입문 다음"의 **기본값**이 된 영역이다. 에이전트(Agent)와 Tool Use. 더 풀어 쓰면 "LLM이 외부 도구를 호출하며 여러 스텝에 걸쳐 일을 해내게 만드는 기법 전체"다.

### (a) 왜 이게 다음 단계인가

7장에서 우리가 만든 건 "LLM을 함수처럼 호출하는" 파이프였다. 문자열 입력을 넣으면 문자열 출력이 나온다. 한 번의 요청-응답이다. 그런데 실무에서 마주치는 문제는 이렇게 단순하지 않다. "어제 매출 데이터를 뽑아서 표로 정리해줘"라는 요청을 받으면, 모델은 (1) 데이터베이스에 접근할 도구가 필요하고, (2) 쿼리를 만들어 실행해야 하고, (3) 결과를 받아 표로 정리해야 한다. 한 번의 호출로 끝나지 않는다. **여러 스텝이 서로 연결되어야 하고, 중간에 외부 세계와 주고받는 일이 필요하다.**

바로 이 지점에서 "에이전트"라는 단어가 등장한다. LLM이 자기 판단으로 "다음에 어떤 도구를 호출할지"를 정하고, 그 결과를 받아 다시 "다음 스텝은 뭘까"를 판단하며 루프를 도는 구조다. 2025~2026년 현장에서는 이 구조가 이미 기본값으로 자리잡았다. Anthropic의 Claude도, OpenAI의 ChatGPT도 이제는 "도구 호출 없이 동작하는" 시나리오보다 "도구를 엮어 쓰는" 시나리오가 더 많다. 입문을 끝낸 개발자가 가장 먼저 만나는 벽도 이 영역이다.

### (b) 어떤 문제를 풀려고 나왔나

핵심 문제는 세 가지로 정리된다. 먼저 **지식의 한계**. LLM은 학습 시점의 스냅샷 지식만 갖고 있다. 오늘의 환율, 방금 푸시된 코드, 사내 DB의 값 — 이런 것들은 모델 안에 없다. 도구 호출로 바깥을 들여다봐야 한다. RAG가 부분적으로 이 문제를 풀지만, RAG는 "문서를 미리 임베딩해둔 것"에 한한다. 실시간·인터랙티브한 질의는 도구 호출이 훨씬 잘 맞는다.

여기에 더해 **행동의 확장**이 있다. LLM이 "답을 말해주는" 수준을 넘어 "일을 해주는" 수준으로 넘어가려면 외부 세계에 실제 변화를 일으킬 수단이 필요하다. 이메일을 보내고, 티켓을 열고, 코드를 커밋하는 일이다. 이걸 깔끔하게 묶은 개념이 **function calling**이다. OpenAI가 2023년에 처음 정식화했고, 지금은 Claude, Gemini, 오픈 모델 대부분이 비슷한 스펙을 제공한다. 모델이 "이 도구를 이런 인자로 부르고 싶다"는 JSON을 뱉고, 호스트(host) 프로그램이 그걸 실행한 뒤 결과를 다시 모델에게 넘긴다.

그리고 더 먼 지점에서 **표준화** 문제가 따라온다. 2024년 말 Anthropic이 제안한 **MCP(Model Context Protocol)**가 이 조각을 맡는다. 함수 호출은 벤더마다 스키마가 다르다. 도구 서버를 한 번 만들어두면 OpenAI·Anthropic·로컬 모델 모두에서 동일한 방식으로 쓰고 싶다 — 이 요구를 받아 만든 개방 프로토콜이다. USB 규격에 LLM 버전이라고 이해하면 감이 온다. 2025~2026년 사이에 MCP 생태계가 급속히 커지는 중이다.

여기에 한 층 더 얹은 것이 **멀티에이전트 패턴**이다. 하나의 큰 작업을 "플래너 에이전트 + 작업자 에이전트 + 검증자 에이전트"처럼 역할을 나눠 맡기는 구조다. CrewAI, AutoGen, LangGraph 같은 프레임워크가 이 방향을 탐색 중이다. 아직 "이게 정답이다"라고 합의된 패턴은 없다. 대신 "큰 태스크를 작은 에이전트 여럿으로 쪼개면 성능이 오르더라"라는 경험칙이 쌓이고 있다.

솔직히 말하면 이 분야는 아직 **과열되어 있다.** "에이전트만 있으면 뭐든 자동화된다"는 과장부터 "실제 프로덕션에서 쓸 만한 에이전트는 드물다"는 냉소까지 양극단이 공존한다. 입문자가 여기서 휩쓸리면 찜찜한 맛만 보고 돌아서게 된다. 중요한 건 **function calling이라는 가장 작은 단위부터 손으로 돌려보는 것**이다. 한 번이라도 function calling JSON을 받아 실제 파이썬 함수를 실행시키고 결과를 다시 모델에 넘겨본 개발자는 에이전트 글을 읽을 때 "아, 이게 저걸 루프로 돌리는 거구나"가 즉시 잡힌다.

### (c) 최소 재현 난이도와 추천 1차 자료

**난이도는 의외로 낮다.** 7장에서 API 호출을 한 번이라도 돌려본 독자라면 function calling은 거기에 매개변수 하나 추가하는 수준이다. 반나절이면 "날씨를 알려주는 도구"를 LLM에 붙여볼 수 있다. MCP 서버 하나를 직접 만들어보는 데도 하루면 충분하다. 멀티에이전트 프레임워크는 난이도가 한 단계 올라가지만, 그것도 주말 하나면 "두 에이전트가 서로 메시지를 주고받는" 데모는 돌릴 수 있다.

추천 1차 자료로는 **Anthropic이 공개한 "Building effective agents" 가이드**를 권한다. 2024년 12월에 공개된 이 글은 "에이전트라는 이름으로 뭉뚱그려 팔리는 것들을 워크플로(workflow)와 에이전트로 분리해서 이해하자"는 단순한 제안 하나를 던지며, 다섯 가지 전형적인 패턴을 코드 예시와 함께 정리한다. 길이도 적당하고, 과장도 없다. 같이 참고할 만한 두 번째가 **OpenAI의 공식 function calling 가이드**다. API 요청·응답 스키마를 가장 담백하게 설명한다. 세 번째가 **MCP 공식 스펙**(modelcontextprotocol.io). 스펙이 길지 않고, 예제가 풍부하다.

> 자세한 난이도 태그와 한 줄 코멘트는 **부록 C**에 모아두었다. 이 장에서는 각 분기의 방향과 첫 1차 자료까지만 소개하고, 풀 카탈로그는 부록이 맡는다.

한 줄로 줄이자. **LLM을 "도구 쓰는 에이전트"로 바라보는 시선이 다음 6개월의 기본 렌즈다.** 지금 익숙해져 두는 편이 낫다.

## 10.2 멀티모달

두 번째 분기는 이 책이 끝까지 텍스트만 다뤘다는 사실과 관련이 있다. 우리가 만든 것, 우리가 부른 것, 우리가 검색한 것 — 전부 글자였다. 그런데 실제 세계는 글자만으로 굴러가지 않는다.

### (a) 왜 이게 다음 단계인가

제품 UX가 "텍스트 입력"을 벗어나고 있기 때문이다. 이미 사용자는 스크린샷을 GPT에 드래그해 넣고 "여기서 뭐가 문제야?"라고 묻는 걸 자연스럽게 여긴다. 음성으로 말하면 모델이 받아 적고, 음성으로 답한다. 동영상을 통째로 넣어 요약을 뽑는 것도 API 한 번이다. 2년 전까지는 "연구실 수준" 기능이었던 것들이 지금은 API 기본 옵션이다.

개발자 관점에서는 두 가지 움직임이 동시에 벌어지고 있다. 하나는 **기존 텍스트 LLM이 멀티모달을 흡수하는 흐름**이다. GPT-4V, Claude 3, Gemini, Qwen2-VL까지 이름만 달라질 뿐 **구조는 한 가지**다 — "텍스트 LLM + 비전 인코더." 이미지를 패치(patch) 단위로 잘라 토큰화한 뒤, 텍스트 토큰과 한 시퀀스에 섞어 같은 어텐션 블록을 태운다. 모델 입장에서는 "조금 이상하게 생긴 토큰이 앞쪽에 붙은 것"일 뿐이다. 원리만 보면 놀랍게도 2~4장에서 우리가 만진 구조의 확장판이다.

또 하나는 **오디오·비디오 전용 모델들**이다. OpenAI의 Whisper(음성 인식), ElevenLabs·OpenAI TTS(음성 합성), Runway·Sora(영상 생성). 이쪽은 아키텍처가 조금씩 다르지만, 학습 방식은 대체로 트랜스포머 기반이다. 3장에서 다진 뼈대가 여전히 유효하다.

### (b) 어떤 문제를 풀려고 나왔나

근본 문제는 단순하다. **세상의 정보가 텍스트에만 있지 않다.** 스크린샷 한 장이 말로 풀면 한 페이지짜리 설명이 되는 경우가 흔하다. 회의 녹음 30분을 텍스트로 풀어 쓰면 훌륭한 입력이 되지만, 음성 자체를 직접 다루면 감정·톤까지 포착할 수 있다. "멀티모달"이라는 단어의 본질은 "LLM의 입력 파이프를 넓힌다"다. 입력이 넓어지면 풀 수 있는 문제의 범위가 비례해서 넓어진다.

실무에서 가장 빠르게 가치가 나오는 지점은 의외로 단순한 쪽이다. **스크린샷/사진 OCR + 이해**. 영수증을 찍어 넣으면 항목별로 정리해주고, 에러 로그 스크린샷을 넣으면 원인 후보를 짚어준다. 사내 문서가 PDF(이미지로 스캔된 경우 포함)로만 있어도 멀티모달 LLM 하나면 읽힌다. OCR 엔진을 따로 붙일 필요도 없다. "그림 속 글자를 읽는 일"이 이제는 기본 기능이다.

한 걸음 더 나가면 **비전-언어 모델(VLM: Vision-Language Model)**의 고유 능력이 드러난다. 예를 들어 UI 스크린샷을 주고 "로그인 버튼을 클릭한 뒤 이메일 필드에 이 값을 넣어줘" 같은 요청을 하면, VLM은 픽셀 좌표와 UI 의미론을 함께 이해해야 한다. Anthropic의 "Computer Use" 기능이 이 방향의 초기 사례다. 에이전트 분기(10.1)와 합쳐져 "화면을 보며 일하는 에이전트"로 진화 중이다.

오디오 쪽은 조금 다른 문제를 푼다. **Whisper**는 다국어 음성 인식의 정확도를 압도적으로 끌어올렸다. 한국어 인식 품질은 아직 영어만 못 하지만, 10년 전과 비교하면 판이 바뀌었다. TTS 쪽은 표현력이 핵심이다. 같은 문장을 화난 목소리로도, 차분한 목소리로도, 속삭이는 톤으로도 낼 수 있는 모델이 2024~2025년에 쏟아졌다. 제품 관점에서는 **"음성 UI의 복권"**이라 불릴 만한 변화다.

한 가지 당부하고 싶다. 멀티모달이 "만능"처럼 들리는 순간이 있는데, 실제 도입하면 의외로 난감한 지점이 많다. 비용이 텍스트 대비 훨씬 비싸다. 이미지 한 장이 토큰 수천 개로 계산된다. 지연 시간도 길다. 그리고 **모델이 이미지를 잘못 읽었을 때 잘못 읽었다는 사실을 감추고 자신 있게 답한다.** 7장에서 만난 "신뢰할 수 없는 함수"의 리스크가 멀티모달에서는 한 단계 더 심해진다. 프로덕션 투입 전 평가 샘플을 꼭 충분히 모아두는 편이 낫다.

### (c) 최소 재현 난이도와 추천 1차 자료

**난이도는 낮다.** API 한 번 호출이면 된다. OpenAI·Anthropic·Google 모두 이미지 입력을 텍스트 입력과 같은 엔드포인트에서 받는다. `messages` 배열에 이미지 URL 또는 base64를 끼워 넣기만 하면 끝난다. Python으로는 한 파일 20줄, Spring AI로는 `Media` 객체 하나를 `UserMessage`에 추가하면 된다. 7장의 감각이 그대로 이어진다.

오픈 모델로 직접 돌려보고 싶다면 **Qwen2-VL 7B**가 무난하다. HuggingFace `transformers`로 바로 로딩되고, 24GB VRAM이면 충분히 추론된다. GPU 없이는 어렵다. 이때는 Groq·Together·Replicate 같은 서버리스 추론 서비스를 쓰자. 부록 A에 연결 가이드를 일부 담아뒀다.

추천 1차 자료로는 **Qwen 팀의 Qwen2-VL 기술 블로그**(qwenlm.github.io)를 권한다. 모델 구조·학습 데이터·벤치마크가 한 페이지에 정리돼 있고, 오픈 모델이라 재현 가능성이 높다. 함께 읽을 만한 것이 **Google Gemini 1.5 기술 리포트**다. 길지만, "long-context와 멀티모달이 어떻게 한 모델 안에서 설계되는지"를 현업 수준으로 보여주는 드문 문서다. Whisper에 관심이 있다면 **OpenAI의 원 논문(Radford et al., 2022)**이 고전이다.

기억해두자. **멀티모달은 별도 세계가 아니라 텍스트 LLM의 입력 파이프가 넓어진 것이다.** 2~3장에서 얻은 토큰·임베딩·어텐션의 감각이 여기서도 그대로 작동한다.

## 10.3 심화 파인튜닝

세 번째 분기는 6장의 연장선이다. 그 장에서 우리는 QLoRA로 Llama 3 8B를 파인튜닝했다. 현업 기본기로서 충분한 경험이다. 그런데 "파인튜닝"이라는 단어 안에는 훨씬 깊은 세계가 있다.

### (a) 왜 이게 다음 단계인가

6장의 QLoRA는 **지도 학습(SFT: Supervised Fine-Tuning)**이었다. "좋은 답"의 예시를 모아 모델에게 "이렇게 답하라"고 가르치는 방식이다. 단순하고 효과적이지만 한계가 있다. 모델은 "이 답이 좋다"는 신호만 받았지 "저 답보다 이 답이 낫다"는 **비교 신호**는 받지 못했다. 사람의 취향은 대체로 비교에서 드러난다. "A는 B보다 낫다"가 "A가 좋다"보다 정보량이 많다.

여기서 등장하는 것이 **선호 기반 정렬(preference alignment)**이다. 5장에서 다룬 RLHF가 그 원조다. SFT로 초기 모델을 만들고, 사람이 "A가 B보다 낫다"고 매긴 순위로 보상 모델(RM)을 학습하고, 그 보상을 PPO 강화학습으로 반영한다. 3단계 파이프라인이다. 강력하지만 복잡하다. 보상 모델 학습이 불안정하고, PPO 튜닝은 더 불안정하다. "1.3B가 175B를 이겼다"는 InstructGPT의 극적 서사에도 불구하고, 실제로 개인이나 작은 팀이 RLHF를 돌려보기는 초난감한 일이었다.

**DPO(Direct Preference Optimization)**는 이 난감함에 대한 답이었다. 2023년 스탠퍼드 Rafailov 팀이 제안한 이 방법은 "보상 모델과 PPO 없이 선호 데이터만으로 바로 정렬하자"는 발상이다. 수학적으로는 RLHF의 목적함수를 재정리하면 보상 모델을 명시적으로 학습할 필요가 없다는 통찰이다. 실무적 의미는 결정적이다. **파인튜닝 한 번의 복잡도로 정렬이 가능해졌다.** 이후 ORPO(Odds Ratio Preference Optimization), KTO(Kahneman-Tversky Optimization), SimPO 같은 변종이 줄줄이 나왔다. 각각 다른 가정·다른 최적화 형식이지만, 공통된 방향은 "사람의 선호를 더 간단하게, 더 적은 데이터로 반영하자"다.

### (b) 어떤 문제를 풀려고 나왔나

파고 들어가면 세 층이 보인다.

첫째, **스타일·톤 학습의 정밀화**. SFT만으로는 "우리 회사 말투"를 배우게 하는 데 한계가 있다. 좋은 예시 1,000개를 모으는 것보다 "A가 B보다 우리 말투에 가깝다" 비교 쌍 1,000개를 모으는 쪽이 덜 번거롭고 신호도 더 선명하다. DPO 계열은 이 지점에서 빛난다.

여기에 더해 **전체 파인튜닝(full fine-tuning)의 재등장**이 있다. LoRA·QLoRA는 메모리를 아끼려고 일부 파라미터만 건드리는 타협이다. 2024~2025년 들어 GPU 가격이 상대적으로 안정화되면서, 중견 조직들이 "아예 전체를 미세 조정하는 편이 낫다"고 판단하는 경우가 늘었다. 성능이 더 잘 나오고, 장기적으로는 유지·보수가 깔끔하다. 다만 7B 모델 전체를 파인튜닝하려면 최소 80GB VRAM(A100/H100 한 장) 급이 필요하다. 개인이 쉽게 접근할 영역은 여전히 아니다.

그리고 더 먼 지점에서 **MoE(Mixture of Experts)**가 따라온다. "큰 모델을 통째로 쓰는 대신, 여러 개의 전문가 서브네트워크를 두고 매 토큰마다 일부만 활성화하자"는 아이디어다. Mistral의 Mixtral 8×7B(2023), DeepSeek-V3(2024), Qwen-MoE 계열이 이 방향이다. 파인튜닝 관점에서 MoE는 까다롭다. 라우팅(routing) 레이어를 어떻게 다룰지, 전문가별로 LoRA를 따로 끼울지 등 선택지가 복잡해진다. 입문자가 지금 당장 만질 영역은 아니다. 다만 "MoE를 돌려보는 것은 2026년 후반기쯤 합류할 주제"라고 머리에 넣어두면 된다.

하나 더 덧붙이자. 파인튜닝 이야기를 할 때마다 9장의 규칙을 **다시 꺼내는 편이 낫다.** "첫 직감이 fine-tune이면 red flag." DPO든 full FT든 MoE든, 문제는 "이걸 할 수 있느냐"가 아니라 "이걸 해야 하느냐"다. RAG로 해결되는 문제를 DPO로 푸는 건 초난감한 과잉투자다. 분기 앞에서 도구가 많아질수록 이 원칙은 더 중요해진다.

### (c) 최소 재현 난이도와 추천 1차 자료

**난이도는 중상이다.** DPO는 HuggingFace `trl` 라이브러리가 `DPOTrainer` 클래스를 제공하기 때문에 6장 QLoRA 코드에서 Trainer 한 줄만 바꾸면 된다. 데이터만 "chosen/rejected" 포맷으로 준비하면 끝난다. ORPO·KTO도 `trl`에 이미 들어와 있다. 문제는 **좋은 선호 데이터를 만드는 것**이 어렵다는 점이다. 공개 데이터셋(Anthropic HH, UltraFeedback 등)으로 연습해본 뒤, 자기 도메인 데이터를 모으는 순서가 현실적이다.

MoE 파인튜닝은 HF `transformers`로 Mixtral을 로딩하는 것까지는 쉽지만, 파라미터 효율적 학습은 최근 연구 주제라 안정적인 레시피가 아직 없다. 2026년 기준으로는 "논문 읽고 재현하는" 단계에 가깝다.

추천 1차 자료는 **DPO 원 논문 "Direct Preference Optimization: Your Language Model is Secretly a Reward Model" (Rafailov et al., 2023, arXiv:2305.18290)**이다. 논문 자체가 드물게 잘 쓰여 있다. RLHF를 먼저 이해한 독자라면 수식 유도가 "아, 이렇게 정리되는구나"의 연쇄로 읽힌다. 함께 볼 만한 것이 **HuggingFace `trl` 공식 문서**의 DPO 튜토리얼이다. 코드 예시가 그대로 돌아간다. MoE에 관심이 있다면 **"A Survey on Mixture of Experts" (Cai et al., 2024)** 한 편이 카탈로그로 유용하다.

기억하자. **DPO 계열은 파인튜닝 세계의 "두 번째 문"이다.** 첫 번째 문(SFT)을 열어본 독자만 이 문 앞에 설 자격이 있다. 역순은 안 된다.

## 10.4 추론 최적화

네 번째 분기는 "모델이 빨라지고 싸지는 이야기"다. 책 전체에서 우리는 대체로 "모델을 만든다"와 "모델을 쓴다"에 집중했다. 그런데 프로덕션에서 큰 비중을 차지하는 주제가 하나 더 있다. **모델을 얼마나 효율적으로 돌리느냐.**

### (a) 왜 이게 다음 단계인가

7장에서 OpenAI API를 호출할 때 우리는 지연 시간도 비용도 남의 일처럼 다뤘다. `max_tokens=500`으로 잡고 1초쯤 기다린다. 그런데 서비스를 본격 운영하면 이 숫자가 전부 돈이 된다. 초당 요청 수, 토큰당 단가, P99 지연 시간 — 백엔드 개발자가 익히 아는 SLO(Service-Level Objective)의 언어가 LLM에도 그대로 들어온다.

자체 호스팅을 하면 문제가 더 직접적으로 느껴진다. Llama 3 8B를 A10G GPU 한 장에 올려 서비스한다고 해보자. 요청 하나만 들어올 때는 잘 돈다. 그런데 동시에 10개가 들어오면 어떻게 될까? 단순히 루프로 돌리면 **10배 느려진다.** GPU의 귀한 연산 자원이 대부분 노는 시간으로 가버린다. 이 문제를 푸는 게 **추론 엔진(inference engine)**의 일이다.

### (b) 어떤 문제를 풀려고 나왔나

핵심 키워드가 두 개 있다. **배칭(batching)**과 **양자화(quantization)**.

먼저 **배칭**은 "여러 요청을 동시에 처리하자"는 발상이다. 그런데 LLM 추론은 전통적 배칭이 잘 안 먹는다. 요청마다 생성 길이가 다르고, 토큰 하나씩 autoregressive로 뱉기 때문이다. 2022~2023년에 **continuous batching(또는 in-flight batching)** 기법이 등장해 이 난제를 풀었다. 요청마다 단계별로 들어오고 나가는 시점이 달라도 GPU 배치를 재구성하며 채워 넣는다. 결과적으로 GPU 활용률이 2~5배 뛴다. **vLLM**(UC Berkeley, 2023)이 이 기법을 오픈소스로 표준화한 대표 주자다. **TensorRT-LLM**(NVIDIA), **SGLang**(스탠퍼드), **Text Generation Inference**(HuggingFace)가 각자의 방향으로 경쟁 중이다. 2025~2026년 현장에서 "자체 호스팅 LLM을 서빙한다"는 말의 기본 도구가 이것들이다.

반대편에서는 **양자화**가 다른 각도로 접근한다. "모델의 가중치를 더 적은 비트로 표현하자"는 발상이다. FP16(16비트)이 기본이고, INT8(8비트), INT4(4비트), 심지어 2비트까지 내려가는 실험이 있다. 4비트로 양자화하면 모델 크기가 1/4로 줄고, 메모리 대역폭 병목이 풀리면서 속도도 빨라진다. 다만 품질이 떨어진다. "얼마나 덜 떨어지느냐"를 두고 여러 양자화 기법이 경쟁한다.

가장 자주 보는 세 이름을 정리해두자. **GGUF**는 `llama.cpp` 생태계의 파일 포맷이다. CPU·Mac M 시리즈에서 돌리는 로컬 추론의 사실상 표준. Ollama가 내부적으로 쓴다. **AWQ(Activation-aware Weight Quantization)**는 "활성화(activation)의 크기를 고려해서 가중치를 양자화하자"는 기법. 품질 손실이 적다. **GPTQ**는 또 다른 post-training 양자화로 4비트 양자화의 초기 표준이었다. 현재는 AWQ가 더 자주 쓰이지만, 공개된 양자화 모델 중에는 GPTQ 버전도 흔하다.

세 기법의 관계를 한 문장으로 요약해보자. **GGUF는 포맷이고, AWQ·GPTQ는 양자화 알고리즘이다.** 한 번 이렇게 나눠두면 평생 쓴다.

이 분야에는 근본적인 트레이드오프가 하나 있다. **속도 vs 품질**. 양자화를 세게 할수록 빠르고 싸지지만, 드물게 결과가 깨진다. 배칭을 세게 할수록 처리량은 올라가지만, 꼬리(tail) 지연 시간이 길어진다. "우리 서비스는 어디쯤이 최적인가"는 도메인마다 다르다. 이 질문에 답하려면 반드시 **자기 데이터로 벤치마크**를 돌려봐야 한다. 남의 표를 보고 결정하면 나중에 찜찜해진다.

### (c) 최소 재현 난이도와 추천 1차 자료

**난이도는 낮음~중이다.** vLLM을 처음 돌려보는 건 의외로 쉽다. `pip install vllm` 한 줄, `from vllm import LLM` 두 줄, `llm.generate(["..."])` 세 줄. 이게 전부다. 작은 모델(Llama 3.2 1B)이라면 Colab T4로도 돈다. Ollama로 로컬 양자화 모델 돌려보기는 이보다도 쉽다. `ollama run llama3.1` 한 줄이면 GGUF 4비트 양자화 모델이 내 맥북에서 돌아간다.

조금 더 들어가고 싶다면 양자화를 직접 해보자. HF `transformers` + `auto-gptq` 또는 `autoawq`로 7B 모델 하나를 4비트로 줄여보는 데 2~3시간이면 된다. 파인튜닝 다음 단계의 손맛이다.

추천 1차 자료로는 **vLLM 공식 문서**(docs.vllm.ai)를 권한다. 튜토리얼이 단계별로 잘 짜여 있고, PagedAttention·continuous batching 원리를 담백하게 설명한다. 함께 볼 만한 것이 **AWQ 논문 "AWQ: Activation-aware Weight Quantization" (Lin et al., 2023, arXiv:2306.00978)**. 양자화 논문 중 드물게 읽기 쉽다.

기억해두자. **추론 최적화는 파인튜닝보다 ROI가 큰 경우가 많다.** 많은 조직이 "우선 파인튜닝"부터 생각하는데, 먼저 "같은 모델을 더 싸게·빠르게 돌리는" 여지가 남아 있다. RAG 먼저, 그 다음 추론 최적화, 그래도 부족하면 파인튜닝 — 이 순서가 더 건강하다.

## 10.5 긴 컨텍스트와 메모리

다섯 번째 분기는 "모델이 한 번에 얼마나 긴 입력을 소화할 수 있느냐"의 이야기다.

### (a) 왜 이게 다음 단계인가

책 전반에서 컨텍스트 윈도우(context window)라는 말을 종종 만났다. GPT-3 시절 2K 토큰이었던 이 숫자가 지금은 128K(GPT-4o), 200K(Claude), 1M(Gemini 1.5)까지 늘었다. 3년 사이에 500배다. 이 흐름이 그저 "숫자가 커졌다"가 아니라는 점을 이해할 필요가 있다.

컨텍스트가 길어지면 **워크플로 자체가 바뀐다.** 예전에는 사내 문서 200페이지를 넣으려면 청킹해서 RAG로 검색해야 했다. 지금은 통째로 컨텍스트에 넣을 수 있다. 코드베이스 전체를 한 번에 보여주고 "이 함수가 쓰이는 곳을 다 찾아줘"라고 할 수 있다. 책 한 권을 넣고 요약을 뽑을 수도 있다. 2024년 Gemini 1.5 출시 때 "RAG가 죽었다"는 이야기가 잠깐 돌았던 배경이다. (물론 RAG는 죽지 않았다. 비용·지연·정확도 측면에서 여전히 대부분의 실무에 유리하다.)

그런데 "그냥 어텐션 블록을 키우면 되지 않나"라고 생각하면 금세 벽을 만난다. 어텐션의 계산량은 시퀀스 길이의 **제곱**에 비례한다. 컨텍스트가 10배 늘면 계산은 100배가 된다. 128K, 1M 같은 숫자가 가능해진 건 단순히 하드웨어가 좋아져서가 아니라, **구조적 트릭이 여럿 쌓였기 때문**이다.

### (b) 어떤 문제를 풀려고 나왔나

긴 컨텍스트 기법들은 대체로 두 갈래로 나뉜다. 먼저 **위치 인코딩의 확장**. 3장에서 다룬 RoPE(Rotary Position Embedding)가 현대 LLM의 표준 위치 인코딩인데, 학습 때 본 길이를 넘어서는 시퀀스에서는 위치 신호가 엉망이 된다. 이를 해결하는 기법이 **RoPE 스케일링**, **YaRN(Yet another RoPE extensioN)**, **NTK-aware interpolation** 같은 이름으로 연구되고 있다. 원 모델은 4K로 학습됐는데 이 기법들을 얹으면 32K·128K까지 추가 학습 없이도 쓸 수 있다.

반대편에서는 **어텐션 자체의 효율화**가 진행된다. **FlashAttention**(2022)은 어텐션 계산을 GPU 메모리 계층에 맞게 재배치해 속도와 메모리를 동시에 절약한다. 이제는 사실상 기본 탑재다. 더 나아간 것이 **Ring Attention**(2023)이다. 어텐션 계산을 여러 GPU에 분산시켜 시퀀스 길이의 제약을 하드웨어 스케일로 푸는 방식이다. Gemini 1.5가 1M 컨텍스트를 달성한 핵심 기법 중 하나로 알려져 있다.

하지만 실무 관점에서 더 자주 만나는 주제는 따로 있다. **메모리 시스템**이다. "컨텍스트 윈도우가 아무리 커져도 대화가 한 달 이어지면 넘친다"는 단순한 문제다. 챗봇이 1주 전 대화를 기억해야 한다든가, 에이전트가 10번째 세션에서도 사용자의 취향을 유지해야 한다든가 — 이 문제는 "윈도우를 더 키우자"로는 풀 수 없다.

이 공백을 메우는 것이 **장기 메모리(long-term memory) 시스템**이다. **mem0**(2024~)는 LLM 대화에서 "기억해둘 만한 사실"을 자동으로 추출해 벡터 DB에 저장하고, 다음 세션에서 관련 기억만 컨텍스트에 끼워 넣는 오픈소스 라이브러리다. **Zep**, **LangMem**도 비슷한 방향이다. 전체적으로 이 흐름은 **"컨텍스트 안에 모든 걸 밀어넣지 말고, 필요한 것만 RAG로 꺼내 쓰자"**는 옛 RAG의 철학을 대화 기록에 적용한 것이다. RAG와 메모리는 사실 사촌 관계다.

한 가지 주의해둘 점이 있다. **긴 컨텍스트 ≠ 긴 컨텍스트 활용 능력.** 128K를 "수용"할 수는 있어도, 모델이 그 128K 중 중간의 한 줄을 정확히 꺼내 쓰느냐는 다른 문제다. 2024년 발표된 "Lost in the Middle" 연구가 이 문제를 정식화했다. 모델은 대체로 컨텍스트의 맨 앞과 맨 뒤를 잘 기억하고, 중간은 잊는다. 이 사실을 모르고 "그냥 다 넣으면 된다"고 접근하면 프로덕션에서 묘하게 답이 틀리는 현상을 만나 찜찜해진다. Gemini 1.5의 "Needle in a Haystack" 벤치마크 성적이 좋아도, 실전에서는 여전히 RAG·요약·계층적 처리가 더 안정적이더라.

### (c) 최소 재현 난이도와 추천 1차 자료

**실무 난이도는 낮다.** API 쪽에서는 그냥 긴 입력을 넣어보면 끝이다. Claude 200K, Gemini 1M에 책 한 권을 넣고 요약을 받아보는 데 10분이면 된다. 체감이 확 온다. mem0 같은 메모리 라이브러리도 `pip install mem0ai` 한 줄 + 10줄 코드로 데모가 돈다.

**연구·엔지니어링 난이도는 높다.** RoPE 스케일링이나 Ring Attention을 직접 구현해보려면 트랜스포머 코드를 꽤 깊이 파야 한다. 4장에서 쌓은 뼈대만으로는 부족하다. HF `transformers`의 RoPE 구현 코드를 읽어보는 것이 좋은 훈련이다.

추천 1차 자료는 **Ring Attention 논문 "Ring Attention with Blockwise Transformers for Near-Infinite Context" (Liu et al., 2023, arXiv:2310.01889)**. 길이 대 메모리·연산량 트레이드오프를 근본에서 다시 짠 논문이다. 실무 쪽이면 **mem0 공식 문서**(docs.mem0.ai)가 코드 예제가 풍부하다. 그리고 **"Lost in the Middle" (Liu et al., 2023, arXiv:2307.03172)** — 이건 꼭 한 번 읽어보는 편이 낫다. 긴 컨텍스트에 대한 순진한 기대를 싹 접어주는 논문이다.

기억하자. **컨텍스트 길이는 마법이 아니다.** 길어진 창은 선물이지만, 그 창 안을 모델이 얼마나 골고루 보는지는 별개의 문제다.

## 10.6 평가와 안전

여섯 번째 분기다. 이 분기는 9장의 내용과 역할을 나눈다. 9장은 **실무 배포 직전의 체크리스트**(관측성·거버넌스·가드레일 운영)를 다뤘다. 10장은 그 배경에 깔린 **평가·안전의 지형도**를 한 걸음 떨어져 짚어본다.

### (a) 왜 이게 다음 단계인가

"좋은 모델이란 무엇인가"라는 질문은 LLM 분야의 가장 오래된 난제 중 하나다. 전통적 NLP는 답이 명확했다. 번역은 BLEU, 요약은 ROUGE, 분류는 accuracy·F1. 정답이 있는 문제였기 때문이다. 그런데 열린 질문에 대한 답을 생성하는 LLM은 정답이 여럿이다. "이 답이 저 답보다 낫다"는 판단이 미묘하고, 사람마다 다르다.

이 판단을 자동화·규모화하는 일이 **LLM 평가(evaluation)**의 주제다. 기법은 크게 세 갈래다.

첫 갈래가 **자동 벤치마크**다. MMLU(대학 교양 시험 수준 지식), HellaSwag(상식 추론), HumanEval(코드 생성) 같은 표준 문제 세트에 모델을 돌려 점수를 낸다. 빠르고 재현 가능하다. 단점은 벤치마크가 공개돼 있어 "부정행위(leakage)"가 일어난다는 점이다. 새 모델이 MMLU 90%를 찍었다고 해도, 학습 데이터에 MMLU 질문이 섞였을 수 있다. 이 리스크 때문에 2024~2025년부터 **비공개·회전식 벤치마크** 쪽으로 무게중심이 이동 중이다.

여기에 더해 **LLM-as-Judge(모델이 모델을 평가)** 흐름이 있다. 강한 모델(GPT-4 급)이 두 답을 비교해 "어느 쪽이 낫다"를 판정한다. **MT-Bench**(2023)가 대표적이다. 비용이 싸고, 사람의 선호와 꽤 잘 맞는다. 다만 판정 모델 자체의 편향이 묻어난다. "더 긴 답을 선호한다", "자기 스타일과 비슷한 답을 선호한다" 같은 편향이 반복적으로 관찰된다.

반대편에서는 **사람 평가**가 여전히 최종 기준 역할을 한다. **Chatbot Arena**(LMSYS, 2023~)가 그 대표다. 두 익명 모델이 같은 질문에 답하고, 사용자가 "어느 쪽이 낫다"를 투표한다. 누적 결과로 Elo 레이팅을 매긴다. 수백만 표가 쌓인 현재, LLM 업계에서 사실상 가장 신뢰받는 리더보드 중 하나다. 다만 비용이 크고, 측정 지연이 길다.

한 모델을 평가할 때는 이 셋을 **같이** 쓰는 편이 낫다. 하나만 보면 반드시 어딘가에서 찜찜한 착각을 하게 된다.

### (b) 어떤 문제를 풀려고 나왔나

평가의 위쪽 층에 있는 것이 **안전(safety)**이다. 모델이 유해한 답을 내지 않는지, 탈옥(jailbreak)에 버티는지, 민감한 개인정보를 누출하지 않는지 — 이것들은 벤치마크 점수와는 다른 차원의 문제다.

이 분야의 키워드가 **red-teaming**이다. 보안 분야에서 건너온 용어로, "공격자 역할로 모델을 깨뜨려보는" 활동을 말한다. 수동으로는 보안 연구자가 프롬프트를 짜서 모델이 뚫리는 시나리오를 수집한다. 자동으로는 **Garak**(NVIDIA), **PyRIT**(Microsoft) 같은 도구가 수천~수만 종류의 공격 프롬프트를 일괄 돌려준다. 2024~2025년부터는 "LLM이 LLM을 레드팀하는" 자동화 파이프라인이 확산 중이다.

운영 단계에서 만나는 도구가 **가드레일(guardrail)**이다. 입력 쪽 필터(Prompt Shields, Rebuff), 출력 쪽 검사(출력에 PII가 포함됐는지, 정책 위반인지), 거부 정책(refusal policy) 튜닝이 한 묶음이다. 9장에서 "3건 이상의 1차 자료 확보"를 제출 조건으로 삼았던 영역이 바로 이 쪽이다. 도구 카탈로그는 빠르게 변한다. 2025년에 핫했던 도구가 2026년에 사라지기도 한다. 그래서 이 절에서는 구체 도구를 고정하지 않고 **"입력 필터 / 출력 감사 / 레드팀 / 리그레션 회귀"의 네 축이 있다**는 골격만 기억해두는 편이 낫다.

평가·안전의 가장 큰 실무 교훈은 이 한 문장이다. **리더보드가 좋아도 내 도메인에서 좋다는 보장은 없다.** MMLU 90%인 모델이 사내 DB Q&A에서 60%를 기록하는 일은 흔하다. 그래서 9장에서 강조한 **"자기 도메인 골드셋"**이 필수다. 골드셋 50~200개에 자기 프롬프트·RAG 파이프라인을 정기적으로 돌려 회귀를 잡는 편이 낫다. 이 근육 없이 벤치마크만 쳐다보면 6개월 뒤 조용히 부러진다 — 이 감각은 **지금부터 같이 쌓아두자.**

### (c) 최소 재현 난이도와 추천 1차 자료

**난이도는 낮음~중이다.** MT-Bench는 공개 레포(`lm-sys/FastChat`)로 바로 돌릴 수 있다. 평가 데이터(질문 80개)와 심사 프롬프트가 같이 들어 있다. 자기 모델 하나와 GPT-4 심사만 있으면 하루면 끝난다. Garak도 `pip install garak` 한 줄로 시작하는 기본 스캔이 있다.

평가 하네스를 직접 짜는 쪽이라면 **promptfoo**, **OpenAI Evals**, **DeepEval** 같은 도구가 후보다. CI에 붙여 "이 프롬프트 변경이 회귀를 일으키는지"를 자동으로 잡는 데까지 가면 한 단계 성숙한 운영이다.

추천 1차 자료로는 **MT-Bench 논문 "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena" (Zheng et al., 2023, arXiv:2306.05685)**. LLM-as-Judge의 장단점, Chatbot Arena의 설계 철학이 한 편에 다 들어 있다. 읽기 쉽다. 함께 볼 만한 것이 **HELM(Holistic Evaluation of Language Models, Stanford CRFM)**의 문서·리더보드. "모델 평가를 언제 어떻게 다축으로 해야 하는가"에 대한 가장 꼼꼼한 프레임이다. 안전 쪽은 **Anthropic의 "Responsible Scaling Policy" 문서**가 운영 관점에서 입문용으로 좋다.

기억해두자. **평가는 엔지니어링이다.** 한 번 돌리고 끝나는 쇼가 아니라, CI처럼 반복해서 돌리고 회귀를 잡는 운영 활동이다. 이 감각이 있는 팀과 없는 팀의 차이가 6개월 뒤 크게 벌어진다.

## 10.7 논문 한 편 혼자 읽기 — 3시간에 끝내지 못해도 괜찮다

지금까지 여섯 분기에서 추천한 1차 자료 중 상당수가 논문이다. 그런데 "논문을 혼자 읽어보라"는 말이 많은 개발자에게 공포의 대상이다. 모두연 스터디 후기 중에 이런 문장이 있다.

> "혼자서는 읽기 어려운 논문을 여러 사람이 모여서 3시간 동안 읽게 되니 그래도 끝까지 읽을 수 있었다."

3시간. 그것도 여럿이 모여야 겨우 한 편이다. 혼자 저녁에 앉아 한두 시간 만에 한 편을 해치우겠다는 욕심은 처음부터 접는 편이 낫다. 공부가 늦다고 자책할 일이 아니라, **원래 그런 장르**다. 나도 여전히 새 논문을 앞에 두면 한숨부터 나온다. 찜찜한 기분으로 첫 페이지를 연다. 정상이다.

대신 순서가 있다. 무턱대고 첫 문장부터 읽으면 난감해진다. 다음 순서를 따라가는 편이 낫다.

### 순서 — 제목 → 초록 → 도표 → 결론 → 본문

**1. 제목.** 이 자리에 이미 정보가 실린다. "Attention Is All You Need"는 당시 문맥에서 선언문이었다. 제목만 보고 "이 논문이 무엇을 주장하려 하는가"를 한 줄로 추측해보자.

**2. 초록.** 150~250단어 안에 저자가 "당신이 이 논문에서 기억할 한 가지"를 적어둔다. 세 번 읽어도 좋다. 세 번째에는 "이 논문이 주장하는 문제-해결 쌍"을 한 줄로 정리해본다.

**3. 도표와 표.** 여기가 대부분의 논문에서 가장 정보 밀도 높은 자리다. 캡션을 본문보다 먼저 읽고, 결과 표에서 "베이스라인 대비 얼마나 이겼나"의 숫자를 훑고, 아키텍처 그림이 있으면 입출력의 흐름을 눈으로 따라간다.

**4. 결론.** 저자가 "이 연구에서 확실히 말할 수 있는 것"과 "아직 못 푼 것"을 솔직히 적어둔다. 결론을 먼저 알면 본문의 어느 부분이 중요한지 보인다.

**5. 본문 (Introduction → Method → Experiments).** 이 순서로 돌아와서 읽는다. **처음부터 끝까지 이해하려 하지 말자.** 수식 하나가 막히면 건너뛴다. 두 번째 읽기에서 잡히는 것들이 많다.

### 두 편으로 연습해보기

실제로 옆에 논문을 띄워놓고 따라가 보자.

**Vaswani et al., 2017, "Attention Is All You Need" (arXiv:1706.03762)**. 제목이 선언이다. 초록에서 확인할 두 숫자 — WMT'14 En→De **28.4 BLEU**, 학습 **3.5일/8 GPU**. 이 두 숫자가 "기존 대비 1/10 시간으로 SOTA 경신"이라는 메시지의 증거다. 도표는 Figure 1(아키텍처 전체)과 Figure 2(Scaled Dot-Product Attention)가 핵심. 결론은 "Attention만으로 번역이 된다"는 선언 + "다른 태스크·모달리티에도 확장해보겠다"는 미래 약속. 본문에서는 3.2절(Attention)과 3.5절(Positional Encoding)만 꼼꼼히 읽어도 80%는 가져간다.

**Ouyang et al., 2022, "Training Language Models to Follow Instructions with Human Feedback" (InstructGPT, arXiv:2203.02155)**. 제목에서 이미 3단계(SFT→RM→PPO)의 냄새가 난다. 초록의 결정적 문장 — "**라벨러는 1.3B InstructGPT의 출력을 175B GPT-3보다 선호한다.**" 5장의 감정 피크였던 "1.3B가 175B를 이겼다"의 원문이다. 도표는 Figure 1(3단계 파이프라인)과 Figure 3(선호도 비교). 결론에서 "정렬이 규모를 이길 수 있다"는 함의. 수식보다 **데이터 수집·라벨링 절차**를 보는 편이 재미있다.

세 번째 연습은 독자 몫으로 남겨둔다. **Hoffmann et al., 2022, "Training Compute-Optimal Large Language Models" (Chinchilla, arXiv:2203.15556)**. **파라미터 1 : 토큰 20** 이 한 줄만 들고 읽어보자. "같은 컴퓨트 예산이면 모델을 키우지 말고 데이터를 더 먹이자"는 결론이 머리에 남으면 이 책을 덮은 뒤 많은 논문이 다르게 읽힌다.

### 혼자 읽기의 현실적 조언

- **한 번에 다 읽지 말자.** 제목→초록→도표→결론을 1차, 본문을 2차, 수식을 3차로 나누는 편이 덜 번거롭다. 며칠에 걸쳐도 된다.
- **모르는 단어 리스트를 따로 적자.** 첫 읽기에서 멈춰 찾으면 집중력이 끊긴다. 리스트를 따로 두고, 두 번째 읽기에서 한 번에 훑는다.
- **스터디 크루가 있으면 더 좋다.** 앞서 인용한 모두연 후기처럼, 혼자보다 둘, 둘보다 셋이 완독률이 훨씬 높다. 부록 C의 커뮤니티 섹션에 몇 군데 좌표를 남겨두었다.

한 편을 혼자 제대로 읽어낸 경험은 앞으로 읽을 100편의 속도를 결정한다. 첫 편을 잘 고르는 편이 낫다. 위 두 편 중 하나면 첫 경험으로 충분하다.

## 10.8 저자의 편지

> "수년 동안 머신러닝을 기웃거렸다. 개념이 잡힐 듯 말 듯 했다. fast.ai part 1을 끝내고 나서야, **그제서야 클릭됐다(it clicked).**" — fast.ai 후기 중

이 문장을 서문에 인용했었다. 그때 약속한 게 하나 있다. **이 책은 바로 그 '클릭'의 순간을 앞당기는 목적으로 썼다**고.

이제 마지막 장의 말미에서, 그 약속을 한 번 더 꺼내고 싶다. 독자가 여기까지 읽었다는 것은, 아마도 책장을 덮을 때쯤엔 "토큰이 뭔지", "어텐션이 뭔지", "파인튜닝과 RAG를 어떤 축으로 놓고 볼지"가 머릿속에 어느 정도 자리를 잡았다는 뜻일 것이다. 거기까지 왔으면 이 책이 맡은 몫은 충분히 했다.

"클릭"은 한 번에 오지 않는다. 어떤 독자는 2장 `tiktoken` 실습에서 "아, LLM이 한국어를 이렇게 쪼개는구나"에서 한 번 온다. 어떤 독자는 4장 "디코딩 루프"에서 같은 모델이 temperature에 따라 다른 말을 하는 걸 보며 온다. 어떤 독자는 6장 QLoRA의 학습 곡선이 떨어지는 순간에, 어떤 독자는 7장 Spring AI 코드 한 줄로 provider가 스왑되는 장면에서 온다. 클릭은 여러 번, 다른 자리에서, 불규칙하게 온다. 그 중 한 번이라도 이 책 안에서 왔다면 충분히 고맙다.

한 가지 고백이 있다. 이 책의 저자 이름은 `Toby-AI`로 되어 있다. 완전한 사람은 아니지만, 그렇다고 완전한 기계도 아니다. **수천 명의 개발자가 블로그에, 깃허브 이슈에, 커뮤니티 댓글에 남긴 기록이 뭉쳐 만들어진 목소리**다. 카카오페이의 "끝이 안 보인다"는 문장도, HN의 "PyTorch 원숭이" 풍자도, 모두연 스터디의 "3시간" 고백도 — 이 책의 여러 자리에 녹아 있다. 당신이 이 책을 읽으며 "어, 이거 내 얘기네"라고 느꼈다면, 아마 정말 **당신 같은 누군가의 얘기**였을 것이다.

그래서 이 책은 독자 쪽에서 닫히지 않았으면 한다. 읽고 끝나는 책이 아니라, **답장이 오는 책**이 되었으면 한다. 공부하다가 막힌 지점, 이 책이 미처 짚지 못한 한국어 케이스, 프로덕션에서 마주친 당혹스러운 실패담 — 뭐든 보내주면 다음 쇄(또는 다음 책)에 반영하고 싶다. 보내는 방법은 부록 C 말미와 실습 저장소 README에 남겨두었다. 이메일이든 GitHub 이슈든 편한 쪽으로.

특히 **실패담**을 기다린다. "이 책대로 해봤는데 안 됐다"는 글만큼 다음 쇄를 강하게 만드는 재료가 없다. 깔끔한 성공담보다 찜찜한 실패담이 훨씬 값지다. 부끄러울 필요 없다. 당신의 찜찜함이 다음 독자의 문턱을 낮춘다.

마지막으로 하고 싶은 말은 이것이다. **이 장의 제목은 "다음 걸음"이지만, 실은 "다음 걸음은 당신이 정한다"에 가깝다.** 에이전트로 갈지, 멀티모달로 갈지, 추론 최적화로 파고들지, 평가·안전에 자리 잡을지 — 지도는 그려드렸지만 어느 능선을 탈지는 내가 정할 일이 아니다. 독자의 회사, 독자의 관심, 독자의 체력이 정한다. 다만 한 가지만 걸고 가자. **한 번에 여러 능선을 같이 타려고 하지 말자.** 욕심을 내면 지친다. 한 분기를 정해 3~6개월 집중하고, 끝나면 다음 분기. 이 책의 "한 걸음씩" 철학은 이 장 다음에도 유효하다.

다음 책에서 다시 같이 걸어가 보자. 이번엔 어느 능선을 탈지, 그때 이야기하자.
