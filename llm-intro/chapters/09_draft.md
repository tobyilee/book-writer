# 9장. 통합 — 어떤 도구를 언제 꺼내는가

8장 말미에서 우리는 파인튜닝-RAG-프롬프트, 세 해법을 한 장면에 겹쳐놓았다. 같은 공통 태스크—"사내 개발 문서 요약/Q&A"—를 4장에서는 from scratch 미니 GPT로, 6장에서는 Llama 3 8B QLoRA로, 7장에서는 API 호출로, 8장에서는 RAG로 풀었다. 네 번의 손 실습이 끝난 지금, 이 네 답이 서로 다른 질문에 대한 답이 아니라 **같은 질문에 대한 서로 다른 답**이었음을 우리는 눈으로 확인했다.

그래서 어떻게 고르지?

이 한 줄이 이 장의 열쇠다. 내일 팀장이 "슬랙 타래 10만 건을 검색해서 답해주는 사내 봇 하나 만들어줘"라고 말한다고 해보자. 4장의 미니 GPT를 꺼내야 할까, 6장의 QLoRA를 해야 할까, 7장의 API로 3일 만에 끝낼까, 8장의 RAG로 갈까. 네 해법이 머릿속에 있는 지금도 답이 즉시 나오지 않는다. 오히려 선택지가 많아진 만큼 결정이 더 난감해졌을 수도 있다.

그러니 이 장에서는 책에 흩어진 조각들을 한 장의 역인덱스로 묶어보자. 결정 플로차트 하나, 3축 트레이드오프 표 하나, 거버넌스 도구 매트릭스 하나. 그리고 한국어 프로덕션에서 실제로 넘어진 세 사례. 이 네 도구를 책상 위에 꺼내놓으면, 새로운 LLM 과제 앞에서 "어디서부터 건드려야 하지?" 하는 첫 1분이 한결 덜 찜찜해진다.

## 9.1 결정 플로차트 — 네 해법은 순서대로 꺼낸다

먼저 한 가지를 분명히 해두자. 네 해법은 **대등한 선택지**가 아니다. 비용·위험·전환 비용 측면에서 분명한 서열이 있고, 그 서열을 뒤집는 순간 프로젝트가 흔들린다. 8장에서 이미 한번 말했지만, 이 장 머리에서 한 번 더 박아두는 편이 낫다. **첫 직감이 "파인튜닝" 또는 "자체 모델"이면 red flag다.** 반대로 첫 직감이 "프롬프트로 일단 돌려보자"면 그 사람은 실무 감각이 있다. 커뮤니티 합의도 정확히 같은 방향이다.

> "If your first instinct is 'let's fine-tune,' that's a red flag; if your first instinct is 'let's see if we can get there with prompting and RAG first,' that signals practical judgment." [커뮤니티]

이 감각을 그림으로 그리면 아래와 같은 플로차트가 된다. 왼쪽이 가볍고 오른쪽이 무겁다. 왼쪽에서 시작해서, **지금 단계로 해결이 안 된다는 증거**가 쌓이면 한 칸씩 오른쪽으로 이동한다. 반대로 점프하지 않는다.

```
   [ 1. 프롬프트만 ]
          │
          │ 기준 미달?
          │ (예: 도메인 지식이 모델 밖에 있다)
          ▼
   [ 2. 프롬프트 + RAG ]
          │
          │ 기준 미달?
          │ (예: 검색해도 톤·구조가 안 맞는다)
          ▼
   [ 3. 프롬프트 + RAG + 파인튜닝 ]
          │
          │ 기준 미달?
          │ (예: 규제·레이턴시·데이터 주권으로 외부 API 불가
          │     + 오픈 모델 FT로도 품질 부족)
          ▼
   [ 4. 자체 모델 ]
     (드물다. 대부분 조직에는 불필요)
```

각 분기의 **이동 트리거**를 한 줄로 정리해두자. 말로 외우지 말고 이 표를 눈으로 익히는 편이 낫다.

| 단계 | 언제 머무는가 | 오른쪽으로 넘어가는 트리거 |
|-----|-------------|----------------------|
| 1. 프롬프트만 | 공개 지식 안에서 답이 나오는 태스크. 톤·구조가 프롬프트로 맞춰짐 | 답변에 최신·내부 문서가 필요 / 환각이 잦다 / 출처 인용이 요구된다 |
| 2. +RAG | 도메인 지식이 외부 문서로 정리 가능. 주기적 갱신 | 검색해도 톤이 어긋난다 / 출력 포맷이 엄격 / 반복 패턴을 모델이 내재화해야 한다 |
| 3. +파인튜닝 | 톤·구조·스키마를 모델에 박아야 함. RAG로 지식, LoRA로 스타일 | 규제로 외부 API 사용 불가 / 레이턴시·단가가 자체 추론보다 비싸다 / 오픈 모델 FT로도 요구 품질 미달 |
| 4. 자체 모델 | 도메인 언어가 공개 모델과 근본적으로 다름 (특수 법률·의료 코퍼스) | — (이 단계로 가는 조직은 이 책의 독자 중 극소수) |

플로차트 옆에 함께 걸어둬야 할 것이 **"파인튜닝 직행 체크리스트"**다. 팀에서 "파인튜닝하자"는 말이 나올 때, 그 말을 멈추기 전에 다음 다섯 질문을 먼저 던져보자. 하나라도 "아니오"가 나오면 한 칸 왼쪽으로 돌아가는 편이 낫다.

```
┌─ red flag 체크리스트 ────────────────────┐
│                                           │
│ □ 같은 태스크를 프롬프트만으로 돌려봤는가? │
│ □ RAG로 최소 파이프라인을 돌려봤는가?      │
│ □ 골드셋 20~100건으로 평가해봤는가?        │
│ □ "데이터가 자주 갱신된다"가 답이 아닌가? │
│ □ 파인튜닝의 유지보수 비용(재학습·버전      │
│   관리·롤백)을 감당할 사람이 있는가?      │
│                                           │
│ 하나라도 "아니오" → 한 칸 왼쪽으로.        │
└──────────────────────────────────────────┘
```

특히 네 번째 질문에 주의하자. 데이터가 자주 갱신되는데 파인튜닝을 고르면, 모델이 배포되는 순간부터 지식이 낡기 시작한다. 6개월 뒤 당신은 재학습 스케줄러와 버전 매트릭스 위에 앉아 있을 것이다. **찜찜하고 번거로운 일이다.** 반면 RAG는 문서 저장소만 갱신하면 지식이 자동으로 새로 흐른다. 운영 관점에서 RAG가 1순위인 이유가 이것이다.

한 가지 오해를 풀어두자. "3번 = 파인튜닝 포함"이 "1·2번을 버린다"는 뜻이 아니다. 3번 단계에서도 여전히 시스템 프롬프트가 존재하고, RAG가 지식을 공급한다. 파인튜닝은 그 위에 **얇은 스타일 레이어**로 얹힌다. 이 누적 구조를 기억해두자. 오른쪽으로 갈수록 도구가 교체되는 게 아니라 **겹쳐진다**.

"4번 자체 모델"이 드물다고 했는데, 그럼에도 로컬·오픈소스 루트를 고집하는 팀이 자주 빠지는 함정 하나는 짚고 넘어가자. 피카부랩스 블로그가 네 가지 함정을 정리해두었다. 문서 파싱—표·차트가 기본 라이브러리에 무시되고 페이지 단편화로 맥락이 소실된다. 기술 스택 선택 피로—LangChain vs LlamaIndex, 임베딩 모델, 벡터 DB 버전 호환성이 매주 흔들린다. 벡터 DB 운영—연구자가 DBA가 된다. 그리고 마지막이 가장 조용히 뼈아프다.

> "오픈소스를 활용하니 비용은 안 들겠지?" [web:피카부랩스]

공짜가 아니다. GPU 하드웨어, 유지보수 인력의 기회비용, 버전 불호환으로 날아가는 이틀—이게 전부 계산에 들어와야 한다. 3번·4번 단계로 이동할 때는 이 숨은 비용을 **표에 적어놓고** 결정하는 편이 낫다. 결정 회의에서 "일단 오픈소스로 가보자"는 말이 나오면, 그 옆에 "유지보수할 사람은 누구이며 언제까지?"를 함께 기록해두자.

## 9.2 비용·지연·품질의 세 축 — GPT-4급 API vs 로컬 Llama 8B

플로차트에서 "기준 미달"이라는 단어를 여러 번 썼다. 이 기준을 결정하는 세 축이 비용(cost)·지연(latency)·품질(quality)이다. 이 셋은 서로 교환 가능한 자원이고, 한 축을 누르면 다른 축이 튀어 오른다. 실무 의사결정의 거의 모든 줄다리기가 이 세 축 위에서 벌어진다.

구체적인 숫자감 없이 말로만 "트레이드오프가 있다"고 하면 감이 오지 않는다. 그러니 **가상의 사내 문서 Q&A 서비스**를 하나 상정하고 두 주요 선택지—GPT-4급 API vs 로컬 Llama 3 8B—의 손익을 거칠게 계산해보자.

가정은 이렇다. 월 요청 100만 건, 평균 입력 1,500토큰(한국어 문서 컨텍스트 포함), 평균 출력 300토큰. 주간 업무시간 피크 기준 QPS 10 내외. 응답 p50 2초, p95 5초를 서비스 수용 기준으로 본다.

| 항목 | GPT-4 계열 API | 로컬 Llama 3 8B (자체 GPU) |
|-----|---------------|---------------------------|
| 월 비용 (추론) | 입력 1.5M × $2.5 + 출력 0.3M × $10 = **$6,750** | GPU 1~2대 감가상각 + 전력 ≈ **$1,500~2,500** |
| 운영 비용 | 거의 0 (벤더 관리) | 엔지니어 0.5~1 FTE (서빙·모니터링·롤백) |
| p50 지연 | 800ms ~ 1.5s (네트워크 포함) | 300~800ms (로컬, vLLM 기준) |
| p95 지연 | 2~4s (벤더 혼잡 영향) | 1~2s (자체 제어 가능) |
| 품질 (한국어 Q&A 체감) | 높음 (현재 최상위 모델) | 중상 (도메인 FT/RAG 필요) |
| 콜드 스타트·배포 | 즉시 | 수일~수주 (인프라·서빙 스택) |
| 규제·데이터 주권 | 벤더 의존 | 내부 네트워크 |

숫자는 대략적이고 벤더 요금·모델 세대에 따라 두 배쯤은 쉽게 흔들린다. 그래도 방향은 분명하다. **추론 단가만 보면 로컬이 싸다. 운영 총비용까지 넣으면 역전될 수 있다.** 0.5 FTE 엔지니어의 연봉·기회비용을 월 $5,000으로 잡으면 위 표의 로컬 쪽에 $5,000이 얹힌다. 그러면 GPT-4급 API가 오히려 싼 경우도 생긴다.

토스증권의 기록을 한 번 곁들여보자. Llama 3 8B를 Transformers 라이브러리로 단순 로딩해서 추론시키니 **단순 질문에도 수십 초**가 걸렸다고 한다.

> "많은 정보를 알아야 됩니다." (ML 엔지니어가 vLLM, Triton, Docker, Kubernetes를 다 알아야 한다는 뜻) [techblog:토스]
> "이런 허들 때문에 여러 모델을 서빙해서 A/B 테스트로 비교하기 어렵습니다." [techblog:토스]

"로컬 모델이 싸다"는 문장 안에는 이 허들이 감춰져 있다. vLLM·KServe·모니터링·배치 튜닝까지 다 설계했을 때 비로소 위 표의 "p50 300~800ms"가 나온다. 맨몸으로 Transformers만 돌려서는 표의 숫자는 꿈도 꾸지 말자. 그러니 **"추론 단가 비교표"만 들고 경영진 설득에 들어가는 건 난감한 일이다.** TCO(총소유비용) 관점의 계산을 한 번 더 거치는 편이 낫다.

세 축을 어떻게 함께 묶을지에 대한 현장 규칙을 하나 정리해두자.

- **MVP 단계 (0~3개월):** API. 품질 최상위 + 운영 부담 0. 월 비용이 올라도 가설 검증이 우선.
- **제품 검증 이후 (3~12개월):** 토큰 사용량과 품질 요구를 측정. 월 비용이 "엔지니어 0.5 FTE × 12개월"을 넘으면 로컬 전환 검토.
- **규제·데이터 주권 요구:** 처음부터 로컬 또는 Azure OpenAI·AWS Bedrock 같은 프라이빗 게이트웨이.
- **하이브리드:** 쉬운 질문은 로컬 모델, 어려운 질문만 API로 에스컬레이션. LY Corp가 제안하는 카스케이드 필터링이 이 아이디어다.

> "카스케이드 필터링으로 최대 98%까지 추론 비용을 절감." [techblog:LY Corp]

숫자는 과장 없이 받아 읽자. 저 98%는 FrugalGPT 인용이며, 모든 태스크에서 재현되는 수치는 아니다. 그래도 "싼 모델이 먼저 걸러낸다"는 구조는 비용 관점에서 유효한 패턴이다. 기억해두자.

SK하이닉스 MSR의 RAG 플랫폼 벤치마크 기록도 3축 감각에 숫자 하나를 보태준다. AWS 기술블로그에 공개된 글에 따르면, RAG Inference는 순수 LLM 대비 **TTFT(Time-To-First-Token)가 약 30% 증가**했다. 검색 단계가 끼어들기 때문이다. 더 인상적인 수치는 벡터 인덱스가 메모리에 못 들어갈 때 일어나는 일이다. **0.088초가 122.06초로 늘어난다.** 약 1,300배. 30GB 인덱스 기준, 스토리지로부터 인덱스를 읽어야 하는 상황에서다. 이 수치를 처음 보면 끔찍한 일이다.

> "벡터 데이터 인덱스를 위한 메모리 용량이 부족한 경우 스토리지로부터 인덱스를 읽어서 검색 시간이 급격히 증가." [techblog:AWS-SK하이닉스]

이 사례가 말하는 바는 단순하다. 3축의 숫자는 **설정을 하나 틀면 자릿수 단위로 뒤집힌다.** 인프라를 스케일업만 믿고 가면 비용적으로 지속 불가능하고, 메모리 산정을 실측 없이 가면 지연 축이 통째로 무너진다. "일단 RAG부터"가 만능이 아니라는 반례이자, 9.3의 평가 하네스와 9.4의 관측성을 이 장이 계속 강조하는 이유이기도 하다.

## 9.3 평가 하네스 — 프롬프트 바꾸면 성능이 어디로 가는가

비용·지연은 숫자로 뜬다. 품질은 그렇지 않다. "GPT-4가 더 낫다"고 말하려면 무엇을 기준으로 더 낫다는 건지, 그 차이가 유의미한지, 프롬프트를 한 줄 바꾸면 이 우위가 유지되는지를 **재현 가능하게** 측정해야 한다. 이 재현 가능성의 기반이 평가 하네스(evaluation harness)다.

이 주제에서 가장 단단한 한국어 1차 자료가 LY Corp의 글이다. 짧은 한 문장을 박아두자.

> "출력을 JSON/CSV/XML로 써 달라'와 같이 출력 형식 관련해 단 한 줄만 수정해도 전체 예측 중 최소 10% 이상이 다른 답으로 바뀌었습니다." [techblog:LY Corp]

**프롬프트 한 줄이 10%의 답을 뒤집는다.** 이 문장이 의미하는 바를 천천히 되새기자. 프롬프트 A에서 프롬프트 B로 바꿨는데 사용자 체감 품질이 올라간 것 같다. 정말 올라갔을까? 10%가 이동했는데, 그중 8%가 좋은 방향이고 2%가 나쁜 방향이었을 수도 있고, 반대로 2%가 좋은 방향, 8%가 나쁜 방향이었을 수도 있다. 골드셋 없는 상태에서는 구분이 불가능하다. 그리고 이 판단을 머릿속 감각으로 "아마 좋아졌다"고 넘어가는 것이 프로덕션 LLM 프로젝트가 은근슬쩍 무너지는 가장 흔한 경로다.

평가 하네스의 최소 구성요소는 다음과 같다.

**1. 골드셋(Gold Set).** 도메인 전문가가 검증한 **질문–정답 쌍** 20~100건. 이 책의 공통 태스크라면 "사내 개발 문서에 대한 실제 질문 + 기대 답변"을 수집한다. LLM이 생성한 실버셋(silver set)으로는 부족하다. 실제 사용자 로그에서 샘플링한 뒤 사람이 정답을 확정하는 편이 낫다. smallcherry 패널 토크가 이 구분을 정확히 짚는다.

> "도메인 전문가가 검증한 Golden Dataset 구축 (LLM 생성 Silver dataset 이상)." [velog:smallcherry]

**2. 자동 지표.** 골드셋에 대해 프롬프트/모델을 바꾸면 자동으로 점수가 나오는 파이프라인.

| 지표 | 언제 쓰는가 | 한계 |
|-----|-----------|-----|
| ROUGE | 요약·짧은 생성. n-gram 겹침 | 단어 표면만 봄. 한국어 교착어에 약함 |
| BLEU | 번역 계열 | 자연어 응답의 품질 판단엔 부적합 |
| BERTScore | 의미 유사도 측정 | 임베딩 모델에 의존. 한국어는 `bge-m3` 같은 것 써야 체감 |
| LLM-as-a-judge | 개방형 답변 품질 | 판정 모델의 편향·비용. 참조 모델과 답변 모델이 같으면 자기편향 |
| 태스크 특화 지표 | RAG의 Hit@k, 분류의 F1, JSON 출력의 스키마 일치율 | 구체적이지만 태스크 밖 일반화 안 됨 |

어떤 한 지표에도 과잉 의존하지 않는 편이 낫다. 보통 **두세 지표를 섞어** 쓴다. 예컨대 사내 문서 Q&A라면 (a) Hit@5로 검색 단계 점검, (b) BERTScore로 생성 답변의 의미 유사도, (c) LLM-as-judge로 "도움이 됐는가"를 묻는 3층 구성이 무난하다.

**3. Human-in-the-loop 샘플링.** 자동 지표만으론 놓치는 결이 있다. 매주 혹은 매 릴리스마다 10~20건을 사람이 직접 읽고 5점 척도로 채점한다. 자동 지표와 사람 점수의 상관관계가 약해지면 지표 자체를 의심해야 한다.

**4. 회귀 방지 CI.** 프롬프트·모델·RAG 청킹 전략을 바꿀 때마다 위 평가가 자동으로 돈다. 전체 점수가 하락하거나 특정 카테고리에서 2σ 이상 빠지면 PR이 막히게 한다. Promptfoo가 이 용도에 적합하다.

7장에서 우리는 `temperature=0`인데도 답이 매번 조금씩 다른 현상을 만났다. 그 "재현성의 부재"가 이 평가 하네스와 정확히 맞물린다. 같은 골드셋·같은 프롬프트·같은 모델로 돌려도 점수가 흔들린다면, 그건 당신 모델이 나빠진 게 아니라 GPU 부동소수점 비결정성이나 배치 효과 때문일 수 있다. 그러니 평가 실행을 **복수 회(보통 3회) 돌려 평균·분산을 함께 기록**하는 편이 낫다. 한 번의 수치에 일희일비하지 말고, 분산이 줄어드는지도 같이 보자. 분산이 크다는 건 평가 자체의 잡음이 크다는 뜻이고, 이 잡음을 뚫고 유의미한 개선을 보려면 골드셋을 키우거나 반복 횟수를 늘려야 한다.

한 가지 더 붙여둘 감각이 있다. smallcherry 패널 토크 후기에 나오는 수치다. 한 사례에서는 문서 요약·목차·Q&A 페어를 **메타데이터로 추가**했더니 검색 정확도가 **+15%p** 올랐다. 키워드 + 벡터 하이브리드, Text-to-SQL 병행 같은 구성도 같이 거론된다.

> "모두 파싱해서 저장하는 건 상황에 따라서 비효율적일 수도 있음."
> "Retrieval한 Context에 상충된 정보가 존재하면 할루시네이션을 유발할 수밖에 없음." [velog:smallcherry]

즉 평가 하네스가 말해주는 것은 "모델을 바꿔야 하는지"뿐 아니라 **"청크·검색·리랭크 중 어디를 만져야 하는지"**다. 숫자가 없으면 이 중 어느 걸 만졌을 때 어느 쪽이 움직이는지 알 수 없고, 결국 프롬프트를 이리저리 흔들며 손감각으로 튜닝하는 난감한 상태로 돌아간다.

여기서 도구 하나를 덧붙여두자. 공개 벤치마크의 위치를 **대조군**으로 참조하고 싶다면 Stanford CRFM의 HELM이 유용하다. 16개 시나리오 × 7개 지표(accuracy, calibration, robustness, fairness, bias, toxicity, efficiency)로 공개 모델을 재현 가능하게 비교한다. 내 모델의 toxicity 점수가 공개 Llama·GPT 계열에 비해 어디쯤인지를 같은 조건으로 볼 수 있다. 다만 이것만으로 "안전하다"고 선언하지 말자. MT-Bench 논문 자체도 명시한다.

> "MT-bench 논문은 helpfulness를 강조하지만, safety는 주로 생략한다." [논문:Zheng2023 — 의역]

즉 MT-Bench 하나로 안전성을 증명하려 들면 안 된다. 도움이 되는지(helpfulness)와 해롭지 않은지(harmlessness)는 다른 질문이고, 다른 벤치마크가 필요하다. 이 이야기는 다음 절로 이어진다.

## 9.4 관측성·감사·거버넌스 — 별도 층으로 두는 이유

평가 하네스가 "내 모델이 얼마나 잘하는가"를 측정한다면, 거버넌스는 "내 모델이 얼마나 안 망가지는가"를 측정한다. 5.6(Observability-first)에서 살짝 건드렸지만, 프로덕션에 LLM을 띄우는 순간 이 층이 사실상 가장 많은 시간을 잡아먹는다. 7·8장에서 우리는 Langfuse·LangSmith 연동만 훑었는데, 그 윗층에 **감사(audit)와 red-teaming**이 붙는다.

먼저 왜 별도 층이 필요한가. "시스템 프롬프트에 '욕설 금지, 개인정보 금지'라고 써두면 되는 것 아니냐"는 반문이 나올 법하다. LY Corp의 글이 이 질문에 가장 또렷하게 답한다. 세 가지 약점이 있다.

1. **Position bias.** 정보가 프롬프트 중간으로 갈수록 모델의 주의가 U자 곡선으로 떨어진다. 안전 규칙을 길게 넣으면 중간의 규칙은 지켜지지 않는다.
2. **False positive 증폭.** 유해 쿼리 거절률만 오르는 게 아니라 무해 쿼리 거절률도 같이 오른다. 정상 질문에 "죄송합니다, 답변드릴 수 없습니다"를 반복하는 챗봇이 생기는 전형적 원인.
3. **Micro-perturbation 민감도.** 앞 절에서 인용한 그 10%. 출력 형식 한 줄 수정에 응답이 뒤집힌다. 안전 규칙을 프롬프트에 묶으면 평범한 UI 변경이 안전성 회귀를 유발할 수 있다.

이 세 가지 이유 때문에 **입력 필터 – 모델 – 출력 감사 – red-team 주기 점검**을 **별도 층**으로 두는 쪽이 현재 업계 합의다. 각 층에 투입할 도구를 한 매트릭스로 정리해두자. 위협 카테고리가 가로축, 도구가 세로축이다.

```
위협 \ 도구      │ Prompt       │ Llama     │ Rebuff    │ Kanana    │ garak /  │ PyRIT /
                 │ Shields      │ Guard 3/4 │           │ Safeguard │ Promptfoo│ NeMo GR
─────────────────┼──────────────┼───────────┼───────────┼───────────┼──────────┼──────────
Prompt injection │      ●       │     ●     │     ●     │     ●     │    ●     │    ●
Jailbreak        │      ●       │     ●     │     -     │     ●     │    ●     │    ●
PII / 개인정보   │      -       │     ●     │     -     │     ●     │    -     │    ●
Toxicity / 혐오  │      -       │     ●     │     -     │     ●     │    ●     │    ●
간접 인젝션      │      ●       │     -     │     -     │     ●     │    ●     │    ●
다회차 공격      │      -       │     -     │     -     │     -     │    -     │    ● (PyRIT)
출력 포맷 이탈   │      -       │     -     │     -     │     -     │    -     │    ● (NeMo)
```

도구 하나하나를 "언제 꺼내는가" 한 줄로 붙여두자. 외우려 들지 말자. 필요할 때 이 페이지를 다시 펼치면 된다.

- **Azure Prompt Shields** — Azure 위에서 이미 돌고 있고, 사용자 입력 + RAG 소스 양쪽의 인젝션을 한 API로 막고 싶을 때. 한국어는 공식 훈련 대상이 아니니 별도 검증 필요.
- **Meta Llama Guard 3 / 4** — 온프렘·프라이빗에서 입출력을 자체 모델로 분류하고 MLCommons 14개 카테고리로 감사 로그를 남기고 싶을 때. Llama 3.1 레퍼런스에 기본 포함.
- **Rebuff** — 가볍게 Python/JS 앱에 인젝션 탐지만 얹고 싶을 때. 단 2025-05 이후 아카이브(read-only)라 **프로덕션 채택은 신중히**, 프로토타입·학습용 권장.
- **카카오 Safeguard by Kanana** — 한국어 서비스에서 영어권 도구의 한국어 성능이 의심스러울 때. `Kanana-Safeguard`(혐오·범죄·자해), `Siren`(성인인증·전문조언·PII·IP), `Prompt`(인젝션·리킹)로 3종 분담. HuggingFace에 Apache 2.0으로 공개.
- **NVIDIA garak** — 배포 전에 "알려진 공격 카테고리에 얼마나 뚫리는가"를 재현 가능한 수치로 뽑고 싶을 때. CLI로 돌려 CI에 붙이기 쉽다.
- **Microsoft PyRIT** — 단발 jailbreak가 아니라 **대화가 흘러가며 무너지는 다회차 공격**까지 자동화하고 싶을 때. Microsoft AI Red Team이 사내에서 실제 쓰는 프레임워크.
- **Promptfoo Red Team** — 선언형 YAML로 red-team 스위트를 관리하고 PR마다 "이번 프롬프트 변경으로 새로 뚫린 공격이 있는가"를 CI가 보게 하고 싶을 때. OWASP LLM Top 10 프리셋 지원.
- **NVIDIA NeMo Guardrails** — 챗봇이 **여러 턴에 걸쳐 주제를 이탈하거나 정책을 우회하는 것**까지 막고 싶고, 가드레일 정책을 Colang DSL로 코드화하고 싶을 때.

한국 독자를 위한 경로 제안은 단순하다. **한국어 입력/출력 분류는 Kanana 시리즈를 1차 후보로**, **red-team 주기 점검은 garak(스캐너) + Promptfoo(CI)**, **다회차 공격과 대화 흐름 제어가 필요하면 PyRIT / NeMo Guardrails**. 영어권 서비스라면 Prompt Shields 또는 Llama Guard를 1차로.

컴플라이언스 관점에서는 KISA의 「인공지능(AI) 보안 안내서」를 회사 법무팀이 참조할 공식 문서로 찜해두자. 다섯 축—데이터 검증·정제, 모델 모니터링·로그관리, 안전한 배포·접근 제어, Adversarial Training, 응답 필터링·검증—으로 국내 규제 맥락을 정리해놓았다. 책이 LLM 엔지니어링 관점의 책이긴 하지만, 규제가 붙는 도메인(금융·의료·공공)에서는 엔지니어링 결정의 제약 조건이 바뀐다는 사실을 기억해두자.

마지막으로 카카오페이증권의 한 문장.

> "증권사에서 AI를 도입하며 가장 어려웠던 측면은 아무래도 보안이었습니다." [techblog:카카오페이]

AWS Bedrock Guardrails로 입출력 민감정보 필터링과 민감 문서 "exclude" 라벨링을 얹었다는 기록이다. 증권이라는 규제 도메인에서 "기술 < 보안"이 실제 체감된 현장의 증언이다. LLM 프로젝트의 어려움이 LLM 자체가 아니라 LLM을 **감싸는 층**에 있다는 사실이, 이 장 전체를 관통하는 메시지이기도 하다.

## 9.5 한국어 실패담 세 건 — 나도 저렇게 될 수 있구나

이제 위에서 쌓은 플로차트·3축·거버넌스 매트릭스를 **실제로 넘어진 세 사례** 위에 겹쳐본다. 세 사례 모두 공개된 한국어 포스트모템·회고에서 가져왔다. 출처와 링크를 본문에 병기해두었으니 책을 덮은 뒤 원문을 꼭 읽어보길 권한다. 현장의 목소리가 이 책보다 훨씬 구체적이다.

### 사례 1. "요즘 AI가 대세니까 한번 해보자" — 파인튜닝으로 직행했다가 돌아온 소규모 팀

> "솔직히 중간중간 후회가 들 정도로 쉽지 않은 프로젝트였다."
> "요즘 AI가 대세니까 한번 해보자."
> — Soyeon, velog, 「[2025 회고] 소규모 팀에서 LLM 서비스 만들기」

**무엇이 틀어졌나.** 주니어 3~4인 팀, AI/LLM 경험 0. 프론트엔드 1인이 풀스택 책임을 졌다. 초반 버전은 단순 키워드 프롬프트 기반이었는데, 환각이 잦고 제품 톤과 어긋났다. 경영진은 "파인튜닝하면 품질 문제가 해결된다"고 믿었고, 조직은 "요즘 AI가 대세니까"라는 열의만으로 착수했다. 회사 데이터는 여러 시스템에 흩어져 있었고 ML 용도로 정제되지 않은 상태였다.

**뭐로 갈아탔나.** 파인튜닝 대신 RAG로 선회. 도메인 지식을 구조화된 레퍼런스로 정리하고, 메타데이터 기반 분류·유사도 검색으로 재설계했다. AI는 수단이라는 원칙을 뒤늦게 받아들였고, 비즈니스 임팩트를 먼저 검증한 뒤 규칙 기반 위에 AI를 전략적으로 얹는 구조로 바꿨다.

**교훈.** 이 사례는 9.1 플로차트의 가장 전형적인 반례다. 1·2단계(프롬프트·RAG)를 건너뛰고 3단계(파인튜닝)로 직행한 팀이 결국 2단계로 돌아온다. "**요즘 AI가 대세니까**"라는 말이 조직에서 들리면, 그건 실패가 예약된 신호로 보는 편이 낫다. 플로차트 옆의 red flag 체크리스트가 딱 이 순간을 위한 것이다.

### 사례 2. "retrieve 결과가 잘려서 나와요" — RAG를 띄웠는데 평가가 없었다

> "retrieve 시, 결과의 content가 완전한 Json 형태가 아니라 일부가 잘려서 나오는 문제가 발생했어요."
> "단순히 최근 사기 거래만을 포함하면 현재 거래 패턴과 다를 수도 있기 때문에 잘못된 판단 결과를 내릴 가능성이 높았던 거예요."
> "프롬프트가 길수록 응답 오류가 늘어나고, 출력 포맷도 안정적으로 유지되지 않는다는 점을 확인."
> — HyunwooKim, 당근 테크블로그, 「당근페이 AI Powered FDS로 가는 여정」

**무엇이 틀어졌나.** 당근페이 FDS(이상거래탐지)에 LLM을 얹는 프로젝트. RAG 적용 시 retrieve 결과의 JSON이 **중간에서 잘려** 반환됐다. 최근 사기 데이터만으로 평가했더니 정탐률·재현률이 **시간이 지날수록 하락**했다. 프롬프트가 길어질수록 응답 오류는 늘었고 출력 포맷은 불안정해졌다. 세 가지 증상 모두 단독으로도 문제였고, 겹치니 운영이 어려웠다.

**뭐로 갈아탔나.** 기본 청킹 전략(자동 분할) 대신 **사기 케이스별로 파일 1개 = 청크 1개**로 분리(`ChunkingStrategy=NONE`). `RetrieveAndGenerate`의 단일 패스 호출 대신 `Retrieve` + `ConverseAPI` 파이프라인으로 분리해, retrieve 결과를 **사람이 설계한 기준으로 조립**한 뒤 프롬프트에 투입했다. 평가도 "최근 데이터"가 아닌 **의미 유사도 기반**으로 재설계.

**교훈.** 이 사례는 9.3 평가 하네스와 9.4 관측성을 동시에 관통한다. RAG를 띄운다고 문제가 풀리는 게 아니다. 청킹 전략·평가 설계·프롬프트 길이가 전부 결정점이고, 이 결정을 **관측 가능하게** 하지 않으면 "최근 데이터만으로 평가해서 시간이 지나니 성능이 떨어진" 같은 조용한 붕괴가 반드시 찾아온다. 금융 도메인에서는 "구조화된 제어 > 엔드투엔드 자동화"가 본능이어야 한다는 사실도 이 기록이 확인해준다.

### 사례 3. "계정과 API키 발급 자체가 병목이 되었어요" — 프롬프트만으로도 충분했는데 다른 데서 막혔다

> "계정과 API키 발급 자체가 병목이 되었어요. 발급한 API 키가 많아지면서 어느 팀이 어떤 서비스를 위해 계정과 키를 사용하는지 관리하기 어려워졌고 보안에 대한 우려도 커졌어요."
> "어떤 계정은 여유가 남아있는데, 다른 계정은 이미 한도에 도달해 호출이 실패하는 상황이 발생."
> — Tommy Park, 당근 테크블로그, 「당근의 GenAI 플랫폼」

**무엇이 틀어졌나.** 기술 실패담은 아니다. 오히려 기술은 잘 됐다. 팀마다 OpenAI·Anthropic·Google 계정을 따로 발급해 프롬프트+API 조합으로 기능을 빠르게 내놓았다. 그런데 그 속도가 너무 빨라 발급된 API 키를 관리할 수 없게 됐다. 어느 팀이 어떤 서비스에서 쓰는지 추적이 안 됐고, 한 계정은 쿼터가 남는데 다른 계정은 이미 한도에 도달해 호출이 실패하는 상황이 반복됐다. 기술이 아니라 **플랫폼 층의 부재**가 병목이었다.

**뭐로 갈아탔나.** 단일 LLM Router(게이트웨이)로 재설계. 계정·키 관리, 쿼터 분산, 감사 로그, 비용 태깅을 한 층에 모았다. 개별 팀은 라우터만 호출하고, 어떤 provider가 뒤에서 응답하는지 모른다. 7장에서 다룬 Spring AI의 "의존성 한 줄로 provider 스왑" 아이디어를 조직 단위로 확장한 구조다.

**교훈.** 이 사례는 플로차트 1단계(프롬프트만)가 **기술적으로는 충분했던** 드문 사례다. 제품은 만들어졌고, 파인튜닝도 필요 없었다. 그런데 조직이 커지자 다른 데서 막혔다. LLM 프로젝트의 난이도가 모델 학습이 아니라 **운영 층위**에 있다는 사실이 이 기록에서 뼈아프게 드러난다. 9.4에서 거버넌스 도구 매트릭스를 그렸던 이유도 바로 이것이다. 프롬프트 한 줄로 만든 기능도, 열 팀이 동시에 쓰기 시작하면 전혀 다른 층의 문제를 만든다.

---

세 사례를 나란히 놓아보면 하나의 공통점이 보인다. **실패의 원인이 "모델"이 아니었다.** 사례 1은 조직 의사결정, 사례 2는 평가·청킹 전략, 사례 3은 플랫폼 거버넌스였다. 이 책은 모델과 모델 주변의 작은 반경만 다뤘지만, 실제 프로젝트에서는 그 반경 바깥의 결정이 프로젝트의 성패를 더 자주 좌우한다. 이 사실은 부끄러워할 것이 아니라 오히려 담담히 받아들이는 편이 낫다. 책 한 권으로 다 해결되지 않는다는 사실이 이 책의 약속이기도 하다.

## 9.6 "일단 일주일 써보고 고르라" — 머리로 고르지 말고 손으로 고르라

지금까지 플로차트, 3축 표, 거버넌스 매트릭스, 세 실패담을 얹었다. 여기까지 와도 한 가지가 남는다. **선택의 근거를 어떻게 만들 것인가.** 표는 일반론이고, 우리 팀의 우리 문제에는 여전히 고유한 윤곽이 있다. 이 윤곽을 잡는 가장 빠른 방법을 마지막에 한 번 더 소환하자.

> "Start simple. Ollama + Llama 3.1 8B. If it does not do what you need after a week of real use, then you understand the gap well enough to make a better choice." [커뮤니티:r/LocalLLaMA]

한 주만 써보자. 진짜 태스크, 진짜 사용자 요청, 진짜 데이터로. 한 주 뒤 당신은 더 이상 "프롬프트로 될까, 파인튜닝이 필요할까"를 머리로 굴리지 않는다. 손에 남은 실패 로그가 다음 단계를 가리킨다. RAG가 필요하다면 retrieve 결과 품질이 모자라서 그렇게 느낀 것이고, 파인튜닝이 필요하다면 프롬프트 엔지니어링의 한계를 열 번쯤 치고 돌아온 경험이 그렇게 말하는 것이다.

머리로 고르지 말고 손으로 고르라. 이 한 줄이 책 전체의 실습 철학이기도 하고, 9장 결정 프레임워크의 실행 규칙이기도 하다. 표와 플로차트는 **손의 기록을 해석하는 자**이지, 손을 대신하는 도구가 아니다.

한 가지만 덧붙이자. 손으로 고른다고 해서 **측정 없이** 고른다는 뜻은 아니다. 한 주짜리 시도에도 최소한의 골드셋 20건과 자동 지표 한두 개는 얹어두는 편이 낫다. 그래야 일주일 뒤 "느낌상 좋아진 것 같다"가 아니라 "Hit@5가 0.62에서 0.78로 올라갔다"라는 문장을 쥐고 다음 결정에 들어갈 수 있다. 9.3의 평가 하네스가 일주일짜리 탐색에도 그대로 필요한 이유가 이것이다.

## 마무리 — 지도 위에 서 있다는 감각

되짚어보자. 이 장에서 같이 얻은 것은 네 가지다.

- **결정 플로차트**: 프롬프트만 → +RAG → +파인튜닝 → 자체 모델. 왼쪽에서 시작해 증거가 쌓이면 한 칸씩 오른쪽으로. 뛰어넘지 않는다. 각 분기의 트리거 표와 "파인튜닝 직행 체크리스트"가 옆에 붙는다.
- **3축 트레이드오프**: 비용·지연·품질. GPT-4급 API vs 로컬 Llama 8B의 손익을 TCO 관점으로 본다. "추론 단가만" 비교하는 일은 초난감하다.
- **거버넌스 매트릭스**: 입력 필터(Prompt Shields, Llama Guard, Rebuff, Kanana Safeguard) × 출력 감사·red-team(garak, PyRIT, Promptfoo, NeMo Guardrails). 한국어 서비스는 Kanana를 1차 후보, red-team 주기 점검은 garak + Promptfoo, 다회차 공격은 PyRIT / NeMo.
- **한국어 실패담 3건**: Soyeon(파인튜닝 직행 후 RAG로 선회) / 당근페이 FDS(RAG의 청킹·평가 재설계) / 당근 GenAI 플랫폼(프롬프트는 됐지만 플랫폼 층이 병목).

네 도구를 책상 위에 올려두면, 새로운 LLM 과제가 들어올 때 첫 1분의 찜찜함이 확실히 줄어든다. 물론 이 네 도구가 결정을 **대신** 해주진 않는다. 손으로 한 주 돌려보고, 손에 남은 증거로 다음 칸을 정하자. 이 책의 두 번째 약속—"머리로 고르지 말고 손으로 고르라"—이 9장의 결론이기도 하다.

이제 책의 지도가 거의 다 그려졌다. 2~3장의 원리, 4~6장의 만드는 궤적, 7~8장의 쓰는 궤적, 그리고 이 장의 결정 프레임워크. 남은 것은 하나다. **이 책을 덮고 나서 어디로 갈 것인가.** 에이전트, 멀티모달, 심화 파인튜닝, 추론 최적화, 긴 컨텍스트, 평가·안전—이 여섯 분기의 지도가 10장에서 기다리고 있다. 같이 넘어가 보자. 이번엔 지도 위에 서 있는 감각으로.
