# 2장. 토큰과 임베딩 — 블랙박스의 첫 층

문자열 하나를 상상해보자. 예컨대 `"안녕하세요, 반갑습니다."` 같은 평범한 한국어 인사 한 줄이다. 우리가 API로 이 문자열을 LLM에 보내면, 모델은 이 글자들을 "읽고" 답을 만든다. 그런데 이 "읽는다"가 도대체 무슨 뜻일까? 신경망 내부에는 문자열 타입 같은 것이 없다. 있는 거라곤 거대한 행렬과 부동소수점 수뿐이다. 그렇다면 모델은 사람의 말을 어디서 어떻게 숫자로 바꿔먹는 것일까?

이 질문이 바로 이 장의 출발점이다. LLM이라는 블랙박스에서 가장 바깥, 가장 만져볼 수 있는 첫 번째 층이 토큰화(tokenization)와 임베딩(embedding)이다. 이 두 단계는 실체가 있다. 코드로 돌려볼 수 있고, 출력된 숫자가 눈에 바로 들어온다. 어텐션이나 트랜스포머처럼 그림을 100번 봐도 손에 잡히지 않는 것들과 다르다. 그러니 블랙박스를 여는 일은 여기서 시작하는 편이 낫다. 같이 첫 층부터 한 꺼풀씩 벗겨보자.

## 2.1 단어를 어떻게 쪼갤 것인가

컴퓨터가 문자열을 다룰 때 첫 번째 결정은 "이 문자열을 어느 단위로 쪼갤 것인가"다. 선택지는 대략 세 가지다. 문자 단위, 단어 단위, 그리고 서브워드(subword) 단위.

**문자 단위**는 단순하다. `"hello"`를 `['h', 'e', 'l', 'l', 'o']`로 자른다. 장점은 분명하다. 어떤 신조어·오탈자·외래어가 들어와도 절대 모르는 글자가 없다. 한국어로 치면 유니코드 한글 11,172자만 알면 된다. 단점도 분명하다. 토큰 하나가 너무 작은 의미만 들고 있어서, 모델이 "사과"라는 단어의 의미를 잡으려면 `사` 토큰과 `과` 토큰을 조합해 학습해야 한다. 시퀀스가 길어지고 학습이 느려진다.

**단어 단위**는 반대 쪽 극단이다. `"I love apples"`를 `['I', 'love', 'apples']`로 자른다. 의미 단위에 가깝고 시퀀스도 짧다. 그런데 번거롭다. 학습 때 보지 못한 단어(`apple` → `apples`의 변형, 오탈자, 신조어)가 나오면 전부 `<UNK>`(Unknown) 토큰으로 퉁쳐야 한다. 영어는 그나마 공백으로 자르면 되는데, 한국어는 교착어라 "사과를", "사과가", "사과와"가 전부 다른 단어다. 어휘 수가 폭발한다.

**서브워드 단위**가 이 두 극단 사이의 절충이다. `"birthplace"`를 `['birth', 'place']`로 자르고, `"unhappiness"`를 `['un', 'happy', 'ness']`로 자른다. 자주 같이 등장하는 문자 조각은 한 덩어리로 묶어두고, 드문 조합은 더 잘게 쪼갠다. 이러면 (a) 모르는 단어가 와도 서브워드 조각으로 표현할 수 있어 `<UNK>`가 거의 사라지고, (b) `"birth"`라는 의미 뿌리는 `"birthday"`, `"birthplace"`, `"rebirth"`가 공유하므로 학습이 효율적이다. 현대 LLM은 예외 없이 서브워드 토큰화를 쓴다.

서브워드 계열의 대표 알고리즘은 세 갈래로 이어지는 계보를 이룬다. 먼저 **BPE(Byte Pair Encoding)**가 있다. 원래는 1994년에 제안된 데이터 압축 기법이었는데, Sennrich 등(2016)이 기계번역에 끌어다 쓴 뒤로 OpenAI GPT 시리즈가 이 위에서 자랐다. 동작 원리는 단순하다. 학습 말뭉치 전체에서 가장 자주 같이 등장하는 문자 쌍을 하나로 병합하고, 이걸 원하는 어휘 크기(vocab size)에 도달할 때까지 반복한다. `l`, `o` → `lo` → `low` → `lower` 식으로 덩어리가 커진다. 그 뒤 Google이 BERT에 얹은 것이 **WordPiece**다. BPE와 거의 같은 구조지만 병합 기준이 "빈도"가 아니라 "언어 모델 확률(likelihood) 기여도"다. BERT 토크나이저에서 `##ing` 같은 `##` 접두사를 본 적이 있다면 바로 그 친구다. `##`은 "이 조각은 단어의 중간 또는 끝에 붙는다"는 표시다. 2018년에는 Google이 **SentencePiece**를 내놓았다. 공백까지도 하나의 문자(`▁`)로 취급하여 "단어 경계"라는 개념 자체를 없앴다. 한국어·일본어·중국어처럼 띄어쓰기가 불규칙한 언어에 특히 잘 맞는다. Llama, T5, Gemma, 한국어 HyperCLOVA 등 최근 다국어 모델이 주로 이 계열이다.

세 알고리즘이 서로 크게 다르지 않다. 공통의 아이디어는 하나다. **자주 같이 나오는 조각은 한 토큰으로 묶고, 드문 조각은 작은 단위로 남겨둔다.** 이 단순한 원리가 GPT부터 Claude, Llama까지 모든 현대 LLM의 입구를 지킨다.

### BPE가 동작하는 방식을 손으로 따라가 보기

말로만 읽으면 찜찜하다. BPE가 실제로 어떻게 어휘를 만드는지 짧게 손으로 따라가 보자. 학습 말뭉치에 다음 네 단어가 있다고 해보자. 빈도는 괄호 안에 적었다.

```
low (5)   lower (2)   newest (6)   widest (3)
```

처음 어휘는 문자 수준이다. `{l, o, w, e, r, n, s, t, i, d}`. 각 단어 끝에 특수 토큰 `</w>`를 붙여 "단어 경계"를 표시하는 게 일반적이지만, 여기서는 간단히 생략한다.

- **1단계:** 가장 자주 같이 등장하는 문자 쌍을 찾는다. `newest`에 `e, s`가 6번, `lowest`·`widest`에도 등장. `(e, s)` 쌍이 9회로 최다. 병합해 `es`를 새 토큰으로 추가.
- **2단계:** 방금 만든 `es`를 포함해 다시 쌍의 빈도를 센다. `(es, t)`가 9회. 병합해 `est` 추가.
- **3단계:** `(l, o)`가 7회(`low`·`lower`). 병합해 `lo` 추가.
- **4단계:** `(lo, w)`가 7회. 병합해 `low` 추가.

이런 식으로 원하는 어휘 크기에 도달할 때까지 반복한다. 결과적으로 어휘에는 `l, o, w, e, r, ..., es, est, lo, low, ...`처럼 **문자와 서브워드가 뒤섞여** 들어간다. 추론 시에는 이 병합 순서를 그대로 다시 적용해 입력 문자열을 토큰으로 바꾼다. 병합 규칙 리스트가 토크나이저의 "상태"인 셈이다. GPT-4의 `cl100k_base`는 이 병합을 100,000번 넘게 수행한 결과물이다. 실전 어휘가 100,277개인 이유다.

이 단순한 알고리즘이 왜 그렇게 강력할까. `"lower"`가 학습 말뭉치에 없었어도 문제가 없기 때문이다. `low`와 `er`로 쪼개면 그만이다. `"rebirthing"` 같은 긴 합성어도 `re` + `birth` + `ing`으로 표현된다. **새 단어가 와도 작은 조각의 조합으로 표현할 수 있다**는 것, 이것이 서브워드의 진짜 위력이다.

## 2.2 손 실습 1 — `tiktoken`으로 같은 뜻을 두 언어로 쪼개보기

말로만 들으면 서브워드 감각이 오지 않는다. 눈으로 봐야 한다. 아래 스크립트는 OpenAI의 공식 토크나이저 라이브러리인 `tiktoken`을 불러와 영어와 한국어를 같은 뜻의 문장으로 나란히 쪼갠다. 같이 돌려보자.

먼저 환경 준비다. 터미널에서 아래 한 줄이면 끝난다.

```bash
pip install tiktoken
```

아래 코드는 GPT-4 계열이 쓰는 `cl100k_base` 인코더를 불러와 두 문장을 토큰화한다. `encode`는 문자열을 정수 ID 리스트로 바꿔주고, `decode`는 반대 방향이다. 토큰별 실제 문자열을 보기 위해 ID를 하나씩 역디코드한다.

```python
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

en = "I went to the library yesterday and borrowed three books."
ko = "어제 도서관에 가서 책을 세 권 빌렸다."

def show(text: str) -> None:
    ids = enc.encode(text)
    pieces = [enc.decode([i]) for i in ids]
    print(f"원문: {text}")
    print(f"토큰 수: {len(ids)}")
    print(f"조각들: {pieces}")
    print()

show(en)
show(ko)
```

실행 결과는 대략 이런 모양이 된다. 환경에 따라 몇 토큰 차이가 있을 수는 있다.

```text
원문: I went to the library yesterday and borrowed three books.
토큰 수: 11
조각들: ['I', ' went', ' to', ' the', ' library', ' yesterday', ' and', ' borrowed', ' three', ' books', '.']

원문: 어제 도서관에 가서 책을 세 권 빌렸다.
토큰 수: 22
조각들: ['어', '제', ' 도', '서', '관', '에', ' 가', '서', ' 책', '을', ' 세', ' 권', ' ', '빌', '렸', '다', '.']
```

같은 뜻을 담은 두 문장이다. 영어는 11토큰, 한국어는 약 22토큰. 거의 두 배다. 왜 이럴까?

답은 학습 데이터 비율에 있다. `cl100k_base` 어휘 100,277개는 주로 영어 웹 코퍼스에서 BPE로 병합된 결과다. 영어에서 `" library"`는 통째로 한 토큰이 될 만큼 자주 등장했다. 반면 "도서관"은 영어 코퍼스에 거의 없으니 `' 도'` + `'서'` + `'관'`으로 쪼개졌다. 심지어 "어제"의 `'어'`와 `'제'`가 분리됐고, "빌렸다"는 `'빌'`, `'렸'`, `'다'`의 세 조각으로 풀렸다.

대략적인 체감 수치는 이렇다. **영어는 1토큰 ≈ 0.75단어, 한국어는 1토큰 ≈ 0.4~0.5음절** 수준이다. 이게 실무에서 무엇을 뜻하는지 잠시 멈춰 생각해보자. OpenAI·Anthropic 같은 API는 토큰 수 기준으로 과금한다. 같은 분량의 사내 문서를 한국어로 보내면 영어로 보낼 때보다 **요금이 1.7~2배** 더 나올 수 있다. 월 청구서를 보고 나서야 이 사실을 깨닫는다면 꽤 아찔한 일이다. 컨텍스트 윈도우(context window)도 마찬가지다. 128K 토큰 한도는 영어로는 책 한 권이 거뜬히 들어가지만, 한국어로는 훨씬 빡빡하다. 같은 모델을 쓰는데도 한국어 서비스는 비용·지연·컨텍스트 압박을 더 받는다. 이건 기분 탓이 아니라 토크나이저의 통계 때문이다.

한 가지 더 관찰할 것이 있다. 한국어가 조각나는 방식이 **바이트 단위**라는 점이다. `cl100k_base`는 엄밀히 말하면 "바이트 BPE"다. UTF-8로 인코딩된 바이트 열 위에서 병합을 수행한다. 한글 한 글자 `"가"`는 UTF-8에서 3바이트(`0xEA 0xB0 0x80`)다. 그래서 디코드 결과가 `'도'` + `'서'` + `'관'`처럼 음절 단위로 나오는 것처럼 보여도, 내부적으로는 바이트 조각이 묶였다 풀렸다 하고 있다. 이 때문에 한국어 토큰 중엔 간혹 화면에 정상적으로 찍히지 않는 조각이 나올 수도 있다. "반쪽짜리 한글 바이트"가 다음 토큰의 나머지 바이트와 합쳐져야 비로소 한 글자가 완성되는 식이다. 이건 버그가 아니라 설계다.

모델별로 토크나이저가 다르다는 점도 한 번 확인해두자. 아래 코드는 같은 문장을 GPT-3.5가 쓰는 `cl100k_base`와 GPT-4o가 쓰는 `o200k_base`에 각각 넣어 토큰 수를 비교한다.

```python
import tiktoken

text = "어제 도서관에 가서 책을 세 권 빌렸다."

for enc_name in ["cl100k_base", "o200k_base"]:
    enc = tiktoken.get_encoding(enc_name)
    n = len(enc.encode(text))
    print(f"{enc_name:14s}  토큰 수 {n}")
```

실행하면 대략 이런 결과가 나온다.

```text
cl100k_base     토큰 수 22
o200k_base      토큰 수 15
```

`o200k_base`는 어휘가 약 20만 개로 늘어났고, 다국어·한국어 대응이 개선됐다. 같은 문장이 22에서 15로 줄었다. 7토큰 차이가 적어 보일 수 있지만, 하루 수십만 요청을 처리하는 서비스에서는 이 차이가 곧 비용이다.

**요금 체감을 위해 거친 계산을 한 번 해보자.** GPT-4 계열 입력 단가를 1M 토큰당 약 $2.5로 잡고, 사내 개발 문서(한국어) 10,000건을 요약한다고 해보자. 문서 하나당 평균 2,000자라면 한국어 토큰 수는 대략 4,000~4,500토큰. 전체 약 **4,500만 토큰 × $2.5 / 1M = $112.5**. 같은 분량을 영어로 썼다면 토큰이 1.7~2배 적으니 **절반 가까이 싸게** 먹힌다. 여기에 출력 토큰(요약문)까지 더하면 차이는 더 벌어진다. 한국어 서비스의 LLM 비용이 체감보다 빠르게 불어나는 이유가 이것이다.

그렇다면 한국어만 쓰는 프로젝트는 어떻게 해야 할까? 선택지는 네 가지다. (a) 감수하고 쓴다. (b) 한국어 비중이 높은 토크나이저를 쓰는 모델(HyperCLOVA-X, KoGPT, 한국어 튜닝된 Llama·Qwen 계열)을 고른다. (c) 입력을 영어로 번역해서 보낸 뒤 출력만 한국어로 받는 우회 전략을 쓴다. (d) **직접 학습시킨다.** 다음 실습에서 (d)를 맛본다.

## 2.3 손 실습 2 — `minbpe`로 한국어 BPE 토크나이저를 직접 훈련시켜보기

앞 실습에서 남의 토크나이저를 빌려 썼다면, 이번에는 내 손으로 하나 만들어본다. 안드레이 카파티(Andrej Karpathy)의 `minbpe`는 BPE의 핵심 로직만 남긴 최소 구현체다. 그중에서도 `BasicTokenizer`가 가장 작다. 이걸 한국어 텍스트에 훈련시켜 같은 문장이 어떻게 달라지는지 보자.

설치는 git clone 한 줄이다.

```bash
git clone https://github.com/karpathy/minbpe.git
cd minbpe
```

아래 스크립트는 `minbpe/basic.py`의 `BasicTokenizer`를 불러와 짧은 한국어 코퍼스에 학습시킨다. 어휘 크기(`vocab_size`)는 512로 잡았다. UTF-8 바이트 256개 위에 병합 256개가 얹히는 셈이다.

```python
# reference: karpathy/minbpe
from minbpe import BasicTokenizer

# 훈련용 한국어 코퍼스 (실전에서는 수 MB~수십 MB)
corpus = """
어제 도서관에 가서 책을 세 권 빌렸다.
오늘도 도서관에 갔다. 도서관에는 책이 많다.
도서관 사서는 친절했다. 책을 빌리는 일은 즐겁다.
나는 책을 좋아한다. 너도 책을 좋아하니?
""" * 20  # 반복해서 패턴이 잡히도록

tok = BasicTokenizer()
tok.train(corpus, vocab_size=512, verbose=False)

sample = "어제 도서관에 가서 책을 세 권 빌렸다."
ids = tok.encode(sample)
pieces = [tok.decode([i]) for i in ids]

print(f"토큰 수: {len(ids)}")
print(f"조각들: {pieces}")
```

출력은 대략 이렇게 나온다.

```text
토큰 수: 9
조각들: ['어제 ', '도서관', '에 ', '가서 ', '책을 ', '세 ', '권 ', '빌렸다', '.']
```

22토큰이 9토큰으로 줄었다. `cl100k_base`는 `"도서관"`을 세 조각으로 찢었지만, 한국어에 훈련시킨 작은 토크나이저는 `"도서관"`을 **한 덩어리로** 학습했다. `"어제 "`도, `"가서 "`도, `"책을 "`도 통째로 토큰이다. 코퍼스에 자주 같이 등장했기 때문이다.

> 실행하다 `AttributeError: module 'minbpe' has no attribute 'BasicTokenizer'`를 만난다면 당황하지 말자. `minbpe`는 `pip`로 설치되는 패키지가 아니다. 저장소를 clone한 디렉토리 안에서 실행하거나, `sys.path`에 그 경로를 추가해야 한다.

훈련 전후의 차이를 명시적으로 비교해보면 감이 더 잘 온다. 아래 코드는 같은 문장을 (a) 방금 훈련시킨 토크나이저, (b) `tiktoken`의 `cl100k_base`에 각각 넣고 토큰 수를 나란히 출력한다.

```python
import tiktoken

external = tiktoken.get_encoding("cl100k_base")
sentences = [
    "어제 도서관에 가서 책을 세 권 빌렸다.",
    "도서관 사서는 친절했다.",
    "나는 책을 좋아한다.",
]

print(f"{'문장':40s}  {'minbpe':>8s}  {'cl100k':>8s}")
for s in sentences:
    mine = len(tok.encode(s))
    ext = len(external.encode(s))
    print(f"{s:40s}  {mine:>8d}  {ext:>8d}")
```

실행 결과는 대략 이렇다.

```text
문장                                        minbpe    cl100k
어제 도서관에 가서 책을 세 권 빌렸다.             9        22
도서관 사서는 친절했다.                           6        13
나는 책을 좋아한다.                               5         9
```

세 문장 모두 minbpe 쪽이 절반 이하다. 어휘가 내 도메인에 맞춰져 있으니 당연한 결과다. 주의할 점 하나. **이 수치에 혹해서 "그럼 모든 서비스에 커스텀 토크나이저를 쓰자"고 점프하면 안 된다.** 커스텀 토크나이저를 쓰려면 그 토크나이저로 **모델 자체를 처음부터 학습시키거나 파인튜닝**해야 한다. 이미 학습된 GPT-4에 내 토크나이저를 끼우는 건 불가능하다. 어휘 ID와 임베딩 레이어의 row가 1:1 대응으로 묶여 있기 때문이다. 커스텀 토크나이저는 나중에 6장에서 오픈 모델을 파인튜닝할 때, 또는 from scratch로 미니 GPT를 만드는 4장에서 의미가 있다.

훈련된 토크나이저가 만든 병합 규칙은 `merges` 딕셔너리에 이력이 쌓인다. 초반 병합 열 개를 찍어보면 어떤 쌍이 먼저 묶였는지 눈으로 볼 수 있다.

```python
# minbpe BasicTokenizer는 merges 딕셔너리에 병합 이력을 갖고 있다
for (a, b), new_id in list(tok.merges.items())[:10]:
    left = tok.decode([a])
    right = tok.decode([b])
    merged = tok.decode([new_id])
    print(f"({left!r}, {right!r}) -> {merged!r}")
```

출력은 대략 이런 모양이다. 초반 병합은 공백과 빈도 높은 문자 쌍부터 시작된다.

```text
(' ', '도') -> ' 도'
(' 도', '서') -> ' 도서'
(' 도서', '관') -> ' 도서관'
('에', ' ') -> '에 '
('책', '을') -> '책을'
...
```

`"도서관"`이 한 토큰으로 합쳐지는 과정이 세 단계에 걸쳐 보인다. `" 도"` → `" 도서"` → `" 도서관"`. BPE가 말 그대로 **쌍을 하나씩 병합**하는 알고리즘이라는 사실이 여기서 눈에 들어온다.

자, 여기서 중요한 점을 기억해두자. **토큰화 품질은 어휘가 내 도메인에 맞게 학습됐는지에 달려 있다.** 범용 토크나이저는 영어에 최적화돼 있다. 한국어 비중이 큰 서비스라면, 파인튜닝 전에 먼저 "내 모델은 어떤 토크나이저를 쓰는가, 그것이 내 도메인에 맞는가"를 확인하는 편이 낫다. 같은 모델이라도 토크나이저에 따라 비용·지연·품질이 크게 갈린다.

여기까지 오면 첫 질문 하나는 풀렸다. LLM은 문자열을 정수 ID 리스트로 바꿔 받는다. 그런데 정수 ID 자체는 아무 의미도 들고 있지 않다. `도서관`의 ID가 279번이든 5,112번이든, 숫자 자체에는 "책이 있는 장소"라는 뜻이 없다. 모델은 `"도서관"`과 `"서점"`이 의미상 가깝다는 사실을 어떻게 알까? 다음 층으로 내려가보자.

## 2.4 임베딩 — 의미가 가까우면 벡터도 가깝다

토큰 ID는 사전 색인일 뿐이다. 그걸 의미가 있는 무언가로 바꿔야 한다. 그 무언가가 바로 **임베딩(embedding)**이다.

정의는 한 줄이다. 임베딩은 **토큰 ID를 고차원 실수 벡터로 변환한 것**이다. 보통 768차원, 1024차원, 4096차원 같은 수로 간다. GPT-3의 경우 한 토큰이 12,288차원짜리 실수 벡터로 표현된다. 차원 수는 모델마다 다르다.

중요한 건 차원 수가 아니라 **벡터 공간의 성질**이다. 잘 학습된 임베딩은 이런 성질을 갖는다.

> **의미가 가까운 토큰은 벡터 공간에서도 가깝다.**

"도서관"과 "서점"은 비슷한 자리에 찍히고, "도서관"과 "자동차"는 멀리 떨어져 있다. "가깝다"는 보통 코사인 유사도(cosine similarity)로 잰다. 두 벡터가 이루는 각도가 작을수록 1에 가깝다.

이걸 눈으로 확인해보자. HuggingFace의 `sentence-transformers`가 편하다. 설치는 역시 한 줄이다.

```bash
pip install sentence-transformers
```

아래 스크립트는 다국어 임베딩 모델 `paraphrase-multilingual-MiniLM-L12-v2`를 불러와 몇 개 단어의 벡터를 구하고, `"도서관"`과의 코사인 유사도를 내림차순으로 보여준다.

```python
from sentence_transformers import SentenceTransformer, util

model = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")

query = "도서관"
candidates = ["서점", "학교", "책", "사서", "자동차", "바나나", "주차장", "독서실"]

q_vec = model.encode(query, convert_to_tensor=True)
c_vec = model.encode(candidates, convert_to_tensor=True)
scores = util.cos_sim(q_vec, c_vec)[0]

ranked = sorted(zip(candidates, scores.tolist()), key=lambda x: -x[1])
for word, score in ranked:
    print(f"{word:8s}  유사도 {score:.3f}")
```

출력은 이런 모양이 된다. 모델 버전에 따라 소수점 둘째 자리는 흔들릴 수 있다.

```text
독서실      유사도 0.712
서점        유사도 0.684
사서        유사도 0.651
책          유사도 0.598
학교        유사도 0.512
주차장      유사도 0.287
자동차      유사도 0.194
바나나      유사도 0.121
```

"독서실", "서점", "사서"가 상위다. "책", "학교"는 중간. "자동차"와 "바나나"는 바닥이다. 벡터 공간이 의미적 관계를 꽤 그럴듯하게 반영한다는 걸 눈으로 확인할 수 있다. 놀라운 점은 누가 "도서관과 독서실은 의미가 비슷하다"고 가르친 적이 없다는 사실이다. 모델은 방대한 텍스트에서 "도서관"과 "독서실"이 비슷한 문맥에 등장하는 것을 보고, 두 벡터를 자연스럽게 가까이 배치했다. **분포 가설(distributional hypothesis)**이라는 오래된 언어학 원리다. "비슷한 문맥에 나오는 단어는 비슷한 의미를 갖는다."

이쯤에서 유명한 실험 하나를 재현해보자. 2013년 Mikolov의 word2vec 논문이 보여준 고전이다.

> `king - man + woman ≈ queen`

벡터끼리 더하고 빼는 산술 연산에 의미론이 나타난다는 관찰이다. 같은 감각으로 한국어에서도 시도해본다. 아래 코드는 `"서울 - 한국 + 일본"`을 계산한 뒤 후보 도시 중 가장 가까운 벡터를 찾는다.

```python
def closest(vec, pool, topk=3):
    pool_vecs = model.encode(pool, convert_to_tensor=True)
    sims = util.cos_sim(vec, pool_vecs)[0]
    ranked = sorted(zip(pool, sims.tolist()), key=lambda x: -x[1])
    return ranked[:topk]

seoul = model.encode("서울", convert_to_tensor=True)
korea = model.encode("한국", convert_to_tensor=True)
japan = model.encode("일본", convert_to_tensor=True)

target = seoul - korea + japan

pool = ["도쿄", "오사카", "베이징", "부산", "뉴욕", "파리", "후쿠오카"]
for word, score in closest(target, pool):
    print(f"{word:6s}  유사도 {score:.3f}")
```

결과는 실행에 따라 갈린다. 어떤 때는 `"도쿄"`가 1위로 오르고, 어떤 때는 `"오사카"`나 `"후쿠오카"`가 먼저 오기도 한다. 문장 임베딩 모델은 단어 수준 word2vec보다 잡음이 많아서 항상 깔끔하게 떨어지진 않는다. 그래도 일본 쪽 도시들이 상위를 차지하는 경향은 분명히 보인다. 벡터 공간이 "수도-국가" 관계를 어느 정도 축으로 잡아뒀다는 뜻이다.

물론 이 연산이 언제나 성립하는 것은 아니다. "삼촌 - 남자 + 여자 = 이모"는 잘 되지만, "파리 - 프랑스 + 베트남 = 하노이"는 덜 선명하다. 편향이나 학습 데이터 편중으로 이상한 결과가 나올 때도 있다. 기억해두자. **임베딩은 통계의 그림자다.** 세상을 완벽하게 반영하는 수학적 진리가 아니라, 학습 말뭉치 안에서의 공기(共起) 패턴이다. 그래서 편향도 같이 학습되고, 도메인이 바뀌면 의미도 바뀐다. 이 사실을 잊으면 나중에 "모델이 왜 이렇게 대답하지?" 하는 지점에서 번번이 뒷맛이 찜찜해진다.

### 왜 굳이 고차원인가

여기서 찜찜한 의문이 하나 생긴다. 왜 768차원, 4096차원 같은 숫자일까? 100차원으론 안 될까? 10,000차원은 더 좋을까?

직관적으로 이렇게 생각해보자. 의미는 여러 축을 동시에 갖는다. "왕"이라는 단어는 단수/복수, 남성/여성, 인물/사물, 권력/평민, 현대/고대 같은 수많은 축 위에 동시에 놓여 있다. 각 축을 하나의 차원으로 잡는다고 해도 수백 개는 필요하다. 실제로 모델이 각 차원을 사람이 이해할 수 있는 단일 개념으로 학습하는 건 아니다. 차원들은 **뒤엉켜(entangled)** 있다. 그래서 여유 있게 주는 편이 낫다.

너무 작으면 다른 의미가 같은 자리에 겹친다. 너무 크면 계산 비용이 치솟고 과적합이 난다. 경험적으로 768~4096 사이 어디쯤이 대형 모델에는 좋은 지점으로 알려져 있다. BERT base가 768, GPT-3가 12,288, Llama 3 8B가 4,096이다. 참고 삼아 기억해두자.

### 문장 임베딩과 토큰 임베딩은 다르다

앞의 실습에서 쓴 `sentence-transformers`는 **문장 전체**를 하나의 벡터로 만든다. 엄밀히 말하면 LLM 내부의 "토큰 임베딩 레이어"와는 다르다. 문장 임베딩 모델은 보통 BERT 계열 모델로 토큰별 벡터를 뽑은 뒤 평균 풀링(mean pooling)해서 문장 벡터 하나로 압축한다. 두 개념의 층위가 다르다.

LLM 내부의 임베딩 레이어는 **토큰 하나당 벡터 하나**를 뱉는다. `"어제 도서관에 가서 책을 세 권 빌렸다."` 9토큰이라면 9개의 벡터가 나와 `[9, 768]` 모양 행렬이 된다. 이 행렬이 이후 어텐션 블록의 입력이다. 반면 RAG 같은 검색 시스템에서 쓰는 임베딩은 보통 문장·문단 단위다. 출처도 다르고 용도도 다르다. 8장에서 RAG를 다룰 때 `bge-m3` 같은 한국어 품질 좋은 문장 임베딩 모델을 다시 만날 것이다. 같은 "임베딩"이란 단어를 쓰지만 층위가 다르다는 점은 지금 머릿속에 넣어두자.

## 2.5 Java 개발자의 머릿속에서는

여기까지 오면 머릿속이 조금 헝클어졌을 수 있다. 익숙한 언어로 한 번 환기해보자. 평소 Spring Boot와 JPA를 쓰는 독자라면 이렇게 떠올려도 좋다.

> **토큰 ID는 `Map<String, Integer>`의 value, 임베딩은 `float[768]` row다.**

아래 코드는 토크나이저와 임베딩 레이어의 모양을 Java로 스케치한 것이다. 실제 구현은 훨씬 복잡하지만 자료구조의 골격은 이 정도다.

```java
// 토크나이저: 문자열을 정수 ID로 매핑하는 사전
public class Tokenizer {
    private final Map<String, Integer> vocab;   // "도서관" -> 279
    private final String[] inverseVocab;        // 279 -> "도서관"

    public int[] encode(String text) { /* BPE 병합 적용 */ }
    public String decode(int[] ids)   { /* inverseVocab 조회 */ }
}

// 임베딩: ID를 고차원 실수 벡터로 변환
public class EmbeddingLayer {
    // vocabSize 행 × embeddingDim 열의 행렬
    private final float[][] weight;   // [100277][768] 같은 모양

    public float[] lookup(int tokenId) {
        return weight[tokenId];        // ID번째 row를 그대로 꺼낸다
    }
}
```

모델 내부의 "임베딩 레이어"는 결국 거대한 2차원 배열이다. 행은 어휘 크기(vocabulary size)만큼, 열은 모델의 은닉 차원(hidden dim)만큼이다. GPT-4급 모델이라면 `float[100000][4096]` 수준. 대략 16억 개의 float이다. 이 배열은 학습 과정에서 경사 하강법으로 조금씩 갱신된다. "도서관"과 "독서실"의 row가 비슷한 방향으로 수렴하는 건 누가 규칙으로 박아넣은 게 아니라, 수많은 훈련 스텝을 거치며 자연스럽게 가까워진 결과다.

Java 개발자에게 친숙한 비유를 하나 더 보태자면, **임베딩 조회(lookup)는 `HashMap.get(key)`와 시간 복잡도가 같다.** O(1)이다. 토큰 ID가 들어오면 그 번째 row를 꺼내기만 하면 된다. 연산 비용이 드는 건 그 다음부터다. 768차원 벡터를 받아 어텐션 가중치를 계산하고, 레이어를 쌓아 올리는 일이 LLM 계산량의 본체다. 그 이야기는 다음 장의 몫이다.

한 가지 더 기억해둘 만한 점이 있다. 임베딩 레이어의 가중치 행렬은 **학습 가능한 파라미터**다. 우리가 JPA의 `@Entity` 테이블을 읽기/쓰기 둘 다 하듯이, 학습 중에는 이 행렬의 값이 경사 하강법으로 꾸준히 업데이트된다. 초기엔 랜덤이다. "도서관"의 row와 "서점"의 row가 처음엔 아무 관계도 없다. 수십억 토큰을 흘려보내며 "이 두 단어가 비슷한 문맥에 등장한다"는 신호를 받을 때마다 두 row가 서서히 같은 방향으로 당겨진다. 이 과정을 오래 반복한 결과가 GPT나 Claude 같은 모델의 입구에 놓인 거대한 임베딩 테이블이다. **행렬 자체가 곧 훈련의 기록물**인 셈이다.

실무에서 한 가지 더 만나게 될 개념이 있다. **타이드 임베딩(tied embedding)**이라는 기법이다. 입력 쪽 임베딩 행렬과 출력 쪽 "언어 모델 헤드(LM head)"의 가중치 행렬을 **같은 것**으로 묶는 트릭이다. 입력에서 "도서관" → 벡터로 바꿀 때 쓰는 행렬과, 출력에서 "어떤 토큰을 낼지" 확률을 계산할 때 쓰는 행렬이 하나로 공유된다. 파라미터 수가 줄고, 학습도 더 안정적이다. GPT-2 이후 대부분의 decoder-only 모델이 이 방식을 쓴다. 아주 깊게 들어가진 않지만, 이름을 들어본 적 있다는 정도로 기억해두자. 4장에서 미니 GPT를 직접 만들 때 다시 마주친다.

## 2.6 그래서, 순서는 어떻게 알까

이 장에서 확인한 것을 정리해보자. 문자열이 들어오면 토크나이저가 서브워드 단위로 쪼개 정수 ID 리스트를 만든다. 임베딩 레이어가 각 ID를 고차원 실수 벡터로 바꾼다. 이제 문장 `"어제 도서관에 가서 책을 세 권 빌렸다."`는 모델 내부에서 대략 `[9, 768]` 모양의 행렬로 존재한다. 9행은 토큰 9개, 768열은 각 토큰의 의미 벡터다.

그런데 여기서 한 가지가 빠졌다. 눈치챘을지 모르겠다. 행렬만 봐서는 "어제"가 맨 앞에 왔는지, "빌렸다"가 맨 앞에 왔는지 구분이 되지 않는다. 임베딩 레이어는 **어떤 토큰이 어떤 의미인가**는 알려주지만 **그 토큰이 몇 번째 자리에 있었는가**는 담지 않는다. 9개 토큰을 아예 순서를 바꿔서 뒤섞어 넣어도 임베딩 결과 행렬의 집합은 똑같다.

사람 말은 순서가 중요하다. "개가 사람을 물었다"와 "사람이 개를 물었다"는 같은 단어들로 이루어졌지만 전혀 다른 뉴스다. 순서 정보가 사라지면 모델은 두 문장을 구분할 수 없다. 그런데 RNN처럼 순차 처리를 하지 않는 트랜스포머에서는, 순서가 **저절로** 들어오지 않는다. 누군가 명시적으로 집어넣어야 한다.

자, 여기까지가 2장의 경계다. 토큰을 숫자로 바꾸긴 했고, 의미의 좌표도 얻었다. 그런데 순서는 어떻게 알지? 이 질문이 다음 장의 열쇠다. 3장은 위치 인코딩(Positional Encoding) 이야기부터 시작해 트랜스포머의 핵심인 어텐션까지 한 걸음씩 밟아간다.

## 마무리 — 이 장에서 같이 얻은 것

한 번 되짚어보자.

- 문자·단어·서브워드 중에서 현대 LLM은 서브워드를 쓴다. BPE, WordPiece, SentencePiece 계보가 있고, 원리는 모두 "자주 같이 나오는 조각을 묶는다"로 요약된다.
- `tiktoken`으로 같은 뜻의 영어·한국어 문장을 쪼개봤다. 한국어는 영어 대비 **약 1.7~2배 더 많은 토큰**을 먹는다는 걸 숫자로 확인했다. 이건 API 요금·컨텍스트 창·지연에 그대로 반영되는 실무 문제다.
- `minbpe`로 한국어 도메인에 맞춘 작은 토크나이저를 직접 훈련시켜봤다. 토크나이저가 도메인에 맞으면 같은 문장도 훨씬 짧게 표현된다.
- 임베딩은 "의미가 가까우면 벡터도 가깝다"는 공간을 만든다. `sentence-transformers`로 `"도서관"`과 `"독서실"`이 벡터 공간에서 실제로 이웃임을 확인했다. 수도-국가 관계 같은 구조도 벡터 산술로 어느 정도 잡힌다.
- Java로 치면 토크나이저는 `Map<String, Integer>` 사전, 임베딩 레이어는 `float[vocab][dim]` 행렬이다. 조회 비용은 O(1).

그리고 하나 남긴 질문이 있다. **순서는 어떻게 들어가는가.** 이 질문을 들고 다음 장으로 넘어가자. 거기서 트랜스포머의 두 기둥—위치 인코딩과 어텐션—을 같이 열어본다.
