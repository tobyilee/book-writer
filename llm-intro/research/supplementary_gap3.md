## 1. 정면 실패담 (Case Studies — 9장 본문용)

각 항목은 9장 케이스 스터디 3건 후보다. **(a) 무엇이 틀어졌나 / (b) 시작할 때 잘못 가정한 것 / (c) 대신 했어야 했던 것** 구조로 추출했다.

### 1.1 소규모 팀의 LLM 서비스 실패 회고 (velog / soyeon)

- **출처:** [2025 회고] 소규모 팀에서 LLM 서비스 만들기 · Soyeon · 2025-12-30 · https://velog.io/@soonmac/llm-service-in-a-small-team
- **플랫폼:** velog
- **(a) 무엇이 틀어졌나:**
  - 주니어 3~4인 팀, AI/LLM 경험 0. 프론트엔드 1인 저자가 풀스택 책임.
  - 초반 버전이 "단순 키워드 프롬프트"에 의존 → 환각, 제품 톤 불일치, 도메인-출력 느슨한 결합.
  - 회사 데이터가 여러 시스템에 흩어져 있고 ML 용도로 정제되지 않은 상태.
- **(b) 잘못된 가정:**
  - 경영진: "파인튜닝하면 품질 문제가 해결된다". 리소스·실현성·실제 비즈니스 니즈 계산 누락.
  - 조직: "**요즘 AI가 대세니까 한번 해보자**"라는 열의만으로 착수. 필요한 리소스 합의 없음.
- **(c) 대신 했어야 했던 것:**
  - 파인튜닝 대신 RAG로 선회. 도메인 지식을 구조화된 레퍼런스로 정리, 메타데이터 기반 분류·유사도 검색.
  - AI는 수단. 비즈니스 임팩트를 먼저 검증. 규칙 기반 위에 AI를 전략적으로 얹기.
  - 기술 해법 대신 UX 해법을 우선 고려.
- **직접 인용 (서문/1장/9장 소재):**
  > "솔직히 중간중간 후회가 들 정도로 쉽지 않은 프로젝트였다" [velog:soyeon]
  > "요즘 AI가 대세니까 한번 해보자" [velog:soyeon]
- **9장에서의 쓰임:** "첫 직감이 fine-tune이면 red flag" 플로차트의 **가장 전형적인 반례**. 입문 팀이 파인튜닝으로 직행했다가 RAG로 선회하는 서사.

### 1.2 당근페이 FDS에 LLM 적용하기 (Medium / 당근 테크블로그)

- **출처:** 당근페이 AI Powered FDS로 가는 여정: 룰엔진 구축부터 LLM 적용까지 · HyunwooKim · 2025-11-24 · https://medium.com/daangn/%EB%8B%B9%EA%B7%BC%ED%8E%98%EC%9D%B4-ai-powered-fds%EB%A1%9C-%EA%B0%80%EB%8A%94-%EC%97%AC%EC%A0%95-%EB%A3%B0%EC%97%94%EC%A7%84%EA%B5%AC%EC%B6%95%EB%B6%80%ED%84%B0-llm-%EC%A0%81%EC%9A%A9%EA%B9%8C%EC%A7%80-725c3a1c388d
- **플랫폼:** 당근 테크블로그 (Medium)
- **(a) 무엇이 틀어졌나:**
  - RAG 적용 시 retrieve 결과의 JSON이 **중간에서 잘려** 반환.
  - 최근 사기 데이터만으로 평가했더니 정탐률·재현률이 **시간이 지날수록 하락**.
  - 프롬프트가 길어질수록 응답 오류 증가, 출력 포맷 불안정.
- **(b) 잘못된 가정:**
  - 기본 청킹 전략(자동 분할)이 JSON 구조를 이해한다는 가정.
  - 최근성 ≈ 관련성. "최근 사기 데이터가 현재 거래 패턴을 대표할 것"이라는 가정.
  - `RetrieveAndGenerate`의 단일 패스 호출이 고위험 도메인에 충분하다는 가정.
- **(c) 대신 했어야 했던 것:**
  - 사기 케이스별 **파일 1개 = 청크 1개**로 분리. `ChunkingStrategy=NONE`.
  - `Retrieve` + `ConverseAPI` 파이프라인으로 분리. retrieve 결과를 **사람이 설계한 기준으로 조립** 후 프롬프트에 투입.
  - 최근성이 아닌 **의미 유사도** 기반 RAG로 재설계.
- **직접 인용:**
  > "retrieve 시, 결과의 content가 완전한 Json 형태가 아니라 일부가 잘려서 나오는 문제가 발생했어요" [techblog:당근]
  > "단순히 최근 사기 거래만을 포함하면 현재 거래 패턴과 다를 수도 있기 때문에 잘못된 판단 결과를 내릴 가능성이 높았던 거예요" [techblog:당근]
  > "프롬프트가 길수록 응답 오류가 늘어나고, 출력 포맷도 안정적으로 유지되지 않는다는 점을 확인" [techblog:당근]
- **9장에서의 쓰임:** "관측성·평가 설계 누락 시 무슨 일이 일어나는가"의 **수치 있는 프로덕션 사례**. 금융 도메인은 "구조화된 제어 > 엔드투엔드 자동화"라는 9장 테제의 근거.

### 1.3 SK하이닉스 MSR의 RAG 플랫폼 성능 평가 (AWS Tech Blog)

- **출처:** SK하이닉스의 RAG 플랫폼 구축 및 성능 평가/분석 연구 사례 · HwiKyoung Kim 외 · 2025-02-24 · https://aws.amazon.com/ko/blogs/tech/sk-hynix-rag-platfrom-analysis-evaluation/
- **플랫폼:** AWS Korea Tech Blog (SK하이닉스 MSR 공동 저자)
- **(a) 무엇이 틀어졌나:**
  - 온프레미스 환경에서 "타당성·정합성의 판단 한계" 존재. 확장성·유연성·용도 전환이 어려움.
  - 데이터셋이 커지면 **TTFT(Time-To-First-Token) 약 30% 증가** (RAG가 순수 LLM 대비).
  - 벡터 인덱스가 메모리에 못 들어갈 때 **검색 시간 약 1300배 증가** (0.088초 → 122.06초). 30GB 인덱스 기준.
- **(b) 잘못된 가정:**
  - "온프레미스 자체 SW 플랫폼이면 연구용으로 충분하다"는 전제.
  - "벡터 DB는 설치하면 끝"이라는 인프라 경시.
- **(c) 대신 했어야 했던 것:**
  - 단순 인프라 스케일업은 비용적으로 지속 불가. **성능과 비용을 고려한 최적화 솔루션** 필요.
  - 메모리-스토리지 상호작용을 **실측**으로 검증하는 벤치마크 설계.
- **직접 인용:**
  > "기존에 On-Premise 환경에서 연구를 진행해 왔으나 자체 구축한 SW 플랫폼의 타당성 및 정합성의 판단 한계가 존재하였고, 인프라 구축에 있어 확장성, 유연성, 용도 전환 등에 어려움이 존재" [techblog:AWS-SK하이닉스]
  > "RAG Inference는 기존 LLM 서비스 대비 지식 데이터 검색을 수행해야 하며, 이에 의해서 TTFT(Time-To-First-Token)가 약 30% 증가" [techblog:AWS-SK하이닉스]
  > "벡터 데이터 인덱스를 위한 메모리 용량이 부족한 경우 스토리지로부터 인덱스를 읽어서 검색 시간이 급격히 증가" [techblog:AWS-SK하이닉스]
- **9장에서의 쓰임:** 9장 "비용·지연·품질 3축 트레이드오프" 절의 **수치 있는 벤치마크 사례**. "일단 RAG부터"가 만능이 아니라는 반례.

### 1.4 카카오페이증권 춘시리 AI봇 — "비전문가도 개발 가능, 하지만 보안은 가장 어려웠다"

- **출처:** 페이증권의 업무도우미 AI봇을 소개합니다! 근데 이제 춘식이를 곁들인 · Terra · 2025-01-24 · https://tech.kakaopay.com/post/choonsiri/
- **플랫폼:** 카카오페이 기술블로그
- **(a) 무엇이 틀어졌나:**
  - 도입 과정에서 가장 어려운 부분이 "기술"이 아니라 **보안**이었음.
  - 초기 위키 Q&A 중심으로 설계했지만, 출시 후 보니 **일반 질문 기능**이 가장 많이 쓰임 (사내에서 외부 AI 도구가 보안 정책으로 막혀 있어서).
  - 서울 리전에서 Anthropic Claude만 가능, 모델 다양화 제약.
- **(b) 잘못된 가정:**
  - "AI 개발 난이도가 가장 큰 장벽"이라는 가정.
  - 사용자들이 "문서 기반 Q&A"만 원할 거라는 가정.
- **(c) 대신 했어야 했던 것:**
  - **AWS Bedrock Guardrails**로 입출력 민감정보 필터링, 민감 문서에 "exclude" 라벨링.
  - 개발 시간의 대부분은 LLM 로직이 아니라 **Slack 포매팅·API 연동**이라는 사실을 사전에 반영.
- **직접 인용 (서문·1장·10장 공감 인용 소재):**
  > "증권사에서 AI를 도입하며 가장 어려웠던 측면은 아무래도 보안이었습니다" [techblog:카카오페이]
  > "일반질문 기능을 통해 업무 도중 다양한 문제를 빠르게 해결할 수 있었습니다" [techblog:카카오페이]
  > "비전문가도 충분히 개발할 수 있다는 사실을 몸소 체험" [techblog:카카오페이]
- **9장·서문에서의 쓰임:** 서문 "LLM을 처음 현업에 붙일 때 실제로 뭐가 어려운가"의 역전 서사. **기술 < 보안/거버넌스**라는 9장 거버넌스 절 도입부 인용.

### 1.5 피카부랩스 — 로컬 LLM RAG 시스템 4가지 함정

- **출처:** 로컬 LLM으로 RAG 시스템 구축, 직접 해보고 알게 된 4가지 함정 · Seunghwan Kim · 2025-09-09 · https://peekaboolabs.ai/blog/local-llm-rag-system-pitfalls
- **플랫폼:** 피카부랩스 블로그 (벤더 블로그지만 실무 시행착오 기록이 구체적)
- **(a) 무엇이 틀어졌나:**
  - 문서 파싱 — 표·차트가 기본 라이브러리에 무시됨. 페이지 단편화로 맥락 소실.
  - 기술 스택 선택 피로 — LangChain vs LlamaIndex, 임베딩 모델, 벡터 DB 버전 호환성.
  - 벡터 DB 운영 — 설치는 시작에 불과. 대용량에서 인덱스 최적화 필요, 연구자가 DBA가 됨.
  - "공짜"의 숨은 비용 — GPU 하드웨어, 유지보수 기회비용.
- **(b) 잘못된 가정:**
  - "**오픈소스를 활용하니 비용은 안 들겠지?**"
  - "설치하면 끝"이라는 벡터 DB 경시.
  - 문서 파싱이 "텍스트 추출 = 쉬운 일"이라는 가정.
- **(c) 대신 했어야 했던 것:**
  - 파싱/임베딩/벡터 DB가 통합된 사전 구성 솔루션 채택 (벤더 제품 추천).
  - 혹은 DIY할 거면 **DBA 수준의 유지보수 리소스를 사전 할당**.
- **직접 인용:**
  > "오픈소스를 활용하니 비용은 안 들겠지?" [web:피카부랩스]
- **9장에서의 쓰임:** 9장 "결정 플로차트 — 언제 로컬/오픈소스를 꺼내는가"의 **경계 조건**. "공짜가 아니다"의 구체적 불릿.

### 1.6 RAG 프로젝트는 왜 실패하는가? — 패널 토크 후기 (velog)

- **출처:** RAG 프로젝트는 왜 실패하는가?: RAG를 잘 다루고 싶은 엔지니어를 위한 패널 토크 후기 · smallcherry · https://velog.io/@smallcherry/RAG-%ED%94%84%EB%A1%9C%EC%A0%9D%ED%8A%B8%EB%8A%94-%EC%99%9C-%EC%8B%A4%ED%8C%A8%ED%95%98%EB%8A%94%EA%B0%80-RAG%EB%A5%BC-%EC%9E%98-%EB%8B%A4%EB%A3%A8%EA%B3%A0-%EC%8B%B6%EC%9D%80-%EC%97%94%EC%A7%80%EB%8B%88%EC%96%B4%EB%A5%BC-%EC%9C%84%ED%95%9C-%ED%8C%A8%EB%84%90-%ED%86%A0%ED%81%AC-%ED%9B%84%EA%B8%B0
- **플랫폼:** velog (goorm 기술 블로그 패널 요약)
- **(a) 무엇이 틀어졌나:**
  - PDF·표·이미지를 **전부 파싱해 저장**하는 전략이 항상 효율적이지 않음.
  - Retrieval 한 컨텍스트에 **상충되는 정보**가 있으면 할루시네이션 유발.
  - 85~90% 정확도를 달성해도 **고객 기대 불일치**로 불만 발생.
- **(b) 잘못된 가정:**
  - "임베딩 모델 하나로 모든 도메인 커버 가능".
  - "벡터 검색만으로 충분, 키워드 필요 없음".
  - "높은 정확도 수치 = 고객 만족".
- **(c) 대신 했어야 했던 것:**
  - **도메인 전문가가 검증한 Golden Dataset** 구축 (LLM 생성 Silver dataset 이상).
  - 문서 요약·목차·Q&A 페어를 **메타데이터로 추가** (한 사례에서 검색 정확도 +15%p).
  - 키워드 + 벡터 하이브리드, Text-to-SQL 병행.
  - 고객에게 **RAG의 기술적 한계를 선제적으로 설명**.
- **직접 인용:**
  > "모두 파싱해서 저장하는 건 상황에 따라서 비효율적일 수도 있음" [velog:smallcherry]
  > "Retrieval한 Context에 상충된 정보가 존재하면 할루시네이션을 유발할 수 밖에 없음" [velog:smallcherry]
  > "고객에게 RAG의 기술적 특성과 한계를 먼저 이해시키는 것이 중요함" [velog:smallcherry]
- **9장에서의 쓰임:** 9장 플로차트에서 **"RAG는 만능이 아니다"** 분기 조건의 근거. 평가 하네스 절의 "Golden vs Silver dataset" 언급.

## 2. 전환점 이야기 (Turning Points — 서문·1장·10장용)

초기의 혼란 → 조직 전체의 방향 전환으로 이어진 이야기들. 기술보다 감정·서사 결이 살아 있다.

### 2.1 당근 GenAI 플랫폼 — "계정과 API키 발급 자체가 병목이 되었어요"

- **출처:** 당근의 GenAI 플랫폼 · Tommy Park · 2025-12-19 · https://medium.com/daangn/%EB%8B%B9%EA%B7%BC%EC%9D%98-genai-%ED%94%8C%EB%9E%AB%ED%8F%BC-ee2ac8953046
- **전환점:** 팀마다 OpenAI·Anthropic·Google 계정을 따로 발급 → API 키 관리 불가 → 보안 우려 → **LLM Router로 단일 게이트웨이 재설계**.
- **직접 인용:**
  > "계정과 API키 발급 자체가 병목이 되었어요. 발급한 API 키가 많아지면서 어느 팀이 어떤 서비스를 위해 계정과 키를 사용하는지 관리하기 어려워졌고 보안에 대한 우려도 커졌어요." [techblog:당근]
  > "어떤 계정은 여유가 남아있는데, 다른 계정은 이미 한도에 도달해 호출이 실패하는 상황이 발생" [techblog:당근]
- **쓰임:** **서문/1장**에서 "LLM을 쓰기 시작하면 처음에는 기술이 아닌 계정·키·비용 관리부터 막힌다"는 현실 인용. 10장 에필로그 "내가 다룰 그 다음 문제"의 구체 예시.

### 2.2 토스증권 GPU 클러스터 도입기 — "요리하라고 해서 왔는데 프라이팬이 없어요"

- **직접 인용 (제목이 이미 공감 인용):**
  > "요리하라고 해서 왔는데 프라이팬이 없어요" [techblog:토스]
- **쓰임:** **서문 오프닝 후보**. 은유가 강하고 짧아 인용 풀 최상위. "LLM 도입하라고 해서 시작했는데 인프라부터 없더라"는 독자 공감축.

### 2.3 토스증권 LLM 서빙 — "수십 초씩 기다려야 합니다"

- **출처:** LLM 쉽고 빠르게 서빙하기 · Kim Min-gyu · 2024-11-26 · https://toss.tech/article/llm-serving

- **직접 인용:**
  > "많은 정보를 알아야 됩니다" (ML 엔지니어가 vLLM, Triton, Docker, Kubernetes를 다 알아야 함) [techblog:토스]
  > "이런 허들 때문에 여러 모델을 서빙해서 A/B 테스트로 비교하기 어렵습니다" [techblog:토스]
- **쓰임:** 9장 "관측성·비용" 절, **"기술적으로 가능한 것과 실제로 운영 가능한 것 사이의 간극"** 예시.

### 2.4 카카오페이 백엔드 개발자의 LLM 내부 원리 — "끝이 보이지 않기도 합니다"

- **쓰임:** **1장 오프닝 공감 인용**. 이 책의 독자와 정확히 같은 직군(백엔드 개발자)이 같은 감정을 공유한 기록.

## 5. 리서치 한계

- **인터뷰 없음:** 계획 단서에 따라 인터뷰는 수집하지 않음. 사적이고 감정적인 실패담(예: "팀장에게 설득당해 파인튜닝 했다가 3개월 날렸다" 같은 1인칭 고백)은 상대적으로 약함.
- **최근성 편향:** 수집된 글의 80%가 2024~2025년. 2022~2023년 초기 LLM 도입기 회고는 커뮤니티 파편이 많고 프로덕션 포스트모템 형태로는 드묾.
- **조직 편향:** 토스·당근·카카오페이·SK하이닉스 같은 **대기업/강한 엔지니어링 조직** 비중이 높음. 중소기업·스타트업 실패담은 velog/Disquiet의 개인 회고 1~2건(soyeon 등)으로 대표.
- **OKKY·네이버 카페 실무 토론:** 주제별 장문의 실패담은 거의 없었고, 짧은 질답 위주. 인용 소재로는 약함.
- **네이버 d2 / 라인 Engineering:** d2.naver.com에서 LLM **실패** 회고글은 발견되지 않음 (LY 기술블로그 가드레일 글은 발견, 이는 ④로 이관). HyperCLOVA X 개발기는 공개된 회고보다 홍보성 글 위주.
- **익명 게시판:** OKKY·블라인드 등의 익명 주장은 신뢰성 리스크 때문에 본 수집에서 제외.
