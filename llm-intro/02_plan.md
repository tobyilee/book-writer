- **장르:** 기술 입문서 (원리 50% + 실습 40% + 에세이형 내러티브 10%). 순수 실습서도 순수 이론서도 아님. 현장 합의인 **"튜토리얼 → 그림 → 논문 → 코드 재구현"의 4단 파이프라인**(Reference 5.1)을 한 권에 압축.
- **분량:** 전체 본문 약 **230,000~270,000자 (공백 포함, A5 기준 약 380~440쪽)**. 서문·에필로그·부록 포함 시 430~490쪽.
  - 의미 있는 깊이 우선. 다만 "구현 코드 통째 복붙"으로 분량 부풀리지 않음. 긴 코드는 핵심 20~40줄만 본문, 전체는 저장소 링크.
- **난이도:** **중급 (선수 지식: Python 기본 문법, 학부 선형대수·미분 기초, HTTP API 호출 경험, Java Spring 기초).** 첫 장에서 선수 지식 자가 체크 박스 제공.
- **독자 여정:**
  - **진입 상태:** "ChatGPT/Claude는 써봤지만 내부는 블랙박스. 트랜스포머 그림을 10번 봐도 감이 안 오고, 공부를 시작하기도 전에 **어디서부터 뭘 해야 할지 몰라서 지쳐 있다.**"
  - **중간 각성:** 4장 말미 — 직접 짠 100~300줄짜리 미니 GPT가 한국어 토큰을 이어붙이고, 디코딩 전략을 바꾸면 같은 모델에서 다른 문장이 나오는 걸 보며 "LLM이 결국 조건부 확률 + 샘플링이구나"를 몸으로 체감.
  - **출구 상태:**
    1. 토큰 → 임베딩 → 어텐션 → 디코딩의 파이프라인을 **자기 언어로** 설명할 수 있다.
    2. nanochat/build-nanogpt 스타일로 미니 모델을 **from scratch 한 번, 파인튜닝 한 번** 돌려봤다.
    3. Spring AI/LangChain4j로 **Java 서비스에서 LLM을 소비하는 코드를 쓸 수 있다.**
    4. "파인튜닝 vs RAG" 같은 실무 판단에서 **본인 근거로** 선택할 수 있다.
    5. 다음 단계(에이전트·멀티모달·심화 파인튜닝)를 읽을 지도가 생긴다.

## 내러티브 아크

이 책은 "**왜 지금 혼란스러운가**"에서 시작해 "**그래서 이제 어디로 가면 되는가**"로 끝난다. 중간의 전진 동력은 **"개념 → 손으로 확인 → 다시 개념 → 더 큰 손 실습"**의 나선형, 그리고 4~8장을 관통하는 **하나의 공통 태스크**("사내 개발 문서 요약/Q&A")다.

**1장 (Why):** 독자의 현재 감정—"탭 40개", "그림 100번"—을 인정하는 데서 출발. 시중 자료가 난립하는 이유를 지도로 그리고(튜토리얼/시각/논문/코드 4계층), 이 책이 그 사이 어디를 달리는지 선언한다. "수학 먼저 vs 코드 먼저"(Reference 4.1) 논쟁을 여기서 열고, **"우리는 코드 먼저, 단 6장에서 수학으로 한 번 돌아온다"**를 약속한다. **"사내 개발 문서 요약/Q&A"를 공통 태스크로 예고** — 4·6·7·8장에서 같은 문제를 네 가지 해법으로 푼다는 지도를 먼저 보여준다.

**2장 (What—원리 1부: 토큰과 의미 벡터):** "토큰과 임베딩"으로 입문의 가장 실체 있는 층부터 만진다. `tiktoken`·`minbpe`로 한국어를 쪼개보며 **"birthplace = birth + place"**를 눈으로 확인. **"위치는 어떻게 넣는가?"라는 질문은 3장으로 넘긴다** — 2장은 순수하게 "의미 벡터" 직관에만 집중.

**3장 (What—원리 2부: 어텐션과 트랜스포머):** **Positional Encoding을 도입부에서 한 번에 정리한 뒤** 어텐션으로 진입. 시각적 비유(회의실/도서관 사서 Q/K/V)를 앞세우고, 코드 구현은 chunk 단위로 쪼개 "한 블록씩" 쌓는다. 3B1B/Alammar 자료를 본문과 직접 연결—"이 그림을 먼저 보고 오라"는 외부 포인터가 아니라 본문 안에서 함께 걷는다. encoder/decoder/enc-dec 3종 비교표는 유지하되 **BERT는 "검색·임베딩 생성에 지금도 쓰인다" 한 줄**만, 깊은 논의는 8장 RAG 임베딩 절(bge-m3)로 이관.

**4장 (How—실습 1부: From Scratch + 디코딩 루프):** 2·3장의 이해를 손으로 고정한다. **Karpathy "Let's build GPT" 영상/build-nanogpt 저장소**를 뼈대로 삼아 Bigram → 멀티헤드 어텐션 → 미니 트랜스포머를 한 챕터 안에서 쌓는다. nanoGPT deprecation(2025-11) 사실을 본문 박스로 고지하고 **nanochat/build-nanogpt로 실습 경로를 재연결**. 공통 태스크를 **축소판**으로 이식 — "사내 개발 문서"를 모사한 작은 한국어 코퍼스로 학습. 장 말미에 **"디코딩 루프" 절을 신설**(1.5~2K자): 같은 모델의 출력층에서 Greedy → Temperature → Top-k → Top-p를 한 장면 안에서 실행하고 결과를 나란히 비교. 이 경험이 5장의 "왜 Instruction Tuning이 필요한가"로도, 7장의 API 파라미터로도 이어지는 **이중 연결 고리**가 된다.

**5장 (What—원리 3부: 스케일과 정렬):** 미니 모델에서 "왜 큰 모델이 필요한가"로 자연 이행. Pretraining → Scaling Laws (Kaplan→Chinchilla→Llama의 3막 서사, Reference 2.2) → Emergent Ability → Instruction Tuning → RLHF → Constitutional AI. **"1.3B InstructGPT가 175B GPT-3를 이긴"** 사건을 서사의 절정으로. 4장에서 base 모델 sampling의 한계를 몸으로 봤기 때문에 "왜 정렬이 필요한가"가 이론이 아닌 체감으로 들어온다. 이 장은 이해 중심, 코드는 의사코드 위주.

**6장 (How—실습 2부: 파인튜닝):** 5장의 원리를 **오픈 모델에 얹는다**. 공통 태스크를 이번엔 **Llama 3 8B QLoRA**로. HuggingFace `transformers`+`peft`+`trl`+`bitsandbytes`를 조합. [웹:SK-DevOcean]·[웹:unfinishedgod] 실전 기록을 디딤돌로. 4장이 "from scratch 교육용"이었다면 6장은 **현업 기본기**—VRAM 가이드라인과 Colab/Unsloth 대안을 정면으로.

**7장 (How—실습 3부: 소비·서비스 연결):** 지금까지는 "모델을 만든다" 관점. 이 장에서 "모델을 쓴다" 관점으로 회전한다. 공통 태스크를 **API 호출 한 줄**로 같은 모양으로 풀어본다. 4장에서 손으로 본 디코딩 루프가 API 파라미터로 어떻게 매핑되는지(`temperature`, `top_p`, `max_tokens`, `finish_reason`) 이어 받는다. **"Temperature=0인데도 답이 다르다"**(Reference 6.5)의 당혹감을 정면으로 — GPU 부동소수점 비결정성, 배치 효과. Python(LangChain/LlamaIndex) 예제 후, **같은 기능을 Spring AI `ChatClient`와 LangChain4j로 재현**—provider 스왑을 Maven 의존성 한 줄로. 이 장이 Java 독자에게 "이 책을 사길 잘했다"를 느끼게 하는 지점.

**8장 (How—실습 4부: RAG 최소 구현):** 공통 태스크의 마지막 해법. "파인튜닝 vs RAG" 판단 체크리스트를 펴고, **실무 1원칙 "RAG 먼저"**(Reference 4.2)를 실습으로 보여준다. 임베딩 → 벡터 검색(pgvector 또는 Chroma) → 컨텍스트 주입 → 출처 인용. Python으로 최소 파이프라인을 만든 후 Spring AI VectorStore로 이식. 3장에서 미룬 BERT 계열 임베딩 모델(`bge-m3` 등) 논의가 여기서 본격화.

**9장 (Synthesis):** 2~8장에서 만든 부분들을 통합 사고로 묶는다. **"첫 직감이 fine-tune이면 red flag"**(Reference 5.4)를 실무 의사결정 플로차트로 정리. 네 해법이 **같은 공통 태스크의 서로 다른 답**이었다는 감각이 플로차트의 최강 근거. 관측성(Observability-first, Reference 5.6), 비용, 지연, 평가(evaluation harness)를 한 자리에. 한국어 실패담 케이스 스터디와 거버넌스·red-teaming은 **"집필 제출 조건"**(아래 "보완 리서치 제출 조건" 절 참조).

**10장 (Next):** 이 책 다음을 안내한다. 에이전트·멀티모달·심화 FT·추론 최적화·긴 컨텍스트·평가·안전 6개 분기를 **지도**로. 각 분기마다 (a) 왜 다음 단계인가, (b) 어떤 문제를 풀려고 나왔나, (c) 최소 재현 난이도 + 추천 1차 자료. **"논문 한 편 혼자 읽기" 1.5페이지 팁 박스**를 저자의 편지 앞에 흡수. 에필로그 톤.

**서문/프롤로그:** "끝이 안 보인다"(카카오페이) 인용 + "수년 동안 배회했다… fast.ai part 1 후 it clicked"(Reference 6.8) 인용으로 연다. **"이 책은 이렇게 읽는다"** 2페이지 가이드를 서문 말미에 포함 — 독자 유형별(원리파/실습파/Java 서비스파) 읽기 경로.

- **독자가 얻는 것:** 현재 자기 위치를 지도 위에 찍을 수 있는 좌표. "내가 이상한 게 아니었구나" 안도.

### 2장. 토큰과 임베딩 — 블랙박스의 첫 층
- **핵심 질문:** LLM은 단어를 어떻게 "숫자"로 바꾸는가? 왜 "birthplace"가 "birth + place"로 쪼개지는가? 한국어는?
- **주요 내용:**
  - 문자·단어·서브워드 토큰화 비교. BPE/WordPiece/SentencePiece 계보 (Reference 1.1)

### 3장. 어텐션과 트랜스포머 — 그림이 아니라 이야기로
- **핵심 질문:** 위치 정보는 어떻게 들어가는가? 어텐션이 도대체 무엇을, 어떻게 보고 있는가? 트랜스포머 박스 그림을 해체하면 뭐가 남는가?
- **주요 내용:**
  - **Positional Encoding 도입부에서 일괄 정리** — 사인/코사인 주기 함수, RNN과 달리 순서가 자동으로 안 들어오는 이유, "왜 더하는가 vs concat인가" 한 줄 (Reference 1.2)
  - **Q/K/V 스토리:** "질문을 던진다(Query) → 간판을 본다(Key) → 해당 선반의 내용을 가져온다(Value)" 도서관/회의실 비유 (Reference 1.3)
  - **3박자 서술:** ① 시각 비유 → ② 수식 한 줄(Scaled Dot-Product Attention) → ③ PyTorch 10줄 구현

  - Transformer 아키텍처 3종(encoder-only/decoder-only/enc-dec) 비교표 (Reference 1.4) — **현대 LLM이 왜 decoder-only인지** 명시. **BERT는 "검색·임베딩에 지금도 쓰인다" 한 줄**로만, 깊은 논의는 8장으로 이관
  - Vaswani 2017 탄생 서사 (Reference 2.1) — "RNN 지우고 Attention만" 왜 병렬화 혁명이었는지. Java `Stream.parallelStream` 비유
  - **원문 자료 연결 박스:** 본문 읽고 나서 3Blue1Brown Ch.5-6 → Illustrated Transformer 한국어판 순 권장 (숙제 아님, 보너스)
- **독자가 얻는 것:** 트랜스포머 그림을 보면 "저기 Q·K·V가 있고 저기 레지듀얼이 있구나"를 **말로 풀 수 있음.**

## 핵심 설계 결정 요약

### 1. "간단한 모델을 만들면서 실습" 해석 — **A안 + B안 조합 (Hybrid)**

- **4장 = A안** (nanoGPT/nanochat/build-nanogpt 스타일 **from scratch**). 교육적 가치 극대화, "LLM이 조건부 확률 + 샘플링"임을 몸으로 체감하는 각성 챕터. **디코딩 루프 절**을 말미에 두어 7장으로의 다리도 겸한다.
- **6장 = B안** (HuggingFace 기반 오픈 모델 QLoRA **파인튜닝**). 현업 기본기, 실무에서 실제로 하게 되는 일에 근접.
- **C안 (Llama/Mistral 로컬 실행)은 채택하지 않음**. 대신 7장 Spring AI의 Ollama 스왑 데모와 부록 A 세팅 가이드에 녹여 넣음.
- 근거: 커뮤니티 논쟁 4.6("작은 모델 직접 훈련 vs 오픈소스 파인튜닝") 현장 합의가 **"교육용 nanoGPT + 실무용 파인튜닝 병행"**. 이 책은 그 합의를 그대로 채택.
- nanoGPT deprecation(2025-11) 사실을 4장 도입 박스로 명시. `nanochat`/`build-nanogpt` 저장소를 실습 저장소로 채택.

### 2. 4장 ↔ 5장 순서 — **현행(4→5) 유지**

- **근거 (계획 문서에 명기):**
  - (a) 서문의 "코드 먼저" 선언과의 일관성. 5→4로 뒤집으면 bottom-up이 되어 **저자의 명시적 약속과 정면 충돌**한다.
  - (b) 4장에서 Bigram → Self-Attention → Transformer의 학습 곡선을 본 독자는 5장 Scaling Laws의 "N·D·C 거듭제곱"을 **그래프가 아닌 경험**으로 읽는다.
  - (c) **"1.3B가 175B를 이긴" 감정 피크**가 5장에 오려면 독자가 "1.3B를 내 손으로 만드는 게 이런 거구나"를 이미 체감했어야 한다. 역전하면 감정 피크가 무뎌진다.
- 더하여 v2에서 4장 말미 "디코딩 루프"가 신설되면서, base 모델 샘플링의 한계를 5장 Instruction Tuning으로 이어주는 **감각적 다리**가 하나 더 생겼다.

### 3. Python vs Java 비율 — **Python 75% / Java 25%, Java는 두 지점에 집중**

- **주 언어 Python:** 2·3·4·5·6·8장 실습 전부 Python.
- **Java 전면 등장 2개 지점:**
  - **7장(서비스 연결):** Spring AI + LangChain4j로 **본격 Java 실습**. Python 먼저 쓰고 같은 기능을 Java로 재작성하는 2단 구조. 이 장은 Java 비중 50%+.
  - **8장(RAG) 후반:** Spring AI `VectorStore`로 Python 파이프라인 이식. Java 비중 30%.
- **기타 장의 Java:** "Java로 치면 ~" 비유 박스만 (2장 Map 비유, 3장 Stream.parallelStream 비유 등).
- 근거: 대상 독자가 "Java·Python 둘 다 다루는 중급 개발자"이고 리서치 2.5·4.3·5.5가 **"학습·연구는 Python, 서비스 통합은 Java"** 현장 합의를 지지.

### 4. GPU 없는 독자 대응 — **실습 챕터마다 대안 경로 박스 의무화**

- **4장, 6장, 7장(로컬 모델 절), 8장(임베딩 생성 절)** 말미에 **"GPU 없을 때"** 박스.
- 4장: Colab 무료 T4 / Kaggle Notebook / M-series MPS
- 6장: Colab T4 + Unsloth + QLoRA / Kaggle 2×T4 / HF AutoTrain Spaces
- 7장: Ollama (Mac M 시리즈 8GB VRAM으로 Llama 3.1 8B) + API 대안
- 8장: 임베딩은 API(OpenAI `text-embedding-3-small`) 또는 Ollama `bge-m3` 대안
- 근거: Reference 5.2, 6.6. "하드웨어 탓하지 말고 무료 티어로 한 사이클"이 현장 규범.

### 5. 챕터 수 — **본문 10장 + 서문 + 부록 3개**

- 부록 D(논문 읽는 법) 제거 후 A/B/C만 남김. 책 본문에서 "논문을 직접 읽어야 한다"고 요구하는 챕터가 없으므로 별도 부록의 독자 수요가 약했다. 10장 "저자의 편지" 앞 1.5페이지 박스로 흡수.
- 실습 4개(4·6·7·8)와 원리 3개(2·3·5)의 교차 배치가 리드미컬: **원리 → 실습 → 원리 → 실습 → 원리 → 실습 → 실습 → 통합 → 다음**. 6·7·8 연속 실습의 피로도는 **장별 "지도" + 7장 내장 콜아웃 + 8장 말미 콜아웃**(핵심 설계 6번 참조)으로 흡수.

### 6. 연속 실습 피로도 해법 — **세 장치, 별도 쉼터 챕터 반대**

저자 질문 Q1에 대한 답. 6·7·8 연속 실습의 진짜 문제는 분량이 아니라 **관점 회전이 장 안에 숨어 있다는 점**이다. 별도 쉼터 챕터를 끼우면 10장짜리 책의 목차 리듬이 어그러진다. 대신 세 장치를 내장한다.

- **(a) 각 장 도입부 "지금 우리가 어디에 있나" 1페이지 지도:** 2~8장 전체 여정 도식에 현재 장을 하이라이트. 4·6·7·8장 각각 상단에 동일 포맷으로 배치.
- **(b) 7장 중간 "지금까지 만든 것 vs 지금부터 쓰는 것" 2페이지 회고 콜아웃:** 4·6장의 "만드는 궤적"과 7장부터의 "쓰는 궤적"을 한 장면에 겹친다. 관점 회전을 명시적으로.
- **(c) 8장 말미 "파인튜닝-RAG-프롬프트 3각을 한 장면에 겹쳐보기" 콜아웃:** 6·7·8의 세 해법을 같은 공통 태스크의 세 축으로 시각화. 9장 플로차트로의 다리.

7장·8장 순서 교체(Q1-b)는 반대 — RAG가 API 응답 구조·벡터 DB 소비를 전제하므로 7장 선행이 필연.

### 7. 공통 실습 태스크 — **"사내 개발 문서 요약/Q&A"**

4장(축소판 from scratch) → 6장(Llama 3 8B QLoRA) → 7장(API) → 8장(RAG)이 **같은 문제에 대한 네 가지 해법**. 9장 결정 플로차트는 이 감각을 근거로 삼는다.

- **4장:** 공통 태스크를 모사한 작은 한국어 코퍼스(**모두의 말뭉치 현대어 일부 + 나무위키 덤프 소량, 1~3MB, 저장소 README에 다운로드·전처리 스크립트**. 세종실록은 고어·한자 난이도 때문에 제외)로 from scratch 학습.
- **6장:** 같은 태스크를 QLoRA로 Llama 3 8B에 얹기.
- **7장:** 같은 태스크를 API 호출로.
- **8장:** 같은 태스크를 RAG 파이프라인으로.

이 연속성을 각 장 "주요 내용"에 명시했다. 네 가지 해법의 출력 품질·비용·구현 난이도 비교가 9장의 가장 강력한 근거가 된다.

### 8. 문체 3층 구획 원칙 (신설)

Q5 답변. 토비 문체 일관성과 실습 장의 명료성을 동시에 살리기 위한 구획.

- **(1) 장 도입·전환 문단 = 청유형.** "같이 돌려봅시다", "여기서 한 번 확인해봅시다". 독자와 같이 걷는 감각.
- **(2) 코드 블록 바로 앞 1~2문단 = 평어 서술.** "아래 스크립트는 토크나이저를 한국어 텍스트에 학습시킨다." 명료·담담.
- **(3) 코드 주석·에러 해설·터미널 로그 설명 = 매뉴얼체.** "이 오류는 디바이스 이동이 누락됐을 때 발생한다." 불필요한 수사 없음.
- **(4) 장 말미 "독자가 얻는 것" = 다시 청유형.** 수미상관으로 장을 닫는다.

실습 4개 장(4·6·7·8)의 "주요 내용" 목록에 **"문체 3층 구획 적용"**을 명시했다. 1·5·9·10장·서문은 원래 청유형 중심이라 별도 표기 없이 유지.

추가 효과: 외부 한국어 자료 인용이 부족해도 저자 본인의 문체로 메워지는 구조가 만들어진다. 공감 인용은 서문·1·9·10장의 청유형 레이어에 집중 배치(보완 리서치 ③의 5~10건), 실습 장은 저자 체험담과 3층 구획으로 충당.
