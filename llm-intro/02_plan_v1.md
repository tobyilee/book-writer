## 책 특성

- **장르:** 기술 입문서 (원리 50% + 실습 40% + 에세이형 내러티브 10%). 순수 실습서도 순수 이론서도 아님. 현장 합의인 **"튜토리얼 → 그림 → 논문 → 코드 재구현"의 4단 파이프라인**(Reference 5.1)을 한 권에 압축.
- **분량:** 전체 본문 약 **220,000~260,000자 (공백 포함, A5 기준 약 360~420쪽)**. 서문·에필로그·부록 포함 시 420~480쪽.
  - 의미 있는 깊이 우선. 다만 "구현 코드 통째 복붙"으로 분량 부풀리지 않음. 긴 코드는 핵심 20~40줄만 본문, 전체는 저장소 링크.
- **난이도:** **중급 (선수 지식: Python 기본 문법, 학부 선형대수·미분 기초, HTTP API 호출 경험, Java Spring 기초).** 첫 장에서 선수 지식 자가 체크 박스 제공.
- **독자 여정:**
  - **진입 상태:** "ChatGPT/Claude는 써봤지만 내부는 블랙박스. 트랜스포머 그림을 10번 봐도 감이 안 오고, 공부를 시작하기도 전에 **어디서부터 뭘 해야 할지 몰라서 지쳐 있다.**"
  - **중간 각성:** 4장 말미 — 직접 짠 100줄짜리 Bigram/Mini-Transformer가 한국어 토큰을 이어붙이는 걸 보고 "LLM이 결국 조건부 확률이구나"를 몸으로 체감.
  - **출구 상태:**
    1. 토큰 → 임베딩 → 어텐션 → 디코딩의 파이프라인을 **자기 언어로** 설명할 수 있다.
    2. nanochat/build-nanogpt 스타일로 미니 모델을 **from scratch 한 번, 파인튜닝 한 번** 돌려봤다.
    3. Spring AI/LangChain4j로 **Java 서비스에서 LLM을 소비하는 코드를 쓸 수 있다.**
    4. "파인튜닝 vs RAG" 같은 실무 판단에서 **본인 근거로** 선택할 수 있다.
    5. 다음 단계(에이전트·멀티모달·심화 파인튜닝)를 읽을 지도가 생긴다.

## 내러티브 아크

이 책은 "**왜 지금 혼란스러운가**"에서 시작해 "**그래서 이제 어디로 가면 되는가**"로 끝난다. 중간의 전진 동력은 **"개념 → 손으로 확인 → 다시 개념 → 더 큰 손 실습"**의 나선형.

**1장 (Why):** 독자의 현재 감정—"탭 40개", "그림 100번"—을 인정하는 데서 출발. 시중 자료가 난립하는 이유를 지도로 그리고(튜토리얼/시각/논문/코드 4계층), 이 책이 그 사이 어디를 달리는지 선언한다. "수학 먼저 vs 코드 먼저"(Reference 4.1) 논쟁을 여기서 열고, **"우리는 코드 먼저, 단 6장에서 수학으로 한 번 돌아온다"**를 약속한다.

**2~3장 (What—원리 1부: 블랙박스 열기):** 2장은 "토큰과 임베딩"으로 입문의 가장 실체 있는 층부터 만진다. `tiktoken`·`minbpe`로 한국어를 쪼개보며 **"birthplace = birth + place"**를 눈으로 확인. 3장은 트랜스포머·어텐션. 시각적 비유(회의실/도서관 사서 Q/K/V)를 앞세우고, 코드 구현은 chunk 단위로 쪼개 "한 블록씩" 쌓는다. **3B1B/Alammar 자료를 본문과 직접 연결**—"이 그림을 먼저 보고 오라"는 외부 포인터가 아니라 본문 안에서 함께 걷는다.

**4장 (How—실습 1부: From Scratch):** 2·3장의 이해를 손으로 고정한다. **Karpathy "Let's build GPT" 영상/build-nanogpt 저장소**를 뼈대로 삼아 Bigram → 멀티헤드 어텐션 → 미니 트랜스포머를 한 챕터 안에서 쌓는다. nanoGPT deprecation(2025-11) 사실을 본문 ⚠️ 박스로 고지하고 **nanochat/build-nanogpt로 실습 경로를 재연결**. 한국어 셰익스피어(세종실록/나무위키 덤프 일부)로 학습하여 한국어 토큰이 이어지는 걸 본다. GPU 없는 독자용 **Colab T4 / Kaggle / M-series MPS 경로**를 챕터 말미 박스로.

**5장 (What—원리 2부: 스케일과 정렬):** 미니 모델에서 "왜 큰 모델이 필요한가"로 자연 이행. Pretraining → Scaling Laws (Kaplan→Chinchilla→Llama의 3막 서사, Reference 2.2) → Emergent Ability → Instruction Tuning → RLHF → Constitutional AI. **"1.3B InstructGPT가 175B GPT-3를 이긴"** 사건을 서사의 절정으로. 이 장은 이해 중심, 코드는 의사코드 위주.

**6장 (How—실습 2부: 파인튜닝):** 5장의 원리를 **오픈 모델에 얹는다**. HuggingFace `transformers`+`peft`+`trl`+`bitsandbytes`로 **Gemma 또는 Llama 3 8B를 한국어 요약/톤 변환 태스크로 QLoRA 파인튜닝**. [웹:SK-DevOcean]·[웹:unfinishedgod] 실전 기록을 디딤돌로. 4장이 "from scratch 교육용"이었다면 6장은 **현업 기본기**—VRAM 가이드라인과 Colab/Unsloth 대안을 정면으로.

**7장 (How—실습 3부: 소비·서비스 연결):** 지금까지는 "모델을 만든다" 관점. 이 장에서 "모델을 쓴다" 관점으로 회전한다. OpenAI/Anthropic API의 파라미터(`temperature`, `top_p`, `max_tokens`, `finish_reason`)를 **백엔드 함수 호출로** 다룬다. **"Temperature=0인데도 답이 다르다"**(Reference 6.5)의 당혹감을 정면으로 — GPU 부동소수점 비결정성, 배치 효과(Thinking Machines Lab). Python(LangChain/LlamaIndex) 예제 후, **같은 기능을 Spring AI `ChatClient`와 LangChain4j로 재현**—provider 스왑을 Maven 의존성 한 줄로. 이 장이 Java 독자에게 "이 책을 사길 잘했다"를 느끼게 하는 지점.

**8장 (How—실습 4부: RAG 최소 구현):** "파인튜닝 vs RAG" 판단 체크리스트를 펴고, **실무 1원칙 "RAG 먼저"**(Reference 4.2)를 실습으로 보여준다. 임베딩 → 벡터 검색(pgvector 또는 Chroma) → 컨텍스트 주입 → 출처 인용. Python으로 최소 파이프라인을 만든 후 Spring AI VectorStore로 이식. **이 챕터는 리서치 공백 영역(Reference 8.3)이므로 "보완 리서치 필요" 마커 포함**.

**9장 (Synthesis):** 2~8장에서 만든 부분들을 통합 사고로 묶는다. **"첫 직감이 fine-tune이면 red flag"**(Reference 5.4)를 실무 의사결정 플로차트로 정리. 관측성(Observability-first, Reference 5.6), 비용, 지연, 평가(evaluation harness)를 한 자리에. 이 장은 "어떤 프로젝트를 받으면 이 책의 뭘 꺼내 쓸지" 역인덱스 역할.

**10장 (Next):** 이 책 다음을 안내한다. 에이전트(function calling, tool use), 멀티모달, 심화 파인튜닝(DPO, ORPO, 전체 파인튜닝), MoE, 긴 컨텍스트, 로컬 추론 엔진(vLLM, SGLang, llama.cpp)을 **지도**로. 각 분기마다 **3줄 소개 + 추천 1차 자료 1건**. 에필로그 톤.

**서문/프롤로그:** "끝이 안 보인다"(카카오페이) 인용 + "수년 동안 배회했다… fast.ai part 1 후 it clicked"(Reference 6.8) 인용으로 연다. **"이 책은 이렇게 읽는다"** 2페이지 가이드를 서문 말미에 포함 — 독자 유형별(원리파/실습파/Java 서비스파) 읽기 경로.

**부록:**
- A. 환경 세팅 (Python 3.11+ venv/uv, CUDA/MPS 체크, Ollama 설치, Spring Boot 프로젝트 스캐폴딩)
- B. 용어집 (한국어·영문 병기, 1·5장 용어 40여 개)
- C. 더 읽을거리 (참고문헌 7.1~7.6 재편성, 난이도 태그 병기)
- D. (선택) 논문 빠르게 읽는 법 — "논문 하나 읽는 데 3시간"(Reference 6.7)의 응답으로

  - **논쟁 3선 개봉:** (a) 수학 vs 코드, (b) 파인튜닝 vs RAG, (c) Python vs Java — 각 논쟁의 저자 입장을 한 줄로 선언하고, 본격 논의는 각 장에서

- **독자가 얻는 것:** 현재 자기 위치를 지도 위에 찍을 수 있는 좌표. "내가 이상한 게 아니었구나" 안도.

### 2장. 토큰과 임베딩 — 블랙박스의 첫 층
- **핵심 질문:** LLM은 단어를 어떻게 "숫자"로 바꾸는가? 왜 "birthplace"가 "birth + place"로 쪼개지는가? 한국어는?
- **주요 내용:**
  - 문자·단어·서브워드 토큰화 비교. BPE/WordPiece/SentencePiece 계보 (Reference 1.1)
  - **손 실습 1:** `tiktoken`으로 영어/한국어 문장 토큰화 관찰 — 한국어 교착어 특성이 토큰 수에 미치는 영향을 눈으로 확인
  - **손 실습 2:** Karpathy `minbpe`의 BasicTokenizer를 한국어 텍스트에 훈련시켜보기 (10~20줄)
  - 임베딩의 직관: "의미가 가까우면 벡터도 가깝다"를 `gensim` 또는 HF `sentence-transformers` 시연 (유사도 Top-K)
  - Positional Encoding 도입 이유와 수식 감각 (사인/코사인 — "왜 이게 등장했나만" 먼저, 유도는 3장에서)
  - **Java 비유 박스:** "토큰 ID는 `Map<String, Integer>`의 value, 임베딩은 `float[768]` row"
- **독자가 얻는 것:** LLM이 사람 말을 어떻게 받아먹는지에 대한 첫 체감. 한국어 토큰 효율 문제의 실감.
- **예상 분량:** 20,000자

### 3장. 어텐션과 트랜스포머 — 그림이 아니라 이야기로
- **핵심 질문:** 어텐션이 도대체 무엇을, 어떻게 보고 있는가? 트랜스포머 박스 그림을 해체하면 뭐가 남는가?
- **주요 내용:**
  - **Q/K/V 스토리:** "질문을 던진다(Query) → 간판을 본다(Key) → 해당 선반의 내용을 가져온다(Value)" 도서관/회의실 비유 (Reference 1.3)
  - **3박자 서술:** ① 시각 비유 → ② 수식 한 줄(Scaled Dot-Product Attention) → ③ PyTorch 10줄 구현
  - Multi-Head Attention을 "여러 시점으로 동시에 본다"로 설명, 헤드별 역할 분화 직관
  - Transformer 아키텍처 전체를 encoder-only/decoder-only/enc-dec 3종으로 나눠 (Reference 1.4) — **현대 LLM이 왜 decoder-only인지** 명시
  - Vaswani 2017 탄생 서사 (Reference 2.1) — "RNN 지우고 Attention만" 왜 병렬화 혁명이었는지. Java `Stream.parallelStream` 비유
  - **원문 자료 연결 박스:** 본문 읽고 나서 3Blue1Brown Ch.5-6 → Illustrated Transformer 한국어판 순 권장 (숙제 아님, 보너스)
- **독자가 얻는 것:** 트랜스포머 그림을 보면 "저기 Q·K·V가 있고 저기 레지듀얼이 있구나"를 **말로 풀 수 있음.**
- **예상 분량:** 28,000자

## 핵심 설계 결정 요약

### 1. "간단한 모델을 만들면서 실습" 해석 — **A안 + B안 조합 (Hybrid)**

- **4장 = A안** (nanoGPT/nanochat/build-nanogpt 스타일 **from scratch**). 교육적 가치 극대화, "LLM이 조건부 확률"임을 몸으로 체감하는 각성 챕터.
- **6장 = B안** (HuggingFace 기반 오픈 모델 QLoRA **파인튜닝**). 현업 기본기, 실무에서 실제로 하게 되는 일에 근접.
- **C안 (Llama/Mistral 로컬 실행)은 채택하지 않음**. 대신 7장 Spring AI의 Ollama 스왑 데모와 부록 A 세팅 가이드에 녹여 넣음. **독립 챕터화하기엔 C안은 "다른 챕터의 연장"에 가깝다**는 판단.
- 근거: 커뮤니티 논쟁 4.6("작은 모델 직접 훈련 vs 오픈소스 파인튜닝") 현장 합의가 **"교육용 nanoGPT + 실무용 파인튜닝 병행"**. 이 책은 그 합의를 그대로 채택.
- ⚠️ **nanoGPT deprecation(2025-11)** 사실을 4장 도입 박스로 명시. `nanochat`/`build-nanogpt` 저장소를 실습 저장소로 채택.

### 2. Python vs Java 비율 — **Python 75% / Java 25%, Java는 두 지점에 집중**

- **주 언어 Python:** 2·3·4·5·6·8장 실습 전부 Python.
- **Java 전면 등장 2개 지점:**
  - **7장(서비스 연결):** Spring AI + LangChain4j로 **본격 Java 실습**. Python 먼저 쓰고 같은 기능을 Java로 재작성하는 2단 구조. 이 장은 Java 비중 50%+.
  - **8장(RAG) 후반:** Spring AI `VectorStore`로 Python 파이프라인 이식. Java 비중 30%.
- **기타 장의 Java:** "Java로 치면 ~" 비유 박스만 (2장 Map 비유, 3장 Stream.parallelStream 비유 등).
- 근거: 대상 독자가 "Java·Python 둘 다 다루는 중급 개발자"이고 리서치 2.5·4.3·5.5가 **"학습·연구는 Python, 서비스 통합은 Java"** 현장 합의를 지지.

### 3. GPU 없는 독자 대응 — **실습 챕터마다 대안 경로 박스 의무화**

- **4장, 6장, 7장(로컬 모델 절), 8장(임베딩 생성 절)** 말미에 **"GPU 없을 때"** 박스.
- 4장: Colab 무료 T4 / Kaggle Notebook / M-series MPS
- 6장: Colab T4 + Unsloth + QLoRA / Kaggle 2×T4 / HF AutoTrain Spaces
- 7장: Ollama (Mac M 시리즈 8GB VRAM으로 Llama 3.1 8B) + API 대안
- 8장: 임베딩은 API(OpenAI `text-embedding-3-small`) 또는 Ollama `bge-m3` 대안
- 근거: Reference 5.2, 6.6. "하드웨어 탓하지 말고 무료 티어로 한 사이클"이 현장 규범.

### 4. 챕터 수 — **본문 10장 + 서문 + 부록 4개**

- 5~10개 가이드라인의 **상한에 근접**. 주제 폭(토큰·어텐션·트랜스포머·스케일링·정렬·파인튜닝·API·RAG·통합·다음)이 넓어 9장 이하로 압축하면 각 장이 과밀해지고, 11장 이상으로 늘리면 독자 호흡이 흐트러진다.

## 리뷰어에게 던지는 질문 (plan-reviewer 집중 검토 요청)

### Q1. 7·8장 연속 실습의 독자 피로도
**이슈:** 6장(파인튜닝) → 7장(API+Java) → 8장(RAG) 3개 장이 연속 실습이며, 각 장이 26K~34K자로 무겁다. 독자가 7·8장을 한 번에 읽기 힘들 수 있다.
**검토 요청:** (a) 7·8장 사이에 짧은 "호흡 챕터"를 넣을지, (b) 7장과 8장 순서를 바꿔 "가벼운 RAG 먼저, 무거운 Java 서비스 연결 나중"으로 할지, (c) 현재 안대로 두되 두 장 사이 1~2페이지 "쉼터"만 둘지.

### Q2. 5장(스케일·정렬) vs 4장(from scratch) 순서
**이슈:** 현재 **4장(실습 from scratch) → 5장(원리 사전학습/RLHF)** 순서. 논리적으로는 "사전학습 원리를 먼저 알고 미니 모델을 돌리는 게" 자연스러울 수 있다. 반대 근거는 "작은 모델 돌려본 경험이 있어야 5장 Scaling Laws의 감이 온다".
**검토 요청:** 4장과 5장의 순서 역전(5→4)이 학습 곡선에 더 좋을지, 현재(4→5) 실습-각성-이론심화 구조가 더 좋을지. 토비 문체("청유형 같이 걸어봅시다")와 어느 쪽이 맞는지도.

### Q3. 4장의 한국어 코퍼스 선택
**이슈:** 현재 "세종실록/나무위키 덤프 일부"로 제안. 저작권·토큰 효율·학습 안정성 측면에서 (a) 영어 셰익스피어만(Karpathy 원형) 유지하고 한국어는 예시로만 짧게, (b) 공개 한국어 코퍼스(AI Hub 국립국어원 모두의 말뭉치) 일부, (c) 저자가 만든 작은 한국어 텍스트(에세이/블로그) 선택지가 있다.
**검토 요청:** 어느 안이 "독자 재현 가능성 + 한국어 체감 + 법적 안전성"의 균형에 맞는지.

### Q4. 부록 D(논문 읽는 법)의 포함 여부
**이슈:** Reference 6.7 "3시간 동안 논문 읽기" 원석을 활용하려 부록 D를 제안했으나, **이 책의 범위를 넘어서는 주제**일 수 있다. 10장 "다음 걸음"에 1페이지 팁으로 흡수하거나, 완전히 빼는 선택지.
**검토 요청:** 부록 D 유지/흡수/제거 중 어느 쪽이 책의 스코프 규율에 맞는지.

### Q5. 책 전체에서 "Toby-AI" 저자 톤과 실습 챕터의 마찰
**이슈:** 1·5·9·10장과 서문은 토비 문체(청유형·평어체·수사적 질문)가 자연스럽다. 하지만 4·6·7·8장처럼 **코드가 본문의 60%를 차지하는 장**에서는 청유형이 어색해질 수 있다. "같이 해봅시다" → `pip install`의 전환이 부자연스러울 위험.
**검토 요청:** 실습 챕터에서 문체를 (a) 동일하게 유지, (b) 코드 블록 앞뒤는 평어 서술/코드 블록은 담담한 매뉴얼체로 구획, (c) 장마다 톤 비율을 다르게 허용 중 어떤 원칙이 적절한지. 토비 문체 일관성 vs 실습 명료성의 균형점.
