# 맨손으로 짓는 AI 에이전트

> 같은 에이전트를 세 가지 추상화 레벨로 세 번 다시 짓는다. 그러면 추상이 무엇을 대신해 주는지 비로소 보인다.

- **저자:** Toby-AI
- **부제:** Python SDK에서 LangChain, LangGraph까지 — 같은 에이전트를 세 번 다시 짓는다
- **버전:** v1.0.0
- **발행일:** 2026-05-16
- **언어:** 한국어 (ko)
- **분량:** 본문 약 12만 5천 자 (14장 + 부록 3편, 코드 블록 별도)
- **라이선스:** CC BY-NC-SA 4.0 (저작자 표시·비상업적 이용·동일조건 변경허락)

## 이 책은 무엇인가

2026년의 "AI 에이전트"는 더 이상 안개 속 단어가 아니다. Simon Willison은 *"agent라는 단어가 마침내 합의에 가까운 정의를 가지게 되었다"*고 적었고, 그 정의는 단순하다 — **LLM이 도구를 부르는 루프**. 이 책은 그 한 줄을 손으로 만져 볼 수 있는 코드로 옮기는 중급 실전 가이드다.

특이한 점은 같은 에이전트를 *세 번* 다시 짓는다는 것이다. Part 1에서 OpenAI/Anthropic SDK만 가지고 ReAct 루프·메모리 잘라내기·무한 루프 방어·비용 캡까지 약 200줄짜리 `mini_agent.py`를 손으로 짠다. Part 2에서 같은 에이전트를 LangChain과 LCEL로 다시 짠다 — 줄어든 줄 수와 늘어난 추상의 깊이를 같이 측정한다. Part 3에서 LangGraph로 또 짓는다. "재시도 한 번"이라는 한 줄짜리 요구사항이 LCEL에서 어떻게 막히고 그래프에서 어떻게 풀리는지를 손으로 겪는다.

왜 세 번이나 짓느냐고 물을지 모르겠다. *추상이 무엇을 대신해 주는지 알려면 한 번은 자기가 짤 코드를 직접 봐야 하기 때문*이다. LangChain 튜토리얼을 따라가도 동작하는 챗봇 하나는 금세 만든다. 그런데 그 챗봇이 어느 날 멈추거나, 이상한 답을 하거나, 청구서가 갑자기 늘었을 때 안을 들여다보지 못한다. 라이브러리 코드를 추적해 보면 결국 그 안에서 만나는 건 Part 1에서 짤 *바로 그 코드*다. 이왕 만날 거라면, 라이브러리 안에서가 아니라 자기가 짠 `mini_agent.py` 안에서 먼저 만나두는 편이 낫다.

이 책은 입장도 가진다. LangChain은 "골라 쓰는 라이브러리이지 통째로 삼키는 종교가 아니"고, 에이전트는 "자율성이 비용이라는 사실을 안 사람만이 제대로 짤 수 있"으며, 멀티 에이전트는 회의적으로 보되 필요한 자리에서는 LangGraph의 supervisor 패턴이 답이다. 6장에서는 *"$0.08짜리 작업이 4시간 무한 루프로 $2,847이 된 실화"*를 통해 비용 캡과 무한 루프 방어가 왜 6장 분량을 받아야 하는지를, 14장에서는 Willison의 **lethal trifecta**(사적 데이터 + 신뢰할 수 없는 콘텐츠 + 외부 통신)를 통해 보안이 왜 마무리 챕터의 절반을 차지하는지를 정직하게 다룬다.

## 누구를 위한 책인가

- **Python을 알고 LLM 기초가 있는 개발자.** 프롬프트를 한 번 짜 본 적이 있고, OpenAI 또는 Anthropic API를 한 번 호출해 본 적이 있다면 충분하다.
- **에이전트는 처음 만드는 사람.** 1장에서 "에이전트가 무엇인지"부터 못 박고 들어간다. 한 번도 LangChain을 써 본 적이 없어도 좋다 — 오히려 7장의 비교가 더 충격적일 것이다.
- **LangChain은 써 봤지만 왜 그렇게 생겼는지 모르는 사람.** Part 1을 끝내고 7장에 도착하면 *어디서 보일러플레이트가 줄고 어디서 추상이 늘어났는지*가 손에 잡힌다.
- **LangChain 비판도, 옹호도 들어 본 사람.** 9장에서 6가지 비판(추상화 과잉·breaking change·로깅 부재 등)을 정면으로 다루고, Hamel Husain의 "골라 쓰기" 입장을 책의 입장으로 채택한다.

책을 덮을 즈음, 독자는 손에 세 개의 `mini_agent`를 쥐고 있다 — v3 정본, `mini_agent_lc`, `mini_agent_lg`. 셋 다 자기 프로젝트의 출발점으로 쓰면 된다.

## 무엇을 얻게 되는가

- **에이전트의 작업 정의 한 줄을 명확히 잡는다.** "LLM이 도구를 부르는 루프" — 1장에서 못 박고 책 전체가 이 정의 위에서 굴러간다.
- **약 200줄짜리 `mini_agent.py`를 손으로 짠다.** SDK 한 줄 호출에서 출발해 도구 호출, ReAct 루프, 메모리 관리, 무한 루프/컨텍스트 폭주/에러 처리/비용 캡까지 차곡차곡 쌓는다.
- **ReAct를 넘어선 루프 변종을 안다.** Plan-and-Solve, Reflexion, Tree of Thoughts — 언제 무엇을 꺼내야 하는지의 기준이 생긴다.
- **LangChain의 가치와 비용을 측정 가능한 좌표로 본다.** 줄 수가 아닌 import 그래프 깊이, 추상의 단 수, 부작용으로 비교한다.
- **LangGraph가 *왜* 필요한지를 손으로 겪는다.** "검색 결과가 비면 재시도 한 번"이라는 단순 요구사항이 LCEL에서 막히는 30~50줄짜리 시퀀스를 직접 부딪힌다.
- **체크포인터·HITL·멀티 에이전트의 진짜 모양을 안다.** 11장에서 supervisor/swarm을 회의적 시선과 함께 다루고, 12장에서 상태 저장과 사람 끼우기를 코드로 본다.
- **평가 4종과 trace 읽는 법을 익힌다.** 13장에서 에이전트가 *제대로 동작하는지*를 측정하는 좌표와, *망가지는 순간을 보는 법*을 본다.
- **보안과 비용 통제를 프로덕션 가까이 가져온다.** OWASP 4종, dual-LLM, lethal trifecta, 비용 캡 — 14장 한 챕터로 운영 직전까지 끌어올린다.

## 차례

### Part 1. 맨손으로 짓는 에이전트

1. **에이전트라는 단어가 마침내 뜻을 가지게 되었다** — "에이전트"가 무엇이며, 2026년 현재 가장 잘 동작하는 에이전트는 어떻게 생겼는가
2. **함수 하나 부르기 — Tool Use의 정확한 모양** — LLM이 도구를 부른다는 게 정확히 어떤 입출력의 모양인가
3. **도구를 여러 번, 스스로 결정해서 — ReAct 루프 짓기** — 모델이 도구를 한 번이 아니라 *필요한 만큼* 부르게 하려면 코드가 어떻게 바뀌어야 하는가
4. **ReAct 너머 — 루프 모양의 변종들** — ReAct가 잘 안 통하는 영역에서 어떤 루프가 시도되었고, 무엇을 언제 꺼내야 하는가
5. **기억의 모양 — 대화 history는 어디서 잘리고, 어디로 흘러가는가** — 에이전트의 "기억"은 결국 무엇이며, 컨텍스트 윈도우를 넘기지 않으려면 어떤 전략을 쓰는가
6. **무한 루프와 청구서 — 에이전트가 망가지는 다섯 가지 모양** — 프로덕션 직전에 반드시 손에 익혀야 할 안전장치는 무엇이고, 어디에 끼워 넣어야 하는가

### Part 2. LangChain — 추상화의 가치와 비용

7. **같은 에이전트, 두 번째로 짓기 — LangChain과 LCEL** — Part 1의 `mini_agent`를 LangChain으로 다시 짜면 어디가 줄어들고 어디가 늘어나는가
8. **LangChain이 진짜로 빛나는 자리 — 통합, 도구 생태계, LangSmith** — LangChain의 어떤 부분은 "내가 다시 짤 가치가 없는 것"인가
9. **LangChain의 그늘 — 비판, 옹호, 그리고 골라 쓰기** — LangChain을 *얼마나* 받아들이고 *어디서* 거절할지 어떻게 정할까

### Part 3. LangGraph — 상태와 그래프로 다시 생각하기

10. **LCEL의 한계와 그래프의 등장 — mini_agent를 세 번째로 짓기** — LangChain의 chain만으로 안 되는 진짜 상황은 무엇이고, 그래프는 그것을 어떻게 푸는가
11. **여러 에이전트가 같이 일할 때 — Supervisor, Swarm, 그리고 회의 한 줌** — 멀티 에이전트는 진짜로 더 똑똑한가, 아니면 그냥 더 복잡한가
12. **상태를 저장하고, 사람을 끼우고, 다시 깨우기 — Checkpointer와 HITL** — 장시간 대화·승인 단계·실패 후 재개를 코드 한 줄로 추가하려면 무엇이 있어야 하는가

### Part 4. 운영으로 가는 길 — 측정과 안전

13. **측정과 관찰성 — 평가 4종과 trace를 읽는 법** — 에이전트가 *제대로 동작하는지*를 어떻게 측정하고, *망가지는 순간*을 어떻게 보는가
14. **안전하게 운영하기 — 보안, lethal trifecta, 비용 통제** — 에이전트가 *나를 공격하는 상황*을 어떻게 막고, *나도 모르게 청구서가 폭주하는 상황*을 어떻게 막는가

### 부록

- **부록 A. 한국어 환경 팁** — 한국어 프롬프트·tool description·평가셋의 짧은 보강
- **부록 B. 다루지 않은 길로 가는 지도** — 로컬 모델, MCP 심층, AutoGen/MetaGPT/CrewAI 등으로 가는 진입 가이드
