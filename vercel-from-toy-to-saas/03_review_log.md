# Plan Review Log — Round 1

## 종합 판정

**Revise** — 4단계 성장 내러티브 골격은 탄탄하고 9개 챕터의 책임 분배도 잘 짜여 있다. 다만 (a) 2장이 너무 무겁고, (b) 단계 챕터(3-6장)와 횡단 챕터(7-8장) 사이 중복이 통제되지 않은 채 "약속"만 있고, (c) 사용자 요구 9가지 중 "활용 기능"의 횡단 노출 — 특히 AI Gateway·OG Image·Cron Jobs — 이 어느 챕터에도 home이 없다. 큰 골격은 유지하되 위 3가지를 손보면 충분히 빌드 가능한 계획이다.

## 차원별 평가

### 1. 커버리지 vs 사용자 요구: ⚠️

사용자가 명시한 9가지 요구를 매핑해보면:

| 요구 | 매핑 챕터 | 상태 |
|------|----------|------|
| 개념 | 2장 | ✅ |
| 사용법 (CLI·Git·API) | 4장(env pull), 부록 A | ⚠️ Git 연동 자동 배포의 첫 흐름이 어디에도 본격적으로 안 잡힘. 1장 도입 후 곧장 2장 추상화로 점프 |
| 활용 기능 | 분산 (4장 Edge Middleware, 5장 Edge Config·Turborepo, 8장 WAF·BotID) | ⚠️ **AI Gateway / AI SDK가 어느 챕터에도 명시되지 않음** (Ref §2.8). OG Image도 없음. Cron Jobs는 3장에서 한도만 언급되고 6장 이상에서 어떻게 쓰는지 안 다룸 |

| 논쟁 사례 | 8장(2026 사고), 9장(Lock-in, Edge), 7장(청구 폭탄) | ✅ |

**Critical 누락 — AI Gateway / AI SDK**: 레퍼런스 §2.8과 §8.3에 비중 있게 등장하고, 2026년 책 시점에서 Vercel의 가장 새로운 차별화 기능이다. 입문자가 "Vercel을 왜 지금 배워야 하는가"의 답 중 하나인데 챕터 제목·주요 내용 어디에도 없다.

**Should — Cron Jobs의 성장 곡선**: 3장에서 "1일 1회 한도가 답답해지는 순간"으로 다음 단계 신호로만 등장한다. 사이드/스타트업 단계에서 cron이 실제로 무엇을 돌리는지(빌드 후 ISR 갱신, scheduled 알림, 백그라운드 동기화), 함수 quota를 차감한다는 사실의 비용 함의가 아무 챕터에도 없다.

### 2. 챕터 균형: ⚠️

분량 분포는 12 / 22 / 18 / 22 / 24 / 22 / 22 / 22 / 24p. 합 188p. 후반부 평균 22-24p로 일관된 건 좋다. 다만:

- **2장(22p)이 가장 위험하다.** 함수 런타임 3종 + 렌더링 4종 + 빌드 파이프라인 + 엣지 네트워크 + Preview + Next.js 관계 — 이 책의 추상화 절반이 한 챕터에 압축돼 있다. 입문자가 3장으로 넘어가기 전에 지칠 가능성이 높다.
- **5장(24p)도 무겁다.** 5대 항목 + 케이스 2개 + 외부 보강 패턴 3개 + Edge Config + Turborepo + 보안 룰북. 한 챕터에 도구 7개가 새로 등장한다.
- **9장(24p)** 은 의도적으로 길지만 대안 매트릭스 + 학술 백본 + 비상 탈출 + Edge 논쟁이라 dense하다. 페이싱 노트의 "비판·여행 가이드 형식이라 가볍게 느껴지도록"이 실제로 가능할지 미지수.

**Should — 2장을 둘로 쪼개거나 일부를 9장 학술 부록으로 이전**: 함수 런타임·렌더링 전략의 *심층* 비교는 6장(SaaS의 비용 시뮬레이션)이나 7장(과금)으로 미루고, 2장은 "한 장의 추상화 지도"에 집중하는 게 입문자 친화적.

### 3. 독자 진입점: ✅

1장 12페이지의 "처음 막힌 자리"는 잘 잡았다. 5분 배포의 황홀함 → 환경변수·도메인·청구 화면의 막막함이라는 공감 도입은 Toby 스타일과 결이 맞고, 4단계 척추를 미리 약속하는 구조도 자연스럽다. 위치 진단 체크리스트(PV·결제·팀원·시크릿)는 입문자가 자기 좌표를 빠르게 잡게 한다.

다만 **Nice-to-have**: 1장에서 "이 책이 답하지 *않는* 것"을 한 단락 추가하면 좋다. Next.js 자체 문법, React 서버 컴포넌트 깊이, AWS 비교 — 이 책이 다루는 범위와 안 다루는 범위를 1장에서 못 박으면 후반부 9장의 대안 비교가 더 깔끔하게 들어간다.

### 4. 중복·낭비: ❌

이게 가장 큰 약점이다. 현재 계획의 "2막은 단계, 3막은 횡단"이라는 구조 자체는 좋은데, **같은 도구가 여러 챕터에서 어느 깊이로 다뤄지는지에 대한 명확한 분담선이 없다.**

구체 예:

- **Sensitive 환경 변수 플래그**: 3장(켜기), 4장("의심 시 모두 켜기"), 8장(2026 사고 + 두 종류의 구조적 차이) — 3번 등장. 8장이 home이 되어야 하는데 3·4장에서 이미 운영 룰까지 다루면 8장이 반복으로 느껴진다.
- **Spend Management**: 4장("만들자마자 켜는 한 가지", 디폴트 OFF), 7장("임계 알림 + 자동 일시정지 + 웹훅 한 페이지 가이드") — 4장에서 이미 의식 약속을 받았는데 7장에서 한 번 더 한다. 자칫 같은 메시지의 두 번 반복.
- **WAF / BotID / Attack Challenge Mode**: 5장(보안 룰북), 7장(Attack Challenge Mode 무료), 8장(WAF Managed Rulesets, BotID, Attack Challenge Mode 다시) — 8장의 "다시"라는 표현이 이미 자기 인식하고 있을 정도.
- **Edge Middleware**: 4장(Basic Auth/rate limit/redirect), 8장("무거운 로직 넣지 말 것") — 두 번째가 너무 가볍다.
- **Fluid compute**: 2장(런타임 3종), 6장("AI 에이전트 같은 I/O-bound 함수에 진짜 이득"), 7장("과금에서 I/O 대기 무료") — 분산된 게 자연스럽지만 2장·6장·7장 세 곳 어디도 "여기서 결판낸다"는 느낌이 없다.

**Critical — 도구별 home 챕터를 명시하라**: 각 도구는 "단계 챕터에서는 *언제 켜는가*만 다루고 *어떻게 쓰는가*는 home 챕터에서 다룬다" 같은 분담 규칙을 02_plan.md 본문에 명시해야 한다. 지금은 "단계 챕터에서 한 페이지씩 짧게, 본격 깊이는 7·8장"이라는 한 줄밖에 없다.

### 5. 흐름·페이싱: ⚠️

4막 구조 자체는 잘 작동한다. 다만:

- **3막(7·8장)의 톤 전환**: 페이싱 노트는 "단계 챕터에서 흩어져 있던 메모를 한 곳에 모아 운영 의식의 두 기둥으로 굳힌다"고 약속한다. 그러나 현재 7장 주요 내용은 "함수 과금 3축 손계산", "Bandwidth가 새는 이유"부터 시작 — 이건 새 정보이지 *모으기*가 아니다. 마찬가지로 8장도 "환경 변수 두 종류"부터 새로 정의한다. **약속과 실제 챕터 시작이 어긋난다.**
- **9장의 닫음 약속**: 페이싱 노트는 "마지막 페이지는 다시 1장의 새벽 장면으로 돌아와"라고 한다. 좋다. 그러나 9장 주요 내용 9개 중 그 *콜백*이 명시되어 있지 않다. 계획서에서는 보장되지 않는 약속이 된다.
- **6장 → 7장 전환**: 6장이 SaaS의 결정(Enterprise / 멀티 / 셀프) 으로 닫고, 7장이 다시 비용 기초(Hobby/Pro/Enterprise 한 장 정리)로 돌아온다. 단계 진입 후 백트래킹하는 인상.

**Should — 7·8장 도입을 "이미 흩어진 메모를 모은다"는 콜백으로 명시**: 7장 첫 절은 "지금까지 단계마다 한 페이지씩 비용을 이야기했다. 여기서는 그 페이지들을 한 자리에 놓고…"로 시작한다고 계획에 적어두면 톤 전환 위험이 줄어든다.

**Should — 9장 마지막 절을 별도 항목으로 못 박기**: 주요 내용에 "10. 닫는 글: 다시 그 새벽으로 — 같은 한 줄, 다른 무게"를 명시 추가.

### 6. 구체성: ⚠️

대부분 챕터의 "주요 내용"은 1챕터를 쓸 만한 구체성을 갖췄다. 특히 3·4·5장의 단계 챕터, 7장의 비용은 좋다. 다만 약한 챕터:

- **6장 SaaS 단계**: "한도 협상 — Vercel과 어떻게 대화하고 무엇을 준비해 가는가"는 추상적이다. 레퍼런스에 협상 패턴 1차 자료가 없다(§3.4는 "영업 컨택"만). 이 절을 쓸 근거가 부족하다.
- **6장 운영 의식**: "OpenTelemetry, 함수별·라우트별 사용량 분해, 사고 대응 런북" — 사고 대응 런북은 어떤 사고를 가정하나? 레퍼런스에 SaaS 단계의 운영 런북 사례가 없다. 추상으로 끝날 가능성.
- **8장 컴플라이언스**: "비기술 의사결정에 어떻게 들어오는가"는 한 줄로는 너무 모호하다. 어떤 비즈니스 시나리오에서? B2B 계약? 의료? 금융?

**Should — 6장 "한도 협상"을 삭제하거나 "Enterprise 컨택 시 준비 자료" 정도로 좁히기**: 1차 자료 없이 협상 노하우를 책에 쓰면 독자가 시도했을 때 안 통할 위험.

### 7. 반례·논쟁 균형: ✅

전반적으로 잘 짜였다. 9장의 Lock-in 두 관점, 7장의 청구 폭탄 케이스, 8장의 2026 사고가 비판적 시선의 무게를 충분히 잡고 있다. 1-2장은 옹호 톤(이게 어떻게 작동한다)이고, 3-6장은 중립 운영서, 7-8장은 비판 강도 ↑, 9장은 비판·대안. 4막 구조 자체가 균형을 보장한다.

다만 **Nice-to-have — 옹호 측 강한 사례를 8장이나 9장에 한 절 추가**: 책이 비판으로 점점 기우는 인상을 막으려면 "Vercel이 진짜 빛나는 자리"의 단단한 사례 한두 개(예: Claude.ai의 Vercel 운영, AI Gateway 통합 케이스)가 9장 후반에 들어가면 균형이 더 좋다. 지금은 1장 도입의 "황홀함"이 옹호의 거의 전부.

5. **[MED] 9장 마지막 절 "다시 그 새벽으로" 명시** — 페이싱 노트에만 약속된 1장 콜백을 9장 주요 내용에 항목으로 박아 보장.

6. **[MED] 6장 "한도 협상" 삭제 또는 좁히기** — 1차 자료 없는 추상 절. "Enterprise 컨택 시 준비 자료" 정도로만 남기기.

7. **[LOW] 1장에 "이 책이 답하지 않는 것" 한 단락** — Next.js 문법, RSC 깊이, AWS 비교의 범위를 못 박아 9장 대안 비교의 무게를 명확히.

8. **[LOW] 9장 후반에 옹호 측 강한 사례 한 절** — 균형을 위해 Vercel이 진짜 빛나는 자리(Claude.ai 운영, AI Gateway 통합)의 사례를 한 절 추가.

## Round 1 → Round 2 변경 로그 (book-planner)

리뷰 판정 **Revise**에 대응해 02_plan.md를 다음과 같이 갱신했다.

### HIGH 1 — AI Gateway / AI SDK / OG Image / Cron Jobs home 부여
- **5장**에 "AI 기능을 붙일 때 — Vercel AI SDK·AI Gateway" 한 절 신설. 다중 모델 단일 API + 마진 투명도 비판을 균형 있게.
- **6장** Fluid 절을 "AI 워크로드에 주는 진짜 이득"으로 5장과 직결. I/O 무료 과금이 AI 에이전트 비용 곡선에 미치는 차이를 본격 다루기.
- **4장**에 "Cron Jobs 다회 사용 패턴" 한 단락(scheduled fetch·ISR 갱신·DB 정리·알림) + "OG Image Generation" 한 단락(Next.js `ImageResponse` + Vercel 엣지) 신설.
- 도구 분담 표에 4종 모두 명시.

### HIGH 2 — 도구 분담 규칙을 표로 본문에 못박음
- 부록과 내러티브 아크 사이에 **도구 분담 표** 신설. 15개 주요 도구(Sensitive 플래그·Spend Management·Cron Jobs·OG Image·Edge Middleware·Edge Config·Turborepo·WAF·BotID·Attack Challenge Mode·AI Gateway·Fluid·런타임 3종·Deployment Protection·Image Optimization)에 대해 *처음 등장 챕터 / Home(깊이) / 핵심 한 줄* 3열로 정리.
- 2장 "역할 약속"에 표를 장 끝에 본문으로 삽입한다고 명시.
- 4·5·6장 도입에 "분담 규칙" 박스를 추가해 단계 챕터에서는 *언제·왜 켜는가*까지, 메커니즘은 7·8장에서임을 매번 환기.
