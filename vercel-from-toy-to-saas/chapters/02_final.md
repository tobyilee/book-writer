# 2장. Vercel을 한 번 더 정의해 보자

"Frontend Cloud."

Vercel 홈페이지 첫 화면이 두 단어로 자기를 소개한다. 정작 그 두 단어 뒤에 무엇이 묶여 있는지 자기 언어로 설명할 수 있는 사람은 의외로 적다. 1장에서 우리는 새벽에 `vercel deploy`를 누르고 5분 만에 사이트가 떠오르는 황홀함을 같이 떠올렸다. 그 한 줄 뒤에 정확히 무엇이 돌아가고 있었을까? 빌드 서버일까, CDN일까, 함수일까, 아니면 그 모두일까?

이 장은 그 답을 *지도*로 그려보려고 한다. 한 가지 미리 약속해두자. 이 지도는 어디까지나 *전체 윤곽*이다. 각 추상화에 대해 "이런 것이 있다, 어디서 본격적으로 다룰 예정이다" 정도까지만 짚고 넘어간다. 함수 런타임 3종을 처음 만나면 곧장 깊게 비교해보고 싶고, ISR을 보면 `revalidate` 동작을 따져보고 싶지만, 그건 책의 후반에서 *결정의 자리*에 도달했을 때 다시 펼쳐보자. 지금 그 모든 디테일을 한꺼번에 꺼내면 지도가 아니라 미로가 된다.

대신 이 장이 끝날 때 독자가 갖게 되는 것은 두 가지다. 하나는 같은 동료가 "Vercel이 뭐야?"라고 물을 때 한 문단으로 답할 수 있는 능력. 다른 하나는 책의 후속 챕터에서 어떤 도구가 어디서 깊게 다뤄지는지 한눈에 알 수 있는 *분담 표*다. 마지막 절(2.8)이 그 표다. 그 표 한 장이 있으면, "Spend Management가 4장에서도 나왔는데 7장에서 또 나오네?" 같은 혼란을 피할 수 있다.

자, 그럼 Vercel이 매니지드해주는 것이 정확히 무엇부터 살펴보자.

## 2.1 Vercel은 한 줄로 무엇을 묶었는가

질문을 다시 던져보자. Vercel은 호스팅인가, CDN인가, 서버리스 플랫폼인가?

답은 "그 모두를 한 번의 git push 뒤에 묶어놓은 매니지드 PaaS"다. 풀어 쓰면 이렇다 — Git 저장소를 연결하면 push 한 번으로 빌드, 배포, 글로벌 CDN 캐싱, SSL 인증서 자동 발급, preview URL 생성, 그리고 서버리스 함수 실행까지 한 흐름으로 처리된다. 이 한 줄을 통째로 받아들이고 시작하자. 입문자가 Vercel을 *부분적으로* 보면 헷갈린다. "그냥 정적 호스팅이지?" 같은 첫인상은 절반만 맞다.

조금 다른 각도로 보면, Vercel은 *프론트엔드 개발자가 평소 하던 일*에 매니지드 인프라를 한 겹 덧씌운 제품이다. Git에 push하던 습관 그대로, `next build`를 돌리던 흐름 그대로 두면, 그 위에서 빌드 서버·CDN·함수 런타임이 알아서 정렬한다. 35개 이상의 프레임워크에 대해 이런 통합을 자동으로 해준다. 빌드 명령, 라우팅 규약, 이미지 최적화, 환경 변수 주입을 표준 추상화로 감싸 놓았기 때문에, Next.js든 Astro든 SvelteKit이든 비슷한 경험으로 진입할 수 있다. 다만 솔직히 말해두자. Next.js는 Vercel이 직접 메인테인하기 때문에 가장 깊게 통합된다. *first-class*라는 말이 다른 프레임워크에는 다소 무게가 다르다. 이 비대칭이 만들어내는 vendor lock-in 논쟁은 9장에서 본격적으로 따져보자.

여기서 한 가지 짚고 넘어가자. "매니지드"라는 단어가 만들어주는 안도감은 진짜다. 인증서 갱신을 잊어버려 사이트가 죽는 사고는 Vercel을 쓰면 거의 일어나지 않는다. 대신 *반대 방향의 사고*가 새로 생긴다. 청구서가 갑자기 늘어난다든가, preview URL이 검색 엔진에 노출된다든가 하는 종류다. 매니지드의 비용은 항상 "내가 안 챙겨도 되지만, 그래서 내가 모르는 사이에 일이 벌어진다"는 형태로 돌아온다. 이 책의 후반부 절반은 이 반대편 비용을 다룬다.

## 2.2 엣지 네트워크와 PoP — 글로벌이 빠르다는 말의 조건

Vercel이 글로벌 사용자에게 빠르다고 하는 이유는 자체 PoP(Point of Presence)에 정적 자산과 CDN 캐시, 그리고 Edge Functions를 분산해두기 때문이다. PoP는 쉽게 말해 사용자와 가까운 자리에 미리 놓아둔 작은 서버 거점이다. 한국 사용자가 한국 PoP에서 받아간다면, 미국 데이터센터를 왕복하지 않으니 당연히 빠르다. 2025 대비 2026년에는 비미국권 TTFB가 눈에 띄게 개선됐다는 비교 자료도 있다.

그런데 *항상* 빠른가? 여기서부터 조심하자. PoP 수만 보면 Cloudflare가 300개 이상으로 더 많고, 글로벌 사용자 분포에 따라 체감 속도가 달라진다. 게다가 데이터베이스가 한 region에만 있으면, Edge에서 서버까지 빠르게 도달해도 결국 그 region까지 한 번 더 왕복해야 한다. "Edge라서 빠르다"가 아니라 "Edge에서 받을 게 다 Edge 가까이 있을 때 빠르다"가 정확한 문장이다. 이 *조건부* 논쟁은 9장에서 학술 자료(Cassel et al. 2025의 placement-cost-latency triangle 같은)를 곁들여 본격적으로 다뤄보자. 지금은 "Vercel = 글로벌 분산 = 무조건 빠름"이라는 단순한 그림을 머릿속에서 지우는 정도면 충분하다.

엣지 네트워크가 의미 있는 자리는 분명히 있다. 정적 자산과 캐시된 페이지, 그리고 fetch·헤더 조작 같은 가벼운 Edge 함수들이다. 이 셋은 사용자 가까이서 처리될수록 분명한 이득이 있다. 그 너머는 9장에서 다시 보자.

## 2.3 빌드 파이프라인 — push 한 번 뒤에 무엇이 일어나는가

Next.js 앱을 push했다고 해보자. 그 push 뒤에 어떤 일이 차례로 벌어질까?

흐름을 한 장으로 정리하면 이렇다. (1) Git에 push가 일어나면 Vercel이 webhook을 받는다. (2) 새로운 빌드 컨테이너가 떠서 의존성을 설치하고 `next build`(또는 프레임워크별 빌드 명령)를 돌린다. (3) 빌드 결과물이 Vercel 인프라에 배포되고, 해당 브랜치 또는 PR에 *preview URL*이 발급된다. (4) main 브랜치로 merge되면 같은 흐름이 한 번 더 돌면서 *production*으로 승격된다.

이 흐름의 핵심 두 가지를 미리 짚자. 첫째, push마다 preview URL이 자동으로 만들어진다는 사실. 디자인 리뷰, QA, 스테이크홀더 데모에 그 URL을 그대로 던져주면 된다. 둘째, main과 preview는 *같은 코드 경로지만 다른 환경*이다. 같은 컨테이너 이미지가 다른 환경 변수와 다른 도메인으로 떠오른다. 이 둘의 분리가 운영 의식의 출발점이다 — preview에 production 시크릿을 그대로 부어버리면 모든 PR이 production-급 폭발 반경을 갖게 된다. 이 *위험*은 8장에서 본격적으로 결판을 보자.

모노레포라면 한 가지가 더 추가된다. Turborepo Remote Cache를 켜두면 affected workspace만 재빌드한다. 즉, 빌드 시간이 변경된 패키지에 비례한다. 이 패턴은 5장에서 스타트업 단계의 *빌드 시간 단축*과 함께 다룰 예정이다. 지금은 "기본 흐름은 push → preview → merge → production이고, 모노레포는 affected만 빌드한다" 정도로 머릿속에 넣어두자.

## 2.4 함수 런타임 3종 — Node, Edge, Fluid가 무엇이고 언제 등장하는가

여기서부터 잠깐 호흡을 가다듬자. 이 절은 *짧게 끝낼 것*이다. 함수 런타임 3종은 Vercel을 이해할 때 가장 헷갈리기 쉬운 개념이고, 자세히 들어가면 한 챕터를 통째로 잡아먹는다. 그래서 이 장에서는 "이런 게 있다, 언제 등장한다" 한 문단씩으로 끝내고, 본격적인 비교는 6장에서 SaaS 단계의 비용·운영 결정과 함께 다루기로 약속하자.

**Node runtime**은 우리가 익숙한 그 Node.js다. Node API 전체를 사용할 수 있고, region을 고정해서 배치할 수 있다. 데이터 변환, 파일 처리, 외부 SDK 호출처럼 Node 생태계의 라이브러리를 그대로 가져다 쓰고 싶을 때 적합하다. Pro 플랜의 Fluid 위에서는 max duration이 800초까지 갈 수 있어 길게 도는 작업도 받아준다.

**Edge runtime**은 글로벌 분산되는 가벼운 런타임이다. 사용자 가까이서 실행되니까 빠르다. 단, 제약이 *제법 강하다* — filesystem 접근이 안 되고, `require()`도 직접 못 부르며, 대부분의 Node API가 빠져 있다. 25초 안에 응답이 시작돼야 하고 최대 300초 스트리밍이다. 처음 Edge에 익숙한 라이브러리를 import했다가 "왜 안 돌지?" 하고 좌절하는 일이 많은데, 이게 난감한 이유는 잠시 후 알아보자. Edge에서는 fetch, crypto.subtle, Web Streams 정도만 가능하다고 가정하고 시작하는 편이 낫다.

**Fluid compute**는 2025년에 신설되어 2026년 신규 배포의 기본값으로 자리잡은 새 모델이다. 단일 인스턴스가 다중 요청을 동시에 처리하고, bytecode 최적화와 pre-warming으로 cold start 영향을 완화한다. 가장 결정적인 차이는 과금이다 — CPU 시간 ms 단위로만 청구하고, *I/O 대기 시간은 무료*다. AI 에이전트나 외부 API를 길게 호출하는 함수에서 비용 곡선이 눈에 띄게 펴진다.

여기까지가 *지도 깊이*다. 더 들어가고 싶은 마음을 잠시 눌러두자. Node API 제약이 실제로 어떤 라이브러리에서 어떻게 막히는지(pdfkit, sharp, bcrypt 같은 단골 사례), Fluid의 단일 인스턴스 다중 처리 메커니즘이 어떻게 작동하는지, AI 워크로드에서 실제 비용 절감이 얼마쯤 되는지 — 이 *심층 비교*는 6장에서 본격적으로 다룬다. 학술 맥락(Mahmoudi et al. 2024의 cold start 메타분석)도 그쪽에서 함께 짚어보자.

## 2.5 렌더링 전략 4종 — SSG, SSR, ISR, PPR이 무엇인지

Next.js 문서를 처음 펼치면 머리를 띵하게 만드는 약어 4개가 있다. 하나씩 한 줄 정의로만 짚어보자.

**SSG (Static Site Generation)**는 빌드 타임에 페이지를 정적 파일로 만들어 두는 방식이다. 배포 후에는 그냥 정적 파일이니 가장 싸고 가장 빠르다. 블로그처럼 자주 바뀌지 않는 콘텐츠에 어울린다.

**SSR (Server-Side Rendering)**은 요청이 들어올 때마다 서버에서 페이지를 렌더해 돌려준다. 동적이지만 매번 함수를 깨우니 비용이 올라간다.

**ISR (Incremental Static Regeneration)**은 정적 페이지에 *재생성 주기*를 붙인 것이다. `revalidate` 옵션으로 "이 페이지는 N초 지나면 백그라운드에서 다시 만들어줘"라고 표시한다. 본질은 RFC 5861의 stale-while-revalidate 패턴을 프레임워크 레벨로 통합한 것이다. 정적의 빠름과 동적의 신선함을 절충하려는 시도다.

**PPR (Partial Prerendering)**은 가장 새로 등장한 모델이다. 한 페이지 안에 정적 shell과 동적 영역(Suspense holes)을 같이 두고, 단일 응답으로 둘을 결합해서 보낸다. 50ms 미만의 TTFB와 부분 동적성을 동시에 달성하는 첫 모델이라는 평을 받는다. Next.js 15에서 stable이 되었고, 16의 cacheComponents flag에서는 기본값이다.

여기서도 마음을 가다듬자. ISR의 `revalidate = 60`을 보면 "60초마다 재생성"으로 읽고 싶어진다. 그런데 이게 *흔한 오해*다. 실제로는 "60초가 지난 뒤 *다음 요청이 들어오면* 백그라운드에서 재생성"에 가깝다. 첫 요청은 stale 응답을 받는다. 이런 종류의 함정은 9장에서 결판을 보자. PPR의 동적 영역이 실제로는 어떻게 캐시되고 어떻게 invalidate되는지 같은 디테일도 그쪽으로 미루자. 지금은 "이름 4개와 한 줄 정의"만 머리에 넣어둬도 충분하다.

## 2.6 Preview Deployment — 모든 PR에 따라붙는 도메인

Vercel을 써 본 사람이라면 누구나 한 번쯤 받아본 그 URL이 있다. `my-app-git-fix-typo-toby.vercel.app` 같은 형태. PR을 올리면 자동으로 만들어지는 그 preview URL이다.

이 기능의 강점은 분명하다. 동료에게 "이 PR 한번 봐줘"라고 할 때 코드 diff뿐 아니라 *실제로 떠 있는 사이트*를 같이 던져줄 수 있다. 디자이너가 최종 결과물을 브라우저에서 확인할 수 있고, 비개발자도 클릭해서 만져볼 수 있다. 이 동반 워크플로 자체가 Vercel을 다른 PaaS와 차별화하는 큰 축이다.

그런데 같은 강점이 *그대로* 위험이 된다. preview URL은 기본적으로 인터넷에 떠 있다. `*.vercel.app` 도메인은 검색 엔진 크롤러에 인덱싱될 수 있고, URL 패턴을 추측해서 접근하는 경우도 있다. 만약 그 preview에 production 데이터베이스 시크릿이 그대로 들어있다면? 모든 PR이 production-급 폭발 반경을 갖게 된다는 표현이 과장이 아니다. 솔직히 찜찜한 그림이다.

그래서 이 *위험*은 8장에서 본격적으로 다룬다. preview에 production 시크릿을 재사용하지 않는 실전 워크플로, robots.txt와 Deployment Protection(SSO, Password, Allowlist) 설정, 환경별 시크릿 분리. 이런 도구가 거기서 깊게 결판이 난다. 지금은 "preview는 강력한 자산이고, 동시에 잘못 두면 가장 새기 쉬운 자리다" 한 문장만 챙겨두자.

## 2.7 Next.js와 Vercel — 왜 Next.js만 first-class처럼 보일까

이 질문은 Vercel을 쓰는 사람이라면 누구나 한 번쯤 부딪힌다. 같은 Frontend Cloud인데 왜 Next.js만 유난히 매끄럽게 돌아가는 것 같을까?

답은 단순하다. Next.js는 Vercel이 직접 메인테인하는 프레임워크다. 35+ 프레임워크 어댑션이 있다고 해도, 회사가 직접 만들고 운영하는 프레임워크와 third-party 어댑터 사이에는 엔지니어링 거리의 차이가 있다. ISR, Image Optimization, PPR, Edge Middleware 같은 기능이 Vercel에서 가장 매끄럽게 돌아가는 이유다. 정확히 같은 기능이 다른 호스트에서 *작동은 하지만 reduced fidelity*인 경우가 적지 않다.

여기서 두 관점이 부딪힌다. 한쪽은 "Next.js를 쓰는 순간 사실상 Vercel에 종속된다"고 본다. 다른 쪽은 "Next.js의 70%가 Vercel 외부에서 운영된다, Walmart, Nike, Claude.ai 같은 사례가 있다"고 반박한다. Next.js 16.2의 stable Adapter API가 Cloudflare·Netlify·Amplify·Google·OpenNext 공동 설계로 자리 잡으면서 풍경이 다시 바뀌고 있기도 하다.

이 *논쟁*은 9장에서 두 관점을 나란히 놓고 결판을 본다. 비상 탈출 옵션(`output: "standalone"` Docker 빌드)을 평소에 손에 두는 법, OpenNext의 위치와 한계, 결정 휴리스틱 4줄 같은 것들이 모두 거기 모여 있다. 지금은 "Next.js와 Vercel은 *유난히 가까운 관계*이고, 그 가까움이 강점이자 위험 요인이다"라는 양면을 머릿속에 같이 두자. 한쪽만 보면 판단이 한쪽으로 기운다.

## 2.8 도구 분담 표 — 어디서 무엇을 깊게 다루는가

지금까지 등장한 도구가 꽤 된다. Sensitive 환경 변수 플래그, Spend Management, Edge Middleware, Cron Jobs, WAF, BotID, Fluid compute, AI Gateway, Deployment Protection, Image Optimization⋯ 책 전체를 통과하면서 같은 도구가 여러 챕터에 등장한다. 같은 도구를 두세 번 만나면 "어, 이거 아까도 봤는데 어디서 깊게 다뤘더라?" 하고 헷갈리기 쉽다.

그래서 한 가지 약속을 해두자. **각 도구는 단계 챕터(3~6장)에서 *언제·왜* 켜는가까지만 다루고, 메커니즘·설정 페이지·사고 분석은 *home 챕터*에서 결판을 본다.** 이 규칙으로 같은 도구가 여러 번 등장해도 깊이를 헷갈리지 않게 한다. 표로 정리하면 이렇다.

| 도구 | 처음 등장 | Home (깊이) | 핵심 한 줄 |
|------|-----------|-------------|-----------|
| Sensitive 환경 변수 플래그 | 3장 (켜기) | **8장** (구조·사고·회전 절차) | UI에서 다시 못 읽는 게 단점이 아니라 보안 자산 |
| Spend Management | 4장 (왜 켜는가, 디폴트 OFF) | **7장** (임계·자동 일시정지·웹훅 설정) | Pro 만들자마자 켜는 한 가지 |
| Cron Jobs | 3장 (한도 한계) | **4장** (실제 쓰임 패턴·quota 차감) | 함수 quota를 차감한다는 비용 함의 |
| OG Image Generation | — | **4장** (사이드 SEO/공유) | Next.js `ImageResponse` + Vercel 엣지의 동적 카드 |
| Edge Middleware | 4장 (Basic Auth·rate limit·redirect) | **8장** (무거운 로직 금지·메커니즘) | fetch/헤더 조작 전용, 그 이상은 Route Handler |
| Edge Config | 5장 (read-mostly 메타데이터 분리) | **5장** | p50 1ms 미만, 피처 플래그·차단 IP·리다이렉트 룰 |
| Turborepo Remote Cache | 5장 | **5장** | 모노레포 affected 빌드, invalidation 함정은 9장 |
| WAF Custom Rules | 5장 (켜기) | **8장** (Managed Ruleset·BotID 메커니즘) | Bot/AI Bot/BotID/Custom 4종 |
| BotID | 5장 (켜기) | **8장** | invisible fingerprint, Middleware metadata로 rate limit 결합 |
| Attack Challenge Mode | 5장 (켜기) | **7장** (무료·차단 트래픽 미과금) | 모든 플랜 무료 가드레일 |
| AI Gateway / AI SDK | **5장** (스타트업의 AI 붙이기) | **5장** + 6장 호명 (Fluid I/O 무료의 AI 이득) | 다수 모델 단일 API + 마진 투명도 비판 |
| Fluid compute | 2장 (개요) | **6장** (메커니즘 심층) + 7장 (과금 손계산) | I/O 대기 무료, AI 워크로드 비용 곡선 변화 |
| 함수 런타임 3종 | 2장 (개요) | **6장** (심층 비교) | Node API 전체 vs Edge 제약 vs Fluid 다중 처리 |
| Deployment Protection | 3장 (켜기) | **8장** (SSO·Password·Allowlist·preview 인덱싱 차단) | preview를 production-급 폭발 반경으로 만들지 말기 |
| Image Optimization | 3장 (`<Image>` 등장) | **5장** (외부 CDN 분리 80% 절감) + 7장 (LLM 크롤러 폭증) | 가장 폭증하기 쉬운 단일 항목 |

이 표를 한 번 훑고 나면 같은 도구가 여러 챕터에 등장해도 헷갈리지 않을 것이다. 3장에서 Sensitive 플래그를 켜고 지나가면서 "이건 8장에서 사고 해부와 함께 깊게 본다"라고 적혀 있다는 사실을 기억해두자. 5장에서 BotID를 켤 때 "메커니즘은 8장"이라는 사실을, 4장에서 Spend Management를 켜라고 권할 때 "임계 알림 설정 페이지는 7장"이라는 사실을 머리 한쪽에 둬두면 좋다.

## 마무리

지도 한 장을 같이 그렸다. Vercel은 Git push 한 번으로 빌드·배포·CDN·SSL·preview·서버리스 함수를 한 흐름에 묶는 매니지드 PaaS다. 글로벌 PoP가 *조건이 맞으면* 빠르게 응답한다. 빌드 파이프라인은 push → preview → merge → production의 흐름을 따른다. 함수는 Node·Edge·Fluid 세 런타임이 있고, 렌더링은 SSG·SSR·ISR·PPR 네 가지가 있다. preview는 강력하지만 새기 쉬운 자리이고, Next.js는 first-class지만 그 가까움이 양면이다.

이 지도가 가벼워 보였다면 그게 의도다. 후속 챕터들은 이 윤곽 위에 구체적인 *결정의 자리*를 하나씩 얹는다. 3장은 토이 단계의 가드레일, 4장은 Pro로 올라가는 트리거, 5장은 외부 보강과 AI 기능, 6장은 SaaS의 분기 결정과 함수 런타임 심층 비교, 7장은 비용의 결판, 8장은 보안의 결판, 9장은 함정·대안·옹호의 균형. 같은 개념이 점점 깊어지는 나선형으로 책이 진행된다.

다음 장은 가장 가벼운 단계, 토이 프로젝트로 내려간다. 무료 플랜(Hobby)으로 어디까지 살아남을 수 있고, 어디서부터 결제 카드와 ToS의 신호가 깜빡이는지 한 번 살펴보자.
