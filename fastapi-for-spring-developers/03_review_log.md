#### Critical

**#1 — 3장 분할 또는 슬림화**
- 선택: **(B) 한 챕터로 두되 슬림화**.
- 결정: 본 챕터의 가치는 "Spring `@RestController` + DTO + `@Valid` 삼중주가 한 함수로 합쳐지는 *체감*"이다. 두 챕터로 쪼개면 그 체감이 분산된다. 13챕터 구조도 보존된다.
- 처리: Pydantic 디테일을 핵심으로 추리고(`field_validator` 핵심 1개만), Query/Path/Body/Header 세부 표는 부록 A로 이동. APIRouter 본격 사용은 4장 짧은 절 + 12장 통합 프로젝트로 이동.

**#2 — 레이어드 아키텍처/도메인 폴더 구조 가르치는 자리 명시**
- 결정: 5장 말미 한 절(약 1,500자)로 reference §4.3의 by-feature 폴더 구조 + Spring 패키지 구조 매핑을 명문화. 4·6·7·9·10장의 모든 작은 예제가 이 구조를 따른다고 책 차원에서 약속.
- 위치를 5장으로 잡은 이유: 폴더 분리의 동기가 가장 강한 챕터가 5장(엔티티·레포·서비스·라우터 분리)이라 학습 부담이 자연스럽다. 4장 끝은 DI 분량이 이미 무거워 부담.

**#3 — 백그라운드 작업 자리**
- 결정: 8장 §8.7 독립 절(약 2,500자)로 흡수. 9장이 아닌 8장으로 간 이유는 `BackgroundTasks`/`ARQ`가 모두 async 컨텍스트와 직접 연결되기 때문. 9장(예외·관측성)은 횡단관심사 챕터라 호흡이 다르다.
- 처리: 비교 표(BackgroundTasks vs APScheduler vs ARQ vs Celery) + 어느 시점에 BackgroundTasks를 벗어나야 하는지의 판단 기준. 예제로 "이메일 발송을 BackgroundTasks → ARQ로 옮기는 1단계 마이그레이션".

#### Should

**#4 — 9장 예제에 도메인 부여**
- 결정: "에러 핸들러 데모" → **"주문 API의 도메인 예외 계층"**. 5가지 실패 모드(`OrderNotFound`, `InsufficientStock`, `PaymentDeclined`, `IdempotencyConflict`, `RateLimitExceeded`)를 도메인 예외로 모델링. reference §4.4와 자연스럽게 연결.
- 결제 대신 주문을 고른 이유: 결제는 외부 게이트웨이 의존성이 강해 학습이 산만해질 수 있다(원안에서 12장 도메인 선정 시 같은 이유로 결제를 제외했음). 주문은 Spring 출신에게 가장 친숙한 도메인.
