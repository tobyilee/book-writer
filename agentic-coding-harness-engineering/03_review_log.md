## 1. Executive Summary

이 계획은 **이론적 골격(2부)과 한국어 생태계(5부 21장)는 한국어권 1위 수준으로 강하지만, 운영(operations) 층위가 종합 바이블 약속에 비해 얇다.** 비용 통제·관측·CI/CD·팀 도입·모델 회귀 같은 "현업 개발자가 책장에서 두 번째로 펼치는 페이지"가 거의 없다. 4부 보안은 4장 56쪽으로 종합 바이블 기조에서 가장 가벼운 부 중 하나로 끝나는데, 책의 차별화 포인트(Behaviour harness·하네스 보안·평가 회피)를 4부가 짊어진다는 점에서 어울리지 않는다. 22장(메타 하네스)이 마지막인데, 5가지 출구 능력을 다시 호출하는 마무리 외에 "현장의 목소리"(인터뷰·실패담)가 없어 종합 바이블치고 닫는 맛이 약하다. 부록은 충실하나 **운영 런북 부록**과 **관측·비용 부록**이 빠져 있어 본문의 운영 공백을 부록으로도 못 메운다. **결론: 일부 보강 후 진행. 신규 챕터 1~2개 + 운영 부록 1~2종을 추가하면 종합 바이블 약속이 완성된다.**

## 2. Strengths

구체적으로 잘된 점부터 짚는다(짧게).

### 2.1 이론적 골격이 단단하다
- **2부 5장 → 6장 → 7장**의 흐름(CAR → Guides·Sensors → ACI 4원칙)이 레퍼런스 §2의 권고를 정확히 반영한다.
- 특히 7장에서 ACI를 "책의 진짜 척추"로 명시한 것은 레퍼런스 §12.2 권고와 합치한다.
- CAR ↔ Guides·Sensors 매핑(레퍼런스 §2.2 마지막 단락)을 5장에서 어떻게 끼울지 명시되어 있는 점도 좋다.

### 2.2 1부의 "거울" 설계가 영리하다
- 2장 Hashimoto 6단계로 자기 진단 → 22장 마지막에서 5가지 출구 능력 재호출 — 책 전체가 진단·재진단의 폐루프가 된다.
- 이 구조는 종합 바이블 정체성과 잘 맞는다 (책 한 권이 "당신의 다음 단계"를 가리킨다).

### 2.3 한국어 생태계 21장이 가장 알차다
- revfactory·instructkr·Haandol·MadPlay 4 진영을 모두 1차 자료까지 동반.
- 한국 환경 특수 이슈 4종(한글 토큰·사내 보안·한글 vs 영문 컨벤션·대기업 사례)도 본문에 반영.
- 단, **한 챕터로 충분한가**는 §3.4에서 비판한다.

### 2.4 부록 6종 구성 자체는 합리적이다
- A(템플릿 갤러리), B(MCP 카탈로그), C(Inspect 키트), D(페이퍼 20선), E(한국어 로드맵), F(용어집).
- 레퍼런스 §12.6 권고(부록을 두껍게)와 정확히 일치.
- 다만 **빠진 부록 2종**은 §4에서 추가 권고한다.

### 2.5 Toby 스타일 적용 원칙(§9)이 명문화되어 있다
- "22개 중 18개는 수사적 질문 또는 상황 가정으로 시작" 같은 정량 기준은 집필 단계에서 매우 유용하다.
- 외래어 절제·수동태 회피·영어 인용 처리 규칙도 구체적이다.

## 3. Critical Issues (반드시 고쳐야 할 항목, 우선순위 순)

### [Critical-1] 운영(Operations) 층위가 통째로 빠져 있다

**현상**: 종합 바이블이라 자칭하는 책에 "오늘 출근해서 펼치는 페이지"가 없다. 다음 6개 운영 토픽이 본문 어디에도 정면 챕터로 다뤄지지 않는다.

| 운영 토픽 | 현재 계획에서의 위치 | 문제 |
|----------|---------------------|------|
| **비용 통제·throttling·요금 모니터링** | 12장 마지막 단락에 "무한 루프의 비용 통제 휴리스틱"이라고 한 줄만 등장 | 종합 바이블에서 한 줄은 곤란. CFO/팀장이 책을 검토할 때 가장 먼저 묻는 페이지. |
| **관측·메트릭·로깅** (LangSmith, Helicone, Langfuse, OpenTelemetry GenAI Semantic Conventions) | 1장 Osmani 6 카테고리 중 "관측"이 한 단어 등장 후 사라짐 | Böckeler의 Sensors와도 직결되는데 챕터 없음. |
| **모델 버전 회귀 운영 노하우** | 4장에서 "Sonnet 4.5 → 4.6 회귀"가 도입 훅으로만 등장 | 회귀를 어떻게 *탐지*·*롤백*·*평가 회귀 회피*하는지의 운영 패턴이 빠짐. |
| **CI/CD 통합 — PR 자동 리뷰·자동 백포트·deploy preview에서의 에이전트** | 어디에도 없음 | 4부 17장(격리)에서 한 줄 정도만. |
| **팀 도입·온보딩·승인 프로세스** | 어디에도 없음 | 책 정체성이 "현업 개발자"이므로 팀 도입은 반드시 다뤄야 한다. |
| **디버깅·재현성·결정성** (temperature, seed, cache hit rate, traces 비교) | 어디에도 없음 | 회귀 사고를 재현하는 능력이 없으면 보안 4부도 공허하다. |

**근거**:
- 레퍼런스 §10에서 LangSmith/Helicone/Langfuse가 누락되어 있어 계획이 그대로 따라간 것으로 보이지만, 종합 바이블이라면 *레퍼런스가 빠뜨린 것을 본문이 메워야* 한다.
- Osmani의 6 카테고리 중 마지막이 "관측·비용 메트릭"인데(레퍼런스 §1.2), 1장에서 6개를 나열만 하고 본문 22장 어디서도 "관측·비용"이 단독 챕터를 받지 못한다.
- OpenAI 1M LOC 실험(레퍼런스 §5.2)에서 "Garbage Collection 에이전트"가 22장으로 갔지만, 그 실험의 핵심 운영 노하우(PR 리뷰 워크플로, CI 가이드)는 어디에도 없다.

**제안**:
- **신규 챕터 — "관측과 비용: Sensors의 다른 얼굴" (3부 또는 4부 사이)** 약 14쪽.
  - 5축 메트릭(latency, token, cost, success rate, regression delta)
  - 트레이스 도구 비교 (LangSmith·Helicone·Langfuse·Phoenix)
  - OpenTelemetry GenAI Semantic Conventions 소개
  - 비용 알람·throttling·model fallback 패턴
  - 모델 버전 회귀 탐지 회로 (4.5→4.6 사고를 케이스 스터디)
- **신규 챕터 — "팀에 하네스를 도입하기" (5부 21장 직후 또는 22장 직전)** 약 12쪽.
  - 온보딩 단계 (1주차·2주차·1개월 체크리스트)
  - 승인 프로세스 (도구 권한·MCP 서버 화이트리스트·시크릿 관리)
  - PR에서 에이전트 사용 정책 (라벨링·작성자 표기·diff 크기 한도)
  - 비용 분배·과금 리포팅
  - 갈등 사례 (시니어 vs 주니어, 보안팀 vs 개발팀)
- 22장에서 "Humans steer. Agents execute → Humans design environments. Agents iterate"로 닫기 전에, 위 두 챕터가 들어와야 22장의 무게가 산다.

### [Critical-2] 4부(보안)가 종합 바이블 약속에 비해 가볍다

**현상**: 4부는 4장 56쪽으로 1·5부와 동일하다. 그런데 책의 **차별화 포인트 3개 중 2개**(하네스 자체 보안, 평가 회피)가 4부에 몰려 있다.

**근거**:
- 레퍼런스 §8(샌드박싱·보안)이 5절(Trifecta·Shai-Hulud·격리·MicroVM·Sandbagging)이고 분량이 §7(장기 실행)과 비슷하다.
- 그런데 계획에서는 3부(장기 실행) 5장 75쪽 vs 4부(보안) 4장 56쪽 — **공급(레퍼런스)과 본문 분배가 어긋난다.**
- 한국어권에서 정면으로 다루는 첫 책이라면, 보안만큼은 두꺼워야 후속 책이 따라 잡기 어려운 해자가 된다.

**제안**:
- 4부에 **신규 챕터 추가 — "프롬프트 인젝션 공격 카탈로그와 방어"** 약 12~14쪽.
  - Direct vs indirect injection
  - 도구 description injection (레퍼런스 §8.3 마지막 단락)
  - 데이터 위치 분리·canary token·output channel whitelist
  - GitHub MCP 익스플로잇 분해 (Willison 사례)
  - 한국 환경 사례 — 사내 위키·Jira 인젝션 시뮬레이션
- 또는 **17장(격리 3축)을 두 챕터로 분할**:
  - "격리 3축 (이론·아키텍처)" — Inspect·Firecracker·gVisor 비교
  - "격리 운영 레시피 (Docker·devcontainer·sandbox-exec 실습)" — 30분 PoC 단계별
- 결과적으로 4부가 5장 70쪽으로 늘어나면 종합 바이블 무게가 산다.

### [Critical-3] 22장 마무리가 종합 바이블치고 가볍다 (12쪽)

**현상**: 22장 메타 하네스 + 미래 전망 + 5가지 출구 능력 호출 + Hashimoto 인용으로 닫는다. 12쪽. 22개 챕터 중 가장 짧다.

**근거**:
- 종합 바이블의 마지막 챕터는 보통 **"현장의 목소리"** 또는 **"5년 후"** 류로, 책의 모든 챕터를 한 번 더 호출하는 무게가 있어야 한다.
- 현재 22장은 "메타 하네스"라는 토픽으로 닫는데, 이건 책 전체의 무게를 결산하기엔 좁은 주제다.
- DDIA(*Designing Data-Intensive Applications*)의 마지막 장 *Future of Data Systems*가 60쪽인 것을 떠올리면, 22장은 "5년 뒤의 하네스" 류로 더 두꺼워야 한다.

**제안**:
- 22장을 두 부분으로 분할 (또는 22장을 두껍게 + 23장을 신설):
  - 22장: **메타 하네스 — 스스로 진화하는 시스템** (16~18쪽). revfactory의 자동 갱신 후크, OpenAI Garbage Collection을 코드 수준에서 분해. 메타 하네스의 위험(자기 강화 편향)도.
  - 23장(에필로그): **현장의 목소리·5년 뒤** (10~12쪽).
    - 한국·해외 실무자 5~7명 인터뷰 압축 (revfactory 황민호·instructkr Sigrid·Haandol·toss tech·우아한형제들 + Hashimoto/Böckeler 인용 모자이크)
    - 5가지 출구 능력 자가 점검표
    - "에이전트가 또 같은 실수를 하면…" Hashimoto 인용으로 닫기
- 이 구조는 *Designing Data-Intensive Applications*가 *Conclusion + The Future of Data Systems*로 닫는 패턴과 동형.

### [Critical-4] 풀쿼트 분포가 한 인용에 쏠릴 위험

**현상**: 레퍼런스 §11이 9 카테고리에 약 25개 풀쿼트를 정리했다. 그런데 계획의 챕터별 인용 매핑을 보면:
- Hashimoto의 "Anytime an agent makes a mistake…" → 1·2·8·22장 4번 등장 가능성 (반복 위험).
- Osmani의 "Agent = Model + Harness" → 1·4장 (괜찮음).
- Böckeler "When the agent struggles…" → 6장 (1회만, 다행).
- "Humans steer. Agents execute" → 22장 (1회만, 좋음).

**근거**:
- 종합 바이블에서 같은 인용을 4장에서 반복하면 인용의 무게가 닳는다.
- 챕터별 *시그니처 인용*을 한 개씩 정해야 독자가 "그 챕터=그 인용" 매핑을 갖게 된다.

**제안**:
- 22개 챕터에 **레퍼런스 §11의 25개 풀쿼트를 1:1로 배분 (1장 = 1 시그니처 인용)**, 나머지는 본문 안에서 한 번만 사용.
- Hashimoto의 "mistake → engineer a solution"은 **2장(6단계)** 시그니처로 고정. 22장은 **OpenAI의 "Humans steer. Agents execute"**로 충분히 무게 있다.
- 각 챕터 도입 또는 종결 직전에 시그니처 인용 1개를 박아 챕터의 정체성을 만든다.

### [Critical-5] 실습이 "코드/설정 점검" 일변도

**현상**: 22장의 실습을 훑어보면 다음 패턴이 압도적이다:
- "자기 프로젝트의 X를 Y로 분류해 보기" (1·2·5·6·14·17장 등)
- "X를 자기 환경에 적용하고 결과 측정" (8·9·10·11·12·13·19장 등)

**근거**:
- 종합 바이블 독자는 코드 점검만 22번 하면 지친다.
- 일부 챕터는 **시나리오 시뮬레이션·인터뷰·디스커션** 형식이 더 잘 맞는다.
  - 4장(모델보다 중요한 하네스): "팀 회의 시나리오 — Sonnet 4.6 회귀를 어떻게 보고할 것인가" 롤플레이.
  - 14장(단일 vs 다중): 가상 토론 — Walden Yan과 Anthropic이 한 회의실에 있다면.
  - 15장(Lethal Trifecta): 자기 봇의 trifecta 진단을 *팀원과 함께 색칠하기* (페어 워크).
  - 21장(한국 생태계): revfactory·claw-code 코드 30분 읽기 + **메인테이너에게 디스코드로 질문 1개 보내기** (실제 커뮤니티 진입).

**제안**:
- 22개 실습 중 **최소 5개를 시나리오·인터뷰·디스커션 형식**으로 다양화.
- 부록 A 직전에 "실습 가이드 — 22개 실습을 어떻게 4주 커리큘럼으로 묶는가"를 부록 E와 통합.

## 6. Style Notes (Toby 스타일 적용 비판)

### [Critical-Style] 일부 챕터의 "스타일 적용 포인트"가 너무 일반적

**문제 챕터들**:
- **5장**: "왜 같은 에이전트가 어떤 날은 멀쩡하고 어떤 날은 미쳐 날뛸까?" — 이 챕터는 CAR 분해라는 *추상적 분류 작업*인데 도입은 감정적 훅 — 본문과 어긋난다.
  - 권고: "AGENTS.md를 백 줄 썼는데도 같은 실수를 한다고 해보자. 그 백 줄은 어디에 속할까? 아니, 우리가 어떤 *축*으로 분해해야 빈자리가 보일까?"
- **6장**: "에이전트가 또 같은 곳에서 막혔다 — 짜증부터 내지 말고, 무엇이 빠졌는지 물어보자" — 좋은 도입이지만 챕터 내용(사이버네틱 폐루프)을 따라잡지 못한다.
  - 권고: 본문 중간에 "1948년 Wiener가 사이버네틱스를 정의할 때, 그가 본 것은 무엇이었을까?" 류 역사적 수사 질문을 1개 추가.
- **17장**: "에이전트가 `rm -rf`를 시도했다는 사연을 본 적 있을 것이다" — 시작은 좋다. 그런데 *공감*에서 *행동*으로 넘기는 다리가 약하다.
  - 권고: "그 사연이 우리 일이 되지 않게 — 우리는 30분 안에 격리를 짤 수 있다. 진짜로? 그렇다, 진짜로."

### [Should-Style] 강한 비유가 필요한 챕터에 비유가 없다

- **3장(평가·런타임 동형성)**: 추상이 강한데 비유가 없다. "동형성"이 무엇인지 *말의 마구 비유*로 다시 풀자 — "마구가 평가 코스에서도, 실제 길에서도 같은 형태인 것처럼…"
- **10장(Context Rot)**: "긴 컨텍스트의 배신"이라는 부제는 좋지만 본문 비유가 약하다. 권고 — "5만 단어짜리 회의록을 받았다. 거기서 *세 줄짜리 결정*을 다시 찾으려면 어떻게 할까? 모델도 같은 일을 한다."
- **18장(Sandbagging)**: "평가 모드를 인지하는 모델"이라는 추상에 *학생이 시험관 표정을 보고 답을 바꾸는 비유*를 한 절로.

### [Should-Style] 챕터별 "시그니처 감정 단어" 매핑

- §9.3에서 "난감하다·찜찜하다·번거롭다·끔찍한 일이다" 4개를 챕터당 1회 이상 등장시킨다고 했는데, **챕터별로 어느 단어가 어울리는지** 사전에 정해 두면 일관성이 산다.
  - 8장(거대 AGENTS.md): "찜찜하다"
  - 11장(2-prompt): "끔찍한 일이다" (다음날 처음부터 다시)
  - 15장(Trifecta): "난감하다"
  - 17장(격리): "번거롭다" → "그런데 이 번거로움이 우리를 살린다"

- 21장(한국 생태계)을 "책 전체에서 가장 청유형 강하게"로 못 박은 건 좋다.
