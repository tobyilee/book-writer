# 15장. 관측과 비용 — Sensors의 다른 얼굴

## 15.1 측정되지 않는 하네스는 운영되지 않는다

3부의 마지막 챕터에서 우리가 마주할 사실은 — 번거롭다.

지금까지 우리는 컨텍스트 부패에 대응하고(10·11장), 루프를 단순하게 만들고(12장), 평가자를 분리하고(13장), 단일 vs 다중의 합의를 잡아 왔다(14장). 모든 자리에서 우리는 *결정*을 내렸다. 그러나 그 결정들이 *실제로 잘 작동하고 있는지*를 — 우리는 어떻게 아는가.

답은 단순하지만 번거롭다. **숫자를 봐야 한다.** Latency가 어떻게 변하고 있는지, 토큰이 어디서 새고 있는지, 비용이 어느 자리에서 폭주하는지, 성공률이 어제와 비교해 떨어졌는지. 그리고 가장 잔인한 자리 — *모델 회귀가 조용히 들어왔을 때*, 우리가 *먼저* 알 수 있는지.

이 챕터의 한 줄은 그래서 이렇게 적는다.

> *측정되지 않는 하네스는 운영되지 않는다.*

이 한 줄이 산업 합의에 가깝다. Birgitta Böckeler의 *Sensors are not optional*이 같은 자리를 가리킨다.[^bockeler-main] 6장에서 우리는 Sensors가 *피드백 폐루프의 절반*이라는 것을 봤다. 그러나 6장의 Sensors는 *에이전트 자기 학습용*이었다. 15장의 Sensors는 *우리(사람) 운영용*이다. 같은 단어, 다른 얼굴.

번거롭다. 정말 번거롭다. 5축 메트릭을 정의하고, 트레이스 도구를 붙이고, OpenTelemetry 컨벤션을 따르고, 비용 알람을 설정하고, 회귀 골든 케이스를 짠다. 한 주 짜리 일이다. 그러나 이 번거로움이 — *우리를 살린다*. 그래서 15장이 3부의 마지막 자리에 있다. 척추가 움직이는 마지막 단계 — *움직임이 정량으로 관측되는* 자리.

## 15.2 5축 메트릭 — 정의·계산식·임계 알람

먼저 *무엇을* 재는가부터. 다섯 축이 표준이다.

| 축 | 정의 | 계산식 | 임계 알람 |
|----|------|--------|----------|
| **Latency** | 사용자 요청부터 첫 토큰까지의 시간 (TTFT) + 완료까지의 시간 (E2E) | p50 / p95 / p99 분위로 측정 | p95가 *전주 평균의 1.5배*를 넘으면 알람 |
| **Token usage** | 입력 + 출력 + (cached) 토큰 합 | 작업당 / 일별 / 모델별 분리 | 일일 총 토큰이 *전주 평균의 2배* 이상이면 알람 |
| **Cost** | 모델 단가 × 토큰. (현재 Anthropic·OpenAI·Google 단가 기준) | 작업당 / 사용자당 / 모델별 | 일일 비용이 *예산 상한의 80%*를 넘으면 사전 알람, 100%면 throttling |
| **Success rate** | 작업이 명세된 종료 조건을 만족한 비율 | (성공 작업 수) / (전체 작업 수) | 7일 이동평균이 *기준선의 -10%* 떨어지면 알람 |
| **Regression delta** | 골든 케이스 평가에서 *어제 대비* 점수 변화 | 어제 점수 - 오늘 점수, 케이스별 | 회귀가 *3개 이상* 발생하면 즉시 알람 |

다섯 축의 *우선순위*는 도메인에 따라 다르지만, 일반 코딩 워크플로에서 가장 *조용히* 망가지는 축은 5번째 — *Regression delta*다. 1~4축은 사용자가 *불편을 호소*해서 알게 된다. "느려졌다", "비싸졌다", "실패한다." 그러나 회귀 — *모델이 조용히 멍청해진 것* — 는 사용자가 *호소하지 않을 때도* 일어난다. 그래서 우리가 *먼저* 측정해야 한다.

15.6에서 회귀 운영 케이스 스터디를 본다. 그 전에 *어떤 도구로* 이 5축을 모으는가를 보자.

## 15.3 트레이스 도구 4종 — 핵심 차이

LLM 운영 트레이스 도구의 4대 후보가 있다. 각자 강점이 다르다.

| 도구 | 호스팅 모델 | 강점 | 약점 |
|------|------------|------|------|
| **LangSmith** | SaaS (LangChain Inc.) | LangGraph·LangChain 통합 1급, UI가 가장 친숙 | 자체 호스팅 불가능, 데이터가 외부로 나감 |
| **Helicone** | SaaS + 자체 호스팅 가능 | 프록시 기반 — *코드 수정 거의 없음*, OpenAI/Anthropic 호환 | LangGraph 같은 그래프 트레이스는 약함 |
| **Langfuse** | 자체 호스팅 1급 (오픈소스) | 데이터 주권 보장, 가격 통제 | 셋업이 약간 무겁다 |
| **Phoenix (Arize)** | 자체 호스팅 (오픈소스) + Arize SaaS | OpenInference 표준 채택, *프로덕션 모니터링*이 강함 | 작은 프로젝트엔 다소 과함 |

선택 휴리스틱을 한 줄로 정리하면:

- **빠른 PoC** = LangSmith. 한 시간 안에 띄운다.
- **데이터 주권 필수** = Langfuse. 자체 호스팅 + 오픈소스.
- **코드 수정 최소화** = Helicone. API 엔드포인트만 바꾸면 끝.
- **프로덕션급** = Phoenix. OpenInference 위에 안정.

우리 환경의 정답은 보통 *둘의 조합*이다. 개발 단계는 LangSmith나 Helicone, 프로덕션은 Phoenix 또는 Langfuse. *둘의 데이터를 합치는 표준*이 다음 절의 자리다.

## 15.4 OpenTelemetry GenAI 시맨틱 컨벤션

도구가 4종인데 트레이스 데이터 형식이 *다 다르면*, 우리가 도구를 *바꿀 수 없다*. 한 도구에 락인되는 것이 트레이스의 *원래 비용*이 된다.

이 락인을 푸는 표준이 **OpenTelemetry GenAI Semantic Conventions**다. 2025년부터 OpenTelemetry 커뮤니티가 *LLM 호출 trace의 표준 속성*을 정의하기 시작했다. 핵심 속성 셋.

```
gen_ai.system               = "anthropic" | "openai" | "google" | ...
gen_ai.request.model        = "claude-sonnet-4-6" | "gpt-5" | ...
gen_ai.usage.input_tokens   = 1234
gen_ai.usage.output_tokens  = 567
gen_ai.response.id          = "msg_abc123..."
```

이 속성을 도구가 *공통으로* 인식한다면, 우리는 LangSmith에서 출력한 trace를 Phoenix가 읽을 수 있다. *벤더 락인이 풀린다*. Phoenix와 Langfuse가 이 표준을 채택했고, LangSmith·Helicone도 호환을 진행 중이다.

실천 한 줄: **새 도구를 도입할 때 *OpenTelemetry GenAI 컨벤션 호환*을 1순위 기준으로.** 이 한 줄이 1~2년 뒤 우리에게 도구를 *바꿀 자유*를 준다. 컨벤션을 따르는 도구의 trace는 다음 도구가 *자동으로* 읽는다.

## 15.5 비용 알람 · throttling · model fallback

5축 중 *가장 빠르게* 우리를 깨우는 축이 비용이다. 잘못된 프롬프트 하나가 무한 루프에 들어가면 — 12.4에서 본 *Principal Skinner의 부재* 상황 — 한 시간에 수백 달러가 날아간다. 자다가 일어나 청구서를 본 새벽이 누구나 한 번 있다.

비용을 한 줄로 통제하는 패턴 셋.

**(1) 사전 알람.**

비용이 *예산 상한의 80%*를 넘는 순간 슬랙으로 알람. 운영자가 *경고 받는* 자리. 이 자리에서 운영자가 *수동으로* 작업을 멈출 수 있다. Anthropic Admin API, OpenAI Usage API가 일별 사용량을 노출하므로, 한 시간 cron이면 충분하다.

```python
# 의사 코드
budget_limit = 100  # USD/day
current_usage = anthropic_admin.get_today_usage()
if current_usage > budget_limit * 0.8:
    slack_notify(f"Anthropic 비용 경고: ${current_usage}/{budget_limit}")
if current_usage > budget_limit:
    slack_notify(f"Anthropic 비용 *상한 도달*: throttling 시작")
    enable_throttling()
```

**(2) Throttling.**

상한 *도달 시* 자동으로 호출 빈도를 제한한다. 모든 API 호출 앞에 *큐*를 두고, 큐의 처리 속도를 제한. 비용이 다음 시간(또는 다음 날)으로 *분산*된다. 작업이 *완전히* 멈추는 것은 아니지만 *천천히* 진행된다. 사용자에게는 *지연 알림*. 운영자에게는 *원인 조사 시간*.

**(3) Model fallback.**

비싼 모델(Sonnet 4.6) → 싼 모델(Haiku 4.5) 자동 전환. 비용 임계 도달 시 *현재 진행 중인 작업*은 그대로 두고, *새로 들어오는 작업*은 fallback 모델로 라우팅. 13장의 Architect/Editor 패턴이 *비용 차원에서* 자연스러워지는 자리. 일상 작업의 80%는 Haiku로도 충분하다는 사실을 도메인 측정으로 검증해 두면, fallback이 *품질 손실 없이* 비용을 30~50% 줄인다.

이 셋을 *순서대로* 구현하자. (1) 사전 알람만 있어도 야간 청구서 사고는 80% 막힌다. (2) throttling은 *주말 사고*를 막는다. (3) fallback은 *사용자 경험*을 보존하면서 비용을 줄인다.

## 15.6 운영 케이스 스터디 — Sonnet 4.5→4.6 회귀

4장에서 우리는 *forward bridge*를 받았다. Sonnet 4.5→4.6 회귀 사건이 일어났을 때, 그 사건이 *왜 일어났고 어떻게 운영적으로 대응해야 하는지*가 15장의 자리에서 마무리된다고 약속했다. 이제 그 자리다.

**사건의 골격.**

2026년 2월, Anthropic이 Sonnet 4.6을 출시했다.[^claude-46-card] 시스템 카드 기준 4.5보다 *전반적으로 우월*. Terminal-Bench 2.0 점수도 올랐다. 그런데 — 일부 코드 도메인에서, 특히 *한국어로 적힌 PRD*와 *복잡한 시스템 프롬프트*가 결합된 환경에서, 4.6이 4.5보다 *조용히 떨어졌다*. 사용자 호소가 산발적으로 들어왔다. "어제까지 잘 되던 작업이 오늘 이상해졌다."

이 회귀가 *조용한* 이유는 평균 점수는 올랐기 때문이다. Anthropic이 *전반적*으로 측정한 벤치는 다 좋았다. 그러나 *우리 도메인의 작업*에서는 일부 회귀가 있었다. **모델 회귀는 *전체 평균*이 아니라 *우리 도메인*의 사실이다.**

**탐지·롤백·재평가의 운영 패턴.**

회귀를 *먼저* 알고, *되돌리고*, *다시 평가하는* 회로를 우리가 갖고 있어야 한다.

```
탐지 → 롤백 → 재평가
```

각 단계의 코드 수준 모습.

**(1) 탐지 — 골든 케이스 30개 + 일일 자동 평가.**

부록 H가 다룰 골든 케이스 카탈로그를 미리 짠다. *우리 도메인*의 작업 30개. 정답이 명확하고, 평가가 자동화 가능한 케이스만. 매일 새벽 자동 cron으로 30개를 *현재 모델*로 돌리고 점수를 기록.

```python
# 의사 코드
for case in golden_cases:
    result = current_model.run(case.prompt)
    score = case.evaluator(result)
    db.insert(date=today, case_id=case.id, score=score)

yesterday_avg = db.avg_score(yesterday)
today_avg = db.avg_score(today)
if today_avg < yesterday_avg - 0.10:
    alert(f"회귀 의심: 어제 {yesterday_avg}, 오늘 {today_avg}")
```

15.2의 *Regression delta* 축이 여기서 산출된다.

**(2) 롤백 — 모델 버전 핀 (pin).**

회귀가 탐지되면 *이전 모델*로 즉시 되돌린다. 단, 이게 가능하려면 *모든 호출에 모델 버전이 명시*되어야 한다. `claude-sonnet-4-6` 같은 *별칭*이 아니라 *날짜 도장* (`claude-sonnet-4-6-20260217`) 식의 핀. 별칭은 *다음 날* 다른 버전을 가리킬 수 있고, 이 silent pinning이 회귀의 *근본 원인*인 경우가 많다.

```python
# 잘못된 예
model = "claude-sonnet-4-6"  # 별칭, 시간이 지나면 가리키는 게 바뀔 수 있음

# 권장
model = "claude-sonnet-4-6-20260217"  # 날짜 핀, 결정적
```

롤백은 한 줄 변경이다. `20260217` → 이전 안정 핀. 30초 안에 가능하다.

**(3) 재평가 — 회귀가 *어떤 케이스*에서 일어났나.**

골든 케이스 30개 중 *몇 개*가 회귀했는지가 다음 결정의 단서다. 3개 이상이면 모델 자체의 문제로 본다. 1~2개면 그 케이스가 *원래* 경계 케이스였을 가능성. 회귀한 케이스의 *공통 패턴*을 본다 — 한국어 비율이 높았는가, 시스템 프롬프트가 길었는가, 도구 호출이 많았는가. 이 패턴이 *우리 도메인의 회귀 단서*가 된다.

Anthropic이나 OpenAI에 *피드백을 보낼* 자리도 여기다. "한국어 + 긴 시스템 프롬프트 환경에서 회귀를 봤다"는 한 줄이, 다음 모델 출시 때 *반영될 수 있다*. 회귀는 *우리만의 발견*으로 두지 말고, 1차 벤더에 *돌려보내자*.

**이 케이스가 보여 주는 핵심.**

회귀는 *조용히* 들어온다. 평균은 올라가도 우리 도메인은 떨어질 수 있다. 그래서 *우리 도메인의 골든 케이스*를 *일일 자동 평가* 회로 위에 올리는 일이 — 번거롭지만 — 우리를 살린다.

이 회로 없이 운영하는 것은, 정량 게이트 없이 코드 푸시하는 것과 같다. 단순한 단위 테스트가 *지금* 빠지면 우리는 알아챌 수 있다. 모델 회귀 탐지 회로가 빠지면 — 우리는 *몇 주 뒤* 사용자 호소가 누적되어서야 알아챈다. 그 *몇 주*가 우리 신뢰의 비용이다.

## 15.7 시그니처 인용 박스

> ### 시그니처 인용
>
> > "Sensors are not optional in a cybernetic harness — they are the *only* way to close the loop."
> >
> > *사이버네틱 하네스에서 Sensors는 옵션이 아니다 — 폐루프를 닫는 *유일한* 방법이다.*
> >
> > — Birgitta Böckeler, *Harness engineering for coding agent users* [^bockeler-main]
>
> 이 한 줄이 우리에게 박는 다섯 가지 결정:
> 1. **5축 메트릭을 *모두* 측정한다.** Latency · Token · Cost · Success rate · Regression delta.
> 2. **OpenTelemetry GenAI 컨벤션 호환을 *1순위 기준*으로 도구를 고른다.** 락인을 사전에 푼다.
> 3. **비용 알람·throttling·fallback의 *셋*을 순서대로 구현한다.** 사전 알람 → throttling → fallback.
> 4. **모델 버전을 *날짜 핀*으로 박는다.** 별칭은 silent pinning의 원인.
> 5. **골든 케이스 30개 + 일일 자동 평가.** 회귀를 *먼저* 알자.

## 15.8 실습 — 5축 대시보드 첫 버전

> ### 실습: 자기 워크플로에 5축 대시보드의 *3축*을 일일 그래프로 띄우기
>
> **목적:** 5축 메트릭이 *추상*에서 *그래프*가 되는 자리를 자기 손으로 만든다. 회귀가 들어오는 날, *우리가 먼저 알 수 있는지*를 본다.
>
> **준비물.**
> - LangSmith·Helicone·Langfuse·Phoenix 중 한 개 (15.3에서 고른 도구).
> - 자기 코딩 에이전트 (Claude Code, Codex CLI, Cursor — 어느 것이든).
> - 골든 케이스 5개. (30개는 다음 분기 목표, 첫 시작은 5개로.)
>
> **절차 (3시간).**
>
> 1. **(30분) 트레이스 도구 셋업.** 도구 가이드 따라 한 시간 안에 띄운다. PoC 단계는 LangSmith가 가장 빠르다. 자기 에이전트의 모든 모델 호출이 *trace*로 잡히는지 확인.
>
> 2. **(60분) 골든 케이스 5개 짜기.** 자기 도메인의 *정답이 분명한* 작업 5개. 각 케이스마다:
>    - 입력 prompt
>    - 기대 출력 (또는 *통과 조건*)
>    - 자동 평가 함수 (예: "출력에 'TODO'가 없으면 통과", "JSON 파싱 가능하면 통과")
>
>    한국어 케이스 2개, 영어 케이스 3개 정도가 균형이 맞다. 10장의 한국어 토큰 비용이 회귀 패턴에 영향을 주는지 보려면 *섞어 두자*.
>
> 3. **(45분) 일일 자동 평가 cron.** 30분 cron으로 다음을 돌린다.
>    ```bash
>    # 의사 코드
>    for case in 5_golden_cases; do
>      result=$(claude --file $case.prompt)
>      score=$(eval_function $result)
>      curl -X POST $METRICS_DB -d "{date: today, case: $case, score: $score}"
>    done
>    ```
>
>    매일 새벽 1번 자동 실행. 점수가 DB(혹은 Google Sheets라도)에 누적.
>
> 4. **(45분) 3축 그래프 띄우기.** 트레이스 도구의 대시보드에 다음 셋:
>    - **Token usage** — 일별 토큰 사용량. 케이스별, 모델별 분리.
>    - **Cost** — 일별 비용. 예산 상한선과 같은 그래프에.
>    - **Regression delta** — 일별 골든 케이스 평균 점수. 어제 점수와의 *차이*를 보조 그래프로.
>
>    Latency와 Success rate는 다음 단계 (다음 분기) 목표.
>
> **무엇을 보는가.**
>
> 일주일 운영하면서 다음을 본다.
>
> - 토큰 사용량의 *주중·주말 패턴*이 보이는가. 보이면 *내가 작업하지 않는 시간에도* 토큰이 쓰이고 있을 가능성. 어디서 새는지 trace로 추적.
> - 비용이 *예측 가능한* 곡선인가, *튀는* 날이 있는가. 튀는 날은 *왜* 튀었는지 trace를 본다.
> - 골든 케이스 점수가 *흔들리는가*. 같은 모델·같은 케이스·다른 점수가 나오면 — 모델의 *결정성*이 문제. *시드 고정*이나 *temperature=0* 적용을 검토.
>
> **결과 한 줄 메모.**
> ```
> 5축 대시보드 v1, 일주일 운영.
> 발견 1: 주말 토큰 +30% (자동 cron이 멈추지 않고 있었음).
> 발견 2: 화요일 비용 2배 (실험 작업 1건이 무한 루프에 빠짐, Principal Skinner 부재).
> 발견 3: 한국어 케이스 점수가 영어 케이스보다 0.5점 낮게 안정 — 회귀가 아니라 *지속적 격차*.
> 다음 분기 목표: 골든 케이스 5 → 30, Latency·Success rate 추가, fallback 구현.
> ```

이 실습을 한 번 하고 나면, *측정되지 않는 하네스*가 *얼마나* 위험한지가 *느낌*으로 와 닿는다. 그 느낌이 15장의 진짜 산출물이다.

## 15.9 마무리 — 번거로움이 우리를 살리는 자리

이 챕터에서 우리가 얻은 것은 한 가지 *번거로움의 정당화*다. 5축 메트릭, 4종 트레이스 도구, OpenTelemetry 컨벤션, 비용 알람·throttling·fallback, 골든 케이스 일일 평가. 한 주 짜리 일이고 — 솔직히 — 코드를 짜는 *재미*는 별로 없는 일이다. 그러나 이 번거로움이, *조용한 회귀가 들어오는 날*에 우리를 *먼저* 알게 한다.

Sonnet 4.5→4.6 회귀 사건이 그 자리를 보여 줬다. 평균은 올라도 우리 도메인은 떨어질 수 있다. 우리가 *우리 도메인의 골든 케이스*를 갖고 있지 않으면, *몇 주 뒤* 사용자 호소가 누적되어서야 안다. 그 몇 주의 신뢰 비용이 — 한 주 짜리 셋업 노동보다 — 비싸다. 거래는 명확하다.

이 거래를 받아들이는 자리에서 3부가 끝난다.

3부의 척추를 정리해 보자. 우리는 — 컨텍스트는 부패한다는 사실을 받아들였고(10장), 매 세션을 깡통으로 보고 디스크가 메모리를 대체하게 했고(11장), 루프를 한 줄 bash로 단순화시켰고(12장), 평가자를 분리해 자기 평가의 함정을 끊었고(13장), 단일 vs 다중의 합의를 잡았고(14장), 그 모든 결정을 *5축 메트릭 위*에 올렸다(15장). 9장까지가 *서 있는* 척추였다면, 15장 끝의 척추는 *걷고, 측정되고, 회귀를 먼저 아는* 척추다.

기억해두자. 측정되지 않는 하네스는 운영되지 않는다. 그리고 측정의 *번거로움*은 우리 신뢰의 비용을 *예방하는* 가장 싼 보험이다. 한 주를 투자하자. 다음 분기에 우리는 회귀를 *먼저* 안다.

이제 정량으로 지킬 수 있게 됐다면 — 다음 자리는 공격이다. 누군가 *우리 하네스 자체*를 노리는 자리. 4부의 첫 챕터, *Lethal Trifecta — Simon Willison의 위협 모델*로 간다. 측정되는 척추 위에 *방어 레이어*를 한 겹 더 입힌다.
