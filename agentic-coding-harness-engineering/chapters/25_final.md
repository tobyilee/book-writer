# 25장. 팀에 하네스를 도입하기 — 1주차·2주차·1개월

이런 자리에 자주 선다. 혼자서 Claude Code나 Cursor를 쓰면서 생산성이 한 단계 올라간다. 흥분된다. 다음 주 팀 회의 자리에서 "이거 우리 팀 다 같이 써봅시다"라고 말한다. 그 다음 무슨 일이 일어나는가? *별일이 안 일어난다.*

시니어는 "내가 직접 짜는 게 더 빠른데"라며 시큰둥하다. 주니어는 "에이전트에 너무 의존하는 거 아닌가요"라며 조심스럽다. 보안팀이 "어 잠깐, 그게 외부 API 호출하는 거예요? 개인정보 어디로 가요?"라며 회의를 또 잡자고 한다. CFO가 "한 사람당 월 $200이라는데, 30명이면 6천 달러. ROI 자료 좀 주세요"라고 메일을 보낸다.

솔직해지자. **기술 도입은 코드보다 사람의 합의가 어렵다.** 25장은 그 사실을 인정한 자리에서 출발한다. 우리가 알게 된 것을 5명·30명·300명 팀에 *어떻게* 옮겨심을지의 자리다.

## 1주차 — 개인 환경 표준화

가장 많이 실패하는 자리가 1주차다. 이유는 — *너무 많이 한꺼번에* 하려고 해서다. 1주차의 핵심은 *모든 팀원의 개인 환경이 같은 출발선에 서게 하는 것*이다. 그것 하나만 한다.

**1주차 개인 환경 체크리스트.**

- [ ] 1차 도구 1개 설치 (Claude Code 또는 Codex CLI 또는 Cursor 중 하나)
- [ ] 회사 인증 + 청구 계정 설정
- [ ] 사내 LLM 게이트웨이가 있다면 그쪽으로 라우팅 설정
- [ ] AGENTS.md(또는 CLAUDE.md) 1개 작성 — 100줄 이내
- [ ] 자기 자주 쓰는 도구 5개를 시스템 프롬프트에 정리
- [ ] 첫 PR을 에이전트 도움으로 만들어 보기 (라벨 `agent-assisted` 부여)

이 6개가 1주차의 전부다. 더 욕심내지 말자. 6개 중에서 가장 어려운 게 *AGENTS.md 100줄 이내*다. 8장에서 본 "큰 AGENTS.md의 실패"를 1주차부터 새겨야 한다. 처음부터 500줄을 짜겠다는 욕심이 나오는데 — 그러면 다음 주에 *아무도 그 파일을 안 읽는다*. 100줄 안에 들어가는 자리만 적자. 나머지는 부록 A의 디렉토리 분산 패턴으로 갈라 둔다.

**1주차 회의 1개.** 30분짜리 한 번이면 충분하다. 안건은 두 가지 — *우리 팀의 1차 도구를 어느 것으로 정할 것인가*, *AGENTS.md의 첫 100줄에 무엇을 넣을 것인가*. 길어지면 안 된다. 결정을 내리고 헤어진다. 결정이 안 나면 — 일단 한 도구로 잠정 결정하고, 2주 뒤에 다시 본다.

## 2주차 — 팀 도구 권한 + MCP 화이트리스트 + 시크릿 관리

1주차가 *개인*의 자리였다면 2주차는 *팀*의 자리다. 보안팀과 처음 부딪히는 자리이기도 하다.

**2주차 팀 차원 체크리스트.**

- [ ] 사용 가능 MCP 서버 화이트리스트 합의 (5~10개로 시작)
- [ ] 도구 권한 등급 정의 (3단계 권장: 읽기 / 쓰기 / 외부 통신)
- [ ] 시크릿 관리 정책 합의 (단기 토큰, 환경별 분리)
- [ ] 격리 1축 적용 — Docker 또는 devcontainer 표준화
- [ ] 사내 LLM 게이트웨이 통합 (있다면)
- [ ] 비용 한도 알람 설정 (개인별 일일 한도)

MCP 화이트리스트가 가장 중요하다. 부록 B의 25종 카탈로그에서 *우리 팀에 필요한 5~10개*만 고른다. 욕심내면 안 된다. 16장에서 본 Lethal Trifecta — *Private data + Untrusted content + External communication* — 의 위험은 MCP 서버 조합에서 가장 자주 터진다. 적게 시작해서 필요할 때 늘리는 편이 낫다.

도구 권한 등급은 3단계가 적당하다.

- **읽기 등급** — 파일 읽기, 검색, DB SELECT 정도. 모두 허용.
- **쓰기 등급** — 파일 쓰기, 커밋, PR 생성. 권한 게이트(사람 확인) 필수.
- **외부 통신 등급** — 외부 API 호출, 이메일 발송, Slack 메시지. 화이트리스트 + 감사 로그 + 사람 확인 필수.

이 3단계를 AGENTS.md에 명시하면 — 에이전트가 매번 어느 등급의 작업을 하는지 *자기가 알 수 있다*. "이 작업은 외부 통신 등급이라 사용자 확인이 필요합니다"라고 자동으로 묻는다.

**2주차 회의 2개.** 첫 번째는 보안팀과 *공동으로*. MCP 화이트리스트와 도구 권한 등급을 함께 정한다. 두 번째는 팀 내부에서 — *시크릿 관리 + Docker 표준화*. 보안팀과의 회의를 *먼저* 잡는 편이 낫다. 그 결정이 우리 팀 회의의 입력이 되니까.

## 1개월 — PR 라벨링·diff 한도·작성자 표기·비용 분배

1개월에 도달하면 — 우리 팀이 에이전트를 일상적으로 쓰는 자리에 와 있다. 이 자리에서 *프로세스*가 자리잡아야 한다. 안 그러면 다음 분기 회고 자리에서 또 *체감* 얘기로 돌아간다.

**1개월 프로세스 체크리스트.**

- [ ] PR 라벨링 합의 — `agent-assisted` / `agent-authored` / `human-only`
- [ ] diff 한도 합의 — 에이전트 PR의 최대 변경 라인 수 (가령 500줄)
- [ ] 작성자 표기 정책 — 커밋 메시지 + PR 본문의 *Co-Authored-By* 정책
- [ ] 비용 분배 — 팀별·프로젝트별 토큰 비용 분배 메커니즘
- [ ] 5축 메트릭 대시보드 — 15장의 latency·token·cost·success rate·regression delta
- [ ] 첫 평가 회귀 알람 — 22장의 미니 벤치 15개 자동 실행
- [ ] 월간 리포트 템플릿 — 부록 G와 짝

PR 라벨링이 가장 중요한 자리다. 3단계가 적당하다.

- **`agent-authored`** — 에이전트가 *주도적으로* 작성. 사람은 프롬프트를 던지고 결과를 검토.
- **`agent-assisted`** — 에이전트가 *부분적으로* 도움. 사람이 절반 이상 작성.
- **`human-only`** — 에이전트 도움 없이 사람이 작성.

이 라벨링은 회고 자리에서 결정적이다. "에이전트 도입 효과"를 묻는 자리에서 *어떤 PR이 어느 등급이었는가*가 데이터로 남는다. 라벨이 없으면 — 또 *체감* 얘기로 돌아간다.

작성자 표기는 보통 *Co-Authored-By* 트레일러로 푼다. 가령 `Co-Authored-By: Claude (Anthropic) <noreply@anthropic.com>`. 이 한 줄이 — 6개월 뒤 *어떤 코드가 누가 만든 것인가*를 git blame으로 추적할 수 있게 한다.

비용 분배는 — 솔직히 — 가장 골치 아픈 자리다. 사내 LLM 게이트웨이가 있다면 게이트웨이에서 팀별 분배가 자연스럽다. 없다면 — *프로젝트별 API 키*로 갈라 두는 편이 낫다. 한 명이 여러 프로젝트를 오가면 키를 갈아 끼워야 한다는 부담이 있지만, 비용 가시성이 명확해진다.

## 갈등 사례 1 — 시니어 vs 주니어

가장 흔한 갈등이다. 시니어는 "내가 직접 짜는 게 더 빠른데, 굳이 에이전트를 끼워야 하나?"라고 한다. 주니어는 "에이전트에 너무 의존하면 실력이 안 늘 것 같아 불안해요"라고 한다.

두 입장 모두 *부분적으로* 맞다. 그래서 더 어렵다.

시니어의 우려는 — *컨텍스트 스위칭 비용*이다. Hashimoto가 같은 자리에서 한 줄을 박았다.

> "Turn off agent desktop notifications. Context switching is very expensive."[^hashimoto]

시니어가 머릿속에 이미 코드 그림이 다 그려져 있는 자리에서, 에이전트의 결과를 *검토*하는 작업은 *직접 짜는* 작업보다 더 무거울 수 있다. 그게 시니어의 진짜 우려다.

주니어의 우려는 — *학습 박탈*이다. 에이전트가 답을 척척 내놓으면, *왜 그게 답인지*를 따져볼 시간이 없다. 6개월 뒤에 자기가 짠 코드가 아니라 *에이전트가 짠 코드*만 가득한 자리에 서면 — 솔직히, 끔찍한 일이 될 수 있다.

이 갈등의 해법이 *기술적 해법*이 아니다. **역할 분담의 해법**이다.

- **시니어** — *생성*보다 *검토와 통제*에 시간을 쓴다. 에이전트가 만든 PR의 리뷰어 역할, AGENTS.md의 큐레이터 역할, 평가 미니 벤치의 골든 케이스 작성자 역할. 시니어는 1차 생성을 *위임*하고, 자기 시간을 *시스템 설계*에 쓴다.
- **주니어** — *생성*에서 에이전트와 짝지어 일하되, *왜 그 답인지를 항상 묻는다*. AGENTS.md에 "에이전트의 답에 *왜?*를 1번 이상 묻고 그 답을 PR 본문에 적는다"는 규칙을 박아두면, 학습 박탈 우려가 줄어든다.

이 분담을 1개월 회의에서 합의해 두면 — 갈등이 *사라지지는 않지만* 줄어든다.

## 갈등 사례 2 — 보안팀 vs 개발팀

보안팀과 개발팀은 *목적이 다르다*. 개발팀은 속도와 생산성을, 보안팀은 위험 최소화를 우선한다. 하네스 도입 자리에서 둘이 부딪히는 게 자연스럽다.

가상 회의록 한 장면.

> **개발팀:** Claude Code에 GitHub MCP 서버를 붙이려고 합니다. PR 자동 리뷰 워크플로 짜려고요.
>
> **보안팀:** 그 MCP가 우리 private repo 접근 권한 있어요?
>
> **개발팀:** 네, 그래야 의미가 있어서요.
>
> **보안팀:** 그게 동시에 외부 API 호출하잖아요. 그러면 Lethal Trifecta예요. 안 됩니다.
>
> **개발팀:** ... 그럼 무엇을 하라는 건지...

이런 자리에서 가장 자주 무너진다. 둘이 서로 *목적*만 말하고 *구체*를 안 말해서다.

해법은 — **16장의 Lethal Trifecta 분해**를 회의 자료로 들고 가는 것이다.

> "The lethal trifecta for AI agents: private data, untrusted content, and external communication."[^willison-trifecta]

이 셋 중 **두 개로 줄이면 위험이 사라진다**. 그러면 회의가 이렇게 바뀐다.

> **개발팀:** GitHub MCP를 *읽기 등급*으로만 깔겠습니다. 외부 PR 코멘트 작성은 *별도 도구*로 빼서, 그쪽은 private repo 데이터를 못 보게 분리하면 어떨까요?
>
> **보안팀:** 그러면 trifecta가 풀리네요. 진행하셔도 됩니다. 다만 두 도구 사이 데이터가 컨텍스트로 *우회 흘러가지* 않는지 분기마다 감사하겠습니다.

같은 회의가 — *해법이 있는 회의*가 된다. 그 차이가 16장의 분해를 들고 가느냐 마느냐다.

## 갈등 사례 3 — CFO vs 팀장

세 번째 갈등이 가장 어렵다. CFO가 "한 사람당 월 $200, 30명이면 6천 달러. ROI 자료 주세요"라고 메일을 보낸 자리다.

ROI를 *체감*으로 답하면 안 된다. 22장의 미니 벤치 15개와 1개월 PR 라벨링 데이터가 — 이 자리에서 우리를 살린다.

가상 답변 한 장면.

> **팀장:** 9월 한 달 PR 데이터입니다. 총 142개 PR 중 `agent-authored` 47개, `agent-assisted` 73개, `human-only` 22개. 22개의 `human-only` PR 평균 리드 타임이 4.2일, `agent-assisted` 73개의 평균이 2.1일, `agent-authored` 47개의 평균이 1.3일입니다.
>
> **팀장:** 그리고 우리 도메인 평가 골든 15개 점수가 6월 9/15에서 9월 12/15로 올랐습니다. 이건 코드 품질도 같이 올라갔다는 신호로 봐도 됩니다.
>
> **CFO:** ... 흥미롭네요. 그럼 비용 6천 달러 중 *얼마가 어느 효과를 만들었는지* 더 분해할 수 있어요?

이 자리에서 *체감*이 아니라 *데이터*로 답하는 게 가능해진다. 25장 1개월 체크리스트 — PR 라벨링과 비용 분배와 5축 메트릭 — 이 모두 *이 회의를 위해* 깔아둔 자리다. CFO가 메일을 보낸 그 다음 날 — 솔직하게 — 데이터가 손에 있어야 한다. 없으면 또 다음 분기로 미뤄진다.

## 시그니처 인용 박스

> "Each component of the harness encodes an assumption about what the model cannot do on its own."
>
> *하네스의 각 구성 요소는 모델이 스스로 할 수 없는 것에 대한 가정을 인코딩한다.*
>
> — **Anthropic**, *Harness design for long-running application development*[^anthropic-harness2]

이 한 줄을 *팀 도입 자리*에서 다시 읽자. 우리가 1주차·2주차·1개월에 깔아 둔 모든 줄 — AGENTS.md의 100줄, MCP 화이트리스트의 7개, PR 라벨링의 3단계, 5축 메트릭의 6번째 축 — 이 모두 **모델이 스스로 할 수 없는 것에 대한 가정**을 인코딩한 자리다. 모델은 우리 팀의 PR 컨벤션을 *모른다*. 모델은 우리 회사의 보안 정책을 *모른다*. 모델은 CFO가 다음 주에 어떤 데이터를 요구할지 *모른다*. 그 모름을 메우는 게 우리가 하는 일이다.

5장에서 처음 만난 이 인용이, 25장에서 *팀 도입의 거울*로 다시 돌아온다. 5장에서 우리는 이 인용을 *시스템 설계 차원*에서 봤고, 25장에서는 *조직 차원*에서 본다. 같은 인용의 두 얼굴이다.

## 실습 박스 — 자기 팀에 4주 도입 로드맵 초안 작성

이번 주 안에 끝낼 수 있는 실습이다.

자기 팀(가상 또는 실제) 30명을 가정하고, 1주차·2주차·1개월의 회의록 *초안*을 직접 작성해 보자. 형식은 자유다. 다음 정보가 들어가면 된다.

**1주차 회의 (30분).**
- 안건 1 — 우리 팀 1차 도구 결정. 후보 3개와 결정 근거.
- 안건 2 — AGENTS.md 첫 100줄에 들어갈 내용. 6 카테고리 매핑.
- 합의 사항과 다음 주까지 각자 할 일.

**2주차 회의 1 (보안팀과, 60분).**
- 안건 — MCP 화이트리스트 5~10개와 도구 권한 등급 3단계 합의.
- Lethal Trifecta 진단 — 16장 분해를 자료로 들고 가서 우리 팀 후보 도구 조합을 진단.
- 합의 사항과 보안팀이 분기별로 감사할 항목.

**2주차 회의 2 (팀 내부, 30분).**
- 안건 — 시크릿 관리 + Docker 표준화.
- 합의 사항.

**1개월 회의 (60분).**
- 안건 1 — PR 라벨링 3단계 합의.
- 안건 2 — diff 한도와 작성자 표기 정책.
- 안건 3 — 5축 메트릭 대시보드의 첫 화면 디자인.
- 안건 4 — CFO에게 분기마다 보낼 리포트 템플릿.

이 4개 회의록 초안을 *솔직하게* 적자. 자기 팀의 시니어가 어떤 우려를 낼지, 보안팀이 어떤 질문을 던질지, CFO가 어떤 숫자를 요구할지 — 미리 *그 입장에서* 적어 보자. 그러면 실제 회의 자리에 갔을 때 *덜 당황한다*.

이 실습은 30~60분이면 끝난다. 그 시간 투자가 — 다음 분기에 *몇 주의 회의 시간*을 줄여 준다.

## 정리해보자

기억해두자. **혼자 잘 쓰는 건 쉽다. 팀이 잘 쓰는 건 다른 문제다.** 25장의 한 문장이다.

1주차에 개인 환경을 표준화한다. 2주차에 보안팀과 합의해 MCP 화이트리스트와 도구 권한 등급을 정한다. 1개월에 PR 라벨링·diff 한도·작성자 표기·5축 메트릭·미니 벤치를 깐다. 갈등 사례 셋 — 시니어 vs 주니어, 보안팀 vs 개발팀, CFO vs 팀장 — 은 *기술적 해법*이 아니라 *역할 분담*과 *데이터*로 푼다.

조직 정치의 무게가 — 솔직히 — 번거롭다. 코드보다 어렵다. 다만 그 무게를 *체크리스트로 분해*하면 — 한 주씩 해결할 수 있다. 25장의 핵심은 *다음 월요일 아침에 들고 갈 수 있는 도구*를 만들어 두는 것이다. 4주 회의록 초안이 그 도구다.

팀에 옮겨심었다면 — 그 다음은 *하네스가 스스로 자라는 것*이다. 26장이 메타 하네스다.
