# 22장. SWE-Bench·Terminal-Bench·OSWorld — 자기 하네스를 정량 측정하기

이런 자리에 자주 앉게 된다. 분기 회고 자리. 매니저가 묻는다. "올해 우리 에이전트 도입의 *효과*가 어떻게 됐죠?" 누군가는 PR 리드 타임이 줄었다고 말하고, 누군가는 주관 만족도 설문 결과를 띄운다. 누군가는 — 좀 머쓱하게 — "체감으로는 분명히 좋아진 것 같은데요…"라고 말한다. 그 자리에서 모두가 고개를 끄덕인다. 그리고 회의가 끝난다.

찜찜하다. 분명히 좋아진 것 같은데, *정말 그럴까?* 우리가 한 달 전 모델 버전 올리면서, 새 MCP 서버 붙이면서, AGENTS.md를 50줄 깎으면서 — 어느 변경이 어느 효과를 냈는지 분간이 안 된다. 체감만으로는 모자라다. 숫자가 필요하다.

22장은 그 숫자의 자리다. 산업이 만들어 둔 벤치들을 한번 훑고, 그 위에 *우리 도메인에 맞는 미니 벤치*를 올리는 워크플로를 본다.

## 왜 외부 벤치만으로는 모자란가

먼저 솔직하게 인정하자. SWE-Bench Verified 점수가 60%에서 75%로 올랐다는 외부 발표가, 우리 회사의 사내 시스템 리팩토링 작업에 *그대로* 적용된다는 보장은 없다. METR이 2025년 7월에 정리한 *How Does Time Horizon Vary Across Domains?*가 명료하게 못 박은 사실이 이거다.[^metr-domains] 한 도메인의 능력 향상이 다른 도메인에 그대로 옮겨붙지 않는다.

그래서 22장은 두 갈래로 간다. 첫째, 산업이 만들어둔 벤치를 알자. 우리 도메인과 맞닿는 부분이 분명히 있고, 자기 하네스의 위치를 *대충이라도* 잡으려면 외부 좌표가 있어야 한다. 둘째, 그 위에 자기 도메인 미니 벤치 15개를 얹자. 외부 벤치만 쳐다보면 우리는 영원히 *남의 점수*만 본다. 그러면 분기 회고 때 또 체감 얘기로 돌아간다.

## SWE-Bench 시리즈 — 가장 자주 듣게 되는 이름

SWE-Bench는 2023년 Princeton과 OpenAI 공동 연구로 시작했다. 12개 인기 Python 레포에서 실제 GitHub 이슈 2,294개를 모아 패치 채점 프레임을 만들었다.[^swe-agent-paper] 핵심 아이디어는 단순하다 — 사람이 닫은 이슈와 그 패치가 통과한 테스트 묶음을 정답지로 쓴다. 모델이 만든 패치가 같은 테스트를 통과하면 정답.

그 후 형제·자매 벤치들이 줄줄이 나왔다.

| 벤치 | 특징 |
|------|------|
| **SWE-Bench (2023)** | 원본. Python 12개 레포, 2,294개 이슈 |
| **SWE-Bench Lite** | 300개 서브셋. 빠른 평가용 |
| **SWE-Bench Verified (2024-08)** | OpenAI가 사람 검증으로 잡음 제거한 500개. 2026년 현재 상위 모델 75%+ |
| **SWE-Bench Multimodal** | 스크린샷·UI 변화 포함 |
| **SWE-Bench Live** | GitHub 최신 이슈로 연속 갱신. 학습 데이터 오염 방지 |

이 네 변종이 다 의미가 있다. *Verified*는 잡음 제거판이고 가장 자주 인용된다. *Live*는 학습 데이터 오염을 피하려고 매주 최신 이슈로 갱신한다. *Multimodal*은 스크린샷이 들어가는 UI 작업까지 펴 보고, *Lite*는 빠른 회귀 테스트용으로 쓴다.

흥미로운 사실이 있다. *Verified*에서 모델이 75%를 넘기는 자리에서도, *Live*는 점수가 자주 더 낮게 나온다. 같은 모델인데 — 왜? 학습 시점 이후 이슈는 모델이 못 본 자리니까. 모델 점수가 높다는 게 *학습된 패턴을 잘 외운다*는 뜻과 *진짜 새로운 이슈를 푼다*는 뜻 사이에는 분명한 간격이 있다. 자기 하네스를 평가할 때 이 간격을 잊지 말자.

## Terminal-Bench 2.0 — 89개 어려운 터미널 과제

SWE-Bench가 *깃허브 이슈*를 보는 자리라면, Terminal-Bench는 *터미널 안에서의 일*을 본다. 2026년 Snorkel·Stanford·Laude·Anthropic이 공동으로 만든 Terminal-Bench 2.0은 89개의 어려운 터미널 과제로 구성된다.[^tb2] 각 과제마다 고유 환경, 사람이 작성한 풀이, 검증 테스트가 짝지어 있다.

평가 대상이 흥미롭다. 상용 CLI인 Claude Code·Codex CLI·Gemini CLI 셋과, OSS인 OpenHands·Mini-SWE-Agent·Terminus 2가 같은 자리에서 비교된다. *모델*이 아니라 *하네스*가 비교된다는 점에서 4장의 명제 — "모델보다 중요한 하네스" — 가 다시 한 번 검증되는 자리다.

또 하나 중요한 건 **Harbor**다. Terminal-Bench의 평가 하네스를 OSS 프레임워크로 분리해 *다른 벤치도 같은 형식으로 운영*할 수 있게 만들었다.[^tb2] 이 분리는 결정적이다. 평가 하네스가 OSS가 됐다는 얘기는, 우리가 사내 도메인 벤치를 만들 때 *바닥부터 다시 짜지 않아도 된다*는 뜻이다. Harbor 위에 우리 과제만 올리면 된다.

## OSWorld·WebArena — GUI 에이전트의 자리

GUI 에이전트는 또 다른 영역이다. OSWorld는 실제 OS GUI 위에서 369개 과제를 정의한다.[^osworld] 에이전트가 마우스·키보드·스크린샷으로 상호작용한다. WebArena는 같은 사상을 웹 환경판으로 확장한다.

이 자리는 — 솔직히 — 코딩 에이전트와는 거리가 좀 있다. 우리가 다루는 작업이 IDE/CLI 안에 있다면 OSWorld 점수는 그닥 와닿지 않는다. 다만 에이전트가 *Confluence를 보며 사내 위키에서 정보를 찾고, Jira에 이슈를 생성하고, Notion 문서를 갱신*하는 종류의 일을 한다면 — 갑자기 OSWorld가 *우리 일*이 된다.

자기 도메인을 솔직하게 그려 보자. 우리 에이전트가 *어디까지* 일하기를 바라는가? IDE 안만? 아니면 IDE 밖의 협업 도구까지? 그 답에 따라 어느 벤치가 자기 좌표인지가 달라진다.

## METR Time Horizons — 시간으로 능력을 잰다는 발상

METR(Model Evaluation & Threat Research)이 만든 Time Horizons 벤치는 발상이 다르다. *과제 정답률*이 아니라 *과제 길이*로 능력을 잰다.[^metr-time-horizon] [^metr-time-horizon11] HCAST + RE-Bench + SWAA 합산으로 1초~16시간 범위의 작업을 만들어 두고, "50% 확률로 성공하는 인간 기준 작업 시간"을 모델별로 잰다.

추세 발견이 충격적이었다. 6년간 7개월마다 2배. 2024년 이후 — 4개월로 가속(논쟁 진행 중). Time Horizon 1.1(2026-01)에서 그 추세가 재확인됐다.[^metr-time-horizon11]

다만 한계도 명백하다. 소프트웨어/리서치 영역에 한정된 벤치다. 도메인 일반화는 *보장되지 않는다*.[^metr-domains] 시그니처 인용이 여기서 나온다.

> "Time horizon doubles every 7 months — but only on the tasks the harness can already see."
>
> *시간 지평은 7개월마다 두 배가 된다 — 다만, 하네스가 이미 볼 수 있는 과제 위에서만.*
>
> — METR

이 단서가 중요하다. *하네스가 볼 수 있는 과제*에서만 그 가속이 일어난다. 우리 도메인의 과제를 하네스가 *아예 못 보는* 형태로 두면, 그 가속의 혜택은 우리에게 안 온다. 이게 자기 도메인 미니 벤치를 만들어야 하는 *진짜* 이유다.

## Inspect Evals — 200+ 사전 빌드 평가

UK AI Safety Institute가 만든 Inspect는 에이전트 평가의 사실상 표준 프레임워크가 되어 가고 있다.[^inspect-evals] 200개 넘는 사전 빌드 평가가 들어 있고, 외부 에이전트(Claude Code, Codex CLI, Gemini CLI)와 통합 가능하며, 50명 넘는 커뮤니티 컨트리뷰터, METR·Apollo 등 주요 안전 연구소가 채택했다.

Inspect의 가치는 *평가를 표준화한 인프라*에 있다. 우리가 사내에서 평가를 시작할 때 *바닥부터* 짤 필요가 없다는 뜻이다. Docker·Kubernetes·Modal 백엔드를 그대로 쓰고, 결과 리포팅도 표준 포맷으로 받는다. 그 위에 우리 도메인 과제만 얹으면 된다.

부록 C가 Inspect 평가 시작 키트다. 도메인별 미니 벤치 만들기 워크플로(15~30 골든 케이스)를 거기에 자세히 풀어 둔다. 본문에서는 *왜* 자기 도메인 미니 벤치를 만들어야 하는지에 집중하자.

## 자기 도메인 미니 벤치 — 15개 골든 케이스 워크플로

자, 본격적인 자리다. 회사의 사내 시스템 — 가령 결제 모듈의 리팩토링, 사내 위키 검색, Jira 한국어 이슈 분류 — 같은 자리에서, 외부 벤치는 우리에게 *대략적인* 신호만 준다. 우리가 진짜 알고 싶은 건 *우리 도메인에서 우리 하네스가 잘 굴러가는가*다.

15~30개 골든 케이스로 시작하자. 30개 이상은 첫 라운드에 욕심이다. 적게 시작하자. 워크플로는 다음과 같다.

**1단계 — 도메인 풀 정의 (1일).** 자기 팀이 *자주 하는 작업* 5종을 적어 보자. 가령 PR 리뷰, 결제 모듈 버그 수정, 사내 위키 검색, 한글 커밋 메시지 생성, Jira 이슈 분류. 각 5종에 *3개씩* 골든 케이스를 만들면 15개다.

**2단계 — 입력·출력·검증 정의 (2일).** 각 골든 케이스마다 입력(자료, 환경, 초기 상태), 출력(기대 결과), 검증 함수(자동 채점 또는 사람 검토 가이드)를 정의한다. 자동 채점이 안 되는 자리(가령 코드 리뷰 메시지 품질)는 *시간이 지나도 일관된 사람 검토*를 담보해야 한다. 단순 LGTM/Reject 두 단계가 좋다 — 5단계 평가는 사람이 일관성을 못 지킨다.

**3단계 — Inspect 위에 올리기 (1일).** Inspect의 task 형식으로 변환하고, Docker 백엔드로 격리 실행 환경을 잡는다. 이 단계는 부록 C가 자세히 다룬다.

**4단계 — 베이스라인 측정 (반나절).** *현재* 하네스로 15개를 한 번 돌린다. 점수가 8/15든 12/15든 상관없다 — 그 숫자가 출발점이다.

**5단계 — 변경 시 회귀 측정 (정기).** AGENTS.md를 고칠 때, 모델 버전을 올릴 때, MCP 서버를 추가할 때, 매번 15개를 다시 돌린다. *delta*가 +1인지 -1인지를 본다. 4장에서 배운 회귀 탐지를 자기 도메인에서 굴리는 자리다.

미니 벤치의 진짜 목적은 *분기 회고 자리에서 그 표를 띄울 수 있게* 하는 것이다. "체감으로는…" 대신 "우리 결제 도메인 골든 15개에서 6월에 9/15였고 9월에 12/15입니다." 이렇게 말할 수 있게 하자. 그러면 회의의 분위기가 바뀐다.

## 한국어 도메인 5종 — 부록 H의 짝

부록 H가 한국어 도메인 골든 케이스 5종을 카탈로그로 정리해 뒀다. 본문에서 미리 한 번 늘어놓자. 우리 환경에 *필요한* 평가 종류라서다.

1. **한글 코드 리뷰 메시지 평가** — 같은 PR diff에 대해 한국어로 일관된 톤의 리뷰 메시지가 나오는가
2. **한글 커밋 메시지 평가** — 변경 내용을 정확히 요약하는 한국어 커밋 메시지가 나오는가
3. **한국어 주석 정리 평가** — 영어 주석과 한국어 주석이 섞인 코드베이스에서 일관되게 정리할 수 있는가
4. **사내 위키(한국어) 검색 평가** — Confluence/Notion의 한국어 문서를 RAG로 검색할 때 정확도
5. **Jira 한국어 이슈 분류 평가** — 한국어로 적힌 Jira 이슈를 라벨·우선순위로 분류

이 다섯이 *한국어 환경 특수* 평가다. 글로벌 벤치에는 안 들어 있다. 우리가 직접 만들어야 한다. 부록 H가 각 평가의 골든 케이스 6개씩, 총 30개를 카탈로그로 정리한다. 본문에서는 자리만 잡아두자 — 한국어 평가는 *우리 책임*이다.

## 시그니처 인용 박스

> "Time horizon doubles every 7 months — but only on the tasks the harness can already see."
>
> *시간 지평은 7개월마다 두 배가 된다 — 다만, 하네스가 이미 볼 수 있는 과제 위에서만.*
>
> — **METR**, *Measuring AI Ability to Complete Long Tasks*[^metr-time-horizon]

이 한 줄에 22장 전부가 압축돼 있다. 외부 벤치의 가속은 사실이다. 다만 *우리 하네스가 보지 못하는 과제*는 그 가속에서 빠져 있다. 그러니 자기 도메인을 *하네스가 볼 수 있게* 만드는 일이 우리 몫이다 — 그게 미니 벤치 15개의 일이다.

## 실습 박스 — 자기 도메인 골든 15개를 Inspect로 돌리기

이번 주말 안에 끝낼 수 있는 실습이다.

**도메인 5종 × 3개 = 15 골든 케이스.** 자기 팀의 자주 하는 작업 5종을 솔직하게 적자. 가령 (1) PR 리뷰 메시지 작성, (2) 사내 결제 API의 버그 수정, (3) Jira 한국어 이슈 분류, (4) 한글 README 갱신, (5) 통합 테스트 실패 분석. 각 5종에 3개씩 *실제* 사례를 골든 케이스로 만든다 — 이미 사람이 푼, 검증된 사례를 가져오자.

**Inspect로 변환.** 부록 C의 시작 키트를 따라 task 정의를 짠다. 검증 함수가 자동화 어려우면 *LGTM/Reject* 2단계로 사람 검토를 정한다. 검토자는 *고정해야 한다* — 매번 다른 사람이 보면 일관성이 깨진다.

**현재 하네스로 베이스라인 측정.** 15개를 한 번 돌린다. 점수를 기록한다. 그게 출발점이다.

**한 가지 변경, 한 번 회귀.** 다음 주에 AGENTS.md 한 줄을 고친다고 하자. 그 변경 *직전*에 15개를 한 번 더 돌리고, *직후*에 또 한 번 돌린다. delta를 본다.

이 실습의 핵심은 *완벽한 평가 인프라*를 만드는 게 아니다. *분기 회고 자리에서 띄울 수 있는 첫 표*를 만드는 거다. 15개의 작은 진실이 50개의 큰 체감을 이긴다.

## 정리해보자

기억해두자. **체감으로는 좋아진 것 같다 — 그런데 정말 그럴까?** 이 질문을 분기마다 한 번씩 진지하게 던져 보자. 그 답이 "잘 모르겠다"라면, 우리는 평가 하네스를 잘못 짠 게 아니라 *아예 안 짠* 것이다.

외부 벤치는 좌표를 준다. SWE-Bench Verified·Live, Terminal-Bench 2.0 + Harbor, OSWorld·WebArena, METR Time Horizons. 그 위에서 자기 도메인 미니 벤치 15개를 얹자. Inspect 위에 올려서 회귀 측정을 자동화하자. 한국어 도메인 5종은 글로벌 벤치에 없으니 우리가 만들어야 한다.

22장의 진짜 메시지는 — 평가가 *부록*이 아니라는 것이다. 평가는 5축 메트릭(15장)에 들어가는 6번째 축이고, 4장의 회귀 탐지가 굴러가게 하는 동력이고, 다음 분기 회고 자리에서 우리 입을 바꿔 주는 도구다.

평가가 손에 들어왔다면 — 다음은 *어떤 하네스를 어디에 놓을지*를 봐야 한다. 23장의 도구 풍경이다.
