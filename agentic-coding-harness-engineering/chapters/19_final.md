# 19장. 격리 3축 — 이론과 아키텍처

> *"Tooling, host, and network — three orthogonal axes of agent isolation."*
> "툴링·호스트·네트워크 — 에이전트 격리의 세 직교 축."
> — UK AISI *Inspect Sandboxing Toolkit* (정리 인용) [^inspect-sandbox]

에이전트가 `rm -rf`를 시도했다는 사연을 어디선가 본 적이 있을 것이다. 운 좋게 막힌 사연도 있고, 운 나쁘게 막히지 못한 사연도 있다. 그 사연이 *우리* 일이 되지 않게 — 우리는 30분 안에 격리를 짤 수 있다. 진짜로? 그렇다, 진짜로. 다만 *세 축*을 모두 알고 있어야 한다. 한 축만 알면 30분이 1시간이 되고, 그 한 축마저 *잘못된 축*이면 1시간이 모래 위의 삽질이 된다.

이 챕터는 짧다 — 8쪽 분량이다. 19장이 *이론*이고 20장이 *운영*이다. 둘로 나눈 이유는 단순하다 — 격리는 *이론으로 받아들인 다음 운영으로 짜야* 한다. 운영부터 들어가면 어느 축이 비었는지 모른 채 한 축만 두껍게 쌓는다. 이론을 먼저 받아들이면 30분 안에 *세 축에 골고루 한 줄씩* 짤 수 있다.

19장에서는 UK AISI Inspect Sandboxing Toolkit이 정리한 *Tooling·Host·Network* 3축을 받아들인다. 그리고 Firecracker MicroVM, gVisor, devcontainer, sandbox-exec 네 가지를 한 표 위에서 비교한다. 마지막에 자기 환경을 3축 매트릭스에 매핑하는 점검을 한다.

번거롭게 들리는가? 번거로울 수 있다. 그러나 19장의 번거로움은 20장에서 *한 시간 안에 손에 쥐는 운영*으로 갚는다.

## 왜 3축인가

UK AISI(영국 AI Safety Institute)가 Inspect Sandboxing Toolkit을 발표하면서 정리한 격리 모델은 *세 축*으로 직교한다 [^inspect-sandbox] [^aisi-sandbox]. 직교(orthogonal)라는 말이 핵심이다 — 세 축이 *서로 독립적*이라는 뜻이고, 한 축의 격리가 다른 축의 격리를 *대체하지 못한다*는 뜻이다.

세 축은 다음과 같다.

| 축 | 통제 대상 | 한 줄 정의 |
|----|-----------|------------|
| **Tooling** | 모델이 접근 가능한 *도구 집합과 코드 실행 권한* | "에이전트가 *무엇을* 할 수 있는가" |
| **Host** | 모델이 *호스트 시스템*을 탈출·침해하지 못하게 | "에이전트가 *어디에서* 살고 있는가" |
| **Network** | 외부(인터넷 포함) *통신*의 통제, egress 화이트리스트 | "에이전트가 *어디로* 나갈 수 있는가" |

세 축이 직교한다는 것을 사고 모드로 보면 분명해진다. 각 축이 빠졌을 때 *어떤* 사고가 일어나는지를 따로 그릴 수 있다.

- *Tooling만 격리됐다면.* 도구 집합은 잘 좁혔는데 호스트 격리가 없으면, 한 도구의 출력이 호스트의 임의 파일을 건드린다. *Tooling이 통과시킨* 셸 명령 한 줄이 호스트의 `/etc/`를 망친다.
- *Host만 격리됐다면.* 컨테이너는 잘 짰는데 도구가 무엇이든 줘 버리면, 컨테이너 안에서 모든 게 일어난다. 호스트는 살아 있지만 컨테이너 안의 *작업물* — 작업 중이던 코드, 환경변수, 임시 파일 — 이 다 망가진다.
- *Network만 격리됐다면.* 외부로 못 나가게 잘 잠갔는데 도구·호스트 격리가 없으면, *내부에서* 일어나는 사고를 막지 못한다. `rm -rf` 같은 자해는 네트워크와 무관하다.

이 세 사고 모드는 *서로 다르다*. 그래서 한 축의 격리로 다른 두 축을 메울 수 없다. 모든 축에 *최소 한 줄*은 있어야 한다.

UK AISI의 권고는 분명하다 — *Three axes — and you must close all three.* 세 축 모두 닫아라. 닫는 두께는 환경마다 다를 수 있지만, *닫혀 있어야 한다*는 사실은 같다.

## Tooling 축 — 무엇을 할 수 있는가

이 축은 *도구의 표면적*을 좁힌다. 7장(ACI 4원칙)·9장(도구·MCP·서브에이전트)에서 다룬 자리와 같은 자리다. 다만 거기서는 *효율성*의 시점이었고, 여기서는 *격리*의 시점이다.

이 축에서 할 일은 다음 셋이다.

1. *최소 권한 도구 집합.* 에이전트가 정말 필요한 도구만 노출한다. 셸을 통째로 주는 대신 *명시적인 명령 화이트리스트*를 둔다. 파일시스템을 통째로 주는 대신 *작업 디렉토리만* 노출한다.
2. *위험한 도구의 게이트.* `rm`, `git push --force`, `curl`, `npm publish` 같은 *결과를 되돌리기 어려운* 도구는 *사람의 승인 게이트*를 건다. 자동 실행 화이트리스트에 들어 있지 않은 도구는 자동 차단.
3. *MCP 서버의 권한 표면 검증.* 18장에서 본 description injection의 표적이 정확히 이 자리다. MCP 서버를 깔 때 *어떤 도구가 무엇을 할 수 있는지* 자기 손으로 한 번 읽고 분류한다. description을 *데이터*로 봐서는 안 된다.

이 축은 *코드 수준*에서 짠다. 도구 정의 파일(MCP server 설정, AGENTS.md의 도구 섹션, 시스템 프롬프트의 도구 목록)에서 한 줄씩 좁힌다.

## Host 축 — 어디에서 살고 있는가

이 축은 *프로세스 격리*다. 에이전트가 도는 *호스트*를 작은 상자 안에 가둔다. 상자 안에서 무슨 일이 일어나도 상자 밖은 안전하다.

호스트 격리에는 여러 *깊이*가 있다.

| 깊이 | 기술 | 트레이드오프 |
|------|------|--------------|
| *얕음* | macOS sandbox-exec, devcontainer, chroot | 시작 빠름(초 단위), 성능 손실 적음, 격리 약함 |
| *중간* | Docker, Linux user namespaces, gVisor | 시작 보통(초~분), 성능 손실 작음, 격리 보통 |
| *깊음* | Firecracker MicroVM, KVM, 풀 VM | 시작 느림(100ms~분), 성능 손실 보통, 격리 강함 |

여기서 핵심 한 가지 — *격리의 깊이는 트레이드오프*다. 깊은 격리는 안전하지만 무겁다. 얕은 격리는 가볍지만 뚫린다. 자기 환경에 *맞는 깊이*를 선택해야 하고, 그 선택은 다음 두 변수에 달렸다.

1. *위험의 크기.* 에이전트가 다루는 것이 *프라이빗 데이터의 핵심*이라면 깊은 격리가 필요하다. 사이드 프로젝트의 toy code라면 얕은 격리로도 충분하다.
2. *시작 빈도.* 에이전트가 *세션마다* 새로 떠야 한다면 시작 시간이 중요하다. 한 번 띄워서 오래 쓴다면 시작 시간은 덜 중요하다.

이 두 변수의 곱이 깊이를 결정한다. 한국 회사에서 흔한 케이스를 두 개 보자.

- *사이드 프로젝트* — 위험 작음 + 시작 빈도 높음 → *얕음* (devcontainer + 작업 디렉토리 격리).
- *결제·인증 모듈* — 위험 큼 + 시작 빈도 보통 → *중간 또는 깊음* (Docker + egress proxy + ephemeral secrets, 또는 Firecracker MicroVM).

여기서 짚고 가자 — 깊은 격리가 *언제나* 옳지는 않다. 깊은 격리를 짠다고 한 달을 보내는 동안, 얕은 격리로 *오늘부터* 막을 수 있는 사고를 못 막는다. 완벽함은 시작의 적이다.

## Network 축 — 어디로 나갈 수 있는가

이 축은 *egress 통제*다. 에이전트가 외부로 보낼 수 있는 *목적지*를 명시적으로 화이트리스트한다.

이 축에서 할 일은 다음 셋이다.

1. *Egress 화이트리스트.* 외부로 나가는 모든 호출을 *프록시 한 군데를 통과*시킨다. 프록시는 화이트리스트된 도메인만 통과시키고, 나머지는 거절한다. 16장의 trifecta 1차 방어에서 본 *output channel whitelist*가 여기 해당한다.
2. *DNS 통제.* 도메인이 아니라 IP를 직접 박은 호출도 막아야 한다. DNS도 *내부 DNS만* 쓰게 강제한다. 외부 DNS로의 직접 호출은 차단.
3. *단기 시크릿(ephemeral secrets) 결합.* 외부 API 키는 *세션 단위로 발급되는 단기 토큰*으로 굴린다. 한 번 노출돼도 영향 시간이 짧다.

Network 축은 *뚫리기 가장 쉬운* 축이기도 하다. 한 줄짜리 환경변수 추가, 한 줄짜리 호스트 파일 수정, 한 줄짜리 DNS 변경이 화이트리스트를 우회할 수 있다. 그래서 Network 축은 *Host 축과 함께* 짜야 한다 — 호스트 격리 안에서 네트워크 통제가 *변경 불가능한 상태*로 박혀 있어야 한다. 두 축이 직교하면서도 *결합되어* 안전을 만든다.

## 네 가지 격리 기술 비교

이론을 손에 쥐려면 *실제 기술*을 한 표에 놓고 비교하는 게 빠르다. 코딩 에이전트 환경에서 자주 쓰이는 네 가지를 본다.

| 기술 | 격리 깊이 | 시작 시간 | 메모리 | 강점 | 약점 |
|------|-----------|----------|--------|------|------|
| **Firecracker** [^inspect-sandbox] | 깊음 (KVM 위 MicroVM) | ~100ms | 5MB+ | 깊은 격리 + 빠른 시작. AWS Lambda의 기반. | 리눅스 전용. 운영 복잡도 보통. |
| **gVisor** | 중간 (사용자 공간 커널) | 초 단위 | MB 단위 | syscall 표면적 좁힘. Docker과 함께 쓰기 쉬움. | 일부 syscall 미지원. 성능 손실 5~30%. |
| **devcontainer** | 얕음 (Docker + VS Code 표준) | 초 단위 | Docker 기반 | 개발 워크플로 통합 일류. 학습 곡선 낮음. | Docker daemon 신뢰. 격리 깊이 얕음. |
| **sandbox-exec (macOS)** | 얕음 (Apple SBPL) | 즉시 | 무시 | macOS 네이티브. Claude Code·Codex CLI 기본 통합. | Apple 비공식 API(deprecated 표기). macOS 한정. |

이 표에서 짚을 점 두 가지.

첫째, *Claude Code와 Codex CLI*가 macOS sandbox-exec과 Linux user namespaces를 *기본으로* 통합한다 [^claude-code] [^openai-harness]. 즉 *얕은 격리는 자동으로 따라온다*. 30분 PoC의 출발점이 이미 손에 쥐어져 있는 셈이다. 이걸 모르면 처음부터 짜고 있을 수 있다 — 그러지 말자.

둘째, *깊이의 선택은 정답이 없다*. 네 가지 모두 *어딘가에서는* 옳다. 결제 시스템의 자동 배포 에이전트라면 Firecracker가 답일 수 있다. 사이드 프로젝트의 코딩 에이전트라면 devcontainer로 충분하다. macOS 머신의 일상 작업이라면 sandbox-exec이 가장 가볍게 시작하는 길이다.

## 3축에 어디가 비었는가 — 자가 진단

이 챕터의 진짜 출구는 *자기 환경의 3축 매트릭스를 그리는 것*이다. 표 한 장이면 된다.

| 축 | 우리 환경의 현재 상태 | 빈 자리 |
|----|----------------------|---------|
| Tooling | (적어 보자) | (어떤 도구가 화이트리스트 없이 풀려 있는가) |
| Host | (적어 보자) | (어떤 격리도 없이 호스트에서 직접 도는가) |
| Network | (적어 보자) | (어떤 외부 호출이 프록시 없이 나가는가) |

세 줄을 다 채워 보면, 보통 *한 축이 비어 있다*. 가장 흔한 패턴은 다음 둘이다.

- *Network는 어느 정도 막혀 있는데 Tooling이 비어 있다.* 사내 방화벽은 있지만 에이전트의 도구 화이트리스트는 없는 케이스. 결과 — 사내 안에서 일어나는 자해 사고에 무방비.
- *Tooling은 좁힌 듯한데 Host가 비어 있다.* MCP 서버를 골라 깔았지만 호스트 격리는 없는 케이스. 결과 — 한 도구의 버그 한 줄이 호스트를 흔들 수 있다.

세 번째 흔한 패턴은 *세 축이 다 비어 있는* 경우다. 이 경우 — 솔직히 말하자 — 평소에 사고가 *안 났던 것이 운*이다. 운에 의존한 시스템은 운이 떨어지면 무너진다.

20장에서 30분 PoC 세 개를 본다. 그 PoC가 *오늘 빈 축*을 한 줄씩 채우는 첫 걸음이다.

## 실습 — 3축 매트릭스 자가 진단

10분짜리 점검이다. 종이 한 장이면 된다.

**1단계 (3분).** 위의 3축 표를 자기 환경에 옮겨 적는다. *현재 상태*를 솔직하게 적는다. 모르면 *모른다*고 적는다.

**2단계 (4분).** 각 축에서 *빈 자리*를 한 줄씩 적는다. 빈 자리가 한 축에 두 개 이상이면 둘 다 적는다. 어느 축이 *가장* 비어 있는가?

**3단계 (3분).** 가장 빈 축에 *오늘 끼울 수 있는 한 줄*을 적는다. *이번 주* 안에 끼울 수 있는 크기여야 한다. 한 달 걸리는 액션은 액션이 아니다.

이 표를 보관하자. 20장의 PoC를 짤 때, 이 표가 *어느 PoC부터* 손에 쥘지 정해 준다.

## 기억해두자

- *격리는 3축이다.* Tooling(무엇을), Host(어디서), Network(어디로). 세 축이 직교하므로 한 축이 다른 두 축을 대체하지 못한다.
- *세 축 모두 닫아야 한다.* UK AISI의 권고. 닫는 두께는 환경마다 달라도, *닫혀 있어야* 한다는 사실은 같다.
- *깊이의 선택은 위험과 시작 빈도의 곱이다.* Firecracker가 언제나 옳지 않고, sandbox-exec이 언제나 부족하지도 않다.
- *Claude Code·Codex CLI는 얕은 격리를 기본 통합한다.* 출발점이 이미 손에 있다. 그것부터 활용하자.
- *3축 매트릭스를 한 장 그려 보자.* 빈 축이 다음 챕터(20장) PoC의 출발점이다.

이론을 봤으니 — 30분 안에 손에 쥐는 운영 레시피로 가자. 20장에서 Docker·devcontainer·sandbox-exec 세 가지 PoC를 짠다.
