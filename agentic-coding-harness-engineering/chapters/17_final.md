# 17장. Shai-Hulud — 자기 복제하는 npm 웜의 해부

> *"What started as one stolen token cascaded into 25,000+ repositories in 72 hours."*
> "한 번의 토큰 탈취가 72시간 만에 25,000개가 넘는 저장소로 번졌다."
> — Wiz Research [^shaihulud-wiz]

남의 일이라 생각했다면 다시 보자. 우리 `npm install`도 다 거기서 거기다.

처음 기사를 봤을 때를 떠올려 보자. 2025년 9월 어느 날 슬랙에 누가 링크를 던졌다. "npm 공격 또 났대." 우리는 한숨을 한 번 쉬고, "아 그러게, 요즘 이런 게 많네" 정도로 넘겼다. 11월에 변종이 나왔을 때도 비슷했다. 25,000개 저장소가 영향을 받았다는 숫자 앞에서 잠시 멈칫했지만, 우리 일이라는 실감은 약했다. 2026년 2월 SANDWORM_MODE가 나왔을 때, 그제서야 좀 다른 기분이 들었다 — *악성 MCP 서버*가 LLM API 키를 노렸다는 대목 앞에서.

그게 우리 일이다. 우리가 매일 깔고, 매일 호출하고, 매일 컨텍스트에 끼워 넣는 그 모든 도구의 한가운데가, 정확히 그 사고들이 노린 자리였다.

이 챕터는 사고를 *시간 순으로* 분해한다. 추상이 아니라 분해로 가르친다. 1.0(2025-09)에서 어떤 메커니즘이 작동했는지, 2.0(2025-11)에서 무엇이 자동화·확장됐는지, SANDWORM_MODE(2026-02)에서 *왜 LLM 키가 새 표적이 됐는지*를 따라간다. 그리고 Elastic의 대응 가이드를 손에 쥔다 — 사고가 났을 때 *오늘 밤 안에* 해야 할 일이 무엇인지. 마지막은 페어 워크. 자기 프로젝트의 MCP 의존성 트리를 그리고, 신뢰 경계를 표시한다 — 팀원과 함께.

끔찍한 일이다. 그 끔찍함을 미리 한 번 보면, 사고는 *덜* 끔찍해진다.

## 1.0 — 2025년 9월: 자기 복제의 시작

Palo Alto Unit 42가 처음 보고한 *Shai-Hulud* 캠페인은 2025년 9월에 등장했다 [^shaihulud-unit42]. 이름은 듄(Dune)의 모래벌레에서 가져왔다. 거대한 몸을 모래 밑에 숨기고, 표면 진동을 따라 한 번에 솟구쳐 무는 그 벌레다. npm 생태계 위를 같은 식으로 움직였다.

작동 메커니즘을 단순화해 보자.

**1단계 — 자격증명 탈취.** 어떤 npm 패키지의 `postinstall` 후크에 악성 코드가 심긴다. 사용자가 그 패키지를 깔면 후크가 실행된다. 후크는 사용자 머신을 뒤져 npm 토큰, GitHub 토큰, AWS 자격증명을 수집한다. 토큰이 어디 있는지는 표준화돼 있다 — `~/.npmrc`, `~/.config/`, 환경변수, CI 러너의 시크릿 마운트 위치. 후크는 이것을 외부 서버로 빼낸다.

**2단계 — 자기 복제.** 탈취한 npm 토큰으로 *피해자가 메인테이너인 다른 패키지들*에 접근한다. 그 패키지들에 새 버전을 발행한다. 새 버전의 `postinstall`에는 같은 악성 코드가 들어 있다. 다른 사용자들이 그 새 버전을 깔면, 그들의 토큰도 털린다. 그들의 패키지에 또 발행된다. 그렇게 모래벌레가 모래 밑에서 번진다.

**3단계 — 가속.** 자동화된 스크립트가 도는 동안, 인간 메인테이너가 *모르는 사이에* 자기 패키지에 악성 버전을 발행한다. 발행자 이름이 정상이고, 변경 로그도 평범해 보인다. 다른 사용자가 의심할 이유가 없다. 모래 밑이라 진동이 안 나오는 것이다.

이 1.0이 2025년 9월에 수백 개 패키지를 감염시켰다. Unit 42는 자기 복제 메커니즘을 정확히 짚었고, 즉시 IOC(indicators of compromise)를 공개했다.

여기서 중요한 점은, *어떤 패키지도 처음부터 악성으로 만들어지지 않았다*는 사실이다. 정상 메인테이너의 정상 패키지가, 그 메인테이너의 토큰이 한 번 털린 그 순간부터 악성 전파의 발사대가 됐다. 우리가 매일 의존하는 그 신뢰의 그물이 — 그 그물 자체가 — 한 가닥의 끊어짐으로 통째로 흔들렸다.

찜찜하지 않은가? 우리도 어딘가에서 그 메인테이너 중 한 명이다.

## 2.0 — 2025년 11월: 25,000 저장소

두 달 뒤 11월에 변종이 나왔다. Wiz Research가 *Shai-Hulud 2.0*이라 불렀다 [^shaihulud-wiz].

규모가 차원이 달랐다.

> **수치(Wiz 보고 기준).**
> - 영향 받은 저장소: **25,000개 이상**
> - 영향 받은 사용자: **350명 이상**
> - 자동 재발행된 npm 패키지: **수백 개**
> - 사고 인지에서 핵심 전파까지: **72시간 이내**

72시간이라는 숫자에서 잠깐 멈춰 보자. 우리 팀이 사고를 인지하고, 의사결정 회의를 잡고, 패치를 깔고, 토큰을 회수하는 데 보통 72시간 안에 끝나는가? 솔직히 끝나기 어렵다. 그런데 공격은 그 안에 25,000개 저장소를 돌았다. *우리가 일어나기 전에 일이 끝나 있다*는 뜻이다.

2.0에서 새로 들어온 것은 *자동화의 깊이*였다. 1.0이 사람의 손이 한두 번 끼는 반자동이었다면, 2.0은 거의 완전 자동이었다. 토큰 탈취 → 의존성 그래프 수집 → 새 버전 빌드 → 발행 → 자기 복제 — 이 사이클이 한 머신에서 분 단위로 돌았다. 모래벌레가 한 번 솟구치는 동작이, 한 번이 아니라 수천 번 동시에 일어난 것이다.

또 하나, 2.0이 노린 자리가 *CI 파이프라인*이라는 점이 중요하다. 개인 개발자 머신뿐 아니라 GitHub Actions, GitLab CI, CircleCI 같은 자동화 환경의 시크릿 마운트를 표적으로 삼았다. CI 러너에는 보통 *팀 전체의 토큰*이 들어 있다. 한 개인의 토큰보다 훨씬 큰 권한이다. 그래서 한 번 털리면 한 팀 전체가 털린다.

이쯤 되면 끔찍한 일이다. 그리고 끔찍함의 진짜 자리는 — 우리가 매일 굴리는 CI가 정확히 그 표적이라는 것이다. 매일 자동으로 도는 우리 파이프라인이 *공격자의 자동화 자산*과 같은 자리에 있다. 자동화가 우리 편이라고 안심하는 순간이 가장 위험하다.

## SANDWORM_MODE — 2026년 2월: LLM이 표적에 들어오다

세 번째 변종이 2026년 2월에 등장했다. 이름은 *SANDWORM_MODE*. Socket과 SC Media 보고에 따르면, 이전 두 변종과 *동일한 자기 복제 패턴*에 *새 표적*이 추가됐다 [^sandworm].

새 표적은 — *LLM API 키*와 *MCP 서버*다.

이 변종은 npm 패키지의 `postinstall`에서 시작하지만, 추가로 다음을 한다.

1. *Anthropic·OpenAI·Google API 키*가 있는 자리를 뒤진다 — `~/.anthropic/`, `~/.openai/`, `OPENAI_API_KEY` 같은 환경변수, IDE 설정 파일. LLM 키가 있으면 그것을 빼낸다.
2. *MCP 서버 설정 파일*을 찾는다 — `~/.config/claude/mcp.json`, `~/.cursor/mcp.json` 같은 자리. 거기 등록된 MCP 서버 목록을 읽는다.
3. *악성 MCP 서버*를 그 목록에 끼워 넣는다. 다음번 에이전트가 켜질 때 그 서버가 같이 로드된다. 악성 MCP 서버는 *도구 description에* 한 줄짜리 prompt injection을 심어 놓는다 — 16장에서 본 그 모양이다.

여기서 일이 진짜 무서워진다. 1.0과 2.0은 *자격증명*이 표적이었다. 자격증명은 회수할 수 있다. 토큰을 갈고, 키를 회전하면 끝이다. 그런데 SANDWORM_MODE는 *컨텍스트 자체*를 오염시킨다. 악성 MCP 서버가 도구 description으로 prompt injection을 흘려 넣으면, 그건 자격증명 회전으로 풀리지 않는다. 에이전트의 *세계관*이 오염된 상태이고, 우리는 그것을 매일 신뢰하면서 코드를 짠다.

이게 4부 인트로에서 말한 "하네스 자체가 공격 표면이 됐다"는 문장의 진짜 의미다. 마구가 공격받기 시작한 것이다.

여기서 한 가지 짚고 가자 — 1.0의 자격증명 탈취, 2.0의 CI 파이프라인 자동화, SANDWORM_MODE의 LLM·MCP 표적 — 이 셋이 *각각 독립적으로* 일어나는 게 아니다. 같은 자기 복제 골격 위에 *표적의 깊이*가 한 단계씩 더해진 것이다. 다음 변종이 어디로 갈지는 모르지만, 패턴은 분명하다 — *우리가 매일 신뢰하는 자리*를 한 단계씩 더 깊이 노린다.

## 의미 — 하네스의 공급망

세 사고를 시간 순으로 보고 나면, 한 가지 깨달음으로 수렴한다.

*하네스는 공급망이다.*

이 한 줄을 받아들이면 시야가 바뀐다. 우리는 지금까지 "내 코드의 공급망"을 걱정했다. `package.json`의 의존성 트리, `requirements.txt`의 핀 버전, lockfile의 무결성. 그런데 *하네스의 공급망*은 따로 있다.

| 공급망 차원 | 검증 대상 |
|------------|------------|
| MCP 서버 | description, 도구 정의, 핀 버전 |
| 도구 정의 | 셸 명령어, HTTP 엔드포인트, 권한 표면 |
| 에이전트 메모리 | AGENTS.md·CLAUDE.md의 출처와 갱신 이력 |
| CI 파이프라인 | 시크릿 마운트, 캐시, 자동 발행 자격증명 |
| 의존성 패키지 | npm·pip·gem의 lockfile, supply-chain 검증 |
| 외부 데이터 입력 | 이슈 본문, 위키 페이지, RSS 피드 — 16장에서 다룬 untrusted content |

이 여섯 차원이 모두 supply-chain 검증 대상이다. 하나라도 빠지면, Shai-Hulud의 다음 변종이 그 빠진 자리로 들어올 준비를 한다.

여기서 하나 강조하자. *하네스 공급망 검증*은 새로운 종류의 일이 아니다. 이미 우리가 코드 공급망에서 하던 일을 — 의존성 핀, 무결성 해시, 정기 audit — 하네스의 모든 차원으로 *확장*하면 된다. 단, 확장의 *목록*을 분명히 가지고 있어야 한다. 위 표가 그 목록의 시작점이다.

## Elastic의 대응 가이드 — 사고가 났을 때 오늘 밤

Elastic은 2025년 11월 2.0 사고 이후 *Navigating the Shai-Hulud Worm 2.0*이라는 대응 가이드를 공개했다 [^shaihulud-elastic]. 이 가이드의 정수를 *우리 팀이 오늘 밤 한다고 가정하고* 옮겨 본다.

**1단계 — 노출 평가 (1시간).** 자기 환경에서 의심 패키지의 흔적을 찾는다. lockfile 검색, `~/.npmrc`의 토큰 인덱스 확인, 최근 24시간 내 이상 발행 이력. Wiz·Unit 42·Elastic이 공개한 IOC 목록이 시작점이다.

**2단계 — 자격증명 회전 (2시간).** 의심이 조금이라도 있다면 *모든 토큰을 회전한다*. npm, GitHub, GitLab, AWS, GCP, Anthropic·OpenAI 키. CI 러너의 시크릿도 같이. 회전 순서는 *외부 통신이 가능한 토큰 → 외부에서 들어오는 토큰* 순이다. 외부로 빠져나갈 위험이 큰 것부터.

**3단계 — 의존성 핀과 격리 (2시간).** 영향 패키지의 *직전 안전 버전*으로 lockfile을 핀한다. 의심이 풀릴 때까지 *해당 패키지가 깔린 환경에서 에이전트를 떼어 둔다*. 격리 환경에 옮겨 둘 수 있으면 더 좋다 — 19장과 20장에서 격리를 다룬다.

**4단계 — MCP 서버 점검 (1시간).** SANDWORM_MODE 이후로는 *모든 MCP 서버 설정 파일을 diff*한다. 등록된 서버의 *signature*가 어제와 같은지, *도구 description*이 어제와 같은지 확인한다. 이상이 보이면 즉시 그 서버를 떼고, 그 서버를 통해 처리한 최근 작업의 컨텍스트를 *전체 폐기*한다.

**5단계 — Post-mortem과 재발 방지 (1주).** 사고가 *어떤 게이트를 통해 들어왔는지* 한 줄로 정리한다. 그 게이트가 다음 사고에도 열려 있을 가능성이 있는지 평가한다. 영구적인 방어 — supply-chain 자동 audit, MCP description diff, 토큰 자동 회전 — 를 *작업 항목*으로 만들고 책임자를 지정한다.

이 가이드의 시간 견적은 빠듯하다. 사고가 *터진 다음에* 이 다섯 단계를 첫 24시간 안에 끝내는 건 어렵다. 그래서 핵심은 *사고가 나기 전에* 1·2·3단계의 도구 세트를 *준비해 두는 것*이다. 토큰 회전 스크립트, lockfile 핀 절차, 격리 PoC — 사고가 날 때 만들면 늦다. 미리 만들어 두자.

## 한 가지 더 — supply chain 의식의 한국 환경 적용

한국 회사의 사내 환경에는 한 가지 추가 변수가 있다. 사내 npm 미러, 사내 pip 미러, 사내 GitLab 패키지 레지스트리 같은 *중간 캐시*다.

중간 캐시는 두 얼굴이 있다. *방어선*이 될 수도 있고, *증폭기*가 될 수도 있다.

- *방어선* — 중간 캐시가 외부에서 들어오는 모든 패키지를 *지연 동기화*한다면, Shai-Hulud 같은 사고가 외부에서 일어났을 때 캐시가 그 자리에서 *동기화를 멈출 수 있다*. 사내에는 직전 안전 버전이 그대로 남는다.
- *증폭기* — 중간 캐시가 *자동 미러링*만 하고 검증을 안 하면, 외부 사고가 사내로 *그대로* 들어온다. 사내라는 안심 위에 외부 위험이 그대로 얹힌다.

자기 회사의 중간 캐시가 어느 쪽인지를 *오늘 점검하자*. 캐시 운영자에게 물어보자: "외부에서 새 버전이 발행되면 사내 미러는 몇 분 만에 동기화하고, 그 사이에 어떤 검증이 끼는가?" 답을 모르면 캐시는 증폭기일 가능성이 높다. 답을 받았다면, 그 답이 우리 1차 방어선의 한 줄이 된다.

## 실습 — 페어 워크: MCP 의존성 트리와 신뢰 경계

이 실습도 *팀원과 함께* 한다. 16장의 도구 매트릭스 색칠과 같은 짝이다.

**준비 (5분).** 자기 프로젝트의 *모든* 에이전트 설정을 한 곳에 모은다.

- `~/.config/claude/` 또는 `~/.cursor/` 같은 사용자 설정의 MCP 서버 목록
- 프로젝트 디렉토리의 MCP 설정 파일
- `package.json` 또는 그에 상응하는 의존성 매니페스트
- CI 파이프라인 설정의 자동 도구 호출

**의존성 트리 그리기 (20분).** 한 사람이 화이트보드(또는 Miro·draw.io) 앞에 선다. 다른 사람이 옆에서 같이 본다. 다음 노드를 그린다.

- 루트 노드: 자기 에이전트
- 1차 노드: 직접 등록한 MCP 서버 N개
- 2차 노드: 각 MCP 서버의 *내부 의존성* — npm 패키지, pip 모듈, 외부 API 호출
- 3차 노드: 그 의존성의 *추가 의존성* (lockfile 한 단 깊이까지)

여기서 그림이 어지러워질 것이다. 그게 정상이다. 어지러운 그림이 *현실의 supply chain*이다.

**신뢰 경계 표시 (15분).** 그림 위에 다음 색으로 경계를 그린다.

- *내부* (우리 팀 또는 우리 회사가 100% 통제) — 파란 선
- *반-신뢰* (공인된 벤더, 핀된 버전, 정기 audit 있음) — 노란 선
- *외부* (검증되지 않은 OSS, 미러만 거치는 외부 패키지) — 빨간 선

빨간 선 안에 들어 있는 노드 중 *프라이빗 데이터에 접근하는* 것은 무엇인가? 16장의 trifecta 진단을 같이 본다. 빨간 선 + 프라이빗 데이터 = Shai-Hulud의 다음 변종이 들어올 자리다.

**진단과 액션 (15분).** 그림 위에 *오늘 끼울 수 있는 한 줄짜리 방어*를 색깔별로 적는다.

- 빨간 선의 패키지 핀 강화 — lockfile 무결성 해시 추가
- MCP 서버 description의 *어제와 오늘 diff* 자동화 — cron 또는 CI에 한 줄
- CI 러너의 시크릿 마운트 *최소 권한* 재검토 — 한 토큰의 권한 표면 좁히기
- LLM API 키의 *사용 모니터링* 알람 — Anthropic·OpenAI 콘솔의 사용량 임계치 설정

이 실습의 결과물은 그림 한 장과 액션 항목 3~5개다. 액션 항목은 *이번 주* 안에 끝낼 수 있는 크기여야 한다. 한 달 걸리는 액션은 액션이 아니라 계획이다.

## 기억해두자

- *Shai-Hulud는 한 번의 사고가 아니다.* 1.0(2025-09) → 2.0(2025-11, 25,000+ 저장소) → SANDWORM_MODE(2026-02, 악성 MCP)의 진화 곡선이 있다.
- *72시간 안에 일이 끝난다.* 우리 사고 대응 회의 시간보다 빠르다. 그래서 *대응 도구 세트는 사전에* 준비해야 한다.
- *하네스는 공급망이다.* MCP 서버, 도구 정의, 에이전트 메모리, CI 파이프라인, 의존성 패키지, 외부 데이터 입력 — 여섯 차원 모두 검증 대상이다.
- *자격증명 회전만으로 안 풀리는 사고가 있다.* SANDWORM_MODE처럼 *컨텍스트 자체*가 오염되면, 토큰을 갈아도 prompt injection은 살아 있다.
- *한국 환경의 중간 캐시는 양날의 검이다.* 방어선이 될 수도 있고 증폭기가 될 수도 있다. 운영자에게 물어보고 답을 받아 두자.

공급망에서 들어오는 공격도 위험하지만, 텍스트 한 줄로 들어오는 공격이 더 흔하다. 18장 — 인젝션 공격이 몇 가지 모양이고 각각을 어떻게 막는지, *카탈로그*를 펼친다.
