# 03 리뷰 로그 — 다시, C — Apple Silicon에서 ARM 커널까지

## 리뷰 라운드 1

리뷰 대상은 `02_plan.md` 초안 (15장 구성, 약 16만 6천 자 예상). 책 약속은 "C 전문가로 다시 출발하는 개발자를 위한 종합 바이블". 단순 입문서가 아니다. 그 약속을 척도 삼아 5축으로 검토한다.

### 전체 인상

먼저 칭찬은 짧게 하고 넘어간다. 내러티브 아크(5막)와 챕터 간 흐름 한 줄 표는 잘 짜였다. "낯설다 → 정비됐다 → 떨린다 → 안전하다 → 해냈다"는 정서 곡선은 토비 문체와 결이 잘 맞고, Ch12를 정점으로 두는 구조도 명료하다. 그러나 종합 바이블이라 부르려면 빠진 토픽이 있고, 일부 챕터는 분량 균형이 어긋난다. 그리고 챕터 카운트의 일관성에 모순이 있다 (제목은 "14장"인데 실제로는 Ch15까지 있다).

---

### Critical (반드시 해소 권장)

**[Ch 카운트 불일치] 책 특성/검증 체크리스트는 "14장"·"15장 각각"이라고 모두 쓰여 있고, 실제 챕터 목록은 Ch15까지 15개다.**
- 이유: 산출 책의 정체성을 결정하는 메타 정보가 자기모순. 책 약속에도 영향.
- 구체 제안: 본문 헤더 "## 책 특성"의 "14장 구성"을 "15장 구성"으로 정정하고, "챕터 평균 1만 2천~1만 5천 자" 계산도 15장 기준으로 재산정. 검증 체크리스트의 "15장 각각"은 그대로 두면 된다.

**[Ch3+Ch4 강결합 & Ch4 갑작스러움] Apple Silicon 셋업이 Ch3(툴체인) + Ch4(IDE)로 강하게 묶여 있는데, Ch4가 끝난 직후 Ch5에서 갑자기 포인터로 점프한다.**
- 이유: Ch3·Ch4는 환경 구축이 본질, Ch5부터는 언어 본질로 모드 전환. 모드 전환 사이에 짧은 "C 멘탈 모델로 들어가기"가 없어 독자가 환경 구축 모드에서 곧장 포인터 모드로 끌려간다. 레퍼런스 §8(멘탈 모델 전환)이 명시적으로 자기 챕터를 못 가졌다.
- 구체 제안: Ch4 끝부분 또는 Ch5 도입부에 §8의 "사고 전환 체크리스트"와 표(Java/Python vs C 7행 비교)를 짧게 흡수. 길이가 부담되면 Ch5 도입 2~3쪽으로 충분. 별도 챕터를 만들 필요는 없다 (그러면 15장이 16장 됨).

**[누락 토픽: ABI/calling convention] Ch13이 AArch64 ABI를 짧게 다루지만, 호스트 챕터에 ABI/calling convention 자체에 대한 챕터적 다룸이 없다.**
- 이유: 종합 바이블 약속. 옛 C 경험자가 Java/Python에서 돌아오면 "함수 호출이 실제로 어떻게 일어나는가" (스택 프레임, 인수 전달 레지스터, 리턴 주소)에 대한 직관이 약하다. 베어메탈 챕터에서 갑자기 X0–X7이 튀어나오면 무리.
- 구체 제안: Ch8에 §"함수 호출의 실제 — 스택 프레임과 calling convention" 2~3쪽 추가. 또는 Ch5 끝부분에 "함수 포인터 → 함수 호출 메커니즘"으로 자연스럽게 연결. 베어메탈 직전(Ch11)에서 호스트 ABI 대비 AArch64 ABI 비교는 그대로 두되, 호스트 측에 다리 한 개를 더 둔다.

**[누락 토픽: 고정폭 정수·엔디언·비트 조작] 베어메탈 챕터에서 `uint32_t`가 펑펑 쓰이는데, `<stdint.h>` 고정폭 타입·엔디언·비트 조작이 명시적 토픽으로 못 다뤄지고 있다.**
- 이유: 레퍼런스 §8.3이 "K&R 시절 `int`/`long`을 64비트에서 깨먹은 역사"를 명시했고, 베어메탈에서는 MMIO 레지스터 비트 마스킹·시프트·`__builtin_bswap`이 일상이다. Java/Python 출신은 비트 조작 손이 가장 굳어 있다.
- 구체 제안: Ch5 또는 Ch7에 §"고정폭 정수와 비트 조작 — `<stdint.h>`·시프트·마스크·엔디언" 한 절(4~5쪽) 삽입. Ch7(UB)에 붙이면 "signed overflow가 왜 UB인데 unsigned는 아닌가"와 자연스럽게 연결된다.

### Should (강하게 권장)

**[Ch9 비대화] Ch9 "안전망"이 ASan·UBSan·TSan·MSan·clang-tidy·scan-build·cppcheck·-fanalyzer·clang-format·libFuzzer·빌드 매트릭스를 모두 1만 2천 자에 담는다.**
- 이유: 1만 2천 자에 도구 10개. 한 도구당 1쪽 미만. 종합 바이블에서 안전망은 책의 안전 약속 핵심이므로 더 두꺼워야 한다.
- 구체 제안: Ch9를 1만 5천~1만 7천 자로 늘리고, 정적 분석(4개 도구)을 한 절로 묶어 비교 표 + 워크플로 2~3개 시연. clang-format은 짧게(반쪽). libFuzzer는 한 절(2~3쪽)로 미니 튜토리얼 — fuzz target 작성·실행·corpus 관리. 추가 분량 3천~5천 자는 Ch15(8천 자 → 6천 자) 슬림화 또는 Ch1(1만 자 → 9천 자) 압축으로 흡수 가능.

**[Ch10 산만함] Ch10이 NUL 종결 문자열·C23 안전 함수·컨테이너 자작·POSIX 사다리·libc 구현체 차이를 한 챕터에 다 담는다.**
- 이유: "표준 라이브러리의 빈자리"라는 한 주제로 통일되어 있지만, 실제 다루는 영역(문자열·컨테이너·이식성)이 너무 흩어진다. 1만 2천 자로는 각각 얕다.
- 구체 제안 (택1):
  - (안A) Ch10 분량을 1만 5천 자로 늘리고 절 구조를 강화 — §1 문자열, §2 컨테이너 자작, §3 POSIX와 외부 라이브러리, §4 libc 이식성 — 으로 명확히 묶기.
  - (안B) "POSIX·외부 라이브러리" 절을 Ch15로 옮기고 ("실세계 C 프로젝트를 위한 다음 도구들" 부록 성격), Ch10은 문자열+컨테이너에 집중. Ch15는 "다음 걸음"에 어울리는 자리이므로 자연스럽다.
  - (안C) Ch10에서 libc 이식성을 빼서 Ch8(빌드 사이클) 또는 Ch11(freestanding) 도입부로 흡수. Ch10은 "표준 문자열·메모리 API의 함정 + 컨테이너 자작"으로 좁힌다.
  - 추천: 안A. 추가 분량은 안전망과 함께 묶어 흡수.

**[Ch3 + Ch11 sanitizer 중복 vs 흩어짐] sanitizer는 Ch3 hello.c 시범, Ch6 누수 잡기, Ch7 UB 잡기, Ch9 본격 챕터, Ch11 베어메탈 제약까지 다섯 곳에 분산된다.**
- 이유: 흐름상 자연스러우나 메시지가 분산되면 독자가 "왜 켜야 하는지 = 어디서 켜야 하는지" 머릿속 모델이 흔들린다.
- 구체 제안: Ch3에서 hello에 `-fsanitize=address,undefined`를 켜는 한 줄을 보일 때, 짧은 사이드바("Ch9에서 본격 설명")로 미루는 표지를 명시. Ch6·Ch7에서도 동일하게 "Ch9 예고"라는 명시적 포인터를 둔다 (현 계획은 Ch6에 "Ch9 예고" 한 줄이 있는데 일관되지 않다).

**[Ch12 → Ch13 전환의 위험] Ch12가 "한 줄씩 해부"하고 Ch13이 다시 "부트 흐름 해부"한다. 같은 코드를 두 번 해부하는 셈.**
- 이유: 챕터 의도는 다르다 (Ch12 = 첫 부팅·도달, Ch13 = 부팅 코드의 한 줄씩·링커 스크립트·ABI). 그러나 표현이 비슷하면 독자는 중복을 느낀다.
- 구체 제안:
  - Ch12 "한 줄씩 해부"라는 표현을 의도적으로 약화 — Ch12는 "동작하는 그림을 손에 넣기"가 목표. 코드는 보여주고 큰 구조만 짚되, 모든 줄을 설명하지 않는다.
  - Ch13에서 비로소 한 줄씩 들어간다 — `mpidr_el1`의 비트, `.bss` 클리어 루프, 링커 스크립트의 VMA/LMA, EL2→EL1 전환을 본격 해부.
  - 챕터 약속을 명시적으로 분리: Ch12 "동작 + 큰 그림", Ch13 "왜 그렇게 짰는가". 현 계획 본문에 이 분리가 흐릿하다.

**[Ch15 톤 약화] Ch15가 8천 자에 인터럽트·MMU·태스크 스위치·동시성·SIMD·드라이버·TrustZone·Rust 다리까지 다 담는다.**
- 이유: "다음 길라잡이"의 성격은 좋지만 8개 토픽 × 8천 자는 한 토픽당 1천 자. 짧게 짚는 게 의도라 해도 너무 펑펑 뿌리고 끝난다.
- 구체 제안: Ch15를 (a) 다음 단계 4개(인터럽트·MMU·태스크 스위치·시스템 콜)에 집중하고 각 단계가 "왜 필요한지 + 어디서 시작하는지"만 2쪽씩 = 8쪽. (b) "책에서 못 다룬 영역"은 한 절로 묶고, (c) Rust 다리는 마지막 한 페이지로 축약. 분량은 6천 자 정도로 줄여 Ch9 보강 분량을 만든다.

### Nice-to-have (개선되면 좋음, 선택)

**[제목 후보의 "베어메탈"이라는 단어가 1번 제목에 없다]**
- 이유: 책 약속에 "ARM Kernel 자작"이 있는데, 추천된 1번 제목 "Apple Silicon에서 ARM 커널까지"는 "커널까지"라고 도착지를 약속한다. 그러나 베어메탈이라는 단어는 부제에 없어 검색·발견성에서 약간 손해. (단 1번 제목의 정서가 가장 강력하다는 추천 논리는 동의.)
- 구체 제안: 그대로 유지. 띠지 카피나 카탈로그 설명에서 "베어메탈"을 명시하면 충분. 변경 권하지 않음.

**[누락 토픽: MISRA·CERT C 가이드라인]**
- 이유: 종합 바이블 약속이 강하지만, 임베디드/안전 분야에서 자주 거론되는 MISRA·CERT C·CWE는 언급되지 않는다. 다만 대상 독자가 자기 커널 학습 목적이므로 산업 표준은 부록·외전 수준이면 충분.
- 구체 제안: Ch9 "정적 분석" 절 끝에 짧은 박스(1쪽) — "clang-tidy의 `cert-*` 카테고리가 CERT C에서 왔다. MISRA C는 임베디드 인증 영역에서 표준" — 정도로 노출만 시킨다. 챕터를 만들 필요는 없다.

**[누락 토픽: `_Generic`·atomic·thread·`restrict`·VLA 폐기/부활]**
- 이유: Ch2가 C23 새 기능을 카탈로그식으로 나열하지만, `_Generic`/atomic/`<threads.h>`/`restrict`/VLA의 의미와 사용 시점에 대한 깊이 있는 다룸이 책 어디에도 없다. "C 전문가" 약속을 생각하면 얕다.
- 구체 제안:
  - `restrict`: Ch7(UB·strict aliasing) 절 끝에 한 절(1~2쪽). aliasing 가정의 명시적 약속이라는 맥락에서 자연스럽다.
  - `_Generic`: Ch10(libc 빈자리) 또는 Ch8(헤더와 링커)에서 "타입 디스패치를 직접 — `tgmath.h` 흉내내기" 짧은 데모(1~2쪽).
  - atomic·thread: Ch15 "다음 걸음"에서 "동시성" 절을 살리되, 본문에서는 다루지 않는다고 명시. 별도 챕터는 책의 초점(베어메탈 부팅)에서 벗어나므로 추가하지 않음을 권장.
  - VLA: Ch2에서 "C99에 들어왔다가 C11에서 선택적으로 후퇴, C23에서 일부 부활" 한 단락. 그대로 추가.

**[누락 토픽: lldb 디버깅 깊이]**
- 이유: Ch4에서 lldb 기본 명령(`break set -n main`, `step`, `print`)을 짚지만, 베어메탈에서 QEMU `-s -S` + 원격 디버깅은 Ch12 예제 Makefile에 `make debug`만 있다. 디버깅 워크플로의 두께가 얕다.
- 구체 제안: Ch4 끝에 "lldb 실용 가이드 — 브레이크포인트·watchpoint·`expr`·`thread backtrace`" 한 절(2~3쪽). Ch12에서는 QEMU 원격 디버깅 사이드바 1쪽으로 살린다.

**[누락 토픽: `compile_commands.json` 깊이]**
- 이유: Ch4와 Ch8에 양쪽으로 등장하지만 어디서도 깊게 다뤄지지 않는다. clangd/clang-tidy/scan-build의 공통 입구이므로 한 번 정면으로 다룰 가치가 있다.
- 구체 제안: Ch4 끝부분에 "왜 모든 모던 C 도구가 이 파일을 요구하는가" 한 절(1~2쪽). 또는 Ch8의 빌드 시스템 사다리 절 끝에 흡수.

**[Ch1 분량 약함] Ch1이 1만 자, 16~18쪽. 환영 챕터로는 적절하지만 "K&R 시절에 알던 것 중 깨진 것" 카탈로그가 책의 정서적 닻이므로 더 묵직해도 좋다.]**
- 이유: 책이 시작되는 정서적 무게는 Ch1에서 결정된다.
- 구체 제안: Ch1을 1만 2천 자로 확장. K&R에서 깨진 관용구를 5~7개 정도 코드 예제로 보이는 한 절(`void f()`, `realloc(ptr,0)`, signed overflow 가정, K&R 함수 선언 스타일, 1980년대 `int` 가정 등)을 신설.

**[Ch6 `__attribute__((cleanup))` 위치]**
- 이유: 좋은 도구. 다만 GNU 확장이 베어메탈에서 핵심이라는 사실(Ch2가 명시)을 생각하면 Ch6에 묶지 말고 Ch11과 묶어도 좋다.
- 구체 제안: 현재 위치 유지. Ch11에서 다시 언급하는 짧은 콜백 한 줄 추가.

**[example/ 디렉토리 — 다른 사용자의 접근성]**
- 이유: `example/`은 잘 짜였다. 다만 macOS 외 환경에서 `aarch64-elf-gcc`가 다른 패키지명·다른 설치 경로일 수 있다.
- 구체 제안: `example/README.md`에 "Linux 사용자의 경우 (`gcc-aarch64-linux-gnu` 등)"·"WSL 사용자의 경우" 짧은 한 절 추가. 책 본문에 넣을 필요는 없다.

### 평행한 의견 (저자 결정 존중 영역)

- **Ch4를 별도 챕터로 두는 것 — 유지 권장.** vi에서 모던 IDE로의 전환이 책 약속의 일부이고, Java/Python 출신은 IDE 셋업이 학습 속도를 결정한다. Ch3에 합치자는 안도 가능하지만 현 분리가 더 명료.
- **Ch11을 별도 챕터로 두는 것 — 유지 권장.** Ch10 → Ch12 직행은 점프가 크다. Ch11이 "정서적 다리"라는 계획의 자체 평가가 정확.
- **Ch14를 Raspberry Pi 4로 한정한 것 — 유지 권장.** 실제 보드는 한 종으로 좁히는 게 책 약속을 명확하게 한다. Cortex-M(STM32)이나 다른 보드는 Ch15에서 한 줄 노출만.

### 우선순위별 변경 권고 — 한 줄 요약

**Critical (4개, 반드시):**

### 결론

**수용 권장 변경 — 총 15개 (Critical 4, Should 5, Nice-to-have 6).**

**현 계획 유지 가능 여부 — 부분 유지 가능, 전면 재설계 불필요.**

내러티브 아크와 챕터 순서는 견고하다. 5막 구조, 정서 곡선, Ch12 정점 설계, example/ 트리는 그대로 가도 좋다. 다만 Critical 4개(특히 카운트 정정과 멘탈 모델 흡수, ABI·비트 조작 보강)는 종합 바이블 약속을 지키려면 반드시 들어가야 한다. Should 5개는 챕터 균형과 약속 명료화를 위한 것으로 강하게 권장. Nice-to-have는 저자의 분량 여력에 따라 선별.

이 변경을 모두 반영해도 챕터 수는 15장 그대로, 전체 분량은 약 17만~18만 자 수준(목표 17만~21만 자 범위 내)에 안착할 것으로 본다. 챕터를 추가하거나 순서를 바꾸는 변경은 없다 — 기존 챕터 내 절 보강과 분량 재배치만으로 충분하다.

`book-planner`의 다음 라운드에서 이 15개 항목 중 어디까지 수용할지 회신을 기다린다.

### 반영된 변경

**Critical (4건):**
1. **카운트 통일.** "14장 구성" → "15장 구성"으로 정정. 본문 두 곳("14장 끝에 독자가 어디", "14장 여정의 지도")도 모두 "15장"으로 통일. 분량 표기도 15장 평균으로 재산정.
2. **Ch5 도입부 멘탈 모델 절 추가.** "(도입부) 멘탈 모델 전환 — Java/Python의 거리" 1~2쪽 분량으로 신설. Java GC·NPE·try/catch, Python ref counting·KeyError와 C의 거리를 7행 비교 표로. 레퍼런스 §8 흡수. Ch5 분량 1만 3천 → 1만 4천 자.
3. **Ch8에 ABI/calling convention 절 추가.** "함수 호출의 실제 — ABI와 calling convention" 2~3쪽. x86-64 SysV vs AArch64 PCS 정면 비교(X0~X7, X8 indirect, X29 FP, X30 LR). `objdump -d`로 프롤로그·에필로그 한 명령씩 읽기. example에 `abi_peek/` 데모 추가. Ch8 분량 1만 2천 → 1만 3천 자. Ch12·Ch13에서 X0~X7이 갑자기 튀어나오지 않게 다리 놓음.
4. **Ch7에 `<stdint.h>`·비트 조작·엔디언 절 추가.** "고정폭 정수와 비트 조작" 4~5쪽. K&R `int`/`long` 가정 깨짐 → `uint8_t`/`uint16_t`/`uint32_t`/`uint64_t`, signed/unsigned 비대칭, 비트 마스크·시프트 관용구, 엔디언 검사·`__builtin_bswap32`. example에 `bits_demo/` 추가. Ch7 분량 1만 3천 → 1만 4천 자. Ch12 MMIO와 Ch13 부팅 코드 어셈블리 읽기에 다리 놓음.

**Should (5건):**

Ch9 분량 1만 2천 → 1만 6천 자(+4천). example에 `Makefile.matrix` 5열 메타 타깃 추가.
