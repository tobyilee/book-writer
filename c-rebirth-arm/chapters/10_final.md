# 10장. 표준 라이브러리의 빈자리 — string·io·container 한계와 대안

Java 시절을 떠올려 보자. 단어를 세고 싶으면 `HashMap<String, Integer>` 한 줄, 정렬하고 싶으면 `Collections.sort`, 정규식이 필요하면 `java.util.regex`. JDK가 한 줄 거리에 다 갖춰 두었다. Python은 더하다 — `dict`, `list`, `collections.Counter`까지 표준 라이브러리에 있다.

같은 손으로 C 표준 라이브러리(`libc`)의 헤더 목록을 펴 보자. `<string.h>`에는 `strlen`·`strcpy`·`strcmp` 같은 NUL 종결 문자열 함수가 들어 있고, `<stdio.h>`에 `printf` 가족이 있고, `<stdlib.h>`에 `malloc`/`free`가 있다. **그리고 거기서 끝난다.** 동적 배열도, 해시맵도, 정규식도, 소켓도 없다. C11에서야 스레드가, C23에 와서야 checked arithmetic이 표준에 들어왔다.

이 사실을 처음 마주하면 난감하다. "그럼 매번 직접 만들어 쓰라는 건가?"라는 질문이 솟구치는데, 사실 그게 C 표준이 들고 있는 입장이다 — **필요한 추상은 그 추상이 필요한 사람이 짠다.** 이번 장에서는 그 빈자리를 정직하게 들여다보자. 무엇이 비어 있는지, 어디까지를 표준만으로 갈 수 있는지, 그 너머를 무엇으로 메우는지.

## 문자열의 한계 — NUL 종결의 두 얼굴

C 문자열의 정체는 한 문장으로 정리된다 — **NUL(`'\0'`) 바이트로 끝나는 `char` 배열.** 길이를 따로 저장하지 않는다. 그래서 `strlen("hello")`는 `'\0'`을 만날 때까지 한 글자씩 센다 — O(n). 1만 자짜리 문자열의 길이를 알고 싶으면 1만 번을 센다. Java의 `String.length()`가 O(1)이었던 시절이 그립다면, 그 그리움은 정확한 감각이다.

길이를 모른다는 사실은 단순한 성능 문제가 아니다. **버퍼 오버플로의 고전적 원천**이 여기에서 시작된다. 이 코드를 보자.

```c
char dst[8];
strcpy(dst, "hello, world");  // 12바이트 + NUL = 13바이트가 8바이트 칸에 들어간다
```

`strcpy`는 `dst`의 크기를 모른다. 알 길도 없다 — 인수는 그냥 `char *`다. 그저 NUL을 만날 때까지 복사할 뿐이라, 옆에 자리잡은 변수든 스택의 리턴 주소든 그대로 덮어쓴다. Java에서 `ArrayIndexOutOfBoundsException`을 봤다면, C는 같은 자리에서 **침묵한다**. 운이 좋으면 segfault, 운이 나쁘면 30분 뒤 무관해 보이는 위치에서 터진다.

그래서 "길이를 받는 변종"이 등장했다. 정석은 다음 셋이다.

| 함수 | 보장 | 함정 |
|------|------|------|
| `strncpy(dst, src, n)` | n 바이트까지만 복사 | **NUL로 끝낸다는 보장이 없다.** src가 n보다 길면 dst에 NUL이 안 붙는다 |
| `snprintf(dst, n, "%s", src)` | n-1 바이트 + NUL 보장 | 가장 안전하지만, 잘렸는지 여부를 리턴값으로 직접 확인해야 한다 |
| `strlcpy(dst, src, n)` (BSD·Apple) | n-1 바이트 + NUL 보장. 잘리면 원본 길이 리턴 | **표준이 아니다.** Apple·BSD·musl엔 있고 glibc엔 (오래) 없었다 |
| `memccpy(dst, src, c, n)` | c 바이트를 만나면 거기서 멈춤 | C23에 정식 표준화됨. 안전하지만 인터페이스가 낯설다 |

이 표를 처음 보면 `strncpy`의 어색함이 눈에 박힌다. **이름은 안전해 보이는데 NUL 종결을 보장하지 않는다.** 1970년대에 정해진 의미를 표준이 그대로 끌고 왔기 때문이다. 그래서 모던 C에서 `strncpy`를 쓰면서 NUL 종결까지 챙기려면 한 줄이 더 붙는다.

```c
strncpy(dst, src, sizeof dst - 1);
dst[sizeof dst - 1] = '\0';   // 잊지 말자. strncpy는 이 한 줄을 책임지지 않는다
```

찜찜한 관용구다. 이 찜찜함 때문에 BSD 사람들이 `strlcpy`를 만들었고, Apple·musl이 받아들였고, glibc는 한참을 거부하다 결국 받아들였다. C23에 정식으로 들어가지는 않았지만 "사실상 표준"으로 간주해도 무리가 없다 — Linux 커널 안에서도 광범위하게 쓰인다.

그렇다면 신규 코드에서는 어떻게 가는 게 좋을까? **권장은 `snprintf` 또는 (있을 때) `strlcpy`.** 두 함수 모두 다음 둘을 보장한다 — (1) 목적 버퍼를 넘지 않는다, (2) 결과는 항상 NUL로 끝난다. 그게 우리가 원하는 두 가지 전부다.

C23은 이 영역에 작은 보강을 더 얹었다. `memset_explicit`은 컴파일러가 "어차피 값을 안 쓰니 memset 호출을 지워도 된다"는 최적화를 못 하게 막는다 — 비밀번호 바이트를 지울 때 쓴다. `strdup`이 드디어 정식 표준이 됐다 (POSIX에는 오래 있었다). `<stdckdint.h>`의 `ckd_add`/`ckd_mul`로 오버플로를 깔끔하게 잡을 수 있다. 작지만 이런 보강이 쌓이면 안전 코딩의 부담이 천천히 줄어든다.

여기서 한 가지 더 짚어 두자. **C 문자열은 바이트 배열이지 문자 배열이 아니다.** Java의 `String`이 UTF-16 코드 유닛 배열이고 Python 3의 `str`이 코드 포인트 배열인 것과는 결이 다르다. C에서 `"안녕"`을 들고 있으면 그건 UTF-8로 인코딩된 6바이트 바이트 시퀀스다 (NUL 종결 포함 7바이트). `strlen("안녕")`은 2가 아니라 6을 리턴한다 — 글자 수가 아니라 바이트 수다.

이걸 의식하지 않으면 "한글 한 자가 두 자처럼 들어왔다"는 식의 혼란이 생긴다. 모던 C 코드는 보통 다음 규칙으로 산다 — **저장과 입출력은 UTF-8 바이트로, 글자 단위 조작이 필요할 때만 코드 포인트로 변환.** 글자 단위가 필요한 일은 의외로 드물다. 파일 입출력·소켓 전송·로깅까지 다 바이트 단위면 충분하다. 그러니 한글이 안 깨지게 다루겠다는 결심이 곧 UTF-8을 그대로 들고 다니겠다는 결심이 된다.

문자열을 손에 잡기 시작했으니, 다음 빈자리로 가 보자.

## 컨테이너 부재 — 자작 패턴이 일상이다

배열, 리스트, 맵, 셋. 다른 언어에서 한 줄로 끝나던 일이 C에서는 통째로 부재한다. **C 표준 라이브러리에는 컨테이너가 없다.** `qsort`와 `bsearch` 정도가 다인데, 그것조차 `void *`와 비교 함수 포인터를 직접 넘기는 방식이다.

이 사실 앞에서 다른 언어 출신은 두 가지 반응을 보인다. 첫째는 "그럼 외부 라이브러리 가져다 쓰면 되지" — 맞다. uthash, sds, klib 같은 헤더-온리 라이브러리가 있다. 둘째는 "직접 짤 수 있는 건 직접 짠다" — 이쪽이 C 문화에 더 가깝다. 작은 프로젝트, 베어메탈, 학습용 코드는 매번 직접 만든다. 우리 책의 example 디렉토리도 그 길을 따른다.

그렇다면 직접 만드는 동적 배열은 어떻게 생겼을까? 정석 한 장을 보자. 책의 `example/ch10-libc-gap/vec.c`에 같은 코드가 들어 있다.

```c
struct vec_t {
    int   *data;
    size_t len;
    size_t cap;
};

static int vec_grow(vec_t *v)
{
    size_t new_cap = (v->cap == 0) ? 4 : v->cap * 2;
    int *new_data = realloc(v->data, new_cap * sizeof *new_data);
    if (!new_data) return -1;
    v->data = new_data;
    v->cap = new_cap;
    return 0;
}

int vec_push(vec_t *v, int value)
{
    if (v->len == v->cap) {
        if (vec_grow(v) != 0) return -1;
    }
    v->data[v->len++] = value;
    return 0;
}
```

핵심은 셋이다. **데이터 포인터·길이·용량 세 필드**, **용량이 차면 두 배로 늘리기**, **`realloc` 실패는 호출자에게 -1로 전달.** Java의 `ArrayList`가 안에서 하는 일도 본질적으로 이것이다. 다만 Java는 GC가 옛 배열을 알아서 회수하고, C는 `realloc`이 옛 영역을 알아서 풀어 준다 (실패 시 옛 포인터는 유효하게 살아 있다는 점을 잊지 말자 — `realloc` 결과를 같은 변수에 바로 대입하면 실패 시 옛 메모리를 잃는다).

해시맵은 좀 더 복잡하다. 책의 `map.c`에는 open addressing + FNV-1a 해시 + load factor 0.75 리해시까지 200줄 안에 들어 있다. 한 조각만 끄집어내 보자.

```c
static uint32_t fnv1a(const char *s)
{
    uint32_t h = 2166136261u;
    while (*s) {
        h ^= (uint8_t)*s++;
        h *= 16777619u;
    }
    return h;
}
```

FNV-1a는 한 줄짜리 비암호용 해시다. 4줄에 한 함수가 끝난다. 충돌 해결에는 두 방식이 있는데 — **chaining**(슬롯마다 연결 리스트)과 **open addressing**(빈 자리 찾을 때까지 다음 슬롯으로 이동) — 베어메탈에 더 가까운 쪽은 open addressing이다. 메모리 할당을 슬롯당 한 번씩 안 해도 되니까.

흥미로운 점은 삭제다. open addressing에서 슬롯을 그냥 비워 버리면 같은 해시값을 갖는 다른 키를 못 찾게 된다 — probing 사슬이 끊기기 때문이다. 그래서 **tombstone**(묘비)이라는 제3의 상태를 둔다. "지워졌지만 사슬은 살아 있다"는 표시다. 책의 `map.c`에서 그 처리를 직접 확인할 수 있다.

매크로로 제네릭 컨테이너를 만드는 길도 있다. `klib`의 `khash.h`가 대표적인 예인데, 매크로 인자로 타입을 받아서 같은 코드를 여러 타입에 펼친다.

```c
KHASH_MAP_INIT_STR(name2age, int)   // string → int 맵
```

이 한 줄이 매크로 펼침을 거치면 수백 줄의 인라인 함수로 부풀어 오른다. C에서 generic을 흉내내는 정석 중 하나다. 다만 매크로 디버깅이 어렵고 컴파일 에러가 끔찍해진다는 단점이 있다 — 그래서 학습 단계에서는 한 타입에 한 자료구조를 직접 짜는 편이 낫다.

C23이 이 영역에 작은 길을 하나 더 텄다. `typeof` 키워드가 정식 표준에 들어왔으니, 매크로 안에서 인수의 타입을 끌어다 쓸 수 있다.

```c
#define SWAP(a, b) do {           \
    typeof(a) tmp = (a);          \
    (a) = (b);                    \
    (b) = tmp;                    \
} while (0)
```

C++의 템플릿만큼 깔끔하지는 않지만, 그래도 GNU 확장에 의존하지 않고 같은 일을 할 수 있게 됐다. 이런 작은 보강이 쌓여 모던 C의 표정이 천천히 바뀐다.

자작 컨테이너에는 흔히 깔리는 함정이 셋 있다. **첫째, `realloc` 실패 처리를 잊는다.** `v->data = realloc(v->data, n)` 한 줄로 끝내면 NULL이 돌아왔을 때 옛 메모리도 잃는다. 임시 변수에 받아서 NULL 체크 후 대입하는 패턴이 정석이다. **둘째, 포인터 무효화를 잊는다.** 동적 배열이 커지면 `realloc`이 메모리를 옮겨 둘 수 있어, 옛 주소를 들고 있던 포인터가 갑자기 무효가 된다 — Java의 `ArrayList`처럼 인덱스로만 접근하는 게 안전하다. **셋째, 비교/해시 함수의 일관성을 안 챙긴다.** 해시 함수가 같은 값을 두 번 호출해도 같은 결과여야 하고, `==`가 참인 두 키는 해시값도 같아야 한다 — 이 두 약속이 깨지면 맵이 조용히 망가진다.

이 세 함정은 ASan/UBSan을 켜고 빌드하면 대부분 잡힌다. 책의 `example/ch10-libc-gap/Makefile`이 기본으로 `-fsanitize=address,undefined`를 켜 두는 이유다 — 만들면서 동시에 안전망을 깔아 둔다는 9장의 약속이 여기까지 이어진다.

이쯤에서 솔직하게 정리해 두자. **C에서 자료구조를 직접 짜는 일은 부담스럽지만, 그 부담이 우리가 코드에서 보고 있는 것을 좁힌다.** Java의 `HashMap`을 쓰면서 그 안의 해시 함수, 충돌 처리, 리사이즈 정책을 의식하는 사람은 드물다. C에서 직접 짜고 나면 그 셋이 손에 잡힌다. 그게 베어메탈로 가는 발판이 된다 — 거기서는 라이브러리도 없으니 자료구조를 직접 짜는 일이 일상이 된다.

## POSIX vs C 표준 — 의존성 사다리

여기서 잠시 멈춰 한 가지 구분을 정리해 두자. **C 표준 라이브러리**와 **POSIX**는 같은 것이 아니다. 자주 헷갈리는 자리다.

C 표준 라이브러리(libc)는 ISO C가 정의한다 — `<stdio.h>`, `<string.h>`, `<stdlib.h>`, `<math.h>` 같은 헤더. 운영체제와 무관하게 어디든 있다고 약속된 영역이다.

POSIX는 그 위에 얹는 유닉스 계열 운영체제 표준이다 — `<sys/socket.h>`(소켓), `<pthread.h>`(스레드 — C11 이전부터), `<unistd.h>`(`fork`/`exec`/`read`/`write`), `<sys/stat.h>`(파일 시스템). macOS와 Linux는 다 POSIX 인터페이스를 들고 있고, Windows는 안 들고 있다 (WSL이나 MSYS 같은 호환 레이어를 통해서만).

그러니 같은 "C 표준 라이브러리에 없는 것"이라도 사정이 다르다.

| 필요 | C 표준에 있나? | POSIX에 있나? | 그 외 |
|------|--------------|-------------|------|
| 정규식 | 없음 | `<regex.h>` (POSIX 확장 정규식) | PCRE2, RE2 |
| 소켓·네트워킹 | 없음 | `<sys/socket.h>`로 풍부 | libcurl, libuv |
| 스레드 | C11부터 `<threads.h>` | `<pthread.h>` | libuv, OpenMP |
| 파일 시스템 (디렉토리 열기) | 없음 | `<dirent.h>`, `<sys/stat.h>` | — |
| 동적 배열·해시맵 | 없음 | 없음 | uthash, klib, sds, stb |
| JSON·YAML 파싱 | 없음 | 없음 | cJSON, jansson, yaml-cpp |

이 표를 머리에 박아 두면 "내가 지금 어디에 의존하고 있는가"가 보인다. 사다리는 이런 구조다 — **C 표준 → POSIX → 외부 라이브러리**. 위로 올라갈수록 기능이 늘고 이식성이 줄어든다. 베어메탈로 갈 거라면 사다리의 맨 아래, C 표준만으로 어디까지 갈 수 있는지를 알아야 한다 (사실 베어메탈에서는 그 C 표준조차 일부만 쓸 수 있다 — 다음 장의 주제다).

macOS에서 일하는 우리는 POSIX 영역에 자유롭게 발을 디딘다. `pthread.h`도, `sys/socket.h`도, `fork`도 다 동작한다. 다만 **POSIX 안에서도 미세하게 다르다.** Apple의 BSD 계열 POSIX와 Linux의 GNU 확장이 똑같지 않다. `epoll`은 Linux에만 있고, `kqueue`는 BSD/Apple에만 있다. 이런 차이를 가리려고 보통 `libevent`/`libuv` 같은 추상 레이어를 한 겹 얹는다.

베어메탈 ARM 커널을 향해 가는 우리의 여정에서는, **POSIX 사다리를 아예 잘라 버리는** 순간이 곧 온다 (다음 장이다). 호스트 OS가 없으면 `fork`도 `pthread`도 의미가 없다 — 우리가 OS이기 때문이다. 그러니 지금 이 단계에서 POSIX 의존을 의식적으로 격리해 두는 습관이 도움이 된다. POSIX 호출은 따로 모아 `os_posix.c` 같은 한 파일에 넣어 두고, 핵심 로직은 C 표준만으로 짜는 패턴이 그래서 권장된다. 베어메탈 빌드에서는 `os_posix.c`를 빼고 `os_bare.c`를 끼우면 같은 로직이 그대로 흐른다.

## libc 구현체 차이 — 같은 표준, 다른 모서리

마지막 빈자리는 표면에 잘 안 드러나지만 한번 부딪히면 길게 따라온다 — **같은 C 표준이라도 구현체에 따라 미묘하게 다르다.**

세 구현체가 흔하다.

- **glibc** (GNU libc) — Linux 데스크톱·서버의 사실상 표준. 가장 풍부한 확장.
- **musl** — Linux의 가벼운 대안. Alpine, Docker 이미지, 정적 링킹 친화적.
- **Apple libSystem** — macOS의 BSD 계열 libc. `dyld` 동적 링커와 묶여 있다.

(이 외에도 Windows의 MSVCRT, 임베디드의 newlib·picolibc 등이 있다.)

표준이 같으니 `strlen`·`strcpy`·`malloc`은 다 똑같이 동작한다. 그런데 모서리에서 갈라진다. 몇 예를 보자.

- `strlcpy`/`strlcat`은 Apple과 musl엔 있고 glibc엔 (2.38 이전) 없었다. 그래서 Linux로 옮기면 컴파일이 깨진다.
- `qsort_r`(reentrant qsort)는 인수 순서가 glibc와 BSD가 다르다 — 같은 이름인데 첫 인수와 마지막 인수가 뒤바뀐다. 한쪽 코드를 다른 쪽에 그대로 옮기면 자료를 망친다.
- `getline`은 POSIX 2008에 표준화됐지만, 옛 macOS 헤더는 `_POSIX_C_SOURCE`나 `_DARWIN_C_SOURCE` 매크로를 정의해야 노출됐다.
- `realpath(path, NULL)` — NULL을 두 번째 인자로 넘기면 함수가 직접 버퍼를 할당해 준다. glibc·Apple·musl 모두 지원하지만 옛 BSD에서는 안 됐다.

이런 차이를 다루는 정석은 **`_POSIX_C_SOURCE`·`_GNU_SOURCE`·`_DARWIN_C_SOURCE` 같은 feature test macro를 컴파일 옵션에 명시**하는 것이다.

```c
#define _POSIX_C_SOURCE 200809L   // POSIX 2008 인터페이스 노출
#include <unistd.h>
```

이 매크로는 `#include`보다 먼저 정의해야 효과가 있다. 보통은 컴파일러 플래그(`-D_POSIX_C_SOURCE=200809L`)로 박는다.

이식성을 챙기는 코드는 보통 다음 패턴을 따른다 — 헤더 어딘가에 `compat.h`를 한 장 두고, 함수가 없는 플랫폼을 위한 polyfill을 거기에 모은다.

```c
/* compat.h */
#if !defined(__APPLE__) && !defined(__MUSL__)
#  if __has_include(<bsd/string.h>)
#    include <bsd/string.h>      // libbsd 패키지가 깔린 Linux
#  else
size_t strlcpy(char *dst, const char *src, size_t size);   // 직접 polyfill
#  endif
#endif
```

이런 한 줄의 분기를 코드에 묻어 두면, CI에서 macOS와 Linux를 모두 도는 일이 무난해진다. 두 플랫폼을 동시에 도는 GitHub Actions matrix가 정석이다.

```yaml
strategy:
  matrix:
    os: [macos-latest, ubuntu-latest]
```

CI에서 두 OS를 다 도니까, 모서리 차이가 머지 전에 잡힌다. 한쪽에서만 빌드되는 코드를 한 달 묵힌 뒤 마이그레이션하다 깨지는 일을 막아 준다.

`__has_include` (C23부터 표준, 그 이전 GCC/Clang 확장)도 이런 분기를 깔끔하게 다듬는 데 쓴다.

```c
#if __has_include(<bsd/string.h>)
#  include <bsd/string.h>
#elif !defined(__APPLE__) && !defined(__OpenBSD__)
   /* glibc 2.38 이전이거나 libbsd가 없는 환경 */
   size_t strlcpy(char *dst, const char *src, size_t size);
#endif
```

`#ifdef __APPLE__` 같은 플랫폼 매크로보다 **"기능이 있는가"**를 직접 묻는 쪽이 더 정확하다. 같은 Linux라도 musl과 glibc가 다르고, 같은 glibc라도 2.38 전후가 다르다. 플랫폼이 아니라 기능을 묻는 분기가 미래에 잘 늙는다.

여기서 한 가지 솔직하게 인정해 두자. **이식성은 부가 기능이 아니라 의식적 결정이다.** "내 코드는 모든 곳에서 도는 게 좋다"는 추상적 바람만으로는 안 된다. 어떤 플랫폼·어떤 표준 버전·어떤 libc 구현체를 지원할지를 명문화하고, CI에서 그 조합을 매 PR마다 돌려야 한다. 그 비용이 부담스러우면 지원 플랫폼을 좁히는 게 정직하다 — "리눅스만 지원합니다", "Apple Silicon만 지원합니다"라고 README 첫 줄에 쓰는 편이, 깨지는 코드를 모든 곳에서 굴리는 것보다 낫다.

## 한 발 더 — 외부 라이브러리를 빌릴 때

빈자리를 직접 메우는 일이 모든 경우에 옳지는 않다. **이미 잘 짜인 헤더-온리 라이브러리가 충분히 있다.** 한 번쯤 손에 익혀 두면, 자작과 외부 사이에서 매번 의식적으로 고를 수 있게 된다.

| 영역 | 라이브러리 | 특징 |
|------|----------|------|
| 해시맵 | `uthash` (Troy D. Hanson) | 매크로 기반, 구조체 필드 하나만 추가하면 해시 가능 |
| 동적 문자열 | `sds` (Salvatore Sanfilippo, Redis 출신) | 길이 정보를 헤더에 보관, `char *`처럼 쓰면서도 안전 |
| 일반 유틸 | `klib` (Heng Li) | `khash`, `kvec`, `kstring` — 작고 빠르다 |
| 미디어·이미지 | `stb` (Sean Barrett) | `stb_image`, `stb_truetype` — 한 헤더 한 기능 |
| JSON | `cJSON` | 작고 의존 없음. 임베디드에도 잘 들어간다 |

이 라이브러리들의 공통점이 있다 — **하나같이 작다.** 한 헤더(또는 헤더 + 한 파일)로 끝난다. 빌드 시스템을 건드리지 않고 프로젝트에 떨어뜨려 쓸 수 있다. 이게 C 생태계의 묘한 미덕이다. 큰 의존을 만들지 않는 작은 도구들이 잘 살아남는다.

다만 외부 라이브러리를 가져올 때마다 묻는 게 좋다 — **"이걸 의존하면 베어메탈로 옮길 때 깨지나?"** 동적 할당을 쓰는 라이브러리는 베어메탈에서 우리가 직접 마련한 할당자에 연결해야 한다. `printf`를 부르는 라이브러리는 우리의 `mini_printf`로 갈아 끼워야 한다. **이식성은 의존성의 모양으로부터 시작된다.** 헤더-온리 라이브러리가 사랑받는 이유가 거기에 있다 — 안에서 무엇을 부르는지 한눈에 보이니까.

## 마무리

C 표준 라이브러리의 빈자리를 네 각도에서 살펴봤다. **문자열은 NUL 종결 한 줄에 모든 함정이 들어 있고**, **컨테이너는 부재 자체가 기본값**이며, **POSIX는 그 위에 얹는 별도 표준**이고, **같은 libc도 구현체별로 모서리가 다르다.**

이 빈자리들을 처음 보면 부담이 크게 느껴진다. 하지만 한 번 자기 손으로 `vec`과 `map`을 짜 보고 나면, 다른 언어에서 `ArrayList`와 `HashMap`이 안에서 하던 일이 손에 잡히는 감각이 따라온다. 그게 C가 가진 묘한 보상이다 — 추상이 적은 만큼, 보고 있는 것이 좁아진다.

기억해 두자.

- **`strcpy`는 잊자.** 신규 코드에서는 `snprintf` 또는 `strlcpy`를 기본값으로.
- **자료구조는 작게 시작하자.** 한 프로젝트에 컨테이너 한 종류면 충분한 경우가 의외로 많다. 부족해지면 그때 늘리면 된다.
- **POSIX와 C 표준을 의식적으로 구분하자.** 매크로 한 줄로 노출 범위를 통제할 수 있다는 사실을 잊지 말자. POSIX 의존은 한 파일에 모아 두는 편이 좋다.
- **이식성은 CI matrix로 잡자.** 한쪽에서만 빌드되는 코드는 다른 쪽에서 결국 깨진다. `__has_include`와 feature test macro가 손에 익으면 코드가 곱게 늙는다.
- **자작 컨테이너에는 ASan을 같이 두자.** 안전망 없이 손으로 짠 자료구조는 한 줄 실수로 한 시간을 태운다. 9장에서 짜 둔 매트릭스가 여기서 다시 일한다.

다음 장에서는 더 좁은 세계로 들어간다. **표준 라이브러리 자체가 없는** 세계 — freestanding 모드. `printf`도 `malloc`도 `errno`도 없는 그곳에서, 우리가 가진 게 무엇이고 무엇을 직접 마련해야 하는지를 본다. 베어메탈 ARM 커널로 가는 길의 마지막 다리다.
