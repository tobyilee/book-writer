# 5장. 포인터를 다시 마주하다 — 메모리의 별명

오랜만에 펼친 C 책의 한 장을 보다가, "포인터" 세 글자 앞에서 손이 잠깐 멈춘다고 해보자. 옛 기억엔 `*`와 `&`가 어지럽게 얽혀 있고, 그 시절엔 분명히 이해했다고 자부했는데 지금은 어딘가 흐릿하다. Java로 10년, JavaScript로 5년, Python을 거치는 동안 우리는 메모리 주소라는 것을 한 번도 직접 만질 필요가 없었다. 객체는 어디엔가 살고 있었고, 변수는 그 객체를 가리키는 이름이었으며, 더 이상 필요 없어지면 가비지 컬렉터가 알아서 정리해 줬다.

이제 그 안락한 동네에서 잠깐 떠나야 한다. 그렇다고 너무 겁먹지는 말자 — 우리는 다시 처음부터 배우는 게 아니다. 흐릿해진 옛 기억을 다섯 시나리오로 다시 정돈하면 손은 곧 따뜻해진다. 그러나 그 다섯 시나리오에 들어가기 전에, 더 먼저 해야 할 일이 있다. **어디서 어디로 자리를 옮겨야 하는지를 짚는 일이다.**

## 멘탈 모델 전환 — Java/Python의 거리

C에 다시 발을 들이는 사람이 가장 먼저 겪는 어려움은 문법이 아니다. **어떤 마음가짐으로 코드를 읽어야 하는지**를 잊었다는 점이다. 우리가 Java나 Python에서 자유롭게 쓰던 직관 몇 가지가 C에서는 그대로 통하지 않는다. 한 번 표로 정면에 두고 가자.

| 축 | Java / Python | C |
|----|---------------|---|
| 메모리 회수 | GC가 알아서 | `free`를 손으로 |
| null 접근 | `NullPointerException` 던짐 | UB — 운에 따라 segfault 또는 침묵 |
| 배열 경계 밖 접근 | `ArrayIndexOutOfBoundsException` | UB — 옆 메모리를 조용히 깨뜨릴 수 있음 |
| 타입 안전 | 강타입 + 런타임 검사 | 정적 + 약타입, 묵시 변환이 흔함 |
| 에러 전달 | 예외 throw/catch | 반환 코드 + `errno` |
| 변수가 가리키는 것 | 객체 참조 (간접) | 메모리 셀의 별명 (직접) |
| 실행 모델 | 인터프리터 / JIT | 컴파일 → 링크 → 정적 바이너리 |

이 일곱 줄 중에서 가장 충격적인 줄은 두 번째와 세 번째다. **Java는 잘못된 접근에 비명을 지른다. C는 침묵한다.** 운이 좋으면 segfault로 즉시 죽기라도 하지만, 운이 나쁘면 옆 메모리가 조용히 깨지고 한참 뒤 전혀 엉뚱한 곳에서 문제가 터진다. 그 추적이 끔찍하다 — 원인과 증상 사이의 거리가 한 함수가 아니라 수십 함수, 때로는 수십 분의 실행 시간이다.

Java 개발자가 C 코드를 처음 만지고 가장 자주 듣는 말이 있다. "코드가 갑자기 망가져요." 그건 코드가 갑자기 망가진 게 아니다. **한참 전에 망가진 코드의 결과가 이제야 보이는 것이다.** 이 시간차가 C 디버깅의 본질이고, 이 책의 9장에서 다룰 sanitizer가 정확히 이 시간차를 좁히는 도구다.

좋은 소식은 우리가 평생 이 어둠 속을 더듬으며 살아야 하는 건 아니라는 점이다. 도구가 있다. ASan, UBSan, lldb 디버거. 나쁜 소식은 그 도구를 켜기 전까지는 C가 침묵한다는 사실 자체는 변하지 않는다는 점이다. 이게 우리 멘탈 모델의 첫 번째 자리 옮기기다.

두 번째 자리 옮기기는 메모리에 관한 것이다. Java에서 `String s = new String("hello")`라고 쓸 때, `s`는 `String` 객체를 **가리키는** 참조다. 객체 자체는 힙 어딘가에 있고, `s`는 그 어딘가의 손가락질이다. 객체가 어디 있는지, 어떻게 배치되었는지는 우리가 알 일이 아니다. JVM이 알아서 한다.

C에선 다르다. `int x = 42;`라고 쓰면 어딘가의 메모리 셀에 `42`라는 값이 들어가고, **`x`는 그 셀의 별명이다.** 셀 자체에 주소가 있고, 그 주소를 우리가 직접 얻을 수 있고(`&x`), 다른 변수에 담을 수 있고(`int *p = &x`), 그 변수를 통해 셀을 다시 만질 수 있다(`*p = 7`). 이게 포인터다. 별 거 아니다. 별명의 별명일 뿐이다.

이 단순한 직관 — **"포인터는 메모리의 별명이다"** — 이게 이 챕터의 출발점이자 도착점이다. 이걸 손에 잡으면 나머지는 그 변주다.

세 번째 자리 옮기기는 에러 처리에 관한 것이다. Java나 Python에서 우리는 `try/catch`로 예외를 잡는 게 자연스럽다. 코드의 정상 흐름과 예외 흐름이 분리되어 있고, 한참 떨어진 함수의 실패도 한 곳에서 받을 수 있다. 깔끔한 모델이다.

C에는 예외가 없다. **에러는 반환 값으로 전달한다.** `0`이 성공, `-1`이 실패인 라이브러리도 있고, `NULL`이 실패인 라이브러리도 있고, POSIX는 `-1`을 반환하고 `errno`에 자세한 원인을 적어 두는 패턴이 많다. 라이브러리마다 컨벤션이 달라서 헷갈리는데, 한 가지 일관된 점은 — **모든 함수 호출 다음에 결과를 검사해야 한다**는 점이다.

```c
FILE *fp = fopen("data.txt", "r");
if (!fp) {
    perror("fopen");
    return -1;
}
// ... 사용 ...
fclose(fp);
```

이 단순한 패턴이 C 코드의 모양새다. Java에서 `try` 블록 한 번으로 묶었을 일이 C에서는 매 호출 다음 검사로 흩어진다. 코드 양이 늘어 보이지만, 흐름이 더 명시적이다 — 어디서 실패할 수 있는지가 한눈에 보인다. 우리가 6장에서 `goto cleanup`이라는 관용구를 배우는 이유가 여기에 있다.

## 첫 번째 시나리오 — 주소를 눈으로 보기

말로만 하면 막연하다. 코드로 들어가자. 이 챕터의 예제 디렉터리(`example/ch05-pointers/`)에 다섯 시나리오를 한 파일에 모은 `pointers.c`가 있다. 빌드하고 돌려 보자.

```bash
make && make run
```

첫 시나리오는 가장 단순하다.

```c
static void addr_basic(void) {
    int x = 42;
    int *p = &x;

    printf("&x      = %p\n", (void *)&x);
    printf("p       = %p\n", (void *)p);
    printf("*p      = %d\n", *p);

    *p = 7;
    printf("after *p=7, x = %d\n", x);
}
```

실행하면 이런 출력이 나온다(주소는 매번 다르다).

```
&x      = 0x16b17565c
p       = 0x16b17565c
*p      = 42
after *p=7, x = 7
```

**무슨 일이 일어났는가.** `x`라는 변수를 위해 메모리 한 셀이 잡혔고, 그 셀에 `42`가 들어갔다. 그 셀의 주소가 `0x16b17565c`다(스택 어딘가다 — 이 책의 6장에서 다룬다). `&x`는 그 주소를 묻는 연산자다. `p`는 그 주소를 담은 변수, 그러니까 포인터다. `*p`는 `p`가 가리키는 자리의 값을 보거나(`*p`를 읽기), 거기에 쓰거나(`*p = 7`)다.

마지막 줄이 핵심이다. `p`를 통해 셀에 `7`을 썼을 뿐인데, `x`도 `7`이 됐다. 당연하다 — `p`와 `x`는 같은 셀의 두 별명이니까. 이 한 줄을 머리에 박아 두면 포인터의 80%가 손에 들어온 셈이다.

`(void *)`로 캐스팅한 게 약간 거슬릴 수 있다. `printf`의 `%p` 포맷은 명세상 `void *`를 받는다고 정해져 있어서, 다른 포인터 타입을 그대로 넘기면 컴파일러가 깐깐하게 경고한다. 이 책에서는 그 약속을 지킨다.

## 두 번째 시나리오 — 포인터의 포인터

이제 한 단계 깊이 들어간다. 함수가 새 객체를 만들어 호출자에게 돌려주고 싶다고 해보자. Java라면 단순하다 — `return new Box()`. 반환 값으로 객체를 돌려주면 끝이다.

C에서는 두 길이 있다. 첫째, 함수가 `int *`를 반환한다(이게 더 흔하다). 둘째, 함수가 `int **`를 인자로 받아 그 자리에 새 포인터를 써 준다. 두 번째 패턴이 어색해 보이지만, 함수의 반환값을 이미 다른 용도(에러 코드)로 쓸 때 종종 등장한다.

```c
static void make_int(int **out, int seed) {
    *out = malloc(sizeof **out);
    if (*out) **out = seed * seed;
}

static void pointer_to_pointer(void) {
    int *box = NULL;
    make_int(&box, 9);
    if (box) {
        printf("make_int(9) → %d\n", *box);   // 81
        free(box);
    }
}
```

여기서 잠깐 멈춰서, `*out = malloc(sizeof **out)` 한 줄을 천천히 읽자. **`out`은 `int **`다 — int 포인터를 가리키는 포인터.** `*out`은 한 단계 풀어서 `int *`를 가리킨다. 거기에 `malloc`의 결과(또 다른 `int *`)를 대입한다. `**out`은 두 단계 풀어서 `int`다. 그래서 `sizeof **out`은 `int` 크기를 묻는다. `**out = seed * seed`는 그 `int` 셀에 값을 쓴다.

처음 보면 별 두 개가 어지럽다. 그러나 별 하나 = 한 단계 역참조, 라는 규칙만 잡으면 손이 풀린다.

호출 쪽에서는 `&box`로 "내 `box` 변수의 주소를 줄게, 거기에 새 포인터를 써 줘"라고 부탁한다. 함수가 끝나면 `box`는 더 이상 NULL이 아니다 — `malloc`이 돌려준 힙 주소가 들어 있다. 그래서 `*box`로 그 자리의 `81`을 읽을 수 있고, 다 쓰면 `free(box)`로 돌려준다.

소유권 약속을 다시 짚자. `make_int`는 새 메모리를 잡아 호출자에게 넘긴다. **그 메모리의 `free` 책임은 호출자에게 있다.** 함수 이름에 `make_`라는 접두어가 붙은 것 자체가 "이건 새로 만들어 주는 거니까 네가 받아 가서 처리해라"라는 약속이다. 이 약속의 작명 규칙은 다음 장에서 좀 더 다듬는다.

## 세 번째 시나리오 — 배열의 감쇠

이제 옛 K&R 시절에도 헷갈렸을 법한 자리에 도착했다. 배열과 포인터의 관계다.

먼저 단순한 사실. **C 안에서 `arr[i]`는 컴파일러가 `*(arr + i)`로 풀어 본다.** 둘은 같은 식이다. `arr[2]`라고 쓰든 `*(arr + 2)`라고 쓰든, 심지어 `2[arr]`이라고 써도(별로 권하진 않지만) 같은 결과다. 덧셈은 가환적이니까. 그래서 우리는 배열이 사실 포인터의 친척이라는 사실을 인정해야 한다.

그런데 친척이지 같은 사람은 아니다. 그 차이가 함수에 배열을 넘길 때 드러난다.

```c
static void show_size(int arr[]) {
    printf("inside show_size, sizeof(arr) = %zu (포인터 크기)\n",
           sizeof arr);
}

static void array_decay(void) {
    int xs[] = { 1, 2, 3, 4, 5 };
    size_t n = sizeof xs / sizeof xs[0];

    printf("in main, sizeof(xs) = %zu, n = %zu\n", sizeof xs, n);
    show_size(xs);
}
```

이 코드를 빌드하면 컴파일러가 친절하게 경고를 띄운다.

```
warning: sizeof on array function parameter will return size of 'int *'
         instead of 'int[]' [-Wsizeof-array-argument]
```

이 경고를 보고 처음에 우리는 잠깐 당황한다. 분명히 `void show_size(int arr[])`라고 적었으니 `arr`는 배열이어야 하지 않나? 컴파일러가 친절하게 답을 알려준다 — **함수 매개변수에서 `int arr[]`는 사실상 `int *arr`다.** C가 배열을 함수에 넘길 때 그 자리에서 포인터로 "감쇠"시킨다. 이걸 영어로 array-to-pointer decay라고 한다.

실행 결과를 보면 명확해진다.

```
in main, sizeof(xs) = 20, n = 5
inside show_size, sizeof(arr) = 8 (포인터 크기)
```

`main`에서는 `sizeof(xs)`가 `20`(int 다섯 개 × 4바이트). 그러나 `show_size` 안에서는 `sizeof(arr)`가 `8`(64비트 머신의 포인터 크기). **배열은 함수 경계를 넘으면서 길이 정보를 잃는다.** 그래서 C 함수에 배열을 넘길 때는 **반드시 길이를 같이 넘긴다.** `void func(int *arr, size_t n)`이 표준 관용구다.

이 한 가지가 옛 C 시절에 우리가 많이 데었던 자리다. `strcpy(dst, src)`가 `dst`의 크기를 모르고, `memcpy(dst, src, n)`이 그래서 `n`을 받는 이유가 여기다. 길이를 잃었으니 길이를 따로 들고 다닐 수밖에 없다.

컴파일러의 `-Wsizeof-array-argument` 경고가 이 함정을 잡아 주는 게 얼마나 고마운지 모른다. 옛 시절에는 이 경고가 없어서 같은 코드가 조용히 어딘가에서 사고를 냈다. 모던 C 도구가 우리에게 준 작은 안전망이다.

## 네 번째 시나리오 — 함수 포인터

Java에서 람다, Python에서 일급 함수에 익숙한 사람이라면 함수 포인터는 그 친척으로 보면 된다. 함수도 메모리 어딘가에 코드로 적혀 있고, 그 코드의 시작 주소를 변수에 담을 수 있다.

```c
static int add(int a, int b) { return a + b; }
static int mul(int a, int b) { return a * b; }

typedef int (*binop_t)(int, int);

static int apply(binop_t op, int a, int b) { return op(a, b); }

static void func_pointer(void) {
    printf("apply(add, 3, 4) = %d\n", apply(add, 3, 4));   // 7
    printf("apply(mul, 3, 4) = %d\n", apply(mul, 3, 4));   // 12
}
```

`typedef int (*binop_t)(int, int)` 한 줄이 어지러우면 풀어 읽어 보자. 이 줄은 "**`binop_t`는 `int` 두 개를 받아 `int`를 돌려주는 함수의 포인터 타입이다**"라는 뜻이다. 별표가 있어서 포인터, 괄호 안에 매개변수가 두 개, 앞의 `int`가 반환 타입.

이 typedef가 없으면 `apply`의 시그니처를 적기가 끔찍해진다.

```c
int apply(int (*op)(int, int), int a, int b);
```

이게 더 자주 만나는 모양새다. 손에 익숙해질 때까지는 typedef로 한 번 이름을 붙여 두고 쓰는 편이 낫다.

호출 부분에서 `add`라는 이름 자체가 함수의 주소로 자연히 변환된다. `&add`라고 명시적으로 써도 같은 뜻이다(함수 이름은 배열 이름과 비슷하게, 식 안에서 자동으로 주소로 감쇠한다). `apply(add, 3, 4)` 안에서 `op`는 `add`의 주소를 받고, `op(a, b)`는 `(*op)(a, b)`와 같다 — 그 주소의 함수를 호출한다.

이 패턴이 그렇게 자주 쓰이느냐고? 아주 자주 쓰인다. `qsort`의 비교 함수, 이벤트 콜백, 인터럽트 핸들러 — C 코드에서 "동작을 인자로 넘긴다"는 거의 다 함수 포인터다. Java의 람다나 JS의 콜백 같은 자리에 이 친구가 있다. 다만 클로저(주변 변수 캡처)는 없다. C 함수 포인터는 그저 함수의 주소일 뿐, 주변 환경을 같이 들고 다니지 않는다. 그게 필요하면 별도 컨텍스트 포인터를 함께 넘긴다(`qsort_r`의 패턴이 그렇다).

이 책의 베어메탈 챕터에서 인터럽트 벡터 테이블을 다룰 때 함수 포인터가 본격 등장한다. 벡터 테이블은 한 번 보면 "이게 함수 포인터의 배열이구나"가 손에 들어온다. 지금 익혀 둔 이 모양새가 그때 그대로 쓰인다.

## 다섯 번째 시나리오 — `void *`로 일반화

마지막 시나리오는 살짝 위태롭다. `void *`다.

`void *`는 "어떤 타입을 가리키는지 모르는 포인터"다. 일반 컨테이너, 일반 알고리즘, 메모리 블록 복사 같은 일에 쓴다. 어떤 포인터로든 변환되고, 어떤 포인터로부터 받을 수 있다 — 한 가지 제약은 **`void *`를 직접 역참조할 수는 없다**는 점이다. 타입을 모르면 한 셀이 몇 바이트인지도 모르니까.

```c
static void copy_blob(void *dst, const void *src, size_t n) {
    unsigned char *d = dst;
    const unsigned char *s = src;
    for (size_t i = 0; i < n; ++i) d[i] = s[i];
}

static void void_pointer(void) {
    int from = 0xC0FFEE;
    int to   = 0;
    copy_blob(&to, &from, sizeof to);
    printf("copy_blob 후 to = 0x%X\n", to);  // 0xC0FFEE
}
```

이 `copy_blob`은 사실상 `memcpy`의 단순 흉내다. 두 가지를 짚자.

첫째, `void *dst`를 `unsigned char *d`로 받아 옮긴다. **메모리를 바이트 단위로 만질 때는 항상 `unsigned char`**다. `char`는 부호가 처리계마다 다르고, `signed char`는 비트 패턴을 그대로 보존하지 않을 수 있다. `unsigned char`만이 메모리의 비트를 그대로 비추는 안전한 시야다.

둘째, `const void *src`의 `const`는 "이 자리는 읽기만 하고 쓰지 않을 것"이라는 약속이다. 그러면 호출자가 안심하고 const 데이터를 넘길 수 있다. 우리도 함수 본문에서 `s[i] = ...`를 쓸 수 없다 — 컴파일러가 막는다.

`void *`의 진짜 힘은 일반 컨테이너에서 빛난다. 해시맵 한 구현이 키와 값으로 `void *`를 받으면, 같은 코드로 정수 맵, 문자열 맵, 구조체 맵을 모두 만들 수 있다. 다만 타입 안전성을 잃는 대신이다. 잘못된 타입을 넘기면 컴파일러가 막아 주지 않고, 런타임에 메모리를 잘못 해석한다. 모던 C 코드는 이 약점을 `_Generic`(C11) 같은 도구로 보강하지만, 그건 다음 장의 일이다.

### `const`의 위치 함정 — 한 번에 짚고 가자

`const` 이야기가 나왔으니 잠깐 옆으로 새서, 옛 C 시절에 누구나 한 번씩 데었던 자리를 짚자. **`const`의 위치 함정**이다.

```c
const int *p1;        // p1이 가리키는 자리는 const. p1 자체는 옮길 수 있다.
int * const p2;       // p2가 가리키는 자리는 보통, p2 자체는 옮길 수 없다.
const int * const p3; // 둘 다 const. 자리도, 가리키는 곳도 못 바꾼다.
```

이걸 빨리 읽는 비결이 하나 있다. **`*`를 기준으로 좌우를 가른다.** `*`의 왼쪽에 `const`가 붙으면 "가리키는 자리"가 const다. `*`의 오른쪽에 `const`가 붙으면 "포인터 자체"가 const다.

그래서 `const int *p1`은 "가리키는 자리가 const인 포인터". `int * const p2`는 "고정된 포인터(이동 불가), 가리키는 자리는 보통".

함수 시그니처에서 어느 쪽을 자주 보느냐? 거의 다 `const int *p` 모양새다. 함수가 인자로 받은 데이터를 읽기만 할 거면 그 시그니처에 `const`를 박아 둔다. 호출자도 안심, 컴파일러도 그 약속이 깨지면 잡아낸다.

`int * const p`는 자주 안 쓴다. 보통 함수 안에서 한 포인터를 절대 다른 곳으로 옮기지 않겠다고 자기 자신과 약속할 때나 쓴다. 이 모양새를 자주 보는 자리가 하나 있긴 한데, 베어메탈 코드에서 MMIO 레지스터 주소를 잡을 때다. `volatile uint32_t * const UART_DR = (uint32_t *)0x09000000`처럼. 주소는 못 바꾸고, 그 자리는 휘발성이고. 이 책 12장에서 다시 만난다.

## 다섯 시나리오를 묶는 한 가지 그림

다섯 시나리오를 차례로 손으로 짚었다. 그 결과를 한 그림으로 요약해 두고 가자.

- **시나리오 1 — 주소 출력.** 변수는 메모리 셀의 별명이고, 그 셀에는 주소가 있다.
- **시나리오 2 — 포인터의 포인터.** 함수가 새 객체를 만들어 호출자에게 넘길 때 쓰는 한 가지 방법.
- **시나리오 3 — 배열의 감쇠.** 배열은 함수 경계를 넘으면서 길이 정보를 잃는다. 길이는 따로 들고 다니자.
- **시나리오 4 — 함수 포인터.** 함수도 주소가 있다. 그 주소를 변수에 담아 동작을 인자로 넘긴다.
- **시나리오 5 — `void *`.** 타입을 모르는 일반 포인터. 강력하지만 타입 안전성과 맞바꾼다.

이 다섯이 익으면 트리, 연결리스트, 해시맵, 콜백 시스템 — C 코드의 거의 모든 자료구조와 패턴이 자연스러워진다. 옛 K&R 시절의 흐릿한 기억이 다시 또렷해지는 자리가 정확히 여기다.

## 자주 만나는 함정

복귀하는 사람이 첫 몇 주에 가장 자주 만나는 포인터 함정을 짚어 두자. 미리 알면 한 번씩은 피해 갈 수 있다.

**초기화 안 된 포인터.** `int *p;` 한 줄을 적어 두고 곧장 `*p = 5;`를 쓰면 정말 끔찍한 일이 벌어진다. `p`는 메모리 어디를 가리킬지 모르는 쓰레기 값이다. 어딘가 다른 변수의 메모리를 덮어쓸 수도 있고, 운이 좋으면 segfault로 즉시 죽는다. **포인터 변수는 선언과 동시에 NULL이나 유효한 주소로 초기화하자.** `int *p = NULL;` 한 줄을 잊지 말자.

**해제 후 사용.** `free(p)` 다음에 `*p`를 쓰면 use-after-free다. 9장의 ASan이 잡아 주지만, 처음부터 짝짓는 습관을 들이는 편이 낫다. 흔히 쓰는 작은 트릭 — `free(p); p = NULL;`. NULL 역참조는 즉시 segfault라 적어도 침묵의 사고는 안 된다.

**`*p`와 `p[0]`의 혼동.** 둘은 같다. 단일 객체를 가리키는 포인터든 배열의 첫 원소를 가리키는 포인터든, 첫 원소를 읽으려면 둘 중 어느 쪽이든 좋다. 그러나 의미는 다르게 읽힌다 — `*p`는 "이 자리 하나", `p[0]`은 "배열의 첫 원소". 의도를 코드 모양새로 드러내자.

**NULL과 0의 미묘한 차이.** C에서 `NULL`은 보통 `(void *)0`으로 정의된다. 정수 0과 거의 같지만, `printf("%p", 0)`처럼 가변 인자로 그냥 0을 넘기면 컴파일러가 보장해 주지 못한다. 명시적으로 `(void *)0` 또는 `NULL`을 넘기자. C23부터는 `nullptr`이라는 새 키워드가 들어와서 이 미묘함이 조금 정리됐다.

**포인터 산술의 단위.** `p + 1`은 한 바이트 뒤가 아니라 **한 원소 뒤**다. `int *p`라면 `p + 1`은 4바이트 뒤, `double *p`라면 8바이트 뒤. C가 알아서 타입 크기만큼 곱해 준다. 이 자동 곱셈이 편하지만 가끔 함정이다 — `void *`는 크기를 모르니 산술이 정의되지 않는다(GCC는 확장으로 허용하지만 표준에선 UB다).

이 다섯 함정만 머리 한쪽에 둬도 첫 몇 주의 사고가 절반으로 준다. 나머지 절반은 6장에서 다룰 메모리 관리, 7장에서 다룰 UB가 책임진다.

## 마무리

이제 손에 두 가지가 들렸다. **포인터가 메모리 셀의 별명이라는 한 줄짜리 직관**, 그리고 **그 직관을 다섯 시나리오로 실연해 본 손가락의 기억**.

옛 K&R 시절의 흐릿한 기억이 다시 또렷해졌다면 절반은 성공이다. 그러나 아직 한 가지가 남았다. 포인터 끝에 매달린 그 메모리는 어디서 와서 어디로 가는가? 누가 사 와서, 누가 돌려주는가? **GC가 없는 세계에서 누수, 이중 해제, use-after-free는 어떻게 피하는가?**

다음 장이 그 자리다. 메모리는 직접 사 와서 직접 돌려준다 — 이 단순한 약속을 코드 수준에서 어떻게 표현하는지, 안전망은 어떻게 짜는지를 짚는다. 이 챕터의 다섯 시나리오가 손에 들어왔다면, 다음 챕터의 모든 코드가 익숙한 모양새로 펼쳐진다.
