# 2장. 표준의 미로 — C89에서 C23까지

새 C 프로젝트의 첫 줄을 두드린다고 상상해보자. 빈 디렉토리에 `main.c` 한 파일을 만들고, 그 옆에 `Makefile`을 만들기 시작한다. 첫 줄은 보통 이렇게 시작한다.

```make
CFLAGS = -Wall -Wextra -Wpedantic -std=???
```

여기서 손이 멈춘다. **`-std=` 뒤에 뭘 적어야 하지?** `c89`라고 적으면 옛날 책 그대로의 C가 된다. `c99`나 `c11`은 친숙하지만 모던하다고 부르긴 애매하다. `c17`은 안정적이라 들었고, `c23`은 가장 최신이지만 어딘가 위험해 보인다. `gnu23` 같은 변종도 있다는데 어떻게 다를까. 그렇다고 아예 안 적으면? 컴파일러가 기본값을 알아서 골라준다는데, 그 기본값을 신뢰해도 될까?

이 작은 망설임이 사실은 C 재시작자가 가장 먼저 만나는 진짜 결정이다. 한번 정해두면 프로젝트의 색깔이 바뀐다. 라이브러리 의존성도, 동료들과의 약속도, 컴파일러가 우리에게 알려주는 경고의 양도, 다 이 한 줄에서 결정된다. 살펴보자.

## 표준 타임라인 — 한 장의 지도부터

말로 다 풀기 전에 큰 그림부터 보자. C의 공식 표준은 그렇게 많지 않다. 다섯 개라고 외워둬도 큰 문제가 없다.

| 약칭 | 정식 명칭 | 발표 | 핵심 추가 |
|------|----------|------|----------|
| K&R C | 비공식 | 1978 | 언어 그 자체. 함수 선언이 인자 없는 K&R 스타일 |
| C89 / ANSI C | ANSI X3.159-1989 (= ISO C90) | 1989/1990 | 함수 프로토타입, `const`, `volatile`, `enum`, 표준 라이브러리 정비 |
| C99 | ISO/IEC 9899:1999 | 1999 | `long long`, `<stdbool.h>`, VLA, `//` 주석, `inline`, flexible array member |
| C11 | ISO/IEC 9899:2011 | 2011 | `_Generic`, `<threads.h>`, `<stdatomic.h>`, 익명 struct/union, 개선된 유니코드 |
| C17 / C18 | ISO/IEC 9899:2018 | 2018 | **신규 기능 없음.** C11의 결함 정정만 한 안정화 릴리스 |
| **C23** | ISO/IEC 9899:2024 | **2024-10** | `bool`/`true`/`false`/`nullptr`/`static_assert`가 키워드, `_BitInt(N)`, `typeof`, `auto`, `constexpr`, `[[attribute]]`, `<stdbit.h>`, `<stdckdint.h>`, `#embed`, `%b` printf |

이 표를 한 번 훑었다면 절반은 끝났다. 외울 필요는 없다. 다만 몇 가지 사실은 머리 어디엔가 박아두자.

첫째, **C89(= ISO C90)가 "ANSI C"라는 별명을 가진 진짜 표준의 출발점**이다. K&R 첫 판은 그 이전이라 표준이 아니라 사실상 표준이다. 오래된 책의 표지에 "ANSI C"라고 적혀 있다면 그건 C89/C90을 가리킨다. 둘이 아주 미세하게 다르긴 한데(편집상의 차이), 보통 같은 거라고 봐도 무방하다.

둘째, **C17은 새 기능이 없다.** 단지 C11의 결함을 모아 정정한 안정화 릴리스다. 그래서 많은 프로젝트가 "안정 + 충분히 모던"이라는 이유로 C17을 기본값으로 쓴다. 이 책도 호스트 챕터의 기본값을 C17로 잡는다. 합리적 선택지다.

셋째, **C23은 2024년 10월에 정식 발표됐다.** C99 이후 가장 큰 폭의 개정 중 하나로 평가된다. 그 변화가 무엇인지는 잠시 뒤에 절을 따로 두고 보자.

## C99 — 작지만 결정적인 한 발

C89에서 C99로 넘어오는 동안 무엇이 바뀌었는가. 큰 줄기는 네 가지다.

**1. `long long`이 들어왔다.** 64비트 정수가 정식 이름을 얻은 시점이다. 그 전에는 컴파일러마다 `__int64` 같은 비표준 확장을 썼다. 지금은 `<stdint.h>`의 `int64_t`/`uint64_t`를 쓰는 게 권장이지만, 그 출발점이 C99의 `long long`이라는 사실은 기억해둘 만하다.

**2. `<stdbool.h>`가 추가됐다.** C89에는 `bool` 타입이 없었다. `int`를 진위값처럼 쓰는 게 관행이었고, 0이 거짓이고 0이 아니면 참이었다. C99부터 `<stdbool.h>`를 인클루드하면 `bool`, `true`, `false`를 쓸 수 있게 됐다. 사실은 `_Bool`이라는 진짜 키워드를 매크로로 가린 것이지만, 사용자 입장에서는 자연스러워졌다.

**3. 가변 길이 배열(VLA)이 들어왔다.** `void foo(size_t n) { int arr[n]; ... }` 같은 코드가 합법이 됐다. 그런데 이게 곧 골치 아픈 손님이 됐다 — 스택을 동적으로 깎아 먹는다, MSVC는 지원하지 않는다, 임베디드에서는 위험하다는 등의 이유로. 결국 C11에서 VLA는 **옵셔널 기능으로 강등됐다.** "지원하지 않아도 표준 컴파일러로 인정한다"는 뜻이다. 그래서 모던 C에서는 VLA를 안 쓰는 게 권장이다. 알아두자.

**4. 표현이 풍부해졌다.** `//` 한 줄 주석이 합법이 됐고, `for (int i = 0; ...)`처럼 반복문 안에서 변수를 선언할 수 있게 됐다. `inline` 키워드가 들어왔고, struct 끝에 가변 길이 배열을 선언하는 flexible array member (`int data[];`)도 추가됐다. 복소수(`<complex.h>`)와 가변 인자 매크로(`__VA_ARGS__`)도 이 시기에 들어왔다.

C99까지 도착하면 우리가 "옛날 C"라고 떠올리는 모습과 모던 C의 모습이 한 번 갈라진다. 이 표준 하나로 표현의 자유가 꽤 넓어진 셈이다.

## C11 — 멀티스레드의 시대가 표준에 도착하다

C11이 가져온 변화는 한마디로 정리할 수 있다. **언어가 멀티스레드의 시대를 인정했다.** 그 전까지는 pthread(POSIX 스레드)나 Windows 스레드 API를 직접 다뤘다 — 즉 OS 종속이었다. C11이 다음 세 가지를 표준에 박았다.

- **`<threads.h>`** — `thrd_create`/`thrd_join`/`mtx_lock` 같은 표준 스레드 API. POSIX보다는 얇은 추상.
- **`<stdatomic.h>`** — `atomic_int`, `atomic_load`, `atomic_store`, 메모리 순서(`memory_order_seq_cst` 등). lock-free 자료구조를 만지는 사람들에게 결정적.
- **`thread_local`/`_Thread_local`** — 스레드 지역 저장소 키워드.

다만 솔직하게 한 가지 짚어두자. **`<threads.h>`는 시장에 늦게 도착했다.** macOS의 libSystem은 오래 지원하지 않았고, 여전히 pthread를 쓰는 코드가 압도적이다. 표준이 약속한 거랑 실제 현장에서 쓰는 거랑 거리가 있는 부분이다. `<stdatomic.h>`는 그래도 비교적 잘 정착했다.

C11이 추가한 다른 작은 것들도 짚어두자. `_Generic` 키워드가 들어와서 매크로로 흉내 내던 타입 분기를 언어 수준으로 올릴 수 있게 됐고, 익명 struct/union이 합법이 됐다(중첩된 struct에서 한 단계 .이름을 생략할 수 있게 됐다). `_Static_assert`가 들어와서 컴파일 타임에 단언이 가능해졌다. `aligned_alloc`/`aligned` 같은 정렬 관련 도구도 추가됐다.

C17은 앞서 말했듯이 신규 기능 없이 C11의 결함 정정만 한 릴리스다. 그래서 C11과 C17을 거의 같은 거라고 봐도 무방하다. 다만 컴파일러 입장에서는 C17이 "다듬어진 C11"이라 더 깔끔하다. 표준이 안정됐다는 인상을 주려고 C17을 기본값으로 다는 도구가 많다(예를 들어 CMake 같은 곳에서 `CMAKE_C_STANDARD 17`).

## C23 — 가장 큰 변화, 가장 큰 함정

이제 진짜 새로운 손님, C23이다. 2024년 10월 31일에 정식 발표됐고, 가장 큰 폭의 개정 중 하나로 평가된다. 어떤 게 들어왔는지 큰 줄기를 보자.

**키워드 승격.** 옛날에는 매크로나 헤더 의존이었던 것들이 정식 키워드가 됐다.

- `bool`, `true`, `false` — `<stdbool.h>` 없이도 쓸 수 있다.
- `nullptr` — 정수 0과 묵시 변환되지 않는 진짜 널 포인터 리터럴.
- `static_assert` — C11의 `_Static_assert`를 줄인 이름. 헤더 없이 그냥 쓸 수 있다.
- `thread_local` — `_Thread_local`의 축약.

**임의 폭 정수 `_BitInt(N)`.** 이게 임베디드 진영에서는 꽤 큰 사건이다.

```c
typedef _BitInt(8) byte_t;     // 정확히 8비트
typedef _BitInt(24) addr24_t;  // 정확히 24비트
typedef _BitInt(128) hash_t;   // 128비트
```

레지스터의 비트 필드를 정확한 폭으로 표현할 수 있다. 옛날에는 비트 필드를 `unsigned int : 24` 같은 식으로 struct 안에 박았지만, `_BitInt(N)`은 변수로 쓰는 정수다. 임베디드·암호 코드에서 큰 도움이 된다.

**타입 추론과 컴파일 타임 상수.**

- `auto x = some_func();` — C++스러운 타입 추론.
- `typeof(expr)` — 표현식의 타입을 추출. GCC 확장이었던 게 표준에 들어왔다.
- `constexpr int N = 42;` — 컴파일 타임 상수 정의.

**통일된 어트리뷰트 문법.** 이전까지 `__attribute__((nodiscard))`(GCC)나 `__declspec(nodiscard)`(MSVC)로 갈렸던 게 `[[nodiscard]]` 한 문법으로 정리됐다.

```c
[[nodiscard]] int must_check(void);
[[deprecated("use foo2 instead")]] void foo(void);
```

**새 표준 헤더 두 개.**

- `<stdbit.h>` — 비트 카운트, 엔디언 변환 등. `stdc_count_ones`, `stdc_bit_width`, `stdc_leading_zeros` 같은 함수. 비트 연산을 손으로 짜지 않아도 된다.
- `<stdckdint.h>` — 체크된 정수 산술. `ckd_add`, `ckd_sub`, `ckd_mul`이 오버플로 여부를 bool로 돌려준다. signed overflow는 UB이기 때문에, 이게 표준에 들어온 건 사실상 안전 보강이다.

**문법 사소한 보강.**

- `%b`/`%B` printf — 이진 출력. 이전엔 직접 비트 시프트로 만들었던 작업.
- `#embed` — 바이너리 파일을 통째로 배열 리터럴로 가져오기. 폰트 파일·아이콘 등을 코드에 임베드할 때 편하다.
- `__VA_OPT__` — 가변 인자 매크로의 빈 인자 처리.
- 16진 부동소수 리터럴, 디지트 구분자 `'`(가독성용), 이진 리터럴 `0b101010` 등.

목록이 길게 보이지만 한 가지 핵심만 잡고 가자. **C23은 큰 폭의 갱신이고, 그 안에 진짜 쓸모 있는 도구들이 들어 있다.** `nullptr`, `[[nodiscard]]`, `<stdckdint.h>` 셋만 해도 안전한 C를 짜는 데 큰 도움이 된다. 새 프로젝트라면 시도해 볼 가치가 충분하다.

다만 — 모든 변화에는 함정이 따라온다. 이게 다음 절의 주제다.

## C23의 함정 — 조용히 깨지는 옛 코드

C23의 변화 중에 한두 개는 옛 코드를 조용히 부순다. **경고도 없이 동작이 달라지는** 종류라 가장 무섭다. 두 장면만 보자.

### `realloc(ptr, 0)`이 UB가 됐다

C89부터 C11까지 폭넓게 통하던 관용구가 있었다.

```c
ptr = realloc(ptr, 0);   // 옛날 의미: free(ptr)과 같다, 또는 작은 블록을 돌려준다
```

`realloc(ptr, 0)`이 사실상 `free(ptr)`과 같다고 가정한 코드가 꽤 있다. 라이브러리 안에서 메모리 풀을 비우는 패턴, 동적 배열을 0 크기로 줄이는 패턴 등이다. **C23부터 이 동작이 정의되지 않은 동작(UB)이 됐다.** 컴파일러가 무슨 짓을 해도 표준이 허용한다. 운이 좋으면 그대로 free처럼 동작할 거고, 운이 나쁘면 옆 메모리가 깨지거나 컴파일러가 그 코드 경로를 통째로 지워버린다.

ACM Queue의 "Catch-23" 칼럼이 이걸 두고 "표준 위원회가 오래된 코드의 발치를 슬그머니 빼는 결정"이라며 정면으로 비판했다. 그 칼럼의 입장이 옳다 그르다를 떠나서, **옛 코드를 C23으로 마이그레이션할 때 가장 자주 부딪힐 문제 중 하나**라는 건 사실이다. 7장에서 UB를 본격적으로 다룰 때 한 번 더 이야기하자. 지금은 머리 어디엔가 박아두자.

### 빈 괄호 함수 선언의 의미가 바뀌었다

1장에서 짧게 짚었던 그 변화다. K&R 시절에 흔했던 한 줄.

```c
void f();
```

옛 의미는 "인자 정보가 없다, 임의 인자를 받을 수 있다"였다. 그래서 `f(1, 2, 3)`이 통과했다(실제 `f`의 시그니처와 안 맞으면 런타임에 망가졌지만). 인자 없음을 명시하려면 `void f(void)`라고 적어야 했다.

**C23부터 `void f()`는 `void f(void)`와 같다.** 인자 없음이라는 뜻이다. 자기 코드에 `void` 명시 없는 빈 괄호 선언이 잔뜩 있다면, 모던 컴파일러에 통과시킬 때 경고 무더기를 만나게 된다. 한 번 정리해두면 끝이지만, 정리하기 전에는 좀 번거롭다.

### 함정에 대처하는 자세

함정 두 개를 봤지만 너무 겁먹지는 말자. 우리에겐 친구가 있다. 컴파일러의 `-Wall -Wextra -Wpedantic` 트리오다.

```bash
clang -std=c23 -Wall -Wextra -Wpedantic main.c
```

이 트리오를 켜두면 함정이 도착하기 전에 컴파일러가 미리 손을 들어 알려준다. 빈 괄호 함수 선언이 보이면 잔소리를 한다. `realloc(ptr, 0)`이 보이면 도구에 따라 잡힌다. 그러니 **표준을 올리면 경고 옵션도 같이 올리자.** 이게 모던 C의 머슬 메모리다.

## `-std=` 한 줄을 어떻게 정할 것인가

이제 본론이다. 우리 프로젝트의 `-std=`에 뭘 적어야 하는가? 살펴보자.

### 합리적 기본값

먼저 가장 폭넓게 통하는 답부터 적어두자.

> 새 프로젝트라면 **`-std=c23 -Wall -Wextra -Wpedantic`** 또는 **`-std=c17 -Wall -Wextra -Wpedantic`** 으로 시작하는 편이 낫다.

C17이 더 안전하고, C23이 더 모던하다. 둘 사이의 거리는 생각보다 좁다 — C17은 C11의 정정판이라 C23만큼 도구 갱신이 빠르지 않을 뿐이다. 어느 쪽을 골라도 큰 차이는 없다. 다만 한 가지 기준이 도움이 된다.

**의존하는 외부 라이브러리가 어디까지 따라잡았는지를 보자.** Apple Clang 16+, Homebrew LLVM 19+, GCC 14+ 정도면 C23의 상당 부분이 들어와 있다. 그러나 libc(GNU libc, musl, Apple libSystem 등)는 컴파일러보다 늦게 따라잡는 게 일반적이다. `<stdbit.h>`나 `<stdckdint.h>` 같은 새 헤더가 어디서는 동작하고 어디서는 안 될 수 있다. 그래서 **이식 라이브러리는 여전히 C11/C17 가정**으로 짜는 게 현실적 절충이다.

다음 표를 머리 어디엔가 박아두자.

| 상황 | 권장 `-std=` |
|------|-------------|
| 새 애플리케이션 / 자기 도구 | `-std=c23`. 안전 보강(`nullptr`, `<stdckdint.h>`)을 적극 쓰자 |
| 다른 곳에서 갖다 쓸 라이브러리 | `-std=c17`. 컴파일러·플랫폼 호환성 우선 |
| 옛 코드 마이그레이션 | 일단 `-std=c17`로 빌드 통과시키고, 점진적으로 `-std=c23`로 올리기 |
| Linux 커널 같은 베어메탈 / 시스템 코드 | `-std=gnu17` 또는 `-std=gnu23`. GNU 확장 필요 |
| MSVC를 지원해야 한다 | `-std=c11`. MSVC의 C 지원이 보수적 |

### MSVC라는 변수

한 가지 더 짚어두자. **MSVC는 C에 보수적이다.** C99 일부도 늦게 지원했고, C11/C17의 일부만 들어왔으며, C23은 부분 지원이다. 그래서 윈도우와 macOS·리눅스 양쪽을 다 지원해야 하는 코드라면 `-std=c11` 정도가 안전한 절충선이다. 다행히 이 책은 Apple Silicon Mac과 ARM 베어메탈을 다루므로, Clang과 GCC 둘만 신경 쓰면 된다. MSVC 걱정은 미뤄두자.

### `gnu*` 변종은 무엇인가

표준 이름에 `gnu`가 붙은 변종이 있다.

- `-std=gnu89`, `-std=gnu99`, `-std=gnu11`, `-std=gnu17`, `-std=gnu23`

이게 뭘 뜻하느냐. **해당 표준 + GNU 확장**이다. GNU 확장이라는 게 무엇인가? GCC와 Clang이 표준에 더해 추가로 지원하는 비표준 기능 묶음이다. 대표적인 게 `__attribute__((aligned(16)))`, 명령 표현식(`({ ... })`), 0 길이 배열, `typeof`(C23 이전), 인라인 어셈블리 `__asm__` 같은 것들이다.

표준만 쓰겠다면 `-std=c17`이 깔끔하다. 그런데 **베어메탈 코드에서는 사실상 `gnu*`가 표준**이다. 왜 그러느냐. 베어메탈에서는 인라인 어셈블리, 링커 스크립트와 연동되는 `__attribute__((section("...")))`, `__attribute__((aligned))`, `__attribute__((weak))` 같은 GNU 확장이 거의 필수다. 표준 C만으로는 부팅 코드를 짤 수 없다. 그래서 이 책의 베어메탈 챕터(11~15장)는 `gnu17` 또는 `gnu23`을 쓴다. 그게 표준이 아니라 함정인 게 아니라 — **베어메탈 영역의 사실상 표준이라는 점이 표준이 정한 한계**라고 받아들이자.

## 표준 매트릭스로 손에 익히기 — 같은 코드, 다섯 시대

여기까지 표준 이야기가 길었다. 손을 한 번 더럽혀보자. 이번 장의 예제 디렉토리(`example/ch02-std-matrix/`)에 같은 코드를 다섯 표준으로 빌드하는 매트릭스가 들어 있다. `std_demo.c`라는 작은 파일 하나, 그리고 `c23_only.c`라는 C23 전용 파일 하나가 있다.

먼저 `std_demo.c`의 핵심은 짧다.

```c
#include <stdio.h>

static int square(int x) { return x * x; }

int main(void) {
    int total = 0;
    int i;
    for (i = 1; i <= 5; i++) total += square(i);
    printf("sum of squares 1..5 = %d\n", total);
    printf("compiled with __STDC_VERSION__ = %ld\n",
#ifdef __STDC_VERSION__
           (long)__STDC_VERSION__
#else
           0L
#endif
    );
    return 0;
}
```

별것 없다. 1부터 5까지의 제곱 합을 출력하고, `__STDC_VERSION__` 매크로 값을 함께 찍는다. 핵심은 **이 코드가 C89, C99, C11, C17, C23 다섯 표준 모두에서 깔끔하게 컴파일된다**는 점이다. `make matrix` 한 줄이면 다섯 번 빌드되고 다섯 번 실행된다.

```
--- -std=c89 빌드 ---
sum of squares 1..5 = 55
compiled with __STDC_VERSION__ = 0
--- -std=c99 빌드 ---
sum of squares 1..5 = 55
compiled with __STDC_VERSION__ = 199901
--- -std=c11 빌드 ---
sum of squares 1..5 = 55
compiled with __STDC_VERSION__ = 201112
--- -std=c17 빌드 ---
sum of squares 1..5 = 55
compiled with __STDC_VERSION__ = 201710
--- -std=c23 빌드 ---
sum of squares 1..5 = 55
compiled with __STDC_VERSION__ = 202311
```

`__STDC_VERSION__`의 값이 다섯 다른 숫자로 찍힌다. C89에는 이 매크로가 정의되지 않을 수 있어서 0이 찍힌다. 실용적인 가치는 없는 데모지만 — 그 숫자 다섯 개가 컴파일러가 우리에게 알려주는 "지금 어느 시대 C로 짓고 있다"는 자기 신고라는 점, 그 점 하나를 손으로 만지는 게 목적이다.

반대편에 `c23_only.c`가 있다. 이 파일은 C23 전용 기능을 두 가지 쓴다.

```c
typedef _BitInt(8) byte_t;

static void greet(const char *who) {
    if (who == nullptr) {
        printf("(이름 없는 손님)\n");
        return;
    }
    printf("hello, %s\n", who);
}
```

`_BitInt(8)`이라는 임의 폭 정수와 `nullptr`이라는 키워드. 둘 다 C23부터다. `make c23-only`로 빌드하면 깔끔하게 통과한다.

```
hello, c23
(이름 없는 손님)
_BitInt(8) 최댓값을 담은 변수: 127
```

이 코드를 일부러 C17로 빌드해보자. `make c23-only-as-c17`은 그 시도를 자동화해준다.

```
c23_only.c:19:16: error: use of undeclared identifier 'nullptr'
   19 |     if (who == nullptr) {
      |                ^~~~~~~
1 warning and 2 errors generated.
(예상대로 컴파일이 거부되었다. 이게 표준 차이의 증거다.)
```

`nullptr`이 C17에는 키워드가 아니기 때문에, 컴파일러가 "그런 식별자 못 본다"고 정면으로 거부한다. **이게 표준 차이의 손에 잡히는 증거다.** `-std=`에 뭘 적느냐가 빈말이 아니라는 사실이 여기서 분명해진다.

이 매트릭스를 한 번 직접 두드려보고 다음 장으로 가자. `make matrix`로 다섯 시대를 가로질러보고, `make c23-only-as-c17`로 일부러 실패를 한번 만나보자. 표준이라는 게 우리 손에 닿는 실체라는 감각이 그 짧은 의식에서 생긴다.

## 마무리

이 장에서는 표준의 미로를 한 장의 지도로 펼쳐 봤다. K&R에서 C23까지 큰 줄기를 짚었고, C99·C11·C23의 핵심 추가 사항을 정리했고, C23의 함정 두 곳(빈 괄호 함수 선언, `realloc(ptr, 0)`)도 미리 봤다. 마지막에는 `-std=` 한 줄을 어떻게 정할지를 함께 고민했다.

**기억해두자.**

- **새 프로젝트는 `-std=c23 -Wall -Wextra -Wpedantic`으로 시작하는 편이 낫다.** 안전 보강(`nullptr`, `<stdckdint.h>`)이 손에 들어온다.
- **이식 라이브러리는 `-std=c17`로 짜자.** 컴파일러·플랫폼 호환성을 우선 챙길 때.
- **베어메탈은 `-std=gnu17` 또는 `-std=gnu23`이 사실상 표준이다.** 인라인 어셈블리와 `__attribute__`이 필수라서 그렇다.
- **C23의 함정 두 곳을 잊지 말자** — 빈 괄호 함수 선언의 의미 변화, `realloc(ptr, 0)`의 UB화. 옛 코드를 마이그레이션할 때 가장 자주 부딪힌다.

표준의 지도가 손에 들어왔다. 그렇다면 이제 그 표준을 빌드해줄 컴파일러는 어디서 오는가? Apple Silicon Mac에 어떤 도구를 깔고, 무엇을 PATH에 올려야 다음 장의 모든 예제가 동작하는가? 다음 장은 Day 0 — 환경 셋업의 한 시간이다. 손에 든 표준을 빌드 가능한 형태로 옮겨놓자.
