# 13장. 부트 흐름 해부 — `boot.S` 한 줄, 링커 스크립트 한 섹션

QEMU 시리얼 창에 `hello, aarch64`가 처음으로 떠오르던 순간을 떠올려보자. 손가락이 키보드 위에서 잠시 멈추고, 작은 글자 열네 개에 자기도 모르게 시선이 박혔던 그 순간 말이다. 그런데 며칠 지나서 그 코드를 다시 펼치면 묘하게 찜찜한 감정이 따라온다. 분명 내가 짠 코드인데, 한 줄 한 줄 짚어보라고 하면 어디부터 입을 열어야 할지 막막하다. `mrs   x0, mpidr_el1`은 도대체 무엇을 읽는 명령이며, 왜 그 다음에 `and x0, x0, #3`이 따라붙는가. `ldr x1, =stack_top`의 `=` 기호는 어떤 의미인가. 링커 스크립트에서 `KEEP(*(.text.boot))`이라는 표현은 왜 굳이 `KEEP`을 두르는가. 이런 질문에 자신 있게 답하지 못한 채로 다음 장의 보드 부팅으로 넘어간다면, 그 다음에 만나게 될 디버깅의 밤이 두 배로 길어진다.

그래서 한번 멈춰 서자. 12장이 "네 파일이 어떻게 협력해서 첫 글자에 도달하는가" 라는 동작의 그림이었다면, 이번 장은 그 그림 위에 한 줄씩, 한 섹션씩 돋보기를 들이대는 시간이다. **같은 코드 베이스를 두 번 보지만, 시선의 깊이가 다르다.** 12장이 동작이라면 13장은 이유다. 그리고 이번 장의 끝에서 우리는 부팅 코드의 각 줄을 누가 물어도 한 문장으로 설명할 수 있게 되어 있을 것이다. 그 자신감이 14장에서 진짜 라즈베리 파이 보드를 만질 때 결정적인 무기가 된다.

## 부팅 직후의 CPU — 모든 것이 미정의 상태다

자, 호스트 OS의 안락한 세계를 떠나서 베어메탈로 발을 디딘 직후의 풍경을 그려보자. QEMU에 `kernel.elf`를 던지고 가상 시간이 0에서 1로 넘어간 그 순간, CPU 위에 무엇이 놓여 있을까. Java 출신이라면 "JVM이 알아서 초기화해주는 객체 같은 것"을 떠올리고 싶겠지만, 베어메탈에는 그런 친절이 없다. **레지스터 31개가 모두 의미 없는 값으로 채워져 있고**, 스택 포인터(`SP`)가 가리키는 곳이 어딘지 아무도 보장해주지 않으며, 캐시는 비활성화 상태이고, MMU는 꺼져 있어 가상 주소와 물리 주소가 일대일로 대응한다. `.bss` 섹션의 전역 변수들이 0으로 초기화되어 있을 거라는 호스트 시절의 약속도 여기서는 자동으로 지켜지지 않는다.

그렇다면 한 가지 질문을 해보자. AArch64 CPU가 켜진 직후의 **예외 레벨(Exception Level, EL)** 은 무엇일까. AArch64는 EL0(사용자)·EL1(커널)·EL2(하이퍼바이저)·EL3(시큐어 모니터)이라는 네 단계의 권한 모드를 가지고 있고, 부팅 직후에 어느 레벨로 들어가는지는 펌웨어와 부트로더의 약속에 따라 달라진다. QEMU `virt` 보드는 기본적으로 **EL2**로 진입한다. 라즈베리 파이 4의 GPU 펌웨어 역시 보통 EL2로 떨궈준다. 우리가 만들 작은 커널은 EL1에서 도는 것이 일반적이므로, 엄밀히는 EL2에서 EL1으로 내려오는 전환 코드가 필요하다. 다만 12장의 가장 단순한 골격에서는 이 전환을 생략하고 EL2에서 곧장 실행해도 `hello, aarch64`를 찍는 데까지는 무리가 없다. 이번 장의 뒷부분에서 EL 전환의 의미를 다시 들여다보자.

그리고 또 하나 기억해두자. 베어메탈 부팅 직후에는 **`printf`도, `malloc`도, `errno`도 없다.** 11장에서 충분히 떨었던 그 사실이 13장에서 다시 살아 돌아온다. 그런데 더 무서운 것은, **C 언어의 일반 함수조차 아직 호출할 수 없다**는 사실이다. 함수 호출은 스택을 필요로 하고, 스택은 `SP` 레지스터가 가리키는 메모리 영역인데, `SP`가 어디를 가리키는지 우리가 직접 정해주기 전까지는 함수 호출 자체가 위험하다. 이게 왜 `boot.S`라는 어셈블리 파일이 반드시 먼저 필요한지에 대한 답이다 — **C 함수를 안전하게 호출할 수 있는 최소 환경을 어셈블리로 직접 만들어 줘야 한다.**

## `boot.S`를 한 줄씩 — 일곱 줄에 담긴 일곱 가지 약속

12장에서 본 `boot.S`를 다시 한번 펼쳐보자. 한 화면에 들어오는 짧은 코드인데, 한 줄씩 의미를 풀어보면 일곱 개의 약속이 차곡차곡 쌓여 있다.

```asm
.section ".text.boot"
.globl _start
_start:
    mrs   x0, mpidr_el1     // 1) CPU ID를 읽는다
    and   x0, x0, #3        // 2) 하위 2비트만 남긴다
    cbnz  x0, halt          // 3) 0이 아니면 멈춰라
    ldr   x1, =stack_top    // 4) 스택 꼭대기 주소를 가져온다
    mov   sp, x1            // 5) SP 레지스터에 박는다
    bl    kernel_main       // 6) C 함수를 호출한다
halt:
    wfe
    b     halt              // 7) 보조 코어들은 여기서 잠든다
```

먼저 시작점이다. `.section ".text.boot"` 지시문이 의미하는 바는 "이어지는 코드를 `.text.boot`이라는 이름의 섹션에 모아둬라" 라는 어셈블러에 대한 약속이다. 평범한 `.text` 섹션과 달리 이 섹션은 링커 스크립트에서 **반드시 ELF의 맨 앞**에 배치된다. 그 이유는 단순하다 — QEMU나 GPU 펌웨어가 ELF를 로드한 직후 가장 먼저 점프하는 곳이 ELF의 시작 주소이고, 그곳에 우리의 진입점 `_start`가 있어야 부팅이 성공한다. 이 약속은 잠시 후 링커 스크립트 절에서 자세히 다시 살펴보자.

다음 줄인 `.globl _start`는 "`_start`라는 심볼을 외부에서도 보이게 만들겠다"는 선언이다. 링커가 이 심볼을 ENTRY 지점으로 인식하려면 전역으로 노출되어 있어야 한다.

이제 본격적인 일곱 가지 약속이다. 첫 번째 줄, `mrs x0, mpidr_el1`. `mrs`는 "**Move from System Register**"의 약자로, 시스템 레지스터의 값을 일반 레지스터로 복사하는 명령이다. `mpidr_el1`(Multiprocessor Affinity Register)은 현재 코드를 실행 중인 CPU 코어의 정체성을 담은 시스템 레지스터로, 하위 비트들에 코어 번호가 들어 있다. 결과적으로 이 한 줄은 "내가 몇 번 코어인지 알아내자"는 의미다.

두 번째 줄, `and x0, x0, #3`. 비트 마스킹이다. `0b11`(십진수 3)과 `AND` 연산을 해서 **하위 2비트만 남긴다**. 라즈베리 파이 4나 QEMU virt 보드처럼 코어가 4개인 시스템에서는 코어 번호가 0·1·2·3 중 하나이므로 2비트면 충분하다. 7장에서 익혔던 비트 마스크 관용구(`val & 0x3`)가 여기서 처음 실전 무대에 오른다. 7장에서 미리 손을 풀어두지 않았다면 이 한 줄이 무엇을 하는지 짐작하는 데도 시간이 걸렸을 것이다. 미리 다리를 놓아둔 셈이다.

세 번째 줄, `cbnz x0, halt`. `cbnz`는 "**Compare and Branch if Non-Zero**"의 약자로, `x0`가 0이 아니면 `halt` 레이블로 점프하라는 명령이다. 직역하면 "내가 0번 코어가 아니라면 잠들어라". 이 한 줄이 바로 **primary core(0번 코어)만 부팅 코드를 통과시키고, 보조 코어들은 무한 루프에 가두는** 결정적 약속이다. 만약 이 줄이 없다면 네 개의 코어가 동시에 `kernel_main`을 호출하고, 같은 `SP`를 공유하다가 즉시 스택이 뒤엉켜 망가진다. 멀티코어 베어메탈의 첫걸음은 이렇게 "내가 누구인지 보고 통과 여부를 결정한다"는 한 줄에서 시작한다.

네 번째와 다섯 번째 줄을 묶어서 보자. `ldr x1, =stack_top`과 `mov sp, x1`. `ldr`은 메모리에서 값을 읽어오는 명령인데, `=symbol` 형태를 쓰면 어셈블러가 그것을 "이 심볼의 주소"로 풀어서 처리한다. 즉 `stack_top` 심볼의 **주소**를 `x1`에 넣는 것이다. 그 다음 줄에서 `mov sp, x1`으로 그 주소를 `SP` 레지스터에 옮긴다. **이 두 줄이 끝나는 순간, 우리는 처음으로 안전한 스택을 손에 넣는다.** 이제부터 C 함수를 호출해도 된다.

여섯 번째 줄, `bl kernel_main`. `bl`은 "**Branch with Link**"의 약자로, 점프하기 전에 현재 PC(Program Counter)의 다음 명령 주소를 `X30`(link register, 줄여서 `LR`)에 저장한 뒤 분기한다. 8장의 ABI 절에서 미리 손에 익혀둔 그 약속이다 — AArch64 PCS에서 리턴 주소는 X30에 보관된다. 그래서 `kernel_main` 내부에서 `ret` 명령으로 돌아올 수 있게 된다. 그런데 잠깐, 우리 `kernel_main`은 사실 `for (;;) { }` 무한 루프로 끝나서 절대 돌아오지 않는다. 그래도 컴파일러는 함수 호출 규약을 지키느라 `LR`을 저장한다. 형식과 실제가 어긋나는 미세한 지점이지만, 베어메탈에서는 이런 어긋남이 일상이다.

마지막 일곱 번째, `halt:` 레이블 아래의 `wfe`와 `b halt`. `wfe`는 "**Wait For Event**"로, 외부 이벤트가 올 때까지 코어를 저전력 상태로 잠재우는 명령이다. 이벤트가 와서 깨어나도 그 다음 줄 `b halt`가 다시 `halt` 레이블로 분기시키므로 영원히 갇힌다. 사실상의 무한 잠재우기다. 보조 코어들이 부팅 시점에 잘못된 곳으로 흘러가서 메모리를 망가뜨리지 않도록 안전하게 보관하는 자리다.

일곱 줄에 담긴 일곱 약속. 이걸 한 호흡으로 외워보자 — **누구냐 → 통과냐 → 스택을 깔자 → C로 가자 → 나머지는 잠재우자.** 부트로더 코드는 이런 약속의 짧은 시(詩)에 가깝다.

### `mpidr_el1`을 조금 더 깊이

`mpidr_el1`을 단순히 "코어 번호 레지스터"라고만 부르고 지나가긴 아쉽다. 이 레지스터의 비트 구조를 한 번만 들여다보자. AArch64 사양에서 64비트 레지스터인데, 우리가 관심 있는 부분은 하위 32비트, 특히 그 안의 **Aff0**(Affinity Level 0) 8비트다. 멀티프로세서 시스템에서는 코어가 클러스터로 묶이는데, Aff0이 같은 클러스터 안의 코어 번호, Aff1이 클러스터 번호, Aff2/Aff3이 더 상위 그룹을 가리킨다. 라즈베리 파이 4의 BCM2711처럼 코어 4개가 한 클러스터에 있는 단순한 SoC에서는 Aff0의 하위 2비트만 보면 코어 번호가 나온다. 그래서 `and x0, x0, #3`(즉 `0b11`과 AND)으로 충분한 것이다.

| 비트 범위 | 의미 |
|----------|------|
| `[7:0]`   | Aff0 — 같은 클러스터 내 코어 번호 |
| `[15:8]`  | Aff1 — 클러스터 번호 |
| `[23:16]` | Aff2 — 상위 그룹 |
| `[30]`    | U bit — Uniprocessor 시스템 여부 |
| `[31]`    | RES1 — 항상 1 |
| `[39:32]` | Aff3 — 최상위 그룹 (큰 시스템에서) |

큰 서버급 SoC(예: 64코어 Ampere Altra)에서는 Aff1·Aff2까지 봐야 정확한 코어 식별이 가능하지만, 라즈베리 파이 4 같은 4코어 단일 클러스터 시스템에서는 Aff0의 2비트만 봐도 충분하다는 점을 기억해두자. 7장에서 비트 마스크를 익혀둔 손이 여기서 다시 한번 쓸모를 발한다. 만약 멀티 클러스터 시스템으로 나아간다면 마스크가 `0xFF`로 바뀌고, 추가로 Aff1도 검사해야 한다.

## `.bss` 클리어 — 호스트 시절의 무료 점심이 사라지다

12장의 가장 단순한 골격에서는 `.bss` 클리어 코드를 생략했다. 왜냐하면 우리의 작은 `kernel.c`는 전역 변수를 거의 쓰지 않았고, 0으로 초기화된 전역 변수에 의존하는 코드 경로가 없었기 때문이다. 그러나 이건 운이 좋은 경우다. 일반적인 C 프로그램에서 다음 코드를 보자.

```c
static int counter;          // .bss에 들어간다 — 호스트에서 자동으로 0
static char buffer[1024];    // 마찬가지

void increment(void) {
    counter++;
}
```

호스트 환경에서는 OS의 ELF 로더가 `.bss` 섹션의 가상 주소 영역에 **0으로 채워진 페이지**를 매핑해준다. 그래서 C 표준이 "초기화 없이 정의된 전역 변수는 0으로 초기화된 것처럼 동작한다"고 약속할 수 있다. 그런데 베어메탈에서는 그 친절한 로더가 없다. **GPU 펌웨어나 QEMU는 ELF의 `.bss` 섹션을 메모리에 적재해주지 않는다.** 정확히 말하면, `.bss`는 ELF 파일 안에 실제 바이트로 들어 있지 않다(공간만 예약된 BSS = Block Started by Symbol). 따라서 그 영역의 메모리는 **부팅 직후 무엇이 들어 있을지 아무도 모른다**. 운이 좋으면 0일 수도 있고, 운이 나쁘면 이전 부팅 잔재가 남아 있을 수도 있다.

그렇다면 어떻게 해야 할까. 답은 명확하다. **`boot.S`에서 `kernel_main`을 호출하기 직전에 `.bss` 영역을 0으로 직접 채워줘야 한다.** 13장의 두 번째 스테이지에서 추가하는 코드가 바로 그것이다.

```asm
    // .bss 0으로 클리어
    ldr   x1, =__bss_start
    ldr   x2, =__bss_end
clear_bss:
    cmp   x1, x2
    b.ge  bss_done
    str   xzr, [x1], #8         // *x1 = 0; x1 += 8;
    b     clear_bss
bss_done:
    bl    kernel_main
```

한 줄씩 보자. `ldr x1, =__bss_start`와 `ldr x2, =__bss_end`는 BSS 섹션의 시작·끝 주소를 각각 `x1`, `x2`에 담는다. 이 두 심볼은 우리가 짠 코드에는 없다 — **링커가 만들어주는 심볼**이다(잠시 후 링커 스크립트 절에서 정확히 어떻게 만드는지 본다). `cmp x1, x2`로 둘을 비교하고, `b.ge bss_done`은 "`x1 >= x2`(부호 있음)면 끝났으니 빠져나가라"는 조건 분기다. `str xzr, [x1], #8`은 한꺼번에 두 가지 일을 한다 — `xzr`(항상 0인 zero register)을 `x1`이 가리키는 주소에 8바이트 단위로 쓰고, **쓴 다음에 `x1`을 8만큼 증가**시킨다. ARM 어셈블리의 post-increment 주소 모드라는 관용구다. 이 한 줄이 곧 C의 `*((uint64_t*)p)++ = 0;`에 해당한다.

8바이트씩 끊어 클리어하는 이유는 단순한 성능 — AArch64는 64비트 머신이므로 8바이트 정렬된 메모리 접근이 자연스럽고 빠르다. 만약 BSS 섹션의 크기가 8의 배수가 아니라면 어떻게 될까. 좋은 질문이다. 그래서 링커 스크립트에서 `.bss` 섹션 끝에 `. = ALIGN(8);` 또는 `. = ALIGN(16);`을 박아두는 관습이 있다 — 8이나 16의 배수로 패딩을 강제해서 8바이트 단위 클리어가 안전하게 끝나도록 만든다. 패딩의 몇 바이트가 0이 되더라도 어차피 그 자리는 우리가 쓰지 않을 영역이라 문제가 없다.

### `.data`는 어떻게 되는가

`.bss`를 0으로 채웠으면 `.data`는 어떨까. `.data`는 **초기값이 명시된 전역 변수**가 들어가는 섹션이다.

```c
static int active_count = 7;     // .data에 들어간다 — 초기값 7
```

호스트 환경에서는 ELF 로더가 `.data`의 내용을 파일에서 메모리로 복사해준다. 베어메탈에서는 두 가지 경우로 갈린다. 첫째, QEMU `virt` 보드처럼 ELF를 통째로 메모리에 로드해주는 환경에서는 `.data`도 함께 적재되므로 추가 작업이 필요 없다. 12장의 골격이 `.data` 복사 코드를 생략한 것은 이 첫 번째 경우 덕분이다. 둘째, 라즈베리 파이 4처럼 GPU 펌웨어가 `kernel8.img`라는 **flat binary**(ELF가 아니라 순수한 바이트열)를 로드하는 환경에서는 이야기가 약간 달라진다 — `kernel8.img`에 `.text`와 `.data`가 한 덩어리로 들어가 있으므로 어차피 메모리에 함께 적재된다. 더 복잡한 임베디드 시나리오, 예컨대 Flash에 코드를 두고 RAM으로 `.data`를 복사해야 하는 마이크로컨트롤러에서는 본격적인 `.data` 복사 코드가 필요하지만, 우리의 64비트 Cortex-A 계열에서는 이 단순화가 통한다.

기억해두자. **`.bss` 클리어는 거의 항상 필요하고, `.data` 복사는 환경에 따라 다르다.** 부트 코드가 어떤 환경을 가정하느냐에 따라 이 두 단계의 모양이 달라진다.

## EL2 → EL1 전환 — 권한을 한 단계 낮추는 의식

이제 13장의 세 번째 스테이지로 가자. EL2에서 EL1으로 내려오는 전환 코드다. 그런데 잠깐, 왜 우리는 굳이 EL2에서 EL1으로 내려와야 할까. 그냥 EL2에서 계속 돌면 안 되는가.

이론적으로는 그래도 된다. 우리가 짠 작은 커널이 `hello`만 찍고 끝나는 수준이라면 EL2든 EL1이든 결과는 같다. 그러나 조금만 더 나아가서 인터럽트 벡터 테이블을 설정하고 타이머를 켜고 사용자 모드(EL0)에서 도는 프로세스를 띄우려 한다면, EL의 차이가 결정적이 된다. EL2의 인터럽트 벡터와 EL1의 인터럽트 벡터는 **다른 시스템 레지스터**에 등록되어야 하고, 페이지 테이블도 다른 레지스터(`TTBR0_EL2` vs `TTBR0_EL1`)를 쓴다. 일반적인 OS는 EL1에서 도는 것을 약속으로 하므로, 우리도 그 약속에 미리 합류하는 편이 낫다. 라즈베리 파이 OS, Linux, FreeBSD 모두 EL1을 커널 EL로 사용한다.

EL 전환 코드는 어떻게 생겼을까. 한 토막을 보자.

```asm
    // 현재 EL이 EL2인지 확인하고, 맞으면 EL1으로 내려간다
    mrs   x0, CurrentEL
    lsr   x0, x0, #2            // CurrentEL 필드는 [3:2] 비트
    cmp   x0, #2
    b.ne  skip_el2              // EL2가 아니면 그냥 통과

    // HCR_EL2: EL1을 AArch64로 실행하도록 설정
    mov   x0, #(1 << 31)        // RW 비트: EL1을 AArch64로
    msr   hcr_el2, x0

    // SPSR_EL2: ERET로 돌아갈 때의 모드를 EL1h(Stack Pointer EL1 사용)로
    mov   x0, #0x3c5            // DAIF 마스킹 + EL1h
    msr   spsr_el2, x0

    // ELR_EL2: ERET가 점프할 주소를 다음 레이블로
    adr   x0, skip_el2
    msr   elr_el2, x0

    eret                        // EL1으로 떨어진다
skip_el2:
    // 이제 EL1에서 도는 코드
```

한 호흡에 보기엔 무거운 코드다. 가벼운 마음으로 짚어보자.

먼저 `mrs x0, CurrentEL`로 현재 EL을 읽는다. `CurrentEL` 시스템 레지스터의 비트 `[3:2]`에 현재 EL 값이 들어 있으므로, `lsr x0, x0, #2`(logical shift right, 오른쪽으로 2비트 시프트)로 그 값을 하위로 끌어내린다. 7장에서 손에 익힌 시프트가 또 한 번 등장한다. `cmp x0, #2`와 `b.ne skip_el2`로 EL2가 아니면 전환 코드를 건너뛴다 — 이미 EL1이라면 굳이 내려갈 필요가 없다.

이제 핵심이다. EL 전환은 일반 함수 호출처럼 `bl`이나 `br`로 할 수 없다. 대신 **예외 복귀(Exception Return) 메커니즘을 흉내내는 방식**으로 진행한다. 정확히 말하면, 마치 우리가 EL2에서 어떤 예외를 처리하고 있다가 EL1으로 돌아가는 것처럼 CPU를 속이는 것이다. 그러려면 세 개의 레지스터를 세팅해야 한다.

`HCR_EL2`(Hypervisor Configuration Register, EL2)는 EL2에서 EL1을 어떻게 다룰지 결정한다. `RW` 비트(31번 비트)를 1로 두면 "EL1은 AArch64로 실행한다"는 약속이 된다. 이걸 안 켜면 EL1이 AArch32(32비트)로 떨어져서 우리의 64비트 코드가 갑자기 32비트 모드에서 깨어나 폭주한다. 끔찍한 일이다.

`SPSR_EL2`(Saved Program Status Register, EL2)는 "예외에서 돌아갈 때 복원할 상태"를 담는다. `0x3c5`라는 매직 넘버는 두 부분으로 구성된다. 하위 4비트 `0b0101`은 EL1h 모드(EL1 + Stack Pointer로 SP_EL1 사용)를 의미하고, 상위의 `0x3c0`은 `DAIF`(Debug·SError·IRQ·FIQ) 인터럽트 마스킹을 모두 켠다는 뜻이다. 즉 "EL1으로 돌아가면 인터럽트는 일단 끈 상태로 시작하라"는 안전 약속이다. 인터럽트 벡터 테이블이 아직 준비되지 않은 상태에서 인터럽트가 들어오면 즉시 폭주하므로, 이 마스킹은 필수다.

`ELR_EL2`(Exception Link Register, EL2)는 "예외 복귀가 점프할 주소"를 담는다. `adr x0, skip_el2`로 `skip_el2` 레이블의 주소를 PC 상대 방식으로 계산해서 `x0`에 넣고, 그것을 `ELR_EL2`에 옮긴다. 이제 마지막 `eret` 명령이 모든 약속을 한꺼번에 발동시킨다 — CPU가 `SPSR_EL2`에서 새 모드(EL1h)를 가져오고, `ELR_EL2`에서 새 PC(`skip_el2`)를 가져와서, 마치 EL1에서 던진 예외에서 돌아온 것처럼 EL1의 `skip_el2`로 떨어진다. 이게 ARM AArch64의 EL 전환 의식이다.

복잡해 보인다. 그렇다, 처음 보면 복잡하다. 그래도 한 번만 이 의식을 코드로 적어놓으면 모든 베어메탈 프로젝트에서 거의 그대로 재사용된다. s-matyukevich의 raspberry-pi-os 튜토리얼, Linux 커널의 `arch/arm64/kernel/head.S`, 그 외 거의 모든 AArch64 부트 코드가 이 패턴을 따른다. 한 번 익혀두면 평생 쓰는 의식인 셈이다.

## 링커 스크립트 — 메모리 지도를 손으로 그리다

이제 시선을 코드에서 메모리 배치로 옮기자. 12장에서 우리가 본 `linker.ld`는 짧은 한 페이지짜리 파일이었다.

```ld
ENTRY(_start)
SECTIONS
{
    . = 0x40000000;
    .text : { KEEP(*(.text.boot)) *(.text*) }
    .rodata : { *(.rodata*) }
    .data : { *(.data*) }
    .bss : { *(.bss*) }
}
```

이 다섯 줄이 결정하는 것은 무엇일까. **메모리 어디에 무엇이 놓일지의 지도다.** 한 줄씩 풀어보자.

`ENTRY(_start)`는 ELF의 진입점 심볼을 `_start`로 지정한다. ELF 헤더의 `e_entry` 필드에 이 심볼의 주소가 박힌다. QEMU나 디버거가 ELF를 로드할 때 어디로 점프해야 할지 이 한 줄에서 알아낸다.

`SECTIONS { ... }` 블록 안의 첫 줄, `. = 0x40000000;`. `.`은 **현재 위치 카운터(location counter)** 라고 부르는 링커의 특별한 변수로, "이 다음에 배치될 섹션의 시작 주소"를 의미한다. 0x40000000이라는 숫자는 QEMU `virt` 보드의 RAM이 시작되는 물리 주소다. 왜 하필 이 주소냐고 묻는다면, QEMU의 가상 보드 설계가 그렇게 정해두었기 때문이다 — RAM이 0x40000000부터 시작하고, 그 아래 영역은 MMIO와 부트 ROM이 차지한다. 라즈베리 파이 4(64비트 모드)에서는 이 값이 `0x80000`으로 바뀐다(8장에서 미리 본 GPU 펌웨어의 로드 주소다). **링커 스크립트의 한 숫자가 보드를 바꾼다.** 14장에서 이 한 숫자를 어떻게 바꿔치울지 다시 본다.

`.text : { KEEP(*(.text.boot)) *(.text*) }`. 이 한 줄에 두 가지 약속이 압축되어 있다.

첫째, `.text` 출력 섹션 안에 들어갈 것을 두 가지로 명시한다 — `*(.text.boot)`(모든 입력 파일의 `.text.boot` 섹션)을 먼저, 그 다음에 `*(.text*)`(모든 입력 파일의 `.text`로 시작하는 모든 섹션). 순서가 결정적이다. `.text.boot`이 먼저 나오므로 우리의 `_start` 진입점이 출력 ELF의 맨 앞에 자리를 잡는다. 만약 이 순서를 바꿔서 `*(.text*) *(.text.boot)`이라고 적으면, `_start`가 ELF의 중간 어딘가로 밀려 들어가서 `e_entry` 주소가 어긋나거나 QEMU가 엉뚱한 주소로 점프하게 된다.

둘째, `KEEP(...)` 지시자가 둘러싸고 있다. 이 친구는 링커의 **dead code elimination**(`--gc-sections`로 죽은 섹션을 잘라내는 최적화)으로부터 해당 섹션을 보호하는 약속이다. 우리의 `_start`는 어셈블리에서 정의되었고, C 코드 어디서도 직접 호출되지 않는다 — CPU가 점프해 들어오는 것이지 함수처럼 불리는 것이 아니기 때문이다. 그래서 링커는 `_start`를 "사용되지 않는 죽은 코드"라고 오해할 수 있다. 그걸 잘라버리면 ELF가 진입점 없는 죽은 파일이 된다. 끔찍한 일이다. `KEEP(...)`이 그 사고를 막아준다. 베어메탈 링커 스크립트의 거의 모든 경우에 진입점 섹션은 `KEEP`으로 감싸진다. 잊지 말자.

`.rodata`, `.data`, `.bss`는 각각 읽기 전용 데이터, 초기값 있는 전역 변수, 0으로 초기화되는 전역 변수가 들어가는 출력 섹션이다. `*(.rodata*)`처럼 와일드카드를 쓰는 것은 컴파일러가 만들 수 있는 여러 변종 섹션 이름(`.rodata.str1.1`, `.rodata.cst8` 등)을 모두 한 출력 섹션에 모으겠다는 약속이다.

### `__bss_start`와 `__bss_end` 심볼 만들기

`.bss` 클리어 코드에서 우리가 참조했던 `__bss_start`와 `__bss_end` 두 심볼은 어디서 오는가. 링커 스크립트에서 우리가 직접 만들어야 한다. 12장의 가장 단순한 골격에는 이 심볼이 없었으나, 13장의 두 번째 스테이지로 가면 다음과 같이 보강된다.

```ld
ENTRY(_start)
SECTIONS
{
    . = 0x40000000;
    .text : { KEEP(*(.text.boot)) *(.text*) }
    .rodata : { *(.rodata*) }
    .data : { *(.data*) }
    . = ALIGN(8);
    __bss_start = .;
    .bss : { *(.bss*) *(COMMON) }
    . = ALIGN(8);
    __bss_end = .;
    . = ALIGN(16);
    stack_bottom = .;
    . += 0x4000;                /* 16KB 스택 */
    stack_top = .;
}
```

먼저 `. = ALIGN(8);`. **현재 위치 카운터를 8의 배수로 정렬**한다. 8바이트씩 끊어 `.bss`를 클리어할 때 안전한 경계를 만들어준다. 그 다음 `__bss_start = .;`이 핵심이다. **이 시점의 위치 카운터 값을 `__bss_start`라는 심볼에 박아넣는다.** 즉 `.bss` 섹션이 시작하기 직전의 주소를 심볼로 만들어서, 어셈블리 코드에서 `ldr x1, =__bss_start`로 참조할 수 있게 해주는 다리다. `.bss` 섹션 본문이 끝난 직후에 다시 `__bss_end = .;`로 끝 주소를 박는다. 이렇게 만들어진 두 심볼이 어셈블리에서 `ldr`로 끌려와 BSS 클리어 루프를 굴린다.

이어서 `stack_bottom`과 `stack_top` 심볼은 스택 영역의 경계를 정의한다. `. += 0x4000;`은 위치 카운터를 16KB만큼 진행시켜서 그만큼의 빈 공간을 예약하고, `stack_top = .;`이 스택의 꼭대기 주소를 박는다. AArch64의 스택은 **위에서 아래로 자라므로**(decrement before push의 관례) `SP`에 `stack_top` 주소를 넣어주는 것이 맞다. 12장의 골격에서는 `.bss` 섹션 안에 `stack`/`stack_top`을 어셈블리로 정의했지만, 13장의 보강된 스크립트에서는 링커가 직접 책임을 진다. 책임의 위치가 한 단계 위로 올라가는 셈이다.

### VMA와 LMA — 같지 않을 수 있다

링커 스크립트에서 한 번쯤 마주치는 까다로운 개념이 **VMA(Virtual Memory Address)** 와 **LMA(Load Memory Address)** 의 분리다. 평소에는 둘이 같다. 즉 "메모리에 적재되는 주소"와 "실행 중에 참조되는 주소"가 동일하다. 그런데 임베디드 시스템에서는 갈라지는 경우가 있다.

전형적인 예가 Cortex-M 마이크로컨트롤러다. 초기값 있는 전역 변수(`.data` 섹션)는 영구 저장 장치인 **Flash에 적재(LMA)** 되지만, 부팅 직후에 그 내용을 RAM으로 복사해서 **실행 중에는 RAM 주소(VMA)** 로 참조한다. 링커 스크립트에서 `.data : AT(LMA_in_flash) { *(.data*) }` 같은 표현으로 두 주소를 분리해서 알려준다. 우리의 Cortex-A 베어메탈은 RAM에서 코드를 실행하므로 VMA와 LMA가 같아 별도 표기가 필요 없지만, 임베디드 영역으로 나아가면 이 분리를 자주 만나게 된다. 머릿속에 이런 분리가 있다는 사실만 기억해두자. 우리 책에서는 깊이 들어가지 않는다.

## 어셈블리에서 C로 — ABI라는 약속

이제 어셈블리와 C가 만나는 경계 — `bl kernel_main` 한 줄 — 가 어떻게 정확히 작동하는지 들여다보자. 8장에서 미리 손에 익혀둔 **AArch64 PCS**(Procedure Call Standard, 함수 호출 규약)가 여기서 그대로 살아 돌아온다.

핵심 약속을 한 표로 다시 모아보자.

| 레지스터 | 역할 |
|---------|------|
| `X0`~`X7` | 처음 8개의 정수/포인터 인수 (그 이상은 스택으로) |
| `X8` | indirect result location (큰 구조체 반환 시) |
| `X9`~`X15` | caller-saved 임시 |
| `X16`~`X17` | IP0/IP1 — 인트라프로시저 콜 임시 |
| `X18` | 플랫폼 예약 (macOS는 사용) |
| `X19`~`X28` | callee-saved 임시 |
| `X29` (FP) | Frame Pointer — 스택 프레임 체인 |
| `X30` (LR) | Link Register — 리턴 주소 |
| `X31` | SP 또는 XZR (문맥에 따라) |

`bl kernel_main`이 실행되는 순간 무슨 일이 벌어지는가. CPU가 다음 명령의 주소(`bl` 명령 다음 줄의 주소)를 `X30`에 저장하고, 그 다음 `kernel_main` 심볼의 주소로 점프한다. `kernel_main` 안에 들어가면 컴파일러가 만든 프롤로그가 시작된다 — 보통 `stp x29, x30, [sp, #-16]!`(FP와 LR을 스택에 동시에 푸시) 한 줄과 `mov x29, sp`(새 FP를 현재 SP로 갱신) 한 줄로 짧게 구성된다. 함수의 본문이 끝나면 에필로그가 `ldp x29, x30, [sp], #16`(FP와 LR을 다시 팝)으로 복원하고 `ret`(`X30`이 가리키는 주소로 점프)으로 마무리한다.

우리 `kernel_main`은 `for (;;) {}` 무한 루프로 끝나니까 사실 에필로그가 실행될 일이 없다. 그래도 컴파일러는 보수적으로 프롤로그·에필로그를 만들어 둔다. `objdump -d kernel.elf`로 디스어셈블해보면 이 약속이 그대로 보인다. 한번 직접 눈으로 확인해보자.

```
$ aarch64-elf-objdump -d kernel.elf

kernel.elf:     file format elf64-littleaarch64

Disassembly of section .text:

0000000040000000 <_start>:
    40000000:   d53800a0    mrs     x0, mpidr_el1
    40000004:   92400400    and     x0, x0, #0x3
    40000008:   b5000060    cbnz    x0, 40000014 <halt>
    4000000c:   58000061    ldr     x1, 40000018 <halt+0x4>
    40000010:   9100003f    mov     sp, x1
    40000014:   d503205f    wfe

0000000040000020 <kernel_main>:
    40000020:   a9bf7bfd    stp     x29, x30, [sp, #-16]!
    40000024:   910003fd    mov     x29, sp
    ...
```

각 명령이 32비트 고정폭이고(`d53800a0` 한 워드가 한 명령), 우리 소스 코드의 한 줄이 어셈블리 한 줄과 거의 일대일로 대응한다는 사실이 한눈에 보인다. AArch64는 가변 길이 명령(예: x86-64의 1바이트부터 15바이트까지)이 없는 깔끔한 RISC 설계라서 디스어셈블 결과가 읽기 쉽다. 베어메탈을 공부하기에 좋은 환경 중 하나다.

여기서 한 번 더 짚어두자. `bl kernel_main` 직전에 `SP`가 정확히 `stack_top`을 가리키도록 우리가 어셈블리에서 세팅해두었기 때문에, `kernel_main`의 프롤로그 `stp x29, x30, [sp, #-16]!`이 `stack_top - 16` 위치에 FP/LR을 저장한다. 만약 우리가 `mov sp, x1` 한 줄을 빼먹었다면 어떻게 될까. `SP`는 부팅 직후의 미정의 값이고, 그 미정의 주소에 데이터를 쓰는 순간 시스템이 어디로 가서 뭘 망가뜨릴지 아무도 모른다. **베어메탈의 첫 번째 사고가 바로 이 자리에서 자주 일어난다.** 잊지 말자 — `kernel_main` 호출 전에 `SP`가 유효한 주소를 가리키고 있는지 반드시 확인해야 한다.

## `objdump`와 `readelf`로 ELF의 속을 들여다보기

부팅 코드를 한 줄씩 해부하면서 함께 익혀둘 만한 도구가 두 가지 있다. **`objdump`** 와 **`readelf`** 다. 둘 다 GNU binutils의 일부이고 `aarch64-elf-` 접두사를 단 크로스 버전으로 베어메탈 ELF를 분석할 수 있다.

먼저 `objdump -d kernel.elf`. 위에서 본 디스어셈블 결과를 만들어주는 명령이다. 자주 같이 쓰는 옵션이 몇 개 있다.

| 옵션 | 효과 |
|------|------|
| `-d` | `.text`만 디스어셈블 |
| `-D` | 모든 섹션을 디스어셈블 (`.data`까지) |
| `-S` | 소스 코드와 어셈블리를 섞어서 출력 (`-g`로 컴파일했을 때) |
| `-h` | 섹션 헤더 목록만 |
| `--disassemble=_start` | 특정 심볼만 |

`-S` 옵션이 특히 쓸모 있다. C 소스 한 줄이 AArch64 명령 몇 개로 풀어지는지 한눈에 보이기 때문이다. 가령 `uart_putc(c)` 함수 호출이 어떻게 `mov w0, ...` + `bl uart_putc` 두 줄로 컴파일되는지 확인할 수 있다. 7장에서 익힌 시프트와 마스크가 컴파일러가 만들어주는 명령에서 그대로 등장하는 모습은 한 번쯤 직접 봐둘 가치가 있다.

다음으로 `readelf -a kernel.elf`. ELF 파일의 모든 메타데이터를 사람이 읽을 수 있는 형태로 풀어주는 도구다. 우리가 13장에서 가장 자주 들여다볼 부분은 두 가지다.

```
$ aarch64-elf-readelf -h kernel.elf
ELF Header:
  Magic:   7f 45 4c 46 02 01 01 00 ...
  Class:                             ELF64
  Data:                              2's complement, little endian
  Type:                              EXEC (Executable file)
  Machine:                           AArch64
  Entry point address:               0x40000000
  ...
```

`Entry point address: 0x40000000`. 우리가 `ENTRY(_start)` + 링커 스크립트의 `. = 0x40000000`으로 박아둔 약속이 정확히 ELF 헤더에 반영되었음을 한눈에 확인할 수 있다. QEMU가 이 값을 보고 PC를 0x40000000으로 세팅한 다음 부팅을 시작한다.

```
$ aarch64-elf-readelf -S kernel.elf
There are 12 section headers, starting at offset 0x...:

Section Headers:
  [Nr] Name              Type             Address           Offset
       Size              EntSize          Flags  Link  Info  Align
  [ 0]                   NULL             0000000000000000  ...
  [ 1] .text             PROGBITS         0000000040000000  ...
       0000000000000050  0000000000000000  AX       0     0     8
  [ 2] .rodata           PROGBITS         0000000040000050  ...
       000000000000000f  0000000000000000   A       0     0     1
  [ 3] .bss              NOBITS           0000000040001000  ...
  ...
```

각 섹션이 어디 주소에 자리를 잡았는지, 크기가 얼마인지, 어떤 플래그(A=allocate, X=execute, W=write)를 가졌는지가 한 표에 정리되어 나온다. `.bss`의 타입이 `NOBITS`라는 점에 눈을 두자. **NOBITS는 "ELF 파일 안에 실제 바이트가 없다"는 의미다.** 즉 `.bss`는 크기만 예약되어 있고 내용은 파일에 들어 있지 않다. 이게 곧 부팅 직후 `.bss`를 직접 0으로 채워야 하는 이유와 정확히 맞아떨어진다. 도구가 보여주는 사실이 우리가 어셈블리로 짠 약속과 일치하는 모습을 한 번 직접 보고 나면, 부트 코드의 모든 줄이 갑자기 서로 묶여 의미를 갖기 시작한다.

자, 13장의 두 도구를 일상 습관으로 들여보자. **새 부트 코드를 짤 때마다 `objdump -d`로 진입점을 확인하고, `readelf -h`로 ELF 헤더의 진입 주소를 확인한다.** 이 두 명령이 우리가 의도한 메모리 배치와 실제 ELF의 일치 여부를 한순간에 알려준다.

## 세 단계로 직접 분해해보기 — `ch13-boot-anatomy/`

이번 장의 핵심을 손에 박는 방법은 단 하나다. 13장 예제 디렉토리 `example/ch13-boot-anatomy/`에 들어가서 직접 빌드해보는 것이다. 12장의 골격을 세 개의 스테이지로 쪼개두었다.

**Stage 1 — `stage1-asm-only/`.** 가장 단순한 형태다. `boot.S`가 `_start` 진입점을 갖고, primary core를 골라낸 뒤 곧장 무한 루프에 빠진다. `kernel.c`도 `linker.ld`도 없다 — 어셈블리 한 파일과 링커 스크립트, Makefile뿐이다. **이 스테이지가 빌드되고 QEMU에서 무한 루프에 빠진다는 사실이 확인되면, 부팅의 가장 작은 단위가 손에 잡힌 셈이다.** 화면에는 아무것도 안 찍히지만, `objdump -d`로 ELF의 진입점이 우리가 의도한 곳에 있는지, `readelf -h`로 `Entry point address`가 0x40000000인지 확인할 수 있다. **CPU가 점프해 들어와서 우리가 짠 코드를 실행하다가 영원히 잠든다** — 이게 베어메탈의 최소 단위다.

**Stage 2 — `stage2-bss-stack/`.** `.bss` 클리어와 스택 포인터 셋업이 더해진다. 링커 스크립트에 `__bss_start`/`__bss_end`/`stack_top` 심볼이 추가되고, `boot.S`에 BSS 클리어 루프와 `mov sp, x1` 두 줄이 들어간다. `kernel.c`는 여전히 없다 — 어셈블리만으로 "C 함수를 호출할 준비가 끝난 상태"까지 도달한다. 빌드해서 QEMU에서 돌려보면 역시 화면에는 아무것도 안 뜨지만, `objdump -d`로 BSS 클리어 루프가 우리 의도대로 만들어졌는지 확인할 수 있다.

**Stage 3 — `stage3-main-call/`.** 마침내 12장과 완전히 동등한 상태다. `kernel.c`가 추가되고 `bl kernel_main`이 들어가서 시리얼에 `hello, aarch64`가 찍힌다. Stage 2와의 차이가 정확히 "C 함수를 한 번 호출한다"는 것 하나뿐임을 코드를 비교해보면 알 수 있다.

세 스테이지를 차례로 빌드해보는 것의 진짜 목적은 단순히 코드를 보는 것이 아니라, **"부팅에 꼭 필요한 것"과 "있으면 좋은 것"의 경계를 손에 익히는 것**이다. Stage 1만 있어도 부팅 자체는 한다 — 진입점이 있고, 무한 루프가 있다. Stage 2는 거기에 안전한 스택을 더해서 C 함수 호출의 발판을 마련한다. Stage 3에서야 비로소 화면에 글자가 나온다. 이 점진적 누적이 손에 익으면, 14장에서 라즈베리 파이 4로 옮길 때 "어디서 막혔는가"를 정확히 추적할 수 있는 능력이 생긴다.

각 스테이지마다 자체 `Makefile`이 있고, 최상위 `Makefile`이 세 스테이지를 차례로 빌드한다. `make stage1`·`make stage2`·`make stage3`로 개별 빌드도 되고, `make all`로 한 번에 셋 다 빌드할 수 있다. `make run-stage3`로 마지막 단계를 QEMU에서 실행해 글자가 찍히는지 직접 확인해보자. `make objdump-stage3`로 디스어셈블 결과를 텍스트 파일로 떨궈서 천천히 들여다보는 것도 권한다.

## 마무리 — 한 줄도 마법이 아니다

13장의 출발점에서 던졌던 질문으로 돌아가자. **`mrs x0, mpidr_el1`은 무엇을 읽는 명령인가.** 이제 우리는 단답으로 답할 수 있다 — 현재 CPU 코어의 affinity 레지스터를 읽어서 primary core인지 골라내기 위한 첫 단계다. **`KEEP(*(.text.boot))`은 왜 `KEEP`을 두르는가.** 링커의 dead code elimination으로부터 진입점 섹션을 보호하기 위해서다. **`.bss` 클리어가 왜 필요한가.** 베어메탈에는 호스트 OS의 ELF 로더가 없어서 NOBITS 섹션이 자동으로 0으로 채워지지 않기 때문이다. **EL2에서 EL1으로 어떻게 내려가는가.** `HCR_EL2`·`SPSR_EL2`·`ELR_EL2` 세 시스템 레지스터를 세팅하고 `eret`로 떨어진다.

처음에는 일곱 줄 어셈블리와 다섯 줄 링커 스크립트가 마법처럼 보였다면, 이제는 한 줄도 마법이 아니다. 모두가 명확한 이유와 정해진 약속의 산물이다. 그리고 이 약속들은 라즈베리 파이 4든, 다른 Cortex-A 보드든, 우리가 앞으로 만날 모든 AArch64 베어메탈 환경에서 거의 그대로 재사용된다. 한 번 손에 박아두면 평생 쓰는 자산이다.

기억해두자. **어셈블리는 짧고 정직하다.** 호스트 C의 추상이 가렸던 메모리·레지스터·CPU의 실제 동작이 여기서는 한 줄씩 그대로 드러난다. 처음엔 그 적나라함이 부담스럽지만, 두 번 세 번 들여다보면 오히려 호스트 코드보다 더 솔직하고 예측 가능하다는 사실을 알게 된다. 디버깅도 마찬가지다 — `objdump -d`로 명령 하나하나를 따라가는 베어메탈 디버깅은, 한 줄에 무엇이 숨어 있을지 모를 거대한 호스트 프레임워크의 디버깅보다 오히려 단순한 경우가 많다.

다음 장에서는 시선을 옮긴다. QEMU 가상 보드에서 손에 익힌 부팅 코드를 이번에는 **실제 손에 잡히는 라즈베리 파이 4**로 옮긴다. 메모리 시작 주소가 0x40000000에서 0x80000으로 바뀌고, MMIO UART의 종류와 주소가 달라지고, 부팅 펌웨어가 ELF 대신 flat binary(`kernel8.img`)를 요구한다. 그러나 13장에서 익힌 일곱 줄의 약속 — 누구냐 → 통과냐 → 스택 → C → 잠재우자 — 은 보드가 바뀌어도 그대로 통한다. 손에 들고 가자.
