# 7장. 정의되지 않은 동작 — C가 너를 어떻게 배신하는가

밤 11시, 어제까지 멀쩡히 돌던 코드가 갑자기 이상한 값을 찍는다. 컴파일러를 바꿨거나, `-O0`에서 `-O2`로 옮겼거나, 그도 아니면 옆자리 동료가 무관해 보이는 함수에 `static`을 한 줄 붙였을 뿐인데, 어딘가에서 숫자가 어긋난다. 디버거를 붙여 보면 더 황당하다 — 분명히 보이는 `if (p != NULL)` 가드가 통째로 사라져 있거나, 반복문이 한 바퀴 더 돌고 있거나, 방금 출력한 변수가 그다음 줄에서는 다른 값이다.

Java였다면 어땠을까. `ArrayIndexOutOfBoundsException`이 떨어지고 스택 트레이스가 친절히 사고 현장을 가리켰을 것이다. Python이었다면 인터프리터가 `IndexError`로 즉시 발을 잡았을 것이다. 그런데 C는 침묵한다. 침묵하다가, 가장 곤란한 시점에, 가장 추적하기 어려운 방식으로 무너진다.

이게 정의되지 않은 동작(Undefined Behavior, UB)이라는 손님이다. C 표준이 일부러 비워 둔 칸이고, 컴파일러가 그 빈칸을 자유롭게 채워 쓰는 자리다. 이 장에서는 그 빈칸을 들여다보자. 두려워서 피하려는 게 아니라, 어디에 함정이 있는지 알고, 어떤 도구가 함정을 비춰 주는지 손에 익히려는 것이다. 그리고 이어지는 후반부에서는 베어메탈로 가는 다리 — 고정폭 정수, 비트 조작, 엔디언 — 를 한 절로 묶어 깔아 두자. Ch12에서 ARM CPU의 MMIO 레지스터를 손가락으로 두드릴 때, 이 절의 손풀이가 그대로 쓰인다.

## "정의되지 않음"은 무슨 뜻인가

먼저 단어부터 풀어 보자. C 표준에는 세 가지 어색한 동작 등급이 있다. 구현 정의 동작(implementation-defined), 명세되지 않은 동작(unspecified), 그리고 정의되지 않은 동작(undefined). 앞 두 가지는 그래도 어느 정도 예측이 된다. 구현 정의는 컴파일러가 어떻게 하는지 문서로 약속하고, 명세되지 않은 동작은 표준이 여러 합리적 선택지 중 하나를 자유롭게 고를 수 있게 해 준다.

세 번째, 정의되지 않은 동작이 진짜 손님이다. 표준은 이렇게 말한다 — "여기서 일어나는 일에 대해 우리는 아무것도 보장하지 않는다." 그게 무슨 의미일까. 컴파일러 입장에서 보면 면죄부다. UB가 발생하는 코드 경로는 **존재해서는 안 되는 경로**다. 존재해서는 안 되는 경로라면, 컴파일러는 그 경로가 실행되지 않는다는 가정 아래 최적화를 마음껏 할 수 있다.

이 부분이 Java/Python 출신에게 가장 비현실적으로 들리는 대목이다. 코드에 적힌 if 문이 사라진다고? 그렇다. 사라진다. 컴파일러가 "이 if 안에 들어오는 경로는 UB를 거쳐야만 도달 가능하다 — 그렇다면 이 경로는 실행되지 않으므로 if 자체가 죽은 코드다"라고 추론하면, if 블록이 통째로 없어진다. 그리고 사라진 코드가 NULL 가드였다면, 그 가드를 통과해 NULL 포인터가 그대로 역참조된다.

> John Regehr는 자기 블로그 *The Strict Aliasing Situation is Pretty Bad*에서 이렇게 정리했다. "C의 정의되지 않은 동작은 컴파일러에게 주어진 거대한 가정의 출처다. 컴파일러는 UB를 발견하면 그 코드 경로가 실행될 수 없다고 가정하고, 그 가정으로부터 다른 모든 가정을 끌어낸다." 무서운 말이다. 한 번 UB를 흘리면, 그 주변의 코드까지 줄줄이 영향을 받는다는 뜻이기 때문이다.

이 무서움을 직시하자. 그렇다면 어떻게 해야 할까? 두 단계로 가자. 첫째, 자주 부딪히는 UB를 종류별로 알아 둔다. 둘째, UBSan/ASan 같은 도구로 그 UB가 실행 시점에 잡히도록 빌드를 잡아 둔다. 이 장에서는 첫째를 본격적으로 다루고, 둘째는 9장에서 깊게 들어간다. 다만 예제는 미리 sanitizer를 켜고 돌려서 손맛을 보자.

## UB 카탈로그 — 흔한 손님 다섯

C에서 자주 부딪히는 UB는 사실 그렇게 많지 않다. 종류로 따지면 200가지가 넘지만, 실무에서 자주 마주치는 손님은 다섯 명 정도다. 한 명씩 인사하자.

### signed 정수 오버플로

```c
int big = 2000000000;
int sum = big + big;   /* INT_MAX(약 21억)를 넘는다 */
```

`int`가 32비트라면 `INT_MAX`는 약 21억 4748만이다. 위 코드의 `big + big`은 그것을 훌쩍 넘는다. 자, 어떻게 될까? Java라면 정의된 wrap-around가 일어나 음수가 된다. C에서도 일어나기는 한다 — 그러나 그건 "**우연히** 그렇게 동작했을 뿐"이다. 표준은 signed 정수 오버플로를 UB로 못 박았다.

왜 굳이? 컴파일러가 더 공격적인 최적화를 할 수 있게 하기 위해서다. `for (int i = 0; i <= n; i++)` 같은 반복문에서, 컴파일러가 "signed 오버플로는 일어나지 않는다"고 가정할 수 있으면 루프 전개나 벡터화에서 자유가 커진다. 그 대가로, 정말 오버플로가 발생하면 코드의 어딘가가 조용히 깨진다.

`unsigned`는 다르다. 표준이 wrap을 정의해 줬다. `UINT_MAX + 1`은 0이다. 이 비대칭은 모던 C 코딩의 첫 번째 습관 형성 포인트다 — **카운터·인덱스에 signed를 쓰면 일단 한 번 더 생각해 보자**. 외부에서 들어오는 사이즈, 길이, 카운트는 `size_t`(unsigned)가 정석에 가깝다.

### NULL 역참조

```c
int *p = NULL;
int v = *p;   /* UB */
```

Java였다면 `NullPointerException`이 단정한 자세로 떨어졌을 것이다. C에서는 운에 맡기는 게임이다. 운이 좋으면 OS의 가상 메모리 보호가 작동해 SIGSEGV가 떨어지고 프로세스가 깨끗하게 죽는다. 운이 나쁘면 — 임베디드에서는 흔하다 — NULL 주소(0)에 진짜로 무언가가 매핑되어 있어 거기 있던 값이 변수로 들어온다. 더 나쁜 경우, 컴파일러가 "이 코드가 도달하면 UB이므로 도달할 수 없다"고 추론해 그 윗줄의 NULL 체크를 통째로 지워 버린다.

NULL 체크가 사라지는 사고는 실제로 자주 일어났다. 가장 유명한 사례는 Linux 커널의 `tun_chr_poll` 버그다. 포인터를 먼저 역참조한 뒤에 NULL 체크를 했는데, 컴파일러가 "역참조 뒤에 NULL이라는 건 UB이므로 NULL 체크는 의미가 없다"고 보고 그 체크를 지웠다. 그게 보안 취약점이 됐다. 같은 패턴을 우리 코드에 심어 두고 있지 않다고 확신할 수 있을까? 좀 찜찜하다. 그래서 NULL 체크는 항상 역참조 **전에** 두고, 정말 NULL이면 안 되는 자리에는 `assert(p != NULL)`이나 매크로로 한 줄을 박아 두자.

### 배열 경계 밖 접근

```c
int arr[5] = {1, 2, 3, 4, 5};
int v = arr[7];   /* 길이는 5, 유효 인덱스는 0~4 */
```

이 한 줄이 침묵하는 게 C의 가장 정직한 모습이다. 옆 스택 프레임에 있던 다른 변수의 값이 그대로 묻어 나오거나, 운이 나쁘면 다음 함수 호출의 리턴 주소까지 닿는다. Java가 익숙한 사람에게는 가장 받아들이기 어려운 침묵이다. 그러나 동시에 C가 빠른 이유 중 하나이기도 하다 — 모든 배열 접근에 경계 체크를 끼우지 않기 때문이다. 우리가 그 비용을 직접 부담하기로 약속한 셈이다.

다행히 도구가 있다. ASan은 스택 배열의 양옆에 빨간 띠(redzone)를 깔아 두고, 그 띠를 건드리는 순간 `stack-buffer-overflow`로 깔끔히 잡아낸다. 이게 도구의 손맛이다. Ch9에서 본격적으로 손에 익히겠지만, 이 장 예제에서 미리 한 번 보고 가자.

### 초기화 안 된 변수 읽기

```c
int x;
if (x > 0) { /* ... */ }   /* x를 초기화하지 않았다 */
```

자동 변수(스택에 잡히는 지역 변수)는 **초기화되지 않은 채 태어난다**. 그 자리에는 직전 스택 프레임이 남기고 간 쓰레기 비트가 그대로 있다. 그걸 읽어서 분기에 쓰면, 분기 방향이 매번 달라진다. MSan(MemorySanitizer)이 이걸 잡는 게 본업인데, 안타깝게도 macOS에서는 제한적이다. UBSan의 `-fsanitize=initialize`와 컴파일러의 `-Wuninitialized` 경고가 1차 방어선이다. 더 단순한 길은 **선언과 동시에 초기화**하는 습관이다. C23의 `auto x = 0;` 같은 표현이 그 습관을 가볍게 도와준다.

### `volatile` 누락

이건 호스트 코드에서는 잘 안 보인다. 그러나 Ch12에서 MMIO 레지스터를 만지기 시작하면 정면으로 부딪힌다. 한 줄로만 말해 두자 — **메모리 매핑된 하드웨어 레지스터는 `volatile`을 빼면 컴파일러가 접근을 통째로 최적화해 날려 버린다**. 같은 주소에 같은 값을 두 번 쓰는 코드는 컴파일러 눈에는 "한 번만 써도 된다"로 보이기 때문이다. 그런데 그 한 번을 두 번 써야 하드웨어가 동작하는 경우가 있다. 끔찍한 일이다. 자세한 이유와 패턴은 Ch12 MMIO 절에서 다시 만나자.

## 컴파일러가 코드를 지운다 — 실제 사례

위 다섯이 카탈로그라면, "왜 갑자기 망가지는지 모르겠다"는 그 느낌은 어디서 올까? 컴파일러가 UB를 가정으로 끌어다 쓰는 한 가지 패턴을 직접 보자.

```c
int divide(int a, int b) {
    int q = a / b;       /* b == 0 이면 UB */
    if (b == 0) {
        return -1;       /* 0으로 나눈 케이스 처리 */
    }
    return q;
}
```

읽어 보면 의도가 보인다 — `b`가 0이면 -1을 돌려주려는 가드다. 그런데 컴파일러는 이렇게 추론한다. "`a / b`가 이미 실행됐다. 그게 정의되려면 `b != 0`이었어야 한다. 따라서 이 함수에 들어와 그 줄을 통과했다면 `b != 0`이다. 그렇다면 아래의 `if (b == 0)` 가드는 절대 참이 될 수 없다 — 죽은 코드다. 지워 버리자." 그래서 `-O2`로 최적화하면 그 if 블록이 통째로 사라진다.

해결책은 명백하다. 가드를 **나누기 전에** 두면 된다. 그러나 같은 패턴이 더 미묘한 모습으로 코드 곳곳에 숨어 있다 — 포인터 역참조 뒤의 NULL 체크, 시프트 뒤의 시프트 양 검사, 인덱스 접근 뒤의 경계 검사. 한 번 의심하기 시작하면 끝이 없다. 그래서 좋은 자세는 두 가지다. 첫째, **검사는 사용 전에** 두는 습관. 둘째, **UBSan을 일상 빌드의 기본값으로** 두는 빌드 구성. 이 둘이 같이 가야 마음이 편해진다.

## Strict Aliasing — 같은 비트를 다른 안경으로 보면 안 된다

UB 카탈로그 중에서도 한 명, strict aliasing은 별도의 절을 받을 자격이 있다. 가장 추적하기 어렵고, 가장 자주 "왜 됐지?" 하다가 "왜 안 되지?"로 바뀌는 손님이기 때문이다.

규칙은 한 줄로 쓸 수 있다. **한 객체의 값은 그 객체의 정의된 타입, 그것과 호환되는 타입, 또는 문자 타입(`char`/`signed char`/`unsigned char`)을 통해서만 접근해야 한다.** 이걸 어기면 UB다.

말은 추상적이지만 사례는 구체적이다.

```c
int bits = 0x40490FDB;          /* 비트로는 π의 IEEE-754 표현 */
float *as_float = (float *)&bits;
float pi = *as_float;           /* UB — strict aliasing 위반 */
```

`int` 자리에 들어 있는 비트를 `float`라는 다른 안경으로 들여다보려는 시도다. 우리는 "어차피 비트는 똑같으니 그냥 캐스트로 끝내면 되지 않나?" 하고 생각한다. 그런데 컴파일러는 다르게 본다. 컴파일러 입장에서 `int*`와 `float*`는 **서로 다른 타입의 객체를 가리킨다**고 가정한다. 그 가정 아래 두 포인터를 통한 접근을 자유롭게 재배치하고, 레지스터에 캐싱하고, 순서를 바꾼다. 우리가 같은 메모리를 두 안경으로 보고 있다는 사실은 컴파일러에게는 보이지 않는다.

그 결과 무슨 일이 일어날까? 디버그 빌드(`-O0`)에서는 그냥 동작한다. 비트가 그대로 읽히고 π 비슷한 값이 찍힌다. 그러다 `-O2`로 올리는 순간, 컴파일러가 두 접근의 순서를 뒤집거나 어느 한쪽을 캐시에 묶어 두면서, 갑자기 엉뚱한 값이 나오기 시작한다. "디버그에서는 됐는데 릴리즈에서는 안 된다" — 이 문장이 strict aliasing의 정체다.

### 안전한 비트 재해석 — `memcpy`가 정석

그렇다면 어떻게 해야 할까? 비트를 그대로 다른 타입으로 보고 싶을 때, 답은 단정하다. **`memcpy`로 비트만 옮긴다**.

```c
int bits = 0x40490FDB;
float pi;
memcpy(&pi, &bits, sizeof pi);   /* 같은 크기의 비트를 그대로 복사 */
```

`memcpy`는 바이트 단위로 동작하기 때문에 strict aliasing 규칙에 걸리지 않는다. 그리고 모던 컴파일러는 이 패턴을 알아보고 실제 호출 없이 인라인으로 비트를 옮긴다. 성능 손해도 없다. C23은 여기서 한 걸음 더 나가 `<stdbit.h>`에 비트 조작 헬퍼를 정리했지만, 비트 재해석의 경우는 `memcpy`가 여전히 표준 관용구다.

union을 쓰는 길도 있다. `union { int i; float f; }` 식으로 묶어 한쪽으로 쓰고 다른 쪽으로 읽는 것이다. C99 이후로는 합법이지만(C++에서는 여전히 UB) 코드 의도가 흐려지기 쉽다. 비트 재해석을 한 번만 하는 자리라면 `memcpy`로 가는 편이 깔끔하다.

### `char *`는 특권 안경이다

위 규칙의 예외가 하나 있다. **`char`/`signed char`/`unsigned char` 포인터는 어떤 객체든 들여다볼 수 있다.** 이건 표준이 일부러 열어 둔 뒷문이다. 메모리 덤프, 시리얼라이즈, 직렬화 같은 작업에서 바이트 단위로 객체를 만지작거리려는 자리에서 살아남게 해 주려는 의도다.

```c
uint32_t reg = 0x12345678;
unsigned char *bytes = (unsigned char *)&reg;
for (int i = 0; i < 4; i++) {
    printf("%02x ", bytes[i]);  /* 합법, UB 아님 */
}
```

이 코드가 무엇을 찍는지가 곧 다음 절의 주제 — 엔디언과 직결된다.

### `-fno-strict-aliasing`은 진통제일 뿐

여기서 한 가지 의문이 든다 — "리눅스 커널은 strict aliasing을 안 지킨다는데?" 맞는 말이다. Linus Torvalds가 GCC의 strict aliasing 가정을 매우 싫어한 게 유명한 일화고, Linux 커널 빌드는 `-fno-strict-aliasing`을 켜고 돌아간다. 그렇다면 우리도 그 옵션을 켜면 되지 않을까?

물론 가능하다. 하지만 그건 진통제다. 통증을 없애는 게 아니라 가려 두는 것이다. `-fno-strict-aliasing`이 켜져 있어도 코드 자체는 여전히 UB를 품고 있다. 다른 컴파일러, 다른 버전, 다른 타깃으로 옮기는 순간 잠재된 통증이 되살아난다. 작은 프로젝트라면 처음부터 `memcpy` 관용구를 들이는 편이 낫다. 진통제는 정말 거대한 레거시를 옮길 때만 꺼내자.

## 고정폭 정수와 비트 조작 — 모던 C의 기본 자세

여기까지가 UB의 카탈로그였다. 이제 베어메탈로 가는 다리를 깔자. 다리는 세 부분이다 — 고정폭 정수, 비트 마스크와 시프트, 엔디언. 이 셋이 Ch12에서 ARM PL011 UART의 데이터 레지스터에 한 글자를 써 넣는 그 한 줄을 떠받친다.

### `int`/`long`의 시절은 갔다

K&R 시절에는 `int`가 좋은 디폴트였다. 그 시절 시스템에서는 `int`가 워드 크기에 거의 일치했고, 코드를 한 머신에서 다른 머신으로 옮길 때도 큰 사고가 안 났다. 그런데 그 약속은 32비트에서 64비트로 옮겨 가면서 깨졌다. LP64(Linux/Mac)에서는 `int`가 32비트, `long`이 64비트다. LLP64(Windows)에서는 `int`도 `long`도 모두 32비트고, 64비트는 `long long`이다. 같은 `long`이 운영체제에 따라 크기가 다르다. 이걸 모르고 짠 K&R 시절 코드가 64비트로 옮겨오며 침묵 속에 잘려 나간 일이 한두 번이 아니다.

그래서 모던 C는 다른 자세를 택한다. **폭이 중요한 자리에는 폭을 명시한다.** `<stdint.h>` 헤더가 그 도구함이다.

```c
#include <stdint.h>

uint8_t  byte    = 0xFFu;          /* 8비트, 0~255 */
uint16_t halfword = 0xCAFEu;       /* 16비트 */
uint32_t word    = 0xDEADBEEFu;    /* 32비트 */
uint64_t dword   = 0x0123456789ABCDEFull;  /* 64비트 */

int32_t signed_word = -1;          /* 명시적 32비트 부호 있는 정수 */
```

이름이 모든 걸 말해 준다. `u`가 붙으면 unsigned, 숫자가 폭이다. 임베디드, 베어메탈, 네트워크 패킷 파서, 파일 포맷 파서 — 폭이 의미를 갖는 모든 자리에서 `<stdint.h>` 타입이 정답에 가깝다. 그리고 ARM 레지스터처럼 "이 비트는 항상 32비트 워드여야 한다"는 자리에서는 `uint32_t`가 자기소개를 끝낸다.

여기에 한 마디 더 — 카운터·인덱스는 어떻게 할까? 정석은 `size_t`다. unsigned이고 플랫폼의 자연스러운 워드 크기를 따라간다. 음수가 의미 있는 카운트라면 `ssize_t`(POSIX)나 `ptrdiff_t`를 고민한다.

### 비트 마스크와 시프트 — 한 비트씩 손에 익히자

레지스터를 만진다는 건 비트를 만진다는 뜻이다. 32비트 워드 중에서 7번 비트만 1로 켜고, 12~15번 비트에서 값을 읽고, 나머지는 그대로 두는 — 이런 작업이다. 다섯 가지 관용구를 손에 익혀 두자.

```c
uint32_t reg = 0x00000000u;

/* (1) 비트 켜기 — OR */
reg |= (1u << 7);             /* 7번 비트 = 1 */

/* (2) 비트 끄기 — AND with NOT */
reg &= ~(1u << 7);            /* 7번 비트 = 0 */

/* (3) 비트 토글 — XOR */
reg ^= (1u << 3);             /* 3번 비트를 뒤집는다 */

/* (4) 비트 읽기 — 시프트와 마스크 */
uint32_t baudrate_field = (reg >> 12) & 0xFu;   /* 12~15번 비트를 4비트 값으로 */

/* (5) 비트 필드에 값 쓰기 — 끄고 켜기 */
reg = (reg & ~(0xFu << 12)) | ((newval & 0xFu) << 12);
```

처음 보면 어지럽다. 그런데 패턴이 다섯 개뿐이다. 한 시간만 손으로 따라 써 보면 입력이 자연스럽게 흘러나온다. 특히 (5)번 패턴 — "한 필드를 먼저 끄고, 그 자리에 새 값을 OR로 끼우는" — 이 베어메탈 드라이버 코드에서 가장 자주 보는 손동작이다. Ch12에서 UART 컨트롤 레지스터를 만질 때 다시 만난다.

한 가지 조심할 점이 있다. `1 << 7`이 아니라 `1u << 7`이라고 명시적으로 unsigned 리터럴을 쓰는 습관이다. 왜냐하면 signed 정수에서 부호 비트(31번)까지 시프트가 닿으면 UB가 되기 때문이다. `1 << 31`은 32비트 `int`에서 부호 비트를 1로 설정하려는 시도가 되는데, 이건 표준이 금지한다. `1u << 31`은 unsigned이므로 정의된 동작이다. 작은 차이지만 베어메탈 드라이버 코드에서는 매번 나오는 차이다.

### 엔디언 — 같은 비트도 다르게 읽힌다

이제 마지막 다리, 엔디언이다. 32비트 정수 `0x12345678`을 메모리에 한 워드로 저장했다고 하자. 그 정수의 첫 바이트(메모리에서 가장 낮은 주소)에 있는 값이 `0x12`인가, `0x78`인가? 이 질문의 답이 엔디언이다.

- **빅 엔디언:** 큰 자리(MSB)가 먼저. 첫 바이트는 `0x12`.
- **리틀 엔디언:** 작은 자리(LSB)가 먼저. 첫 바이트는 `0x78`.

ARM은 양쪽 모두를 지원하지만, Apple Silicon과 Raspberry Pi 4를 포함한 대부분의 모던 ARM 시스템은 **리틀 엔디언**으로 동작한다. x86도 리틀 엔디언이다. 그래서 호스트에서 베어메탈로 가는 흐름에서는 엔디언이 같다는 점이 다행이다. 다만 네트워크 패킷을 다루기 시작하면 갑자기 빅 엔디언과 만나야 한다 — 네트워크 바이트 오더가 빅 엔디언이기 때문이다.

호스트의 엔디언을 직접 확인하는 한 줄을 알아 두자.

```c
#include <stdio.h>
#include <stdint.h>

int main(void) {
    uint32_t probe = 0x01020304u;
    unsigned char *p = (unsigned char *)&probe;
    printf("first byte = 0x%02x\n", p[0]);
    if (p[0] == 0x04) printf("little endian\n");
    else if (p[0] == 0x01) printf("big endian\n");
    return 0;
}
```

`unsigned char *`로 객체를 들여다보는 건 strict aliasing 규칙의 예외였다는 점을 기억하자. 그래서 이 코드는 합법이다.

컴파일 시점에 엔디언을 분기하고 싶다면 GCC/Clang의 빌트인 매크로를 쓴다.

```c
#if __BYTE_ORDER__ == __ORDER_LITTLE_ENDIAN__
    /* 리틀 엔디언 경로 */
#elif __BYTE_ORDER__ == __ORDER_BIG_ENDIAN__
    /* 빅 엔디언 경로 */
#endif
```

바이트를 뒤집어야 할 때는 GCC/Clang이 제공하는 `__builtin_bswap16`/`bswap32`/`bswap64`가 빠르고 정확하다. POSIX에서는 `htonl`/`ntohl`(호스트 ↔ 네트워크)로 같은 일을 한다. 한 줄로 끝나는 자리에 비트 시프트를 손으로 짜지 말자 — 컴파일러 빌트인은 보통 단일 명령어로 컴파일된다. 손코드는 더 느리고 더 실수하기 쉽다.

## 도구 없이는 침묵, 도구가 있으면 즉시 발각

지금까지 본 UB들은 도구의 도움 없이는 침묵 속에 코드를 망친다. 그러나 도구를 켜는 순간 즉시 잡힌다. 한 번 손맛을 보자. 예제는 `example/ch07-ub/` 폴더에 있다.

```c
/* signed 정수 오버플로 */
int big = 2000000000;
int sum = big + big;
```

이걸 `make ubsan`으로 빌드하고 돌려 보자.

```
$ make ubsan
$ ./ub_zoo overflow
ub_zoo.c:25:19: runtime error: signed integer overflow:
  2000000000 + 2000000000 cannot be represented in type 'int'
SUMMARY: UndefinedBehaviorSanitizer: undefined-behavior
signed overflow result = -294967296
```

UBSan이 정확히 그 줄, 그 연산을 가리킨다. 같은 코드를 `make unsafe`로 sanitizer 없이 빌드하면? 그냥 `-294967296`이라는 wrap된 값이 찍히고 끝난다. 문제가 있는지조차 알 수 없다. 이게 침묵의 모습이다.

NULL 역참조도 마찬가지다.

```
$ ./ub_zoo null
ub_zoo.c:44:13: runtime error: load of null pointer of type 'int'
SUMMARY: UndefinedBehaviorSanitizer: undefined-behavior
AddressSanitizer:DEADLYSIGNAL
ERROR: AddressSanitizer: SEGV on unknown address 0x000000000000
```

UBSan이 NULL 역참조라고 진단하고, ASan이 뒤이어 SEGV를 깔끔하게 보고한다. 어디서, 어떤 타입의 포인터가, 어떤 주소를 건드렸는지가 한눈에 들어온다.

OOB 읽기도 다르지 않다.

```
$ ./ub_zoo oob
ub_zoo.c:54:13: runtime error: index 7 out of bounds for type 'int[4]'
SUMMARY: UndefinedBehaviorSanitizer: undefined-behavior
ERROR: AddressSanitizer: stack-buffer-overflow
READ of size 4 at 0x... thread T0
```

UBSan이 "인덱스 7이 `int[4]`의 경계 밖"이라고 정확히 짚어 주고, ASan이 스택 버퍼 오버플로로 한 번 더 잡는다. 두 도구가 같이 켜져 있을 때의 안도감이 이렇다.

자, 그렇다면 strict aliasing 위반은? 안타깝게도 이건 런타임 도구로는 잡기가 어렵다. UBSan에는 `-fsanitize=alignment`나 `-fsanitize=object-size` 같은 친척이 있지만, strict aliasing 위반 자체를 잡는 옵션은 없다. 컴파일러의 `-Wstrict-aliasing` 경고가 1차 방어선이고, 그다음은 코드 리뷰와 관용구 — "비트 재해석은 항상 `memcpy`로" — 가 방어선이다. 도구로 못 잡는 손님이 가장 무섭다는 사실을, 다시 한 번 기억해 두자.

## 챕터를 닫으며

이 장에서 만난 손님들을 정리하자. signed 오버플로, NULL 역참조, OOB 접근, 초기화 안 된 변수, `volatile` 누락 — 다섯 개의 UB. 그리고 한 명의 별도 게스트, strict aliasing. 마지막으로 베어메탈로 가는 다리 셋 — `<stdint.h>`의 고정폭 정수, 비트 마스크와 시프트 다섯 관용구, 엔디언.

C가 침묵한다는 사실은 두려운 일이지만, 도구가 그 침묵을 깨 준다는 사실은 위안이 된다. UBSan과 ASan을 켠 빌드를 일상으로 두는 습관이 이 장의 가장 큰 자산이다. 9장에서 이 도구들을 빌드 매트릭스의 정점에 두고 본격적으로 손에 익히자.

그리고 비트와 엔디언의 손풀이는 5장에서 깔아 둔 길로 이어진다. 함수 호출이 실제로 어떻게 일어나는지, `.c`가 어떻게 `.o`로 묶이고 한 실행파일이 되는지를 다음 장에서 본다. 8장이 끝나면 우리는 빌드 사이클의 네 단계를 한 손에 잡게 되고, 함수가 인수를 어떤 레지스터에 싣고 호출되는지 — AArch64에서 `X0`부터 `X7`까지의 그 약속이 — 손에 들어온다. 그 약속은 베어메탈에서 어셈블리와 C가 만나는 자리에서 그대로 다시 쓰인다.
