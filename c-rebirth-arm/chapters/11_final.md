# 11장. 베어메탈로 가는 다리 — Freestanding C와 호스트 없는 세계

`printf` 한 줄을 쓴다고 해 보자. 우리 입장에서는 한 줄짜리 일이지만, 그 한 줄 뒤에는 거대한 협력이 숨어 있다. `printf`는 포맷 문자열을 파싱하고 인수를 끌어다가 바이트 시퀀스를 만든다. 그 시퀀스를 stdout이라는 파일 디스크립터로 흘려보낸다. stdout은 운영체제의 시스템 콜 — `write(1, buf, n)` — 로 변환된다. 시스템 콜은 커널로 점프하고, 커널은 TTY 드라이버를 거쳐 터미널 에뮬레이터로 바이트를 전달한다. 터미널 에뮬레이터가 픽셀 위에 글자 모양을 그린다.

이 협력의 사슬 어디에선가 한 마디라도 끊어지면 `printf`는 안 도는 함수다. **그리고 베어메탈에서는 사슬 전체가 없다.**

이걸 처음 마주하면 어떤 기분이 드는지 정직하게 들여다보자. 호스트 OS 위에서 자란 손은 "당연한 것"이 한가득이다. `malloc`을 부르면 메모리가 나온다. `printf`를 부르면 글자가 찍힌다. `errno`가 어딘가에 있고, `exit(0)`가 프로그램을 끝낸다. 베어메탈은 이 "당연한 것"의 목록을 한 줄씩 지워간다. 처음엔 난감하고, 그다음엔 무엇이 남았는지를 찬찬히 확인하게 된다.

이번 장에서는 그 정리 작업을 한다. **`-ffreestanding` 플래그 한 줄이 무엇을 약속하고 무엇을 거두어 가는지**를 손에 잡히는 형태로 풀자. 다음 장에서 QEMU에 첫 글자를 찍기 위한 마음의 준비다.

## Hosted vs Freestanding — C 표준이 정의하는 두 환경

C 표준(ISO/IEC 9899)은 사실 두 가지 환경을 정의한다. **hosted implementation**과 **freestanding implementation**이다.

| 축 | Hosted | Freestanding |
|----|--------|--------------|
| 진입점 | `int main(void)` 또는 `int main(int, char**)` | 구현체가 정의 — 보통 `_start` |
| 표준 라이브러리 | 전체 — `<stdio.h>`, `<stdlib.h>`, `<string.h>`, ... | 일부만 — 타입·매크로 헤더 7~8개 |
| 동적 메모리 | `malloc`/`free` 사용 가능 | 직접 마련해야 함 (또는 안 쓴다) |
| OS 콜 | `read`/`write`/`open`/`exit` 등 가능 | 없음 — CPU 위에서 직접 돈다 |
| 빌드 환경 | 보통 OS 위 | CPU만 있는 곳 (베어메탈), 또는 다른 OS의 커널 자체 |

호스트 환경은 우리가 지금까지 살아온 곳이다. 운영체제가 코드 아래에 깔려 있고, 그 위에서 우리 프로그램이 동작한다. `main`이 끝나면 OS가 자원을 회수한다. 메모리 누수가 있어도 프로세스가 끝나면 OS가 정리한다.

freestanding 환경은 그 아래에 깔리는 OS가 없다. **우리가 그 OS의 자리에 있다.** 또는 OS 자체를 짜고 있다. 또는 OS 없이 도는 임베디드 펌웨어를 짜고 있다. 어느 쪽이든 위에서 어떤 가정도 빌려 올 수 없다.

`-ffreestanding` 플래그를 컴파일러에 건네면 두 가지가 바뀐다.

1. **컴파일러가 "표준 라이브러리가 있다"는 가정을 끈다.** `printf`가 정의돼 있다고 가정하고 부르거나, `memcpy`를 자동으로 끼워 넣거나 하지 않는다 (`-fno-builtin`을 같이 켜는 이유다).
2. **진입점이 `main`이라는 가정을 끈다.** `_start`든 `__entry`든 `boot`든 우리가 정한 심볼이 진입점이 될 수 있다.

`-nostdlib`은 한 걸음 더 간다. **링커가 표준 라이브러리를 안 끼운다.** `libc.a`도 `libgcc.a`(컴파일러 런타임 헬퍼)도 자동으로 안 붙는다. 부족한 게 있으면 우리가 직접 마련해야 한다. `-nostartfiles`는 거기에 더해 `crt0.o`(`_start`에서 `main`으로 가는 다리)도 빼라는 지시다.

이 셋을 합친 빌드 명령은 보통 이런 모양이다.

```bash
clang --target=aarch64-none-elf \
  -ffreestanding -nostdlib -nostartfiles \
  -mcpu=cortex-a72 -O2 -Wall \
  -c kernel.c -o kernel.o
```

여기서 한 가지 솔직한 진단을 해 두자. **`-ffreestanding`을 켰다고 자동으로 안전해지지는 않는다.** 그냥 컴파일러가 가정 몇 개를 끄는 것뿐이다. 그 가정의 빈자리를 어떻게 메우는지는 전적으로 우리 몫이다.

## 안전한 헤더, 위험한 헤더

freestanding에서도 쓸 수 있는 표준 헤더가 있다. C 표준이 명시한 목록은 다음과 같다.

| 헤더 | 무엇을 들고 있나 |
|------|---------------|
| `<stddef.h>` | `size_t`, `ptrdiff_t`, `NULL`, `offsetof` |
| `<stdint.h>` | `uint8_t`, `int32_t`, `uintptr_t` 등 고정폭 정수 |
| `<stdbool.h>` | `bool`, `true`, `false` (C23부터는 키워드라 헤더 필요 없음) |
| `<limits.h>` | `INT_MAX`, `CHAR_BIT`, `UINT_MAX` 등 한계값 |
| `<float.h>` | 부동소수 한계값 (`FLT_MAX` 등) |
| `<stdarg.h>` | `va_list`, `va_start`, `va_arg` (가변 인수 처리) |
| `<iso646.h>` | `and`, `or`, `not` 같은 키워드 별칭 (거의 안 씀) |

특징을 보면 한눈에 들어온다 — **다 "타입과 매크로" 헤더지 "함수" 헤더가 아니다.** 함수 호출이 끼어 있는 헤더는 자동으로 위험 영역이다. 표준 라이브러리가 없는 곳에서 그 함수를 부르면 링크가 깨지기 때문이다.

그렇다면 위험한 헤더는?

- `<stdio.h>` — `printf`, `fopen`, `fread`. 전부 호스트 OS에 의존.
- `<stdlib.h>` — `malloc`, `free`, `exit`, `system`. 동적 할당과 프로세스 제어 전부.
- `<string.h>` — `memcpy`, `strlen` 같은 함수. 흥미롭게도 **컴파일러가 자체 구현으로 인라인해 주는 경우도 있다** — 그래서 `-fno-builtin`을 같이 켠다. 직접 쓰고 싶으면 `mini_memcpy`/`mini_strlen`을 자기 손으로 짠다.
- `<time.h>` — 시간 함수. OS가 시간을 들고 있어야 동작한다.
- `<errno.h>` — `errno`라는 전역 변수가 어딘가 있어야 한다.

freestanding 코드는 위 헤더들을 **건드리지 않는다는 결심**으로 시작한다. 이 결심을 코드 레벨에서 강제하려면 빌드 시스템에 헤더 경로를 막아 두는 방법도 있다 — `-nostdinc -isystem <minimal-include-dir>`로 표준 헤더 디렉토리를 비표준 위치만 보게 묶어 두는 패턴이다. 다만 학습 단계에서는 그 정도로 엄격할 필요는 없다. 안 부르면 그만이다.

## 표준 라이브러리 없음 — 무엇을 직접 마련해야 하나

`printf`가 없다. `malloc`도 없다. `errno`도 없다. 이 셋 없이 어떻게 사는가? 직접 만들거나, 안 쓰거나, 둘 중 하나다.

### 출력 — `mini_printf`의 자리

베어메탈의 첫 출력은 `printf`가 아니라 **UART에 한 글자씩 쓰기**다. 12장에서 자세히 보겠지만, 코드의 모양은 이렇게 단순하다.

```c
#define UART0_DR (volatile uint32_t *)0x09000000UL

static void uart_putc(char c) {
    *UART0_DR = (uint32_t)(unsigned char)c;
}

static void uart_puts(const char *s) {
    while (*s) uart_putc(*s++);
}
```

UART 데이터 레지스터에 바이트를 쓰면 그 바이트가 시리얼 라인을 따라 호스트(QEMU 터미널, 또는 진짜 보드의 UART 핀)로 흘러간다. **OS가 끼어 있지 않다.** 우리 코드와 하드웨어 레지스터 사이에 다른 누구도 없다.

`printf`처럼 포맷 문자열을 받고 싶으면 직접 `mini_printf`를 짜야 한다. 핵심은 `<stdarg.h>`로 가변 인수를 풀고, `%d`/`%x`/`%s` 같은 변환 지시자를 손으로 처리하는 일이다. 100~200줄이면 쓸만한 `mini_printf`가 나온다. 책의 example 디렉토리에는 그 골격이 들어 있다.

### 메모리 — `malloc`의 자리

`malloc`은 부르지 않는다. 정적 변수(데이터/BSS 영역)와 스택만으로 사는 게 베어메탈의 첫 단계다.

```c
static char buffer[4096];   // .bss 또는 .data에 잡힌다
```

이런 정적 버퍼가 핵심 자원이 된다. 4KB든 4MB든 빌드 타임에 결정한 크기로 잡아 두고 쓴다. 동적 할당이 정 필요해지면 그때 **자기 할당자**를 짠다. 가장 단순한 형태는 bump allocator다.

```c
static uint8_t arena[1024 * 1024];  // 1MB
static size_t  arena_top = 0;

void *bump_alloc(size_t n) {
    if (arena_top + n > sizeof arena) return NULL;
    void *p = &arena[arena_top];
    arena_top += n;
    return p;
}
```

`free`는 없다. 또는 전부 한 번에 비우는 `bump_reset`만 있다. 단순한 만큼 쓸 곳도 좁지만, 부팅 초기에 페이지 테이블이나 인터럽트 벡터를 잡을 때처럼 "한 번 만들고 다시 안 푼다"는 자리에는 충분하다.

더 본격적인 할당자(freelist, slab, buddy)는 OS의 한 챕터가 되는 주제다. 우리 책의 범위는 아니다 — 그 자리는 15장의 "다음 걸음"에서 살짝 가리킨다.

### 에러 처리 — `errno`의 자리

`errno`는 호스트 libc가 들고 있는 전역 변수다. 베어메탈에는 없다. 그러면 에러는 어떻게 전달하나? 두 가지 정석이 있다.

1. **리턴값에 에러 코드를 실어 보낸다.** `int read_byte(uart_t *u, uint8_t *out)` — 성공 0, 실패 음수. 출력은 out 파라미터로.
2. **enum으로 에러 종류를 명시한다.** `typedef enum { OK, ERR_TIMEOUT, ERR_BAD_ARG } status_t;` — 가독성이 좋아지고 의도가 코드에 드러난다.

이 두 패턴은 호스트 코드에서도 권장되지만, 베어메탈에서는 **유일한 선택지**가 된다. 예외는 없다. 예외 던지기/잡기는 C에 원래 없고, C++에서도 베어메탈에서는 비활성화하는 게 보통이다.

## 런타임 초기화의 책임

호스트 환경에서는 `main`이 시작되기 전에 무대 준비가 끝나 있다. OS가 프로세스 메모리를 잡고, `crt0`가 `argv`/`argc`/`envp`를 세팅하고, 전역 변수들이 초기화되고, 표준 입출력 버퍼가 열린다. 우리는 `main`의 첫 줄부터 일하면 된다.

freestanding 환경에서는 **그 무대 준비를 우리가 한다.** 무엇을 해야 하나?

### `.bss` 클리어

C 표준은 약속한다 — "초깃값 없이 선언된 정적/전역 변수는 0으로 시작한다." 호스트에선 OS가 그 약속을 지킨다 (프로세스 메모리는 0으로 초기화된 페이지로 채워진다). 베어메탈에선 우리가 지켜야 한다.

`.bss` 섹션이 그 변수들이 잡히는 자리다. 부팅 직후 그 영역을 0으로 미는 코드를 우리가 짠다. 보통 `boot.S`에서 한다.

```asm
ldr     x0, =__bss_start
ldr     x1, =__bss_end
1:  cmp     x0, x1
    b.ge    2f
    str     xzr, [x0], #8        // 8바이트씩 0 쓰기, 포인터 자동 증가
    b       1b
2:
```

`__bss_start`와 `__bss_end` 심볼은 링커 스크립트가 정의해 준다. 이 한 토막이 없으면 우리의 `static int counter = 0;`이 0이 아닐 수 있다. 끔찍한 일이다 — 디버깅 한참 후에야 알아챈다.

### `.data` 복사

`static int x = 42;`처럼 초깃값이 있는 정적 변수가 들어가는 자리가 `.data`다. ROM에 굽는 펌웨어라면 `.data`는 처음엔 ROM에 있고 (LMA = Load Memory Address), 실행 시점에 RAM으로 복사돼야 한다 (VMA = Virtual Memory Address). 그 복사를 우리가 한다.

QEMU virt 보드처럼 처음부터 RAM에 적재되는 경우는 이 복사가 필요 없다 — LMA와 VMA가 같다. 그래서 우리 12장 예제에서는 `.data` 복사 코드를 생략한다. 진짜 보드(라즈베리 파이 4)로 옮길 때 이 일이 다시 나타난다 (14장).

### 스택 포인터 설정

CPU가 부팅 직후 가진 레지스터 값은 미정의다. **스택 포인터(`sp`)에도 쓰레기가 들어 있다.** 첫 함수 호출 한 줄을 하기 전에 우리가 `sp`를 유효한 메모리 영역으로 가리켜 줘야 한다.

스택 영역은 보통 링커 스크립트에서 잡는다.

```ld
. = ALIGN(16);
. = . + 0x10000;        /* 64KB 스택 */
stack_top = .;
```

그리고 `boot.S`에서 그 심볼을 `sp`에 싣는다.

```asm
ldr     x0, =stack_top
mov     sp, x0
```

이 두 줄이 빠지면 첫 `push`/`bl` 한 줄에서 우주가 끝난다. 그것도 조용히. UART에 글자가 안 찍히고, breakpoint도 못 거는 상태로 멍하니 멈춘다.

### 진입점 — `_start`와 `main`의 관계

호스트 환경에서 `main`은 사실 첫 함수가 아니다. `_start`(또는 `__libc_start_main`)가 먼저 돌고, 그 함수가 `main`을 부른다. 우리가 `main`을 짜면 컴파일러 + libc가 `_start`까지의 다리를 자동으로 끼워 준다.

freestanding 환경에서는 그 다리가 없다. `_start`를 우리가 직접 짠다. 그리고 `_start`에서 `kernel_main`(또는 우리가 정한 이름)을 부른다. 12장의 `boot.S`에 정확히 그 모양이 들어 있다.

```asm
.globl _start
_start:
    /* CPU 상태 정리 */
    bl    kernel_main      // C 코드로 점프
halt:
    wfe
    b     halt
```

`main`이라는 이름은 의미가 없어진다. 어떤 이름이든 좋다 — `kernel_main`, `boot_main`, `app_entry`, 우리 마음대로.

## 컴파일러 옵션 — 한 줄씩 의미

베어메탈 빌드의 컴파일러 옵션은 길어진다. 한 줄씩 의미를 짚어 보자.

```
-ffreestanding         # 표준 라이브러리 가정 끄기, main이 진입점이 아닐 수 있음
-nostdlib              # 링커가 libc.a/libgcc.a 자동 링크 끄기
-nostartfiles          # crt0.o 자동 끄기
-fno-builtin           # 컴파일러가 자체 memcpy/memset 인라인 끄기
-mgeneral-regs-only    # 부동소수/SIMD 레지스터 사용 끄기 (베어메탈 초기엔 FP 셋업 안 함)
-mcpu=cortex-a72       # 타깃 CPU 명시 — 라즈베리 파이 4
-fno-stack-protector   # 스택 보호기 끄기 (호스트 libc 의존)
-fno-asynchronous-unwind-tables   # 예외 unwind 정보 끄기 (C에선 어차피 안 씀)
-O2 -Wall -Wextra      # 평소 옵션 그대로
```

이 셋 중에서 **`-mgeneral-regs-only`**가 처음 보는 사람에겐 낯설 텐데, 이유는 명확하다. AArch64는 FP/SIMD 레지스터(V0~V31)를 별도로 들고 있고, 그것들을 쓰려면 부팅 직후 활성화 코드가 필요하다. 부팅 첫 단계에서는 그 셋업이 없으니, 컴파일러가 그 레지스터를 안 쓰도록 막아 둔다. 나중에 FP를 켤 준비가 되면 이 플래그를 풀어 준다.

`-fno-stack-protector`도 익숙해질 가치가 있다. 호스트 빌드의 stack smashing protector는 호스트 libc 안의 `__stack_chk_fail` 함수를 부른다 — 베어메탈엔 그 함수가 없으니 링크가 깨진다. 보호기를 끄거나, 직접 구현해서 끼우거나, 둘 중 하나다. 학습 단계에서는 끄는 게 단순하다.

## Sanitizer는 어디로 가나

9장에서 책의 빌드 매트릭스 중심에 둔 ASan/UBSan은 베어메탈에서 어떻게 되나? 답은 정직하다 — **그대로는 안 쓴다.**

ASan과 UBSan은 런타임 라이브러리(libclang_rt, libubsan)를 끼워서 동작한다. 그 런타임이 무엇에 의존하는가? 동적 메모리(`malloc`/`free`), 스레드(`pthread_mutex_*`), 시그널(`SIGSEGV` 처리), 표준 입출력(에러 메시지를 stderr에 찍음). **이 의존이 다 호스트 libc 위에 서 있다.** freestanding 빌드에 `-fsanitize=address`를 그냥 켜면 링크가 와르르 깨진다.

그래서 베어메탈 코드의 안전망은 **호스트 측 단위 테스트**로 우회한다. 같은 로직을 호스트 환경에서 빌드해서 ASan을 켜 테스트한다.

```
src/
  uart.c          ← 베어메탈에서 부른다
  uart.h
  ring_buffer.c   ← 베어메탈에서 부른다
  ring_buffer.h
test/
  test_ring_buffer.c   ← 호스트에서 ASan으로 빌드, ring_buffer.c와 함께
```

링 버퍼나 문자열 처리 같은 "OS 의존이 없는 로직"은 호스트와 베어메탈에서 똑같이 동작한다. 그 로직만 떼어내 호스트에서 ASan으로 두들겨 보고, 베어메탈에서는 같은 코드를 그대로 가져다 쓴다. 안전망을 거꾸로 매다는 셈이다.

베어메탈 자체의 동적 검사가 필요하면 **하드웨어 워치포인트**나 **MMU 페이지 보호**가 다음 자리에 들어온다. 이게 진짜 OS를 짤 때의 한 챕터다 — 우리 책 너머의 일이다.

## C++을 사실상 못 쓰는 이유

베어메탈에서 C++을 쓸 수 있냐는 질문이 자주 따라온다. 답은 "원칙적으로 가능하지만 매우 좁다"다.

- `new`/`delete`는 동적 할당에 의존 → 우리가 마련한 할당자에 연결해야 함
- 예외(`throw`/`catch`)는 런타임 unwind 코드에 의존 → 끄거나(`-fno-exceptions`) 직접 마련
- RTTI(`dynamic_cast`, `typeid`)는 런타임 타입 정보 테이블에 의존 → 끄거나(`-fno-rtti`) 직접 마련
- STL은 거의 다 동적 할당 사용 → 못 씀
- 전역 객체의 생성자는 `_start` 전에 자동 호출 → freestanding에서는 우리가 명시적으로 호출해야 함

이 모든 것을 정리하면 "C++이라기보다 C에 객체 문법만 더한 것"이 된다. 그럴 거면 처음부터 C로 가는 게 무난하다. 그래서 OS·임베디드의 베어메탈 코드는 압도적으로 C다. 가끔 Rust의 `#![no_std]`가 끼어들 뿐이다 (15장 후반에 다시 짚는다).

## 손에 잡히는 골격 — `libmini.a`

이번 장의 example 디렉토리에는 호스트에서 빌드되는 `libmini.a`가 들어 있다. `main` 없이, 라이브러리 하나만 나오는 빌드다.

```
example/ch11-freestanding/
├── mini.h         ← mini_memset, mini_memcpy, mini_strlen, mini_strcmp, mini_u64_to_dec 선언
├── mini.c         ← 위 함수들의 구현 — libc 호출 0
├── Makefile       ← -ffreestanding -nostdlib로 .a 만들기
└── (libmini.a)    ← 빌드 산물
```

`make verify`를 한 번 돌려 보자. `nm`이 라이브러리의 외부 의존을 보여 준다.

```
$ make verify
--- libmini.a 안의 정의된 심볼 (T = text/code) ---
0000000000000000 T _mini_memset
0000000000000094 T _mini_memcpy
0000000000000144 T _mini_memcmp
000000000000016c T _mini_strlen
0000000000000184 T _mini_strcmp
00000000000001b4 T _mini_u64_to_dec

--- libmini.a가 외부로 부르는 함수 (U = undefined) ---
비어 있으면 libc 의존이 0이라는 뜻 — 베어메탈에 그대로 가져다 쓸 수 있다.
```

"U" 줄이 비어 있다. **이 라이브러리는 외부로 누구도 부르지 않는다.** 호스트 빌드인데도, 진짜 freestanding의 모양을 갖췄다. 이 골격이 다음 장에서 부르는 베어메탈 코드의 출발점이 된다.

여기서 한 가지 흥미로운 사실이 나온다. `-fno-builtin`을 안 켜면 컴파일러가 우리의 `mini_memset`을 호출하는 코드를 보고 "어차피 같은 일이니 진짜 `memset`을 부르자"고 슬쩍 바꿔치기할 수 있다. 빌드된 라이브러리에 `U memset`이라는 줄이 등장한다 — 안 부른 줄 알았는데 부르고 있다. 그래서 베어메탈 빌드의 정석에 `-fno-builtin`이 들어가는 거다. 잊지 말자.

## 멘탈 시프트 — 무엇을 잃고 무엇을 얻었나

이 장의 핵심은 코드보다 마음가짐이다. 호스트에서 베어메탈로 옮겨가면서 **무엇을 잃고 무엇을 얻는지** 한 번 정리해 두자.

| 잃은 것 | 얻은 것 |
|---------|--------|
| `printf`의 편리함 | 한 글자가 화면에 닿기까지의 사슬에 대한 시야 |
| `malloc`의 무한해 보이는 메모리 | 메모리 한 바이트마다 보이는 출처 |
| OS가 챙겨주는 `.bss` 초기화 | 부팅 첫 30줄에 대한 완전한 통제 |
| `errno`와 표준 에러 메시지 | 리턴 코드로 일관된 에러 흐름 |
| 풍부한 표준 라이브러리 | "필요한 건 직접 만든다"는 결정권 |
| 빠른 빌드/실행 사이클 | 한 줄의 코드가 의미하는 것에 대한 무게감 |

이 표의 오른쪽 칸이 베어메탈의 보상이다. 왼쪽 칸이 비용이고. 두 칸을 정직하게 같이 보면, 베어메탈은 "더 어려운 호스트"가 아니라 **다른 종류의 일**임이 드러난다. 더 적은 추상으로 더 가까이에서 일한다. 손이 더 많이 가지만, 보이는 것이 명확해진다.

이걸 평생 하라는 얘기는 아니다. 호스트에서 일하는 게 대부분의 경우에 옳다 — 추상은 인간이 발명한 가장 강력한 도구 중 하나다. 다만 한 번쯤은 추상을 걷어낸 자리에서 일해 보는 경험이 다른 어떤 학습에서도 안 주는 감각을 준다. 베어메탈로 가는 이유는 거기에 있다.

## 마무리

이번 장을 한 줄로 요약하면 — **`-ffreestanding -nostdlib`은 "표준 라이브러리가 없다고 가정해 줘"라는 컴파일러에게 거는 한 마디다.** 그 한 마디가 작아 보여도 따라오는 책임이 크다. 진입점부터 `.bss` 클리어, 스택 셋업, 입출력, 에러 처리까지 — 호스트가 해 주던 일을 한 줄씩 우리가 메운다.

기억해 두자.

- **freestanding은 "기능이 적어졌다"가 아니라 "가정이 줄었다"다.** 그 가정을 어떻게 메울지가 우리 결정이다.
- **안전한 헤더는 타입·매크로 헤더뿐.** 함수가 들어 있는 헤더는 의식적으로 피하자.
- **`.bss` 클리어, 스택 셋업, 진입점 — 셋은 항상 우리가 한다.** 빠뜨리면 조용히 망가진다.
- **Sanitizer는 호스트 측 단위 테스트로 우회한다.** 같은 로직을 호스트와 베어메탈 양쪽에서 빌드할 수 있게 짜는 습관이 도움이 된다.
- **C++은 사실상 못 쓴다.** 베어메탈은 C의 자리다.

다음 장이 책의 정점이다. 네 개의 파일 — `boot.S`, `kernel.c`, `linker.ld`, `Makefile` — 이 어떻게 협력해서 QEMU 시리얼에 첫 글자를 찍는지, 큰 그림을 한 번에 그려 보자. 줄별 해부는 13장의 몫이지만, 12장이 끝나면 우리는 OS 없이 ARM CPU 위에서 직접 도는 코드를 손에 쥐게 된다.
