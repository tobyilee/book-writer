# 4장. vi를 떠나며 — 모던 에디터·IDE 고르기

한참 만에 C 소스 파일을 다시 열었다고 해보자. 화면에 `hello.c`가 떠 있고, 손은 무의식적으로 `:wq`를 친다. 그런데 옆자리 동료의 화면을 흘끗 보니 사정이 다르다. 마우스로 변수를 가리키니 정의가 팝업으로 뜨고, 함수 이름에 커서를 두고 단축키 하나로 호출 그래프가 펼쳐진다. 빌드를 누르면 따로 터미널을 열지 않아도 디버거가 정확히 그 줄에서 멈추고, 변수 값이 옆 패널에 나란히 줄을 선다.

가만히 보고 있자니 마음 한구석이 찜찜해진다. 우리가 vi에 머무는 동안 세상은 어디까지 갔단 말인가. 그렇다고 화려한 IDE 앞에서 새로 적응을 시작하기엔 손에 익은 vi 키바인딩이 너무 깊이 박혀 있다. 일단 모니터에서 손을 떼지 않고도 모든 게 끝나는 그 흐름이 익숙하다.

이 챕터는 그 사이 어딘가에 자리를 잡는 일에 관한 것이다. vi의 손맛을 완전히 포기하지 않으면서, 모던 도구가 가져다 주는 자동 완성·정의 점프·디버거 통합은 누릴 수 있는 지점이 어디인지. 결론부터 말하자면 답은 하나가 아니다. 그러나 "재시작용 기본값"은 분명히 있다.

## 결론을 먼저 — VS Code + clangd

복귀하는 사람에게 가장 권할 만한 조합은 **VS Code + clangd**다. 이유는 셋이다.

첫째, **무료다.** 라이선스 걱정 없이 회사 노트북·개인 노트북·서버 어디든 같은 환경을 깐다.

둘째, **clangd가 LSP 서버로 동작한다.** 자동 완성, 정의로 이동, 참조 찾기, 리네이밍, 인라인 진단 — IDE에서 기대하는 것이 거의 다 들어 있다. 그리고 clangd는 LLVM 프로젝트의 일부라 컴파일러와 같은 본바닥에서 자란 도구다. 검사 결과가 실제 빌드와 어긋날 일이 적다.

셋째, **베어메탈 cross-compile 환경에서도 그대로 동작한다.** 이 책의 후반부에서 우리는 `aarch64-elf-gcc`로 빌드되는 코드를 다룬다. 다른 많은 IDE가 호스트 컴파일러에 묶여 있는 반면, clangd는 `compile_commands.json` 하나만 있으면 어떤 툴체인의 코드든 인덱싱해준다. Apple Silicon에서 macOS 바이너리를 만들든, Mac에서 ARM ELF를 만들든, 같은 에디터가 그대로 인식한다.

물론 이게 모두에게 정답이라는 뜻은 아니다. 후보를 한 표로 정리하고, 누구에게 어떤 선택이 맞는지 솔직히 말해 보자.

### 후보 비교

| 도구 | 강점 | 약점 | 어울리는 사람 |
|------|------|------|----------------|
| **VS Code + clangd** | 무료, 빠름, 확장 자유, cross 환경 친화 | 디버거 통합이 약간 손 감 (CodeLLDB) | **재시작 기본값** |
| **CLion** | C/C++ 전용 1급 도구, 리팩토링·정적 분석 최강 | 유료, 인덱싱 무거움 | 어느 정도 익숙해진 뒤 큰 프로젝트로 |
| **Xcode** | Apple Silicon 네이티브, Instruments 동봉 | 베어메탈 cross 워크플로엔 거의 무용 | iOS/macOS 앱 만드는 사람 |
| **Neovim + clangd + DAP** | vi 근육 그대로, 초고속, SSH 친화 | 설정 비용 큼 | vi 손맛을 못 놓는 사람 |
| **Zed** | Rust로 작성된 신예 초고속 에디터 | C 생태계 통합이 아직 얕음 | 가볍게 시도해 볼 만 |
| **Cursor** | VS Code 포크 + AI 통합 | 클라우드 의존, 베어메탈에서 자주 미끄러짐 | AI 짝코딩 선호자 |

이 표에서 가장 솔직해야 하는 칸은 **"어울리는 사람"**이다. 적당히 모두에게 다 맞는다고 둘러대고 싶지만, 그건 결국 누구에게도 안 맞는다는 말이 된다. 그러니까 한 명씩 짚어 보자.

**Xcode는 좋은 도구다. 하지만 우리 책의 워크플로엔 거의 안 맞는다.** 이 책의 정점은 Mac 위에서 도는 macOS 바이너리가 아니라, ARM CPU 위에서 OS 없이 도는 ELF 바이너리다. Xcode의 프로젝트 모델은 처음부터 끝까지 Apple 플랫폼을 가정한다. 베어메탈 ELF를 만들겠다고 Xcode 프로젝트를 비틀어 짜기 시작하면 손가락이 굳는다. 다만 메모리 프로파일링이 필요할 때 **Instruments**를 별도로 띄워 쓰는 건 충분히 가치 있다. 에디터로는 빠지고, 분석 도구로는 합류한다 — 이 정도의 분업이 좋다.

**CLion은 한 단계 위로 올라가는 선택이다.** 자작 커널의 코드가 수천 줄을 넘기고, 리팩토링과 디버거 GUI의 도움이 절실해지는 시점에 CLion으로 옮겨도 늦지 않다. 그러나 시작부터 CLion으로 들어가면 라이선스 비용도 그렇고, 인덱싱 대기 시간도 그렇고, 첫 몇 챕터의 호흡이 무거워진다. 처음엔 VS Code, 익숙해진 뒤 CLion — 이 사다리가 자연스럽다.

**Cursor는 매력적이지만 함정도 분명하다.** AI 짝코딩의 흡인력은 크다. 그러나 베어메탈 코드는 AI가 가장 자주 미끄러지는 영역 중 하나다. 인라인 어셈블리, 링커 스크립트, MMIO 비트 조작 — 이런 영역에선 학습 데이터의 절대량이 적어서 LLM이 그럴듯한 헛소리를 늘어놓을 가능성이 높다. 그럴듯한 헛소리만큼 디버깅이 끔찍한 일은 없다. 호스트 측 일반 C 코드까지는 좋지만, 이 책의 후반 챕터에서는 AI 자동 완성을 잠시 끄고 자기 머리로 짠 코드만 통과시키는 편이 안전하다.

**Neovim은 vi 손에서 손을 떼고 싶지 않은 독자를 위한 진지한 선택이다.** clangd LSP 클라이언트(`nvim-lspconfig`)와 디버거 어댑터 프로토콜(`nvim-dap`)이 안정되어 있어서, 마음먹고 설정만 잡으면 IDE에 거의 근접하게 끌어올릴 수 있다. 다만 그 "설정만 잡으면"이라는 단서가 만만치 않다. 처음 며칠은 코드를 짜는 시간보다 에디터를 길들이는 시간이 길다. 그래도 그 길을 자기 힘으로 한 번 닦으면 일생을 같이 가는 도구가 된다.

이 책은 그래서 본문 예제를 VS Code + clangd를 기준으로 설명한다. 단, Neovim 사용자도 같은 `compile_commands.json`을 그대로 쓸 수 있도록 코드 구조를 짠다. 도구가 둘이어도 인덱싱의 진입점은 하나다.

## VS Code + clangd 셋업

자, 손을 더럽혀 보자. 먼저 세 가지를 깐다.

```bash
brew install llvm                 # clangd, clang-tidy, clang-format, lldb
# VS Code는 https://code.visualstudio.com 또는 `brew install --cask visual-studio-code`
```

그다음 VS Code에서 확장 세 개를 깔자.

- **clangd** (게시자: `llvm-vs-code-extensions` — vscode-clangd)
- **CodeLLDB** (게시자: `vadimcn` — Rust로 짠 LLDB 프런트엔드)
- **CMake Tools** (게시자: Microsoft)

여기서 한 가지 함정이 있다. Microsoft가 공식으로 푸는 **`C/C++` 확장**(게시자: `ms-vscode`)이 있는데, 이게 clangd와 같은 자리에서 다툰다. 두 개를 동시에 켜면 진단 표시가 두 번 뜨고, 자동 완성이 어느 쪽에서 오는지 헷갈리고, 점프가 가끔 엉뚱한 곳으로 간다. 정말 끔찍한 일이다.

해결책은 단순하다. **MS C/C++ 확장은 끄고, clangd 하나만 둔다.** clangd의 README에도 같은 경고가 박혀 있다. 두 확장이 같은 워크스페이스에서 동시에 동작하면 동작이 미정의가 된다(는 말은 좀 과장이지만 체감상 그렇다).

확장이 깔렸으면 마지막 한 조각, **`compile_commands.json`**이 필요하다. 이게 clangd에게 "어떤 컴파일러 옵션으로 어떤 파일을 빌드하라고 지시했는지" 알려 주는 데이터베이스다. 이게 없으면 clangd는 헤더 경로도 모르고, `-std=` 플래그도 모르고, 결국 `<stdio.h>`조차 못 찾는다고 빨갛게 줄을 긋는다. 정말 답답하다.

이 파일을 만드는 길은 셋이다.

1. **CMake 한 줄.** `cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON` 이면 `build/compile_commands.json`이 자동으로 떨어진다. 프로젝트 루트로 심볼릭 링크를 걸어 두면 clangd가 바로 찾는다.
2. **Bear.** `brew install bear` 후 `bear -- make` 하면 평범한 Make 빌드를 가로채서 컴파일 명령을 모두 기록해 준다. Makefile만 있는 작은 프로젝트에 잘 어울린다.
3. **손으로 짓는다.** 파일이 두 개·세 개뿐인 미니 프로젝트라면 그냥 JSON을 직접 쓰는 게 가장 단순하다. 이 챕터의 예제가 그 방식을 보여준다.

세 길 중 무엇을 골라도 좋다. 다만 `compile_commands.json`이 어떻게 생겼는지는 한 번쯤 직접 들여다보는 편이 낫다. 그래야 어느 날 clangd가 갑자기 헤더를 못 찾는다고 우길 때, 자기 손으로 한 줄을 고쳐 살릴 수 있다.

### 예제 — geom 미니 라이브러리

이 챕터의 예제 디렉터리(`example/ch04-editor-setup/`)에는 작은 라이브러리가 들어 있다. 헤더(`geom.h`) 하나와 소스(`geom.c`) 하나, 그리고 그것을 쓰는 `main.c` 하나다. 굳이 작게 만든 이유가 있다. 첫째, IDE 셋업을 검증하기엔 작을수록 좋다. 둘째, 헤더와 소스가 분리된 구조는 다음 챕터에서 다룰 "선언과 정의의 분리"의 가장 작은 표본이다.

`geom.h`는 2D·3D 벡터 길이와 정규화 함수만 선언한다.

```c
typedef struct { double x, y; } vec2_t;
typedef struct { double x, y, z; } vec3_t;

double vec2_length(vec2_t v);
double vec3_length(vec3_t v);
vec2_t vec2_normalize(vec2_t v);
vec3_t vec3_normalize(vec3_t v);
```

`geom.c`는 `<math.h>`의 `sqrt`로 길이를 구하고, 정규화는 길이로 나눈다. 길이가 0인 입력은 그대로 돌려보내 0으로 나누는 사고를 막는다.

빌드와 실행은 `make && make run`이다. 출력은 단순하다.

```
|a| = 5.000
|b| = 3.000
a/|a| = (0.600, 0.800)
```

`(3, 4)`의 길이가 `5`로 나오면 학교 다닐 때의 직각삼각형이 떠올라 잠깐 반갑다. 그러나 이 예제의 진짜 목적은 거기가 아니다. **에디터에서 `vec2_length`라는 식별자에 커서를 두고 `F12`(또는 `gd`)를 눌렀을 때, `geom.c`의 정의로 정확히 점프하는가** — 이 한 가지를 확인하는 일이다.

만약 점프가 동작하지 않거나 자동 완성이 비어 있다면, clangd가 헤더 경로를 못 찾고 있다는 뜻이다. 이 예제는 그 점검을 손쉽게 만들기 위해 `make compdb`라는 작은 헬퍼를 둔다.

```bash
make compdb
```

이 한 줄이 같은 디렉터리에 `compile_commands.json`을 만든다. 내용을 열어 보면 두 줄짜리 JSON이다 — 각각 `geom.c`와 `main.c`를 어떤 옵션으로 빌드하는지가 적혀 있다. 이게 clangd 인덱싱의 출발점이다.

이 단계가 끝나면 한 가지 자잘한 확인이 남는다. 같은 디렉터리에 둔 `.clangd` 파일이다.

```yaml
CompileFlags:
  Add: [-Wall, -Wextra, -Wpedantic, -std=c23]

Diagnostics:
  UnusedIncludes: Strict
```

이건 clangd가 인덱싱할 때 추가로 어떤 옵션을 적용할지를 지시한다. `compile_commands.json`이 이미 옵션을 적어 두지만, 거기에 빠진 경고를 보강하거나 미사용 `#include`를 잡아내는 등의 강화 설정을 여기에 둔다. 작은 파일이지만 에디터 안에서 느끼는 깐깐함이 한 단계 올라간다.

### compile_commands.json 잠깐 들여다보기

여기서 한 번 호흡을 가다듬고, 우리가 방금 만든 `compile_commands.json`이 어떻게 생겼는지 직접 살펴보자.

```json
[
  {
    "directory": "/Users/me/.../ch04-editor-setup",
    "file": "geom.c",
    "command": "clang -std=c23 -Wall -Wextra -Wpedantic -O0 -g -c geom.c"
  },
  {
    "directory": "/Users/me/.../ch04-editor-setup",
    "file": "main.c",
    "command": "clang -std=c23 -Wall -Wextra -Wpedantic -O0 -g -c main.c"
  }
]
```

별 거 없다. 항목 하나가 곧 컴파일 명령 하나다. `directory`는 그 명령을 어디서 실행했는지, `file`은 무엇을 빌드했는지, `command`는 어떤 옵션으로 빌드했는지. clangd는 이 JSON을 읽어 "아, 이 파일은 C23 표준으로, 이 경로에서, 이 경고와 함께 컴파일되는구나"를 알아낸다. 그 정보로 코드를 분석한다.

이게 단순한 만큼 강력하다. CMake가 깔려 있지 않아도, Make 빌드가 복잡하지 않아도, 이 한 JSON만 있으면 clangd가 동작한다. cross 컴파일러로 빌드되는 코드도, 호스트 컴파일러와 cross 컴파일러가 섞인 코드도, 각 파일이 어떤 컴파일러로 어떤 옵션과 함께 빌드되는지 그 명령줄만 그대로 적어 두면 clangd는 곧이곧대로 이해한다.

이 책의 후반에서 우리가 베어메탈 코드를 다룰 때, 호스트용 테스트 코드와 ARM 베어메탈 코드가 한 프로젝트에 섞여 있을 것이다. 그때도 `compile_commands.json` 하나면 충분하다. 한 파일은 `clang -target arm64-apple-darwin ...`으로, 다른 파일은 `aarch64-elf-gcc -ffreestanding ...`으로 빌드된다고 적어 두면 clangd가 알아서 두 그룹을 따로 인덱싱한다.

### 빌드 시스템 — Make에서 CMake로, 또는 그 자리에 머물기

`compile_commands.json` 이야기가 자연스럽게 빌드 시스템 이야기로 흘러간다. 이 책의 예제는 의도적으로 **Make**를 기본으로 쓴다. 왜냐하면 베어메탈 영역의 거의 모든 튜토리얼이 Make 기반이고, Makefile 한 장을 처음부터 끝까지 읽을 수 있는 사람과 그렇지 않은 사람의 학습 속도가 크게 갈리기 때문이다.

그러나 호스트 측 프로젝트가 커지기 시작하면 Make로 모든 걸 처리하기엔 손이 시리다. 그때 자연스럽게 사다리에 오를 도구가 **CMake**다. CMake의 가치는 한 줄로 요약된다 — `cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON` 한 줄이면 빌드 디렉터리에 `compile_commands.json`이 떨어진다. clangd가 더 이상 손을 탈 일이 없다.

다른 빌드 시스템도 있다. **Meson**은 Python 기반이고 GNOME·systemd 쪽이 즐겨 쓴다. **Ninja**는 백엔드 실행기로, CMake/Meson이 Ninja 빌드 파일을 만들어 그 위에서 돈다. 모두 좋은 도구지만, 처음엔 Make에서 시작해서 프로젝트가 자라면 CMake로 옮기는 정도가 가장 자연스러운 사다리다.

이 책이 베어메탈 챕터에서도 끝까지 Make를 고수하는 이유는 단순하다. 베어메탈 코드는 컴파일러 호출과 링커 호출이 적나라하게 드러나야 학습 가치가 있다. CMake가 그 호출을 추상해 가리면, 부팅 코드가 어떻게 만들어지는지 손에 들어오지 않는다. 그래서 두 영역을 의도적으로 분리한다 — 호스트 코드가 커지면 CMake로, 베어메탈 코드는 Make로.

## 디버거 — lldb로 충분하다

코드를 짜다 보면 단순히 인쇄 디버그로는 한계가 오는 순간이 온다. 변수가 어느 순간 0이 되었는지, 어느 함수가 그 변경을 만들었는지, 스택은 지금 어떻게 쌓여 있는지 — 이런 질문에는 디버거가 답한다.

macOS의 정통 디버거는 **lldb**다. Xcode 토대 위에 자라서 Apple Silicon에서 코드 서명 문제 없이 그대로 동작한다. gdb를 옛 추억으로 기억하는 사람도 있겠지만, Apple Silicon에서 gdb는 코드 서명을 직접 만지지 않으면 디버깅 권한을 못 얻는다. 손이 많이 가는 길이다. lldb로 가는 편이 낫다.

명령은 gdb와 닮았지만 미묘하게 다르다.

```bash
lldb ./geom_demo
(lldb) breakpoint set --name main      # 또는 b main
(lldb) run                              # 또는 r
(lldb) step                             # 또는 s
(lldb) print v                          # 또는 p v
(lldb) bt                               # backtrace
(lldb) quit
```

gdb의 `b`/`r`/`s`/`p`가 거의 그대로 통한다. 다만 `print` 출력 형태나 컨테이너 들여다보기는 lldb 쪽이 더 풍부하다.

CodeLLDB 확장을 깔아 두었으면 굳이 터미널에서 lldb를 칠 일도 많이 줄어든다. VS Code 왼쪽의 디버그 패널에서 "Run and Debug" 버튼을 누르고, 첫 실행 때 launch 설정을 LLDB로 잡아 주면 된다. 변수 패널, 워치 패널, 호출 스택이 한 화면에 줄을 선다. 이 단계가 손에 익으면 `printf` 디버깅 시간이 절반으로 준다.

`.vscode/launch.json`은 처음 한 번만 짜 두면 그 다음부터는 잊고 쓴다. geom 예제용 최소 설정은 이렇게 생겼다.

```json
{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Debug geom_demo",
            "type": "lldb",
            "request": "launch",
            "program": "${workspaceFolder}/geom_demo",
            "args": [],
            "cwd": "${workspaceFolder}",
            "preLaunchTask": "build"
        }
    ]
}
```

`preLaunchTask: "build"`는 디버그를 시작하기 전에 빌드 작업을 한 번 돌려라는 뜻이다. 그 빌드 작업은 `.vscode/tasks.json`에 `make`를 호출하도록 적어 두면 된다. 한 번 짜 두면 F5 한 번에 빌드와 디버거 시작이 묶여 돈다. 일과의 마찰이 한 단계 줄어든다.

한 가지 당부할 점이 있다. **디버거 빌드는 반드시 `-O0 -g`로 한다.** 최적화가 켜 있으면 컴파일러가 변수를 레지스터에 잡아 두고 어딘가에 합쳐 버려서, 디버거가 "이 변수는 최적화로 사라졌다"고 매정하게 답한다. 정말 끔찍한 일이다. 이 챕터의 예제 Makefile이 처음부터 `-O0 -g`로 묶여 있는 이유가 거기에 있다.

## Neovim 분기 — vi 손이 그리운 독자에게

이 절은 짧다. Neovim으로 가는 길은 그 자체로 한 권의 책이 필요한 주제고, 이 책의 범위가 아니다. 그러나 손이 자꾸 `:wq`로 가는 독자를 위해 출발점만 짚자.

요지는 셋이다.

1. **clangd LSP 클라이언트를 깐다.** `nvim-lspconfig` 또는 `mason.nvim`을 통해 `clangd`를 LSP 서버로 등록하면 자동 완성·정의 점프가 같은 식으로 동작한다. `compile_commands.json`을 보는 눈은 VS Code와 똑같다.
2. **디버거는 `nvim-dap` + `nvim-dap-ui`로.** lldb를 어댑터로 잡으면 GUI에 가까운 디버깅을 터미널 안에서 누릴 수 있다. CodeLLDB와 같은 프로토콜이다.
3. **lazy.nvim**으로 플러그인을 모듈화한다. 한 파일에서 모두 끄고 켜기 좋다.

진지하게 이 길을 갈 거라면 `LazyVim`이나 `AstroNvim` 같은 잘 다듬어진 배포판으로 출발하는 편이 낫다. 자기 발판부터 짜겠다는 마음이 강하면 처음부터 직접 짜는 것도 좋고. 어느 쪽을 가든, 첫 며칠은 코드보다 설정 파일을 더 자주 본다는 사실은 각오하자.

## 첫 디버깅 세션 — geom 예제로 한 바퀴

말로만 설명하면 손에 안 잡힌다. 우리 예제로 한 바퀴 돌아 보자.

먼저 `geom.c`의 `vec2_length` 함수에 중단점을 건다. 에디터에서 그 함수의 `return` 줄 옆 여백을 클릭하면 빨간 점이 박힌다. F5를 누른다. 빌드가 한 번 돌고, 디버거가 떠서 그 빨간 점에서 멈춘다.

왼쪽 변수 패널에 `v.x`와 `v.y`가 보인다. `v.x = 3.0`, `v.y = 4.0`. 여기서 한 단계 더 — 워치 패널에 `v.x * v.x + v.y * v.y`를 적어 넣으면 `25.0`이 뜬다. `sqrt(25.0)`이 5.0이라는 사실은 우리가 짠 코드의 출력에서 이미 봤지만, 디버거 안에서 한 단계씩 변수의 값이 변해 가는 모습을 직접 보는 건 다른 경험이다.

F10(스텝 오버)을 누르면 한 줄씩 진행된다. F11(스텝 인)을 누르면 `sqrt` 같은 호출 안으로 따라 들어간다 — 다만 `<math.h>`의 `sqrt`는 시스템 라이브러리라 소스가 없어서 어셈블리로 떨어진다. 굳이 거기까지는 안 가도 좋다.

호출 스택 패널을 보면 우리가 지금 어디 있는지가 한눈에 들어온다. `main → vec2_length → sqrt`. 각 줄을 클릭하면 그 호출 시점의 변수 상태로 옮겨가 본다. 이게 디버거가 주는 가장 큰 선물이다 — 코드의 어느 한 순간을 동결시켜 둘러볼 수 있는 능력.

처음엔 어색하다. 그러나 한 번 손에 익으면 "어, 이 변수가 갑자기 NULL이네"라는 의문에 1분 안에 답을 찾을 수 있다. printf 디버깅으로는 10분이 걸리는 일이다. 시간 차이가 누적되면 학습 곡선의 기울기가 달라진다.

## 자잘한 함정 몇 가지

마지막으로, 셋업하다가 한 번씩은 부딪히는 자잘한 함정을 모아 둔다. 미리 알고 가면 시간을 한참 아낀다.

- **`#include <stdio.h>`에 빨간 줄.** 거의 항상 `compile_commands.json` 부재 또는 위치 문제다. 프로젝트 루트나 빌드 디렉터리에 있는지 확인하고, `.clangd`의 `CompilationDatabase` 설정을 본다. 그래도 안 되면 한 번 VS Code를 재시작해 보자.
- **두 LSP가 동시에 진단을 띄운다.** MS C/C++ 확장이 켜져 있을 가능성이 높다. clangd 하나만 남기자.
- **Mac에서 디버거가 권한이 없다고 한다.** lldb는 Xcode CLT가 깔리면 권한 문제가 거의 없지만, 외부에서 받은 바이너리(예: 다른 사람이 보낸 `.elf`)는 코드 서명 문제로 디버그가 막힐 수 있다. 자기 컴파일러로 만든 바이너리는 안전하다.
- **헤더가 너무 많은 프로젝트.** clangd 인덱싱이 처음에 한참 돈다. 인덱스가 한 번 완성되면 그 다음부터는 빠르다. 첫 인덱싱 시간은 커피 한 잔을 마실 정도로 두고 보자.
- **베어메탈 코드를 인덱싱하려는데 헤더를 못 찾는다.** `compile_commands.json`에 cross 컴파일러의 `-isysroot` 또는 `-I` 옵션이 그대로 들어가 있어야 한다. 이 책 후반에서 베어메탈 빌드를 다룰 때 다시 짚는다.

## 마무리

vi의 손맛이 완전히 사라지진 않는다. VS Code 안에서 `Ctrl+Shift+P`로 "Vim emulator"를 검색해 봐도 좋고, Neovim으로 한 발 더 깊이 들어가도 좋다. 우리가 떠나는 건 vi의 정신이 아니라 vi만으로 모든 걸 해결하던 시절의 단조로움이다.

이제 손에는 두 가지가 들렸다. 코드를 짤 에디터(VS Code + clangd 또는 Neovim + clangd), 그리고 멈춰 보고 들여다볼 디버거(lldb 또는 CodeLLDB). 이 둘이 있으면 다음 챕터부터 만날 모든 예제를 자기 손으로 따라 실행하고, 막히는 부분에서 정확히 멈춰 변수 값을 들여다볼 수 있다.

다음 장에선 도구의 손에서 떨어져, 다시 C 본질의 한복판으로 들어간다. 첫 손님은 **포인터**다. 옛 K&R 시절의 흐릿한 기억이 Java의 객체 참조와 Python의 변수 이름표 사이 어딘가에서 길을 잃어 있다면, 그 자리를 다시 정돈하자. 별로 무섭지 않다 — 다섯 시나리오면 손이 다시 따뜻해진다.
