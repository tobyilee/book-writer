## 잘된 점 (3~5)

1. **레퍼런스의 핵심 차별화 포인트가 챕터 단위로 정확히 매핑됨.** 카카오페이 정산 4단계(refs §4.3) → 6장 backbone, `JobOperator#recover()`(§3.8) → 7장+10장, graceful shutdown(§3.9) → 5장+10장, ChunkOrientedStepBuilder 재설계(§3.3) → 5장. 레퍼런스가 강조한 6의 변경점이 사실상 누락 없이 챕터로 흡수됐다.

2. **3막 구조의 동기부여가 분명하다.** "왜 굳이 프레임워크냐 → 모델 이해 + 6 변경점 → 운영·마이그레이션"의 흐름이, 대상 독자(이미 직접 짜본 사람)의 회의를 1막에서 정면으로 받아주고 마지막에 운영 자신감으로 닫는다는 약속이 명확하다. 특히 1장 도입부 훅(새벽 3시 SQL을 짜는 장면)은 독자 진입 상태와 정확히 맞다.

3. **챕터 도입부 훅이 모두 한국 1차 자료에 묶여 있다.** 1·3·4·6·7·10·11장 훅이 OKKY 글번호, 카카오페이/우아한형제들 사례, GitHub Discussion #4532 등 구체적 출처에 매여 있어 "감정적 공감 표현"이 추상적 수사로 흘러가지 않는다. 토비 스타일 챕터 작성가가 그대로 받아 쓸 수 있는 형태.

4. **5장(6 변경점)을 4장(모델) 다음에 둔 배치 결정이 옳다.** 모델을 모르면 변경의 의도가 안 보인다는 판단이 합리적이고, 5장 1번(타임라인)부터 9번(fault-tolerance 재구성)까지 9개 소절이 레퍼런스 §3.1~§3.20을 누락 없이 커버한다.

5. **리서치 한계를 챕터에 명시적으로 반영했다.** 252줄~259줄에서 SCDF·JFR·MongoDB JobRepository·Boot 4 통합의 한계를 어느 챕터에서 어떻게 처리할지 한 줄씩 적어둔 것은 정직한 운영. 독자에게 "여기는 신기능 소개 톤"이라고 신호 주는 의도가 챕터 본문에서도 살아날 가능성이 높다.

## 개선 필요 사항 (우선순위 순)

### Critical (반드시 반영)

**C1. [3장] @StepScope 함정이 어디에도 없다.**
레퍼런스 §7.1이 "한국 입문자가 자주 부딪히는 문제 6개" 중 하나로 `@StepScope`를 명시적으로 들었고(OKKY #380648 인용), 1장 도입부도 "직접 짠 배치가 부딪히는 5가지 문제"를 약속한다. 그런데 12개 챕터 어디에도 `@StepScope` 절이 없다. JobParameters를 빈에 주입하는 사실상 유일한 표준 메커니즘이고, 늦은 빈 생성·NPE는 한국 커뮤니티 빈출 함정이다.
- **제안:** 3장 7번("흔한 실수") 옆에 `@StepScope` 절을 신설하거나, 4장에 "Reader/Writer를 JobParameters로 매개화하기" 절을 추가해 거기서 `@StepScope`를 다룬다. 12장 안티패턴 체크리스트에도 한 줄 추가.

**C2. [5장] 도메인 모델 record화·primitive long 전환·v5 직렬화 비호환이 5장에서 빠지고 11장에만 있다.**
레퍼런스 §3.15 "도메인 모델 불변성·JobParameters 재설계"는 6의 메이저 의미 변화 중 하나(엔티티 ID `Long`→`long`, JobParameter record화, JobParameters를 `Set<JobParameter>` 기반으로 재설계, **v5 실패 인스턴스의 v6 재시작 불가**)인데, 5장 9개 소절에서 이걸 다루는 항목이 없다. 11장 2번에서 호환성 경고만 짧게 나온다.
- **제안:** 5장에 "도메인 모델·JobParameters 재설계" 소절(10번)을 추가. 5장에서 "왜 record로 바뀌었나" 의도를 보여줘야 11장의 마이그레이션 경고가 납득된다. 5장 분량 38쪽 → 40쪽 선에서 흡수 가능.

**C3. [12장] 안티패턴 9개가 본문 챕터들과 이미 중복된다 — 새로운 가치를 못 만든다.**
12장 1번이 나열한 9개 안티패턴(트랜잭션 미설계 / 메모리 폭주 / 비멱등 Writer / cursor + 외부 API / 멀티스레드 + stateful Reader / JobParameters 미스 / 자동 실행 충돌 / graceful shutdown / 메타 누적)은 모두 4·6·7·8·10장에서 이미 다룬 것이다. "압축 체크리스트"로 자리매김했지만, 20쪽 챕터를 떠받치기엔 약하다. 자칫 본문 회상록이 된다.
- **제안 A (스코프 축소):** 12장을 14쪽으로 줄이고, 안티패턴 체크리스트 + 다른 도구 비교 + 다음 학습 로드맵에만 집중. 줄어든 6쪽은 5장(C2 흡수)과 11장(아래 S1)에 분배.
- **제안 B (스코프 변경):** 12장을 "운영 1년차 회고 — 5번째 잡을 만들 때 보이는 패턴"으로 톤 전환. 단순 안티패턴 나열이 아니라 "처음엔 모든 잡에 retry 걸었는데 1년 지나니 실패 분류 기준이 바뀌었다" 같은 시간 축의 회고로. 토비 스타일 에세이형 마무리에 더 어울린다.

**C4. [전체] chunk-oriented vs tasklet 비중을 9:1로 잡았는데, tasklet이 어디서 1을 차지하는지 챕터에 안 보인다.**
레퍼런스 §1.3은 명시적으로 "9 대 1 비중으로 chunk-oriented가 본류"라고 했고, 2장 6번에서도 그렇게 약속한다. 그런데 tasklet의 1할 자리 — 파일 압축, 디렉터리 정리, 외부 명령 실행 — 이 어디서도 실제로 다뤄지지 않는다. 4장은 ItemReader/Processor/Writer만, 6장 5가지 유스케이스도 모두 chunk 모델이다.
- **제안:** 6장 마지막에 "유스케이스 6: Tasklet이 어울리는 일들"을 짧게 (3쪽 정도) 추가. 또는 4장 끝에 "tasklet이 필요한 순간" 절을 둔다. 약속한 비중을 본문에서 지켜야 신뢰가 생긴다.

### Should (반영 권장)

**S1. [11장] 마이그레이션 11개 소절이 26쪽에 다 들어가지 못한다 — 분량이 부족하다.**
11장은 5→6 마이그레이션 11개 소절(체크리스트, 패키지 이동, JobLauncher→JobOperator, EnableBatchProcessing 분리, modular deprecated, 시퀀스 이름 변경, fault-tolerance 재작성, ChunkOrientedTasklet 전환, Jackson 2 정리, 단계적 전략) 각각이 코드 변환 예시까지 보여줘야 의미가 있다. 26쪽으로는 항목당 2.4쪽 — 매우 빠듯하다. 차별화 포인트로 내건 챕터(특성 4번)가 주마간산 위험.
- **제안:** 11장을 30~32쪽으로 늘리고, 12장에서 6쪽을 가져온다(C3 제안 A와 짝). 또는 11장 11번("단계적 마이그레이션 전략")을 잡 한 개의 실제 마이그레이션 시나리오(코드 diff 4~5개 + 메타데이터 검증 SQL)로 풀어 분량 정당화.

**S2. [9장] 알람 설계 부분이 얇다.**
9장 7번("알람 설계: 잡 실패율, Step 평균 처리 시간 편차, Skip 누적 추세")이 한 줄로 처리됐는데, 이건 운영 챕터의 핵심 결과물이다. 카카오페이 사례가 "잡이 1시간 → 2시간이 됐다"는 도입부 훅과 직접 연결되는 답이 9장 7번이어야 한다.
- **제안:** 9장에 "알람 임계값을 어떻게 정하는가 — 정상 범위 + 추세 + 파편적 실패"를 한 절로 분리하고, PromQL 표현식 2~3개 예시 제공. 분량 26쪽 → 28쪽.

**S3. [6장] 유스케이스 5개가 한 챕터(35쪽)에 욱여넣어졌다 — 평균 7쪽이면 진짜 backbone은 카카오페이뿐이고 나머지는 스케치가 된다.**
정산 사례(카카오페이)가 4단계 개선 + 코드 diff까지 가면 단독으로 12~15쪽 필요하다. 그러면 ETL·통계·파일 ETL·대용량 일괄은 평균 5쪽으로 압축된다 — 이건 결정 트리 카드 수준이지 backbone이 아니다.
- **제안 A:** 6장을 정산 사례 단독 챕터(20쪽)로 축약하고, 나머지 4개 유스케이스를 "유스케이스 패턴 카탈로그" 부록(15쪽)으로 분리. 본문 챕터 수는 11+부록.
- **제안 B:** 6장을 그대로 두되 "정산 = backbone, 나머지 4개 = 결정 트리 카드"라는 구조를 챕터 도입부에서 명시. 독자가 기대를 보정할 수 있게.

**S4. [7장] outbox 패턴을 한 줄로 처리하지 말 것.**
7장 7번(외부 부수효과 outbox)은 "송금이 두 번 나갔다"는 운영 트라우마를 전제하는 절인데, 한 줄로 끝난다. 레퍼런스 §6.4가 "단순 멱등으로는 부족 — outbox 테이블, 멱등 키, dedupe 저장소 같은 추가 장치 필요"라고 명시했음.
- **제안:** outbox + 멱등 키 + dedupe 저장소 3가지를 비교하는 작은 표 + 송금 잡 시나리오 코드 예시 1개. 7장 32쪽 안에서 흡수 가능.

**S5. [10장] K8s 챕터의 모니터링·알람 부분이 9장과 모호하게 겹친다.**
10장 9번 "실 운영 체크리스트"가 "graceful shutdown, recover 연결, 정리 잡, 모니터링, 알람"을 나열하는데, 모니터링·알람은 9장의 영토다. "K8s 환경에서만 의미 있는 모니터링 결정"(예: Pod 단위 메트릭, K8s Event과 JobExecution 상태 매칭)이 아니면 9장으로 보낸다.
- **제안:** 10장 9번을 "운영 워크플로우 자동화 — recover API + 정리 잡을 K8s 리소스로 표현하기"로 좁히고, 모니터링은 9장 참조로 처리.

### Nice-to-have (선택)

**N1. [2장] 30쪽 vs 1장 22쪽 — 입구가 무겁다.**
2장이 가장 두꺼운 입문 챕터(1·2·3장 평균 27쪽 중 30쪽)인데, 첫 잡을 띄우기 전에 모델 설명만 30쪽 읽는 것은 대상 독자(이미 직접 짜본 사람)에게 답답할 수 있다.
- **제안:** 2장 6번(chunk vs tasklet)·7번(직접 짠 코드 매핑)을 3장 또는 4장에 분산하면 24쪽 선으로 정리되고, 3장과의 자연스러운 흐름이 살아난다.

**N2. [전체] "직접 짠 코드 → Spring Batch로 매핑" 시각이 2장에서만 등장하고 사라진다.**
2장 7번이 "내가 짜던 그 부분이 여기로 옮겨진다"고 약속하는데, 이 매핑 시각이 4·5·7장에서도 살아나면 책 전체가 일관된다("내가 짜던 retry 코드가 6에선 이렇게 들어간다", "내가 SQL로 NOT IN 했던 그 자리가 ExecutionContext가 한다").
- **제안:** 각 챕터 끝에 "직접 짰을 때 vs Spring Batch 6" 작은 박스 코너 추가. 책 전체에 일관된 시각적 모티브가 생긴다.

**N3. [2장] JobInstance vs JobExecution을 그림 없이 글만으로 설명하면 입문자가 막힌다.**
레퍼런스 §2.2의 JobInstance:JobExecution = 1:N 관계는 텍스트만으로는 매번 헷갈리는 부분.
- **제안:** 2장에 다이어그램 사용을 챕터 노트에 명시. (책 산출 시점에 텍스트 기반 도식 또는 ASCII art로 처리 가능)

**N4. [12장] 다른 도구 비교 표가 3·4·5번에 분산되어 있다.**
Airflow(3), Quartz/SCT(4), SCDF/SCT 학습 주제(5)가 따로 노는데, "어떤 문제엔 무엇을 쓰는가" 한 표로 통합하는 게 독자에게 더 쓸모 있다.
- **제안:** "스케줄링 vs 잡 신뢰성 vs DAG 오케스트레이션 vs 단발성 작업" 4축 결정 표 1개로 통합.

## 추가 권장 토픽

다음 항목은 레퍼런스에 있고 대상 독자에게 가치 있는데 계획에서 누락 또는 비중이 낮다.

1. **`@StepScope` / `@JobScope` 사용법과 함정** (C1과 동일) — 4장 또는 3장에 한 절.
2. **테스트 전략** — 레퍼런스 §3.20에서 "JUnit 4 지원 deprecated"가 나오고, `spring-batch-test` 자체를 어떻게 쓰는지(JobLauncherTestUtils, MetaDataInstanceFactory)가 한국 자료에 빈약하다. 12장 또는 7장에 "잡을 어떻게 테스트할 것인가" 한 절.
3. **JobRepository 정리 잡 자체의 구현** — 10장 5번이 "정기 정리 잡 필요"라고 말하지만 그 정리 잡을 Spring Batch로 짜는 예시(메타→메타) 자체가 흥미로운 사례다.
4. **CommandLineJobOperator로 외부 스케줄러 통합** — 5장 8번에서 신규 CLI 도구만 언급하는데, K8s CronJob의 `command:`에 어떻게 들어가는지가 10장과 직접 연결된다.
5. **금융 정산 reconciliation 패턴** — 레퍼런스 §4.6이 다중 소스 ingest + intelligent matching + mismatch 처리 3요소를 명시했는데, 6장 정산은 카카오페이 성능 사례에 집중하느라 reconciliation 자체의 패턴이 빠진다. 이벤트 소싱 결합 시 60% 단축 보고도 인용 가치 있음.

## 전체 평가

**등급: B+**

**한 줄 이유:** 레퍼런스의 핵심 자산(카카오페이 4단계 + 6 변경점 + K8s 운영 + 마이그레이션)을 챕터 단위로 정확히 가져왔고 3막 구조의 동기부여가 분명하다 — 다만 (1) `@StepScope`·테스트·tasklet 등 약속과 본문 사이의 누락과 (2) 6장(유스케이스 5개)·11장(마이그레이션 11항목)·12장(중복) 세 챕터의 분량/스코프 부정합이 남아 있어 A로 가려면 한 라운드 더 손봐야 한다.

**A로 가는 길:** Critical 4개 모두 반영 + Should 5개 중 최소 3개 반영. 특히 **C1(@StepScope), C3(12장 재설계), S1(11장 분량 확장)** 이 세 가지가 핵심.
