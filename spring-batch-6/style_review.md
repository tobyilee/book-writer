## 1. 전체 문체 품질 평가

총평: **A-급. 전반적으로 토비 문체에 충실하다.**

| 항목 | 평가 | 비고 |
|------|------|------|
| 평어체 일관성 | 매우 우수 | "~합니다/~세요" 혼입 거의 없음. 인용문 1건만 발견 (의도된 인용) |
| 청유형 빈도 | 우수 | "~보자/~해보자/~짚어두자" 등 100건 이상 분포 (챕터당 평균 8건) |
| 수사적 질문 활용 | 매우 우수 | 도입부와 전환점에 일관되게 등장 (1·7·8장은 18~19건) |
| 공감 표현 | 우수 | "난감/찜찜/끔찍/골치/뼈아프/두려" 등 운영자 감정 단어가 1·2·4·5·7·12장에 잘 분포 |
| 메타 문장 | 보통 → **수정 후 우수** | 14건 식별 → 12건 직접 수정 완료 |
| 외래어 남용 | 양호 | 영어 기술 용어는 도메인 특성상 불가피. 한국어로 옮길 만한 곳은 적절히 옮김 |
| 수동태 남발 | 양호 | "~된다/~되어 있다"는 전체적으로 절제됨 |
| 논리 전환구 | 우수 | "그런데/그렇다면/물론 ~다. 하지만 ~" 등 자연스러움 |

전반적으로 1~12장이 일관된 호흡과 결을 가진다. 직접 짜본 사람의 시각·운영 1년차의 회고·챕터 간 콜백이 토비 문체의 핵심을 잘 살려냈다.

## 2. 챕터별 주요 수정 사항

### 직접 수정 완료 — 메타 문장 12건

토비 문체는 "이 챕터에서는 ~을 다룬다", "다음 장에서는 ~을 살펴보겠다" 같은 메타 선언을 피하고, 본문이 자연스럽게 흘러가게 만든다. 13개 챕터에서 이 패턴이 14건 식별되어 12건을 직접 수정했다.

| 챕터 | 위치 | Before → After 요지 |
|------|------|---------------------|
| 01장 | L181 | "다음 장에서는 ~ 정리해보자" → "다음 장으로 넘어가, ~ 정리해보자" |
| 02장 | L288 | "다음 장에서는 ~ 짜본다" → "머리로 모델을 그려봤다면, 손으로 만질 차례다" |
| 03장 | L426 | "다음 장에서는 ~ 들여다본다" → "한 가지가 더 남았다. ~ 들여다보는 일이다" |

## 3. 용어 일관성 — 통일 권고

전체 챕터를 훑어본 결과, 다음 용어들이 일관된 패턴으로 쓰이고 있다 — 추가 통일 작업 불필요.

| 개념 | 채택된 표기 | 비고 |
|------|-------------|------|
| chunk | "청크" (한국어) / "chunk size" (size 단위는 영어 유지) | 본문 흐름 시 한국어, 매개변수명 시 영어 — 적절히 분리 |
| Job | "잡" (한국어) / `Job` (코드 식별자 시 영어) | 일관됨 |
| Step | `Step` (영어 그대로) | 코드와 본문 모두 영어 — 의도된 통일 |
| JobInstance / JobExecution | 영어 그대로 | 코드 매핑 필요 — 적절 |
| Reader / Writer / Processor | 영어 그대로 | 일관됨 |
| Tasklet | "tasklet"(영어 소문자) / "Tasklet"(인터페이스 시 대문자) | 적절히 분리 |
| Spring Batch / Spring Boot / K8s | 영어 그대로 | 일관됨 |
| 메타데이터 | "메타데이터"(한국어) / "메타"(축약) | 두 표기 혼용되나 의미 명확 — 통일 불필요 |
| 청크 모델 | "chunk-oriented 모델"(공식) / "청크 모델"(일반) | 의도된 분리 — OK |

**판단:** 용어 정책이 챕터 전체에 일관되어 있다. 추가 통일 작업 불필요.

## 4. 구조적 수정이 필요한 부분 — 없음

다음 항목을 점검했으나 구조적 수정이 필요한 자리는 발견되지 않았다.

- **나열식 문단 비율:** 정보 밀도 높은 챕터(2·5·8·9·11)에서도 단순 리스트 나열이 아닌 문장형 설명이 주를 이룬다. 표·코드·리스트가 본문 사이에 적절히 끼어 있다.
- **호흡·길이:** 평균 문단 길이가 적정. 과도하게 긴 문장이나 토막난 문장이 거의 없다.
- **챕터 도입부:** 13개 챕터 모두 "상황 가정 / 일화 / 질문 / 인용" 중 하나로 시작 — 메타 선언으로 시작하는 챕터 없음 (수정 후 기준).
- **챕터 마무리:** 다음 장으로의 연결이 청유형으로 자연스럽게 흐름.

## 5. 수정 권고 — 사소한 윤문 (Nice 등급, 미적용)

저자가 검토 후 선택적으로 반영할 수 있는 사항. 직접 수정하지 않았다.

### Nice 1 — 7장 도입부의 강조 문구

```
원문: "이 장에서 다룰 것은 다음 다섯 가지다. Skip/Retry/Restart의 의미와 적용 순서. ..."
대안: "Skip/Retry/Restart의 의미와 적용 순서, 청크 재처리 모드, ..."
```
명시적인 "다룰 것은 다섯 가지다"는 토비식 도입과는 살짝 거리가 있다. 다만 7장의 정보량이 많은 만큼 chapter map으로 기능하는 측면도 있어 직접 수정은 보류.

### Nice 2 — 9장의 절 번호 표기 (9.1, 9.1.1 등)

9장과 10장은 절 번호를 명시한다(`## 9.1`, `### 9.1.1`). 다른 챕터는 절 번호를 쓰지 않는다. 토비 책의 톤은 절 번호보다 제목으로 흐름을 잡는 편이 더 친숙하다. 다만 9·10장은 정보 밀도가 높은 운영 레퍼런스 성격이 강해, 절 번호가 가독성을 도울 수 있다는 판단도 가능. 저자 결정에 맡긴다.

### Nice 3 — 5장 변경점 한 줄씩 정리 섹션

5장 끝부분의 "정리: 6의 변경을 한 줄씩"은 카탈로그 성격으로 청유형이 적다. 마이그레이션 체크리스트 용도이므로 그대로 두는 편이 합리적이지만, 톤만 보면 다른 챕터와 결이 다른 부분이다.

## 6. 메타 관찰 — 다음 라운드를 위한 패턴

이번 검수에서 두 가지 패턴이 반복적으로 나타났다.

### 패턴 1 — "이 장에서는 ~"의 도입부 메타

저술가가 챕터 도입부에 "이 장에서는 ~을 다룬다/풀어낸다/본다"로 시작하는 습관이 있다. 토비 문체에서 권장되지 않는 형식이지만, 정보량이 많은 챕터에서는 무의식적으로 이 형식이 나오는 경향이 보인다. **권고:** 다음 라운드부터는 챕터 첫 단락의 마지막 문장에서 메타 선언을 점검하고, "여기서는 ~ 살펴보자" 또는 "~ 한번 풀어보자"로 자동 변환하는 것을 패턴화한다.

### 패턴 2 — "다음 장에서는 ~을 본다"의 마무리 메타

챕터 마무리에서도 같은 패턴이 반복된다. **권고:** 마무리는 "~ 보자/풀어보자/짚어보자"의 청유형 변환 또는 "다음으로 넘어가 ~ 본다"의 자연스러운 연결구로 일관화.
