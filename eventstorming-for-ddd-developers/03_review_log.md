# Plan Review Log

## 리뷰 요약

전반적으로 단단한 계획이다. 특히 "워크숍 → 코드 → 팀" 3막 구조가 청중의 자기 진단(이미 코드를 짤 줄 안다)을 정확히 짚었고, 5~8장의 "코드의 방"이 책의 척추로서 reference의 가장 풍부한 영역(§5·§6)을 그대로 가져왔다. 가장 강한 점은 색상 grammar를 Spring 토큰으로 환생시키는 일관된 매핑 약속이다. 가장 약한 점은 (a) planner가 자체 신고한 챕터 9의 근거 빈약, (b) 챕터 11이 너무 많은 안티패턴·도구·관점을 한 챕터에 욱여넣은 비대칭, (c) 챕터 5와 6 사이 Aggregate→Bounded Context 전이 지점에서 독자가 "Aggregate를 잡았는데 왜 다시 모듈을 자르지?"라는 인지적 점프를 겪을 위험이다.

전체 분량은 12장 × 평균 10,000자 = 약 12만 자로 일관되며, 부록 3개의 역할 분담도 적절하다. 다만 챕터 7(13,000자)과 챕터 9·10(9,000~10,000자)의 무게 차이가 reference 근거의 두께 차이를 정직하게 반영한다는 점은 좋은 신호이자, 동시에 챕터 9의 보강 필요성을 입증한다.

## 5개 축별 평가

### 1. 청중 정합성

- **좋은 점:**
  - 제목 후보 A("Spring 개발자를 위한 EventStorming")는 청중의 검색어와 정확히 일치한다. 후보 B/C를 "톤"으로 흡수하는 추천 전략도 영리하다.
  - 1장의 자기 진단 장치(컬리 회고 + 오토피디아 실패담)가 청중의 출발선을 정확히 짚는다. "DDD 책 1~2권 읽었지만 폴더 구조는 그대로"라는 한 줄은 청중의 거울이다.
  - 12장 결말에서 "월요일 아침"으로 착륙하는 약속은 실용서 청중에게 결정적이다.

- **우려:**
  - 5~8장은 "Spring + React/Next + DDD 진심" 청중에게 정확히 꽂히지만, 1~4장은 *일반 DDD 입문자에게도 통할 수준*으로 일반화되어 있다. 특히 2장(색상 grammar)과 3장(Big Picture 6단계)은 Brandolini 요약에 가깝다. 청중이 "이건 영어책 번역과 뭐가 다른가"를 묻기 쉬운 구간.
  - 5장 후반의 Vernon 4규칙 + Servienti 관점은 *DDD 책 4~5권 읽은 시니어*에게는 익숙한 영역이다. 중급 독자(*DDD 책 1~2권 통과* 기준선)에게는 새롭지만, 시니어에게는 review에 가까울 수 있다 — 청중의 두 끝단(중급 vs 시니어) 사이 정합성이 조금 흔들린다.
  - 8장 BFF/GraphQL 논쟁은 책의 본류(EventStorming → 코드)에서 약간 빗겨난 곁가지로 보인다. Spring 백엔드 중심 청중이라면 *RSC + React Query*만으로도 충분하고, BFF/GraphQL 논쟁은 부록 C 옆에 짧은 사이드바로 빼는 게 본문 밀도를 높인다.

- **권고:**
  - 2장과 3장에 각각 *Spring 프로젝트의 OrderService 폴더 구조*를 한 페이지짜리 도입 시나리오로 박아라. 색상 grammar 설명 전에 "지금 너의 `OrderService.java` 안에 무엇이 있는가"부터 물어, 청중이 책 첫 30페이지 안에 자기 코드와 sticky를 겹쳐 보게 만든다.
  - 5장은 청중의 두 끝단을 다 잡기 위해 *"이미 Vernon을 읽은 독자는 5.4를 건너뛰어도 좋다"* 류의 명시적 reading guide를 챕터 첫머리에 한 줄 넣어라. 시니어 독자에게 정직하게 페이스를 양보하는 것이 신뢰를 만든다.

### 2. 내러티브 아크

- **좋은 점:**
  - 3막 구조(워크숍 → 코드 → 팀)가 명료하고, 각 막 안에서도 점진적 심화(2장 색상 → 3장 Big Picture → 4장 Process Modeling)가 자연스럽다.
  - 4장 끝에서 5장으로 "Big Picture는 발견, Software Design은 결정"이라는 격(格)의 차이를 약속한 것이 좋은 다리다.

- **우려:**
  - **5장 → 6장의 인지적 점프가 가장 위험하다.** 5장은 "Aggregate를 발견해 JPA 클래스로 내려보내기"인데, 6장은 다시 한 단계 위로 올라가 "Pivotal Event로 Bounded Context를 자르고 Spring Modulith 모듈로 옮긴다"이다. 독자 입장에서는 "방금 Aggregate를 잡았는데 왜 다시 모듈 경계를?"라는 순서 역행 감각이 생긴다. 사실 reference §3.2의 매핑상 Bounded Context가 Aggregate보다 *상위 결정*이므로, 6장이 5장 *앞*에 와야 자연스럽다.
  - 9장(Hotspot → 백로그)은 3막(팀의 방)의 첫 챕터인데, 직전 8장(Read Model/프론트)에서 *코드 흐름*의 절정을 찍은 다음 갑자기 "리팩토링 backlog"로 전환된다. 막 사이 전환부의 호흡이 짧다.
  - 11장(facilitator 함정)과 12장(한국 팀)이 모두 "현장의 실패담"을 다루어 톤이 겹친다. 11장이 일반화된 안티패턴이라면 12장은 한국 사례 — 그러나 독자 입장에서 두 챕터 모두 "사례·교훈" 색채라 끝부분이 약간 늘어진다.

- **권고:**
  - **챕터 5와 6을 순서 교체**하라. 6장 "Pivotal Event → Bounded Context → Spring Modulith module"을 먼저 두고, 그 안에서 발견된 모듈 하나를 5장 "Aggregate → JPA"가 줌인하는 구조가 reference §3.1의 흐름(Big Picture → Process Modeling → Software Design)과 정확히 일치한다. 현재 순서는 reference의 시퀀스를 거스른다.
  - 9장 도입부에 8장과의 다리("코드를 만들었으니 이제 워크숍의 핫핑크를 PR로 옮길 차례다") 한 단락을 명시적으로 박아라.

### 3. 챕터 균형

- **좋은 점:**
  - 평균 10,000자 ± 2,000자 안에서 대체로 일관된다. 7장(13,000자)이 가장 무거운 것은 reference §5의 두께가 그만큼이어서 정직한 반영이다.
  - 부록 A(Spring Modulith 코드 템플릿)가 7장의 코드 무게를 분담해 본문 흐름을 가볍게 만들어준다.

- **우려:**
  - **11장이 비대칭으로 비대하다.** 7개 안티패턴 + Baas-Schwegler/Deep Democracy + Evans Whirlpool + Example Mapping/Domain Storytelling + EventStorming vs Event Modeling — 한 챕터에 다섯 개의 독립된 주제가 들어와 있다. 11,000자로는 각각이 한 단락 수준으로 압축되어 "코드 한 조각이 살아 있기" 약속을 지키기 어렵다.
  - **9장 분량 추정(10,000자)이 reference 근거(§4.3, 약 1.5페이지) 대비 크다.** planner 본인이 신고한 약점과 일치한다.
  - 2장(10,000자)에서 "9개 색상 + alpha-omega 문장 + Visible Legend + Java 타입 비유 + 색상 강박 + 코드 스니펫"을 다 다루기엔 분량이 빠듯하다. 색상 강박은 11장에서 다시 등장하므로 2장에서는 가볍게 언급만.

- **권고:**
  - 11장을 둘로 쪼개라. 11A "facilitator의 함정(개발자가 빠지는 7가지)"은 안티패턴에만 집중하고 ~8,000자로 줄인다. EventStorming vs Event Modeling 비교와 Whirlpool은 분리해 11B 또는 12장 직전의 짧은 막간 챕터(또는 부록 D)로 옮긴다. 또는 11장은 안티패턴 7개로 줄이고, vs Event Modeling은 **부록 D 또는 12장의 한 절**로 흡수한다.
  - "코드 한 조각이 살아 있느냐"를 점검해라 — 챕터 10(원격), 11(facilitator), 12(한국 팀)는 본질상 워크숍/조직 챕터라 코드 스니펫 약속이 불가능하다. 이건 약속을 깰 것이 아니라, 책 서두에 "후반부는 코드보다 사람 얘기다"를 명시적으로 약속하면 된다.

### 4. 커버리지 vs 중복

- **좋은 점:**
  - reference의 12개 섹션이 챕터에 대체로 골고루 배치되었다. §5(Spring 생태계)와 §6(프론트)이 5~8장에 두텁게, §9(한국 사례)가 1장·12장에 자기 진단/결말 양쪽으로, §7·§8이 10·11장에 자연스럽게 들어갔다.
  - 부록 C(Brandolini ↔ Spring 용어집)가 책 전반에 흩어진 매핑을 한 표로 묶어주는 좋은 장치다.

- **우려:**
  - **5.5 Hexagonal Architecture (Tom Hombergs)가 챕터 배치에서 사실상 빠졌다.** reference §5.5는 "책의 후반부 척추가 될 수 있다"고 자평했는데, 정작 챕터 목록에 Hexagonal/Ports & Adapters를 본격 다루는 자리가 없다. 7장에서 한 단락 정도 언급할 수는 있어도, 한 챕터 또는 한 섹션의 비중은 받아야 할 주제다.
  - **2장 색상 강박 ↔ 11장 색상 강박 안티패턴** 중복. 같은 주제를 두 챕터가 다룬다. 2장은 grammar 설명 맥락, 11장은 facilitator 안티패턴 맥락이므로 완전 중복은 아니나, 같은 예시를 두 번 보면 독자가 데자뷔를 느낀다.
  - **11장 vs 8장의 "관점 충돌" 처리** — 8장에서 Rodrigo Estrada의 BFF 반대 시각, 11장에서 Servienti·Evans 등 다양한 관점을 묶는 방식이 일관되지 않다. 책 전체에 "관점 충돌"을 어떻게 표기할지 한 가지 시각적 컨벤션(예: "다른 시각" 박스)을 잡으면 reader friction이 줄어든다.

- **권고:**
  - **5장 또는 6장에 "5.x Hexagonal Architecture로 패키지 구조 잡기" 절(약 2,000자)을 추가**하라. Domain은 plain POJO, port·adapter로 자르는 컨벤션은 이 책 청중에게 사실상 필수다. reference §5.5의 EventStorming 색깔 ↔ Port/Adapter 매핑(Blue→Inbound port, Pink→Outbound adapter)을 그대로 가져오면 된다.
  - 2장의 색상 강박은 *grammar 설명을 위한 짧은 예시* 한 줄로 줄이고, 본격 처방은 11장에 양보한다.

### 5. 표적 점검: 챕터 9 "Hotspot → 백로그"

- **진단:**
  - planner 본인의 자체 신고가 정확하다. reference §4.3은 "원서가 명시적으로 다루지 않는 부분 — 책의 차별화 포인트"라 적었지만, 실제 자료는 Tornhill 1권 + Feathers 1권 + 추상적 매핑(워크숍 hotspot ∩ Git churn) 정도다. 한국 Spring 프로젝트의 *구체 사례* 인용이 reference에 없다 — §9에 등장하는 사례들(컬리·우아한형제들·카카오·오토피디아) 모두 1장과 12장에 이미 배정되어 있고, 어느 것도 "워크숍 hotspot → PR로 옮긴 사례"라는 정확한 각도로 인용 가능한 자료가 아니다.
  - 그럼에도 *개념적 다리*로서 이 챕터의 가치는 크다. 청중이 "워크숍을 돌리고 나면 다음 월요일에 뭘 하지?"라는 가장 절박한 질문에 답하는 자리다. 이 자리를 비워둘 수는 없다.

- **옵션 평가:**
  - **(a) 강화한다:** 추가 리서치로 한국 Spring 모놀리스의 hotspot 리팩토링 사례 1~2개를 보강해야 한다. 시간·리서치 비용이 가장 크다. 그러나 보강에 성공하면 책의 차별화 포인트로 가장 강해진다.
  - **(b) 부록 D로 빼고 9장은 transitional로 축소:** 9장을 ~5,000자의 짧은 다리 챕터(개념 + 워크숍 hotspot ↔ 코드 hotspot 매핑 그림 한 장 + arrow voting)로 줄이고, "Sprout Method로 Aggregate 떼어내기" 같은 구체 코드 시나리오는 부록 D로 빼는 방식. 위험 부담이 가장 낮다.
  - **(c) 다른 챕터로 흡수:** 9장 내용을 11장(안티패턴) 또는 12장(한국 팀)에 흡수. 단점은 9장의 *위치적 가치*(8장 코드 → 12장 팀의 다리)가 사라진다는 점. 흐름이 끊긴다.

- **권고: (b) — 9장을 transitional 챕터로 축소.**
  - 이유 1: planner의 자체 신고가 솔직했고, reference 보강 없이 10,000자를 쓰면 *추측에 기댄 사례*가 들어갈 위험이 크다. 그건 책 전체의 신뢰를 깎는다.
  - 이유 2: 9장의 *위치적 역할*(8장 ↔ 10·11·12장 사이 다리)은 충분히 살아 있다. 짧은 챕터로도 그 역할은 가능하다.
  - 이유 3: Sprout Method 코드 시나리오를 부록 D로 빼면, *코드를 약속한 챕터*(5~8장)와 *워크숍/조직을 약속한 챕터*(9~12장)의 정체성이 더 명확해진다.
  - 구체 권고: 9장을 ~6,000자로 축소하고, Sprout Method 실전 시나리오는 부록 D로 분리. 후속 판본에서 한국 사례 인터뷰가 확보되면 9장을 다시 늘리는 것을 명시한다. 또는 (a)를 선택할 거라면, 챕터 저술 *전에* community-research를 1라운드 더 돌려 한국 Spring 팀의 hotspot 리팩토링 사례를 보강하는 비용을 감수해야 한다.

## 권고된 수정 사항 (우선순위 순)

1. **[Critical] 챕터 5와 6의 순서 교체.** 6장(Pivotal Event → Bounded Context → Spring Modulith) → 5장(Aggregate → JPA) 순서로 바꾼다. reference §3.1의 시퀀스(Big Picture → Software Design)와 일치하고, 독자의 인지 순서(큰 경계 → 작은 단위)와도 맞는다.
2. **[Critical] 챕터 9를 transitional로 축소(~6,000자) + 부록 D 신설.** Sprout Method 코드 시나리오를 부록 D로 빼고, 9장은 워크숍 hotspot ↔ 코드 hotspot의 *개념적 다리*에 집중한다.
3. **[Should] 챕터 11을 슬림화.** 7개 안티패턴에 집중하고, EventStorming vs Event Modeling 비교는 12장 직전 짧은 막간 또는 부록으로 분리. ~8,000자로 줄인다.
4. **[Should] Hexagonal Architecture를 5장 또는 6장에 한 절(~2,000자) 추가.** reference §5.5의 매핑(Blue→Inbound port, Pink→Outbound adapter)을 그대로 본문에 박는다.
5. **[Should] 챕터 8의 BFF/GraphQL 논쟁 비중 축소.** RSC + React Query 매핑에 집중하고, BFF/GraphQL은 사이드바 또는 부록 C 옆 짧은 절로 옮긴다.
6. **[Nice-to-have] 2장·3장에 "Spring 프로젝트의 OrderService 폴더" 도입 시나리오 추가.** 청중이 책 첫 30페이지 안에 자기 코드와 sticky를 겹쳐 보게 한다.
7. **[Nice-to-have] 5장 첫머리에 reading guide 한 줄.** Vernon을 이미 읽은 시니어 독자에게 5.4를 건너뛸 자유를 명시.
8. **[Nice-to-have] "관점 충돌" 시각적 컨벤션 도입.** Servienti·Estrada·Frankel 등 반대 시각이 등장하는 모든 자리에 일관된 박스/표기.

- **챕터 12의 결말 약속 ("월요일 아침")** — 실용서 청중에게 결정적 착륙점이다. 톤·구조 모두 유지.
- **제목 후보 A 선택 + B/C를 톤으로 흡수하는 전략** — 마케팅과 책의 영혼을 모두 잡았다.
- **부록 C 용어집** — Brandolini ↔ Spring 매핑을 한 표로 묶는 발상은 이 책의 핵심 가치를 가장 압축적으로 보여주는 장치다. 분량(3,000자) 그대로 유지.
