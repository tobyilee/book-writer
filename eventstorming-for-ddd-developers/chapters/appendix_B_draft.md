# 부록 B. Miro/Mural EventStorming 보드 세팅

10장에서 짚은 도구 선택과 ground rule은 *왜·언제·어떻게 분리할지*에 대한 이야기였다. 부록 B는 *어떻게 실제 보드를 세팅하는지*만 짧게 본다. 첫 원격 워크숍을 잡는 facilitator가 *전날 밤 30분*에 따라 할 수 있는 체크리스트로 정리한다.

## Miro 보드 세팅 — 단계별 체크리스트

**1단계. 보드 만들기.** Miro에 로그인하고 *New Board*. 무료 플랜에서도 3개 보드까지 무료다. 첫 시도라면 무료로 시작하자. 보드 이름은 *"{팀명}-{도메인}-bigpicture-{날짜}"* 정도가 깔끔하다 — 6개월 뒤에 다시 열어볼 때 찾기 쉽다.

**2단계. 공식 템플릿 골라넣기.** Miro 검색창에 *"Event Storming"*을 치면 두 개의 공식 템플릿이 뜬다. *Event Storming Process Modelling*과 *Event Storming Software Design*. 첫 워크숍이라면 *Process Modelling*을 골라 *Import*한다. swimlane frame과 색상 sticker 팔레트가 미리 깔린 상태로 보드가 열린다. 이걸 *그대로 쓰지는 않는다* — 다음 단계에서 우리 도메인에 맞게 조정한다.

**3단계. swimlane 골격 깔기.** 보드 왼쪽 끝에서 오른쪽 끝까지 *4~5개의 frame*을 가로로 깐다. 각 frame이 한 *phase*를 의미한다. 예를 들어 주문 도메인이라면 *주문 접수 → 결제 → 출고 → 배송 → 정산* 다섯 단계. 이게 *Anticipate Structure* 패턴의 실전 적용이다(10장 참조). frame 제목만 적어두고, 안은 비워둔다. 워크숍 시작할 때 사람들이 그 frame *안에* sticky를 붙이기 시작한다.

**4단계. legend(범례) 박기.** 보드 *왼쪽 상단*에 색상 legend frame을 하나 만든다. 9가지 색의 sticker를 각각 한 장씩 박고, 옆에 *Domain Event / Command / Actor / Policy / External System / Aggregate / Read Model / Hotspot / Opportunity* 라벨을 붙인다. 워크숍 중 참가자가 *이거 무슨 색이지?*가 헷갈릴 때 그 자리를 가리키면 된다. Brandolini가 *"Visible Legend"*를 그렇게 강조한 이유가 원격에서 더 크다 — 화면 너머에서는 *물어볼 자리*가 그 frame이다.

**5단계. sticker 팔레트 준비.** 보드 *왼쪽 하단*에 색상별 sticker 묶음을 한 두 장씩 *미리 깔아둔다*. 참가자가 sticker를 *복사해서* 자기 자리로 옮긴다. 매번 *Add → Sticky note → Color 변경* 클릭을 안 거쳐도 되니 마찰이 줄어든다. 한국 팀에서 워크숍 초기 *클릭 마찰*이 발화 의지를 깎는다 — 이걸 줄이는 작은 디테일이 큰 차이를 만든다.

**6단계. voting 도구 준비.** 보드 *오른쪽 하단*에 파란 *arrow sticker*를 사람 수만큼 *× 2* 깐다(arrow voting용). Brandolini의 *arrow voting*은 워크숍 끝의 의식이고(원서 ch.4 §1881), 원격에서도 그대로 작동한다. 사람들이 자기 *블루 화살표*를 hotspot에 끌어다 붙인다.

**7단계. 권한 설정.** *Share* 버튼 → *Edit* 권한을 참가자 전원에게 준다. *Comment* 권한만 주면 sticky를 못 붙인다. 외부 도메인 전문가가 회사 계정이 없는 경우 *Anonymous edit*을 켜야 한다. 이건 보안 정책상 막힌 회사가 많으니 *전날* 확인해두자. 워크숍 당일 *권한 문제로 5분이 사라지는* 일이 가장 흔한 사고다.

**8단계. 카메라 위치 안내.** 보드 *상단 중앙*에 *"카메라 켜주세요"* 안내 sticky를 하나 박아둔다. 10장에서 ground rule로 잡은 카메라 정책의 시각적 알림. 시작 5분 전 들어온 참가자가 그 sticky를 보고 카메라를 켠다 — 진행자가 매번 말로 요청하지 않아도 된다.

**9단계. 보드 archive 정책.** 워크숍이 끝난 *직후*에 보드를 *복제*해서 한 부는 *원본*으로 두고, 한 부는 *작업본*으로 쓴다. *Iterate on Copy* 패턴(원서 ch.11)의 실전 적용. 원본은 다시 손대지 않는다. 작업본 위에서 사후 정리·refactor 토론을 한다. 6개월 뒤 원본을 다시 열어보면 *그날의 모델*이 그대로 살아 있다.

## Mural 보드 세팅 — 차이점만

Mural을 쓴다면 큰 흐름은 같다. 다른 점만 짧게 짚자.

- *Templates → EventStorming*에서 공식 템플릿을 부른다. 종류는 적지만 *facilitation 모드*가 강력하다.
- *Summon* 기능을 미리 익혀두자. 모든 참가자의 화면을 한 점으로 *동기화*한다. *Explicit Walkthrough* 단계에서 결정적이다 — 한 사람이 narrator가 되어 이야기를 풀 때, 다른 사람들의 시선이 흩어지는 문제를 잡아준다.
- *Private mode*가 silent ideation에 적합하다. 일정 시간 동안 *남의 sticky를 안 보고* 자기 영역에만 집중하게 한다. 끝나면 한꺼번에 공개된다. Brandolini의 silent ideation 정신과 잘 맞는다.

Mural은 *facilitator 전문가용*이라는 인상이 강하다. 첫 워크숍이라면 Miro가 더 친숙하고, Mural은 *세 번째 워크숍쯤* 도입해도 늦지 않다.

## FigJam과 Excalidraw — 저비용 대안

예산이 빠듯하거나 보안 정책이 까다로운 한국 팀이 자주 묻는다. *Miro/Mural 말고는 없나?* 두 가지 대안을 짧게 짚자.

**FigJam**은 Figma 진영의 협업 화이트보드. 디자인팀이 이미 Figma를 쓰는 회사에서 자연스럽다. EventStorming 전용 템플릿은 약하지만 — 커뮤니티 템플릿이 몇 개 있다 — 색상 sticker와 frame 기능은 충분하다. 디자이너와 개발자가 *같은 도구*에서 워크숍을 한다는 부수 효과가 크다. 한국에서 Toss·Karrot 같은 곳에서 자주 보이는 선택이다.

**Excalidraw**는 *직접 호스팅*까지 가능한 오픈소스다. 보안 정책상 외부 SaaS가 막힌 조직에서 매력적이다. UI는 단순하고, sticker 색상도 손수 설정해야 한다. *공식 EventStorming 템플릿이 없다*는 게 가장 큰 약점 — 매번 직접 깔아야 한다. 한 번 깔아둔 템플릿을 *팀의 git 저장소에 json으로* 박아두는 패턴이 자주 쓰인다.

둘 다 *공식 템플릿의 풍부함*에서는 Miro/Mural에 못 미친다. 첫 워크숍에 권하지는 않고, *익숙해진 다음에 옮길 도구*로 두자.

## 작은 디테일 몇 가지

마지막으로 *보드 세팅 외*에 자주 잊는 디테일을 짧게 모은다.

- **워크숍 *전*에 보드 링크를 슬랙에 한 번 더 보내자.** 시작 5분 전 *링크 못 찾았어요*가 가장 흔한 사고다.
- **보드 안에 *parking lot* frame을 하나 두자.** 11장 함정 1에서 짚은 *코드 얘기로 빠지는* 문제를 잡아주는 자리다. 코드 아이디어가 나오면 그 frame으로 옮긴다.
- **boards.miro.com 단축 URL을 미리 만들자.** 회사 도메인이 길면 슬랙·캘린더에서 끊긴다.
- **워크숍 *후* 보드 PDF export를 받아두자.** Miro 계정이 사라지거나 권한이 회수되어도 *기록은 남는다*. 한국 회사에서 *계정 정리* 정책 때문에 워크숍 자료가 사라지는 사고가 의외로 흔하다.

이 부록은 짧다. 의도적으로 짧다. 보드 세팅은 *기술적 디테일*이지, 워크숍의 본질이 아니다. 이 30분 체크리스트를 마치고 본문 10장으로 돌아가, *왜 분리하는지·언제 모이는지*에 시간을 더 쓰자. 도구는 거들 뿐이다.
