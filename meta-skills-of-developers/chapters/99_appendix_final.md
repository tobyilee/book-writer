# 【부록】 AI 코딩 도구의 분기 — Aider / Cursor / Claude Code

9장이 *AI 시대의 메타 스킬을 어떻게 결합할 것인가*였다면, 부록은 한 발 내려와 실용 질문 하나만 본다. *지금 — 2026년 초 — 어떤 도구를, 어떤 기준으로 고를 것인가.* AI 코딩 도구 시장은 6개월 단위로 풍경이 바뀐다. 그래서 부록은 *오래 가는 원리*와 *오늘의 스냅샷*을 함께 적었다. 1년 뒤 다시 펼쳤을 때 점검 체크리스트로 쓰는 편이 낫다.

## 1. 세 도구의 정체성

세 도구는 같은 일을 한다 — *AI를 옆에 두고 코드를 쓴다*. 그런데 그 옆에 *어떻게 두는가*가 셋 다 다르다. 그 차이가 도구 비교의 90%다.

**Aider는 git에 붙어 있는 페어 코더다.** CLI에서 `aider` 한 줄로 실행하고 파일을 지정하면 바로 대화가 시작된다. 변경분이 만들어지면 즉시 git 커밋으로 떨어진다. *diff, commit, branch*라는 기존 git 습관이 그대로 유지된다는 게 가장 큰 강점이다. 멀티 모델 호환이라 Claude·GPT·로컬 LLM 어느 것이든 붙여 쓸 수 있다. 학습 비용이 낮고, 결과물의 *추적성*이 git 그 자체다. 약점도 분명하다. 자율성이 약하다. 큰 변경을 한 번에 일임하기 어렵고, 반복 루프나 테스트 자동화는 직접 짜야 한다. 페어 파트너로 옆에 *앉혀 두고 한 단계씩* 같이 가는 도구로 보자.

**Cursor는 IDE 안에 살고 있다.** VS Code를 fork한 AI-native 편집기로, 단축키·확장·키맵을 그대로 쓰면서 그 위에 AI 인터랙션이 얹힌다. Tab 자동완성, Cmd-K 인라인 편집, 채팅 패널 — 이 세 가지가 일상을 차지한다. 진입 장벽이 매우 낮다. *내가 쓰던 IDE에 AI가 들어왔다*는 감각이라 처음 30분이면 익숙해진다. 약점은 *편집기 안*에 갇혀 있다는 점이다. 멀티파일 자율 변경이나 장시간 백그라운드 작업에는 본질적으로 덜 적합하다. 코드를 *한 줄씩 보면서* 쓰고 싶은 사람, IDE 중심으로 움직이는 사람에게 가장 자연스럽다.

**Claude Code는 자율 에이전트다.** 터미널에서 실행되지만 단순한 채팅 프롬프트가 아니라 *프로그래밍 가능한 환경*에 가깝다. 슬래시 커맨드로 작업을 정의하고, 서브 에이전트로 분기시키고, MCP로 외부 도구를 연결하고, hook으로 이벤트에 반응한다. 한 번의 지시로 *수십 개 파일을 읽고, 변경하고, 테스트를 돌리고, 실패하면 고쳐 다시 돌리는* 루프를 자율적으로 돈다. 강점은 명확하다. 큰 작업을 통째로 일임할 수 있고, 그 일임의 *공정 자체*를 설계할 수 있다. 약점도 명확하다. 학습 곡선이 가장 가파르다. 슬래시 커맨드·서브 에이전트·MCP·hook은 처음 보는 사람에게 낯선 단어다. 비용도 가장 많이 든다. 통제권을 잃고 *AI가 멋대로 한 일*을 사후 점검하느라 시간이 더 들 위험도 있다. 자율성을 *부릴 줄 아는 사람*에게만 가치가 있다는 게 정직한 표현이다.

세 도구의 자리가 다르다는 점을 기억해두자. *어느 게 좋은가*가 아니라 *어떤 워크플로우에 살고 있는가*가 답을 정한다.

## 2. 선택 기준 네 가지

도구를 고를 때 던질 질문은 사실 네 개면 충분하다.

**첫째, 나는 어디서 코드를 가장 많이 수정하는가.** IDE 안에서 한 파일을 열고 천천히 다듬는 시간이 길다면 Cursor가 자연스럽다. 터미널 안에서 git·테스트·빌드를 동시에 굴리는 시간이 길다면 Aider나 Claude Code가 자리 잡기 좋다. 자기가 평소에 *어떤 창*에 가장 오래 머무는지 한 주만 관찰해보자. 답은 거기서 나온다.

**둘째, 페어로 가고 싶은가, 자율 에이전트에게 일임하고 싶은가.** 한 줄씩 같이 보면서 가는 게 마음 편한 사람과, 큰 덩어리를 던져두고 결과만 받는 게 효율적인 사람으로 갈린다. 같은 사람이라도 작업의 종류에 따라 둘 사이를 오간다. 리팩터링·탐색·디버깅 같은 *생각이 많이 필요한 작업*은 페어가 낫고, 보일러플레이트·일괄 변경·테스트 작성 같은 *반복적이지만 정확해야 하는 작업*은 자율이 낫다. Aider는 페어, Cursor는 그 사이, Claude Code는 자율 쪽으로 무게 중심이 기운다.

**셋째, 개인 환경인가, 팀 환경인가.** 팀 표준이 정해져 있다면 그 표준을 따르는 게 맞다. 팀이 Cursor를 쓰는데 혼자 Claude Code를 쓰면 *프롬프트 자산·컨벤션·코드 리뷰 패턴*이 공유되지 않는다. 자유도가 높은 환경 — 작은 팀, 사이드 프로젝트, 1인 운영 — 이라면 자기 손에 가장 잘 맞는 것을 골라도 된다. *팀이 안 쓰는 도구를 혼자 우기는 비용*은 생각보다 크다.

**넷째, 새 워크플로우를 6개월마다 점검할 의지가 있는가.** *지금 잘 맞는 도구가 1년 뒤에도 잘 맞으리라는 보장*이 없다. 점검 의지가 있다면 어느 도구를 골라도 큰 손해를 보지 않는다. 점검 의지가 없다면 *학습 비용이 가장 낮은 쪽*으로 가는 편이 안전하다 — 처음에는 Cursor나 Aider가 그 자리에 가깝다.

네 질문에 답해보면 자기 자리가 거의 정해진다. 답이 명확하지 않다면 셋 다 일주일씩 써 보자. 일주일 뒤에 *돌아가고 싶은 도구*가 답이다.

## 3. 2026년 초 채택 추세

숫자 두 개만 기억해두자.

JetBrains 2026 Developer Survey에서 Claude Code와 Cursor가 *각각 18%의 직장 사용 점유율*을 차지했다. 그런데 *most loved* 응답에서는 Claude Code 46%, Cursor 19%다. 점유율은 비등한데 *애정*은 두 배 이상 갈렸다(JetBrains, "2026 Developer Ecosystem Survey," 2026). 점유율은 *지금 누가 쓰고 있는가*고, 애정은 *내년에 누가 더 깊이 쓸 가능성이 큰가*에 가까운 신호다. 사용자 확장의 폭과 사용자 만족의 깊이가 각각 다른 도구로 갈리는 풍경이 2026년 초의 모습이다.

또 하나 — Anthropic의 Claude Code 매출은 2025년 7월 기준 직전 분기 대비 5.5배로 뛰었고, 11월에 연환산 10억 달러를 찍었다. 2026년 초에는 25억 달러를 넘었다(Faros AI, "Best AI Coding Agents for 2026: Real-World Developer Reviews," 2026). *시장이 자율 에이전트형 워크플로우 쪽으로 빠르게 기울고 있다*는 신호로 받아들이는 편이 좋다. 그렇다고 Cursor가 곧 사라진다는 뜻은 아니다. *IDE 중심 사용자*는 사라지지 않는다.

Aider는 매출 데이터가 공개되어 있지 않지만, 오픈소스 프로젝트로서 꾸준히 사용자 층을 유지하고 있다. *git 습관을 유지하고 싶은 사람*과 *멀티 모델·로컬 모델 사용자*가 안정적으로 모인다. 셋 중 가장 *조용히* 자리를 지키는 도구다.

추세 데이터는 *2026년 초 스냅샷*이다. 1년 뒤 이 부록을 다시 펼친다면 이 절의 숫자는 갱신해야 한다. 추세는 *기억하지 말고 점검하자*.

## 4. 도구 무관 — 같은 원리 다섯

세 도구를 늘어놓고 보면 결국 같은 결론에 도달한다. *도구를 무엇으로 고르든 적용되는 원리*가 따로 있다는 것. 9장에서 다룬 다섯 원칙이 그것이다. AI as Multiplier, Trust-but-verify, Prompt as core competency, Spec-first development, Every engineer is a manager. 다섯 원칙은 Aider에서도, Cursor에서도, Claude Code에서도, 그리고 1년 뒤에 등장할 새 도구에서도 동일하게 적용된다. 어느 도구를 쓰든 *명세를 잘 쓰는 사람*이 더 좋은 결과를 얻고, 어느 도구를 쓰든 *AI 출력을 검증하는 루틴*을 가진 사람이 더 안전한 코드를 만든다.

7장에서 cue와 시스템을 구분했다. *도구는 cue, 원리는 시스템이다*. cue는 6개월·1년 단위로 바뀌어도 시스템은 거의 바뀌지 않는다. *Cursor가 좋아? Claude Code가 좋아?*를 6개월째 검색만 하고 있는 사람을 가끔 본다. 그럴 시간에 *내 명세를 한 번 더 다듬는* 편이 낫다. 명세는 도구가 바뀌어도 살아남고, 도구는 명세가 없으면 무력하다.

도구 의존성도 한 줄 짚어두자. 어떤 도구든 *없으면 일을 못 하는 상태*가 되면 곤란하다. AI 도구가 잠시 다운될 수도 있고, 회사가 보안 이유로 사용 금지를 거는 일도 있다. *도구가 사라져도 60% 속도로는 일을 끝낼 수 있는 상태*를 자기 안에 유지해두자. 의존이 아니라 증폭이 되어야 한다는 명제는, 도구 선택의 맥락에서 더 분명하게 적용된다.

## 5. 마무리 — 1년에 한 번 결정 일지에

도구 선택은 *2년 단위로 다시* 보자. 6개월 단위로 매번 갈아타는 건 비용이 크고, 5년 동안 한 도구만 고집하는 것도 시장 변화를 못 따라간다. 1년에 한 번, *지금 쓰는 도구가 여전히 내 워크플로우에 맞는가*를 자기 결정 일지(3장)에 짧게 적어두자. 도구를 고른 이유, 6개월 동안 좋았던 점, 불편했던 점, 다음 점검 시기 — 이 네 줄이면 충분하다.

이 네 줄을 적는 행위가 7장의 *도구화*, 6장의 *학습*, 3장의 *결정*을 한 자리에 모은다. AI 코딩 도구는 평생 다룰 새로운 환경이다. 환경을 다루는 메타 스킬은 변하지 않는다. 그게 부록이 책의 마지막 페이지에 놓인 이유다.
