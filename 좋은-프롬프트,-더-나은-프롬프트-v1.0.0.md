# 좋은 프롬프트, 더 나은 프롬프트 — 작성부터 평가까지

> 프롬프트를 감으로 쓰는 단계를 벗어나, 두 후보 중 어느 쪽이 실제로 더 나은지 수동·자동 eval로 측정해 선택하는 법을 다룬다.

- **저자:** Toby-AI
- **버전:** v1.0.0
- **발행일:** 2026-06-01
- **언어:** 한국어 (ko)
- **분량:** 13장 + 머리말 / 맺음말 / 부록 A / 참고문헌 — 본문 약 76,000자

## 이 책은 무엇인가

시중의 프롬프트 안내서 대부분은 "이렇게 쓰세요" 팁 모음에서 멈춘다. 이 책은 한 걸음 더 나아간다. 2026년 현재 실제로 쓰이는 세 모델 — Claude Opus 4.8, GPT-5.5, Gemini 3.5 Flash — 의 공식 가이드를 1차 근거로 삼아 각 모델의 실제 작동 방식을 짚는다. 그리고 책의 진짜 무게는 **평가(eval)**에 있다. "이 프롬프트가 더 낫다"는 느낌을 어떻게 점수로 옮기고, 코드 없이도 채점하며, 필요하면 자동화해 회귀 테스트로 지켜내는지를 10~12장에서 구체 예시로 보여준다.

두 가지 점에서 기존 자료와 다르다. 첫째, 모델 간 비교가 피상적 나열이 아니라 "같은 작업을 세 가지 언어로 번역"하는 실용 수준이다. XML 구조를 요구하는 Claude, 추론 깊이와 답변 길이를 독립으로 제어할 수 있는 GPT-5.5, temperature를 건드리면 성능이 망가지는 Gemini — 각 모델의 결을 이해하고 나서야 비교가 의미를 갖는다. 둘째, 10·11·12장은 "환불 정책 요약"이라는 단일 작업을 끝까지 끌고 간다. 손으로 채점하고(10장), 자동화하고(11장), 품질·비용·지연을 함께 놓고 최종 선택한다(12장). 매 장 새 예시를 꺼내지 않으므로 12장의 클라이맥스 효과가 살아있다.

휘발성 수치(가격·context 창·파라미터 기본값)는 부록 A 스냅샷 표 한 곳에 격리해 책 본문의 수명을 늘렸다. 모델이 바뀌어도 원칙은 유효하다.

## 누구를 위한 책인가

AI 도구를 매일 쓰지만 결과가 들쭉날쭉한 이유를 설명하지 못하는 사람에게 적합하다. 코드 없이도 읽힌다 — 1~9장은 일반 사용자가 바로 적용할 수 있는 패턴을 담고, 10~12장 eval 파트도 개념 트랙(코드 없음)과 코드 트랙(Python/YAML)을 명시적으로 분리해 두었다. 비개발자는 개념 트랙만 읽고도 "더 나은 프롬프트를 측정으로 가려낸다"는 워크플로우를 온전히 이해할 수 있다.

개발자라면 9장 코딩 에이전트(Claude Code·Cursor·Codex, 컨텍스트 5계층·MCP)와 11장 pytest/promptfoo/CI 자동화 파이프라인에서 추가 수확을 얻는다. 세 모델을 혼용하는 제품을 만들거나, 프롬프트를 코드처럼 버전 관리하고 싶은 팀에게도 실용적인 기준점이 된다.

진입 상태가 "프롬프트를 감으로 쓰고, 어떤 날은 좋고 어떤 날은 엉뚱한 답이 나와도 왜인지 모른다"라면, 출구 상태는 "성공 기준을 먼저 정의하고, 모델 특성에 맞춰 프롬프트를 구조화하며, 두 후보 중 어느 쪽이 나은지 수동·자동 eval로 측정해 선택하고, 회귀 테스트로 그 품질을 유지한다"이다.

## 무엇을 얻게 되는가

- Claude·GPT-5.5·Gemini 3.5 Flash 각각에 맞는 프롬프트 작성 방식과 그 차이를 이해하는 감각
- 두 프롬프트 후보 중 어느 쪽이 더 나은지 느낌이 아니라 rubric·점수·체크리스트로 가려내는 수동 평가 능력
- pytest/promptfoo/LLM-as-judge로 프롬프트를 자동 테스트하고 회귀를 막는 파이프라인(개발자)
- 품질만이 아니라 비용·지연까지 함께 저울질해 최적 프롬프트를 선택하는 의사결정 프레임
- 일상 글쓰기·학습·코딩 에이전트 각 시나리오에서 바로 쓸 수 있는 프롬프트 패턴 레퍼토리
- 안티패턴·프롬프트 인젝션·버전 신선도 규율을 포함한 프롬프트 운용 원칙

## 차례

1. **1장. 감으로 쓰던 프롬프트, 이제는 설계한다** — 2026년 프롬프트 엔지니어링의 핵심 변화와 이 책 전체의 사이클 지도
2. **2장. 좋은 프롬프트의 해부학** — 역할·맥락·지시·few-shot·출력 형식, 어떤 모델에도 통하는 일곱 가지 부품
3. **3장. Claude Opus 4.8 — XML과 effort의 언어** — XML 1급 시민, effort로 추론을 켜는 법, prefill 폐기 마이그레이션
4. **4장. GPT-5.5 — 외과수술 같은 지시** — reasoning_effort와 verbosity를 독립으로 제어하는 두 개의 레버
5. **5장. Gemini 3.5 Flash — 온도를 건드리지 마라** — temperature 1.0 고정 원칙, few-shot 필수, 멀티모달 강점
6. **6장. 세 모델을 한자리에 — 비교와 멀티모델 전략** — 같은 작업을 세 가지 프롬프트로 번역하고, 비용·지연까지 고려한 모델 선택
7. **7장. 일상 프롬프트 — 글쓰기·요약·이메일·번역·기획** — 막연한 요청을 구체적인 주문으로 바꾸는 패턴
8. **8장. 학습 프롬프트 — 가르치게 만들기** — 정답 자판기를 소크라테스식 튜터로 전환하는 법
9. **9장. 코딩 에이전트 프롬프트 — Claude Code·Cursor·Codex** — 컨텍스트 5계층, 첫 턴에 task·intent·constraints를 충분히 싣는 법
10. **10장. 더 나은가, 어떻게 아는가 — 수동 평가** — "환불 정책 요약" 두 후보를 rubric·A/B·체크리스트로 손으로 채점
11. **11장. 프롬프트도 코드처럼 테스트한다 — 자동 평가** — 같은 작업을 pytest/promptfoo/LLM-as-judge로 자동화하고 CI에 연결
12. **12장. 작성→평가→비교→선택, 한 바퀴 돌려보기** — 품질·비용·지연을 함께 놓고 최종 선택, 채택안을 회귀 테스트셋에 고정
13. **13장. 함정을 피하고, 안전하게 — 안티패턴과 마무리** — 흔한 실수, 프롬프트 인젝션, 책 전체를 관통하는 한 바퀴 원칙
