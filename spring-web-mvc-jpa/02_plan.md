# React/Next.js 개발자를 위한 Spring MVC + JPA 백엔드 입문 — 저술 계획

> 검색 시점: 2026-05-25 기준 / 입력 레퍼런스: `01_reference.md` / 장르: tech-book (profiles/tech-book)

## 제목 후보

1. **프런트엔드 개발자를 위한 Spring 백엔드 입문** *(slug: `spring-for-frontend-devs`)*
   - 톤: 정직하고 직설적인 입문서. 군더더기 없는 정공법 제목.
   - 포지셔닝: "내가 누구인지(프런트 개발자)"와 "무엇을 배우는지(Spring 백엔드)"를 한 줄에 못 박는다. 서점·검색에서 대상 독자가 즉시 자기 책임을 알아본다. 안전하지만 차별점(AI 페어코딩·신선도 함정)이 제목에 안 드러나는 게 약점.

2. **React에서 Spring으로: 백엔드를 처음 만드는 프런트엔드 개발자의 안내서** *(slug: `react-to-spring`)*
   - 톤: 여정(journey) 서사. "건너간다"는 이동의 은유를 전면에 세운 입문서 톤.
   - 포지셔닝: 독자가 이미 선 자리(React)에서 출발해 목적지(Spring)로 데려간다는 이 책의 핵심 교수법 — "프런트가 아는 것에서 출발해 백엔드로 확장" — 을 제목이 그대로 말한다. 독자 여정 1번 항목과 정확히 호응. `fetch`를 아는 사람을 위한 책임이 분명해진다.

3. **마법 걷어내기: AI와 함께 배우는 Spring MVC + JPA** *(slug: `spring-minus-the-magic`)*
   - 톤: 도발적이고 관점이 선 제목. 레퍼런스의 현장 합의 키워드 "minus the magic"(어노테이션 마법 걷어내기)과 AI 페어코딩이라는 두 차별점을 전면에.
   - 포지셔닝: "Spring은 마법처럼 보이지만 그 마법을 걷어내고 보여주겠다 + AI가 주는 코드도 마법이 아니라 검증 대상"이라는 책의 두 기둥을 압축. 가장 개성 있지만, "Spring MVC+JPA 입문"이라는 정보가 부제로 밀려 검색 노출은 1·2번보다 약할 수 있음.

**추천: 2번 (`react-to-spring`)** — 이 책의 가장 강력한 차별점은 "프런트엔드 개발자가 이미 아는 것에서 출발한다"는 교수법이고(사용자가 명시한 독자 여정 1번), 제목이 그 약속을 직접 한다. 대상 독자가 "이건 내 책"이라고 느끼는 호소력이 1번보다 강하고, 3번보다 "무엇을 배우는지"가 분명하다. AI 페어코딩·신선도 함정은 부제·표지 카피·서문에서 보강하면 된다.

## 책 특성

- **장르:** tech-book (입문서 / 안내서 성격. 활성 프로필: `profiles/tech-book/`)
- **난이도:** 초급 → 초중급. 자바 문법·SQL 기본은 전제하되, 백엔드 개념·Spring·JPA·보안은 무전제로 아주 쉬운 예부터 쌓는다. 챕터가 진행될수록 점진적으로 난이도를 올린다.
- **예상 분량:** 14개 챕터 × 평균 약 5,300자 = 본문 약 74,000자(한글). 입문서로서 챕터당 4,000~6,500자 범위(개념 도입 챕터는 짧게, 연속 프로젝트가 누적되는 중반 챕터는 길게). 부록·서문·에필로그 포함 시 약 82,000자. (리뷰 P1 반영: 보안을 9·10장으로 분할해 13→14장)
- **독자 여정 (진입 → 출구):**
  - **진입 상태:** React/Next.js로 클라이언트를 능숙하게 만들지만, 서버는 "`fetch`가 때리는 블랙박스". 자바 문법과 SQL은 알지만 백엔드 실무 경험 0. Cursor로 AI 페어코딩은 해봤으나 Spring 생성 코드의 옳고 그름을 판별하지 못함.
  - **출구 상태:** Spring Boot 3.x로 REST API를 직접 설계·구현하고, JPA로 DB에 영속화하고, JWT·세션 두 방식의 보안을 적용하고 트레이드오프를 설명할 수 있다. Thymeleaf SSR을 CSR과 비교해 언제 무엇을 쓸지 판단한다. 무엇보다 **AI가 준 Spring 코드를 읽고, 버전 신선도 함정(javax→jakarta 등)을 잡아내고, SQL 로그로 동작을 검증할 수 있는** 백엔드 입문자가 된다.

## 내러티브 아크

이 책은 **하나의 연속 프로젝트 — "할 일 관리 API(TaskBoard)"** 를 14개 챕터에 걸쳐 빈 프로젝트에서 인증·화면까지 키워나가는 구조다. 사용자가 명시한 독자 여정 6단계를 그대로 척추로 삼되, AI 페어코딩과 버전 신선도는 별도 챕터 + 본문 사이드바의 이중 구조로 책 전체에 녹인다.

흐름의 논리는 이렇다. **1장**은 독자가 이미 선 자리에서 출발한다 — 프런트 개발자가 매일 쓰는 `fetch`를 뒤집어 "서버 입장의 HTTP"로 멘탈 모델을 전환한다. 이론이 아니라 익숙한 것의 재해석이라 진입 장벽이 낮다. CORS도 여기서 "프런트가 겪는 증상"으로만 꺼내 두고, "이걸 막는 것도 푸는 것도 서버의 책임"이라는 씨앗을 심어 5장에서 회수한다. **2장**은 이 책의 학습 방법론 자체를 세팅한다 — Spring을 배우는 데 AI를 어떻게 동반자로 쓰고, 왜 AI를 믿으면 안 되는지(신선도 함정의 예고편). 다만 "코드 안 짜는 장"이 1·2장 연속으로 이어지면 행동 지향적 프런트 독자의 동기가 식으므로, 2장은 핵심 학습 계약으로 압축하고 신선도 함정의 구체 사례는 3장 첫 실습(import 검수)과 묶어 체감 진입을 앞당긴다.

**3장**부터 손이 움직인다. 먼저 npm 세계에서 온 독자를 위한 빌드·실행·디버깅 오리엔테이션("Gradle = package.json + npm scripts")을 깐 뒤, `start.spring.io`로 프로젝트를 만들고 첫 엔드포인트를 띄운다 — 레퍼런스의 현장 합의("이론 선행 금물, 동작하는 엔드포인트 먼저")를 그대로 따른다. **4장**에서 DI/IoC를 — 단, 이론을 앞세우지 않고 3장에서 이미 쓴 코드의 "마법"을 사후에 걷어내며 설명한다. **5장**은 REST API를 제대로 만든다(DTO·검증·전역 예외). 여기서 서버 측 CORS 설정(`CorsConfigurationSource`)과 최소 슬라이스 테스트(`@WebMvcTest`/MockMvc)를 함께 도입한다 — 둘 다 프런트 독자가 실제로 막히는 지점이자, 11·13장이 회수할 기반이다. 여기까지가 "메모리에 사는, 테스트와 CORS를 갖춘 API".

**6~8장**은 JPA를 *기존 API에 점진적으로 붙인다*. 6장에서 엔티티·Repository로 DB를 연결하고, 7장에서 연관관계와 영속성 컨텍스트의 "마법"(더티 체킹·지연 로딩)을, 8장에서 그 마법이 운영에서 터지는 지점(N+1·LazyInitException)과 SQL 로그 규율을 다룬다. 8장에는 repository 슬라이스 테스트(`@DataJpaTest`)를 더해 "SQL을 눈으로 보는" 규율을 테스트로 굳힌다. 편의→누수의 순서가 자연스럽다.

**9~11장**은 보안이다. 입문자에게 Spring Security는 단일 최대 난관이라, 리뷰 지적대로 한 장에 욱여넣지 않고 도약을 세 단계로 완충한다. **9장**은 보안의 기초 — 인증/인가 개념과 "필터체인이 요청을 가로채는 법"이라는 새 멘탈 모델, 그리고 최소 설정만. **10장**은 JWT(stateless)를 구현하며 신선도 함정의 클라이맥스(`WebSecurityConfigurerAdapter` 제거, `authorizeHttpRequests`)를 터뜨린다. **11장**은 세션(stateful)을 *같은 프로젝트에 두 번째로* 적용하고 트레이드오프를 비교한다 — 프런트에게 직접 와닿는 "토큰 어디 저장?" 갈등을 진입점으로. 쿠키 기반 인증과 5장의 CORS `allowCredentials`가 여기서 연결된다. 단절을 막기 위해 10장 끝에 "이게 유일한 답은 아니다(11장에서 다른 길)"라는 예고를 심는다. **12장**은 Thymeleaf로 관리자 화면을 붙이며 SSR vs CSR을 프런트 관점에서 정리한다.

**13장**은 책의 클라이맥스 — 이제 백엔드를 아는 독자가 Claude Code 등으로 *실제 기능 하나를 스펙→설계→구현→테스트*로 완주하는 워크플로우. 1~12장에서 쌓은 검증 능력이 여기서 무기가 된다. 특히 5·8장에서 미리 익힌 테스트가 "구현→테스트" 단계의 실체를 떠받친다(테스트를 본 적도 없이 "테스트하라"는 모순 제거). **14장**은 여정을 종합하고 3.x→4.x 마이그레이션 전망으로 닫는다.

**버전 신선도(javax→jakarta, WebSecurityConfigurerAdapter 제거, authorizeRequests→authorizeHttpRequests)** 는 1급 주제로, 2장에서 개념을 심고, 10장(Security)에서 구체 함정으로 터뜨리고, 13·14장에서 AI 검증·마이그레이션으로 회수한다 — 책 전체를 관통하는 보조 서사다.

## 챕터 목록

### 1장. fetch 너머에 누가 있나 — 클라이언트의 HTTP를 서버의 HTTP로 뒤집기
- **핵심 질문:** 내가 매일 쓰는 `fetch`의 반대편에는 무슨 일이 일어나고 있을까?
- **주요 내용:**
  - 프런트가 이미 아는 것(메서드·상태코드·헤더·JSON 바디)을 명시적으로 출발점으로 선언
  - 같은 요청을 "보내는 쪽"이 아니라 "받는 쪽"에서 보기 — 라우팅·역직렬화·검증·처리·직렬화의 5단계
  - 상태코드(200/201/400/401/404/500)를 "받는 것"에서 "내가 결정하는 책임"으로
  - **CORS는 "증상"으로만 꺼내고 닫지 않는다** — 프런트가 매일 보는 그 빨간 콘솔 에러를 "이미 아는 것"으로 선언하되, "이걸 *막는 것도 푸는 것도 서버의 책임*"이라는 씨앗을 심어 5장(서버 측 CORS 설정)으로 명시적으로 넘긴다. (리뷰 P0-1: 1장에서 닫지 말 것)
  - REST는 규약이 아니라 아키텍처 스타일이라는 예고(Fielding, 깊이는 5장·부록으로 유보)
  - 쿠키/세션이 서버 측에서 수명을 갖는다는 개념의 씨앗(9~11장 예고)
- **독자가 얻는 것:** "서버는 블랙박스가 아니라 내가 통제할 5단계 파이프라인"이라는 새 멘탈 모델
- **스캐폴드:** 개념 도입형 (상황 가정 → 익숙한 것 뒤집기)
- **AI 페어코딩 학습 포인트(사이드바):** AI에게 "이 fetch 요청을 서버가 어떻게 받는지 설명해줘"라고 물어 멘탈 모델을 보강하는 법. AI 설명을 RFC 9110 같은 1차 근거와 대조하는 습관의 예고.
- **1차 근거(P2):** 메서드·상태코드의 권위를 위해 RFC 9110(HTTP Semantics) 조항을 사이드바/각주 한 줄로 직접 인용(입문자 부담 고려해 본문 비대화는 피함).
- **예상 분량:** 약 4,500자

### 2장. AI와 함께 배운다는 것 — 동반자이자 의심 대상
- **핵심 질문:** Cursor·Claude Code가 Spring 코드를 척척 만들어주는데, 나는 무엇을 더 배워야 하나?
- **주요 내용:**
  - 이 책의 학습 계약: AI는 스캐폴드·보일러플레이트·설명에 강하고, 비즈니스 로직·검증·판단은 독자 몫
  - **버전 신선도 함정의 정체** — 인터넷·AI는 구버전 코드를 자신 있게 준다(`javax.*`, `WebSecurityConfigurerAdapter`, `authorizeRequests`). 왜 이런 일이 생기는지(학습 데이터의 시점)
  - 왜 이 책은 Spring Boot 3.x로 가르치는가 — 3.x가 프로덕션 주류(학습 기준), 4.0.6이 이미 최신 stable, 3.5 OSS 지원 2026-06-30 종료라는 사실을 숨기지 않고 명시
  - AI 코드 검증 루틴의 첫 소개: 읽기 → 버전 식별 → 동작 확인(SQL 로그·실행)
  - 프롬프트에 버전 명시하기, "Ask me questions"로 모호성 되묻기
- **독자가 얻는 것:** AI를 학습 가속기로 쓰되 맹신하지 않는 구체적 작업 습관. 이후 모든 사이드바의 토대.
- **스캐폴드:** 비교 대조형 (기대 vs 불안) + 개념 도입형
- **AI 페어코딩 학습 포인트:** 이 장 자체가 AI 페어코딩 학습법의 본진. 신선도 검증·프롬프트 위생을 명문화.
- **예상 분량:** 약 4,500자 (리뷰 P1 반영: 5,500→4,500자로 압축. "코드 안 짜는 장"이 1·2장 연속되는 것을 완화 — 핵심 *학습 계약*만 이 장에 두고, 신선도 함정의 *구체 예시*(`javax.*` import 등)는 3장 첫 실습에서 코드와 함께 보여주는 쪽으로 이관. 내용 삭제가 아니라 체감 진입을 3장으로 당기는 재배치)
- **신선도 주의:** 버전 수치(4.0.6/2026-04-23, 3.5 EOL 2026-06-30)는 fact-checker가 `01_reference.md` §6과 대조 필수. "(사실 확인 필요)" 마크 대상.

### 3장. 첫 엔드포인트 — 빌드 도구부터 동작하는 서버까지
- **핵심 질문:** 이론은 잠시 미루고, npm 세계에서 온 내가 가장 빨리 응답하는 서버를 어떻게 띄우나?
- **주요 내용:**
  - **빌드·실행·디버깅 오리엔테이션(리뷰 P0-2, 이 장 앞 소절 신설):** 자바 빌드 경험이 없는 프런트 독자를 위한 첫 다리.
    - **Gradle = `package.json` + npm scripts**, `build.gradle`의 `dependencies` = `package.json`의 `dependencies`, `./gradlew bootRun` = `npm run dev`, `./gradlew build` = `npm run build`
    - Gradle 권장(이유 간단히: Kotlin/Groovy DSL·증분 빌드), Maven은 존재만 언급
    - 의존성을 *어디에* 추가하는가(`build.gradle`), 실행·재시작, IDE(IntelliJ)에서 디버거 붙이기 기초
    - "0줄이면 첫 좌절" — 별도 장은 과하나 이 소절은 필수
  - `start.spring.io`로 프로젝트 생성(Java 17/21, Spring Boot 3.x, Web·Validation 의존성) — 무엇을 고르고 왜인지
  - 프로젝트 구조 훑기, `@RestController` + `@GetMapping`으로 첫 `/api/tasks` 응답
  - 자동설정(auto-configuration)이 보일러플레이트를 없애는 마법 — "지금은 그냥 동작한다, 4장에서 걷어낸다"
  - Next.js API routes / 서버액션과의 1:1 멘탈 매핑
  - 연속 프로젝트 TaskBoard의 첫 삽 — 메모리에 사는 할 일 목록 반환
- **독자가 얻는 것:** 손으로 띄운 첫 Spring 서버 + "빌드/실행/디버깅을 npm에 빗대 이해"하는 안정감. "할 수 있다"는 자신감.
- **스캐폴드:** 개념 도입형 (동작 우선)
- **AI 페어코딩 학습 포인트:** AI에게 프로젝트 스캐폴드를 생성시키되, **버전을 프롬프트에 명시**("Spring Boot 3.x, jakarta namespace")하고 생성된 `build.gradle`/import를 직접 검수. AI가 `javax.*`를 주면 잡아내는 첫 실전(2장에서 압축한 신선도 함정의 *구체 예시*가 여기서 코드로 처음 등장).
- **예상 분량:** 약 5,500자 (빌드 오리엔테이션 소절 추가로 5,000→5,500)

### 4장. 마법 걷어내기 — DI와 IoC가 실제로 하는 일
- **핵심 질문:** 3장에서 아무것도 `new` 하지 않았는데 어떻게 객체들이 연결됐을까?
- **주요 내용:**
  - 3장 코드를 되짚으며 "누가 컨트롤러를 만들고 의존성을 넣었나" 추적 — 이론이 아니라 사후 해부
  - IoC/DI 정의(Fowler) — 객체가 협력자를 직접 만들지 않고 컨테이너가 주입
  - 프런트 다리: props/context로 의존성 내려주기, NestJS `@Injectable`/Angular DI의 같은 계보
  - **생성자 주입**을 권장으로 — 왜 필드 주입보다 나은가(테스트·불변성)
  - Service 계층을 도입해 TaskBoard를 컨트롤러-서비스로 분리(다음 JPA 연결의 발판)
- **독자가 얻는 것:** "Spring의 마법"의 정체를 본 안도감. 어노테이션을 무서워하지 않게 됨.
- **스캐폴드:** 문제-해결형 (마법 → 걷어내기) + 개념 도입형
- **AI 페어코딩 학습 포인트:** AI에게 "왜 필드 주입 대신 생성자 주입?"을 묻고 답을 Fowler 글과 대조. AI가 만든 빈 와이어링을 읽고 "무엇이 무엇으로 치환되는지" 추적하는 연습.
- **예상 분량:** 약 5,000자

### 5장. 제대로 된 REST API — DTO·검증·전역 예외·CORS·첫 테스트
- **핵심 질문:** 동작은 하는데, 잘못된 요청·에러를 어떻게 품위 있게 다루고, 내 React 앱에서 부르면 왜 막히나?
- **주요 내용:**
  - DTO 정의 — 왜 엔티티를 그대로 노출하지 않는가(검증은 DTO 계층에서)
  - `@Valid` + Jakarta Bean Validation(Hibernate Validator), 실패 시 `MethodArgumentNotValidException`
  - `@RestControllerAdvice` + `@ExceptionHandler`로 전역 예외 처리, `ResponseEntity`로 구조화 응답(타임스탬프·상태·필드별 메시지)
  - 상태코드를 의도적으로 설계(201 Created, 400, 404, 409 등) — 1장에서 심은 "내가 결정하는 책임"의 실현
  - **서버 측 CORS 설정 소절(리뷰 P0-1):** 1장에서 "증상"으로 심은 CORS를 여기서 *서버 책임*으로 회수.
    - "왜 브라우저만 막고 Postman/curl은 통과하나" — 동일 출처 정책을 프런트 관점으로
    - `@CrossOrigin`(좁은 범위) vs `CorsConfigurationSource` 빈(전역·권장) — React/Next dev 서버(`localhost:3000`)에서 Spring(`localhost:8080`) 호출 시나리오
    - preflight(OPTIONS)·허용 메서드/헤더, `allowCredentials`는 쿠키 인증과 엮이므로 11장(세션)과 크로스레퍼런스
  - REST 엄밀 정의 vs 실무 용법 짧은 정리(HATEOAS는 부록 C로)
  - **첫 테스트 소절(리뷰 P0-3):** `@WebMvcTest` + MockMvc로 컨트롤러 슬라이스 테스트 1~2개(성공 200/201, 검증 실패 400).
    - 프런트 다리: **MockMvc = supertest** — "요청 던지고 상태코드·바디 단언"하는 같은 모양
    - 13장 "구현→테스트" 워크플로우가 회수할 씨앗(테스트를 본 적도 없이 "테스트하라"는 모순 제거)
  - TaskBoard에 생성·수정·삭제·검증을 추가 — 메모리 API 완성(CORS·테스트 포함)
- **독자가 얻는 것:** 견고한 REST API 설계 능력. 검증·예외를 프레임워크에 위임하고, CORS를 서버에서 풀고, 컨트롤러를 테스트하는 법.
- **스캐폴드:** 문제-해결형 (찜찜한 에러 처리 → 전역 핸들러; CORS 에러 → 서버 설정)
- **AI 페어코딩 학습 포인트:** AI에게 DTO·검증·CORS 설정·MockMvc 테스트를 생성시키고, 검증 규칙이 비즈니스 요구와 맞는지, CORS가 과도하게 열려 있지 않은지(`allowedOrigins("*")` + credentials 조합 금지) 직접 검수(AI는 스캐폴드, 규칙·보안은 사람). 엣지 케이스를 AI에게 되묻기.
- **예상 분량:** 약 6,500자 (CORS·테스트 소절 추가로 6,000→6,500)

### 6장. 메모리에서 데이터베이스로 — JPA를 붙이다
- **핵심 질문:** 서버를 끄면 사라지는 데이터를, 어떻게 DB에 눌러 담나?
- **주요 내용:**
  - 임피던스 불일치 개념(객체 ↔ 관계형) — ORM이 왜 필요하고 왜 완벽하지 않은지(Neward 예고)
  - `@Entity` 매핑, `JpaRepository` 인터페이스 선언만으로 CRUD — Repository 패턴(Evans)과 프록시 구현
  - 프런트 다리: Prisma/Drizzle와의 비교 — 쿼리를 직접 안 쓰는 대신 무엇을 잃나
  - 5장의 메모리 TaskBoard를 *같은 API 계약을 유지한 채* DB 영속화로 교체 — 점진적 부착의 핵심 실습
  - H2(개발)로 시작, application.yml 설정
- **독자가 얻는 것:** API를 깨지 않고 데이터를 영속화하는 경험. ORM의 본질 이해.
- **스캐폴드:** 개념 도입형 + 사례 분석형
- **AI 페어코딩 학습 포인트:** AI는 엔티티·Repository 생성에 강함 — 스캐폴드를 맡기되 생성된 엔티티 매핑이 의도한 스키마와 맞는지, `javax.persistence`가 아니라 `jakarta.persistence`인지 검수.
- **예상 분량:** 약 6,000자

### 7장. 영속성 컨텍스트라는 출석부 — 더티 체킹과 지연 로딩의 마법
- **핵심 질문:** `save()`를 안 했는데 값이 바뀌어 저장됐다. 무슨 일이 일어난 건가?
- **주요 내용:**
  - 영속성 컨텍스트 = 트랜잭션 동안 엔티티를 추적하는 "출석부" 비유
  - 더티 체킹 — 변경 감지 후 커밋 시 자동 UPDATE
  - 연관관계 매핑(`@OneToMany`/`@ManyToOne`) — TaskBoard에 "프로젝트-할일" 관계 추가
  - 지연 로딩(LAZY) vs 즉시 로딩(EAGER)의 기본 개념
  - `@Transactional`의 역할과 경계 — 왜 트랜잭션 밖에선 컨텍스트가 없나
- **독자가 얻는 것:** JPA의 "비가시 동작"을 예측 가능한 모델로 전환. 다음 장의 함정을 이해할 토대.
- **스캐폴드:** 개념 도입형 (비유 중심)
- **AI 페어코딩 학습 포인트:** AI가 만든 연관관계 매핑이 양방향/단방향 중 무엇이고 왜 그런지 묻기. "이 코드가 실행하는 SQL을 보여줘"로 비가시 동작을 가시화.
- **예상 분량:** 약 6,000자

### 8장. 운영에서 터지는 것들 — N+1과 SQL 로그 규율
- **핵심 질문:** 개발할 땐 멀쩡하던 API가 왜 운영에서 느려지나?
- **주요 내용:**
  - N+1 문제 — 개발 땐 데이터가 적어 안 보이고 운영에서 터진다
  - **SQL 로그를 켜서 직접 보는 실습** — JPA 규율의 핵심
  - N+1 해법 4종: ① JOIN FETCH ② `@EntityGraph` ③ 배치 페치(`default_batch_fetch_size`) ④ DTO 프로젝션
  - JOIN FETCH + 페이징의 함정(중복 row/페이징 깨짐) → 배치 페치 우회
  - JPA 함정 3종 정리: 더티 체킹 타이밍 · `@Transactional` self-invocation 무효 · 트랜잭션 밖 지연 로딩(LazyInitializationException)
  - ORM 추상화의 누수(Neward "베트남") — 결국 SQL을 봐야 하는 순간
  - **repository 테스트 소절(리뷰 P0-3):** `@DataJpaTest`로 repository 슬라이스 테스트 1개 — 쿼리가 의도한 결과를 주는지, N+1 해법 적용 전후로 발생 SQL 수가 줄었는지를 테스트로 확인. 5장 컨트롤러 테스트와 짝을 이뤄 "테스트로 검증한다"는 13장 약속의 두 번째 기둥.
- **독자가 얻는 것:** JPA를 "마법"이 아니라 "SQL을 생성하는 도구"로 다루는 실무 감각. repository를 테스트로 묶는 법.
- **스캐폴드:** 문제-해결형 + 사례 분석형
- **AI 페어코딩 학습 포인트:** AI가 준 Repository 메서드가 N+1을 유발하는지 SQL 로그로 검증. "이 코드의 N+1을 고쳐줘"라고 했을 때 4종 중 무엇을 택했고 적절한지 판단하는 능력.
- **저술 시 보강(리뷰 P1, 레퍼런스 §7 한계):** 오프닝이 추상론("운영에서 터진다")에 그치지 않도록 **국내 회사 기술블로그 N+1 포스트모템 1건(우아한형제들·토스·카카오 tech 등)을 챕터 저술 단계에서 추가 수집**해 구체 사건으로 연다. 실행 보장 차원에서 명시.
- **예상 분량:** 약 6,500자 (repository 테스트 소절은 함정 정리 압축으로 흡수, 총량 유지)

### 9장. 보안의 기초 — 인증·인가와 필터체인이 요청을 가로채는 법
- **핵심 질문:** "이 요청을 보낸 게 누구이고, 무엇을 할 권한이 있나"를 서버는 어떻게 가려내나?
- **주요 내용(리뷰 P1: 보안 과부하 해소 위해 9·10장으로 분할 — 이 장은 *개념과 멘탈 모델*만):**
  - 인증(authentication) vs 인가(authorization) 기본 개념 — 둘을 헷갈리지 않게
  - **필터체인이라는 새 멘탈 모델** — 요청이 컨트롤러에 닿기 전에 줄지어 선 필터들이 가로챈다. 프런트 다리: Next.js `middleware.ts`/Express 미들웨어 체인과 같은 모양
  - Spring Security 6.x 최소 설정 — `SecurityFilterChain` 빈으로 "무엇을 열고 무엇을 막을지"만 선언(아직 JWT/세션 구현 전, 기본 인증으로 감 잡기)
  - `authorizeHttpRequests`/`requestMatchers`로 경로별 인가 규칙
- **독자가 얻는 것:** 보안을 "마법 어노테이션"이 아니라 "요청을 가로채는 필터 줄"로 이해. 10·11장 구현의 토대.
- **스캐폴드:** 개념 도입형 (비유 중심 — 필터체인)
- **AI 페어코딩 학습 포인트:** AI에게 "Spring Security 필터체인이 요청을 처리하는 순서를 설명해줘"를 묻고, 생성된 최소 설정이 6.x 람다 DSL인지 확인(구버전 함정의 본격 처리는 10장).
- **예상 분량:** 약 4,500자

### 10장. JWT로 인증하기 — stateless 구현과 신선도 함정의 클라이맥스
- **핵심 질문:** 로그인한 사용자를 서버가 *상태 없이* 어떻게 매 요청 알아보나?
- **주요 내용:**
  - 프런트 진입점: "토큰을 localStorage vs httpOnly 쿠키 어디에 저장?" — 프런트가 직접 겪는 갈등
  - JWT(RFC 7519) 구조와 stateless 모델 — 공유 스토어 없이 수평 확장, 짧은 액세스 토큰 + 리프레시
  - Spring Security 6.x로 JWT 인증 필터 구성 — 9장에서 세운 필터체인에 인증 필터를 끼운다. TaskBoard에 로그인·보호된 엔드포인트 추가
  - **신선도 함정 1급 처리(클라이맥스):** AI에게 Security 설정을 시키면 십중팔구 `WebSecurityConfigurerAdapter`를 상속한 *구버전* 코드를 준다 → `SecurityFilterChain` 빈 + 람다 DSL로, `authorizeRequests`/`antMatchers`를 `authorizeHttpRequests`/`requestMatchers`로 잡아 고치는 before/after 실전
  - **단절 완충 예고(리뷰 P1):** 장 끝에 "JWT가 유일한 답은 아니다 — 즉시 로그아웃·폐기가 필요하면? 다음 장에서 다른 길(세션)을 본다"는 한 줄로 11장과 한 호흡으로 묶는다
- **독자가 얻는 것:** stateless 인증을 직접 구현. Security 6.x 최신 설정을 구버전과 구별하는 눈.
- **스캐폴드:** 문제-해결형 (AI의 구버전 코드 → 6.x로 교정)
- **AI 페어코딩 학습 포인트:** **이 장이 신선도 함정의 클라이맥스.** `WebSecurityConfigurerAdapter` 구버전 코드를 잡아내 6.x 람다 DSL로 고치는 것이 핵심 실전. → 부록 B 신선도 체크리스트로 확인.
- **예상 분량:** 약 6,000자
- **신선도 주의:** Security 6.x API 시그니처는 fact-checker가 `01_reference.md` §3·§5와 대조 필수.

### 11장. 다른 길 — Spring Security 세션(stateful)과 트레이드오프
- **핵심 질문:** JWT가 답이라더니, 왜 2025년 들어 다시 세션을 이야기하나?
- **주요 내용:**
  - 세션(stateful) 방식 — 서버 스토어에 세션 저장, 쿠키로 세션 ID 전달
  - 같은 TaskBoard에 *세션 방식을 두 번째로* 적용 — 한 프로젝트에서 두 모델을 체험(10장 끝 예고를 회수)
  - 쿠키 인증과 5장 CORS의 연결: `allowCredentials(true)` + 정확한 `allowedOrigins`(와일드카드 금지) — 프런트가 실제로 막히는 지점
  - 트레이드오프 정면 비교: 즉시 폐기·"모든 기기 로그아웃"·XSS 측면(httpOnly 쿠키 > localStorage)
  - "매 요청 Redis/DB로 JWT 검증 = stateless 이점 잃은 stateful 재발명"이라는 현장 비판
  - 의사결정 가이드: 여러 프런트/모바일 서빙 API라면 JWT, 즉시 폐기·단순함 우선이면 세션
- **독자가 얻는 것:** 보안 방식을 상황에 따라 선택하고 그 이유를 설명하는 판단력.
- **스캐폴드:** 비교 대조형
- **AI 페어코딩 학습 포인트:** AI에게 "JWT와 세션 중 우리 상황엔 뭐가 맞아?"를 묻되, AI의 추천을 우리 요구사항(폐기 필요성·확장성)에 비춰 검증. AI는 일반론, 결정은 사람.
- **예상 분량:** 약 5,500자

### 12장. 서버가 그리는 화면 — Thymeleaf와 SSR vs CSR
- **핵심 질문:** React를 쓰던 내가 왜 굳이 서버 렌더링 템플릿을 배워야 하나?
- **주요 내용:**
  - Thymeleaf 기본 — "데이터 + 템플릿 → 완성 HTML"의 단순한 모델
  - TaskBoard에 간단한 관리자 화면(목록·상세)을 SSR로 추가
  - 프런트 관점의 SSR vs CSR: Next.js SSR/서버컴포넌트와의 멘탈 매핑
  - "굳이 Thymeleaf?" 논쟁 — 관리자·간단 CRUD·SEO 페이지는 SSR이 빠르고 단순, 프런트/백 버저닝·API 계약 불필요(Deblauwe)
  - 결론: 퇴보가 아니라 도구 선택 — 상황에 맞는 렌더링 전략
- **독자가 얻는 것:** 렌더링 전략을 상황별로 판단. SSR을 무기 하나로 추가.
- **스캐폴드:** 비교 대조형
- **AI 페어코딩 학습 포인트:** AI에게 Thymeleaf 템플릿을 생성시키되, 데이터 바인딩 표현식이 의도대로인지 검수. 프런트 경험(JSX)과의 차이를 AI에게 설명받아 빠르게 적응.
- **예상 분량:** 약 5,000자

### 13장. 스펙에서 동작까지 — Claude Code로 기능 하나를 완주하기
- **핵심 질문:** 백엔드를 이제 좀 아는 내가, AI와 함께 실제 기능을 처음부터 끝까지 어떻게 만드나?
- **주요 내용:**
  - 스펙 주도 워크플로우: **스펙 → 설계 → 구현 → 테스트**
  - Cursor(IDE) + Claude Code(터미널 장기작업) 조합, Plan Mode(Shift+Tab)로 스펙 초안·기존 코드 탐색
  - AI에게 컨텍스트 주는 법: 기존 TaskBoard 코드·버전·컨벤션을 프롬프트에 명시, "Ask me questions"로 모호성 해소
  - **"테스트" 단계의 실체(리뷰 P0-3 회수):** 5장(`@WebMvcTest`/MockMvc)·8장(`@DataJpaTest`)에서 이미 본 테스트를 AI 워크플로우의 검증 도구로 통합 — "테스트를 본 적도 없이 테스트하라"는 모순 제거. AI 생성 코드에 슬라이스 테스트를 붙여 동작을 단언
  - **AI 생성 Spring 코드 검증 종합** — 1~12장에서 익힌 모든 검증 능력의 총동원: 버전(jakarta·Security 6.x)·SQL 로그(N+1)·DTO 검증·트랜잭션 경계·테스트
  - 실전 시연: TaskBoard에 새 기능(예: 태그·검색) 하나를 AI와 완주, 생성 코드를 단계별 검증
- **독자가 얻는 것:** AI 페어코딩 실전 워크플로우를 손에 넣고, 쌓은 검증 능력을 한 흐름으로 통합.
- **스캐폴드:** 사례 분석형 + 종합 정리형
- **AI 페어코딩 학습 포인트:** 이 장 자체가 AI 페어코딩 실전의 본진(2장과 수미상관). 학습법이 아니라 실무 워크플로우로서.
- **예상 분량:** 약 7,000자

### 14장. 여기서 어디로 — 여정 정리와 3.x에서 4.x로
- **핵심 질문:** 입문서를 덮은 나는 이제 무엇을 할 수 있고, 다음 한 걸음은 어디인가?
- **주요 내용:**
  - 여정 되짚기: fetch 뒤집기 → REST(+CORS·테스트) → JPA → 보안 세 단계 → SSR → AI 워크플로우가 어떻게 한 흐름이었나(나열 아닌 연결)
  - 통합 관점: TaskBoard 한 프로젝트가 어떻게 모든 개념을 엮었나
  - **버전 신선도 서사의 회수:** 3.x로 배운 토대 위에서 Spring Boot 4.x / Framework 7로의 마이그레이션 전망(jakarta는 이미 익혔으니 다음은 Jackson 3·API versioning 등) — "왜 3.x로 시작했는지"가 여기서 완결
  - 다음 학습 단계: 테스트 심화·배포·관찰 가능성 등 입문 너머의 지도
  - 당부: AI는 동반자이되 검증은 끝까지 사람의 몫
- **독자가 얻는 것:** 학습의 지도와 다음 방향. 입문자에서 "스스로 더 배울 수 있는 사람"으로.
- **스캐폴드:** 종합 정리형
- **AI 페어코딩 학습 포인트:** AI를 활용한 지속 학습 루틴 — 새 버전·새 라이브러리를 AI와 함께 따라잡되 1차 소스로 검증하는 평생 습관.
- **휘발성 경고(P2):** 마이그레이션 항목(Jackson 3·API versioning 등)은 "2026-05 기준, 빠르게 바뀔 수 있다" 안전 문구를 붙인다. fact-checker 대조 대상.
- **예상 분량:** 약 5,000자

## 부록 (선택)

- **부록 A. 프런트엔드 ↔ Spring 멘탈 모델 대조표** — 레퍼런스 §2 표를 독자용 빠른 참조 카드로(fetch↔서버, NestJS DI↔Spring DI, Prisma↔JPA, 토큰 저장 등)
- **부록 B. 신선도 함정 체크리스트** — AI/인터넷 코드에서 잡아내야 할 구버전 패턴 목록(`javax.*`→`jakarta.*`, `WebSecurityConfigurerAdapter`, `authorizeRequests`→`authorizeHttpRequests`, `antMatchers`→`requestMatchers`)과 3.x 기준 치환표
- **부록 C. REST의 엄밀한 정의(Fielding)와 HATEOAS** — 본문에서 유보한 깊이를 원하는 독자용
- **본문↔부록 크로스레퍼런스(리뷰 P2):** 부록이 죽지 않도록 본문에서 명시적으로 가리킨다. 10장 AI 사이드바 → "부록 B 신선도 체크리스트로 확인", 각 프런트 다리 등장 시 → "부록 A 대조표 참조", 5장 REST 정리 → "더 깊이는 부록 C".

## 평가 기준 자가 점검 (점진성 · 연속 프로젝트 · 프런트 대비)

- **점진성:** 난이도 곡선이 개념(1~2) → 동작·빌드(3) → 구조(4~5) → 영속화(6~8) → 보안 3단계(9 기초→10 JWT→11 세션) → 화면(12) → AI 실전(13) → 종합(14)으로 단조 증가. 보안을 9·10·11장으로 쪼개 입문자 최대 난관의 도약을 완충(리뷰 P1). 각 장은 직전 장의 산출물 위에 쌓인다.
- **연속 프로젝트:** TaskBoard 단일 프로젝트가 3장(빈 프로젝트)→5장(메모리 API+CORS+테스트)→6장(DB 영속화)→7장(연관관계)→8장(repo 테스트)→9~11장(보안 기초·JWT·세션)→12장(관리자 화면)→13장(AI로 기능 확장)으로 끊김 없이 성장. "API 계약을 유지한 채 DB를 점진적으로 붙인다"는 사용자 요구를 6장에서 핵심 실습으로 명시.
- **프런트 대비 설명:** 1장(fetch 뒤집기·CORS 증상), 3장(Gradle=package.json·Next API routes), 4장(NestJS/Angular DI), 5장(MockMvc=supertest), 6장(Prisma/Drizzle), 9장(필터체인=Next middleware), 10장(토큰 저장 위치), 12장(Next SSR)에서 매 핵심 개념마다 프런트 멘탈 모델을 다리로 사용.
- **AI 페어코딩의 이중 구조:** 별도 챕터(2장 학습법·13장 실전) + 전 챕터 사이드바(각 장 끝 "AI 페어코딩 학습 포인트"). 사용자 요구 6·7번 충족.
- **버전 신선도 1급 처리:** 2장(개념)→3·6장(import 검수)→10장(Security 클라이맥스)→13·14장(검증·마이그레이션)으로 책 전체 관통. 사용자가 명시한 모든 함정(javax→jakarta, WebSecurityConfigurerAdapter 제거, authorizeRequests→authorizeHttpRequests) 포함.
- **입문자 실패 지점 3종 보강(리뷰 P0):** ① CORS 서버 측 — 1장 증상 제시 → 5장 `CorsConfigurationSource` 설정 → 11장 `allowCredentials` 연결. ② 빌드·실행·디버깅 — 3장 앞 소절(Gradle=npm). ③ 테스트 본문 — 5장(`@WebMvcTest`)·8장(`@DataJpaTest`)에서 도입 → 13장 워크플로우가 회수.

## 레퍼런스 활용·한계 메모

- 레퍼런스는 밀도가 높고 버전 경계가 명확해 계획에 충실히 반영함. 별도 리서치 보강 없이 진행 가능.
- 단, 레퍼런스 §7이 밝힌 한계가 **챕터 저술 단계에서 보강 필요**:
  - 챕터 오프닝용 생생한 한국 커뮤니티 인용(OKKY/velog 원문) — 1·8장 오프닝에서 추가 수집 권장
  - 회사 1차 N+1/트랜잭션 포스트모템(우아한형제들·토스·카카오) — 8장 사례 보강 가치 큼(리뷰 P1: 8장 오프닝 추상화 방지)
  - RFC 1차 조항 인용(9110·6265·7519) — 1·10장에서 보강 권장(리뷰 P2)
- 버전 수치(4.0.6, 3.5 EOL 등)는 빠르게 변하므로 fact-checker가 저술 시점 기준 재확인 필수.
