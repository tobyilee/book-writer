## 리뷰 라운드 1

### 잘된 점 (먼저 인정)

- **연속 프로젝트(TaskBoard)의 점진적 성장 설계가 탄탄하다.** 3장(빈 프로젝트)→5장(메모리 API)→6장(DB 영속화, "API 계약 유지한 채 교체")→7장(연관관계)→9·10장(인증 두 방식)→11장(화면)→12장(AI 확장)으로 끊김이 거의 없다. 특히 6장의 "같은 API 계약을 유지한 채 메모리→DB로 교체"는 점진적 부착 요구를 정확히 잡았다.
- **신선도 함정의 3막 구조(2장 개념 심기 → 9장 클라이맥스 → 12·13장 회수)가 설득력 있다.** javax→jakarta, WebSecurityConfigurerAdapter 제거, authorizeRequests→authorizeHttpRequests가 모두 명시됐고, "AI가 십중팔구 구버전을 준다 → 잡아내는 게 핵심 실전"이라는 9장 설계는 이 책의 진짜 차별점이다.
- **AI 페어코딩 이중 구조(2·12장 별도 챕터 + 전 장 사이드바)가 억지스럽지 않다.** 사이드바가 매 장 "그 장의 개념에 맞는" 검증 행위(6장 import 검수, 8장 N+1 SQL 로그 검증)로 특화돼 장식이 아니라 학습에 붙는다.
- **레퍼런스 충실도가 높다.** Fielding·Fowler·Neward·Evans 1차 문헌, N+1 4종, Security 6.x 시그니처, JWT vs 세션 2025 논쟁이 해당 장에 정확히 매핑됐다.

### P0 (필수 — 승인 전 반드시 반영)

- **[1장 + 신규/5장] CORS가 "이미 아는 것"으로 닫히고 서버 측 처리가 책 어디에도 없다.** 1장 주요 내용(라인 53)에서 CORS를 프런트가 아는 것으로 선언만 하고, 그 뒤로 다시 등장하지 않는다. 그러나 이 책의 독자는 **React/Next 앱에서 Spring API를 때리는 순간 CORS 에러를 100% 만난다.** 입문자가 가장 흔히 막히는 지점이며(평가기준 6 명시), 프런트 독자에게는 특히 치명적이다. 레퍼런스에는 직접 항목이 없으나 §2 표(`fetch`↔서버)의 자연스러운 연장선이다. → **5장(REST 완성) 또는 9장(보안) 끝에 `@CrossOrigin`/`CorsConfigurationSource` 서버 측 설정 소절을 추가**하고, "왜 브라우저만 막고 Postman은 통과하나"를 프런트 관점으로 풀 것. 보안 쿠키(10장)와도 엮인다(쿠키 기반 인증 시 `allowCredentials`).

- **[전반 / 신규 3장 보강] 빌드 도구(Gradle/Maven)·실행·디버깅 오리엔테이션이 통째로 빠졌다.** 3장이 `start.spring.io`에서 바로 엔드포인트로 직행하는데, 자바 빌드 경험이 없는 프런트 독자는 `npm`/`package.json` 세계에서 왔다. **"Gradle = package.json + npm scripts, `./gradlew bootRun` = `npm run dev`"** 같은 멘탈 매핑 없이는 3장에서 첫 좌절을 겪는다. 평가기준 6이 명시한 누락 지점이다. → 3장에 "프로젝트 구조 훑기" 소절(라인 81)을 확장해 **빌드 도구 선택(Gradle 권장)·의존성 추가 위치·실행 명령·IDE에서 디버거 붙이기**를 프런트 도구 대비로 짧게 넣을 것. 별도 장까지는 불필요하나 현재처럼 0줄은 안 된다.

- **[12장 vs 13장] 테스트가 12장 워크플로우의 한 단어로만 등장하고 실체가 없다.** 12장은 "스펙→설계→구현→**테스트**"를 표방하는데(라인 200), 책 전체에 테스트를 가르치는 본문이 없다. 13장에서 "테스트 심화"를 *다음 단계*로 미룬다(라인 216). 그러면 12장의 "테스트"는 검증 능력의 종합이라는 클라이맥스 주장과 모순된다 — 독자는 `@SpringBootTest`/`@WebMvcTest`/MockMvc를 한 번도 본 적 없이 "테스트로 검증하라"는 말을 듣는다. → **5장 또는 8장에 최소한의 슬라이스 테스트(`@WebMvcTest` 컨트롤러 1개, repository 테스트 1개)를 한 소절 심어** 12장이 회수할 기반을 만들 것. 풀 TDD는 범위 밖이 맞지만, "0"은 12장의 약속을 깬다. 프런트 독자는 Jest/Vitest 경험이 있으니 "MockMvc = supertest" 다리도 가능.

### P1 (권장 — 반영 시 책의 완성도가 크게 오름)

- **[9장 단독 7,000자 + 신선도 클라이맥스 = 입문자 과부하 위험].** 9장은 (a)인증/인가 개념 (b)토큰 저장 갈등 (c)JWT 구조 (d)Security 6.x 필터체인 구현 (e)신선도 함정 클라이맥스를 한 장에 욱여넣는다. 입문자에게 Spring Security는 단일 최대 난관인데, 7,000자 한 장에서 "보안 개념 + 미들웨어/필터라는 새 멘탈 모델 + 최신 API + 함정 잡기"가 동시에 쏟아진다 — 점진성(평가기준 2)이 여기서 깨진다. → **9장을 둘로 쪼개는 안을 검토**: 9a "보안의 기초 — 인증/인가와 Security 필터체인이 요청을 가로채는 법"(개념+최소 설정), 9b "JWT로 stateless 인증 구현 + 신선도 함정". 13장 체제를 14장으로 늘리거나, 11장(Thymeleaf, 5,000자)을 가볍게 유지해 총량을 맞출 수 있다. 최소한 9장 안에서 "필터체인이 무엇인지"를 별도 소절로 떼어 도약을 완충할 것.

- **[10장] "JWT가 답이라더니 왜 다시 세션"이라는 핵심 질문(라인 172)이 9장 전개와 충돌할 소지.** 9장이 JWT를 "인증의 첫 문"으로 먼저 깊게 구현한 뒤 10장에서 뒤집으면, 입문자는 "그럼 9장에 쓴 시간은?"이라는 혼란을 겪을 수 있다. 레퍼런스 §4.2는 두 방식을 *대등한 트레이드오프*로 본다. → 9장 도입부에 "여기선 JWT를 먼저 보지만, 이게 유일한 답은 아니다(10장에서 다른 길)"라는 **예고 한 줄**을 심어 두 장을 한 호흡으로 묶을 것. 현재 10장 핵심 질문의 도발 톤은 좋으나, 9장에 복선이 없으면 단절로 읽힌다.

- **[2장 위치] 학습 방법론(2장)을 손이 움직이기 전(3장)에 배치한 건 합리적이나, 입문자 동기 면에서 위험.** 1장(개념)→2장(또 메타적 학습법)으로 두 장 연속 "아직 코드를 안 짠다"가 이어진다. "할 수 있다"는 자신감은 3장에서야 온다. 프런트 독자는 행동 지향적("일단 렌더부터", 레퍼런스 §2 학습순서 관점)이다. → 2장을 현재 5,500자에서 **압축(4,000자대)**하거나, 핵심 학습 계약만 2장에 두고 신선도 함정의 구체 사례는 3장 첫 실습(import 검수)과 더 강하게 묶어 "읽기만 하는 구간"을 줄일 것. 내용을 빼라는 게 아니라 체감 진입을 앞당기라는 제안.

- **[8장] 회사 1차 N+1 포스트모템 보강 — 레퍼런스 §7이 명시한 한계.** 8장은 N+1을 다루지만 오프닝 소재가 추상적일 위험이 있다. 레퍼런스 §7은 우아한형제들·토스·카카오 실전 사례 미수집을 한계로 밝혔다. → 챕터 저술 단계에서 8장 오프닝에 **국내 회사 기술블로그 N+1 사례 1건을 추가 수집**할 것(계획 라인 246에 이미 메모됨 — 실행 보장 차원에서 재확인). 입문자에게 "운영에서 터진다"는 추상론보다 구체 사건이 훨씬 와닿는다.

### P2 (선택 — 여유가 되면)

- **[1장] RFC 9110/6265를 "예고"로만 둔 건 입문서로서 타당하나, 1차 인용 보강 여지.** 레퍼런스 §7이 RFC 본문 미수집을 한계로 밝혔다. 1·9장에서 RFC 조항을 한 줄이라도 직접 인용하면 권위가 올라간다. 입문자 부담을 고려해 사이드바/각주 수준으로.

- **[부록 활용] 부록 A(멘탈 모델 대조표)·B(신선도 체크리스트)가 매우 유용한데 본문에서 참조가 끊길 위험.** 각 장 사이드바나 마무리에서 "→ 부록 B 체크리스트로 확인" 식 **본문↔부록 크로스레퍼런스**를 명시하면 부록이 죽지 않는다.

- **[11장] "退보" 한자 표기(라인 191)** — 입문서 한글 톤에 이질적. "퇴보"로 표기 통일 권장(style-checklist 외래어/표기 항목).

- **[13장] 마이그레이션 전망(3.x→4.x)이 빠르게 휘발하는 정보** — Jackson 3·API versioning 등 구체 항목은 "2026-05 기준, 빠르게 바뀔 수 있다" 안전 문구를 붙일 것(style-checklist 휘발성 경고 항목). fact-checker 대조 대상으로 이미 메모됨.

## 플래너 반영 로그 (라운드 1 피드백 → 02_plan.md 갱신, 2026-05-25)

리뷰 P0 3건 전부 + P1 4건 + P2 전 항목을 반영해 `02_plan.md`를 13→14장으로 갱신했다. 변경 내역과 사유:

### P0 (필수) — 전부 반영
- **P0-1 CORS 서버 측:** 1장 주요 내용에서 CORS를 "이미 아는 것"으로 닫던 항목을 *"증상으로만 꺼내고 서버 책임으로 5장에 넘긴다"* 로 재서술. 5장 제목을 "…·CORS·첫 테스트"로 확장하고 **서버 측 CORS 설정 소절**(`@CrossOrigin` vs `CorsConfigurationSource`, "왜 브라우저만 막고 Postman은 통과하나", preflight, `allowCredentials`) 신설. 11장(세션)에서 쿠키 인증 + `allowCredentials(true)`/`allowedOrigins` 와일드카드 금지를 5장 CORS와 명시적으로 연결.
  - *사유:* 이 독자는 React/Next 앱으로 Spring API 호출 시 CORS를 100% 만나는데 책 어디에도 서버 측 해법이 없었음. 프런트 독자 특화 필수 공백.
- **P0-2 빌드·실행·디버깅:** 3장 제목을 "첫 엔드포인트 — 빌드 도구부터 동작하는 서버까지"로 바꾸고 **앞 소절 신설** — Gradle=`package.json`+npm scripts, `./gradlew bootRun`=`npm run dev`, 의존성 추가 위치, IDE 디버거 붙이기. 분량 5,000→5,500자.
  - *사유:* npm 세계에서 온 독자가 start.spring.io 직후 첫 좌절하는 지점. 0줄→필수 소절.
- **P0-3 테스트 본문:** 5장에 `@WebMvcTest`/MockMvc 슬라이스 테스트 소절(프런트 다리 "MockMvc=supertest"), 8장에 `@DataJpaTest` repository 테스트 소절 신설. 13장(구 12장) "구현→테스트" 단계가 이 둘을 *회수*하도록 명문화.
  - *사유:* 13장이 "스펙→설계→구현→테스트"를 표방하는데 본문 테스트가 0이면 "테스트를 본 적도 없이 테스트하라"는 모순. 5·8장에 미리 심어 해소.

### P1 (권장) — 전부 반영
- **9장 분할:** 구 9장(보안 기초+JWT+필터체인+신선도, 7,000자)을 **9장(보안 기초·필터체인, 4,500자) / 10장(JWT·신선도 클라이맥스, 6,000자)** 으로 분할. 세션은 11장(5,500자)으로 밀림. 책 13→14장. 필터체인을 "Next.js middleware/Express 미들웨어 체인"에 빗대 별도 멘탈 모델 장으로 독립.
  - *사유:* 입문자 단일 최대 난관에 개념+미들웨어+최신 API+함정이 한 장에 폭발 → 점진성 붕괴. 3단계로 완충.
- **9→10 단절 완충:** 10장(JWT) 끝에 "JWT가 유일한 답은 아니다 — 다음 장에서 다른 길(세션)" 예고 한 줄 명시.
- **2장 압축:** 5,500→4,500자. 핵심 *학습 계약*만 2장에 두고 신선도 함정의 *구체 예시*(`javax.*` import 등)는 3장 첫 실습으로 이관 — "코드 안 짜는 장" 1·2장 연속을 완화, 체감 진입을 3장으로 당김.
- **8장 회사 사례:** 오프닝 추상화 방지 위해 국내 회사 기술블로그 N+1 포스트모템 1건을 *저술 시 수집*하도록 "저술 시 보강" 항목으로 실행 보장 명시.

### P2 (선택) — 전부 반영
- 1·10장에 RFC(9110/7519) 1차 조항을 사이드바/각주 한 줄로 인용 권장 명시.
- 부록 A·B·C가 죽지 않도록 **본문↔부록 크로스레퍼런스**(10장→부록 B, 프런트 다리→부록 A, 5장→부록 C) 명시.
