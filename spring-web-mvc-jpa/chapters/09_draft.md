# 9장. 보안의 기초 — 인증·인가와 필터체인이 요청을 가로채는 법

지금 우리 TaskBoard는 누구에게나 활짝 열려 있다. 한번 생각해보자. `localhost:3000`에서 부르든, `curl`로 때리든, 지구 반대편 누군가가 우연히 주소를 알아내든 — 들어오는 요청은 죄다 컨트롤러까지 무사통과한다. 할 일을 만들고, 고치고, 지우는 일을 아무나 할 수 있다. 내 할 일 목록을 옆자리 동료가 마음대로 지워버려도 서버는 군말 없이 200을 돌려준다. 생각만 해도 아찔하지 않은가?

그렇다면 이런 의문이 자연스럽게 따라온다. "이 요청을 보낸 게 대체 누구이고, 그 사람이 이 일을 할 권한이 있나"를 서버는 어떻게 가려낼 수 있을까? 그리고 더 궁금한 건 이것이다. 그 검사를 *어디서* 해야 할까? 컨트롤러마다 맨 앞에 "너 누구야?"를 묻는 `if` 문을 박아 넣어야 하나? 그건 아무리 봐도 번거롭고 빠뜨리기 쉽다.

이번 장에서는 보안의 코드를 짜기 전에, 먼저 그 *멘탈 모델*부터 세워보자. 인증과 인가가 무엇이고 어떻게 다른지, 그리고 Spring Security가 요청을 어떤 방식으로 가로채는지 — 이 둘만 손에 잡히면 다음 두 장(JWT·세션)이 훨씬 수월해진다. 실제 토큰이나 세션 구현은 잠시 미뤄두자. 오늘은 그림을 그리는 날이다.

## 너 누구야, 그리고 그걸 해도 돼? — 인증과 인가

보안 이야기를 시작하면 가장 먼저 헷갈리는 두 단어가 있다. **인증(authentication)**과 **인가(authorization)**다. 영어로 보면 둘 다 'auth'로 시작해서 더 헷갈린다. 하지만 가르는 기준은 의외로 단순하다.

상황을 하나 그려보자. 회사 건물에 들어간다고 해보자. 로비에서 경비원이 사원증을 보자고 한다. 사원증을 대고 "저는 이 회사 직원 김아무개입니다"가 확인되는 것 — 이게 **인증**이다. "네가 누구인지"를 증명하는 단계다. 그런데 사원증이 통과됐다고 해서 모든 문이 열리는 건 아니다. 일반 직원 카드로는 서버실 문이 안 열린다. "김아무개라는 건 알겠는데, 너는 *서버실에 들어갈 권한*은 없어" — 이게 **인가**다. "네가 무엇을 할 수 있는지"를 따지는 단계다.

정리하면 이렇다. 인증은 **신원 확인**이고("너 누구야?"), 인가는 **권한 확인**이다("그걸 해도 돼?"). 순서도 자연스럽게 정해진다. 누구인지 모르면 무엇을 할 수 있는지도 따질 수 없으니, 인증이 먼저고 인가가 나중이다.

우리 TaskBoard에 빗대보자. 로그인해서 "나는 사용자 A다"가 확인되는 게 인증이고, "사용자 A는 자기 할 일은 지울 수 있지만 남의 할 일은 못 지운다"가 인가다. 둘은 분명히 다른 일이다. 이 점을 처음부터 또렷이 갈라두면, 앞으로 보안 코드를 읽을 때 "지금 이건 신원을 따지는 코드인가, 권한을 따지는 코드인가"를 늘 분간할 수 있다. 기억해두자. 이 둘을 헷갈리면 보안 설정 전체가 뿌옇게 보인다.

## 검사는 컨트롤러에 닿기 전에 — 필터체인이라는 새 그림

자, 이제 핵심 질문으로 돌아가자. 그 신원 확인과 권한 확인을 *어디서* 할까?

앞서 잠깐 떠올린 방법 — 컨트롤러마다 맨 앞에 검사 코드를 박는 방법 — 을 다시 보자. 이건 5장에서 검증을 `if` 덩어리로 흩뿌리던 것과 똑같은 실수다. 엔드포인트가 늘어날수록 같은 검사가 복사되고, 어느 하나에서 빠뜨리는 순간 그게 곧 보안 구멍이 된다. 컨트롤러는 "할 일을 만든다"는 본업에만 집중해야 하는데, 매번 "너 누구야?"부터 묻느라 지저분해진다. 뒷맛이 영 찜찜하다.

그렇다면 어떻게 해야 할까? 검사를 컨트롤러 *바깥*으로, 그것도 요청이 컨트롤러에 닿기 *전*으로 끌어내면 된다. 바로 이 지점에서 Spring Security의 핵심 개념이 등장한다. **필터체인(filter chain)**이다.

이 그림은 사실 프런트 개발자에게 그리 낯설지 않다. Next.js를 써봤다면 `middleware.ts`를 떠올려보자. 페이지나 API 라우트에 요청이 도달하기 *전에* 가로채서 "이 사람 로그인했나? 안 했으면 로그인 페이지로 돌려보내자"를 처리하던 그 파일 말이다. Express를 써봤다면 `app.use(authMiddleware)`로 줄줄이 끼워 넣던 미들웨어 체인이 바로 그것이다. 요청이 최종 핸들러에 닿기 전에, 줄지어 선 미들웨어들이 차례로 요청을 만지고 통과시킬지 말지를 정한다.

Spring Security의 필터체인도 정확히 같은 모양이다. 요청 하나가 들어오면, 컨트롤러에 닿기 전에 **줄지어 선 필터들이 차례로 요청을 가로챈다.** 어떤 필터는 "이 요청에 인증 정보가 들어 있나"를 살피고, 어떤 필터는 "이 경로에 접근할 권한이 있나"를 따진다. 줄을 통과한 요청만 비로소 컨트롤러에 도착한다. 통과하지 못한 요청은 컨트롤러 근처에도 못 가고 401(인증 안 됨)이나 403(권한 없음)으로 되돌려진다.

```
요청 → [필터1] → [필터2] → [인증 필터] → [인가 필터] → ... → 컨트롤러
                                  ↑
                  여기서 막히면 컨트롤러는 구경도 못 한다
```

이 그림이 손에 잡히면 보안이 한결 덜 무섭다. Spring Security는 "마법 어노테이션 덩어리"가 아니라, **요청 앞에 쳐둔 검문소들의 줄**이다. 우리가 할 일은 그 검문소들에게 "무엇을 통과시키고 무엇을 막을지"를 일러주는 것뿐이다. 다음 장에서 JWT를, 그다음 장에서 세션을 다룰 텐데, 그건 결국 "이 줄 어딘가에 *어떤* 검문소를 끼워 넣느냐"의 이야기다. 줄 자체의 모양은 지금 그려둔 이 그림 그대로다.

## 최소 설정 — 무엇을 열고 무엇을 막을지만 선언하자

그림을 그렸으니, 이제 그 검문소들에게 말을 거는 가장 단순한 방법을 보자. Spring Security를 프로젝트에 더하면(즉 `build.gradle`에 `spring-boot-starter-security` 의존성을 넣으면), 놀라운 일이 벌어진다. 아무 설정도 안 했는데 *모든* 요청이 갑자기 막힌다. 서버를 띄우면 콘솔에 임시 비밀번호가 찍히고, 어떤 페이지를 열어도 로그인 창부터 들이댄다.

이게 당황스러울 수 있지만, 사실 친절한 기본값이다. Security의 철학은 "일단 전부 잠그고, 열어줄 것만 명시적으로 열어라"이기 때문이다. 안전한 쪽으로 기울어 있는 셈이다. 그렇다면 우리가 할 일은, 이 잠긴 문 중에서 "무엇을 열고 무엇을 닫을지"를 직접 선언하는 것이다.

그 선언을 담는 그릇이 바로 `SecurityFilterChain` 빈이다. 아주 단순한 형태부터 보자.

```java
package com.example.taskboard.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.web.SecurityFilterChain;

@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/tasks/**").authenticated()
                .anyRequest().permitAll()
            )
            .httpBasic(basic -> {});  // 지금은 가장 단순한 기본 인증으로 감만 잡는다

        return http.build();
    }
}
```

> **(사실 확인 필요)** 위 `SecurityFilterChain` 빈 구성, `authorizeHttpRequests`·`requestMatchers`·`httpBasic` 람다 DSL의 메서드 시그니처가 Spring Security 6.x 기준으로 정확한지 레퍼런스 §3·§5와 대조 바람.

코드가 짧으니 한 줄씩 같이 읽어보자. 우리가 만드는 건 `SecurityFilterChain`이라는 빈 하나다. 4장에서 익힌 그 빈, 맞다. Security에게 "내 필터체인은 이렇게 구성해줘"라고 건네는 설계도라고 생각하면 된다.

`authorizeHttpRequests`는 "요청을 인가하는 규칙을 여기 적겠다"는 뜻이다. 그 안에서 `requestMatchers("/api/tasks/**")`로 경로를 집어 "이 경로로 오는 요청은"이라고 운을 떼고, `.authenticated()`로 "인증된 사용자만 통과시켜라"라고 못 박는다. 그리고 `.anyRequest().permitAll()`로 "그 밖의 나머지 요청은 다 열어둬라"라고 마무리한다. 읽어 내려가면 거의 문장처럼 읽힌다 — "할 일 API는 로그인한 사람만, 나머지는 누구나."

마지막의 `httpBasic`은 지금 단계에서 "가장 단순한 인증 방식으로 일단 감만 잡자"는 뜻이다. 브라우저가 사용자 이름과 비밀번호를 물어 실어 보내는, 가장 원시적인 방식이다. 실전에서 우리가 쓸 건 아니다. 진짜 인증 — JWT와 세션 — 은 다음 두 장의 몫이다. 여기서는 "필터체인에 인가 규칙을 선언한다는 게 이런 모양이구나"만 눈에 익히면 충분하다.

여기서 잠깐, 신선도 이야기를 하나 짚고 가자. 만약 AI에게 "Spring Security 설정 코드를 짜줘"라고 하면, 십중팔구 `WebSecurityConfigurerAdapter`라는 클래스를 *상속*하고 `configure(HttpSecurity)` 메서드를 *오버라이드*하는 코드를 자신 있게 내놓을 것이다. 그리고 그 안에는 `authorizeRequests`나 `antMatchers` 같은 이름이 보일 것이다. 이건 모두 **구버전 패턴**이다. `WebSecurityConfigurerAdapter`는 Spring Security 6.x에서 아예 제거됐고(우리가 쓰는 Spring Boot 3.x가 이 6.x를 품는다), `authorizeRequests`는 `authorizeHttpRequests`로, `antMatchers`는 `requestMatchers`로 이름이 바뀌었다. 우리가 위에서 쓴 게 바로 새 이름들이다.

지금은 "그런 함정이 있구나" 정도만 알아두자. 이 신선도 함정을 *제대로* 터뜨려 before/after로 잡아내는 건 다음 장, JWT를 구현하면서다. 거기가 진짜 클라이맥스다. 오늘은 그저 "AI가 주는 Security 코드는 특히 더 의심하고 봐야 한다"는 예고편만 마음에 담아두면 된다.

## 비밀번호는 절대 날것으로 두지 말자 — BCrypt

인증 이야기가 나온 김에, 코드는 아직 안 짜더라도 개념 하나는 미리 심어두자. 사용자를 인증하려면 결국 비밀번호를 어딘가에 저장해둬야 한다. 그런데 비밀번호를 입력받은 그대로, 즉 날것의 평문으로 데이터베이스에 저장하면 어떻게 될까?

끔찍한 일이다. 만에 하나 DB가 통째로 유출되면, 모든 사용자의 비밀번호가 그대로 적의 손에 들어간다. 게다가 사람들은 같은 비밀번호를 여기저기서 돌려쓰는 경우가 많으니, 우리 서비스의 유출이 그 사람의 다른 계정까지 줄줄이 무너뜨린다. 이건 상상만으로도 아찔하다.

그래서 비밀번호는 **절대 날것으로 저장하지 않는다.** 대신 한 방향으로만 뭉개버리는 함수를 통과시켜 알아볼 수 없는 형태로 만든 뒤 저장한다. 이걸 **해싱**이라 하고, Spring Security 진영에서 비밀번호 해싱의 사실상 표준으로 쓰는 게 **BCrypt**다. BCrypt의 영리한 점은, 일부러 계산을 *느리게* 만들어 무차별 대입 공격을 어렵게 한다는 데 있다. 빠른 게 미덕인 세상에서 일부러 느린 함수라니 묘하지만, 보안에서는 그 느림이 곧 방패다.

쓰는 모양도 어렵지 않다. `PasswordEncoder`라는 빈을 하나 등록해두고, 비밀번호를 저장할 때 인코딩하고, 로그인 시 들어온 비밀번호와 대조할 때 이 인코더에게 맡기면 된다.

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

> **(사실 확인 필요)** `BCryptPasswordEncoder` 클래스명과 `PasswordEncoder` 빈 등록 방식이 Spring Security 6.x 기준으로 정확한지 레퍼런스 §3·§5와 대조 바람.

지금은 이 인코더를 본격적으로 쓰진 않는다. 다음 장에서 로그인을 구현할 때 비로소 활약한다. 다만 **"비밀번호는 해싱해서 저장한다, 그 도구가 BCrypt다"** 이 한 문장만은 지금 단단히 새겨두자. 입문자가 가장 흔히 저지르는 보안 실수가 바로 비밀번호를 평문으로 다루는 것이다. 이 함정만 피해도 절반은 한 셈이다.

> **AI 페어코딩 학습 포인트**
>
> 보안만큼은 AI에게 코드를 받기 전에 *질문부터* 던져보자. 예컨대 "Spring Security 6.x 필터체인이 요청을 처리하는 순서를 설명해줘"라고 물어, 오늘 우리가 그린 "검문소의 줄" 그림을 보강하는 것이다. 그다음 AI가 내놓은 설정 코드를 받으면 두 가지를 *반드시* 확인하자. 첫째, `WebSecurityConfigurerAdapter`를 상속하고 있지는 않은가(상속한다면 구버전이다). 둘째, `authorizeHttpRequests`·`requestMatchers` 같은 6.x 람다 DSL을 쓰고 있는가, 아니면 `authorizeRequests`·`antMatchers` 같은 옛 이름인가. 프롬프트에 "Spring Boot 3.x, Spring Security 6.x 기준으로" 한 줄을 못 박아두면 함정에 빠질 확률이 줄어든다. 다만 그래도 AI는 옛 코드를 섞어 줄 때가 많으니, 우리 눈으로 거르는 습관을 들이는 편이 낫다. 신선도 함정의 본격적인 before/after 교정은 부록 B 신선도 체크리스트와 함께, 다음 장에서 정면으로 다룬다.

## 마무리

이번 장에서 우리는 코드를 거의 짜지 않았다. 대신 앞으로 두 장을 버틸 *그림 두 장*을 손에 넣었다. 하나는 **인증과 인가의 구분** — "너 누구야?"(신원)와 "그걸 해도 돼?"(권한)는 다른 질문이라는 것. 다른 하나는 **필터체인** — Spring Security는 마법 덩어리가 아니라, 요청이 컨트롤러에 닿기 전에 줄지어 가로채는 검문소들의 줄이고, 그 모양은 Next.js 미들웨어나 Express 미들웨어 체인과 같다는 것. 그리고 그 검문소에게 "무엇을 열고 무엇을 막을지"를 일러주는 그릇이 `SecurityFilterChain` 빈이라는 것까지 보았다. 비밀번호는 BCrypt로 해싱해 저장한다는 원칙도 미리 심어두었다.

기억해두자. 보안이 무섭게 느껴지는 건 대개 "요청이 어디서 어떻게 검사되는지"가 안 보이기 때문이다. 그 흐름을 그림으로 그려두면, 그다음부터는 "이 줄 어디에 어떤 필터를 끼울까"의 문제로 단순해진다.

다음 장에서는 드디어 그 줄에 *진짜 인증 필터*를 끼운다. 로그인한 사용자를 서버가 상태 없이(stateless) 어떻게 매 요청마다 알아보는지 — **JWT**로 직접 구현해본다. 그리고 그 과정에서, 오늘 살짝 예고한 그 신선도 함정 — `WebSecurityConfigurerAdapter`를 들고 오는 AI의 구버전 코드 — 을 정면으로 마주해 6.x 방식으로 고쳐낸다. 이 책 전체를 관통하는 신선도 서사의 클라이맥스다. 그럼, 오늘 그린 그림을 들고 다음 장으로 넘어가 보자.
