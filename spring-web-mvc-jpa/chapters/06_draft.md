# 6장. 메모리에서 데이터베이스로 — JPA를 붙이다

서버를 잘 띄워두고 TaskBoard에 할 일을 차곡차곡 쌓았다고 해보자. "장보기", "운동하기", "이 장 끝내기"까지 정성껏 넣었다. 그런데 코드를 한 줄 고쳐 서버를 다시 띄우는 순간, 방금 넣은 할 일들이 흔적도 없이 사라진다. `Ctrl+C` 한 번에 모든 것이 백지가 된다. 처음 겪으면 정말 허탈하다. 분명히 잘 동작했는데, 데이터가 증발한 것이다.

왜 이런 일이 벌어질까? 지금까지 우리의 할 일 목록은 자바 객체로, 서버 프로세스의 **메모리(RAM)** 안에 살고 있기 때문이다. 메모리는 빠르지만 휘발성이다. 프로세스가 죽으면 그 안의 모든 것이 함께 사라진다. 프런트로 치면, 새로고침 한 번에 날아가는 `useState` 값과 똑같다. 화면 안에서만 잠깐 사는 상태. 우리에게 필요한 건 서버를 껐다 켜도, 심지어 서버가 통째로 죽었다 살아나도 남아 있는 데이터다. 그러려면 데이터를 **메모리 바깥의 영구적인 저장소**, 즉 데이터베이스에 눌러 담아야 한다.

자, 그렇다면 여기서 가장 중요한 질문을 던져보자. **우리가 5장까지 공들여 만든 그 API를, 데이터베이스를 붙이느라 다 갈아엎어야 할까?** 컨트롤러도, DTO도, 검증 규칙도 처음부터 다시? 다행히 그렇지 않다. 이 장의 핵심은 바로 여기에 있다 — **바깥에서 보이는 API 계약은 그대로 둔 채, 안쪽 저장소만 메모리에서 데이터베이스로 갈아 끼우는 것.** 클라이언트(우리의 React 앱)는 아무것도 달라진 걸 못 느끼는데, 데이터는 이제 영영 사라지지 않는다. 이 점진적 부착이 오늘의 주인공이다.

## 객체와 표(table)는 원래 사이가 나쁘다

데이터베이스 이야기를 꺼내면 곧장 SQL이 떠오를 것이다. `INSERT INTO tasks ...`, `SELECT * FROM tasks WHERE ...` 같은 것들. 프런트를 하면서도 한 번쯤은 써봤을 그 SQL 말이다. 그런데 잠깐 멈추고 생각해보자. 우리가 자바에서 다루는 건 `Task`라는 **객체**다. 필드를 갖고, 메서드를 갖고, 다른 객체를 참조한다. 반면 관계형 데이터베이스가 다루는 건 행과 열로 이루어진 **표**다. 이 둘은 생긴 게 너무 다르다.

객체의 세계에는 상속이 있고, 객체가 객체를 직접 가리키는 참조가 있다. 하지만 표의 세계에는 그런 게 없다. 표에는 외래 키(foreign key)라는 숫자 하나로 다른 표를 가리킬 뿐이다. 객체는 메모리 주소로 "같은 객체인지"를 따지지만, 표는 기본 키(primary key) 값으로 "같은 행인지"를 따진다. 이렇게 객체 모델과 관계형 모델 사이에 벌어진 근본적인 간극을, 어려운 말로 **임피던스 불일치(impedance mismatch)**라고 부른다. 이름은 거창하지만, 뜻은 단순하다. "객체랑 표는 원래 사이가 안 좋다"는 것이다.

그래서 옛날 개발자들은 이 간극을 손으로 메웠다. `SELECT`로 행을 읽어와 한 칼럼씩 꺼내 객체 필드에 일일이 옮겨 담고, 저장할 땐 객체 필드를 다시 `INSERT` 문자열로 조립했다. 이 지루하고 실수투성이인 변환 작업을 자동으로 해주겠다고 나선 도구가 바로 **ORM(Object-Relational Mapping)**, 객체-관계 매핑이다. 자바 진영의 ORM 표준이 **JPA(Jakarta Persistence API)**이고, 그 구현체로 가장 널리 쓰이는 것이 **Hibernate**다. Spring Data JPA는 이 위에 한 겹을 더 씌워 우리 손을 한층 덜어준다.

여기서 한 가지는 솔직하게 못 박아두자. **ORM은 마법이 아니다.** ORM은 우리가 객체를 다루면 그에 맞는 SQL을 대신 생성해 데이터베이스에 보내주는 도구일 뿐이다. SQL을 없애주는 게 아니라, 우리 대신 SQL을 써주는 것이다. 이 차이가 결정적이다. 객체와 표의 간극을 ORM이 메워주긴 하지만, 그 추상화는 완벽하지 않다. 편할 땐 한없이 편하지만, 어느 순간 ORM이 만들어낸 SQL이 우리 기대와 다르게 동작해 발목을 잡는 날이 온다. 테드 네워드(Ted Neward)는 일찍이 이 ORM의 양면성을 두고 "컴퓨터 과학의 베트남"이라 부르기도 했다(2006). 편의를 좇아 들어갔다가 빠져나오기 힘든 수렁이 될 수 있다는 경고다.

그러니 우리가 가질 자세는 분명하다. **ORM이 생성하는 SQL을 끝까지 볼 줄 알아야 한다.** 이건 협박이 아니라 약속이다. 이 장에서는 일단 ORM의 편안함을 마음껏 누리되, 그 아래에서 어떤 SQL이 만들어지는지 곁눈질하는 습관부터 들이자. 그 SQL이 본격적으로 우리를 괴롭히는 이야기는 8장에서 정면으로 다룬다.

> 프런트에서 Prisma나 Drizzle을 써본 적 있다면, JPA가 그리 낯설지만은 않을 것이다. 셋 다 "쿼리를 직접 안 쓰고 객체/모델로 데이터를 다룬다"는 같은 약속을 한다. 다만 결을 짚어두자. Prisma는 우리가 어떤 동작을 할지 비교적 또렷하게 드러내는 편이고, JPA는 영속성 컨텍스트라는 보이지 않는 장치(7장에서 다룬다)가 더 많은 일을 알아서 처리한다. 편한 만큼 "내가 시키지 않은 SQL이 나가는" 순간이 더 잦다는 뜻이기도 하다. 프런트↔Spring 대조는 부록 A 대조표도 함께 보자.

## 객체에 "이건 테이블이다"라고 표식 붙이기 — @Entity

이론은 이쯤 해두고, 손을 움직이자. 지금 우리의 할 일은 평범한 자바 객체다. 이 객체에게 "너는 데이터베이스의 한 행(row)에 대응한다"고 알려주는 일부터 시작한다. 그 표식이 바로 `@Entity` 어노테이션이다.

3장에서 우리는 `com.example.taskboard` 패키지에서 출발했다. 그 자리에 `Task`라는 엔티티 클래스를 만들어보자.

```java
package com.example.taskboard;

import jakarta.persistence.Entity;
import jakarta.persistence.GeneratedValue;
import jakarta.persistence.GenerationType;
import jakarta.persistence.Id;

@Entity
public class Task {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    private boolean done;

    protected Task() {
        // JPA가 내부적으로 쓰는 기본 생성자
    }

    public Task(String title) {
        this.title = title;
        this.done = false;
    }

    // getter / 상태 변경 메서드는 지면 관계상 생략
    public Long getId() { return id; }
    public String getTitle() { return title; }
    public boolean isDone() { return done; }

    public void markDone() { this.done = true; }
}
```

한 줄씩 짚어보자. 클래스 위의 `@Entity`는 "이 클래스는 데이터베이스 테이블과 짝을 이룬다"는 선언이다. 별다른 설정을 안 하면 JPA는 클래스 이름을 따 `task`라는 이름의 테이블을 가정한다. 필드 하나하나는 테이블의 칼럼(column)이 된다. `title`은 `title` 칼럼, `done`은 `done` 칼럼이 되는 식이다.

`@Id`가 붙은 `id` 필드가 보이는가? 이건 "이 필드가 이 테이블의 기본 키(primary key)다"라는 뜻이다. 아까 말했듯 표의 세계에서는 기본 키 값으로 행을 구별하니, 모든 엔티티에는 반드시 기본 키가 있어야 한다. 그 아래 `@GeneratedValue(strategy = GenerationType.IDENTITY)`는 "이 키 값은 내가 정하지 않을 테니, 데이터베이스가 행을 넣을 때 자동으로 1, 2, 3... 하고 채워달라"는 부탁이다. 프런트로 치면, 새 항목을 만들 때 우리가 `id`를 직접 만들지 않고 서버가 채워주길 기대하던 것과 같은 감각이다.

여기서 신선도 함정 하나를 미리 짚고 가자. import 문을 다시 보면 전부 `jakarta.persistence.*`로 시작한다. **`javax.persistence`가 아니다.** 3장에서 한 번 겪었듯, AI나 오래된 블로그는 십중팔구 `javax.persistence.Entity`를 자신만만하게 내놓는다. 그 코드를 그대로 Spring Boot 3.x 프로젝트에 붙이면 컴파일조차 되지 않는다. 처음 보면 "분명 시키는 대로 했는데 왜 안 되지?" 하고 한참 헤매게 된다. 정말 난감한 순간이다. 그러니 엔티티를 만들 때마다 import 첫 줄이 `jakarta`로 시작하는지 눈으로 확인하는 습관을 들이자. 이 작은 버릇 하나가 한 시간짜리 삽질을 막아준다.

> 그런데 왜 `protected Task()`라는 빈 생성자가 필요할까? JPA가 데이터베이스에서 읽어온 행을 객체로 되살릴 때, 일단 빈 객체를 하나 만든 뒤 값을 채워 넣기 때문이다. 그래서 인자 없는 기본 생성자가 반드시 필요하다. 다만 우리 코드에서 실수로 빈 `Task()`를 만드는 걸 막으려고 `protected`로 살짝 잠가둔 것이다. 지금은 "JPA가 요구하는 약속" 정도로만 알아두면 충분하다.

## 인터페이스 한 줄로 CRUD가 끝난다 — JpaRepository

엔티티를 정의했으니, 이제 이 객체를 데이터베이스에 저장하고 다시 꺼내올 통로가 필요하다. 직접 SQL을 쓰던 시절이라면 `INSERT`, `SELECT`, `UPDATE`, `DELETE`를 일일이 손으로 적었을 것이다. 그런데 Spring Data JPA의 세계에서는 그 통로를 만드는 데 코드를 거의 쓰지 않는다. 인터페이스 하나만 선언하면 된다.

```java
package com.example.taskboard;

import org.springframework.data.jpa.repository.JpaRepository;

public interface TaskRepository extends JpaRepository<Task, Long> {
}
```

이게 전부다. 정말이다. `JpaRepository<Task, Long>`를 상속하는 빈 인터페이스 하나를 선언했을 뿐인데, 우리는 방금 `save()`(저장), `findById()`(하나 조회), `findAll()`(전체 조회), `deleteById()`(삭제) 같은 기본 CRUD 메서드 한 묶음을 통째로 손에 넣었다. 꺾쇠 안의 `Task`는 "이 저장소가 다루는 엔티티"이고, `Long`은 "그 엔티티의 기본 키 타입"이다.

이쯤 되면 의문이 생기는 게 정상이다. **구현 코드를 한 줄도 안 썼는데, 어떻게 `save()`가 실제로 동작한다는 걸까?** 인터페이스는 그냥 약속일 뿐, 알맹이가 없지 않은가? 3장에서 느꼈던 그 마법 같은 찜찜함이 또 고개를 든다. 좋은 감각이다. 그 정체를 잠깐 들춰보자.

비밀은 이렇다. 우리가 인터페이스만 선언하면, Spring Data JPA가 애플리케이션을 띄울 때 **그 인터페이스를 구현한 객체(프록시proxy)를 자동으로 만들어** 컨테이너에 등록한다. 우리는 그 만들어진 구현체를 4장에서 배운 의존성 주입으로 건네받아 쓰기만 하면 된다. 직접 만든 적 없는 구현체가 어디선가 와서 일을 해주는 것이다.

사실 이건 새로운 발상이 아니다. "데이터 저장소에 접근하는 통로를 하나의 객체로 추상화한다"는 생각은 에릭 에반스(Eric Evans)가 2003년 책 『도메인 주도 설계』에서 정리한 **리포지토리(Repository) 패턴**의 구현이다. 마치 컬렉션(리스트)에서 객체를 넣고 빼듯이 데이터를 다루게 하되, 그 뒤에서 벌어지는 SQL 대화는 숨기는 것이다. Spring Data JPA는 이 고전적인 패턴을 "인터페이스만 선언하면 구현은 알아서"라는 형태로 극단까지 편하게 만들어준 셈이다.

## 메모리를 데이터베이스로 갈아 끼우기 — 점진적 부착의 핵심

이제 가장 중요한 대목이다. 5장에서 우리는 TaskBoard의 비즈니스 로직을 서비스 계층(`TaskService`)으로 분리해뒀다. 컨트롤러는 HTTP를 받아 DTO로 주고받고, 실제 할 일을 저장하고 꺼내는 일은 서비스가 맡는 구조다. 그때 서비스 안에서는 데이터를 메모리, 예컨대 `List`나 `Map` 같은 자바 컬렉션에 담아두고 있었을 것이다.

이제 그 메모리 저장소를 방금 만든 `TaskRepository`로 갈아 끼운다. **단, 컨트롤러와 DTO는 손대지 않는다.** 바로 이게 점진적 부착의 핵심이다. 그림으로 그리면 이렇다.

```
[변경 전]  컨트롤러 → 서비스 → 메모리(List/Map)
[변경 후]  컨트롤러 → 서비스 → TaskRepository → 데이터베이스
                ↑             ↑
            (그대로)      (여기 안쪽만 교체)
```

서비스 코드는 이런 모양으로 바뀐다. 메모리에 직접 담고 꺼내던 부분을, 리포지토리 호출로 옮기는 것이다.

```java
package com.example.taskboard;

import org.springframework.stereotype.Service;
import java.util.List;

@Service
public class TaskService {

    private final TaskRepository taskRepository;

    // 4장에서 배운 생성자 주입 — Spring이 구현체를 건네준다
    public TaskService(TaskRepository taskRepository) {
        this.taskRepository = taskRepository;
    }

    public Task create(String title) {
        Task task = new Task(title);
        return taskRepository.save(task);   // 메모리에 add 하던 자리
    }

    public List<Task> findAll() {
        return taskRepository.findAll();    // 메모리에서 꺼내던 자리
    }

    public Task findById(Long id) {
        return taskRepository.findById(id)
                .orElseThrow(() -> new TaskNotFoundException(id));
    }
}
```

여기서 잠깐 멈추고 음미해보자. 컨트롤러는 여전히 `taskService.create(...)`를 부르고, 여전히 같은 DTO로 응답한다. 클라이언트가 보내는 요청도, 받는 JSON 응답도 한 글자 안 바뀌었다. 우리의 React 앱은 서버 안에서 메모리가 데이터베이스로 바뀌었다는 사실을 **전혀 눈치채지 못한다.** 그런데 이제 서버를 껐다 켜도 데이터는 멀쩡히 남아 있다. 바깥 계약은 그대로, 안쪽 실체만 교체. 이 깔끔한 갈아 끼우기가 가능한 건, 우리가 컨트롤러·서비스·저장소를 계층으로 잘 나눠둔 덕분이다. 5장에서 흘린 땀이 여기서 빛을 발한다.

이것이 실무에서 코드를 키워나가는 건강한 방식이다. 한 번에 전부 갈아엎지 않고, 바깥 약속을 지키면서 안쪽을 한 겹씩 개선한다. 기억해두자. **좋은 계층 분리는 "나중에 갈아 끼울 자유"를 미리 사두는 일이다.**

## H2 데이터베이스와 application.yml — 일단 동작부터

그런데 한 가지 빠진 게 있다. 데이터베이스에 저장한다고 했는데, 정작 그 데이터베이스는 어디 있는가? 진짜 데이터베이스(MySQL이나 PostgreSQL 같은)를 깔고 설정하는 일은, 입문 단계에서 첫 좌절을 부르기 딱 좋다. 설치하고, 계정 만들고, 접속 정보 맞추다 보면 정작 배우려던 JPA는 시작도 못 한다. 정말 번거로운 일이다.

그래서 개발과 학습 단계에서는 **H2**라는 가벼운 데이터베이스를 쓰자. H2는 자바로 만들어진 데이터베이스라 설치가 따로 필요 없고, 의존성에 추가하는 것만으로 서버가 뜰 때 함께 메모리 안에서 깜빡 살아난다. 설정도 거의 없이 곧장 SQL을 받아준다. "진짜 데이터베이스를 붙이기 전, 손쉽게 연습할 수 있는 모래밭" 정도로 생각하면 된다.

먼저 `build.gradle`의 `dependencies`에 두 줄을 더한다. JPA를 쓰겠다는 선언과, H2 데이터베이스다.

```groovy
dependencies {
    // 3장부터 있던 것들
    implementation 'org.springframework.boot:spring-boot-starter-web'

    // 이번 장에서 추가
    implementation 'org.springframework.boot:spring-boot-starter-data-jpa'
    runtimeOnly 'com.h2database:h2'
}
```

다음으로 설정이다. 3장에서 우리 프로젝트에는 `application.properties` 파일이 있었다. 같은 자리에 같은 역할을 하면서 더 읽기 좋은 형식인 `application.yml`로 바꿔 쓰는 경우가 많으니, 여기서는 YAML로 적어보자(둘 중 무엇을 쓰든 효과는 같다).

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:taskboard      # 메모리에 뜨는 H2
    username: sa
    password:
  jpa:
    hibernate:
      ddl-auto: create-drop          # 시작 시 테이블 생성, 종료 시 삭제
    show-sql: true                    # 실행되는 SQL을 콘솔에 찍어준다
```

설정값 몇 개만 골라 짚어보자. 나머지는 지금 깊이 알 필요 없다.

`ddl-auto`는 "애플리케이션이 뜰 때 테이블 구조(DDL)를 어떻게 다룰지"를 정한다. 여기 적은 `create-drop`은 "서버가 뜰 때 엔티티에 맞춰 테이블을 새로 만들고, 서버가 꺼질 때 깨끗이 지운다"는 뜻이다. 매번 백지에서 시작하니 연습에는 편하다. 다만 단어 그대로 데이터를 지우는 옵션이라, **진짜 데이터를 보존해야 하는 운영 환경에서는 절대 쓰면 안 된다.** 운영에서 `create-drop`이나 `create`를 켜두면 서버를 띄울 때마다 데이터가 통째로 날아가는 끔찍한 사고가 난다. 이 점은 꼭 기억해두자. 운영에서는 보통 테이블을 건드리지 않는 `validate`나 `none`을 쓰고, 스키마 변경은 별도 도구로 관리한다. (이 부분은 운영 정책에 따라 갈리니, 실제 배포 전에 공식 문서를 함께 확인하는 편이 낫다.)

`show-sql: true`는 우리가 앞서 한 약속과 곧장 맞닿는다. 이 옵션을 켜두면, JPA가 객체를 다룰 때 **실제로 어떤 SQL을 만들어 데이터베이스에 보내는지** 콘솔에 그대로 찍어준다. 우리가 `taskRepository.save(task)` 한 줄을 부르면, 콘솔에는 이런 SQL이 흐른다.

```sql
insert into task (done, title, id) values (?, ?, default)
```

`findAll()`을 부르면 또 이런 게 찍힌다.

```sql
select t1_0.id, t1_0.done, t1_0.title from task t1_0
```

여기서 작은 깨달음이 온다. 우리가 자바 메서드를 불렀을 뿐인데, 그 아래에서 ORM이 우리가 알던 그 SQL을 대신 써서 보내고 있었던 것이다. `save()`는 `INSERT`로, `findAll()`은 `SELECT`로. 객체 세계의 명령이 표 세계의 SQL로 번역되는 그 현장을, 이제 우리는 두 눈으로 본다. 마법처럼 보이던 것이 사실은 "내가 아는 SQL을 자동으로 써주는 일"이었음을 확인하는 순간이다.

이 `show-sql`을 켜두는 습관은 입문자에게 특히 귀하다. ORM이 무슨 SQL을 만드는지 늘 곁눈질하다 보면, 8장에서 다룰 N+1 같은 함정도 "어, 왜 SELECT가 이렇게 여러 번 나가지?" 하고 스스로 알아차릴 수 있게 된다. 지금은 그 눈을 길들이는 첫 단계다.

> 한 가지 더. 우리가 쓴 H2는 `mem`, 즉 메모리 위에 뜬다. "데이터를 영속화한다더니 또 메모리냐"고 의아할 수 있다. 맞다, H2를 이렇게 쓰면 서버가 꺼질 때 데이터가 사라진다. 다만 핵심은 달라졌다. 이제 데이터는 **자바 객체가 아니라 데이터베이스가 관리하는 진짜 테이블**에 들어 있고, 우리는 `ddl-auto`를 `validate`로 바꾸고 `url`만 진짜 데이터베이스(MySQL·PostgreSQL 등)로 갈아 끼우면 **코드 한 줄 안 고치고** 영구 저장으로 넘어갈 수 있다. 메모리에서 데이터베이스로의 진짜 이사는 이미 끝났다. 실제 데이터베이스로 옮기는 일은 접속 정보를 바꾸는 설정 문제로 줄어든 것이다.

## AI에게 엔티티를 맡길 때 — 매핑은 의심하고, 네임스페이스는 확인하자

::: tip AI 페어코딩 학습 포인트
엔티티와 리포지토리를 만드는 일은 AI가 가장 잘하는 영역 중 하나다. 정형화된 패턴이라 스캐폴드를 척척 뽑아준다. 그래서 더더욱, 받아든 코드를 그냥 믿고 넘어가기 쉽다. 바로 거기에 함정이 있다.
:::

AI에게 "Task 엔티티와 JPA 리포지토리를 만들어줘"라고 부탁한다고 해보자. 그럴듯한 코드가 순식간에 나온다. 그런데 그대로 붙이기 전에 두 가지는 꼭 우리 눈으로 검수하자.

첫째, **네임스페이스다.** 3장에서 겪은 그 함정이 엔티티에서 가장 자주 튀어나온다. AI가 `import javax.persistence.Entity`, `import javax.persistence.Id`를 줬다면 그건 구버전 코드다. Spring Boot 3.x에서는 전부 `jakarta.persistence.*`여야 한다. import 첫 줄만 봐도 1초 만에 판별할 수 있으니, 받자마자 가장 먼저 확인하자. 프롬프트에 "Spring Boot 3.x, jakarta 네임스페이스"라고 못 박아두면 실수가 줄지만, 그래도 마지막 확인은 사람의 몫이다.

둘째, **매핑이 내 의도와 맞는지다.** AI는 우리 머릿속 스키마를 모른다. `title`을 길이 제한 없는 칼럼으로 잡았는지, 꼭 있어야 할 값을 `nullable`로 열어뒀는지, 기본 키 생성 전략이 우리가 쓰는 데이터베이스와 맞는지 — 이런 건 AI가 일반론으로 채워 넣은 것이라, 우리 요구와 어긋날 수 있다. 그러니 받아든 엔티티를 읽으며 "이 매핑이 내가 원하는 테이블 모양과 같은가?"를 따져보자. 확신이 안 서면, 앞서 켜둔 `show-sql`로 실제 생성되는 테이블·SQL을 눈으로 확인하면 된다. **AI는 스캐폴드를 만들고, 그 스캐폴드가 우리 의도와 맞는지 판단하는 건 우리다.** 이 장에서도 그 원칙은 흔들리지 않는다.

## 마무리

이번 장에서 우리는 TaskBoard의 데이터를 휘발하는 메모리에서 데이터베이스로 옮겨 담았다. 객체와 표 사이의 임피던스 불일치를 ORM이 메워준다는 것, 그러나 ORM은 마법이 아니라 "SQL을 대신 써주는 도구"일 뿐이라는 것을 짚었다. `@Entity`로 객체에 테이블의 표식을 붙였고, `JpaRepository` 인터페이스 한 줄 선언으로 CRUD를 손에 넣었으며, 무엇보다 **API 계약을 그대로 둔 채 안쪽 저장소만 갈아 끼우는** 점진적 부착을 직접 해봤다. H2와 `application.yml`로 손쉽게 환경을 띄웠고, `show-sql`을 켜 우리가 부른 자바 메서드가 어떤 SQL로 번역되는지 두 눈으로 확인했다.

그런데 데이터를 저장하고 꺼내다 보면, 머지않아 고개를 갸웃하게 하는 일을 만난다. 분명 `save()`를 부르지 않았는데 값이 바뀌어 저장돼 있거나, 객체 하나를 꺼냈을 뿐인데 연결된 다른 객체까지 슬그머니 따라오는 것이다. JPA가 보이지 않는 곳에서 부지런히 일을 하고 있다는 증거인데, 그 "보이지 않는 일꾼"의 정체를 모르면 동작을 예측할 수가 없어 영 찜찜하다.

다음 장에서는 그 일꾼의 이름을 부른다. 트랜잭션이 도는 동안 우리 엔티티를 하나하나 추적하는 **영속성 컨텍스트**, 그리고 그것이 부리는 더티 체킹과 지연 로딩의 마법을 정면으로 들여다보자. 오늘 저장한 이 `Task` 하나가, 그 해부의 첫 표본이 된다.
