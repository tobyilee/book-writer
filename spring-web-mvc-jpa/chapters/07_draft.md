# 7장. 영속성 컨텍스트라는 출석부 — 더티 체킹과 지연 로딩의 마법

6장 끝에서 작은 약속을 하나 남겼다. 분명 `save()`를 부르지 않았는데 값이 바뀌어 저장되더라는 그 수상한 일 말이다. 직접 한번 겪어보자. 다음 코드를 보자.

```java
@Service
public class TaskService {

    private final TaskRepository taskRepository;

    public TaskService(TaskRepository taskRepository) {
        this.taskRepository = taskRepository;
    }

    @Transactional
    public void completeTask(Long id) {
        Task task = taskRepository.findById(id)
                .orElseThrow(() -> new TaskNotFoundException(id));
        task.markDone();   // done = true 로 바꾸기만 했다
        // save() 를 부르지 않았다!
    }
}
```

`completeTask`가 한 일을 천천히 따라가 보자. 데이터베이스에서 `Task` 하나를 꺼내(`findById`), 그 객체의 `done` 값을 `true`로 바꿨다(`markDone`). 그게 전부다. 어디에도 `taskRepository.save(task)`가 없다. 그런데 이 메서드를 호출하고 나서 데이터베이스를 들여다보면, 해당 행의 `done` 칼럼이 멀쩡히 `true`로 **바뀌어 저장돼 있다.** 콘솔에는 우리가 시킨 적 없는 `update task set ...` SQL이 떡하니 찍혀 있다.

처음 이걸 보면 정말 묘하다. 내가 시키지도 않은 `UPDATE`가 나갔으니, 좋게 보면 똑똑한 것이고 나쁘게 보면 통제를 벗어난 것이다. 좋은 감각이라면 여기서 살짝 불안해야 한다. **내가 시키지 않은 일을 프레임워크가 알아서 해주는 건, 그 원리를 모르는 한 언제 어떻게 터질지 모르는 시한폭탄과 같다.** 그러니 이 장에서 우리가 할 일은 분명하다. JPA가 보이지 않는 곳에서 부리는 그 "마법"의 정체를 똑똑히 들여다보는 것이다. 정체를 알고 나면, 마법은 더 이상 마법이 아니라 예측 가능한 동작이 된다.

## 출석부를 든 선생님 — 영속성 컨텍스트

마법의 한복판에는 **영속성 컨텍스트(persistence context)**라는 것이 있다. 이름이 거창하니 일단 비유부터 들자.

학창 시절 교실을 떠올려보자. 선생님은 수업이 시작되면 출석부를 펼친다. 출석부에는 그 시간에 들어온 학생들의 명단이 적혀 있고, 선생님은 누가 자리를 옮겼는지, 누가 졸고 있는지를 그 명단에 기록해둔다. 수업이 끝날 때, 선생님은 출석부에 적힌 변동 사항을 정리해 학적부에 한꺼번에 반영한다.

영속성 컨텍스트가 바로 이 **출석부**다. 트랜잭션이 시작되면 — 수업이 시작되면 — JPA는 출석부를 하나 펼친다. 우리가 `findById`로 데이터베이스에서 `Task`를 꺼내오면, JPA는 그 객체를 출석부에 적어 넣는다. "이 시간에 이 학생이 들어왔다"고 명단에 올리는 것이다. 그 뒤로 JPA는 출석부에 올라온 객체들을 **계속 지켜본다.** 누가 무엇이 바뀌었는지를. 그리고 트랜잭션이 끝날 때 — 수업이 끝날 때 — 변동된 내용을 데이터베이스에 한꺼번에 반영한다.

이 출석부는 또 하나 영리한 일을 한다. **1차 캐시** 역할이다. 한 트랜잭션 안에서 같은 `Task`를 두 번 조회한다고 해보자.

```java
Task t1 = taskRepository.findById(1L).orElseThrow(...);
Task t2 = taskRepository.findById(1L).orElseThrow(...);
// t1 == t2 ? 그렇다, 같은 객체다
```

두 번째 `findById`는 데이터베이스에 다시 묻지 않는다. 이미 출석부에 올라와 있는 그 객체를 그대로 돌려준다. 선생님이 이미 출석 부른 학생을 두고 또 부르지 않는 것과 같다. 그래서 `t1`과 `t2`는 단지 값이 같은 게 아니라, 메모리 상에서 **완전히 같은 객체**다. 한 트랜잭션 안에서 같은 식별자(id)를 가진 엔티티는 항상 같은 객체임을 보장받는다는 뜻인데, 객체 세계와 표 세계의 동일성을 JPA가 출석부로 이어주고 있는 셈이다.

> 프런트에서 React Query나 SWR을 써봤다면, "한 번 가져온 데이터를 키로 캐싱해두고 같은 키면 다시 안 부른다"는 감각이 익숙할 것이다. 영속성 컨텍스트의 1차 캐시도 결이 비슷하다. 다만 범위가 다르다. React Query의 캐시는 화면(컴포넌트 트리)이 사는 동안 유지되지만, 영속성 컨텍스트의 캐시는 **딱 한 트랜잭션 동안만** 살아 있다가 트랜잭션이 끝나면 통째로 사라진다. 이 "수명"의 차이가 뒤에서 중요해지니 기억해두자.

## save() 없이도 저장되는 이유 — 더티 체킹

이제 맨 앞의 수수께끼로 돌아가자. `save()`를 안 했는데 어떻게 `UPDATE`가 나갔을까?

출석부 비유를 이어가자. 선생님은 수업 시작 때 학생들의 상태를 출석부에 적어둔다. 철수는 1분단, 깨어 있음. 영희는 3분단, 졸고 있음. 이게 **스냅샷(snapshot)**, 즉 처음 모습의 사진이다. 수업이 끝날 무렵 선생님은 지금 교실 상태와 처음 적어둔 스냅샷을 **하나하나 비교한다.** 철수가 2분단으로 옮겨 앉았네? 그럼 학적부에 "철수 자리 변경"을 반영한다. 아무도 안 옮겼으면 아무것도 안 한다.

JPA의 **더티 체킹(dirty checking)**, 우리말로 **변경 감지**가 정확히 이것이다. JPA는 `findById`로 엔티티를 출석부에 올릴 때, 그 엔티티의 **처음 값을 따로 복사해 스냅샷으로 보관해둔다.** 우리가 `task.markDone()`으로 `done`을 `true`로 바꾸면, 그건 그냥 평범한 자바 객체의 필드를 바꾼 것일 뿐 — 이 순간엔 아무 SQL도 안 나간다. 그러다 트랜잭션이 끝나기 직전, JPA는 출석부에 올라온 모든 엔티티를 한 바퀴 돌며 **지금 값과 처음 스냅샷을 비교한다.** `done`이 `false`에서 `true`로 바뀌었네? 그럼 변경된 그 필드를 반영하는 `UPDATE` SQL을 만들어 데이터베이스에 보낸다.

그러니 `save()`를 부를 필요가 없었던 것이다. 더 정확히 말하면, **영속성 컨텍스트가 관리하는 엔티티는 값을 바꾸는 것만으로 변경이 추적되고, 트랜잭션이 커밋될 때 자동으로 반영된다.** 이 동작이 처음엔 "어, 편한데?" 싶다가도, 원리를 모르면 정반대 함정이 된다. 별생각 없이 조회한 엔티티의 필드를 잠깐 만졌을 뿐인데, 의도치 않은 `UPDATE`가 슬그머니 나가는 일이 생기는 것이다. 운영 데이터가 나도 모르게 바뀌어 있으면 그건 정말 아찔한 일이다. 그러니 이렇게 기억해두자. **조회한 엔티티는 "살아 있다." 건드리면 흔적이 남는다.**

> 그렇다면 `save()`는 언제 쓰나 하는 의문이 들 것이다. 좋은 질문이다. `save()`는 출석부에 **아직 없는** 새 엔티티를 처음 등록할 때(즉 `INSERT`) 필요하다. 6장에서 `create`가 새 `Task`를 만들어 `save()`로 저장했던 게 그 경우다. 반면 이미 출석부에 올라온(조회해온) 엔티티를 수정할 때는, 위에서 봤듯 굳이 `save()`를 부르지 않아도 더티 체킹이 알아서 반영한다. 물론 명시적으로 `save()`를 한 번 더 불러도 잘못된 건 아니다. 다만 "조회한 걸 고칠 땐 save가 필수"라고 착각하진 말자.

## 엔티티에도 생애가 있다 — transient, persistent, detached

여기까지 오면 한 가지가 또렷해진다. 같은 `Task` 객체라도, **출석부에 올라와 있느냐 아니냐에 따라 JPA가 대하는 태도가 전혀 다르다.** 그래서 JPA는 엔티티가 처한 상태를 몇 가지로 구분한다. 어려워 보이지만 출석부 비유로 풀면 쉽다.

먼저 **비영속(transient)** 상태다. `new Task("장보기")`로 방금 만든 객체를 떠올려보자. 이건 그냥 자바 객체일 뿐, 아직 출석부에 이름이 없다. JPA는 이 객체의 존재조차 모른다. 교실 문밖에 서 있는, 아직 출석 부르기 전의 학생이다.

다음은 **영속(persistent)** 상태다. 이 객체를 `save()`로 저장하거나 `findById`로 조회해 출석부에 올린 순간, 엔티티는 영속 상태가 된다. 이제 JPA가 지켜본다. 더티 체킹의 대상이 되고, 1차 캐시에 살고, 값을 바꾸면 흔적이 남는다. 출석 불린 학생이다. 지금까지 이야기한 "마법"은 전부 이 영속 상태에서 일어난 일이다.

마지막은 **준영속(detached)** 상태다. 트랜잭션이 끝나 출석부 자체가 닫히면, 그 안에 있던 엔티티들은 전부 출석부에서 떨어져 나와 준영속이 된다. 수업이 끝나 교실 밖으로 나간 학생이다. 한때 출석부에 있던 학생이 맞지만, 이제 선생님은 그를 지켜보지 않는다. **준영속 엔티티는 값을 아무리 바꿔도 더티 체킹이 일어나지 않는다.** 출석부 밖의 일이라 JPA가 모르기 때문이다.

```
new Task(...)          →  비영속(transient)   : 출석부에 없음
save() / findById()    →  영속(persistent)    : 출석부에 있음, 추적됨
트랜잭션 종료           →  준영속(detached)    : 출석부에서 떨어짐, 추적 안 됨
```

이 상태 구분이 왜 중요할까? **준영속 엔티티를 영속 엔티티로 착각하는 데서 수많은 버그가 시작되기 때문이다.** "분명 값을 바꿨는데 데이터베이스에 반영이 안 돼요"라는 흔한 하소연의 정체가 바로 이것이다. 트랜잭션 밖에서, 즉 출석부가 이미 닫힌 뒤에 엔티티 값을 바꿨으니 더티 체킹이 일어날 리가 없다. 거꾸로 앞서 본 "안 시켰는데 UPDATE가 나갔다"는 영속 상태에서 무심코 값을 건드린 경우다. 같은 객체라도 어느 상태에 있느냐가 모든 걸 가른다. 기억해두자.

## 객체가 객체를 가리키게 하기 — 연관관계 매핑

지금까지 `Task` 하나만 다뤘다. 그런데 현실의 데이터는 혼자 살지 않는다. 할 일에는 그 일을 맡은 사람이 있고, 사람은 여러 할 일을 가진다. TaskBoard에도 이제 그런 관계를 들여보자. `Task`에 **담당자(User)**를 붙이는 것이다.

6장에서 짚었듯, 표의 세계에서 다른 표를 가리키는 방법은 **외래 키(foreign key)** 하나뿐이다. `task` 테이블에 `user_id`라는 칼럼을 두고 거기에 사용자의 기본 키 값을 적어두는 식이다. 그런데 우리가 자바에서 다루고 싶은 건 숫자가 아니라 **객체 참조**다. `task.getUser().getName()`처럼 객체를 타고 들어가고 싶다. 이 간극 — 표는 외래 키, 객체는 참조 — 을 메워주는 게 **연관관계 매핑**이다.

먼저 간단한 `User` 엔티티를 하나 만들자.

```java
package com.example.taskboard;

import jakarta.persistence.Entity;
import jakarta.persistence.GeneratedValue;
import jakarta.persistence.GenerationType;
import jakarta.persistence.Id;

@Entity
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    protected User() {}

    public User(String name) {
        this.name = name;
    }

    public Long getId() { return id; }
    public String getName() { return name; }
}
```

이제 `Task`가 `User`를 가리키게 하자. 할 일 하나에는 담당자 한 명, 한 명의 담당자는 여러 할 일을 가질 수 있다. 그러니 `Task` 쪽에서 보면 **여러(Many) 할 일이 한(One) 사용자를** 향한다. 이걸 `@ManyToOne`이라고 적는다.

```java
import jakarta.persistence.ManyToOne;
import jakarta.persistence.FetchType;
import jakarta.persistence.JoinColumn;

@Entity
public class Task {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;
    private boolean done;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User user;

    // 생성자·getter 등은 6장과 동일, 지면상 생략
}
```

`@ManyToOne`은 "이쪽이 Many, 가리키는 저쪽이 One"이라는 선언이다. `@JoinColumn(name = "user_id")`는 "이 관계를 `task` 테이블의 `user_id`라는 외래 키 칼럼으로 표현하라"는 지시다. 이렇게 해두면 `task.getUser()`로 객체를 자연스럽게 타고 들어갈 수 있게 된다. 외래 키 숫자를 객체 참조로 바꿔주는 일을, JPA가 대신 해주는 것이다.

### 단방향, 양방향, 그리고 "연관관계의 주인"

여기까지는 `Task`만 `User`를 안다. `User`는 자기에게 어떤 할 일이 달려 있는지 모른다. 한쪽만 상대를 가리키니 이걸 **단방향(unidirectional)**이라 한다. 대부분의 경우엔 이 단방향만으로 충분하다.

그런데 가끔은 반대쪽에서도 타고 들어가고 싶다. `user.getTasks()`로 그 사람의 할 일 목록을 통째로 보고 싶은 것이다. 그러려면 `User`에도 `Task` 목록을 들고, 서로가 서로를 가리키게 해야 한다. 이걸 **양방향(bidirectional)**이라 한다.

```java
import jakarta.persistence.OneToMany;
import java.util.ArrayList;
import java.util.List;

@Entity
public class User {
    // ... 앞의 필드들 ...

    @OneToMany(mappedBy = "user")
    private List<Task> tasks = new ArrayList<>();
}
```

여기서 입문자를 가장 자주 헷갈리게 하는 개념이 등장한다. `@OneToMany`에 붙은 저 `mappedBy = "user"`다. 이게 무슨 뜻일까?

다시 표의 세계를 떠올려보자. `Task`와 `User`의 관계를 나타내는 외래 키는 어디 있었나? `task` 테이블의 `user_id` 칼럼, **딱 하나뿐이다.** 그런데 자바에서는 `Task.user`와 `User.tasks`, **두 군데서** 같은 관계를 들여다보고 있다. 표에는 외래 키가 하나인데 객체에는 참조가 둘이다. 여기서 JPA가 난감해한다. **"둘 중 누구의 변경을 보고 외래 키를 갱신해야 하지?"**

그래서 JPA는 둘 중 하나를 **연관관계의 주인(owner)**으로 정하게 한다. 주인은 외래 키를 직접 들고 있는 쪽, 즉 `@JoinColumn`이 붙은 `Task.user`다. 주인이 아닌 쪽(`User.tasks`)에는 `mappedBy`를 붙여 **"나는 그저 거울일 뿐이다. 외래 키의 진짜 주인은 저쪽(`user` 필드)이다"**라고 선언한다. 그러니 외래 키를 바꾸고 싶으면 반드시 **주인 쪽**, 즉 `task.setUser(someUser)`를 건드려야 한다. 거울인 `user.getTasks().add(task)`만 해서는 데이터베이스에 외래 키가 반영되지 않는다.

이 대목에서 많은 입문자가 정말 찜찜한 시간을 보낸다. "분명 `user.getTasks().add(...)`로 추가했는데 DB에 반영이 안 돼요!" 하고 한참을 헤맨다. 원인은 거의 항상 같다. 주인이 아닌 거울 쪽만 건드린 것이다. 그러니 양방향을 쓸 땐 이 한 줄을 새겨두자. **외래 키를 가진 쪽이 주인이고, 데이터베이스를 바꾸려면 주인을 건드려야 한다.**

물론 솔직히 덧붙이자면, 양방향은 편한 만큼 신경 쓸 게 늘어난다. 그래서 실무에서도 "정말 양쪽에서 다 타고 들어가야 하나?"를 먼저 자문하고, 꼭 필요할 때만 양방향으로 가는 편이 낫다. 한쪽 방향으로 충분하면 단방향이 단순하고 안전하다.

## 어떤 필드는 접근할 때 쿼리가 나간다 — 지연 로딩

이제 또 다른 "마법"을 들여다볼 차례다. 6장 끝에서 예고했던, "객체 하나를 꺼냈을 뿐인데 연결된 다른 객체까지 슬그머니 따라온다"는 그 일이다. 정확히는 따라올 수도, 안 따라올 수도 있는데 — 그걸 가르는 게 **로딩 전략**이다.

상황을 가정해보자. 우리가 `taskRepository.findById(1L)`로 `Task` 하나를 꺼냈다. 이 `Task`는 `User`를 가리키고 있다. 그렇다면 `Task`를 꺼낼 때, 그에 딸린 `User`까지 **지금 당장** 데이터베이스에서 함께 읽어와야 할까? 아니면 우리가 실제로 `task.getUser()`를 부르는 **그 순간까지 미뤄뒀다가** 그때 읽어와야 할까?

전자가 **즉시 로딩(EAGER)**, 후자가 **지연 로딩(LAZY)**이다. 말 그대로다. 즉시 로딩은 "엔티티를 꺼낼 때 연관된 것도 즉시 다 가져와라", 지연 로딩은 "연관된 건 진짜 쓸 때까지 게으르게(lazy) 미뤄둬라"다.

지연 로딩이 어떻게 동작하는지 코드로 따라가 보자.

```java
@Transactional
public void printUserName(Long taskId) {
    Task task = taskRepository.findById(taskId).orElseThrow(...);
    // 이 시점엔 select * from task ... 만 나갔다. user 는 안 읽었다.

    System.out.println(task.getUser().getName());
    // 바로 이 순간! getUser() 를 실제로 건드리니
    // 그제서야 select * from users where id = ? 가 나간다.
}
```

`findById`로 `Task`를 꺼낼 때는 `task` 테이블만 조회한다. `user`는 건드리지 않는다. 그러다 우리가 `task.getUser().getName()`으로 사용자 이름을 실제로 꺼내려는 **그 순간**, JPA가 비로소 `users` 테이블에 추가 쿼리를 날려 데이터를 채운다. 이게 지연 로딩이다.

그렇다면 어떻게 이런 일이 가능할까? `findById` 시점엔 `user`를 안 읽었다면서, `task.getUser()`에는 도대체 뭐가 들어 있던 걸까? 비밀은 **프록시(proxy)**다. JPA는 진짜 `User` 객체 대신, 겉모습만 `User`인 **가짜 대역 객체**를 슬쩍 넣어둔다. 이 대역은 속이 텅 비어 있다가, 누군가 `getName()` 같은 메서드를 부르는 순간 "아, 이제 진짜 데이터가 필요하구나" 하고 그제서야 데이터베이스에 쿼리를 날려 속을 채운다. 6장에서 봤던 리포지토리 프록시와 같은 발상이다. JPA는 이렇게 가짜 객체를 곳곳에 심어두고, 필요한 순간에만 진짜 일을 시킨다.

### LAZY를 기본으로, 그런데 함정이 하나

그렇다면 즉시 로딩과 지연 로딩 중 무엇을 골라야 할까? 정답에 가까운 권장이 있다. **연관관계는 기본적으로 지연 로딩(LAZY)으로 두는 편이 낫다.** 왜냐고?

즉시 로딩을 떠올려보자. `Task` 하나를 꺼낼 때마다 그에 딸린 `User`가 항상 함께 끌려온다. 당장 사용자 정보가 필요 없는데도 말이다. 할 일 100개를 목록으로 조회하는데, 쓰지도 않을 사용자 정보 100명분이 매번 따라온다면 그건 분명 낭비다. 게다가 즉시 로딩은 우리가 예상하기 어려운 쿼리를 뒤에서 뭉텅뭉텅 만들어내는 일이 많아, 다음 장에서 다룰 N+1 문제의 단골 원인이 되기도 한다. 그래서 `@ManyToOne`의 기본값이 즉시 로딩임에도 불구하고, 위 코드에서 우리는 일부러 `fetch = FetchType.LAZY`를 명시했다. (참고로 `@OneToMany`는 기본값이 이미 지연 로딩이다.)

그런데 지연 로딩에는 입문자를 거의 반드시 한 번은 울리는 함정이 도사리고 있다. 앞에서 영속성 컨텍스트는 **트랜잭션 동안에만** 살아 있다고 했던 걸 기억하는가? 지연 로딩은 그 컨텍스트가 살아 있을 때만 작동한다. 진짜 데이터가 필요한 순간 추가 쿼리를 날리려면, 쿼리를 날릴 무대(트랜잭션과 컨텍스트)가 아직 열려 있어야 하기 때문이다.

문제는 이렇게 터진다. 트랜잭션이 이미 끝나버린 뒤 — 예컨대 컨트롤러가 응답을 만들면서 — 아직 안 읽은 프록시의 `getUser()`를 건드리는 것이다. 무대가 이미 닫혔는데 거기서 쿼리를 날리려 하니, JPA는 손을 들고 `LazyInitializationException`이라는 예외를 던진다. 개발할 땐 멀쩡하던 코드가 어느 순간 이 예외로 빨갛게 터지면, 처음 겪는 사람은 정말 당황스럽다. "방금 전까지 잘 됐는데?" 싶은 것이다.

지금은 이 함정의 **존재**만 또렷이 알아두자. 정체는 이미 우리 손에 있다. 지연 로딩은 영속성 컨텍스트가 살아 있어야 작동하고, 컨텍스트는 트랜잭션과 운명을 같이한다. 그러니 **트랜잭션 밖에서 지연 로딩을 건드리면 터진다.** 이 함정이 운영에서 어떻게 모습을 드러내는지, 그리고 어떻게 길들이는지는 8장에서 N+1 문제와 함께 정면으로 다룬다. 여기서는 "왜 터지는지"의 원리까지만 손에 쥐고 가자. 원리를 알면 8장의 해법이 암기가 아니라 납득이 된다.

## 모든 마법의 무대 — @Transactional이라는 경계

지금까지 이야기한 모든 것 — 출석부, 더티 체킹, 1차 캐시, 지연 로딩 — 에는 한 가지 공통된 전제가 있었다. **트랜잭션이 열려 있어야 한다**는 것이다. 영속성 컨텍스트는 트랜잭션과 함께 태어나 트랜잭션과 함께 죽는다. 출석부는 수업 시간에만 펼쳐져 있다.

그 트랜잭션의 경계를 긋는 게 바로 `@Transactional` 어노테이션이다. 이 장 맨 앞 코드에서 `completeTask` 위에 슬쩍 붙어 있던 그 녀석이다. 메서드에 `@Transactional`을 붙이면, Spring은 그 메서드가 시작될 때 트랜잭션을 열고(출석부를 펼치고), 메서드가 정상적으로 끝나면 커밋한다(변동을 학적부에 반영하고 출석부를 닫는다). 도중에 예외가 터지면 롤백한다(변동을 없던 일로 돌린다).

```java
@Transactional
public void completeTask(Long id) {
    // ── 여기서 트랜잭션 시작, 출석부 펼침 ──
    Task task = taskRepository.findById(id).orElseThrow(...);
    task.markDone();
    // ── 메서드 끝, 커밋: 더티 체킹이 UPDATE 를 날리고 출석부 닫음 ──
}
```

이제 맨 앞의 수수께끼가 완전히 풀린다. `completeTask`에 `@Transactional`이 붙어 있었기에 그 안에서 출석부가 펼쳐졌고, `findById`로 꺼낸 `task`가 출석부에 올라 영속 상태가 됐고, `markDone()`으로 바꾼 값이 메서드가 끝나는 순간 더티 체킹으로 감지돼 자동 `UPDATE`로 반영된 것이다. `save()`가 없어도 저장된 이유가 여기에 있었다.

거꾸로, **`@Transactional`이 없으면 이 모든 마법이 작동하지 않는다.** 출석부가 펼쳐지지 않으니 더티 체킹도 없고, 지연 로딩을 건드리면 무대가 없어 터진다. 그래서 보통 데이터를 다루는 서비스 계층의 메서드에 `@Transactional`을 붙여 "여기부터 여기까지가 한 트랜잭션이다", 곧 "여기까지가 출석부가 펼쳐진 시간이다"라는 경계를 명확히 긋는다. 조회만 하는 메서드라면 `@Transactional(readOnly = true)`를 붙여 "읽기만 한다"고 알려주면, JPA가 더티 체킹용 스냅샷을 만들지 않아 살짝 가벼워진다.

`@Transactional`을 둘러싼 더 미묘한 함정들 — 예컨대 같은 클래스 안에서 메서드를 호출하면 트랜잭션이 안 먹는 문제 — 도 있지만, 그건 8장에서 함께 정리하자. 지금은 이 한 가지가 핵심이다. **`@Transactional`은 영속성 컨텍스트라는 무대의 막을 올리고 내리는 경계다.** 이 경계를 의식하는 순간, JPA의 모든 동작이 "언제 무대가 열려 있나"라는 하나의 질문으로 환원된다. 기억해두자.

## AI에게 연관관계를 맡길 때 — 방향과 주인을 되물어라

::: tip AI 페어코딩 학습 포인트
연관관계 매핑은 AI가 코드를 척척 뽑아주는 영역이지만, 동시에 **AI의 선택이 우리 의도와 가장 자주 어긋나는** 영역이기도 하다. 엔티티 매핑 한 줄의 차이가 수십 개의 예상 못 한 쿼리로 번지기 때문이다.
:::

AI에게 "Task와 User를 연관관계로 묶어줘"라고 부탁하면, 그럴듯한 매핑 코드가 순식간에 나온다. 그런데 그대로 받아들이기 전에, 세 가지는 반드시 우리 입으로 되물어보자.

첫째, **이게 단방향이야, 양방향이야? 그리고 왜?** AI는 묻지도 않았는데 양방향으로 만들어 놓는 경우가 많다. 하지만 앞서 봤듯 양방향은 연관관계의 주인을 신경 써야 하고, 관리할 지점이 늘어난다. "우리가 정말 양쪽에서 다 타고 들어가야 하나?"를 AI에게 되묻고, 필요 없으면 단방향으로 단순하게 가자. AI는 일반론으로 풍성한 쪽을 택하기 쉽지만, 단순함의 가치는 우리가 판단해야 한다.

둘째, **로딩 전략이 LAZY로 돼 있나?** AI가 `@ManyToOne`을 만들면서 `fetch` 설정을 생략하면, 그건 기본값인 즉시 로딩(EAGER)이 된다. 앞서 이야기했듯 이건 다음 장에서 다룰 N+1 문제의 씨앗이 되기 쉽다. 그러니 생성된 `@ManyToOne`에 `fetch = FetchType.LAZY`가 명시돼 있는지 눈으로 확인하자. 없으면 우리가 채워 넣으면 된다.

셋째, 무엇보다 **이 코드가 실제로 어떤 SQL을 만드는지 보여달라고 하자.** 6장에서 켜둔 `show-sql`이 여기서 진가를 발휘한다. "이 연관관계로 Task 목록을 조회하면 어떤 SQL이 나가?"라고 AI에게 물어 답을 받되, 그 답을 곧이곧대로 믿지 말고 **직접 실행해 콘솔에 찍히는 SQL과 대조하자.** AI의 설명과 실제 동작이 어긋나는 순간이 바로 우리가 배우는 순간이다. 잊지 말자. AI는 매핑의 스캐폴드를 만들고, 그 매핑이 어떤 SQL로 번역되는지 검증하는 건 끝까지 우리 몫이다.

## 마무리

이번 장에서 우리는 JPA의 "보이지 않는 일꾼"의 정체를 똑똑히 들여다봤다. 핵심은 **영속성 컨텍스트**라는 출석부 하나였다. 트랜잭션이 열리면 펼쳐지고 닫히면 사라지는 이 출석부가, 우리가 본 모든 마법의 무대였다.

출석부에 올라온 엔티티(영속 상태)는 처음 모습의 스냅샷과 끊임없이 비교당하고, 값이 바뀌면 트랜잭션이 끝날 때 **더티 체킹**으로 자동 반영된다 — 그래서 `save()` 없이도 저장됐다. 같은 엔티티는 1차 캐시 덕에 한 트랜잭션 안에서 늘 같은 객체로 보장된다. 엔티티는 출석부에 들고 나는 데 따라 **비영속·영속·준영속**이라는 생애를 갖는다. 객체가 객체를 가리키는 **연관관계 매핑**에서는, 외래 키를 가진 쪽이 주인이고 주인을 건드려야 데이터베이스가 바뀐다는 것을 새겼다. 그리고 연관된 엔티티를 진짜 쓸 때까지 미뤄두는 **지연 로딩**은 프록시라는 대역을 심어두고 동작하되, 출석부가 닫힌 트랜잭션 밖에서 건드리면 터진다는 것까지 봤다. 이 모든 것의 막을 올리고 내리는 경계가 `@Transactional`이었다.

여기까지 오면 JPA는 더 이상 마법이 아니다. "언제 출석부가 펼쳐져 있나, 이 엔티티는 출석부에 있나, 이 필드는 진짜 읽혔나"를 묻는 순간, 모든 동작이 예측 가능해진다.

그런데 솔직히 털어놓자면, 우리는 아직 마법의 **편안한 면**만 봤다. 개발할 땐 데이터가 적어 멀쩡하던 코드가, 운영에 올라가 데이터가 쌓이는 순간 갑자기 느려 터지는 일이 있다. 분명 똑같은 코드인데 말이다. 조회 한 번에 SQL이 수십, 수백 개씩 줄지어 나가는 그 악명 높은 **N+1 문제**, 그리고 방금 예고한 `LazyInitializationException`이 운영에서 모습을 드러내는 순간 말이다. 다음 장에서는 마침내 그 어두운 면을 정면으로 마주한다. `show-sql`로 길들인 그 눈으로 직접 SQL 로그를 들여다보며, 마법이 어디서 터지고 어떻게 길들이는지를 손으로 익혀보자. 오늘 만든 이 `Task`—`User` 연관관계가, 그 해부의 첫 표본이 된다.
