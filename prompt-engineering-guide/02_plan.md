# 최신 AI 모델을 위한 프롬프트 엔지니어링 저술 계획

> 기준 시점: 2026-06-01 / 장르: tech-book / 레퍼런스: `01_reference.md`
> 핵심 차별점: **세 모델(Claude Opus 4.8 · GPT-5.5 · Gemini 3.5 Flash) 공식 가이드 1차 근거 + 실행 가능한 프롬프트 eval(수동·자동)**

## 제목 후보

1. **프롬프트 엔지니어링 2026: 세 모델을 길들이는 법** — 슬러그 `prompt-engineering-2026`
   톤: 실전·현행성 강조, 약간의 위트("길들이는"). 포지셔닝: "지금 이 순간의 모델(Claude Opus 4.8·GPT-5.5·Gemini 3.5 Flash)에 정확히 맞춘 가이드". 버전 신선도가 셀링 포인트인 독자에게 강하게 꽂힌다. 단점은 모델명이 바뀌면 제목이 빨리 늙는다.

2. **좋은 프롬프트, 더 나은 프롬프트 — 작성부터 평가까지** — 슬러그 `better-prompts-eval`
   톤: 차분하고 방법론 중심. 포지셔닝: 이 책의 진짜 차별점인 **eval(평가)**을 제목 전면에 세운다. "프롬프트를 어떻게 잘 쓰지?"를 넘어 "내 프롬프트가 정말 더 나은지 어떻게 아는가?"라는 한 단계 깊은 질문을 던지는 독자에게 적합. 모델명에 묶이지 않아 수명이 길다.

3. **프롬프트의 기술: 일상부터 코딩 에이전트까지** — 슬러그 `the-art-of-prompting`
   톤: 폭넓고 친근한 입문서. 포지셔닝: 일반 사용자~개발자를 모두 품는 넓은 우산. 활용 시나리오(일상·학습·코딩)의 스펙트럼을 강조해 입문 독자에게 진입장벽이 낮다. 단점은 eval이라는 차별점이 제목에서 안 드러난다.

**추천:** **후보 2 — 좋은 프롬프트, 더 나은 프롬프트 (slug: `better-prompts-eval`)**
이유: 시중의 프롬프트 책 대부분은 "이렇게 쓰세요" 팁 모음에서 멈춘다. 이 책의 가장 큰 가치는 "더 나은 프롬프트를 **측정으로 가려내는**" eval 워크플로우다. 제목이 그 차별점을 정면으로 내세우면서도 모델명에 묶이지 않아 수명이 길다. 부제로 세 모델을 명시하면 현행성도 챙길 수 있다(예: 부제 "Claude Opus 4.8·GPT-5.5·Gemini 3.5 Flash 시대의 프롬프트 작성과 평가"). 빠른 출판·현행성 마케팅이 더 중요하면 후보 1로 전환 가능.

## 책 특성

- **장르:** tech-book (입문서~중급, 일부 개발자 심화 — 에세이형 기술서에 가까운 친근한 평어체)
- **분량:** 13장 + 머리말/맺음말 + 부록 A(모델 스냅샷 표). 본문 약 13장 × 평균 5,000자 ≈ 65,000자(한글) 내외. 코드/YAML/표 블록 다수 → 조판 시 약 200~240p. (eval·코딩 장은 코드 비중이 커 상한 6,000자, 일부는 분할 가능)
- **난이도:** 초급→중급. 1~9장은 코드 없이도 읽힘(일반 사용자), 10~12장 eval과 9장 코딩 에이전트는 Python/YAML 예시 포함(개발자 심화). 난이도 곡선이 완만히 상승.
- **독자 여정(할 수 있게 되는 것):**
  진입 상태 — "프롬프트를 감으로 쓰고, 결과가 들쭉날쭉해도 왜인지 모른다. 모델마다 뭐가 다른지도 모른다."
  → 출구 상태 — "성공 기준을 먼저 정의하고, 모델 특성에 맞춰 프롬프트를 구조화하며, 두 프롬프트 중 어느 것이 더 나은지 **수동·자동 eval로 측정해 선택**하고, 회귀 테스트로 그 품질을 지켜낸다. 일상·학습·코딩 어느 시나리오에서도 자기 워크플로우를 갖는다."

## 내러티브 아크

이 책은 **기초 → 모델별 → 활용 시나리오 → 평가 → 종합**의 다섯 마디로 흐른다.

먼저 **1~2장(기초)**에서 "왜 지금 프롬프트 엔지니어링인가"를 묻는다. 2026년의 핵심 변화 — 추론을 CoT 프롬프트로 강제하던 시대에서 `effort`/`thinking_level` 같은 **파라미터로 추론량을 제어하는 시대**로의 이동 — 를 먼저 깔고, 그 위에 좋은 프롬프트의 보편 골격(역할·맥락·지시·few-shot·구조 구분자)을 세운다. 이 두 장은 이후 모든 장의 공통 어휘가 된다.

**3~6장(모델별)**은 그 골격이 모델마다 어떻게 달라지는지를 다룬다. Claude(XML·effort·prefill 폐기), GPT-5.5(reasoning_effort·verbosity·surgical 지시), Gemini 3.5 Flash(temperature 1.0·few-shot 필수·멀티모달)를 각각 한 장씩, **공식 가이드를 1차 근거로** 파고든다. 모델 장을 먼저 따로 본 뒤 6장에서 나란히 비교하면, 독자는 "같은 작업도 모델마다 다르게 말해야 한다"는 감각을 얻는다. 비교 장을 모델 장 뒤에 둔 건, 각 모델을 충분히 안 상태라야 대조가 의미 있기 때문이다.

**7~9장(활용 시나리오)**은 배운 것을 현장에 적용한다. 일상(글쓰기·요약·이메일·번역)에서 시작해 학습(개념 설명·소크라테스식·연습문제 생성)을 거쳐 개발자 코딩 에이전트(Claude Code/Cursor/Codex·컨텍스트 5계층·MCP)로 난이도를 끌어올린다. 일반 독자는 7~8장에서 충분한 수확을 얻고, 개발자는 9장에서 심화한다.

**10~12장(평가 — 책의 심장)**은 "그래서 내 프롬프트가 정말 더 나은가?"라는 질문에 답한다. 세 장은 **하나의 러닝 예시("환불 정책 요약")를 공유한다** — 10장에서 그 작업의 프롬프트 A·B를 손으로 채점(rubric·A/B·체크리스트)하고, 11장에서 같은 작업을 자동화(pytest 결정론 채점·promptfoo·LLM-as-judge·회귀·CI)하며, 12장에서 **"작성→평가→비교→선택"** 워크플로우로 완주한다. 매 장 새 작업을 꺼내지 않고 한 작업을 끝까지 끌고 가므로 12장 클라이맥스 효과가 살고, 11장 코드 절벽 앞에서도 일반 독자가 "이미 본 작업의 자동화"로 받아들여 진입장벽이 낮아진다. 선택의 축은 품질만이 아니라 **비용·지연까지** 함께 저울질한다(12장 비교표). 12장이 사실상 이 책의 클라이맥스다.

**13장(종합)**은 안티패턴·흔한 실수·프롬프트 인젝션/안전을 짚고, 모든 장의 실을 하나로 묶어 독자가 자기만의 반복 가능한 프롬프트 작업 흐름을 갖도록 닫는다.

오프닝 기법은 13개 장에 분산 배정했다 — '상황 가정'은 4개 장(전체의 약 31%, 1/3 이하)으로 제한했고, 인접한 두 장은 같은 기법을 쓰지 않는다.

## 챕터 목록

### 1장. 감으로 쓰던 프롬프트, 이제는 설계한다
- **핵심 질문:** 똑같은 질문인데 왜 어떤 날은 멋진 답이, 어떤 날은 엉뚱한 답이 나올까? 2026년의 프롬프트 엔지니어링은 예전과 무엇이 달라졌나?
- **주요 내용:**
  - 프롬프트 엔지니어링의 정의와 핵심 사이클: 성공 기준 정의 → eval 설계 → 작성·개선 → 측정·반복 [Anthropic]
  - in-context learning이라는 학술적 뿌리(파라미터를 안 바꾸고 입력만으로 동작 변경) [Brown 2020]
  - 2026년의 시대 변화: "CoT로 추론 강제" → "파라미터(effort/reasoning_effort/thinking_level)로 추론량 제어" — 세 공식 가이드 공통 권고
  - 이 책의 지도: 기초→모델별→활용→평가, 그리고 "더 나은 프롬프트는 측정으로 가린다"는 약속
- **독자가 얻는 것:** 프롬프트를 우연이 아니라 설계의 대상으로 보게 되고, 책 전체의 여정과 핵심 사이클을 머리에 담는다.
- **오프닝 기법:** 수사적 질문 ("같은 질문에 왜 답이 매번 다를까?")
- **예상 분량:** 4,500자

### 2장. 좋은 프롬프트의 해부학
- **핵심 질문:** 모델이 헷갈리지 않는 프롬프트는 어떤 부품으로 이뤄지는가?
- **주요 내용:**
  - 보편 골격: 역할(role) → 맥락/목표(왜) → 지시(순서) → 입력 데이터 → 예시(few-shot) → 출력 형식/제약 → self-check
  - 구조 구분자: XML 태그 vs Markdown — 무엇을 분리하고 왜 분리하나
  - few-shot/multishot의 원리와 3~5개 권장 [Brown 2020], CoT의 원리 [Wei 2022]와 "2026엔 파라미터로 대체" 단서
  - "동료가 읽고 안 헷갈리면 모델도 안 헷갈린다"(Anthropic golden rule)
  - "하지 마라"보다 "하라" — 긍정 지시 원칙
- **독자가 얻는 것:** 어떤 모델에도 통하는 프롬프트 기본기. 이후 모델별 장의 공통 어휘.
- **오프닝 기법:** 정의 뒤집기 ("좋은 프롬프트는 '긴 프롬프트'가 아니다 — 오히려 군더더기를 덜어낸 것에 가깝다")
- **예상 분량:** 5,500자

### 3장. Claude Opus 4.8 — XML과 effort의 언어
- **핵심 질문:** Claude에게 일을 제대로 시키려면 무엇을 Claude의 방식대로 말해야 하나?
- **주요 내용:**
  - XML 태그가 1급 시민: `<instructions>`/`<context>`/`<example>`/`<document>` 구조화, few-shot은 `<example>` 3~5개
  - `effort` 파라미터(`low`~`max`): 코딩·에이전트 `xhigh`, "얕은 추론이면 프롬프트로 우회 말고 effort를 올려라" — 이전 어떤 Opus보다 중요
  - adaptive thinking(기본 OFF → `thinking:{type:"adaptive"}`)
  - 매우 문자적인 지시 해석 → 범위 명시. verbosity 자동조절. 긍정 예시로 톤 튜닝
  - **prefill 폐기(4.6+, 400 에러)** → Structured Outputs/직접 지시/tool calling으로 마이그레이션
  - **[I2] 휘발성 수치는 본문에 박지 않는다** — 가격·context 창·effort 기본값 등은 부록 "모델 스냅샷 표"로 가리킨다(`{스냅샷 표 참조}`). 본문 산문은 수명 긴 원칙(XML 1급 시민, effort로 추론을 켠다, prefill 폐기 같은 *철학·구조*)만 담는다.
- **독자가 얻는 것:** Claude 전용 프롬프트 작성·마이그레이션 능력.
- **오프닝 기법:** 상황 가정 ("같은 지시를 Claude에 던졌는데 의외로 '시킨 것만 딱' 했다고 해보자")
- **예상 분량:** 5,800자

### 4장. GPT-5.5 — 외과수술 같은 지시
- **핵심 질문:** GPT-5.5의 추론 깊이와 답변 길이를 어떻게 따로따로 조절하나? 왜 모순된 지시가 가장 해로운가?
- **주요 내용:**
  - `reasoning_effort`(`none`~`xhigh`)로 사고 깊이+도구 호출 의향 제어 (GPT-5.5 기본 `medium`, 5.2 기본 `none` 차이 명기)
  - `verbosity`를 추론 길이와 **독립**으로 제어 + 자연어 오버라이드("글로벌 low + 코딩 툴 high")
  - agentic eagerness 제어: 줄이려면 effort↓+early-stop, 늘리려면 effort↑+persistence
  - "surgical precision" — 모순·모호 프롬프트가 가장 해롭다, instruction hierarchy, prompt optimizer
  - tool preambles, Markdown 기본 미사용(필요시 재명시), Responses API + `previous_response_id`
  - (GPT-5/5.2 cookbook에서 외삽한 부분 명시 — 레퍼런스 §7 한계 반영, "사실 확인 필요" 후보)
  - **[I2]** context 창·max output·knowledge cutoff·reasoning_effort 기본값 등 휘발성 수치는 부록 스냅샷 표로 격리. 본문은 "reasoning_effort와 verbosity를 독립으로 돌린다" 같은 원칙만.
- **독자가 얻는 것:** GPT-5.5의 두 레버(reasoning_effort·verbosity)를 분리해 다루는 감각.
- **오프닝 기법:** 실패 장면(in medias res) ("프롬프트에 '간결하게'와 '아주 자세히'를 둘 다 써놨더니, 모델이 둘 사이에서 길을 잃는다")
- **예상 분량:** 5,800자

### 5장. Gemini 3.5 Flash — 온도를 건드리지 마라
- **핵심 질문:** 왜 Gemini만 유독 temperature를 1.0으로 두라고 경고할까? 멀티모달·롱컨텍스트의 강점을 어떻게 끌어내나?
- **주요 내용:**
  - `thinking_level`(`minimal`~`high`, 기본 high) — CoT 강제 대신 high+단순 프롬프트
  - **temperature 1.0 고정 강력 권장** — 낮추면 looping·성능 저하(세 모델 중 유일한 명시적 경고)
  - few-shot을 **항상 포함**("없으면 덜 효과적") — 세 가이드 중 가장 강하게 밀어붙임
  - `media_resolution`로 멀티모달 토큰 제어(이미지 high/PDF medium/비디오 low~medium), 멀티모달·long context 강점
  - 기본 간결·직접적 → conversational 원하면 명시 steer. 구조화는 XML 또는 MD 중 한 형식 일관
  - (Gemini 3 Flash와 3.5 Flash 혼동 금지 경고 — 레퍼런스 버전 노트 반영)
  - **[I2]** thinking_level 기본값·media_resolution 토큰 수치(이미지 1120/PDF 560 등)·출력 속도 같은 휘발성 수치는 부록 스냅샷 표로 격리. 본문은 "temperature 1.0을 건드리지 마라, few-shot을 항상 넣어라" 같은 수명 긴 원칙만.
- **독자가 얻는 것:** Gemini 특유의 함정(낮은 temperature)과 강점(멀티모달·few-shot)을 다루는 법.
- **오프닝 기법:** 충격적 수치·사실 ("temperature를 0.2로 낮췄더니 모델이 같은 문장을 반복하며 맴돈다 — Gemini가 유일하게 '건드리지 말라'고 경고하는 손잡이다")
- **예상 분량:** 5,500자

### 6장. 세 모델을 한자리에 — 비교와 멀티모델 전략
- **핵심 질문:** 같은 작업을 세 모델에 시킬 때 무엇을 어떻게 다르게 말해야 하나? 어떤 일에 어떤 모델을 고를까?
- **주요 내용:**
  - 축별 비교표 해설: 추론 제어 / 구조 선호 / few-shot / temperature / verbosity / prefill / 강점
  - 같은 프롬프트의 세 모델 변환 실습(하나의 작업 → 3가지 모델 맞춤 프롬프트)
  - 모델 선택 가이드: 장기 에이전트·코딩(Claude)·instruction following(GPT-5.5)·멀티모달·속도·롱컨텍스트(Gemini)
  - **[C1] 비용·지연 관점 한 절(신설):** effort/reasoning_effort/thinking_level을 올리면 추론 토큰이 늘어 비용·지연이 함께 오른다 — 모델·파라미터 선택은 "최고 품질"이 아니라 "이 작업에 맞는 추론량"을 고르는 일이다. 같은 작업이라도 빠른·저렴 모드(Gemini의 속도 우위, Claude fast mode)와 고품질 모드 사이에서 tradeoff가 갈린다. (구체 수치는 부록 스냅샷 표 참조 — 본문은 원칙만) 이 관점은 10~12장 eval에서 "선택의 한 축"으로 다시 만난다(전방 콜백).
  - 멀티모델 제품의 함정: 출력 길이·verbosity 철학 차이 → 모델별 튜닝 필수
  - "기법의 원리는 이해하되 최신 모델에선 파라미터 우선"이라는 통합 관점
- **독자가 얻는 것:** 모델 간 이식과 선택의 의사결정 프레임.
- **오프닝 기법:** 비교 대조형 진입 — 선택지 제시 ("같은 요약 작업, 세 모델에 똑같이 던지면 결과가 셋 다 다르게 나온다 — 무엇이 갈랐을까?")
- **예상 분량:** 5,500자

### 7장. 일상 프롬프트 — 글쓰기·요약·이메일·번역·기획
- **핵심 질문:** 매일 쓰는 일에서 "괜찮은 답"을 "정확히 원하는 답"으로 끌어올리려면?
- **주요 내용:**
  - 역할 + 구체 맥락 + 출력 형식 3종 세트(여행 플래너 표 예시 등)
  - 글쓰기/요약/이메일 톤 조정/번역 뉘앙스/기획 브레인스토밍 각각의 프롬프트 패턴
  - 모델 메모: Gemini 간결 기본(수다 원하면 명시), Claude 길이 자동조절, GPT-5.5 `verbosity`
  - 흔한 실패와 교정: 막연한 요청 → 구체 제약으로
- **독자가 얻는 것:** 비개발자도 바로 쓰는 일상 프롬프트 레퍼토리.
- **오프닝 기법:** 인용·일화 ("'AI한테 이메일 써달라고 하면 꼭 번역기 돌린 티가 나요' — 한 동료의 하소연에서 시작해보자")

### 8장. 학습 프롬프트 — 가르치게 만들기

### 9장. 코딩 에이전트 프롬프트 — Claude Code·Cursor·Codex
- **핵심 질문:** 코드를 한 줄씩 받아쓰던 시대를 넘어, 에이전트에게 일을 "맡기려면" 무엇을 어떻게 줘야 하나?
- **주요 내용:**
  - 설정 5계층: CLAUDE.md(항상 켜진 고밀도 컨텍스트)·SKILL.md(온디맨드 플레이북)·MCP·.cursorrules·AGENTS.md [agensi 2026, 커뮤니티 — 보조 표시]

  - (커뮤니티 출처는 보조로 표시 — 공식 가이드와 일치하는 부분만 채택)
- **독자가 얻는 것:** 코딩 에이전트를 위한 컨텍스트 설계·워크플로우(개발자 심화).
- **오프닝 기법:** 정의 뒤집기 ("코딩 에이전트 프롬프트는 '코드를 더 잘 받는 법'이 아니다 — '코드를 안 받고 결과를 받는 법'에 가깝다")

### 10장. 더 나은가, 어떻게 아는가 — 수동 평가
- **핵심 질문:** "이 프롬프트가 더 낫다"는 느낌을 어떻게 점수로 바꾸나?
- **[C2] 러닝 예시 도입 — "환불 정책 요약":** 10·11·12장을 관통하는 **단일 작업**을 이 장에서 처음 세운다. "고객용 환불 정책을 정중하게 요약하되 환불 불가 조건을 빠뜨리지 않는다"는 작업에 프롬프트 A·B 두 후보를 만든다. (레퍼런스 §4.2 promptfoo YAML의 "30일"·llm-rubric 소재와 동일 — 11장에서 그대로 자동화된다.) 이 장에서는 이 두 후보를 **손으로** 채점한다.
- **주요 내용:**
  - rubric 채점: 기준별 1~5점(정확성·완결성·형식·톤·안전), Anthropic Console 5점 척도 — 환불 요약 A·B에 실제 적용
  - A/B 비교와 **위치 편향** → 순서 무작위화 [Zheng 2023]
  - 체크리스트 평가(형식·금칙어·길이·사실성: "30일" 명시했는가, 환불 불가 조건 누락 없는가)
  - Console 워크플로우: 성공기준 → `{{변수}}` 프롬프트 → 테스트케이스 → side-by-side 채점 → 버전 반복
  - 성공 기준을 먼저 적는 습관(1장 사이클 콜백)
- **독자가 얻는 것:** 코드 없이도 프롬프트 품질을 체계적으로 채점하는 법. 이후 11·12장에서 자동화될 바로 그 작업을 손으로 먼저 경험한다.
- **오프닝 기법:** 수사적 질문 ("'이게 더 나은 것 같아'라는 느낌, 그 느낌을 어떻게 남에게 증명할까?")
- **예상 분량:** 5,000자

### 11장. 프롬프트도 코드처럼 테스트한다 — 자동 평가
- **핵심 질문:** 프롬프트를 바꿀 때마다 품질이 회귀하지 않는지 자동으로 지키려면?
- **[C2] 러닝 예시 이어받기:** 10장에서 손으로 채점한 **그 "환불 정책 요약" 작업**을 이 장에서 자동화한다. 새 작업을 꺼내지 않는다 — 독자는 "10장에서 봤던 그 A·B를 이제 코드로 채점"하는 연속 흐름을 탄다. 11장의 모든 코드/YAML 예시는 환불 요약 한 작업으로 통일한다.
- **[I1] 이중 트랙 구조 장치(책 최대 코드 절벽 완충):** 장을 두 트랙으로 명시 분리한다. ① **개념 트랙(먼저, 코드 없이)** — 결정론 채점·LLM-as-judge·회귀의 *개념 지도*만으로 "자동 eval이 무엇을 왜 하는가"를 일반 독자가 이해하게 한다. ② **코드 트랙(뒤, 개발자용)** — pytest/promptfoo/CI 워크스루. 코드 박스에 "개발자 트랙" 표식을 달아, 비개발자가 코드 블록을 건너뛰어도 개념은 온전히 잡히도록 시각적으로 설계한다. 도입부에 "코드가 부담스러우면 개념 트랙만 읽고 12장으로 넘어가도 좋다"는 안내를 둔다.
- **주요 내용:**
  - (개념 트랙) 결정론 채점이란 무엇인가 / judge에게 채점을 맡긴다는 발상 / 회귀 테스트가 지키는 것 — 코드 없이
  - (코드 트랙) 결정론적 채점: 정규식·문자열 포함("30일")·JSON schema·정답 일치 — pytest 회귀 테스트 예시
  - (코드 트랙) promptfoo: 환불 요약 프롬프트 A/B + `llm-rubric` + 결정론 assertion YAML, `npx promptfoo eval && view`
  - LLM-as-judge: 단일 강한 judge 고정, 편향 가드레일(position/verbosity/self-enhancement → 순서 무작위·길이 정규화·judge≠피평가 모델), 인간 라벨 보정 [Zheng 2023]
  - 테스트셋 + 회귀 + CI 통합(OpenAI Evals model-graded 2단계)
  - "결정론 먼저, 통과분만 LLM-rubric" 비용 절약 순서
- **독자가 얻는 것:** (개발자) 자동 eval 파이프라인을 직접 돌리는 능력. (일반 독자) 코드를 건너뛰어도 자동 eval의 원리·언제 쓰는지를 잃지 않는다.
- **오프닝 기법:** 충격적 수치·사실 ("GPT-4 judge가 사람 평가와 80% 이상 일치했다 — 그렇다면 채점을 기계에 맡겨도 될까?")
- **예상 분량:** 6,000자

### 12장. 작성→평가→비교→선택, 한 바퀴 돌려보기
- **핵심 질문:** 지금까지 배운 작성과 평가를 하나의 워크플로우로 어떻게 꿰는가?
- **[C2] 러닝 예시 완주:** 10·11장에서 따라온 **그 "환불 정책 요약" 작업**을 여기서 작성→평가→비교→선택으로 끝까지 돌린다. 독자는 처음 보는 작업이 아니라 "이미 손으로 채점하고(10장) 자동화한(11장) 그 A·B"를 이제 의사결정까지 완주하므로 클라이맥스 효과가 극대화된다.
- **주요 내용:**
  - 통합 사례: "환불 정책 요약"에 프롬프트 A vs B (10·11장과 동일 작업)
  - 1단계 결정론 assertion으로 탈락 거르기 → 2단계 rubric+LLM-rubric로 품질 비교 → 3단계 **품질·비용·지연 종합**으로 채택
  - **[C1] "품질 vs 비용 vs 지연" 비교표 예시(신설):** 환불 요약 A·B(필요시 모델·effort 변형까지)를 행으로, **품질 점수 / 1k건당 비용 / p50 지연 / 채택 여부**를 열로 나란히 둔 표를 제시. "최고 품질이 아니라 최적 지점을 고른다"를 실증한다 — 예컨대 품질이 5% 높지만 비용·지연이 3배인 후보는 탈락. effort/verbosity가 비용·지연을 어떻게 밀어올리는지(3~5장 콜백)를 표에서 눈으로 확인. (레퍼런스 §4.3 "점수+비용+지연 종합" 근거)
  - **채택안을 회귀 테스트셋에 고정 = 프롬프트 버전관리·재현성** (모델 교체 시 같은 셋으로 재평가)
  - 수동·자동을 언제 섞을지, 멀티모델(3·4·5장 모델별 차이)을 eval에 반영
  - 한 장에서 책 전체(작성·모델·평가)를 관통하는 클라이맥스
- **독자가 얻는 것:** 반복 가능한 자기만의 프롬프트 개선 루프 — 책의 핵심 산출물. "더 나은"을 품질만이 아니라 비용·지연까지 함께 저울질해 고르는 감각.
- **오프닝 기법:** 상황 가정 ("환불 정책을 요약하는 프롬프트 두 개를 손에 들고 있다고 해보자 — 둘 다 그럴듯한데, 무엇을 배포할까?")
- **예상 분량:** 6,000자

### 13장. 함정을 피하고, 안전하게 — 안티패턴과 마무리
- **핵심 질문:** 흔한 실수와 위험(프롬프트 인젝션 등)을 어떻게 피하고, 이 책을 덮은 뒤 무엇을 습관으로 가져갈까?
- **주요 내용:**
  - 안티패턴·흔한 실수: 모순된 지시, "하지 마라" 남발, 출처 없는 수치, CoT 과잉(파라미터로 충분한데), 모델 무관 복붙
  - 프롬프트 인젝션·안전 간단 소개(신뢰 경계, 사용자 입력 격리, 시스템 지시 보호) — 입문 수준
  - 커뮤니티 "베스트 프롬프트" 라이브러리 품질 편차 주의 → 공식 우선·검증 습관
  - 버전 신선도 규율: 가격·context·파라미터·기본값은 출판 직전 1차 소스 재확인
  - 종합 회고: 성공 기준 → 모델 맞춤 작성 → 측정으로 선택 → 회귀로 유지라는 한 바퀴, 독자가 가져갈 관점
- **독자가 얻는 것:** 위험 회피 감각 + 평생 가는 프롬프트 작업 원칙.
- **오프닝 기법:** 인용·일화 ("'프롬프트는 코드처럼 다뤄라' — 책 전체를 한 줄로 줄이면 이 말이 남는다. 마지막으로, 그 코드가 깨지는 흔한 방식부터 짚어보자")
- **예상 분량:** 5,000자

## 부록 A. 모델 스냅샷 표 (2026-06-01 기준) — [I2] 휘발성 수치 물리적 격리
