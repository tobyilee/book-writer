## 3. 내러티브 아크

이 책은 "회의론에서 출발해 공학으로 착지한다." 1장은 "프롬프트 엔지니어링은 곧 사라진다"는 회의론을 정면으로 받아 어디까지 맞고 어디부터 틀렸는지 판을 깐다. 2장은 조건화로서의 프롬프트와 세 레버(명확성·맥락·형식)를 심어 이후 모든 기법의 뿌리를 만든다.

3장은 **단일 호출에서 끝나는 기법**을 체계로 정리한다. Zero-shot, Few-shot, CoT, Self-Consistency, Role prompting, PoT/PAL — 한 번의 API 호출 안에서 완결되는 모든 것. 여기서 Least-to-Most, Self-Refine, Reflexion, CoVe는 의도적으로 빼두었다. 이들은 본질적으로 **다단계 호출**이므로 8장에서 다룬다.

4장(Context Engineering)은 "프롬프트는 문자열이 아니라 아키텍처"라는 관점으로 독자를 끌어올린다. 5장 RAG·환각 완화는 컨텍스트로 진실을 말하게 하는 법, 6장 구조화 출력·도구 사용은 컨텍스트를 기계가 읽는 형태로 주고받는 법이다. 여기까지 읽으면 독자는 시스템 단위 프롬프트를 설계할 수 있다.

7장(모델 비교)은 **이제야** 등장한다. 독자가 Context Engineering·RAG·구조화 출력을 체험한 뒤에야 "같은 프롬프트가 Claude와 GPT에서 왜 다르게 동작하는가"를 실감한다. 7장은 4–6장의 실전 산출물을 공급사별로 번역하는 관점의 장이다.

8–9장은 "프롬프트를 넘어 파이프라인과 루프로"의 도약이다. 8장 **프롬프트 자동 최적화**는 수동 설계에서 데이터 기반으로 — DSPy/OPRO/TextGrad를 중심 소재로 삼고, 체이닝·라우팅·map-reduce, 런타임 self-improving(Self-Refine·Reflexion·CoVe) vs 빌드타임 optimizer(APE·DSPy·TextGrad)의 구분을 명확히 한다. 9장 에이전틱 프롬프팅은 ReAct부터 MCP까지, 책에서 가장 현대적인 지대를 다룬다.

10–12장은 공학적 엄밀성의 요구다. 10장 Evals는 Hamel Husain의 "Evals가 프롬프트 엔지니어링의 본체" 선언을 받아 골든셋 구축 워크스루를 중심에 둔다. 11장 보안은 prompt injection에 집중하고, 12장 운영은 캐싱·버저닝·관측성·배포 위생으로 현장에 착지한다.

13장은 1장의 회의론을 다시 연다. 12개 챕터를 지나온 독자의 답은 이제 1장과 다르다. 월요일 아침 무엇을 다르게 할지 구체적 다음 걸음으로 마무리.

**챕터 간 callback 포인트:**
- 1장 회의론 → 13장 수미상관
- 2장 세 레버(명확성·맥락·형식) → 4·5·6장에서 고도화
- 3장 CoT → 7장(모델별 CoT 금지), 8장(루프로의 발전), 10장(효용 측정), 13장(thinking 모델 이후의 운명)
- 4장 Context Engineering → 5·6·8·9장에서 각 하위 꼭지로 확장
- 8장 DSPy → 10장 Evals가 전제, 12장 운영에서 파이프라인 배포로 완성
- 11장 실패 사례(Air Canada, Chevrolet) → 12장 운영 위생

## 4. 챕터 목록

### 1장. 왜 지금, 프롬프트 엔지니어링인가

- **핵심 질문:** "모델이 곧 다 알아들을 텐데, 지금 이걸 배워야 할까?"
- **주요 내용:**
  - 회의론의 두 축: Sam Altman "5년 뒤 없어진다" + r/ML "folk wisdom" 비판
  - 반론: Wei 2022 CoT (17.9% → 56.9%), DSPy 5–40%p — "같은 모델, 다른 프롬프트"의 정량 격차
  - Ethan Mollick의 역설: 모델이 강해질수록 프롬프트 레버리지는 커진다
  - 협의 vs 광의: "Context Engineering"으로의 이름 이동 (Karpathy, Willison, HN 2025)
  - 이 책의 포지션: 팁은 휘발되지만 원칙·평가·운영은 남는다
- **독자가 얻는 것:** 이 책을 끝까지 읽을 명분. 동료가 "그거 의미 있냐"고 물을 때 답할 3문장.
- **예상 분량:** 13,000자 (약 17p)
- **참조 섹션:** §1.1, §1.2, §1.3, §7.1, §10
- **오프닝 hook (1인칭 고백체 — 수미상관용 유지):** "2024년 말, 한 친구가 이런 말을 했어요. '프롬프트 엔지니어링? 그거 1년 뒤엔 아무도 안 할 일 아니야?' 저는 대답을 못 했습니다. 이 책은 그날의 대답을 늦게나마 하는 책입니다."

### 2장. 프롬프트는 왜 작동하는가 — 첫 번째 원리들

- **핵심 질문:** "모델은 내 지시를 어떻게 '이해'하는가, 그리고 나는 그 과정에 어떻게 개입하는가?"
- **주요 내용:**
  - 프롬프트의 공학적 정의 (Liu 2021): p(y|x) 재구성 템플릿으로서의 조건화
  - 세 레버: 명확성(Clarity), 맥락(Context), 형식(Format)
  - 측정 가능성 원칙 — "간결하게" → "3문장 이하"
  - 긍정문 선호, 실패 경로 명시("If not in doc, say 모름")의 근거
  - Lost-in-the-Middle 편향 첫 소개 (맨앞·맨뒤 강조)
- **독자가 얻는 것:** 나머지 12개 챕터의 뿌리가 되는 세 레버. 프롬프트를 볼 때 "어느 레버가 흔들리는가"로 진단하는 눈.
- **예상 분량:** 15,000자 (약 20p)
- **참조 섹션:** §1.1, §2.1–2.6, §2.4
- **오프닝 hook (수사적 질문 진입 — S1 다양화):** "같은 모델에게 똑같이 '요약해 줘'라고 했을 때, 왜 어떤 날은 세 문장이 오고 어떤 날은 열 문장이 올까요? 이 질문에 답할 수 있다면 프롬프트 엔지니어링의 절반은 끝난 겁니다."

### 3장. 단일 호출의 기법들 — Zero-shot, Few-shot, CoT, Role

- **핵심 질문:** "한 번의 API 호출 안에서, 프롬프트가 낼 수 있는 최대치는 어디까지인가?"
- **주요 내용:**
  - Zero-shot / Instruction-following: FLAN, InstructGPT 계보
  - Few-shot / ICL: GPT-3 원 논문 + 실무 주의 (예시 순서·편향·동적 선택)
  - Chain-of-Thought: Wei 2022, Kojima "Let's think step by step", Auto-CoT
  - Self-Consistency (Wang 2022): 다수결 샘플링
  - PoT / PAL: 코드 생성 후 실행
  - Role prompting의 혼재된 근거 (Zheng 2023 "Is 'A Helpful Assistant' the Best Role?")
  - **2025년의 반전** (S2 재배치): reasoning 모델(o1/o3, Claude thinking, Gemini Thinking)에서 CoT 명시 금지 권고, 최신 모델에서 zero-shot이 few-shot에 경쟁적인 현상
  - Turpin 2023의 "CoT가 진짜 추론인가" 논쟁을 각주급 소개
  - **이 장에서 다루지 않는 것:** Least-to-Most, Self-Refine, Reflexion, Chain-of-Verification (다단계 호출이므로 8장)
- **독자가 얻는 것:** "단일 호출 기법 결정 트리" — 과제 앞에서 어느 기법이 먼저 손에 잡혀야 하는지.
- **예상 분량:** 17,000자 (약 22p)
- **참조 섹션:** §3.1, §3.2, §3.3, §3.4, §3.8, §7.3, §7.4, §7.5, §2.2
- **오프닝 hook (숫자·사실 진입 — S1 다양화):** "GSM8K 17.9% → 56.9%. 이 한 줄이 프롬프트 엔지니어링을 학문으로 만들었습니다. Wei 2022의 이 격차가 없었다면 이 책도 없었을 거예요."

### 4장. Context Engineering — 프롬프트는 문자열이 아니다

- **핵심 질문:** "왜 2025년에 '프롬프트 엔지니어링'이 '컨텍스트 엔지니어링'으로 개명되고 있는가?"
- **주요 내용:**
  - Karpathy 2025 선언, Willison·Husain·LangChain 계보
  - 컨텍스트의 여섯 구성요소: system prompt + RAG 결과 + 메모리(장·단기) + 툴 정의 + 이전 turn + 구조화 사용자 입력
  - 신호 대 잡음비 원칙 — 1M 토큰이어도 다 넣지 말 것 (RULER)
  - 프로덕션용 시스템 프롬프트 템플릿 뼈대 (§6.1)
  - 프롬프트 캐싱 관점: system을 앞에, 변하는 입력을 뒤에
  - **5·6·8·9장의 지도**: RAG(5), 구조화·툴(6), 자동 최적화·체이닝(8), 에이전트(9)가 모두 이 장의 하위 꼭지임을 밝힘
  - Long Context Engineering: Lost in the Middle / RULER / Needle-in-a-Haystack — C2 재구성으로 5장의 긴 문맥 절이 이곳으로 흡수됨
- **독자가 얻는 것:** "프롬프트 한 덩어리"에서 "컨텍스트 아키텍처"로의 관점 이동. 책 후반부의 지도.
- **예상 분량:** 15,000자 (약 19p)
- **참조 섹션:** §5.10, §6.1, §6.2, §6.6, §5.7, §2.4, §10-2
- **오프닝 hook (인용 진입 — S1 다양화):** "'Prompt engineering is the new term for what we used to call software engineering, when the compiler is an LLM.' — Andrej Karpathy, 2025. 이름만 바뀐 게 아닙니다. 관점이 바뀐 거예요."

### 5장. RAG와 환각 — 컨텍스트로 진실을 말하게 하기

- **핵심 질문:** "모델이 모르는 것을 답하지 않게 하고, 아는 것은 정확히 말하게 하려면?"
- **주요 내용 (C2 재구성: 환각 완화를 주 꼭지로):**
  - RAG의 원 아이디어 (Lewis 2020) 다시 읽기
  - 실무 패턴: Query rewriting / HyDE, re-ranking, cite-while-answer, context compression
  - Anthropic Contextual Retrieval (2024)
  - **환각 완화 레시피 3–5개 (신설 독립 절):**
    - (a) 실패 경로 지시 — "모르면 '모름'" (§2.6)
    - (b) Chain-of-Verification (Dhuliawala 2023) — 스스로 검증 질문 생성·답변
    - (c) Self-RAG 패턴 — 검색 필요성 자체를 모델이 판단
    - (d) Semantic entropy (Farquhar 2024, Nature) — 환각 탐지의 측정법
    - (e) FActScore / TruthfulQA / HaluEval — 환각 평가 벤치
  - 인용 강제 프롬프트 레시피와 "citation hallucination"(Perplexity 사례)의 한계
  - 한국어 도메인(법률·금융)의 용어집 주입 패턴
  - RAG vs Fine-tuning (§7.6): 지식은 RAG, 스타일은 fine-tuning, 행동은 system prompt
- **독자가 얻는 것:** "환각을 어떻게 줄이나"의 단일 주소. 인용 기반 답변 시스템의 최소 레시피.
- **예상 분량:** 17,000자 (약 22p)
- **참조 섹션:** §3.11, §5.2, §3.9 (CoVe), §2.6, §7.6, §8.3, §9
- **오프닝 hook (역사적 일화/뉴스 진입 — S1 다양화):** "2024년 2월, 몬트리올 법원은 Air Canada에게 고객에게 환불을 지급하라고 명령했다. 이유: 챗봇이 '환불 가능'이라 답했기 때문이다. 사실이 아니었다. 챗봇이 지어낸 말이었다. 그날 이후 환각은 법적 책임의 문제가 됐다."

### 6장. 구조화 출력과 도구 사용 — 말고 받는 법

- **핵심 질문:** "자유로운 텍스트 대신 안정적인 JSON을 받고, 모델이 실제로 외부 세계와 상호작용하게 하려면?"
- **주요 내용:**
  - 세 공급사 API: OpenAI Structured Outputs(strict), Anthropic tool_use/prefill, Gemini responseSchema
  - 스키마 설계 원칙: 풍부한 description, oneOf·재귀 회피, 의미 있는 필드명
  - 라이브러리 생태계: Instructor, BAML, Outlines, LangChain typed output

### 8장. 프롬프트 자동 최적화 — 수동 설계에서 데이터 기반으로

> **C1 반영:** 이전 "체이닝과 워크플로우" 장을 "프롬프트 자동 최적화"로 재명명하고, DSPy/OPRO/TextGrad를 중심 소재로 승격. 런타임 self-improving(Self-Refine·Reflexion·CoVe)과 빌드타임 optimizer(APE·DSPy·TextGrad·BAML)를 명확히 구분. Least-to-Most·Self-Refine·Reflexion은 이 장에만 등장(3장 중복 제거, C3 반영).

- **핵심 질문:** "프롬프트를 손으로 고칠 것인가, 컴파일할 것인가 — 그리고 언제 무엇을 고를 것인가?"
- **주요 내용:**
  - **파트 1 — 다단계 호출로의 도약 (체이닝·워크플로우):**
    - Anthropic "Chain complex prompts": 추출 → 분석 → 요약
    - 패턴 카탈로그: Map-reduce, Router(분류→분기), Orchestrator-worker
    - 언제 쪼개고 언제 통합할 것인가 — 지연·비용·정확도의 트레이드오프
  - **파트 2 — 런타임 self-improving 루프** (3장에서 의도적으로 제외한 기법 집합):
    - Least-to-Most (Zhou 2022): 자동 분해의 원형
    - Self-Refine (Madaan 2023): 자기 피드백→수정
    - Reflexion (Shinn 2023): 언어적 반성 메모리
    - Chain-of-Verification (Dhuliawala 2023): 검증 질문 생성 — 5장 환각 완화와 callback
    - **런타임 self-improving vs 빌드타임 optimizer의 결정적 구분:** 전자는 추론 시간마다 루프를 돎(비용·지연), 후자는 개발 시간에 한 번 최적화하고 추론은 짧음
  - **파트 3 — 빌드타임 자동 최적화 (이 장의 중심 소재):**
    - **APE** (Zhou 2022): 프롬프트 후보를 LLM이 생성·평가
    - **OPRO** (Yang 2023, DeepMind): 최적화 궤적을 LLM에 피드. "Take a deep breath" 에피소드
    - **DSPy** (Khattab 2023): 선언형 모듈 → 자동 프롬프트·few-shot 컴파일. Signature·Module·Teleprompter
    - **TextGrad** (Yuksekgonul 2024): 텍스트 그래디언트로 프롬프트·few-shot·하이퍼파라미터 동시 튜닝
    - BAML의 선언형 관점 — 프롬프트를 함수 시그니처로
  - **파트 4 — 하이브리드 실무:** 초기 템플릿은 사람이 설계, 미세 조정은 optimizer로. 대형 제품팀의 현실(§7.2).
  - 10장 Evals와의 의존: 자동 최적화는 eval 없이는 작동하지 않는다 — 8·10장이 쌍
- **독자가 얻는 것:** "프롬프트를 컴파일한다"는 은유를 머릿속에 심기. 수동 vs 자동 결정 트리. 월요일에 DSPy 파일럿을 시도할 최소 레시피.
- **예상 분량:** 18,000자 (약 23p) — C1로 분량 14,000 → 18,000 확대
- **참조 섹션:** §3.7, §3.9, §3.10, §3.13, §7.2, §10-5
- **오프닝 hook (인용 진입 — S1 다양화):** "OPRO 논문의 한 문장이 커뮤니티를 놀라게 했다. '최적의 프롬프트 접두사는 Take a deep breath였다.' 사람이 고안한 수많은 팁을 제치고, LLM이 LLM에게 숨 한 번 쉬라고 말하는 게 가장 효과적이었다. 이 장은 그런 발견을 당신 손으로 만드는 장이다."

### 10장. 평가(Evals) — 측정하지 않으면 개선할 수 없다

- **C1 (자동 최적화 비중):** 8장을 "체이닝과 워크플로우"에서 **"프롬프트 자동 최적화 — 수동 설계에서 데이터 기반으로"**로 재명명. DSPy/OPRO/TextGrad/APE를 중심 소재로 승격하고 4파트(체이닝 / 런타임 self-improving / 빌드타임 optimizer / 하이브리드 실무)로 재구성. 분량 14,000 → 18,000자. 런타임 vs 빌드타임 구분 명시.
- **C2 (환각 단일 주소):** 5장을 "RAG와 긴 문맥"에서 **"RAG와 환각 — 컨텍스트로 진실을 말하게 하기"**로 재구성. Chain-of-Verification, Self-RAG, semantic entropy(Farquhar 2024), FActScore/HaluEval/TruthfulQA를 독립 절로. Long Context Engineering은 4장 Context Engineering으로 흡수.
- **C3 (3장·8장 중복 제거):** 3장에서 Least-to-Most, Self-Refine, Reflexion, CoVe를 **전부 제거**하고 8장에만 배치. 3장은 "단일 호출 기법" (Zero/Few-shot, CoT, Self-Consistency, PoT/PAL, Role)로 스코프 고정. 각 장 본문에 "이 장에서 다루지 않는 것" 박스로 경계 명시.
- **S1 (오프닝 hook 다양화):** v1에서 9개였던 1인칭 고백체를 3개(1·9·13장)로 축소. 나머지 10개는 수사적 질문(2·10장), 숫자/사실(3장), 인용(4·8장), 역사적 일화/뉴스(5·11장), 코드·에러(6·12장), 반대 입장 인용(7장)으로 교체. 리뷰어가 요구한 4가지 톤(코드/에러·역사적 일화·반대 입장·수사적 질문) 모두 충족.
