# 9장. 에이전트 프롬프팅 — 모델에게 손과 발을 주는 법

Cursor가 내 코드베이스를 수정하는 걸 처음 본 날, 무서웠습니다. 어떻게 저렇게 "알아서" 움직이지? 파일을 열고, 읽고, 수정안을 제시하고, 심지어 테스트까지 돌린다. 몇 달 뒤 Claude Code가 등장해 터미널에서 직접 서브 에이전트를 스폰하고, 브랜치를 만들고, 커밋 메시지를 다듬는 걸 봤을 때는 감탄과 불안이 반반이었어요. 그 뒤로 에이전트 시스템 프롬프트를 하나씩 뜯어봤습니다. 마법은 없었어요. **설계**가 있었을 뿐입니다.

이 장은 그 설계의 문법에 관한 이야기다. 모델이 툴을 고르고, 계획을 세우고, 실패에서 회복하게 하려면 프롬프트를 어떻게 써야 하는지 — 2022년 ReAct 논문에서 2025년 Claude Code의 SubAgent 패턴까지, 에이전트 프롬프팅의 사실상 표준이 어떻게 굳어졌는지를 따라가 보자.

한 가지 먼저 당부해 두고 싶다. 에이전트는 **긴 컨텍스트로 돌아가는 게 아니라 좋은 프롬프트로 돌아간다.** 100만 토큰 창이 있어도 목표가 모호하고 툴 매니페스트가 뭉개져 있으면 에이전트는 엉뚱한 곳을 헤맨다. 반대로 16k 토큰 창이어도 5부 구조가 제대로 잡혀 있으면 꽤 안정적으로 작동한다. 이 장 내내 이 메시지를 반복할 생각이다.

## 9.1 ReAct — 에이전트의 공통 조상

에이전트 이야기를 시작할 때 건너뛸 수 없는 한 편의 논문이 있다. Yao et al., "ReAct: Synergizing Reasoning and Acting in Language Models" (arXiv:2210.03629, 2022)다. ReAct가 등장하기 전까지 LLM의 추론(Chain-of-Thought)과 행동(tool use, API call)은 따로 놀고 있었다. 추론은 종이 위에서만 진행됐고, 행동은 외부 API로 내보내기만 했다. 둘이 만나지 못했다.

ReAct는 간단한 아이디어로 이 둘을 엮었다. 모델이 한 턴에 **Thought → Action → Observation** 세 항목을 번갈아 쓰게 한 것이다.

```
Question: Colorado orogeny의 동부 구역이 확장되는 고도 범위는?

Thought 1: 먼저 Colorado orogeny가 무엇인지 검색해 동부 구역의 지리를
         파악해야겠다.
Action 1: search[Colorado orogeny]
Observation 1: The Colorado orogeny was an episode of mountain building
             in Colorado and surrounding areas...

Thought 2: 동부 구역에 대한 언급이 없다. 더 구체적으로 찾아보자.
Action 2: search[Colorado orogeny eastern sector]
Observation 2: The eastern sector extends into the High Plains and
             is called the Central Plains orogeny.

Thought 3: 이제 High Plains의 고도 범위를 찾으면 된다.
Action 3: search[High Plains elevation]
Observation 3: The High Plains rise from around 1,800 ft in the east
             to over 7,000 ft in the west.

Thought 4: 답을 찾았다.
Action 4: finish[1,800 to 7,000 ft]
```

이 포맷이 왜 그렇게 큰 변화를 만들었을까? 세 가지 이유가 있다.

첫째, **추론이 행동을 유도하고, 행동이 추론을 교정한다.** Thought 1은 막연하지만, Observation 1을 받아 Thought 2에서 "더 구체적으로 찾자"는 판단이 나온다. 인간 연구자가 검색창을 켜고 하는 일과 정확히 같다.

둘째, **실패가 드러난다.** 추론만 하는 모델은 틀린 답도 확신에 차서 말한다(환각). 그런데 Action의 결과인 Observation이 "아무 결과 없음"이라고 답하면 모델은 "아, 질문을 바꿔야 하는구나"를 알게 된다. 실패가 프롬프트 안에서 가시화되는 것이다.

셋째, **프롬프트만으로 구현된다.** ReAct는 추가 학습이 없다. Few-shot 예시 몇 개로 모델에게 Thought/Action/Observation 포맷을 가르치면 끝이다. 2022년 당시 이게 얼마나 강력한 발견이었는지를 상상해 보자. 에이전트를 "파인튜닝"해야 한다고 믿던 시기였다. Yao et al.은 "아니다. 프롬프트로 된다"라고 선언한 셈이다.

오늘날 우리가 쓰는 거의 모든 에이전트 — LangChain, LlamaIndex, OpenAI function calling, Anthropic tool use — 는 이 ReAct 루프의 변형이다. Thought가 `reasoning` 필드로, Action이 `tool_call`로, Observation이 `tool_result`로 이름이 바뀌었을 뿐, 구조는 동일하다.

그렇다면 우리는 2026년에 ReAct를 어떻게 다시 읽어야 할까? 원래 논문처럼 Thought를 문자열로 프롬프트에 박아 넣을 필요는 이제 거의 없다. 함수 호출 API가 그 역할을 대신한다. 대신 **"왜 ReAct가 작동하는가"의 세 이유**를 기억하자. 추론과 행동의 교차, 실패의 가시화, 프롬프트만으로 구현. 이 세 기둥이 흔들리면 에이전트는 엉망이 된다.

## 9.2 에이전트 시스템 프롬프트 — 사실상 표준 5부 구조

본격적으로 손을 움직여 보자. 에이전트에게 시스템 프롬프트를 써야 한다면, 무엇을 담아야 할까?

여러 공식 자료의 합의점을 추리면 **5부 구조**가 나온다. Anthropic의 "Building effective agents" (2024-12), OpenAI의 "A practical guide to building agents" (2025), Lilian Weng의 "LLM-powered Autonomous Agents" (2023) — 세 글의 강조점은 조금씩 다르지만, 다음 다섯 부분은 공통적으로 등장한다.

1. **Role & Goal** — 너는 누구이며 무엇을 이룰 것인가
2. **Tool Manifest** — 네가 쓸 수 있는 도구의 목록과 사용법
3. **Policy** — 무엇을 하고, 무엇을 하지 말아야 하는가
4. **Output Format & Stop Condition** — 어떤 형식으로 응답하고 언제 멈추는가
5. **Trajectory Examples** — 성공적인 다단계 실행의 예시

하나씩 풀어 보자.

### 9.2.1 Role & Goal — 문장 세 개로 정의되는 정체성

에이전트에게 "너는 유능한 조수다"라고 말하는 건 쓸모없다. 너무 모호해서 모델이 아무 역할이나 채워 넣는다. 대신 **미션 한 문장 + 책임 범위 한 문장 + 성공 정의 한 문장**을 써 보자.

```
# Role
너는 `release-notes-writer`다. GitHub PR 목록을 읽고 제품 사용자를
위한 릴리스 노트를 작성한다.

# Goal
주어진 PR 번호들에서 사용자 영향(user-facing change)을 추출해
마크다운 릴리스 노트로 만든다. 내부 리팩토링이나 의존성 범프는
포함하지 않는다.

# Success criteria
- 각 PR에 한 줄 요약(≤80자)과 PR 링크가 있다
- "Breaking changes" 섹션이 가장 위에 온다
- 사용자 관점 용어만 사용한다 (예: "DB migration" → "로그인 지연")
```

왜 이게 효과적인가? 에이전트가 중간에 길을 잃을 때 돌아올 **닻(anchor)**이 있기 때문이다. 루프 15회째에 "다음엔 뭘 하지?"를 자문하는 순간, 모델은 system prompt의 Goal 문장을 다시 읽는다. Success criteria가 명확하면 "지금 작업이 거기 맞는가"를 판단할 수 있다. 없으면 엉뚱한 방향으로 더 깊이 들어간다.

### 9.2.2 Tool Manifest — 설명이 문서다

에이전트에게 툴을 주는 방법은 공급사별로 조금씩 다르지만(OpenAI function calling, Anthropic tool_use, Gemini function calling), 본질은 같다. **JSON 스키마 + description**이다. 그리고 이 description이 프롬프트의 숨은 주인공이다.

초보 실수를 살펴보자. 난감한 description의 전형이다.

```json
{
  "name": "get_data",
  "description": "Gets data",
  "parameters": {
    "type": "object",
    "properties": {
      "id": {"type": "string"}
    }
  }
}
```

이러면 모델은 툴을 언제 써야 할지 판단할 수 없다. "data"가 뭔지, "id"가 어떤 형식인지, 실패하면 어떻게 되는지 — 전부 모른다. 결과는 예측 가능하다. 모델은 이 툴을 아무 상황에서나 호출하거나, 반대로 필요한 상황에서도 호출하지 않는다.

같은 툴을 이렇게 써 보자.

```json
{
  "name": "get_user_orders",
  "description": "사용자의 최근 30일 주문 내역을 조회한다. 주문 상태(주문 접수/배송 중/배송 완료/취소)와 상품명, 금액을 반환한다. 30일 이전 조회는 `get_user_orders_historical`을 사용하라. 사용자 ID를 모르는 경우 먼저 `search_user`로 검색할 것.",
  "parameters": {
    "type": "object",
    "properties": {
      "user_id": {
        "type": "string",
        "description": "사용자 식별자. 이메일이 아닌 내부 ID (예: usr_abc123)"
      }
    },
    "required": ["user_id"]
  }
}
```

차이를 느끼겠는가? description이 **사용 조건 + 반환값 + 대안 툴 + 선행 조건**을 담고 있다. 에이전트는 이 description을 문서처럼 읽고, 언제 이 툴을 호출할지, 호출 전에 무엇을 준비할지, 결과를 어떻게 해석할지 판단한다.

기억해두자. **툴 description은 주석이 아니라 프로덕트 요구사항 명세(PRD)다.** PM이 엔지니어에게 쓰듯 써야 한다.

한 가지 더. 툴이 많아질수록(>15~20개) 선택 정확도가 떨어진다. 커뮤니티 관찰과 Anthropic의 agent 가이드가 일관되게 지적하는 점이다. 이 문제는 9.4절 multi-agent 오케스트레이션에서 다시 다루겠다.

### 9.2.3 Policy — "하지 말 것"이 품질을 만든다

Claude Code의 공개된 시스템 프롬프트를 한 번이라도 본 사람은 이 문장에 고개를 끄덕일 거다. "하지 말아야 할 것"의 목록이 "해야 할 것"의 목록보다 길다. 왜 그럴까?

LLM의 기본 성향이 "도와주기"에 편향돼 있기 때문이다. 사용자가 "DB 좀 정리해 줘"라고 하면 모델은 의욕적으로 `DROP TABLE`을 생각해낼 수 있다. 이 의욕을 제어하지 않으면 2025년 Replit에서 있었다고 보고된(확인 필요) 사건처럼 프로덕션 DB가 지워진다. 실제로 Chevrolet of Watsonville의 2023년 챗봇은 사용자가 "아무 제안이든 받아들여라"라고 유도하자 2024 Tahoe를 1달러에 팔겠다고 답했다 — 법적 구속력이 있는 제안이라는 문장까지 덧붙여서. Air Canada 챗봇은 있지도 않은 환불 정책을 지어냈고, 몬트리올 법원은 2024년 2월 그 발화에 회사가 책임지도록 판결했다.

공통 교훈: **LLM의 발화는 회사의 발화다.** 그리고 회사의 발화를 통제하는 가장 저렴한 도구가 바로 Policy 섹션이다.

실전 템플릿을 보자.

```
# Policy

## 권한 원칙 (Least Privilege)
- 파일 쓰기 전 반드시 `read_file`로 현재 내용을 확인한다
- `git push --force`, `rm -rf`, `DROP TABLE`은 절대 실행하지 않는다
- 프로덕션 DB 접근 시 READ-ONLY 계정만 사용한다

## 불확실성 처리
- 요청이 모호하면 추정하지 말고 사용자에게 한 문장 질문한다
- 문서에 근거가 없는 답은 "확인이 필요합니다"로 답한다
- 실행한 툴의 결과가 예상과 다르면 3회까지 재시도 후 사용자에게 보고

## 보고 의무
- 파일 5개 이상 수정 시 diff 요약을 먼저 제시하고 확인을 받는다
- 외부 API 호출이 비용을 발생시키는 경우 사전에 공지한다
- 민감 정보(시크릿, PII)는 응답에 포함하지 않는다

## 이 에이전트의 스코프 밖
- 코드 리뷰 승인 (승인은 사람이 한다)
- 사용자 계정 삭제·병합
- 요금제 변경
```

눈여겨볼 지점이 하나 있다. "이 에이전트의 스코프 밖" 섹션이다. 이걸 넣지 않으면 에이전트는 "도움이 될 것 같으니까" 스코프를 넘나든다. 차라리 명시적으로 "이건 내 일이 아니다"를 가르쳐 주는 편이 낫다.

### 9.2.4 Output Format & Stop Condition — 언제 멈추는가

에이전트의 실패 모드 중 가장 번거로운 게 뭔지 아는가? 무한 루프다. 툴을 호출하고, 결과를 보고, 또 호출하고, 또 보고 — 멈추지 않는다. Hamel Husain의 블로그 댓글이나 r/LocalLLaMA 스레드에는 "에이전트가 30분째 탐색 중이다" 같은 호소가 주기적으로 올라온다.

그 원인의 대부분은 **stop condition이 프롬프트에 없기 때문**이다. 모델은 "언제 충분한가"를 모른다. 가르쳐 주자.

```
# Output Format & Stop Condition

## 완료 기준
다음 조건 중 하나라도 충족되면 최종 답변을 생성하고 종료한다:
- 요청된 모든 PR을 처리했다
- 사용자 확인이 필요한 결정이 나왔다 (이 경우 질문 문장을 출력)
- 5회 연속 툴 호출이 새로운 정보를 제공하지 못했다 (dead-end 감지)

## 최종 답변 포맷
```markdown
## Release Notes — v{version}

### Breaking changes
- ...

### New features
- ...

### Fixes
- ...
```

## 중간 진행 보고
툴 10회 호출마다 한 줄로 진행 상황 요약 (예: "PR 5/12 완료")
```

세 가지 포인트를 짚고 가자. 첫째, 완료 기준을 **복수로** 명시했다. 성공 종료뿐 아니라 "사용자 확인 필요"와 "dead-end"도 종료 사유다. 이걸 넣지 않으면 에이전트는 "완벽한 답"을 찾으려고 영원히 헤맨다. 둘째, 최종 포맷을 스켈레톤으로 보여 줬다. 모델은 구조를 보면 구조대로 채운다. 셋째, 중간 보고 규칙이 있다. 장시간 실행되는 에이전트를 관측(observability)하려면 반드시 필요하다.

### 9.2.5 Trajectory Examples — 하나의 완주 예시

마지막 5부는 **예시 궤적** 한두 개다. 입력부터 최종 출력까지 에이전트가 어떻게 움직이는지를 보여 주는, 말하자면 튜토리얼이다. Few-shot 예시가 단일 호출에서 포맷을 학습시키듯, trajectory example은 에이전트에게 **루프의 리듬**을 학습시킨다.

짧게 한 번만 보자.

```
# Example trajectory

User: "PR #142, #143, #145에 대한 릴리스 노트를 써 줘."

Turn 1:
  tool_call: get_pr(pr_number=142)
  tool_result: {title: "Add OAuth login", labels: ["feature", "user-facing"], ...}

Turn 2:
  tool_call: get_pr(pr_number=143)
  tool_result: {title: "Refactor auth middleware", labels: ["internal"], ...}
  # 내부 리팩토링이므로 최종 노트에 포함하지 않음

Turn 3:
  tool_call: get_pr(pr_number=145)
  tool_result: {title: "Fix login timeout", labels: ["bug", "user-facing"], ...}

Turn 4:
  최종 답변:
  ```markdown
  ## Release Notes

  ### New features
  - OAuth 로그인 지원 (#142)

  ### Fixes
  - 로그인 지연 문제 해결 (#145)
  ```
```

예시를 잘 고르는 팁 하나. **"판단"을 포함하는 궤적을 고르자.** 위 예시에서 Turn 2는 PR을 조회하고도 최종 노트에 **포함하지 않기로** 결정한다. 이런 판단 예시가 있어야 모델이 규칙을 내면화한다. 단순한 성공 궤적만 보여 주면 모델은 툴을 기계적으로 호출하기만 한다.

자, 5부 구조를 한 번에 모아 보자. 실제로 프로덕션에 투입할 만한 에이전트 시스템 프롬프트는 대략 이렇게 생겼다.

```
# Role
# Goal
# Success criteria

# Tools
<tool manifest with rich descriptions>

# Policy
## 권한 원칙
## 불확실성 처리
## 보고 의무
## 스코프 밖

# Output Format & Stop Condition
## 완료 기준
## 최종 답변 포맷
## 중간 진행 보고

# Example trajectory
<1–2 worked examples>
```

"이거 너무 길지 않나?"라고 생각할 수 있다. 2024년 중반까지는 그 걱정이 합당했다. 토큰당 요금이 비쌌고, system prompt가 매 호출마다 과금됐다. 그러나 2024-08 Anthropic이, 2024-10 OpenAI가 **prompt caching**을 표준화하면서 그림이 바뀌었다. 앞에 둔 긴 system prompt는 캐시에 태우면 90%까지 싸진다. 길게 쓰는 걸 두려워하지 말자. 대신 뒤에 가변 입력을 두어 캐시 히트를 최대화하면 된다.

## 9.3 To-Do List 패턴 — 계획을 컨텍스트에 유지하기

에이전트가 길을 잃는 또 다른 흔한 실패 모드가 있다. 세 번째 툴 호출쯤 되면 "원래 내가 뭘 하려 했더라?"를 잊어버리는 것이다. 긴 컨텍스트 모델(Gemini 2M, Claude 200k)도 이 문제에서 자유롭지 않다. Lost-in-the-Middle(Liu et al., arXiv:2307.03172) 편향 때문에 맨 처음에 준 Goal 문장이 흐릿해진다.

해결책이 흥미롭다. 에이전트가 **자기 계획을 자기 컨텍스트에 쓰게** 하는 거다. Claude Code가 TodoWrite 툴을 쓰는 이유가 여기에 있다. 에이전트는 작업을 시작할 때 해야 할 일을 목록으로 뽑고, 하나씩 수행하면서 상태를 업데이트한다.

패턴은 간단하다.

```
# 프롬프트에 추가할 지시

## Planning
복잡한 작업(3단계 이상 예상)을 시작하기 전에:
1. 작업을 To-do 목록으로 쪼갠다 (`create_todo_list` 툴 사용)
2. 각 항목을 수행하면서 상태를 업데이트한다 (`update_todo` 툴 사용)
3. 예상치 못한 발견이 있으면 목록을 재작성한다

## To-do 항목 작성 규칙
- 구체적 동사로 시작 ("파일 읽기"가 아니라 "src/auth.py의 login() 함수 읽기")
- 각 항목은 15분 이내에 완료 가능한 크기로
- 의존성이 있으면 순서를 명시
```

왜 이게 효과적일까? 세 가지 이유가 있다.

첫째, **계획이 프롬프트 맨 앞이 아니라 "최근 대화"에 머물게 된다.** Lost-in-the-Middle의 "맨 뒤"에 계속 계획이 붙어 있으니 recency bias의 덕을 본다. 둘째, **상태가 외부화된다.** 모델이 머릿속에서만 계획을 유지하면 쉽게 흐트러지지만, 명시적 목록이 있으면 매 턴 이를 참조한다. 셋째, **사용자가 관찰할 수 있다.** To-do가 업데이트되는 걸 사용자가 보면 "이 에이전트가 뭐 하는 중인지" 불안이 줄어든다. 신뢰는 관측성에서 나온다.

한 가지 주의. To-do 목록을 강제하면 **단순한 일까지 거창한 계획으로 만드는** 부작용이 생긴다. "PR 하나만 요약해 줘"에 에이전트가 7단계 계획을 세우면 번거롭다. 그래서 "3단계 이상 예상되는 경우"라는 조건을 지시에 넣는 거다. 단순한 과제는 바로 수행하고, 복잡한 과제만 계획한다.

Devin(Cognition Labs, 2024)이 공개한 데모를 본 독자라면 이 패턴이 낯설지 않을 것이다. Devin은 화면에 "Current plan" 창을 띄우고 실시간으로 항목을 업데이트한다. 사용자는 에이전트의 상태를 시각적으로 추적한다. Claude Code의 Todo 관리도 같은 철학이다. 에이전트가 "생각하는 척"이 아니라 "기록하는" 것이 중요하다.

## 9.4 Multi-Agent 오케스트레이션 — 하나가 안 되면 여럿으로

툴이 많아질 때(>15~20개) 에이전트 선택 정확도가 떨어진다고 앞서 언급했다. 이 문제를 정면으로 풀려면 어떻게 해야 할까?

한 가지 방법은 **계층화**다. 하나의 거대한 에이전트 대신, 리드 에이전트가 서브 에이전트에게 일을 위임하는 구조. 이걸 Orchestrator-Worker 패턴, 혹은 Multi-Agent 시스템이라고 부른다. Anthropic의 "Building effective agents" 글은 이 패턴을 여러 형태로 정리한다 — Routing, Parallelization, Orchestrator-workers, Evaluator-optimizer.

가장 실용적인 형태인 **Orchestrator-Workers**를 살펴보자.

```
┌─────────────────┐
│  Lead Agent     │  ← 사용자와 대화, 계획 수립, 서브 에이전트 스폰
│  (Orchestrator) │
└────────┬────────┘
         │ spawn + message
         ├──────────┬──────────┬──────────┐
         ▼          ▼          ▼          ▼
    ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
    │ Agent1 │ │ Agent2 │ │ Agent3 │ │ Agent4 │
    │ Search │ │ Code   │ │ Writer │ │ Review │
    └────────┘ └────────┘ └────────┘ └────────┘
```

리드 에이전트는 큰 그림을 본다. "릴리스 노트를 쓰려면 (a) PR을 모으고 (b) 사용자 영향을 판단하고 (c) 문구를 다듬고 (d) 검토해야 한다"까지 계획한다. 각 단계를 서브 에이전트에게 넘긴다. 서브 에이전트는 좁은 범위에서 툴 5~7개만 다루면 되니까 선택 정확도가 회복된다.

리드-서브 사이의 메시지 프로토콜이 이 패턴의 핵심이다. 다음 필드를 고정하자.

```json
{
  "from": "lead-agent",
  "to": "code-search-agent",
  "task_id": "find-auth-bug-001",
  "goal": "src/auth/ 디렉토리에서 JWT 검증 로직을 찾아 파일 경로와 함수명을 반환하라",
  "constraints": [
    "파일 수정하지 말 것",
    "최대 10분 이내 완료",
    "결과를 JSON 형식 {files: [{path, function_name, line_range}]}로 반환"
  ],
  "context": {
    "repo_path": "/workspace/myapp",
    "related_issue": "#1234"
  }
}
```

왜 이런 고정 프로토콜이 필요할까? **서브 에이전트가 리드의 전체 컨텍스트를 받지 않아야 하기 때문이다.** Multi-agent의 함정 중 하나가 "컨텍스트 폭발"이다. 리드가 쌓은 10만 토큰을 서브에게 그대로 넘기면 토큰 비용이 지수적으로 폭증한다. 대신 goal + constraints + minimal context만 전달하고, 서브는 격리된 컨텍스트에서 좁은 작업을 수행한 뒤 요약만 반환한다.

Claude Code의 SubAgent 패턴이 정확히 이렇다. 메인 세션이 Task 툴로 서브 에이전트를 스폰할 때, 메인 컨텍스트는 넘어가지 않는다. 서브는 받은 프롬프트만으로 격리 실행되고, 최종 메시지를 메인에 돌려준다. 토큰 효율과 실수 격리를 동시에 얻는 구조다.

실전에서 멀티 에이전트가 **오히려 해가 되는 경우**도 있다. 간단한 과제를 지나치게 쪼개면 리드-서브 간 통신 오버헤드가 순이익을 잡아먹는다. Anthropic의 가이드도 "단일 에이전트로 충분하면 단일 에이전트를 써라"라고 못 박는다. 멀티 에이전트는 (a) 툴 수가 20개를 넘을 때, (b) 병렬화로 지연을 줄이고 싶을 때, (c) 역할별 격리(예: 검색 에이전트와 실행 에이전트 분리)가 필요할 때 — 이 세 경우에만 고려하자.

## 9.5 에이전트의 실패 모드 5가지

에이전트를 실제로 돌려 보면 반복적으로 마주치는 실패 패턴이 있다. 알고 있으면 디버깅이 훨씬 빨라진다.

### 9.5.1 Context Explosion — 컨텍스트가 부풀어 오른다

툴을 호출할 때마다 결과가 컨텍스트에 쌓인다. 20번째 호출쯤 되면 프롬프트 길이가 10만 토큰을 넘어간다. 비용은 차치하고, Lost-in-the-Middle 때문에 모델이 원래 Goal을 놓친다. 더 심하면 컨텍스트 윈도우를 초과해 오래된 내용이 잘려 나가고, 그 시점부터 에이전트는 방금 한 일을 잊어버린다.

대책 세 가지.

1. **요약 주기적 삽입**: 툴 10회마다 "지금까지의 상황 요약" 메시지를 리드가 스스로 작성하게 한다.
2. **Observation 필터링**: 긴 툴 결과(예: 파일 전체 내용)는 원본을 저장하고 컨텍스트에는 관련 부분만 인용한다.
3. **세션 재시작**: 특정 토큰 임계치에 도달하면 현재 상태를 To-do 목록에 세이브하고 새 세션을 시작한다.

### 9.5.2 Infinite Loop — 무한 루프

앞서 9.2.4에서 다뤘다. Stop condition을 복수로 명시하고, "N회 연속 동일 툴 호출 시 종료" 같은 하드 가드를 프롬프트에 박자. 프롬프트만으로 불안하면 실행 환경에 실제 최대 턴 수 제한을 걸자. 에이전트 프레임워크 대부분이 `max_iterations` 옵션을 제공한다.

### 9.5.3 Tool Misuse — 엉뚱한 툴 호출

사용자가 "주문 내역 보여 줘"라고 했는데 에이전트가 `delete_user_order`를 호출하는 사례. 실제로 일어난다. 원인은 대개 **툴 이름의 모호성** 또는 **description의 부실**이다.

방지책. 이름을 동사로 짓고 의도를 담자 (`get_user_orders_readonly` vs `manage_orders`). Description 첫 문장에 반드시 "무엇을 하는지"와 "반환값"을 쓰자. 위험한 툴에는 dry-run 모드를 두고 기본값을 dry-run으로 두자.

### 9.5.4 Hallucinated Tool Call — 존재하지 않는 툴을 부른다

모델이 "이쯤이면 `send_email` 툴이 있겠지"라고 추정하고 존재하지 않는 툴을 호출하는 경우. Tool manifest에 없는 함수다. 실행 레이어는 에러를 반환하지만, 모델이 에러를 무시하고 같은 환각 툴을 반복 호출하는 경우도 있다.

방지책. (a) 실행 레이어에서 존재하지 않는 툴 호출 시 **명시적이고 짧은 에러 메시지**를 반환한다(`Tool 'send_email' is not available. Available tools: [...]`). (b) 프롬프트 Policy에 "available tools 외에는 호출하지 말 것, 필요한 툴이 없으면 사용자에게 요청"을 명시한다. (c) 에러가 같은 턴에 3번 반복되면 자동 종료한다.

### 9.5.5 Silent Drift — 미묘한 목표 이탈

가장 찜찜한 실패다. 에이전트가 에러 없이 돌아가는데, 결과를 보면 원래 요청과 미묘하게 다르다. "코드 리뷰"를 시켰는데 리팩토링까지 해버렸거나, "데이터 요약"을 시켰는데 요약+분석+추천까지 줬다.

원인은 대개 LLM의 "도움 편향"이다. 모델은 "더 해 주고 싶다." 프롬프트에 Success criteria를 명시하고, 그 범위를 넘으면 멈추라고 강조해야 한다. 9.2.3의 "스코프 밖" 섹션이 이걸 막는다.

## 9.6 실전 사례 — Cursor, Devin, Claude Code

추상적 원칙만 보면 어렵다. 실제 에이전트 제품들이 어떻게 했는지 세 사례만 빠르게 보자.

**Cursor의 에이전트 모드.** Cursor는 IDE 안에서 파일을 읽고 편집하는 에이전트다. 시스템 프롬프트의 공개된 일부 지침에 이런 문장이 있다 — "edit as minimal diffs." 파일을 통째로 다시 쓰지 말고, 가능한 한 작은 diff로 편집하라는 지시다. 왜 이게 중요할까? 큰 rewrite는 (a) 리뷰하기 어렵고 (b) 기존 주석이나 포맷을 지우고 (c) 테스트를 망가뜨린다. Cursor 팀은 이를 경험으로 배웠고, 원칙을 프롬프트에 박았다. 툴을 주는 것만큼 **"툴을 어떻게 쓰는가"의 규범**이 중요하다는 사례다.

**Devin의 계획 구조.** Cognition Labs의 Devin은 장시간 실행(수 시간~수 일) 에이전트를 표방했다. 시연에서 Devin은 화면 분할로 "Plan" 창을 띄우고 실시간으로 항목을 갱신한다. To-do 패턴(9.3)의 극단적 구현이다. 완전 검증된 상용 수준인지는 커뮤니티에서 논쟁 중이다(일부 데모가 에디팅됐다는 지적, 확인 필요). 다만 **"계획을 외부화하면 에이전트 신뢰도가 오른다"는 원리**만큼은 업계가 받아들인 것으로 보인다.

**Claude Code의 SubAgent 패턴.** 이 책의 저술 하네스 자체가 이 패턴을 쓴다. 메인 오케스트레이터가 `research-lead`, `planner`, `chapter-writer`, `style-guardian` 같은 서브 에이전트를 스폰한다. 각 서브는 자기 역할에 맞는 system prompt, tool subset, success criteria를 받고 격리 실행된다. 메인은 서브의 최종 요약만 받아 다음 단계를 결정한다. 9.4의 Orchestrator-Workers 패턴이 실제로 돌아가는 모습이다.

세 사례의 공통점은 무엇일까? **모두 "무엇을 할지"보다 "어떻게 할지"와 "하지 말아야 할 것"에 프롬프트의 무게가 실려 있다.** 에이전트의 자유도를 높이면 능력도 올라가지만 리스크도 함께 커진다. 가드레일을 프롬프트로 심는 것이 2026년 에이전트 제품의 본체 작업이다.

## 9.7 8장과의 연결 — 에이전트 루프는 런타임 Self-Improving이다

잠시 8장 이야기를 꺼내 보자. 8장에서 우리는 런타임 self-improving(Self-Refine, Reflexion, CoVe)과 빌드타임 optimizer(DSPy, TextGrad, APE)를 구분했다. 에이전트 루프 자체가 런타임 self-improving의 실전 예다.

Thought → Action → Observation 사이클은 본질적으로 **자기 피드백 루프**다. Observation이 모델이 기대한 결과와 다르면 Thought에서 계획이 수정된다. Reflexion의 "언어적 반성" 개념은 에이전트 프롬프트의 "failure recovery" 지침으로 녹아든다.

```
# Policy (추가)

## 실패 복구
- 툴 호출이 예상과 다른 결과를 반환하면:
  1. "무엇이 달랐는가"를 한 문장으로 기록
  2. 원인 가설을 세 개 이내로 세움
  3. 가장 가능성 높은 가설로 다음 툴 호출 조정
- 같은 실패가 3회 반복되면 접근 방식을 근본적으로 바꾸거나
  사용자에게 보고
```

에이전트 프롬프팅과 자동 최적화는 양쪽에서 같은 목표로 수렴한다. **"모델이 스스로 교정하도록 구조를 만드는 것."** 런타임은 매 호출에서, 빌드타임은 eval 루프에서. 에이전트를 설계한다면 두 축을 모두 의식하자.

## 9.8 MCP와 프로토콜의 시대 — 에이전트 생태계의 미래

지금까지 다룬 프롬프트 구조는 "단일 시스템 안에서" 에이전트를 설계하는 이야기였다. 그런데 2024년 말부터 흐름이 바뀌고 있다. 에이전트가 **여러 시스템을 가로질러** 일하는 시대다.

2024-11 Anthropic이 공개한 **MCP(Model Context Protocol)**는 툴 정의를 표준화하려는 시도다. "이 서버는 이런 툴을 제공한다"를 JSON으로 선언하면, 어떤 LLM이든 그 서버의 툴을 쓸 수 있다. 비유하자면 에이전트판 USB다. 개별 에이전트마다 툴을 하드코딩할 필요가 없어진다.

OpenAI는 **Responses API**로 에이전트 상태 관리를 표준화했고, Google은 **A2A(Agent-to-Agent) 프로토콜**로 에이전트 간 통신 포맷을 제안했다. 2026년 4월 현재 세 프로토콜이 경쟁 중이고, 생태계가 어느 쪽으로 수렴할지는 아직 안갯속이다.

이 흐름이 프롬프트 엔지니어링에 주는 의미는 두 가지다.

첫째, **툴 description이 점점 더 중요해진다.** MCP 서버 하나가 여러 LLM 클라이언트에 의해 소비되면, description이 유일한 "사용법 문서"가 된다. 9.2.2에서 강조한 description 품질이 에이전트 생태계의 공공재가 된다.

둘째, **에이전트 시스템 프롬프트가 얇아진다.** 과거에는 툴 매니페스트를 system prompt에 통째로 넣었다. MCP 시대에는 툴이 런타임에 동적으로 로드된다. 시스템 프롬프트는 Role/Goal/Policy 같은 "에이전트 정체성"에 집중하고, 툴 정보는 외부에서 주입된다.

어느 프로토콜이 이기든 한 가지는 분명하다. **에이전트는 계속 늘어난다.** 그리고 늘어날수록 **프롬프트의 뼈대가 얼마나 단단한가**가 에이전트 간 호환성과 신뢰성을 결정할 것이다.

## 9.9 처음이라면 이것만 — 에이전트 최소 템플릿

학생 독자, 그리고 이제 막 에이전트를 만들려는 개발자에게 출발점을 제시한다. 아래 템플릿은 9.2의 5부 구조를 최소 분량으로 압축한 것이다. 복붙해서 시작하자.

```python
AGENT_SYSTEM_PROMPT = """
# Role
너는 {역할}이다. {한 줄 미션}.

# Goal
{구체적 목표 한 문장}
Success criteria:
- {측정 가능 기준 1}
- {측정 가능 기준 2}
- {측정 가능 기준 3}

# Tools
아래 툴만 사용할 수 있다. 존재하지 않는 툴은 호출하지 말 것.
사용 전 description을 반드시 읽을 것.
(툴 목록은 API의 tools 파라미터로 별도 전달됨)

# Policy
## 권한 원칙
- 파일 수정 전 read로 현재 상태 확인
- 파괴적 작업(rm, drop, force-push)은 절대 금지

## 불확실성 처리
- 모호하면 사용자에게 한 문장 질문
- 근거 없는 답은 "확인 필요합니다"

## 스코프 밖
- {이 에이전트가 하지 않는 것들}

# Output Format & Stop Condition
## 완료 기준
- 목표 달성 또는
- 사용자 확인이 필요한 결정 발생 또는
- 5회 연속 툴 호출이 진전 없음

## 최종 답변 포맷
{원하는 마크다운/JSON 포맷}

## 중간 보고
툴 10회마다 한 줄로 진행 상황 요약

# Planning (작업이 3단계 이상일 때)
1. To-do 목록을 먼저 작성
2. 항목별 수행 및 상태 업데이트
3. 예상치 못한 발견 시 목록 재작성

# Example trajectory
{판단을 포함하는 성공 궤적 1개}
"""
```

이걸 그대로 프로덕션에 쓰지는 말자. **시작점**일 뿐이다. 자기 도메인에 맞게 Policy와 Success criteria를 다듬고, 툴 description을 풍부하게 쓰고, 실패 사례를 모아 trajectory example을 보강하자. 그게 에이전트 프롬프트 엔지니어링의 일상적 업무다.

## 9.10 마무리 — 에이전트는 긴 컨텍스트가 아니라 좋은 프롬프트로 돌아간다

이 장의 한 줄 메시지다. 다시 쓸게.

**에이전트는 긴 컨텍스트가 아니라 좋은 프롬프트로 돌아간다.**

200k, 1M, 2M — 컨텍스트 윈도우 경쟁이 벌어진다. 하지만 경쟁하는 회사들이 공개한 에이전트 제품(Claude Code, ChatGPT with tools, Gemini in Workspace)을 열어 보면 하나같이 **정성껏 설계된 시스템 프롬프트**가 심장에 있다. 역할과 목표, 툴 매니페스트와 사용 규범, 정책과 실패 복구, 출력 포맷과 정지 조건, 궤적 예시. 다섯 기둥이 서 있어야 에이전트가 안정적으로 움직인다.

ReAct가 2022년에 가르쳐 준 게 결국 이거다. 추론과 행동을 교차시키고, 실패를 가시화하고, 프롬프트만으로 구현하라. 그 원리는 4년이 지난 2026년에도 유효하다. MCP나 A2A 같은 프로토콜이 생태계를 바꿔도, 한 에이전트의 내부에서는 여전히 프롬프트 엔지니어가 5부 구조를 다듬고 있다.

월요일 아침에 할 일 세 가지를 남기며 장을 닫는다.

1. **지금 쓰고 있는 에이전트의 시스템 프롬프트를 꺼내 9.2의 5부 구조에 매핑해 보자.** 빠진 부분이 있다면 그곳이 실패가 가장 먼저 터질 곳이다.
2. **툴 description을 PRD처럼 다시 써 보자.** 사용 조건, 반환값, 대안 툴, 선행 조건 — 네 항목이 있는가?
3. **stop condition을 복수로 명시하자.** "목표 달성"만으로는 부족하다. "사용자 확인 필요"와 "dead-end 감지"가 있어야 한다.

다음 10장은 에이전트를 포함한 모든 LLM 시스템의 공통 기반인 **평가(Evals)**다. Hamel Husain의 선언대로, Evals가 프롬프트 엔지니어링의 본체다. 이 장에서 설계한 에이전트가 정말 좋아지고 있는지 — "같다"와 "다"의 거리를 어떻게 메울지 — 다음 장에서 함께 풀어 보자.

## 학생용: 처음이라면 이것만

- 에이전트 = ReAct 루프 + 툴 + 시스템 프롬프트. 마법 없다.
- 시스템 프롬프트 **5부 구조**(Role/Tools/Policy/Output/Examples)를 외우자.
- **첫 에이전트는 툴 3~5개로 시작.** 15개 넘어가면 선택 정확도가 급락한다.
- To-do 패턴을 일찍 붙이자. 에이전트가 길을 잃는 대부분의 문제가 여기서 해결된다.
- 무한 루프 방어는 프롬프트와 `max_iterations` **둘 다**로.
