## 1. 책 제목 후보 (3개)

### 후보 A — **하네스 엔지니어링: Claude Code를 팀의 무기로 길들이기**
- **부제:** 모델은 수렴하고, 차이는 하네스에서 만들어진다
- **포지셔닝:** 메타-엔지니어링 정통서. "Harness Engineering"이라는 용어 자체를 한국어 엔지니어링 담론에 정착시키는 것을 목표로 하는, 다소 야심 찬 톤.
- **꽂히는 독자:** 이미 Claude Code를 몇 달 써 봤고, "이게 왜 어떤 팀에선 되고 어떤 팀에선 안 되는가"를 설명할 언어를 찾고 있는 테크리드.
- **약점:** "하네스"라는 단어가 한국어로 직관적이지 않다. 마구간·말·등산 이미지가 먼저 떠올라 1차 거부감이 있을 수 있다.

### 후보 B — **AI 코딩 에이전트 운영의 기술: Claude Code로 5–10명 팀이 실패하지 않는 법**
- **부제:** CLAUDE.md, Hooks, Subagents, Skills, MCP를 한 시스템으로 묶기
- **포지셔닝:** 실용서·운영 가이드. 의사결정 프레임과 안티패턴 카탈로그를 강조하는, 가장 안전하고 잘 팔릴 톤.
- **꽂히는 독자:** "이번 분기에 우리 팀에 Claude Code를 본격 도입해야 하는데 사고 안 나게 해야 한다"는 미션을 받은 플랫폼 엔지니어·매니저.
- **약점:** "어떻게"에는 강하지만 "왜 지금 이게 진짜 새로운 직무인가"라는 도발이 약하다. 또 다른 운영 가이드 한 권으로 묻힐 위험.

### 후보 C — **Claude Code, 팀의 운영체제로 만들기**
- **부제:** 모델이 아니라 하네스를 설계하는 사람
- **포지셔닝:** 에세이형 기술서. Karpathy의 "LLM as a new OS" 비유와 Martin Fowler의 사이버네틱 모델을 한국어 독자에게 풀어내는, 사색적이지만 실무를 놓지 않는 톤.
- **꽂히는 독자:** 기술 트렌드의 큰 그림을 잡으려는 시니어. 책장에 꽂아두고 분기마다 다시 읽고 싶어 할 만한 톤.
- **약점:** 사색이 강해지면 "이번 주 우리 팀이 무엇을 해야 하나"에 답하는 페이지가 줄어든다. 대상 독자가 원하는 즉효성이 약해진다.

### 추천: **후보 A**
이유 세 가지.
1. 리서치 종합본의 핵심 명제 — "모델은 수렴하고 차별점은 하네스로 옮겨간다(98.4% 하네스 가설, Opus 같은 모델이 Claude Code 77% vs Cursor 93%, 16%p 격차)" — 가 곧 책의 thesis다. 제목이 thesis를 직접 발화해야 한다.
2. "Harness Engineering"은 2025–2026년에 영어권에서 새로 정착 중인 용어다. 한국어 책이 이 용어를 정착시키면 독자에게 어휘를 선물하는 효과가 크다. "프롬프트 엔지니어링"이 그랬듯 단어 자체가 시장을 만든다.
3. 부제가 후보 B의 운영적 효용("길들이기")과 후보 C의 사색적 야심("모델은 수렴")을 동시에 담는다. 표지가 양쪽 독자를 끈다.

## 3. 내러티브 아크

책은 **3부 구성**으로 흐른다.

**Part I — 왜 하네스인가 (Chapter 1–2)**: 모델이 수렴하는 시점에 차이를 만드는 것은 하네스라는 명제를 설득한다. CVSS 9.4 사고·"98.4% 하네스 가설"·동일 모델 16%p 격차 같은 강한 신호로 시작해, "Agent = Model + Harness" 등식과 사이버네틱 모델·OODA 루프 등 정의 프레임을 한 챕터에 박는다. 1부의 인지적 전환은 **"내가 모델 선택과 프롬프트만 보고 있었는데, 사실 책임 영역은 그 바깥의 시스템 설계였다"**.

**Part II — 하네스의 네 축 (Chapter 3–7)**: 한 축이 한 챕터다. (3) 멀티 에이전트 오케스트레이션 — Generator-Verifier, Initializer+Coder, Fork-Join, Agent Teams. (4) 안전과 권한 — 7-mode·ML 분류기·Prompt Injection 사고·Gateway. (5) 자동화 — Hooks, PR 게이트, Block-at-submit, PR-from-anywhere. (6) 관측가능성과 비용 — OpenTelemetry, $13/dev/day 실데이터, 게이트웨이 캡. (7) 도메인 지식 코드화 — CLAUDE.md 디시플린, Skills, Ratchet Pattern. 2부의 인지적 전환은 **"하네스는 모호한 베스트 프랙티스가 아니라 네 축으로 분해 가능한 엔지니어링 대상이다"**.

**Part III — 운영의 정치학 (Chapter 8–10)**: 기술이 아닌 사람·조직·미래를 다룬다. (8) 안티패턴 카탈로그 — 운영에 흩어두지 않고 한 챕터에 모아 임팩트 확보. (9) 사례 비교 — Anthropic 내부, IMWEB, Toss/올리브영, GeekNews 5–10명 팀. (10) 도구 생태계와 1–2년 미래 — Claude Code vs Codex vs Cursor vs Aider, Managed Agents, 시니어/주니어 격차. 3부의 인지적 전환은 **"하네스 엔지니어링은 도구 책의 마지막 장이 아니라, 다음 2년의 직무 정체성이다"**.

### 책 전체를 관통하는 모티브

**"톱니바퀴(ratchet)"**. Addy Osmani가 던진 한 줄 — "각각의 에이전트 실수가 영구적 제약이 된다" — 이 책 전체의 통일 이미지가 된다. 팀의 암묵지가 한 번 명시화되면 절대 뒤로 가지 않는, 한 방향으로만 도는 톱니바퀴. 매 챕터가 끝날 때 "이 챕터에서 너의 톱니바퀴에 추가해야 할 칸은 무엇인가?"를 묻는 짧은 박스로 닫는다. 책 마지막 장에서 이 톱니바퀴가 곧 **팀 자체**가 된다는 점을 회수한다.

부속 모티브는 **"지휘자(conductor)"**. GeekNews "5인 팀처럼" 글에서 가져온 비유. 인간 엔지니어는 더 많이 코드를 쓰는 사람이 아니라, 여러 인스턴스를 동시에 지휘하는 사람으로 정체성이 바뀐다. 마지막 장에서 이 비유가 회수된다.

## 4. 챕터 목록

### Chapter 1: 왜 하네스인가 — 모델 너머로 시선을 옮기자

- **핵심 질문:** 동일한 모델이 한 도구에선 77%, 다른 도구에선 93%로 작동한다면, 우리는 그동안 무엇을 보고 있던 것일까?
- **이전 챕터로부터의 다리:** (책의 시작) 서문에서 던진 "지금 너의 팀에 Claude Code를 도입한 지 두 달, 누구는 10배 빨라졌다 하고 누구는 어제 PR을 통째로 되돌렸다. 왜 이렇게 결과가 갈리는가?"라는 질문을 받아 챕터 1이 첫 답을 내놓는다.
- **주요 내용:**
  - 오프닝 일화 1 (공포): PR 제목에 악성 지시가 박혀 GitHub Action이 자기 API 키를 코멘트로 게시한 CVSS 9.4 사고 (W15) — 1장에서는 짧은 묘사(반쪽)로만 보여주고, 본격 해부는 4장으로 넘긴다
  - 오프닝 일화 2 (가능성): GeekNews "9시 → 4탭 → 11:30 PR" 한국 팀 일화 (C1) — 외국 사고만으로 시작하는 인상을 피하고, "공포 → 가능성 → 증명"의 3단 도입을 만든다
  - "Agent = Model + Harness" 등식, "98.4%가 하네스" (P1), 동일 Opus 모델 Claude Code 77% vs Cursor 93% (W23)
  - "Skill Issue 가설": 더 좋은 모델을 기다리는 것은 핵심을 빗나간다 (W1)
  - 수렴 명제: "Top coding agents look more like each other than their underlying models" (W7)
  - 책의 thesis 선언: 차별점은 모델이 아니라 하네스에서 만들어진다, 그리고 그 책임은 너에게 있다
  - 책 전체 지도(1쪽 분량 약속): 3부 구성·4축·두 모티브를 한 페이지에
- **독자가 끝나고 할 수 있게 되는 것:** (1) 동료에게 "왜 우리 팀이 더 빠른가/느린가"를 모델 탓이 아닌 하네스의 언어로 설명할 수 있다. (2) 하네스 투자가 모델 업그레이드 대기보다 ROI가 높다는 주장을 데이터로 뒷받침할 수 있다.
- **활용할 리서치 자원:** §1.2 (Skill Issue, 98.4%, 16%p 격차), §3.1.4 (CVSS 9.4 사고 — 짧게), §7.4 (수렴 명제), C1 (GeekNews 한국 팀 일화)
- **예상 분량:** 11,000자
- **챕터 톤:** 도입은 공포(CVSS) → 가능성(한국 팀) → 증명(수치) 3단으로. 중반은 수치 폭격(독자가 "어, 진짜?"하고 멈춰서게), 마지막은 thesis 선언과 책 전체 지도(1쪽). CVSS 사고 묘사는 짧게 — 풀이는 4장에서.

### Chapter 2: 하네스의 해부학 — 우리가 다루는 것의 정의

- **핵심 질문:** "하네스를 잘 짜자"고 말할 때, 우리는 정확히 무엇을 짜는 것일까?
- **이전 챕터로부터의 다리:** 1장에서 "하네스가 차이를 만든다"는 명제를 받아들인 독자에게, 이제 그 하네스가 무엇으로 구성되는지 정의를 박아준다. 이후 모든 챕터의 공용 어휘를 여기서 확정.
- **주요 내용:**
  - 4계층 어휘 정리: Prompt / Context / Harness / Scaffolding (W17 인용 "Prompt is a subset of Context, Context is a subset of Harness")

    - **Skill = WHAT** (특정 종류의 작업을 어떻게 하는가, progressive disclosure로 필요할 때 로드)
    - **Subagent = WHO** (분리된 컨텍스트와 역할, "이 작업은 다른 머리로")
    - **메인+Task = DYNAMIC** (메인이 그때그때 위임, 미리 정의 안 된 일회성 작업)
    - 한 분기 트리: "항상 적용? → CLAUDE.md / 작업 종류 정의? → Skill / 분리된 머리 필요? → Subagent / 일회성 위임? → Task"
    - **3장 Subagent 논쟁의 회수**: Anthropic 권장 vs Shankar "brittle" 비판이 이 4지선다 안 어디에 위치하는지 명시. 독자가 자기 팀 워크플로 X를 4지선다에 매핑
    - Skill vs Subagent: WHAT vs WHO 분기 (W24) — 이 분기가 4지선다의 한 부분
    - 4지선다 결정 워크시트: 독자가 자기 팀의 워크플로 5개를 4칸 중 어디에 둘지 즉석에서 적어내게
  - Plugin Marketplace로 사내 표준 배포 (W28) — 4지선다에서 결정한 자산을 어떻게 팀 전체에 자동 배포할지
  - **[신설·챕터 마지막 절] "한 시간 온보딩" 워크시트 — 톱니바퀴 모티브의 가장 강한 회수 지점**
    - 신규 입사자가 월요일 9시에 받는 1시간 절차표:
      1. (0–10분) repo clone + `.claude/` 폴더 trust → Plugin marketplace 자동 설치 (hooks/skills/commands 일괄)
      2. (10–25분) CLAUDE.md 60줄 통독 → 팀 헌법 학습
      3. (25–45분) `/work` 명령으로 첫 작업 실행, Generator-Verifier 워크플로 체험
      4. (45–60분) 발견한 첫 어색함을 CLAUDE.md에 PR로 추가 — 신입의 첫 ratchet 칸
    - "한 사람이 떠난 후에도 그의 학습이 다음 사람에게 자동으로 전수되는" — ratchet의 본질을 이 워크시트가 직접 구현. 책 전체 모티브를 챕터 마지막에 행동 가능한 형태로 회수
    - Plugin marketplace + `.claude/` trust 효용을 챕터화 — 추상이 아닌 절차로
- **독자가 끝나고 할 수 있게 되는 것:** (1) 자기 팀의 CLAUDE.md를 60줄로 다이어트하는 작업을 시작한다. (2) 첫 Skill을 작성해 팀 git에 commit한다. (3) **어떤 지식을 어디(CLAUDE.md / Skill / Subagent / 메인+Task)에 둘지 즉석에서 결정할 수 있다.** (4) 신규 입사자 한 시간 온보딩 절차표를 자기 팀 버전으로 작성해 다음 입사자에게 적용한다.

- **한국 팀 인용** — 매 챕터 최소 1회 GeekNews/velog/CIO Korea 라운드테이블 인용. 영어권 자료에 짓눌리지 않게.

## 7. 위험 요소와 대응

### 7.1 진부해지기 쉬운 곳

- **Chapter 2 어휘 정의** — "프롬프트/컨텍스트/하네스 4계층"이 자칫 IT 입문서스러워질 수 있다.
  - **대응:** 정의 표 직후에 "그러니까 너의 어제 PR 회고에서 '이건 컨텍스트 문제였다'와 '이건 하네스 문제였다'를 가른다면…" 식의 즉시 적용 다리를 둔다.
- **Chapter 5 Hook 예시** — 인터넷에 흔한 prettier hook 같은 게 또 나오면 식상.
  - **대응:** 흔한 예시는 1–2개로 제한하고, 나머지는 한국 환경 특수 시나리오(사내 보안 검사, Java/Spring CI 통합) 위주.

### 7.2 너무 어려워지기 쉬운 곳

- **Chapter 3 학술 인용** — Reflexion, Voyager, AgentForge가 한꺼번에 나오면 5–10명 팀 테크리드가 부담.
  - **대응:** 각 논문 인용 직후 "이게 너의 어제 PR 리뷰에선 어떤 모양이었는가" 한 단락 의무화. 학술 명칭은 box로 분리하고 본문은 한국어 평어체 유지.
- **Chapter 6 OpenTelemetry** — exporter·collector·signal 용어가 갑자기 쏟아지면 관측 비전공 독자 이탈.
  - **대응:** OTLP 셋업의 가장 단순 경로(Claude Code → SigNoz 셀프호스트 도커 한 줄)부터 시작, 고급 토픽은 "더 나아가고 싶다면" 박스로.

### 7.3 Claude Code 버전 변화에 가장 빨리 늙는 곳

- **Chapter 4의 7-mode 권한 시스템** — Anthropic이 권한 모델을 바꾸면 즉시 노후화
- **Chapter 5의 GHA Action 사양** — `claude-code-action` v1 → v2 등 버전 변경 빈번
- **Chapter 10의 도구 비교 매트릭스** — 6개월 내 변동 가능성 매우 높음
- **Chapter 6의 OpenTelemetry 환경변수명**

**방어 전략:**
1. **버전 명시 의무화** — "본 책은 2026년 4월 시점의 Claude Code N.M 기준이다" 박스를 모든 영향 챕터 첫머리에
2. **추상 패턴을 우선, 구체 명령어를 차선** — 예: "PreToolUse 차단 패턴" 개념을 먼저 설명, 정확한 매처 문법은 보조 박스
3. **공식 문서 링크 부록** — 모든 영향 섹션 끝에 "최신 사양은 [code.claude.com/docs/...]" 명시
4. **GitHub repo 동반** — 본문 코드 예시를 책 출간 후 1년간 유지보수 가능한 repo에 두고 본문엔 경로만 (예: `book-harness-engineering/ch5/hooks/format-hook.json`). 명령어 변경 시 repo만 갱신
