# Chapter 5. 자동화의 결정성 — Hook으로 박을 것과 CLAUDE.md로 권할 것

3장에서 우리는 두 종류의 충돌을 갈라서 보았다. 하나는 한 개발자의 노트북 안에서 네 개의 워크트리가 서로 부딪히는 충돌이었고, 다른 하나는 다섯 명 개발자의 Claude Code 인스턴스가 같은 시각에 main 브랜치로 푸시하려는 충돌이었다. 앞의 것은 워크트리·브랜치 디시플린으로 풀었다. 뒤의 것은 그날 그 자리에서 풀지 않고 미뤄 두었다. 5장이 그 빚을 갚는 자리다. "5명이 동시에 main을 향해 달려드는 경우"는 결국 PR 게이트의 문제이고, PR 게이트는 결정성 도구가 가장 무겁게 박혀야 할 지점이기 때문이다.

상상해 보자. 네 시 반쯤 어느 작은 팀의 main 브랜치 위로 다섯 개의 PR이 거의 동시에 도착한다. 두 개는 사람이 손으로 쓴 PR이고, 세 개는 각자의 Claude Code 인스턴스가 만든 PR이다. 셋 중 하나는 Slack에서 "이 버그 좀 잡아줘"라고 던진 한 줄에서 출발했고, 또 하나는 Jira 알림이 자동으로 트리거한 PR이다. 마지막 하나는 누가 시켰는지 본인도 잘 기억나지 않는 PR이다. 어디까지를 자동으로 통과시키고 어디부터 사람이 멈춰 세워야 하는가? 이 질문이 5장의 핵심 질문이다. 매번 반드시여야 하는 일과 권장사항인 일을 어떻게 구분해, 어느 만큼을 톱니로 박아 두고 어느 만큼을 권고로 풀어 둘 것인가.

## CLAUDE.md는 권고, Hook은 강제

먼저 한 줄을 머리에 박아 두자. 커뮤니티에 떠도는 말이지만, 운영 도장으로 새겨 넣을 만하다. **"CLAUDE.md is the suggestion, Hooks is the requirement."** [W14, C2-팁38] CLAUDE.md에 적힌 규칙은 시스템 프롬프트의 일부로 들어가 모델의 자율성에 호소한다. 모델은 대체로 따른다. 그러나 "대체로"라는 말이 이미 답을 말해 준다. 한 보고는 CLAUDE.md 규칙의 준수율을 80% 언저리로, hook의 준수율을 100%로 표현했다 [W14]. 8할이면 충분한 룰이 있고, 8할로는 곤란한 룰이 있다. 그 두 부류를 똑같이 CLAUDE.md에 적어 두는 순간, 자동화의 결정성은 가장 약한 쪽 평균으로 수렴한다.

그래서 결정 규칙은 단순하다. **매번 반드시여야 하는 것은 hook으로, 사람만이 가르칠 수 있는 도메인 지식은 CLAUDE.md로** [W14, W4]. Linter, formatter, 시크릿 차단, 위험 명령 차단 — 이 종류는 모델의 양심에 맡기지 않는 편이 낫다. "LLM은 linter보다 비싸다. Hook으로 결정성을 확보하면, instruction budget을 사람만이 가르칠 수 있는 것에 쓸 수 있다" [W4]. 좋은 정렬이다. 결정적인 일은 결정적인 도구에, 모호한 일은 모호함을 견딜 수 있는 자리에.

그렇다면 어떤 일이 "매번 반드시"인가? 답을 짧게 줄여 보자.

> 같은 잘못이 반복해서 일어났는가? 한 번이라도 그것이 빌드를 깼는가, 비밀을 흘렸는가, 사람을 새벽 세 시에 깨웠는가? 그렇다면 hook이다. 그렇지 않다면 CLAUDE.md에서 시작해 보고, 사고가 한 번 나면 hook으로 옮겨 박는다.

## PostToolUse 자동 포매팅 — 작은 톱니 한 칸

가장 흔하게 박히는 톱니부터 살펴보자. 모델이 코드를 편집할 때마다 자동으로 포매터를 돌려 정렬을 맞추는 hook이다. 결과 차이가 작아 보이지만, 이 한 칸이 PR 리뷰 화면을 바꾼다.

```jsonc
// .claude/settings.json — 편집 직후 자동 포매팅. 실패해도 에이전트는 계속.
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "prettier --write \"$file\" || true" }
        ]
      }
    ]
  }
}
```

여기서 가장 중요한 두 글자는 코드 자체가 아니라 끝의 `|| true`다. [C2-팁39] 포매터는 가끔 문법 깨진 임시 파일에서 실패한다. 만약 실패가 비-제로 종료 코드로 그대로 전달되면, 에이전트는 자기 편집이 거부됐다고 오해하고 같은 자리를 다시 고치기 시작한다. 한참을 헛도느라 토큰만 태운다. `|| true`는 "포매팅 실패는 에이전트의 책임이 아니다"라는 선언이다. 결정성을 박되, 그 결정성이 모델의 사고 흐름에 끼어들지는 않게 한다.

이 디시플린은 hook을 쓸 때마다 되살아나는 질문이기도 하다. **"이 hook이 실패했을 때, 에이전트가 멈춰야 하는가, 아니면 무시하고 가야 하는가?"** 무겁게 멈춰야 하는 hook(시크릿 차단, 위험 명령 차단)에는 `|| true`를 절대 붙이지 말자. 가볍게 정렬만 맞추는 hook에는 반드시 붙이자. 차이를 헷갈리는 순간 자동화는 가장 짜증 나는 모양으로 자기 다리를 걸어 넘어뜨린다.

한 가지 더. PostToolUse 자동 포매팅은 너의 팀이 hook을 처음 들이는 자리로도 좋다. 위험이 거의 없고, 효과는 PR 차이 화면에서 즉시 보인다. 새 도구를 팀에 들이는 첫 주에 가장 큰 거부감을 일으키는 것은 사실 거창한 차단이 아니라 "내 코드가 또 이상하게 정렬됐다"는 식의 사소한 짜증이다. 그 짜증을 첫 주에 한 번에 없애 두면, 그다음 hook들에 대한 팀의 인내심이 한참 늘어난다.

## PR 게이트로서 GHA + Claude Code Action

PostToolUse가 에이전트의 손가락 끝을 잡는다면, PR 게이트는 에이전트의 결과물 전체를 잡는 자리다. 둘은 같은 hook 사고방식의 적용 거리만 다른 형태다. Anthropic이 공식으로 내놓은 `claude-code-action`과 `claude-code-security-review`가 이쪽의 표준 출발점이다. [W31] PR이 열리거나 push될 때 변경 diff를 읽고, prompt injection·SQL injection·XSS·auth bypass 같은 OWASP Top 10 항목을 의미 단위로 검사한다. 단순 정규식 매칭이 아니다. 의미를 본다.

```yaml
# .github/workflows/claude-review.yml — PR 열림/푸시 시 의미 단위 보안 리뷰.
name: claude-review
on:
  pull_request:
    types: [opened, synchronize]
permissions:
  contents: read
  pull-requests: write
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - uses: anthropics/claude-code-security-review@v1
        with:
          api-key: ${{ secrets.ANTHROPIC_API_KEY }}
          comment-mode: inline
```

비용 감각도 한번 잡아 두자. 보고된 수치는 **PR당 약 $0.04, 하루 20 PR이면 월 $24** 수준이다. [W31] 개발자 한 시간 인건비에 한참 못 미친다. 한 명이 한 시간 동안 손으로 못 잡을 보안 결함 한 건만 잡아도, 한 해 분 비용을 한 번에 회수한다는 계산이 선다. "Claude로 PR 리뷰를 자동화하면 비싸지지 않을까?"라는 질문은 자주 듣는데, 적어도 보안 리뷰 단계에서는 그 걱정을 한 번 접어 두어도 괜찮다.

타이밍은 두 가지를 고를 수 있다. PR 오픈 1회만 보거나, 매 push마다 보거나. [W31] 후자를 고르면 Claude가 "이 부분 이렇게 고치자"라고 제안한 코멘트를 본인이 다음 push에서 자동 resolve해 주는 모양도 만들 수 있다. 단, 매 push 모드는 비용이 비례해 늘어난다. 활동량이 많은 PR에서는 5~10회 리뷰가 도는 것도 흔하다. 한국 사정으로 다시 환산하면 PR 하나에 50원, 매 push 5회면 250원 수준이다. 큰 돈은 아니지만, 비용은 늘 보이지 않는 자리에서 누적된다 — 6장에서 다시 만나게 될 주제다.

폐쇄망 환경이라면 어떻게 할까? 한국의 많은 금융·공공 팀이 GitHub 클라우드 대신 사내 GitLab이나 사내 GHA Runner를 쓴다. 그때도 본질은 같다. PR 이벤트에 webhook을 걸고, 사내망 안에서 Claude Code SDK를 돌려 결과를 코멘트로 붙이는 한 단계만 추가하면 된다.

```yaml
# .gitlab-ci.yml — 사내 GitLab Runner에서 PR(=MR) diff를 사내 게이트웨이 거쳐 검사.
stages: [review]
claude-review:
  stage: review
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
  image: registry.corp.local/ci/claude-code:latest
  script:
    - git fetch origin "$CI_MERGE_REQUEST_TARGET_BRANCH_NAME"
    - git diff "origin/$CI_MERGE_REQUEST_TARGET_BRANCH_NAME"...HEAD > /tmp/diff.patch
    - claude --print --no-tools \
        --system-file ./.claude/security-review.md \
        < /tmp/diff.patch > /tmp/review.md
    - ./scripts/post-mr-comment.sh /tmp/review.md
```

이미지 레지스트리 주소는 사내 망 안의 것이고, API 호출 역시 사내 게이트웨이를 거친다. 핵심은 한 줄 — "PR 이벤트에 결정성 검사 한 칸을 더하는 것"이다. 도구 이름은 GitLab이든 Bitbucket이든 사내 Bamboo든 바뀌어도 좋다.

## Block-at-submit — 일하는 도중에는 막지 말자

Hook이 좋다고 모든 자리에 박는 게 능사일까? 여기서 한 가지 반대 의견을 들어 보자. Shrivu Shankar는 자기 운영 경험을 정리하면서 이렇게 잘라 말했다. **"plan 도중에 막으면 에이전트가 혼란하고 좌절한다. 끝까지 일하게 하고, commit 단계에서 결과만 검사하라."** [W30]

찜찜한 말처럼 들릴 수 있다. 차단할 수 있을 때 차단하는 게 안전하지 않은가? 그러나 실제로 작업해 보면 알 수 있다. 모델이 plan 도중에 한 번 차단당하면, 차단의 의미를 자기 사고에 통합하느라 한참을 헤맨다. 같은 자리에서 다른 방식을 시도하다 다시 막히고, 그러다 작업의 큰 그림을 잃는다. 정말 위험한 명령(rm -rf, 시크릿 외부 송출, 운영 DB 직접 조작)이라면 plan 도중에라도 PreToolUse로 즉시 막아야 한다. 4장에서 박은 톱니가 그 자리다. 그러나 그것이 아니라면 — 즉, "이 디자인은 우리 컨벤션과 다르다"든가 "이 변수명은 우리 프로젝트 스타일이 아니다" 같은 일이라면 — 끝까지 일하게 두고 commit 직전에서 한 번에 보는 편이 낫다.

이 전략에 이름을 붙이자: **block-at-submit**. 차단 시점을 도구 호출 순간이 아니라 결과 제출 직전으로 미루는 것이다. 시점 하나만 옮겼을 뿐인데, 자동화의 결정성과 에이전트의 자율성이 동시에 살아난다.

## Pre-commit 게이트 패턴 — `/tmp` 파일 트릭

block-at-submit을 가장 우아하게 구현하는 패턴이 하나 있다. Shankar가 직접 운영하면서 정리한 것인데, **테스트가 통과해야만 생기는 임시 파일을 commit의 입장권으로 쓰는** 방법이다. [W30]

```jsonc
// .claude/settings.json — git commit은 /tmp/agent-pre-commit-pass 파일이 있을 때만.
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash(git commit*)",
        "hooks": [
          {
            "type": "command",
            "command": "test -f /tmp/agent-pre-commit-pass || { echo 'run tests first'; exit 2; }"
          }
        ]
      },
      {
        "matcher": "Bash(npm test*|pytest*|./gradlew test*)",
        "hooks": [
          { "type": "command", "command": "rm -f /tmp/agent-pre-commit-pass" }
        ]
      }
    ]
  }
}
```

```bash
# scripts/test-and-pass.sh — 테스트 통과 시에만 입장권 파일을 남긴다.
set -e
npm test
touch /tmp/agent-pre-commit-pass
echo "tests passed; commit gate open"
```

설정의 운영 방식은 이렇다. 테스트 명령(`npm test`, `pytest`, `./gradlew test` 등)이 시작될 때 입장권 파일을 일단 지운다. 테스트가 끝나고 통과하면 별도 스크립트가 입장권 파일을 만든다. 에이전트가 `git commit`을 시도하면 PreToolUse hook이 입장권 파일이 있는지 확인하고, 없으면 차단한다. 자연스럽게 "테스트를 통과해야 commit 가능"이라는 룰이 박힌다. CLAUDE.md에 "반드시 테스트를 먼저 실행하세요"라고 적어 두는 것과는 차원이 다른 결정성이다.

이 패턴을 한국 환경에 맞게 변주해 보자. Java/Spring 팀이라면 `./gradlew check` 통과를 입장권으로 삼고, 프론트엔드 팀이라면 `pnpm test && pnpm lint`를, 데이터 팀이라면 dbt 테스트나 SQL linter 통과를 입장권으로 삼을 수 있다. 핵심은 너의 팀이 "이걸 안 하고는 절대 commit하면 안 된다"고 합의한 단 한 가지 검사를 골라, 그 검사의 통과만이 commit의 입장권이 되도록 만드는 것이다.

기억해두자. 이 트릭의 가장 큰 미덕은 모델에게 "왜 막혔는지"를 설명하기 쉽다는 점이다. 차단 메시지에 `run tests first`라는 한 줄만 박아 두면, 모델은 자연스럽게 테스트를 먼저 실행하는 흐름으로 돌아온다. 차단의 결정성과 모델의 자율성이 같이 살아 있는 모양이다.

## PR-from-anywhere — Slack에서 푸시 버튼 하나로 PR까지

자동화가 이쯤 되면 한 걸음 더 나갈 수 있다. PR을 만드는 트리거를 IDE 안에서가 아니라, 팀의 의사소통이 일어나는 자리에서 거는 패턴이다. **PR-from-anywhere**. [W30] Slack의 한 마디, Jira의 새 티켓, CloudWatch의 알림 — 이 신호들이 Claude Code SDK를 호출해 작업을 시작하고, 끝에 가서는 테스트가 통과한 PR을 만들어 다시 Slack으로 답장한다.

```python
# slack_to_pr.py — Slack 슬래시 명령 한 줄을 받아 Claude Code로 작업 → PR 반환.
import os, subprocess
from slack_bolt import App

app = App(token=os.environ["SLACK_BOT_TOKEN"])

@app.command("/fixit")
def handle_fix(ack, command, say):
    ack()
    issue = command["text"]  # 예: "로그인 폼이 빈 비번으로도 통과해요 — repo: web/login"
    say(f"받았다. 작업 시작: {issue}")
    out = subprocess.run(
        ["claude", "--print",
         "--allowed-tools", "Read,Edit,Bash(npm test*),Bash(git*)",
         "--system-file", ".claude/agents/bug-fixer.md",
         "--input", issue],
        capture_output=True, text=True, timeout=1800,
    )
    pr_url = out.stdout.strip().splitlines()[-1]
    say(f"PR 생성: {pr_url}")
```

여기서 두 가지 디시플린이 살아 있어야 한다. 첫째, **`--allowed-tools`로 도구를 명시적으로 좁힌다.** Slack에서 트리거된다는 것은 작업 의도가 짧은 한 문장이라는 뜻이고, 짧은 문장은 오해의 여지가 크다. 도구를 좁혀야 오해가 큰 사고로 번지지 않는다. 둘째, **이 PR은 자동 머지가 아니다.** 자동 트리거된 PR은 바로 다음 절에서 다룰 PR 정책의 가장 까다로운 사례 — "인간 prompter가 사실상 없는 PR" — 으로 분류해 사람 눈을 두 번 거친다.

PR-from-anywhere의 부수 효과 중 좋은 것 하나를 적어 두자. 회사 전체 GHA 로그를 정기적으로 분석해 "에이전트가 자주 막히는 지점"을 추출하고, 그것을 다시 CLAUDE.md나 사내 CLI 문서에 반영하는 루프다. [W30] 한 인용을 그대로 옮기면 이렇다 — "Query logs since 5 days ago; identify what agents got stuck on; fix issues; submit PR." 자동화가 자기 자신의 부족한 부분을 학습 데이터로 삼는 모양이다. 너의 팀이 한 달쯤 자동화를 굴려 본 후에, 한 번 시도해 볼 만한 다음 단계다.

## 논쟁점 — Hook 적극 vs Block-at-submit만

이쯤에서 한 번 멈추고 솔직해지자. Hook을 적극 박는 진영과 block-at-submit만으로 충분하다는 진영 사이에는 진짜로 의견 차가 있다. 양쪽 다 합리적이다. [W14, W30]

**Hook 적극 진영**의 주장은 단순하다. 100%와 80%의 차이는 작아 보이지만, 사고는 그 20%에서 난다. 시크릿 차단, 위험 명령 차단, 자동 포매팅처럼 결정적이어야 마땅한 일은 결정적인 도구로 박아야 한다. CLAUDE.md에 적어 두고 모델의 양심에 기대는 자체가 거버넌스의 약점이다.

**Block-at-submit 진영**의 주장도 만만치 않다. 과한 hook은 에이전트의 자기 교정 능력 자체를 망친다. 모델은 자기 작업 결과를 테스트하고 잘못을 발견해 고치는 능력을 가지고 있는데, 그 능력은 "끝까지 일해 본 다음에" 발휘된다. 일하는 도중에 자꾸 끊으면 모델은 매번 새 시작점에서 헤매고, 결과적으로 자율성과 결정성을 둘 다 잃는다.

내 결론은 양쪽을 가르는 한 줄이다. **결정성을 위반했을 때 사고가 나는 것은 hook으로, 결정성을 권하는 정도면 충분한 것은 block-at-submit으로.** 시크릿이 외부로 새 나가면 그날의 사고이고, 변수명이 컨벤션과 다르면 그날의 짜증이다. 같은 톱니로 잡을 수 없다. 한쪽은 PreToolUse 차단이 답이고, 다른 한쪽은 PR 게이트의 코멘트가 답이다.

자기 팀이 어느 쪽으로 더 기울어 있는지는 한 가지 질문이면 알 수 있다 — "지난 분기, 우리 팀에서 사고에 가까웠던 사건이 모델의 plan 단계에서 일어났는가, 아니면 commit 직전에 일어났는가?" 답이 앞쪽이면 hook을 더 박아야 하고, 뒤쪽이면 PR 게이트를 더 다듬어야 한다.

## PR 리뷰 정책 재설계 — AI가 코드를 쓰는 시대의 거버넌스

지금까지가 도구의 결정성이었다면, 여기부터는 사람의 결정성이다. AI가 코드를 쓰는 시대의 PR 리뷰는 예전의 PR 리뷰와 같은 모양으로 굴러갈 수 없다. 같은 도장이지만, 새로 새기는 편이 낫다. 짧지만 무거운 다섯 가지 결정을 정리해 두자.

**첫째, "인간 prompter 없는 PR은 2명 승인."** Shrivu Shankar의 정책이다. [W30] 인간 prompter가 있는 PR — 즉 사람이 자기 책상에서 Claude에게 "이거 고쳐"라고 시켜서 만든 PR — 은 그 사람이 자기 의도를 가지고 결과를 1차 검토한 셈이라 1인 승인으로 충분하다. 그러나 Slack 슬래시 명령이나 Jira 알림에서 자동 트리거된 PR은 사실상 인간 prompter가 없다. 의도를 가진 사람이 결과를 한 번도 보지 않은 상태로 PR이 도착한다. 이때는 두 명의 다른 시각이 한 PR을 같이 봐야 사고가 줄어든다. "왜 2명이고 왜 prompter 유무로 갈리는가?" — 답은 의도와 검증의 거리에 있다. prompter가 곧 1차 검증자다. 그가 없는 PR은 1차 검증이 비어 있는 PR이다.

**둘째, AI 흔적의 표기 정책.** 커밋 메시지나 PR 본문에 "Generated with Claude Code"를 명시할 것인가, 숨길 것인가. 여기엔 두 힘이 부딪힌다. 한쪽은 투명성이다 — 누가 무엇을 만들었는지 정직하게 적어야 검토자의 시선이 거기 맞춰진다. 다른 한쪽은 거부감이다. 한 학계 보고에 따르면 일부 응답자는 "PR에 AI 코드처럼 보이면 평가에 부정적"이라는 우려를 표한다. [P10] 너의 팀에서 이 거부감이 실재한다면, 명시적 표기는 정직한 동시에 위험하다. 그렇다면 답은 한 가지다 — **조직 리더가 "AI 사용은 받아들여진다"는 문화를 먼저 만든다.** 그 문화 위에서야 표기는 부담이 아니라 정직이 된다. 거부감을 그대로 두고 표기만 강제하면, 사람들은 흔적을 지우는 데 시간을 쓴다.

**셋째, 자동 머지 허용 범위.** 세 단계로 나누는 편이 낫다. (1) **룰셋만 통과해도 자동 머지** — 린터·포매터·타입 검사·테스트 같은 결정적 통과만으로 머지가 가능한 영역. 문서 오타 수정, 의존성 패치 버전 업, 자동 생성된 i18n 추가 같은 것들이다. (2) **사람 1명 승인 후 머지** — 일반적인 기능 변경, 일반적인 버그 픽스. (3) **절대 자동 머지 금지** — 인증·인가 로직, 결제·계좌, 데이터 마이그레이션, 외부에 보이는 API 시그니처 변경. 이 세 단계의 경계를 어디에 그을지가 너의 팀의 거버넌스 핵심이다. "자동 머지 vs 항상 사람 게이트"라는 흑백 논쟁은 [W29, W33] 양쪽 모두 부분만 옳다. 변경 종류별로 다른 답을 줘야 한다.

**넷째, intent 공유 템플릿.** AI slop PR이 가장 위험한 자리는 "왜 이 변경이 필요한가?"가 비어 있을 때다. [W15] 사람의 PR도 비어 있을 때가 있지만, 사람은 적어도 머릿속에서는 알고 있다. AI가 만든 PR은 작성자조차 잠시 후면 자기 의도를 잊는다. 그래서 PR 템플릿에 한 칸을 추가하자.

```markdown
<!-- .github/PULL_REQUEST_TEMPLATE.md — 의도 명시를 강제하는 한 칸 -->
## 내가 시킨 것 (Intent)
- 한 줄로: ____
- 트리거: [수동 / Slack / Jira / 알림 / 기타]
- 비-범위: 이 변경이 의도적으로 손대지 않은 영역 ____

## 변경 요약
- ____

## 검증
- [ ] 단위 테스트 통과
- [ ] 인접 영역 회귀 테스트
- [ ] (해당 시) AI 도구 사용 명시
```

특히 "비-범위" 한 줄이 결정적이다. AI는 시키지 않은 것까지 손대는 경향이 있고, 그것이 사고로 번진다. "이 변경은 인증 로직과 무관하다"라는 한 줄이 PR에 박혀 있으면, 리뷰어는 인증 부분의 변경을 발견했을 때 즉시 반려할 명분을 가진다.

**다섯째, 96/48% 검증 격차에 대한 보정.** 한 보고에 따르면, 96%의 개발자가 AI 코드를 완전히 신뢰하지 않는다고 답하면서, 실제로 검증하는 비율은 48%에 그친다. [W33] 신뢰는 안 한다고 말하면서 검증은 안 하는 격차다. 이 격차는 정책으로 보정해야 한다. 가장 단순한 보정은 PR 본문에 검증 체크리스트를 강제하고, AI 흔적이 있는 PR은 체크리스트의 항목을 한 칸 더 두는 것이다. "AI 도구가 만든 변경이라면, 인접 영역 회귀 테스트를 명시해 주세요." 한 줄이지만, 무겁다.

이 다섯 가지를 한 장의 정책 문서로 정리해 두자. 너의 팀이 책을 덮은 다음 날 작성할 수 있을 만큼 짧게. **9장에서는 Anthropic·Toss·IMWEB·Shankar 네 팀의 PR 정책을 매트릭스로 비교한다. 그때 너의 팀의 다섯 가지 결정도 같은 매트릭스 안에 넣어 비교해 보자.** 너의 정책이 어느 팀에 가까운지, 그리고 어디서 너의 팀만의 색을 낼 것인지가 거기서 보일 것이다.

## 자동화 워크시트 — 너의 팀의 톱니에 추가할 한 칸

이 장의 작업을 너의 팀에 옮기려면 한 장의 표가 있으면 충분하다. 빈 표를 그려 놓고, 팀이 모인 자리에서 한 줄씩 채워 보자.

| 결정성을 어디에 박을까 | 후보 | 도구 | 실패 시 동작 |
|---|---|---|---|
| 매번 반드시 — 사고로 번진다 | 시크릿 외부 송출 차단 | PreToolUse hook | 즉시 차단(exit 2) |
| 매번 반드시 — 정렬 문제 | 포매팅 | PostToolUse hook | 무시(`\|\| true`) |
| commit에서 한 번 — 테스트 통과 | 단위 테스트 통과 | Pre-commit gate (`/tmp` 트릭) | 차단 메시지로 안내 |
| PR에서 한 번 — 의미 검사 | OWASP Top 10 의미 검사 | GHA + Claude Code Action | PR 코멘트 |
| 권고면 충분 — 도메인 지식 | 우리 팀 컨벤션 | CLAUDE.md | 모델 자율 |

표 자체가 결론이다. 결정성을 박을 자리는 네 개 — PreToolUse, PostToolUse, pre-commit, PR 게이트 — 그리고 권고로 풀 자리는 하나, CLAUDE.md다. 너의 팀이 들이고 싶은 룰 하나를 가져와 이 표의 어느 줄에 들어가는지 답하면, 그 룰의 구현 방식이 거의 자동으로 결정된다.

## 마무리

이 장에서 너는 다섯 칸을 손에 쥐었다.

(1) **CLAUDE.md는 권고, hook은 강제.** 두 룰을 같은 자리에 적지 말자.
(2) **PostToolUse 자동 포매팅의 `|| true` 디시플린.** 결정성과 자율성이 충돌하지 않게.
(3) **PR 게이트로서 GHA + Claude Code Action.** PR당 약 50원, 월 3만 원 안팎의 작은 청구서로 큰 안전을 산다.
(4) **Block-at-submit과 pre-commit `/tmp` 트릭.** 일하는 도중에는 막지 말고, 결과 직전에 한 번에 보자.
(5) **PR 리뷰 정책 한 장.** 2명 승인의 경계, AI 흔적 표기, 자동 머지 3단계, intent 템플릿, 96/48 보정.

여기까지의 자동화는 빠른 만큼 비싸진다. 빠른 자동화가 쌓이면 그다음 달 청구서가 무거워진다. **6장은 그 청구서를 펼쳐 보는 자리다.** 보이지 않으면 통제할 수 없으니, 우리가 이번 달 Claude Code에 얼마를 쓰고 있는지, 그 돈이 어떤 작업으로 흘러가는지 한 눈에 답할 수 있는 관측 체계를 갖춰 보자. 한국의 5인 팀 시나리오 세 가지를 표로 그려 보면, 이 장의 자동화가 너의 통장에 어떤 모양으로 도착할지가 또렷이 보일 것이다.
