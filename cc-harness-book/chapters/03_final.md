# Chapter 3. 멀티 에이전트 오케스트레이션 — 한 명의 천재 vs 다섯의 직공

상상해 보자. 너의 팀이 어제 PR 리뷰에서 두 종류의 사고를 동시에 겪었다. 하나는 한 시니어 개발자의 화면에서 일어났다. 그는 워크트리를 네 개 띄워 놓고 동시에 작업했는데, 세 번째 워크트리에서 변경한 마이그레이션 스크립트와 첫 번째 워크트리의 모델 클래스가 묘하게 어긋난 채 main에 합쳐졌다. 본인 머릿속에서도 헷갈렸으니, 사람이 코드를 보고 잡아내기는 더 어려웠다. 또 하나는 다른 자리에서 일어났다. 다섯 명이 각자 Claude Code 인스턴스를 띄우고 거의 같은 시각에 main으로 PR을 던졌다. 한 사람의 PR이 머지된 직후 다음 PR이 충돌난 채로 자동 머지를 시도했고, 새벽에 빌드가 망가졌다. 두 사고는 표면적으로 비슷해 보인다. "에이전트가 코드를 망쳤다"라는 한 줄로 요약된다. 하지만 원인은 완전히 다르다. 다른 문제이고, 다른 해법이 필요하다.

이 둘을 처음부터 분리해서 부르자. **워크트리 격리**는 한 사람의 병렬이다. 한 명의 개발자가 같은 저장소에서 여러 작업을 동시에 굴리려 할 때 생기는 문제이며, 해법은 워크트리·브랜치 디시플린이다. 반면 **멀티 에이전트 오케스트레이션**은 한 작업의 협업이다. 같은 결과물을 만들기 위해 여러 에이전트가 역할을 나눠 일하는 구조이고, 해법은 Generator–Verifier 분리, 컨텍스트 격리, 메모리 설계다. 그리고 5–10명 팀이 main에 동시 push하는 세 번째 종류의 충돌이 있다. 이것은 워크트리도 멀티 에이전트도 아닌, **PR 게이트와 conflict 정책**의 영역이다. 이 책은 그 이야기를 5장에서 본격적으로 다룬다. 지금 장에서는 앞 두 가지에 집중하자. 한 사람이 4개의 탭을 띄울 때, 그리고 한 작업을 5명의 직공이 나눠 맡을 때, 무엇이 어떻게 다르고 왜 후자가 자주 더 안정적인지를.

이 장의 핵심 질문은 단 하나다. **왜 가장 똑똑한 단일 모델 하나보다, 역할이 분리된 평범한 모델 다섯이 더 안정적으로 일하는가?** 답을 미리 흘리자면 이렇다. 똑똑한 한 모델은 자기 결과를 객관적으로 못 본다. 만든 자에게 검증을 맡기면 둘 다 망가진다. 학계와 실무가 같은 결론으로 수렴하고 있다. 자, 함께 살펴보자.

## 만든 자에게 검증을 맡기지 말자 — Generator-Verifier 패턴

같은 사람이 코드를 작성하고 자기 코드를 리뷰하는 회사가 있다고 해보자. 솔직히 말하면, 그건 회사가 아니라 1인 작업장이다. 우리가 코드 리뷰를 굳이 다른 사람에게 맡기는 이유가 무엇인가? 코드를 쓰는 동안의 머릿속과, 그것을 검증하는 머릿속은 다른 종류의 주의력을 요구하기 때문이다. 작성자는 자기가 의도한 것을 본다. 리뷰어는 자기가 보지 못한 것을 본다. 두 시선은 본질적으로 다르다.

LLM 에이전트도 같은 한계를 가진다. 같은 에이전트에게 코드를 만들게 하고, 그 자리에서 "이거 맞아?"라고 물으면, 모델은 자기가 방금 쓴 출력을 정당화하는 방향으로 답한다. 인간이 자기 변명을 잘 만드는 것과 똑같다. 그래서 학계와 실무는 한목소리로 권한다. **만들기와 검증하기를 다른 에이전트에게 맡기자.** 이른바 **Generator–Verifier 패턴**이다 [W3, W22, P4].

수치를 보면 차이가 어이없을 정도다. AgentForge 연구는 Planner / Coder / Tester / Debugger / Critic 다섯 역할을 분리하고 듀얼 메모리를 도입한 멀티 에이전트 구성을 SWE-bench Lite에서 측정했다. 결과는 40.0% 해결률. 같은 모델로 단일 에이전트만 굴렸을 때보다 +26~28pt 향상이다 [P4]. 다른 도메인 연구는 더 극적이다. Multi-Agent LLM Orchestration 논문은 인시던트 대응 추천 품질을 비교했는데, 단일 에이전트의 actionable 추천률은 1.7%였다. 멀티 에이전트로 분리한 경우는 100%였다 [W22]. 80배·140배 향상이라는 수치를 보면 처음엔 통계 작난이 아닌가 싶지만, 한 가지를 생각하면 납득이 간다. 단일 에이전트는 절반 이상의 답변에서 "지금 당장 무엇을 할지"를 제시하지 못한 채 추상적인 분석으로 빠진다. 검증하는 두 번째 시선이 없으니, 첫 시선이 만족하는 어디쯤에서 멈추는 것이다. 끔찍한 일이다.

> **[더 읽기] Reflexion — Verbal Reinforcement Learning**
>
> Generator–Verifier의 학술적 원형 중 하나는 Reflexion이다 [P5]. 에이전트가 작업을 실패하면 그래디언트를 갱신하지 않고도 "왜 실패했는지"를 자연어로 자기에게 적어둔다. 이 회고가 episodic memory에 저장되고, 다음 시도에 컨디셔닝된다. 그래디언트 없는 강화학습이라는 의미에서 "verbal RL"이라 부른다. 그 이전 세대인 ReAct(Reason–Act–Observe loop), 이후의 Self-Refine, CRITIC, Chain-of-Verification까지 모두 같은 generate→critique→refine 루프의 변형이다 [P7]. Voyager는 Minecraft에서 자동 커리큘럼과 iterative prompting, 그리고 학습된 절차를 모은 **skill library**로 평생학습을 시연했다 [P6]. 오늘날 우리가 말하는 "스킬"이라는 단어의 학술적 뿌리가 여기에 있다.

학술 인용이 멀게 느껴지는가? 너의 어제 PR 리뷰를 떠올려보자. 같은 사람이 PR을 올리고, 같은 사람이 머지 버튼을 누른 적이 한 번이라도 있다면, 이미 너는 단일 에이전트 모드의 위험을 본능적으로 알고 있는 것이다. 우리는 사람의 일에는 그 분리를 강하게 요구하면서, 에이전트에게는 너무 자주 잊는다. 만든 에이전트와 검증하는 에이전트를 분리해 보자. 이 한 줄짜리 결정이, 멀티 에이전트 오케스트레이션의 첫 번째 톱니다.

조금 더 깊이 들어가보자. 분리한다고 해서 무조건 좋아지는가? 아니다. **검증자에게 무엇을 보게 할지**를 정하지 못하면, 두 번째 시선은 그저 첫 시선의 메아리가 된다. AgentForge가 +26pt를 만든 핵심은 다섯 역할이 단지 "다섯 개의 의견"이 아니라 **각자 다른 출력 형식·다른 평가 기준**을 강제했다는 점이다 [P4]. Tester는 실행 결과만 본다. Critic은 코드 가독성과 책임 분리만 본다. Debugger는 실패한 케이스 재현만 한다. 같은 코드를 보면서도 보는 각도가 다르다. 그래서 Verifier 프롬프트를 짤 때 한 가지를 의식해야 한다. **"이 검증자는 어떤 종류의 결함을 가장 잘 잡는가"를 한 줄로 적을 수 있어야 한다.** 그 한 줄을 못 적겠다면, 그 검증자는 사실 Generator의 거울일 가능성이 높다.

## 두 가지 실패 모드 — Initializer + Coder의 이유

장시간 자율 작업을 시켜본 적이 있는가? 한 번 돌려놓으면 두 가지 종류의 사고가 반복된다. Anthropic 엔지니어링 블로그가 정리한 표현을 그대로 빌리자면, **Over-ambition**과 **Premature completion**이다 [W2].

Over-ambition은 첫 번째 인스턴스가 너무 많은 것을 한 번에 하려고 할 때 일어난다. "이 기능을 만들어줘"라고 말하면 에이전트는 친절하게도 데이터 모델 정의·마이그레이션·API·프론트엔드·테스트·문서화를 동시에 시작한다. 결과는 뻔하다. 컨텍스트 윈도가 다 차기 전에 목표는 절반쯤 완성된 어딘가에서 흩어지고, 어느 파일도 머지 가능한 상태가 아니다. 찜찜하기 짝이 없다.

Premature completion은 그 다음 인스턴스가 일으킨다. 두 번째 호출에서 에이전트는 진행 파일과 git 로그를 보고 "아, 어느 정도 끝났네"라고 판단해 버린다. 사람이라면 미완성이라는 걸 한눈에 봤을 텐데, 모델은 "feature list 5개 중 3개에 체크가 있으니 마지막 두 개만 다듬으면 끝"으로 오판한다. 결과적으로 완성되지 않은 채 "완료" 보고가 올라온다. 이 또한 끔찍한 일이다.

해법은 의외로 깔끔하다. **두 가지 다른 프롬프트로 두 가지 다른 역할을 만들자.**

- **Initializer Agent (1회만 실행)** — `init.sh`, `claude-progress.txt`, JSON feature list, 첫 git commit. "프로젝트의 처음을 정리하는 사람"이다.
- **Coding Agent (반복 실행)** — 진행 파일과 git 로그를 읽고, feature list에서 한 개를 골라, 깨끗한 상태로 commit한다. "정해진 한 칸을 채우는 사람"이다.

Anthropic은 이 패턴이 어디서 영감을 얻었는지를 솔직하게 말한다. "유능한 소프트웨어 엔지니어들이 매일 무엇을 하는지를 본 데서 나왔다" [W2]. 그들은 한 번에 모든 걸 하지 않는다. 시작 단계에서 골격을 세우고, 본 작업에서는 하루 한두 개의 작은 commit을 깨끗하게 남긴다. 우리가 좋은 엔지니어에게 자연스럽게 기대하는 그 패턴을 에이전트에게도 그대로 옮긴 것이다.

테스트와 관련해 한 가지 인상적인 문구가 있다. Anthropic이 Coding Agent 프롬프트에 박아 넣은 문장이다.

> "It is unacceptable to remove or edit tests because this could lead to missing or buggy functionality." [W2]

테스트를 삭제하거나 수정하는 것은 "받아들일 수 없다"고 명시적으로 말한다. 모델은 평소에 "테스트를 고치자"는 유혹을 자주 받는다. 빨간 줄이 떠 있으면 그것을 없애고 싶어 한다. 그렇다면 어떻게 해야 할까? "테스트는 진실의 원천이고, 너는 코드를 고쳐서 통과시켜야 한다"는 메시지를 시스템 프롬프트에 못 박는 편이 낫다. 이 한 줄이 자율 루프 한 시간을 살린다.

여기서 한 가지 의문이 생긴다. 그렇다면 우리가 사람에게 시키는 거의 모든 패턴을 에이전트에게도 옮길 수 있다는 말인가? 어느 정도는 그렇다. 다만 "옮길 수 있다"와 "옮기는 비용이 합리적이다"는 다르다. 비용 이야기는 곧 다시 하자.

## 한 사람의 병렬 — git worktree로 4탭을 굴리다

이번엔 시선을 옮겨보자. 멀티 에이전트가 아니라, **한 사람이 여러 인스턴스를 동시에 굴리는 시나리오**다. 같은 저장소에서 여러 갈래의 작업을 동시에 진행하고 싶을 때, 우리는 git worktree를 쓴다. Claude Code도 이 패턴을 일급 시민으로 지원한다. `claude --worktree feature-auth` 한 줄이면 격리된 디렉토리, 전용 브랜치, 독립된 파일 시스템이 한 번에 마련된다 [C2-팁15, W26].

한국 개발자 한 명이 GeekNews에 공유한 오전 2시간 사례를 보자 [C1].

> "9시에 버그 보고서를 받았다. 4개 탭을 동시에 띄웠다. 한 탭은 버그를 수정하고, 한 탭은 어제 밤에 올라온 PR을 리뷰하고, 한 탭은 changelog를 정리하고, 한 탭은 관련 라이브러리의 배경 지식을 조사했다. 9시 20분에는 4개 작업이 모두 진행 중이었고, 11시에는 PR이 자동 생성되어 있었다. 11시 30분에 사람이 최종 리뷰를 했다."

오전 두 시간 만에 네 가지 일이 끝난다. 처음 들으면 비현실적으로 들리지만, 잘 보면 어떤 일도 새롭지 않다. 4명의 사람이 각자 한 일을 했을 뿐이다. 다만 그 4명이 한 명의 머리에서 동시에 굴러갔다는 점이 다르다. 이게 바로 **Fork-Join 병렬**의 전형이다 — 작업을 분기시키고, 끝나면 다시 사람이 모아 머지한다.

다만 잊지 말자. 이 패턴은 **한 사람의 병렬**이지, **한 작업의 협업**이 아니다. 4개 탭은 서로 협력하지 않는다. 각자 다른 작업을 한다. 따라서 이 패턴이 풀어주는 문제는 "한 사람이 한 시점에 하나의 일밖에 못한다"는 시간 제약이지, "한 작업의 품질을 높이는 분업"이 아니다. 둘을 헷갈리면 무엇이 어디에 좋은지 잘못 배운다.

그리고 이 한 사람의 병렬에는 한 사람의 함정이 있다. 4개의 워크트리가 같은 파일을 만지면 어떻게 될까? 각자 따로 commit하다가 머지 시점에 충돌이 폭발한다. 그래서 한 사람이 4탭을 굴릴 때 가장 중요한 디시플린은 단순하다. **각 워크트리의 작업 영역을 미리 그어두자.** 어제 작업한 GeekNews 케이스도 자세히 보면 "버그 수정 / PR 리뷰 / changelog / 배경 조사"로 영역이 깔끔하게 갈려 있다. 같은 모듈을 두 탭이 동시에 만지지 않게 한 것이다.

서브에이전트에도 같은 격리를 줄 수 있다. 프론트매터에 `isolation: worktree`를 박으면 그 서브에이전트는 자기만의 worktree에서 일한다 [C2-팁35]. 큰 리팩토링이나 마이그레이션처럼 다른 작업과 충돌할 가능성이 큰 일을 시킬 때 유용하다. 다만 worktree는 디스크 공간과 IDE 인덱싱을 추가로 먹는다. 3~5개가 적정선이라는 권고가 일반적이다 [C2-팁15]. 10개 탭을 띄워 놓고 만족스러워하던 순간이 있다면, 한 번쯤 의심해 보자. 정말로 10개가 동시에 진행되고 있는가, 아니면 8개가 너의 머릿속에서 잠자고 있는가.

## 한 작업의 협업 — Agent Teams와 그 청구서

워크트리가 한 사람의 병렬이라면, **Agent Teams는 한 작업의 협업**이다. Claude Code의 실험적 기능인 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`을 켜면, 팀 리더가 팀원에게 작업을 분배한다 [C2-팁20]. 같은 파일 수정 시 충돌도 자동으로 회피한다. 처음 보면 "와, 진짜 5명 팀이 일하는 것 같네"라며 신기해할 만하다.

하지만 청구서 이야기를 해야 한다. 운영 데이터를 분석한 보고에 따르면, Agent Teams는 표준 세션의 약 **7배 토큰**을 소비한다 [W18]. 각 teammate가 자기 컨텍스트 윈도와 별도 인스턴스를 유지하기 때문이다. 한 작업을 다섯이 나눠 했으니 다섯 명분의 컨텍스트가 따로 채워지고, 거기에 리더의 조율 컨텍스트가 더해진다. 하루에 표준 세션으로 $30을 쓰던 개발자가 Agent Teams를 켜는 순간 $200을 쓸 수 있다는 의미다.

그럼 7배의 청구서는 언제 받을 만한가? 이걸 결정하는 단순한 기준이 있다. **이 작업에 한 명의 시선으로는 잡지 못할 종류의 결함이 있는가?** 만약 그렇다면 Generator–Verifier 두 시선만으로도 충분한지 먼저 따져보자. 그래도 더 필요하다면 Planner / Coder / Tester / Debugger / Critic 다섯 시선을 들이는 게 맞다. AgentForge가 보여준 +26~28pt가 그 자리에서 나온다 [P4]. 반대로 단순한 버그 수정이나 리팩토링에 다섯 시선을 들이는 건 사치다. 그럴 거면 한 시선으로 짧게 끝내고, 사람이 PR에서 두 번째 시선을 보태는 편이 낫다.

기억해두자. **워크트리 4탭은 한 사람의 시간을 늘린다. Agent Teams는 한 작업의 시선 수를 늘린다.** 둘은 다른 자원이고 다른 청구서를 만든다. 자기 팀의 어제 충돌이 어느 쪽에 속하는지부터 분류하는 것이, 도구를 잘 쓰기 위한 첫걸음이다.

물론 Agent Teams 외에도 "한 작업의 협업"을 구현하는 길은 많다. 메인 에이전트가 여러 서브에이전트를 차례로 호출하는 정적인 분업 구조도 가능하고, 메인이 자체 `Task()` 호출로 그때그때 동적 분업을 만드는 방식도 있다. 토큰 비용은 선택한 구조에 따라 천차만별이다. 정적인 두세 단계 분업이라면 1.5배 정도, 다섯 역할의 Agent Teams 풀이라면 7배에 가깝다. 처음부터 가장 풍부한 구조를 도입할 필요는 없다. **두 시선으로 시작해, 정말 부족할 때 한 시선씩 늘려가자.** 시선의 수를 줄이기보다 늘리기가 훨씬 쉽고, 줄이는 결정은 거의 항상 어떤 비용 사고를 겪고 나서 내려진다. 미리 그 사고를 피해도 된다.

## 컨텍스트 격리 — 리서치는 편집 못 하게, 플래너는 실행 못 하게

한 작업을 여러 에이전트가 협업할 때 가장 흔한 실수 하나를 짚자. 모든 에이전트에게 똑같은 권한과 똑같은 도구를 주는 것이다. "혹시 모르니 다 줘 두자"는 충동인데, 이게 시간이 지나면 빠짐없이 사고를 만든다. 리서치 에이전트가 검색 결과를 정리하다가 무심코 파일을 수정한다거나, 플래너가 "그래, 이왕 계획 세웠으니 한번 돌려보자"며 위험한 명령을 실행해 버린다거나.

12개 핵심 하네스 패턴 중 하나가 **Context-Isolated Subagents**다 [W23]. 정의는 단순하다. **리서치 에이전트는 편집할 수 없게, 플래닝 에이전트는 실행할 수 없게 도구 권한 자체를 잘라둔다.** 그렇게 하면 두 가지가 동시에 좋아진다. 첫째, 권한 사고가 원천 차단된다. 둘째, 각 에이전트의 컨텍스트가 그 역할에 필요한 것만으로 좁혀지면서 추론 품질이 올라간다. 혹시라도 리서치 에이전트가 코드 편집의 유혹을 받아 흩어지지 않는다.

이 패턴은 메인 에이전트의 컨텍스트를 깨끗하게 유지하는 효과도 가진다. 메인은 보통 사용자와 직접 대화하며 큰 결정과 합의 라인을 잡는다. 그 위에 리서치·테스트·리뷰 같은 보조 작업을 직접 떠안으면, 메인의 컨텍스트가 보조 작업의 부산물로 빠르게 더러워진다. 보조 작업은 격리된 서브에이전트로 넘기고, 메인은 결과 요약만 받는 편이 낫다. 같은 200K 컨텍스트라도, 깨끗한 200K와 더러운 200K는 추론 품질이 전혀 다르다.

여기서도 너의 어제 PR 리뷰를 떠올려보자. 시니어 한 명이 모든 일을 다 하던 시절, 우리는 그가 "하나만 부탁드립니다"가 끝없이 쌓이면서 결국 어떤 일도 깊이 못 보던 광경을 자주 봤다. 컨텍스트 오염이라는 학술적 표현을 굳이 빌리지 않아도, 우리는 본능적으로 안다. 일은 좁은 책임자에게 맡기는 편이 낫다는 것을. 에이전트도 같다.

권한 격리에는 한 가지 부수 효과가 더 있다. **사고가 났을 때 추적이 쉬워진다.** 모든 에이전트가 모든 도구를 가진 환경에서 한 번 사고가 나면, 누가 어떤 의도로 무엇을 했는지를 사후에 재구성하기가 끔찍할 정도로 어렵다. 반면 리서치 에이전트가 read 권한만 가지고, 편집 에이전트가 그 결과를 받아 write를 수행하는 구조라면, 로그만 봐도 "어디서 무엇이 잘못됐는지"가 한눈에 들어온다. 권한은 사고 예방 도구이기도 하지만, 사고 사후 진단 도구이기도 하다. 5–10명 팀에서 새벽 2시에 부팅된 자율 작업이 무엇을 했는지를 다음 날 아침에 빠르게 재구성할 수 있는가. 이 질문에 자신 있게 답할 수 없다면, 권한 격리가 아직 부족한 것이다.

도구 측면에서 한 가지 더 짚자. 12개 패턴 중 또 하나가 **Single-Purpose Tool Design**이다 [W23]. FileReadTool, FileEditTool처럼 도구를 좁은 스코프로 잘게 쪼개 두는 방식이다. 하나의 거대한 "FileTool"이 read·write·delete를 다 하면, 권한 부여가 binary가 된다 — 다 주거나 하나도 안 주거나. 반면 단일 목적 도구로 분해하면 read는 모두에게, write는 일부 에이전트에게, delete는 사람 게이트로 두는 등 입자도가 미세해진다. 권한 설계는 도구 설계와 한 몸이다. 함께 생각하자.

## 논쟁점 — Subagent를 만들까, 메인에 위임할까

지금까지 우리는 "역할을 분리하자"라는 큰 방향을 따라왔다. 그런데 어떻게 분리할 것인가에서는 진영이 갈린다. 책의 긴장감을 만드는 자원은 한쪽 입장만 옹호하지 않는 것이니, 두 입장을 다 짚고 가자.

**관점 A: Subagent로 명시적으로 분리하자.**

Anthropic이 공식적으로 권장하는 방향이다 [W14, C2-팁35]. `.claude/agents/` 디렉토리에 서브에이전트를 정의해 두면 컨텍스트가 격리되고, 각각에 좁은 권한과 전문 프롬프트를 줄 수 있다. 팀이 SOP화하기에도 좋다. "테스트 작성은 `tester` 서브에이전트에게, 보안 리뷰는 `security-reviewer` 서브에이전트에게"처럼 책임이 명확해진다. 새 팀원도 "어느 에이전트가 무엇을 하는가"라는 지도를 보고 합류할 수 있다.

**관점 B: 메인에게 위임시키자. Subagent 분할은 brittle하다.**

Shrivu Shankar는 정반대의 주장을 한다 [W30]. 그의 표현을 그대로 옮기자면, 커스텀 서브에이전트는 brittle한 해법이다. 인간이 정의한 rigid workflow를 강제로 박아두는 셈이라, 모델이 더 똑똑해질수록 그 박힌 워크플로가 오히려 발목을 잡는다는 것이다. 그의 대안은 이렇다. CLAUDE.md를 잘 써 두고, 메인 에이전트가 자체 `Task()` 기능으로 동적으로 위임하게 하자. 그러면 모델이 그때그때의 맥락에 맞는 분업을 스스로 결정한다.

또 하나의 비판이 있다. Skill·Subagent 호출 신뢰성 자체에 대한 회의다. 한 분석은 "Claude Code는 너무 똑똑해서 어차피 서브에이전트를 잘 부르지 않는다"고 지적한다 [W24]. 커뮤니티가 audit한 214개 스킬 중 **73%가 100점 만점에 60점 미만**이었다는 보고도 있다. 만들어 두고도 안 불려지거나, 불려도 품질이 안 나오는 케이스가 누적되면 "저거 진짜 쓰기는 쓰는 거야?"라는 의심이 팀 안에 자라난다. 찜찜한 일이다.

그렇다면 어떻게 결정해야 할까? 결론을 응축한 4지선다 결정 프레임은 이 책의 7장에서 본격적으로 다룬다. 지금은 1차 판별 기준 정도만 들고 가자.

- **컨텍스트 격리가 본질인가?** (예: 리서치는 편집 못 하게) → Subagent
- **권한 분리가 본질인가?** (예: 보안 리뷰는 push 못 하게) → Subagent
- **빈도가 높고 SOP화가 필요한가?** → Subagent (팀 자산화)
- **그때그때 맥락에 따라 분업이 바뀌는가?** → 메인 + Task 동적 위임
- **CLAUDE.md만으로 충분히 메인이 알아서 할 수 있는가?** → 메인 + Task

두 진영 다 옳다. 다만 어느 쪽을 기본값으로 둘지는 팀의 성향과 작업의 종류에 따라 다르다. 안전·결정성을 더 중시하는 팀은 Subagent를, 유연성·속도를 더 중시하는 팀은 메인 + Task를 선호한다. 너의 팀이 어느 쪽이 더 중요한지부터 합의하자. 그 합의 위에 7장의 4지선다를 얹으면 된다.

한 가지만 더. 이 논쟁은 시간이 지나면 자연스럽게 한쪽으로 기울 가능성이 있다. 모델이 좋아질수록 동적 위임이 더 안정적으로 동작하기 때문이다. 반대로 거버넌스 부담이 큰 조직(금융·의료·공공)은 명시적 Subagent 쪽으로 더 무겁게 기울 가능성이 있다. "누가 무엇을 할 수 있는가"를 감사·증빙해야 하는 환경에서 동적 위임은 추적이 까다롭다. 너의 팀이 어떤 환경에 있는지에 따라 1년 뒤의 답이 달라진다는 점도 기억해두자.

## 한국의 5인 팀 — 오케스트라 지휘자 모델

오케스트레이션이라는 단어가 추상적으로 들린다면, 한국 개발자가 GeekNews에 공유한 다섯 명 팀 운영 모델을 보자 [C1]. 이 팀은 사람을 "오케스트라 지휘자"로 자리매김했다. 에이전트들이 연주자다. 진행 흐름은 이렇다.

```
/issues  → GitHub 이슈 자동 생성 (요건 정리 단계)
/work    → 이슈 기반 개발 + PR 생성 (구현 단계)
/review  → PR 리뷰 + 개선 (검증 단계)
사람     → UX, 코드 스타일 최종 확인 (오전 11:30 회의)
```

세 개의 슬래시 커맨드와 한 번의 사람 회의가 하루를 만든다. 흥미로운 점은, 이 흐름이 우리가 앞서 본 학술 패턴들과 자연스럽게 맞물린다는 것이다. `/issues`는 Initializer 역할이다 — 이슈로 골격을 정의한다. `/work`는 Coder 역할이다. `/review`는 Verifier 역할이다. 11:30의 사람 리뷰는 그 위의 Critic 시선이다. AgentForge의 5역할이 한국 개발자의 일상에 그대로 내려앉은 모양새다.

같은 분이 인용한 "2명 팀이 월 $400으로 대규모 팀 산출물을 달성"이라는 표현을 들으면 처음엔 과장처럼 들린다 [C1]. 하지만 잘 보면 그 수치는 모델 비용 청구서다. 사람의 시간은 별도다. 두 명이 하루 8시간 일하면서, 그 위에 4탭의 에이전트와 3개의 슬래시 커맨드와 11:30의 회의가 더해진다. 결과물은 다섯 명 팀처럼 보이지만, 안을 들여다 보면 **사람 둘 + 에이전트 다수 + 디시플린 하나**의 합이다. 디시플린이 빠지면 같은 청구서로 절반의 결과만 나온다. 이게 핵심이다.

조금 더 솔직히 말하자면, 이 모델의 진짜 비용은 청구서가 아니라 **사람의 인지 부하**다. 4탭의 작업 흐름을 머리에 담고, 각 탭이 어디까지 갔는지를 11:30에 한 번에 종합하려면 사람 쪽 컨텍스트 윈도가 적지 않게 소모된다. 한국 개발자의 후기에서도 "주니어 개발자가 이해도 없이 PR을 올리는 사례가 늘었다"는 한 줄이 등장한다 [C4]. 4탭을 굴리면서 동시에 코드 이해까지 자신의 머리 안에 두는 일은 시니어에게도 쉬운 일이 아니다. 그래서 이 패턴을 도입할 때 한 가지 권하고 싶은 게 있다. **하루의 첫 30분과 마지막 30분은 4탭을 다 닫고 사람의 노트만 쓰자.** 시작 30분은 무엇을 분기시킬지 결정하는 시간이고, 마지막 30분은 분기시킨 것을 회수하는 시간이다. 그 두 끝에서 머리를 비워 두지 않으면, 다섯 명 팀의 산출물을 만들면서 두 사람의 머리는 5명 팀의 피로를 함께 받게 된다. 이건 한국 개발자만의 이야기가 아니다. 어느 팀이든 똑같이 부딪히는 자리다.

## 너의 톱니에 추가할 한 칸

> **[톱니 박스] 멀티 에이전트 시작을 위한 5칸 체크리스트**
>
> 1. **분류부터 한다.** 어제의 충돌이 (a) 한 사람의 병렬 사고인지, (b) 한 작업의 시선 부족인지, (c) 5명의 main push 충돌인지 종이에 적어보자. 답에 따라 해법이 다르다. (c)는 5장.
> 2. **한 워크플로를 골라 Generator/Verifier로 쪼갠다.** 가장 자주 도는 작업 하나를 고르자. PR 리뷰가 가장 흔한 후보다. 만드는 에이전트와 검증하는 에이전트 프롬프트를 분리해, 단일 모드와 7일간 비교해 보자.
> 3. **Initializer + Coder를 시도해본다.** 새 기능 하나를 골라, 첫 commit만 만드는 Initializer 프롬프트와 매 호출마다 한 feature씩 다루는 Coder 프롬프트를 따로 작성해 보자. "테스트 삭제 금지" 한 줄을 Coder에 박아두자.
> 4. **워크트리 3탭으로 시작한다.** 4탭 이상은 다음 주에. 영역이 겹치지 않는 세 개 작업을 골라 워크트리에 분산해보자. 같은 모듈을 두 탭이 만지지 않게 사전에 합의하자.
> 5. **Subagent vs 메인 위임 1차 분기를 정한다.** "권한 분리·컨텍스트 격리가 본질인 작업"만 Subagent로, 나머지는 메인 + Task로 시작한다. 한 달 뒤 7장 4지선다로 다시 점검한다.

## 마무리

이 장에서 우리는 두 종류의 동시성을 분리해 봤다. 한 사람의 병렬과 한 작업의 협업. 하나는 워크트리와 디시플린의 문제이고, 다른 하나는 Generator–Verifier·Initializer–Coder·컨텍스트 격리의 문제다. 둘은 다른 청구서를 받는다. 워크트리는 디스크와 사람의 주의력으로, Agent Teams는 7배의 토큰으로.

학계가 보내준 메시지는 단순하다. **만든 자에게 검증을 맡기지 말자.** 실무가 보내준 메시지도 단순하다. **한 번에 모든 걸 하려는 에이전트는 미완성을 남기고, 너무 일찍 끝났다고 판단하는 에이전트는 망가진 결과를 남긴다.** 이 두 메시지가 만나는 자리가 멀티 에이전트 오케스트레이션이다. 다섯의 평범한 직공이 한 명의 천재보다 더 안정적인 이유는, 다섯 명에게는 서로의 결과를 의심하는 두 번째·세 번째 시선이 자연스럽게 끼어들기 때문이다. 천재 한 명에게는 그 시선이 없다.

다음 장에서는 **안전과 권한**을 본격적으로 다룬다. 여러 에이전트가 동시에 일한다는 그림을 받아들였다면, 그 동시 실행이 우리 시스템을 부수지 않게 하는 첫 관문이 권한 설계다. `rm -rf`, `git push --force`, `DROP TABLE`을 막는 PreToolUse 훅, 7-mode 권한, ML 분류기, 그리고 한 보안 연구자가 PR 제목 한 줄로 API 키를 빼낸 CVSS 9.4 prompt injection 사고. 이 모두를 본격적으로 해부해 보자. 멀티 에이전트의 자유를 누리려면, 자유를 가두는 울타리부터 세워야 한다는 이야기를 다음 장에서 함께 살펴보자.
