# 계획 리뷰 로그 (Round 1)

## 종합 판단 (한 줄)

**보완 후 통과** — thesis와 3부 구조, 두 모티브의 배치까지 골격은 견고하다. 다만 (a) MCP 운영, (b) Subagent vs Skill 결정 프레임, (c) 워크트리/세션 격리, (d) 새 멤버 온보딩, (e) PR 리뷰 정책 재설계 — 이 다섯 가지가 챕터 안에 묻혀 있어 5–10명 팀 테크리드가 이 책을 들고 월요일 아침에 실행하기에는 한 단계 부족하다.

---

## 강점 (계획이 잘한 것 — 그대로 보존해야 함)

- **3부 구성의 인지적 점진**(Why → 4축 분해 → 운영 정치학)이 명확하다. 1·2장이 thesis 박는 데 집중하고, 8장(안티패턴)을 운영 챕터에 흩지 않고 독립 챕터로 모은 것은 임팩트 측면에서 옳다. 운영 가이드가 자주 빠지는 함정(안티패턴이 본문 곳곳에 흩어져 인상이 흐려지는 것)을 정확히 피했다.
- **Thesis가 데이터로 박힌다.** 98.4% 하네스, 동일 Opus 16%p 격차, CVSS 9.4 사고가 1장에 모두 배치되어 있다. "왜 지금 이 책인가"를 첫 30분에 증명할 수 있는 강한 도입재다.
- **두 모티브의 배치 설계.** 톱니바퀴는 1장 도입–7장 명시화–매 챕터 마무리 박스로 회수되고, 지휘자는 9장 도입–10장 회수로 닫는다. 모티브가 산발하지 않고 구조 안에 결합되어 있다.
- **위험 요소 섹션의 솔직함.** §7에서 "어디가 가장 빨리 늙을까"를 챕터별로 명시하고 GitHub repo·버전 박스·추상 패턴 우선이라는 5가지 방어 전략을 박은 것은 6개월 주기 노후화를 다룰 책으로서 책임감 있는 설계다.
- **분량 배분의 비대칭.** 1·6장 11,000자(가볍게) / 3·9장 14,000자(무겁게)로 의도적 비대칭을 둔 것이 좋다. "균등 분량 = 균등 중요도"의 흔한 함정을 피했다.

## 우려와 보완 제안 (Critical)

### C1. MCP가 어디에도 깊이 다뤄지지 않는다

- **이슈:** 7개 빌딩블록 표(2장)에는 MCP가 한 줄 등장하지만, **이후 9개 챕터 어디에도 MCP 운영을 정면으로 다루는 절이 없다.** 4장 안전 챕터에 Gateway 패턴이 잠깐 등장하고, 8장 안티패턴에 "MCP 미리 다 설치", "MCP를 API 추상화로 오용"이 한 줄씩 있을 뿐이다.
- **왜 문제인가:**
  1. 5–10명 팀이 가장 먼저 부딪히는 거버넌스 결정이 `.mcp.json` 공유·토큰 관리·서버 신뢰 경계다. Shankar가 "Jira·AWS·GitHub MCP를 simple CLI로 교체"한 결정의 무게(§6.2, §8.8)는 한 줄로 처리될 사안이 아니다.
  2. 리서치는 §8.8에 **MCP 역할 논쟁(추상화 vs 보안 게이트웨이)** 을 정식 논쟁점으로 분리해 두었다. 책의 논쟁점 자원이 챕터에 회수되지 않으면 §8 전체가 사장된다.
  3. Willison의 "Skills > MCP" 인용(§3.5.4 / §8.8)이 7장에서만 회수되면, MCP 자체의 실무 운영을 다루지 않는 책이 된다. 그러면 "운영서"라는 포지셔닝이 약해진다.
- **구체적 제안:** **4장(안전과 권한)에 "MCP 운영" 절을 독립 섹션으로 신설.** 분량 +1,500자. 다음 5가지를 다룬다:
  1. `.mcp.json` 팀 공유 vs `.mcp.local.json` 개인 — gitignore 디시플린
  2. 토큰 관리 — Gateway에 일임 vs OAuth 위임 vs 환경변수
  3. 서버 신뢰 경계 — sandboxed vs unsandboxed, 우리 팀이 MCP 서버를 자체 호스팅할 때 vs 공식 서버 쓸 때
  4. **MCP vs CLI 결정 트리** — Shankar 사례를 빌어 "이건 MCP일 일인가, bash 명령일 일인가"
  5. §8.8 논쟁점(추상화 vs 게이트웨이) 양면 제시 → 독자가 자기 팀 맥락에서 결정

### C2. Subagent vs Skill 결정 프레임이 두 챕터에 흩어져 있다

- **이슈:** Subagent 논쟁은 3장(W30 vs W24), Skill vs Subagent 분기는 7장(WHAT vs WHO)에 있다. **두 결정이 같은 자리에 모이지 않는다.** 5–10명 팀이 가장 헷갈려하는 지점이 정확히 이 두 결정이 만나는 곳이다.
- **왜 문제인가:** 실무에서는 "이걸 Skill로 할까, Subagent로 할까, 그냥 CLAUDE.md에 박을까, 아니면 메인이 Task()로 위임하게 할까" — **네 갈래 결정**을 한 자리에서 내려야 한다. 계획대로면 독자는 3장을 읽고 7장을 읽고 다시 3장을 펴고 자기 팀 화이트보드에서 합쳐야 한다.
- **구체적 제안:** **7장 후반부에 "결정 프레임: CLAUDE.md / Skill / Subagent / 메인+Task 4지선다"를 한 표·한 분기 트리로 박는다.** Skill = WHAT, Subagent = WHO, CLAUDE.md = ALWAYS, Task = DYNAMIC 같은 4축. 3장의 Subagent 논쟁은 그대로 두되, 7장 결정 프레임에서 "3장에서 본 그 논쟁을 너의 워크플로 X에 어떻게 적용할까"로 회수. 추가 분량 +800자, 7장 13,000 → 13,800.

### C3. 새 멤버 온보딩이 누락되었다

- **이슈:** 5–10명 팀의 가장 흔한 운영 질문 중 하나 — "신규 입사자가 첫 주에 우리 하네스를 어떻게 받아 한 시간 안에 일을 시작하게 할 것인가" — 가 어느 챕터에도 정면으로 없다. 7장 Skill을 "onboarding guide for a new hire"라고 인용하지만(§3.5.4), 그건 비유일 뿐 실제 온보딩 절차는 다루지 않는다.
- **왜 문제인가:**
  1. 톱니바퀴 모티브의 가장 강력한 회수 지점이다. 한 사람이 떠난 후에도 그의 학습이 다음 사람에게 자동으로 전수되는 — 이게 ratchet의 본질인데, 계획에는 추상으로만 남아 있다.
  2. Plugin Marketplace 사내 배포(§4.6, W28)와 직접 연결되는 실용 가치 — `.claude/` 폴더 한 번 trust로 모든 hooks/skills/commands가 자동 설치된다는 운영 효용 — 이 챕터화되지 않으면 marketplace 토픽 자체가 7장 끝에 단편적으로 한 줄로 묻힌다.
- **구체적 제안:** 두 가지 중 택1.
  - **(권장) 7장 마지막 절에 "한 시간 온보딩" 워크시트 신설.** "신규 입사자가 월요일 9시에 받는 것: (1) repo clone, (2) `.claude/` trust, (3) Plugin marketplace auto-install, (4) 첫 30분 안에 돌려볼 `/work` 명령" — 한 페이지 절차표. 분량 +700자.
  - **(차선) 9장 사례 비교에 "IMWEB 인프라팀이 신규 멤버 1명을 어떻게 받았는가" 미니 케이스 추가.**

### C4. 워크트리/세션 격리가 멀티 에이전트에 묻혀 있다

- **이슈:** 워크트리(§3.3.4)는 3장 Fork-Join 한 절에만 등장한다. 그러나 워크트리는 사실 **멀티 에이전트와 다른 차원의 운영 토픽**이다 — 한 사람이 한 명의 Claude Code 인스턴스를 쓰더라도 3–5개 워크트리를 병렬로 굴리는 게 표준 워크플로다(C1, GeekNews "9시 → 4탭 → 11:30 PR" 사례).
- **왜 문제인가:** 5–10명 팀에서 발생하는 두 종류의 충돌이 다르다. (1) 한 개발자의 4개 워크트리가 서로 부딪히는 경우, (2) 5명 개발자의 Claude Code 인스턴스들이 main에 동시에 push하려는 경우. 전자는 워크트리·브랜치 디시플린, 후자는 PR 게이트·conflict 정책. 3장은 (1)을 (2)인 양 다루고, (2)는 5장 PR 게이트에 묻힌다.
- **구체적 제안:** 3장에 **"워크트리 격리 vs 멀티 에이전트 오케스트레이션"의 차이를 명시하는 한 단락**을 신설. "워크트리는 한 사람의 병렬, 멀티 에이전트는 한 작업의 협업. 둘은 다른 문제이며 다른 해법이다." 그리고 5장 PR 게이트 절에서 "이제 한 팀의 5명이 동시에 push할 때"라는 이전 챕터로부터의 다리를 명시.

### C5. PR 리뷰 정책 재설계가 안티패턴(8장)에 묻혀 있다

- **이슈:** "AI가 코드를 쓰는 시대의 PR 리뷰가 어떻게 바뀌어야 하는가"는 §3.2.2(Shankar의 "인간 prompter 없는 PR은 2명 승인"), §4.5(P10: "AI 코드 흔적 거부감"), §6.4("AI slop PR", "GitHub kill switch"), §4.1(96/48% 검증 격차)에 흩어져 있다. **계획은 5장(자동화)에 PR 게이트로 묶고, 8장 안티패턴 "사람 측"에 AI slop을 던진다.** 그러나 PR 리뷰 정책은 자동화 토픽도 아니고 안티패턴 한 줄도 아니다 — **거버넌스 정책 재설계**다.
- **왜 문제인가:** 테크리드가 책을 덮은 다음 날 가장 빨리 손대야 하는 문서가 PR 리뷰 정책이다. "리뷰어가 누구냐, 몇 명 승인이냐, AI 흔적은 표시하느냐, intent는 어떻게 공유하느냐, 자동 머지는 어디까지 허용하느냐" — 이게 한 절에 모이지 않으면 책의 적용 가치가 떨어진다.
- **구체적 제안:** **9장(사례 비교)에 "PR 정책 비교 매트릭스"를 추가.** Anthropic / Toss / IMWEB / Shankar의 PR 정책을 한 표로 (필수 승인 수, AI 흔적 표기, 자동 머지 범위, 리뷰어 역할). 이미 있는 비교 매트릭스 옆자리에 한 표 더. 또는 5장 끝에 "PR 정책 재설계" 1,000자 짜리 미니 절을 두고 9장이 사례로 받는다.

## 권고 (Nice to have)

### N1. 1장 도입의 도발성을 한 단계 더

- **이슈:** 1장 오프닝이 CVSS 9.4 사고로 시작하는 것은 좋은데, 그 직후 "98.4% 하네스 → 16%p 격차 → Skill Issue 가설 → 수렴 명제"로 수치 4발이 연속이다. 좋은 도입이지만 **한국 독자에게 더 꽂히는 일화 하나가 더 있으면** 외국 사례·외국 수치만 쏟아지는 인상을 줄일 수 있다.
- **구체적 제안:** GeekNews "9시 → 4탭 → 11:30 PR"(C1) 또는 IMWEB "10배 평가"(C5) 일화를 1장 1.5번째 자리에 배치. CVSS(공포) → 한국 팀 일화(가능성) → 수치(증명) 3단 도입.

### N2. 6장 비용 챕터에 "한국 환경 환산" 한 표 더

- **이슈:** 6장에 "$13/dev → 원화" 환산을 한다고 명시되어 있으나, 한국 팀이 가장 알고 싶은 것은 **"우리 팀 5명 한 달에 얼마"** — 즉 시나리오별 월 비용 시뮬레이션이다.
- **구체적 제안:** "5인 팀 시나리오 3종" 표 한 장. (a) 라이트(MAX_THINKING_TOKENS 캡, /effort low 디폴트) ~$300/월, (b) 표준 ~$900/월, (c) 헤비(Agent Teams 활용) ~$2,500/월. 한국 환경 부가세·환율 가정 명시.

### N3. 9장 사례 비교에 "실패한 도입" 1건 추가 시도

- **이슈:** 9장 5개 사례가 모두 성공 사례다. §9 리서치 한계에서도 "조직이 도입 실패해 롤백한 사례는 거의 없음"이라고 인정한다. 그러나 5개 다 성공이면 책이 한쪽으로 기운다.
- **구체적 제안:** 명시적 롤백 사례를 못 찾는다면 **"shadow AI proliferation" 일반화 일화**(W11) 또는 GeekNews 부정 후기(C4 "주니어 PR 이해 없이 올린다") 중 하나를 미니 사례로 격상. 또는 9장 도입에서 "이 5개는 모두 성공 보고이므로 생존편향이 있다"는 메타 코멘트를 솔직히 박는다.

### N4. 10장 마무리감을 위해 "이 책 자체의 ratchet" 회수

- **이슈:** 10장이 도구 비교·미래 시그널·시니어/주니어로 가다가 마지막에 "톱니에 추가할 한 칸"으로 끝나는데, **책 전체가 ratchet 모티브였으므로** 마지막 의식이 좀 더 강해야 한다.
- **구체적 제안:** 10장 마지막 1쪽을 "이 책을 덮고 너의 팀 `.claude/CLAUDE.md`에 추가할 한 줄"이라는 단일 행동으로 닫는다. 모든 챕터가 박스로 했던 것을 책 전체 닫는 의식으로 격상.

### N5. 부록에 "결정 트리 모음"

- **이슈:** 책 전체에 결정 분기가 여러 개 등장한다 — Subagent vs Skill, Hook vs CLAUDE.md, 메인 vs 위임, MCP vs CLI 등. 부록에 이걸 모은 한 페이지가 있으면 책의 reference value가 크게 올라간다.
- **구체적 제안:** 부록에 "결정 트리 7선" 한 페이지. 본문 각 챕터에서 해당 분기마다 "→ 부록 D.3 참조".

## 챕터별 의견

| 챕터 | 의견 | 우선순위 |
|------|------|---------|
| 1. 왜 하네스인가 | 도입 강도 좋음. 한국 팀 일화 1개를 1.5번째 자리에 끼우면 더 좋음(N1). 책 전체 지도가 마지막에 있다고 했는데 너무 길어지면 1장의 속도가 느려진다 — 지도는 1쪽 분량 약속. | High |
| 2. 하네스의 해부학 | 정의·표 챕터로서 위험은 "지루함". 위험요소(§7.1)에 대응이 있으나, **§2.4 Subagent vs Skill을 여기서 미리 한 번 박아두면 3·7장에서 회수가 자연스럽다.** 현재 계획은 7장에서 처음 분기를 다루는데, 2장 어휘 챕터에 한 줄 정도 미리 등장시켜야 forward reference 부담이 줄어든다. | Med |
| 3. 멀티 에이전트 | 학술 인용 4개(Reflexion·Voyager·AgentForge·Survey)가 한 챕터에 몰린 것이 가장 부담스럽다. 위험요소에 대응이 있으나, **AgentForge와 Reflexion 중 하나는 "더 읽기" 박스로 빼고 본문에선 한 개만 깊이.** 워크트리 구분(C4)을 명시. | High |
| 4. 안전과 권한 | MCP 운영 절 신설 필수(C1). 분량 12,000 → 13,500. CVSS 9.4 사고를 1장에서 한 번 보여주고 4장에서 풀어 해부하는 구조는 좋음 — 단 **1장의 사고 묘사는 짧게(반쪽)**, 4장에서 풀게. | High |
| 5. 자동화의 결정성 | PR 정책 재설계 미니 절(C5) 또는 9장 매트릭스로 보완. 코드 비중 최대 챕터인 만큼 **한국 환경 코드 예시(사내 GHA, 폐쇄망 GitLab)를 1–2개 의무화**. | Med |
| 6. 관측·비용 | 가장 빨리 늙을 챕터 중 하나(환경변수명 변경). 추상 패턴 우선 원칙을 가장 엄격히 적용해야 함. 한국 환경 5인 팀 시뮬 표(N2). | Med |
| 7. 도메인 지식 코드화 | 책 모티브 회수 챕터로서 가장 중요. 결정 프레임 4지선다(C2)와 한 시간 온보딩(C3) 추가. 분량 13,000 → 14,500까지 늘려도 무방. | High |
| 8. 안티패턴 카탈로그 | 8개 안티패턴이 모두 카탈로그 톤이면 단조로울 수 있다. **3개는 "사고 회고 톤"(서사형), 5개는 "체크리스트 톤"** 으로 양식 다양화. AI slop PR은 8장에 두되 5/9장에서 거버넌스로 다시 회수. | Med |

| 10. 도구·조직·미래 | 마무리감 강화(N4). "도구 비교 매트릭스는 6개월 늙을 것"을 본문에서 한 번 더 인정 — 늙음 자체를 자료적 가치로 프레이밍한 §7.3 방어전략 5번을 챕터 내에서도 회수. | Med |

## 최종 권고

**다음 라운드에서 반드시 다뤄야 할 것 (우선순위 순):**

1. **MCP 운영을 4장에 독립 섹션으로 박는다 (C1)** — `.mcp.json` 공유, 토큰 관리, 신뢰 경계, MCP vs CLI 결정 트리, §8.8 논쟁점 회수. 5–10명 팀 거버넌스의 가장 큰 공백.
2. **결정 프레임 4지선다(CLAUDE.md / Skill / Subagent / 메인+Task)를 7장에 한 표·한 분기 트리로 응축 (C2)** — 현재 3장과 7장에 흩어진 결정을 한 자리에 모은다.
3. **새 멤버 한 시간 온보딩 워크시트를 7장 마지막에 (C3)** — Plugin marketplace + `.claude/` trust 한 번으로 자동 설치되는 운영 효용을 챕터화. 톱니바퀴 모티브의 가장 강력한 회수 지점.
4. **PR 리뷰 정책 재설계를 9장 비교 매트릭스로 격상 또는 5장 미니 절 (C5)** — 안티패턴 한 줄로 끝낼 일이 아니다. 테크리드가 책을 덮은 다음 날 가장 빨리 손댈 문서.
5. **워크트리 격리와 멀티 에이전트 오케스트레이션의 차이를 3장에서 명시 (C4)** — 한 사람의 병렬과 한 작업의 협업은 다른 문제이며 다른 해법이라는 한 단락.

**부수적으로 고려할 것:**
- 1장 도입에 한국 팀 일화 1개 (N1)
- 6장 5인 팀 비용 시뮬 표 (N2)
- 9장 실패·생존편향 메타 코멘트 (N3)
- 10장 마무리 의식 강화 (N4)
- 부록 결정 트리 모음 (N5)

이 5개 Critical을 다음 라운드에서 반영하면, 계획은 보완을 끝내고 챕터 저술 단계로 넘어갈 만하다.
