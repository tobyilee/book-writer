## 1. 책 제목 후보

각 후보는 본 의미(capability overhang)와 서브 의미(능력 격차/역량 활용 지체)를 한국어 화자에게 와닿게 풀어내는 방식이 다르다. 슬러그는 영문으로 명시했다.

### 후보 1 — **『AI의 숨은 능력 — 우리가 아직 꺼내지 못한 90%』**
- 슬러그: `ai-hidden-capability`
- 톤: 대중적·낙관적·자기계발 친화. 잡지/서점 평매대 친화.
- 포지셔닝: 일반 독자가 집어 들고 싶어지는 제목. "내가 못 쓰는 거구나"라는 자기 의심을 "꺼내면 되는 거구나"라는 호기심으로 전환.
- 약점: capability overhang의 학술적·전문가적 무게감이 다소 옅어진다. 전문가 독자에게는 가벼워 보일 수 있음.

### 후보 2 — **『능력 과잉 — AI는 이미 우리보다 멀리 와 있다』**
- 슬러그: `capability-overhang`
- 톤: 사회비평·담론서 톤. 칼럼·논픽션 느낌.
- 포지셔닝: 용어 자체를 한국어로 정착시키려는 시도. 부제가 "능력 격차" 의미를 양면으로 잡아준다(AI가 가진 능력 vs 우리가 끌어내는 능력).
- 약점: "능력 과잉"이라는 표현이 한국 독자에게 낯설다. 첫 문장에서 용어 도입 부담.

### 후보 3 — **『AI를 다 못 쓰고 있다 — 모델·사람·도구·조직·평가 사이의 격차』**
- 슬러그: `not-using-ai-fully`
- 톤: 직설적·문제 진단형. 실용서·기술서 톤.
- 포지셔닝: 부제가 책의 5층 구조를 그대로 노출. 개발자·전문가 친화. 책 내용과 가장 정확히 부합.
- 약점: 제목이 다소 평면적. 첫인상의 강도가 후보 1·2보다 약함.

**추천: 후보 1 『AI의 숨은 능력 — 우리가 아직 꺼내지 못한 90%』**

세 독자층 중 가장 크고 진입 장벽이 낮은 일반 독자를 책의 문 앞으로 끌고 오는 데 후보 1이 가장 유리하다. "숨은 능력"이라는 표현은 일반 독자의 '내가 잘못 쓰는 건가?' 자기 의심(커뮤니티 패턴 1)을 직접 건드린다. 부제 "90%"는 정확한 수치가 아니라 *비유*로 작동하며, 책 본문에서 "사실 90%는 검증할 수 없는 추정"이라는 메타 진술로 곧장 받아칠 수 있어서 책의 출발점으로 적합하다. 개발자·전문가 독자에게는 책 2장에서 "capability overhang"이라는 정식 용어를 도입하고, 부록에서 학술적 무게를 보강하는 방식으로 보완한다.

## 2. 책 특성

- **장르**: 에세이형 기술·사회 비평서. 절반은 개념·진단(논픽션), 절반은 실용 가이드. 챕터 후반부로 갈수록 실용 비중이 커진다.
- **분량**: 약 320~360페이지 (한글 글자 수 18만~21만 자). 챕터 평균 12,000~17,000자, 도입·결말 챕터는 6,000~10,000자. EPUB 기준 본문만. (v1 대비 9장 → 10장으로 확장하며 분량 상향.)
- **난이도**: 중급. 1~3장은 일반 독자 친화(초급), 4·5장은 보편(중급 진입), 6장은 개발자 친화(중급), 7·8장은 전문가 친화(상급), 9·10장은 다시 보편적 결론. 전체적으로 "AI를 한 번이라도 써본 한국 독자"가 진입 가능한 수준을 base로 잡고, 챕터별로 점진적 심화 후 마지막에 다시 좁아진다.
- **독자 여정**:
  - 진입 상태: *"AI가 대단하다는데 나는 왜 잘 못 쓰지? 동료들은 5배 빠르다는데. 어쩌면 나는 이미 뒤처지고 있는 건가?"* — 자기 의심·불안·체념의 혼합. (커뮤니티 패턴 1·7)
  - 출구 상태: *"격차는 실재한다. 하지만 그건 내 결함이 아니라 5층의 구조다. 모델·사람·도구·조직·평가가 동시에 미숙하다. 그 안에서 내가 다음 한 걸음으로 무엇을 할지 안다. 그리고 *쓰지 않기로 한 선택*도 그 안에서 정당하다."* — 진단을 받아들이고 자기 위치의 다음 행동이 명확해진 동시에, 비사용의 합리성도 입체적으로 본 상태.

## 3. 내러티브 아크

책은 세 막으로 흐른다. v1과 비교해 2막을 5개 챕터(4·5·6·7·8장)로 확장해 5층 원인을 1:1 매핑하고, 3막을 *서사 챕터 + 사회 챕터* 2개로 분리했다.

**1막(1~3장) — 격차를 발견하고 명명한다.** 1장에서 한국·해외 실제 장면(60세 개발자가 Claude Code로 재기, 시니어가 150,000줄 후 사용 중단, 한국 직장인 51.8% 사용·42.2% 대체 우려)을 한 화면에 펼쳐 *같은 도구를 두고 정반대 경험이 공존한다*는 직관을 만든다. 끝 절에서 *짧은 역사 도입*으로 GPT-2부터 2026 Frontier Lag까지의 12년을 6장면으로 압축해 2장으로 넘어가는 다리를 만든다. 2장에서 그 격차에 이름을 붙인다 — capability overhang. Jack Clark의 두 인용을 축으로 어원, 인접 개념 8개 표, 그리고 *5층(모델/사람/도구/조직/평가)이 4~8장의 골격이 된다는 메타 약속*을 박는다. 3장에서 격차의 원인을 *모델 안에서*가 아니라 *모델 바깥에서* 찾는 시각으로 전환한다 — 같은 모델인데 누구는 잘 쓰고 누구는 못 쓰는 이유. AlphaProof·Claude Code·Brittlebench 사례.

**2막(4~8장) — 다섯 차원에서 해부한다, 그러나 독자 진입 장벽을 점진적으로 올리며.** 4장은 *사람*(인지·UX) — mental model, 신뢰 보정, 자기 평가와 실측의 괴리, 상상력 병목, 그리고 *비사용의 합리성*(개인 차원). 5장은 *모델 안* — emergent capability, sandbagging, RLHF default mode, noise injection. 4장에서 사람 바깥의 시야를 잡아준 다음 5장에서 모델 안으로 잠깐 들어가는 이 순서가, v1의 *모델 안 → 사람*보다 정서 곡선과 맞는다(리뷰 P1-A 반영). 6장은 *도구*(컨텍스트 엔지니어링과 하네스). 7장은 *조직*(BCG 가치 격차, Tinholt 3원인, 한국 조직 격차, OECD 재교육 시차). 8장은 *평가의 시차* — Frontier Lag, METR time-horizon, configuration underspecification, 모델 카드, 그리고 *한국어 LLM frontier-lag 전용 절*. 5층이 4·5·6·7·8장에 1:1 매핑된다(리뷰 P0-A 반영).

**3막(9~10장) — 그래서 어떻게 살 것인가.** 9장은 *세 사람의 하루* — 일반 사용자 민지, 개발자 준호, 전문가 서영이 같은 작업(분기 보고서·복잡한 리팩토링·한국어 LLM 평가)을 어떻게 다르게 끌어내는지 *장면 세 개*로 보여준다. 원칙은 장면 안에서 자연스럽게 추출되며, 27개 원칙 나열은 부록 A로 이관된다(리뷰 P0-B 반영). 10장은 *격차의 다음 10년* — 양쪽 관점, 책의 입장, 회의주의의 합리성을 *집단 차원*으로 확장(4장의 개인 차원을 받음), 한국 사회 디지털 디바이드의 다음 형태, 그리고 마지막 청유. 청유는 "오늘 한 가지 작업"이 아니라 *5층 구조를 한 줄로 압축한 행동*으로 강화된다(리뷰 P2 반영).

정서 곡선: 1장 *불안·자기 의심* → 2장 *안도(이건 나만의 문제가 아니다)* → 3장 *호기심의 전환(바깥을 보자)* → 4·5장 *자기 인식과 모델 안 들여다보기* → 6·7·8장 *집중(어떻게 작동하는가 / 누구의 책임인가)* → 9장 *실행감(내가 할 수 있는 게 보인다)* → 10장 *주체성(나는 어떤 사람이 될 것인가, 쓰지 않는 자유까지 포함해서)*.

- **주요 내용**:

  - 시니어 개발자가 150,000줄을 만든 뒤 AI를 끊은 theSeniorDev 사례
  - 한국 직장인 51.8%가 매일 AI를 쓰지만 42.2%는 자기 업무 대체를 걱정 — 사용과 두려움의 공존
  - "동료는 5배 빠르다는데 나는 왜 안 되지?"라는 HN의 자기 의심 thread
  - 챕터 마지막 절 — **짧은 역사 도입**: GPT-2(2019)·GPT-3(2020)·Wei emergence(2022-06)·Clark overhang 명명(2022-11)·GPT-4(2023-03)·AlphaProof IMO(2024-07)·Claude Code(2025-02)·BCG 가치 격차 보고(2025-09)·Frontier Lag(2026-04)의 6~9장면을 3페이지로 압축. 2장으로 넘어가는 다리. *(리뷰 P2 "역사 흐름 매핑 누락" 반영)*
- **사례·인용 후보**:
  - GeekNews 27295 (60세 개발자), theSeniorDev blog (150,000줄)
  - 한경 매거진 2025-08 (한국 51.8% 통계), ZDNet 2025-09-12 (대체 우려 42.2%)
  - HN item 44095189 ("Anyone struggling to get value out of coding LLMs?")
  - 삼일PwC 헤비유저 임금 격차 자료
  - 역사 도입에는 레퍼런스 1.3절의 9개 마일스톤
- **독자가 얻는 것**: "이 격차는 내 결함이 아니라 시대의 풍경이다"라는 첫 정서적 안도 + 12년 역사의 척도.
- **예상 분량**: 9,000~11,000자 (도입 챕터 + 역사 절 포함)
- **대상 독자 가중치**: 일반 50% / 개발자 30% / 전문가 20%

### 2장. 능력 과잉이라는 이름 — capability overhang의 정체
- **핵심 질문**: 이 격차에 이름을 붙인다면 무엇이라 부를 수 있는가? 그 이름의 무게는 어디서 오는가?
- **주요 내용**:
  - Jack Clark이 Import AI 310에서 명명한 "capabilities overhang"의 두 인용
  - 137건의 emergent ability(Wei et al. 2022)와 그 반박(Schaeffer "Mirage") — *간단히만 소개*하고 자세한 모델 안 논의는 5장으로 미룸
  - GPT-4 자체가 overhang을 안고 출시되었다는 자기참조
  - 인접 개념 8가지(hardware overhang, elicitation gap, deployment gap, AI literacy gap, sandbagging, configuration underspecification, imagination bottleneck, Software 3.0)를 한 표로 정리
  - 책에서 쓸 정의 — "AI 시스템의 잠재 능력과 사용자·개발자·조직이 실제로 끌어내는 능력 사이의 격차"
  - **메타 약속**: 이 격차는 *모델/사람/도구/조직/평가* 5층에서 동시에 발생한다. 4~8장이 한 층씩 다룬다. (4~8장 = 5층 1:1 매핑)
- **사례·인용 후보**:
  - Clark, Import AI 310 (2022-11-28) — 첫 정식 명명
  - Clark, Import AI 321 (2023-03-21) — GPT-4 자체가 overhang
  - Wei et al. 2022 (arXiv:2206.07682) / Schaeffer 2023 (arXiv:2304.15004)
  - The Verge 2022-12-08 (사전 등재 기준점), Wiktionary 표제어
- **독자가 얻는 것**: 격차를 가리키는 정식 어휘 + 책 전체의 골격을 한 번에 보는 지도.
- **예상 분량**: 12,000~14,000자
- **대상 독자 가중치**: 일반 35% / 개발자 35% / 전문가 30%

### 3장. 같은 모델, 다른 모델 — 격차는 모델 바깥에 있다
- **핵심 질문**: 우리가 다 같은 GPT, 다 같은 Claude를 쓴다. 그런데 결과는 왜 이렇게 다른가?
- **주요 내용**:
  - AlphaProof/AlphaGeometry 2 사례 — Gemini base 동일, scaffold가 IMO 은메달을 만들었다
  - Claude Code 사례 — Simon Willison 인용("2025년 가장 영향력 있는 사건"), 같은 모델인데 채팅과 터미널·tool loop의 결과 차이
  - Brittlebench가 보여주는 12% 변동, 단일 perturbation으로 모델 순위 63% 사례에서 뒤집힘
  - "같은 모델이 사용자마다 다른 모델이다"는 핵심 명제로 챕터를 마무리
  - 이 명제가 가져오는 함의 — 격차의 책임이 모델에만 있지 않다, 그리고 우리에게도 있다는 부담. 4장(사람)으로 넘기는 다리.
- **사례·인용 후보**:
  - DeepMind 2024-07 (AlphaProof IMO), Nature 2025 논문
  - Simon Willison 2025-12-31 ("2025: Year in LLMs")
  - Brittlebench arXiv:2603.13285, POSIX arXiv:2410.02185
  - Cursor vs Claude Code 분담 정착 (Builder.io, NxCode, TrueFoundry)
- **독자가 얻는 것**: "모델 탓"이나 "내 탓"에 머물지 않는 제3의 시야 — 인터페이스·문맥·하네스가 결과를 만든다.
- **예상 분량**: 12,000~14,000자
- **대상 독자 가중치**: 일반 30% / 개발자 45% / 전문가 25%

### 4장. 사람 안에는 무엇이 들었나 — 의심·신뢰·상상력, 그리고 쓰지 않을 자유
- **핵심 질문**: 같은 도구를 두고 누구는 일찍 포기하고 누구는 끝까지 끌어내는 차이는, 도구가 아니라 사람의 무엇에서 오는가? 그리고 *쓰지 않기로 한 사람*은 무엇을 본 것인가?
- **주요 내용**:
  - 사용자의 mental model 부재와 의인화의 함정 (arXiv:2510.02660)
  - 신뢰 보정 실패와 과의존 (CHI 2024 "Are You Really Sure?", CHB 2024)
  - "상상력 병목"(NEXT Conf 2023) — 새 도구를 기존 작업에 끼워 넣을 뿐 새 작업을 발명하지 않는다
  - **한국 사례 절**: 한국 직장인 51.8%가 *무엇에 쓰는가*의 빈약함(이메일·번역·요약 등 단발 작업 위주). 헤비유저 18.2%는 *어떤 작업을 발명*했는가. *(리뷰 P1-B 반영, 한국 사례 분산)*
  - 자기 평가와 실측의 괴리 — 더 빠르다고 느끼지만 실제로는 더 느린 경우 (Pragmatic Engineer Cursor 분석, METR)
  - "프롬프트 설명할 시간에 직접 짜는 게 빠르다"는 합리화의 정체

### 5장. 모델 안에는 무엇이 들었나 — emergent · sandbagging · default mode
- **핵심 질문**: 모델은 정말 우리가 보는 것 이상으로 잘 하는가? 그 능력은 왜 평소엔 보이지 않는가? — 4장에서 사람을 들여다본 다음, 잠깐 모델 안으로 들어가 본다.
- **주요 내용**:
  - Wei et al.의 137건 emergence 사례를 일반 독자 언어로 풀이 — "갑자기 할 줄 알게 되는 일"
  - Schaeffer 반박의 함의 — 능력은 있지만, *어떻게 측정하느냐가 능력을 보이게도 안 보이게도 한다*. (측정 문제는 8장에서 본격화한다는 예고)
  - RLHF default mode — "친절하고 안전한 어시스턴트"로 가둬진 모델의 한 단면
  - Sandbagging (van der Weij et al. 2024) — 모델이 일부러 능력을 숨기는 현상
  - Noise injection (arXiv:2412.01784) — 숨겨진 능력을 추출하는 방법
  - **한국어 default mode 한 절**: 한국어로 응답할 때 더 보수적·격식적·길어지는 패턴 관찰. 1차 자료 부족 시 *정보 제한*으로 명시하고, 챕터 저술 단계에서 OKKY·velog 1차 관찰 보강 권고. *(리뷰 P1-B 반영)*

- **사례·인용 후보**:

### 8장. 측정의 시차 — 무엇이 능력인가는 어떻게 재느냐의 함수다
- **핵심 질문**: 학계와 산업이 *AI가 무엇을 할 수 있는가*를 답한다고 주장한다. 그 답은 얼마나 *지금*의 답인가? 우리는 어떤 시차를 두고 능력을 보고 있는가?
- **주요 내용**:
  - **벤치마크 포화** — Stanford HAI 2025 AI Index가 보고하는 주요 벤치마크의 최고 모델 포화. 새 능력을 측정할 도구의 부족.
  - **Configuration underspecification** — Frontier Lag (arXiv:2605.04135) 112,303건 평가 논문 감사. 학계가 답한다고 주장하지만 실은 *수개월·수년 전, 더 싸고 elicit이 덜 된 모델*을 측정.
  - **시간 horizon 평가** — METR (arXiv:2503.14499) "50% task-completion time horizon" 지표. AI가 50% 성공률로 끝낼 수 있는 작업의 인간 평균 시간. 7개월마다 doubling.
  - **Prompt sensitivity가 metric을 흔든다** — POSIX (arXiv:2410.02185). 단일 점수로 capability를 표현하는 것의 부적절성.
  - **Sandbagging eval** — 5장의 sandbagging을 *측정자 입장*에서 다시 본다. 활성화 노이즈, password-locked capability.
  - **모델 카드의 위치** — Claude 4.7 카드, GPT-5 시스템 카드, Gemini 2.5 카드의 capability table. *1차 자료가 어디 있고 어떻게 읽어야 하는가*. 본문에 카드 한 페이지를 발췌·해석. *(리뷰 누락 주제 1 반영, 1차 자료 보강 필수)*
  - **한국어 LLM frontier-lag 전용 절** — 영어 벤치마크 포화와 한국어 도메인 평가의 비대칭. 한국 책 독자가 *자기 언어로 평가된 적 없는 모델*을 어떻게 평가할 것인가. 1차 자료 부족 시 *정보 제한*으로 명시하되, 한국 평가 데이터셋 사례(KMMLU 등 가능한 범위) 1~2건은 짚는다. *(리뷰 누락 주제 2 반영)*
  - **"잘 쓴다"는 것의 측정 가능성 논쟁** — 레퍼런스 5.2절을 8장 결말에 통합. *한 사람의 best practice가 다른 사람에게 안 통하는데 그것을 *측정*하는 게 가능한가.* 9장의 처방 챕터로 넘기는 다리. *(리뷰 P2 "5.2 측정 가능성 누락" 반영)*
- **사례·인용 후보**:
  - Frontier Lag 2026 (arXiv:2605.04135)
  - METR arXiv:2503.14499 (50%-time-horizon)
  - POSIX arXiv:2410.02185
  - Stanford HAI 2025 AI Index
  - van der Weij et al. 2024 (arXiv:2406.07358) — 측정 관점 재인용
  - Anthropic Claude 4.7 model card, OpenAI GPT-5 system card (보강 후 1차 인용)
  - KMMLU 또는 한국어 LLM 평가셋 1차 자료(가능 범위)
- **독자가 얻는 것**: "AI가 무엇을 할 수 있나"라는 질문 자체를 *측정의 함수*로 보는 시야. 모델 카드를 직접 읽을 수 있는 최소 도구.
- **예상 분량**: 13,000~15,000자
- **대상 독자 가중치**: 일반 20% / 개발자 25% / 전문가 55%

### 9장. 세 사람의 하루 — 사용자·개발자·전문가는 같은 작업을 어떻게 다르게 끌어내는가
- **핵심 질문**: 격차를 진단했다면, *세 사람의 책상 위에는 어떤 풍경이 펼쳐져 있는가*? 그 풍경 안에서 *원칙*은 어떻게 살아 움직이는가?
- **주요 내용**:
  - **챕터 도입의 메타 자기 검증**: *같은 원칙이 모두에게 통하지 않는다*는 사실(8장 결말의 "측정 가능성" 논쟁을 받음)을 9장 도입에서 인정한 다음 시작. *(리뷰 P2 "측정 가능성"을 9장 도입에 흡수)*
  - **장면 1 — 일반 사용자 민지의 분기 보고서.** 가설적 인물이 분기 보고서 초안을 ChatGPT/Claude/Gemini로 끌어낸다. 출력 형식 강제, 역할 부여, few-shot 2~5개, 도구 분담, 검증 가능 작업부터 — 이 원칙들이 *민지의 한 시간* 안에서 자연스럽게 등장한다.
  - **장면 2 — 개발자 준호의 복잡한 리팩토링.** Cursor 80% + Claude Code 20% 분담, plan-then-execute, 컨텍스트 위생, rules 파일, sub-agent, **AI 생성 ⇄ 인간 검증 루프** — 6장의 검증 루프 모티프가 준호의 작업에서 *작동하는 모습*을 본다.
  - **장면 3 — 전문가 서영의 한국어 LLM 평가.** scaffold가 곧 capability, configuration 공개, harness benchmark 분리, sandbagging 가설, capability surface 시각화, *한국어·한국 도메인 평가의 부재*를 자기 연구에서 어떻게 다룰지. 8장의 frontier-lag 논의가 서영의 실제 결정으로 변환된다.
  - **세 장면의 교차** — 같은 *검증 루프* 원리가 세 사람에게 다르게 작동한다. 같은 *컨텍스트 엔지니어링* 원리가 세 사람에게 다른 형태로 적용된다. 원칙은 *복사할 수 있는 것*이 아니라 *번역해야 하는 것*임을 챕터 결론에서 명시.
  - 27개 원칙 전체 나열은 **부록 A**로 이관. 본문은 *장면·서사·근거*, 부록은 *압축 체크리스트*. *(리뷰 P0-B 핵심 반영 + P2 "부록 A와의 분담" 명시)*
- **사례·인용 후보**:
  - Anthropic Claude prompting best practices, Anthropic context engineering
  - GeekNews 27677 (50 best practices), Disquiet William Jung
  - Brynjolfsson 2023 (저숙련 +35%, 고숙련 +0%) — 시니어 특화 처방의 근거
  - 삼일PwC 헤비유저 임금 격차 — 민지의 자기 위치 자각
  - Karpathy "Iron Man suit", AI 생성–인간 검증 루프
- **독자가 얻는 것**: 자기 그룹의 처방을 *장면 속에서* 본다. 다른 두 그룹의 처방을 *남의 것으로* 읽으면서 자기 처방의 한계도 본다. 부록 A에서 압축 체크리스트로 책상 옆에 둔다.
- **예상 분량**: 16,000~18,000자
- **대상 독자 가중치**: 일반 33% / 개발자 33% / 전문가 33% (균등)

### 10장. 격차의 다음 10년 — 그래서 우리는 어떻게 살 것인가
- **핵심 질문**: overhang은 좁혀지는가, 더 벌어지는가? 그 안에서 한 사람의 선택은 무엇을 의미하는가? — 4장에서 *개인의 비사용 합리성*을 다뤘다면, 10장에서는 *집단의 회의주의*까지 확장한다.
- **주요 내용**:
  - 양쪽 관점의 무게 — 좁혀짐(컨텍스트 엔지니어링 표준화·METR doubling) vs 벌어짐(BCG 5%만 substantial value·한국 헤비유저 임금 격차)
  - 책의 입장: *두 격차*를 분리해서 보자 — 모델·도구 측면은 좁혀지지만, *사람이 자기 능력으로 통합하는 속도의 격차*는 벌어진다
  - **집단 차원의 회의주의의 합리성** — QuitGPT 700,000명, sycophancy 비용, Pentagon deal 같은 정책 우려. 4장의 *개인 비사용*을 받아 *집단의 합리적 거리두기*로 확장. *(리뷰 P1-C 핵심 반영, 4장↔10장 2단 배치 완성)*
  - 한국 사회의 디지털 디바이드 다음 형태 — 인프라는 세계 최고, 정보 판별·AI 활용은 격차
  - 새로운 격차의 사회적 함의 — Brynjolfsson 단기 평준화 vs 한국 헤비유저 프리미엄의 장기 격차
  - **마지막 청유** — 책 전체의 5층 구조를 한 줄로 압축한 행동. v1의 "오늘 한 가지 작업을 골라 끝까지 끌어내 보자"보다 무게를 올려, *"자기 도메인에서 검증 가능한 작업 하나를 고르고, 그 작업의 컨텍스트를 한 번이라도 더 정교하게 짜보자. 그리고 다음 작업에서는, 그 정교함을 자기 도구의 일부로 만들자"* 같은 *5층 원리를 행동으로 옮긴 청유*. 단, *쓰지 않기로 한 독자에게는 그 선택의 정당성을 한 번 더 명시*하고 닫는다. *(리뷰 P2 "청유 강도" 반영)*
- **사례·인용 후보**:
  - MIT Tech Review 2026-02 (QuitGPT 캠페인)
  - Brynjolfsson QJE 2025, 한국 삼일PwC 임금 격차
  - KDI / 한국노동연구원 디지털 디바이드 분석
  - Clark Import AI 397·438 (capability overhang 계속 진행 중)
  - Karpathy "Iron Man suit", Software 3.0
- **독자가 얻는 것**: 격차를 *통계*가 아니라 *자기 선택*으로 다시 보는 시야. 사용/비사용을 양극으로 두지 않는 입체적 주체성.
- **예상 분량**: 9,000~11,000자 (결말 챕터는 1장 호흡으로 짧게)

## 7. 리뷰 라운드 1 반영 — 변경 항목과 이유

### P0 (필수 반영, 둘 다 처리)

- **P0-A 5층 = 5챕터 회복**: v1의 7장(조직+평가 통합)을 v2에서 **7장(조직) + 8장(측정의 시차)**으로 분리. 2장의 메타 약속("4~8장이 5층을 1:1 매핑한다")이 회복됨. 9장→9장 그대로, v1 9장 → v2 10장으로 결말 챕터 번호 이동.
- **P0-B 8장 27원칙 나열 재설계**: v1의 8장(27원칙 나열)을 v2에서 **9장 "세 사람의 하루"**로 재설계. 가설적 인물 민지(사용자)·준호(개발자)·서영(전문가)의 *장면 3개*로 원칙을 서사 안에 녹임. 27개 원칙 전체 나열은 부록 A로 이관. *부록 A vs 9장 분담 = 압축 체크리스트 vs 서사·원리*로 명시.

### P1 (가능한 한 반영, 셋 다 처리)

- **P1-A 4장 위치**: v1의 4장(모델 안)과 5장(사람) 순서를 **교환**. v2의 4장 = 사람(인지·UX + 비사용 합리성), 5장 = 모델 안. 3장에서 "모델 바깥"으로 시야를 옮긴 다음 4장에서 사람으로, 5장에서 잠깐 모델 안을 들여다본 뒤 6장(도구)으로 다시 바깥. 정서 곡선 자연화.
- **P1-B 한국 사례 분산**: 4·5·6·7장에 한국 사례 절을 명시적으로 추가. **4장** = 한국 직장인 51.8%/18.2%의 *질적 해석*(무엇에 쓰는가의 빈약함, 헤비유저의 작업 발명). **5장** = 한국어 LLM의 default mode 한 절(정보 제한 가능성 명시). **6장** = 한국 헤비유저 개발자의 워크플로우 한 절(Disquiet William Jung 또는 GeekNews 27295의 *기술적 디테일*). **7장** = 한국 조직 격차(KRIVET·KDI·KoreaDeep)를 한 절 이상.
- **P1-C 회의주의 격상**: v1에서 9장의 한 절로 묻혀 있던 "쓰지 않을 자유"를 **4장(개인의 비사용 합리성) + 10장(집단의 합리적 회의주의)**의 2단 배치로 격상. 책의 출구 약속("쓰지 않을 자유도 같은 무게로")이 분량으로도 지켜짐.

### P2 + 누락 주제 (선택적 반영, 모두 흡수)

- **P2 역사 흐름**: 1장 마지막 절에 *12년 역사 6~9장면*을 짧은 도입으로 추가. 2장과의 다리.
- **P2 6장 두께**: AI-native 인프라(llms.txt/MCP/Skills) 절을 6장에서 7장으로 이관해 6장 슬림화. 6장은 *프롬프트→컨텍스트→하네스→검증 루프→Software 3.0*의 단조 흐름.
- **P2 측정 가능성(5.2) 누락**: 8장 결말에 "잘 쓴다는 것의 측정 가능성" 논쟁 한 절. 그리고 9장 도입에 "같은 원칙이 모두에게 통하지 않는다"는 메타 자기 검증.
- **P2 부록 A 8장 분담 모호**: 부록 A를 *9장의 서사에서 추출한 압축 체크리스트 + 도구별 1페이지 use case*로 재정의. 본문 9장 = 서사·원리, 부록 A = 압축·도구별.
- **P2 마지막 청유 강도**: 10장의 마지막 청유를 *5층 원리를 행동으로 옮긴 청유*로 강화. *쓰지 않기로 한 독자에게도 한 번 더 정당성을 명시*하고 닫는다.
- **누락 주제 1 모델 카드**: 8장에 *모델 카드 직접 발췌·해석* 한 절. Claude 4.7 / GPT-5 / Gemini 2.5 카드의 capability table을 1차 자료로 보강.
- **누락 주제 2 한국어 LLM frontier-lag**: 8장에 *한국어 LLM 평가의 부재*를 전용 절로. 부재 자체를 본문에 명시.
- **누락 주제 3 Karpathy 검증 루프**: 6장에 *AI 생성 ⇄ 인간 검증 루프 전용 절*. 그리고 9장 장면 2에서 *작동하는 모습*으로 재등장. 두 번 다룸으로써 책의 핵심 원리로 부각.

### 미반영 / 다른 해결책 제시

- **리뷰 챕터별 코멘트 — 1·2·3장 분량 점프**: "1장(6,000자) → 2장(16,000자) → 3장(15,000자)의 가파른 점프" 지적. v2에서는 1장을 9,000~11,000자(역사 절 추가)로 늘리고, 2장을 12,000~14,000자, 3장을 12,000~14,000자로 조정해 점프를 완만하게 함.
