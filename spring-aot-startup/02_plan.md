## 책 특성

- **장르:** 에세이형 기술서 (토비 스타일). 단순 카탈로그가 아니라, 각 기술의 *역사·동기·트레이드오프·실전 적용*까지 이야기로 묶는다.
- **분량:** 410~450쪽 / 한글 약 41~45만 자 (13 본 챕터 + 서문 + 에필로그 + 부록 5개)
- **난이도:** 중급+ ~ 고급 진입. Spring Boot 운영 경험 + JVM 기본기 전제.
- **독자 여정:**
  - **진입 상태:** Spring Boot 운영을 1~3년 해본 백엔드 개발자. 컨테이너로 배포하면서 cold start, 메모리, 스케일링이 비용·SLA 문제가 됐다. AOT·Native·GraalVM·CRaC·Leyden이라는 키워드가 트위터/InfoQ에서 자꾸 보이는데 무엇이 무엇인지, 자기 프로젝트에 무엇이 맞는지 답을 못 한다.
  - **출구 상태:** "내 워크로드에는 X를 도입한다, 그 이유는 5단계 비용 중 어디를 줄이고 어떤 trade-off를 받아들이기 때문이다"라고 동료 앞에서 결정 트리를 그리며 설명할 수 있다. 각 기술의 함정(리플렉션·열린 파일 핸들·시드 공유·warm-up 스파이크 등)을 사전에 안다. 진단 도구로 자기 앱의 5단계 비용 프로파일을 측정할 수 있다. *D단계(JIT warm-up)*에 대해서도 Azul·Alibaba·OpenJDK가 어떻게 다르게 풀었는지 안다.
- **차별점 (이 책이 다른 자료와 다른 점):**
  1. **5단계 비용 모델**을 책 전체의 통일된 비교 축으로 사용. 다른 책·블로그는 기술별로 따로 설명한다.
  2. **결정 트리**(11장)를 책의 클라이맥스로 두고, 모든 챕터가 거기로 수렴.
  3. **트레이드오프와 함정**을 일급 시민으로 다룬다 — 단순 "어떻게 켜는가"가 아니라 "켰을 때 무엇이 깨지는가".
  4. **토비 스타일의 역사적·철학적 맥락** — Quarkus·Micronaut의 도발, Spring이 늦었던 이유, "리플렉션은 적인가 도구인가"의 30년 논쟁, "OpenJDK가 늦은 동안 Azul·Alibaba는 무엇을 했나"의 산업사.
  5. **D단계 전담 챕터**(9장) — 시장의 다른 자료가 다 빠뜨리는 JIT warm-up을 정면으로 다룬다.
  6. **한국 백엔드 컨텍스트** — 한국 컨퍼런스·OKKY·velog 후기를 적극 인용하고, 검증되지 않은 영역은 그렇다고 명시.

## 내러티브 아크

이 책은 세 막짜리 드라마다.

**1막 (1~3장) — 문제 공간을 세운다.** 1장에서 "내 앱이 6초 만에 뜨는데, 그게 무엇이 문제인가"를 *체험적 톤*으로 묻는다. 컨테이너, K8s 오토스케일링, Lambda cold start, 카나리 배포의 warm-up spike — 모두 같은 시대적 압박이다. 학술 인용(Manner 2024)은 한 줄 각주로 가볍게 처리하고, 본문 톤은 "내가 본 6초"의 1인칭 서사로 유지한다. 2장에서 "Java는 왜 느리게 시작하는가"를 다섯 단계 비용으로 *개념적으로* 분해한다 (A. JVM 부팅 / B. 클래스 로딩 / C. 프레임워크 초기화 / D. JIT warm-up / E. 첫 요청). **진단 도구는 한 문단 포인터로 만 다루고 "자세한 측정은 12장에서"라는 약속만 남긴다** — 이렇게 해야 2장이 *순수 개념 챕터*로 깨끗해지고, 12장이 "약속한 측정의 회수"라는 서사적 의미를 얻는다. 3장에서 시야를 옆으로 돌려 Quarkus와 Micronaut를 본다 — 그들은 어떻게 빌드 타임으로 갔는가, Spring은 왜 늦었는가. *반사실*("만약 Spring이 2018년에 호환성을 깨고 갔다면?") 한 절을 추가해 토비식 깊이를 살린다. 이 챕터가 4장 이후의 모든 기술을 "Spring의 응답"으로 읽게 만드는 장치다.

**2막 (4~9장) — 기술 풀스택을 깊이 본다.** 5단계 비용을 줄이는 도구를 한 챕터씩 차근차근. 4장 CDS·AppCDS·Dynamic CDS (A·B 비용 — 가장 쉬운 첫걸음). 5장 Spring AOT 본격 해부 (C 비용). 여기에 "Reflection: 30년 자산을 어떻게 마이그레이션하는가"의 사변 블록을 *5장 전반부에* 두어 카탈로그성을 깨뜨리고, "왜 lazy init은 답이 아닌가" 사이드바를 도입부에 끼워 Reference 1.6/4.3을 회수한다. 6장 GraalVM Native Image 첫 챕터 — closed-world, points-to analysis, heap snapshotting의 철학. 7장 Native Image 실전 — 트러블슈팅, PGO, 빌드 시간, Liberty Mutual 심층. 8장 CRaC와 SnapStart (A~E 통째 우회). 본문을 명시적으로 두 부로 쪼개 **전반부: CRaC 원리와 Spring 통합 (EKS·on-prem 독자용) / 후반부: SnapStart의 산업 확장 (Lambda 독자용)** 구조로 만들어, 한 챕터 안에서도 두 독자층이 자기 길을 찾게 한다. **9장은 신설 D단계 전담 챕터 — "warm-up의 시대: Azul, Alibaba, OpenJ9가 먼저 푼 문제"**. Azul ReadyNow와 Cloud Native Compiler, Alibaba Dragonwell JWarmup과 더블 일레븐 사례, OpenJ9 SCC를 *산업사*로 다룬다. "OpenJDK가 늦은 동안 누가 어떻게 풀었나"라는 토비식 곁가지 서사다. 그리고 10장에서 OpenJDK의 응답 — Project Leyden JEP 483/514/515과 premain의 미래. 9·10장의 분리로 D단계가 책의 비교 격자 위에서 *주력으로* 자리잡고, 동시에 "비주류 답안 vs OpenJDK 정공법"의 두 노선이 명확해진다. Native Image를 6·7장 두 챕터로 쪼갠 이유와 동일한 논리 — *철학·산업사* 챕터와 *실전·정공법* 챕터를 분리해야 토비식 무게가 산다.

**3막 (11~13장) — 결정과 운영, 그리고 미래.** 11장에서 책의 클라이맥스 — **결정 트리**를 펼친다. 단, *분석 페이지* 일변도로 가지 않고, 도입을 "한 팀이 18개월 걸려 Native에 갔다가 결국 JVM으로 돌아온 가상 케이스 스터디"로 연다. 그리고 "분기를 잘못 탔을 때 무엇이 일어나는가" 섹션 — *틀린 결정의 비용*을 한 절로 다룬다. 마지막은 Netflix가 *Native를 고르지 않았다*는 결정에 대한 메타 회고로 묶어 "결정하지 않는 것도 결정이다"라는 토비식 마무리. 12장은 진단·트러블슈팅 — "이미 결정했다면, 그 결정이 옳았는지 어떻게 *확인*하나?"로 프레이밍해 11장 *후*의 진단으로 자연스레 잇는다. `BufferingApplicationStartup`, async-profiler, JFR, JIT 컴파일 로그로 자기 앱의 5단계 비용 프로파일을 측정한다. 13장은 미래 — Leyden premain의 AOT Code Compilation, Virtual Threads와의 *관계 없음* 반례형 사변("흔한 오해: 둘이 같이 가야 cloud-native다. 그러나..."), ZGC와의 상호작용, "10년 뒤 JVM은 어떻게 시작할까". 마지막 한 페이지는 토비식 회고 — "결국 시작 시간은 우리가 컨테이너에 얹은 비용을 다시 줄이려는 노력일 뿐".

**클라이맥스는 11장 결정 트리.** 1~10장이 거기로 수렴하고, 12~13장이 거기서 발산한다.

## 챕터 목록

### 1장. 6초의 무게: 왜 시작 시간이 다시 문제가 됐나
- **핵심 질문:** Spring Boot 앱이 6초 만에 뜨는데, 그게 *왜* 비용인가?
- **주요 내용:**
  - 모놀리식 시대와 컨테이너 시대의 시작 시간 의미 변화 — "기동은 한 번"에서 "기동이 끊임없는 운영 사건"으로
  - K8s HPA scale-up, Lambda cold start, 카나리 배포의 warm-up spike — 세 가지 압박을 1인칭 체험담으로
  - 시작 시간 비용을 돈으로 환산하면: Lambda init duration 과금, Pod 부팅 동안 idle 인스턴스 비용
  - (각주 처리) Manner et al. 2024의 cold start 학술 분류 — 본문은 체험 톤 유지
  - 이 책이 답하려는 질문 4개 (무엇을 줄일지·무엇을 골라야 할지·언제 도입할지·운영 중 무엇이 깨지는지)
- **5단계 비용 다루는 부분:** 전체 개요 (A~E)
- **독자가 얻는 것:** 자기 앱의 시작 시간을 *문제로 인식하는 언어*. "그저 느린 게 아니라 5단계 비용의 합이며, 각 단계가 분리 가능한 비용이다"라는 시각.
- **예상 분량:** 25~28쪽 / 25,000자

### 2장. Java는 왜 느리게 시작하는가: 다섯 단계의 해부
- **핵심 질문:** "시작 시간"이라는 한 단어 아래 어떤 5단계가 숨어 있는가?
- **주요 내용:**
  - A단계 JVM 부팅 — HotSpot 자체 초기화, 코어 클래스 로딩
  - B단계 클래스 로딩·링킹 — 파싱·검증의 비용
  - C단계 프레임워크 초기화 — `@ComponentScan`, `@Conditional`, 리플렉션
  - D단계 JIT warm-up — Tier 0 인터프리터에서 Tier 4 C2까지의 여정, 그리고 *warm-up spike*가 SLA를 깨는 패턴
  - E단계 첫 요청 처리 — hot path 진입 비용
  - 빈 앱에서 출발해 Spring Boot Web 앱까지, 각 단계가 얼마나 부풀어 오르는지 추적
  - **(축약) 진단 도구는 한 문단 포인터로만** — "각 단계를 어떻게 *볼 수 있는지*는 12장에서 본격적으로 다룬다" 한 줄 약속만
- **5단계 비용 다루는 부분:** A·B·C·D·E (총론, 개념)
- **독자가 얻는 것:** 어떤 기술이 어느 단계를 줄이는지를 *자기 머릿속 격자*로 정리. 이후 모든 챕터를 이 격자에 얹을 준비.
- **예상 분량:** 26~28쪽 / 26,000자 *(v1에서 4,000자 축소 — 진단 도구 부분을 12장으로 회수)*

### 3장. 그들은 어떻게 빌드 타임으로 갔는가: Quarkus, Micronaut, 그리고 Spring의 응답
- **핵심 질문:** Spring은 왜 AOT에 늦었는가? 그리고 어떻게 따라잡았는가?
- **주요 내용:**
  - 2018년 동시에 등장한 Quarkus와 Micronaut — 다른 출발점, 같은 답
  - Micronaut의 APT 기반 DI 코드 생성 — 리플렉션을 *원천에서* 제거
  - Quarkus의 augmentation phase — 빌드 페이즈 자체의 도입
  - "리플렉션은 본질적으로 느린 게 아니라, 정보가 부족해서 느린 것"이라는 통찰
  - Spring이 늦었던 이유 — 광범위한 reflection API 노출, 30년 호환성, BeanDefinition 모델
  - Spring의 응답 전략: 모델은 유지하고 *처리 시점*을 옮긴다 — Spring AOT의 설계 선택
  - **(신설) 반사실 사변 1~2쪽** — "만약 Spring이 2018년에 호환성을 깨고 갔다면? — 없었던 길에서 비춰보는 자산 마이그레이션 문제"
- **5단계 비용 다루는 부분:** C (주), 역사적 맥락
- **독자가 얻는 것:** 이후 모든 Spring 기술을 "왜 *이런 모양으로* 만들어졌는가"의 맥락에서 읽을 수 있는 시야.
- **예상 분량:** 30~32쪽 / 30,000자 *(반사실 절 추가로 v1 대비 +2,000자)*

### 4장. 첫걸음, 그러나 절반의 답: CDS 계보
- **핵심 질문:** 코드 한 줄 안 바꾸고도 시작 시간을 15~30% 줄일 수 있다면 — 왜 그리고 어떻게?
- **주요 내용:**
  - CDS (JDK 1.5) → AppCDS (JEP 310) → Dynamic CDS (JEP 350)의 진화
  - 무엇이 공유되는가: 클래스 메타데이터의 메모리 매핑
  - `-XX:ArchiveClassesAtExit`의 실제 워크플로우 — 훈련 단계, 운영 단계
  - Spring Boot 3.3의 `-Dspring.context.exit=onRefresh` 통합 — 빈 초기화까지만 돌리는 영리한 트릭
  - 실측: 4.9s → 2.4s (51% 감소) 사례 분석
  - 한계: JIT 코드는 안 공유, 메모리 절감은 제한적 — "절반의 답"인 이유
  - 컨테이너 이미지에 CDS 아카이브 빌드하기 (Maven/Gradle 통합)
- **5단계 비용 다루는 부분:** A, B
- **독자가 얻는 것:** Stage 0에서 당장 적용할 수 있는 가장 안전한 첫걸음. 그리고 왜 이걸로는 끝이 아닌지의 감.
- **예상 분량:** 25~28쪽 / 25,000자

### 5장. Spring AOT: 런타임을 빌드 타임으로 옮기다
- **핵심 질문:** Spring AOT는 *정확히 무엇을* 빌드 타임에 해주는가, 그리고 *어떻게*? — 그리고 30년 자산을 어떻게 들고 가는가?
- **주요 내용:**
  - **(신설 사이드바 ⓐ — 도입부) "왜 lazy init은 답이 아닌가" 1~2쪽** — Spring Boot 2.2의 `spring.main.lazy-initialization=true`가 시작 시간을 깎는 가장 직관적 답으로 보이지만 *왜 좋은 답이 아닌가*. 첫 요청 latency 증가, fail-fast 위배, 로드밸런서 readiness 신호 왜곡 (Reference 1.6/4.3 회수)
  - **(신설 사변 블록 ⓑ — 전반부 1/3) "Reflection: 30년 자산을 어떻게 마이그레이션하는가"** — Reference 4.2의 "리플렉션은 적인가 도구인가" 논쟁을 5장에서 *먼저* 던진다 (6장 Native 철학 챕터에서는 더 좁게 *closed-world 관점*만 다룸). Spring이 reflection API를 30년 노출했다는 부채, RuntimeHints API가 그 부채를 *재설계 없이* 다루려는 영리한 타협
  - `ApplicationContextAotGenerator`의 작동 — `refreshForAotProcessing(hints)`이 빈을 *정의만 하고 인스턴스화는 안 하는* 이유
  - 생성되는 세 산출물: `*__BeanDefinitions.java` 소스, RuntimeHints JSON, `ApplicationContextInitializer`
  - 핵심 API 투어: `BeanFactoryInitializationAotProcessor`, `BeanRegistrationAotProcessor`, `RuntimeHintsRegistrar`, `@Reflective`
  - **자주 놓치는 사실: JVM 모드(`-Dspring.aot.enabled=true`)에서도 동작한다** — Native 없이 10~20% 단축
  - 깨지는 패턴 갤러리: `BeanDefinitionRegistry`로 런타임 빈 등록, 인터페이스 리턴 타입의 `@Bean`, 환경변수 기반 `@Profile`, 람다 빈 supplier
  - `META-INF/spring/aot.factories`로 라이브러리 통합 — 본인 라이브러리에 힌트 부여하기
  - Stage 1 도입 가이드
- **5단계 비용 다루는 부분:** C (주력)
- **독자가 얻는 것:** Spring AOT를 *블랙박스에서 화이트박스*로 전환. 자기 코드가 AOT 친화적인지 판단하고, 깨지는 패턴을 미리 수정할 수 있다. *그리고 Spring AOT가 한 시대의 부채를 어떻게 타협으로 풀었는지*의 사변까지 얻는다.
- **예상 분량:** 38~40쪽 / 38,000자 *(v1 35,000자 → 사변 블록 + Lazy 사이드바로 +3,000자 증액)*

### 6장. closed-world의 결단: GraalVM Native Image의 철학
- **핵심 질문:** 왜 GraalVM은 "런타임에 로드될 코드는 빌드 시점에 다 알려져야 한다"고 요구하는가?
- **주요 내용:**
  - Substrate VM이 OpenJDK가 *아닌* 이유 — 별도 런타임이라는 결단
  - points-to analysis 개념적 설명 (Wimmer et al. OOPSLA 2019)
  - heap snapshotting — static initializer 결과를 빌드 타임 힙으로 직렬화
  - copy-on-write로 매핑되는 사전 채워진 힙 — 시작이 빠른 진짜 이유
  - reachability metadata 5개 섹션 (reflection / jni / resources / serialization / proxies)
  - 1st-party vs 3rd-party 힌트 — `oracle/graalvm-reachability-metadata` 리포지토리의 역할
  - **(5장과의 분업 명시) closed-world 관점에서의 reflection** — 5장에서 던진 30년 자산 사변을 *Native가 어떻게 더 좁게 좁히는가*로 받음
  - 시작 50ms의 의미 — 인터프리팅 단계가 없다는 것
- **5단계 비용 다루는 부분:** A·B·C·D 일괄 제거 (E는 남음)
- **독자가 얻는 것:** Native Image가 *어떤 결단을 요구하는지* 철학적으로 이해. 다음 챕터의 실전 트러블슈팅이 왜 그렇게 어려운지의 배경.
- **예상 분량:** 32~35쪽 / 32,000자

### 7장. Native Image 실전: 빌드 시간, 트러블슈팅, PGO
- **핵심 질문:** GraalVM Native Image를 실제로 운영할 때 무엇이 깨지고, 어떻게 진단하는가?
- **주요 내용:**
  - 빌드 시간 1~10분의 현실 — dev loop을 깨지 않는 CI 구조
  - 트러블슈팅 5선:
    1. `--verbose --no-fallback` 빌드 실패 진단
    2. `NoSuchMethodException` → `reflect-config.json` 보강
    3. 리소스 못 찾음 → 패턴 등록
    4. JDK 프록시 오류 → `registerJdkProxy`
    5. Tracing Agent(`-agentlib:native-image-agent`)로 자동 힌트 생성과 그 한계
  - PGO의 이론과 현실 — Enterprise 한정, instrumented 바이너리는 운영 불가, ML 추론 프로파일이 자동 대안
  - **Liberty Mutual 사례 심층 분석** — 5,770ms → 655ms (9배), OCI 컨테이너 회귀, profiles 동적 해석 불가
  - Spring profiles와 Native Image — 빌드 타임 고정 vs 환경변수 외부화
  - 한국 커뮤니티의 실패담 (velog, OKKY 인용) — 무엇이 깨졌고 왜 포기했나
  - 채택 결정의 첫 질문: "사내에 GraalVM 전문성을 가진 사람이 1명 이상 있는가?"
  - GraalVM 라이선스(Oracle GFTC, Enterprise vs CE) 1페이지 정리
- **5단계 비용 다루는 부분:** A·B·C·D (실전 트레이드오프)
- **독자가 얻는 것:** Native Image를 *낭만 없이* 본다. 어디서 멈춰야 할지, 어디까지 가도 되는지의 감각.
- **예상 분량:** 35~38쪽 / 35,000자

### 8장. 시간 여행: CRaC와 SnapStart, 살아 있는 JVM을 동결하다
- **핵심 질문:** "이미 다 시작된 상태"를 디스크에 저장했다가 그 지점부터 다시 시작할 수 있다면 — 무엇이 가능해지고 무엇이 깨지는가?
- **주요 내용:**
  - **(명시적 두 부 구조 — 챕터 도입에 길 안내)** "전반부는 EKS·on-prem 독자를 위한 CRaC, 후반부는 Lambda 독자를 위한 SnapStart. 두 길은 같은 기술적 뿌리(CRIU)에서 다른 운영 결과로 갈라진다."
  - **[전반부: CRaC 원리와 Spring 통합]**
    - CRIU의 원리 — 사용자 공간 체크포인트/복원이 어떻게 가능한가 (Galletta et al. 2024 인용)
    - CRaC의 `org.crac.Resource` API — `beforeCheckpoint` / `afterRestore`의 의미
    - Spring 통합의 영리함 — `Lifecycle` 인터페이스 재활용, Boot 3.4의 자동화
    - 가장 강력한 제약: 체크포인트 시점에 *열린 파일 핸들·소켓이 없어야 한다*
    - Spring PetClinic 84% 단축 (1,010ms) — baseline 대비 시각화
    - EKS·on-prem 운영 파이프라인: "체크포인트 → 배포 → restore"의 CI/CD 통합
  - **[후반부: SnapStart의 산업 확장]**
    - **AWS Lambda SnapStart** — Firecracker microVM 스냅샷 + 커스텀 CRaC 컨텍스트
    - SnapStart 함정 갤러리: `java.util.Random` 시드 공유, DB 커넥션 재사용, 캐싱된 timestamp
    - 보안 우려 — 시크릿이 체크포인트 이미지에 포함될 수 있는 위험 (Deleuze, Spring Blog 2023)
    - 프레임워크 비교 — Spring vs Quarkus vs Micronaut on SnapStart
    - "당신은 어느 부에 속하나" 길 안내 박스 (8장 마무리)
- **5단계 비용 다루는 부분:** A·B·C·D·E 일괄 우회
- **독자가 얻는 것:** "코드 변경 거의 없이 80%+ 단축"이라는 매력의 *진짜 비용*을 안다. Lambda냐 EKS냐로 길이 갈리는 분기점도.
- **예상 분량:** 38~40쪽 / 38,000자 *(v1 35,000자 → 두 부 구조 명시·길 안내 박스로 +3,000자)*

### 9장. warm-up의 시대: Azul, Alibaba, OpenJ9가 먼저 푼 문제 *[신설 챕터]*
- **핵심 질문:** OpenJDK가 늦은 동안 누가 어떻게 *D단계*를 풀었는가? 그들의 답을 우리는 왜 외면했고, 지금 무엇을 배워야 하는가?
- **주요 내용:**
  - 1막: warm-up이 비용이 된 산업 — 금융 트레이딩, 광고 입찰, 알리바바 더블 일레븐. *왜 같은 문제를 다른 산업이 따로 풀었나*
  - **Azul Zing/Prime** — Falcon JIT(LLVM 기반), ReadyNow의 프로파일 직렬화, Cloud Native Compiler의 *동일 메서드 100대 인스턴스 1회 컴파일* 패턴 (Reference 3.5)
  - **Alibaba Dragonwell + JWarmup** — 2016년 사내 시작, 2019년 GA. 사전 실행 단계에서 hot method 프로파일 기록, 정상 실행 시 main 호출 전 사전 컴파일. ElasticHeap과의 패키징 (Reference 3.3)
  - **Eclipse OpenJ9 Shared Class Cache + AOT** — `-Xshareclasses`로 *AOT 컴파일된 네이티브 코드까지* 공유 메모리 캐싱. 시작 42% 단축, 풋프린트 66% 감소. 운영 검증된 솔루션이 왜 채택률은 낮은가
  - **사변: "OpenJDK는 왜 늦었는가"** — 표준이라는 책임의 무게, 라이선스, 호환성 부담. OpenJDK가 늦은 동안 비주류 답이 *실전을 풀었다*는 산업사적 의미
  - **상용 vs 오픈소스의 결** — Azul은 상용·전문 지원, Dragonwell은 오픈소스·중국 생태계, OpenJ9은 오픈소스·IBM 색채. 도입 결정 시 사내 라이선스·지원 모델 고려
  - 다음 챕터로의 다리: "그리고 OpenJDK가 응답했다 — Project Leyden"
- **5단계 비용 다루는 부분:** D (전담 — 책 전체에서 유일하게 D를 *주력*으로 다루는 챕터)
- **독자가 얻는 것:** D단계가 *실재하는 비용*임을 안다. 상용·비주류 답안의 존재를 알고, 자기 워크로드에서 warm-up이 문제라면 Leyden 외에도 선택지가 있음을 안다. *산업사적 시야*도 보너스.
- **예상 분량:** 30~33쪽 / 30,000자 *(신설 — D단계 챕터 부재 비판에 대한 정공법 응답)*

### 10장. Leyden의 점진주의: condenser와 AOT 캐시의 길
- **핵심 질문:** Azul·Alibaba가 푼 문제에 OpenJDK는 어떻게 응답했는가? GraalVM의 결단 없이도 Native에 가까운 시작을 얻을 수 있는가?
- **주요 내용:**

  - JEP 515 (JDK 25) — AOT Method Profiling. JIT가 부팅 즉시 프로파일 사용 — *9장 ReadyNow/JWarmup의 OpenJDK 정공법 응답*

  - **GraalVM vs Leyden — 기술적 측면 비교만**

- **독자가 얻는 것:** Leyden을 GraalVM의 대안이 아니라 *다른 길*로 본다. 9장의 비주류 답안과 Leyden을 *같은 D단계 풀이의 다른 노선*으로 보는 시야. 점진 도입을 원하는 팀에게 첫 선택지.
- **예상 분량:** 32~35쪽 / 32,000자

### 11장. 결정 트리: 당신의 워크로드에 무엇이 맞는가
- **핵심 질문:** 여덟 가지 기술 앞에서, 내 프로젝트에는 무엇을 골라야 하는가? — 그리고 *잘못 고르면 무엇이 깨지는가*?
- **주요 내용:**
  - **(신설 — 도입 가상 케이스 스터디 5~6쪽)** "한 팀이 18개월 걸려 Native에 갔다가 결국 JVM으로 돌아온 이야기" — 가공된 1인칭 서사. 의도(cold start 단축), 초기 성공(스테이징에서 50ms), 균열의 시작(라이브러리 호환성, 빌드 시간 폭증, GraalVM 전문가 퇴사), 후퇴 결정, 그 결정에서 *얻은 것*. 가상임을 명시하되 Reference 7.1 한국 사례 부재의 정직한 보완
  - 5단계 비용 격자 위에 8개 기술 일괄 배치 (한 페이지 비교표 — CDS / Spring AOT / Native / CRaC / SnapStart / Azul / Dragonwell / Leyden)
  - "어디서 비용을 치를 것인가" 매트릭스 — 빌드 시간 / 메모리 / 시작 시간 / 피크 성능 / 디버깅 / 동적 기능 (Reference 2.1)
  - **결정 트리 본격 펼침** (리서치 4.1):
    - 분기 1: 짧은 수명 함수형 워크로드인가? → Native vs SnapStart
    - 분기 2: 긴 수명 백엔드 API인가? → CRaC vs Leyden vs 그냥 JVM+AOT
    - 분기 3: 배치/CLI인가? → CDS+Leyden
    - 분기 4: D단계가 1순위 병목인가? (트레이딩·광고 입찰) → Azul/Dragonwell의 비주류 답안
  - 추가 변수 4개: 개발팀 규모·역량 / 라이브러리 다양성 / 컴플라이언스 / 인프라(Linux 가능 여부)
  - 시나리오 사례 3개로 트리 실행:
    1. Lambda 위 결제 API
    2. EKS 위 사용자 프로필 마이크로서비스
    3. CronJob으로 도는 야간 정산 배치
  - **(신설) "분기를 잘못 탔을 때의 비용" 섹션 3~4쪽** — Reference 4.1 결정 트리의 각 분기를 *틀리게 갔을 때* 무엇이 일어나는가. Native를 골랐는데 라이브러리 다양성이 컸을 때, CRaC을 골랐는데 Linux가 아니었을 때, lazy init을 골랐는데 SLA가 엄격했을 때. *틀린 결정의 비용*을 가시화
  - **Netflix 사례 깊이 분석 — 메타 회고로 마무리** — 왜 그들은 Native를 *고르지 않았는가*, 그래도 15% 이득. "결정하지 않는 것도 결정이다"는 토비식 마침표
  - 단계별 도입 로드맵 (Stage 0~3)
- **5단계 비용 다루는 부분:** 모든 단계 종합
- **독자가 얻는 것:** 책의 클라이맥스. 자기 프로젝트에 대해 *직접 결정 트리를 그릴 수 있다*. 동료를 설득할 언어와 근거를 가진다. *틀린 결정의 비용*까지 가시화한 무게를 지닌다.
- **예상 분량:** 42~45쪽 / 42,000자 *(v1 38,000자 → 가상 케이스 스터디 + 틀린 결정 비용 섹션 추가로 +4,000자)*

### 12장. 진단의 기술: 결정 후, 그 결정이 옳았는지 확인하기
- **핵심 질문:** 이미 결정을 내렸다면, 그 결정이 옳았는지 어떻게 *측정*하나? — 그리고 내 앱의 5단계 비용을 어떻게 *수치로* 분해하나?
- **주요 내용:**
  - **(도입 프레이밍 — 11장과의 연결)** "결정은 끝났다. 이제 확인이다. 측정 없이 옳음을 주장할 수 없다."
  - 2장에서 약속했던 측정 도구의 *회수*
  - `BufferingApplicationStartup` + `/actuator/startup` — 빈별 트리 분석 실습
  - `FlightRecordingApplicationStartup` + JFR — 본격 프로파일링
  - `-Xlog:class+load=info` 로그 읽기 — B단계 측정
  - `-XX:+PrintCompilation` 로그 — D단계 JIT 측정
  - **async-profiler로 시작 flame graph** 그리기 (Krzysztof Slusarski 가이드 인용)
  - CDS 효과 측정 — `-Xshare:on`, `-XX:+PrintSharedArchiveAndExit`
  - Native Image에서의 진단 — JFR 제한, native-image-agent의 활용
  - K8s 측 진단 — HPA `cpu-initialization-period`, readiness probe의 함정
  - **트러블슈팅 시나리오 5개**: 빈 등록 폭발, 라이브러리 자동설정 누락, 컨테이너 cold cache, JIT thrashing, GC 첫 사이클
- **5단계 비용 다루는 부분:** 측정의 메타 — 모든 단계
- **독자가 얻는 것:** 측정 능력. "체감"이 아니라 "수치"로 자기 앱을 안다. *결정 후의 검증* 역량.
- **예상 분량:** 32~35쪽 / 32,000자

### 13장. 다음 10년: AOT Code, Loom, ZGC, 그리고 시작이라는 개념의 종말
- **핵심 질문:** 10년 뒤 JVM 애플리케이션은 어떻게 "시작"할까? 아니, "시작"이라는 단계가 남아 있긴 할까?
- **주요 내용:**
  - Leyden premain의 AOT Code Compilation — 컴파일된 네이티브 코드까지 캐시. OpenJDK가 GraalVM과 수렴하는 미래
  - **(반례형 사변) Virtual Threads와의 *관계 없음*** — "흔한 오해: Loom과 AOT는 같이 가야 cloud-native다. 그러나..." Loom은 *처리량* 문제, AOT는 *시작* 문제. 직접 관계 없음을 명시하고, 둘이 *공존했을 때* 어떤 결과가 나오는지를 사변으로
  - Virtual Thread의 함정 — `synchronized` 블록의 carrier pinning (시작 시간보다 운영에서 발견되는 함정)
  - ZGC·Shenandoah와의 상호작용 — JEP 483 + ZGC 호환성 (JDK 26), 저지연 GC와의 시너지
  - 미답 영역: 한국 비주류 라이브러리의 호환성, ROI 계산, 라이선스 변화 (Oracle GFTC)
  - 다음 세대 서버리스 — Firecracker microVM의 진화, Knative와의 통합
  - **마지막 회고:** 결국 우리가 하고 있는 일은, 컨테이너에 얹은 비용을 다시 줄이려는 노력일 뿐
  - 독자에게 남기는 한 문장: "당신의 부팅을 다시 보라. 그것은 시간이 아니라 결정의 합이다."
- **5단계 비용 다루는 부분:** 미래 — 단계 경계 자체의 흐려짐
- **독자가 얻는 것:** 책을 덮었을 때 *시야가 한 단계 위로 올라간* 감각. 다음 1~2년 follow-up의 길잡이. Loom과 AOT가 *짝이 아니라는* 반례 사변까지.
- **예상 분량:** 26~28쪽 / 26,000자
