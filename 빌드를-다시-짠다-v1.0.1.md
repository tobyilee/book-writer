## 이 책은 무엇인가

이 책은 `build.gradle.kts`를 의존성 한두 줄 추가하는 자리로만 써오던 Spring Boot 백엔드 개발자가 그 파일을 도구 그 자체로 키우는 여정을 다룬다. Gradle 9.5와 Spring Boot 4.0이 같은 시기에 한 번씩 큰 정리를 끝낸 지금, 8.x 시절의 관행을 그대로 9.x에 올리면 곳곳에서 미묘한 마찰이 생긴다. 어차피 한 번 다시 짤 거라면 새 위에 새것으로 짓는 편이 낫다 — 이 책은 그 새 짓기를 함께 한다.

전체가 `shop`이라는 가상 Spring Boot 앱 한 채를 따라간다. 1장의 `shop`은 의존성 한 줄짜리 Hello World 빌드 스크립트다. 18장의 `shop`은 멀티 모듈로 쪼개져 있고, `buildSrc`와 `build-logic` included build로 build logic이 모듈화돼 있고, 커스텀 Gradle 플러그인이 git SHA를 자동으로 박아주고, Configuration Cache로 incremental 빌드가 빠르고, GitHub Actions에 Build Scan과 dependency graph가 자동으로 올라가고, GraalVM Native Image로 OCI 컨테이너에 들어가고, 의존성은 verification metadata와 lockfile로 묶여 있다. 같은 앱이 자라는 흐름을 따라가는 동안 독자는 두 가지를 동시에 얻는다 — Gradle 9.5와 Spring Boot 4.0의 정확한 사고 모델, 그리고 회사 빌드를 한 단계 위로 올릴 도구 상자.

레퍼런스가 아니라는 점은 일찍 분명히 해둔다. 레퍼런스가 필요하면 공식 Gradle User Guide가 가장 좋다. 이 책이 하려는 일은 그 레퍼런스의 단어들이 한 앱이 자라는 흐름 안에서 어떤 자리에 들어가는지를 자기 언어로 잡는 것이다. 모든 챕터에 실제로 빌드되는 코드가 동반되고, 챕터마다 "이 챕터에서 자라는 앱 상태"를 한 줄로 박아둔다.

## 누구를 위한 책인가

Spring Boot로 개발은 해봤지만 `build.gradle.kts`는 의존성 한두 줄 더하는 영역으로만 써온 백엔드 개발자를 정면으로 끌어안는다. 특히 다음 셋 중 하나라도 짚이는 사람.

- Maven 출신이라 Gradle의 멘탈 모델이 어색한 사람. `dependencyManagement`가 어디로 갔는지, `parent pom` 같은 게 있는지, `mvn`과 `./gradlew`의 사고방식이 어떻게 다른지 매번 다시 검색한다.
- 멀티 모듈을 시도했다가 `bootJar` 함정에 한 번 빠져본 사람. library 모듈에서 빈 jar가 나왔고, 그 사고가 왜 일어났는지 명확히 짚지 못한 채 어찌어찌 넘긴 적이 있다.
- Configuration Cache라는 단어를 들어봤지만 켤 엄두를 못 낸 사람. 켜면 회사 빌드가 깨질 것 같아서, 또는 위반을 진단할 도구를 모른다.

진입 상태는 의존성 한 줄 추가 수준의 빌드 스크립트 사용자이고, 출구 상태는 멀티 모듈을 Convention Plugin으로 정리하고, Configuration Cache 위반을 진단해 호환되게 고치고, 커스텀 Task와 Plugin을 Provider/Property 기반으로 만들고, GraalVM Native와 Dependency Verification까지 운영 환경에 적용할 판단 기준을 갖는 빌드 운영자다.

## 무엇을 얻게 되는가

- Gradle의 Settings · Project · Task · Configuration · 3-phase Lifecycle을 자기 언어로 설명할 수 있다.
- Kotlin DSL의 lazy Property/Provider API로 build script를 쓰고, eager 패턴을 골라낼 수 있다.
- Version Catalog + BOM(platform) 조합으로 의존성을 다스리고, `io.spring.dependency-management` 플러그인의 자리를 안다.
- 멀티 모듈 Spring Boot 프로젝트를 `subprojects {}` 안티패턴 없이 Convention Plugin으로 정리한다.
- `bootJar` 함정을 5분 안에 진단하고 두 가지 처방 중 적합한 것을 고른다.
- Configuration Cache를 켜고, 위반 로그를 읽고, 자기 빌드를 호환되게 고친다.
- 커스텀 Task와 Plugin을 Provider/Property API 기반으로 만들고, git SHA 자동 주입 같은 실용 plugin을 직접 짠다.
- GitHub Actions에 `gradle/actions/setup-gradle`로 빌드 캐시 · Build Scan · Dependency Graph를 붙인다.
- GraalVM Native Image, Dependency Verification, Locking, Repository Content Filtering을 운영 환경에 적용할 판단 기준을 갖는다.
- Gradle 9.x 마이그레이션 체크리스트를 자기 회사 빌드에 적용한다.

## 차례

### Part I — Gradle 사고방식

1. **왜 build.gradle.kts를 다시 짜야 하는가** — 빌드 스크립트가 곧 운영의 일부인 이유, Gradle 9.5 + Spring Boot 4.0의 현재, 책 전반의 9가지 약속.
2. **Gradle의 사고 모델 — Settings, Project, Task, Lifecycle** — Gradle은 빌드를 어떻게 객체로 본 다음 task graph로 옮기는가.
3. **Kotlin DSL 그리고 그 함정** — Groovy DSL과 미묘하게 다른 지점, type-safe accessor의 등장 조건, `=` vs `.set()`.

### Part II — 단일 모듈 Spring Boot를 제대로 빌드한다

4. **Maven에서 옮겨오는 다리** — `pom.xml`의 모든 개념이 Gradle에서 어디로 가는지 1:1로 매핑한다.
5. **의존성을 다스리는 정석 — Version Catalog와 BOM** — declaration과 resolution은 직교한다. 둘 다 쓴다.
6. **Spring Boot Gradle 플러그인의 내부 — bootJar, bootRun, bootBuildImage** — 플러그인 한 줄이 정확히 무엇을 일으키는가. Spring Boot 3.x → 4.x 마이그레이션 포함.
7. **테스트를 분리한다 — JVM Test Suite로 통합 테스트** — 단위 테스트와 통합 테스트를 같은 `test` task에 욱여넣지 않는 법.

### Part III — 규모를 키운다

8. **모듈을 쪼갠다 — settings 구조와 프로젝트 간 의존** — 단일 모듈을 `app + domain + order + payment`로 옮긴다 (그리고 9장의 미스터리를 남긴다).
9. **bootJar 함정** — library 모듈에서 빈 jar가 나오는 사고를 5분 안에 진단하고 두 가지 처방으로 막는다.
10. **Convention Plugin으로 build logic을 모듈화한다** — `subprojects {}` 안티패턴 없이 4개 모듈의 build.gradle.kts를 한 줄로 줄인다.
11. **buildSrc 한계를 넘는다 — build-logic included build와 Composite Build** — buildSrc 변경 시 전체 out-of-date를 끊고, 외부 라이브러리와 동시 개발하는 패턴까지.

### Part IV — 빌드를 도구로 만든다

12. **커스텀 Task와 Plugin — Provider/Property로 lazy하게** — 빌드 스크립트의 소비자에서 생산자로. git SHA 자동 주입 plugin을 처음부터 만든다.
13. **Configuration Cache를 켠다** — 위반을 진단하고 자기 빌드를 호환되게 고치는 단계별 처방.
14. **CI에 올린다 — GitHub Actions와 Build Scan** — `gradle/actions/setup-gradle`로 빌드 캐시 · Build Scan · Dependency Graph를 한 번에.

### Part V — 운영의 무게

15. **GraalVM Native Image로 패키징한다** — JIT vs Native의 트레이드오프, 운영 환경 패키징 판단 기준.
16. **의존성 보안 — Verification, Locking, Repository Content Filtering** — supply chain을 잠그는 운영 패턴 셋.
17. **Gradle 9.x로 옮긴다 — 마이그레이션 노트 모음** — 7.x · 8.x 빌드를 9.x로 안전하게 올리는 체크리스트.
18. **이제 어디로 갈 것인가** — Isolated Projects, Develocity, 책에서 다루지 않은 영역 안내와 다음 단계.

부속으로 **머리말 · 차례 · 맺음말 · 참고문헌 · 판권**.

## 표기·실습 컨벤션

- **Kotlin DSL이 기본이다.** Groovy DSL은 차이가 클 때만 박스로 짧게 병기한다.
- **마이그레이션 노트 박스 / Maven 비교 박스 / 함정 박스**로 흐름과 디테일을 분리한다. 박스를 건너뛰어도 본문은 매끄럽게 흐른다.
- **모든 챕터의 코드는 `ch{NN}-{slug}/` 폴더에 산다** — `ch01/`, `ch04-bootapp/`, `ch08-multimodule/`, `ch09-convention/`, `ch11-composite/`, `ch12-custom-plugin/`, `ch13-config-cache/`, `ch14-ci/`, `ch15-native/`, `ch16-security/`. 챕터마다 그 챕터가 어느 폴더에서 동작하는지 도입부에 박아둔다.

## v1.0.1 변경 사항

v1.0.0 대비 두 군데의 사실 검증이 마무리됐다.

- **9장 사고 회고 박스** — "한국의 한 결제 SaaS 팀" 같은 미검증 일화 인용을 빼고, 공개 출처(Spring Boot 이슈 [#16689](https://github.com/spring-projects/spring-boot/issues/16689), Gradle Discuss 포럼 [thread 41459](https://discuss.gradle.org/t/41459))를 인용해 같은 사고가 어디서나 반복되는 패턴임을 강조하는 형태로 재작성.
- **15장 `RuntimeHintsAgent` 박스** — agent의 정확한 좌표를 `org.springframework:spring-core-test`로 확정하고, Spring Boot 4가 전용 Gradle 플러그인을 따로 노출하지 않는다는 점을 반영해 `-javaagent`로 직접 붙이는 Kotlin DSL 예제로 교체. `@EnabledIfRuntimeHintsAgent` 어노테이션 사용 패턴 추가.

본문 챕터 구성과 분량은 v1.0.0과 동일하다.
