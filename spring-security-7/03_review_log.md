# 저술 계획 리뷰 — 1차

## 종합 평가

**강점**
- 리서치 §1~§9의 거의 모든 토픽이 챕터로 매핑돼 있다. 매핑 누락이 거의 보이지 않고, §5 매핑 절(계획 §5)이 명시적이라 추적 가능하다.
- 토대(2·3장) → 시나리오(4~7장) → 인가(8장) → 횡단(9·10·11장) → 마이그레이션(12장) → 트렌드+통합(13장) 순서가 토비 시리즈 특유의 "어휘 먼저, 시나리오 다음" 흐름을 잘 잇는다. 2·3장에서 만든 어휘로 이후 챕터가 말하게 한다는 의도가 명확하다.
- 7.0의 핵심 신기능(Passkeys/OTT/MFA, DPoP, `AuthorizationManager` 신모델, `csrf(spa)`)이 각자의 자리를 갖고 흩어져 있지 않다. 특히 MFA를 7장 안에서 OTT·Passkeys와 묶어 "패스워드 너머"라는 한 챕터 콘셉트로 모은 결정은 좋다.
- 실무 함정(§7)을 각 시나리오 챕터 머리에 분산 배치한 설계가 토비 톤(공감 진입)과 잘 맞는다. "찜찜하다", "난감하다" 같은 정서 어휘를 챕터 머리에 깔겠다는 §6 메모도 일관된다.
- 차별점 6가지가 구체적이다. 특히 5/6→7 마이그레이션을 별도 챕터로 빼고, Passkeys/OTT/MFA를 표면 소개에 그치지 않고 ceremony·운영 함정까지 가겠다고 약속한 부분은 시중 책과 분명히 다르다.

**약점**
- **테스팅 챕터가 통째로 빠져 있다.** `@WithMockUser`, `SecurityMockMvcRequestPostProcessors`, OAuth2 클라이언트/자원 서버 테스트 지원, Reactive 보안 테스트(`WebTestClient` mutateWith)는 시니어가 매일 부딪히는 영역인데 13개 챕터 어디에도 절(節) 단위로 잡혀 있지 않다.
- **13장이 과적재됐다.** Zero Trust + BFF + DPoP + Secret 회전 + Actuator + 관측·감사 + 논쟁점 + 통합 실전. 44~50p로 한 챕터에 묶기엔 성격이 다른 다섯 묶음이 한 자리에 모인 모양새다.
- **마이그레이션(12장)의 위치가 문제다.** 본 책의 명시적 1차 독자는 "Spring Security 5/6을 표면적으로만 써본 사람"이다. 이 독자가 7.0으로 점프할 때 가장 먼저 부딪히는 빨간 빌드 에러는 책 후반까지 미뤄둘 수 없다.
- **"보안 기본 지식"이 1장에서 너무 가볍게 처리된다.** 1장 핵심 질문이 이미 "7.0 GA가 가져온 변화"로 좁혀져 있어, OWASP A01·A02·A07 / 인증 vs 인가 / 세션 vs 토큰 트레이드오프 / Zero Trust 멘탈 모델 같은 보안 사고의 좌표가 처음 등장할 자리가 없다. 13장에서 Zero Trust를 처음 만나는 건 늦다.
- **8장(인가)이 4·5·6·7장의 인증 결과물(권한·SCOPE·Authority)을 모두 받아 처리하기에는 단일 챕터다.** URL + Method + Role Hierarchy + 도메인 객체 + OWASP A01까지 한 챕터에 응축됐는데, 4~7장이 만들어낸 권한 매핑 다양성을 고려하면 분량 34~40p로 빠듯하다.

**한 줄 권고: 부분 수정.** 큰 골격은 통과. 단 (1) 13장 분할, (2) 마이그레이션 챕터 위치 재고, (3) 테스팅 챕터 신설, (4) 1장에 "보안 멘탈 모델" 절 보강 — 이 넷은 필수.

## 구체적 피드백

### 1. 커버리지 누락

리서치 §1~§9 매핑 자체는 촘촘하다. 다만 다음이 챕터 단위로 잡히지 않았다.

- **테스팅.** 리서치에는 명시적 §이 없지만 시니어 독자의 "내부 아키텍처 + 시나리오별 활용을 깊이 이해하고 싶다"는 출구 상태를 충족하려면 테스트가 빠질 수 없다. Spring Security 공식 문서의 testing 절(`@WithMockUser`, `@WithUserDetails`, `SecurityMockMvcRequestPostProcessors`, `oauth2Login()` / `jwt()` mock, Reactive `mutateWith`)이 본 계획 어디에도 챕터/절로 잡혀 있지 않다. 이건 단순 누락이지 의도된 배제로 보기 어렵다.
- **HTTPS/TLS 운영 맥락.** 9장에서 HSTS는 다루지만, "왜 HSTS가 HTTPS 종단을 가정하는지", "역프록시·로드밸런서 뒤에서 `X-Forwarded-Proto` 처리", 7.0이 도입한 `LoginUrlAuthenticationEntryPoint`의 상대 경로 리다이렉트 같은 운영 맥락(리서치 §3.2 Web)이 본문에서 어디에 사는지 불분명하다.
- **Custom Filter 추가 패턴.** 4.12절의 "커스텀 `AuthenticationProvider`"는 3장에 흡수되지만, "커스텀 `Filter`를 체인 어느 자리에 끼울 것인가"(`addFilterBefore`/`addFilterAfter`)는 시니어가 자주 묻는 질문인데 챕터 단위 답이 보이지 않는다. 2장에서 한 절로 잡으면 좋다.
- **Error Handling 일관 처리.** `AuthenticationEntryPoint`, `AccessDeniedHandler` 커스터마이즈, ProblemDetail(`@ControllerAdvice`)과의 정합은 4·5·8장 어디에 살아야 하는지 불분명하다. 9장에 한 절로 묶거나 8장 끝에 두는 게 자연스럽다.
- **GraphQL/gRPC 보안.** 본 책 범위 밖으로 의도적으로 배제했다면 1장 사용법 안내에 그 사실을 한 줄 명시하길 권한다. 시니어 독자가 "이 책으로 GraphQL 보안도 답을 얻을 수 있나" 기대했다가 끝까지 못 찾는 경험은 피해야 한다.

### 2. 챕터 균형 문제

- **13장(44~50p)이 가장 과적재.** Zero Trust 멘탈 모델 + BFF + DPoP + Secret 회전 + Actuator + 관측·감사 + 논쟁점 + 통합 실전 — 한 챕터의 약속이 8개. 토비 시리즈 평균(한 챕터 한 메시지)에 비추면 너무 많다. 후반 절반이 "통합 실전 워크스루"인데 이게 사실상 별도 챕터급 분량이다.
- **2장(36~42p)과 5장(36~42p)이 토대/시나리오 챕터 중 가장 무겁다.** 2장은 토대로서 적정. 5장은 JWT 함정 4건이 본문에 통합돼 있어 무겁지만 "가장 위험한 토픽이라 가장 두껍게"라는 의도가 적절하다. 둘 다 통과.
- **7장(38~44p)이 OTT + Passkeys + MFA 세 모델을 한 챕터에 묶는다.** 세 모델 모두 ceremony·운영 함정까지 깊이 가겠다는 약속이라면 분량이 빠듯하다. 다만 "패스워드 너머의 인증"이라는 메시지의 응집도가 높아 한 챕터 유지에 찬성. 단 절 분량 균형은 저술 시 면밀히 봐야 한다(셋이 1:1:1이 아닐 가능성 큼).
- **8장(34~40p)이 인가의 모든 것을 담는다.** URL + Method + Role Hierarchy + 도메인 객체 + OWASP A01 + 논쟁점(SpEL 잠식) — 5묶음이 한 챕터. Method Security 함정(§7.1 함정 2)까지 본문에 들어간다. 분량 34~40p가 빠듯해 보인다.
- **9·10장 통합 가능성?** 9장 24~28p + 10장 26~30p = 50~58p. 둘 다 "횡단 관심사"라는 이름표를 공유하지만 합치면 13장만큼 과적재된다. 분리 유지가 옳다. 다만 두 챕터 사이의 "다음 챕터 예고" 연결을 명시적으로 두자.

### 3. 내러티브 흐름

도입 → 토대 → 시나리오 → 인가 → 횡단 → 마이그레이션 → 트렌드 흐름 자체는 자연스럽다. 다만 두 군데가 걸린다.

- **12장 마이그레이션의 후반 위치.** "5/6 코드 빨갛게 물드는 경험"을 1장에서 정서적으로 깔아두고 그 답을 12장까지 미루는 건 길다. 1차 독자가 책을 펴는 1차 동기는 "내 6.x 코드를 7로 옮기는 법"일 가능성이 높다. 그 답이 책 12/13에 가서야 정리된다면, 도중에 책을 덮을 위험이 있다. 두 가지 해결책 중 하나:
  - (A) 1장에 "5/6 → 7 핵심 변경 한눈 요약" 박스를 두고, 12장 본편을 그대로 후반에 두기 (현재 안의 보강)
  - (B) 마이그레이션 전 페이지를 별도 챕터 2번째 자리(현 2장 앞 또는 3장 뒤)로 끌어올리기 — 단 이 경우 2·3장 어휘 없이 마이그레이션 표를 읽어야 해서 비추.
  - **권고: (A).** 본편 12장은 유지하되, 1장에 "이 책을 어디서부터 읽을 것인가" 챕터 의존도 지도와 함께 마이그레이션 한눈 요약을 둔다.
- **10장(세션·쿠키·컨텍스트)과 11장(Reactive) 사이의 흐름.** 10장 끝의 "Reactive로의 다리"가 잘 쓰여 있다. 다만 9장(CSRF·CORS·헤더)이 10장 앞에 오는 게 자연스러운지는 한 번 더 고민. CSRF는 세션 흐름의 산물이다. 즉 10장(세션)을 9장(CSRF) 앞에 두는 것도 합리적. 단 9장의 `csrf(spa)` 한 줄과 13장 BFF의 연결을 끊지 않으려면 현 순서 유지가 더 낫다. 통과.

### 4. 선행 지식 의존도

큰 문제 없다. 다만 두 군데를 짚는다.

- **3장의 `AuthorizationManager` 어휘가 2장에서 미리 언급된다.** 2장 필터 11번 안에 `AuthorizationFilter`가 등장하는데, 그 시점에서 `AuthorizationManager`가 무엇인지 독자는 모른다. 2장에서 `AuthorizationFilter` 설명을 "이름만 짚고 3장에서 본격적으로"로 명시적으로 차단해 두는 게 좋다.
- **7장 MFA가 8장 `AuthorizationManagerFactory.allAuthorities`를 미리 쓴다.** 계획 §7장 "선택적 MFA" 절에서 `AuthorizationManagerFactory.allAuthorities("FACTOR_PASSWORD", "FACTOR_OTT")` 코드가 등장하는데, 이 빌더의 의미는 8장에서 본격적으로 다뤄진다. 7장 시점에 독자가 처음 만나면 어색하다. 7장에서 이 코드는 "8장에서 자세히 다룰 `AuthorizationManagerFactory`의 미리보기"라고 명시하거나, 8장에 짧은 박스 cross-ref를 두라.

### 5. 차별점 검증

차별점 6가지를 챕터들이 실제로 입증하는가?

1. **"7.0 GA 기준 첫 한국어 심층 레퍼런스"** — 1장+12장+13장+7장 조합으로 OK. 통과.
2. **"5/6→7 마이그레이션 함정 별도 챕터"** — 12장으로 OK. 다만 §3에서 지적했듯 1장 보강이 필요.
3. **"Passkeys, OTT, MFA, DPoP 각각 별도 챕터"** — **여기서 약속과 실제가 어긋난다.** 계획은 차별점에서 "각각 별도 챕터"라고 썼지만, 실제 챕터는 7장에 OTT+Passkeys+MFA 셋을 묶었고 DPoP는 13장에 한 절로 들어간다. 두 가지 정합 방법:
  - (A) 차별점 문구를 "Passkeys/OTT/MFA를 한 챕터에 1급으로 묶고, DPoP는 자원 서버 보호의 핵심 기법으로 13장에 둠"으로 정직하게 수정
  - (B) DPoP를 5장(JWT 자원 서버) 안의 한 절로 끌어올림 — DPoP는 본질적으로 자원 서버 토큰 검증 강화이므로 5장이 자연스러운 자리
  - **권고: (B)+(A) 혼합.** DPoP는 5장으로 옮기고, 13장에는 Zero Trust 원칙 4(sender-constrained)와 DPoP의 연결만 짧게 cross-ref. 차별점 문구는 그대로 두되 7장 묶음 모델을 분명히.
4. **"Reactive 보안 별도 챕터"** — 11장. 통과.
5. **"실무 함정 사례를 챕터 머리에 배치"** — 4·5·6·8·9·10·12장 머리에 분산 배치. 통과. 다만 §6 톤 메모의 "챕터 머리에 이런 상황 가정해보자 한 단락"이 실제 챕터별로 어떤 함정과 짝지어지는지 1차 저술 전에 한 번 더 매핑하라.
6. **"최종 챕터에서 SPA+BFF+자원 서버+IdP 통합 시나리오"** — 13장 후반. 통과. 단 §2의 13장 과적재 문제 해결 필요.

### 6. 누락 가능 토픽 점검

- **Testing (`@WithMockUser`, `SecurityMockMvcRequestPostProcessors`)** — **누락.** 신설 권고.
- **HTTPS/TLS, 역프록시 헤더 처리** — 부분 누락. 9장 보강 또는 13장 운영 절에 흡수.
- **Custom `Filter` 추가** — 부분 누락. 2장 끝 한 절로 권고.
- **Error Handling 일관 처리** — 부분 누락. 9장 또는 8장 끝에 한 절로 권고.
- **GraphQL/gRPC 보안** — 의도적 배제로 보임. 1장에 "이 책이 다루지 않는 것" 한 줄 명시 권고.
- **Spring Authorization Server 서버 측 운영** — 부분 다룸(1장·12장·13장 논쟁점). 본격 운영(클라이언트 등록·grant 구현·token customization)은 다루지 않음. 의도적이라면 1장 명시. "리서치 한계 §11.2/§11.6" 인용하면 정직하다.
- **Cloud(K8s/Spring Cloud)와의 통합** — Spring Cloud Gateway BFF가 13장에 나오지만 K8s Service Mesh/mTLS와의 관계는 빠짐. 후반 트렌드 박스로 한 단락이면 충분.
- **"보안 기본 지식"(OWASP/인증vs인가/세션vs토큰)** — 1장에 한 줄씩 흩어져 있을 뿐, 전용 절이 없다. 1장에 "보안 멘탈 모델 5분" 절을 새로 두거나, 2장 머리 박스로 두라.

### 7. 챕터별 분량 적절성 — 13장 특별 검토

13장 44~50p는 책 평균(32p)의 1.5배. 절 구성을 보면:

- Zero Trust 7 tenets 매핑 (이론)
- BFF 패턴 (구조)
- DPoP (RFC 9449 기반 구현)
- Secret/키 회전 (운영)
- Actuator 사고 사례 (운영)
- 관측·감사 (운영)
- 논쟁점 정리 (메타)
- 통합 실전 워크스루 (실습) ← 사실상 별도 챕터

**분할 권고:**
- 13장: "Zero Trust·BFF·DPoP — 트렌드와 표준" (Zero Trust + BFF + DPoP만, 28~32p)
- 14장: "운영의 모서리 — 키 회전·Actuator·관측·감사" (또는 12장과 13장 사이) (20~24p)
- 15장: "한 권을 한 시나리오로 — 통합 실전 워크스루" (28~32p)

**대안(보수적 분할):**
- 13장: 트렌드(Zero Trust + BFF + DPoP + 논쟁점) — 30p
- 14장(신설): 통합 실전 워크스루 — 22~26p
- 운영(키 회전·Actuator·관측) 묶음은 14장 후반에 운영 체크리스트로 흡수

분할 후 책 전체 분량은 460~540p로 살짝 늘지만 챕터별 약속이 또렷해진다.

### 8. 마이그레이션 챕터(12장) 위치 — 재논의

§3에서 지적한 (A) 안을 구체화한다.

- **현 12장은 위치 유지.** 본문 전체에서 분산된 변경점을 종합하는 "재방문" 챕터 성격이 강하고, 토대(2·3장) 어휘 없이는 deprecation 표가 의미를 못 가진다.
- **1장에 신설할 절(節):** "5/6 코드는 7에서 무엇이 빨갛게 변하는가 — 핵심 함정 5건 미리보기"
  - 리서치 §7.1의 함정 1~5(Stack Overflow 2019 복붙, `@EnableGlobalMethodSecurity`, `.and()`, mvc/ant 혼합, `WebSecurityCustomizer` 누락)를 표로 한눈에
  - 각 함정의 "본격 답"이 12장 어느 절인지 페이지 cross-ref
  - 이로써 5/6 출신 독자가 1장만 읽고도 "내 빨간 줄을 어디서 답 찾을 수 있다"는 안심을 얻는다
- **추가: 책 앞부분 "이 책을 어디서부터 읽을 것인가" 챕터 의존도 지도(이미 1장 계획에 있음)에 두 가지 독자 경로를 명시:**
  - 경로 1 (처음 배우는 독자): 1→2→3→4→5→...→13장 통독
  - 경로 2 (5/6 마이그레이션 독자): 1→12→2→3→...→13장
  - 경로 3 (특정 신기능만): 1→2→3→해당 챕터

### 9. Toby 톤과 정합성

대체로 좋다. 챕터 제목과 핵심 질문이 문답식·평어체 톤과 잘 맞는다. 다만 두 군데가 걸린다.

- **8장 핵심 질문 "비즈니스 규칙이 어디에 살아야 하는가"** — 좋다. 그러나 챕터 제목 "인가의 모든 것 — URL·Method·Role Hierarchy·도메인 객체"가 핵심 질문보다 카탈로그식이다. 토비 톤이라면 제목도 질문을 향해야 한다. 대안:
  - "인가는 어디에 살아야 하는가 — URL부터 도메인 객체까지" (질문형)
  - "권한은 어디에 사는가 — 인가의 5층 구조" (수사적)
- **9장 제목 "CSRF·CORS·보안 헤더 — 가장 헷갈리는 셋"** — 좋다. 통과.
- **13장 제목 "Zero Trust·BFF·DPoP — 그리고 모든 것을 합친 한 시나리오"** — 두 가지를 한 제목으로 묶었는데, §7에서 권고한 분할을 따르면 자연스럽게 풀린다.
- **5장 핵심 질문 "issuer-uri 한 줄이 어떻게 안전한 자원 서버를 만들고, 무엇을 안 막아 주는가?"** — 토비 톤의 정점. 그대로.

### 10. 시중 책 대비 차별점 — 한 번 더 가혹하게

Baeldung 식 튜토리얼은 "이 코드를 이렇게 짜면 동작한다"에서 멈춘다. Spring in Action은 "각 기능을 한 챕터씩 둘러본다". 본 책의 약속은 그것보다 두 칸 깊게 — "왜 그렇게 동작하는가" + "어디서 멈춰야 하는가" + "현장에서 어떻게 실패하는가" + "7.0에서 어떻게 다시 그렸는가".

이 약속을 실제 챕터가 입증하려면:

- 매 챕터의 "함정" 절이 단순 코드 비교를 넘어 "왜 그렇게 짜는 게 자연스러워 보였는가"의 심리까지 짚어야 한다 (토비 시리즈 특유). 계획 §6 톤 메모의 "찜찜하다/난감하다"가 이 약속의 부분 이행.
- 매 시나리오 챕터 끝의 "이 모델은 어떤 위협을 막고 어떤 위협엔 무력한가" 정리(§2 내러티브 메모)가 약속의 핵심. 모든 시나리오 챕터(4·5·6·7장)에 일관되게 들어가는지 저술 단계에서 확인.
- 13장의 통합 실전이 진짜로 "처음부터 끝까지 빌드"인지, 아니면 "각 챕터 발췌 모음"인지가 차별점의 시금석. 13장 분할(§7) 후 통합 실전 챕터를 별도로 두면 그 약속을 지키기 쉬워진다.

## 우선순위가 매겨진 변경 요청

1. **[필수] 13장을 분할.** 44~50p 한 챕터 안에 트렌드 + 운영 + 통합 실전이 함께 들어가는 건 토비 시리즈의 "한 챕터 한 메시지" 원칙에 어긋난다.
   - **어떻게:** 13장(트렌드: Zero Trust+BFF+DPoP+논쟁점, 28~32p) + 14장 신설(통합 실전 워크스루, 22~28p). 운영 묶음(키 회전·Actuator·관측·감사)은 14장 후반 "운영 체크리스트" 절로 흡수.
   - **왜:** 책의 클로징 메시지("이 책을 덮으면 무엇을 만들 수 있는가")를 통합 실전 챕터가 단독으로 책임지게 하면 약속이 또렷해진다.

2. **[필수] 테스팅 챕터 또는 절을 신설.** 시니어 독자의 "심층 활용" 약속에서 테스트 누락은 빈틈이다.
   - **어떻게(권고 A):** 새 챕터 "Spring Security 테스팅 — `@WithMockUser`부터 OAuth2 mock까지" (20~24p)를 8장과 9장 사이에 삽입. 인가 다룬 직후 자연스러움.
   - **어떻게(권고 B, 현 골격 보존형):** 8장 끝에 한 절(6~8p) "인가 결정을 어떻게 테스트하는가" + 5장 끝에 한 절 "JWT 자원 서버 테스트". 다만 Reactive 보안 테스트는 11장 끝에 별도 절 필요.
   - **권고: A.** 챕터 수가 14 → 15(분할 포함)로 늘지만, 시니어 독자에게 가장 자주 인용될 챕터가 될 가능성 높음.

3. **[필수] 1장에 "5/6 코드는 7에서 무엇이 빨갛게 변하는가" 절 추가.**
   - **어떻게:** 1장 본문에 리서치 §7.1 함정 1~5 한눈 표 + 12장 cross-ref. 챕터 의존도 지도에 "5/6 마이그레이션 독자 경로"를 명시.
   - **왜:** 1차 독자가 책 12장까지 기다리지 않고도 1장에서 "내 빨간 줄의 답이 12장에 있다"는 안심을 얻게 한다.

4. **[필수] 1장에 "보안 멘탈 모델 5분" 절 신설.**
   - **어떻게:** OWASP Top 10 2021(A01·A02·A07) + 인증 vs 인가 + 세션 vs 토큰 + Zero Trust 한 줄 정의를 3~4p로 응축. 13장에서 Zero Trust를 처음 만나는 게 아니라, 1장에서 좌표를 박은 뒤 13장에서 그 좌표 위에 Spring Security 매핑을 얹는 흐름.
   - **왜:** 책 전체의 보안 사고 좌표축을 1장이 책임지게 해서, 이후 모든 챕터의 결정이 그 좌표 위에서 읽힌다.

5. **[필수] DPoP를 5장으로 이동.**
   - **어떻게:** 13장의 DPoP 절을 5장 후반(JWT 자원 서버) "토큰 sender-constraining — DPoP" 절로 옮긴다. 13장에는 "Zero Trust 원칙 4와 DPoP의 연결"만 cross-ref로 한 단락.
   - **왜:** DPoP는 본질적으로 자원 서버 토큰 검증 강화 메커니즘이다. 5장 "JWT 함정 9건" 다음 절로 두면 토큰 보호의 자연스러운 진화 흐름이 된다. 13장 분할에도 도움.

6. **[권고] 8장 분할 검토.** 분량 34~40p에 5묶음을 응축하면 빠듯하다.
   - **어떻게:** 8장(URL + Method + Role Hierarchy, 26~30p) + 8.5장 또는 9장(도메인 객체 보안 + OWASP A01 + 논쟁점, 16~20p) 분할. 또는 8장 분량을 38~44p로 늘리는 방식.
   - **왜:** 인가는 본 책의 사실상 두 번째 토대(첫 번째는 2·3장). 응축보다 펼치는 게 토비 시리즈답다.

7. **[권고] 7장 MFA 절에서 `AuthorizationManagerFactory` 미리보기 처리 명시.**
   - **어떻게:** 7장 MFA 절 마지막에 "이 `allAuthorities`의 작동 원리는 8장에서 자세히"라는 한 단락 + 8장에 "MFA에서 미리 본 `AuthorizationManagerFactory`를 다시" 한 단락 cross-ref.
   - **왜:** 선행 지식 의존도(§4) 문제 해결.

8. **[권고] 2장에 "커스텀 `Filter`는 어디에 끼우는가" 절 추가.**
   - **어떻게:** 2장 끝(필터 11종 설명 직후)에 `addFilterBefore`/`addFilterAfter` + Order 결정 패턴 한 절(4~6p).
   - **왜:** 시니어가 자주 묻는 질문. 2장 어휘 직후가 가장 자연스러운 자리.

9. **[권고] 9장에 "Error Handling 일관 처리" 한 절 추가.**
   - **어떻게:** `AuthenticationEntryPoint`/`AccessDeniedHandler` 커스터마이즈 + ProblemDetail 정합 한 절(4~6p).
   - **왜:** REST API 운영의 흔한 빈틈. 9장 헤더·CORS와 묶이는 운영 관심사.

10. **[선택] 8장 챕터 제목을 질문형으로.**
   - **어떻게:** "권한은 어디에 사는가 — 인가의 5층 구조" 같은 제목.
   - **왜:** 토비 톤의 정합.

11. **[선택] 1장 "이 책이 다루지 않는 것" 명시.**
   - **어떻게:** 1장 끝에 한 단락 — GraphQL 보안, gRPC 보안, Spring Authorization Server 서버 측 깊은 운영, K8s Service Mesh와의 통합은 본 책 범위 밖. 리서치 한계 §11.2/§11.6 인용으로 정직하게.
   - **왜:** 시니어 독자가 끝까지 안 나오는 토픽을 기다리는 시간을 절약.

## 라운드 1에 대한 book-planner 응답 — 변경 적용 내역

리뷰의 필수 5건·권고 4건·선택 2건을 모두 02_plan.md에 반영했다. 챕터 수는 13 → **15** 로 늘었고, 본문 분량은 412~480p → **470~546p** 로 늘어났다. 사용자가 제시한 12~15 챕터 범위의 상한에 위치한다. 변경의 한 줄 요지와 이유를 항목별로 정리한다.

### 필수 항목 (5/5 반영)

1. **[필수 1] 13장 분할.** 트렌드(14장) + 통합 실전(15장)으로 분리. 운영 묶음(키 회전·Actuator·관측·감사·논쟁점)은 14장 후반 절들로 흡수.
   - *적용 이유:* 한 챕터에 8개 약속이 들어가던 구조를 풀어, 클로징 챕터(15장)가 "한 권을 한 시나리오로"라는 단일 메시지를 단독으로 책임지게 함. 토비 시리즈의 "한 챕터 한 메시지" 원칙 회복.
   - *DPoP 처리:* 5번 항목과 정합 — 14장에서는 Zero Trust 원칙 4(sender-constrained) 매핑 안의 한 단락 cross-ref로만.

2. **[필수 2] 테스팅 챕터 신설.** 12장으로 22~26p. 위치는 Reactive(11장) 직후·마이그레이션(13장) 직전.
   - *적용 이유:* 시니어 독자가 "심층 활용" 약속에서 가장 자주 인용할 챕터. 본문 챕터(인증·인가·Reactive)의 어휘가 모두 깔린 시점에서 "모든 결정을 회귀 가능한 안전망에 묶는 법"으로 응집.
   - *내용 구성:* `@WithMockUser`/`@WithUserDetails`/`@WithSecurityContext` 의미 차이, `SecurityMockMvcRequestPostProcessors`, `oauth2Login()`/`jwt()` mock, `WebTestClient.mutateWith(mockUser())`, MockMvc 통합 패턴, 7.0의 testing 변경점, Reactive·Method Security 테스트 모두 포함.

3. **[필수 3] 1장 "5/6 코드는 7에서 무엇이 빨갛게 변하는가" 절 추가.** 함정 5건(§7.1) 한눈 표 + 13장 cross-ref. 챕터 의존도 지도에 "마이그레이션 우선 독자 경로(1→13→2→3→…)" 명시.
   - *적용 이유:* 1차 독자(5/6 출신)가 1장만 펴고도 "내 빨간 줄의 답이 13장에 있다"는 안심을 얻는 것. 마이그레이션 본편을 후반에 유지하면서도 진입 시점 동기 부여를 보강.

4. **[필수 4] 1장 "보안 멘탈 모델 5분" 절 신설.** OWASP A01·A02·A07 / 인증 vs 인가 / 세션 vs 토큰 / Zero Trust 한 줄 정의를 3~4p로 응축.
   - *적용 이유:* 책 전체의 보안 사고 좌표축을 1장이 책임지게 해서, 이후 모든 챕터의 결정이 이 좌표 위에서 읽히게 함. 14장(트렌드)에서 Zero Trust를 처음 만나는 게 아니라, 1장에서 좌표를 박고 14장에서 그 위에 매핑하는 흐름.
   - *분량 영향:* 1장 22~26p → 28~32p. 도입이 약간 길어지지만 어휘 기반의 가치가 크다.

5. **[필수 5] DPoP를 5장으로 이동.** 5장 후반 "JWT의 한계와 sender-constrained 토큰: DPoP" 절로. 14장에서는 한 단락 cross-ref.
   - *적용 이유:* DPoP는 본질적으로 자원 서버 토큰 검증 강화 메커니즘이다. JWT 함정 4건 절 다음에 두면 "Bearer → sender-constrained"의 토큰 보호 진화 흐름이 한 챕터 안에서 완결된다. 13장 분할에도 도움.
   - *차별점 문구 정직화:* 차별점 3번을 "Passkeys/OTT/MFA를 7장에 1급으로 묶고, DPoP는 5장 JWT 자원 서버 후반에 배치"로 수정. 약속과 실제의 어긋남을 제거.

### 권고 항목 (4/4 반영)

6. **[권고 6] 8장 응축 우려 — 응축 유지하되 분량 늘리고 ACL 박스 축소.**
   - *결정:* 8장 분할 대신 본문 분량을 34~40p → 36~42p로 늘리고, 도메인 객체 ACL을 본문 절이 아닌 짧은 박스로 축소.
   - *이유:* 8장 분할은 "권한은 어디에 사는가"라는 단일 질문의 응집도를 깨뜨림. ACL을 박스로 축소하면 핵심 5묶음(URL/Method/Role/Factory/Error Handling)에 본문이 집중된다. Error Handling 절을 새로 두면서도 분량은 +2p로 안전한 폭으로 조정.

7. **[권고 7] 7장 MFA의 `AuthorizationManagerFactory` 미리보기 처리.** 7장 MFA 절 마지막에 "8장에서 자세히 다룰 `AuthorizationManagerFactory`의 미리보기"라고 명시 + 8장에 역방향 cross-ref 박스.
   - *이유:* 선행 지식 의존도(§4) 문제 해결. 독자가 어색함을 느끼지 않게 양방향 cross-ref로 묶음.

8. **[권고 8] 2장에 커스텀 Filter 절 추가.** 2장 끝에 "내가 만든 Filter를 끼우는 법"(`addFilterBefore`/`addFilterAfter` + Order 패턴, 4~6p).
   - *이유:* 시니어가 자주 묻는 질문. 2장 어휘 직후가 가장 자연스러운 자리. 2장 분량 36~42p → 38~44p.

9. **[권고 9] 9장에 Error Handling 절 추가 → 8장으로 재배치 결정.**
   - *결정 수정:* 리뷰 권고는 9장이었지만, 인가 결정의 짝(`AuthenticationEntryPoint`/`AccessDeniedHandler`)이라는 의미적 결속이 더 강해 **8장 후반** "거부를 어떻게 표현할 것인가" 절(6~8p)로 배치. 9장에는 HTTPS/TLS 운영 맥락(역프록시·`X-Forwarded-Proto`) 3~4p 절을 별도로 추가.
   - *이유:* 8장 인가 결정과 9장 헤더·CORS의 짝 의미를 살리는 동시에, 9장에 누락 지적된 HTTPS/TLS 운영 절을 채움. 두 가지 권고를 동시에 충족.

### 합의 미달 / 라운드 2 잠재 논점 — 현재 없음

리뷰의 11개 항목(필수 5 + 권고 4 + 선택 2)을 모두 반영했고, 권고 9에서 9장 → 8장으로의 자리 변경 외에는 모든 권고를 원안대로 수용했다. 권고 9의 자리 변경은 리뷰가 "9장 또는 8장 끝"으로 열어 두었으므로 합의 미달이 아닌 자유 판단 영역의 결정.

라운드 2에서 재논의가 필요할 가능성이 있는 잠재 논점:
- **8장 분량 36~42p로 5묶음 + Error Handling이 여전히 빠듯한가?** — 저술 단계에서 ACL 박스가 어느 정도 비중을 가져야 충분한지 재점검 필요.
- **12장 테스팅 22~26p가 7.0의 testing 변경점·Reactive·Method Security·통합 시나리오를 모두 담기에 충분한가?** — 저술 단계에서 24~28p로 늘릴 여지 열어둠.
- **14장과 15장의 분리가 너무 작위적이지 않은가?** — 14장(트렌드+운영)과 15장(통합 실전)이 자연스럽게 이어지는지는 저술 단계에서 14장 끝 → 15장 첫 단락의 다리 글에서 검증 필요.

위 잠재 논점은 모두 라운드 2 트리거가 아닌 저술 단계에서 자연 점검되는 사항이다. 라운드 2 리뷰가 있다면 위 세 가지를 우선 점검 대상으로 권한다.
