# 02_plan.md 비판적 리뷰

## 1. 종합 평가 (한 단락)

초안은 9개 영역을 모두 챕터에 매핑했고, 의심→발견→도구→실무→결단의 곡선이 일관되게 흐르며, callback 계획·인용 자료 매핑·Spring 비교 포인트가 챕터별로 명시되어 있어 전체 골격은 단단하다. 특히 8장(마이그레이션)을 절정으로 두고 5개 사례 + Uber 후퇴를 균형 있게 배치한 점, 4장(NestJS) 진입 도입을 t2.micro CPU 후기로 잡은 감각, 1장→8장 Walmart convoy effect 회수 설계는 책의 정체성을 잘 보여준다. 그러나 **(1) 보안과 테스트가 9개 영역 외 누락 영역인데도 사용자에게 결정을 맡긴 채 본 계획에 통합되지 않은 점, (2) 마이그레이션 사례 6개가 8장에 모두 몰려 1장이 1차 자료의 무게 없이 추상적으로 시작되는 위험, (3) 9장의 분량(26p)·범위(설득+커리어+모놀리스+Bun/Deno+JVM 화해+다음 한 걸음)이 균형을 잃은 점, (4) "결정 트리"·"체크리스트" 같은 형식 어휘가 평어체 본문 흐름과 충돌할 가능성**이 가장 시급하다. 개선 요청 6개와 사용자 결정이 필요한 항목 3개로 정리한다.

## 2. 강점 (구체적으로)

1. **9개 영역 매핑이 빠짐없이 됐다.** §4.1~§4.9가 1~9장에 일대일에 가깝게 들어갔고, Top 10 헷갈림 지점(§2)은 1·2·4·5·7장에 분산됐다. 자기 점검 체크리스트가 이를 검증한다.
2. **4장(NestJS) 도입을 t2.micro 후기(§4.4 dev.to/digvijay)로 시작한 선택이 탁월하다.** "27~29% → 6~7%"라는 구체 수치가 Spring 출신의 통념을 즉시 흔든다. 4장의 핵심 질문 ("어디서 Spring이 그리워지고, 어디서 NestJS가 더 가뿐한가")도 양면 톤이라 비교서의 정체성에 맞다.
3. **5장에서 "Hibernate의 마법이 사라진 자리"라는 프레이밍이 탁월하다.** lazy loading·dirty checking·`@Transactional` AOP 부재를 결함이 아니라 "다른 설계"로 풀겠다는 의도가 명시됐고, Prisma 이슈 #4991 인용까지 잡혀 있다. Java 출신의 정서적 진입 장벽을 정면으로 다룬다.
4. **1장→8장 Walmart convoy effect callback이 책의 시그니처가 될 수 있다.** "이벤트 루프를 막지 마라"라는 추상 권고가 8장에서 구체적 사례로 회수되는 구조는 비교 가이드의 기억에 남는 장치다.
5. **6장 디버깅 도구 매핑 표를 챕터의 "표지 페이지"로 두겠다는 결정이 명료하다.** JVM ↔ Node 매핑이 §4.5 표 그대로 들어가고, 본문이 그 표를 풀어내는 구성은 시니어 독자가 책을 사고 가장 먼저 펴볼 페이지가 된다.
6. **분량 합계가 목표 범위 안에 있다.** 8장 50p·4장 44p의 가중이 책의 메시지에 비례한다. 챕터별 글자 수까지 명시한 정밀도도 좋다.
7. **"DI는 진짜 DI인가" Hacker News 논쟁을 회피하지 않고 4장 본문에 넣겠다는 자세가 시니어 독자를 존중한다.** 입문서가 아니라는 신호를 본문 흐름으로 준다.

## 3. 개선 요청 (우선순위 순)

### 요청 1: 보안·테스트 챕터 누락을 7장 안에 흡수하지 말고 명시적으로 처리하라 — Must-fix

**문제:** 사용자가 명시한 9개 영역에 보안과 테스트가 빠져 있다. 초안은 자체 점검(§7-3, §7-4)에서 이 사실을 인지하면서도 "사용자 판단 필요"로 미뤄두고 본 계획에는 7장 일부에만 흐릿하게 흡수했다. 시니어 백엔드를 대상으로 하는 실무 가이드에서 보안(JWT/OAuth/CORS/Helmet/Rate Limit/의존성 취약점)과 테스트(Jest/Vitest/Supertest/`@nestjs/testing`/계약 테스트)가 빠지면 "운영 체크리스트"라는 약속이 깨진다.

**구체적 수정안:**
- **테스트:** 4장 12불릿에 "NestJS 테스트 — `@nestjs/testing` Test.createTestingModule, Spring `@SpringBootTest`/`@MockBean`과의 매핑, Jest vs Vitest, Supertest로 컨트롤러 통합 테스트, 트랜잭셔널 롤백 패턴" 절을 추가하고 4장 분량을 44p → 48p로 늘려라. 8장의 마이그레이션 사례에서 "테스트 커버리지가 어떻게 옮겨갔는가" 한 단락도 같이.
- **보안:** 7장에 "보안 운영"이라는 독립 절(약 6p)을 신설하라. 7장 36p → 42p. 다룰 항목: Helmet 미들웨어, CORS 정책, JWT vs 세션, OAuth(Passport.js·`@nestjs/passport`)와 Spring Security 매핑, Rate Limiting(`express-rate-limit`·`@nestjs/throttler`), `npm audit`·Snyk·Dependabot 의존성 스캔, Spring Security `@PreAuthorize` ↔ NestJS 가드 비교.
- 새 챕터를 만들지는 말라(분량·내러티브 곡선이 깨진다). 4장과 7장에 흡수가 옳다.

**근거:** §6의 팁 1·2·6번은 보안·테스트의 부재를 메우지 못한다. §4.4의 "NestJS는 테스트하기 쉽다"라는 §5.2 호불호 항목, Spring Security 출신 시니어가 가장 먼저 던지는 질문(JWT 가드를 어떻게 짜는가)을 본문이 답하지 않으면 시니어 대상 가이드의 신뢰가 흔들린다.

### 요청 2: 1장에 1차 자료 무게를 더 실어 추상으로 시작하지 마라 — Should-fix

**문제:** 1장 10불릿이 모두 개념 설명(이벤트 루프 단계, 워커 스레드, GC 비교)이고 실제 운영 일화가 없다. Walmart convoy effect를 "힌트로 살짝 흘린다"고 했지만, 1장 본문에서 독자를 붙잡을 일화가 너무 약하다. 8장에서 회수하기 전까지 6개 챕터를 기다려야 한다.

**구체적 수정안:**
- 1장 도입에 §3.4 Walmart의 "10~80ms convoy effect"를 한 페이지로 정직하게 미리 풀어라. "단일 스레드가 막히면 이런 일이 벌어진다"의 구체적 그림을 8장이 아니라 1장에서 한 번 보여주고, 8장에서는 같은 사례를 "왜 이런 일이 벌어지지 않게 SSR을 CDN으로 빼야 했는가"의 결정 차원으로 다시 풀어라.
- §3.3 Netflix flame graph 사례("정규식 한 줄이 핫스팟")도 1장이 아니라 6장 도입의 시그니처 일화로 더 강하게 활용하라. 6장 도입 부분이 현재 "JVM 도구 ↔ Node 도구 매핑 표"로 시작하는데, 표보다 일화로 시작하는 게 6장 핵심 질문("어디서 도구의 사상이 갈라지는가")에 더 맞는다.

**근거:** Toby 스타일 가이드의 "상황 가정" 패턴이 추상 도입보다 강하게 작동한다. 1장에서 의심을 정직하게 꺼낸다는 계획의 의도와도 부합한다. callback 회수가 아니라 "두 번 다른 각도로 본다"의 구도가 비교서 톤에 더 맞다.

### 요청 3: 9장이 너무 많은 것을 떠안고 있다. 분량을 26p → 32~34p로 늘리거나 범위를 줄여라 — Should-fix

**문제:** 9장 10불릿이 (1) 도입 설득 템플릿, (2) 채용·온보딩 학습 경로, (3) 코드 리뷰 가이드 체크리스트, (4) modular monolith 권고, (5) 후퇴 신호 측정, (6) 커리어 시장 신호, (7) Bun·Deno, (8) JVM 진영과의 화해, (9) 독자 유형별 액션, (10) 닫는 말을 모두 다룬다. 26페이지에 들어갈 양이 아니다. 게다가 8장이 50p로 절정을 만들고 9장이 26p로 급격히 가벼워지면 "결단의 구간"이라는 내러티브 약속이 흔들린다.

**구체적 수정안:**
- 9장 분량을 32~34p로 늘리거나, 다음 두 항목을 에필로그(현재 6~8p)로 옮겨라:
  - "JVM 진영과 결별이 아니다" 절 → 에필로그로. 에필로그의 "두 런타임을 같이 돌리는 백엔드 개발자"라는 정체성 메시지와 자연스럽게 합쳐진다.
  - "Bun·Deno의 위치" → 9장 본문에서 박스 형태가 아니라 짧은 절로 두되, 분량은 2p 이내로 엄격히 제한.
- 9장의 "코드 리뷰 가이드 체크리스트"는 7장 운영 체크리스트와 항목이 겹친다(graceful shutdown, 구조화 로깅 등). 7장→9장 callback이 "압축·재인용"이라고 했지만, 실제 본문에서는 같은 항목을 두 번 보게 된다. 9장 체크리스트는 "Spring 출신이 PR 리뷰에서 자주 놓치는 것"으로만 좁히고, 7장 운영 체크리스트와 다른 축으로 잘라라(예: 코드 스타일·타입 안전·예외 전파·쿼리 누락).

**근거:** 8장 50p 직후 9장 26p로 떨어지는 곡선은 분량 균형의 문제다. 프롤로그 8~10p + 에필로그 6~8p로 양 끝을 잡아두는 구조라면 에필로그가 정체성 메시지를 더 무겁게 받아도 된다.

### 요청 4: 마이그레이션 사례 6개가 8장에 몰린 위험을 일부 분산하라 — Should-fix

**문제:** PayPal·LinkedIn·Netflix·Walmart·당근마켓·Uber가 8장에 모두 들어간다. 8장이 절정인 건 맞지만, 1장(런타임)·3장(활용 패턴)·6장(디버깅)·7장(운영) 본문에 사례가 거의 없어서 5~7장이 도구 카탈로그처럼 읽힐 위험이 있다.

**구체적 수정안:**
- §3.2 LinkedIn의 "다른 서비스와 대화하기"라는 BFF 강점은 3장(활용 패턴) 도입에 한 단락 인용하라. 3장이 단순 도구 나열이 아니라 "Node가 어디서 강점을 보이는가"의 관점으로 시작된다.
- §3.3 Netflix flame graph는 요청 2에서 제안한 대로 6장 도입으로 옮기거나 강화.
- §3.1 PayPal의 "Express 비강제성 → Kraken.js 컨벤션 강제"는 4장(NestJS) 도입에 짧게. 4장 t2.micro 후기와 결합해 "구조 강제의 필요성"을 이중으로 보여줄 수 있다.
- 8장에서는 6개 사례 모두 그대로 다루되, 다른 챕터에서 미리 본 사례는 "1장에서 본 그 Walmart 이야기를 다시 보자"의 형식으로 들어가 callback을 통해 책의 통일감이 강해진다.

**근거:** §3 전체가 8장에만 인용되면 사례의 해석 폭이 좁아진다. 같은 사례를 다른 각도(1장의 런타임 한계, 3장의 BFF 적합성, 6장의 도구, 8장의 결정)에서 보는 구도가 비교서의 무게를 균등하게 만든다.

### 요청 5: "결정 트리"·"의사결정 표"·"체크리스트" 형식 어휘를 본문 흐름으로 풀어라 — Should-fix

**문제:** 3장(패턴 선택 가이드 의사결정 표), 4장(NestJS/Fastify 선택 표), 5장(선택 가이드), 8장(결정 트리), 9장(코드 리뷰 가이드 체크리스트) — 각 챕터에 형식 어휘가 반복 등장한다. 스타일 가이드의 평어체+청유형+수사적 질문 톤과 충돌한다. "결정 트리를 따라가자"라는 형식 권고는 Toby 스타일 본문이 아니라 컨설팅 자료 톤이다.

**구체적 수정안:**
- 표·체크리스트는 그대로 두되, 본문에서는 "이때는 어떻게 결정해야 할까?"의 수사적 질문으로 진입한 뒤 산문으로 시나리오를 풀어라. 표는 그 산문 끝에 정리 자료로 붙는 형태.
- 4장 결정 가이드는 "팀 규모 5명 이하 + 마이크로서비스 3개 이하면 Fastify가 더 가볍다. 그 위로 올라가면 NestJS의 모듈 그래프가 짐을 덜어준다"의 산문체로 풀고, 표는 본문 끝에 한 페이지 정리.
- 9장 코드 리뷰 체크리스트는 본문에서는 "PR 리뷰에서 가장 자주 데이는 5가지 함정을 살펴보자"의 형식으로 풀고, 체크리스트 자체는 챕터 끝에 한 페이지 부록 박스.

**근거:** Toby 스타일은 산문 흐름 안에 의사결정이 녹는 톤이다. "결정 트리"라는 형식이 평어체와 부딪치면 책이 "기술 컨설팅 보고서"처럼 읽히고, 비교 가이드의 정체성을 흐린다.

### 요청 6: 한국 사례 보강을 본 계획에 정식 작업으로 포함시켜라 — Should-fix

**문제:** §3.6은 당근마켓 1건만 1차 자료가 단단하고, 토스·인프런·LINE은 §8-2 한계로 명시되어 있다. 본 계획 §7-(3)-4번에서 "Phase 4 전에 보강 리서치를 한 번 더 돌리면 좋다"고 적었지만, 본 계획 단계에서 이 작업을 명시적 To-do로 잡아두지 않으면 8장 본문 작성 시점에 누락된다.

**구체적 수정안:**
- 8장 인용 자료 목록에 "[보강 필요] 토스·인프런·LINE·우아한형제들·카카오 1차 자료"를 명시 항목으로 추가하라.
- 6개 사례를 "확정된 사례"와 "보강 후 추가 검토" 두 묶음으로 나눠라. 당근마켓 1건만으로 한국 사례 대표성이 약하므로, 8장 6개 사례 중 한 자리를 비워두고 보강 결과에 따라 채우는 구도가 안전하다.
- 만약 보강에서 1차 자료가 못 나오면 8장은 6개가 아니라 5개 사례 + "한국 사례의 자료 한계" 절로 정직하게 가는 안도 명시해두라.

**근거:** §8-2 한계의 자체 자각을 본 계획에서 작업으로 전환해야 Phase 4(챕터 저술)에서 누수가 안 생긴다.
