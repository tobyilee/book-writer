# 1장. 단일 스레드는 정말 일감을 감당할 수 있는가

> **V8과 libuv, 이벤트 루프, 그리고 Spring 서블릿 스레드 모델과의 헤어짐**

블랙 프라이데이 새벽, 월마트의 트래픽 그래프가 평소의 몇 배로 솟아오르는 그 순간을 떠올려보자. 모바일에서 들어오는 요청이 전체의 70%를 넘기고, 한 페이지를 서버에서 렌더링하는 데 10~80ms가 걸린다. 평상시 같으면 그저 평범한 응답 시간이지만, 이날의 트래픽 곡선 위에서는 의미가 다르다. 페이지 한 번 그리는 사이에 다른 요청들이 차곡차곡 큐에 쌓인다. 앞 사람의 작업이 뒷사람을 줄세우는 — 운영자들이 "convoy effect"라 부르는 — 그 함정 위에 단일 스레드 런타임이 노출된다.

이 광경을 처음 들으면 의심이 먼저 든다. 단일 스레드 한 개가 어떻게 수만 커넥션을 받아내고, 또 그 한 개가 막혔다는 건 도대체 무슨 의미일까? Spring 출신의 우리에게는 살짝 어색한 그림이다. 우리가 익숙한 세계에서는 톰캣이 스레드 풀을 들고 있었고, 한 요청이 한 스레드를 점유했고, 옆 스레드는 옆 일을 했다. 한쪽이 지연되어도 다른 쪽은 자기 페이스로 일을 처리해줬다. JDBC 호출이 길어져도 그건 그 한 사용자만의 문제였다. 그런데 Node.js의 그림은 이와 다르다. 일감을 들고 도는 메인 스레드가 한 개뿐이고, 그 한 개가 막히면 모든 요청이 같이 멈춘다. 적어도 진술의 표면만 보면 그렇다.

그렇다면 이 모델은 어떻게 트래픽을 감당하는 것일까? 막히면 어떤 일이 벌어지고, 막히지 않게 만들려면 어디를 봐야 할까? Spring Boot에서 풀던 동시성 문제는 Node에서 똑같이 풀리는가, 아니면 사고 회로를 갈아끼워야 하는가? 이 장에서는 Node.js 런타임의 본체 — V8과 libuv — 를 들여다보면서 이 의심을 풀어보자. Spring 서블릿 시절의 직관을 그대로 들고 가면서, 어디서 그 직관이 그대로 통하고, 어디서 잠깐 갈아끼워야 하는지를 함께 짚는다. 결론부터 말하면, 단일 스레드는 일감을 충분히 감당한다. 다만 한 가지 조건이 붙는다. 그 조건이 무엇인지 풀어가는 게 이 장의 일이다.

## 월마트의 컨보이, 단일 스레드의 첫 비명

월마트의 사례를 조금 더 가까이서 살펴보자. 이들은 모바일 트래픽이 전체의 70%를 차지하는 환경에서 페이지 렌더링을 서버사이드로 끌어왔다. 클라이언트가 약하고 네트워크가 느린 환경(2G·3G가 아직 흔하던 시절)에 SPA를 통째로 던지는 대신, 서버에서 HTML을 그려 뱉어주는 모델로 다시 돌아온 것이다. 이걸 Node.js로 옮긴 뒤 한동안은 거의 거짓말 같은 시기가 있었다. 55%의 트래픽을 처리하는 동안 CPU가 1% 안팎으로 도는 수치가 나왔다. Black Friday 단 하루 동안 15억 달러어치 거래가 그 위로 흘렀다. JVM에서 같은 트래픽을 받아낼 때 흔히 보던 GC 곡선과 풀 사용률 그래프가, 여기서는 거의 평평했다. "이게 정말 가능한가" 싶은 정직한 놀라움이 운영팀에서 나왔다고 한다.

그런데 어느 시점부터 사고가 났다. 페이지 렌더링이 이벤트 루프 위에서 도는 동안, 그 사이로 들어오는 다른 요청들이 차곡차곡 큐에 쌓였다. 한 페이지에 10~80ms — 평소에는 무시해도 될 수치였다. 그러나 트래픽이 평소의 몇 배가 되니 사정이 달라졌다. 앞 작업이 끝나기 전엔 뒷 작업이 시작되지 못한다. 이게 운영팀이 회고에서 못 박은 그 단어, **convoy effect**다. 차량 행렬이 앞차의 속도에 묶여 다 같이 느려지듯, 이벤트 루프 위의 콜백들이 앞 콜백의 처리 시간에 그대로 묶인 것이다. 평균 응답시간은 멀쩡해 보이는데 p95·p99가 무너지기 시작한다. 80ms × 100건이 직렬로 줄을 서면 마지막 요청은 8초를 기다린다. 응답을 못 받은 사용자는 새로고침을 누르고, 그 새 요청이 또 큐의 끝에 붙는다. 한 번 무너지면 회복이 어려운 종류의 사고다.

월마트는 결국 fruit-loops와 hula-hoop이라는 자체 도구로 페이지를 사전 렌더링해 CDN에 박아두는 전략을 택했다. 동적으로 그릴 수밖에 없는 부분만 서버에서 처리하고, 나머지는 캐시 가장자리로 빼낸 것이다. 이 결정의 전체 그림 — 왜 그 시점에 그렇게 결정했는지, 마이그레이션은 어떻게 단계별로 굴렸는지 — 은 8장에서 다시 만난다. 1장에서 우리가 짚을 것은 그 결정의 토대가 되는 메커니즘이다. **왜 페이지 한 장이 모든 사용자를 줄 세우는가**, 그 한 가지다.

이 일화에는 부록 같은 사실이 하나 더 있다. 월마트의 엔지니어들은 이 시기를 거치면서 자체 프레임워크 hapi(현재 Hapi.js)를 만들어냈다. Express만으로는 거대 트래픽 환경의 운영 디테일이 충분치 않다고 봤기 때문이다. 다시 말해, **운영하면서 프레임워크가 진화하는 그림**이 펼쳐졌다. JVM 진영에서 Spring이 Servlet API의 부족분을 메우며 자라온 그림이, 같은 모양으로 Node 진영에서도 일어났다. 세대가 다른 두 런타임이 같은 길의 다른 단계를 걷고 있을 뿐이라는 인상을 주는 일화다. 이 책에서 다루는 NestJS도 그 진화의 한 가지 분기다(4장에서 본격적으로 마주한다).

여기서 잠시 Spring의 그림을 다시 꺼내자. 톰캣이 받는 요청을 머릿속에 그려보면, `server.tomcat.threads.max=200`이 디폴트에 가까웠고, 한 요청이 도착하면 풀에서 스레드 하나가 빠져나와 요청을 끝까지 들고 갔다. JDBC가 막히면 그 스레드만 막혔다. 옆 스레드는 옆 일을 했다. 한 사용자가 느린 쿼리를 던졌다고 해서 다른 사용자가 같이 멈추는 일은, 풀이 고갈되지 않는 한 일어나지 않았다. 풀이 모자라면 풀을 키웠다. 그 모델 위에서 우리는 오랫동안 백엔드 직관을 길러왔다. 그래서 Node.js의 "한 스레드가 막히면 모두가 같이 줄을 선다"는 진술을 처음 들으면 살짝 찜찜해진다. 톰캣이라면 스레드 풀을 늘리거나 비동기 서블릿(`@Async`)을 쓰면 풀려나갔던 문제가, 여기서는 도대체 어떻게 풀린다는 말인가.

답은 **그 한 스레드가 무엇을 하느냐**에 있다. 톰캣 스레드는 요청 시작부터 끝까지를 자기 시간으로 쓴다. JDBC가 응답할 때까지 자리에서 잠을 잔다. blocking I/O가 그대로 스레드를 점유한다. 반면 Node의 메인 스레드는 자기가 직접 잠들지 않는다. I/O가 필요해지면 OS에 위임하고, 곧장 다음 콜백으로 넘어간다. **잠을 잘 자리가 자기 자리가 아니다.** 이 한 줄이 두 모델을 가르는 핵심이다. 결국 Node의 메인 스레드가 처리하는 일은 (1) 새 커넥션 수락, (2) 들어온 데이터의 콜백 디스패치, (3) 실행 중인 JS 코드의 한 청크씩의 실행 — 이 셋이 거의 전부다. JDBC 응답을 기다리는 동안 이 스레드는 다른 사용자의 요청을 받아 들이고, 다른 사용자의 콜백을 실행한다. 그래서 빠르다. 그래서 또한, 한 청크가 너무 길어지면 모든 사용자가 그 청크가 끝나기를 기다리게 된다.

이 그림을 본격적으로 풀어보려면 우선 V8과 libuv의 자리를 정리해야 한다. 어디서 JS가 돌고, 어디서 I/O가 도는지를 명확히 그려두지 않으면 "막힌다"의 의미가 모호한 상태로 남는다.

## V8 엔진의 자리 — JIT은 익숙한 풍경이다

Node.js 런타임의 본체는 두 부품으로 이루어진다. 하나는 V8, 또 하나는 libuv다. V8은 JavaScript를 머신 코드로 컴파일해 돌리는 엔진이고, libuv는 비동기 I/O와 이벤트 루프를 담당하는 C 라이브러리다. 두 부품을 합쳐 단일 프로세스로 묶은 것이 우리가 `node` 명령으로 띄우는 그 런타임이다. 이 두 부품이 한 메인 스레드 위에서 협력해 도는 그림이 머릿속에 잡히면 절반은 끝난 셈이다.

V8의 동작은 JVM 출신이라면 의외로 익숙하다. JIT 컴파일이 그렇다. V8은 코드를 처음에는 인터프리터(Ignition)로 돌리다가, 자주 호출되는 함수는 옵티마이저(TurboFan)가 머신 코드로 컴파일해 캐시한다. 핫스팟이 식별되면 더 공격적으로 인라이닝하고, 가정이 깨지면 — 예를 들어 객체의 모양이 바뀌거나, 기대하지 않은 타입의 인자가 들어오면 — 디옵티마이즈해서 다시 인터프리터로 떨어뜨린다. 이 흐름은 JVM의 HotSpot이 C1·C2 컴파일러를 단계적으로 적용하고, 가정이 깨지면 디옵트하는 그림과 거의 같은 모양이다. "처음엔 느리다가 점점 빨라진다", "워밍업이 필요하다"는 직관이 그대로 통한다. 람다 콜드 스타트에서 Node가 200ms 미만으로 도는 반면 Spring Boot가 3~10초씩 걸리는 그 차이는 JIT 자체의 이야기라기보다 부팅 시점의 클래스 로딩과 컨텍스트 초기화의 무게 차이에 가깝다. JIT의 따뜻해지는 모양은 둘 다 비슷하다.

다만 V8이 JVM과 결정적으로 갈리는 지점이 두 가지 있다. 하나는 **Hidden Class와 Inline Caching**이라는 최적화 기법이다. JS 객체는 Java 클래스처럼 모양이 컴파일타임에 박혀 있지 않다. `obj.x = 1`을 한 다음에 `obj.y = 2`를 추가할 수 있고, 또 `obj.x`를 지울 수도 있다. 이런 자유를 그대로 두면 객체 접근이 매번 해시 룩업이 되어 느려진다. 그래서 V8은 같은 모양으로 만들어진 객체들에 자동으로 "히든 클래스"를 부여하고, 자주 호출되는 접근 경로는 인라인 캐시로 박아둔다. 코드 한 토막으로 보면 직관이 잡힌다.

```javascript
// JS — 같은 모양의 객체와 다른 모양의 객체
function makeFast() {
  // 생성자 안에서 모든 필드를 한 번에 초기화 — 같은 hidden class
  return { id: 0, name: '', email: '' };
}

function makeSlow() {
  const obj = {};
  obj.id = 0;        // hidden class 1
  obj.name = '';     // hidden class 2 (transition)
  obj.email = '';    // hidden class 3 (transition)
  return obj;
}

// fast 군은 같은 hidden class를 공유하므로 V8이 inline cache로 접근을 인라이닝
// slow 군은 각 단계마다 transition을 일으키며 IC를 무효화시킨다
```

두 함수가 만들어내는 객체는 표면적으로 같다. `{id, name, email}`을 가진 평범한 객체다. 그런데 V8 입장에서는 다르다. `makeFast`는 항상 같은 모양의 객체를 한 방에 만들어내므로 같은 히든 클래스로 묶인다. `makeSlow`는 필드를 하나씩 추가하면서 그때마다 새로운 히든 클래스로 천이한다. 후속 코드에서 `obj.id`에 접근할 때 `makeFast` 군은 인라인 캐시 한 줄로 바로 메모리 오프셋에 접근하지만, `makeSlow` 군은 캐시 미스가 일어나며 느려진다. Java 개발자가 이걸 알아두면 좋은 이유는 한 가지다 — **객체의 모양을 도중에 바꾸면 V8의 최적화가 깨진다.** 생성자에서 모든 필드를 한 번에 초기화하고, 나중에 필드를 추가하지 않는 편이 낫다. Java처럼 클래스 정의에 필드를 다 적어두는 우리의 습관이, 여기서도 그대로 좋은 습관으로 작동한다는 것이 의외다.

또 하나 짚어둘 것은 **인자 타입의 일관성**이다. V8의 옵티마이저는 함수 호출 시 인자의 타입을 추적해 타입 가드를 박아둔다. `function add(a, b) { return a + b; }`라는 함수를 항상 number로만 호출하면 옵티마이저는 정수 덧셈으로 인라이닝한다. 그런데 어느 날 갑자기 string을 넘기면 가드가 깨지며 디옵트가 일어난다. Java라면 컴파일러가 타입을 보장해주니 이런 상황 자체가 없지만, JS는 동적 타입이라 런타임이 직접 가드를 한다. TypeScript를 쓰면 코드 상에서는 보호가 되지만 V8 입장에서는 여전히 동적 호출이다. 그래서 핫 패스 함수의 인자 타입을 흐트러뜨리지 않는 게 미세 최적화의 한 축이 된다. 평소엔 신경쓸 일이 없지만, 프로파일링하다가 디옵트가 자주 보이면 의심해볼 자리다.

`node --trace-deopt`로 띄우면 디옵트가 일어나는 함수와 이유를 콘솔로 받아볼 수 있다. JVM의 `-XX:+PrintCompilation`과 비슷한 자리에 있는 도구다. 처음엔 의외로 자주 도는 게 보여 살짝 놀라겠지만, 운영에 문제가 될 정도로 흔한 일은 아니다. 정상 트래픽의 핫 패스에서 반복적으로 디옵트가 일어나는 게 보일 때만 손을 대면 된다.

다른 결정적 차이는 **GC**다. V8은 New Space(Young Generation)와 Old Space로 힙을 나누고, 짧은 수명의 객체는 Scavenge라는 빠른 minor GC로, 긴 수명의 객체는 Mark-Sweep-Compact 계열의 major GC로 처리한다. 두 세대로 나뉜 큰 그림은 JVM의 G1과 닮아 있지만 결정적인 차이가 있다. V8은 단일 프로세스·단일 힙을 강하게 가정한다. 기본 힙 한도가 1.5GB 정도이고(64-bit 환경의 기본값), `--max-old-space-size`로 늘리더라도 4~8GB 부근에서 운영을 멈추는 게 보통이다. JVM에서 32GB·64GB 힙을 설정하고 ZGC로 백 밀리초 단위 STW를 받아내는 그 그림은, V8 위에서는 자연스럽지 않다. 이게 두 런타임의 운영 패턴을 갈라놓는 핵심이다 — JVM은 큰 힙 한 개로 묵직하게 가는 모델, Node는 작은 힙 여러 개로 가로 확장하는 모델. Walmart가 트래픽을 처리하던 그 그림에서 인스턴스를 가로로 늘리는 게 자연스러웠던 이유가 여기 있다. PM2 cluster나 컨테이너 한 개당 한 프로세스의 패턴이 Node 운영의 기본기로 굳어진 배경도 같다.

GC의 운영 차원은 7장에서 본격적으로 다루지만, 1장 단계에서 새겨둘 직관은 한 줄이다. **Node 인스턴스가 1.5GB를 넘기 시작하면 가로로 쪼개는 결정을 먼저 검토한다.** JVM 시절에는 힙을 키우는 게 자연스러운 첫 카드였다면, Node에서는 인스턴스를 늘리는 게 자연스러운 첫 카드다. 둘은 다른 도구지만 푸는 문제는 같다 — 메모리 압박을 어떻게 분산할 것인가.

여기까지 정리하면, V8은 JVM과 닮은 면이 많다. JIT, 세대 GC, 워밍업, 이런 직관은 그대로 가져가도 된다. 다만 **객체 모양을 자주 바꾸지 말 것**과 **큰 힙을 들고 가지 말 것**, 이 두 가지만 새로 새겨두면 된다. 이제 진짜 새로운 부품, libuv를 들여다보자.

## libuv와 이벤트 루프 — 한 스레드의 정체

libuv가 하는 일을 한 줄로 줄이면 이렇다. **OS의 비동기 I/O 멀티플렉서(Linux의 epoll, BSD/macOS의 kqueue, Windows의 IOCP)를 추상화해, 한 스레드 위에서 수만 fd의 이벤트를 받아내는 프레임워크.** Linux의 `epoll_wait` 시스템 콜이 N개의 fd 이벤트를 한 번에 돌려준다는 점을 이용해, 단일 스레드로 C10K 문제를 정면 돌파한 모델이다. Spring/톰캣 시절의 직관으로 보면 어색하지만, 사실 우리가 알던 **Java NIO와 Netty가 같은 길을 먼저 걸었다.** Spring WebFlux의 Reactor도 결국 그 위에 서 있다. Node는 그 모델을 런타임의 기본값으로, 또 동시에 거의 유일한 옵션으로 가져온 것뿐이다. JVM이 blocking과 non-blocking을 둘 다 들고 가는 데 비해, Node는 처음부터 non-blocking 한 길로만 굴러간다.

이벤트 루프의 한 틱은 6단계로 구성된다. 그림으로 정리하면 이렇다.

```
   ┌────────────────────────────────────────┐
   │  timers           — 만료된 setTimeout/setInterval
   ├────────────────────────────────────────┤
   │  pending callbacks — TCP 에러 등 OS가 미룬 콜백
   ├────────────────────────────────────────┤
   │  idle / prepare    — 내부용
   ├────────────────────────────────────────┤
   │  poll              — 실제 I/O 콜백 (epoll_wait)
   ├────────────────────────────────────────┤
   │  check             — setImmediate 콜백
   ├────────────────────────────────────────┤
   │  close callbacks   — socket.on('close', ...) 등
   └────────────────────────────────────────┘
                ↓ 단계 사이마다 nextTick / microtask 큐를 비운다
                └→ 다음 틱
```

각 단계는 자기에게 할당된 콜백 큐를 비우거나, 일정 한도까지 처리한 뒤 다음 단계로 넘어간다. 가장 중요한 단계는 **poll**이다. 여기서 `epoll_wait`(또는 OS별 대응)이 호출되어 새로 도착한 I/O 이벤트를 받아오고, 그 이벤트의 콜백이 그 자리에서 실행된다. 등록된 fd에 받을 데이터가 없으면 poll은 잠시 잠을 잘 수도 있다(타임아웃 안에서). 한 틱이 다 돌면 다시 timers 단계로 돌아간다. 이게 우리가 흔히 듣는 "이벤트 루프"의 실체다.

여기서 자주 헷갈리는 게 두 개 있다. **`setImmediate`와 `setTimeout(fn, 0)`의 차이**, 그리고 **`process.nextTick`이 어디로 끼어드는가**다. 짧은 코드 한 토막으로 살펴보자.

```javascript
// JS — 실행 순서가 보이지 않는다면 이벤트 루프를 다시 봐야 한다
setTimeout(() => console.log('A: timeout 0'), 0);
setImmediate(() => console.log('B: immediate'));
process.nextTick(() => console.log('C: nextTick'));
Promise.resolve().then(() => console.log('D: microtask'));
console.log('E: sync');
```

이 다섯 줄을 그대로 돌리면 출력은 `E → C → D → A → B` 또는 `E → C → D → B → A` 순서로 찍힌다. E가 가장 먼저 나오는 건 동기 코드라 당연하다. 그다음 C(nextTick)와 D(microtask)가 나오는 이유는, **이벤트 루프의 단계 사이마다 nextTick 큐와 마이크로태스크 큐를 먼저 비우기 때문**이다. nextTick이 microtask보다 우선이라는 작은 디테일도 따라온다. 그리고 A의 timer와 B의 immediate는 각각 timers 단계와 check 단계로 들어가는데, 두 단계가 한 틱에서 어느 쪽이 먼저 도는지는 호출 시점의 미세한 타이밍에 따라 갈린다. 보통 메인 모듈 바깥에서는 timer가 먼저 나오고, I/O 콜백 안에서는 immediate가 먼저 나온다. 처음 보면 살짝 난감하지만, 한 번 머릿속에 그림이 잡히면 디버깅할 때 유용한 직관이 된다.

여기서 짚어둘 것은 **microtask와 nextTick이 단계 사이에 끼어든다**는 점이다. async/await의 `await` 한 줄은 사실 함수를 거기서 잘라내, 마이크로태스크 큐로 콜백을 등록한다. 그래서 `await`가 많은 코드도 실은 한 틱 안에서 마이크로태스크가 빈번하게 실행되는 그림으로 도는 것이다. Java의 CompletableFuture가 ForkJoinPool로 콜백을 던지는 모양과는 결이 다르다. CompletableFuture는 별도 스레드 풀이 콜백을 가져가지만, async/await는 같은 메인 스레드 위에서 마이크로태스크 큐를 통해 콜백이 돌아온다. **코드 모양은 비슷한데 실행 모델은 근본적으로 다르다.** 이걸 처음 정리하면 살짝 헷갈리고, 한 번 정리되면 평생 도움이 된다.

한 가지 운영 함정도 같이 짚자. `process.nextTick`과 `Promise.resolve().then`을 무한 재귀로 등록하면 — 예를 들어 콜백 안에서 또 nextTick을 등록하면 — **이벤트 루프의 다음 단계로 영영 넘어가지 못한다.** poll 단계가 다시 돌지 못하니 새 I/O 이벤트가 도착해도 처리되지 않는다. 결과는 컨보이 효과의 다른 형태다. 사용자 입장에서는 "서버가 응답을 안 한다"가 똑같이 보인다. 그래서 nextTick은 **꼭 필요할 때만, 비재귀적으로** 쓰는 편이 낫다. setImmediate가 더 안전한 선택이 되는 자리가 종종 있다.

setImmediate를 좀 더 구체적으로 써보자. I/O 콜백 안에서 `setTimeout(fn, 0)`과 `setImmediate(fn)` 중 어느 쪽을 쓰는 게 자연스러운가? 짧은 시연 코드가 답을 준다.

```javascript
// JS — I/O 콜백 안에서는 setImmediate가 항상 먼저 실행된다
const fs = require('node:fs');

fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});
// 출력은 항상: immediate → timeout
```

이유는 이렇다. `fs.readFile`의 콜백이 실행되는 순간은 이미 poll 단계 안이고, 그 다음 단계가 check 단계(setImmediate가 실행되는 자리)다. timer는 다음 틱의 timers 단계까지 기다려야 한다. 그래서 **I/O 콜백 이후에 작은 비동기 후속 작업을 큐에 던지고 싶으면 setImmediate가 더 일관된 선택**이 된다. 0ms timer를 쓰면 운영체제 timer 해상도에 따라 1ms 이상의 미세한 지연이 끼는 경우도 있다. 큰 차이는 아니지만, 일관된 직관을 위해 알아두면 좋다.

## I/O는 OS 레벨에서 어떻게 도는가 — epoll의 그림

조금 더 안쪽으로 들어가보자. 이벤트 루프가 단일 스레드 위에서 수만 커넥션을 처리한다는 진술은 사실 OS 레벨의 비동기 I/O 인터페이스에 의지한다. Linux의 `epoll_create`, `epoll_ctl`, `epoll_wait`이라는 세 시스템 콜이 그 핵심이다. 짧게 그림으로 옮기면 이렇다.

```
[애플리케이션 코드]                  [커널]
  서버 소켓 listen()       ←→   listen_fd 등록
  epoll_create()            ←→   epoll 인스턴스 생성
  epoll_ctl(ADD, listen_fd) ←→   listen_fd를 epoll이 감시
  ...새 커넥션 도착...
  epoll_wait()              ←→   준비된 fd 목록 반환
  accept()                  ←→   클라이언트 fd 획득
  epoll_ctl(ADD, client_fd) ←→   client_fd를 추가로 감시
  ...데이터 도착...
  epoll_wait()              ←→   client_fd가 readable 상태로 반환
  read()                    ←→   non-blocking으로 읽음 → 콜백 실행
```

핵심은 `epoll_wait` 한 번이 N개의 준비된 fd를 한 번에 돌려준다는 점이다. 톰캣 시절의 BIO(blocking I/O) 모델은 한 fd에 한 스레드를 붙여 그 스레드가 read에서 잠을 자며 데이터를 기다렸다. 1만 개의 fd가 동시에 열려 있으면 1만 개의 스레드가 필요했다(혹은 풀이 그만큼 커야 했다). 그래서 C10K가 진짜로 어려운 문제였다. epoll의 등장 이후 한 스레드가 1만 개의 fd를 한꺼번에 감시하는 그림이 가능해졌고, libuv는 그 위에 콜백 디스패칭 레이어를 얹은 것이다. 그래서 Node의 메인 스레드는 "잠을 자는 스레드"가 아니라 "이벤트 디스패처"의 정체성을 갖는다.

이 차이를 한번 받아들이면 운영 직관이 통째로 바뀐다. 톰캣에서 봤던 "active threads" 그래프 — 풀이 얼마나 차 있는지를 보던 그 그래프 — 가 Node에서는 사라진다. 대신 **이벤트 루프 lag**이 그 자리를 대체한다. 풀의 포화도가 아니라 단일 스레드의 처리 지연이 핵심 메트릭이 되는 것이다. 같은 관심사("얼마나 막혔는가")를 다른 도구로 보는 셈이다. 곧 이 lag 지표를 측정하는 코드를 함께 살펴본다.

여기서 우리에게 익숙할 수도 있는 다른 그림 하나를 같이 꺼내자. **Spring WebFlux와 Reactor**. JVM 진영에서 비동기·non-blocking 모델을 정면으로 받아들인 결과물이다. WebFlux도 결국 Netty 위에서 Reactor가 이벤트 루프를 돌리고, 콜백 체인을 `Mono`/`Flux` 연산자로 표현한다. 사고 회로 자체는 Node와 거의 같다. **non-blocking I/O + 이벤트 루프 + 비동기 콜백 체인.** 도구의 이름과 문법이 달라졌을 뿐, 본체는 같은 모델이다. WebFlux를 한 번이라도 운영해본 경험이 있다면, Node로 넘어오는 그 거리는 사실 짧다. 톰캣 기반 Spring MVC에서 WebFlux로 옮길 때보다 오히려 더 짧을 수도 있다. JS의 async/await이 Reactor의 `flatMap` 체인보다 코드 모양이 단순하다는 평이 흔한 이유도 같다. 두 진영의 비동기 흐름이 결국 같은 곳을 향해 수렴하고 있다는 인상이 1장 단계에서 한 번 잡히면, 이후 챕터들이 한결 자연스럽게 읽힌다.

조금 더 시각적으로 두 모델을 마주 세워보자.

```
[Spring MVC (BIO) — Tomcat 200 스레드 풀]
  요청 ──→ [스레드#1] ──→ JDBC 호출 (스레드 #1 잠) ──→ 응답
  요청 ──→ [스레드#2] ──→ JDBC 호출 (스레드 #2 잠) ──→ 응답
  요청 ──→ [스레드#3] ──→ ...
  ...풀이 200개까지...

[Node.js (libuv 이벤트 루프)]
  요청1 ──┐
  요청2 ──┤
  요청3 ──┼→ [메인 스레드 — 콜백 디스패처] ←─→ [epoll/kqueue]
  ...     │      │
  요청N ──┘      ├→ JDBC 응답 콜백 (1)
                 ├→ HTTP 응답 콜백 (2)
                 └→ 파일 읽기 콜백 (3, 워커 풀에서 도착)
```

같은 워크로드를 두 모델이 다른 방식으로 풀어낸다. Spring MVC는 N개의 동시 요청을 N개의 스레드로 받아 각자 잠재워두는 모델이고, Node는 한 개의 메인 스레드가 콜백을 디스패치하며 epoll/kqueue가 잠을 대신 자주는 모델이다. 사용자에게 보이는 결과는 비슷하지만, 자원 사용 그림은 완전히 다르다. 1만 동시 커넥션 시나리오를 그려보면 Spring MVC는 1만 스레드를 만들 수도 없고(보통 풀이 200~500), 만든다 해도 컨텍스트 스위칭 오버헤드가 폭발한다. Node는 메인 스레드 한 개가 그 1만 fd를 epoll로 감시하며 도는 게 자연스러운 운영점이다. 가상 스레드는 Spring 진영에서 이 그림을 비슷하게 따라잡으려는 답이다.

## 워커 스레드 풀 — 비동기의 뒷마당, 그리고 "스레드 풀"이라는 같은 이름의 다른 자리

여기까지 이야기를 들으면 이런 의문이 생긴다. **그렇다면 파일 I/O는?** 디스크 read는 OS가 epoll로 잘 다뤄주지 않는다. 정확히 말하면, 리눅스의 일반 파일 I/O는 비동기 인터페이스가 오랫동안 빈약했다. `io_uring`이 도착한 비교적 최근까지, 디스크 read는 사실상 blocking으로만 가능했다. 이걸 Node가 어떻게 처리하느냐가 단일 스레드 모델의 작은 비밀이다.

답은 단순하다. **libuv가 워커 스레드 풀을 들고 있다.** 기본 4개. 환경변수 `UV_THREADPOOL_SIZE`로 늘릴 수 있다. 파일 I/O, DNS 조회의 일부(`dns.lookup`), `crypto`의 `pbkdf2`나 `scrypt` 같은 CPU성 연산은 이 풀에 던져진다. 메인 스레드는 그저 "이 일 좀 해줘"라고 위임하고 콜백을 등록한 뒤 곧장 다음 일을 한다. 작업이 끝나면 워커가 결과를 가지고 돌아오고, 메인 스레드는 다음 틱의 poll 단계에서 그 콜백을 실행한다.

이걸 알면 두 가지가 바로 정리된다. 첫째, **네트워크 I/O는 이 풀을 거치지 않는다.** TCP·HTTP 같은 네트워크 fd는 epoll/kqueue로 직접 처리되므로 메인 스레드 위에서 다 돌아간다. 그래서 네트워크 I/O가 많은 워크로드에서 Node가 압도적으로 가벼운 것이다. Walmart가 55%의 트래픽을 1% CPU로 받아낸 그 그림이, 이 사실 위에 서 있다. 둘째, **파일 I/O가 많거나 crypto 연산이 많으면 4개의 워커가 병목이 된다.** 이때는 풀 사이즈를 늘리거나, 진짜로 무거운 작업이라면 Worker Threads로 분리해야 한다. 이 둘은 다른 부품이다. 잠시 뒤에 본다.

DNS 한 가지를 짚어두자. 우리가 흔히 쓰는 `dns.lookup`은 OS의 `getaddrinfo`를 부르는데, 이게 동기적인 시스템 콜이다. 그래서 libuv 워커 풀을 점유한다. 외부 API를 자주 호출하는 서비스에서 DNS lookup이 갑자기 느려지면 워커 4개가 다 점유되고, 그 사이에 들어온 다른 파일 I/O가 줄을 선다. 운영에서 한 번 데보면 평생 기억에 남는 종류의 사고다. 대안은 `dns.resolve` 계열 API를 쓰는 것이다. 이건 DNS 쿼리를 직접 보내는 비동기 구현이라 워커 풀을 쓰지 않는다. **Spring 출신이 처음 운영하면서 "왜 DNS가 느려지는데 파일 I/O까지 같이 느려지지?"라는 의문을 갖는 자리**가 바로 이 워커 풀 공유 지점이다.

Spring 출신이 여기서 한 번 헷갈린다. 톰캣의 그림에서는 "스레드 풀"이라는 말이 곧 요청 처리 스레드를 가리켰다. 풀이 커지면 동시에 받을 수 있는 요청이 늘어났다. 그런데 Node에서 "스레드 풀"이라고 하면 보통 두 가지를 동시에 가리킬 수 있다. 하나는 **libuv의 워커 풀** — 파일·DNS·crypto의 뒷마당. 또 하나는 **Node 12부터 정식 도입된 Worker Threads** — 별도 V8 인스턴스를 가진 진짜 분리된 실행 환경. 이름이 비슷해서 매뉴얼을 읽다 보면 자주 헷갈린다. 정리하면 이렇다. **요청을 받는 메인 스레드는 한 개로 고정**, **파일·crypto·DNS 같은 시스템 작업의 뒷마당이 libuv 워커 풀(기본 4개)**, **CPU 바운드 작업을 직접 분리할 때 쓰는 게 Worker Threads(개수는 사용자 결정)**. 이 셋이 다른 부품이다. 책 전체에서 "워커 풀"과 "Worker Threads"를 일관되게 구분해 쓰겠다.

이 차이를 처음에 정리해두면 운영 메트릭을 보는 눈이 다르게 잡힌다. 톰캣에서 보던 "active threads" 그래프 대신, Node에서는 **이벤트 루프 lag**이 핵심 지표가 된다. 그리고 워커 풀이 포화됐는지를 따로 보는 메트릭도 필요해진다(이건 6장에서 본격적으로 다룬다).

## 이벤트 루프를 막는다는 것의 실체 — 코드 한 줄의 무게

자, 그렇다면 처음에 던졌던 의심으로 돌아오자. 단일 스레드가 막힌다는 건 무슨 의미일까? 코드 한 줄로 시연해보자.

```javascript
// JS — 한 줄이 모든 사용자를 줄세우는 그림
const http = require('node:http');

http.createServer((req, res) => {
  if (req.url === '/heavy') {
    // CPU 바운드 — 메인 스레드를 점유한다
    const start = Date.now();
    while (Date.now() - start < 2000) { /* spin */ }
    res.end('done after 2s of blocking');
  } else {
    res.end('hello, world');
  }
}).listen(3000);
```

이 서버를 띄워두고 두 개의 터미널을 열자. 한쪽에서 `curl localhost:3000/heavy`를 호출하고, 동시에 다른 쪽에서 `curl localhost:3000/`을 반복 호출해보자. 무슨 일이 벌어지는가? `/heavy`가 도는 2초 동안, **`/`로 들어오는 모든 요청도 같이 멈춘다.** 응답이 없다. 새 커넥션은 OS 레벨에서 받아들여지지만, 그 콜백이 메인 스레드 위에서 실행되지 못한다. 이게 우리가 도입에서 본 월마트의 convoy effect를 코드 한 줄로 재현한 모습이다. 페이지 한 번 그리는 데 80ms가 걸리는 동안, 100개의 요청이 동시에 들어오면 마지막 요청은 8초를 기다리는 셈이다. 평소엔 문제 안 되던 코드가 트래픽이 몰리는 순간 운영 사고로 모양을 바꾼다.

같은 시나리오를 `ab`(Apache Bench) 같은 부하 도구로 정량으로 보면 더 분명해진다. 평소 `ab -n 1000 -c 50 localhost:3000/`은 p99이 5ms 안팎으로 나올 것이다. 그런데 위의 `/heavy` 한 번을 그 와중에 끼워 넣으면 p99이 2초를 넘긴다. 단순히 한 사용자만 느려지는 게 아니라, **그 2초 동안 도착한 모든 요청의 응답시간이 같이 부풀어 오른다.** 이걸 그래프로 보면 컨보이의 모양이 그대로 그려진다. 한 차량이 멈추니 뒤따르던 모든 차량이 같이 멈추는 그 그림이다.

Spring/톰캣에서는 이런 일이 안 벌어진다. 한 스레드가 2초 동안 자고 있어도, 풀의 다른 스레드가 다른 요청을 받는다. 풀이 200개라면 199개는 멀쩡히 돌아간다. 그래서 우리는 그동안 "동기 코드의 무게"에 둔감해도 됐다. JDBC가 막혀도, 이미지 변환 한 번이 200ms 걸려도, 풀이 충분하면 그냥 돌아갔다. **하지만 Node에서는 그 둔감함이 운영 사고가 된다.** 동기 crypto 한 번, 큰 JSON.parse 한 번, 정규식 catastrophic backtracking 한 번이 그대로 모든 사용자를 줄 세운다. 운영에서 자주 문제가 되는 건 이런 모양들이다.

- **`crypto.pbkdf2Sync`** 같은 동기 API 사용. 비밀번호 해시가 100ms씩 걸린다면, 그게 그대로 메인 스레드를 점유한다. `pbkdf2`(콜백/Promise 버전)는 워커 풀로 빠져나가지만 `pbkdf2Sync`는 여기에 박힌다.
- **수 MB짜리 JSON.parse / JSON.stringify**. 직렬화 한 번이 수백 ms를 잡아먹을 수 있다. 큰 응답은 streaming JSON으로 풀거나, 스키마 기반 직렬화(Fastify의 빌트인이 그 자리)를 쓰는 편이 낫다.
- **재앙적 백트래킹을 일으키는 정규식**. `/^(a+)+$/`에 `aaaaaaaaaaaaaaaaX` 같은 입력이 들어오면 V8의 정규식 엔진이 지수 시간 동안 메인 스레드를 잡는다. Netflix가 한 번 호되게 당했고, 6장에서 그 사례를 일화로 다시 본다.
- **`fs.readFileSync` 같은 동기 파일 I/O**. 작은 설정 파일을 부팅 시점에만 읽는 정도라면 괜찮지만, 요청 핸들러 안에서 동기로 부르면 즉시 컨보이다.
- **무거운 템플릿 엔진 렌더링**. Pug/Handlebars 같은 템플릿이 큰 데이터를 받아 수 ms 이상 걸리면, 그게 메인 스레드 점유다. 월마트의 사례가 정확히 여기 속한다 — SSR 렌더링이 곧 CPU 바운드 작업이고, 큰 트래픽 위에서 컨보이의 본체가 된다.

같은 일감을 동기로 짠 코드와 비동기로 짠 코드가 운영 환경에서 어떻게 갈리는지를 짧은 비교로 보자. 사용자 비밀번호를 검증하는 핸들러를 두 가지로 짜보면 이렇다.

```typescript
// TS — 같은 작업, 두 가지 방식. 운영 결과는 천지차이
import { pbkdf2, pbkdf2Sync } from 'node:crypto';
import { promisify } from 'node:util';

const pbkdf2Async = promisify(pbkdf2);

// (X) 메인 스레드를 잡는 동기 호출 — 모든 요청이 줄을 선다
function verifyBlocking(password: string, salt: Buffer, expected: Buffer): boolean {
  const derived = pbkdf2Sync(password, salt, 100_000, 64, 'sha512');
  return derived.equals(expected);
}

// (O) libuv 워커 풀로 위임 — 메인 스레드는 다음 일을 한다
async function verifyNonBlocking(password: string, salt: Buffer, expected: Buffer): Promise<boolean> {
  const derived = await pbkdf2Async(password, salt, 100_000, 64, 'sha512');
  return derived.equals(expected);
}
```

두 함수의 표면은 거의 같지만, 운영에서는 정반대의 결과를 만든다. `verifyBlocking`은 메인 스레드 위에서 100ms씩 도는 동기 작업이라 트래픽이 몰리는 순간 컨보이를 만든다. `verifyNonBlocking`은 같은 작업을 워커 풀로 보내고 그동안 메인 스레드는 다른 콜백을 처리한다. 100ms × 100건이라면 전자는 마지막 사용자가 10초를 기다리고, 후자는 워커 4개가 병렬로 처리하니 약 2.5초 안에 다 끝난다(워커 풀 사이즈를 고려하지 않은 단순 계산). **이 차이가 동시성 모델의 본체다.** Spring 출신이 처음 Node 코드를 보면 `Sync` 접미사가 붙은 API와 안 붙은 API가 같이 등장하는 게 어색한데, 그 어색함의 정체가 여기 있다. 동기 API는 거의 부팅 시점이나 스크립트성 코드에서만 쓰이고, 요청 핸들러 안에서는 비동기 API가 기본값이다.

그래서 등장하는 운영 격언이 "**Don't block the event loop.**"이다. Node 공식 문서가 한 페이지를 통째로 할애해 강조하는 그 격언이다. 처음 보면 추상적인 권고처럼 들리지만, 위 코드 한 줄을 직접 띄워보면 그 격언이 왜 추상이 아니라 운영 함정의 본체인지 단번에 이해된다. 이걸 막지 못하면 단일 스레드는 트래픽을 못 감당한다. 막아내는 한, 단일 스레드는 정말 잘 감당한다. **조건은 그저 한 가지다 — 메인 스레드를 길게 점유하는 코드를 쓰지 말 것.** 이게 V8 + libuv 모델의 거의 유일한 청구서다.

그렇다면 **막혔는지 어떻게 아는가?** 이 지점에서 첫 번째 운영 도구가 등장한다.

## 이벤트 루프 lag을 측정하는 법

Node 12부터 표준 라이브러리에 들어온 `perf_hooks.monitorEventLoopDelay()`가 답이다. 짧은 TS 코드로 운영 메트릭을 어떻게 뽑는지 살펴보자.

```typescript
// TS — strict mode. 이벤트 루프 lag의 분포를 주기적으로 찍는다
import { monitorEventLoopDelay, IntervalHistogram } from 'node:perf_hooks';

const histogram: IntervalHistogram = monitorEventLoopDelay({ resolution: 20 });
histogram.enable();

setInterval((): void => {
  // 단위는 나노초 — 밀리초로 환산해서 본다
  const p50 = histogram.percentile(50) / 1e6;
  const p95 = histogram.percentile(95) / 1e6;
  const p99 = histogram.percentile(99) / 1e6;
  const max = histogram.max / 1e6;

  console.log(
    `loop lag — p50: ${p50.toFixed(2)}ms  ` +
    `p95: ${p95.toFixed(2)}ms  ` +
    `p99: ${p99.toFixed(2)}ms  ` +
    `max: ${max.toFixed(2)}ms`,
  );
  histogram.reset();
}, 5000);
```

이 5초짜리 샘플러를 운영 환경에 띄워두면 이벤트 루프가 얼마나 매끄럽게 도는지가 그래프로 보인다. 경험적으로 **p99 lag이 50ms를 넘으면 의심**이 시작된다. 정상적으로 도는 서비스는 보통 5ms 이하에서 머문다. 50ms를 넘기 시작하면 어딘가에서 동기 작업이 메인 스레드를 잡고 있다는 신호다. 100ms를 넘으면 이미 사용자에게 체감되고, 1초를 넘으면 사고다. 8장에서 후퇴 신호 체크리스트를 다룰 때 이 지표가 다시 등장하니, 1장 단계에서 이 그림 하나만 머릿속에 잡아두자.

원리는 간단하다. `monitorEventLoopDelay`는 내부적으로 짧은 간격(`resolution`)으로 timer를 등록하고, 그 timer가 실제로 호출된 시각과 예상 시각의 차이를 히스토그램에 기록한다. 이벤트 루프가 막혀 있으면 timer 콜백이 늦게 실행되고, 그 지연이 그대로 lag 값이 된다. 톰캣에서 GC 멈춤을 측정하던 그림과 본질적으로 같다. **"멈춤이 길어지면 의심한다"**는 원칙은 두 런타임이 똑같이 따른다. 다만 측정 도구의 이름과 단위가 살짝 바뀔 뿐이다.

라이브러리를 쓰지 않고 직접 단순 lag 검출만 하고 싶으면 한 줄로도 가능하다.

```typescript
// TS — 가장 단순한 lag 검출. 100ms 인터벌로 예상보다 늦은 시간을 잰다
const INTERVAL_MS = 100;
let prev: number = Date.now();

setInterval((): void => {
  const now = Date.now();
  const lag = now - prev - INTERVAL_MS;
  if (lag > 50) {
    console.warn(`[loop-lag] ${lag}ms behind expected`);
  }
  prev = now;
}, INTERVAL_MS);
```

원리가 같다. 100ms마다 깨어야 하는 콜백이 늦게 깨어나면, 그 지연이 lag이다. 운영 환경에 본격적으로 박을 거라면 `perf_hooks` 쪽이 분포까지 돌려주니 더 풍부하지만, 빠르게 디버깅용으로 띄우기엔 후자도 충분하다. Java의 `MicrometerJVMGcMetrics`나 `VisualVM`에서 GC 멈춤 시간을 보던 그 자리에, Node에서는 이벤트 루프 lag이 들어선다. 같은 자리, 다른 이름이다. 운영 직관은 그대로 옮겨가도 된다. 6장에서 디버깅 도구 체인을 옮기는 이야기를 본격적으로 다루면, 이 한 도구가 사실 빙산의 일각임을 알게 된다.

운영에 박을 때 한 가지 디테일을 새겨두자. 이 lag 메트릭은 Prometheus, Datadog, OpenTelemetry 같은 표준 메트릭 백엔드로 그대로 흘려보내는 편이 낫다. JVM에서 우리가 GC 시간을 알람으로 묶어뒀듯, Node에서는 이벤트 루프 lag p99을 알람으로 묶는다. 7장에서 운영 체크리스트로 다시 만나는 항목이다.

## Worker Threads — 진짜로 무거운 일을 옮기는 자리

이벤트 루프를 막지 말라는 격언을 받아들이면, 자연스러운 다음 질문이 따라온다. **그러면 진짜로 무거운 CPU 작업은 어디서 돌리지?** 이미지 리사이즈, PDF 생성, 큰 파일의 SHA-256 해싱, 무거운 정규식 처리, 머신러닝 추론. 이런 작업을 메인 스레드에서 돌리면 확실히 막힌다. libuv 워커 풀에 던지자니 풀의 본래 용도는 그게 아니고, 풀 사이즈를 마구 늘리는 것도 좋은 답이 아니다.

Node 10.5부터 들어온 **Worker Threads**가 이 자리를 메운다. 이름이 헷갈리지만 libuv 워커 스레드 풀과는 완전히 다른 부품이다. Worker Threads는 **별도의 V8 인스턴스를 가진, 진짜로 분리된 JS 실행 환경**이다. 메모리도 따로, 이벤트 루프도 따로 돈다. 메인 스레드와는 `MessagePort`로 메시지를 주고받는다. 한 마디로 같은 프로세스 안에 작은 별도 Node가 하나 더 도는 그림이다.

```typescript
// TS — main.ts: 무거운 작업을 워커로 위임한다
import { Worker } from 'node:worker_threads';
import { fileURLToPath } from 'node:url';

function runHeavyTask(input: number): Promise<number> {
  return new Promise((resolve, reject) => {
    const worker = new Worker(
      fileURLToPath(new URL('./heavy.worker.ts', import.meta.url)),
      {
        workerData: input,
        execArgv: ['--require', 'ts-node/register'],
      },
    );
    worker.once('message', (value: number) => resolve(value));
    worker.once('error', reject);
    worker.once('exit', (code: number) => {
      if (code !== 0) reject(new Error(`worker exited ${code}`));
    });
  });
}

// 사용 예 — 메인 이벤트 루프는 그대로 돈다
const result = await runHeavyTask(40);
console.log('fibonacci(40) =', result);
```

워커 쪽은 별도 파일에 두고, `parentPort`로 결과를 돌려보낸다. 이렇게 분리하면 메인 스레드는 자기 일을 계속 하고, 무거운 계산은 옆 V8 인스턴스에서 따로 돈다. 모양만 보면 Java의 ForkJoinPool에 작업을 던지는 그림과 닮았다. **다만 비용이 다르다.** ForkJoinPool의 워커 스레드는 같은 JVM·같은 힙을 공유한다. 객체 참조를 그대로 던질 수 있다. 반면 Worker Threads는 V8 인스턴스 자체가 따로다. 메모리도 격리된다. 데이터를 넘기려면 직렬화하거나(`postMessage`의 기본 동작 — structured clone), `SharedArrayBuffer`로 명시적으로 공유해야 한다. 한 워커를 띄우는 데 메모리도 수십 MB 단위로 들고, 시작 비용도 무시 못 한다. JVM의 스레드와는 무게가 한참 다르다.

그래서 Worker Threads의 운영 패턴은 보통 **풀로 만들어 재사용**한다. `piscina` 같은 라이브러리가 사실상의 표준이다. ThreadPoolExecutor에 작업을 던지듯, 풀에 작업을 던지고 결과를 받는다. 코드 모양도 우리가 익숙하던 그림에 가깝다.

```typescript
// TS — piscina로 워커 풀 재사용
import Piscina from 'piscina';
import { resolve } from 'node:path';

const pool = new Piscina({
  filename: resolve(__dirname, 'heavy.worker.js'),
  minThreads: 2,
  maxThreads: 8,
  idleTimeout: 30_000,
});

// 라우트 핸들러 안에서 동시에 여러 작업을 돌려도 메인 루프는 멀쩡
const [pdf, thumbnail] = await Promise.all([
  pool.run({ html, kind: 'pdf' }),
  pool.run({ image, kind: 'thumbnail' }),
]);
```

이게 ThreadPoolExecutor의 직관과 거의 같다. `corePoolSize`, `maximumPoolSize`, `keepAliveTime`을 설정하던 그 그림 위에 `minThreads`, `maxThreads`, `idleTimeout`이 거의 1:1로 매핑된다. **다만 한 가지는 분명히 새겨두자 — 가벼운 CPU 작업이라면 Worker Threads는 오히려 손해다.** 직렬화 비용과 워커 시작/메시지 전달 비용이 작업 자체보다 클 수 있다. 진짜로 무거운, 수백 ms 이상 걸리는 작업에만 어울리는 도구다. 이게 Java/Spring에서 ThreadPoolExecutor를 거의 모든 비동기 작업에 막 던지던 습관과 살짝 다른 운영 직관이다. 짧은 비동기는 async/await 자체로 풀고, 긴 CPU 작업만 워커로 옮긴다 — 이게 Node 쪽의 자연스러운 패턴이다.

운영 차원에서 또 하나 짚어둘 것은, **Worker Threads도 무한정 늘리면 V8 인스턴스의 합산 메모리가 곧 호스트 메모리를 잡아먹는다**는 점이다. 한 워커당 50MB라면 16개 풀은 800MB다. 메인 프로세스의 1.5GB 힙에 더해진다. 그래서 Node 운영의 가로 확장 직관은 워커 풀에서도 그대로 적용된다 — **풀을 키우는 것보다 인스턴스를 가로로 늘리는 게 보통 더 자연스럽다.** 한 인스턴스의 자원 한계를 빨리 인정하는 게 Node 운영의 일종의 미덕이다.

## Java Virtual Threads와 async/await — 같은 목적지, 다른 길

여기까지 오면 자연스러운 비교가 떠오른다. **JDK 21의 가상 스레드는 어떤 자리에 들어가는가?** 같은 동시성 문제를 푸는 또 하나의 답이고, Java 진영에서 비교적 최근에 받아든 무기다. 두 모델을 짧게 마주 세워보자.

Java의 가상 스레드(Project Loom의 결과물)는 사용자에게 "스레드가 무한히 싸다"는 환상을 준다. 동기적인 코드 그대로 — `Thread.sleep`, blocking JDBC 호출, blocking HTTP 클라이언트 — 를 그대로 쓰면서, 백만 개의 동시 요청을 받아낼 수 있다. JVM 런타임이 가상 스레드를 OS 스레드(캐리어 스레드) 몇 개에 다중화해주기 때문이다. blocking 호출이 들어오면 가상 스레드가 캐리어에서 잠시 떨어져 자고, 캐리어는 다른 가상 스레드를 받아 실행한다. 코드 모양은 옛날의 톰캣과 똑같다. 한 요청 = 한 (가상)스레드. 다만 그 스레드가 사실은 캐리어 스레드 위에서 yield되는 가상의 것이다. Spring Boot 3.2부터는 임베디드 톰캣에 가상 스레드를 그대로 끼워 쓸 수 있다. 코드 한 줄도 안 바꾸고 처리량이 몇 배로 뛰는 그림이 가능하다(워크로드에 따라).

Node의 async/await은 다른 길로 같은 곳에 도착한다. **코드 모양 자체를 비동기로 바꿔놓는 길이다.** `await` 한 줄은 함수를 거기서 잘라내, 콜백으로 마이크로태스크 큐에 다시 등록한다. 한 메인 스레드 위에서 마이크로태스크들이 끊임없이 들고 나며 수만 커넥션의 진행 상태를 한꺼번에 들고 있는다. 코드 모양이 동기처럼 보이지만, 사실 컴파일러가 그 모양을 비동기 상태 기계로 풀어낸 것이다. 이 점에서 Kotlin의 코루틴과 더 가깝다. Kotlin의 `suspend fun` ↔ JS의 `async function`, Kotlin의 `delay(1000)` ↔ JS의 `await sleep(1000)`. 두 언어의 문법 모양이 거의 거울이다.

두 모델 중 어느 쪽이 더 나을까? 솔직히 말하면 **둘 다 훌륭하다.** 가상 스레드는 기존 동기 코드를 거의 그대로 옮길 수 있다는 점에서, 수십 년 쌓인 Java 자산을 살리는 데 강하다. 동시성 모델 자체가 직관적이다 — "한 요청 = 한 스레드"라는 그림은 신규 엔지니어에게 가르치기도 쉽다. async/await은 비동기 작업의 흐름을 코드 위에 명시적으로 보여준다는 점에서, 어디서 yield되는지가 한눈에 들어온다. `await`이 박힌 자리가 곧 다른 작업이 끼어들 수 있는 자리다. 사고 회로가 다르다. **백엔드 개발자로서 우리가 두 진영을 다 들고 가야 하는 이유**가 여기 있다. 한쪽이 다른 쪽을 대체하는 그림이 아니라, 같은 동시성 문제를 다른 사고 회로로 푸는 두 가지 길이다. 선호의 문제이지 우열의 문제가 아니다.

다만 두 모델의 함정이 모양이 다르다는 점을 짚어두자. 가상 스레드는 `synchronized` 블록 안에서 pinning되면 캐리어 스레드를 잡는다. JNI 호출이나 네이티브 stack frame 안에서도 비슷하게 pin된다. 우리가 JDK 21로 옮기면서 ReentrantLock으로 갈아끼우는 작업을 하게 되는 이유다. async/await은 다른 함정이다. `await`를 빼먹으면 Promise가 대기되지 않은 채 흘러가버린다. `forEach` 안에서 `await`를 쓰면 동작하지 않고, `Promise.all`을 빠뜨리면 직렬화된다. 안 잡힌 rejection이 프로세스를 죽인다. 같은 류의 함정을 풀어가는 학습 곡선이 두 모델 모두에 있다. Spring 출신이 Node로 넘어올 때 이 학습 곡선의 절반 이상이 async/await의 함정에서 나온다. 2장에서 이 함정의 모양을 정직하게 다시 본다.

한 가지 운영 차원의 차이도 있다. 가상 스레드는 JVM이 캐리어 풀을 자동으로 관리해주니 운영자가 별도 메트릭을 새로 학습하지 않아도 된다. 기존 GC·스레드 메트릭이 그대로 통한다. async/await은 이벤트 루프 lag, 워커 풀 포화도 같은 새 메트릭을 익혀야 한다. Spring 출신이 가지는 작은 학습 비용이 여기서 나온다. 책 전체에서 이 새 메트릭들을 하나씩 정리해 가지만, 1장 단계에서는 lag 한 개만 머릿속에 박아두면 된다.

## GC 한 번 더 — 큰 힙을 안 다루는 이유

V8의 GC를 앞서 짧게 다뤘는데, 운영 차원에서 한 번 더 들여다볼 만하다. 우리가 JVM에서 익숙했던 그림은 이렇다. 한 인스턴스에 큰 힙(8GB, 16GB, 더 가면 32GB)을 박아두고, G1이나 ZGC가 STW 시간을 백 밀리초 또는 그 이하로 막아준다. 한 프로세스가 두꺼운 일감을 들고 가는 모델이다. ZGC가 도착한 뒤로는 100GB 힙도 거의 정지 시간 없이 돌리는 사례가 등장한다. Spring Boot 운영자에게 "힙 키워" 같은 조언이 자연스러웠던 이유다.

V8은 이 그림을 따라가지 않는다. 기본 힙이 작고(보통 1.5GB 안팎), `--max-old-space-size`로 늘려도 4~8GB 부근에서 운영을 멈추는 게 일반적이다. 왜 그럴까? 두 가지 이유가 겹친다. 첫째, **V8의 major GC는 큰 Old Space를 stop-the-world로 처리할 때 멈춤 시간이 빠르게 늘어난다.** ZGC 같은 동시 GC가 아직 V8에는 없다. Incremental Marking, Concurrent Sweeping 같은 기법으로 멈춤을 줄이긴 하지만, 본질적으로 큰 힙에 강한 컬렉터는 아니다. 둘째, **Node의 운영 패턴 자체가 "프로세스를 가로로 늘리는" 방향으로 굳어졌다.** 한 프로세스가 큰 힙을 들고 가는 게 아니라, 여러 프로세스(또는 컨테이너)를 띄워 트래픽을 분산하는 모델이다. PM2 cluster, K8s pod, 컨테이너 가로 스케일링이 모두 이 패턴 위에 있다.

이걸 알면 운영 결정 두 가지가 자연스럽게 따라나온다. 첫째, **Node 인스턴스를 거대하게 키우려 하지 말고, 자잘하게 여러 개로 쪼개는 편이 낫다.** 톰캣에서 쓰던 "한 인스턴스 16GB"의 직관 대신, "한 인스턴스 1~2GB, 인스턴스 N개"로 옮긴다. CPU 코어 한 개당 한 프로세스가 보통의 출발선이다(8 vCPU 박스라면 8개 프로세스, PM2 cluster가 자동으로 처리해준다). 메모리 한도는 컨테이너 단위로 박아두고, 그걸 넘기면 K8s가 재시작하도록 둔다.

둘째, **메모리가 OOM으로 죽으면 다시 띄우면 그만**이라는 직관을 가져도 된다. JVM에서는 OOM이 사고였지만, Node 운영에서는 K8s가 재시작하는 흔한 이벤트로 다루는 편이 자연스럽다. 단, **graceful shutdown만 잘 박아두면 된다.** SIGTERM을 받았을 때 (1) 새 커넥션 거절, (2) 진행 중인 요청 완료 대기, (3) 정해진 grace period 후 종료 — 이 세 단계만 갖추면 트래픽이 끊기지 않는다. 톰캣 시절의 graceful shutdown과 본질이 같다. 도구만 바뀌었을 뿐 푸는 문제는 똑같다. 이 디테일은 7장에서 다룬다.

여기서도 살짝 찜찜한 지점이 있긴 하다. 큰 힙으로 한 인스턴스를 묵직하게 굴리던 우리에게는 "여러 개로 쪼갠다"는 운영 패턴이 처음엔 어색하다. 인스턴스 간 캐시는 어떻게 공유하지? 세션은? in-memory 캐시는? 답은 대부분 **외부로 빼낸다**이다. Redis, Memcached, 데이터베이스로. JVM 시절에 EhCache나 Hazelcast로 풀던 자리가, Node에서는 외부 캐시로 자연스럽게 옮겨간다. 도구는 바뀌었지만 푸는 문제는 똑같다 — **멀티 노드 환경의 상태 공유.** 우리가 익숙한 그 문제다. 인스턴스가 무거우냐 가벼우냐의 차이일 뿐, 멀티 노드의 본질은 두 런타임 모두 같다.

또 한 가지, V8 운영에서 자주 쓰는 플래그를 짧게 정리하자.

- `--max-old-space-size=2048` — Old Space 한도(MB). 컨테이너 메모리에 맞춰 박는다.
- `--max-semi-space-size=64` — Young Generation의 한 반쪽 크기. 단명 객체가 많은 워크로드에 키운다.
- `--inspect=0.0.0.0:9229` — Chrome DevTools 디버거 연결.
- `--heapsnapshot-signal=SIGUSR2` — 신호 받으면 힙 스냅샷 저장.

JVM의 `-Xmx`, `-Xms`, `-XX:NewRatio`에 대응되는 자리들이다. 이름만 바뀐 셈이다. 6장에서 디버깅 도구 매핑 표로 다시 만난다.

## 마무리 — 단일 스레드는 감당한다, 다만 조건이 붙는다

처음 던진 질문으로 돌아오자. 단일 스레드는 정말 일감을 감당하는가? 답은 **그렇다, 하지만 조건이 있다**이다. 조건은 두 줄로 요약된다.

첫째, **메인 스레드를 막지 마라.** CPU 바운드 작업, 동기 crypto, 큰 JSON 파싱, 정규식 backtracking, 동기 파일 I/O — 이들이 메인 스레드 위에 올라오는 순간 모든 사용자가 줄을 선다. Walmart의 convoy effect는 추상이 아니라 코드 한 줄에서 발생하는 운영 함정이다. 이걸 못 막으면 단일 스레드는 트래픽을 못 감당한다. 막아내는 한, 단일 스레드는 정말 가볍게 잘 감당한다. Walmart가 1% CPU로 55%의 트래픽을 받아낸 그 그림이 그 증거다.

둘째, **막혔는지 측정하라.** `perf_hooks.monitorEventLoopDelay()`로 p99 lag을 보는 습관, 50ms 임계를 머릿속에 두는 직관, 막혔을 때 Worker Threads로 옮길 수 있는 분리 능력. 이 셋이 운영 기본기다. 톰캣 시절에 GC 멈춤 그래프를 보던 자리에, 이제 이벤트 루프 lag 그래프가 들어선다. 같은 자리, 다른 이름이다.

이 두 줄을 잡으면, V8 + libuv 모델은 정말 가볍고 강력한 런타임이 된다. 네트워크 I/O가 압도적으로 가볍고, 컨테이너 가로 확장이 자연스럽고, 콜드 스타트가 짧고, 람다/엣지/서버리스 환경에 자연스럽게 들어맞는다. JVM이 잘하는 자리(큰 힙, 두꺼운 일감, blocking 라이브러리 자산)와는 다른 자리에서 강하다. 두 런타임이 서로를 대체하기보다 보완한다는 그림이, 1장 단계에서 한 번 잡히는 게 좋다.

운영 차원에서 1장이 남기는 메트릭 체크리스트도 짧게 정리해두자. 7장에서 본격적으로 다시 만나지만, 이름만이라도 머릿속에 박아두면 좋다.

- **이벤트 루프 lag p50/p95/p99** — `perf_hooks.monitorEventLoopDelay()`로 측정. p99 50ms가 첫 번째 의심 임계.
- **libuv 워커 풀 활용도** — DNS·파일·crypto가 풀을 점유하는지. 4개 기본 풀이 자주 포화되면 `UV_THREADPOOL_SIZE`를 키운다.
- **V8 힙 사용량** — `process.memoryUsage().heapUsed`와 `heapTotal`. Old Space 한도에 가까워지면 GC 빈도가 폭증한다.
- **GC 멈춤 시간** — `--trace-gc`로 콘솔 출력, 또는 `v8.getHeapStatistics()`. 한 번에 100ms 이상 멈추면 의심.
- **프로세스당 메모리** — RSS 기준. K8s OOMKilled 빈도와 같이 본다.

이 다섯 개가 톰캣 시절의 active threads, 풀 사용률, GC 그래프, 힙 사용량의 자리에 들어선다. 이름만 바뀌었을 뿐 보는 관심사는 똑같다.

한 가지 짚어둘 게 더 있다. **이 책의 정체성을 1장에서 한 번 못 박아두자.** 우리는 Spring을 떠나 Node로 갈아타려는 게 아니다. 두 번째 런타임을 받아들이는 것이다. JVM 자산은 그대로, 거기에 Node가 잘하는 자리(IO 바운드, BFF, 엣지, SSR, 서버리스)를 한 손 더 갖는다. 1장이 의심에 답한다면("정말 일감을 감당하는가" — 그렇다, 조건이 붙는다), 9장은 그 답을 조직 안에서 풀어낼 언어로 옮긴다. 에필로그에서는 두 런타임을 같이 굴리는 백엔드 개발자의 모양으로 닫는다. 책 전체가 같은 메시지를 세 자리에서 다시 박는다 — **도구만 바뀌었을 뿐, 우리는 여전히 백엔드 개발자다.**

월마트가 1% CPU로 55%의 트래픽을 받아낸 그 그림과, 페이지 렌더링이 컨보이를 만들어낸 그 그림은 사실 같은 동전의 양면이다. 단일 스레드 모델의 가벼움과 단일 스레드 모델의 함정이 같은 메커니즘에서 나온다. 메인 스레드가 잠을 자지 않고 콜백을 디스패치하기만 하기에 가볍고, 메인 스레드 한 개라서 그 한 개가 잡히면 전부 멈춘다. 이 양면을 받아들이는 게 Node 운영의 출발점이다. 8장에서는 같은 Walmart 사례를 결정 차원으로 다시 만난다 — "왜 SSR을 CDN으로 빼야 했는가", "어디서부터 마이그레이션했는가"의 운영 결정. 1장에서 우리가 본 것은 그 결정의 토대가 되는 메커니즘이다. 같은 사례를 두 각도에서 한 번씩 본다는 약속은 책 전체에 걸쳐 깔린다.

이 장에서 짚은 것을 한 줄로 정리하면 이렇다. **도구는 바뀌었다. JVM의 OS 스레드 풀이 V8 + libuv의 단일 메인 스레드 + 워커 풀로 모양을 바꿨다. 하지만 우리가 풀고 있는 동시성 문제 자체는 같다.** 멈춤을 측정하는 것, 무거운 일을 분리하는 것, 가로로 확장하는 것, 외부 캐시로 상태를 빼는 것 — 이 모든 백엔드 직관이 그대로 통한다. 톰캣에서 active thread 그래프를 보던 사람이 Node에서 이벤트 루프 lag 그래프를 본다. ThreadPoolExecutor를 설정하던 사람이 piscina 풀을 설정한다. CompletableFuture로 비동기를 풀던 사람이 async/await으로 푼다. 도구의 이름이 바뀌고 디테일이 살짝 다를 뿐, 사고 회로의 본체는 옮겨가지 않는다. 이게 이 책이 처음부터 마지막까지 끌고 갈 메시지다. 우리는 도구만 바꿨을 뿐, 여전히 백엔드 개발자다.

자, 그렇다면 그 도구의 표면을 살펴볼 차례다. JavaScript와 TypeScript는 Java/Kotlin과 어디까지 닮았고, 어디서부터 다른 사고 회로를 요구할까? 클로저와 익명 클래스, 화살표 함수의 `this`, 구조적 타입과 명목적 타입, 그리고 Promise가 CompletableFuture와 모양만 닮은 그 함정들을 다음 장에서 정면으로 들여다보자.
