# 8장. Java/Spring에서 Node.js로 — 마이그레이션 전략

기술 블로그를 한참 읽다 보면 마음이 묘하게 기울어지는 순간이 온다. PayPal이 갈아탔고, LinkedIn이 갈아탔고, Netflix가 BFF를 떼어냈고, Walmart가 Black Friday를 Node로 받아냈다는 글들이 줄지어 보인다. 한국에서도 줌인터넷이 코드 70%를 줄였고, 당근이 푸시알림을 떼어냈고, 직방의 자바 개발자가 NestJS에 적응했다는 회고가 쌓여 있다. 그 글들을 차례로 읽다 보면 어느새 이런 결론에 도달하기 쉽다. "Spring이 무거운 게 맞았다. 우리도 옮겨야겠다."

그런데 같은 검색창에 "Uber Node 후퇴"를 쳐 보자. 2018년 Uber Engineering 블로그가 나온다. RTAPI를 Node로 만들었던 그 회사가 신규 권장 스택에서 Node + HTTP/JSON을 빼냈다는 글이다. 이유는 단순하다. 새로 들어오는 엔지니어들이 Node 사내 라이브러리를 익히는 시간이 너무 오래 걸려서. "갈아탔다"는 사례만 모아서 보면 한쪽으로 쉽게 기우는데, 같은 책장에 후퇴 사례를 한 권 더 꽂아 두면 그림이 다르게 보인다.

이 장에서 하려는 일은 그 그림을 똑바로 펴 두는 것이다. 1·3·4·6장에서 우리는 사례 네 건을 도입 일화로 짧게 만났다. 1장 도입에서 Walmart의 convoy effect를 봤고, 3장 도입에서 LinkedIn이 "다른 서비스와 대화하기"가 강점이라는 교훈을, 4장 도입에서 PayPal의 Express 비강제성과 사내 Kraken.js 컨벤션 강제 일화를, 6장 도입에서 Netflix가 production에서 정규식 한 줄을 flame graph로 잡아낸 이야기를 봤다. 이번엔 그 사례들을 결정 차원에서 다시 보자. 거기에 한국 사례 다섯과 Uber 후퇴까지 합쳐서, "어디부터 자를 것인가, 그리고 어떤 신호가 보이면 멈출 것인가"의 결정 기준을 산문으로 정리하려 한다.

먼저 한 가지 약속하자. 이 장은 "Node로 가야 한다"의 일방적 권유가 아니다. "어디부터 어떻게 가야 하는가"를 묻고, 그 질문 끝에 "갈 필요가 없는 곳도 있다"는 답까지 같이 두는 자리다. 사례 열 건을 다 본 다음, Martin Fowler의 Strangler Fig 패턴을 단계별로 풀고, NestJS BFF 한 조각의 최소 코드를 직접 적어 본다. 그러고 나서야 결정 가이드와 후퇴 신호, 6개월 로드맵을 이어 붙인다. 챕터가 길지만, 길어야 하는 이유가 있다. 도구의 결정은 보통 한 장의 표로 끝내고 싶어지는데, 실무에서는 한 장의 표가 가장 위험한 결정 도구다. 사례를 충분히 본 다음에야 그 표를 다시 그릴 수 있다.

한 가지 더 미리 정리하자. 이 장에서 자주 쓰는 "BFF"라는 줄임말이 처음인 독자를 위해 한 단락. **Backend for Frontend**는 클라이언트(웹·모바일·SSR 페이지) 한 종류만을 위해 설계된 백엔드 계층이다. 모놀리스의 핵심 도메인 서비스를 그대로 두고, 그 앞에 "이 클라이언트가 한 페이지를 그리기 위해 필요한 데이터를 합산해 주는 얇은 층"을 새로 둔다. PayPal의 계정 개요 페이지, Netflix의 UI 서버, 줌인터넷의 모바일 줌이 모두 이 BFF 자리다. Sam Newman이 책 *Building Microservices*에서 이 이름을 정착시켰는데, 마이그레이션 사례 거의 모두가 이 자리에서 시작한다는 사실이 우연이 아니다. 가장 안전한 자리이고, 가장 빠른 가치가 나오는 자리이고, Node의 강점이 가장 잘 맞는 자리다. 우리 책의 8장이 이 자리를 한가운데에 두고 사례 열 건을 본다.

## 사례 1 — PayPal 계정 개요 페이지

PayPal이 첫 Node 적용지로 고른 페이지가 어디였는지 보면 의외다. "minor application이 아니라 account overview page"였다. PayPal 사용자들이 가장 많이 들어오는 그 페이지를 첫 시범 대상으로 골랐다. 작은 곳에서 안전하게 검증하고 싶은 욕심을 누른 결정이다. 큰 페이지에서 검증해야 의미가 있다는 판단을 먼저 한 셈이다.

비교 조건도 흥미롭다. Java 팀 5명이 1월에 시작했고, Node.js 팀 2명이 3월에 시작했다. 같은 기능을 만드는 일이었다. 결과는 PayPal Tech 블로그에 그대로 남아 있다. Node.js 팀이 동등하거나 더 빠르게 완성했다. 처리량은 Node가 Java 대비 2배 RPS를 보였고, 그것도 단일 코어 vs 5코어 비교였다. 응답 시간은 35% 줄었고, 페이지가 약 200ms 빨라졌다. 코드는 33% 줄었고, 파일 수는 40% 줄었다.

이 수치를 곧이곧대로 일반화하면 곤란하다. 같은 사람들이 두 번째로 만드는 일은 처음보다 빠르게 마련이다. 5명과 2명의 차이도 단순한 인력 비교가 아니다. 그래도 한 가지 신호는 분명히 읽힌다. "사용자에게 가장 많이 노출되는 페이지를 BFF처럼 떼어 Node로 옮겼더니, 같은 일을 더 적은 사람이 더 빠르게 만들었다." 4장 도입에서 짧게 만났던 그 사례를 결정 차원에서 다시 보면 이렇게 정리된다.

그런데 4장에서 보았던 다른 한 면도 같이 기억해야 한다. PayPal은 옮긴 직후 곧 다른 문제를 만났다. Express가 너무 자유로웠다. 5명짜리 작은 팀에서야 좋았는데, 팀이 커지니 어떤 사람은 라우터를 이렇게 쓰고 어떤 사람은 저렇게 썼다. 미들웨어 순서, 에러 처리, 응답 직렬화, 무엇 하나 정해진 모양이 없었다. 결국 사내에서 Kraken.js라는 컨벤션 강제 프레임워크를 직접 만들어야 했다. Node로 옮겨서 행복해진 게 아니라, 옮긴 다음에 "강제할 무언가"를 다시 만들어 끼워야 했다는 뜻이다.

이 두 면을 같이 보면 PayPal의 교훈이 뚜렷해진다. **Spring이 무거웠던 게 사실이지만, "Express만으로 충분하다"는 것도 사실이 아니었다.** 큰 팀이 큰 도메인을 운영할 때는 어떤 형태로든 컨벤션 강제가 필요하다. 그래서 우리가 4장에서 NestJS를 길게 다룬 이유가 여기서 회수된다. NestJS는 PayPal이 Kraken.js로 직접 풀어야 했던 그 문제를 미리 풀어 둔 프레임워크다. 모듈 그래프와 의존성 주입, 인터셉터와 가드와 파이프, 모든 게 정해진 모양으로 들어 있다.

테스트 자산은 어떻게 옮겨갔을까. PayPal은 마이그레이션 회고에서 테스트 이전을 별도 절로 풀지 않았지만, 글의 행간을 보면 신규 코드는 Node에서 새로 짜고, 기존 Java JUnit 자산은 그대로 두는 폴리글랏 패턴이었던 것으로 읽힌다. 4장에서 NestJS 테스트 절을 길게 풀어 둔 이유가 여기에 있다. **마이그레이션은 코드 이전이지 테스트 이전이 아니다.** Java JUnit 자산을 Jest로 일괄 변환하려는 충동을 누르고, 새 코드는 새 테스트로 덮는 게 정직하다.

## 사례 2 — LinkedIn 모바일 백엔드

3장 도입에서 우리는 LinkedIn 사례의 한 단락을 만났다. "Node의 강점은 다른 서비스와 대화하기"라는 교훈이었다. 이번엔 그 사례의 수치까지 같이 펴 보자.

LinkedIn은 모바일 백엔드를 Ruby on Rails에서 Node.js로 옮겼다. 출발점은 Mongrel 인스턴스의 한계였다. Mongrel은 단일 스레드 프로세스인데 메모리 누수가 누적되며 각 인스턴스가 300MB까지 부풀었다. JSON 직렬화 성능도 떨어졌다. 모바일 클라이언트가 자주 호출하는 API의 특성상, 직렬화 자체가 핫패스에 들어 있었다는 점이 결정적이었다.

옮긴 뒤의 결과는 HighScalability와 InfoQ에 정리돼 있다. 서버 30대가 3대로 줄었다. 동시에 10배 부하 헤드룸을 확보했고, 처리는 약 20배 빨라졌다. 메모리 풋프린트도 작아졌고, 코드는 60K 라인에서 2K 라인으로 줄었다. 30분의 1, 20배, 30분의 1. 이 세 숫자가 한 사례에서 동시에 나오는 일은 흔치 않다.

여기서 잠깐 멈춰서, "왜 이 정도로 줄었을까"를 솔직하게 따져 보자. 코드가 60K에서 2K로 줄었다는 사실은 단순히 "Node가 더 간결해서"로 설명되지 않는다. Rails 시절의 코드 베이스에 누적된 우회 처리, 사내 패치, 더 이상 안 쓰는 분기들이 함께 정리됐다고 보는 게 맞다. 두 번째로 만들 때는 첫 번째 만들 때 하지 않아도 됐던 일들을 빼고 시작할 수 있다. 처리 속도 20배도 같은 맥락이다. 단순 런타임 비교가 아니라, 도메인 모델을 깔끔하게 다시 짠 효과가 같이 들어 있다.

그래도 핵심 교훈 하나는 단단하다. **"다른 서비스와 대화하는 일이 주력일 때, Node는 그 일을 매우 잘한다."** 모바일 백엔드는 자체 비즈니스 로직보다 플랫폼 API와 DB와의 대화가 비중이 크다. 요청 하나에 외부 서비스 호출이 다섯 개, 열 개씩 붙는다. 동기적으로 짜면 5초가 걸릴 일이 비동기로 풀면 500ms에 끝난다. Java도 WebClient나 reactive로 같은 일을 할 수 있지만, "기본 모드가 비동기"인 런타임에서 짜는 코드는 여전히 더 단순하다. async/await 한 줄이 그 차이를 만든다.

이 사례가 결정 차원에서 우리에게 주는 신호는 분명하다. 우리 팀의 어느 경계에서 외부 서비스 호출이 가장 많이 일어나는지, 그 경계의 응답 시간이 호출 합산의 직렬화 합인지, 그 답을 알면 BFF 분리의 첫 후보가 보인다. 모바일 클라이언트를 위한 API 게이트웨이, SSR을 위한 페이지 서버, 서드파티 통합 허브 — 이런 곳들이 LinkedIn 패턴이 가장 잘 맞는 자리다.

## 사례 3 — Netflix UI 서버

6장 도입에서 우리는 Netflix가 production에서 정규식 한 줄을 flame graph로 잡아낸 이야기를 봤다. 도구로 측정하지 못하는 것은 결정할 수 없다는 메시지의 사례였다. 이번엔 그 회사가 BFF 분리를 어떻게 결정했는지 다시 보자.

Netflix UI 서버의 출발점은 의외로 작은 불편이었다. 시작 시간이 40분 걸렸다. Java 모놀리스의 빌드와 부팅이 그렇게 됐던 거다. 개발자가 한 줄을 고치고 확인하려면 한 번에 40분을 기다려야 했다. 개발 사이클이 늦으니 코드 품질도 따라 늦어지는 악순환이 생긴다. "이건 도구가 우리를 막고 있다"는 신호다.

Netflix가 고른 길은 두 가지였다. 첫째, Java 모놀리스 자체를 옮기지 않았다. 마이크로서비스(Java/Groovy)는 그대로 두고, 그 앞에 BFF 레이어를 Node.js로 새로 만들었다. 둘째, BFF는 Restify 기반에 도커 컨테이너로 분리했다. 컨테이너 단위로 BFF가 있고, 컨테이너가 Java 마이크로서비스를 프락시한다.

결과는 Netflix Tech Blog에 정리돼 있다. 시작 시간은 40분에서 1분 미만으로 떨어졌고, 응답 시간은 약 70% 줄었다. 응답 시간 70%는 단순한 성능 향상이 아니다. UI 서버에서 70%가 줄었다는 건 사용자가 페이지를 클릭하고 나서 보이기까지의 시간이 그만큼 줄었다는 뜻이다. 시작 시간 40분이 1분이 됐다는 건 개발 사이클이 분 단위로 들어왔다는 뜻이다. 두 효과가 합쳐지면 같은 팀이 같은 시간에 훨씬 많은 실험을 할 수 있게 된다.

그런데 이 사례에서 가장 중요한 건 수치가 아니다. **"Java 마이크로서비스를 그대로 두고 BFF만 분리했다"**는 결정의 모양이다. Netflix는 Node로 갈아탄 게 아니다. Node를 옆자리에 들였다. 핵심 도메인은 여전히 Java/Groovy가 책임지고, UI 합산과 사용자 응답만 Node가 받는다. 이게 이 책 전체에서 우리가 반복해 온 메시지의 가장 단단한 사례다. **두 런타임이 같이 돈다. 한쪽이 다른 쪽을 대체하는 게 아니라, 옆자리에 들어와 자기 일을 한다.**

테스트 자산은 어떻게 됐을까. Netflix는 BFF에 대한 계약 테스트(contract test) 패턴을 적극 활용했다는 점이 여러 글에서 시사돼 있다. Java 마이크로서비스가 노출하는 API 계약을 BFF가 소비자로서 테스트하고, BFF가 노출하는 API 계약을 모바일 클라이언트가 다시 소비자로서 테스트한다. 이렇게 양쪽을 계약 테스트로 막아 두면, BFF가 안전하게 진화할 수 있다. 4장에서 본 NestJS의 `Test.createTestingModule`과 Supertest 통합 테스트 패턴이 이 자리에서 그대로 쓰인다.

flame graph 일화도 같이 회수해 두자. 6장 도입에서 우리는 Netflix가 Node 도입 후에 곧장 flame graph 도구를 직접 만들어 production CPU 핫패스를 분석한 일화를 봤다. 그 일이 어떤 신호인지 다시 보자. **Node로 갈 때는 동시에 측정 도구가 따라가야 한다.** Java에는 JFR과 async-profiler와 VisualVM이 이미 있어서 측정이 자연스럽지만, Node는 직접 chain을 짜야 한다. clinic.js, 0x, perf_hooks, 이벤트 루프 lag 모니터, 이 도구들을 같이 들이지 않으면 production에서 정규식 한 줄에 또 발이 걸린다.

## 사례 4 — Walmart Black Friday

1장 도입에서 우리는 Walmart의 convoy effect를 봤다. "페이지 한 번 렌더링에 10~80ms가 걸리고, 그 사이 들어온 다른 요청들이 줄을 선다." 단일 스레드가 막히면 무슨 일이 벌어지는지 1장이 직접 보여 준 그 일화다. 이번엔 같은 사례를 "왜 SSR을 CDN으로 빼야 했는가"의 결정 차원에서 다시 보자.

Walmart 모바일 트래픽은 전체의 70%를 차지했다. IO 바운드 작업에 더해 서버사이드 렌더링까지 도입한 시점이었다. 모바일 페이지를 서버에서 미리 렌더링해 보내는 일을 Node 서버들이 받았다. 결과는 Hacker News와 Changelog 인터뷰에 정리돼 있다. **55% 트래픽이 Node 서버로 갈 때 CPU가 약 1%였다. Black Friday에 15억 달러어치를 처리했다.** 이 두 숫자만 보면 Node가 모든 일을 다 해낸 것처럼 보인다.

여기서 1장에서 본 convoy effect 일화를 다시 펼쳐 보자. 페이지 한 번 렌더링에 10~80ms가 걸린다. 그 사이 들어오는 다른 요청들은 이벤트 루프가 비기를 기다리며 줄을 선다. 트래픽이 적을 때는 문제가 안 보이는데, 트래픽이 어느 임계를 넘으면 줄이 줄을 만들어 사고가 터진다. 그 임계를 어떻게 지났을까.

답은 두 가지 도구였다. 첫째, fruit-loops와 hula-hoop으로 사전 렌더링을 했다. 사용자 요청이 들어왔을 때 그 자리에서 SSR을 돌리는 게 아니라, 사용 가능한 슬롯에서 미리 렌더링을 돌려 두는 모델이다. 둘째, 사전 렌더링한 결과를 CDN 캐시에 적극적으로 올렸다. 모바일 페이지의 상당 부분은 사용자별 데이터가 아니라 카탈로그·랭킹·프로모션 같은 공유 데이터였다. 그 부분을 CDN이 받으면 Node 서버는 정말로 사용자별 동적 부분만 처리하면 된다.

이 사례에서 결정 차원의 신호를 두 가지 읽어 두자. 하나, **SSR을 Node에 모두 맡기면 안 된다.** SSR은 본질적으로 CPU 작업이다. 단일 스레드 위에 CPU 작업을 쌓으면 convoy effect는 시간 문제다. SSR을 사전 렌더링과 CDN 캐시 두 층으로 분리하는 결정이 이 한계를 우회한다. 둘, **Black Friday 같은 트래픽 피크를 가정하지 않은 설계는 위험하다.** 평소 CPU 1%라는 수치는 평균이다. 피크 시점의 p99 응답 시간을 봐야 한다. 6장에서 본 perf_hooks의 monitorEventLoopDelay가 여기서 회수된다. p99 lag가 50ms를 넘는 순간이 convoy effect의 첫 신호다.

이 사례가 우리 결정에 주는 답은 이렇다. SSR이 비즈니스 핵심이고 트래픽 피크가 큰 서비스라면, Node BFF만으로는 부족하다. CDN 전략과 사전 렌더링 전략을 같이 짜야 한다. 그 결정을 같이 못 하겠다면, BFF 분리 자체를 미루는 편이 낫다. 절반만 한 결정은 평소엔 잘 돌다가 피크에서 무너진다.

## 사례 5 — 당근마켓 푸시 알림

이제 한국 사례로 넘어가자. 첫 번째는 당근마켓이다. 1차 자료가 가장 단단한 한국 사례다.

당근마켓의 출발점은 명확했다. 모놀리식 Rails가 푸시알림 트래픽을 견디지 못하기 시작했다. 채팅 알림, 키워드 알림, 금주의 인기글 — 사용자가 늘어날수록 푸시 트래픽이 한 곳에 몰렸다. 모놀리스를 통째로 옮기는 결정 대신, 푸시알림 한 조각만 떼어내는 결정을 했다. 당근 테크 블로그의 Hwasoo Cho 글에 그 결정 과정이 1차로 남아 있다.

새 서비스는 NestJS 마이크로서비스 한 개로 만들었다. TypeScript로 짰고, Redis를 큐로 썼다. 결과는 깔끔하다. **초당 1,500 RPS를 누락 없이 처리했다. 별도 인프라 추가 없이.** 1,500 RPS는 큰 숫자가 아니지만, "누락 없이"가 핵심이다. 푸시 알림은 누락이 사용자 경험에 직접 닿는 도메인이다. 알림이 안 오면 사용자가 모르고, 사용자가 모르면 다시 안 들어온다. 그 누락률을 0에 수렴시킨 게 결과다.

이 사례에서 결정 차원의 신호는 두 개다. 하나, **모놀리스 전체를 옮기는 결정과 한 조각만 떼는 결정은 다른 결정이다.** 당근은 Rails 모놀리스를 그대로 두고 푸시만 떼어냈다. Strangler Fig 패턴의 가장 작은 단위에 해당한다. 이 결정은 "모놀리스가 한계인 영역과 한계가 아닌 영역을 분리해서 본다"는 사고에서 나온다. 둘, **트래픽 폭주 + I/O 바운드 + 누락 무관용**의 세 조건이 겹치는 곳이 BFF 분리의 자연스러운 첫 후보다. 푸시알림이 정확히 이 조건이었다.

당근의 두 번째 사례도 짚어 두자. 같은 회사가 5개 언어를 굴리다가 Go·TypeScript 양대 스택으로 통합한 회고다. Ruby·Java·Python·Go·TypeScript를 함께 쓰던 팀이 운영 부담을 견디지 못해 정리했다. 정형적 비즈니스 로직(채팅·피드·이미지·인프라)은 Go로, 사용자 대면 서비스(광고·커머스·동네 프로모션)는 TypeScript로. 결제·금융 영역은 Java를 유지하는 폴리글랏 절충이었다.

이 두 번째 사례의 의미가 깊다. **"한 언어로 통일"이 목표가 아니다.** 도메인별로 적합한 언어가 다르다는 사실을 받아들이고, 그 안에서 운영 가능한 수(보통 2~3개)로 줄이는 게 현실적인 단순화다. JS/TS의 강점은 프론트와 백엔드가 한 언어를 공유한다는 점이라고 글에 명시돼 있다. 우리가 9장에서 다룰 "두 런타임을 같이 돌리는" 메시지의 한국 사례 버전이 여기서 미리 보인다.

## 사례 6 — 줌인터넷 모바일 줌

한국 사례 중에서 정량 데이터가 가장 풍부한 곳이 줌인터넷이다. 2020년 6월, 줌인터넷 테크팀이 회사 블로그에 "모바일 줌 SpringBoot → NodeJS 전환기"라는 글을 정성껏 남겼다.

출발점부터 한국 회사다운 솔직함이 있다. 모바일 줌은 "API Aggregation + Frontend Serving"이 주력이었다. 즉, 자체 비즈니스 로직보다 다른 백엔드 API들을 모아서 가공하고 모바일 페이지로 렌더링해 보내는 일이 중심이었다. Spring Boot가 너무 무겁고 과하다고 판단했다. 여기까지 보면 LinkedIn·Netflix와 같은 결의 결정이다.

흥미로운 건 그 다음이다. Vue SSR을 Java 환경에서 돌릴 길을 진지하게 시도했다. J2V8을 써 봤더니 앱이 멈췄다. Nashorn은 CommonJS 미지원에 10배 느렸다. Puppeteer는 Chrome 인스턴스를 별도로 띄워야 해서 인프라가 별도로 필요했다. 글에 "Java 환경에서의 SSR은 실서비스에 무리"라고 결론을 적었다. 이 한 줄이 결정의 무게를 보여 준다.

결과 수치는 같은 하드웨어에서 비교됐다. **TPS 약 40% 증가, 메모리 사용량 50% 이상 감소, 백엔드 코드 1,608라인에서 472라인으로 약 71% 축소.** SSR 도입으로 SEO 기반 PV가 늘었고, 갤럭시 노트2 같은 저사양 단말에서 페이지 렌더링 시간이 약 3초 단축됐다. TypeScript 도입과 데코레이터 표준 라이브러리로 코딩 스타일을 강제화했다고 명시돼 있다.

이 수치가 한국 사례에서 귀한 이유가 있다. 글로벌 사례들은 회사 규모가 너무 커서 우리 팀에 그대로 대입하기가 망설여진다. PayPal의 5명 vs 2명, LinkedIn의 30대 vs 3대, 이런 수치는 영감은 주지만 우리 의사결정에는 직접 안 맞는다. 줌인터넷의 1608라인 vs 472라인은 다르다. **우리 팀의 한 모듈도 그 정도 규모다.** 이 수치를 가지고 우리 팀의 어느 모듈이 같은 결정 후보가 될 수 있을지 그릴 수 있다.

여기서 줌인터넷 글이 명시한 균형 하나를 같이 옮겨오자. 글의 마지막에 "그러나 결제·금융 모놀리식 백엔드까지 같은 결론을 내는 건 위험"이라고 적혀 있다. 이 문장이 8장 전체의 균형 톤을 한 줄로 요약한다. **"프론트 가공 + API 집계" 같은 얇은 BFF 계층에는 Spring Boot가 과한 도구다. 그러나 결제·금융 모놀리식 백엔드까지 같은 결론을 내는 건 위험하다.** 같은 회사 안에서도 도메인별로 결정이 갈린다.

## 사례 7 — 직방 Spring 개발자의 1인칭 적응기

직방의 김동영 글은 2022년 직방 미디엄 채널에 올라온 1차 회고다. Spring Framework로 주로 개발하던 자바 개발자가 직방에 합류한 뒤 NestJS로 전환한 경험을, 프레임워크 사고방식의 1:1 매핑으로 정리했다. 한국에서 가장 가깝게 느껴지는 회고다.

이 글이 귀한 이유는 정량 수치가 없는 데 있다. 정성적 매핑이 그 자체로 가치다. 글에 정리된 매핑을 그대로 옮겨 보자.

- `@Controller` ↔ `@Controller` 데코레이터
- `@Bean` ↔ `@Injectable`
- IoC Container ↔ NestJS Module
- `@Interceptor` ↔ NestInterceptor
- Exception Handler ↔ Exception Filter
- Argument Resolver / Validator ↔ Pipe
- Spring Security 기능 ↔ Guard

일곱 가지 컴포넌트가 거의 1:1로 매핑된다. 글에 "구조가 매우 유사해 적응이 빨랐다"고 적혀 있다. TypeScript 코드 작성 속도가 Java보다 빨랐다는 평가도 같이 있다. 우리가 4장에서 NestJS와 Spring Boot의 정면 비교를 길게 한 이유가 여기서 회수된다. **Java 개발자가 Node로 갈 때 NestJS가 가장 자연스러운 다리다.** 그건 추상적 주장이 아니라 직방 같은 회사에서 실제로 일하는 한국 시니어 개발자의 1인칭 증언이다.

균형 한 줄도 같이 옮기자. Spring의 패키지 스캐닝 기반 자동 등록과 달리 NestJS는 모듈 선언과 `providers` 배열을 직접 적어야 한다. **명시성은 높지만 보일러플레이트는 더 많다.** 4장에서 우리가 다뤘던 그 트레이드오프가 직방 회고에서 같은 모양으로 나온다. 같은 호불호 양면이 결정 차원에서 어떻게 작동하는지 보자. NestJS는 보일러플레이트를 받아들이는 대신 모듈 그래프의 명시성을 얻는다. 큰 팀이 큰 도메인을 운영할 때 그 명시성이 갈수록 귀해진다. Spring 출신이 NestJS를 처음 만나면 "왜 이렇게 손이 많이 가나"로 시작했다가, 6개월 뒤에는 "그래서 모듈을 보면 의존 그래프가 한눈에 보인다"로 바뀐다. 직방 회고가 그 6개월의 압축이다.

## 사례 8 — 인프랩의 Express 졸업

인프랩 향로(Hyangro) CTO가 2024년 4월에 회사 기술 블로그에 올린 글이 있다. "개발 파트 소개 - 1. 백엔드 파트". 이 글의 진짜 가치는 마이그레이션 동기 6가지를 명시적으로 적어 둔 데 있다. **"Express를 졸업할 시점"의 신호 체크리스트가 1차 자료로 정리된 보기 드문 글이다.**

배경부터 보자. 인프랩은 초기 빠른 성장을 위해 JavaScript + Express + 함수형 라이브러리(FxJS, FxSQL) 조합으로 시작했다. 8명 백엔드 팀, 연 50~300% 성장하는 인프런 플랫폼. 강의·동영상·채용·결제·커뮤니티가 한 팀에서 책임지는 단계였다. 이 단계에서 한계가 왔다. 글에 명시된 동기 6가지를 정확히 옮긴다.

1. 신규 입사자 학습 곡선
2. IDE 자동완성·리팩터링 부족
3. 타입 안정성 부재
4. 함수형 JS 인력 풀 협소
5. 서드파티 호환성
6. 커뮤니티 자료 부족

이 여섯 가지를 차례로 읽어 보면 흥미로운 점이 있다. 모두가 "실행 시점의 성능"이 아니라 **"개발 단계의 마찰"**이다. 인프랩이 졸업한 이유는 Express가 느려서가 아니라, 신규 입사자가 합류하기 어려워서, IDE가 도움을 못 주어서, 타입이 없어서다. 이게 우리 팀의 결정 차원에서 매우 중요하다. **"실행 시점의 성능"이 아닌 "팀 차원의 마찰"이 결정 신호다.**

신규 스택은 NestJS Monorepo + TypeORM/MikroORM + Jest/SuperTest + 도메인 주도 설계 + Jenkins/GitHub Actions. 우리가 4·5·7장에서 다룬 도구들이 한 회사의 결정에 그대로 모여 있다. 함수형 JS에서 OOP NestJS로의 전환은 패러다임 자체를 바꾼 드문 사례다. JS → TS, Express → NestJS의 전환 동기 6가지가 그대로 우리 책의 "Express 졸업 시점" 신호 체크리스트로 작동한다.

이 사례에서 결정 차원의 신호 한 줄을 못박자. **"처음엔 익숙한 방식으로 빠르게, 규모가 커지면 구조에 투자한다."** 인프랩이 처음부터 NestJS로 시작했다면 8명 팀이 그렇게 빨리 만들지 못했을 가능성이 높다. 그러나 같은 8명이 50~300% 성장 단계에서 같은 방식을 유지했다면 깨졌을 것이다. **결정의 시점이 다르다.** 우리 팀이 어느 단계에 있는지를 먼저 확인해야, 어느 도구로 갈지를 결정할 수 있다.

## 사례 9 — 네이버파이낸셜 페이와 토스, 그리고 LINE

한국 사례 마지막 묶음은 짧게 가자. 네이버파이낸셜의 페이 플랫폼은 2022년에 김태우의 글로 한 줄을 남겼다. Java Spring 모놀리식에서 Node.js MSA + Kubernetes로 옮긴 사례다. 핵심 throughput·latency 수치는 비공개라 정성적으로만 인용 가능한데, 한 가지 수치가 강력하다. **유휴 인스턴스 메모리 비교: Java Spring ≈ 400MB vs Node.js ≈ 25MB.**

이 16배 차이를 어떻게 봐야 할까. JVM이 비효율적이라는 뜻이 아니다. JVM은 시작 시 미리 잡아 두는 메모리가 크고, 그 위에서 점진적 최적화가 일어나는 모델이다. 평소 안정 운영에서는 Java가 훨씬 부드럽게 돈다. 그러나 컨테이너 오토스케일 환경, 즉 인스턴스가 자주 떴다 졌다 하는 환경에서는 다르다. **"유휴 인스턴스 한 대당 메모리"는 K8s 비용 모델에 직접 영향을 준다.** Pod이 100개 떠 있을 때 한 Pod당 400MB와 25MB는 4GB와 2.5GB의 차이가 아니다. 40GB와 2.5GB의 차이다.

네이버파이낸셜 글은 또 다른 사실 하나를 검증했다. **"Node.js 서버로도 대규모 CPU 연산이 가능함을 검증"했다고 글에 명시돼 있다.** 정량 수치는 비공개지만 이 한 줄이 의미가 깊다. "Node는 I/O 바운드만 잘한다"는 통념을 1차 자료에서 반박한 한국 사례다. 단, 글의 마지막 균형이 중요하다. "안정성·신뢰성" 도메인이라는 특성을 고려해 Spring을 같이 운영하는 폴리글랏 모델이라고 명시돼 있다. 금융권에서도 Node를 쓰지만, Spring을 빼는 게 아니라 옆자리에 두는 결정이다.

토스는 마이그레이션 동기 글이 1차로는 발견되지 않았지만, NestJS Custom Decorator 글(송현지, 2022)이 토스가 NestJS를 본격 운영함을 확인해 준다. Spring의 AOP에 익숙한 개발자에게 NestJS의 데코레이터 + DiscoveryModule 조합이 동등한 표현력을 제공한다는 사실을, 토스 1차 자료가 보여 준 셈이다. 글에 정리된 패턴이 metadata 태깅 → DiscoveryModule lookup → 프로토타입 등록의 3단계인데, 이 모양은 Spring의 `BeanPostProcessor` 또는 `BeanFactoryPostProcessor`가 하는 일과 거의 정확히 같은 역할이다. AOP의 `@Aspect`로 Spring 출신이 익숙해진 그 표현력이, NestJS에서는 Custom Decorator + DiscoveryModule로 같은 무게를 낸다.

LINE Engineering도 LandPress Content 같은 사내 헤드리스 CMS와 SSR용 WAS를 Node.js + Express로 운영한다. 마이그레이션 동기·수치는 비공개지만 "한국 빅테크의 Node 도입 양상"을 확인하는 1차 신호다.

이 한국 사례 묶음의 결론을 한 줄로 정리하자. **한국 빅테크 중 Spring/Java가 본진인 회사도 많지만(카카오·우아한형제들), Node를 옆자리에 들인 회사도 같이 늘고 있다(당근·줌·직방·인프랩·네이버파이낸셜·토스·라인).** 한쪽으로 단정할 수 있는 시점이 아니다. 두 진영이 같이 도는 시점이다.

여기서 자료의 한계 하나를 정직하게 짚어 두자. 한국 사례 1차 회고는 글로벌 사례에 비해 정량 수치가 빈약하다. 줌인터넷이 가장 풍부한 수치를 남겼고, 당근의 1500 RPS와 네이버파이낸셜의 메모리 16배 차이가 그 다음이다. 직방·인프랩·토스·LINE은 정성 회고 또는 운영 패턴 자료에 가깝다. 카카오·우아한형제들·쿠팡은 Spring이 본진이라 Node 마이그레이션 회고 글이 1차로는 발견되지 않는다. 이 사실을 받아들이고 자료를 본다는 게 중요하다. **"한국 빅테크 모두가 Node로 옮긴다"는 단정은 1차 자료에 근거가 없다.** "한국에서도 일부 회사가 일부 도메인에서 Node를 운영한다"가 1차 자료가 받쳐 주는 정직한 한 줄이다. 우리 결정도 그 한 줄 위에서 시작해야 한다.

## 사례 10 — Uber의 후퇴

이제 균형추를 단단히 박을 사례가 남았다. Uber다.

Uber는 LAMP에서 벗어나 실시간 처리에 Node.js를 도입한 초기 회사 중 하나였다. RTAPI(Real-Time API) 게이트웨이를 Node로 구현했다. 모바일 클라이언트가 자주 호출하는 API를 비동기로 받는 자리에 Node가 잘 맞았다. 이 결정만 보면 LinkedIn 사례와 같은 결이다.

그런데 2018년에 Uber Engineering이 흥미로운 글을 올렸다. RTAPI 회고였다. 새로 들어오는 엔지니어들이 사내 Node.js 라이브러리와 운영 패턴을 익히는 데 너무 오래 걸린다는 사실을 인정했다. Go·Java로 옮긴 팀이 늘었고, 결국 신규 권장 스택에서 "Node.js + HTTP/JSON 프레임워크"가 빠졌다. 글에 직접 적힌 결론이다. **"Node.js + HTTP/JSON 프레임워크"는 신규 권장 스택에서 제외됐다.**

이 한 줄을 어떻게 읽어야 할까. "Node가 나쁘다"가 아니다. **"기술 선택의 정답은 시점·조직·도메인에 따라 다르다"**가 정답이다. Uber는 초기에 빠르게 만들어야 했고 Node가 그 일을 해냈다. 그러나 회사가 커지고 신규 엔지니어 유입이 빨라지면서, 사내 라이브러리 부채와 온보딩 비용이 결정의 무게중심을 옮겼다. 같은 회사가 같은 도구에 대해 7년 사이에 다른 결정을 내릴 수 있다.

이 사례가 결정 차원에서 우리에게 주는 신호 세 개를 못박자. 하나, **사내 라이브러리 부채가 누적되면 어떤 도구든 무거워진다.** Express의 비강제성이 PayPal에서 Kraken.js를 만들게 했듯, Node 운영에서 사내 패치가 쌓이면 외부 라이브러리 업데이트가 어려워진다. Node 생태계는 빠르게 변하기 때문에 이 부채의 누적 속도가 Java보다 빠를 수 있다. 둘, **신규 엔지니어 온보딩 시간은 결정 신호다.** 새 사람이 합류해서 production에 코드를 올리기까지 걸리는 시간이 일정 임계를 넘으면, 그 자체가 도구를 다시 봐야 한다는 신호다. 셋, **마이그레이션은 단방향이 아니다.** Node로 갔다가 일부를 되돌릴 수도 있다. 되돌리는 결정이 패배가 아니다. 데이터에 기반한 합리적 재조정이다.

이 사례를 사례 묶음의 마지막에 둔 이유가 있다. 앞에서 본 9개 사례는 모두 성공 사례 또는 진행 중 사례다. 한 책에 그것만 모아 두면 자연스럽게 한쪽으로 기운다. **Uber 사례 한 건이 그 기울기를 바로잡는 균형추 역할을 한다.** 우리가 다음 절에서 후퇴 신호를 산문으로 풀고 체크리스트로 정리하는 까닭도 여기에 있다. 후퇴 신호를 측정하지 못하는 마이그레이션은 Uber의 7년을 답습할 위험이 있다.

한 가지를 더 짚어 두자. Uber가 "Node를 다 빼냈다"가 아니다. **신규 권장 스택에서 빠졌을 뿐이다.** 기존 서비스들은 여전히 Node로 돈다. 다만 새 서비스를 만들 때 Node를 첫 후보로 추천하지 않는다는 결정이다. 이 결의 차이가 중요하다. 마이그레이션 결정도, 후퇴 결정도, "전부 또는 전무"가 아니다. **한 도메인은 Node로 두고, 다른 도메인은 Java/Go로 가고, 또 다른 도메인은 결정을 미룬다. 이 폴리글랏 모양이 큰 회사의 정상 상태다.** 당근이 Go·TypeScript·Java 셋을 같이 굴리는 모양과, Uber가 Node·Go·Java 셋을 같이 굴리는 모양이 같은 결의 결정이다.

## Strangler Fig 패턴 — Martin Fowler의 정의에서 출발하자

사례 열 건을 다 봤다. 이제 그 사례들이 공유하는 패턴을 이름 붙여 보자. Martin Fowler가 2004년에 글 한 편으로 정리한 그 이름이 있다. **Strangler Fig Application.**

이름은 보르네오와 호주 우림에 자라는 식물에서 왔다. 무화과 씨앗이 큰 나무 꼭대기에 떨어져 자라는데, 자라면서 뿌리가 기존 나무를 감싸 내려와 흙에 닿는다. 시간이 지나면 무화과가 큰 나무를 천천히 둘러싸고, 결국 큰 나무는 죽고 무화과만 남는다. 이 모양을 Fowler가 마이그레이션 패턴 이름으로 빌려 왔다. **새 시스템이 기존 모놀리스를 감싸 내려와 천천히 대체하되, 큰 사고 없이.**

핵심 아이디어는 단순하다. 기존 모놀리스 앞에 라우터/프락시(API Gateway)를 둔다. 기능을 하나씩 새 서비스로 빼낸다. 빠진 기능에 대한 트래픽만 새 서비스로, 나머지는 모놀리스로 라우팅한다. 시간이 지나면서 라우팅 규칙이 점점 새 서비스 쪽으로 기운다. 마지막에 모놀리스가 비어 있게 되면, 그제서야 모놀리스를 끈다. 한 번에 갈아엎는 빅뱅이 아니라, 여러 작은 결정의 누적이다.

IBM Developer와 Microservices.io, AWS Prescriptive Guidance가 이 패턴을 단계별로 정리해 두었다. 우리는 다섯 단계로 풀어 보자.

### 1단계 — API Gateway 분리

```
[클라이언트]
    |
    v
[NGINX / Envoy / AWS API Gateway]   <-- 새로 들어옴
    |
    v
[Spring 모놀리스]   (그대로)
```

가장 먼저 하는 일이 API Gateway를 끼워 넣는 것이다. 이 단계에서는 아무 기능도 옮기지 않는다. 라우팅만 담당하는 게이트웨이가 모놀리스 앞에 들어와, 모든 요청이 일단 게이트웨이를 거친다. 모놀리스 입장에서는 호출 경로 한 단계가 추가됐을 뿐이다.

이 단계에서 Spring 모놀리스에 들어가는 변경은 거의 없다. 게이트웨이가 보낸 X-Forwarded-For 같은 헤더를 처리하는 필터 정도가 추가될 수 있다. 그게 끝이다. 이 단계는 그래서 가장 안전한 단계다. **변경이 거의 없는데 미래 결정의 자유도가 크게 늘어난다.**

NGINX, Envoy, AWS API Gateway 중 어느 걸 고르느냐는 운영 환경에 달렸다. K8s 환경에서는 Envoy 또는 그 위에 얹은 Istio가 자연스럽고, AWS Lambda 중심이면 API Gateway가 자연스럽고, 단일 호스트면 NGINX가 가볍다. 7장에서 본 K8s 운영 매뉴얼에서 Ingress Controller가 이 자리를 이미 차지하고 있을 가능성이 높다.

### 2단계 — 공유 DB 단계 (새 Node 서비스가 모놀리스 DB를 직접 읽기)

```
[클라이언트]
    |
    v
[API Gateway]
    /        \
   v          v
[Spring]   [Node 서비스 (신규)]
   |          |
   +-----+----+
         |
         v
    [공유 DB]   <-- 둘 다 같은 DB
```

새 Node 서비스를 만든다. 게이트웨이가 일부 라우트를 새 서비스로 보낸다. 이 단계에서 가장 빠르게 가는 길은, 새 Node 서비스가 모놀리스의 DB를 그대로 직접 읽는 것이다. 같은 PostgreSQL, 같은 테이블, 같은 스키마.

이 단계의 장점은 빠르다는 것이다. 데이터 이전 없이 곧장 새 서비스가 돌기 시작한다. 단점도 분명하다. **결합이 강하다.** 모놀리스의 DB 스키마가 새 서비스에도 묶이기 때문에, 모놀리스에서 컬럼 하나를 바꾸면 새 서비스도 깨진다. Spring 모놀리스에 들어가는 변경은 여기서 시작된다. 스키마 변경 시 새 서비스 팀과 합의해야 한다는 운영 룰이 추가되고, 트랜잭션 경계가 두 서비스에 걸친다는 사실을 인정해야 한다.

이 단계는 임시 단계다. 5장에서 우리가 본 Prisma의 명시적 `$transaction`이 여기서 회수된다. Spring의 `@Transactional`이 모놀리스 한쪽에서만 작동하기 때문에, 새 Node 서비스에서 일어나는 쓰기는 별도 트랜잭션이 된다. 두 트랜잭션 사이의 일관성을 어떻게 맞출 것인가가 다음 단계로 가는 추진력이다.

### 3단계 — DB 분리 (CDC와 이벤트 스트림)

```
[클라이언트]
    |
    v
[API Gateway]
    /        \
   v          v
[Spring]   [Node 서비스]
   |          |
   v          v
[모놀리스 DB] [신규 DB]
   |          ^
   +-Debezium-+
   +--Kafka---+
```

DB를 분리한다. 새 Node 서비스가 자기 DB를 갖는다. 그러나 모놀리스 DB와 새 DB는 별도 시스템이라 데이터 일관성을 어떻게 맞출지가 문제다. 답은 CDC(Change Data Capture)와 이벤트 스트림이다. Debezium 같은 도구가 모놀리스 DB의 binlog를 읽어 Kafka 토픽으로 흘려보내고, 새 Node 서비스가 그 토픽을 구독해 자기 DB를 갱신한다.

```typescript
// 새 Node 서비스의 Kafka 컨슈머 한 조각
import { Consumer, Kafka } from 'kafkajs';

const kafka = new Kafka({
  clientId: 'order-bff',
  brokers: process.env.KAFKA_BROKERS!.split(','),
});

const consumer: Consumer = kafka.consumer({ groupId: 'order-bff-cdc' });

await consumer.connect();
await consumer.subscribe({ topic: 'monolith.public.orders', fromBeginning: false });

await consumer.run({
  eachMessage: async ({ message }) => {
    const evt = JSON.parse(message.value!.toString());
    // Debezium의 일반 envelope: { before, after, op: 'c' | 'u' | 'd' }
    if (evt.op === 'c' || evt.op === 'u') {
      await orderProjection.upsert(toOrderProjection(evt.after));
    } else if (evt.op === 'd') {
      await orderProjection.remove(evt.before.id);
    }
  },
});
```

이 코드가 보여 주는 건, **모놀리스 DB의 변화를 실시간으로 새 시스템에 복제하는 모델**이다. 새 서비스는 더 이상 모놀리스 DB를 직접 읽지 않는다. 이벤트로 전달된 데이터의 자기 카피를 자기 DB에서 읽는다. 이렇게 하면 모놀리스가 죽거나 스키마가 바뀌어도 새 서비스가 안전하다.

Spring 모놀리스에 들어가는 변경은 작다. Debezium은 binlog에 기반하기 때문에 모놀리스 코드는 그대로다. 다만 binlog가 켜져 있어야 하고, replica slot이 하나 더 늘어 DB 부하가 약간 증가한다. 이게 이 단계의 운영 비용이다.

이 단계가 가장 어렵다. 데이터 일관성 모델을 결정해야 한다. **eventual consistency를 받아들일 것인가, 강한 일관성을 다른 방식으로 보장할 것인가.** Spring의 `@Transactional`이 한 줄로 풀던 일을 Saga 패턴이나 outbox 패턴으로 다시 풀어야 한다. 5장에서 본 Prisma의 명시성이 여기서 본격적으로 살아난다. 트랜잭션 경계가 명시적이라야, 분산된 자리에서도 그 경계를 추적할 수 있다.

### 4단계 — 트래픽 점진 이전 (feature flag, canary, shadow traffic)

```
[클라이언트]
    |
    v
[API Gateway + Feature Flag]
    /        \   (예: 5% 트래픽이 Node로)
   v          v
[Spring]   [Node 서비스]
```

새 Node 서비스가 데이터 면에서 안정됐으면, 이제 트래픽을 점진적으로 옮긴다. 한 번에 100% 전환하면 사고가 났을 때 롤백할 자리가 없다. 그래서 세 가지 도구를 같이 쓴다.

**Feature flag.** 게이트웨이 또는 새 서비스 안에서 사용자/요청별로 분기한다. 1%로 시작해 5%, 10%, 50%, 100%로 점진 이전. LaunchDarkly나 Unleash 같은 도구를 쓸 수도 있고, Redis 키 하나로 시작할 수도 있다.

**Canary.** 새 버전을 일부 인스턴스에만 배포해 동일 트래픽을 받게 한다. 7장에서 본 K8s의 Argo Rollouts나 Flagger가 이 일을 한다. 새 인스턴스의 에러율·응답 시간이 기존 인스턴스 대비 임계 안에 있을 때만 점진 이전이 진행된다.

**Shadow traffic (mirroring).** 사용자 응답에는 영향을 주지 않으면서 트래픽을 새 서비스에도 복제해 보낸다. 새 서비스의 응답은 버리고, 메트릭만 모은다. 가장 안전한 검증 모드다. Envoy의 mirror_policy 또는 NGINX의 mirror 디렉티브로 설정할 수 있다.

```typescript
// NestJS BFF에서 feature flag 기반 분기 (NestJS Guard의 가벼운 형태)
import { Injectable, NestMiddleware } from '@nestjs/common';

@Injectable()
export class CanaryRouter implements NestMiddleware {
  constructor(private readonly flags: FlagService) {}

  async use(req: Request, res: Response, next: () => void) {
    const userId = req.headers['x-user-id'] as string | undefined;
    const inCanary = await this.flags.isUserInCanary('order-bff-v2', userId);
    if (inCanary) {
      // 새 Node 서비스로
      return next();
    }
    // 모놀리스로 프록시
    return this.proxyToMonolith(req, res);
  }

  private async proxyToMonolith(req: Request, res: Response) {
    const upstream = await fetch(`${process.env.MONOLITH_URL}${req.url}`, {
      method: req.method,
      headers: req.headers as any,
      body: req.body ? JSON.stringify(req.body) : undefined,
    });
    res.status(upstream.status);
    upstream.body?.pipe(res);
  }
}
```

이 단계에서 Spring 모놀리스에 들어가는 변경은 여전히 작다. 게이트웨이가 새 서비스로 트래픽을 보낼 뿐, 모놀리스는 자기 일을 한다. 다만 새 서비스가 처리한 요청에 대해 **모놀리스가 같은 사용자의 다른 요청을 받았을 때 일관된 응답을 줘야 한다.** 캐시 무효화, 세션 동기화 같은 운영 룰이 추가된다.

### 5단계 — Anti-Corruption Layer

```
[모놀리스 도메인 모델]
       ↓ Spring REST API
[Anti-Corruption Layer (NestJS BFF 안)]
       ↓ 새 도메인 모델로 변환
[새 시스템의 도메인 모델]
```

마지막 단계는 코드 안의 패턴이다. Anti-Corruption Layer라는 이름은 Eric Evans의 도메인 주도 설계에서 왔다. **레거시 모놀리스의 도메인 개념이 새 시스템에 그대로 새어 들어오지 않게 막는 번역 층이다.**

왜 이게 필요한가. 모놀리스의 도메인 모델은 보통 오래됐고, 그 사이에 누적된 사내 개념이 들어 있다. 그 개념을 새 시스템이 그대로 이어받으면, 새 시스템도 같은 부채를 짊어진다. 그래서 새 시스템 입구에서 한 번 변환을 거친다. 모놀리스의 응답을 그대로 쓰지 않고, 새 도메인 모델로 매핑한다.

```typescript
// 모놀리스 응답 (Spring 측 DTO)
interface MonolithOrderDto {
  ord_id: number;
  ord_state: 'P' | 'F' | 'C' | 'X'; // Pending, Fulfilled, Cancelled, eXpired
  cust_no: string;
  ord_amt: number;
  ord_dt: string; // 'YYYYMMDD'
  ext_attr_json: string; // 사내 누적 부채
}

// 새 시스템의 도메인 모델
type OrderStatus = 'pending' | 'fulfilled' | 'cancelled' | 'expired';

interface Order {
  id: string;
  status: OrderStatus;
  customerId: string;
  amount: number;
  orderedAt: Date;
}

// Anti-Corruption Layer
function toOrder(dto: MonolithOrderDto): Order {
  const statusMap: Record<MonolithOrderDto['ord_state'], OrderStatus> = {
    P: 'pending',
    F: 'fulfilled',
    C: 'cancelled',
    X: 'expired',
  };

  return {
    id: String(dto.ord_id),
    status: statusMap[dto.ord_state],
    customerId: dto.cust_no,
    amount: dto.ord_amt,
    orderedAt: parseYyyyMmDd(dto.ord_dt),
    // ext_attr_json은 의도적으로 받지 않는다. 새 시스템에 부채를 들이지 않는다.
  };
}
```

이 코드의 마지막 줄이 핵심이다. `ext_attr_json`을 받지 않는다. 모놀리스에 누적된 사내 부채가 새 시스템에 새어 들어오는 걸 의도적으로 막는다. 새 시스템이 그 데이터를 정말 필요로 한다면, 어떤 도메인 개념인지 다시 분석해서 새 모델로 정의한다. 받지 않는 게 받는 것보다 어렵지만, 이 결정이 새 시스템의 5년 부채를 결정한다.

Spring 모놀리스에 들어가는 변경은 없다. 모놀리스는 자기 모양 그대로 응답한다. 변환은 새 시스템 안에서만 일어난다.

## NestJS BFF Skeleton — 최소 코드로 시작하자

이제 다섯 단계를 머리에 넣었다. Spring 모놀리스 앞단에 NestJS BFF를 두는 최소 코드를 직접 적어 보자. 이 코드 한 덩어리가 위 다섯 단계의 1·2·5단계를 동시에 보여 준다.

```typescript
// src/orders/orders.module.ts
import { Module, HttpModule } from '@nestjs/common';
import { OrdersController } from './orders.controller';
import { OrdersService } from './orders.service';
import { MonolithOrderClient } from './monolith-order.client';

@Module({
  imports: [HttpModule],
  controllers: [OrdersController],
  providers: [OrdersService, MonolithOrderClient],
})
export class OrdersModule {}
```

```typescript
// src/orders/orders.controller.ts
import { Controller, Get, Param } from '@nestjs/common';
import { OrdersService } from './orders.service';
import { Order } from './order.types';

@Controller('orders')
export class OrdersController {
  constructor(private readonly orders: OrdersService) {}

  @Get(':id')
  async getOrder(@Param('id') id: string): Promise<Order> {
    return this.orders.findById(id);
  }
}
```

```typescript
// src/orders/orders.service.ts
import { Injectable, NotFoundException } from '@nestjs/common';
import { MonolithOrderClient } from './monolith-order.client';
import { toOrder } from './order.acl';
import { Order } from './order.types';

@Injectable()
export class OrdersService {
  constructor(private readonly client: MonolithOrderClient) {}

  async findById(id: string): Promise<Order> {
    const dto = await this.client.fetchOrder(id);
    if (!dto) throw new NotFoundException(`Order ${id} not found`);
    return toOrder(dto); // anti-corruption layer
  }
}
```

```typescript
// src/orders/monolith-order.client.ts
import { HttpService, Injectable } from '@nestjs/common';
import { firstValueFrom, catchError } from 'rxjs';
import { MonolithOrderDto } from './order.types';

@Injectable()
export class MonolithOrderClient {
  constructor(private readonly http: HttpService) {}

  async fetchOrder(id: string): Promise<MonolithOrderDto | null> {
    const url = `${process.env.MONOLITH_URL}/api/v1/orders/${id}`;
    const res$ = this.http.get<MonolithOrderDto>(url, {
      timeout: 2000, // Spring 모놀리스가 느릴 때 대비
      headers: { 'X-Source': 'order-bff' },
    });

    try {
      const { data } = await firstValueFrom(res$);
      return data;
    } catch (err: any) {
      if (err.response?.status === 404) return null;
      throw err; // 4xx/5xx는 그대로 전파, GlobalExceptionFilter에서 받음
    }
  }
}
```

이 네 파일이 보여 주는 그림을 다시 정리하자. **Controller는 새 시스템의 도메인 언어로 응답한다.** Service가 그 일을 받아 모놀리스 클라이언트를 호출하고, 응답을 anti-corruption layer로 한 번 변환한다. Client는 모놀리스의 Spring REST API와 대화하는 자리다. 모놀리스는 자기 일을 그대로 한다. 새 시스템 안에서만 새 모델로의 번역이 일어난다.

이 코드 위에 4·5장에서 본 도구들이 차곡차곡 얹힌다. ValidationPipe로 요청을 검증하고, GlobalExceptionFilter로 예외 모양을 통일하고, Pino로 구조화 로깅을 하고, OpenTelemetry로 분산 트레이스를 찍는다. 4장 NestJS 테스트 절에서 본 `Test.createTestingModule`로 OrdersService의 단위 테스트를 짜고, Supertest로 OrdersController의 통합 테스트를 짠다. 이 모든 게 한 NestJS 프로젝트 안에서 자연스럽게 모인다. Spring 출신이 4장에서 NestJS를 익혀 둔 이유가 여기서 회수된다.

여기에 graceful shutdown 코드를 첫날부터 넣자. 7장에서 다룬 운영 매뉴얼이 마이그레이션 코드에서도 똑같이 작동한다.

```typescript
// src/main.ts
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.enableShutdownHooks(); // SIGTERM 받으면 onModuleDestroy/onApplicationShutdown 호출
  await app.listen(3000);
}
bootstrap();
```

`enableShutdownHooks` 한 줄이 K8s 환경에서 SIGTERM 처리의 시작이다. 이거 없이 BFF를 띄우면 deploy 때마다 in-flight 요청을 잘라먹는다. 이런 함정을 첫날부터 막아 두자.

## 결정 가이드 — 어디부터 자를 것인가

사례 열 건과 Strangler Fig 5단계를 봤다. 이제 우리 팀의 결정으로 돌아오자. **"어디부터 자를 것인가"**의 답을 산문으로 풀어 보자.

가장 단단한 답은 **"BFF부터 시작하라"**다. 이건 PayPal·Netflix·줌인터넷이 같은 결정을 한 자리다. 사용자에게 가장 가까운 경계, 외부 서비스 호출이 가장 많은 경계, 응답 시간이 사용자 경험에 직접 닿는 경계 — 이 세 조건이 겹치는 자리가 BFF다. PayPal의 계정 개요 페이지, Netflix의 UI 서버, 줌인터넷의 모바일 줌, 모두 같은 모양이다. 이 자리는 본질적으로 I/O 바운드고, 외부 API 합산이 핵심이고, 단일 스레드 비동기 모델이 잘 맞는다.

핵심 도메인 서비스는 마지막에 두자. 이건 Netflix가 Java 마이크로서비스를 그대로 둔 결정의 핵심이다. **결제·주문·사용자 인증 같은 핵심 도메인은 옮기지 않거나, 가장 마지막에 옮긴다.** 이 도메인들은 비즈니스 룰이 누적된 곳이고, Spring의 트랜잭션 모델과 ecosystem 두께가 가장 잘 빛나는 곳이다. BFF가 안정된 다음, 운영 데이터가 쌓이고 팀의 Node 숙련도가 일정 수준에 도달한 다음, 그제서야 핵심 도메인 분리 결정을 다시 본다. 그 결정이 5년 뒤에 안 와도 된다. 옆자리에 두는 결정으로 충분할 수 있다.

CPU 바운드 작업은 따로 처리하자. 이미지 변환, PDF 생성, 해시·암호화 같은 작업은 단일 스레드 Node에 그대로 얹으면 convoy effect의 원인이 된다. 1장에서 우리가 본 그 함정이다. 답은 두 갈래다. **하나는 worker_threads로 같은 프로세스 안에서 별도 스레드로 분리하는 것이고, 다른 하나는 아예 다른 언어 마이크로서비스로 분리하는 것이다.** 당근마켓이 5개 언어를 Go·TypeScript 양대 스택으로 정리하면서 비즈니스 로직은 Go로 보낸 결정이 두 번째 갈래의 사례다. Go는 멀티코어 활용이 자연스럽고, 결제·금융 영역에서는 Java를 그대로 쓰는 폴리글랏 절충이 합리적이다.

테스트 자산 이전은 신중하게 보자. **Java JUnit 자산을 일괄로 Jest로 변환하려는 충동을 누르자.** 그 충동은 항상 비싸게 끝난다. 신규 코드는 새 테스트로 덮고, 기존 코드는 기존 테스트로 보호한다. 4장의 NestJS 테스트 패턴이 신규 BFF 코드에서 작동하고, 모놀리스 안의 JUnit 자산은 그대로 둔다. 두 자산이 한 시스템에 공존하는 게 정상이다. Netflix가 BFF에 contract test를 두른 그 모양이 우리 팀에서도 같은 모양으로 작동한다.

데이터 일관성 결정도 미리 못박자. Strangler Fig 3단계에서 Saga·outbox 패턴을 끌어왔는데, 이 결정은 **사실상 비즈니스 결정이지 기술 결정이 아니다.** "결제 직후의 재고 차감이 즉시 일관돼야 하는가, 30초 안에 일관되면 되는가"의 답이 아키텍처를 결정한다. 답을 모르면 모놀리스에 두는 게 안전하다.

여기서 비교 관점 두 개를 명시적으로 짚어 두자. 첫째, **AOP 로직이 어떻게 옮겨가는가.** Spring의 `@Transactional`, `@Cacheable`, `@Async`, `@Retryable` 같은 어노테이션이 모놀리스 안에서 횡단 관심사로 박혀 있다면, Node 쪽에서는 그 일들이 NestJS 인터셉터·가드·파이프와 anti-corruption layer로 흩어진다. 같은 일을 같은 자리에서 받지 않는다는 게 핵심이다. 트랜잭션 경계는 Prisma의 `$transaction`으로, 캐시는 명시적 Redis 호출로, 비동기는 `await`과 BullMQ 큐로, 재시도는 라이브러리(p-retry, axios-retry) 또는 명시적 try/catch로 분해된다. **"한 어노테이션이 받던 일이 네 자리로 흩어진다"**가 평균적인 모양이다. 이 흩어짐을 어색하게 보지 말자. 4장에서 본 명시성의 가치가 여기서 회수된다.

둘째, **Spring 트랜잭션이 분산 자리에서 어떻게 분해되는가.** 모놀리스 안에서는 한 메서드의 `@Transactional`이 DB 트랜잭션 한 개로 모든 일을 묶었다. BFF가 분리되고 DB가 분리되면, 그 한 개가 두세 개로 쪼개진다. 답은 두 갈래다. 하나는 **Saga 패턴**이다. 일을 단계별로 분해하고, 각 단계가 실패했을 때 보상 트랜잭션을 발행한다. 다른 하나는 **outbox 패턴**이다. 도메인 이벤트를 자기 DB의 outbox 테이블에 같이 기록하고, 별도 publisher가 그 이벤트를 Kafka로 흘려보낸다. 두 패턴 모두 "강한 일관성에서 eventual consistency로의 의도적 후퇴"다. 이 후퇴를 받아들일 수 없는 도메인이 있다. 그 도메인이 핵심 도메인이고, 핵심 도메인을 마지막에 두라는 권고가 여기서 결의 무게를 받는다.

여기까지를 표 한 장으로 정리하면 이렇다. 표는 본문 위에 올라가는 게 아니라, 본문을 다 읽은 다음에 펴 보는 정리 자료다.

| 도메인 영역 | 권장 결정 | 사례 근거 |
|---|---|---|
| 모바일/웹 BFF, SSR, API Aggregation | Node BFF로 분리 (1순위) | PayPal, Netflix, 줌인터넷, LinkedIn |
| 푸시·알림·스트림 같은 IO 바운드 + 누락 무관용 | Node 마이크로서비스로 분리 | 당근마켓 |
| 결제·주문·인증 같은 핵심 도메인 | 모놀리스 유지 또는 가장 마지막 | Netflix(Java 유지), 네이버파이낸셜(폴리글랏) |
| 이미지·PDF·암호화 같은 CPU 바운드 | worker_threads 또는 다른 언어 | 당근(Go), 일반 권고 |
| 테스트 자산 | 신규 코드는 새 테스트, 기존은 보호 | PayPal·Netflix(contract test) |

이 표 위에 한 줄 더 적어 두자. **"결정은 한 번에 다 안 한다."** 1순위만 결정하고 나머지는 데이터를 모은 다음에 다시 본다.

## 후퇴 신호 — 멈춰야 할 때를 어떻게 알 것인가

이제 책의 균형추를 단단히 박자. Uber 사례에서 우리는 마이그레이션이 단방향이 아님을 봤다. **그렇다면 어떤 신호가 보이면 멈춰야 할까.**

여섯 가지 신호를 산문으로 풀어 보자.

첫째, **신규 엔지니어 온보딩 시간**이다. 새로 들어온 엔지니어가 production에 첫 코드를 올리기까지 걸리는 시간을 측정하자. Spring 시절보다 Node로 넘어온 뒤 그 시간이 길어졌다면, 사내 라이브러리 부채가 쌓였거나 운영 패턴이 표준화되지 못했다는 신호다. Uber가 정확히 이 신호로 후퇴를 결정했다. 임계는 팀마다 다르지만, "한 달 안에 첫 PR이 production에 올라가지 못한다"가 한 자리 신호다.

둘째, **이벤트 루프 lag p99**다. 6장에서 본 perf_hooks의 monitorEventLoopDelay와 loopbench, event-loop-lag이 이 자리에서 측정 도구로 작동한다. **p99 lag가 50ms를 넘는 순간이 의심 신호다.** 한 번 넘으면 사고가 아니다. 자주 넘으면 사고다. 이 신호가 지속되면 CPU 바운드 작업이 BFF에 새어 들어와 있다는 뜻이고, 그 작업을 worker_threads나 다른 언어로 분리해야 한다. 분리할 의지가 없으면 후퇴 신호로 간다.

셋째, **CPU 바운드 비중**이다. clinic doctor와 0x를 정기적으로 돌려 BFF의 CPU 사용 패턴을 본다. 핫패스가 정규식·JSON 파싱·동기 crypto·sync 파일 I/O 어디인지 확인한다. 이 항목들이 핫패스에 정착해 있고 분리 결정이 비싸다면, BFF의 적합성 자체를 다시 봐야 한다. Walmart의 SSR이 CDN과 사전 렌더링으로 우회한 이유가 여기다. 우회할 수 없는 종류의 CPU 작업이 누적되면, 그 도메인은 Node가 잘못된 자리다.

넷째, **운영 사고 빈도**다. 7장에서 본 운영 체크리스트를 전부 갖췄는데도 사고가 줄지 않는다면, 도구의 문제가 아니라 도메인의 문제다. 모니터링·알람·헬스체크·graceful shutdown이 전부 잘 깔려 있는데도 월간 사고 수가 늘면, **그 도메인이 Node에 안 맞는 자리일 가능성을 인정해야 한다.** 이건 결정의 무게가 가장 큰 신호다. 도구를 더 깔아도 안 된다면, 결정 자체를 되돌려야 할지 검토해야 한다.

다섯째, **커뮤니티/사내 라이브러리 부채**다. Node 생태계의 빠른 변화 속에서 사내에서 fork한 라이브러리, 보안 패치를 미룬 의존성, 더 이상 유지되지 않는 npm 패키지에 묶인 코드가 누적된다. 한 줄짜리 npm 패키지로 시작한 left-pad 사고를 기억하자. 이 부채를 줄일 의지와 인력이 없으면, **부채가 일정 임계를 넘는 순간 운영 비용이 가속도를 받는다.** Uber가 같은 자리에서 후퇴했다.

여섯째, **테스트 안정성**이다. 4장에서 우리가 NestJS 테스트 절을 길게 풀어 둔 이유가 여기서 회수된다. Jest 테스트가 자주 깨지고, flaky test가 늘고, CI가 빨간색에서 안 돌아온다면, 코드 품질이 아니라 시스템 설계의 신호일 수 있다. 비동기 경계가 제대로 정리되지 않았거나, 모킹 모델이 일관되지 않았거나, 분산 시스템의 일관성 가정이 깨진 자리가 테스트에서 먼저 드러난다.

이 여섯 가지 신호를 어떻게 측정할지가 다음 질문이다. **6장의 디버깅 도구로 측정하지 못하는 것은 결정할 수 없다.** clinic.js, perf_hooks, OpenTelemetry, Pino, Sentry, 이 도구들이 신호를 숫자로 바꿔 준다. 숫자가 없으면 신호도 없다. 신호가 없으면 멈춰야 할 시점도 모른다.

체크리스트로 정리하면 이렇다.

```
□ 신규 엔지니어가 한 달 안에 production PR을 올리는가?
□ 이벤트 루프 lag p99가 50ms 이내인가?
□ BFF의 CPU 핫패스에 정규식/JSON/crypto/sync I/O가 누적되지 않았는가?
□ 월간 운영 사고 수가 줄거나 정체하는가? (늘면 위험)
□ 사내 fork 라이브러리와 보안 패치 누락 의존성이 분기별로 줄어드는가?
□ Jest 테스트의 flaky 비율이 1% 이내인가?
```

이 체크리스트의 6항목 중 2개 이상이 6개월 연속 빨간불이라면, 후퇴 결정을 진지하게 검토할 시점이다. **후퇴는 패배가 아니다. 데이터에 기반한 합리적 재조정이다.**

후퇴를 검토할 때 한 가지 더 염두에 둘 것은, **후퇴의 모양도 점진적이라는 점이다.** "이번 분기에 새로 만드는 서비스 한 개를 Java로 짓는다"가 후퇴의 가장 작은 단위다. 기존 Node BFF를 갈아엎는 결정이 아니다. Uber도 그 모양으로 후퇴했다. 신규 서비스 추천 스택에서만 Node를 빼냈고, 기존 서비스는 그대로 돈다. 우리도 같은 결의 결정을 할 수 있다. 후퇴라는 단어가 무거우면 "재조정"이라고 부르자. 결정의 무게가 같아도 단어의 무게가 다르다.

## 학술적 뒷받침 — IEEE 2021 Green Button 사례

마지막으로 한 학술 사례를 짧게 인용하자. IEEE 2021의 "Microservice Migration Using Strangler Fig Pattern: A Case Study on the Green Button System"이다.

이 논문은 캐나다의 Green Button 에너지 데이터 공유 시스템을 모놀리스에서 마이크로서비스로 옮긴 사례를 1차 보고로 정리한 글이다. Spring/Node 같은 특정 스택에 묶이지 않고, **Strangler Fig 패턴 자체를 단계별로 적용했을 때 무엇이 작동하고 무엇이 깨지는지를 학술적으로 보고한 보기 드문 자료다.**

이 논문에서 우리에게 가장 가치 있는 부분은 단계 사이의 전환 비용에 대한 보고다. 1단계 API Gateway 분리는 가장 안전했지만, 그 자체로는 가치를 내지 못한다는 점. 2단계 공유 DB는 빠른 가치를 내지만 결합 비용이 누적된다는 점. 3단계 DB 분리가 가장 어렵고 시간이 오래 걸린다는 점. 이 보고가 우리 결정의 기대치를 현실적으로 잡아 준다. **DB 분리에 6개월이 걸린다는 사실을 미리 알고 시작하는 것과, 모르고 시작하는 것은 다르다.**

학술 사례 한 편이 우리 결정의 권위가 되지는 않는다. 그러나 사례 열 건과 산업 가이드(IBM·AWS·Microservices.io)와 학술 보고가 같은 패턴을 가리킨다는 사실은, 이 패턴이 일시적 유행이 아님을 확인해 준다. **2004년에 Martin Fowler가 이름을 붙인 패턴이 2024년에도 같은 모양으로 작동한다는 건, 패턴이 도구의 변화를 견뎌 내는 종류의 자산이라는 뜻이다.** 우리가 8장에서 사례를 길게 본 이유도 같다. 도구가 또 한 번 바뀌어도 패턴은 남는다. 패턴 위에 새 도구를 얹는 결정이 도구만 보고 내리는 결정보다 5년 뒤에 후회가 적다.

## 6개월 마이그레이션 로드맵 — 샘플

이제 모든 걸 한 자리에 모아 6개월 로드맵 샘플로 정리하자. 이건 정답이 아니다. 출발점이다. 우리 팀의 도메인과 인력에 맞게 조정해야 한다.

### 1개월차 — BFF 셋업

NestJS 프로젝트 하나를 띄운다. 4장에서 본 모듈·DI·인터셉터·가드 패턴으로 기초 구조를 잡는다. Pino 로깅, OpenTelemetry, Helmet, JWT, ValidationPipe, GlobalExceptionFilter를 첫날부터 넣는다. graceful shutdown도 첫날부터. 7장의 운영 체크리스트를 그대로 적용한다.

API Gateway를 모놀리스 앞에 둔다. 처음에는 모든 요청이 모놀리스로 가는 라우팅만 한다. NGINX 또는 K8s Ingress가 자연스럽다. 이 시점까지 모놀리스에 들어가는 변경은 거의 없다.

테스트 인프라를 같이 짠다. `Test.createTestingModule` 기반 단위 테스트와 Supertest 기반 통합 테스트의 패턴 두 개를 정착시킨다. CI에 Jest와 ESLint, Prettier를 건다. 이 한 달이 다음 다섯 달의 기반이다. 빨리 가려고 이 한 달을 줄이면 나중에 다 비싸게 돌아온다.

### 2~3개월차 — 한 도메인 이전 (BFF 후보)

가장 작은 BFF 후보 한 개를 고른다. 사용자에게 가까운 자리, 외부 호출 합산 자리, IO 바운드. 줌인터넷 사례의 1608라인 자리 정도가 적합하다. 이 한 도메인을 NestJS BFF로 옮긴다.

처음에는 공유 DB 단계로 시작한다. 새 BFF가 모놀리스 DB를 직접 읽는다. anti-corruption layer를 입구에 둔다. 모놀리스의 DTO가 새 시스템에 그대로 새지 않게 막는다. 이 한 결정이 5년 부채를 결정한다.

게이트웨이에서 feature flag를 걸어 1% 트래픽으로 시작한다. 메트릭(에러율, p99 응답 시간, 이벤트 루프 lag)이 임계 안에 들어오면 5%, 10%, 50%, 100%로 점진 이전. 이 두 달이 끝나면 한 도메인이 Node BFF에서 안정적으로 돌고 있는 상태가 된다.

### 4개월차 — 운영 안정화

이 한 달은 새 코드를 안 짜는 달이다. 운영 데이터를 모으고, 운영 룰을 정착시키고, 사내 라이브러리와 운영 매뉴얼을 정리한다. 신규 엔지니어 온보딩 가이드를 적는다. 7장에서 본 K8s 운영, Sentry 알람, 헬스체크 임계, 모두 이 시점에 정착시킨다.

후퇴 신호 체크리스트를 처음으로 측정한다. 6항목 중 빨간불이 있는지, 있다면 무엇 때문인지 분석한다. **이 분석이 다음 도메인 결정의 입력이다.**

### 5~6개월차 — 다음 도메인 또는 DB 분리

후퇴 신호가 모두 초록불이면 다음 도메인 이전을 시작한다. 첫 도메인보다 살짝 큰 영역, 또는 옆에 닿아 있는 영역으로. 첫 도메인 이전 경험이 두 번째 이전을 빠르게 만든다.

후퇴 신호 중 하나라도 빨간불이면, 그 빨간불을 먼저 해결한다. 새 도메인을 추가하지 않는다. **빨간불 위에 새 도메인을 얹으면, 새 도메인이 빨간불의 원인을 더 키운다.**

DB 분리(3단계)는 빠르면 6개월차, 보통 1년차 이후다. Debezium·Kafka 인프라를 도입할지 결정해야 하고, eventual consistency 모델을 받아들일지 결정해야 한다. 이 결정의 무게가 크기 때문에, 첫 6개월에는 보통 공유 DB 단계에 머무른다. 그게 Strangler Fig가 "천천히 자란다"의 의미다.

### 6개월 로드맵 표

| 시기 | 핵심 작업 | 산출물 | 후퇴 신호 측정 |
|---|---|---|---|
| 1개월 | NestJS BFF 셋업, API Gateway 도입 | 빈 BFF + 운영 도구 체인 | - |
| 2~3개월 | 첫 도메인 이전 (공유 DB, 1%→100%) | 한 도메인이 BFF에서 운영 | 첫 측정 |
| 4개월 | 운영 안정화, 매뉴얼 정착 | 운영 룰 + 온보딩 가이드 | 6항목 전수 측정 |
| 5~6개월 | 다음 도메인 또는 빨간불 해결 | 두 번째 도메인 또는 안정화 | 6항목 재측정 |

이 로드맵이 우리 팀에 그대로 맞는 건 아니다. 우리 팀의 도메인 크기, 인력, Spring 자산 규모에 따라 1개월 단위가 2개월이 될 수도 있고, 3개월이 1.5개월로 줄 수도 있다. **시간보다 중요한 건 단계의 순서다.** 셋업 → 첫 도메인 → 안정화 → 다음 결정. 이 순서를 지키지 않은 마이그레이션은 대부분 첫 도메인 시점에 깨진다.

흔히 빠지는 함정 두 개를 미리 적어 두자. 하나, **첫 달에 BFF 두 개를 동시에 띄우려는 충동.** 한 도메인의 운영 데이터를 모으기 전에 두 번째 도메인을 시작하면, 한 번에 두 곳에서 사고가 터지고 어느 쪽이 원인인지 분리하기가 어렵다. 욕심을 누르고 한 도메인을 끝까지 본 다음에 두 번째를 시작하자. 둘, **운영 안정화 한 달을 줄이려는 충동.** 새 코드를 안 짜는 한 달이 아깝게 보이지만, 그 한 달이 사내 라이브러리·운영 룰·온보딩 가이드·후퇴 신호 측정의 첫 사이클을 정착시킨다. 이 사이클이 없으면 두 번째 도메인 이전이 첫 번째와 같은 비용으로 다시 시작된다. 첫 번째에서 배운 게 두 번째에 누적되지 않는다는 뜻이다. 그 누적이 없으면 6개월 뒤에도 같은 자리에 서 있게 된다.

## 마무리 — 그리고 9장으로

8장이 길었다. 길어야 했던 이유를 한 번 더 짚자. 도구의 결정은 보통 한 장의 표로 끝내고 싶어진다. 그 충동에 따르면 우리 팀의 5년이 한 장의 표 위에 놓인다. 그 표는 보통 우리 도메인을 모르고, 우리 인력을 모르고, 우리 운영 데이터를 모르는 사람이 만든 표다. **사례를 충분히 본 다음에야 그 표를 다시 그릴 수 있다.**

우리가 본 사례 열 건을 마지막으로 한 줄씩 정리하자. PayPal은 큰 페이지에서 검증했고, Express의 비강제성을 Kraken.js로 강제했다. LinkedIn은 외부 서비스 대화 자리에서 30대를 3대로 줄였다. Netflix는 Java 마이크로서비스를 그대로 두고 BFF만 분리했다. Walmart는 SSR을 CDN과 사전 렌더링으로 두 층 분리했다. 당근마켓은 푸시 한 조각만 떼어 1500 RPS를 누락 없이 받았다. 줌인터넷은 1608라인을 472라인으로 줄이며 TPS 40%를 더 받았다. 직방의 자바 개발자는 Spring 7개 컴포넌트를 NestJS에 1:1로 매핑했다. 인프랩은 Express 졸업의 6가지 동기를 명시했다. 네이버파이낸셜은 메모리 400MB와 25MB의 K8s 비용 차이를 보였다. **그리고 Uber는 같은 자리에서 후퇴를 결정했다.**

이 열 줄이 우리 결정의 시작점이다. 끝점이 아니다. 이 줄들 위에 우리 팀의 도메인 한 줄을 더 적고, 우리 팀의 인력 한 줄을 더 적고, 우리 운영 데이터 한 줄을 더 적은 다음, 그제서야 결정의 표를 다시 그린다. 그 표가 우리 팀의 5년을 받칠 수 있는 표다.

한 가지를 더 기억해 두자. 사례를 잘 읽는 일과 결정을 잘하는 일은 다르다. 사례를 많이 읽은 사람이 결정을 잘하는 게 아니다. **자기 팀의 데이터를 정직하게 본 사람이 결정을 잘한다.** PayPal의 2배 RPS는 우리 팀의 RPS가 아니다. Netflix의 70% 응답 감소는 우리 팀의 응답 시간이 아니다. 줌인터넷의 71% 코드 축소는 우리 팀의 코드 베이스가 아니다. 사례는 가능성의 범위를 보여 주는 자료일 뿐, 우리 팀의 답을 대신 만들어 주지 않는다. 6장에서 우리는 도구로 측정하는 법을 익혔다. 그 도구로 우리 팀의 숫자를 모은 다음에야, 사례의 숫자와 비교가 가능해진다. 측정 없는 마이그레이션은 사례를 흉내 내는 일에 그친다. 측정한 위에서 시작한 마이그레이션은 우리 팀의 결정이 된다.

한 가지 더 짚자. 8장의 결정은 한 번 내리면 끝나는 결정이 아니다. **6개월마다 다시 보는 결정이다.** 운영 데이터가 쌓이면 결정의 입력이 바뀌고, 인력이 늘거나 줄면 결정의 무게가 바뀌고, 사례가 더 쌓이면 결정의 참조점이 바뀐다. 8장의 표는 한 번 그리고 끝나는 표가 아니라, 분기별로 다시 펴 보는 표다. 그 사실을 받아들이고 시작하자. 한 번에 완벽한 답을 찾으려는 충동을 누르고, 단계별로 답을 다시 그릴 준비를 하자.

여기까지가 8장의 일이었다. 그런데 한 가지 일이 남았다. 우리가 어디부터 자를지를 결정했다고 해서, 그 결정이 조직 안에서 자동으로 작동하는 게 아니다. **기술 선택 회의에서 "왜 Node인가"를 한 페이지로 답해야 하고, PR 리뷰에서 Spring 출신이 자주 놓치는 함정 다섯 가지를 막아야 하고, 신규 엔지니어 온보딩 4~6주 학습 경로를 짜야 하고, 후퇴 신호를 조직과 문화 차원에서 측정해야 한다.** 이게 9장의 일이다. 8장이 기술적 결정이었다면, 9장은 그 결정을 조직과 커리어 차원으로 끌어올리는 자리다. 도구의 결정에서 사람의 결정으로 한 단계 올라가자.
