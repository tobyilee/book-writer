참조: `01_reference.md` (12 섹션, 573 라인)
대상 독자: agentic / vibe coding 사용 개발자

## 강점

1. **5박자 내러티브가 단순하고 강하다.** 충격(1·2) → 이해(3·4) → 화해(5·6) → 통합(7) → 실천(8·9) → 종합(10). 책 한 권 분량에서 독자의 인지 곡선을 끌고 가는 구조로 무리가 없고, 챕터 간 콜백 포인트도 6개 명시적으로 깔려 있어 편집 단계에서 잃을 위험이 적다.

2. **사용자가 명시한 7개 핵심 요구가 모두 챕터로 매핑된다.** 정의(2장), 발생 메커니즘(3장), 미사라짐(4장), 환각=기능(5장), 인간 환각(6장), capability overhang(7장), 개발자 검증·자동화 생산성(8·9장). 빠진 요구는 없다.

3. **레퍼런스의 §1~§10이 거의 1:1로 챕터에 흡수된다.** §11(참고문헌)·§12(리서치 한계)는 부속·운영으로 빠진 게 자연스러우며, 본문 10챕터에 §1~§10이 고르게 분산됐다. 슬롭스쿼팅·OpenAI 2025 paper·CoVe·SelfCheckGPT·Semantic Entropy·RAG·harness engineering 같은 중요 토픽이 모두 명시적으로 들어가 있다.

4. **"왜 사라지지 않는가"를 독립 챕터(4장)로 잡았다.** 사용자가 도발적 질문으로 제기한 부분을 한 챕터 통째로 답한다. OpenAI 2025 paper 세 명제를 풀고, 검증 기법의 한계(0%로 못 감)까지 다음 챕터 전 못 박는 구성. 사용자 요구의 "철저한 분석"에 가장 직접적으로 응한 챕터다.

5. **8장 검증 전략을 3층(정형·자기·외부 grounding)으로 분류한 것이 깔끔하다.** 레퍼런스 §8.1~§8.6의 도구 카탈로그가 산만하게 흩어질 위험이 있었는데, 3층 분류 + Anthropic 공식 5가지 권장 + HITL이 한 챕터에 응축되어 독자가 "내 환경에 무엇을 깔지" 결정하기 쉽다.

## 약점·문제점

### Critical

#### C1. 8장이 비대하고, 카탈로그-검증-Anthropic 5가지가 한 챕터에 압축되어 밀도 폭발 위험

8장은 명목 4,800자(전체 중 최대)인데, 내용은 (1) vibe vs agentic 구분, (2) 환각 패턴 카탈로그 표, (3) 실무 사고 4건 깊이 분석(슬롭스쿼팅·Cursor·Devin·Claude Code), (4) 한국 후기, (5) 정형 검증 5종, (6) 자기 검증 5종(CoVe·SelfCheckGPT·Semantic Entropy·Verification subagent·Best-of-N), (7) 외부 grounding 4종(RAG·tool use·sandbox·HITL), (8) Anthropic 공식 5가지를 다 담는다. 단순 합계로도 4,800자에 못 들어간다. **레퍼런스 §7 전체 + §8 전체를 한 챕터에 흡수하려는 욕심**이다.

비교: 1·2장은 각 3,800자에 §1 또는 §7.2~§7.4 일부만 담는다. 챕터 균형이 깨졌다.

근거: 5축 중 **챕터 균형**과 **독자 적합도**. 4,800자에 8가지 묶음을 압축하면 표·리스트가 빽빽해지고 토비 문체의 평어체 호흡(설명-비유-감정 공감-청유)이 들어갈 자리가 사라진다.

**제안:** 8장을 **8a. 환각 카탈로그와 정형 검증** / **8b. 자기 검증·외부 grounding·HITL** 두 챕터로 분리. 각 3,800~4,200자. 카탈로그·실무 사고 4건·정형 검증을 8a로, 자기 검증·grounding·sandbox·HITL·Anthropic 5가지를 8b로. 이렇게 하면 본문은 10챕터 → 11챕터, 분량은 4,800 → 8,000자(+3,200자)로 늘어나지만 사용자 요구 "충분한 분량(깊이 있는 주제)"에도 부합한다.

#### C2. 사용자 요구 "충분한 분량" 충족 의문 — 본문 41,000자는 깊이 있는 주제에 박하다

레퍼런스 §1~§10을 본문 10챕터에 담는데 챕터당 평균 4,100자다. 사용자가 명시한 7개 축(정의·발생·미사라짐·기능·인간 환각·overhang·검증/자동화)을 각각 다루면서 사례 깊이·논문 풀이·인간 인지과학·논쟁 4관점까지 다 하기엔 챕터당 4,100자가 답답하다. 비교 기준: 토비 책 평균(`book-planning` 가이드의 한 챕터 ~5,000자) 대비도 짧다.

특히 4장(OpenAI 2025 paper 세 명제 + 정보 이론 + 검증 기법 한계 + Anthropic 솔직함)에 4,300자, 6장(분리뇌 + 기억 재구성 + 서사적 자아 + CSIRO + Amodei 5개 묶음)에 4,000자는 각 주제 하나당 800자에 못 미친다. 사색적 챕터가 yapping 위험으로 빠지진 않지만 **사례 묘사·은유 + 논문 풀이 + 사용자 감정 공감**을 토비 문체로 다 깔기엔 옹색하다.

근거: 사용자 요구 7번 "충분한 분량" + 책 특성에 명시된 "깊이 있는 주제".

**제안:** 챕터당 평균 4,800~5,200자로 상향, 본문 총 50,000~55,000자 목표로 재조정. 위 C1 분리(8장 → 8a/8b)와 결합하면 11챕터 × 약 4,800자 = 약 52,800자, EPUB 300페이지급으로 자연스럽게 확장된다.

#### C3. 6장(인간 환각)과 5장(환각=기능) 사이의 논리 연결이 plan에 안 보인다

5장 끝(generalization-hallucination trade-off) → 6장 머리(분리뇌)의 점프가 plan에서는 "화해" 박자라는 추상으로만 묶여 있다. 콜백 포인트도 "5장 trade-off → 7장 overhang", "6장 사회적 합의 → 8·9장 자동 검증"으로 6장과 5장의 직접 연결이 없다.

독자 시선에서: "환각은 기능이다(5장)"를 받아들이고 → "인간도 환각한다(6장)"로 넘어갈 때, 두 챕터가 *왜* 인접해야 하는지 책에서 명시되지 않으면 6장이 인지과학 외도처럼 보일 위험이 있다.

근거: 5축 중 **내러티브 흐름**. 5박자 구조의 "화해" 두 챕터가 서로 다리를 안 놓으면 책의 가장 도발적인 모티프가 약해진다.

**제안:** 6장 도입에 "5장에서 본 vector blending이 인간 뇌에도 있다"는 한 줄 다리를 명시. 또는 5장 말미에 "이 메커니즘이 LLM만의 것일까?"라는 청유형 질문으로 6장으로 자연 이행. plan의 콜백 포인트에 "5장 vector blending ↔ 6장 좌반구 interpreter"를 추가.

### Should

#### S1. 1장 도입이 사례 4개로 시작 — 첫 인상 과부하

1장은 슬롭스쿼팅 + Cursor 봇 + Claude Code Issue #10628 + Amodei vs Anthropic 47% 충돌 4개 사례를 다 담고 + 책의 입장 선언 + 챕터 지도까지 3,800자에 한다. 토비 책의 1장 평균은 *한 사례*에 깊이 들어가 독자를 잡는 패턴이 많다 (`toby-book-writing-style.md`의 "감정적 공감 표현"·"상황 가정"이 살 자리).

**제안:** 슬롭스쿼팅 한 사례를 1장 핵심에 두고(인물·시간·금액·다운로드 수까지), Cursor 봇은 짧은 곁가지, Claude Code Issue는 8장(자가 환각 카탈로그)으로 이관. Amodei vs Anthropic 충돌은 1장 후반의 책 입장 선언 직전 미끼로 짧게. 1장이 3,800자에서 사례 1+곁가지로 호흡이 살아난다.

#### S2. 9장 "Parallel agents·verification subagent·harness 3단계"가 한 챕터에 또 다시 압축

9장 4,400자에 (1) 두려움→안 씀→overhang 잠김 논리, (2) 자동 검증 표준 루프, (3) parallel agents, (4) verification subagent 2단계, (5) harness 3단계 진화(prompt → context → harness), (6) 비유(말·고삐·울타리), (7) 환각 자산화, (8) 인지 분배 8가지를 담는다. 8장과 같은 양상.

C1을 8장에서 해결하면 9장도 자연스레 부담을 나눠야 한다. 9장은 *생산성 극대화*에 집중하고, harness engineering 3단계 진화·자산화 워크플로는 별도 다룰 만한 무게가 있다.

**제안:** 9장은 자동 검증 루프 + parallel/subagent + 환각 자산화에 집중하고(4,400자 그대로 가능), harness engineering 진화 3단계는 10장 종합 직전 새 챕터(9.5장 또는 10장 앞에 짧은 챕터)로 빼거나, 8b장(C1 분리)의 후반에 흡수. 단, 9장의 핵심 약속(생산성 극대화)을 흐리지 말 것.

#### S3. 레퍼런스 §12(리서치 한계)가 책 본문에 전혀 안 들어간다

§12에는 8가지 리서치 한계가 있다 — 멀티모달 환각 미커버, 법조계 케이스, 한국어 LLM 환각 데이터 부족, mechanistic interpretability 깊이 부족 등. 이 중 **법조계 환각 케이스**(가짜 판례 인용으로 변호사 제재)는 1장 도입에 끌어쓸 매력 있는 외부 사례다. 개발자 독자에게도 "코드만의 문제가 아니다"라는 환기가 된다.

또한 **한국어 LLM 환각**(KT 믿음·HyperCLOVA·Kanana)은 한국 독자 대상 책이라면 부록이나 보강 1단락에 들어가야 한다. plan에 흔적이 없다.

**제안:** 1장 슬롭스쿼팅 옆에 법조계 가짜 판례 1단락 추가(짧게, 도발용). 한국어 LLM 환각은 8장 한국 vibe coding 후기 옆에 1단락 또는 부록 §B로 보강. 또는 에필로그에서 "이 책이 덜 다룬 것들"로 솔직히 명시.

#### S4. 챕터 핵심 질문에 "어떻게/무엇을" 답하는지가 약한 챕터들

- 2장 "환각이라 부르는 현상은 정확히 무엇이고, 어떻게 분해되는가?" → 답: 4축 분류 + confabulation 어휘. OK.
- **5장 "환각률 0%인 모델은 어떤 모델인가?" → "검색엔진"이라는 짧은 답이 챕터 맨 위에 박혀버려 독자가 5장 끝까지 읽을 동기가 약해진다.**
- **7장 "환각을 다룰 줄 알면 무엇이 풀리는가?" → 답: "능력 활용"이라는 추상 명제. 7장은 8·9장의 도입부로 기능하는데, 7장 자체의 *insight*가 명확히 안 보인다.**

**제안:** 5장 핵심 질문을 "환각 없는 LLM은 가능한가, 가능하다면 어떤 모습인가?"로 살짝 바꿔 답을 챕터 끝으로 미룬다. 7장 핵심 질문을 "왜 같은 모델·같은 시기에 어떤 팀은 throughput이 3배, 어떤 팀은 그대로인가?"처럼 *증거 기반* 질문으로 바꾸면 capability overhang의 실재성 입증으로 챕터가 한 결로 흐른다.

### Nice-to-have

#### N1. 챕터 번호 호명 일관성

내러티브 아크 본문에서는 "5장에서 trade-off → 7장 overhang으로 연결"이라 적었는데, 콜백 포인트 표에서도 "5장 generalization-hallucination trade-off → 7장 capability overhang"으로 같은 챕터를 가리킨다. 그러나 챕터 목록 5장의 주요 내용에는 trade-off가 §4.1~§4.3 참조로 들어가 있어 챕터 번호와 §번호가 약간 어긋난다. 책 자체에는 문제 없지만 plan 자체 가독성을 위해 챕터 번호 ↔ 레퍼런스 §번호 매핑 표를 한 번 더 점검할 것.

#### N2. 표지 제목 후보 선정 근거의 가독성

제목 후보 3개를 비교한 추천 단락이 깔끔하지만, 후보 2("환각은 버그가 아니다 — 능력 과잉의 시대")가 사용자 요구 "도발적·실무" 양쪽을 잡는 강한 카드인데도 "사색적 챕터의 비중이 작지 않다"는 이유로 밀려난 게 약하게 느껴진다. 본문이 1·2부 에세이 톤·3·4부 가이드 톤이면 후보 2가 오히려 *대상 독자(개발자)*의 첫 인상에 더 맞을 수 있다. 저자 판단 영역이지만 한 번 재고할 여지.

#### N3. Capability overhang의 *측정* 데이터가 7장에 없다

7장은 정의·인용 위주다. 사용자 요구 6번 "AI 능력 극대화 + 자동화로 생산성 극대화"에 직결되는 챕터인데, *얼마나* 끌어쓸 수 있는지의 정량 단서가 빈약하다. 레퍼런스 §6.3 "Coding Overhang"의 RL 신호 부분과 §6.5 Sholto Douglas 인용을 끌어쓰되, 가능하면 Anthropic·OpenAI 발표의 어떤 internal benchmark에서 throughput이 N% 늘었다 같은 수치를 한두 개 끌어오면 챕터의 무게가 다르다. 레퍼런스에 없으면 챕터 저술 단계에서 web-research에 추가 의뢰.

## 종합 판단

**판정: 부분 수정 후 진행 (Should-fix with one Critical split)**

전체 구조·5박자 내러티브·레퍼런스 흡수·사용자 7개 요구 매핑은 흠잡을 데가 없다. 그러나 실질적 진행 전 **두 가지는 반드시 손봐야 한다**.

1. **8장 분리** (Critical C1) — 현재 4,800자에 8묶음을 압축한 8장은 챕터 저술 단계에서 토비 문체가 들어갈 여유가 없어 카탈로그·리스트 책이 될 위험이 크다. 8a/8b로 분리하면 11챕터 구성이 된다.

2. **본문 분량 상향** (Critical C2) — 본문 41,000자는 사용자가 명시한 "충분한 분량(깊이 있는 주제)" 요구에 박하다. 챕터당 평균 4,800~5,200자, 총 50,000~55,000자로 상향. 위 8장 분리와 결합하면 자연스럽게 도달한다.

이 두 가지가 해결되면 5박자 구조 위에 토비 문체로 두꺼운 책 한 권이 깔리는 게 안전하다. 나머지 Should 항목(1장 사례 압축, 5장 핵심 질문 재정의, 6장 다리, harness 분리)은 8장 분리 결정에 따라 자연스럽게 정리될 가능성이 크고, Nice 항목은 챕터 저술 단계에서 부분 흡수 가능.

**평행선 위험 평가:** 위 Critical 2개는 사용자 요구("충분한 분량") 직접 인용이라 저자(book-planner) 입장에서 반박이 어렵다. 합의 가능성 높음. 8장 분리 방식·새 챕터 번호 매기기는 저자 재량.
