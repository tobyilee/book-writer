- **레퍼런스 §7-1(한국 커뮤니티 자료 부족)**을 plan이 "보강 권장 지점"으로 인지하고 있는 건 좋은데, 5장과 9장의 본문에서 한국 사례를 어떤 형태(박스? 미니 인터뷰? 인용?)로 넣을지가 비어 있다. **권고:** 형식만 미리 정해 두면 챕터 저술가가 헤맨다.

### 2. 대상 독자 적합성 (Critical — 2장 / Should — 8장)

**2장이 입문자에게 너무 빠르게 깊다.** "Python 기본 문법을 이제 막 익힌 학습자"가 진입점인데, 2장에서 ZeroMQ 5소켓 이름을 다 외우게 하고 JSON dict-of-dicts 메시지 형식 예시까지 보여주면 첫 100페이지에서 책을 덮을 위험이 있다. 1장에서 "노트북이 왜 좋은지"의 감정선을 막 끌어올린 직후에 네트워크 프로토콜 디테일이 들어오면 호흡이 깨진다.

**권고 (Critical):**
- 2장의 ZeroMQ 5소켓은 "프런트와 커널이 별도 프로세스"라는 큰 그림에 종속시키고, **5소켓 이름은 외우지 말고 그림 한 장으로 보여주는 정도**로 축소.
- 메시지 dict-of-dicts 예시는 본문이 아닌 **"더 들어가고 싶다면" 박스(부록 박스)**로 빼기.
- 대신 "셀을 실행하면 메시지가 어디로 가서 어떻게 돌아오는가"를 **Shell + IOPub 두 소켓만으로 단순화한 라이프사이클 다이어그램**으로 대체.

**8장은 양면이다.** Pimentel 논문의 24.11% / 4.03% 수치는 강력해서 입문자에게도 충격으로 잘 꽂힌다(이건 좋다). 그러나 NBSafety의 "fine-grained lineage tracking"까지 본문에서 풀어 설명하면 부담이다. **권고 (Should):** NBSafety는 한 단락(개념 + 의의)에서 멈추고 자세한 메커니즘은 각주.

### 3. 챕터 균형 (Critical — 7장 분할 검토)

7장이 26,000자로 다른 챕터 평균(약 18,000자)의 1.4배다. 더 큰 문제는 분량보다 **다루는 환경 수의 불균질성**이다.

- Marimo (깊게, reactive runtime + WASM + 슬라이더 데모)
- Deepnote / Hex (각각 별도 철학)
- Databricks (Spark·MLflow·Delta Lake — 다른 챕터들과 청자가 다름. 엔터프라이즈 데이터 엔지니어 관심사)
- 친척 4종 (Polynote, nteract, Zeppelin, Observable)
- 종합 결정 트리 + 환경 비교표 (책의 시그니처 두 개)

이건 **5개 환경 + 4개 친척 + 2개 시그니처 인포그래픽**이 한 챕터에 들어간 것이다. 입문자에겐 "환경 카탈로그"로 읽혀 호흡이 죽는다.

**권고 (Critical) — 두 가지 분할안 중 택일:**

**안 A (분할):** 7장을 두 챕터로 쪼갠다.
- 7장: **Marimo — 노트북을 다시 짜다** (reactive 단독 깊이, 슬라이더 데모, hidden state 차단 메커니즘)
- 8장 (신설): **협업·엔터프라이즈 노트북 — Deepnote, Hex, Databricks** (+ 친척 4종 한 페이지)
- 결정 트리·비교표는 8장(신설) 끝
- 이렇게 하면 hidden state(현 8장)는 9장으로, git/배포는 10장으로, 베스트 프랙티스는 11장으로 밀린다 → 책이 11장이 됨.

**안 B (압축):** 7장을 그대로 두되 친척 4종(Polynote/nteract/Zeppelin/Observable)을 본문에서 빼고 **부록 또는 비교표의 한 행**으로만 처리. Databricks도 "엔터프라이즈가 아니면 건너뛰기 박스" 명시 후 압축. 분량을 22,000자선으로 떨어뜨림.

내 의견은 **안 A 선호.** Marimo는 책의 두 번째 정점(8장 hidden state와 함께)이 될 만한 무게가 있고, Marimo만의 reactive·`.py` 저장·WASM·앱 배포가 Deepnote/Hex/Databricks와 청중·관심사가 다르다. 한 챕터에 묶으면 Marimo가 카탈로그 항목으로 격하된다.

### 4. 내러티브 흐름 (Should)

전체 호흡 1→2→3 도입 → 4~7 환경 → 8 정점 → 9~10 응용은 **개념적으로는 정합**하다. 그러나 두 지점에서 독자가 떨어질 위험이 있다.

- **2장→3장 사이의 골:** 2장이 위 (1)·(2)에서 지적한 대로 무거우면, 3장 매직 8선(즉시 손에 잡히는)으로 넘어갈 때 "어렵다 → 갑자기 쉽다"의 톤 점프가 생긴다. 2장을 가볍게 만들어야 이 부분이 매끄러워진다.
- **6장→7장 사이의 호흡:** 6장 "VS Code/Cursor"의 톤(에디터 통합, 친근함)에서 7장 도입의 Marimo reactive runtime으로 가면 갑자기 학술적·이론적이 된다. **권고:** 7장 도입에 "왜 새로운 노트북이 필요했나"를 1~2페이지로 두어 6장의 실용 톤에서 7장의 철학 톤으로 다리를 놓아라. (이미 핵심 질문이 그것이긴 한데 plan 본문에 명시되지 않음.)

### 5. 중복·여백 (Should — 중복 / Critical — 환경 설정 누락)

**중복 (Should):**
- 매직이 3장 전반과 7장 Databricks(`%sql`/`%scala`) 양쪽에 나온다. 큰 중복은 아니지만, 7장 Databricks 절은 "이건 3장의 매직이 다른 언어로 확장된 형태다"라는 한 줄 콜백을 넣어 명시적으로 연결하는 게 좋다.
- "Cursor + Colab" 워크플로우가 5장 끝과 6장 본문 양쪽에 나온다 — plan에 5장이 "미리 소개"라고 적혀 있긴 하다. 그래도 5장 분량을 줄이려면 5장은 "6장에서 자세히"로 한 문장만 남기고 다 6장으로 보내는 게 깔끔하다.

**누락 (Critical):**
- **환경 설정 — `pip` / `conda` / `venv` / `uv`가 plan 어디에도 없다.** 입문자가 노트북을 시작하려면 가장 먼저 부딪히는 게 환경 설정이다. "Jupyter 설치는 어떻게? 왜 conda를 써? requirements.txt는 뭐야?"가 비어 있으면, 8장에서 "`requirements.txt`로 버전 핀해라"가 떠 있는 말이 된다. **권고:** 4장 도입 또는 부록으로 "환경 설정 미니 가이드"를 추가하라. 한 페이지면 충분하다.

**여백 (Should):**
- **노트북 보안.** plan은 "보강 권장 지점"에서 "이 책의 범위 밖 박스로" 처리한다고 했는데, 이게 박스 한 줄로 끝나는지 4장 JupyterHub 절의 한 단락인지 모호하다. **권고:** 박스 위치·길이·내용을 결정 메모에 명시.
- **실행 컨텍스트의 비용.** 클라우드 노트북(Colab/Databricks)은 무료가 아니다. Colab Pro $11.99도 적었지만, "Databricks 클러스터를 켜둔 채 잊으면 한 달에 얼마"의 현실적 비용 감각이 5장이나 7장에 있어야 입문자가 다친다. **권고:** Databricks 절에 "비용 함정 한 박스".

### 6. 시점 종속성 (Should)

plan이 시점 명시 의무 항목 4개(Colab 가격, Cursor 기능, Marimo Pair, Notebook Wars 2025)를 잘 식별했다. 다만 추가 검토 필요:

- **Notebook v6/v7/`nbclassic` 상태(4장)**도 2026 기준이라 시점 표기 필요.
- **Jupyter AI의 ACP 통합 모델 명단(Claude/Codex/Copilot/Gemini)**도 빠르게 변동 — 10장에 시점 명시.
- **Marimo의 `mo.ui.slider` API**도 2026-05 기준이라는 한 줄. 책 수명을 위해 "API는 변할 수 있다, 핵심은 reactive 모델"이라는 메타 문장 한 줄.

### 7. 시그니처 결정 트리·비교표 (Critical)

7장 끝의 결정 트리가 책의 핵심 인포그래픽이라면 plan에 분기 기준이 너무 추상적이다. 현재 plan에는 다섯 갈래 질문(혼자 EDA / 팀 협업 / TB급 / 출판 / 재현성)만 적혀 있고, **각 질문의 답이 어느 환경으로 떨어지는지**가 비어 있다.

**권고 (Critical):** plan 단계에서 결정 트리의 **모든 종단 노드**(leaf node)를 확정하라. 예:
- "혼자 EDA + 로컬 + GPU 없음" → JupyterLab 또는 VS Code Notebooks
- "혼자 EDA + GPU 필요" → Colab / Kaggle
- "팀 협업 + 비기술 이해관계자에게 공유" → Hex
- "팀 협업 + 셀 코멘트만 필요" → Deepnote
- "TB급 + Spark" → Databricks
- "출판물 산출" → Quarto
- "재현성 최우선 + 새 프로젝트" → Marimo
- "재현성 최우선 + 레거시 Jupyter 자산" → Jupytext + nbdime + Restart and Run All

이게 plan에 안 들어가면 챕터 저술가가 본문에서 처음 결정해야 하고, 그러면 분기가 비일관적으로 나올 위험이 크다. **결정 트리는 plan의 산출물이지 본문 저술의 산출물이 아니다.**

비교표도 마찬가지. 6개 컬럼(저장 포맷 / 실행 모델 / 협업 / 가격 / 강점 / 약점)은 좋은데, **9개 환경 × 6 컬럼 = 54셀의 내용이 plan에 미리 결정**되지 않으면 챕터 저술가가 임의로 채울 위험이 있다.

### 8. Toby 스타일 적용 가능성 (Should)

plan이 잘 짚었다 — "1장과 8장은 감정선이 강해 평어체·청유형이 자연스럽고, 2~3장은 담담하게, 7장은 인포그래픽 중심이라 캡션에 톤 집중." 이 인식 자체에 동의한다.

다만 **8장은 데이터(Pimentel 수치)·논쟁(Joel Grus·Netflix)·해법(reactive·Restart and Run All·NBSafety)의 무게가 있어 평어체가 반대로 가벼워 보일 위험이 있다.** Toby 스타일이 "수다스러운 친구"로 들리면 24.11%라는 충격 수치의 무게가 빠진다.

**권고:** 8장은 **"진지한 평어체"**로 정의해라. 청유형은 "한번 따라가보자" 정도로 자제하고, 수사적 질문은 진단 단락에서만 사용. "그런데 정말 그럴까?", "왜 우리는 매번 이 함정에 걸릴까?" 같은 무게 있는 질문은 좋고, "헤헤 어렵지?" 같은 가벼운 톤은 피한다. 챕터 저술가에게 이 디렉션을 명시적으로 줘야 한다.
