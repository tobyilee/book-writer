# 2장. 셀, 커널, 상태 — 노트북의 해부학

JupyterLab을 띄우고 빈 노트북을 하나 만들었다고 해보자. 첫 셀에 이렇게 친다.

```python
import pandas as pd
df = pd.read_csv("sales.csv")
df.head()
```

`Shift+Enter`. 0.3초쯤 뒤에 표가 셀 아래에 뜬다. 너무 자연스러워서 그 사이에 무슨 일이 벌어졌는지 생각하지 않는다. 그런데 이 0.3초 동안 사실은 두 개의 프로세스가 ZeroMQ 소켓 위에서 JSON 메시지를 주고받았다. 코드는 한 프로세스에서 짜고, 실행은 다른 프로세스에서 됐다. 결과는 또 다른 채널로 돌아왔다.

복잡해 보이지만 본질은 단순하다. 노트북은 두 개의 분리된 프로세스로 돼 있다. 이 한 가지 사실을 손에 잡으면, 어느 노트북 환경을 만나도 흔들리지 않는다. 같이 해부해보자.

## 두 프로세스 모델 — 프런트엔드와 커널

해부에 들어가기 전에 한 가지를 확인해두자. 노트북은 사실 **두 개의 분리된 프로세스**로 구성된다. 우리가 보는 화면이 한 프로세스, 코드를 실제로 돌리는 인터프리터가 다른 프로세스다. 이 둘은 같은 컴퓨터에 있을 수도, 다른 컴퓨터에 있을 수도 있다. 같은 OS 사용자로 돌 수도, 다른 컨테이너에 격리될 수도 있다. 어쨌든 별개다.

이 분리를 인식하지 않으면 노트북의 많은 동작이 마법처럼 보인다. "왜 변수가 셀 사이에 살아 있지?", "왜 노트북을 닫았다 다시 열면 변수가 사라지지?", "왜 같은 노트북을 두 브라우저에서 열어도 동기화되지?" — 이 질문들의 답이 다 두 프로세스 구조에 있다.

먼저 그림 한 장을 머리에 박자.

```
[프런트엔드]                    [커널]
브라우저 / VS Code / Cursor      Python 인터프리터
- 셀 UI 그림                    - 코드 실행
- 마크다운 렌더                 - 변수·임포트·함수 메모리 보관
- 출력 표시                     - 결과 만들어 보냄
                  ↓ ↑
              ZeroMQ 메시지
            (JSON over TCP)
```

프런트엔드는 우리가 보는 화면이다. 셀을 그리고, 마크다운을 렌더링하고, 출력을 표시한다. 키보드 입력을 받고 클릭에 반응한다. 그게 다다. 코드를 직접 실행하지는 않는다.

커널은 우리가 보지 못하는 별도 프로세스다. 프런트엔드가 "이 코드 좀 돌려줘" 하고 보내면 받아서 실행하고, 결과를 다시 돌려준다. 그 사이에 임포트한 모듈, 정의한 변수, 만든 함수를 자기 메모리에 들고 있다. 셀과 셀 사이에 변수가 살아남는 비밀이 여기 있다 — 우리가 새 셀을 칠 때마다 새 인터프리터가 뜨는 게 아니라, 같은 커널이 계속 같은 메모리를 들고 있다.

이 분리가 왜 중요할까. 세 가지 이점이 있다.

**첫째, 언어 독립성.** 프런트엔드는 Python을 모른다. 그냥 코드 문자열을 커널에 보낼 뿐이다. 그래서 같은 JupyterLab에서 R 커널, Julia 커널, Scala 커널이 돌 수 있다. "Jupyter"라는 이름이 Julia + Python + R에서 왔다는 얘기를 1장에서 했었다. 그 다언어성의 비밀이 이 2-프로세스 구조다. 지금은 40개 이상의 언어 커널이 존재한다.

**둘째, 원격 실행.** 커널이 별도 프로세스라는 건, 그 프로세스가 다른 컴퓨터에 있어도 된다는 뜻이다. 내 브라우저는 노트북 한국에 띄워두고 커널은 미국 GPU 서버에서 돌리는 게 가능하다. Colab과 Kaggle이 이 모델로 굴러간다. 우리가 보는 건 웹 페이지지만, 코드를 실제로 돌리는 건 구글 데이터센터의 어느 컨테이너다.

**셋째, 프런트엔드 교체 가능성.** 같은 `.ipynb` 파일을 JupyterLab으로 열든, VS Code로 열든, Cursor로 열든 같은 커널과 통신한다. UI는 갈아치울 수 있는 옷이다. 그래서 책에서 다루는 환경 아홉 가지 중 다섯 개가 결국 같은 Jupyter 커널을 다른 UI로 감싼 것이다.

**넷째, 격리와 안전성.** 코드 실행이 별도 프로세스에서 일어난다는 건, 그 프로세스가 죽어도 UI는 멀쩡하다는 뜻이다. 무한 루프를 돌렸다, 메모리를 다 먹었다 — 커널만 재시작하면 된다. 우리가 짠 노트북, 셀 입력, 출력은 다 프런트엔드(브라우저)가 들고 있어서 잃지 않는다. 만약 한 프로세스 모델이었다면 한 셀의 무한 루프가 노트북 자체를 통째로 날렸을 것이다.

이 네 가지 이점은 그냥 디자인 결정의 부산물이 아니라 노트북이라는 매체가 데이터·ML·교육에서 살아남은 핵심 이유다. Pérez와 Jupyter 팀이 처음부터 이 분리를 의도했고, 그 결정이 십수 년 뒤 Marimo 같은 새 도전자가 나타나도 Jupyter 생태계를 흔들지 못하는 깊은 뿌리가 됐다.

## Shift+Enter를 누르면 벌어지는 일

이제 그 0.3초를 슬로우 모션으로 따라가보자. 본문은 핵심 두 채널만 다룬다 — Shell과 IOPub. 나머지 디테일은 뒤의 박스에 모아뒀다.

`Shift+Enter`를 누르는 순간 프런트엔드는 셀의 코드를 문자열로 뽑는다. 거기에 메타데이터를 붙여 JSON 메시지 하나를 만든다. 대충 이런 모양이다.

```json
{
  "header": {
    "msg_id": "abc-123",
    "msg_type": "execute_request",
    "session": "sess-789",
    "username": "toby",
    "date": "2026-05-06T12:34:56Z",
    "version": "5.3"
  },
  "content": {
    "code": "df = pd.read_csv('sales.csv')\ndf.head()",
    "silent": false,
    "store_history": true
  }
}
```

이 메시지가 **Shell** 채널을 통해 커널에 도착한다. 커널은 받은 코드를 IPython 인터프리터에 넘겨 실행한다. 실행 결과는 두 갈래로 나뉜다.

하나는 **stdout** 같은 출력. `print()`가 만들어내는 텍스트, 마지막 표현식의 값, 그래프 이미지 — 이런 게 **IOPub** 채널을 통해 브로드캐스트된다. 'Pub'은 publish의 줄임이다. 한 명에게 답하는 게 아니라 듣고 있는 모두에게 뿌린다는 뜻이다. 그래서 같은 노트북을 두 브라우저에서 열어두고 한쪽에서 셀을 실행하면, 다른 쪽에도 결과가 뜬다. JupyterHub의 협업 모드도 같은 메커니즘 위에 있다.

다른 하나는 **execute_reply**. "그 셀 실행 끝났어. 성공이야 / 에러야" 라는 짧은 응답. 이건 다시 Shell 채널로 돌아온다. 프런트엔드는 이 응답을 받고서야 셀 옆의 `In [*]` 표시를 `In [3]` 같은 번호로 바꾼다.

그러니까 셀 하나를 누르면 메시지가 이 순서로 흐른다.

```
프런트엔드 → Shell → 커널         (execute_request)
                       ↓
                       실행
                       ↓
커널 → IOPub → 모든 프런트엔드   (stream / display_data / execute_result)
커널 → Shell → 그 프런트엔드     (execute_reply)
```

이게 다다. 본문에서 알아야 할 메커니즘은 이 한 장이면 충분하다. 어느 노트북 환경을 만나도 이 라이프사이클이 깔려 있다.

## 같은 셀을 두 브라우저에서 열면

방금 IOPub가 'publish'라고 했다. 한 명에게 답하는 게 아니라 모두에게 뿌린다. 이게 무슨 의미인지 한 가지 시나리오로 보자.

같은 노트북을 두 브라우저 탭에서 열어둔다. 한 탭은 데스크톱 모니터, 다른 탭은 노트북 PC. 데스크톱에서 셀을 실행한다.

```python
# 데스크톱 탭에서 실행
x = 42
print(x)
```

결과가 데스크톱 탭에 뜬다. 그런데 노트북 PC 탭으로 가서 새 셀에 `print(x)`를 친다.

```python
# 노트북 PC 탭에서 실행
print(x)  # 42
```

값이 살아 있다. 같은 커널을 두 프런트엔드가 공유하고 있기 때문이다. 한 쪽에서 정의한 변수가 다른 쪽에서 그대로 보인다. 출력도 양쪽 탭에 다 뜬다 — IOPub가 모든 구독자에게 브로드캐스트하니까.

이 메커니즘이 JupyterHub의 협업 모드, RTC(Real-Time Collaboration) 확장의 기반이다. 한 노트북을 여러 사람이 동시에 보면서 한 사람이 셀을 실행하면 모두에게 결과가 뜬다. Google Docs의 실시간 편집과 비슷한 사용감이 같은 메커니즘 위에서 굴러간다.

다만 두 사람이 동시에 다른 셀을 만지면서 변수를 공유하는 건 위험하다. 누군가 `df = pd.read_csv("a.csv")`를 돌렸는데 그 직후에 다른 사람이 `df = pd.read_csv("b.csv")`를 돌리면 `df`가 b로 덮인다. 이게 hidden state 문제의 협업 버전이다. Deepnote, Hex 같은 SaaS는 이걸 해결하기 위해 사용자별로 별도 커널을 띄우거나, 셀 단위 잠금 같은 장치를 따로 둔다(8장).

## 커널이 죽거나 멈추면

커널이 별도 프로세스라는 건, 그 프로세스가 따로 죽을 수 있다는 뜻이기도 하다. 무한 루프를 돌렸다, 메모리를 다 먹었다, 아니면 그냥 노트북을 띄워둔 채로 노트북 PC가 잠들었다 — 이런 일이 흔히 생긴다. 셀을 눌러도 반응이 없다. `In [*]`가 한참 깜빡인다. 난감하다.

이때 프런트엔드는 어떻게 알아챌까. **Heartbeat**라는 별도 채널이 있다. 프런트엔드가 일정 간격으로 핑을 보내고 커널이 퐁을 돌려준다. 응답이 끊기면 "커널이 죽었네" 하고 빨간 점을 띄운다.

그럼 죽은 게 아니라 그냥 무한 루프를 돌고 있다면? 메뉴에서 Kernel → Interrupt를 누르면 된다. 이건 **Control** 채널을 통해 인터럽트 신호가 간다. Control은 Shell과 분리돼 있다. 왜냐. Shell은 지금 실행 요청 처리하느라 막혀 있을 수 있는데, 그 와중에 "그만!"이라는 명령이 Shell을 타고 가면 영원히 처리되지 못한다. Control이 따로 있어야 살아 있는 채널을 통해 멈출 수 있다. 작은 디테일이지만 잘 설계됐다.

`input()`을 쓴 코드는 또 다른 채널이 필요하다. `input()`이 사용자 입력을 기다리는 동안, 커널은 "사용자한테 뭐 좀 물어봐줘"라고 프런트엔드에 부탁해야 한다. 이건 **Stdin** 채널이 담당한다.

Shell, IOPub, Heartbeat, Control, Stdin — 이 다섯 채널이 ZeroMQ 위에서 동시에 돈다. 본문에선 Shell과 IOPub만 알아도 충분하다고 했다. 나머지는 호기심 많은 독자를 위해 박스에 모아뒀다(Box A).

## 같은 노트북, 다른 언어 — 커널 다중성

같은 JupyterLab에서 새 노트북을 만들 때 우측 상단에 커널 선택 메뉴가 뜬다. Python 3가 기본이지만, 거기에 R, Julia, Scala, JavaScript, 심지어 Bash까지 띄울 수 있다. 어떻게 이게 가능할까.

프런트엔드는 코드를 그저 문자열로 다룬다고 했다. 무슨 언어인지 신경 쓰지 않는다. 커널이 받아서 자기 인터프리터로 실행할 뿐이다. 그러니 R 인터프리터를 감싼 IRkernel을 띄우면 R 노트북이 되고, Julia 인터프리터를 감싼 IJulia를 띄우면 Julia 노트북이 된다.

각 언어 커널은 따로 설치해야 한다. R 커널을 쓰고 싶으면 R을 깔고 IRkernel 패키지를 설치한 다음 `IRkernel::installspec()`을 한 번 돌리면 된다. 그러면 다음에 JupyterLab을 띄울 때 메뉴에 R이 나타난다.

```r
# R 콘솔에서
install.packages("IRkernel")
IRkernel::installspec(user = TRUE)
```

Julia도 비슷하다.

```julia
# Julia REPL에서
using Pkg
Pkg.add("IJulia")
```

설치 후 JupyterLab을 다시 띄우면 New Launcher에 Python 3 외에 R, Julia가 같이 떠 있다. 노트북별로 어떤 커널을 쓸지 고를 수 있다.

데이터 사이언스를 R로 시작했다가 Python으로 옮기는 사람, 또는 Julia로 수치 계산을 하면서 시각화는 Python으로 하는 사람 — 이런 사람들에게 한 환경에서 두 언어를 오가는 능력은 큰 자산이다. 이게 Jupyter라는 이름에 담긴 약속이다.

조금 더 깊은 시점에서 보자. 커널 인터페이스를 표준화했기 때문에, 누가 새로운 언어를 위한 커널을 만들면 자동으로 모든 Jupyter 호환 환경(JupyterLab, VS Code Notebooks, Cursor, Colab, Hex, Deepnote)에서 그 언어가 돈다. 한 곳에 만든 커널이 모든 UI에서 쓰인다는 건 기여의 가치를 한 번에 수십 배로 키운다. 이게 Jupyter 생태계가 십수 년 동안 활발한 핵심 이유 중 하나다.

다만 한 노트북 안에서 셀마다 다른 언어를 섞을 수는 없다. 한 노트북은 하나의 커널과 1:1로 묶인다. 여러 언어를 섞고 싶으면 Apache Zeppelin이나 Polynote, 그리고 Databricks Notebooks 같은 별도 환경을 봐야 한다 — 8장에서 다룬다. Databricks는 같은 노트북에서 `%sql`, `%python`, `%scala`, `%md` 매직으로 셀별 언어를 바꾼다. 이건 매직이 다른 언어로 확장된 형태인데, 노트북 표준이 아닌 Databricks 자체 확장이다.

## 상태(state)라는 그림자

지금까지 노트북의 해부학을 도구의 시선으로 봤다. 이제 위험한 얘기를 하나 해두자. 바로 **상태(state)**다.

커널은 메모리 안에 변수, 함수, 임포트한 모듈을 들고 있다. 셀을 실행할 때마다 그 메모리가 갱신된다. 이게 우리가 셀 사이에 변수를 공유할 수 있는 이유고, 노트북의 핵심 편의이기도 하다.

그런데 같은 메커니즘이 가장 큰 그림자를 만든다. 셀 세 개를 다음 순서로 친다.

```python
# 셀 1
x = 10

# 셀 2
x = x + 5
print(x)   # 15

# 셀 3
print(x)   # 15
```

여기까진 자연스럽다. 그런데 셀 2를 한 번 더 실행한다. 화면엔 `15`가 또 찍힌다. 그러나 메모리의 `x`는 이제 20이다. 셀 3을 다시 실행하면 `20`이 나온다. 보이는 코드는 `print(x)` 그대로인데 결과가 바뀌었다.

이게 더 심해지는 사례를 보자.

```python
# 셀 1
import pandas as pd
df = pd.read_csv("sales.csv")

# 셀 2 — 어제 잠깐 만들었다가 지운 셀
df = df.drop(columns=["region"])

# 셀 3
df.columns
```

어제 셀 2를 지웠다. 노트북엔 셀 1과 셀 3만 남아 있다. 그런데 `df.columns`를 다시 실행하면 `region` 컬럼이 사라져 있다. 왜냐하면 어제 메모리에서 그 컬럼이 한 번 떨어졌고, 그 상태가 아직 커널에 남아 있기 때문이다.

오늘 노트북을 다시 열고 위에서 아래로 차례대로 실행하면 어제와 다른 결과가 나온다. 그런데 화면의 출력은 어제 실행한 그대로 박혀 있다. 코드와 결과와 메모리 — 이 세 개가 따로 논다.

**보이는 코드 ≠ 메모리 상태.** 이게 hidden state 문제의 본질이다. 노트북이 가진 가장 큰 자유(셀 임의 실행)가 가장 큰 함정(상태와 코드의 분리)을 만든다.

이 분리가 일어나는 경로는 다양하다. 셀을 위→아래 순서가 아닌 임의 순서로 실행할 때. 한 번 정의한 변수를 다른 셀에서 재할당할 때. 셀을 지웠지만 그 셀이 만든 변수가 메모리에 남아 있을 때. 셀 순서를 위로 끌어올렸는데 의존성이 깨졌을 때. 이런 모든 변화가 메모리에 누적된다. 화면의 코드는 한 가지 이야기를 보여주는데, 실제 메모리는 다른 이야기를 들고 있다.

가장 잔인한 사례는 이런 거다. 어제 작업하다가 노트북을 닫았는데 깜빡 저장을 안 했다. 오늘 노트북을 다시 열었더니 어제 본 출력은 다 그대로다 — 출력은 `.ipynb` JSON에 박혀 있으니까. 그래서 마치 어제 작업이 그대로 살아 있는 것 같다. 그런데 사실 커널은 죽었다 다시 시작된 새 커널이고, 메모리는 비어 있다. `df`라는 변수를 쓰는 셀을 실행하면 `NameError: name 'df' is not defined`. 황당하다. 노트북을 며칠씩 띄워두고 작업하는 사람이 정말 흔히 만나는 함정이다.

이 문제는 9장에서 정면으로 다룬다. Pimentel 연구팀이 2019년에 GitHub의 노트북 140만 개를 분석한 결과, 그 중 24.11%만이 에러 없이 실행됐고, 4.03%만이 같은 결과를 재현했다. 이 충격적인 숫자가 어디서 왔는지, 어떻게 막을지, 그리고 Marimo 같은 새 도구가 이 문제를 어떻게 다시 짜고 있는지 — 그게 9장의 절반을 채운다.

지금은 한 가지만 머리에 남기자. **노트북에서 변수를 본다는 것은 메모리 상태를 보는 것이지, 코드를 보는 것이 아니다.** 새 노트북을 받았을 때 가장 먼저 할 행동은 Kernel → Restart Kernel and Run All Cells다. 위에서 아래로 한 번에 돌렸을 때 같은 결과가 나오는지 확인하는 것 — 이게 노트북 위생의 출발점이다.

## 셀 번호 — `In [3]`이 알려주는 것

셀 옆의 `In [3]` 표시도 잠깐 들여다보자. 이 숫자가 뭘 의미할까. 셀이 실행된 순서다. `In [1]`이 첫 번째 실행, `In [2]`가 두 번째, 그렇게 누적된다. 셀 위치 순서가 아니라 실행 순서다.

같은 노트북에서 셀을 위→아래 순서로 한 번씩 실행하면 `In [1]`, `In [2]`, `In [3]`, … 이 위에서 아래로 가지런히 배열된다. 그런데 위에서 아래가 아닌 임의 순서로 실행하면 — 예를 들어 첫 셀, 그다음 마지막 셀, 그다음 가운데 셀 — 번호가 `In [1]`, `In [2]`, `In [3]`이 아니라 `In [1]`, `In [3]`, `In [2]` 같은 식으로 위치와 무관하게 흐트러진다.

새 노트북을 받았을 때 이 번호를 한 번 눈으로 따라가보자. 위에서 아래로 1, 2, 3, … 가지런하게 흐르면 "위에서 아래로 한 번에 돌렸구나" 안심할 수 있다. 번호가 흐트러져 있으면 "임의 순서로 실행한 흔적이다, 다시 돌려보자" 의심의 신호다. 코드 리뷰의 작은 단서로 자주 쓴다.

`In [*]`은 "지금 실행 중"이다. 별이 한참 깜빡이면 그 셀이 처리 중이라는 뜻. 무한 루프거나, 큰 계산이거나, 또는 커널이 멈춘 신호일 수 있다. 별이 사라지고 숫자가 박히면 실행 완료.

`Out [3]`은 그 셀의 마지막 표현식이 만들어낸 값이다. 표현식이 아니라 문(`x = 10`처럼 할당)이면 `Out`은 안 뜬다. `print(x)`도 `Out`을 만들지 않는다 — print는 stdout으로 흐르고, 그건 IOPub의 `stream` 메시지로 따로 뜬다.

이 작은 표시들이 노트북의 상태를 시각적으로 알려주는 단서다. 잘 활용하면 hidden state 점검이 한결 가벼워진다.

## 멘탈 모델 점검 — 새 환경에서 30분

지금까지 본 멘탈 모델이 진짜 어디서나 통하는지 짧은 시나리오로 확인하자.

**시나리오 A — Colab.** 새 Colab 노트북을 만든다. 화면이 JupyterLab과 다르지만, 셀 옆의 `In [1]`, 우측 상단의 커널 상태(연결됨/연결 끊김), 메뉴의 `런타임 → 런타임 다시 시작` — 이 세 가지가 그대로 있다. Restart and Run All도 같은 자리에 있다. 같은 두 프로세스 모델이다. 단지 커널이 내 컴퓨터가 아니라 구글 클라우드의 어느 컨테이너에서 돌 뿐이다.

**시나리오 B — VS Code Notebooks.** 같은 `.ipynb`를 VS Code에서 연다. 셀 표시는 다르게 생겼지만 실행 번호, 변수 탐색기, 커널 선택 — 다 있다. 단축키가 약간 다르지만 30분이면 적응한다. 같은 모델 위에 다른 UI가 얹혔을 뿐이다.

**시나리오 C — Cursor.** AI 에이전트가 셀 안에 들어와 있다. 코드 자동 생성, 셀 자동 수정 — 이런 게 추가됐다. 그러나 그 아래에서 도는 건 같은 Jupyter 커널이고, 같은 두 프로세스 모델이고, 같은 hidden state 함정이 그대로 있다. AI가 셀을 만들어 주더라도 그 셀이 메모리 상태에 미치는 영향은 사람이 짠 셀과 같다.

**시나리오 D — Marimo.** 여기는 다르다. Marimo는 "두 프로세스, 셀 임의 실행, hidden state"라는 모델 자체를 부정한다. 의존 그래프를 추적해 셀 실행 순서를 자동으로 위상정렬하고, 한 셀의 변경이 영향을 주는 다운스트림 셀을 자동 재실행한다. 그래서 30분이 아니라 한두 시간 더 들여야 사고방식이 바뀐다. 7장에서 본격적으로 다룬다.

이 네 시나리오 중 셋이 같은 모델 위에 얹혀 있다는 사실이 이 장의 핵심 효용이다. 한 번 깊이 이해해두면 새 환경 대부분에 빠르게 적응할 수 있다.

## 커널 스펙(kernelspec) — 커널이 어디 사는지

조금 더 깊이 들어가보자. JupyterLab을 띄우면 어떤 커널이 메뉴에 뜰지를 어떻게 결정할까. **커널 스펙**이라는 작은 JSON 파일이 답이다.

자기 컴퓨터에 설치된 커널 목록을 보려면:

```bash
jupyter kernelspec list
```

```
Available kernels:
  python3        /Users/toby/.venv/notebook/share/jupyter/kernels/python3
  ir             /Users/toby/Library/Jupyter/kernels/ir
  julia-1.10     /Users/toby/Library/Jupyter/kernels/julia-1.10
```

각 디렉토리 안에 `kernel.json`이 있다. python3 것을 보면 이런 모양이다.

```json
{
  "argv": [
    "/Users/toby/.venv/notebook/bin/python",
    "-m", "ipykernel_launcher",
    "-f", "{connection_file}"
  ],
  "display_name": "Python 3 (ipykernel)",
  "language": "python"
}
```

`argv`가 핵심이다. JupyterLab이 이 노트북에 커널을 띄울 때 정확히 이 명령을 실행한다. 그러니까 커널 스펙을 직접 만들면 어떤 가상환경이든, 어떤 파이썬이든 노트북 메뉴에 띄울 수 있다.

자주 쓰는 패턴: 프로젝트별로 가상환경을 만들고, 그 환경의 파이썬을 가리키는 커널 스펙을 등록한다.

```bash
# 프로젝트 가상환경 활성화
source .venv/bin/activate

# 그 환경에 ipykernel 설치
pip install ipykernel

# 커널 스펙 등록
python -m ipykernel install --user --name myproject --display-name "Python (myproject)"
```

이제 JupyterLab 메뉴에 "Python (myproject)"가 뜨고, 그 노트북은 자동으로 해당 가상환경에서 돈다. 프로젝트마다 의존성이 다른 데이터 작업에서 거의 필수에 가까운 패턴이다. 부록 A에서 환경 분리와 같이 다룬다.

## 정리 — 어디서든 통하는 멘탈 모델

해부학을 한 바퀴 돌았다. 정리해두자.

- 노트북은 **두 프로세스**다. 프런트엔드는 UI, 커널은 실행 엔진.
- 둘은 **ZeroMQ 메시지**로 통신한다. 본문은 Shell(요청)과 IOPub(결과 브로드캐스트) 두 채널만 기억하면 된다.
- 한 노트북은 **한 커널**과 묶인다. 커널을 갈면 언어가 바뀐다. R, Julia, Scala 모두 가능.
- 커널이 들고 있는 **메모리 상태**가 노트북의 편의이자 가장 큰 그림자다. 보이는 코드와 메모리는 따로 움직인다.

이 멘탈 모델이 잡혀 있으면 4장의 JupyterLab이든, 5장의 Colab이든, 6장의 VS Code Notebooks든, 새 환경에서 30분 안에 적응할 수 있다. UI는 다 다르지만 그 아래 깔린 두 프로세스 구조는 같다. Marimo만 예외인데, 그 예외가 왜 의미 있는지는 7장에서 본다 — Marimo는 의도적으로 이 구조 위에 reactive runtime을 한 층 더 올려 hidden state를 원천 차단한다.

다음 장에서는 어느 노트북 환경에 가도 바로 써먹을 수 있는 공용 무기, **매직 커맨드**를 살펴보자. 환경마다 UI는 달라도 매직은 거의 다 통한다. `%timeit`, `%%writefile`, `%load_ext autoreload` 같은 한 줄짜리 명령이 노트북 작업의 속도를 한 단계 끌어올린다. 같이 보자.

## Box A — 더 깊이: Jupyter 메시징 프로토콜

호기심 많은 독자를 위한 보너스다. 본문 진행과 무관하니 건너뛰어도 좋다. 다만 노트북이 멈췄을 때 디버깅하거나, 자체 커널을 만들거나, JupyterHub를 운영하게 된다면 한 번쯤 들여다볼 가치가 있다.

### 다섯 채널의 역할

본문에서 Shell과 IOPub 두 채널만 다뤘다. 실제로는 다섯 개의 ZeroMQ 소켓이 동시에 돈다.

| 채널 | 소켓 타입 | 방향 | 용도 |
|------|----------|------|------|
| **Shell** | ROUTER/DEALER | 프런트 → 커널, 응답 역방향 | 코드 실행 요청, 변수 인트로스펙션, 자동완성 |
| **IOPub** | PUB/SUB | 커널 → 모든 프런트 | stdout, 표현식 결과, 디스플레이 데이터, 에러 트레이스 |
| **Stdin** | ROUTER/DEALER | 커널 → 프런트, 응답 역방향 | `input()` 같은 사용자 입력 요청 |
| **Control** | ROUTER/DEALER | 프런트 → 커널 | 인터럽트, 셧다운, 디버그 명령 (Shell이 막혀도 따로 도달) |
| **Heartbeat** | REQ/REP | 양방향 | 커널 생존 확인 핑/퐁 |

Shell과 Control이 분리된 이유를 다시 짚어보자. Shell이 긴 작업을 처리하느라 큐가 막혀 있을 때, 사용자가 인터럽트를 보내면 그 메시지도 같이 큐 뒤에 줄을 서버린다. 그러면 영원히 멈출 수 없다. 그래서 인터럽트는 Control이라는 별도 소켓으로 우선 전달되도록 분리해뒀다. 작지만 결정적인 설계다.

### 메시지 구조 한 예시

모든 메시지는 JSON dict-of-dicts다. 다섯 부분으로 구성된다.

```json
{
  "header": {
    "msg_id": "uuid-abc-123",
    "session": "sess-uuid-789",
    "username": "toby",
    "date": "2026-05-06T12:34:56.789012Z",
    "msg_type": "execute_request",
    "version": "5.3"
  },
  "parent_header": { ... },
  "metadata": {},
  "content": {
    "code": "x = 10\nprint(x)",
    "silent": false,
    "store_history": true,
    "user_expressions": {},
    "allow_stdin": true,
    "stop_on_error": true
  },
  "buffers": []
}
```

- `header`는 메시지 ID, 세션 ID, 메시지 타입, 프로토콜 버전을 담는다. 모든 응답은 `parent_header`에 원래 요청의 헤더를 복사해 둬서 어느 요청에 대한 답인지 추적할 수 있다.
- `content`는 메시지 타입에 따라 모양이 다르다. `execute_request`라면 실행할 코드와 옵션이 들어간다. `display_data`라면 MIME 타입별 출력 데이터가 들어간다.
- `buffers`는 큰 바이너리 데이터(이미지 등)를 별도로 보낼 때 쓴다.

응답으로 돌아오는 `execute_reply`는 이렇게 생겼다.

```json
{
  "header": { "msg_type": "execute_reply", ... },
  "parent_header": { "msg_id": "uuid-abc-123", ... },
  "content": {
    "status": "ok",
    "execution_count": 3,
    "user_expressions": {}
  }
}
```

`execution_count`가 셀 옆의 `In [3]` 번호다. `status`가 `error`면 트레이스백이 같이 IOPub로 흐른다.

### 메시지 흐름의 시각화

`Shift+Enter` 한 번을 눌렀을 때 메시지가 어떻게 흐르는지 시퀀스 다이어그램으로 정리하자.

```
프런트엔드          Server          커널
    │                │              │
    │ execute_request│              │
    │───────────────▶│              │
    │                │ Shell        │
    │                │─────────────▶│
    │                │              │ (실행 시작)
    │                │              │
    │                │  status:busy │
    │                │◀─────────────│
    │  status:busy   │              │
    │◀───────────────│              │
    │                │              │
    │                │              │ print("hi")
    │                │  stream:hi   │
    │                │◀─────────────│
    │  stream:hi     │              │
    │◀───────────────│              │ ← IOPub
    │                │              │
    │                │              │ (마지막 표현식)
    │                │ execute_     │
    │                │ result:42    │
    │                │◀─────────────│
    │  execute_      │              │
    │  result:42     │              │
    │◀───────────────│              │
    │                │              │
    │                │  status:idle │
    │                │◀─────────────│
    │  status:idle   │              │
    │◀───────────────│              │
    │                │              │
    │                │ execute_reply│
    │                │◀─────────────│
    │ execute_reply  │              │
    │◀───────────────│              │
    │                │              │
   (셀 [In *] → [In 3])
```

요청 하나에 대해 응답이 여러 번 흐른다. 시작/종료 알림(status), 출력(stream/execute_result), 그리고 마지막에 응답 완료(execute_reply). 프런트엔드는 이 모두를 받아 화면을 갱신한다.

이걸 알아두면 무엇이 좋은가. 가끔 노트북이 "셀이 끝났는데 출력이 안 뜬다" 같은 이상한 상태에 빠질 때, 어느 채널이 막혔는지 추측할 수 있다. status는 왔는데 stream이 안 왔다 → IOPub 어딘가가 끊겼다. execute_reply는 왔는데 status가 안 갱신됐다 → Shell은 멀쩡한데 IOPub의 status 메시지를 놓쳤다. 디버깅의 단서가 된다.

### 직접 메시지를 보내보고 싶다면

`jupyter_client` 라이브러리로 자체 클라이언트를 만들 수 있다.

```python
from jupyter_client import KernelManager

km = KernelManager(kernel_name="python3")
km.start_kernel()
kc = km.client()
kc.start_channels()

msg_id = kc.execute("x = 42; print(x)")
while True:
    msg = kc.get_iopub_msg(timeout=1)
    print(msg["msg_type"], msg["content"])
    if msg["msg_type"] == "status" and msg["content"]["execution_state"] == "idle":
        break

kc.stop_channels()
km.shutdown_kernel()
```

이걸 응용하면 Papermill 같은 노트북 자동 실행 도구를 자기 손으로 만들 수 있다. 실제로 Papermill이 내부에서 이런 패턴으로 노트북을 셀 하나씩 실행한다. 10장에서 Papermill을 다룰 때 다시 만나자.

자세한 스펙은 공식 문서 [jupyter_client messaging](https://jupyter-client.readthedocs.io/en/stable/messaging.html)에 있다.
