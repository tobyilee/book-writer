## Deepnote — Google Docs식 노트북

Deepnote의 첫인상은 단순하다. **"Google Docs로 노트북을 만든다면 이렇게 생겼겠지."** 클라우드 호스팅, 링크 공유, 실시간 다중 편집, 셀 단위 코멘트. 한 노트북에 여러 사람이 동시에 들어와 서로의 커서가 보이는 채로 작업한다. 누가 어느 셀을 수정 중인지 실시간으로 표시된다.

이게 왜 중요한가. 페어 프로그래밍을 노트북에서 해본 사람은 안다. 한 사람이 화면을 공유하면 다른 사람은 보기만 한다. Slack으로 "셀 5의 그 줄 좀 봐봐"를 주고받는다. 어색하다. Deepnote에선 두 사람이 같은 노트북을 띄우고 같이 셀을 만지고 같이 결과를 본다. 옆에 앉아 있는 것처럼.

```python
# Deepnote 노트북의 한 셀 — 누가 수정 중인지 옆에 표시됨
import pandas as pd
df = pd.read_csv("/work/sales.csv")     # ← 동훈이 편집 중
df.head()
```

기술적으로 Deepnote는 **Jupyter 호환을 강하게 강조**한다. `.ipynb` 파일을 그대로 import할 수 있고, 매직 명령어와 ipywidgets도 거의 동작한다. Jupyter에서 쌓아온 자산을 그대로 들고 갈 수 있다는 점이 큰 진입 장벽 완화다.

또 하나의 셀링 포인트는 **링크 공유의 단순함**이다. 노트북 우상단의 "Share" 버튼을 누르고 링크를 복사하면, 그 링크를 받은 사람은 별도 가입·환경 설정 없이 노트북을 바로 본다. (편집 권한이 있으면 같이 편집까지.) 이메일·Slack·문서 어디든 노트북 링크 하나로 공유가 끝난다.

셀 단위 코멘트도 인상적이다. 셀 우측에 댓글 버튼을 누르면 그 셀에 의견을 남길 수 있다. 코드 리뷰의 경험이 노트북에 들어왔다고 봐도 된다. PR 없이도 인라인으로 의견을 주고받을 수 있다.

```
[셀 12: 모델 학습 결과]
─────────────────────────────────────────
accuracy: 0.87
precision: 0.82
recall: 0.91
─────────────────────────────────────────
💬 동훈: precision이 낮은 게 마음에 걸리는데, threshold를 조정해보면?
   ↳ 민지: 좋은 지적. 0.5에서 0.6으로 올려서 다시 돌려봤어 (셀 14)
```

Deepnote의 한계도 짚어두자. **클라우드 종속**이 가장 크다. 노트북이 Deepnote 서버에 산다. 오프라인 작업이 안 되고, 사내 보안 정책상 외부 SaaS에 데이터를 못 올리는 환경이라면 처음부터 후보가 아니다. 가격은 무료 플랜이 있지만 팀 규모·기능에 따라 유료가 빠르게 붙는다. (가격은 빠르게 변하니 본 책의 가격표는 **2026년 5월 기준**이며 결제 전에는 공식 페이지에서 다시 확인하자.)

그래도 한 마디로 정리하자면 이렇다. **"Jupyter의 자산을 그대로 들고, 팀이 같이 쓸 수 있게 만든 도구."** 가벼운 협업·셀 코멘트가 본질적 필요라면 Deepnote가 가장 자연스러운 선택이다.

Deepnote의 또 한 가지 작은 매력은 **AI 어시스턴트** 가 노트북 옆에 박혀 있다는 점이다. 셀에 자연어로 질문을 던지면 (Colab Data Science Agent와 비슷한 느낌으로) 코드를 제안해준다. 5장에서 본 패턴과 같은 결의 도구다. **노트북 + 협업 + AI** 가 한 환경에 묶여 있다는 게 SaaS 노트북의 표준 패키지로 자리 잡았다고 봐도 된다.

또한 Deepnote는 **데이터 소스 커넥터** 가 풍부하다. PostgreSQL·BigQuery·Snowflake·S3 같은 데이터 소스를 노트북 환경에 등록해두면, 셀에서 SQL을 그냥 쳐서 결과를 바로 받는다. 분석가가 데이터에 접근하기 위해 매번 인증·연결 코드를 짜는 마찰이 줄어든다. 팀 단위로 데이터 소스를 공유 자원으로 등록해두면, 새 팀원도 그 자원을 그대로 쓰면 된다.

## Hex — 노트북에서 출발한 데이터 앱

Hex는 한 단계 다른 야심을 가진 도구다. **"노트북은 분석가의 작업 도구이자, 비기술 이해관계자가 보는 리포트이자, 인터랙티브 데이터 앱이다."** 한 산출물에 세 역할이 다 들어 있다.

Hex의 노트북은 익숙한 셀 인터페이스로 시작한다. Python 셀, SQL 셀, 마크다운 셀이 있다. 여기까지는 다른 도구와 비슷하다. 차이는 두 가지에서 시작한다.

**첫째, no-code 스텝.** Python을 못 쓰는 동료를 위해 데이터 변환·필터링·집계를 GUI로 끌어다 쓸 수 있는 셀이 있다. SQL을 못 짜는 PM이 데이터 필터를 추가하는 일도 가능하다. 코드와 GUI 셀이 같은 노트북 안에 섞여 있다.

**둘째, 앱·스토리 모드.** 노트북 작업을 끝내고 "App" 탭으로 전환하면, 노트북이 깔끔한 인터랙티브 대시보드로 변환된다. 코드는 숨겨지고, 사용자가 만든 위젯·차트만 노출된다. 임원에게 링크 하나로 공유하면, 임원은 노트북의 raw 코드를 보지 않고도 인사이트를 인터랙티브하게 탐색한다.

```
[Hex 노트북 — 분석가 뷰]
┌─────────────────────────────────┐
│ 셀 1: SQL 쿼리                  │
│ SELECT * FROM sales WHERE ...   │
│                                 │
│ 셀 2: Python 변환               │
│ df.groupby('region').sum()      │
│                                 │
│ 셀 3: Plotly 차트               │
│ px.bar(...)                     │
└─────────────────────────────────┘
              ↓ "Publish"
[Hex 앱 — 임원 뷰]
┌─────────────────────────────────┐
│ [필터: 지역 ▼ KR]               │
│ [날짜 범위: 2026-01 ~ 2026-05]  │
│                                 │
│ 📊 [매출 차트]                   │
│ 📊 [지역별 비교]                 │
└─────────────────────────────────┘
```

이게 진가를 발휘하는 순간은 분석가가 임원에게 결과를 설득해야 할 때다. 옛날 같으면 노트북을 PDF로 출력해서 슬랙으로 보냈다. 임원이 "이 숫자, 12월까지 본 건가? 11월까지인가?"를 묻고, 분석가가 다시 노트북을 열어 필터를 바꾸고 다시 PDF를 만든다. Hex에선 임원이 직접 필터를 움직이고 답을 찾는다. **분석가의 부담이 줄고, 의사결정의 속도가 올라간다.**

Hex의 또 한 가지 강점은 **브랜칭·리뷰 워크플로우**다. 노트북마다 git처럼 브랜치를 만들 수 있고, 변경 사항을 PR 형태로 리뷰할 수 있다. 데이터팀이 분석 노트북에 코드 리뷰 문화를 도입하기 좋다.

한계도 분명하다. **Jupyter 호환이 약하다.** Hex는 자체 포맷을 쓰기 때문에 기존 `.ipynb` 자산을 가져오는 게 매끄럽지 않다. 학습 곡선도 다른 도구보다 가파르다 — 노트북 + SQL + no-code + 앱 모드를 다 익혀야 진짜 가치가 보인다. 그리고 가격은 유료 중심이다. 무료 플랜은 한정적이다.

한 마디 정리: **"노트북에서 출발해 인터랙티브 리포트와 데이터 앱까지 한 도구로 끝내고 싶을 때."** 비기술 이해관계자가 분석 결과를 직접 만져야 하는 조직에 잘 맞는다.

Hex의 셀 모델을 한 단계 더 들여다보자. 같은 노트북 안에 다음과 같은 종류의 셀이 섞일 수 있다.

- **SQL 셀** — 데이터 소스에 직접 쿼리. 결과는 DataFrame으로 자동 노출.
- **Python 셀** — 일반 노트북 셀과 동일.
- **No-code 변환 셀** — GUI로 필터·집계·조인.
- **Chart 셀** — GUI로 차트 구성. 코드 없이 시각화.
- **Input 셀** — 슬라이더·드롭다운 같은 사용자 입력 위젯.
- **Markdown 셀** — 서사 작성.

이 셀들이 reactive한 의존 그래프로 묶인다. 7장에서 본 Marimo의 reactive 모델과 같은 결이다. **Hex는 노트북에 reactive·SQL·no-code·앱 모드를 한 번에 얹은 SaaS** 라고 보면 정확하다.

이 풍부함은 양날의 검이다. 잘 쓰면 분석가 한 명이 데이터부터 임원 보고까지 한 도구로 끝낼 수 있다. 잘못 쓰면 셀 종류가 너무 많아 노트북이 어수선해지고, 새 팀원의 학습 곡선이 부담스러워진다. **명확한 컨벤션을 팀이 합의하고 시작하는 편이 낫다.**

## Databricks — 빅데이터 엔진 위의 노트북

Databricks는 정체성이 가장 다르다. Deepnote와 Hex가 "노트북을 협업·리포트로 확장"이라면, Databricks는 **"엔터프라이즈 데이터 플랫폼의 프런트엔드가 노트북"** 이다. 노트북 자체보다 그 아래의 인프라가 본질이다.

Databricks의 핵심은 **Spark 클러스터**다. 노트북 셀에 PySpark 코드를 치면, 그 코드가 분산 클러스터에서 실행된다. 한 노트북으로 TB급 데이터를 다룬다.

```python
# Databricks 노트북의 한 셀 — Spark가 클러스터에서 분산 실행
df = spark.read.parquet("s3://my-bucket/sales/year=2026/")
df.groupBy("region").agg({"amount": "sum"}).show()
```

이 한 셀이 클러스터의 수십 노드에서 병렬로 도는 동안, 노트북 UI는 진행 상황을 실시간으로 보여준다. **노트북에서 빅데이터 작업을 한다**는 게 Databricks의 정체성이다.

여기에 **다언어 매직** 이 깔린다. 같은 노트북 안에서 셀마다 언어를 바꿀 수 있다.

```python
# 셀 1 (Python 기본)
df = spark.read.parquet("s3://...")

# 셀 2 (SQL)
%sql
SELECT region, SUM(amount) FROM sales GROUP BY region

# 셀 3 (Scala)
%scala
val agg = spark.sql("SELECT * FROM sales").groupBy("region").count()

# 셀 4 (Markdown)
%md
## 분석 결과 요약
```

3장에서 우리가 `%timeit`·`%matplotlib inline` 같은 IPython 매직을 살펴봤는데, **Databricks의 다언어 매직은 그 매직 시스템이 다른 언어로 확장된 형태**다. 같은 발상이 더 큰 스케일에서 동작한다고 보면 된다. SQL 분석가, Python 데이터 사이언티스트, Scala 엔지니어가 한 노트북에서 각자의 언어로 같은 데이터 위에 작업할 수 있다.

또 한 가지 Databricks의 셀링 포인트는 **MLflow와 Delta Lake의 통합**이다.

- **MLflow:** 실험 추적·모델 버저닝·배포의 표준 도구. Databricks 노트북에서 학습한 모델은 자동으로 MLflow에 기록된다. 하이퍼파라미터·메트릭·아티팩트가 한 곳에 쌓인다.
- **Delta Lake:** ACID 트랜잭션·시간 여행·스키마 진화를 지원하는 데이터 레이크 포맷. 노트북에서 Delta 테이블을 직접 쿼리하고 업데이트할 수 있다.

이 둘이 노트북과 한 몸이 되면 ML 파이프라인 전체가 노트북 안에서 흘러간다. 데이터 적재(Delta) → 전처리 → 학습(Spark MLlib + MLflow) → 평가 → 배포까지. 다른 도구로 갈아타지 않고.

협업 측면에서도 **자동 버저닝·동시 편집·셀 코멘트**가 들어 있다. Deepnote에 비할 수준은 아니지만, 엔터프라이즈 환경에선 충분하다.

> ⚠️ **비용 함정 — Databricks 클러스터를 켜둔 채 잊으면**
>
> Databricks의 가장 무서운 함정은 가격 모델이다. 노트북 자체는 무료에 가깝지만, **클러스터 컴퓨트 사용량은 시간당 종량제**다. 분석가가 노트북에서 작업하다가 점심을 먹으러 가고, 회의에 들어가고, 퇴근하면서 클러스터를 끄지 않으면 — 그 클러스터는 밤새 idle 상태로 비용을 청구한다. 큰 클러스터(고성능 GPU 노드 등)면 한 달에 수천 달러가 그냥 새어나간다.
>
> 이걸 막는 표준 패턴이 있다. **"Auto-termination"** 설정이다. Databricks는 클러스터 설정에서 "30분간 idle이면 자동 종료" 같은 정책을 걸 수 있다. 모든 클러스터에 이 정책을 기본으로 박아두자. 잊지 말자.
>
> Colab Pro도 비슷한 누적 비용 문제가 있다. $11.99/월이 작아 보이지만 1년이면 $144, 5년이면 $720이다. SaaS 노트북 환경의 누적 비용은 한 번씩 점검할 만한 항목이다. (5장에서도 짚었던 얘기다.)

Databricks의 한계는 **일반 EDA에는 과한 인프라**라는 점이다. 노트북 한 채로 100MB CSV를 분석하려고 Spark 클러스터를 띄우는 건 헬리콥터로 동네 슈퍼에 가는 격이다. **TB급 데이터·Spark 인프라·MLflow가 이미 깔린 조직에 들어가야** Databricks가 의미 있게 살아난다. 그렇지 않은 조직에서는 JupyterLab이나 Colab이 더 적합하다.

조금 더 구체적으로 다듬자면, Databricks는 다음 환경 중 하나에 해당할 때 답이 된다.

- **데이터가 이미 Delta Lake/Parquet으로 데이터 레이크에 적재돼 있다.**
- **분석에서 PySpark·SparkSQL이 표준 도구다.**
- **MLOps 인프라(MLflow·Model Registry·Serving)가 이미 사내에 깔려 있다.**
- **TB 단위 데이터가 일상이다.**

이 조건이 맞으면 Databricks는 모든 조각을 한 손에 쥐여주는 도구다. 반대로 한두 GB의 데이터·로컬 분석·1인 ML 실험이 주된 작업이라면 Databricks를 굳이 도입할 이유가 없다.

또 하나, Databricks의 노트북 인터페이스가 **JupyterLab만큼 깔끔하지는 않다**는 점도 짚어두자. 인프라 통합에 무게가 실려 있다 보니, 셀 편집 경험·확장 생태계는 JupyterLab에 비해 다소 둔하다. 데이터·인프라의 무게를 가져가는 대신 노트북 UX의 일부를 양보한 모양새다. 도구의 트레이드오프는 어디에나 있다.

## 친척 4종 — Polynote, nteract, Zeppelin, Observable

여기까지 다룬 세 도구(Deepnote·Hex·Databricks)와 4~7장의 환경(Jupyter·Colab·Kaggle·VS Code·Marimo)을 합치면 노트북 환경의 큰 줄기는 거의 다 잡았다. 그래도 책 한 권으로서 빈 자리가 있다. 각 진영의 친척들이다. 짧게 박스로 정리해두자.

> 📋 **친척 4종 — 알아두면 좋은 노트북들**
>
> | 도구 | 만든 곳 | 정체성 | 강점 | 약점 |
> |------|--------|--------|------|------|
> | **Polynote** | Netflix | reactive·다언어 노트북 | Scala/Python/SQL/Vega 혼합·셀 간 데이터 공유·불변(immutable) 모델로 재현성 | Git 빌트인 미흡·생태계 작음·Netflix 의존 |
> | **nteract** | nteract 커뮤니티 | Jupyter 노트북의 데스크톱 앱 | 오프라인·가벼움·OS 네이티브 UI | 활발한 개발 둔화·기능이 JupyterLab 대비 단순 |

>
> 이들은 각자 자기 자리가 있다. Netflix가 Spark·Vega 기반 데이터 플랫폼을 갖고 있다면 Polynote가 자연스럽고, 데이터 시각화를 D3로 만드는 사람에겐 Observable이 표준이다. Apache Zeppelin은 Hadoop·Spark 시대의 Apache 생태계에선 여전히 살아 있다. nteract는 오프라인 노트북이 필요한 특수 환경에서 의미가 있다. 다만 책의 본 흐름에서는 박스 한 칸으로 인사를 마치자.

## 환경을 가르는 안목 — 4~8장의 닫는 말

4장에서 시작한 환경 투어가 8장 끝에 와서 한 번 닫힌다. 정리해보자.

- **Jupyter 계열(4장)** — 표준이자 레거시. 모든 길이 이 표준 위에서 갈라진다.
- **클라우드(5장)** — GPU의 대중화. 인프라 없이 ML을 시작하는 길.
- **에디터 통합(6장)** — 노트북이 IDE 안으로. 사고의 흐름을 끊지 않는 작업 환경.
- **reactive(7장)** — 노트북을 다시 짜다. hidden state·git·앱 배포를 한 번에 푼 답.
- **협업·엔터프라이즈(8장)** — 1인 도구를 넘은 노트북. 팀·비기술 이해관계자·빅데이터의 작업대.

이 다섯 줄기를 머릿속에 그려두면, 새 노트북 환경이 등장해도 그게 어느 자리에 들어오는지 짚을 수 있다. **도구는 계속 바뀐다. 자리 짓는 안목이 남는다.** 11장의 닫는 말과 호응하는 흐름인데, 우리는 이 안목을 9장으로 바로 가져가야 한다.

## 협업 노트북, 도입의 마음가짐

8장의 세 도구(Deepnote·Hex·Databricks)와 친척 4종을 다 살펴본 다음, 한 가지 마지막 권유를 두고 가자. **협업 도구는 도구만 갈아 끼운다고 협업이 시작되지 않는다.**

흔한 패턴이 이렇다. 회사가 "협업이 부족하다" 싶어 Hex나 Deepnote를 도입한다. 라이선스를 사고, 데이터 소스를 연결하고, 새 노트북 환경을 세팅한다. 그런데 몇 달 지나도 사람들이 여전히 자기 JupyterLab에서 작업하고 결과만 새 환경에 올린다. 협업 기능은 거의 안 쓰인다.

도구가 잘못된 게 아니다. **협업 문화가 도구보다 먼저 자리 잡혀야 한다.** 셀 코멘트를 남기는 습관, PR처럼 노트북 변경을 리뷰하는 습관, 데이터 소스를 팀 자원으로 다루는 습관. 이 습관이 없는 조직에 도구만 들어오면 도구가 외로워진다.

그래서 도입 순서를 한 번 권하자면 이렇다.

1. **먼저, 한두 사람이 한 노트북을 같이 작업하는 습관**을 만든다. JupyterLab + git으로도 충분하다. 셀 단위 코멘트는 PR 코멘트로 우회.
2. **그 습관이 자리 잡으면**, 협업 SaaS의 기능이 자연스럽게 가치를 보인다. 셀 코멘트가 외부 도구로 흩어지지 않고 한 곳에 모인다는 가치.
3. **그 다음에 도구를 도입**하면 빠르게 뿌리내린다.

도구가 사람의 행동을 바꾸기보다, 사람이 도구의 가치를 발견할 때 도입이 성공한다. 이 책의 큰 흐름과도 통하는 얘기다 — **도구는 계속 바뀐다, 안목과 습관이 남는다.**

## 9장 예고 — 가장 큰 그림자로

8장까지 우리는 환경의 풍요를 봤다. 어디에 가도 노트북이 있다. 자기 작업에 맞는 환경을 고를 수 있다. 그런데 환경이 아무리 풍요로워도, **노트북이라는 매체 자체에 깔린 본질적 그림자**가 있다. 7장 도입에서 한번 들여다본 hidden state 문제, 그리고 그것의 큰 형제인 재현성 문제다.

다음 장에서 그 그림자를 정면으로 다룬다. Pimentel 2019의 충격적 수치(노트북 1.4M개 중 24.11%만 에러 없이 실행, 4.03%만 동일 결과 재현), Joel Grus의 강경 비판("I don't like notebooks"), 그리고 reactive 진영(Marimo) · lineage 진영(NBSafety) · 습관 진영(Restart and Run All)의 세 갈래 답. 이 책의 난이도 곡선에서 가장 가파른 챕터다. 진지한 평어체로, 한 단 호흡을 깊이 가다듬고 들어가보자.
