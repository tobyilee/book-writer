# 7장. Marimo — 노트북을 다시 짜다

6장까지의 흐름을 잠시 돌아보자. 우리는 Jupyter의 표준을 깊이 이해했고(4장), Colab으로 GPU의 벽을 넘었고(5장), VS Code와 Cursor로 노트북을 에디터에 끌어들였다(6장). 같은 `.ipynb` 위에서 환경만 바꿔가며 작업을 이어왔다. 그런데 이 모든 환경은 한 가지 공통의 가정 위에 서 있다. **"셀은 어떤 순서로든 실행될 수 있다."** 그게 노트북의 자유로움이고, 동시에 노트북의 가장 큰 그림자다.

이런 장면이 익숙하지 않은가. 셀 5에서 `df = pd.read_csv("v1.csv")`를 실행한다. 한참 작업하다가 데이터가 바뀌어 셀 5를 `df = pd.read_csv("v2.csv")`로 고치고 다시 실행한다. 그런데 이미 셀 8에서 `df['amount']`를 가공한 뒤 다시 `df`라는 변수에 덮어쓴 적이 있다. 셀 5를 실행한 시점에서 셀 8은 안 돌렸다. 셀 6, 7만 실행했다. 이제 노트북의 메모리 안에는 v2의 raw 데이터가 들어 있는 `df`가 있는데, 보이는 코드만 보면 마치 v2의 가공 데이터가 `df`에 있는 것처럼 보인다.

이 상황이 무서운 이유는, **에러가 안 난다**는 점이다. 코드는 멀쩡히 돌아가고 결과도 그럴듯하게 나온다. 다만 그 결과가 의도한 결과인지 아무도 모른다. Restart Kernel and Run All을 해보면 그제야 차이가 드러난다 — 운이 좋으면 같은 결과가 나오고, 운이 나쁘면 어디선가 KeyError가 터진다. 운이 가장 나쁜 경우는 **다른 결과가 나오는데 에러는 안 나는** 것이다. 이걸 우리는 **hidden state 문제**라 부른다. 9장에서 이 문제를 정면으로 다룰 텐데, 이번 장에서는 그것을 처음부터 다시 짜겠다는 도구를 만나본다.

## 다리 — 왜 새 노트북이 필요한가

위 시나리오에서 본 hidden state 문제는 노트북의 부수적 결함이 아니다. **노트북 패러다임의 본질에서 흘러나오는 결과**다. "셀을 임의 순서로 실행할 수 있다"는 자유와 "메모리 상태가 보이는 코드와 일치해야 한다"는 안전성이 본질적으로 충돌한다. 둘 중 하나를 양보해야 한다.

지금까지의 노트북(Jupyter·Colab·VS Code Notebooks)은 자유 쪽을 골랐다. **사용자가 알아서 Restart and Run All을 하라**는 게 답이었다. 좋은 답이다, 잘 지킬 수만 있다면. 하지만 Pimentel 등(2019)의 연구는 이 가정이 지나치게 낙관적임을 보여준다. GitHub의 Jupyter 노트북 1.4M개를 분석했더니, **24.11%만 에러 없이 실행됐고, 그중 4.03%만 동일 결과를 재현**했다. 학생·연구자·실무자 모두 이 함정에 빠진다. 우리만의 부주의가 아니라 매체 자체의 함정이다.

그렇다면 다른 답은 없을까. **자유 대신 안전을 고르면 어떻게 될까.** 셀 사이의 의존 관계를 도구가 추적해서, 어떤 셀을 바꾸면 그 셀에 의존하는 다른 셀들이 자동으로 다시 실행되도록 하면. 그러면 "보이는 코드 ≠ 실제 메모리 상태"라는 함정 자체가 사라진다. 셀 순서를 사람이 신경 쓸 필요도 없다.

이 발상이 **reactive notebook**이다. 그리고 Python 진영에서 이 발상을 가장 본격적으로 구현한 도구가 **Marimo**다.

> 📌 **친척 그룹 — Marimo가 어디서 왔나**
>
> Marimo가 처음 나온 reactive 노트북은 아니다. 같은 발상의 친척이 다른 언어 진영에 먼저 있었다.
>
> - **Pluto.jl (Julia):** Julia 진영의 reactive 노트북. 2020년 무렵부터 자리 잡았다. Marimo보다 먼저 "셀 의존 그래프 + 자동 재실행"의 사용성을 증명했다.
> - **Observable (JavaScript):** 데이터 시각화 중심의 reactive 노트북. D3 창시자 Mike Bostock이 만들었다. 입력이 바뀌면 차트가 즉시 갱신되는 모델로 인터랙티브 시각화 노트북의 기준점이 됐다.
> - **Polynote (Netflix, Scala/Python/SQL):** 불변 데이터 모델로 재현성을 강조한 reactive 노트북. Netflix 내부에서 자라났다.
>
> Marimo는 이 계보의 **Python 진영의 답**이다. 위 세 도구가 각자 자기 언어 생태계에서 보여준 reactive의 가능성을, Python에 본격적으로 가져온 도구다.

## reactive runtime — 셀 사이의 그래프를 그리다

Marimo의 핵심을 한 문장으로 설명하면 이렇다. **셀 사이의 의존 관계를 정적 분석으로 그래프로 그려두고, 어떤 셀이 바뀌면 그 셀의 다운스트림에 있는 모든 셀을 자동으로 다시 실행한다.**

예제로 잡아보자. 노트북에 다음 세 셀이 있다고 하자.

```python
# 셀 1
import pandas as pd
df = pd.read_csv("sales.csv")

# 셀 2
filtered = df[df['region'] == 'KR']

# 셀 3
total = filtered['amount'].sum()
print(f"한국 매출 합계: {total}")
```

Marimo는 이걸 보면서 다음과 같은 의존 그래프를 그린다.

```
[셀 1: df 정의]
       ↓
[셀 2: filtered = df[...]]   ← df에 의존
       ↓
[셀 3: total = filtered[...]] ← filtered에 의존
       ↓
[출력]
```

이제 셀 1을 `df = pd.read_csv("sales_v2.csv")`로 바꿔본다. **자동으로 셀 2와 셀 3이 다시 실행된다.** 사람이 [Shift+Enter]를 세 번 누를 필요가 없다. 셀 2의 `filtered`가 새 데이터로 업데이트되고, 그에 따라 셀 3의 출력도 즉시 갱신된다.

반대로 셀 2만 바꿔본다고 해보자. `filtered = df[df['region'] == 'JP']`로. 셀 1은 바뀌지 않았으니 다시 실행되지 않는다 (raw 데이터 로드는 비싼 작업이니 다행). 셀 3만 다시 실행되어 일본 매출이 출력된다.

이게 reactive runtime의 본질이다. **셀이 무엇에 의존하고 무엇에 영향을 미치는지를 도구가 안다.** 사람은 코드를 짜는 데만 집중하면 되고, 무엇을 다시 실행해야 하는지는 도구가 알아서 한다.

이 모델을 처음 접하면 "그래서 뭐가 다르지?"라는 반응이 자연스럽다. Jupyter에서도 셀 세 개를 차례로 [Shift+Enter]로 누르면 같은 결과가 나오지 않냐는 것이다. 맞다. **결과는 같다, 모든 셀을 처음부터 끝까지 순서대로 실행할 때만.** 문제는 데이터 작업이 그렇게 직선적으로 흘러가지 않는다는 데 있다. 셀 5번에서 무언가 이상한 게 보여서 셀 2로 돌아가 데이터 로드를 바꾸고, 그 다음 셀 7만 실행해서 결과를 확인하고, 다시 셀 4를 만지고, 셀 9를 새로 추가하고… 이런 비선형적 작업이 노트북의 본질이다. Jupyter는 사람이 매번 "어느 셀부터 어느 셀까지 다시 실행해야 하지?"를 머릿속으로 추적해야 한다. Marimo는 그걸 안 하게 한다.

기술적으로는 Python 코드를 AST(Abstract Syntax Tree)로 파싱해서, 각 셀이 어떤 변수를 정의(write)하고 어떤 변수를 참조(read)하는지를 추출한다. 그래프의 노드는 셀, 엣지는 변수 의존이다. 셀이 바뀌면 영향받는 부분 그래프(subgraph)에 위상정렬을 걸어 순서대로 재실행한다.

## hidden state 원천 차단의 메커니즘

이 reactive 모델은 hidden state 문제를 원천에서 차단한다. 어떻게? 두 가지 강한 제약을 도입해서다.

**첫째, 변수의 단일 정의 강제.** 같은 변수 이름을 두 셀이 모두 정의하면 Marimo는 에러를 띄운다. "이 변수가 어느 셀에서 정의되는지 모호하다"는 이유다. 이 제약이 처음에는 답답하게 느껴질 수 있다. Jupyter에선 자유로웠던 것 — 셀 5에서 `df`를 만들고 셀 8에서 `df = df.dropna()`로 덮어쓰고 셀 12에서 또 다른 무언가로 덮어쓰는 — 가 막힌다. 하지만 이 답답함이 정확히 hidden state를 만드는 동작이다. **변수가 어느 시점에 어떤 모양인지 사람이 추적할 수 없게 만드는 동작.** Marimo는 이걸 처음부터 막는다.

대신 변수명을 분리한다. `df_raw`, `df_clean`, `df_filtered` 식으로. 처음에는 변수 이름 짓는 일이 늘어 귀찮지만, 며칠 지나면 노트북이 훨씬 명료해졌다는 걸 느낀다. **변수명 자체가 데이터의 단계(stage)를 기록하는 역할**을 하기 시작한다.

**둘째, 셀 삭제 시 변수도 삭제.** Jupyter에서는 셀을 지워도 그 셀이 정의한 변수가 메모리에 남아 있다. 이게 hidden state의 또 다른 원천이다. 셀에서 `temp_df`를 만들고 작업한 뒤 셀을 지우면, `temp_df`라는 변수가 화면에서는 사라졌지만 메모리에서는 살아 있다. 다른 셀이 `temp_df`를 참조해도 동작한다. 어디서 정의됐는지는 더 이상 코드에 안 보인다.

Marimo는 셀을 지우면 그 셀이 정의한 변수도 메모리에서 같이 지운다. **보이지 않는 변수는 메모리에도 없다.** 화면의 코드와 메모리 상태가 항상 일치한다.

**셋째, 실행 순서 무관성.** 의존 그래프가 위상정렬을 보장하므로, 사람이 셀을 어느 순서로 작성하든·실행하든 결과가 같다. "Restart and Run All"이라는 말 자체가 의미를 잃는다. 매번 모든 게 일관된 상태로 유지되니까.

이 세 제약은 처음에는 답답하게 느껴진다. Jupyter의 자유로움 — 같은 변수를 셀 여러 곳에서 덮어쓰고, 셀을 지웠다가 다른 셀에서 그 변수를 참조하고, 셀 순서를 무시하고 — 이 막힌다. 하지만 그 자유가 정확히 hidden state를 만든다는 점을 깨달으면, 답답함이 안전감으로 바뀐다. **"셀을 자유롭게 실행할 수 있다는 자유가, 메모리 상태를 추적할 책임이라는 비용으로 돌아왔다."** Marimo는 그 비용을 안 내도 되는 길을 텄다.

Marimo의 또 한 가지 작은 안전장치는 **"비싼 셀 캐싱"** 이다. 데이터를 한 번 불러오는 비싼 셀을 매번 재실행하면 작업이 느려진다. Marimo는 셀이 의존하는 변수가 바뀌지 않으면 결과를 캐싱해두고 재실행을 건너뛴다. (`mo.cache` 데코레이터로 명시적으로 표시할 수도 있다.) reactive 모델이 비효율을 부르지 않도록 한 보완책이다.

## `mo.ui.slider` — reactive와 위젯이 결합한 순간

reactive runtime의 진가는 **인터랙티브 위젯**과 만났을 때 폭발한다. Marimo의 위젯은 reactive 그래프의 노드다. 즉, **위젯의 값이 바뀌면 그에 의존하는 셀들이 즉시 다시 실행**된다.

예시를 보자.

```python
# 셀 1
import marimo as mo

# 셀 2: 슬라이더 위젯
x = mo.ui.slider(1, 10, 0.1, label="배율")
x

# 셀 3: x.value를 참조하면 슬라이더 변경 시 자동 재실행
mo.md(f"현재 배율: **{x.value}**")
```

이 노트북을 띄우면 셀 2에 슬라이더 UI가 나타난다. 마우스로 슬라이더를 끌면, 그 값이 셀 3에 즉시 반영된다. **Run을 다시 누를 필요 없다.** 슬라이더가 reactive 그래프의 입력 노드이고, 셀 3은 그 다운스트림 노드다.

이게 ML 실험에서 어떻게 쓰이는가. 하이퍼파라미터를 슬라이더로 노출해두면, 슬라이더만 움직이면서 모델의 메트릭 변화를 즉각 본다.

```python
# 셀 1
import marimo as mo
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import accuracy_score

# (X_train, X_test, y_train, y_test 준비)

# 셀 2: 정규화 강도 슬라이더
C = mo.ui.slider(0.01, 10, 0.01, value=1.0, label="C (역정규화 강도)")
C

# 셀 3: C가 바뀔 때마다 자동 학습·평가
model = LogisticRegression(C=C.value, max_iter=1000)
model.fit(X_train, y_train)
acc = accuracy_score(y_test, model.predict(X_test))
mo.md(f"정확도: **{acc:.4f}**")
```

슬라이더를 0.01에서 10까지 끌어보면 정확도가 어떻게 변하는지 한눈에 보인다. ipywidgets로도 비슷한 게 가능하지만, Marimo에선 콜백 등록 없이 그냥 `C.value`를 참조만 하면 된다. **위젯 = 그래프 노드**라는 모델이 코드를 매우 단순하게 만든다.

> 📌 **API는 변할 수 있다.** `mo.ui.slider(1, 10, 0.1, label="...")` 형태는 **2026년 5월 기준**의 API다. Marimo는 활발하게 진화 중이라 시그니처가 바뀔 수 있다. 핵심은 표면 API가 아니라 **"위젯이 reactive 그래프의 일급 노드"** 라는 모델이다. 이 모델이 잡혀 있으면 API 변경은 docs를 한 번 다시 보는 것으로 흡수된다.

위젯의 종류도 풍부하다. 슬라이더 외에 드롭다운 (`mo.ui.dropdown`), 체크박스 (`mo.ui.checkbox`), 텍스트 입력 (`mo.ui.text`), 파일 업로드 (`mo.ui.file`), 차트 (`mo.ui.altair_chart`) 등이 있다. 위젯 여러 개를 모아 하나의 폼처럼 쓸 수도 있다.

```python
form = mo.ui.batch(
    mo.md("### 모델 설정"),
    {
        "C": mo.ui.slider(0.01, 10, 0.01, value=1.0, label="C"),
        "max_iter": mo.ui.number(100, 5000, 100, value=1000, label="max_iter"),
        "solver": mo.ui.dropdown(["lbfgs", "liblinear", "saga"], value="lbfgs", label="solver"),
    }
)
form
```

이 폼은 사용자에게 깔끔한 UI로 보이고, 다운스트림 셀에서 `form.value['C']`, `form.value['solver']`로 값을 받아 쓴다. **노트북이 곧 인터랙티브 폼**이 된다. ipywidgets로도 비슷한 게 가능하지만, Marimo는 콜백 등록과 `Output` 객체 관리 같은 부수 작업을 다 빼고 값 참조만으로 모든 게 reactive하게 동작한다.

## `.py`로 저장 — git이 정상이 된다

Jupyter의 `.ipynb`는 JSON 직렬화 포맷이라 git diff가 거의 읽을 수 없다는 얘기는 4장에서도, 6장에서도, 그리고 10장에서도 다룬다. 셀 출력·메타데이터·실행 카운트가 한 파일에 다 박혀 있다 보니 코드 한 줄을 바꿔도 디프가 수십 줄로 부풀어 오른다. 이게 노트북 협업의 가장 큰 마찰이다.

Marimo는 처음부터 다른 길을 골랐다. **노트북을 `.py` 파일로 저장한다.** 순수 Python 스크립트다. JSON 메타데이터도, 출력 직렬화도 없다.

`my_notebook.py`를 열어보면 이런 모습이다.

```python
import marimo

__generated_with = "0.x.y"
app = marimo.App()

@app.cell
def __():
    import marimo as mo
    return (mo,)

@app.cell
def __(mo):
    x = mo.ui.slider(1, 10, 0.1, label="배율")
    x
    return (x,)

@app.cell
def __(mo, x):
    mo.md(f"현재 배율: **{x.value}**")
    return ()

if __name__ == "__main__":
    app.run()
```

각 셀이 `@app.cell`로 데코레이트된 함수로 표현된다. 함수의 인자는 그 셀이 의존하는 변수들 (`mo`, `x`)이고, 반환값은 그 셀이 정의하는 변수들이다. 즉, **의존 그래프가 코드 자체에 명시적으로 박혀 있다.** 함수형 프로그래밍의 데이터 흐름과 비슷하다.

이게 git에서 무엇을 의미하는가. **코드 디프가 평범한 Python 디프다.** `git diff my_notebook.py`를 치면 셀 안의 코드 변경이 명료하게 보인다. PR 리뷰가 일반 Python 코드 PR과 동일하게 동작한다. nbdime이나 Jupytext 같은 추가 도구 없이도 협업이 정상화된다.

그리고 한 가지 더. **`python my_notebook.py`로 그냥 스크립트 실행이 가능하다.** 노트북 UI가 없어도, 위에서부터 차례로 모든 셀이 실행된다. CI에서 노트북 검증 (실행 가능한가 / 결과가 일관되나)을 노트북 변환 도구 없이 그냥 `python` 한 줄로 처리할 수 있다. 이게 의외로 큰 가치다. **노트북이 곧 스크립트라면, 자동화·테스트·배포가 모두 표준 Python 도구로 처리된다.**

`pytest`도 그냥 동작한다. 노트북 안의 함수를 임포트해 단위 테스트를 돌릴 수 있다. Jupyter `.ipynb`에선 함수만 추출하려고 nbconvert로 `.py`를 만들거나, `import_ipynb` 같은 별도 도구를 동원해야 했다. Marimo에서는 그런 우회가 필요 없다. **노트북이 곧 모듈이다.**

```python
# tests/test_pipeline.py
from my_notebook import preprocess, train

def test_preprocess_drops_nulls():
    df_clean = preprocess(some_df_with_nulls)
    assert df_clean.isna().sum().sum() == 0
```

이 한 가지 변화가 노트북 작업의 엔지니어링 위생을 크게 끌어올린다. 노트북에서 자란 코드가 자연스럽게 테스트 가능한 모듈이 된다.

## `marimo run` — 노트북이 곧 앱

`.py` 저장의 또 한 가지 결과는, 노트북 자체가 **즉시 웹 앱이 될 수 있다**는 점이다.

```bash
marimo edit my_notebook.py    # 편집 모드
marimo run my_notebook.py     # 앱 모드
```

`marimo run`을 치면 노트북이 웹 페이지로 뜬다. 코드 셀은 숨겨지고, 위젯과 출력만 노출된다. 사용자는 슬라이더를 움직이고, 결과를 본다. **노트북을 Voila·Streamlit 같은 별도 프레임워크로 변환할 필요가 없다.** 같은 파일이 편집 환경이자 실행 환경이고, 동시에 배포 결과물이다.

이 점이 ML 데모를 만들 때 빛을 본다. Jupyter에서는 모델을 학습한 뒤, 데모 페이지를 만들려면 Streamlit으로 코드를 다시 짜거나 Voila로 변환해야 했다. Marimo에선 노트북 자체가 데모다. `marimo run`으로 띄워서 링크로 공유하면 끝이다.

대시보드·내부 도구도 같은 방식으로 만들 수 있다. SQL 쿼리를 실행하고, 결과를 차트로 보여주고, 필터 위젯을 노출하는 — 그런 사내 대시보드를 노트북 한 채로 만들 수 있다. 그리고 그 노트북은 `git diff`가 멀쩡한 진짜 코드다. 10장에서 다룰 배포 도구 비교(Voila / Panel / Streamlit / Marimo) 표에서 Marimo가 차지하는 자리가 바로 여기다.

배포 환경은 일반 Python 서버 환경에 가깝다. `marimo run --port 8080 my_notebook.py`로 띄우면 그 포트에서 웹 앱이 돈다. Docker 이미지로 감싸 ECS·Cloud Run 같은 곳에 올리면 그대로 사내 도구가 된다. 인증·권한이 필요하면 일반 웹 서버처럼 reverse proxy 뒤에 두면 된다. **노트북 환경의 이질성**이 줄어들고, **일반 Python 웹 서비스처럼 다룰 수 있게 된다.**

## SQL 셀 — Python 안의 SQL을 reactive하게

Marimo가 한 번 더 흥미로운 길을 보여주는 곳이 **SQL 셀**이다. 데이터 작업의 큰 비중이 SQL인데, Python 노트북에서 SQL을 쓰려면 보통 다음과 같은 보일러플레이트가 따른다.

```python
query = """
SELECT region, SUM(amount)
FROM sales
WHERE date >= '2026-01-01'
GROUP BY region
"""
df = pd.read_sql(query, con)
df
```

문자열 안의 SQL이 IDE의 SQL 자동완성·문법 체크 도움을 받기 어렵다. Marimo는 **SQL 셀**을 일급으로 지원한다 — 셀 자체를 SQL로 선언하면, 셀 내용은 SQL 문법으로 다뤄지고, 결과 DataFrame은 Python 변수로 자동 노출된다.

```sql
-- SQL 셀
SELECT region, SUM(amount) AS total
FROM sales
WHERE date >= '2026-01-01'
GROUP BY region
```

이 셀의 결과가 자동으로 `_df` 같은 변수에 잡히고, 다음 Python 셀에서 그 변수를 받아 시각화한다. 그리고 reactive 그래프의 일원이라, **위 SQL 셀의 쿼리가 바뀌면 다운스트림 시각화 셀이 자동으로 다시 실행된다.** SQL과 Python을 매끄럽게 오가는 작업 흐름이 자연스럽게 만들어진다.

이 패턴은 8장에서 다룰 Hex의 SQL 셀과 정신적으로 닮아 있다. 협업·SaaS 환경(Hex)이든 로컬 reactive 환경(Marimo)이든, **SQL과 Python을 한 노트북에 섞는 흐름**이 노트북의 다음 단계로 자리잡고 있다.

## WASM — 브라우저에서 설치 없이

Marimo는 한 발 더 나아간다. **Pyodide(WASM 기반 Python)** 위에서 동작하는 빌드를 제공한다. 즉, **서버 없이 브라우저에서 Python 코드가 실행되는 노트북**이다.

`marimo.app`이라는 호스팅 사이트에 접속해보면 알 수 있다. 노트북을 작성하면 그게 그대로 브라우저 안의 Pyodide에서 돌아간다. 서버에 코드가 안 보내진다. 사용자의 브라우저가 Python 인터프리터를 통째로 들고 있다.

이게 무엇을 가능하게 하나. **"링크 누르면 즉시 실행되는 노트북"** 이 BinderHub처럼 컨테이너를 띄울 필요 없이, 정적 호스팅(GitHub Pages, Vercel 등)으로 가능하다. 강의 자료, 튜토리얼, 인터랙티브 블로그 포스트의 배포 비용이 거의 0이 된다. 데이터 사이즈와 라이브러리 호환성에 한계는 있지만 (Pyodide가 모든 Python 패키지를 지원하는 건 아니다), 입문 자료·시각화 데모·작은 분석 노트북에는 충분하다.

이 점도 노트북의 가능성을 한 차원 넓힌다. **노트북이 정적 사이트의 한 컴포넌트가 될 수 있다는 발상.** 책 한 권을 만들 때, 인터랙티브 노트북 페이지를 책의 한 챕터로 끼워 넣는 일이 가능해진다는 의미다.

## 두 관점 — Marimo는 차세대 표준이 될 것인가

Marimo가 좋다는 얘기를 한참 했으니, 이쯤에서 균형을 잡아보자. 노트북 커뮤니티에서 Marimo를 두고 두 관점이 팽팽하게 부딪힌다.

**관점 A — Marimo가 차세대 표준이 될 것이다.**
- hidden state 문제를 원천 차단한다. 이건 Jupyter가 풀지 못한 가장 큰 그림자다.
- `.py` 저장으로 git 협업이 정상화된다. 추가 도구 (Jupytext, nbdime) 없이도.
- `marimo run`으로 앱 배포까지 한 도구로 끝난다. Streamlit·Voila를 따로 배울 필요가 없다.
- WASM 빌드로 브라우저 배포까지 가능하다. BinderHub의 인프라 부담을 우회한다.
- 새 프로젝트를 시작한다면, 이 모든 장점을 누리는 게 합리적이다.

**관점 B — Jupyter 생태계가 너무 깊어 대체가 어렵다.**
- 십수 년간 쌓인 Jupyter 노트북 자료(강의·논문·O'Reilly 책 부록·캐글 노트북)가 압도적이다. 이 자산이 Marimo로 자동 변환되지 않는다.
- ipywidgets·JupyterLab 확장·매직 명령어 생태계가 Marimo와 호환 안 된다.
- 학습 자료가 Jupyter 중심이다. 처음 배우는 사람에게 Marimo부터 권하기는 부담스럽다.
- 협업 환경(Colab·Kaggle·Deepnote·Hex·Databricks)이 모두 Jupyter 호환에 맞춰져 있다. Marimo는 단일 사용자 도구다.

둘 다 맞다. 어느 쪽이 결국 이길지는 시간이 답할 것이다. 다만 한 가지는 분명하다. **reactive 패러다임 자체는 노트북의 다음 단계로 자리 잡았다.** Marimo·Pluto·Polynote·Observable 모두 같은 방향을 가리키고 있다. Jupyter가 끝까지 이긴다 해도, 그 Jupyter는 reactive 요소를 흡수한 형태일 가능성이 높다. (실제로 Jupyter 진영에서도 reactive 확장 시도가 여러 번 있었다.)

## 언제 Marimo를 고를 것인가

이쯤에서 결정 가이드를 하나 잡아두자. **언제 Marimo를 고르는 게 합리적인가.**

| 상황 | Marimo가 좋은가 | 이유 |
|------|----------------|------|
| 새 프로젝트, 의존 자산 없음 | 매우 좋음 | reactive·git·앱 배포가 한 번에 |
| 재현성이 최우선 (생산 데이터 분석) | 매우 좋음 | hidden state 원천 차단 |
| 노트북을 그대로 사내 대시보드로 배포 | 매우 좋음 | `marimo run` 한 줄 |
| 인터랙티브 위젯 중심 EDA | 좋음 | 위젯이 reactive 그래프 노드 |
| 강의·튜토리얼 (브라우저 즉시 실행) | 좋음 | WASM 빌드 |
| 기존 Jupyter 노트북 자산이 많음 | 신중히 | 자동 변환 안 됨, 수동 마이그레이션 필요 |
| Colab·Kaggle 환경 의존 | 부적합 | Marimo는 단일 사용자 로컬 도구 |
| 팀 실시간 협업 필요 | 부적합 | 협업 기능 없음, git 기반 |
| 다언어(R·Julia·SQL 매직) | 부적합 | Python 중심 |

이 표는 절대 기준이 아니다. **새 프로젝트 + 1인 + 재현성 우선 + 앱 배포까지 한 도구로** — 이 조합이라면 Marimo가 거의 무조건 답이다. 거기서 멀어질수록 다른 환경(Jupyter 계열·VS Code·Colab·Hex·Databricks)이 더 잘 맞는다.

한 가지 권하고 싶은 건, **Marimo를 한 번은 직접 써보자**는 것이다. 작은 EDA 노트북 하나라도 좋다. 슬라이더를 만들어 의존 셀이 즉시 갱신되는 걸 보고, 변수를 두 셀에서 정의하려 했다가 막히는 걸 경험하고, `git diff my_notebook.py`로 코드 변경이 깔끔히 보이는 걸 보면, **노트북에 대한 멘탈 모델이 한 단 올라간다.** 그 다음에는 Jupyter로 돌아가도, 무엇이 reactive로 풀릴 수 있는 문제고 무엇이 본질적으로 셀 단위 임의 실행의 결과인지 분간이 된다.

설치도 간단하다. 한 줄이면 된다.

```bash
pip install marimo
marimo edit my_notebook.py    # 새 노트북 편집
```

이 한 줄로 브라우저에 Marimo 편집기가 뜬다. 첫 화면이 Jupyter와 비슷해 보이지만, 셀을 만지다 보면 reactive의 결을 빠르게 느낀다. 변수 이름 충돌·셀 삭제·슬라이더 동작 — 작은 셀 몇 개로 충분히 체감할 수 있다. 30분만 투자하면 된다.

## Jupyter에서 Marimo로 — 마이그레이션의 현실

Marimo의 매력에 끌려 기존 Jupyter 자산을 옮기려 할 때 부딪히는 현실이 있다. **자동 변환은 아직 매끄럽지 않다.**

`marimo convert my_notebook.ipynb`이라는 변환 명령이 있긴 하다. `.ipynb`를 받아 Marimo 형식의 `.py`로 출력해준다. 단순한 노트북은 그대로 동작한다. 하지만 다음과 같은 패턴이 들어 있으면 손으로 손봐야 한다.

- **같은 변수 이름을 여러 셀에서 정의** — Marimo는 이걸 막는다. 변환 후 변수명 분리가 필요하다.
- **`%matplotlib inline` 같은 IPython 매직** — Marimo는 IPython 매직을 지원하지 않는다. 표준 Python으로 다시 써야 한다.
- **ipywidgets 위젯** — Marimo의 `mo.ui` 위젯으로 다시 짜야 한다.
- **셀 순서에 의존한 부수 효과** — 의존 그래프로 다시 표현해야 한다.

작은 노트북이라면 한두 시간 안에 옮길 수 있다. 큰 분석 프로젝트라면 며칠이 걸릴 수도 있다. **자동 변환에 기대지 말고, 노트북을 다시 짠다는 마음으로 들어가는 편이 정신건강에 낫다.** 그 과정에서 hidden state가 끼어 있던 부분이 자연스럽게 드러나는 게 보너스다.

마이그레이션 전략을 한 가지 권한다면, **새 프로젝트만 Marimo로 시작하기.** 레거시 노트북은 그대로 Jupyter로 유지하고, 새로 시작하는 분석은 Marimo로 짠다. 두 환경이 한 조직에 공존할 수 있다. 시간이 지나면서 자연스럽게 비중이 옮겨가는 흐름이 가장 부드럽다.

## 9장에서 다시 만난다

Marimo가 푼 방식 — reactive runtime으로 hidden state를 원천 차단 — 은 노트북의 가장 큰 그림자에 대한 한 진영의 답이다. 이 답은 9장에서 다시 등장한다. 9장은 hidden state와 재현성 문제를 본격적으로 다루는 챕터인데, 거기서 우리는 **세 갈래 해법**을 만난다.

- **Reactive 진영:** Marimo·Pluto·Polynote·Observable. 의존 그래프로 hidden state 자체를 없앤다. (이번 7장)
- **Lineage 진영:** NBSafety. 정적 분석으로 hidden state를 검출만 하고, Jupyter 위에 얹어 쓴다.
- **습관 진영:** Restart and Run All + 작은 노트북 단위 + 모듈 추출. 도구 교체 없이 패턴으로 푼다.

세 답 중 어느 것이 옳은지는 한 챕터를 통째로 들여 따져볼 만한 문제다. 9장에서 그 자리에 다시 만나자.

## 마무리

Jupyter 이후의 노트북은 무엇을 다시 짜고 있는가. **Marimo의 답은 '실행 모델'이다.** 셀을 임의 순서로 실행할 수 있다는 자유 대신, 의존 그래프에 따라 자동으로 일관된 상태가 유지되는 안전을 골랐다. 그 결정이 hidden state·git·앱 배포·브라우저 실행이라는 네 가지 문제를 한 번에 풀어냈다.

물론 모든 답이 그렇듯, 트레이드오프가 있다. 자유의 일부를 잃었고, 기존 Jupyter 자산과는 갈라섰다. 그 트레이드오프를 받아들일 준비가 된 사람에게, Marimo는 새로운 작업 방식을 열어준다.

다음 장에서는 다시 시선을 넓혀본다. 이번엔 1인 도구가 아닌, **팀과 엔터프라이즈의 노트북**이다. Deepnote·Hex·Databricks가 노트북을 어떻게 협업의 작업대로 만들었는지, 그리고 챕터 끝에서 우리는 책 전체의 환경 9개를 한 장의 결정 트리와 비교표로 정리해본다. 자기 작업에 어느 노트북을 골라야 하는지 — 그 답을 손에 쥐고 9장의 가장 깊은 그림자로 들어가자.
