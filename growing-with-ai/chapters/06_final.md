# 6장. 큰 작업을 구조화하기 — 명세(Spec-Driven)와 하네스

금요일 오후, 팀 리드가 슬랙으로 한 줄을 던진다고 해보자. "다음 스프린트에 알림 기능 좀 붙여줘. 사용자가 설정에서 켜고 끌 수 있게." 당신은 옆에 둔 에이전트에게 그대로 옮겨 적는다. "알림 기능 만들어줘. 사용자가 켜고 끌 수 있게." 엔터를 누른다.

AI는 망설임 없이 코드를 쏟아낸다. 파일이 여섯 개 생기고, 모델이 추가되고, 화면에 토글 스위치까지 그려진다. 그런데 결과물을 들여다보는 순간 어딘가 찜찜하다. 알림은 이메일로 가는데, 우리 제품은 푸시 알림을 쓴다. 토글은 사용자별이 아니라 전역 설정으로 만들어졌고, 알림 종류를 구분하는 개념은 아예 없다. 분명히 "알림 기능"을 만들어달라고 했고, AI는 "알림 기능"을 만들어줬는데, 내가 원한 그 알림 기능은 아니다.

이게 누구의 잘못일까? 잠시 멈추고 생각해보자. AI를 탓하기 전에, 우리가 건넨 한 줄을 다시 읽어보자. "알림 기능 만들어줘." 이메일인지 푸시인지, 누구 단위로 켜고 끄는지, 알림 종류가 몇 가지인지 — 정작 중요한 건 아무것도 적혀 있지 않다. 모호한 입력에 대고 AI는 자기 나름의 가장 그럴듯한 빈칸을 채웠을 뿐이다. **모호한 프롬프트로 시작하면, 엉뚱한 결과는 사고가 아니라 예정된 수순이다.**

그렇다면 어떻게 해야 할까? 같은 일을 다르게 시작해보자. 코드를 요청하기 전에, 먼저 무엇을 만들 것인지부터 못 박는 것이다.

```markdown
## 기능: 사용자 알림 설정

### 무엇을
- 사용자가 알림을 종류별로 켜고 끌 수 있다.
- 알림 종류: (1) 댓글 알림 (2) 멘션 알림 (3) 주간 요약
- 전달 채널: 웹 푸시 (이메일은 이번 범위 아님)

### 누구 단위로
- 설정은 사용자(user)별이다. 전역 설정 아님.
- 기본값: 댓글·멘션은 켬, 주간 요약은 끔.

### 하지 않을 것
- 알림 발송 로직 자체는 이번 작업 범위 밖(이미 존재).
- 이메일 채널은 다음 스프린트로 미룸.
```

이 짧은 명세를 먼저 건네고 나서 "이 명세대로 구현해줘"라고 하면, 결과는 전혀 달라진다. 채널을 헷갈릴 일도, 설정 단위를 멋대로 정할 일도 없다. AI가 똑똑해진 게 아니다. **우리가 빈칸을 남기지 않았을 뿐이다.**

이 장은 바로 이 차이에서 출발한다. 큰 작업을 AI와 함께 끌고 갈 때, 우리는 두 가지 질문에 답해야 한다. 첫째, **코드를 쓰기 전에 무엇을 못 박을 것인가** — 이게 명세(spec-driven) 이야기다. 둘째, **컨텍스트 윈도 하나에 다 담기지 않는 큰 작업을 어떻게 끝까지 끌고 갈 것인가** — 이게 하네스(harness) 이야기다. 둘은 따로 노는 두 주제가 아니다. 큰 작업을 구조화한다는 한 우산 아래, 명세가 "무엇을"을 정하면 하네스가 "어떻게 여러 단계에 걸쳐"를 운영한다. 차례로 살펴보자.

## 코드로 직행하지 말고, 명세를 중심에 두자

우리는 오랫동안 "일단 코드부터 짜고 보는" 습관에 익숙했다. 머릿속에 대충 그림이 있으면 에디터를 열고 타이핑을 시작했다. 설계 문서는 코드를 다 짠 다음에, 그것도 누가 시키면 마지못해 썼다. 사람이 직접 짜던 시절엔 이게 그럭저럭 통했다. 코드를 짜는 사람과 의도를 가진 사람이 같은 머릿속에 있었으니까.

그런데 AI에게 구현을 맡기는 순간, 그 둘이 분리된다. 의도는 내 머릿속에 있고, 코드는 AI가 짠다. 이 틈을 메우는 다리가 바로 **명세**다. 명세 주도 개발(Spec-Driven Development, SDD)은 코드로 바로 가지 않고 명세를 작업의 중심에 두는 방법론이다. 큰 흐름은 단순하다.

> **Spec → Plan → Tasks → Implement**
> 명세를 쓰고 → 계획을 세우고 → 태스크로 쪼개고 → 그제야 구현한다.

(martinfowler.com의 SDD 정리, github.blog 2025 기준)

순서를 뒤집어보면 왜 이게 중요한지 또렷해진다. 명세 없이 곧장 Implement로 뛰어들면, AI는 우리가 머릿속에만 갖고 있던 의도를 추측해서 채운다. 추측이 맞으면 운이 좋은 거고, 틀리면 앞서 본 알림 사례처럼 멀쩡해 보이는 엉뚱한 코드가 나온다. 명세를 먼저 쓰면 추측의 여지가 줄어든다. 그뿐 아니다. 명세를 쓰는 동안 **우리 자신의 머릿속도 정리된다.** "주간 요약 알림은 기본값이 켬일까 끔일까?" 같은 질문은, 명세를 쓰려고 앉지 않으면 구현이 절반쯤 진행된 뒤에야 뒤늦게 튀어나온다. 그때 가서 고치려면 정말 난감하다.

물론 이렇게 반박할 수 있다. "명세 쓸 시간에 그냥 코드를 짜는 게 빠르지 않나?" 작은 작업이라면 맞는 말이다. 한 줄짜리 버그 수정에 명세를 쓰는 건 과하다. 하지만 작업이 커질수록 — 여러 파일을 건드리고, 며칠에 걸치고, 나중에 누군가 이어받아야 하는 일일수록 — 명세 없이 시작한 대가는 뒤에서 이자까지 붙어 돌아온다. 기억해두자. **명세는 코드를 늦추는 게 아니라, 다시 짜는 일을 막는다.**

### 도구를 고르기 전에 — SDD를 떠받치는 도구들

SDD를 손으로만 할 수도 있지만, 이 흐름을 아예 제품으로 만든 도구들이 2025~2026년 사이 빠르게 등장했다. 대표적인 둘이 GitHub의 **Spec Kit**과 AWS의 **Kiro**다.

**Spec Kit**은 오픈소스 툴킷이다. 단계는 Constitution → Specify → Plan → Tasks로 흐른다. 맨 앞의 Constitution이 독특한데, 프로젝트가 지켜야 할 원칙(코딩 규칙·아키텍처 제약 같은 것)을 헌법처럼 먼저 박아두는 단계다. 그 위에서 기능 명세(Specify), 계획(Plan), 태스크 분해(Tasks)가 쌓인다. Copilot·Claude Code·Gemini CLI 등 여러 에이전트에서 쓸 수 있다(github.github.com/spec-kit, 2025 기준).

**Kiro**는 AWS가 2025년 중반 내놓은 에이전틱 IDE다. Code OSS 기반이라 VS Code를 쓰던 사람에게 익숙하고, SDD가 IDE에 네이티브로 박혀 있다. 단계는 Requirements → Design → Tasks 셋이다. 요구사항을 적을 때 **EARS 표기법**을 쓰는 게 특징인데, 예를 들어 "WHEN [조건], THE SYSTEM SHALL [동작]" 같은 정형화된 문장 틀로 요구를 적게 만들어 모호함을 줄인다(martinfowler 2025, morphllm 2026 기준).

그렇다면 둘 중 어느 쪽을 골라야 할까? 상황에 따라 다르다. 아래 표를 그대로 가져다, 자기 팀 상황에 비춰 체크해보자.

#### 핵심 산출물 ① — SDD 도구 선택 기준 매트릭스 (2026 기준)

| 판단 축 | GitHub Spec Kit | AWS Kiro |
|---------|-----------------|----------|
| **팀 규모** | 작은 팀·개인부터 OK. 가벼운 툴킷 | 팀 단위 IDE 표준화에 유리 |
| **기존 IDE/환경** | 여러 에이전트(Copilot·Claude Code·Gemini CLI) 위에 얹어 씀 | Code OSS(VS Code 계열) 기반 — IDE를 통째로 바꿈 |
| **규율 강도** | Constitution으로 프로젝트 원칙을 먼저 못 박음 (규율 강함) | EARS 표기법으로 요구 문장 자체를 정형화 (규율 강함, 입력 단계에 집중) |
| **크로스툴 호환** | 높음 — 툴킷이라 에이전트를 갈아끼우기 쉬움 | 낮음 — Kiro IDE 안에서 작동하는 통합형 |
| **이럴 때 고르자** | 쓰던 에이전트를 유지하며 SDD 흐름만 도입하고 싶다 | IDE까지 SDD 중심으로 표준화하고 새로 세팅해도 좋다 |

> ⚠️ **휘발성 경고:** 이 비교는 **2026년 시점** 기준이다. Spec Kit과 Kiro 모두 분기 단위로 단계 이름·기능이 바뀌는 도구다. 도입 전에 각 도구의 공식 문서를 함께 확인하자. 표의 가치는 "어느 도구가 최신이냐"가 아니라 **"무슨 축으로 비교할 것이냐"**에 있다. 도구가 바뀌어도 팀 규모·기존 환경·규율 강도·크로스툴 호환이라는 네 축은 그대로 쓸 수 있다.

도구를 고르는 일에 너무 무게를 두지는 말자. 핵심은 도구가 아니라 **명세를 먼저 쓰는 습관**이다. Spec Kit도 Kiro도 결국 "코드 전에 명세"라는 같은 원칙을 각자의 방식으로 강제할 뿐이다. 손으로 마크다운 파일 하나에 명세를 적는 것만으로도 절반은 이미 한 셈이다.

### 큰 덩어리를 태스크로 쪼개는 법 — PRD와 사용자 스토리

명세를 쓰자고 했는데, 막상 작업이 크면 "무엇을 만들지"를 한 번에 적기가 막막하다. 그래서 중간 단계를 둔다. 큰 작업은 **PRD(제품 요구 문서) → 사용자 스토리 → 구현 가능한 태스크**로 단계적으로 잘게 부순다(softcery 2025 기준).

PRD는 "왜, 무엇을, 누구를 위해"를 적는 윗단 문서다. 사용자 스토리는 그걸 "사용자로서 나는 ~하고 싶다, 왜냐하면 ~" 형태의 작은 조각들로 나눈다. 그리고 각 스토리는 다시 AI 에이전트가 한 번에 처리할 만한 크기의 태스크로 쪼개진다. 아래는 그대로 채워 쓸 수 있는 간단한 틀이다.

```markdown
# PRD: {기능 이름}

## 배경 / 왜
- 어떤 문제를 푸는가? 누가 아쉬워하는가?

## 목표 (이번 범위)
- 이번에 반드시 되는 것 1
- 이번에 반드시 되는 것 2

## 범위 밖 (하지 않을 것)
- 이번엔 안 하는 것 — 명시적으로 적어둔다

## 사용자 스토리
- US-1: 사용자로서 나는 {행동}을 하고 싶다, 왜냐하면 {이유}.
  - 수용 기준: {이게 되면 합격이다}
- US-2: ...

## 태스크 분해
- [ ] T-1: (US-1) {AI가 한 번에 처리할 크기의 일}
- [ ] T-2: (US-1) ...
- [ ] T-3: (US-2) ...
```

여기서 **"범위 밖" 칸을 빼먹지 말자.** 앞의 알림 사례에서 "이메일은 이번 범위 아님"이라고 한 줄 적었던 게 기억나는가? 무엇을 하지 않을지를 명시하는 것은, 무엇을 할지를 적는 것만큼이나 AI의 헛발질을 줄인다. AI는 빈칸을 보면 기어이 채우려 들기 때문이다.

## 한 윈도를 넘는 작업, 하네스로 끌고 가자

자, 이제 명세도 썼고 태스크도 쪼갰다. 그런데 여기서 새로운 문제가 고개를 든다. 작업이 정말 크면, **한 번의 대화(컨텍스트 윈도) 안에 다 끝나지 않는다.** 태스크가 스무 개쯤 되고, 중간에 에이전트가 컨텍스트가 꽉 차서 헤매기 시작하고, 어제 하던 작업을 오늘 이어서 하려는데 AI는 어제 무슨 결정을 했는지 까맣게 잊어버린다. 겪어본 사람은 안다. 정말 아찔한 순간이다. 분명 잘 굴러가던 작업이 컨텍스트 윈도 하나를 넘는 순간 무너진다.

그렇다면 어떻게 해야 할까? 여기서 **하네스(harness)**라는 개념이 등장한다. 하네스는 5장에서 다룬 컨텍스트(context)의 상위 확장으로 이해하면 쉽다. Augment Code의 정리를 빌리면, 셋의 위계는 이렇다.

> **prompt < context < harness**
> - **prompt** — 단 한 번의 상호작용. "이거 해줘" 한 마디.
> - **context** — 한 컨텍스트 윈도 안에서 토큰을 큐레이션하는 일. 5장의 CLAUDE.md/AGENTS.md가 여기다.
> - **harness** — 컨텍스트 리셋·구조화된 핸드오프·페이즈 게이트로 **여러 윈도에 걸친** 목표 지향 작업을 운영하는 골격.

(Augment Code, "Harness Engineering" 기준)

다시 말해, prompt가 한 마디라면, context는 한 대화 분량의 판을 까는 일이고, harness는 그 대화가 여러 번 갈아엎어져도 작업이 처음부터 끝까지 일관되게 굴러가도록 떠받치는 틀이다. 5장에서 한 윈도 안에 무엇을 흘려보낼지를 설계했다면, 이제는 그 윈도를 여러 개 이어 붙여 큰 작업을 완주하는 법을 배우는 셈이다. 명세가 "영속 계약"이라면, 하네스는 그 계약을 여러 윈도에 걸쳐 집행하는 운영 골격이다.

하네스를 떠받치는 세 기둥을 하나씩 보자.

첫째, **컨텍스트 리셋**이다. 한 컨텍스트 윈도는 무한하지 않다. 작업이 길어지면 윈도가 꽉 차고, 그때부터 에이전트는 앞에서 한 말을 흘리기 시작한다. 그러니 한 페이즈가 끝나면 미련 없이 컨텍스트를 비우고 새로 시작하는 편이 낫다. 단, 그냥 비우면 기억까지 통째로 날아간다. 그래서 둘째가 필요하다.

둘째, **구조화된 핸드오프**다. 컨텍스트를 리셋하기 전에, 지금까지 무엇을 했고 다음에 무엇을 해야 하는지를 **문서로 남긴다.** 마치 교대 근무자가 다음 근무자에게 인수인계서를 넘기듯이 말이다. 새 컨텍스트에서 이 문서를 먼저 읽히면, 에이전트는 어제의 자신이 어디까지 했는지를 다시 알게 된다.

셋째, **페이즈 게이트**다. 각 페이즈를 끝낼 때 "이 페이즈를 정말 통과했는가?"를 판정하는 관문을 둔다. 테스트는 통과했는지, 명세대로 됐는지, 다음 페이즈로 넘길 산출물이 갖춰졌는지를 체크하고, 통과하지 못했으면 다음으로 넘어가지 않는다. 게이트가 없으면 어설프게 끝난 페이즈가 다음 페이즈의 토대를 갉아먹는다.

처음 이 세 기둥을 들으면 "그냥 일을 잘게 나누라는 얘기 아닌가" 싶어 다소 번거롭게 느껴질 수 있다. 그 마음 이해한다. 하지만 작업이 며칠에 걸쳐 컨텍스트를 몇 번이나 갈아엎어야 할 때, 이 세 가지 없이 버텨본 사람은 안다 — 어제의 결정이 통째로 증발해 같은 길을 다시 헤매는 게 얼마나 아찔한지. 번거로움을 한 번 치르고 헤맴을 영영 더는 편이 낫다.

### 추상화 계층이 한 칸 올라간다

여기까지 오면 우리가 AI와 일하는 방식 자체가 한 단계 올라갔음을 느낄 수 있다. 한 논문은 이 변화를 **"task-driven copilot에서 goal-driven AI pair programmer로의 전환"**이라고 표현한다(arXiv 2404.10225). 풀어보자.

예전의 코파일럿은 **태스크 주도**였다. "이 함수 자동완성해줘", "이 반복문 채워줘" — 우리가 잘게 쪼갠 태스크를 하나씩 던지면 AI가 받아쳤다. 우리는 여전히 코드 한 줄 한 줄의 수준에서 생각하고 있었다. 반면 하네스 위에서 일하는 AI는 **목표 주도**에 가깝다. "이 명세대로 알림 기능을 완성해줘"라는 목표를 주면, AI가 페이즈를 거치며 스스로 태스크를 처리해 나간다. 우리가 생각하는 수준이 코드 줄에서 **목표·의도**로 한 칸 올라간 것이다.

좋은 변화다. 하지만 여기엔 함정도 있다. 추상화가 올라갈수록, 우리가 직접 들여다보지 않는 영역도 넓어진다는 점이다. 이 이야기는 잠시 뒤 성찰에서 다시 꺼내자.

### 핵심 산출물 ② — 핸드오프 문서 + 페이즈 게이트 체크리스트

말로만 하면 막연하니, 그대로 채워 쓸 수 있는 양식을 손에 쥐여주겠다. 먼저 **핸드오프 문서**다. 한 페이즈를 끝내고 컨텍스트를 리셋하기 직전에, 이 문서를 채워서 다음 페이즈(혹은 다음 날의 자신, 혹은 동료)에게 넘긴다.

```markdown
# 핸드오프: {작업 이름} — Phase {N} → Phase {N+1}
> 작성 시각: {날짜/시각}

## 1. 목표 (변하지 않는 것)
- 이 작업 전체가 달성하려는 것 한 줄
- 참조 명세 파일: {경로 — 예: specs/notifications.md}

## 2. 지금까지 한 것 (Phase {N} 결과)
- 완료한 태스크: T-1, T-2 (체크된 것)
- 만든/바꾼 핵심 파일: {경로 목록}
- 내린 주요 결정: {왜 이렇게 했는지 — 나중에 또 묻지 않도록}

## 3. 다음에 할 것 (Phase {N+1})
- 다음 태스크: T-3, T-4
- 먼저 읽어야 할 파일/맥락: {경로}
- 주의: {빠지기 쉬운 함정·이미 시도해 실패한 방법}

## 4. 미해결 / 막힌 것
- 아직 못 정한 것, 사람의 판단이 필요한 것
```

다음은 **페이즈 게이트 체크리스트**다. 한 페이즈를 "통과"로 판정하려면 아래가 전부 ✅여야 한다. 하나라도 ❌면 다음 페이즈로 넘어가지 않는다.

```markdown
## 페이즈 게이트 체크리스트 — Phase {N}

- [ ] 이 페이즈의 태스크가 모두 완료됐다 (체크리스트와 대조)
- [ ] 테스트가 통과한다 (새로 추가한 테스트 포함)
- [ ] 결과물이 명세와 일치한다 (specs 파일과 한 줄씩 대조)
- [ ] 범위 밖 변경이 끼어들지 않았다 (PRD "하지 않을 것" 대조)
- [ ] 핸드오프 문서를 작성했다 (다음 페이즈가 읽을 수 있게)
- [ ] 다음 페이즈가 의존하는 산출물이 갖춰졌다

→ 전부 ✅면 컨텍스트를 리셋하고 Phase {N+1} 시작.
→ 하나라도 ❌면 이 페이즈에 머문다.
```

이 두 장의 문서가 별것 아닌 듯 보여도, 컨텍스트 윈도를 여러 번 갈아엎는 큰 작업에서는 생명줄 역할을 한다. 에이전트가 기억을 잃어도, 우리가 기억을 파일에 적어뒀기 때문이다. 명세가 "무엇을"을 못 박았듯, 핸드오프와 게이트는 "여기까지 어떻게 왔고 어디로 가는지"를 못 박는다.

## 사례 — 8년차 엔지니어가 순수 바이브 코딩을 포기한 이유

추상적인 이야기를 현장 사례 하나로 묶어보자. 농심 클라우드 기술 블로그에 한 8년차 AI 엔지니어의 전환기가 실렸다(NDS Cloud Tech Blog, 2025/2026 기준). 그는 한동안 순수 바이브 코딩 — 떠오르는 대로 AI에게 던지고 결과를 받는 방식 — 에 기대 일했다. 빠르긴 했다. 하지만 작업이 커질수록 결과가 자꾸 어긋났고, 결국 그는 그 방식을 **포기했다.**

그가 갈아탄 방식은 이 장의 흐름과 정확히 겹친다. 세 단계였다.

먼저 **설계부터 한다.** 코드를 요청하기 전에 무엇을 만들지를 못 박았다 — 이게 우리가 앞에서 본 명세다. 다음으로 **반복 구현은 AI에 위임한다.** 설계가 서면, 그 위에서 반복적인 구현은 과감히 AI에게 넘겼다 — 3장에서 세운 위임의 기준이 여기서 작동한다. 마지막으로 **AI 코드는 반드시 검수한다.** 위임했다고 끝이 아니라, 나온 코드를 자기 눈으로 확인했다 — 이건 7장에서 본격적으로 다룰 검증의 예고편이다.

이 사례가 알려주는 것은 무엇일까? **순수 바이브 코딩이 무너지는 지점이 바로 "큰 작업"이라는 것이다.** 작은 일은 느낌만으로도 되지만, 일이 커지면 느낌은 어긋남으로 돌아온다. 그 어긋남을 막는 게 "설계 먼저 → 위임 → 검수"라는 구조다. 8년차의 노련함은 더 빠르게 타이핑하는 데 있지 않았다. **언제 구조를 세워야 하는지를 아는 데** 있었다. 비슷한 상황에 놓였다면, 우리도 작업의 크기를 먼저 가늠하고 그에 맞는 구조를 세우는 편이 낫다.

## 마무리

명세를 쓰고, 작업을 페이즈로 쪼개고, 핸드오프 문서를 넘기는 동안 무슨 일이 일어났는지 가만히 들여다보자. 우리는 코드를 한 줄도 직접 타이핑하지 않았을지 모른다. 하지만 그 사이 내내 **생각하고 있었다.** 무엇을 만들지, 무엇을 하지 않을지, 어디서 페이즈를 끊을지, 이 결정이 다음 단계에 어떤 짐을 지울지를. 추상화 계층이 코드 줄에서 목표·의도로 한 칸 올라갔다고 해서, 사고가 면제된 것이 아니다. 오히려 사고의 무게중심이 옮겨갔을 뿐이다. 더 높은 곳에서 일한다는 것은 덜 생각한다는 뜻이 아니라, 더 멀리 내다보며 생각한다는 뜻이다. 1장에서 두려워한 "녹스는 감각"은 바로 이 자리, 무엇을 못 박고 어디서 끊을지를 판단하는 자리에서 오히려 단련된다.

기억해두자. 큰 작업을 구조화하는 일은 두 축으로 이뤄진다. **명세로 무엇을 못 박을지를 정하고, 하네스로 한 윈도를 넘는 작업을 끝까지 끌고 가는 것.** 손에 쥔 선택 매트릭스로 도구를 고르고, 핸드오프 양식과 게이트 체크리스트로 작업을 이어 붙이자. 도구 이름은 다음 분기에 또 바뀌겠지만, "코드 전에 명세, 윈도 너머로 핸드오프"라는 두 원칙은 쉽게 낡지 않는다.

그런데 이렇게 빠르고 크게 만들고 나면, 청구서가 날아온다. 그 많은 코드를 누가, 어떻게 검증할 것인가? 다음 장에서는 작성이 빨라진 만큼 무거워진 또 다른 일 — 검증 — 을 마주해보자.
