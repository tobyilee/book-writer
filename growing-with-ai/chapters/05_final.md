# 5장. 에이전트에게 무엇을 흘려보내는가 — 컨텍스트 엔지니어링과 CLAUDE.md/AGENTS.md

같은 회사, 같은 코드베이스, 같은 모델. 그런데 옆자리 동료는 에이전트에게 "결제 실패 시 재시도 로직 추가해줘"라고 한 줄 던지고 거의 그대로 머지할 만한 코드를 받아온다. 당신이 똑같이 한 줄을 던지면, 에이전트는 어디서 본 적도 없는 라이브러리를 `import`하고, 우리 팀이 3년 전에 버린 콜백 패턴으로 코드를 짜고, 테스트는 한 줄도 붙이지 않은 결과물을 내민다. 모델이 다른 것도 아니다. 둘 다 같은 에이전트를 쓴다.

자, 무엇이 다른 걸까? 잠시 멈추고 생각해보자. 프롬프트 한 줄만 보면 둘은 거의 같다. 차이는 그 한 줄 **바깥**에 있다. 동료의 에이전트는 우리 팀의 규칙·금기·패키지 매니저·테스트 관행을 이미 알고 있는 상태에서 그 한 줄을 받았고, 당신의 에이전트는 아무것도 모르는 백지 상태에서 같은 한 줄을 받았다. 같은 질문이라도 누구에게 묻느냐에 따라 답이 달라지듯, 에이전트도 **무엇을 알고 있느냐**에 따라 전혀 다른 코드를 내놓는다. 이 장은 그 "무엇을 알고 있느냐"를 우리가 직접 설계하는 이야기다.

## 프롬프트가 아니라 컨텍스트다

한동안 우리는 좋은 결과를 얻으려면 **프롬프트를 잘 써야 한다**고 믿었다. 마법의 주문처럼 단어를 고르고, "당신은 시니어 개발자다" 같은 역할 부여를 앞에 붙이고, 예시를 한두 개 끼워 넣는 식이었다. 물론 프롬프트도 중요하다. 하지만 에이전트 시대로 넘어오면서 무게중심이 옮겨갔다.

mem0.ai가 2025년에 정리한 컨텍스트 엔지니어링 가이드에는 이런 문장이 나온다. **"오늘날 에이전트 실패의 대부분은 모델 실패가 아니라 컨텍스트 실패다."** Anthropic도 2025년 11월 글에서 비슷한 결을 짚었다(웹: mem0.ai 2025; Anthropic 2025-11). 이 한 문장을 곱씹어보자. 에이전트가 엉뚱한 코드를 줬을 때, 우리는 보통 "모델이 멍청하네" 하고 모델 탓을 한다. 그런데 실제로는 모델이 멍청한 게 아니라, **우리가 모델에게 필요한 것을 흘려보내지 않은 경우**가 훨씬 많다는 이야기다.

프롬프트 엔지니어링이 "한 번의 질문을 잘 쓰는 일"이라면, 컨텍스트 엔지니어링은 그 질문에 앞서 **에이전트의 머릿속에 무엇을 채워두느냐를 설계하는 일**이다. 단일 프롬프트를 넘어, 에이전트에 흘러드는 정보 전체 — 시스템 규칙, 메모리, 검색해 온 문서, 도구의 사용법, 최근 대화, 그리고 지금 시키려는 작업 — 를 의식적으로 관리하는 것. 이게 컨텍스트 엔지니어링이다.

## 흘러드는 것들에는 층이 있다

에이전트에게 흘러드는 정보를 mem0.ai는 여섯 개의 층으로 나눈다(웹: mem0.ai 2025). 길게 늘어놓을 생각은 없다. 이름과 한 줄 정의만 빠르게 훑고, 이 장의 진짜 무게는 그다음으로 넘기자.

- **시스템 규칙** — "넌 이런 식으로 일해라"라는 항상 켜져 있는 지침. 우리가 곧 다룰 CLAUDE.md가 바로 여기 산다.
- **메모리** — 이전 세션이나 대화에서 누적된 사실. "이 프로젝트는 yarn을 쓴다"를 한 번 알려주면 기억하는 것.
- **검색 문서** — 그때그때 끌어오는 외부 지식. 사내 위키, API 문서, 관련 코드 파일.
- **툴 스키마** — 에이전트가 쓸 수 있는 도구의 사용법. 어떤 함수를 어떤 인자로 부를 수 있는지.
- **최근 대화** — 지금 이 세션에서 오간 직전 메시지들.
- **현재 작업** — 방금 당신이 던진 그 한 줄.

각 층은 서로 다른 속도로 변한다(웹: mem0.ai 2025). 시스템 규칙은 거의 안 변하고, 현재 작업은 매번 바뀐다. 그래서 각 층을 **명료하게 유지**하는 게 핵심이다. 여기서 한 가지 의문이 생긴다. 여섯 층 중에 우리가 직접, 손으로, 가장 강하게 통제할 수 있는 층은 어디일까? 바로 첫 번째 층, 시스템 규칙이다. 그리고 그걸 통제하는 가장 실전적인 도구가 프로젝트 루트에 놓이는 영속 컨텍스트 파일, CLAUDE.md다. 이제부터가 이 장의 본론이다.

## 빈약한 컨텍스트가 만드는 엉뚱한 코드

말로만 하면 와닿지 않는다. 직접 보자. 어느 팀에 새로 들어온 개발자가 결제 모듈에 기능 하나를 추가한다고 해보자. 프로젝트 루트에는 이런 CLAUDE.md가 놓여 있다.

```markdown
# 프로젝트

결제 서비스입니다. 코드 짜주세요.
```

이걸 본 순간 어딘가 찜찜하지 않은가? 이 파일은 사실상 아무것도 말해주지 않는다. 그런데 놀랍게도 현장의 많은 CLAUDE.md가 이 수준이거나, 아예 없다. 자, 이 상태에서 개발자가 에이전트에게 한 줄을 던진다.

> "결제 실패 시 3회까지 재시도하는 로직을 추가해줘."

에이전트는 아는 게 없으니 **그럴듯한 일반론**으로 채운다. 결과는 대략 이렇다.

```javascript
const axios = require('axios');

async function retryPayment(paymentData) {
  for (let i = 0; i < 3; i++) {
    try {
      const res = await axios.post('/api/payment', paymentData);
      return res.data;
    } catch (e) {
      console.log('재시도 중...', i);
    }
  }
  throw new Error('결제 실패');
}
```

코드 자체는 돌아갈 것처럼 보인다. 하지만 이 팀의 시니어가 보면 한숨이 나온다. 무엇이 문제일까? 하나씩 짚어보자.

우리 팀은 HTTP 클라이언트로 `axios`를 안 쓴 지 오래다. 내부 `httpClient` 래퍼를 쓰기로 합의했는데, 에이전트는 그걸 알 길이 없으니 가장 흔한 `axios`를 끌어왔다. `console.log`로 재시도를 찍는 것도 우리 로깅 규칙 위반이다. 우리는 구조화 로거를 쓴다. 게다가 재시도 사이에 백오프(backoff)도 없이 즉시 세 번을 때려버린다. 결제 서버 입장에서는 거의 공격이다.

그리고 마지막. 이 모든 걸 덮어버리는 결정적인 문제가 하나 남았다. **테스트가 한 줄도 없다.** 이 팀은 결제처럼 민감한 코드에 테스트를 강제하는데 말이다. 가장 위험한 도메인의 코드를, 검증 장치 하나 없이 받아버린 셈이다.

에이전트가 멍청해서 이렇게 짠 게 아니다. 우리가 아무것도 흘려보내지 않았기 때문이다. 백지 상태에서 "결제 재시도"를 시키면, 에이전트는 인터넷에서 가장 흔히 보이는 패턴으로 빈칸을 메운다. 그게 우리 팀의 약속과 어긋나는 건 당연한 일이다. 모델 실패가 아니라 컨텍스트 실패다. 앞서 본 그 문장이 여기서 생생해진다.

## 컨텍스트를 채우면 출력이 달라진다

그렇다면 어떻게 해야 할까? 답은 단순하다. 에이전트가 **알았어야 할 것**을 CLAUDE.md에 적어두면 된다. 같은 팀의 CLAUDE.md를 이렇게 다시 써보자.

```markdown
# 결제 서비스 (payment-service)

사내 결제 게이트웨이. PG사 3곳(토스/나이스/KCP)을 추상화해 단일 인터페이스로 제공한다.

## 기술 스택
- 런타임: Node.js 20, TypeScript 5 (strict)
- 패키지 매니저: **pnpm** (npm/yarn 사용 금지)
- HTTP 호출: 반드시 `src/lib/httpClient.ts` 래퍼 사용. axios/fetch 직접 호출 금지.
- 로깅: `src/lib/logger.ts`의 구조화 로거. `console.log` 금지.

## 코딩 규칙
- 외부 호출 재시도는 `src/lib/retry.ts`의 `withRetry()`를 쓴다.
  지수 백오프가 내장돼 있으니 직접 for 루프를 돌리지 말 것.
- 결제 관련 코드는 금액·통화를 다루므로 정수(최소 화폐 단위)로만 계산한다. 부동소수점 금지.
- 에러는 throw하지 말고 `Result<T, PaymentError>` 타입으로 반환한다.

## 테스트
- 결제 도메인(`src/payment/**`) 변경은 **단위 테스트 필수**. 테스트 없는 PR은 머지 불가.
- 테스트 러너: Vitest. `pnpm test`로 실행.
- 외부 PG 호출은 `test/mocks/pg.ts`의 목을 쓴다.

## 하지 말아야 할 것
- 결제 로직에 새 외부 의존성 추가 금지 (보안 리뷰 대상).
- 로그에 카드번호·CVC 등 민감정보 출력 금지.
```

이제 똑같은 한 줄을 던져보자. "결제 실패 시 3회까지 재시도하는 로직을 추가해줘." 이번에는 결과가 사뭇 다르다.

```typescript
import { withRetry } from '../lib/retry';
import { httpClient } from '../lib/httpClient';
import { logger } from '../lib/logger';
import type { Result, PaymentError } from './types';

export async function requestPayment(
  req: PaymentRequest,
): Promise<Result<PaymentResult, PaymentError>> {
  return withRetry(
    async () => {
      const res = await httpClient.post('/payments', req);
      return res.body;
    },
    { retries: 3 },
  ).catch((err) => {
    logger.warn('payment.retry.exhausted', { orderId: req.orderId });
    return { ok: false, error: new PaymentError('RETRY_EXHAUSTED') };
  });
}
```

차이가 보이는가? `httpClient` 래퍼를 쓰고, `withRetry`로 백오프를 위임하고, 구조화 로거로 찍고, `Result` 타입으로 반환한다. 카드번호 같은 민감정보도 로그에 없다. 한 줄짜리 프롬프트는 그대로인데, 출력 품질이 통째로 바뀌었다. 우리가 바꾼 건 프롬프트가 아니라 **그 프롬프트가 흘러든 컨텍스트**다. 이게 컨텍스트 엔지니어링의 가장 손에 잡히는 형태다.

> 위 두 코드 예시는 개념을 보여주기 위한 예시이며, 특정 라이브러리의 실제 API 시그니처가 아니다. `withRetry`·`httpClient`는 팀이 자체적으로 만든 래퍼라고 가정한 것이다.

여기서 한 가지 오해를 짚고 넘어가자. "그럼 CLAUDE.md를 길게, 모든 걸 다 적으면 되겠네?"라고 생각하기 쉽다. 하지만 그건 또 다른 함정이다. 컨텍스트도 결국 모델의 한정된 주의력을 나눠 쓴다. 쓸데없는 회사 소개나 안 지키는 규칙을 잔뜩 적어두면, 정작 중요한 규칙이 묻혀버린다. 앞서 "각 층을 명료하게 유지하라"고 한 이유가 이것이다. CLAUDE.md는 **에이전트가 실제로 어겨서 당신을 난감하게 만든 것들**을 중심으로, 짧고 또렷하게 유지하는 편이 낫다. 한 번 어긴 규칙을 한 줄 추가하는 식으로, 살아 있는 문서처럼 키워가자.

## 한 팀, 여러 도구 — AGENTS.md라는 공용어

이쯤에서 현실적인 고민이 하나 생긴다. 우리 팀은 Claude Code만 쓰지 않는다. 누구는 Cursor를, 누구는 Copilot을, 누구는 Gemini CLI를 쓴다. 그런데 CLAUDE.md는 이름부터 Claude Code의 파일이다. 그렇다면 같은 규칙을 도구마다 다른 파일에 똑같이 베껴 적어야 할까? 생각만 해도 번거롭다. 하나 고치면 네 군데를 고쳐야 하고, 어느 하나는 반드시 뒤처진다.

이 문제를 풀려고 등장한 것이 **AGENTS.md**다. 2025년 8월에 공개된 오픈 표준으로, OpenAI·Google·Cursor·Factory 등이 함께 만들었다(웹: AGENTS.md spec 2025-08). 핵심은 단순하다. **어느 에이전트 도구든 읽을 수 있는 공용 컨텍스트 파일**을 하나 두자는 것이다. 프로젝트 루트에 `AGENTS.md`를 두면 Claude Code, Cursor, Copilot, Gemini CLI 등이 크로스툴로 이를 참조한다(2026년 5월 기준; 웹: AGENTS.md spec 2025-08). 도구가 늘어도 규칙은 한 군데만 관리하면 된다.

형식은 거창하지 않다. 사실상 우리가 위에서 쓴 CLAUDE.md와 같은 마크다운이다. 예를 들면 이런 식이다.

```markdown
# AGENTS.md

## 프로젝트
사내 결제 게이트웨이 (payment-service). PG사 3곳을 단일 인터페이스로 추상화.

## 셋업
- 의존성 설치: `pnpm install`
- 개발 서버: `pnpm dev`
- 테스트: `pnpm test` (결제 도메인 변경 시 필수)

## 규칙
- 패키지 매니저는 pnpm. HTTP는 `src/lib/httpClient.ts` 래퍼만 사용.
- 재시도는 `withRetry()`, 로깅은 구조화 로거. console.log·axios 직접 호출 금지.
- 금액은 정수(최소 화폐 단위)로만 계산.

## 주의
- 결제 로직에 새 외부 의존성 추가 금지.
- 로그에 민감정보(카드번호·CVC) 출력 금지.
```

그렇다면 CLAUDE.md는 이제 버려야 할까? 꼭 그렇진 않다. 도구별 특화 지침이 필요할 때는 도구 전용 파일을 따로 두고, 공통 규칙은 AGENTS.md로 모으는 식의 조합이 현실적이다. 어느 쪽이든 원칙은 같다. **같은 규칙을 여러 군데 베껴 쓰지 말자.** 진실의 출처(source of truth)를 하나로 두는 것 — 이건 사람이 읽는 README든 에이전트가 읽는 AGENTS.md든 다르지 않다.

여기서 신선도 한 가지를 짚어두자. 이 표준은 2025년 8월에 막 공개됐고, 도구들의 지원 범위와 세부 동작은 빠르게 바뀌고 있다(2026년 5월 기준). 어떤 파일을 어떤 우선순위로 읽는지는 도구마다 다를 수 있으니, 실제로 도입할 때는 각 도구의 공식 문서를 함께 확인해두자. 이 부분은 곧 또 바뀔 수 있다.

## 컨텍스트를 설계한다는 것은

이 장의 기법을 한 발 떨어져서 보면, 흥미로운 사실이 하나 드러난다. 우리가 CLAUDE.md나 AGENTS.md를 잘 쓰려고 애쓰는 동안, 사실은 에이전트가 아니라 **우리 자신에게** 무언가를 하고 있었다는 점이다.

"우리 팀은 왜 axios를 안 쓰지?", "재시도 정책의 백오프는 정확히 어떻게 되더라?", "결제 금액은 왜 정수로 다뤘더라?" — 에이전트에게 규칙을 적어주려면, 먼저 그 규칙을 우리가 또렷이 알아야 한다. 막연히 "그냥 그렇게 하기로 했었지" 수준으로 남아 있던 암묵지가, 한 줄의 문장으로 적히는 순간 명시지가 된다. AI에게 잘 설명하려다, 정작 내가 우리 시스템을 더 또렷이 알게 되는 것이다. 컨텍스트를 설계하는 일은 곧 내 의도를 명료히 하는 일이다. 3장에서 우리는 무엇을 맡기고 무엇을 익힐지 판단하는 기준을 세웠다. 그 판단을 에이전트에게 전하는 통로가 바로 이 컨텍스트 파일들이다. 잘 흘려보내려 애쓸수록, 흘려보내는 우리 쪽이 더 선명해진다.

## 마무리

같은 모델인데 누구는 좋은 코드를, 누구는 엉뚱한 코드를 받는 이유 — 이제 그 답이 손에 잡힐 것이다. 차이는 모델이 아니라, 그 모델에게 무엇을 흘려보냈느냐에 있었다. 오늘날 에이전트 실패의 대부분은 모델 실패가 아니라 컨텍스트 실패라는 말을, 우리는 빈약한 CLAUDE.md와 잘 정리된 CLAUDE.md의 출력 차이로 직접 확인했다.

기억해두자. 프롬프트 한 줄을 다듬는 데 시간을 쏟기 전에, 그 한 줄이 흘러들 컨텍스트부터 점검하는 편이 낫다. CLAUDE.md든 AGENTS.md든, 진실의 출처를 하나로 두고, 에이전트가 실제로 어긴 것들을 중심으로 짧고 또렷하게 키워가자. 그리고 그 파일들은 곧 바뀔 도구 생태계 위에 놓여 있으니, 도입할 때는 공식 문서를 함께 확인하는 습관을 잊지 말자.

그런데 컨텍스트를 아무리 잘 흘려보내도, 한 가지 벽에 부딪힌다. 정말 큰 작업 — 컨텍스트 윈도 하나에 도무지 담기지 않는 작업은 어떻게 끌고 갈까? CLAUDE.md에 규칙을 적어두는 것만으로는 부족한, 여러 단계에 걸친 큰 작업 말이다. 다음 장에서는 코드를 짜기 전에 무엇을 못 박아야 하는지(명세), 그리고 한 윈도를 넘어서는 작업을 어떻게 구조화해 끝까지 끌고 가는지(하네스)를 함께 살펴보자.
