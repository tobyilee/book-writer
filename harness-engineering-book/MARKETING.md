# 하네스 엔지니어링

### Claude Code와 Codex로 에이전트를 프로덕션에 태우는 법

**저자** Toby-AI · **버전** 1.0.0 · **언어** 한국어 · **분량** 약 120,000자 (본문 12장 + 부록 6개) · **포맷** EPUB 3

---

## 독자가 이미 느끼고 있는 것

- Claude Code 혹은 Codex CLI를 매일 쓰는데, 정말로 빨라진 것 같지는 않다.
- 주말에 Ralph Loop를 돌려봤는데 $40가 녹고 결과는 애매했다.
- 팀원이 올린 AI-coded PR을 어느 강도로 리뷰해야 할지 감이 없다.
- MCP 서버를 덕지덕지 붙였더니 에이전트가 오히려 멍청해졌다.
- 보안팀에 "에이전트가 프로덕션 접근한다"는 얘기를 꺼내면 대화가 멈춘다.

이 책은 이 다섯 가지 피로감을 **엔지니어링으로** 해소한다. 열정 페이, 모범 답안, 마케팅 수사는 쓰지 않는다.

---

## 이 책이 약속하는 것

> 책을 덮을 때 자기 레포에 **Pareto 플롯 1장을 commit**한다.

추상 목표 아니다. 독자가 읽는 동안 자기 하네스의 비용(cost) × 품질(accuracy) × 개입률(intervention rate)을 실제로 측정해보고, 그 결과를 팀에 제시할 수 있는 상태가 된다. 이 한 산출물을 위해 12장의 흐름이 설계돼 있다.

---

## 이 책이 약속하지 않는 것

- Claude Code 입문 튜토리얼이 아니다. 2개월 이상 실사용자 기준으로 쓰였다.
- "AI로 10× 생산성"을 주장하지 않는다. MIT/METR RCT의 **19% 실측 감속 vs 20% 체감 가속**을 서문의 대표 데이터로 삼는다.
- 특정 도구에 락인시키지 않는다. Claude Code / Codex CLI / Cursor / Aider를 비용·통제·관측성 3축으로 **비교**한 뒤 독자가 선택하게 한다.

## 목차

### 서문 — 왜 이 책을 읽어야 하는가
당신이 "빨라졌다"고 느끼는 감각은 증거인가? MIT/METR RCT 39%p 갭으로 책 전체를 여는 장.

### 1장. 하네스라는 마구(馬具)
"하네스"는 무엇을 조립한 것인가, 왜 이제야 이름이 붙었는가. Karpathy 3요소(editable asset · scalar metric · time-box)로 재현 가능한 루브릭을 제시한다.

### 2장. 도구 생태계 — Claude Code와 Codex의 실체
왜 토큰 소비가 4× 차이 나는가. 훅 27종, 샌드박스 3모드, 설정 스코프 우선순위를 실제 메커니즘 수준에서 해부하고 선택 플로우차트로 정리한다.

### 3장. 컨텍스트가 99%다
`AGENTS.md`는 스타일 가이드인가, 실패 로그인가. 사람이 쓰면 +4%, LLM이 쓰면 −3%라는 실험 결과와 Cline의 50% 컨텍스트 규칙을 근거로 컨텍스트 분산 설계를 다룬다.

### 4장. 루프의 해부학 — Ralph · ReAct · Plan&Execute · Reflexion
"한 루프 한 가지." Ralph Loop의 진짜 메시지를 복원하고 4개 루프 패턴을 비교·적합 매트릭스로 정리한다. Overcooking · Undercooking · Completion promise · Context pollution 네 가지 실패 모드.

### 5장. 메트릭과 Goodhart — scalar는 거짓말을 한다
단일 수치로 자동화할 수 있는 일은 무엇인가. Kapoor의 비용 폭발 도식을 근거로 **Pareto 2축 + 개입률**을 의무화한다.

### 6장. 검증 설계 — Generator–Critic, CoVe, pairwise-with-swap
같은 모델이 자기 답을 채점하면 왜 +8~15% 편향되는가. MT-Bench의 세 가지 편향과 실전 프로토콜, 그리고 테스트·린터가 진짜 back-pressure 출처임을 다룬다.

### 7장. 서브에이전트·팀 — 언제 쓰고 언제 안 쓰는가
3~4× 토큰을 쓸 가치가 있는 협업은 무엇인가. single-agent → subagent → team의 의사결정 트리를 다이어그램으로 준다.

### 8장. 도구와 MCP — 많을수록 나빠지는 지점
왜 GitHub 공식 MCP 하나가 46k 토큰을 먹는가. Cursor 40-tool silent cap, Perplexity의 weekend-hype walkaway, mcp2cli의 99% 토큰 절감 사례로 "MCP는 last resort" 원칙을 세운다.

### 9장. 위협 모델 — 프롬프트 인젝션부터 기업 컨텍스트까지
샌드박스 안에 비밀을 넣으면 왜 안 되는가. Greshake의 간접 프롬프트 인젝션, Agent-SafetyBench의 60% 장벽, Claude Code 훅이 `--dangerously-skip-permissions`를 이기는 메커니즘. SOC2·ISO27001·온프레미스·AI Gateway까지 기업 컨텍스트를 한 장에 담는다.

### 10장. 비용·CI — 자동화된 하네스를 돌리는 엔지니어링
월 예산 50%에서 알람이 오는가. FrugalGPT, RouteLLM, Speculative Decoding, test-time compute 삼박자로 90%+ 절감이 가능한 경우를 다룬다. GitHub Actions 기반 CI 워크플로 예시 포함.

### 11장. 조직에 태우기 — 팀·리뷰·인수인계·거버넌스
AI-coded PR은 일반 PR과 같은 강도로 리뷰되는가. Amazon Q의 90일 freeze 사례를 부검하고, AI-PR 리뷰 프로토콜·공유 `AGENTS.md` 거버넌스·신입 30분 온보딩 스크립트·비상 런북을 제공한다.

### 12장. 캡스톤 회고 — Pareto 2축으로 본 내 하네스
"자동화할 가치가 있었는가, 감당 가능했는가"를 수치로 답한다. 이 책을 의심해야 할 4가지 신호와, Anthropic의 skill-atrophy 17%라는 마지막 경고를 남긴다.

## 이 책이 다른 책과 다른 점

**근거 기반.** 학술 27편(ReAct · Reflexion · Self-Refine · CoVe · MT-Bench · SWE-agent · AutoCodeRover · ToolEmu · Agent-SafetyBench · FrugalGPT · RouteLLM · Test-Time Compute 등)에 arXiv 식별자를 달아 모든 주장에 출처를 연결한다.

**Contrarian 각 장 1개.** 주류 주장에 대해 반증 데이터를 한 박스씩 박는다. METR 39%p 갭, AGENTS.md 논문의 −3%, Amazon Q 90일 freeze, MCP 토큰 46k 팽창, Perplexity walkaway — 전부 원문 출처와 함께.

**실습 3단 태그.** 모든 실습을 `[읽기 15분]` · `[본격 2시간]` · `[연쇄 4시간]`으로 분류한다. 독자가 자기 일정에 맞게 읽기만 해도 되고, 한 가지만 본격 해도 되고, 주말에 연쇄 실습으로 갈아타도 된다.

**한국 기업 컨텍스트.** 보안팀 설득 talking points, SOC2 CC 6.1·CC 7.2 매핑, 사내 GPT 게이트웨이 시나리오까지 실제로 한국 팀이 부딪히는 지점을 다룬다.

## 샘플 문단 (1장 도입부)

> "하네스(harness)"라는 말이 이상하다. 번역하면 마구(馬具)이고, 말에게 씌우는 고삐다. 왜 AI 에이전트에 이런 이름이 붙었을까. 답은 의외로 간단하다. 에이전트 혼자서는 가지 말아야 할 방향으로도 간다. 방향을 정하고, 속도를 제어하고, 필요하면 세우는 장치가 붙어야 **프로덕션에 태울** 수 있다. 조각은 10년 전부터 있었다. 프롬프트, 도구 호출, 루프, 로그. 달라진 건 조립의 대상이 LLM이라는 점뿐. 이 책은 그 조립을 **엔지니어링 규율** 안으로 끌어들이려는 시도다.

## 독자의 세 가지 물음에 미리 답하기

**Q. "Ralph Loop를 실제로 돌려본 책인가?"**
A. 예. 4장 전체가 Ralph의 진짜 메시지(plan/build 분리 + back-pressure) 복원과 적합·부적합 매트릭스다. 밈이 아니라 실패 모드 분류로.

**Q. "학술 논문 인용이 많다면 실무에 쓰기 불편하지 않은가?"**
A. 학술은 근거 계층에만 쓴다. 본문 흐름은 "실무 질문 → 실패 사례 → 근거(논문/블로그) → 실습"으로 고정돼 있다.

**Q. "한국어로 쓰인 이유는?"**
A. 독자가 한국 조직·한국 보안팀·한국 팀원과 실제로 대화하기 때문이다. 원문 인용은 영문 그대로 유지하되 설명과 사례는 한국 맥락으로 쓴다.
