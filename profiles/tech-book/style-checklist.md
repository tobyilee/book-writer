# tech-book 스타일 체크리스트

style-guardian이 tech-book 챕터 초안을 검수할 때 쓰는 항목이다. 항목별로 편차를 찾고, 구체적 수정 제안(원문 인용 → 대안)을 작성한다. 과교정 금지 — 한 챕터당 5~10건.

## 문체 항목

빈도 항목은 **하한(floor)이 아니라 자연스러운 범위(band)**로 본다. 부족함만큼이나 **과포화**도 결함이다 — 같은 장치를 기계적으로 채우면 글이 단조로워진다.

| 항목 | 확인할 것 |
|------|----------|
| 오프닝 | 메타 선언으로 시작했는가? → 상황 가정·질문 외에도 scaffolds 오프닝 메뉴(수사적 질문·충격 수치·일화·실패 장면·앞 장 콜백·정의 뒤집기)로 교체 제안. 인접 챕터와 같은 기법이면 변주 제안 |
| 청유형 빈도 | `-자`, `-보자`가 **자연스러운 수준**인가? 과포화(한 문단에 연달아·장 전체에 도배)면 일부를 평어체(`-다`/`-한다`)로 환원 제안 |
| 수사적 질문 | **논리 전환 지점에만** 있는가? 연속으로 던지거나 장식적으로 남발하지 않았는가? → 연속 사용은 통합·삭제 제안 |
| 공감 표현 | "난감하다", "찜찜하다" 등이 **챕터당 1~2회** 수준인가? 일부 챕터는 의도적으로 비워도 좋다. 매 절마다 반복되면 과포화 → 삭제 제안 |
| tic 과포화 (ceiling) | 같은 표현(그렇다면 / 기억해두자 / ~해보자 / 물론 ~다 등)이 **한 챕터에 과다**하거나 **모든 챕터에 기계적으로 반복**되는가? → 변주·삭제 제안 (한 표현이 한 챕터에 너무 자주 보이면 절반은 다른 표현·평어체로) |
| bold 강조 밀도 | `**굵게**` 강조가 과다하지 않은가? **대략 80단어당 1개를 넘으면** 강조의 효과가 사라진다 → 정말 핵심인 곳만 남기고 해제 제안 |
| 권장 어조 | "반드시 ~해야 한다" 같은 강압 표현이 있는가? → "~하는 편이 낫다"로 |
| 외래어 남용 | 불필요한 영어·일본어식 표현은? |
| 수동태 남발 | 영어식 수동태가 자주 보이는가? |
| 논리 전환구 | "그렇다면 ~", "물론 ~다. 하지만 ~" 구사가 적절한가? **과다 반복은 아닌가?** |
| 나열식 문단 | 문단이 리스트로만 채워져 있는가? |
| 호흡·길이 | 문장이 과도하게 길거나 짧지 않은가? |
| 핵심 요약 (선택) | `### 이 장의 핵심`을 쓰기로 한 책이라면, 헤딩 표기가 통일됐고(흔들림 금지) 3~5불릿이며 새 정보가 아닌 결론 압축인가? (선택 규약 — 안 쓴 장을 결함으로 보지 않는다) |
| 클로징 | 중간 장이 매번 "다음 장에서는 ~" 공식으로 닫히는가? → 변주된 이음말 제안. 마지막·종합 장은 닫는 매듭(여정 회고)인가? |

## 사실·신선도 항목 (최신 기술 주제에서 강하게)

voice.md §6 신선도 규율 위반을 잡는다. 문체 항목보다 우선 — 사실 오류는 항상 Critical.

| 항목 | 확인할 것 |
|------|----------|
| 시점 명기 | "최신/요즘"처럼 휘발하는 표현 대신 "{버전}/{연도} 기준"으로 못 박았는가? |
| 출처 있는 수치 | 벤치마크·점유율·성능 수치가 레퍼런스에 근거하는가? 없으면 `(사실 확인 필요)` 주석이 있는가? |
| 추측 API 금지 | 레퍼런스에 없는 API 시그니처·플래그·옵션명을 지어내지 않았는가? |
| 휘발성 경고 | 베타·미정 로드맵에 "빠르게 바뀔 수 있다"는 안전 문구가 있는가? |

`(사실 확인 필요)` 주석이 남아 있으면 Critical로 보고하고, 가능하면 레퍼런스 대조로 해소를 제안한다. (전용 `fact-checker`(v1.4.0~)가 style 합의 후 이 주석을 받아 레퍼런스 대조로 판정·해소한다 — style-guardian은 1차 방어선으로 누락된 주석·출처 없는 수치를 함께 잡는다.)

## 우선순위 라벨

- **Critical**: 톤이 근본적으로 벗어났거나 **사실/신선도 위반**
- **Should**: 흔들리는 지점
- **Nice**: 사소한 윤문
