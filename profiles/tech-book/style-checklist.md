# tech-book 스타일 체크리스트

style-guardian이 tech-book 챕터 초안을 검수할 때 쓰는 항목이다. 항목별로 편차를 찾고, 구체적 수정 제안(원문 인용 → 대안)을 작성한다. 과교정 금지 — 한 챕터당 5~10건.

## 문체 항목

| 항목 | 확인할 것 |
|------|----------|
| 오프닝 | 메타 선언으로 시작했는가? → 상황 가정·질문으로 교체 제안 |
| 청유형 빈도 | `-자`, `-보자`가 충분한가? 너무 강압적인가? |
| 수사적 질문 | 논리 전환 지점에 질문이 있는가? |
| 공감 표현 | "난감하다", "찜찜하다" 등이 2~3회 이상 있는가? |
| 권장 어조 | "반드시 ~해야 한다" 같은 강압 표현이 있는가? → "~하는 편이 낫다"로 |
| 외래어 남용 | 불필요한 영어·일본어식 표현은? |
| 수동태 남발 | 영어식 수동태가 자주 보이는가? |
| 논리 전환구 | "그렇다면 ~", "물론 ~다. 하지만 ~" 구사가 적절한가? |
| 나열식 문단 | 문단이 리스트로만 채워져 있는가? |
| 호흡·길이 | 문장이 과도하게 길거나 짧지 않은가? |

## 사실·신선도 항목 (최신 기술 주제에서 강하게)

voice.md §6 신선도 규율 위반을 잡는다. 문체 항목보다 우선 — 사실 오류는 항상 Critical.

| 항목 | 확인할 것 |
|------|----------|
| 시점 명기 | "최신/요즘"처럼 휘발하는 표현 대신 "{버전}/{연도} 기준"으로 못 박았는가? |
| 출처 있는 수치 | 벤치마크·점유율·성능 수치가 레퍼런스에 근거하는가? 없으면 `(사실 확인 필요)` 주석이 있는가? |
| 추측 API 금지 | 레퍼런스에 없는 API 시그니처·플래그·옵션명을 지어내지 않았는가? |
| 휘발성 경고 | 베타·미정 로드맵에 "빠르게 바뀔 수 있다"는 안전 문구가 있는가? |

`(사실 확인 필요)` 주석이 남아 있으면 Critical로 보고하고, 가능하면 레퍼런스 대조로 해소를 제안한다. (전용 fact-checker는 P2에서 분리 예정 — 그전까지 style-guardian이 1차 방어선.)

## 우선순위 라벨

- **Critical**: 톤이 근본적으로 벗어났거나 **사실/신선도 위반**
- **Should**: 흔들리는 지점
- **Nice**: 사소한 윤문
