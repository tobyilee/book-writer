### 후보 1. *"정적 타입 개발자가 만나는 TypeScript — Java·Kotlin에서 건너온 이를 위한 안내서"*

- **장점**: 독자 정체성을 정면에 세움. *"건너온 이"*의 따뜻한 뉘앙스.

### 후보 2. *"TypeScript의 진짜 모습 — Java/Kotlin 개발자가 알아야 할 JS 위의 타입 레이어"*
- **logline**: TS는 JS 위에 얹은 컴파일타임 도구다. 이 한 문장의 진짜 무게를 풀어내는 책.
- **톤**: 도발적·각성형.
- **장점**: 책의 핵심 메시지(*JS 위의 타입 레이어*)가 제목에 박힘. 1~3장 아크와 정렬.
- **단점**: 도발적이라 학습서 신뢰감이 다소 약함.

### 후보 6 (신규, **추천**). **"왜 TypeScript는 이렇게 생겼는가 — Java/Kotlin 개발자를 위한 언어와 생태계 전환서"**
- **logline**: TS가 *왜* 이런 모양인지를 알면 *어떻게* 쓸지가 자연스럽게 따라온다. Java/Kotlin 베테랑이 TS를 자기 도구로 받아들이기까지의 한 권.
- **톤**: 진지하고 묻는 자세. 답을 던지기 전에 질문을 함께 세운다.
- **포지셔닝**: *"왜 이렇게 생겼는가"*는 책 본문의 핵심 어조와 정확히 일치 — 1장(*"왜 정적 타입 개발자에게 TS를 처음부터 다시 배워야 하는가"*), 3장(*"왜 unsound인가"*), 8장(*"왜 빌드 도구가 이렇게 많은가"*)이 모두 *왜*로 시작한다.
- **부제**: *"언어와 생태계 전환서"*가 *깊이*(언어)와 *폭*(생태계)을 동시에 약속한다.

## 내러티브 아크

이 책은 **"환상 → 본성 → 도구 → 전환 → 응용 → 현장"** 여섯 마디의 호흡으로 흐른다.

먼저 **1장**에서 독자가 가지고 있을 환상을 정중하게 드러낸다. *"TS는 JS에 타입을 붙인 것"*이라는 한 줄 설명이 왜 절반만 진실인지를 *질문으로* 던지고, 이 책이 어떤 약속을 하는지를 보여준다. 다섯 핵심 모델(점진적·구조적·unsound·erased·TC39 정렬)은 이름만 호명하고, 본격 설명은 3장에 양보한다. 1장은 *답이 아니라 질문*을 던지는 챕터다.

**2~3장**은 책의 첫 번째 위기 구간이다. JS를 한 번도 깊게 본 적 없는 정적 타입 개발자에게, JS의 동적 본성(프로토타입·`this`·이벤트 루프·모듈 분열)과 그 위에 얹힌 TS의 정확한 위치를 보여준다. 3장에서 *"다섯 핵심 모델"*이 본격 풀린다 — 컴파일타임의 환상, 의도된 unsoundness, 점진적 타입의 양방향 호환, type erasure, ECMAScript 정렬. *"Java처럼 안전하다"*는 직관이 어디서 깨지는지 손에 잡힌다.

**4~5장**이 책의 *깊이* 약속을 이행하는 핵심이다. 4장은 *타입을 도구로 손에 익히는* 단계 — 추론·좁히기·discriminated union·utility 8개·strict 모드 가족. 5장(신설)은 *타입이 타입을 만드는* 단계 — 제네릭·제약·conditional·`infer`·매핑·템플릿 리터럴 타입. *"왜 Hono의 라우트 정의에서 응답 타입이 자동 추론되는가"*, *"zod의 `z.infer<typeof S>`는 어떻게 스키마에서 타입을 유도하는가"* 같은 *현실 코드의 마법*을 부품 단위로 분해한다. 곡예가 아니라 *현실에서 어떻게 쓰이는가*에 집중한다. 5장이 있어야 *"Kotlin sealed/Scala HKT를 다뤄본 사람이 TS의 표현력을 인정"*할 수 있다.

**6~9장**은 *언어 너머와 전환*이다. 6장 도메인 모델링(*"이 도메인을 어떻게 모델링할까"* — branded·enum·readonly·*에러를 도메인의 일부로*), 7장 비동기, 8장 모듈/빌드/런타임, 9장(신설) 기존 JS 코드베이스를 TS로 옮기는 마이그레이션. 9장은 *책 독자 다수가 입사 후 첫째 달에 마주칠 시나리오*다 — `allowJs`·`@ts-check`·codemod·`*.d.ts` 직접 작성·토스/카카오 사례.

**10~14장**은 응용이다. 10장 CLI(commander·oclif·Bun compile), 11장 데스크톱(Electron·Tauri 중심, 모바일은 짧게), 12장 웹 프론트엔드(컴포넌트와 reactivity의 타입), 13장 웹 백엔드+풀스택(Express·Fastify·Hono·NestJS·Next 깊이), 14장(신설) 테스트와 타입(Vitest·expect-type·Playwright). 각 영역에서 *TS가 어떤 모양으로 사는지의 패턴*을 보여준다 — 단순 카탈로그가 아니다.

**15장**은 종합이다. 한국 개발자가 자주 마주치는 함정의 큰 카테고리(상세는 부록 D), 7개 산업 논쟁의 균형 잡힌 정리, *AI 시대의 TS*(LLM이 TS를 다루는 방식, 사람이 검토해야 할 것), 다음 책·다음 주제. 책을 덮을 때 독자가 *어디로 더 갈 수 있는지*를 지도로 남긴다.

이 흐름은 단순 위계(쉬움→어려움)가 아니라 **"오해 → 정정 → 도구 → 전환 → 적용 → 현장"**의 인지적 순서를 따른다. 각 챕터는 단독으로도 읽히지만, 순서대로 읽으면 *"TS를 안다"*는 진술의 의미가 한 단계씩 깊어지도록 설계했다.

### 5장. 타입을 만드는 타입 — 제네릭·conditional·infer·매핑·템플릿 리터럴 (**신설 P0-1**)

- **Java/Kotlin 시선 박스**: ① Kotlin reified generic ↔ TS의 한계와 우회 ② Java wildcard `? extends`/`? super` ↔ TS variance

### 6장. 이 도메인을 어떻게 모델링할까 — 구조적 타입 위에서 잃지 않는 법

  2. 구조적 타입의 헐거움을 어떻게 도메인 안전하게 봉합하는가?
  3. checked exception 정신구조에서 온 사람이 TS의 throw·Result·Effect를 어떻게 받아들일까?
- **주요 내용**:
  - `interface` vs `type alias` — 언제 무엇을, 그리고 두 개가 같이 있어도 되는 이유
  - **Branded type / nominal trick** — `type UserId = string & { readonly _brand: unique symbol }`. 토스·Effect-ts·zod의 표준 패턴
  - **Kotlin value class와의 비교** — 둘 다 zero-cost wrapper지만 강제 메커니즘이 어떻게 다른가
  - **enum vs union literal** — TS enum의 함정과 union literal의 우위 (커뮤니티 휴리스틱 4)
  - **immutability** — `readonly`, `Readonly<T>`, `as const`, `Object.freeze()`의 위계와 한계
  - **에러를 도메인의 일부로 P1-5** — throw vs Result/Either 패턴(neverthrow, fp-ts, Effect-ts). *checked exception 정신구조*에서 온 Java 개발자가 TS에서 무엇을 얻고 무엇을 잃는가. boundary에서의 throw, 내부에서의 Result.
- **Java/Kotlin 시선 박스**: ① Kotlin `data class`/`copy()` ↔ TS interface + spread ② Kotlin `Result<T>`/Arrow `Either` ↔ neverthrow ③ Java `enum class` ↔ union literal

- **독자가 얻는 것**: 자기 도메인을 TS로 *안전하고 표현력 있게* 모델링하는 패턴. 에러를 *예외*가 아니라 *값*으로 다루는 선택지.

### 9장. 기존 JS 코드를 TS로 옮기기 — 점진적 도입의 패턴 (**신설 P0-3**)

- **핵심 질문**:
  1. 입사 첫째 달, *"기존 JS 코드베이스를 TS로 옮겨라"* 지시를 받으면 어디서부터 시작하는가?
  2. Kristensen-Møller의 *"60%만 자동 추론 가능"*이 뜻하는 *나머지 40%*를 어떻게 처리하는가?
- **주요 내용**:
  - **마이그레이션의 세 길** — (1) 한 번에 전부, (2) 새 파일만 TS, (3) `allowJs`/`checkJs`로 점진. 각자의 비용
  - **`allowJs`·`checkJs`·`@ts-check` 주석** — *.js 파일에서도 타입 체크. JSDoc으로 타입 표현. 토스의 *"JavaScript에서 TypeScript로 바꾸기"* 시리즈 인용
  - **strict 단계 도입 전략** — `noImplicitAny` → `strictNullChecks` → 전체 strict. *언제 무엇을 켤까*
  - **자동 codemod** — ts-migrate(Airbnb), jscodeshift, ts-morph. 도구의 한계
  - **DefinitelyTyped(`@types/*`)** — 외부 라이브러리에 타입을 입히는 정공법
  - **`*.d.ts` 직접 작성** — 라이브러리에 타입이 없을 때, 사내 모듈을 모듈 augmentation으로 보강할 때
  - **마이그레이션 중의 양면 운영** — JS·TS 혼재 코드베이스에서 PR 리뷰·CI·점진적 strict 강화 전략
  - **Kristensen-Møller (2017)의 60% 한계** — 자동화 범위와 사람의 자리
  - **한국 사례** — 토스·카카오·우아한형제들의 마이그레이션 후기 정리

## 책에서 *의도적으로* 빼는 것 (스코프 명시, v2 갱신)

이 책은 *언어와 생태계 전환서*다. 다음은 의도적으로 다루지 않는다.

1. **React 내부 동작의 깊은 분석** — Fiber 아키텍처, Suspense 내부, Concurrent 모드 알고리즘. 12장은 *"TS와 React가 만나는 면"*만 다루고, React 자체의 깊이는 별도 책에 양보.
2. **각 프레임워크의 모든 API** — NestJS·Hono·Express·Fastify의 완전 레퍼런스는 공식 문서가 더 잘한다. 책은 *"왜 이런 모양인가"*에 집중.
3. **빌드 도구 끝장 비교** — Vite vs Turbopack vs Rspack 벤치마크는 6개월마다 뒤집힌다. 8장은 *"왜 분리되어 있나"*만.
4. **타입 레벨 곡예의 끝** — 5장에서 *"현실 코드의 타입 마법 부품 분해"*까지. *문자열을 파싱하는 타입* 같은 *순수 곡예*는 *Type-Level TypeScript*에 양보.
5. ~~JS→TS 마이그레이션 실전 가이드~~ → **9장에서 본문으로 흡수 (P0-3 반영, 빼는 것에서 삭제)**
6. **CSS·디자인 시스템·접근성** — 12장에서 *컴포넌트 타이핑*만, 스타일·디자인 토큰·a11y는 양보.
7. **DevOps·배포·관측** — Docker·Kubernetes·CI·OpenTelemetry는 책 범위 밖. 부록 C에서 npm publish/Bun compile 정도.
8. **함수형 프로그래밍의 깊이** — fp-ts, Effect-ts는 6장에서 *Result/Either 패턴*과 *branded* 표준 사례로만 호명. 본격 FP는 양보.
9. **WebAssembly·Edge Runtime의 깊이** — 11장 Tauri와 13장 Hono(Cloudflare Workers)에서 호명만.
10. **모바일 네이티브 영역의 깊이** — 11장에서 React Native + Expo의 *위치*만 짧게. iOS·Android 네이티브 개발, RN 신 아키텍처(JSI/Fabric) 깊이, TurboModule codegen 실전은 별도 주제. *모바일 챕터 분리는 거부* — 책의 초점은 *언어와 생태계 전환서*이며 *모바일 앱 개발서*가 아니다 (P1-4 반영).
11. **GraphQL의 깊이** — 13장에서 *"REST 외 선택지"* 한 단락으로 호명만. GraphQL 자체는 별도.
