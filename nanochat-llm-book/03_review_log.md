# 계획 리뷰

## 종합 평가 (한 단락)

10장 구조의 *두 클라이맥스(6장 사전학습 + 9장 추론) + 의도된 골짜기(7장 평가)*는 한국어 LLM 책에서 보기 드물게 야심찬 설계고, 4장에 "관성처럼 보이던 결정들"의 출처를 모은 시도도 차별점이 명확하다. 그러나 계획은 **세 가지 거짓말**을 안고 있다 — (1) `gpt.py`를 "481줄"이라고 두 번 부르는데 실제는 **512줄**이고, 그 차이가 4장 라인 인용 범위(`:154-481`)에서 GPT class 본체의 *마지막 30줄(generate 메서드)을 통째로 누락*시키며, (2) 4장 5.5만 자 한 챕터는 "박사 학위 필요 없음" 약속과 충돌할 만큼 *과적재* 됐고, (3) 8장 CPU 실습이 약속하는 "1500 step SFT → 한 줄 답"은 레퍼런스가 README 원문을 인용해 "kindergartener level. Sometimes the model likes it if you first say Hi before you ask"라고 *이미 적어둔 현실*과 정면 충돌한다. 이 세 가지가 최우선 수정 대상이다.

## 차원별 비판

### A. 코드 커버리지

#### 약점/지적
1. **`checkpoint_manager.py`가 어느 챕터에도 명시적으로 들어가 있지 않다.** 그러나 이 파일에는 책의 약속을 떠받치는 *진짜로 운영적인* 로직이 모여 있다 — `_patch_missing_keys`가 옛 체크포인트의 `resid_lambdas`/`x0_lambdas`를 자동으로 채워주는 *아키텍처 변천 흡수*(`checkpoint_manager.py:30-40`), `find_largest_model`이 디렉토리에서 `d{N}` 패턴을 정렬하는 *모델 태그 추측 로직*, optimizer state가 *rank별로* shard로 저장되는 ZeRO-2의 물리적 흔적. 6장 또는 10장의 한 박스가 필요하다.
2. **`tasks/customjson.py`는 65줄이지만 *식별 가능한 진입점*이다.** identity 합성·자기 챗봇·새 능력 추가 모두가 이 한 파일을 통과한다. 게다가 24-32라인의 친절한 에러 메시지("HINT (Oct 21 2025)... If you recently did a git pull...")는 *진짜 카르파시 식의 사용자 친화성*을 보여주는 인용용 코드인데, 계획 8장에서 데이터 믹스만 언급되고 *이 파일이 무엇이고 왜 거기 있는지*는 빠져 있다.
3. **`nanochat/common.py`의 `get_dist_info`·`compute_init_lr_frac`·`COMPUTE_DTYPE` 자동선택 분기**가 계획서에는 *어디에도 명시되지 않았다*. 레퍼런스 §2.2의 첫 항목으로 "컴퓨트 환경의 단일 진실"이라 부른 모듈인데, 4장 init 직전이나 6장 train 루프 도입에서 한 단락 박스로 다루지 않으면 *왜 H100/A100/CPU/MPS에서 같은 코드가 다르게 도는가*에 대한 코드적 답이 책에 없다.
4. **`nanochat/dataset.py`의 parquet shard 다운로드·retry·resume** — 1장 실습이 `python -m nanochat.dataset -n 2`로 시작하는데 이 모듈이 "무엇을 다운받고 어디에 두는지"가 *서술된 곳이 없다.* 부록 B 트러블슈팅("첫 다운로드 끊김 시 resume")만으로는 1장 첫 실습의 *기대*를 만들지 못한다.

#### 권고
- **4장 라인 인용을 다시 검증한다.** `gpt.py`는 512줄이고, GPT class는 154-512. 계획서의 `:154-481`은 generate 메서드(484-512)를 *통째로 누락*시킨다. 추론용 generate가 비록 9장에서 다시 나오긴 하지만, 4장에서 forward 끝까지 보여놓고 generate를 안 보이면 *한 파일 안에서의 책임 분할*이 잘 안 보인다. 라인 범위를 `:154-512`(전체) 또는 `:154-481(forward까지) + :483-512(generate)` 두 인용으로 갈라라.
- **6장 "체크포인트 한 박스"를 명시한다.** `_patch_missing_keys`의 *옛 체크포인트 호환성*과 optimizer rank shard 패턴은 6장이 *재현성에 대한 책*이라는 정체성을 강화한다. 책에서 "어느 챕터에서 진짜 ML 운영을 가르치는가"의 답.
- **8장 첫 단락에 `customjson.py`를 명시한다.** "어디에 내 데이터를 끼우는가"에 대한 *한 줄 짜리 답*. 그 친절한 에러 메시지를 그대로 인용하면 책의 톤과도 어울린다.
- **1장 실습 직전에 `dataset.py` 한 문단**을 박는다. "셰드 한 개가 약 100MB, 다운로드는 ~30초, 2개면 약 1분"이라는 *측정값*을 한 줄 추가하면 독자가 환경 점검을 *진짜로* 할 수 있다.

### B. 챕터 균형과 흐름

#### 약점/지적
1. **4장 5.5만 자는 한 챕터로 못 읽힌다.** 한국어 IT 서적 평균 챕터는 2~3만 자이고 5.5만 자는 *책 한 권의 1/8을 한 챕터에 묻는 결정*이다. 더 큰 문제는 *밀도*다 — RoPE / RMSNorm / QK norm / ReLU² / value embeddings / per-layer scalars / smear / backout / sliding window / untied embedding / softcap / init_weights / estimate_flops 13개 결정이 한 챕터에 들어 있다. 독자가 4장 절반쯤에서 *어디까지 읽었는지 기억하지 못하는* 챕터다.
2. **4장 → 5장은 *두 번째 가파른 오르막*인데 사이에 휴식이 없다.** "외과의의 집중"(4장) → "약간의 경외"(5장)는 톤 변화가 있긴 하지만, 4장에서 *13개 트릭의 외과 수술*을 끝낸 독자에게 Newton-Schulz quintic polynomial을 *바로 다음 챕터*에서 던지면 계획이 약속한 "통제감 → 친숙함" 곡선이 *현실에선 통제감 → 다시 통제 잃음*이 된다.
3. **1장 2만 자는 *너무 가볍다*.** "환경 구축 + 코드 둘러보기 + 두 가지 읽기 방식"이 2만 자에 다 들어가면 한 항목당 약 3천 자 — 워밍업이 아니라 *목차 나열*에 가깝다. 2장의 BPE 충격이 *aha*가 되려면 1장이 더 두꺼워야 한다.
4. **RL이 8장 끝에 끼어 있다.** 8장은 이미 *Midtraining 소멸·SFT 데이터 믹스·손실 마스킹·bestfit-pad·ChatCORE·Identity·SpellingBee*에 4.5만 자를 쓴다. 마지막에 *REINFORCE-ish "GRPO"·token-level normalization·pass@k*를 더 얹는 것은 한 챕터 안에 *두 개의 학습 패러다임*(SL vs RL)을 욱여넣는 일이다. 게다가 계획서가 정해놓은 *클라이맥스 직전*인데, RL은 reward signal과 advantage 개념 도입에 적어도 만 자를 쓰는 *느린* 주제다.

#### 권고
- **4장을 두 챕터로 쪼갠다 (가장 강한 권고).** 4A장 "GPT 모듈의 골격 — embed/forward/init/FLOPs"(약 3만 자, 7개 결정: GPTConfig·forward 흐름·RoPE·RMSNorm·QK norm·MLP·init_weights·estimate_flops) + 4B장 "modded-nanoGPT가 더한 7가지 트릭"(약 2.5만 자: value embeddings·per-layer scalars·smear·backout·sliding window·untied embedding·softcap). 4A는 *외과 해부*, 4B는 *현대화 패치 노트*로 톤이 자연스럽게 갈라진다. 이렇게 하면 5장이 새 챕터(=새 호흡) 직후에 와서 *오르막의 인접성* 문제도 같이 해소된다.
- **만약 한 챕터를 유지한다면**, 4장 중간에 *명시적 호흡 박스* 2개를 박아라. "여기까지 6개 결정. 차 한 잔 마시고 와도 좋다"라는 토비 식의 *친구 같은* 짧은 단락. 이건 5.5만 자 한 챕터를 살리는 *유일한* 스타일적 장치다.
- **1장을 3만 자로 늘리고 한 절을 *손에 잡히는 코드 산책*에 쓴다.** 예: `from nanochat.tokenizer import get_tokenizer; tok = get_tokenizer(); print(tok.encode_special_tokens())` 같은 5줄짜리 *맛보기* 인터랙션을 1장 마지막에 넣어 2장 *aha*의 사다리를 만든다.
- **RL을 8.5장으로 분리한다.** 약 2만 자 짜리 *얇은 챕터*. 8장은 "능력은 데이터로 가르친다"의 *순수한 완성*으로 두고, 8.5장에서 "그렇다면 *추가로* 강화학습은 무엇을 더하는가"를 별도 호흡으로 다룬다. 이렇게 하면 8장이 4.5 → 3만 자로 줄어들고 RL이 자기 *현존*을 갖는다. 9장 클라이맥스 직전 *추가 봉우리*가 생겨 곡선도 풍성해진다.

### C. 대상 독자 적합성

#### 약점/지적
1. **"PyTorch는 쓰지만 LLM 내부는 모름" 가정이 4장 후반에서 깨진다.** value embeddings의 `has_ve(layer_idx, n_layer)` 분기, 12채널 `ve_gate` × `3 * sigmoid`, backout의 "n_layer//2 위치에서 cache 후 빼기"는 *왜 그렇게 해야 하는지*가 직관이 잘 안 잡히는 결정들이다. 계획서는 "near-zero FLOPs로 150M params"라는 *Karpathy 인용*만 제시하는데, 이건 *권위 인용*일 뿐 *이해의 사다리*가 아니다.
2. **5장 Polar Express 5-iteration quintic coefficient는 *수식 없이* 설명 불가능하다.** 계획서는 "시각적 그림과 작은 실습으로 풀어준다"라고 적었지만, `optim.py:82-89`의 coefficient(`(3.49414, -5.91992, 3.42578, ...)`)는 *Newton-Schulz iteration의 단조 수렴 보장*에서 나온 수치다. 수식을 한 줄도 안 쓰면 "이 숫자는 마법이다"로 끝난다 — 그건 책의 약속과 정반대.
3. **8장 ChatCORE 정의가 *처음 보는 독자*에게 자연스럽게 흘러가지 않는다.** 7장 CORE는 *centered + 22 task 평균*인데 8장 ChatCORE는 *centered + 6 task 평균 + baseline 각각 다름*({0.25, 0.25, 0.25, 0.0, 0.0, 0.0}). "같은 단어가 다른 정의" 트릭이 7장-8장 사이에 *경고 없이* 일어나면 독자가 헷갈린다.
4. **"박사 학위 필요 없음" 약속이 가장 위태로워지는 지점은 *5장 + 6장 사이*다.** 4장의 모델 디테일은 시각적 그림과 코드로 견딜 수 있지만, 5장의 Muon 직관 → 6장 Power Lines `Bopt ∝ D^0.383` → T_epoch framework weight decay scaling은 *세 개의 다른 논문*에서 나온 *세 개의 보정 공식*을 연달아 던진다.

#### 권고
- **4장 후반 4개 결정(value embeddings·smear·backout·per-layer scalars)에 *한 줄짜리 직관 메타포*를 붙인다.** 예: "value embeddings는 *attention의 메모장에 같은 단어를 다시 한 번 적어두는 것*". 메타포가 부정확해도 *그 단계에서 견디는* 사다리는 만들어준다. 또는 *이 4개를 4B장(쪼개기 권고대로 분리할 경우)에 모아두고* "이건 직관이 잘 안 잡힌다는 걸 먼저 인정하자"로 시작한다.
- **5장 Muon 챕터에 단 한 줄 수식을 *허용*한다.** `Newton-Schulz: X_{n+1} = a·X_n + b·X_n·X_n^T·X_n + ...` 한 줄. 그 한 줄이 5-iteration quintic의 *coefficient를 왜 튜닝하는지*에 대한 *이해 가능한* 답이다. "수식 없음" 약속이 5장에서만 양보된다는 사실은 서문에서 미리 *예고*하면 약속 자체가 더 정직해진다.
- **7장과 8장 사이에 *반 페이지짜리 다리*를 박는다.** "CORE는 base 모델용 22 task의 평균이고, ChatCORE는 chat 모델용 6 task의 *baseline-보정된* 평균이다. 같은 *centered* 트릭, 다른 task 모음, 다른 baseline." 두 정의를 한 표로 나란히.
- **6장 scaling laws 자동 결정 한 단락을 *명시적으로 박스 처리*한다.** 본문 흐름에서 빼서 "공식 셋, 한눈에"라는 박스. 박스 안에는 (1) Chinchilla 20→nanochat 12 / speedrun 8 (2) Power Lines `B ∝ D^0.383` (3) T_epoch WD scaling. 박스 밖 본문에서는 *어떤 효과가 어떤 그래프로 보이는지*만. 이렇게 하면 *세 공식이 한 단락에 몰리는* 압도감이 줄어든다.

### D. 실습 분배

#### 약점/지적
1. **4장 실습 "d4 meta init → 파라미터 수와 estimate_flops 출력 → forward 통과 확인"은 *추상적이고 내가 만든 것이 안 남는다*.** 5.5만 자 챕터 끝의 실습이 "숫자 두 개 프린트하고 forward 한 번 통과"라면 독자에게 *4장이 끝났다는 감각*이 없다. 게다가 d4는 너무 작아 *forward 결과를 디코딩해도 의미 없는 토큰열*이 나온다.
2. **6장 GPU 실습 "8XH100 3시간 ~$48"는 *대다수 한국 독자에게 비현실적*이다.** Lambda Cloud는 한국 IP에서 결제가 막힐 수 있고, 한국 대다수 독자는 RTX 4090 한 장 또는 A100 시간제(국내 vast.ai 비싸짐) 환경이다. "선택"이라고 적었지만 *책의 가장 짜릿한 순간*을 옵션으로 두면 6장 클라이맥스가 *책 안에서만의 클라이맥스*에 머문다.
3. **8장 실습 "d6 SFT 1500 steps → '한 줄 답'"는 레퍼런스가 *이미 적어둔* README 원문과 충돌한다.** 레퍼런스 §4.1: *"It might even know that the color of the sky is blue. Sometimes the model likes it if you first say Hi before you ask it questions."* d6+5K pretrain+1500 SFT step의 *현실*은 "운이 좋으면 Paris를 말할 수도 있다"인데, 계획서는 "What is the capital of France? → 한 줄 답"이라 *결과를 약속*한다. 만약 모델이 그 답을 못 뱉으면 *책이 거짓말한 것*이다.
4. **9장 실습 "9 곱하기 8은? → calculator 호출 토큰"이 *d6 CPU 모델로 작동하는가*가 검증되지 않았다.** Tool use FSM은 SFT가 GSM8K 4 epochs를 충분히 학습해야 *모델이* `<|python_start|>`를 sample할 줄 알게 된다. d6+1500 step에선 reliably 안 나올 가능성이 높다.

#### 권고
- **4장 실습을 *세 단계로 분해*한다.** (a) d4 meta init → param 수·FLOPs 출력 (현 계획) → (b) `window_pattern`을 "SSSL" vs "L"로 바꿔 attention mask shape 비교 → (c) **`init_weights`의 std=0.8 임베딩을 std=0.1로 바꿔보고 forward output 분포 확인**. 마지막 (c)가 *내가 만든 것*에 가깝다 — *초기화의 영향*을 시각적으로 본다.
- **6장 GPU 실습에 *현실적인 대안 사다리*를 명시한다.** "8XH100 3시간 $48" 옆에 "A100 1장 24시간 ~$24 (d12 미니 런, val_bpb는 1.0 근처)" 또는 "RTX 4090 한 장 + d8 + 12시간"의 *기대 결과 수치*까지 적어준다. 이게 한국 독자에게 "내가 정말 한 번 굴려볼 수 있는가"의 답을 준다.
- **8장 실습 약속을 *솔직하게 낮춘다*.** "1500 SFT step 후 모델은 *대체로* Paris를 안다. 답이 한 단어가 아니라 한 문장이거나 *살짝 어긋난다*. 그게 정상이다"라고 *기대치 조정*을 본문에 *명시*한다. 그리고 *진짜 한 줄 답*을 원하면 *d12 8K step* 또는 *speedrun 모델 다운로드* 옵션을 제공한다. 책이 *현실보다 빛나게 쓰여진* 순간이 9장 클라이맥스의 정직함을 망친다.
- **9장 calculator 데모는 *speedrun d24 SFT 체크포인트로*만 가능하다는 사실을 명시한다.** d6 1500 step에서는 *대화 자체*가 안정적이지 않다. "calculator 호출이 나오려면 충분한 SFT가 필요하다"는 *기술적 사실*을 박스로. 그리고 카르파시가 공개한 sample checkpoint를 다운받아 띄우는 *진짜 대화 가능한* 대안 경로를 제시한다.

### E. 한국어 독자 특수성

#### 약점/지적
1. **부록 D 5천 자는 *너무 짧다*.** 한국어 토크나이저 재학습은 *세 가지 다른 작업*을 요한다 — (1) AI-Hub/KoCommonCrawl 데이터 정제 (2) `tok_train.py`의 `--max-chars` 조정 + special token 호환성 (3) `tok_eval.py`에서 한국어/영어/코드 ratio 비교. 한 작업당 1.5천 자면 *각 단계의 손에 잡히는 실습*은 불가능. 본 책 9~10장 분량을 차지할 만한 주제를 *부록*에 욱여넣었다.
2. **2장의 "한국어 토크나이저 예시" 약속이 *얼마나 충실한가*가 모호하다.** 계획서는 "한국어 토큰화 결과를 보여준다"인데, 실제 32K vocab on FineWeb으로 학습된 토크나이저는 *한국어를 거의 byte 단위로* 쪼갠다 (레퍼런스 §6.3: 한국어 ratio 2-3×). 이걸 *aha* 순간으로 만들 수도, *실망*으로 만들 수도 있다. 계획서가 어느 쪽을 의도하는지 *명시되지 않았다*.
3. **9장 챗봇 데모에서 *한국어 질문*에 대한 약속이 없다.** 책 표지에 "한국어 개발자가 nanochat을 자기 LLM으로 돌린다"는 약속이 깔려 있는데, 9장 실습은 "9 곱하기 8은?"이라 *영어 모델에 한국어 질문*을 하는 *기대치 어긋남*을 어떻게 해결하는지 안 나온다.

#### 권고
- **부록 D를 *부록 D + 부록 E*로 쪼개거나, 부록 D를 1.2만 자로 늘린다.** 또는 *10장 마지막 한 절(약 5천 자)을 "한국어로 가는 길"*에 할당해서 부록과 본문 양쪽에 *짝을 이루게* 한다. 한국어 독자의 정서적 *마무리*가 본문 안에 있어야 한다.
- **2장에 "*같은 한국어 문장이 GPT-2·GPT-4·우리 32K로 어떻게 쪼개지는가*" 비교 표를 *반드시* 넣는다.** 예: "안녕하세요, 저는 토비입니다"가 GPT-2에선 18 토큰, GPT-4에선 8 토큰, nanochat 32K에선 22 토큰. 이 *숫자 자체*가 한국어 독자의 *aha*를 만든다 — 그리고 "왜 한국어가 비싼가"의 답이 자연스럽게 흘러간다.
- **9장 챗봇 데모에 *한국어 질문에 대한 정직한 대답*을 한 박스로.** "한국어로 물어보면 영어로 답하거나 깨진다. 그게 nanochat의 학습 데이터 분포다. 10장 마지막에서 *이걸 어떻게 고치는지* 다룬다." 정직한 박스 하나가 *책의 약속을 지킨다*.

### F. 토비 스타일 가능성

#### 약점/지적
1. **"코드 한 줄씩 따라가기" × "평어체·청유형·수사적 질문"의 *근본적 충돌*.** 4장 13개 결정·5장 Muon coefficient·6장 scaling law 세 공식이 한 챕터 안에서 *코드 인용 빈도*를 높이면, 평어체의 *호흡*이 끊어진다. "*~을 보자*", "*~지 않은가*" 같은 청유형·수사적 질문은 *서술의 흐름*에 의존하는 장치인데, `gpt.py:285-312`를 통째로 인용한 직후엔 그 흐름이 사라진다.
2. **7장 "차분한 회의" 톤이 *3만 자로 유지될 수 있는가*가 의심스럽다.** 평가 챕터는 본질적으로 *정의·공식·표·숫자*가 빽빽한 챕터다. 토비 스타일은 *공감·정서·수사적 질문*에 강한데, "centered accuracy의 정의"를 평어체 청유형으로 풀면 *지루*하거나 *어색*하다. 차라리 이 챕터는 *짧고 농밀한* 1.8만 자가 적당하다.

#### 권고
- **계획서 *서문*에 "*이 책의 코드 인용 규칙*"을 한 절로 명시한다.** "5줄 이하는 본문 흐름에, 5~20줄은 박스로, 20줄 초과는 *코드 펴기 신호*(파일 경로만 한 줄)로 표시한다." 이 규칙이 토비 스타일의 *호흡*을 보호한다. 그리고 4장은 *코드 펴기 신호*가 가장 자주 등장하는 챕터가 된다 — 즉 본문은 *해설 중심*, 코드는 *워크트리에서 펴서 본다*.
- **7장의 정서적 톤을 "차분한 회의"가 아니라 *"숫자를 의심하는 차분한 회의 + 한국어로 의역되지 않는 두 단어"*로 명시한다.** 즉 7장은 *영어 용어를 그대로 두는* 챕터다 (CORE·centered·bpb 같은). 한국어 의역의 압력을 줄이면 평가 챕터의 *건조함*이 *정밀함*으로 읽힌다. 이건 토비 스타일의 *예외적인 사용*으로 처음 등장한다.
- **9장 클라이맥스는 *토비 스타일이 가장 자연스럽게 폭발*한다 (계획대로).** 다만 한 가지 — 클라이맥스의 마지막 단락은 *수사적 질문으로 닫지 말라*. "어떤가, 우리가 만든 모델이 답해주지 않는가?" 같은 닫음은 *과도*다. 차라리 *문장 하나*로 닫는다: "*72라고 답한다.*" 이게 토비 식의 *함축된 흥분*.

### G. 내러티브 곡선

#### 약점/지적
1. **"두 클라이맥스 + 골짜기" 구조 자체는 좋지만, 7장이 *너무 짧다*면 골짜기가 *눈에 안 보인다*.** 3만 자는 6장(5만 자)에 비해 60% 분량 — 의도된 골짜기치고는 *너무 깊지 않다*. 골짜기가 *깊어야* 9장이 *높이* 보인다.
2. **10장 회고가 *새 질문*으로 닫히지 않는다.** 계획서는 "*우리가 처음에 던졌던 질문에 답해본다*"로 닫는다. 그건 *닫힌 원*이다. 책의 *여운*은 *열린 질문*에서 나온다.
3. **"3개 봉우리(4·6·9)"인지 "2개 봉우리(6·9)"인지가 *계획서 그래프*와 본문 설명에서 불일치한다.** 본문에서 "4장이 가장 가파른 오르막"이고 "6장이 전반부 클라이맥스"라고 적었는데, 그래프에선 4장이 6장보다 낮다. 4장은 *오르막*이지 *봉우리*가 아니다. 만약 4장이 봉우리(달성감)면 책 곡선이 *3봉우리*인데, 그 경우 *어떤 봉우리가 어떤 봉우리를 누르는가*가 모호하다.

#### 권고
- **7장을 2.5만 자로 *더 짧게* 만든다.** 골짜기는 *깊고 짧을수록* 직후의 봉우리가 높아진다. 평가 챕터에 들어가는 *모든 정의*를 7장 본문 안에 다 넣으려 하지 말고, "공식 사전"을 부록 C(용어집)에 일부 이관한다. 7장 본문은 *왜 그 정의가 그 모양인가*만 다룬다.
- **10장 마지막을 *3단계 닫음*으로 다시 구성한다.** (a) 처음 질문에 답하는 단락 (계획대로) → (b) *그런데 이 책에서 답하지 못한 세 가지*를 한 박스로 (multimodal, agent loop, 1조 토큰 학습) → (c) 마지막 한 줄은 *읽는 사람의 다음 행동을 가리키는 청유형*: "*당신의 nanochat을 한 번 만들어 두자.*" 이게 *열린 닫음*이다.
- **그래프를 명시적으로 *3봉우리*로 다시 그린다 — 또는 *4장은 봉우리가 아닌 오르막*임을 본문에서 *한 줄로* 명시한다.** 후자가 더 정직하다: "4장은 책에서 가장 가파른 오르막이지만 *봉우리는 아니다*. 봉우리는 6장과 9장이다." 이 한 줄이 계획의 모호함을 없앤다.

### H. 누락된 토픽

#### 약점/지적
1. **`runs/scaling_laws.sh`와 `runs/miniseries.sh`가 10장에서 *한 줄 언급*으로 끝난다.** 미니시리즈는 *같은 코드로 d4부터 d26까지 자동으로 다른 모델을 만드는 nanochat의 핵심 사상*인데, 10장 절 3번에서 `--depth` 다이얼 한 단락에 묻혀 있다. 차라리 10장 안에 *반 페이지짜리 표*("d4 → d8 → d12 → d24의 wall-clock·val_bpb·CORE")를 박는다.
2. **"에러 메시지를 읽는 법" 메타 챕터가 *부록 B 트러블슈팅 안에 묻혀 있다*.** 한국어 독자에게 *진짜 도움 되는* 것은 PyTorch/CUDA 영어 에러 메시지의 *어디를 먼저 보는가*인데, 부록 B는 OOM·dtype·분산 같은 *구체적 시나리오*만 다룬다. 더 위(예: 1장 부록 또는 1.5장 식 짧은 별책)에 *에러 메시지 읽기*가 필요하다.
3. **`nanochat/report.py`가 10장의 *첫 절*인데 *실제 출력 예시*가 없다.** 계획서는 "markdown 자동 보고서"라고만 적었는데, `report.md`의 *실제 한 페이지*(헤더 + Summary 테이블)를 책에 *그대로 인용*하는 것이 *재현성의 증거*를 보여주는 가장 강한 방법이다.
4. **"내가 학습한 모델을 공유하는 법"이 *책 어디에도 없다*.** 10장 자기 챗봇 만들기 후에 *그 모델을 HF Hub에 올리거나 친구에게 보내는* 한 단락이 *책의 마지막 행동*으로 자연스럽다 — `chat_web` 데모를 *남에게 보여주는* 단계.

#### 권고
- **10장 절 3번을 *한 절 → 두 절*로 확장한다.** "10.3.1 `--depth` 다이얼", "10.3.2 미니시리즈 표 한 페이지". 후자는 표 + 한 단락.
- **부록 B를 *부록 B-1(에러 읽기) + B-2(트러블슈팅 시나리오)*로 쪼갠다.** B-1은 *2천 자짜리 메타 가이드* — "CUDA OOM 메시지의 두 줄 어디를 먼저 보는가", "torch.compile fall-back 경고 무시해도 되는가" 같은 *한국어 독자가 멈추는 지점*. 부록이 *길이*가 아니라 *진입 장벽 해소*로 평가받게.
- **10장 첫 절에 `report.md` 한 페이지를 *통째로 인용*한다.** 그 페이지가 *책의 마지막 증거*다.
- **10장 마지막 절에 "*공유*"를 한 단락으로 추가한다.** HF Hub 업로드 명령 한 줄 + `chat_web`을 ngrok으로 외부 노출하는 한 줄. *책 너머의 행동*까지가 책의 마무리다.

### I. 제목과 슬러그

#### 약점/지적
1. **「나노챗 해부학」은 마케팅적으로 *전문가용*으로 들린다.** "해부학"이라는 단어는 *의대생의 첫 학기 교재* 분위기다 — 검색은 잘 안 잡힌다. 일반 개발자가 "ChatGPT 만들기 책"으로 검색했을 때 *이 책이 안 나온다*. 부제에 "카르파시"를 박았지만 *그 이름이 한국 일반 개발자에게 얼마나 검색 키워드인지*가 불투명.
2. **후보 C "$100"이 빠지면서 *측정 가능한 약속*이 표지에서 사라졌다.** 후보 A는 "8천 줄"이라는 *분량 약속*은 있지만 *결과 약속*이 없다. 일반 개발자가 책을 *집는 이유*는 "내가 무엇을 얻는가"의 직관인데, 후보 A는 그 직관이 *분량으로만 표현됐다*.
3. **슬러그 두 개(`nanochat-llm-book` 작업용 + `nanochat-anatomy` 최종) 운영은 *드리프트 위험*이다.** 매니페스트·EPUB·표지·산출 폴더 어디서 어느 슬러그가 쓰이는지 *명시적 표*가 없으면 빌드 단계에서 어긋난다.

#### 권고
- **후보 A를 유지하되 *부제를 강화*한다.** "카르파시의 8천 줄로 배우는 LLM 학습 파이프라인" 대신 "**한 GPU 노드 3시간, $100로 ChatGPT를 만든다 — 카르파시 nanochat 코드 해부**". 부제에 *결과 약속*과 *분량 약속*과 *원작자 이름*이 모두 들어간다. "$100"이 책의 *측정 가능한 약속*으로 돌아온다.
- **또는 후보 A vs C를 *A/B 테스트*로 제안한다.** 표지 시안을 두 개 만들어 SNS에 *클릭률*을 측정. 최종 결정은 데이터로. 계획서 자체에 "*최종 제목은 표지 A/B 테스트로 결정*"이라 적으면 *너무 빨리 확정하지 않는* 정직함이 된다.
- **슬러그 운영 규칙을 *한 표로 명시*한다.** "작업 디렉토리: `nanochat-llm-book/`, EPUB 파일명: `나노챗-해부학-v0.1.0.epub`, 슬러그 매핑은 `book_manifest.json`의 `slug` 필드". 이 표를 계획서 §6 뒤에 박는다.

### J. 분량 추정

#### 약점/지적
1. **38~45만 자는 한국어 IT 서적으로 *상위 5% 두께*다.** 일반적인 한국어 기술서는 25~32만 자, 320페이지 안팎. 45만 자는 *400페이지 + 부록*인데 *처음 출간하는 책의 분량*으로는 *너무 야심차다*. 책의 *완성 위험*이 분량에서 시작된다.
2. **4장 5.5만 자는 *한 챕터로 읽힐 수 있는가*가 의심스럽다 (B.1과 중복).** 한국어 독자의 평균 챕터 읽기 시간은 약 1.5시간. 5.5만 자는 3시간이 넘는다. *한 자리 앉아서 읽지 못하는 챕터*는 책의 *흐름*을 끊는다.
3. **부록 3개 합쳐 3.8만 자**가 계획에 잡혀 있는데, 부록 B(1.5만)·C(1.5만)는 정당하지만 *부록 A 8천 자(디렉토리 맵)*는 *너무 짧고 흩어진다*. 디렉토리 맵은 *책의 첫 페이지 또는 면지*에 한 페이지 그림으로 들어가는 것이 자연스럽다.

#### 권고
- **목표 분량을 *32~38만 자*로 *낮춘다*.** 4장을 4A+4B로 쪼개면 *각각 3만 자대*가 되고, 7장을 2.5만 자로, 8장을 8장+8.5장으로 쪼개면 *전체 분량 압박*이 자연스럽게 해소된다. *38만 자가 한국어 IT 책의 최대치*라는 *암묵적 합의*에 맞추는 것이 *실현 가능성*을 높인다.
- **4장 한 챕터 = 한 자리 + 한 박스(20분 휴식)*로 *명시적 설계*.** 챕터 머리말에 "*이 챕터는 두 자리에 나눠 읽기를 권한다*"라는 *한 줄*을 박는다. 토비 스타일에 자연스럽고, *독자에게 정직한 안내*다.
- **부록 A 8천 자를 *책의 앞 면지 한 페이지 + 부록 A 4천 자*로 쪼갠다.** 면지의 한 페이지 그림이 *책 전체를 들고 다닐 수 있는 지도*다. 부록 A는 *각 모듈의 한 문장 요약*만 남겨 *짧고 검색용*으로.

## 우선순위 권고 (Top 5)

1. **4장을 4A + 4B 두 챕터로 쪼갠다.** 5.5만 자 한 챕터는 *책의 완성 위험*이자 *독자의 포기 위험*. 4A(GPT 골격·RoPE·RMSNorm·QK norm·ReLU²·init·FLOPs)와 4B(modded-nanoGPT 7가지 트릭)로 갈라야 5장 직전에 *새 호흡*이 생긴다. → B.1, J.2 동시 해결.
2. **`gpt.py:154-481` 라인 인용을 *전 코드 검증을 거쳐 다시* 수정한다.** 실제 파일은 512줄이고 generate 메서드(484-512)가 통째로 빠져 있다. 계획서가 *두 번* "481줄"이라 부르는데 그게 *책의 핵심 카피*라 *틀리면 안 되는 숫자*다. → A.1 권고.
3. **8장 CPU 실습의 결과 약속을 *솔직하게 낮춘다*.** "1500 SFT → 한 줄 답"이 *현실에선 안 나올 가능성이 높다*. 레퍼런스가 README 원문을 *이미 인용*해 둔 *기대치*를 본문에 *명시*해야 책의 정직함이 지켜진다. → D.3.
4. **RL을 8.5장으로 분리하고 7장을 2.5만 자로 줄여 *골짜기를 더 깊게*.** SFT(8)와 RL(8.5)을 갈라 *학습 패러다임 차이*가 책에서 *분명*해지고, 7장을 줄여 *6→7→8→8.5→9 곡선*이 *더 큰 진폭*을 갖는다. → B.4, G.1.
5. **2장에 "*같은 한국어 문장의 세 토크나이저 비교 표*"를 *반드시* 박는다.** 한국어 독자의 *aha*가 책 초반에 일어나야 *책의 약속*이 본문에서 *지켜진다*. 부록 D(한국어)에 묻으면 *너무 늦다*. → E.2.

## 반영 불필요 (Top 3)

1. **"제목 「나노챗 해부학」이 마케팅적으로 약하다"는 권고 (I.1)는 무시해도 된다.** 책의 *진짜 차별점*은 "8천 줄 한 작품을 끝까지 따라간다"이고, "해부학"이라는 단어가 *그 차별점에 가장 가까운 메타포*다. 검색 노출은 *부제와 SEO*로 보완하면 되고, *책 자체의 정체성*은 제목에서 흔들리지 말아야 한다. 후보 C의 "$100"은 *너무 짧고 좁은 약속*이라 8~10장의 정렬·자기 챗봇 챕터를 가린다 — 계획서가 이미 옳게 판단했다.
2. **"5장 Muon 챕터에 수식 한 줄을 *허용*한다"는 권고 (C.2)는 *조건부*로만 반영하면 된다.** 만약 토비 스타일을 *엄격히* 지키고 싶다면 수식 대신 *Newton-Schulz의 한 step을 시각화한 작은 numpy 코드*(1024×1024 random matrix → 5번 iteration → SVD와의 거리 곡선)로 대체할 수 있다. 코드는 본 책의 *기본 언어*이고, 수식은 *예외적 언어*다. 어느 쪽이든 *직관 사다리*만 만들어주면 된다.
3. **"10장 마지막을 *새 질문*으로 닫는다"는 권고 (G.2)는 *과도*할 수 있다.** 토비 스타일의 *다정한 작별*은 *닫힌 원*에 더 어울린다. "*당신의 nanochat을 만들어 두자*"라는 *청유형 한 줄*이면 *열린 닫음*이 *과도하지 않게* 달성된다 — 굳이 "*그런데 이 책에서 답하지 못한 세 가지*" 박스를 만들 필요는 없다. 그건 *학술서의 한계 절*이지 *동반자적 에세이*가 아니다.

## 합의 가능 여부

**2회 왕복이 필요하다.** 1회 갱신으로 충분한 항목들(라인 인용 수정, customjson 명시, 한국어 비교 표 추가, RL 8.5 분리, 부록 D 확장)이 있지만, **4장 쪼개기 결정**은 *책 전체 구조와 정서적 곡선 그래프*를 다시 그려야 하는 *구조 변경*이다. book-planner가 4장을 *유지하는* 결단을 내릴 수도 있고(예: "4장 한 챕터에 휴식 박스 2개로 견딘다"), 그 경우 그래프와 챕터당 분량 재산정만 하면 된다. 두 선택지 모두 *합리적*이므로, 라운드 2에서 *어느 쪽을 선택했고 그 선택이 곡선과 분량에 어떻게 반영됐는지*를 확인하면 종결할 수 있다.

8장 실습 약속 낮추기·9장 calculator 데모의 검증 같은 *기술적 사실 충돌*은 *반드시 반영*되어야 하므로, book-planner가 *반영 + 그 사실을 본문에 어떻게 정직하게 노출했는지*를 라운드 2에서 보면 된다.
