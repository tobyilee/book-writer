## 2. 라운드 1 — 개정 요구 (plan-reviewer → book-planner)

5개 Critical + 3개 Should + 2개 Nice-to-have. 핵심:

### Critical
- **[C1]** 3부 케이스 +1 추가 — "스트리밍·협업·글로벌 분산 (Netflix·Figma·Spanner)" 또는 "실시간 협업 단독". 5개 케이스 중 3개가 트랜잭션 도메인으로 겹쳐 다채로움 부족. reference §4.5·§4.11·§4.12의 풍부함이 본문 1단락으로만 등장.
- **[C2]** 운영 챕터 분리 — 13장 22p가 D축 전체를 흡수. 옵션 A(13장 14p + 신설 SLO·관측성·on-call·배포 16p) 또는 옵션 B(케이스 챕터에 운영 sidebar 4p 의무화). **옵션 A 추천.**
- **[C3]** 보안 별도 챕터 신설 — "분산 시스템 보안·인증·인가·secret·망분리" 16p. reference §9가 "다음 책의 주제"라고 한 건 리서치 한계이지 책 구성의 정당성은 아님.
- **[C4]** 14·15장(CRDT·파이프라인)을 3부 케이스로 연결 — 14장 CRDT를 신설 협업 케이스 챕터에 흡수, 15장 파이프라인을 3부의 "광고·추천·검색 indexing" 케이스로 격상.
- **[C5]** 부록 LLM 1페이지를 본문 챕터로 격상 — "AI 시대의 시스템 디자인" 16p 마무리 챕터. §5 5가지 약속의 LLM 시대 재회수.

### Should
- **[S1]** 0장 hook 다시 — "김 모씨" 가상 인물 → "~라고 해보자" 직접 상황 가정.
- **[S2]** 한국 4·6·10 활용 강화 — 16장 카카오톡 fan-out sidebar, 1장 또는 운영 챕터에 JPA N+1 sidebar, 0장에 "이력서용 기술 도입의 함정" 3p 섹션.
- **[S3]** 11장 합의 hook에 Spanner 사례 보강.

### Nice-to-have
- **[N1]** 8장 hook을 Cloudflare 2017 leap second 또는 Stripe 분산 ID 충돌 사건으로 교체.
- **[N2]** 챕터 끝 "약속 회수" 카드 양식 통일.

권장 분량: 23 챕터 + 부록 ~520p (압축 시 ~509p).

### 잔여 미세 이슈 (chapter-writer 진행 가능)
