# Toby 문체 확장 가이드

프로젝트 루트의 `toby-book-writing-style.md`를 기반으로, 저술 실무에서 자주 흔들리는 지점들을 보강한 가이드다.

## 목차

1. 문체의 정체성
2. 자주 쓰는 표현 사전 (확장)
3. 챕터 구조별 어조 조정
4. 코드·기술 설명 시 톤
5. 해서는 안 될 표현
6. 첨삭 예시

---

## 1. 문체의 정체성

Toby 문체의 본질은 **"나와 함께 생각하는 선배 개발자"**의 목소리다. 세 가지 축으로 설명할 수 있다.

| 축 | 의미 |
|----|------|
| 동반자적 어조 | 강의가 아니라 함께 걷는 여정. `-자`, `-보자` 적극 사용 |
| 사고 자극 | 답을 던지기 전 질문을 먼저 던짐. 독자가 스스로 탐색하도록 유도 |
| 감정적 공감 | 기술서라도 감각을 잃지 않음. 좋지 않은 코드에 "찜찜하다"고 표현 |

## 2. 자주 쓰는 표현 사전 (확장)

### 사고 유도 (장 도입부에 자주)
- 살펴보자 / 알아보자 / 생각해보자
- 함께 고민해보자 / 파헤쳐보자
- 잠시 멈추고 생각해보자

### 상황 가정 (본문 진입부)
- ~라고 상상해보자
- ~하는 상황을 생각해보자
- ~라고 해보자
- 당신이 ~를 맡았다고 가정해보자

### 수사적 질문 (논리 전환)
- 왜 그럴까?
- 그렇다면 어떻게 해야 할까?
- 정말 그것으로 충분할까?
- 이 방식에 문제는 없을까?

### 권장 (강압 금지)
- ~하는 편이 낫다
- ~하는 것이 바람직하다
- ~하는 게 좋다
- 가능하면 ~을 따르자

### 당부 (핵심 직전)
- 기억해두자
- 잊지 말자
- 주의해야 한다
- 이 점이 중요하다

### 문제 상황 묘사 (감정적 단어)
- 난감하다 / 초난감한 상황이다
- 찜찜하다 / 뒷맛이 찜찜하다
- 번거롭다
- 끔찍한 일이다
- 아찔하다

### 논리 전환·부연
- 그렇다면 ~
- 물론 ~다. 하지만 ~
- 여기서 한 가지 의문이 생긴다
- 조금 더 깊이 들어가보자

## 3. 챕터 구조별 어조 조정

| 섹션 | 톤 권장 |
|------|---------|
| 오프닝 | 상황 가정 또는 수사적 질문으로 진입. 독자를 몰입시킨다 |
| 개념 도입 | 친절한 평어체 + 일상 비유. "~라고 생각하면 이해하기 쉽다" |
| 문제 지적 | 감정적 공감 단어 배치. 독자가 "맞아, 나도 그랬어" 느끼도록 |
| 해결 제시 | 청유형으로 "~해보자". 강압 없이 초대 |
| 마무리 | "기억해두자" 식 당부 + 다음 장 예고 |

## 4. 코드·기술 설명 시 톤

기술서라도 톤을 지킨다.

**나쁜 예:**
```
이 메서드는 리스트의 첫 번째 요소를 반환한다. 파라미터로 빈 리스트가 전달되면 IndexError가 발생한다.
```

**좋은 예:**
```
이 메서드는 리스트의 첫 번째 요소를 돌려준다. 그런데 빈 리스트가 들어오면 어떻게 될까? 아쉽게도 `IndexError`가 발생한다. 이 점은 꼭 기억해두자.
```

**포인트:**
- 사실만 나열하지 말고 "그런데 ~"로 흐름을 만든다
- 감탄(아쉽게도, 놀랍게도)과 당부(기억해두자)를 배치한다

## 5. 해서는 안 될 표현

| 피해야 할 | 대안 |
|----------|------|
| "반드시 ~해야 한다" | "~하는 편이 낫다", "~하자" |
| "이번 장에서는 ~를 다룬다" | 바로 상황·질문으로 진입 |
| "결론부터 말하자면 ~이다" | "그렇다면 답은 어디에 있을까?" |
| 영어식 수동태 남발 | 능동 평어체로 |
| "상기한 바와 같이" | "앞서 살펴봤듯이" |
| "프로그래머블한 어프로치" 같은 외래어 범벅 | 한국어로 자연스럽게 |

## 6. 첨삭 예시

### 원문 (고압적·건조)
> 의존성 주입은 객체 간 결합도를 낮추기 위한 설계 패턴이다. 생성자 주입이 세터 주입보다 권장된다.

### 토비 스타일로
> 의존성 주입이라는 말이 조금 딱딱하게 들릴 수 있다. 하지만 개념 자체는 단순하다. 객체가 자기에게 필요한 다른 객체를 직접 만들지 않고, 외부에서 건네받는 방식이다. 왜 이렇게 해야 할까? 객체끼리 너무 단단히 얽혀 있으면 나중에 고치기가 정말 난감해지기 때문이다. 그렇다면 어떻게 주입해야 할까? 세터보다는 생성자로 받는 편이 낫다. 기억해두자.

### 원문 (나열식)
> 캐시의 장점은 1) 응답 속도 향상, 2) DB 부하 감소, 3) 비용 절감이다.

### 토비 스타일로
> 캐시를 쓰면 무엇이 좋을까? 먼저 응답 속도가 눈에 띄게 빨라진다. DB에 매번 물어보지 않아도 되니 DB 부하도 줄어든다. 게다가 비용까지 절감된다. 하나를 얻으려다 셋이 따라오는 셈이다.
